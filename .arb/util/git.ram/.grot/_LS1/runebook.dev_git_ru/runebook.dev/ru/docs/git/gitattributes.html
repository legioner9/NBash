gitattributes

Name

gitattributes — определение атрибутов для каждого пути

Synopsis

$GIT_DIR/info/attributes, .gitattributes

Description

Файл gitattributes представляет собой простой текстовый файл, который присваивает именам путей attributes .

Каждая строка в файле gitattributes имеет вид:

pattern attr1 attr2 ...

То есть шаблон, за которым следует список атрибутов, разделенный пробелами. Ведущие и конечные пробелы игнорируются. Строки, начинающиеся с # , игнорируются. Шаблоны, начинающиеся с двойной кавычки, цитируются в стиле C. Если шаблон соответствует рассматриваемому пути, атрибуты, перечисленные в строке, присваиваются этому пути.

Каждый атрибут может находиться в одном из этих состояний для данного пути:

Set

Путь имеет атрибут со специальным значением "true" ; это указывается путем указания только имени атрибута в списке атрибутов.

Unset

Путь имеет атрибут со специальным значением "false" ; это указывается путем указания имени атрибута с префиксом тире - в списке атрибутов.

Установить значение

Путь имеет атрибут с указанным строковым значением; это указывается путем указания имени атрибута, за которым следует знак равенства = , и его значения в списке атрибутов.

Unspecified

Ни один шаблон не соответствует пути, и ничего не говорит о том, имеет ли путь атрибут или нет, атрибут пути считается неопределенным.

Если пути соответствует более одного шаблона, более поздняя строка переопределяет более раннюю. Это переопределение выполняется для каждого атрибута.

Правила сопоставления путей с шаблоном такие же, как и в файлах .gitignore (см. gitignore[5] ), за некоторыми исключениями:

  • негативные модели запрещены

  • шаблоны, соответствующие каталогу, не сопоставляют рекурсивно пути внутри этого каталога (поэтому использование синтаксиса косой черты path/ в файле атрибутов бессмысленно; вместо этого используйте path/** )

При принятии решения о том, какие атрибуты присвоены пути, Git обращается к файлу $GIT_DIR/info/attributes (который имеет наивысший приоритет), файлу .gitattributes в том же каталоге, что и рассматриваемый путь, и его родительским каталогам вплоть до верхнего уровня рабочего дерева (чем дальше каталог, содержащий .gitattributes , находится по указанному пути, тем ниже его приоритет). Наконец, учитываются глобальные и общесистемные файлы (они имеют самый низкий приоритет).

Если файл .gitattributes отсутствует в рабочем дереве, путь в индексе используется как запасной вариант. В процессе оформления заказа используется индекс .gitattributes , а затем файл в рабочем дереве используется в качестве резервного варианта.

Если вы хотите повлиять только на один репозиторий (т. е. назначить атрибуты файлам, которые относятся к рабочему процессу одного пользователя в этом репозитории), атрибуты следует поместить в файл $GIT_DIR/info/attributes . Атрибуты, которые должны контролироваться версиями и распространяться в другие репозитории (т. е. атрибуты, представляющие интерес для всех пользователей), должны помещаться в файлы .gitattributes . Атрибуты, которые должны влиять на все репозитории для одного пользователя, должны быть помещены в файл, указанный в параметре конфигурации core.attributesFile (см. git-config[1] ). Его значение по умолчанию — $XDG_CONFIG_HOME/git/attributes. Если $XDG_CONFIG_HOME не установлен или пуст, вместо него используется $HOME/.config/git/attributes. Атрибуты всех пользователей системы должны быть помещены в файл $(prefix)/etc/gitattributes .

Иногда вам может потребоваться переопределить настройку атрибута пути к состоянию Unspecified . Это можно сделать, указав имя атрибута с префиксом восклицательного знака ! .

Effects

На некоторые операции Git можно влиять путем назначения определенных атрибутов пути. В настоящее время следующие операции поддерживают атрибуты.

Выезд и заезд

Эти атрибуты влияют на то, как содержимое, хранящееся в репозитории, копируется в файлы рабочего дерева при запуске таких команд, как git switch , git checkout и git merge . Они также влияют на то, как Git сохраняет подготовленное вами содержимое в рабочем дереве репозитория на git add и git commit .

text

Этот атрибут помечает путь как текстовый файл, что обеспечивает преобразование конца строки: когда соответствующий файл добавляется в индекс, окончания строк файла нормализуются в индексе до LF. И наоборот, когда файл копируется из индекса в рабочий каталог, окончания его строк могут быть преобразованы из LF в CRLF в зависимости от атрибута eol , конфигурации Git и платформы (см. объяснение eol ниже).

Set

Установка атрибута text в пути включает преобразование конца строки при возврате и извлечении, как описано выше. Окончания строк нормализуются до LF в индексе каждый раз, когда файл возвращается, даже если файл ранее был добавлен в Git с окончаниями строк CRLF.

Unset

Удаление атрибута text для пути указывает Git не пытаться выполнить какое-либо преобразование конца строки при возврате или оформлении заказа.

Установить строковое значение "auto" .

Если для text установлено значение "auto" , Git самостоятельно решает, является ли файл текстовым или двоичным. Если это текст и файл еще не был в формате Git с окончаниями CRLF, окончания строк преобразуются при возврате и извлечении, как описано выше. В противном случае конвертация при регистрации или оформлении заказа не производится.

Unspecified

Если атрибут text не указан, Git использует переменную конфигурации core.autocrlf , чтобы определить, следует ли преобразовать файл.

Любое другое значение приводит к тому, что Git будет действовать так, как если бы text не был указан.

eol

Этот атрибут отмечает путь для использования определенного стиля окончания строки в рабочем дереве при его извлечении. Это имеет эффект, только если установлен text или text=auto (см. выше), но указание eol автоматически устанавливает text , если text не указан.

Установить строковое значение "crlf" .

Этот параметр преобразует окончания строк файла в рабочем каталоге в CRLF при извлечении файла.

Установить строковое значение "lf" .

Этот параметр использует те же окончания строк в рабочем каталоге, что и в индексе при извлечении файла.

Unspecified

Если атрибут eol для файла не указан, окончания его строк в рабочем каталоге определяются переменной конфигурации core.autocrlf или core.eol (см. определения этих параметров в git-config[1] ). Если text установлен, но ни одна из этих переменных не задана, значением по умолчанию является eol=crlf в Windows и eol=lf на всех других платформах.

Обратная совместимость с атрибутом crlf .

Для обратной совместимости атрибут crlf интерпретируется следующим образом:

crlf                text
-crlf                -text
crlf=input        eol=lf

End-of-line conversion

Хотя Git обычно оставляет содержимое файла без изменений, его можно настроить на нормализацию концов строк в LF в репозитории и, при необходимости, на преобразование их в CRLF при извлечении файлов.

Если вы просто хотите, чтобы в вашем рабочем каталоге заканчивались строки CRLF независимо от репозитория, с которым вы работаете, вы можете установить переменную конфигурации "core.autocrlf" без использования каких-либо атрибутов.

[core]
        autocrlf = true

Это не приводит к принудительной нормализации текстовых файлов, но гарантирует, что текстовые файлы, которые вы добавляете в репозиторий, имеют окончания строк, нормализованные до LF при их добавлении, и что файлы, которые уже нормализованы в репозитории, остаются нормализованными.

Если вы хотите гарантировать, что текстовые файлы, которые любой участник добавляет в репозиторий, имеют нормализованные окончания строк, вы можете установить для атрибута text значение "auto" для файлов all .

*        text=auto

Атрибуты позволяют детально контролировать преобразование концов строк. Вот пример, который заставит Git нормализовать файлы .txt, .vcproj и .sh, гарантировать, что файлы .vcproj имеют CRLF, а файлы .sh имеют LF в рабочем каталоге, и предотвратить нормализацию файлов .jpg независимо от их содержимого.

*               text=auto
*.txt                text
*.vcproj        text eol=crlf
*.sh                text eol=lf
*.jpg                -text
Note
 Если преобразование text=auto включено в кросс-платформенном проекте с использованием принудительной отправки в центральный репозиторий, текстовые файлы, содержащие CRLF, должны быть нормализованы.

Из чистого рабочего каталога:

$ echo "* text=auto" >.gitattributes
$ git add --renormalize .
$ git status        # Показать файлы, которые будут нормализованы
$ git commit -m "Introduce end-of-line normalization"

Если в git status появляются какие-либо файлы, которые не следует нормализовать, снимите их атрибут text перед запуском git add -u .

manual.pdf        -text

И наоборот, для текстовых файлов, которые Git не обнаруживает, нормализацию можно включить вручную.

weirdchars.txt        text

Если для core.safecrlf установлено значение "true" или "warn" , Git проверяет, является ли преобразование обратимым для текущей настройки core.autocrlf . Для "true" Git отклоняет необратимые преобразования; для "warn" Git только печатает предупреждение, но принимает необратимое преобразование. Триггеры безопасности предотвращают такое преобразование файлов в рабочем дереве, но есть несколько исключений. Хотя…​

  • Сам git add не трогает файлы в дереве работы, следующая проверка будет, поэтому срабатывает безопасность;

  • git apply для обновления текстового файла с помощью исправления не затрагивает файлы в рабочем дереве, но операция касается текстовых файлов, а преобразование CRLF связано с исправлением несоответствий в конце строки, поэтому безопасность не срабатывает;

  • Сам git diff не трогает файлы в дереве работы, он часто запускается для проверки изменений, которые вы собираетесь внести в следующий git add . Для раннего обнаружения потенциальных проблем срабатывают триггеры безопасности.

working-tree-encoding

Git распознает файлы, закодированные в ASCII или одном из его расширенных наборов (например, UTF-8, ISO-8859-1, …​), как текстовые файлы. Файлы, закодированные в некоторых других кодировках (например, UTF-16), интерпретируются как двоичные, и, следовательно, встроенные инструменты обработки текста Git (например, git diff ), а также большинство веб-интерфейсов Git по умолчанию не визуализируют содержимое этих файлов.

В этих случаях вы можете указать Git кодировку файла в рабочем каталоге с помощью атрибута working-tree-encoding . Если файл с этим атрибутом добавляется в Git , то Git перекодирует содержимое из указанной кодировки в UTF-8. Наконец, Git сохраняет содержимое в кодировке UTF-8 в своей внутренней структуре данных (называемой "the index" ). При оформлении заказа содержимое перекодируется обратно в указанную кодировку.

Обратите внимание, что использование атрибута working-tree-encoding может иметь ряд подводных камней:

  • Альтернативные реализации Git (например, J Git или libgit2) и более старые версии Git (по состоянию на март 2018 г.) не поддерживают атрибут working-tree-encoding . Если вы решите использовать атрибут working-tree-encoding в своем репозитории, то настоятельно рекомендуется убедиться, что все клиенты, работающие с репозиторием, поддерживают его.

    Например, файлы ресурсов Microsoft Visual Studio ( *.rc ) или файлы сценариев PowerShell ( *.ps1 ) иногда кодируются в UTF-16. Если вы объявите *.ps1 в виде файлов как UTF-16 и добавите foo.ps1 с клиентом Git с поддержкой working-tree-encoding , тогда foo.ps1 будет храниться внутри как UTF-8. Клиент без поддержки working-tree-encoding будет извлекать foo.ps1 как файл в кодировке UTF-8. Обычно это вызывает проблемы у пользователей этого файла.

    Если клиент Git , который не поддерживает атрибут working-tree-encoding , добавит новый файл bar.ps1 , то bar.ps1 будет храниться внутри "as-is" (в этом примере, вероятно, как UTF-16). Клиент с поддержкой working-tree-encoding интерпретирует внутреннее содержимое как UTF-8 и попытается преобразовать его в UTF-16 при оформлении заказа. Эта операция завершится неудачно и вызовет ошибку.

  • Перекодирование содержимого в кодировки, отличные от UTF, может привести к ошибкам, поскольку преобразование может быть небезопасным для UTF-8. Если вы подозреваете, что ваша кодировка небезопасна для передачи туда и обратно, добавьте ее в core.checkRoundtripEncoding , чтобы Git проверил кодировку туда и обратно (см. git-config[1] ). SHIFT-JIS (набор японских символов), как известно, имеет проблемы с UTF-8 и проверен по умолчанию.

  • Для перекодирования контента требуются ресурсы, которые могут замедлить определенные операции Git (например, git checkout или git add ).

Используйте атрибут working-tree-encoding только в том случае, если вы не можете сохранить файл в кодировке UTF-8 и хотите, чтобы Git мог обрабатывать содержимое как текст.

В качестве примера используйте следующие атрибуты, если ваши файлы *.ps1 имеют кодировку UTF-16 с меткой порядка байтов (BOM) и вы хотите, чтобы Git выполнял автоматическое преобразование конца строки в зависимости от вашей платформы.

*.ps1                text working-tree-encoding=UTF-16

Используйте следующие атрибуты, если ваши файлы *.ps1 имеют кодировку UTF-16 с прямым порядком байтов без спецификации, и вы хотите, чтобы Git использовал окончания строк Windows в рабочем каталоге (используйте UTF-16LE-BOM вместо UTF-16LE , если вы хотите, чтобы со спецификацией UTF-16 с прямым порядком байтов). Обратите внимание: во избежание двусмысленности настоятельно рекомендуется явно определять окончания строк с помощью eol , если используется атрибут working-tree-encoding .

*.ps1                text working-tree-encoding=UTF-16LE eol=CRLF

Вы можете получить список всех доступных кодировок на вашей платформе с помощью следующей команды:

iconv --list

Если вы не знаете кодировку файла, то угадать кодировку можно с помощью команды file :

file foo.ps1

ident

Если для пути установлен атрибут ident , Git заменяет $Id$ в объекте BLOB-объекта на $Id: , за которым следует 40-значное шестнадцатеричное имя объекта BLOB-объекта, за которым следует знак доллара $ при оформлении заказа. Любая последовательность байтов, которая начинается с $Id: и заканчивается $ в файле рабочего дерева, заменяется на $Id$ при регистрации.

filter

Атрибуту filter можно присвоить строковое значение, которое называет драйвер фильтра, указанный в конфигурации.

Драйвер фильтра состоит из команды clean и команды smudge , любую из которых можно не указывать. При оформлении заказа, если указана команда smudge , команде передается объект blob со стандартного ввода, а ее стандартный вывод используется для обновления файла рабочего дерева. Аналогично, команда clean используется для преобразования содержимого файла рабочего дерева при возврате. По умолчанию эти команды обрабатывают только один большой двоичный объект и завершают работу. Если вместо фильтров clean и/или smudge используется долго работающий фильтр process , то Git может обрабатывать все большие двоичные объекты с помощью одного вызова команды фильтра в течение всего срока действия одной команды Git , например git add --all . Если настроен долго работающий фильтр process , он всегда имеет приоритет над настроенным фильтром с одним BLOB-объектом. См. раздел ниже для описания протокола, используемого для связи с фильтром process .

Одним из применений фильтрации контента является придание контенту формы, более удобной для использования платформой, файловой системой и пользователем. Для этого режима работы ключевая фраза здесь "more convenient" а не "turning something unusable into usable" . Другими словами, цель состоит в том, чтобы, если кто-то отменяет определение драйвера фильтра или не имеет соответствующей программы фильтра, проект все равно можно будет использовать.

Другое использование фильтрации контента — хранить в репозитории контент, который нельзя использовать напрямую (например, UUID, который ссылается на контент true , хранящийся вне Git, или зашифрованный контент) и превращать его в удобную форму при оформлении заказа (например, загрузка внешний контент или расшифровать зашифрованный контент).

Эти два фильтра ведут себя по-разному, и по умолчанию в качестве первого принимается фильтр, придающий содержимому более удобную форму. Отсутствие определения драйвера фильтра в конфигурации или драйвер фильтра, который завершает работу с ненулевым статусом, не является ошибкой, но делает фильтр неработающим.

Вы можете объявить, что фильтр превращает контент, который сам по себе непригоден для использования, в полезный контент, установив для переменной конфигурации filter.<driver>.required значение true .

Примечание. При каждом изменении чистого фильтра репозиторий следует перенормировать: $ git add --renormalize .

Например, в .gitattributes вы можете назначить атрибут filter для путей.

*.c        filter=indent

Затем вы должны определить конфигурации "filter.indent.clean" и "filter.indent.smudge" в вашем .git/config, чтобы указать пару команд для изменения содержимого программ C, когда исходные файлы возвращаются (запускается "clean" ) и извлекаются (никаких изменений не происходит, поскольку команда — "cat" ).

[filter "indent"]
        clean = indent
        smudge = cat

Для достижения наилучших результатов clean не должен дополнительно изменять свои выходные данные, если он запускается дважды ( "clean→clean" должен быть эквивалентен "clean" ), а несколько команд smudge не должны изменять выходные данные clean ( "smudge→smudge→clean" должны быть эквивалентны "clean" ). См. раздел об объединении ниже.

Фильтр "indent" в этом отношении работает хорошо: он не будет изменять входные данные, которые уже имеют правильный отступ. В этом случае отсутствие фильтра загрязнения означает, что чистый фильтр must принимает собственный выходной сигнал, не изменяя его.

Если фильтру must удалось сделать сохраненное содержимое пригодным для использования, вы можете объявить, что это фильтр required , в конфигурации:

[filter "crypt"]
        clean = openssl enc ...
        smudge = openssl enc -d ...
        required

Последовательность "%f" в командной строке фильтра заменяется именем файла, над которым работает фильтр. Фильтр может использовать это при замене ключевых слов. Например:

[filter "p4"]
        clean = git-p4-filter --clean %f
        smudge = git-p4-filter --smudge %f

Обратите внимание, что "%f" — это имя пути, над которым ведется работа. В зависимости от фильтруемой версии соответствующий файл на диске может отсутствовать или иметь другое содержимое. Таким образом, команды smudge и clean не должны пытаться получить доступ к файлу на диске, а лишь действовать как фильтры для содержимого, предоставляемого им на стандартном вводе.

Длительный процесс фильтрации

Если команда фильтра (строковое значение) определена через filter.<driver>.process , то Git может обрабатывать все большие двоичные объекты с помощью одного вызова фильтра в течение всего срока действия одной команды Git . Это достигается за счет использования протокола длительного процесса (описанного в файлеtechnic/long-running-process-protocol.txt).

Когда Git обнаруживает первый файл, который необходимо очистить или стереть, он запускает фильтр и выполняет подтверждение связи. При рукопожатии приветственное сообщение, отправленное Git , — "git-filter-client" , поддерживается только версия 2, а поддерживаемые возможности — "clean" , "smudge" и "delay" .

После этого Git отправляет список пар "key=value" , завершающийся пакетом сброса. Список будет содержать как минимум команду фильтра (в зависимости от поддерживаемых возможностей) и путь к файлу для фильтрации относительно корня репозитория. Сразу после пакета очистки Git отправляет контент, разделенный на ноль или более пакетов pkt-line, а также пакет очистки для завершения содержимого. Обратите внимание, что фильтр не должен отправлять какой-либо ответ до того, как он получит содержимое и последний пакет сброса. Также обратите внимание, что "value" пары "key=value" может содержать символ "=" , тогда как ключ никогда не будет содержать этот символ.

packet:          git> command=smudge
packet:          git> pathname=path/testfile.dat
packet:          git> 0000
packet:          git> CONTENT
packet:          git> 0000

Ожидается, что фильтр ответит списком пар "key=value" , завершающимся пакетом сброса. Если с фильтром проблем нет, список должен содержать статус "success" . Ожидается, что сразу после этих пакетов фильтр отправит содержимое в виде нуля или более пакетов pkt-line и пакета сброса в конце. Наконец, ожидается второй список пар "key=value" , завершающийся пакетом сброса. Фильтр может изменить статус во втором списке или оставить статус как есть при пустом списке. Обратите внимание, что пустой список в любом случае должен завершаться пакетом сброса.

packet:          git< status=success
packet:          git< 0000
packet:          git< SMUDGED_CONTENT
packet:          git< 0000
packet:          git< 0000  # empty list, keep "status=success" unchanged!

Если содержимое результата пусто, то ожидается, что фильтр ответит статусом "success" и пакетом очистки, сигнализирующим о пустом содержимом.

packet:          git< status=success
packet:          git< 0000
packet:          git< 0000  # empty content!
packet:          git< 0000  # empty list, keep "status=success" unchanged!

Если фильтр не может или не хочет обрабатывать контент, ожидается, что он ответит статусом "error" .

packet:          git< status=error
packet:          git< 0000

Если во время обработки в фильтре возникает ошибка, он может отправить статус "error" после того, как контент был (частично или полностью) отправлен.

packet:          git< status=success
packet:          git< 0000
packet:          git< HALF_WRITTEN_ERRONEOUS_CONTENT
packet:          git< 0000
packet:          git< status=error
packet:          git< 0000

Если фильтр не может или не хочет обрабатывать контент, а также любой будущий контент в течение всего срока действия процесса Git , ожидается, что он ответит статусом "abort" в любой точке протокола.

packet:          git< status=abort
packet:          git< 0000

Git не останавливает и не перезапускает процесс фильтрации, если установлен статус "error" / "abort" . Однако Git устанавливает свой код выхода в соответствии с флагом filter.<driver>.required , имитируя поведение механизма filter.<driver>.clean / filter.<driver>.smudge .

Если фильтр выходит из строя во время связи или не соответствует протоколу, Git остановит процесс фильтрации и перезапустит его со следующим файлом, который необходимо обработать. В зависимости от флага filter.<driver>.required Git интерпретирует это как ошибку.

Delay

Если фильтр поддерживает возможность "delay" , то Git может отправить флаг "can-delay" после команды фильтра и имени пути. Этот флаг означает, что фильтр может задержать фильтрацию текущего большого двоичного объекта (например, для компенсации задержек в сети), ответив без содержимого, но со статусом "delayed" и пакетом сброса.

packet:          git> command=smudge
packet:          git> pathname=path/testfile.dat
packet:          git> can-delay=1
packet:          git> 0000
packet:          git> CONTENT
packet:          git> 0000
packet:          git< status=delayed
packet:          git< 0000

Если фильтр поддерживает функцию "delay" , он должен поддерживать команду "list_available_blobs" . Если Git отправляет эту команду, то ожидается, что фильтр вернет список имен путей, представляющих большие двоичные объекты, которые были задержаны ранее и теперь доступны. Список должен завершаться пакетом сброса, за которым следует статус "success" , который также завершается пакетом сброса. Если двоичные объекты для задержанных путей еще не доступны, то ожидается, что фильтр будет блокировать ответ до тех пор, пока не станет доступен хотя бы один большой двоичный объект. Фильтр может сообщить Git , что у него больше нет задержанных больших двоичных объектов, отправив пустой список. Как только фильтр ответит пустым списком, Git перестанет спрашивать. Все большие двоичные объекты, которые Git не получил на этом этапе, считаются отсутствующими и приведут к ошибке.

packet:          git> command=list_available_blobs
packet:          git> 0000
packet:          git< pathname=path/testfile.dat
packet:          git< pathname=path/otherfile.dat
packet:          git< 0000
packet:          git< status=success
packet:          git< 0000

После того как Git получил имена путей, он снова запросит соответствующие BLOB-объекты. Эти запросы содержат путь и пустой раздел содержимого. Ожидается, что фильтр отреагирует размытым содержимым обычным способом, как описано выше.

packet:          git> command=smudge
packet:          git> pathname=path/testfile.dat
packet:          git> 0000
packet:          git> 0000  # пустой контент!
packet:          git< status=success
packet:          git< 0000
packet:          git< SMUDGED_CONTENT
packet:          git< 0000
packet:          git< 0000  # пустой список, оставьте  "status=success"  без изменений!

Example

Давно работающую демонстрационную реализацию фильтра можно найти в contrib/long-running-filter/example.pl , расположенном в основном репозитории Git . Если вы разрабатываете свой собственный длительный процесс фильтрации, переменные среды GIT_TRACE_PACKET могут быть очень полезны для отладки (см. git[1] ).

Обратите внимание, что вы не можете использовать существующую команду filter.<driver>.clean или filter.<driver>.smudge с filter.<driver>.process , поскольку первые две используют другой протокол межпроцессного взаимодействия, чем последний.

Взаимодействие между атрибутами возврата и оформления заказа

В кодовом пути возврата файл рабочего дерева сначала преобразуется с помощью драйвера filter (если он указан и определен соответствующий драйвер), затем результат обрабатывается с помощью ident (если указан), а затем, наконец, с помощью text (опять же, если он указан и применим). .

В кодовом пути проверки содержимое больших двоичных объектов сначала преобразуется с помощью text , а затем ident и передается в filter .

Объединение ветвей с разными атрибутами возврата/выписки

Если вы добавили в файл атрибуты, которые приводят к изменению канонического формата репозитория для этого файла, например, добавление фильтра очистки/размазывания или атрибутов text/eol/ident, слияние всего, где атрибут отсутствует, обычно приводит к конфликтам слияния. .

Чтобы предотвратить эти ненужные конфликты слияния, Git можно указать выполнить виртуальное извлечение и возврат всех трех этапов файла при разрешении трехстороннего слияния, установив переменную конфигурации merge.renormalize . Это предотвращает возникновение ложных конфликтов слияния, вызванных изменениями, вызванными возвратным преобразованием, когда преобразованный файл объединяется с неконвертированным файлом.

Пока "smudge→clean" дает тот же результат, что и "clean" , даже для файлов, которые уже испачканы, эта стратегия автоматически разрешает все конфликты, связанные с фильтрами. Фильтры, которые не действуют таким образом, могут вызвать дополнительные конфликты слияния, которые придется разрешать вручную.

Генерация текста различий

diff

Атрибут diff влияет на то, как Git генерирует различия для определенных файлов. Он может указать Git , следует ли генерировать текстовый патч для пути или рассматривать путь как двоичный файл. Это также может повлиять на то, какая строка отображается в строке заголовка фрагмента @@ -k,l +n,m @@ , указать Git использовать внешнюю команду для создания различий или попросить Git преобразовать двоичные файлы в текстовый формат перед созданием различий.

Set

Путь, для которого установлен атрибут diff , рассматривается как текст, даже если он содержит байтовые значения, которые обычно никогда не появляются в текстовых файлах, например NUL.

Unset

Путь, для которого не установлен атрибут diff , будет генерировать Binary files differ (или двоичное исправление, если двоичные исправления включены).

Unspecified

Путь, для которого атрибут diff не указан, сначала проверяется его содержимое, и если он выглядит как текст и меньше, чем core.bigFileThreshold, он обрабатывается как текст. В противном случае будет сгенерирован Binary files differ .

String

Разница отображается с использованием указанного драйвера различий. Каждый драйвер может указать одну или несколько опций, как описано в следующем разделе. Параметры драйвера различий "foo" определяются переменными конфигурации в разделе "diff.foo" файла конфигурации Git .

Определение внешнего драйвера различий

Определение драйвера различий выполнено в файле gitconfig , а не в файле gitattributes , поэтому, строго говоря, эта страница руководства — неподходящее место для обсуждения этого вопроса. Однако…​

Чтобы определить внешний драйвер различий jcdiff , добавьте раздел в файл $GIT_DIR/config (или файл $HOME/.gitconfig ), например:

[diff "jcdiff"]
        command = j-c-diff

Когда Git необходимо показать вам разницу для пути с атрибутом diff , установленным в jcdiff , он вызывает указанную вами команду с вышеуказанной конфигурацией, т.е. j-c-diff , с 7 параметрами, точно так же, как вызывается программа GIT_EXTERNAL_DIFF . Подробности см. в git[1] .

Настройка внутреннего алгоритма сравнения

Алгоритм сравнения можно установить с помощью конфигурационного ключа diff.algorithm , но иногда может оказаться полезным установить алгоритм сравнения для каждого пути. Например, можно использовать алгоритм сравнения minimal для файлов .json и histogram для файлов .c и т. д. без необходимости каждый раз передавать алгоритм через командную строку.

Сначала в .gitattributes назначьте для путей атрибут diff .

*.json diff=<name>

Затем определите конфигурацию "diff.<name>.algorithm" , чтобы указать алгоритм сравнения, выбрав myers , patience , minimal или histogram .

[diff "<name>"]
  algorithm = histogram

Этот алгоритм сравнения применяется к выводам различий, обращенным к пользователю, таким как git-diff(1), git-show(1), а также используется для вывода --stat . Механизм слияния не будет использовать алгоритм сравнения, установленный с помощью этого метода.

Note
 Если diff.<name>.command определен для пути с атрибутом diff=<name> , он выполняется как внешний драйвер сравнения (см. выше), и добавление diff.<name>.algorithm не имеет никакого эффекта, поскольку алгоритм не передается внешнему драйверу сравнения.

Определение пользовательского заголовка фрагмента

Каждая группа изменений (называемая "hunk" ) в текстовых выводах различий предваряется строкой следующего вида:

@@ -k,l +n,m @@ TEXT

Это называется hunk header . Часть "TEXT" по умолчанию представляет собой строку, начинающуюся с алфавита, подчеркивания или знака доллара; это соответствует тому, что использует вывод GNU diff -p . Однако этот выбор по умолчанию не подходит для некоторого содержимого, и для выбора можно использовать собственный шаблон.

Во-первых, в .gitattributes вы должны назначить атрибут diff для путей.

*.tex        diff=tex

Затем вы должны определить конфигурацию "diff.tex.xfuncname" , чтобы указать регулярное выражение, соответствующее строке, которую вы хотите отображать в качестве заголовка фрагмента "TEXT" . Добавьте раздел в файл $GIT_DIR/config (или файл $HOME/.gitconfig ) следующим образом:

[diff "tex"]
        xfuncname = "^(\\\\(sub)*section\\{.*)$"

Примечание. Анализатор файла конфигурации обрабатывает один уровень обратной косой черты, поэтому вам придется удвоить обратную косую черту; шаблон выше выбирает строку, которая начинается с обратной косой черты и ноль или более вхождений sub , за которым следует section , за которым следует открывающая скобка до конца строки.

Есть несколько встроенных шаблонов, упрощающих это, и tex — один из них, поэтому вам не нужно писать все вышеперечисленное в файле конфигурации (вам все равно нужно включить это с помощью механизма атрибутов через .gitattributes ). Доступны следующие встроенные шаблоны:

  • ada подходит для исходного кода на языке Ada.

  • bash подходит для исходного кода на языке Bourne-Again SHell. Охватывает расширенный набор определений функций оболочки POSIX.

  • bibtex подходит для файлов со ссылками в кодировке BibTeX.

  • cpp подходит для исходного кода на языках C и C++.

  • csharp подходит для исходного кода на языке C#.

  • css подходит для каскадных таблиц стилей.

  • dts подходит для файлов дерева устройств (DTS).

  • elixir подходит для исходного кода на языке Elixir.

  • fortran подходит для исходного кода на языке Фортран.

  • fountain подходит для документов Fountain.

  • golang подходит для исходного кода на языке Go.

  • html подходит для документов HTML/XHTML.

  • java подходит для исходного кода на языке Java.

  • kotlin подходит для исходного кода на языке Kotlin.

  • markdown подходит для документов Markdown.

  • matlab подходит для исходного кода на языках MATLAB и Octave.

  • objc подходит для исходного кода на языке Objective-C.

  • pascal подходит для исходного кода на языке Pascal/Delphi.

  • perl подходит для исходного кода на языке Perl.

  • php подходит для исходного кода на языке PHP.

  • python подходит для исходного кода на языке Python .

  • ruby подходит для исходного кода на языке Ruby.

  • rust подходит для исходного кода на языке Rust.

  • scheme подходит для исходного кода на языке Scheme.

Настройка разницы слов

Вы можете настроить правила, которые git diff --word-diff использует для разделения слов в строке, указав соответствующее регулярное выражение в переменной конфигурации "diff.*.wordRegex" . Например, в TeX обратная косая черта, за которой следует последовательность букв, образует команду, но несколько таких команд можно запускать вместе без промежуточных пробелов. Чтобы разделить их, используйте регулярное выражение в файле $GIT_DIR/config (или файле $HOME/.gitconfig ), например:

[diff "tex"]
        wordRegex = "\\\\[a-zA-Z]+|[{}]|\\\\.|[^\\{}[:space:]]+"

Встроенный шаблон предоставляется для всех языков, перечисленных в предыдущем разделе.

Выполнение текстовых различий двоичных файлов

Иногда желательно увидеть разницу текстовой версии некоторых бинарных файлов. Например, документ текстового процессора можно преобразовать в текстовое представление ASCII и отобразить разницу текста. Несмотря на то, что при этом преобразовании теряется некоторая информация, полученный результат полезен для просмотра человеком (но не может быть применен напрямую).

Опция конфигурации textconv используется для определения программы для выполнения такого преобразования. Программа должна принимать единственный аргумент — имя файла для преобразования и выдавать результирующий текст на стандартный вывод.

Например, чтобы показать разницу с информацией exif файла вместо двоичной информации (при условии, что у вас установлен инструмент exif), добавьте следующий раздел в файл $GIT_DIR/config (или файл $HOME/.gitconfig ):

[diff "jpg"]
        textconv = exif
Note
 Преобразование текста обычно является односторонним преобразованием; в этом примере мы теряем фактическое содержимое изображения и фокусируемся только на текстовых данных. Это означает, что различия, сгенерированные textconv, непригодны для применения. По этой причине только команды git diff и семейства git log (т. е. log, WhatChanged, Show) выполняют преобразование текста. git format-patch никогда не будет генерировать этот вывод. Если вы хотите отправить кому-нибудь преобразованный в текст дифф двоичного файла (например, потому, что он быстро передает внесенные вами изменения), вам следует сгенерировать его отдельно и отправить в качестве комментария в дополнение к обычному двоичному диффу, который вы можете отправлять.

Поскольку преобразование текста может быть медленным, особенно при выполнении большого количества преобразований с помощью git log -p , Git предоставляет механизм кэширования вывода и использования его в будущих различиях. Чтобы включить кэширование, установите переменную "cachetextconv" в конфигурации вашего драйвера различий. Например:

[diff "jpg"]
        textconv = exif
        cachetextconv = true

При этом результат запуска "exif" для каждого большого двоичного объекта будет кэшироваться на неопределенный срок. Если вы измените переменную конфигурации textconv для драйвера различий, Git автоматически аннулирует записи кэша и повторно запустит фильтр textconv. Если вы хотите сделать кеш недействительным вручную (например, потому что ваша версия "exif" была обновлена ​​и теперь выдает лучший результат), вы можете удалить кеш вручную с помощью git update-ref -d refs/notes/textconv/jpg (где "jpg" — это имя драйвера сравнения, как в примере выше). .

Выбор textconv вместо внешнего diff

Если вы хотите показать различия между двоичными или специально отформатированными двоичными объектами в вашем репозитории, вы можете использовать внешнюю команду diff или использовать textconv для преобразования их в текстовый формат с возможностью сравнения. Какой метод вы выберете, зависит от вашей конкретной ситуации.

Преимущество использования внешней команды diff — гибкость. Вы не обязаны находить изменения, ориентированные на строки, и не обязательно, чтобы выходные данные напоминали унифицированный diff. Вы можете находить изменения и сообщать об изменениях наиболее подходящим для вашего формата данных способом.

Текстовое соглашение, по сравнению с этим, имеет гораздо более ограниченные возможности. Вы обеспечиваете преобразование данных в строковый текстовый формат, а Git использует свои обычные инструменты сравнения для генерации выходных данных. Выбор этого метода имеет ряд преимуществ:

  1. Простота использования. Зачастую гораздо проще написать преобразование двоичного кода в текст, чем выполнять собственный анализ различий. Во многих случаях в качестве текстовых фильтров можно использовать существующие программы (например, exif, odt2txt).

  2. Отличительные особенности Git . Выполнив самостоятельно только этап преобразования, вы все равно сможете использовать многие функции сравнения Git , включая раскрашивание, сравнение слов и комбинированные различия для слияний.

  3. Кэширование. Кэширование Textconv может ускорить повторяющиеся различия, например те, которые могут быть вызваны запуском git log -p .

Маркировка файлов как двоичных

Git обычно правильно угадывает, содержит ли большой двоичный объект текстовые или двоичные данные, проверяя начало содержимого. Однако иногда вам может потребоваться отменить его решение, либо потому, что большой двоичный объект содержит двоичные данные позже в файле, либо потому, что содержимое, хотя технически состоит из текстовых символов, непрозрачно для читателя-человека. Например, многие файлы Postscript содержат только символы ASCII, но создают шумные и бессмысленные различия.

Самый простой способ пометить файл как двоичный — отключить атрибут diff в файле .gitattributes :

*.ps -diff

Это приведет к тому, что Git будет генерировать Binary files differ (или двоичный патч, если бинарные исправления включены) вместо обычного diff.

Однако можно также указать другие атрибуты драйвера различий. Например, вы можете использовать textconv для преобразования файлов Postscript в представление ASCII для просмотра человеком, но в противном случае обрабатывать их как двоичные файлы. Вы не можете указать одновременно атрибуты -diff и diff=ps . Решение — использовать опцию конфигурации diff.*.binary :

[diff "ps"]
  textconv = ps2ascii
  binary = true

Выполнение трехстороннего слияния

merge

Атрибут merge влияет на то, как объединяются три версии файла, когда слияние на уровне файла необходимо во время git merge и других команд, таких как git revert и git cherry-pick .

Set

Встроенный драйвер трехстороннего слияния используется для объединения содержимого аналогично команде merge пакета RCS . Это подходит для обычных текстовых файлов.

Unset

Возьмите версию из текущей ветки в качестве предварительного результата слияния и объявите, что слияние содержит конфликты. Это подходит для двоичных файлов, которые не имеют четко определенной семантики слияния.

Unspecified

По умолчанию здесь используется тот же встроенный драйвер трехстороннего слияния, что и в случае, когда установлен атрибут merge . Однако переменная конфигурации merge.default может называть другой драйвер слияния, который будет использоваться с путями, для которых атрибут merge не указан.

String

Трехстороннее слияние выполняется с использованием указанного специального драйвера слияния. Встроенный драйвер трехстороннего слияния можно явно указать, запросив драйвер "text" ; встроенный драйвер "take the current branch" можно запросить с помощью "binary" .

Встроенные драйверы слияния

Определено несколько встроенных низкоуровневых драйверов слияния, которые можно запросить через атрибут merge .

text

Обычное трехстороннее слияние на уровне текстовых файлов. Конфликтные регионы отмечены маркерами конфликта <<<<<<< , ======= и >>>>>>> . Версия из вашей ветки отображается перед маркером ======= , а версия из объединенной ветки — после маркера ======= .

binary

Сохраните версию из своей ветки в рабочем дереве, но оставьте путь в конфликтном состоянии, чтобы пользователь мог разобраться.

union

Запустите трехстороннее слияние на уровне текстовых файлов, но возьмите строки из обеих версий, а не оставляйте маркеры конфликта. При этом добавленные строки в результирующем файле остаются в случайном порядке, и пользователю приходится проверять результат. Не используйте это, если вы не понимаете последствий.

Определение пользовательского драйвера слияния

Определение драйвера слияния выполнено в файле .git/config , а не в файле gitattributes , поэтому, строго говоря, эта страница руководства — неподходящее место для обсуждения этого вопроса. Однако…​

Чтобы определить собственный драйвер слияния filfre , добавьте раздел в файл $GIT_DIR/config (или файл $HOME/.gitconfig ), например:

[merge "filfre"]
        name = feel-free merge driver
        driver = filfre %O %A %B %L %P
        recursive = binary

Переменная merge.*.name дает драйверу удобочитаемое имя.

Значение переменной merge.*.driver используется для создания команды, запускаемой для объединения версии предка ( %O ), текущей версии ( %A ) и версии других ветвей ( %B ). Эти три токена заменяются именами временных файлов, содержащих содержимое этих версий, при построении командной строки. Кроме того, %L будет заменен размером маркера конфликта (см. ниже).

Ожидается, что драйвер слияния оставит результат слияния в файле с именем %A , перезаписав его, и завершит работу с нулевым статусом, если ему удалось объединить их без ошибок, или с ненулевым статусом, если возникли конфликты. Когда драйвер выходит из строя (например, уничтожается SEGV), ожидается, что он завершит работу с ненулевым статусом, превышающим 128, и в таком случае слияние приводит к сбою (что отличается от возникновения конфликта).

Переменная merge.*.recursive указывает, какой еще драйвер слияния следует использовать, когда драйвер слияния вызывается для внутреннего слияния между общими предками, если их несколько. Если этот параметр не указан, сам драйвер используется как для внутреннего, так и для окончательного слияния.

Драйвер слияния может узнать путь, в котором будет сохранен результат слияния, через заполнитель %P .

conflict-marker-size

Этот атрибут управляет длиной маркеров конфликта, остающихся в файле рабочего дерева во время конфликтного слияния. Только положительное целое число имеет значимый эффект.

Например, эту строку в .gitattributes можно использовать, чтобы указать механизму слияния оставлять гораздо более длинные (вместо обычных 7-символьных) маркеры конфликта, когда слияние файла Documentation/git-merge.txt приводит к конфликту.

Documentation/git-merge.txt        conflict-marker-size=32

Проверка ошибок пробелов

whitespace

Переменная конфигурации core.whitespace позволяет вам определить, какие diff и apply должны учитывать ошибки пробелов для всех путей в проекте (см. git-config[1] ). Этот атрибут дает вам более точный контроль над каждым путем.

Set

Обратите внимание на все типы потенциальных ошибок пробелов, известные Git. Ширина табуляции берется из значения переменной конфигурации core.whitespace .

Unset

Ничего не замечайте как ошибку.

Unspecified

Используйте значение переменной конфигурации core.whitespace , чтобы решить, что считать ошибкой.

String

Укажите список распространенных проблем с пробелами, разделенных запятыми, в том же формате, что и переменная конфигурации core.whitespace .

Создание архива

export-ignore

Файлы и каталоги с атрибутом export-ignore не будут добавлены в архивные файлы.

export-subst

Если для файла установлен атрибут export-subst , то Git расширит несколько заполнителей при добавлении этого файла в архив. Расширение зависит от наличия идентификатора фиксации, т. е. если git-archive[1] было присвоено дерево вместо фиксации или тега, то замена не будет выполнена. Заполнители такие же, как и для опции --pretty=format: из git-log[1] , за исключением того, что их нужно обернуть следующим образом: $Format:PLACEHOLDERS$ в файле. Например, строка $Format:%H$ будет заменена хешем фиксации. Однако для каждого архива расширяется только один заполнитель %(describe) , чтобы избежать атак типа «отказ в обслуживании».

Packing objects

delta

Дельта-сжатие не будет выполняться для больших двоичных объектов для путей с атрибутом delta , установленным в значение false.

Просмотр файлов в инструментах графического интерфейса

encoding

Значение этого атрибута определяет кодировку символов, которая должна использоваться инструментами с графическим интерфейсом (например, gitk[1] и git-gui[1] ) для отображения содержимого соответствующего файла. Обратите внимание, что из соображений производительности gitk[1] не использует этот атрибут, если вы вручную не включите кодировку для каждого файла в его параметрах.

Если этот атрибут не установлен или имеет недопустимое значение, вместо него используется значение переменной конфигурации gui.encoding (см. git-config[1] ).

Использование атрибутов макроса

Вы не хотите, чтобы к любому двоичному файлу, который вы отслеживаете, применялись какие-либо преобразования конца строки или создавались текстовые различия. Вам нужно будет указать, например

*.jpg -text -diff

но это может стать обременительным, если у вас много атрибутов. Используя макроатрибуты, вы можете определить атрибут, который, если он установлен, одновременно устанавливает или отменяет настройку ряда других атрибутов. Система знает встроенный атрибут макроса binary :

*.jpg binary

Установка атрибута "binary" также сбрасывает атрибуты "text" и "diff" , как указано выше. Обратите внимание, что атрибуты макроса могут быть только "Set" , хотя установка одного из них может привести к установке или отключению других атрибутов или даже возврату других атрибутов в состояние "Unspecified" .

Определение атрибутов макроса

Пользовательские атрибуты макросов можно определить только в файлах gitattributes верхнего уровня ( $GIT_DIR/info/attributes , файл .gitattributes на верхнем уровне рабочего дерева или глобальные или общесистемные файлы gitattributes), а не в файлах .gitattributes в подкаталогах рабочего дерева. Встроенный атрибут макроса "binary" эквивалентен:

[attr]binary -diff -merge -text

Notes

Git не переходит по символическим ссылкам при доступе к файлу .gitattributes в рабочем дереве. Это обеспечивает единообразие поведения при доступе к файлу из индекса или дерева, а не из файловой системы.

Examples

Если у вас есть эти три файла gitattributes :

(in $GIT_DIR/info/attributes)

a*        foo !bar -baz

(in .gitattributes)
abc        foo bar baz

(in t/.gitattributes)
ab*        merge=filfre
abc        -foo -bar
*.c        frotz

атрибуты, присвоенные пути t/abc , вычисляются следующим образом:

  1. Проверив t/.gitattributes (который находится в том же каталоге, что и рассматриваемый путь), Git обнаруживает, что первая строка совпадает. Атрибут merge установлен. Он также обнаруживает, что вторая строка совпадает, а атрибуты foo и bar не установлены.

  2. Затем он проверяет .gitattributes (который находится в родительском каталоге) и обнаруживает, что первая строка совпадает, но файл t/.gitattributes уже определил, как атрибуты merge , foo и bar должны быть присвоены этому пути, поэтому он оставляет foo и bar неустановленными. Атрибут baz установлен.

  3. Наконец, он исследует $GIT_DIR/info/attributes . Этот файл используется для переопределения настроек в дереве. Первая строка соответствует, и foo установлен, bar возвращается в неопределенное состояние, а baz не установлен.

В результате назначение атрибутов t/abc будет выглядеть следующим образом:

foo        set to true
bar        unspecified
baz        set to false
merge        set to string value "filfre"
frotz        unspecified
