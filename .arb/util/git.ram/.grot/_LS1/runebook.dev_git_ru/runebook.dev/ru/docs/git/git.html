git

Name

git — тупой трекер контента

Synopsis

git [-v | --version] [-h | --help] [-C <path>] [-c <name>=<value>]
    [--exec-path[=<path>]] [--html-path] [--man-path] [--info-path]
    [-p|--paginate|-P|--no-pager] [--no-replace-objects] [--bare]
    [--git-dir=<path>] [--work-tree=<path>] [--namespace=<name>]
    [--config-env=<name>=<envvar>] <command> [<args>]

Description

Git — это быстрая, масштабируемая, распределенная система контроля версий с необычайно богатым набором команд, обеспечивающая как высокоуровневые операции, так и полный доступ к внутренним компонентам.

Для начала см. gittutorial[7] , затем см. giteveryday[7] , где указан минимальный полезный набор команд. Git User’s Manual имеет более подробное описание.

После того, как вы освоили основные понятия, вы можете вернуться на эту страницу, чтобы узнать, какие команды предлагает Git . Вы можете узнать больше об отдельных командах Git с помощью "git help command" . На странице руководства gitcli[7] представлен обзор синтаксиса команд командной строки.

Отформатированную копию последней версии документации Git с гиперссылками можно просмотреть по адресу https://git.github.io/htmldocs/git.html или https://git-scm.com/docs .

Options

-v
--version

Печатает версию пакета Git , из которой взята программа git .

Эта опция внутренне преобразуется в git version ... и принимает те же параметры, что и команда git-version[1] . Если также указан --help , он имеет приоритет над --version .

-h
--help

Печатает краткий обзор и список наиболее часто используемых команд. Если указана опция --all или -a , то печатаются все доступные команды. Если названа команда Git , эта опция откроет страницу руководства для этой команды.

Доступны и другие параметры для управления отображением страницы руководства. См. git-help[1] для получения дополнительной информации, поскольку git --help ... внутренне преобразуется в git help ... .

-C <path>

Запускайте так, как будто git был запущен в <path> , а не в текущем рабочем каталоге. Если задано несколько вариантов -C , каждый последующий неабсолютный -C <path> интерпретируется относительно предыдущего -C <path> . Если <path> присутствует, но пуст, например -C "" , то текущий рабочий каталог остается неизменным.

Этот параметр влияет на параметры, которые ожидают имя пути, например --git-dir и --work-tree , поскольку их интерпретация имен путей будет выполняться относительно рабочего каталога, вызванного параметром -C . Например, следующие вызовы эквивалентны:

git --git-dir=a.git --work-tree=b -C c status
git --git-dir=c/a.git --work-tree=c/b status
-c <name>=<value>

Передайте параметр конфигурации команде. Указанное значение будет переопределять значения из файлов конфигурации. Ожидается, что <имя> будет иметь тот же формат, что указан в git config (подключи, разделенные точками).

Обратите внимание, что пропуск = в git -c foo.bar ... разрешен, и для foo.bar устанавливается значение boolean true (точно так же, как [foo]bar в файле конфигурации). Включение равных, но с пустым значением (например, git -c foo.bar= ... ), устанавливает foo.bar в пустую строку, которую git config --type=bool преобразует в false .

--config-env=<name>=<envvar>

Как и -c <name>=<value> , присвойте переменной конфигурации <name> значение, где <envvar> — это имя переменной среды, из которой можно получить значение. В отличие от -c , здесь нет ярлыка для прямой установки значения в пустую строку, вместо этого сама переменная среды должна быть установлена ​​в пустую строку. Если <envvar> не существует в среде, это ошибка. <envvar> может не содержать знак равенства, чтобы избежать двусмысленности, поскольку <name> содержит его.

Это полезно в случаях, когда вы хотите передать параметры временной конфигурации в git, но делаете это в операционных системах, где другие процессы могут читать вашу командную строку (например, /proc/self/cmdline ), но не вашу среду (например, /proc/self/environ ). Такое поведение используется по умолчанию в Linux, но может отсутствовать в вашей системе.

Обратите внимание, что это может повысить безопасность таких переменных, как http.extraHeader , где конфиденциальная информация является частью значения, но не, например, url.<base>.insteadOf , где конфиденциальная информация может быть частью ключа.

--exec-path[=<path>]

Путь к месту установки основных программ Git . Этим также можно управлять, установив переменную среды GIT_EXEC_PATH. Если путь не указан, git распечатает текущие настройки и затем выйдет.

--html-path

Выведите путь без косой черты, по которому установлена ​​HTML-документация Git, и выйдите.

--man-path

Распечатайте путь руководства (см. man(1) ) к страницам руководства для этой версии Git и выйдите.

--info-path

Распечатайте путь, по которому установлены информационные файлы, документирующие эту версию Git , и выйдите.

-p
--paginate

Направьте весь вывод в less (или, если установлено, в $PAGER), если стандартным выводом является терминал. Это переопределяет параметры конфигурации pager.<cmd> (см. раздел "Configuration Mechanism" ниже).

-P
--no-pager

Не передавайте вывод Git на пейджер.

--git-dir=<path>

Укажите путь к репозиторию (каталог ".git" ). Этим также можно управлять, установив переменную среды GIT_DIR . Это может быть абсолютный или относительный путь к текущему рабочему каталогу.

Указание местоположения каталога ".git" с помощью этой опции (или переменной среды GIT_DIR ) отключает обнаружение репозитория, которое пытается найти каталог с подкаталогом ".git" (именно так обнаруживаются репозиторий и верхний уровень рабочего дерева), и сообщает Git , что вы находитесь на верхнем уровне рабочего дерева. Если вы не находитесь в каталоге верхнего уровня рабочего дерева, вы должны сообщить Git , где находится верхний уровень рабочего дерева, с помощью опции --work-tree=<path> (или переменной среды GIT_WORK_TREE ).

Если вы просто хотите запустить git , как если бы он был запущен в <path> , используйте git -C <path> .

--work-tree=<path>

Задайте путь к рабочему дереву. Это может быть абсолютный путь или путь относительно текущего рабочего каталога. Этим также можно управлять, установив переменную среды GIT_WORK_TREE и переменную конфигурации core.worktree (более подробное обсуждение см. в разделе core.worktree в git-config[1] ).

--namespace=<path>

Установите пространство имен Git . Дополнительную информацию см. в gitnamespaces[7] . Эквивалентно настройке переменной среды GIT_NAMESPACE .

--bare

Рассматривайте репозиторий как пустой репозиторий. Если среда GIT_DIR не установлена, устанавливается текущий рабочий каталог.

--no-replace-objects

Не используйте ссылки на замену для замены объектов Git . См. git-replace[1] для получения дополнительной информации.

--literal-pathspecs

Относитесь к спецификациям путей буквально (т. е. без подстановки и магии путевых спецификаций). Это эквивалентно установке переменной среды GIT_LITERAL_PATHSPECS значения 1 .

--glob-pathspecs

Добавьте магию "glob" во все спецификации путей. Это эквивалентно установке переменной среды GIT_GLOB_PATHSPECS значения 1 . Отключить подстановку для отдельных спецификаций путей можно с помощью магии pathspec ":(literal)" .

--noglob-pathspecs

Добавьте магию "literal" во все спецификации пути. Это эквивалентно установке переменной среды GIT_NOGLOB_PATHSPECS значения 1 . Включение подстановки для отдельных спецификаций пути можно выполнить с помощью магии pathspec ":(glob)" .

--icase-pathspecs

Добавьте магию "icase" во все спецификации пути. Это эквивалентно установке переменной среды GIT_ICASE_PATHSPECS значения 1 .

--no-optional-locks

Не выполняйте дополнительные операции, требующие блокировок. Это эквивалентно настройке GIT_OPTIONAL_LOCKS на 0 .

--list-cmds=group[,group…​]

Список команд по группам. Это внутренняя/экспериментальная опция, которая может быть изменена или удалена в будущем. Поддерживаемые группы: встроенные, parseopt (встроенные команды, использующие параметры синтаксического анализа), основные (все команды в каталоге libexec), другие (все остальные команды в $PATH , имеющие префикс git-), list-<категория> (см. категории в команде). -list.txt), nohelpers (исключить вспомогательные команды), псевдоним и конфигурация (получить список команд из переменной конфигурацииcomplete.commands)

--attr-source=<tree-ish>

Считайте gitattributes из <tree-ish> вместо рабочего дерева. См. gitattributes[5] . Это эквивалентно установке переменной среды GIT_ATTR_SOURCE .

Сброс, восстановление и возврат

Есть три команды со схожими названиями: git reset , git restore и git revert .

  • git-revert[1] — это создание нового коммита, который отменяет изменения, внесенные другими коммитами.

  • git-restore[1] предназначен для восстановления файлов в рабочем дереве из индекса или другого коммита. Эта команда не обновляет вашу ветку. Эту команду также можно использовать для восстановления файлов в индексе из другого коммита.

  • git-reset[1] касается обновления вашей ветки, перемещения кончика для добавления или удаления коммитов из ветки. Эта операция изменяет историю коммитов.

    git reset также можно использовать для восстановления индекса, перекрывающегося с git restore .

Команды низкого уровня (сантехника)

Хотя Git включает в себя собственный слой фарфора, его команд низкого уровня достаточно для поддержки разработки альтернативных фарфоров. Разработчики таких фарфоров могли бы начать с чтения о git-update-index[1] и git-read-tree[1] .

Интерфейс (ввод, вывод, набор параметров и семантика) этих низкоуровневых команд должен быть намного более стабильным, чем команды уровня Porcelain, поскольку эти команды в основном предназначены для использования в сценариях. С другой стороны, интерфейс команд Porcelain может быть изменен, чтобы улучшить взаимодействие с конечным пользователем.

Следующее описание разделяет команды низкого уровня на команды, управляющие объектами (в репозитории, индексе и рабочем дереве), команды, опрашивающие и сравнивающие объекты, и команды, перемещающие объекты и ссылки между репозиториями.

Configuration mechanism

Git использует простой текстовый формат для хранения настроек каждого репозитория и каждого пользователя. Такой файл конфигурации может выглядеть так:

#
# '#' или ';' символ указывает на комментарий.
#

; основные переменные
[core]
        ; Не доверяйте файловым режимам
        filemode = false

; личность пользователя
[user]
        name = "Junio C Hamano"
        email = "[email protected]"

Различные команды считывают из файла конфигурации и соответствующим образом корректируют свою работу. См. git-config[1] для получения списка и более подробной информации о механизме настройки.

Identifier terminology

<object>

Указывает имя объекта для любого типа объекта.

<blob>

Указывает имя объекта большого двоичного объекта.

<tree>

Указывает имя объекта дерева.

<commit>

Указывает имя объекта фиксации.

<tree-ish>

Указывает имя объекта дерева, фиксации или тега. Команда, которая принимает аргумент <tree-ish>, в конечном итоге хочет работать с объектом <tree>, но автоматически разыменовывает объекты <commit> и <tag>, которые указывают на <tree>.

<commit-ish>

Указывает имя объекта фиксации или тега. Команда, которая принимает аргумент <commit-ish>, в конечном итоге хочет работать с объектом <commit>, но автоматически разыменовывает объекты <tag>, которые указывают на <commit>.

<type>

Указывает, что требуется тип объекта. В настоящее время один из: blob , tree , commit или tag .

<file>

Указывает имя файла - почти всегда относительно корня древовидной структуры, которую описывает GIT_INDEX_FILE .

Symbolic identifiers

Любая команда Git , принимающая любой <объект>, также может использовать следующую символическую запись:

HEAD

указывает главу текущей ветки.

<tag>

действительный тег name (т. е. ссылка refs/tags/<tag> ).

<head>

действующая головка name (т. е. номер refs/heads/<head> ).

Более полный список способов написания имен объектов см. в разделе "SPECIFYING REVISIONS" в gitrevisions[7] .

Environment variables

Различные команды Git обращают внимание на переменные среды и изменяют их поведение. Переменные среды, отмеченные как "Boolean" , принимают свои значения так же, как переменные конфигурации со значениями Boolean , например "true" , "yes" , "on" , а положительные числа принимаются как "yes" .

Вот переменные:

Репозиторий Git

Эти переменные среды применяются к основным командам all Git . Примечание: стоит отметить, что они могут использоваться/переопределяться SCMS, расположенной выше Git , поэтому будьте осторожны при использовании стороннего интерфейса.

GIT_INDEX_FILE

Эта переменная среды определяет альтернативный индексный файл. Если не указано, используется значение по умолчанию $GIT_DIR/index .

GIT_INDEX_VERSION

Эта переменная среды указывает, какая версия индекса используется при записи индексного файла. Это не повлияет на существующие индексные файлы. По умолчанию используется индексный файл версии 2 или 3. См. git-update-index[1] для получения дополнительной информации.

GIT_OBJECT_DIRECTORY

Если каталог хранения объектов указан через эту переменную среды, то под ним создаются каталоги sha1, в противном случае используется каталог $GIT_DIR/objects по умолчанию.

GIT_ALTERNATE_OBJECT_DIRECTORIES

Благодаря неизменяемой природе объектов Git старые объекты можно архивировать в общие каталоги, доступные только для чтения. Эта переменная определяет разделенный ":" (в Windows разделенный ";" ) список каталогов объектов Git , который можно использовать для поиска объектов Git . Новые объекты не будут записываться в эти каталоги.

Записи, начинающиеся с " (двойные кавычки), будут интерпретироваться как пути в кавычках в стиле C, с удалением начальных и конечных двойных кавычек и с учетом обратной косой черты. Например, значение "path-with-\"-and-:-in-it":vanilla-path имеет два пути: path-with-"-and-:-in-it и vanilla-path .

GIT_DIR

Если установлена ​​переменная среды GIT_DIR , она указывает путь, который будет использоваться вместо .git по умолчанию для базы репозитория. Параметр командной строки --git-dir также устанавливает это значение.

GIT_WORK_TREE

Задайте путь к корню рабочего дерева. Этим также можно управлять с помощью параметра командной строки --work-tree и переменной конфигурации core.worktree.

GIT_NAMESPACE

Установите пространство имен Git ; подробности см. в gitnamespaces[7] . Параметр командной строки --namespace также устанавливает это значение.

GIT_CEILING_DIRECTORIES

Это должен быть список абсолютных путей, разделенных двоеточиями. Если установлено, это список каталогов, в которые Git не должен заходить при поиске каталога репозитория (полезно для исключения медленно загружающихся сетевых каталогов). Он не будет исключать текущий рабочий каталог или GIT_DIR, установленный в командной строке или в среде. Обычно Git должен прочитать записи в этом списке и разрешить любую символическую ссылку, которая может присутствовать, чтобы сравнить их с текущим каталогом. Однако, если даже этот доступ медленный, вы можете добавить в список пустую запись, чтобы сообщить Git , что последующие записи не являются символическими ссылками и их не нужно разрешать; например, GIT_CEILING_DIRECTORIES=/maybe/symlink::/very/slow/non/symlink .

GIT_DISCOVERY_ACROSS_FILESYSTEM

При запуске в каталоге, который не имеет каталога репозитория ".git" , Git пытается найти такой каталог в родительских каталогах, чтобы найти вершину рабочего дерева, но по умолчанию он не пересекает границы файловой системы. Для этой переменной среды Boolean можно установить значение true , чтобы указать Git не останавливаться на границах файловой системы. Как и GIT_CEILING_DIRECTORIES , это не повлияет на явный каталог репозитория, установленный через GIT_DIR или в командной строке.

GIT_COMMON_DIR

Если для этой переменной задан путь, файлы, не являющиеся рабочими деревьями, которые обычно находятся в $GIT_DIR, вместо этого будут взяты из этого пути. Файлы, специфичные для рабочего дерева, такие как HEAD или индекс, берутся из $GIT_DIR. Подробности см. в gitrepository-layout[5] и git-worktree[1] . Эта переменная имеет более низкий приоритет, чем другие переменные пути, такие как GIT_INDEX_FILE, GIT_OBJECT_DIRECTORY…​

GIT_DEFAULT_HASH

Если эта переменная установлена, алгоритм хеширования по умолчанию для новых репозиториев будет установлен на это значение. Это значение игнорируется при клонировании и всегда используются настройки удаленного репозитория. По умолчанию — "sha1" . См. --object-format в git-init[1] .

Git Diffs

GIT_DIFF_OPTS

Единственная допустимая настройка — "--unified=??" или "-u??" , позволяющая задать количество строк контекста, отображаемых при создании унифицированного различия. Это имеет приоритет над любым значением параметра "-U" или "--unified" , передаваемым в командной строке Git diff.

GIT_EXTERNAL_DIFF

Когда установлена ​​переменная среды GIT_EXTERNAL_DIFF , названная ею программа вызывается для создания различий, а Git не использует встроенный механизм сравнения. Для пути, который добавляется, удаляется или изменяется, GIT_EXTERNAL_DIFF вызывается с 7 параметрами:

path old-file old-hex old-mode new-file new-hex new-mode

where:

<old|new>-file

это файлы, которые GIT_EXTERNAL_DIFF может использовать для чтения содержимого <old|new>,

<old|new>-hex

— это 40-шестнадцатеричные хеши SHA-1,

<old|new>-mode

являются восьмеричным представлением режимов файла.

Параметры файла могут указывать на рабочий файл пользователя (например, new-file в "git-diff-files" ), /dev/null (например, old-file при добавлении нового файла) или временный файл (например, old-file в индексе). GIT_EXTERNAL_DIFF не следует беспокоиться об отключении временного файла — он удаляется при выходе GIT_EXTERNAL_DIFF .

Для несвязанного пути GIT_EXTERNAL_DIFF вызывается с одним параметром <path>.

Для каждого вызова пути GIT_EXTERNAL_DIFF устанавливаются две переменные среды: GIT_DIFF_PATH_COUNTER и GIT_DIFF_PATH_TOTAL .

GIT_DIFF_PATH_COUNTER

Счетчик, отсчитываемый от 1, увеличивается на единицу для каждого пути.

GIT_DIFF_PATH_TOTAL

Общее количество путей.

GIT_FLUSH

Если для этой переменной среды Boolean установлено значение true, то такие команды, как git blame (в инкрементном режиме), git rev-list , git log , git check-attr и git check-ignore , будут принудительно сбрасывать выходной поток после очистки каждой записи. Если для этой переменной установлено значение false, вывод этих команд будет осуществляться с использованием полностью буферизованного ввода-вывода. Если эта переменная среды не установлена, Git выберет буферизованную или ориентированную на запись очистку в зависимости от того, перенаправляется ли стандартный вывод в файл или нет.

GIT_TRACE

Включает общие сообщения трассировки, например, расширение псевдонима, выполнение встроенной команды и выполнение внешней команды.

Если для этой переменной установлено значение "1" , "2" или "true" (сравнение нечувствительно к регистру), сообщения трассировки будут выводиться в поток stderr.

Если для переменной установлено целочисленное значение больше 2 и меньше 10 (строго), то Git интерпретирует это значение как дескриптор открытого файла и попытается записать сообщения трассировки в этот дескриптор файла.

Альтернативно, если для переменной задан абсолютный путь (начинающийся с символа / ), Git интерпретирует его как путь к файлу и попытается добавить к нему сообщения трассировки.

Сброс переменной или установка ее пустой, "0" или "false" (регистронезависимо) отключает сообщения трассировки.

GIT_TRACE_FSMONITOR

Включает сообщения трассировки для расширения монитора файловой системы. См. GIT_TRACE для доступных вариантов вывода трассировки.

GIT_TRACE_PACK_ACCESS

Включает сообщения трассировки для любого доступа к любым пакетам. Для каждого доступа записывается имя файла пакета и смещение в пакете. Это может быть полезно для устранения некоторых проблем с производительностью, связанных с пакетом. См. GIT_TRACE для доступных вариантов вывода трассировки.

GIT_TRACE_PACKET

Включает сообщения трассировки для всех пакетов, входящих или исходящих из данной программы. Это может помочь при отладке согласования объектов или других проблемах протокола. Трассировка отключается для пакетов, начинающихся с "PACK" (но см. GIT_TRACE_PACKFILE ниже). См. GIT_TRACE для получения информации о доступных вариантах вывода трассировки.

GIT_TRACE_PACKFILE

Позволяет отслеживать пакетные файлы, отправленные или полученные данной программой. В отличие от других выходных данных трассировки, эта трассировка является дословной: без заголовков и кавычек двоичных данных. Вы почти наверняка захотите направить его в файл (например, GIT_TRACE_PACKFILE=/tmp/my.pack ), а не отображать его на терминале или смешивать с другими выводами трассировки.

Обратите внимание, что в настоящее время это реализовано только для клиентской стороны клонирования и выборки.

GIT_TRACE_PERFORMANCE

Включает сообщения отслеживания, связанные с производительностью, например, общее время выполнения каждой команды Git . См. GIT_TRACE для доступных вариантов вывода трассировки.

GIT_TRACE_REFS

Включает сообщения трассировки для операций с базой данных ref. См. GIT_TRACE для доступных вариантов вывода трассировки.

GIT_TRACE_SETUP

Позволяет сообщениям трассировки распечатывать .git, рабочее дерево и текущий рабочий каталог после завершения фазы настройки Git . См. GIT_TRACE для доступных вариантов вывода трассировки.

GIT_TRACE_SHALLOW

Включает сообщения трассировки, которые могут помочь в отладке выборки/клонирования мелких репозиториев. См. GIT_TRACE для доступных вариантов вывода трассировки.

GIT_TRACE_CURL

Позволяет создавать полный дамп трассировки всех входящих и исходящих данных, включая описательную информацию, транспортного протокола git . Это похоже на выполнение команды curl --trace-ascii в командной строке. См. GIT_TRACE для получения информации о доступных вариантах вывода трассировки.

GIT_TRACE_CURL_NO_DATA

Если трассировка скручивания включена (см. GIT_TRACE_CURL выше), не выгружайте данные (то есть выгружайте только информационные строки и заголовки).

GIT_TRACE2

Включает более подробные сообщения трассировки из библиотеки "trace2" . Вывод GIT_TRACE2 представляет собой простой текстовый формат, удобный для чтения человеком.

Если для этой переменной установлено значение "1" , "2" или "true" (сравнение нечувствительно к регистру), сообщения трассировки будут выводиться в поток stderr.

Если для переменной установлено целочисленное значение больше 2 и меньше 10 (строго), то Git интерпретирует это значение как дескриптор открытого файла и попытается записать сообщения трассировки в этот дескриптор файла.

Альтернативно, если для переменной задан абсолютный путь (начинающийся с символа / ), Git интерпретирует его как путь к файлу и попытается добавить к нему сообщения трассировки. Если путь уже существует и является каталогом, сообщения трассировки будут записаны в файлы (по одному на каждый процесс) в этом каталоге, названные в соответствии с последним компонентом SID и необязательным счетчиком (во избежание конфликтов имен файлов).

Кроме того, если для переменной установлено значение af_unix:[<socket_type>:]<absolute-pathname> , Git попытается открыть путь как сокет домена Unix. Тип сокета может быть stream или dgram .

Сброс переменной или установка ее пустой, "0" или "false" (регистронезависимо) отключает сообщения трассировки.

Подробную информацию см. в Trace2 documentation .

GIT_TRACE2_EVENT

Этот параметр записывает формат на основе JSON, подходящий для машинной интерпретации. См. GIT_TRACE2 для доступных опций вывода трассировки и Trace2 documentation для получения полной информации.

GIT_TRACE2_PERF

В дополнение к текстовым сообщениям, доступным в GIT_TRACE2 , этот параметр записывает формат на основе столбцов для понимания областей вложенности. См. GIT_TRACE2 для доступных опций вывода трассировки и Trace2 documentation для получения полной информации.

GIT_TRACE_REDACT

По умолчанию, когда трассировка активирована, Git редактирует значения файлов cookie, заголовок "Authorization:" , заголовок "Proxy-Authorization:" и URI паковочного файла. Установите для этой переменной среды Boolean значение false , чтобы предотвратить это редактирование.

GIT_LITERAL_PATHSPECS

Установка для этой переменной среды Boolean значения true приведет к тому, что Git будет обрабатывать все спецификации пути буквально, а не как шаблоны glob. Например, при запуске GIT_LITERAL_PATHSPECS=1 git log -- '*.c' будут выполняться поиск коммитов, которые касаются пути *.c , а не каких-либо путей, которым соответствует glob *.c . Это может вам понадобиться, если вы передаете буквальные пути в Git (например, пути, ранее предоставленные вам git ls-tree , дифференциальный вывод --raw и т. д.).

GIT_GLOB_PATHSPECS

Установка для этой переменной среды Boolean значения true приведет к тому, что Git будет обрабатывать все спецификации пути как шаблоны glob (так называемая магия "glob" ).

GIT_NOGLOB_PATHSPECS

Установка для этой переменной среды Boolean значения true приведет к тому, что Git будет обрабатывать все спецификации путей как литеральные (так называемая магия "literal" ).

GIT_ICASE_PATHSPECS

Установка для этой переменной среды Boolean значения true приведет к тому, что Git будет обрабатывать все спецификации путей без учета регистра.

GIT_REFLOG_ACTION

Когда ссылка обновляется, в журнале ссылок создаются записи для отслеживания причины обновления ссылки (обычно это имя команды высокого уровня, которая обновила ссылку), а также старые и новые значения ссылки. . Скриптованная команда Porcelain может использовать вспомогательную функцию set_reflog_action в git-sh-setup , чтобы установить свое имя для этой переменной, когда она вызывается конечным пользователем в качестве команды верхнего уровня, для записи в теле журнала ссылок.

GIT_REF_PARANOIA

Если для этой переменной среды Boolean установлено значение false, игнорируйте неработающие или неправильно названные ссылки при переборе списков ссылок. Обычно Git пытается включить любые такие ссылки, что может привести к сбою некоторых операций. Обычно это предпочтительнее, поскольку потенциально разрушительные операции (например, git-prune[1] ) лучше прерывать, чем игнорировать испорченные ссылки (и, таким образом, считать историю, на которую они указывают, не стоящей сохранения). Значение по умолчанию — 1 (т. е. будьте осторожны при обнаружении и отмене всех операций). Обычно вам не нужно устанавливать значение 0 , но это может быть полезно при попытке восстановить данные из поврежденного репозитория.

GIT_COMMIT_GRAPH_PARANOIA

При загрузке объекта фиксации из графа фиксации Git выполняет проверку существования объекта в базе данных объектов. Это сделано для того, чтобы избежать проблем с устаревшими графами коммитов, которые содержат ссылки на уже удаленные коммиты, но это приводит к снижению производительности.

По умолчанию используется "false" , который отключает вышеупомянутое поведение. Установка значения "true" включает проверку существования, чтобы устаревшие коммиты никогда не возвращались из графа коммитов за счет производительности.

GIT_ALLOW_PROTOCOL

Если задан список протоколов, разделенных двоеточиями, ведите себя так, как будто для protocol.allow установлено значение never , а для каждого из перечисленных протоколов protocol.<name>.allow установлено на always (переопределяя любую существующую конфигурацию). Подробнее смотрите описание protocol.allow в git-config[1] .

GIT_PROTOCOL_FROM_USER

Установите для этой переменной среды Boolean значение false , чтобы запретить использование протоколов выборки, отправки или клонирования, которые настроены на состояние user . Это полезно для ограничения рекурсивной инициализации подмодуля из ненадежного репозитория или для программ, которые передают потенциально ненадежные URL-адреса командам git . Более подробную информацию см. в git-config[1] .

GIT_PROTOCOL

Только для внутреннего использования. Используется при установлении связи по проводному протоколу. Содержит список ключей, разделенных двоеточием : , с необязательными значениями key[=value] . Наличие неизвестных ключей и значений следует игнорировать.

Обратите внимание, что серверы, возможно, потребуется настроить так, чтобы эта переменная могла передаваться через некоторые транспорты. Он будет распространяться автоматически при доступе к локальным репозиториям (например, file:// или пути файловой системы), а также по протоколу git:// . Для git-over-http он должен работать автоматически в большинстве конфигураций, но см. обсуждение в git-http-backend[1] . Для git-over-ssh может потребоваться настроить ssh-сервер, чтобы клиенты могли передавать эту переменную (например, используя AcceptEnv GIT_PROTOCOL с OpenSSH).

Эта конфигурация является необязательной. Если переменная не распространяется, клиенты вернутся к исходному протоколу "v0" (но могут упустить некоторые улучшения производительности или функции). Эта переменная в настоящее время влияет только на клонирование и выборку; он еще не используется для push-уведомлений (но может быть использован в будущем).

GIT_OPTIONAL_LOCKS

Если для этой переменной среды Boolean установлено значение false, Git выполнит любую запрошенную операцию без выполнения каких-либо дополнительных дополнительных операций, требующих блокировки. Например, это предотвратит обновление индекса git status как побочный эффект. Это полезно для процессов, работающих в фоновом режиме, которые не хотят вызывать конфликт блокировок с другими операциями в репозитории. По умолчанию 1 .

GIT_REDIRECT_STDIN
GIT_REDIRECT_STDOUT
GIT_REDIRECT_STDERR

Только для Windows: разрешить перенаправление стандартных дескрипторов ввода/вывода/ошибок на пути, указанные переменными среды. Это особенно полезно в многопоточных приложениях, где канонический способ передачи стандартных дескрипторов через CreateProcess() невозможен, поскольку для этого потребуется пометить дескрипторы как наследуемые (и, следовательно, каждый порожденный процесс будет наследовать их, возможно, блокируя обычные операции Git ). Основной вариант использования — использование именованных каналов для связи (например, \\.\pipe\my-git-stdin-123 ).

Поддерживаются два специальных значения: off просто закроет соответствующий стандартный дескриптор, а если GIT_REDIRECT_STDERR равен 2>&1 , стандартная ошибка будет перенаправлена ​​на тот же дескриптор, что и стандартный вывод.

GIT_PRINT_SHA1_ELLIPSIS (deprecated)

Если установлено значение yes , после (сокращенного) значения SHA-1 печатается многоточие. Это влияет на индикацию отключенных HEAD ( git-checkout[1] ) и необработанный вывод разницы ( git-diff[1] ). Печать многоточия в упомянутых случаях больше не считается адекватной, и ее поддержка, вероятно, будет прекращена в обозримом будущем (вместе с переменной).

Discussion

Более подробную информацию о следующем можно получить у Git concepts chapter of the user-manual и gitcore-tutorial[7] .

Проект Git обычно состоит из рабочего каталога с подкаталогом ".git" на верхнем уровне. Каталог .git содержит, среди прочего, сжатую базу данных объектов, представляющую полную историю проекта, файл "index" , который связывает эту историю с текущим содержимым рабочего дерева, а также именованные указатели на эту историю, такие как теги и заголовки ветвей. .

База данных объектов содержит объекты трех основных типов: большие двоичные объекты, содержащие данные файлов; деревья, которые указывают на большие двоичные объекты и другие деревья для создания иерархии каталогов; и коммиты, каждый из которых ссылается на одно дерево и некоторое количество родительских коммитов.

Фиксация, эквивалентная тому, что в других системах называется "changeset" или "version" , представляет собой шаг в истории проекта, а каждый родительский элемент представляет собой непосредственно предшествующий шаг. Коммиты с более чем одним родителем представляют собой слияние независимых линий разработки.

Все объекты именуются по хешу SHA-1 их содержимого, который обычно записывается в виде строки из 40 шестнадцатеричных цифр. Такие имена глобально уникальны. За всю историю, предшествовавшую коммиту, можно поручиться, подписав только этот коммит. Для этой цели предусмотрен четвертый тип объекта — тег.

При первом создании объекты хранятся в отдельных файлах, но для повышения эффективности позже их можно сжать вместе в "pack files" .

Именованные указатели, называемые ссылками, отмечают интересные моменты в истории. Ссылка может содержать имя объекта SHA-1 или имя другой ссылки (последнее называется "symbolic ref" ). Ссылки с именами, начинающимися с refs/head/ , содержат имя SHA-1 самого последнего коммита (или "head" ) разрабатываемой ветки. Имена интересующих тегов SHA-1 хранятся в refs/tags/ . Символическая ссылка с именем HEAD содержит имя извлеченной в данный момент ветки.

Индексный файл инициализируется списком всех путей и для каждого пути объектом большого двоичного объекта и набором атрибутов. Объект blob представляет содержимое файла на момент начала текущей ветки. Атрибуты (время последнего изменения, размер и т.д.) берутся из соответствующего файла в рабочем дереве. Последующие изменения в рабочем дереве можно найти путем сравнения этих атрибутов. Индекс может быть обновлен новым содержимым, а новые фиксации могут быть созданы на основе содержимого, хранящегося в индексе.

Индекс также способен хранить несколько записей (называемых "stages" ) для заданного имени пути. Эти этапы используются для хранения различных неслитых версий файла во время слияния.
