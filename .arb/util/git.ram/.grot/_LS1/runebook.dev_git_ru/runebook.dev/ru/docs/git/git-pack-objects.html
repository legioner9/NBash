git-pack-objects

Name

git-pack-objects — Создать упакованный архив объектов.

Synopsis

git pack-objects [-q | --progress | --all-progress] [--all-progress-implied]
        [--no-reuse-delta] [--delta-base-offset] [--non-empty]
        [--local] [--incremental] [--window=<n>] [--depth=<n>]
        [--revs [--unpacked | --all]] [--keep-pack=<pack-name>]
        [--cruft] [--cruft-expiration=<time>]
        [--stdout [--filter=<filter-spec>] | <base-name>]
        [--shallow] [--keep-true-parents] [--[no-]sparse] < <object-list>

Description

Считывает список объектов со стандартного ввода и записывает на диск один или несколько упакованных архивов с указанным базовым именем или упакованный архив на стандартный вывод.

Упакованный архив — это эффективный способ передачи набора объектов между двумя репозиториями, а также эффективный архивный формат доступа. В упакованном архиве объект хранится либо как сжатое целое, либо как отличие от какого-либо другого объекта. Последнюю часто называют дельтой.

Формат упакованного архива (.pack) разработан как автономный, поэтому его можно распаковать без какой-либо дополнительной информации. Следовательно, каждый объект, от которого зависит дельта, должен присутствовать в пакете.

Индексный файл пакета (.idx) создается для быстрого произвольного доступа к объектам в пакете. Размещение индексного файла (.idx) и упакованного архива (.pack) в подкаталоге Pack/ $GIT_OBJECT_DIRECTORY (или в любом из каталогов $GIT_ALTERNATE_OBJECT_DIRECTORIES) позволяет Git читать из архива пакета.

Команда git unpack-objects может читать упакованный архив и расширять объекты, содержащиеся в пакете, в формат "one-file one-object" ; обычно это делается с помощью команд smart-pull, когда пакет создается «на лету» для эффективной сетевой транспортировки их одноранговыми узлами.

Options

base-name

Запишите в пары файлов (.pack и .idx), используя <base-name> для определения имени созданного файла. При использовании этой опции два файла в паре записываются в файлы <base-name>-<SHA-1>.{pack,idx}. <SHA-1> — это хэш, основанный на содержимом пакета, который записывается в стандартный вывод команды.

--stdout

Запишите содержимое пакета (то, что было бы записано в .pack-файл) на стандартный вывод.

--revs

Считайте аргументы версии из стандартного ввода, а не имена отдельных объектов. Аргументы версии обрабатываются так же, как git rev-list с флагом --objects , который использует свои аргументы commit для построения списка выводимых объектов. Объекты в полученном списке упаковываются. Помимо доработок, также принимаются линии --not или --shallow <SHA-1> .

--unpacked

Это подразумевает --revs . При обработке списка аргументов версии, считываемого со стандартного ввода, ограничьте количество упакованных объектов теми, которые еще не упакованы.

--all

Это подразумевает --revs . В дополнение к списку аргументов версии, считанному со стандартного ввода, представьте, что все ссылки под refs/ указаны для включения.

--include-tag

Включите незапрошенные аннотированные теги, если объект, на который они ссылаются, был включен в результирующий пакетный файл. Это может быть полезно для отправки новых тегов собственным клиентам Git .

--stdin-packs

Считайте базовые имена файлов пакетов (например, pack-1234abcd.pack ) из стандартного ввода вместо имен объектов или аргументов версии. Результирующий пакет содержит все объекты, перечисленные во включенных пакетах (те, которые не начинаются с ^ ), исключая любые объекты, перечисленные в исключенных пакетах (начинающиеся с ^ ).

Несовместим с --revs или вариантами, подразумевающими --revs (например, --all ), за исключением --unpacked , который совместим.

--cruft

Упаковывает недоступные объекты в отдельный пакет "cruft" , обозначаемый наличием файла .mtimes . Обычно используется git repack --cruft . Вызывающие программы предоставляют список имен пакетов и указывают, какие пакеты останутся в репозитории, а какие будут удалены (обозначается префиксом - ). Содержимым пакета Cruft являются все объекты, не содержащиеся в сохранившихся пакетах, для которых не истек льготный период (см. --cruft-expiration ниже) или для которых истек льготный период, но которые доступны из другого объекта, который еще не истек.

Когда во входных данных указан пакет, содержащий все доступные объекты (и перечислены все остальные пакеты как ожидающие удаления), соответствующий пакет будет содержать все недоступные объекты (с mtime новее, чем --cruft-expiration ), а также любые недоступные объекты, чье время mtime старше --cruft-expiration . , но доступны из недостижимого объекта, время которого новее, чем у --cruft-expiration ).

Несовместим с --unpack-unreachable , --keep-unreachable , --pack-loose-unreachable , --stdin-packs , а также с любыми другими вариантами, подразумевающими --revs .

--cruft-expiration=<approxidate>

Если указано, объекты удаляются из пакета crft, если их mtime старше <approxidate> . Если не указано (и задан --cruft ), объекты не удаляются.

--window=<n>
--depth=<n>

Эти два параметра влияют на то, как объекты, содержащиеся в пакете, сохраняются с использованием дельта-сжатия. Объекты сначала сортируются внутри по типу, размеру и (необязательно) именам и сравниваются с другими объектами в --window, чтобы увидеть, экономит ли пространство использование дельта-сжатия. --глубина ограничивает максимальную глубину дельты; слишком глубокая обработка влияет на производительность распаковщика, поскольку дельта-данные необходимо применять столько раз, чтобы добраться до нужного объекта.

Значение по умолчанию для --window — 10, а —eep — 50. Максимальная глубина — 4095.

--window-memory=<n>

Эта опция обеспечивает дополнительный лимит сверх --window ; размер окна будет динамически уменьшаться, чтобы не занимать в памяти более <n> байт. Это полезно в репозиториях со смесью больших и маленьких объектов, чтобы не исчерпать память при большом окне, но при этом иметь возможность использовать большое окно для меньших объектов. К размеру можно добавить "k" , "m" или "g" . --window-memory=0 делает использование памяти неограниченным. Значение по умолчанию берется из переменной конфигурации pack.windowMemory .

--max-pack-size=<n>

В необычных сценариях вы не сможете создавать файлы размером больше определенного в вашей файловой системе, и этот параметр можно использовать, чтобы указать команде разделить выходной пакетный файл на несколько независимых пакетных файлов, каждый из которых не превышает заданный размер. К размеру можно добавить суффикс "k" , "m" или "g" . Минимально допустимый размер ограничен 1 МБ. Значение по умолчанию не ограничено, если не установлена ​​переменная конфигурации pack.packSizeLimit . Обратите внимание, что этот вариант может привести к увеличению размера и замедлению репозитория; см. обсуждение в pack.packSizeLimit .

--honor-pack-keep

Этот флаг приводит к тому, что объект, уже находящийся в локальном пакете и имеющий файл .keep, игнорируется, даже если в противном случае он был бы упакован.

--keep-pack=<pack-name>

Этот флаг приводит к тому, что объект, уже находящийся в данном пакете, игнорируется, даже если в противном случае он был бы упакован. <pack-name> — это имя файла пакета без ведущего каталога (например, pack-123.pack ). Эту опцию можно указать несколько раз, чтобы сохранить несколько пакетов.

--incremental

Этот флаг приводит к тому, что объект, уже находящийся в пакете, игнорируется, даже если в противном случае он был бы упакован.

--local

Этот флаг приводит к тому, что объект, заимствованный из альтернативного хранилища объектов, игнорируется, даже если в противном случае он был бы упакован.

--non-empty

Создавайте упакованный архив только в том случае, если он содержит хотя бы один объект.

--progress

Статус выполнения сообщается в стандартном потоке ошибок по умолчанию, когда он подключен к терминалу, если не указан параметр -q. Этот флаг обеспечивает статус хода выполнения, даже если стандартный поток ошибок не направляется на терминал.

--all-progress

Если указан параметр --stdout, отчет о ходе выполнения отображается на этапах подсчета объектов и сжатия, но блокируется на этапе записи. Причина в том, что в некоторых случаях поток вывода напрямую связан с другой командой, которая может захотеть отображать собственный статус выполнения при обработке входящих данных пакета. Этот флаг похож на --progress, за исключением того, что он также вызывает отчет о ходе выполнения для фазы записи, даже если используется --stdout.

--all-progress-implied

Это используется для обозначения --all-progress всякий раз, когда активировано отображение прогресса. В отличие от --all-progress этот флаг сам по себе фактически не вызывает никакого отображения прогресса.

-q

Этот флаг заставляет команду не сообщать о своем ходе выполнения в стандартном потоке ошибок.

--no-reuse-delta

При создании упакованного архива в репозитории, в котором есть существующие пакеты, команда повторно использует существующие изменения. Иногда это приводит к получению слегка неоптимальной упаковки. Этот флаг сообщает команде не использовать повторно существующие отклонения, а вычислять их с нуля.

--no-reuse-object

Этот флаг сообщает команде вообще не использовать повторно существующие данные объекта, включая неделифицированный объект, что приводит к повторному сжатию всего. Это подразумевает --no-reuse-delta. Полезно только в непонятном случае, когда желательно массовое применение другого уровня сжатия упакованных данных.

--compression=<n>

Указывает уровень сжатия для вновь сжатых данных в созданном пакете. Если не указано, уровень сжатия пакета определяется сначала с помощью package.compression, затем с помощью core.compression и по умолчанию равен -1, значению zlib по умолчанию, если ни один из них не установлен. Добавьте --no-reuse-object, если вы хотите обеспечить единый уровень сжатия для всех данных независимо от источника.

--[no-]sparse

Переключите алгоритм "sparse" , чтобы определить, какие объекты включить в пакет, в сочетании с опцией "--revs" . Этот алгоритм обходит только те деревья, которые появляются на путях, вводящих новые объекты. Это может дать значительный выигрыш в производительности при расчете пакета для отправки небольшого изменения. Однако возможно, что в пакетный файл будут добавлены дополнительные объекты, если включенные коммиты содержат определенные типы прямых переименований. Если этот параметр не включен, по умолчанию используется значение pack.useSparse , которое равно true , если не указано иное.

--thin

Создайте пакет "thin" , исключив общие объекты между отправителем и получателем, чтобы уменьшить передачу по сети. Эта опция имеет смысл только в сочетании с --stdout.

Примечание. Тонкий пакет нарушает формат упакованного архива, пропуская обязательные объекты, и поэтому Git непригоден для использования, не делая его автономным. Используйте git index-pack --fix-thin (см. git-index-pack[1] ), чтобы восстановить автономное свойство.

--shallow

Оптимизируйте пакет, который будет предоставлен клиенту с неглубоким репозиторием. Эта опция в сочетании с --thin может привести к уменьшению размера пакета за счет скорости.

--delta-base-offset

Упакованный архив может выражать базовый объект дельты либо как 20-байтовое имя объекта, либо как смещение в потоке, но старые версии Git не понимают последнего. По умолчанию git pack-objects использует только первый формат для лучшей совместимости. Эта опция позволяет команде использовать последний формат для компактности. В зависимости от средней длины дельта-цепочки этот параметр обычно сжимает полученный пакетный файл на 3–5 процентов.

Примечание. Команды Porcelain, такие как git gc (см. git-gc[1] ), git repack (см. git-repack[1] ), передают эту опцию по умолчанию в современном Git , когда они помещают объекты из вашего репозитория в файлы пакета. То же самое делает git bundle (см. git-bundle[1] ), когда создает пакет.

--threads=<n>

Указывает количество потоков, которые будут создаваться при поиске лучших дельта-соответствий. Для этого необходимо, чтобы объекты пакета были скомпилированы с pthreads, в противном случае эта опция игнорируется с предупреждением. Это предназначено для сокращения времени упаковки на многопроцессорных машинах. Однако необходимый объем памяти для окна дельта-поиска умножается на количество потоков. Если указать 0, Git автоматически определит количество процессоров и соответствующим образом установит количество потоков.

--index-version=<version>[,<offset>]

Это предназначено для использования только в наборе тестов. Это позволяет принудительно использовать версию сгенерированного индекса пакета и принудительно использовать 64-битные записи индекса для объектов, расположенных выше заданного смещения.

--keep-true-parents

При этом варианте родители, скрытые трансплантатами, все же упаковываются.

--filter=<filter-spec>

Исключает определенные объекты (обычно большие двоичные объекты) из результирующего пакфайла. См. git-rev-list[1] для действительных форм <filter-spec> .

--no-filter

Отключает любой предыдущий аргумент --filter= .

--missing=<missing-action>

Опция отладки, которая поможет в будущей разработке "partial clone" . Эта опция определяет, как обрабатываются отсутствующие объекты.

Форма --missing=error запрашивает остановку объектов пакета с ошибкой, если обнаружен отсутствующий объект. Если репозиторий является частичным клоном, будет предпринята попытка получить отсутствующие объекты до объявления их отсутствующими. Это действие по умолчанию.

Форма --missing=allow-any позволит продолжить обход объекта, если обнаружен отсутствующий объект. Никакого извлечения отсутствующего объекта не произойдет. Отсутствующие объекты будут автоматически исключены из результатов.

Форма --missing=allow-promisor похожа на allow-any , но позволяет продолжить обход объекта только для отсутствующих объектов ОЖИДАЕМОГО обещания. Никакого извлечения отсутствующего объекта не произойдет. Неожиданный отсутствующий объект вызовет ошибку.

--exclude-promisor-objects

Опустите объекты, о которых известно, что они находятся в удаленном промисоре. (Эта опция предназначена для работы только с локально созданными объектами, поэтому при переупаковке мы по-прежнему сохраняем различие между локально созданными объектами [без .promisor] и объектами из удаленного промисора [с .promisor].) Это используется. с частичным клонированием.

--keep-unreachable

Объекты, недоступные по ссылкам в пакетах, названных с параметром --unpacked=, добавляются в результирующий пакет в дополнение к доступным объектам, которых нет в пакетах, отмеченных файлами *.keep. Это подразумевает --revs .

--pack-loose-unreachable

Упакуйте недоступные незакрепленные предметы (и их незакрепленные аналоги уберите). Это подразумевает --revs .

--unpack-unreachable

Недоступные предметы храните в свободном виде. Это подразумевает --revs .

--delta-islands

Ограничить дельта-соответствия на основе "islands" . См. ОСТРОВА ДЕЛЬТА ниже.

Delta islands

Когда это возможно, pack-objects пытается повторно использовать существующие на диске изменения, чтобы избежать необходимости поиска новых на лету. Это важная оптимизация для обслуживания выборок, поскольку она означает, что сервер может вообще избежать раздувания большинства объектов и просто отправлять байты непосредственно с диска. Эта оптимизация не может работать, когда объект хранится как дельта относительно базы, которой нет у получателя (и которую мы еще не отправляем). В этом случае сервер "breaks" является дельтой и должен найти новый, который имеет высокую стоимость процессора. Поэтому для производительности важно, чтобы набор объектов в дельта-отношениях на диске соответствовал тому, что получает клиент.

В обычном репозитории это обычно работает автоматически. Объекты в основном доступны из ветвей и тегов, и это то, что получают клиенты. Любые отклонения, которые мы обнаруживаем на сервере, скорее всего, будут между объектами, которые есть или будут у клиента.

Но в некоторых конфигурациях репозитория у вас может быть несколько связанных, но отдельных групп подсказок по ссылкам, причем клиенты имеют тенденцию получать эти группы независимо. Например, представьте, что вы размещаете несколько репозиториев "forks" в одном хранилище общих объектов и позволяете клиентам просматривать их как отдельные репозитории через GIT_NAMESPACE или отдельные репозитории, используя механизм альтернатив. Наивный репак может обнаружить, что оптимальная дельта для объекта соответствует базе, которая находится только в другом форке. Но когда клиент выполнит выборку, у него не будет базового объекта, и нам придется на лету находить новую дельту.

Аналогичная ситуация может возникнуть, если у вас есть много ссылок за пределами refs/heads/ и refs/tags/ , которые указывают на связанные объекты (например, refs/pull или refs/changes , используемые некоторыми хостинг-провайдерами). По умолчанию клиенты извлекают только заголовки и теги, а дельты для объектов, найденных только в этих других группах, не могут быть отправлены как есть.

Дельта-острова решают эту проблему, позволяя вам группировать ссылки в отдельные "islands" . Pack-objects вычисляет, какие объекты с каких островов доступны, и отказывается делать дельту из объекта A с базой, которая отсутствует на всех островах A . Это приводит к немного большему размеру пакетов (поскольку мы упускаем некоторые возможности дельты), но гарантирует, что при выборке одного острова не придется пересчитывать дельты на лету из-за пересечения границ островов.

При перепаковке дельта-островами дельта-окно имеет тенденцию забиваться кандидатами, запрещенными конфигом. Переупаковка с помощью большого --window помогает (и не занимает так много времени, как в противном случае, потому что мы можем отклонить некоторые пары объектов на основе островов, прежде чем выполнять какие-либо вычисления над содержимым).

Острова настраиваются с помощью параметра pack.island , который можно указать несколько раз. Каждое значение представляет собой регулярное выражение с левой привязкой, соответствующее именам ссылок. Например:

[pack]
island = refs/heads/
island = refs/tags/

помещает заголовки и теги на остров (имя которого представляет собой пустую строку; дополнительную информацию об именовании см. ниже). Любые ссылки, не соответствующие этим регулярным выражениям (например, refs/pull/123 ), не находятся ни на одном острове. Поэтому любой объект, доступный только из refs/pull/ (но не заголовки или теги), не является кандидатом на использование в качестве основы для refs/heads/ .

Ссылки группируются в острова на основе их "names" , и два регулярных выражения, которые создают одно и то же имя, считаются находящимися на одном острове. Имена вычисляются на основе регулярных выражений путем объединения любых групп захвата из регулярного выражения с тире - между ними. (А если групп захвата нет, то именем является пустая строка, как в приведенном выше примере.) Это позволяет создавать произвольное количество островов. Однако поддерживается только до 14 таких групп захвата.

Например, представьте, что вы храните ссылки для каждого форка в refs/virtual/ID , где ID — числовой идентификатор. Затем вы можете настроить:

[pack]
island = refs/virtual/([0-9]+)/heads/
island = refs/virtual/([0-9]+)/tags/
island = refs/virtual/([0-9]+)/(pull)/

При этом головки и теги для каждой вилки помещаются на отдельный остров (с именем "1234" или аналогичный), а ссылки pull для каждого переходят в отдельный "1234-pull" .

Обратите внимание, что мы выбираем один остров для каждого регулярного выражения, используя порядок "last one wins" (который позволяет конфигурации, специфичной для репозитория, иметь приоритет над общепользовательской конфигурацией и т. д.).

Configuration

Различные переменные конфигурации влияют на упаковку, см. git-config[1] (найдите "pack" и "delta" ).

Примечательно, что дельта-сжатие не используется для объектов, размер которых превышает конфигурационную переменную core.bigFileThreshold , и для файлов с атрибутом delta , для которого установлено значение false.
