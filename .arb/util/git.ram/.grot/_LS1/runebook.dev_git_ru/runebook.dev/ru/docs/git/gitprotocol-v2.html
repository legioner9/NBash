gitprotocol-v2

Name

gitprotocol-v2 — проводной протокол Git , версия 2

Synopsis

<over-the-wire-protocol>

Description

В этом документе представлена ​​спецификация версии 2 проводного протокола Git. Протокол v2 будет улучшен по сравнению с версией v1 следующими способами:

  • Вместо нескольких имен служб одна служба будет поддерживать несколько команд.

  • Легко расширяется, поскольку возможности перемещаются в отдельный раздел протокола, больше не скрываются за нулевым байтом и не ограничиваются размером pkt-строки.

  • Отделите другую информацию, скрытую за NUL-байтами (например, строку агента как возможность, а symrefs можно запросить с помощью ls-refs ).

  • Справочная реклама будет опущена, если это явно не запрошено.

  • команда ls-refs для явного запроса некоторых ссылок

  • Разработан с учетом http и stateless-rpc. Благодаря четкой семантике сброса удаленный помощник http может просто действовать как прокси.

В протоколе v2 связь ориентирована на команды. При первом обращении к серверу будет объявлен список возможностей. Некоторые из этих возможностей будут представлять собой команды, выполнение которых может запросить клиент. После завершения команды клиент может повторно использовать соединение и запросить выполнение других команд.

Packet-line framing

Вся связь осуществляется с использованием кадрирования пакетной линии, как и в версии v1. Дополнительную информацию см. в gitprotocol-pack[5] и gitprotocol-common[5] .

В протоколе v2 эти специальные пакеты будут иметь следующую семантику:

  • 0000 Flush Packet (flush-pkt) — указывает на конец сообщения.

  • 0001 Delimiter Packet (delim-pkt) — разделяет разделы сообщения.

  • 0002 Пакет завершения ответа (response-end-pkt) — указывает конец ответа для соединений без сохранения состояния.

Первоначальный запрос клиента

Обычно клиент может запросить использование протокола v2, отправив version=2 через соответствующий побочный канал для используемого транспорта, что неизбежно устанавливает GIT_PROTOCOL . Дополнительную информацию можно найти в gitprotocol-pack[5] и gitprotocol-http[5] , а также определение GIT_PROTOCOL в git.txt . Во всех случаях ответом сервера является объявление о возможности.

Git Transport

При использовании транспорта git:// вы можете запросить использование протокола v2, отправив "version=2" в качестве дополнительного параметра:

003egit-upload-pack /project.git\0host=myserver.com\0\0version=2\0

SSH и транспортировка файлов

При использовании транспорта ssh:// или file:// переменная среды GIT_PROTOCOL должна быть явно установлена ​​для включения "version=2" . Возможно, сервер потребуется настроить так, чтобы разрешить передачу этой переменной среды.

HTTP Transport

При использовании транспорта http:// или https:// клиент отправляет запрос "smart" info/refs, как описано в gitprotocol-http[5] , и запрашивает использование v2, предоставляя "version=2" в заголовке Git-Protocol .

C: GET $GIT_URL/info/refs?service=git-upload-pack HTTP/1.0
C: Git-Protocol: version=2

Сервер v2 ответит:

S: 200 OK
S: <Some headers>
S: ...
S:
S: 000eversion 2\n
S: <capability-advertisement>

Последующие запросы затем направляются непосредственно в службу $GIT_URL/git-upload-pack . (Это работает так же для git-receive-pack).

Использует опцию --http-backend-info-refs для git-upload-pack[1] .

Возможно, сервер потребуется настроить для передачи содержимого этого заголовка через переменную GIT_PROTOCOL . См. обсуждение в git-http-backend.txt .

Capability advertisement

Сервер, который решает установить связь (на основе запроса клиента) с использованием протокола версии 2, уведомляет клиента, отправляя строку версии в своем первоначальном ответе, за которым следует объявление о своих возможностях. Каждая возможность представляет собой ключ с необязательным значением. Клиенты должны игнорировать все неизвестные ключи. Семантика неизвестных значений оставлена ​​на усмотрение каждого ключа. Некоторые возможности будут описывать команды, выполнение которых может быть запрошено клиентом.

capability-advertisement = protocol-version
      capability-list
      flush-pkt
protocol-version = PKT-LINE("version 2" LF)
capability-list = *capability
capability = PKT-LINE(key[=value] LF)
key = 1*(ALPHA | DIGIT | "-_")
value = 1*(ALPHA | DIGIT | " -_.,?\/{}[]()<>!@#$%^&*+=:;")

Command request

После получения объявления о возможностях клиент может затем выдать запрос на выбор нужной команды с любыми конкретными возможностями или аргументами. Затем существует дополнительный раздел, в котором клиент может предоставить любые параметры или запросы, специфичные для команды. Одновременно можно запросить только одну команду.

request = empty-request | command-request
empty-request = flush-pkt
command-request = command
    capability-list
    delim-pkt
    command-args
    flush-pkt
command = PKT-LINE("command=" key LF)
command-args = *command-specific-arg
command-specific-args are packet line framed arguments defined by
each individual command.

Затем сервер проверит, содержит ли запрос клиента действительную команду, а также действительные возможности, которые были объявлены. Если запрос действителен, сервер выполнит команду. Сервер ДОЛЖЕН дождаться получения всего запроса клиента, прежде чем выдавать ответ. Формат ответа определяется выполняемой командой, но во всех случаях флеш-пкт указывает на конец ответа.

Когда команда завершена и клиент получил весь ответ от сервера, клиент может либо запросить выполнение другой команды, либо разорвать соединение. Клиент может дополнительно отправить пустой запрос, состоящий только из flash-pkt, чтобы указать, что больше запросов делаться не будет.

Capabilities

Существует два разных типа возможностей: обычные возможности, которые можно использовать для передачи информации или изменения поведения запроса, и команды, которые представляют собой основные действия, которые клиент хочет выполнить (извлечение, отправка и т. д.).

Протокол версии 2 по умолчанию не имеет состояния. Это означает, что все команды должны длиться только один раунд и не иметь состояния с точки зрения сервера, если только клиент не запросил возможность, указывающую, что состояние должно поддерживаться сервером. Клиенты НЕ ДОЛЖНЫ требовать управления состоянием на стороне сервера для правильной работы. Это позволяет выполнять простую циклическую балансировку нагрузки на стороне сервера, не беспокоясь об управлении состоянием.

agent

Сервер может объявить возможность agent со значением X (в форме agent=X ), чтобы уведомить клиента о том, что на сервере используется версия X . Клиент может дополнительно отправить свою собственную строку агента, включив возможность agent со значением Y (в форме agent=Y ) в свой запрос к серверу (но он НЕ ДОЛЖЕН этого делать, если сервер не анонсировал возможности агента). Строки X и Y могут содержать любые печатаемые символы ASCII, кроме пробела (т. е. диапазон байтов 32 < x < 127), и обычно имеют форму "package/version" (например, "git/1.8.3.1" ). Строки агента предназначены исключительно для сбора статистики и отладки и НЕ ДОЛЖНЫ использоваться для программного предположения о наличии или отсутствии определенных функций.

ls-refs

ls-refs — это команда, используемая для запроса справочного объявления в v2. В отличие от текущего объявления ссылки, ls-refs принимает аргументы, которые можно использовать для ограничения количества ссылок, отправляемых с сервера.

Дополнительные функции, не поддерживаемые в базовой команде, будут объявлены как значение команды в объявлении о возможностях в виде списка функций, разделенных пробелами: "<command>=<feature 1> <feature 2>"

ls-refs принимает следующие аргументы:

   symrefs
In addition to the object pointed by it, show the underlying ref
pointed by it when showing a symbolic ref.
   peel
Show peeled tags.
   ref-prefix <prefix>
When specified, only references having a prefix matching one of
the provided prefixes are displayed. Multiple instances may be
given, in which case references matching any prefix will be
shown. Note that this is purely for optimization; a server MAY
show refs not matching the prefix if it chooses, and clients
should filter the result themselves.

Если объявлена ​​функция unborn , в запрос клиента можно включить следующий аргумент.

   unborn
The server will send information about HEAD even if it is a symref
pointing to an unborn branch in the form "unborn HEAD
symref-target:<target>".

Вывод ls-refs следующий:

output = *ref
  flush-pkt
obj-id-or-unborn = (obj-id | "unborn")
ref = PKT-LINE(obj-id-or-unborn SP refname *(SP ref-attribute) LF)
ref-attribute = (symref | peeled)
symref = "symref-target:" symref-target
peeled = "peeled:" obj-id

fetch

fetch — это команда, используемая для получения пак-файла в v2. Его можно рассматривать как модифицированную версию выборки v1, в которой удалена рекламная ссылка (поскольку эту роль выполняет команда ls-refs ), а формат сообщения изменен для устранения избыточности и обеспечения возможности легкого добавления будущих расширений.

Дополнительные функции, не поддерживаемые в базовой команде, будут объявлены как значение команды в объявлении о возможностях в виде списка функций, разделенных пробелами: "<command>=<feature 1> <feature 2>"

Запрос fetch может принимать следующие аргументы:

   want <oid>
Indicates to the server an object which the client wants to
retrieve.  Wants can be anything and are not limited to
advertised objects.
   have <oid>
Indicates to the server an object which the client has locally.
This allows the server to make a packfile which only contains
the objects that the client needs. Multiple строки «иметь» могут быть
supplied.
   done
Indicates to the server that negotiation should terminate (or
not even begin if performing a clone) and that the server should
use the information supplied in the request to construct the
packfile.
   thin-pack
Request that a thin pack be sent, which is a pack with deltas
which reference base objects not contained within the pack (but
are known to exist at the receiving end). This can reduce the
network traffic significantly, but it requires the receiving end
to know how to "thicken" these packs by adding the missing bases
to the pack.
   no-progress
Request that progress information that would normally be sent on
side-band channel 2, during the packfile transfer, should not be
sent.  However, the side-band channel 3 is still used for error
responses.
   include-tag
Request that annotated tags should be sent if the objects they
point to are being sent.
   ofs-delta
Indicate that the client understands PACKv2 with delta referring
to its base by position in pack rather than by an oid.  That is,
they can read OBJ_OFS_DELTA (aka type 6) in a packfile.

Если объявлена ​​функция shallow , в запрос клиента можно включить следующие аргументы, а также потенциальное добавление раздела shallow-info в ответ сервера, как описано ниже.

   shallow <oid>
A client must notify the server of all commits for which it only
has shallow copies (meaning that it doesnу меня нет родителей
a commit) by supplying a строка 'shallow <oid>' для каждого такого
object so that the server is aware of the limitations of the
clientистория. Это делается для того, чтобы сервер знал, что
client may not have all objects reachable from such commits.
   deepen <depth>
Requests that the fetch/clone should be shallow having a commit
depth of <depth> relative to the remote side.
   deepen-relative
Requests that the semantics of the "deepen" command be changed
to indicate that the depth requested is relative to the client's
current shallow boundary, instead of relative to the requested
commits.
   deepen-since <timestamp>
Requests that the shallow clone/fetch should be cut at a
specific time, instead of depth.  Internally it's equivalent to
doing "git rev-list --max-age=<timestamp>". Cannot be used with
"deepen".
   deepen-not <rev>
Requests that the shallow clone/fetch should be cut at a
specific revision specified by '<rev>' вместо глубины.
Internally itэквивалент  "git rev-list --not <rev>" .
Cannot be used with "deepen", but can be used with
"deepen-since".

Если рекламируется функция filter , в запрос клиента можно включить следующий аргумент:

   filter <filter-spec>
Request that various objects from the packfile be omitted
using one of several filtering techniques. These are intended
for use with partial clone and partial fetch operations. See
`rev-list` for possible "filter-spec" values. When communicating
with other processes, senders SHOULD translate scaled integers
(e.g. "1k") into a fully-expanded form (e.g. "1024") to aid
interoperability with older receivers that may not understand
newly-invented scaling suffixes. However, receivers SHOULD
accept the following suffixes: 'k', 'm', and 'g' for 1024,
1048576, and 1073741824, respectively.

Если объявлена ​​функция ref-in-want , в запрос клиента можно включить следующий аргумент, а также потенциальное добавление раздела wanted-refs в ответ сервера, как описано ниже.

   want-ref <ref>
Indicates to the server that the client wants to retrieve a
particular ref, where <ref> is the full name of a ref on the
server.

Если рекламируется функция sideband-all , в запрос клиента можно включить следующий аргумент:

   sideband-all
Instruct the server to send the whole response multiplexed, not just
the packfile section. All non-flush and non-delim PKT-LINE in the
response (not only in the packfile section) will then start with a byte
indicating its sideband (1, 2, or 3), and the server may send "0005\2"
(a PKT-LINE of sideband 2 with no payload) as a keepalive packet.

Если объявлена ​​функция packfile-uris , в запрос клиента можно включить следующий аргумент, а также потенциальное добавление раздела packfile-uris в ответ сервера, как описано ниже.

   packfile-uris <comma-separated list of protocols>
Indicates to the server that the client is willing to receive
URIs of any of the given protocols in place of objects in the
sent packfile. Before performing the connectivity check, the
client should download from all given URIs. Currently, the
protocols supported are "http" and "https".

Если объявлена ​​функция wait-for-done , в запрос клиента можно включить следующий аргумент.

   wait-for-done
Indicates to the server that it should never send "ready", but
should wait for the client to say "done" before sending the
packfile.

Ответ fetch разбит на несколько разделов, разделенных пакетами-разделителями (0001), причем каждый раздел начинается с заголовка раздела. Большинство разделов отправляются только при отправке пак-файла.

output = acknowledgements flush-pkt |
  [acknowledgments delim-pkt] [shallow-info delim-pkt]
  [wanted-refs delim-pkt] [packfile-uris delim-pkt]
  packfile flush-pkt
acknowledgments = PKT-LINE("acknowledgments" LF)
    (nak | *ack)
    (ready)
ready = PKT-LINE("ready" LF)
nak = PKT-LINE("NAK" LF)
ack = PKT-LINE("ACK" SP obj-id LF)
shallow-info = PKT-LINE("shallow-info" LF)
 *PKT-LINE((shallow | unshallow) LF)
shallow = "shallow" SP obj-id
unshallow = "unshallow" SP obj-id
wanted-refs = PKT-LINE("wanted-refs" LF)
*PKT-LINE(wanted-ref LF)
wanted-ref = obj-id SP refname
packfile-uris = PKT-LINE("packfile-uris" LF) *packfile-uri
packfile-uri = PKT-LINE(40*(HEXDIGIT) SP *%x20-ff LF)
packfile = PKT-LINE("packfile" LF)
    *PKT-LINE(%x01-03 *%x00-ff)
   acknowledgments section
* If the client determines that it is finished with negotiations by
  sending a "done" line (thus requiring the server to send a packfile),
  the acknowledgments sections MUST be omitted from the server's
  response.

  • Всегда начинается с заголовка раздела "acknowledgments" .

  • Сервер ответит "NAK" , если none идентификаторов объектов, отправленных как строки, были общими.

  • Сервер ответит "ACK obj-id" для всех идентификаторов объектов, отправленных как общие строки.

  • В ответе не может быть одновременно строки "ACK" и строка "NAK" .

  • Сервер ответит строкой "ready" , указывающей, что сервер нашел приемлемую общую базу и готов создать и отправить пакетный файл (который будет найден в разделе пакетного файла того же ответа).

  • Если сервер нашел подходящую точку отсечения и решил отправить строку "ready" , тогда сервер может решить (в целях оптимизации) пропустить любые строки "ACK" , которые он мог бы отправить во время своего ответа. Это связано с тем, что сервер уже определил объекты, которые он планирует отправить клиенту, и дальнейшие согласования не требуются.

       shallow-info section
* If the client has requested a shallow fetch/clone, a shallow
  client requests a fetch or the server is shallow then the
  serverОтвет пользователя может включать раздел с поверхностной информацией.
  shallow-info section will be included if (due to one of the
  above conditions) the server needs to inform the client of any
  shallow boundaries or adjustments to the clients already
  existing shallow boundaries.

  • Всегда начинается с заголовка раздела "shallow-info" .

  • Если запрошена положительная глубина, сервер вычислит набор коммитов, глубина которых не превышает желаемой.

  • Сервер отправляет строку "shallow obj-id" для каждого коммита, родительские элементы которого не будут отправлены в следующем пакетном файле.

  • Сервер отправляет строку "unshallow obj-id" для каждого коммита, который, по мнению клиента, является поверхностным, но перестает быть поверхностным в результате выборки (поскольку его родительские элементы отправляются в следующем пакетном файле).

  • Сервер НЕ ДОЛЖЕН отправлять какие-либо строки "unshallow" для чего-либо, что клиент не указал как часть своего запроса.

       wanted-refs section
* This section is only included if the client has requested a
  ref using a 'want-ref' line and if a packfile section is also
  included in the response.

  • Всегда начинается с заголовка раздела "wanted-refs" .

  • Сервер отправит список ссылок ( "<oid> <refname>" ) для каждой ссылки, запрошенной с использованием строк want-ref .

  • Сервер НЕ ДОЛЖЕН отправлять какие-либо ссылки, которые не были запрошены с использованием линий want-ref .

       packfile-uris section
* This section is only included if the client sent
  «packfile-uris», и на сервере есть хотя бы один такой URI для
  send.

  • Всегда начинается с заголовка раздела "packfile-uris" .

  • Для каждого URI, отправляемого сервером, он отправляет хэш содержимого пакета (выдаваемый индексным пакетом git ), за которым следует URI.

  • Хэши имеют длину 40 шестнадцатеричных символов. Когда Git обновится до нового алгоритма хеширования, возможно, его потребуется обновить. (Он должен соответствовать выводам индексного пакета после "pack\t" или "keep\t" .

       packfile section
* This section is only included if the client has sent 'want'
  lines in its request and either requested that no more
  negotiation be done by sending «готово» или если на сервере есть
  decided it has found a sufficient cut point to produce a
  packfile.

  • Всегда начинается с заголовка раздела "packfile" .

  • Передача пакфайла начинается сразу после заголовка раздела

  • Передача данных файла пакета всегда мультиплексируется с использованием той же семантики, что и возможность side-band-64k из версии протокола 1. Это означает, что каждый пакет в потоке данных файла пакета состоит из ведущей 4-байтовой длины строки pkt (типично формата pkt-line), за которым следует 1-байтовый потоковый код, за которым следуют фактические данные.

     The stream code can be one of:
1 - pack data
2 - progress messages
3 - fatal error message just before stream aborts

server-option

Если объявлено, это означает, что в запрос можно включить любое количество параметров, специфичных для сервера. Это делается путем отправки каждой опции в виде строки возможностей "server-option=<option>" в разделе списка возможностей запроса.

Предоставленные параметры не должны содержать символы NUL или LF.

object-format

Сервер может объявить возможность object-format со значением X (в форме object-format=X ), чтобы уведомить клиента о том, что сервер может работать с объектами, используя алгоритм хеширования X. Если не указано, предполагается, что сервер обрабатывает только SHA-1. Если клиент желает использовать алгоритм хеширования, отличный от SHA-1, ему следует указать строку формата объекта.

session-id=<session id>

Сервер может объявлять идентификатор сеанса, который можно использовать для идентификации этого процесса в нескольких запросах. Клиент также может сообщить серверу свой собственный идентификатор сеанса.

Идентификаторы сеансов должны быть уникальными для данного процесса. Они должны помещаться в строку пакета и не должны содержать непечатаемые символы или символы пробелов. Текущая реализация использует идентификаторы сеансов трассировки2 (подробности см. в api-trace2 ), но это может измениться, и пользователям идентификатора сеанса не следует полагаться на этот факт.

object-info

object-info — это команда для получения информации об одном или нескольких объектах. Его основная цель — позволить клиенту принимать решения на основе этой информации без необходимости полной выборки объектов. Размер объекта — единственная информация, которая поддерживается в настоящее время.

Запрос object-info принимает следующие аргументы:

size
Requests size information to be returned for each listed object id.
oid <oid>
Indicates to the server an object which the client wants to obtain
information for.

Ответ object-info представляет собой список идентификаторов запрошенных объектов и связанной с ними запрошенной информации, каждый из которых разделен одним пробелом.

output = info flush-pkt
info = PKT-LINE(attrs) LF)
        *PKT-LINE(obj-info LF)
attrs = attr | attrs SP attrs
attr = "size"
obj-info = obj-id SP obj-size

bundle-uri

Если объявлена ​​возможность bundle-uri , сервер поддерживает команду Bundle-uri.

В настоящее время эта возможность объявлена ​​без значения (т. е. не "bundle-uri=somevalue" ), в будущем может быть добавлено значение для поддержки расширений всей команды. Клиенты ДОЛЖНЫ игнорировать любые неизвестные значения возможностей и продолжить диалог «bundle-uri», который они поддерживают.

Команда bundle-uri должна быть выполнена до fetch , чтобы получить URI для объединения файлов (см. git-bundle[1] ) в "seed" и сообщить последующую команду fetch .

Клиент МОЖЕТ выдать bundle-uri до или после любой другой допустимой команды. Чтобы быть полезным для клиентов, ожидается, что он будет выдан после ls-refs и до fetch , но МОЖЕТ быть выдан в любое время в диалоговом окне.

ОБСУЖДЕНИЕ Bundle-uri

Целью этой функции является оптимизация потребления ресурсов сервера в обычном случае путем изменения обычного случая выборки очень большого PACK во время git-clone[1] на меньшую инкрементальную выборку.

Это также позволяет серверам добиться лучшего кэширования в сочетании с uploadpack.packObjectsHook (см. git-config[1] ).

Наличие новых клонов или выборок становится более предсказуемым и распространенным решением по сравнению с недавно созданными файлами *.bundle. Серверы могут даже заранее генерировать результаты таких переговоров для uploadpack.packObjectsHook по мере поступления новых запросов.

Один из способов, с помощью которого серверы могут воспользоваться преимуществами этих пакетов, заключается в том, что сервер будет ожидать, что новые клоны загрузят известный пакет, а затем проверят текущее состояние репозитория, используя подсказки по ссылкам, найденные в этом пакете (или пакетах).

ПРОТОКОЛ для Bundle-uri

Запрос bundle-uri не принимает аргументов и, как отмечалось выше, в настоящее время не объявляет значение возможности. Оба могут быть добавлены в будущем.

Когда клиент отправляет запрос command=bundle-uri , ответом является список пар ключ-значение, представленный в виде строк пакета со значением <key>=<value> . Каждый <key> следует интерпретировать как ключ конфигурации из пространства имен bundle.* для создания списка пакетов. Эти ключи сгруппированы в подраздел bundle.<id>. , где каждый ключ, соответствующий данному <id> , вносит атрибуты в пакет, определенный этим <id> . См. git-config[1] для получения подробной информации об этих ключах и о том, как клиент Git будет интерпретировать их значения.

Клиенты ДОЛЖНЫ проанализировать строку в соответствии с указанным выше форматом, строки, не соответствующие формату, СЛЕДУЕТ отбросить. В таком случае пользователь МОЖЕТ быть предупрежден.

Bundle-uri ОЖИДАНИЯ КЛИЕНТА И СЕРВЕРА

URI CONTENTS

Содержимое объявленных URI ДОЛЖНО быть одного из двух типов.

Объявленный URI может содержать файл пакета, который принимает git bundle verify . Т.е. они ДОЛЖНЫ содержать один или несколько справочных советов для использования клиентом, ДОЛЖНЫ указывать предварительные условия (любые) со стандартными префиксами "-" и ДОЛЖНЫ указывать свой "object-format" , если применимо.

Объявленный URI может альтернативно содержать текстовый файл, который будет принимать git config --list (с опцией --file ). Пары ключ-значение в этом списке находятся в пространстве имен bundle.* (см. git-config[1] ).

Bundle-uri ВОССТАНОВЛЕНИЕ ОШИБОК КЛИЕНТА

Клиент ДОЛЖЕН, прежде всего, плавно снижать производительность при ошибках, независимо от того, связана ли эта ошибка с недостатком данных в URI пакета, потому что этот клиент слишком туп, чтобы, например, понять и полностью проанализировать заголовки пакетов и их обязательные отношения или что-то в этом роде. еще.

Операторы серверов должны чувствовать себя уверенно при включении "bundle-uri" и не беспокоиться, если, например, их CDN выйдет из строя, что клонирование или выборка столкнутся с серьезными сбоями. Даже если комплект(ы) сервера неполны или каким-то образом неисправны, клиент все равно должен получить функционирующий репозиторий, как если бы он решил не использовать это расширение протокола.

Все последующие обсуждения взаимодействия клиента и сервера ДОЛЖНЫ помнить об этом.

Bundle-uri СЕРВЕР К КЛИЕНТУ

Порядок возвращаемых URI пакета не имеет значения. Клиенты ДОЛЖНЫ анализировать свои заголовки, чтобы обнаружить содержащиеся в них OID и необходимые условия. Клиент ДОЛЖЕН рассматривать содержимое самих пакетов и их заголовок как окончательный источник истины.

Сервер МОЖЕТ даже возвращать пакеты, которые не имеют прямого отношения к клонируемому репозиторию (либо случайно, либо преднамеренной конфигурации "clever" ), и ожидать, что клиент сам разберется, какие данные ему нужны из пакета ( с), если таковые имеются.

Bundle-uri КЛИЕНТ-СЕРВЕР

Клиент ДОЛЖЕН предоставлять справочные подсказки, найденные в заголовках пакета, в виде строк have в любом последующем запросе fetch . Клиент МОЖЕТ также полностью игнорировать пакеты, если по какой-то причине это считается худшим, например, если пакеты не могут быть загружены, ему не нравятся найденные подсказки и т. д.

КОГДА РЕКЛАМИРУЕМЫЙ ПАКЕТ(Ы) НЕ ТРЕБУЮТ ДАЛЬНЕЙШИХ ПЕРЕГОВОРОВ

Если после выдачи bundle-uri и ls-refs и получения заголовков пакетов клиент обнаруживает, что нужные ему подсказки могут быть полностью получены из рекламируемых пакетов, клиент МОЖЕТ отключиться от сервера Git . Результаты такого clone или fetch должны быть неотличимы от состояния, достигнутого без использования Bundle-uri.

РАННЕЕ ОТКЛЮЧЕНИЕ КЛИЕНТОВ И ВОССТАНОВЛЕНИЕ ОШИБОК

Клиент МОЖЕТ выполнить досрочное отключение, продолжая загружать пакеты (после потоковой передачи и анализа их заголовков). В таком случае клиент ДОЛЖЕН корректно устранить любые ошибки, связанные с завершением загрузки и проверкой пакета(ов).

Т.е. клиенту может потребоваться повторно подключиться и выполнить команду fetch и, возможно, вообще отказаться от использования bundle-uri .

Такое поведение "MAY" определяется как таковое (а не "SHOULD" ) в предположении, что URI рекламного пакета сервера, скорее всего, будет обслуживать относительно большой репозиторий и указывать на URI, которые имеют хорошие шансы оказаться в порядок работы. Клиент МОЖЕТ, например, рассматривать размер полезной нагрузки пакетов как эвристический метод, чтобы определить, стоит ли того раннее отключение, если потребуется вернуться к полному диалогу "fetch" .

КОГДА РЕКЛАМИРУЕМЫЙ ПАКЕТ(Ы) ТРЕБУЮТ ДАЛЬНЕЙШИХ ПЕРЕГОВОРОВ

Клиент ДОЛЖЕН начать согласование ПАКЕТА с сервером через команду "fetch" , используя подсказки OID, найденные в рекламируемых пакетах, даже если он все еще находится в процессе загрузки этих пакетов.

Это допускает агрессивное раннее отключение от любого интерактивного диалога сервера. Клиент слепо верит, что рекламируемые подсказки OID актуальны, и выдает их как строки have , а затем запрашивает любые подсказки, которые ему нужны (обычно из рекламы "ls-refs" ), через линии want . Затем сервер вычислит (надеюсь, небольшой) PACK с ожидаемой разницей между советами из пакета(ов) и запрошенными данными.

Единственное соединение, которое клиент затем должен поддерживать активным, - это одновременно загружаемые пакеты static , когда они и инкрементный ПАКЕТ извлекаются, они должны быть раздуты и проверены. Любые ошибки на этом этапе должны быть корректно устранены, см. выше.

ОСОБЕННОСТИ ПРОТОКОЛА Bundle-uri

Клиент создает список пакетов из пар <key>=<value> , предоставленных сервером. Эти пары являются частью пространства имен bundle.* , как описано в git-config[1] . В этом разделе мы обсудим некоторые из этих ключей и опишем действия, которые клиент будет выполнять в ответ на эту информацию.

В частности, ключ bundle.version определяет целочисленное значение. Единственное допустимое значение на данный момент — 1 , но если клиент видит здесь неожиданное значение, он ДОЛЖЕН игнорировать список пакетов.

Пока bundle.version понятен, все остальные неизвестные ключи МОГУТ игнорироваться клиентом. Сервер гарантирует совместимость со старыми клиентами, хотя новые клиенты могут лучше использовать дополнительные ключи для минимизации загрузок.

Любое обратно несовместимое добавление значения ключа перед URI будет защищено новым значением или значениями bundle.version в самом объявлении возможности bundle-uri и/или новыми будущими аргументами запроса bundle-uri .

Некоторые примеры пар ключ-значение, которые в настоящее время не реализованы, но могут быть реализованы в будущем, включают:

  • Добавьте "hash=<val>" или "size=<bytes>" , чтобы объявить ожидаемый хэш или размер файла пакета.

  • Объявите, что один или несколько файлов пакета одинаковы (например, чтобы клиенты выполняли циклический перебор или иным образом выбирали один из N возможных файлов).

  • Ярлык "oid=<OID>" и ярлык "prerequisite=<OID>" . Для выражения обычного случая пакета с одним советом и без предварительных условий или с одним советом и одним предварительным условием.

    Это позволило бы оптимизировать общий случай серверов, которые хотели бы предоставить один "big bundle" , содержащий только их ветку "main" , и/или ее инкрементальные обновления.

    Клиент, получающий такой ответ, МОЖЕТ предположить, что он может пропустить получение заголовка из пакета по указанному URI и, таким образом, сохранить себе и серверу(ам) запрос(ы), необходимый для проверки заголовков этого пакета или пакетов.
