git-send-pack

Name

git-send-pack — отправлять объекты по протоколу Git в другой репозиторий.

Synopsis

git send-pack [--mirror] [--dry-run] [--force]
                [--receive-pack=<git-receive-pack>]
                [--verbose] [--thin] [--atomic]
                [--[no-]signed | --signed=(true|false|if-asked)]
                [<host>:]<directory> (--all | <ref>…​)

Description

Обычно вместо этого вы предпочитаете использовать git push , который является оболочкой более высокого уровня этой команды. См. git-push[1] .

Вызывает git-receive-pack в возможно удаленном репозитории и обновляет его из текущего репозитория, отправляя именованные ссылки.

Options

--receive-pack=<git-receive-pack>

Путь к программе git-receive-pack на удаленном конце. Иногда полезно при отправке в удаленный репозиторий через ssh, и у вас нет программы в каталоге по умолчанию $PATH.

--exec=<git-receive-pack>

То же, что --receive-pack=<git-receive-pack>.

--all

Вместо того, чтобы явно указывать, какие ссылки обновлять, обновите все главы, существующие локально.

--stdin

Возьмите список ссылок из стандартного ввода, по одной в строке. Если помимо этой опции в командной строке указаны ссылки, то ссылки из стандартного ввода обрабатываются после ссылок в командной строке.

Если вместе с этой опцией указан --stateless-rpc , то список ссылок должен быть в пакетном формате (pkt-line). Каждая ссылка должна находиться в отдельном пакете, а список должен заканчиваться флеш-пакетом.

--dry-run

Делайте все, кроме фактической отправки обновлений.

--force

Обычно команда отказывается обновлять удаленную ссылку, которая не является предком локальной ссылки, использованной для ее перезаписи. Этот флаг отключает проверку. Это означает, что удаленный репозиторий может потерять коммиты; используйте его с осторожностью.

--verbose

Run verbosely.

--thin

Отправьте пакет "thin" , который записывает объекты в разделенной форме на основе объектов, не включенных в пакет, чтобы уменьшить сетевой трафик.

--atomic

Используйте атомарную транзакцию для обновления ссылок. Если какая-либо из ссылок не обновится, то вся передача завершится неудачей без изменения каких-либо ссылок.

--[no-]signed
--signed=(true|false|if-asked)

Подпишите GPG push-запрос для обновления ссылок на принимающей стороне, чтобы его можно было проверить перехватчиками и/или зарегистрировать. Если false или --no-signed , попытка подписи не будет осуществляться. Если true или --signed , передача завершится неудачно, если сервер не поддерживает подписанные отправки. Если установлено значение if-asked , подпишите тогда и только тогда, когда сервер поддерживает подписанные push-уведомления. Отправка также завершится неудачно, если фактический вызов gpg --sign завершится неудачно. Подробную информацию о принимающей стороне см. в git-receive-pack[1] .

--push-option=<string>

Передайте указанную строку в качестве опции push для использования перехватчиками на стороне сервера. Если сервер не поддерживает параметры push, выдается ошибка. Подробности см. в git-push[1] и githooks[5] .

<host>

Удаленный хост для размещения репозитория. Если указана эта часть, git-receive-pack вызывается через ssh.

<directory>

Репозиторий для обновления.

<ref>…

Удаленные ссылки для обновления.

Указание ссылок

Существует три способа указать, какие ссылки обновлять на удаленном конце.

С флагом --all все ссылки, существующие локально, передаются на удаленную сторону. Вы не можете указать какой-либо <ref> , если используете этот флаг.

Без --all и без всякого <ref> обновляются головки, существующие как на локальной стороне, так и на удаленной.

Когда один или несколько <ref> указаны явно (в командной строке или через --stdin ), это может быть либо один шаблон, либо пара таких шаблонов, разделенных двоеточием ":" (это означает, что имя ссылки не может иметь двоеточие в это). Одиночный шаблон <name> — это просто сокращение от <name>:<name> .

Каждая пара шаблонов состоит из стороны источника (до двоеточия) и стороны назначения (после двоеточия). Ссылка для отправки определяется путем поиска совпадения, соответствующего исходной стороне, а место ее отправки определяется с использованием стороны назначения. Правила, используемые для сопоставления ссылки, — это те же правила, которые используются git rev-parse для разрешения символического имени ссылки. См. git-rev-parse[1] .

  • Если <src> не соответствует ни одной из локальных ссылок, это ошибка.

  • Если <dst> соответствует более чем одной удаленной ссылке, это ошибка.

  • Если <dst> не соответствует ни одной удаленной ссылке, либо

    • он должен начинаться с "refs/" ; В этом случае <dst> используется в качестве пункта назначения буквально.

    • <src> == <dst> и ссылка, соответствующая <src>, не должна существовать в наборе удаленных ссылок; ссылка, соответствующая <src> локально, используется в качестве имени места назначения.

Без --force ссылка <src> сохраняется на удаленном компьютере, только если <dst> не существует или <dst> является правильным подмножеством (т. е. предком) <src>. Эта проверка, известная как "fast-forward check" , выполняется во избежание случайной перезаписи удаленной ссылки и потери оттуда коммитов других людей.

В --force проверка перемотки вперед отключена для всех ссылок.

При желании параметр <ref> может иметь префикс со знаком плюс + , чтобы отключить проверку быстрой перемотки вперед только для этой ссылки.
