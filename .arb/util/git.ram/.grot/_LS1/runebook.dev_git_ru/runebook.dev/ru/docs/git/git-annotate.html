git-annotate

Name

git-annotate — аннотировать строки файла информацией о фиксации.

Synopsis

git annotate [<options>] [<rev-opts>] [<rev>] [--] <file>

Description

Аннотирует каждую строку в данном файле информацией из коммита, в котором введена эта строка. Необязательно — аннотации из данной ревизии.

Единственная разница между этой командой и git-blame[1] заключается в том, что они используют несколько разные форматы вывода, и эта команда существует только для обратной совместимости для поддержки существующих сценариев и предоставления более знакомого имени команды для людей, пришедших из других систем SCM.

Options

-b

Покажите пустой SHA-1 для граничных коммитов. Этим также можно управлять с помощью опции конфигурации blame.blankBoundary .

--root

Не рассматривайте корневые коммиты как границы. Этим также можно управлять с помощью опции конфигурации blame.showRoot .

--show-stats

Включите дополнительную статистику в конце вывода об обвинениях.

-L <start>,<end>
-L :<funcname>

Аннотируйте только диапазон строк, заданный <start>,<end> или именем функции regex <funcname> . Может быть указано несколько раз. Допускаются перекрывающиеся диапазоны.

<start> и <end> являются дополнительными. -L <start> или -L <start>, простираются от <start> до конца файла. -L ,<end> простирается от начала файла до <end> .

<start> и <end> могут принимать одну из следующих форм:

  • number

    Если <start> или <end> — число, оно указывает абсолютный номер строки (строки отсчитываются с 1).

  • /regex/

    В этой форме будет использоваться первая строка, соответствующая данному регулярному выражению POSIX. Если <start> является регулярным выражением, поиск будет осуществляться с конца предыдущего диапазона -L , если таковой имеется, в противном случае — с начала файла. Если <start> равен ^/regex/ , поиск будет начинаться с начала файла. Если <end> является регулярным выражением, поиск будет начинаться со строки, заданной <start> .

  • +смещение или -смещение

    Это справедливо только для <end> и будет указывать количество строк до или после строки, заданной <start> .

Если вместо <start> и <end> указан :<funcname> , это регулярное выражение, обозначающее диапазон от первой строки имени функции, соответствующей <funcname> , до следующей строки имени функции. :<funcname> выполняет поиск с конца предыдущего диапазона -L , если таковой имеется, в противном случае — с начала файла. ^:<funcname> выполняет поиск с начала файла. Имена функций определяются так же, как git diff обрабатывает заголовки фрагментов патчей (см. Defining a custom hunk-header в gitattributes[5] ).

-l

Показывать длинные обороты (по умолчанию: выключено).

-t

Показать необработанную временную метку (по умолчанию: выключено).

-S <revs-file>

Используйте версии из файла revs вместо вызова git-rev-list[1] .

--reverse <rev>..<rev>

Идите по истории вперед, а не назад. Вместо отображения ревизии, в которой появилась строка, отображается последняя ревизия, в которой существовала строка. Это требует ряда изменений, таких как START..END, где виновный путь находится в START. git blame --reverse START для удобства принят за git blame --reverse START..HEAD .

--first-parent

Следуйте только первому родительскому коммиту, увидев коммит слияния. Эту опцию можно использовать для определения того, когда строка была добавлена ​​в конкретную ветку интеграции, а не когда она была введена в историю в целом.

-p
--porcelain

Показать в формате, предназначенном для машинного потребления.

--line-porcelain

Покажите фарфоровый формат, но выведите информацию о фиксации для каждой строки, а не только при первом обращении к фиксации. Подразумевается --фарфор.

--incremental

Покажите результат постепенно в формате, предназначенном для машинного использования.

--encoding=<encoding>

Указывает кодировку, используемую для вывода имен авторов и сводок коммитов. Установка значения none приводит к выводу непреобразованных данных. Для получения дополнительной информации см. обсуждение кодирования на странице руководства git-log[1] .

--contents <file>

Аннотируйте, используя содержимое именованного файла, начиная с <rev>, если он указан, и HEAD в противном случае. Вы можете указать - , чтобы команда считывала содержимое файла из стандартного ввода.

--date <format>

Указывает формат, используемый для вывода дат. Если параметр --date не указан, используется значение переменной конфигурации Assessment.date. Если переменная конфигурации обвиняемого.дата также не установлена, используется формат iso. Поддерживаемые значения см. в обсуждении параметра --date на git-log[1] .

--[no-]progress

Статус выполнения по умолчанию сообщается в стандартном потоке ошибок, когда он подключен к терминалу. Этот флаг включает отчеты о ходе выполнения, даже если он не подключен к терминалу. Невозможно использовать --progress вместе с --porcelain или --incremental .

-M[<num>]

Обнаружение перемещенных или скопированных строк в файле. Когда фиксация перемещает или копирует блок строк (например, в исходном файле есть A, а затем B, а фиксация меняет его на B, а затем на A), традиционный алгоритм blame замечает только половину перемещения и обычно винит строки, которые были перемещен вверх (т.е. B) к родительскому коммиту и возлагает вину на строки, которые были перемещены вниз (т.е. A) к дочернему коммиту. При использовании этой опции ответственность за обе группы линий возлагается на родительскую группу за счет выполнения дополнительных проходов проверки.

<num> является необязательным, но это нижняя граница количества буквенно-цифровых символов, которые Git должен обнаружить как перемещение/копирование в файле, чтобы связать эти строки с родительским коммитом. Значение по умолчанию — 20.

-C[<num>]

Помимо -M , обнаруживает строки, перемещенные или скопированные из других файлов, которые были изменены в том же коммите. Это полезно, когда вы реорганизуете свою программу и перемещаете код по файлам. Если эта опция задана дважды, команда дополнительно ищет копии других файлов в фиксации, создающей файл. Если эта опция задана три раза, команда дополнительно ищет копии других файлов в любом коммите.

<num> является необязательным, но это нижняя граница количества буквенно-цифровых символов, которые Git должен обнаружить как перемещение/копирование между файлами, чтобы связать эти строки с родительским коммитом. Значение по умолчанию — 40. Если задано более одного параметра -C , вступит в силу аргумент <num> последнего -C .

--ignore-rev <rev>

Игнорируйте изменения, внесенные редакцией при назначении вины, как будто изменений никогда не было. Строки, которые были изменены или добавлены игнорируемым коммитом, будут связаны с предыдущим коммитом, который изменил эту строку или близлежащие строки. Эту опцию можно указать несколько раз, чтобы игнорировать более одной ревизии. Если установлена ​​опция конфигурации blame.markIgnoredLines , то строки, которые были изменены игнорируемым коммитом и приписаны другому коммиту, будут помечены ? в выводе об обвинениях. Если установлена ​​опция конфигурации blame.markUnblamableLines , то строки, затронутые игнорируемым коммитом, которые мы не могли отнести к другой ревизии, помечаются * .

--ignore-revs-file <file>

Игнорируйте версии, перечисленные в file , которые должны быть в том же формате, что и fsck.skipList . Эту опцию можно повторить, и эти файлы будут обрабатываться после любых файлов, указанных в опции конфигурации blame.ignoreRevsFile . Пустое имя файла "" очистит список оборотов из ранее обработанных файлов.

--color-lines

Цветовые аннотации строк в формате по умолчанию различаются, если они взяты из того же коммита, что и предыдущая строка. Это облегчает различение блоков кода, введенных разными коммитами. Цвет по умолчанию — голубой, и его можно настроить с помощью параметра конфигурации color.blame.repeatedLines .

--color-by-age

Цвет аннотаций линий в зависимости от возраста линии в формате по умолчанию. Параметр конфигурации color.blame.highlightRecent определяет, какой цвет используется для каждого возрастного диапазона.

-h

Показать справочное сообщение.
