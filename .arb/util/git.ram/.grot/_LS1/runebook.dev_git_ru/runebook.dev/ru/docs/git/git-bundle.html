git-bundle

Name

git-bundle — перемещение объектов и ссылок по архиву

Synopsis

git bundle create [-q | --quiet | --progress]
                    [--version=<version>] <file> <git-rev-list-args>
git bundle verify [-q | --quiet] <file>
git bundle list-heads <file> [<refname>…​]
git bundle unbundle [--progress] <file> [<refname>…​]

Description

Создавайте, распаковывайте файлы "bundle" и манипулируйте ими. Пакеты используются для передачи "offline" объектов Git без активного "server" , находящегося на другой стороне сетевого подключения.

Их можно использовать для создания как инкрементальных, так и полных резервных копий репозитория, а также для передачи состояния ссылок из одного репозитория в другой.

Команды Git , которые извлекают или иным образом "read" через такие протоколы, как ssh:// и https:// , также могут работать с файлами пакета. Можно git-clone[1] создать новый репозиторий из пакета, использовать git-fetch[1] для извлечения из него и вывести список содержащихся в нем ссылок с помощью git-ls-remote[1] . Соответствующей поддержки "write" нет, т.е. git push в комплекте не поддерживается.

Примеры использования пакетов см. в разделе "EXAMPLES" ниже.

Bundle format

Пакеты — это файлы .pack (см. git-pack-objects[1] ) с заголовком, указывающим, какие ссылки содержатся в пакете.

Как и сам формат упакованного архива, пакеты могут быть автономными или создаваться с использованием исключений. См. раздел "OBJECT PREREQUISITES" ниже.

Пакеты, созданные с использованием исключений редакций, — это "thin packs" , созданные с использованием параметра --thin для git-pack-objects[1] и разделенные с помощью параметра --fix-thin для git-index-pack[1] .

При использовании исключений версий невозможно создать "thick pack" , и пользователи не должны беспокоиться об этой разнице. При использовании "thin packs" пакеты, созданные с использованием исключений, имеют меньший размер. То, что под капотом находится "thin" , отмечено здесь просто как любопытство и как ссылка на другую документацию.

См. gitformat-bundle[5] для получения более подробной информации и обсуждение "thin pack" в gitformat-pack[5] для получения дополнительной информации.

Options

создать [опции] <файл> <git-rev-list-args>

Используется для создания пакета с именем file . Для этого требуются аргументы <git-rev-list-args> для определения содержимого пакета. options содержит параметры, специфичные для подкоманды git bundle create . Если file равен - , пакет записывается в стандартный вывод.

verify <file>

Используется для проверки правильности файла пакета и его корректного применения к текущему репозиторию. Это включает в себя проверку самого формата пакета, а также проверку существования обязательных коммитов и их полной связи с текущим репозиторием. Затем git bundle печатает список отсутствующих коммитов, если таковые имеются. Наконец, печатается информация о дополнительных возможностях, например "object filter" . Для получения дополнительной информации см. "Capabilities" в gitformat-bundle[5] . Код выхода равен нулю в случае успеха, но будет ненулевым, если файл пакета недействителен. Если file — - , пакет считывается из стандартного ввода.

list-heads <file>

Перечисляет ссылки, определенные в пакете. Если за ним следует список ссылок, распечатываются только ссылки, соответствующие указанным. Если file равен - , пакет считывается со стандартного ввода.

unbundle <file>

Передает объекты в пакете git index-pack для хранения в репозитории, затем печатает имена всех определенных ссылок. Если указан список ссылок, печатаются только ссылки, соответствующие ссылкам в списке. Эта команда на самом деле является водопроводной и предназначена для вызова только с помощью git fetch . Если file — - , пакет считывается из стандартного ввода.

<git-rev-list-args>

Список аргументов, приемлемый для git rev-parse и git rev-list (и содержащий именованную ссылку, см. ОПРЕДЕЛЕНИЕ ССЫЛОК ниже), который определяет конкретные объекты и ссылки для транспорта. Например, master~10..master приводит к упаковке текущей главной ссылки вместе со всеми объектами, добавленными с момента ее 10-го предка. Не существует явного ограничения на количество ссылок и объектов, которые могут быть упакованы.

[<refname>…​]

Список ссылок, используемый для ограничения количества ссылок, указанных как доступные. В основном это полезно для git fetch , который ожидает получить только те ссылки, которые запрошены, а не обязательно все в пакете (в этом случае git bundle действует как git fetch-pack ).

--progress

Статус выполнения сообщается в стандартном потоке ошибок по умолчанию, когда он подключен к терминалу, если не указан параметр -q. Этот флаг обеспечивает статус хода выполнения, даже если стандартный поток ошибок не направляется на терминал.

--version=<version>

Укажите версию пакета. Версия 2 является более старым форматом и может использоваться только с репозиториями SHA-1; более новая версия 3 содержит возможности, допускающие расширения. По умолчанию используется самый старый поддерживаемый формат, основанный на используемом алгоритме хеширования.

-q
--quiet

Этот флаг заставляет команду не сообщать о своем ходе выполнения в стандартном потоке ошибок.

Specifying references

Редакции должны сопровождаться названиями ссылок для упаковки в пакет.

Можно упаковать более одной ссылки и указать более одного набора необходимых объектов. Упакованные объекты — это те, которые не входят в объединение предварительных условий.

Команда git bundle create разрешает ссылочные имена, используя те же правила, что и git rev-parse --abbrev-ref=loose . Каждое предварительное условие может быть указано явно (например, ^master~10 ) или неявно (например, master~10..master , --since=10.days.ago master ).

Все эти простые случаи подходят (при условии, что у нас есть ветка "master" и "next" ):

$ git bundle create master.bundle master
$ echo master | git bundle create master.bundle --stdin
$ git bundle create master-and-next.bundle master next
$ (echo master; echo next) | git bundle create master-and-next.bundle --stdin

Вот эти (и те же примеры --stdin , но опущенные):

$ git bundle create recent-master.bundle master~10..master
$ git bundle create recent-updates.bundle master~10..master next~5..next

Имя редакции или диапазон, правая часть которого не может быть преобразована в ссылку, не принимаются:

$ git bundle create HEAD.bundle $(git rev-parse HEAD)
fatal: Refusing to create empty bundle.
$ git bundle create master-yesterday.bundle master~10..master~5
fatal: Refusing to create empty bundle.

Object prerequisites

При создании пакетов можно создать автономный пакет, который можно разделить в репозитории без общей истории, а также предоставить отрицательные версии для исключения объектов, необходимых в более ранних частях истории.

При передаче ревизии, такой как new , в git bundle create будет создан файл пакета, содержащий все объекты, доступные из ревизии new . Этот пакет можно разделить в любом репозитории, чтобы получить полную историю, ведущую к версии new :

$ git bundle create full.bundle new

Диапазон ревизий, такой как old..new , создаст файл пакета, для которого потребуется наличие ревизии old (и любых объектов, доступных из него), чтобы пакет мог поддерживать "unbundle" :

$ git bundle create full.bundle old..new

Автономный пакет без каких-либо предварительных условий можно извлечь куда угодно, даже в пустой репозиторий, или клонировать из (т. е. new , но не old..new ).

Можно проявить осторожность и привести к тому, что файл пакета будет содержать объекты, уже находящиеся в месте назначения, поскольку они игнорируются при распаковке в месте назначения.

Если вы хотите сопоставить git clone --mirror , который будет включать ваши ссылки, такие как refs/remotes/* , используйте --all . Если вы хотите предоставить тот же набор ссылок, который получит клон непосредственно из исходного репозитория, используйте --branches --tags для <git-rev-list-args> .

Команду git bundle verify можно использовать для проверки наличия в репозитории получателя необходимых предварительных коммитов для пакета.

Examples

Предположим, вы хотите перенести историю из репозитория R1 на машине A в другой репозиторий R2 на машине B. По какой-то причине прямое соединение между A и B не разрешено, но мы можем переместить данные из A в B через какой-то механизм (CD , электронная почта и т. д.). Мы хотим обновить R2, используя разработки, сделанные на мастере ветки в R1.

Чтобы запустить процесс, вы можете сначала создать пакет, не требующий каких-либо предварительных условий. Вы можете использовать тег, чтобы запомнить последний обработанный коммит, чтобы упростить позднее обновление другого репозитория с помощью инкрементного пакета:

machineA$ cd R1
machineA$ git bundle create file.bundle master
machineA$ git tag -f lastR2bundle master

Затем вы переносите file.bundle на целевой компьютер B. Поскольку этот пакет не требует извлечения какого-либо существующего объекта, вы можете создать новый репозиторий на компьютере B, клонировав его:

machineB$ git clone -b master /home/me/tmp/file.bundle R2

Это определит удаленный компьютер с именем "origin" в результирующем репозитории, который позволит вам извлекать и извлекать данные из пакета. В файле $GIT_DIR/config в R2 будет такая запись:

[remote "origin"]
    url = /home/me/tmp/file.bundle
    fetch = refs/heads/*:refs/remotes/origin/*

Чтобы обновить полученный репозиторий mine.git, вы можете получить или извлечь его после замены пакета, хранящегося в /home/me/tmp/file.bundle, инкрементными обновлениями.

Поработав еще немного в исходном репозитории, вы можете создать инкрементный пакет для обновления другого репозитория:

machineA$ cd R1
machineA$ git bundle create file.bundle lastR2bundle..master
machineA$ git tag -f lastR2bundle master

Затем вы переносите пакет на другой компьютер, чтобы заменить /home/me/tmp/file.bundle, и извлекаете его из него.

machineB$ cd R2
machineB$ git pull

Если вы знаете, до какого коммита предполагаемый репозиторий-получатель должен иметь необходимые объекты, вы можете использовать эти знания для указания предварительных условий, давая точку отсечения для ограничения ревизий и объектов, входящих в результирующий пакет. В предыдущем примере для этой цели использовался тег LastR2bundle, но вы можете использовать любые другие параметры, заданные для команды git-log[1] . Вот еще примеры:

Вы можете использовать тег, который присутствует в обоих:

$ git bundle create mybundle v1.0.0..master

Вы можете использовать предварительное условие в зависимости от времени:

$ git bundle create mybundle --since=10.days master

Вы можете использовать количество коммитов:

$ git bundle create mybundle -10 master

Вы можете запустить git-bundle verify , чтобы посмотреть, сможете ли вы извлечь из пакета, созданного с предварительным условием:

$ git bundle verify mybundle

Здесь будет перечислено, какие коммиты вам необходимы для извлечения из пакета, и выдаст ошибку, если их у вас нет.

С точки зрения репозитория-получателя пакет аналогичен обычному репозиторию, из которого он извлекается или извлекается. Например, вы можете сопоставить ссылки при извлечении:

$ git fetch mybundle master:localRef

Вы также можете увидеть, какие ссылки он предлагает:

$ git ls-remote mybundle
