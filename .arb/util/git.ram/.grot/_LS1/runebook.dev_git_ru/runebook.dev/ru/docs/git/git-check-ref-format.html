git-check-ref-format

Name

git-check-ref-format — гарантирует, что имя ссылки имеет правильный формат.

Synopsis

git check-ref-format [--normalize]
       [--[no-]allow-onelevel] [--refspec-pattern]
       <refname>
git check-ref-format --branch <branchname-shorthand>

Description

Проверяет, является ли данный refname приемлемым, и завершает работу с ненулевым статусом, если это не так.

Ссылка используется в Git для указания ветвей и тегов. Заголовок ветки хранится в иерархии refs/heads , а тег — в иерархии refs/tags пространства имен ref (обычно в каталогах $GIT_DIR/refs/heads и $GIT_DIR/refs/tags или в виде записей в файле $GIT_DIR/packed-refs , если ссылки упакованы git gc ).

Git налагает следующие правила на именование ссылок:

  1. Они могут включать косую черту / для иерархической группировки (каталогов), но ни один компонент, разделенный косой чертой, не может начинаться с точки . или заканчиваться последовательностью .lock .

  2. Они должны содержать хотя бы один / . Это обеспечивает наличие такой категории, как heads/ , tags/ и т. д., но фактические имена не ограничиваются. Если используется опция --allow-onelevel , это правило отменяется.

  3. У них нигде не может быть двух последовательных точек .. .

  4. Они не могут содержать управляющие символы ASCII (т. е. байты, значения которых меньше \040 или \177 DEL ), пробел, тильду ~ , каретку ^ или двоеточие : где угодно.

  5. У них нигде не может быть вопросительного знака ? , звездочки * или открытой скобки [ . См. опцию --refspec-pattern ниже, чтобы узнать об исключении из этого правила.

  6. Они не могут начинаться или заканчиваться косой чертой / или содержать несколько последовательных косых черт (исключение из этого правила см. в опции --normalize ниже).

  7. Они не могут заканчиваться точкой . .

  8. Они не могут содержать последовательность @{ .

  9. Они не могут быть одним символом @ .

  10. Они не могут содержать \ .

Эти правила позволяют инструментам на основе сценариев оболочки анализировать ссылочные имена, расширять путь оболочкой, когда ссылочное имя используется без кавычек (по ошибке), а также избегать двусмысленности в некоторых выражениях ссылочного имени (см. gitrevisions[7] ):

  1. Часто используется обозначение .. с двойной точкой, как в ref1..ref2 , и в некоторых контекстах это обозначение означает ^ref1 ref2 (т. е. не в ref1 , а в ref2 ).

  2. Тильда ~ и каретка ^ используются для обозначения постфиксных операций nth parent и peel onion .

  3. Двоеточие : используется, как и в srcref:dstref , для обозначения "use srcref’s value and store it in dstref" в операциях выборки и отправки. Его также можно использовать для выбора определенного объекта, например, с помощью «git cat-file»: "git cat-file blob v1.3.3:refs.c" .

  4. at-open-brace @{ используется в качестве обозначения для доступа к записи журнала обновлений.

С опцией --branch команда принимает имя и проверяет, можно ли его использовать в качестве допустимого имени ветки (например, при создании новой ветки). Но будьте осторожны при использовании предыдущего синтаксиса извлечения, который может относиться к отсоединенному состоянию HEAD. Правило, реализуемое git check-ref-format --branch $name , может быть более строгим, чем то, что говорит git check-ref-format refs/heads/$name (например, тире может появляться в начале компонента ref, но это явно запрещено в начале имени ветки). При запуске с опцией --branch в репозитории входные данные сначала расширяются для «синтаксиса предыдущей проверки» @{-n} . Например, @{-1} — это способ сослаться на последний элемент, который был извлечен с помощью операции "git switch" или "git checkout" . Эта опция должна использоваться фарфорами для принятия этого синтаксиса везде, где ожидается имя ветки, чтобы они могли действовать так, как если бы вы ввели имя ветки. В качестве исключения обратите внимание, что «предыдущая операция извлечения» может привести к присвоению имени объекта фиксации, если N-е последнее извлеченное событие не было веткой.

Options

--[no-]allow-onelevel

Определяет, принимаются ли одноуровневые имена ссылок (т. е. имена ссылок, которые не содержат несколько компонентов, разделенных / ). По умолчанию — --no-allow-onelevel .

--refspec-pattern

Интерпретируйте <refname> как шаблон ссылочного имени для спецификации ссылки (как используется в удаленных репозиториях). Если эта опция включена, <refname> может содержать один * в спецификации ссылки (например, foo/bar*/baz или foo/bar*baz/ , но не foo/bar*/baz* ).

--normalize

Нормализуйте refname , удалив все ведущие символы косой черты ( / ) и свернув ряды соседних косых черт между компонентами имени в одну косую черту. Если нормализованное имя ссылки допустимо, выведите его на стандартный вывод и выйдите со статусом 0, в противном случае выйдите с ненулевым статусом. ( --print — устаревший способ написания --normalize .)

Examples

  • Выведите имя предыдущей извлеченной вещи:

    $ git check-ref-format --branch @{-1}

  • Определите ссылочное имя, которое будет использоваться для новой ветки:

    $ ref=$(git check-ref-format --normalize "refs/heads/$newbranch")||
{ echo "we do not like '$newbranch' as a branch name." >&2 ; exit 1 ; }
