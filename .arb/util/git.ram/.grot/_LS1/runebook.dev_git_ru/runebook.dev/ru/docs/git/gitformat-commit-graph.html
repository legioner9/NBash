gitformat-commit-graph

Name

gitformat-commit-graph — формат графика фиксации Git

Synopsis

$GIT_DIR/objects/info/commit-graph
$GIT_DIR/objects/info/commit-graphs/*

Description

Граф фиксации Git хранит список OID фиксации и некоторые связанные метаданные, в том числе:

  • Номер поколения фиксации.

  • OID корневого дерева.

  • Дата фиксации.

  • Родители коммита, хранящиеся с использованием позиционных ссылок в файле графа.

  • Фильтр Блума коммита, содержащий пути, которые были изменены между коммитом и его первым родителем, если это необходимо.

Эти позиционные ссылки хранятся в виде 32-битных целых чисел без знака, соответствующих позиции массива в списке идентификаторов OID фиксации. Благодаря некоторым специальным константам, которые мы используем для отслеживания родителей, мы можем хранить не более (1 << 30) + (1 << 29) + (1 << 28) - 1 (около 1,8 миллиарда) коммитов.

Файлы графов фиксации имеют следующий формат:

Чтобы разрешить расширения, добавляющие дополнительные данные в график, мы организуем тело в "chunks" и предоставляем двоичную таблицу поиска в начале тела. Заголовок включает в себя определенные значения, такие как количество фрагментов и тип хеша.

Все многобайтовые числа располагаются в сетевом порядке байтов.

HEADER:

4-byte signature:
    The signature is: {'C', 'G', 'P', 'H'}
1-byte version number:
    Currently, the only valid version is 1.
 1-byte Hash Version
     We infer the hash length (H) from this value:
1 => SHA-1
2 => SHA-256
     If the hash type does not match the repositoryхэш-алгоритм,
     commit-graph file should be ignored with a warning presented to the
     user.
1-byte number (C) of "chunks"
1-byte number (B) of base commit-graphs
    We infer the length (H*B) of the Base Graphs chunk
    from this value.

CHUNK LOOKUP:

(C + 1) * 12 bytes listing the table of contents for the chunks:
    First 4 bytes describe the chunk id. Value 0 is a terminating label.
    Other 8 bytes provide the byte-offset in current file for chunk to
    start. (Chunks are ordered contiguously in the file, so you can infer
    the length using the next chunk position if necessary.) Each chunk
    ID appears at most once.
The CHUNK LOOKUP matches the table of contents from
the chunk-based file format, see gitformat-chunk[5]
The remaining data in the body is described one chunk at a time, and
these chunks may be given in any order. Chunks are required unless
otherwise specified.

CHUNK DATA:

OID Fanout (ID: {O, I, D, F}) (256 * 4 байта)

The ith entry, F[i], stores the number of OIDs with first
byte at most i. Thus F[255] stores the total
number of commits (N).

Поиск OID (ID: {O, I, D, L}) (N * H байт)

The OIDs for all commits in the graph, sorted in ascending order.

Данные фиксации (ID: {C, D, A, T }) (N * (H + 16) байт)

  • Первые H байтов предназначены для OID корневого дерева.

  • Следующие 8 байтов предназначены для позиций первых двух родителей i-го коммита. Сохраняет значение 0x70000000, если в этой позиции нет родителя. Если родительских элементов более двух, у второго значения включен самый старший бит, а остальные биты сохраняют позицию массива в фрагменте дополнительного краевого списка.

  • Следующие 8 байтов хранят топологический уровень (номер поколения v1) фиксации и время фиксации в секундах с момента EPOCH. Номер поколения использует старшие 30 бит первых 4 байтов, а время фиксации использует 32 бита вторых 4 байтов вместе с младшими 2 битами младшего байта, сохраняя 33-й и 34-й бит времени фиксации.

Данные генерации (идентификатор: {G, D, A, 2 }) (N * 4 байта) [Необязательно]

  • В этом списке 4-байтовых значений хранятся исправленные смещения даты фиксации для коммитов, расположенные в том же порядке, что и фрагмент данных фиксации.

  • Если исправленное смещение даты фиксации не может быть сохранено в пределах 31 бита, у значения включен самый старший бит, а остальные биты сохраняют позицию исправленной даты фиксации в фрагменте переполнения данных генерации.

  • Блок данных генерации присутствует только в том случае, если файл графа фиксации записан совместимыми версиями Git , а в случае разделенных цепочек графов фиксации самый верхний уровень также имеет фрагмент данных генерации.

Переполнение данных генерации (ID: {G, D, O, 2 }) [Необязательно]

  • В этом списке 8-байтовых значений хранятся исправленные смещения даты фиксации для коммитов с исправленными смещениями даты фиксации, которые не могут быть сохранены в пределах 31 бита.

  • Блок переполнения данных генерации присутствует только в том случае, если присутствует блок данных генерации и по крайней мере одно исправленное смещение даты фиксации не может быть сохранено в пределах 31 бита.

Дополнительный список ребер (идентификатор: {E, D, G, E}) [Необязательно]

This list of 4-byte values store the second through nth parents for
all octopus merges. The second parent value in the commit data stores
an array position within this list along with the most-significant bit
on. Starting at that array position, iterate through this list of commit
positions for the parents until reaching a value with the most-significant
bit on. The other bits correspond to the position of the last parent.

Индекс фильтра Блума (идентификатор: {B, I, D, X}) (N * 4 байта) [Необязательно]

  • i-я запись, BIDX[i], хранит количество байтов во всех фильтрах Блума от фиксации 0 до фиксации i (включительно) в лексикографическом порядке. Фильтр Блума для i-го коммита охватывает диапазон от BIDX[i-1] до BIDX[i] (плюс длина заголовка), где BIDX[-1] равен 0.

  • Блок BIDX игнорируется, если блок BDAT отсутствует.

Данные фильтра Блума (идентификатор: {B, D, A, T}) [Необязательно]

  • Он начинается с заголовка, состоящего из трех беззнаковых 32-битных целых чисел:

    • Версия используемого алгоритма хеширования. В настоящее время мы поддерживаем только значение 1, которое соответствует 32-битной версии хэша murmur3, реализованной точно так, как описано в https://en.wikipedia.org/wiki/MurmurHash#Algorithm , и методу двойного хеширования с использованием начальных значений 0x293ae76f и 0x7e646e2, как описано в https://doi.org/10.1007/978-3-540-30494-4_26 "Bloom Filters in Probabilistic Verification" .

    • Сколько раз путь хешируется и, следовательно, количество битовых позиций, которые в совокупности определяют, присутствует ли файл в коммите.

    • Минимальное количество бит b на запись в фильтре Блума. Если фильтр содержит записи n , то размер фильтра — это минимальное количество 64-битных слов, содержащих n*b бит.

  • Остальная часть фрагмента представляет собой объединение всех вычисленных фильтров Блума для коммитов в лексикографическом порядке.

  • Примечание. Коммиты без изменений или с более чем 512 изменениями имеют фильтры Блума длиной один, при этом либо все биты установлены в ноль, либо в единицу соответственно.

  • Чанк BDAT присутствует тогда и только тогда, когда присутствует BIDX.

Список базовых графиков (идентификатор: {B, A, S, E}) [Необязательно]

This list of H-byte hashes describe a set of B commit-graph files that
form a commit-graph chain. The graph position for the ith commit in this
fileЧанк поиска OID равен i плюс количество коммитов во всех
base graphs.  If B is non-zero, this chunk must exist.

TRAILER:

H-byte HASH-checksum of all of the above.

Historical notes:

Блоки данных генерации (GDA2) и переполнения данных генерации (GDO2) имеют номер 2 в своих идентификаторах блоков, поскольку предыдущая версия Git записывала в эти блоки возможно ошибочные данные с идентификаторами "GDAT" и "GDOV" . Изменяя идентификаторы, новые версии Git будут молча игнорировать эти старые фрагменты и записывать новую информацию, не доверяя неверным данным.
