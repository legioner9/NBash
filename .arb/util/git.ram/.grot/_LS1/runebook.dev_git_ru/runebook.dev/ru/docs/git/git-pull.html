git-pull

Name

git-pull — извлечь и интегрировать с другим репозиторием или локальной веткой

Synopsis

git pull [<options>] [<repository> [<refspec>…​]]

Description

Включает изменения из удаленного репозитория в текущую ветку. Если текущая ветка находится за удаленным, то по умолчанию текущая ветка будет перемотана вперед в соответствии с удаленным. Если текущая ветвь и удаленная разошлись, пользователю необходимо указать, как согласовать расходящиеся ветки с --rebase или --no-rebase (или соответствующим параметром конфигурации в pull.rebase ).

Точнее, git pull запускает git fetch с заданными параметрами, а затем, в зависимости от параметров конфигурации или флагов командной строки, вызывает git rebase или git merge для согласования расходящихся ветвей.

<repository> должно быть именем удаленного репозитория, переданным в git-fetch[1] . <refspec> может называть произвольную удаленную ссылку (например, имя тега) или даже коллекцию ссылок с соответствующими ветвями удаленного отслеживания (например, refs/heads/*:refs/remotes/origin/*), но обычно это имя ветки в удаленном репозитории.

Значения по умолчанию для <repository> и <branch> считываются из конфигурации "remote" и "merge" для текущей ветки, как установлено git-branch[1] --track .

Предположим, существует следующая история и текущая ветка — " master " :

          A---B---C master on origin
         /
    D---E---F---G master
        ^
        origin/master in your repository

Затем " git pull " извлечет и воспроизведет изменения из удаленной ветки master с тех пор, как она отличается от локальной master (т. е. E ) до ее текущего фиксации ( C ) поверх master , и запишет результат в новую фиксацию вместе с именами ветки master . два родительских коммита и сообщение журнала от пользователя с описанием изменений.

          A---B---C origin/master
         /         \
    D---E---F---G---H master

Подробную информацию, в том числе представление и обработку конфликтов, см. в git-merge[1] .

В Git 1.7.0 или более поздней версии для отмены конфликтующего слияния используйте git reset --merge . Предупреждение. В более старых версиях Git запуск git pull с незафиксированными изменениями не рекомендуется: хотя это и возможно, это оставляет вас в состоянии, из которого может быть трудно выйти в случае конфликта.

Если какое-либо из удаленных изменений перекрывается с локальными незафиксированными изменениями, слияние будет автоматически отменено, а рабочее дерево останется нетронутым. Как правило, лучше всего проверить работоспособность любых локальных изменений, прежде чем извлекать или прятать их с помощью git-stash[1] .

Options

-q
--quiet

Это передается как базовому git-fetch для подавления отчетов во время передачи, так и базовому git-merge для подавления вывода во время слияния.

-v
--verbose

Передайте --verbose в git-fetch и git-merge.

--[no-]recurse-submodules[=yes|on-demand|no]

Эта опция определяет, следует ли получать новые фиксации заполненных подмодулей, а также следует ли обновлять рабочие деревья активных подмодулей (см. git-fetch[1] , git-config[1] и gitmodules[5] ).

Если проверка выполняется через перебазирование, локальные фиксации подмодуля также перебазируются.

Если обновление выполняется посредством слияния, конфликты подмодулей разрешаются и извлекаются.

Варианты, связанные с объединением

--commit
--no-commit

Выполните слияние и зафиксируйте результат. Эту опцию можно использовать для переопределения --no-commit. Полезно только при слиянии.

С --no-commit выполните слияние и остановите его непосредственно перед созданием фиксации слияния, чтобы дать пользователю возможность проверить и дополнительно настроить результат слияния перед фиксацией.

Обратите внимание, что ускоренные обновления не создают фиксации слияния, и поэтому невозможно остановить эти слияния с помощью --no-commit. Таким образом, если вы хотите гарантировать, что ваша ветка не будет изменена или обновлена ​​командой слияния, используйте --no-ff с --no-commit.

--edit
-e
--no-edit

Перед выполнением успешного механического слияния вызовите редактор для дальнейшего редактирования автоматически созданного сообщения о слиянии, чтобы пользователь мог объяснить и обосновать слияние. Опцию --no-edit можно использовать для приема автоматически сгенерированного сообщения (обычно это не рекомендуется).

Старые сценарии могут зависеть от исторического поведения, запрещающего пользователю редактировать сообщение журнала слияния. Они увидят открытый редактор при запуске git merge . Чтобы упростить настройку таких сценариев на обновленное поведение, в их начале для переменной среды GIT_MERGE_AUTOEDIT можно установить значение no .

--cleanup=<mode>

Эта опция определяет, как будет очищено сообщение о слиянии перед фиксацией. Дополнительную информацию см. в git-commit[1] . Кроме того, если <mode> присвоено значение scissors , ножницы будут добавлены к MERGE_MSG перед передачей механизму фиксации в случае конфликта слияния.

--ff-only

Обновляйте новую историю только в том случае, если нет расходящейся локальной истории. Это значение по умолчанию, когда не предусмотрен метод согласования расходящихся историй (с помощью флагов --rebase=*).

--ff
--no-ff

При слиянии, а не при перебазировании, указывает, как обрабатывается слияние, если объединенная история уже является потомком текущей истории. Если запрашивается объединение, по умолчанию используется --ff , за исключением случаев объединения аннотированного (и, возможно, подписанного) тега, который не хранится в своем естественном месте в иерархии refs/tags/ , и в этом случае предполагается --no-ff .

С помощью --ff по возможности разрешайте слияние как ускоренную перемотку вперед (обновляйте только указатель ветки, чтобы он соответствовал объединенной ветке; не создавайте фиксацию слияния). Если это невозможно (когда объединенная история не является потомком текущей истории), создайте фиксацию слияния.

С помощью --no-ff создайте фиксацию слияния во всех случаях, даже если вместо этого слияние можно было разрешить как ускоренную перемотку вперед.

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

Подпишите GPG полученный коммит слияния. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

--log[=<n>]
--no-log

Помимо названий ветвей, заполните сообщение журнала однострочными описаниями не более чем из <n> фактических коммитов, которые объединяются. См. также git-fmt-merge-msg[1] . Полезно только при слиянии.

С помощью --no-log не перечисляйте однострочные описания фактических объединяемых коммитов.

--signoff
--no-signoff

Добавьте трейлер Signed-off-by от коммиттера в конце сообщения журнала фиксации. Значение подписания зависит от проекта, за который вы беретесь. Например, он может удостоверить, что коммиттер имеет права представить работу по лицензии проекта или согласен на какое-либо представление участника, такое как Сертификат происхождения разработчика. (См. https://developercertificate.org , который используется в ядре Linux и проектах Git .) Обратитесь к документации или руководству проекта, в который вы вносите свой вклад, чтобы понять, как подписи используются в этом проекте.

Опцию --no-signoff можно использовать для отмены более ранней опции --signoff в командной строке.

--stat
-n
--no-stat

Покажите diffstat в конце слияния. diffstat также контролируется параметром конфигурации merge.stat.

С -n или --no-stat не показывать статистику различий в конце слияния.

--squash
--no-squash

Создайте рабочее дерево и состояние индекса, как если бы произошло реальное слияние (за исключением информации о слиянии), но на самом деле не делайте фиксацию, не перемещайте HEAD и не записывайте $GIT_DIR/MERGE_HEAD (чтобы заставить следующую команду git commit создать фиксацию слияния). Это позволяет вам создать один коммит поверх текущей ветки, эффект которого аналогичен слиянию другой ветки (или большего количества в случае осьминога).

С помощью --no-squash выполните слияние и зафиксируйте результат. Эту опцию можно использовать для переопределения --squash.

С --squash --commit не разрешен и завершится ошибкой.

Полезно только при слиянии.

--[no-]verify

По умолчанию запускаются перехватчики pre-merge и commit-msg. Когда задан --no-verify , они обходятся. См. также githooks[5] . Полезно только при слиянии.

-s <strategy>
--strategy=<strategy>

Используйте данную стратегию слияния; могут быть предоставлены более одного раза, чтобы указать их в том порядке, в котором они должны быть опробованы. Если опция -s отсутствует, вместо нее используется встроенный список стратегий ( ort при объединении одной головы, octopus в противном случае).

-X <option>
--strategy-option=<option>

Передайте конкретную опцию стратегии слияния в стратегию слияния.

--verify-signatures
--no-verify-signatures

Убедитесь, что фиксация объединяемой боковой ветки подписана действительным ключом, т. е. ключом с действительным идентификатором пользователя: в модели доверия по умолчанию это означает, что ключ подписи подписан доверенным ключом. Если кончик фиксации боковой ветки не подписан действительным ключом, слияние прерывается.

Полезно только при слиянии.

--summary
--no-summary

Синонимы --stat и --no-stat; они устарели и будут удалены в будущем.

--autostash
--no-autostash

Автоматически создайте временную запись в тайнике перед началом операции, запишите ее в ref MERGE_AUTOSTASH и примените после завершения операции. Это означает, что вы можете запустить операцию на «грязном» рабочем дереве. Однако используйте с осторожностью: финальное приложение stash после успешного слияния может привести к нетривиальным конфликтам.

--allow-unrelated-histories

По умолчанию команда git merge отказывается объединять истории, не имеющие общего предка. Эту опцию можно использовать для обхода этой безопасности при объединении историй двух проектов, которые начали свою жизнь независимо. Поскольку это очень редкий случай, переменная конфигурации, позволяющая включить это по умолчанию, не существует и не будет добавлена.

Полезно только при слиянии.

-r
--rebase[=false|true|merges|interactive]

Если это правда, после извлечения перебазируйте текущую ветвь поверх восходящей ветки. Если существует ветвь удаленного отслеживания, соответствующая восходящей ветке, и восходящая ветвь была перебазирована с момента последней выборки, при перебазировании используется эта информация, чтобы избежать повторного базирования нелокальных изменений.

Если установлено значение merges , выполните перебазирование с использованием git rebase --rebase-merges , чтобы локальные фиксации слияния были включены в перебазирование (подробности см. в git-rebase[1] ).

Если установлено значение false, объедините вышестоящую ветвь с текущей ветвью.

При interactive включите интерактивный режим перебазирования.

См. pull.rebase , branch.<name>.rebase и branch.autoSetupRebase в git-config[1] , если вы хотите, чтобы git pull всегда использовал --rebase вместо слияния.

Note
 Это потенциально опасный режим работы. Он переписывает историю, что не сулит ничего хорошего, если вы уже опубликовали эту историю. Не используйте эту опцию, если вы внимательно не прочитали git-rebase[1] .
--no-rebase

Это сокращение для --rebase=false.

Параметры, связанные с получением

--all

Соберите все пульты.

-a
--append

Добавьте имена ссылок и имена объектов полученных ссылок к существующему содержимому .git/FETCH_HEAD . Без этой опции старые данные в .git/FETCH_HEAD будут перезаписаны.

--atomic

Используйте атомарную транзакцию для обновления локальных ссылок. Либо все ссылки обновляются, либо в случае ошибки ни одна ссылка не обновляется.

--depth=<depth>

Ограничьте выборку указанным количеством коммитов из кончика истории каждой удаленной ветки. При извлечении в репозиторий shallow , созданный git clone с опцией --depth=<depth> (см. git-clone[1] ), углубите или сократите историю до указанного количества коммитов. Теги для углубленных коммитов не извлекаются.

--deepen=<depth>

Аналогично --Deep, за исключением того, что он определяет количество коммитов от текущей мелкой границы, а не от кончика истории каждой удаленной ветки.

--shallow-since=<date>

Углубите или сократите историю мелкого репозитория, чтобы включить все доступные коммиты после <дата>.

--shallow-exclude=<revision>

Углубите или сократите историю мелкого репозитория, чтобы исключить коммиты, доступные из указанной удаленной ветки или тега. Эту опцию можно указать несколько раз.

--unshallow

Если исходный репозиторий завершен, преобразуйте поверхностный репозиторий в полный, устранив все ограничения, налагаемые мелкими репозиториями.

Если исходный репозиторий неглубокий, извлеките как можно больше данных, чтобы текущий репозиторий имел ту же историю, что и исходный репозиторий.

--update-shallow

По умолчанию при загрузке из мелкого репозитория git fetch отклоняет ссылки, требующие обновления .git/shallow. Эта опция обновляет .git/shallow и принимает такие ссылки.

--negotiation-tip=<commit|glob>

По умолчанию Git будет сообщать серверу о коммитах, доступных со всех локальных ссылок, чтобы найти общие коммиты и попытаться уменьшить размер получаемого пакетного файла. Если указано, Git будет сообщать только о коммитах, доступных из данных советов. Это полезно для ускорения выборки, когда пользователь знает, какая локальная ссылка, вероятно, будет иметь общие фиксации с извлекаемой исходной ссылкой.

Эту опцию можно указать более одного раза; если да, то Git сообщит о коммитах, доступных из любого из заданных коммитов.

Аргументом этой опции может быть обобщение имен ссылок, ссылка или (возможно, сокращенный) SHA-1 коммита. Указание glob эквивалентно указанию этой опции несколько раз, по одному для каждого соответствующего имени ссылки.

См. также переменные конфигурации fetch.negotiationAlgorithm и push.negotiate , описанные в git-config[1] , и опцию --negotiate-only ниже.

--negotiate-only

Ничего не извлекайте с сервера, а вместо этого печатайте предки предоставленных аргументов --negotiation-tip=* , которые у нас есть общие с сервером.

Это несовместимо с --recurse-submodules=[yes|on-demand] . Внутри это используется для реализации опции push.negotiate , см. git-config[1] .

--dry-run

Покажите, что будет сделано, не внося никаких изменений.

--porcelain

Выведите выходные данные на стандартный вывод в удобном для анализа формате для сценариев. Подробности см. в разделе ВЫХОД в git-fetch[1] .

Это несовместимо с --recurse-submodules=[yes|on-demand] и имеет приоритет над параметром конфигурации fetch.output .

-f
--force

Когда git fetch используется со спецификацией <src>:<dst> , он может отказаться обновлять локальную ветку, как описано в части <refspec> документации git-fetch[1] . Эта опция отменяет эту проверку.

-k
--keep

Сохраните скачанный пакет.

--prefetch

Измените настроенную спецификацию ссылок, чтобы поместить все ссылки в пространство имен refs/prefetch/ . См. задачу prefetch в git-maintenance[1] .

-p
--prune

Перед извлечением удалите все ссылки на удаленное отслеживание, которые больше не существуют на удаленном компьютере. Теги не подлежат сокращению, если они извлекаются только из-за автоматического следования тегам по умолчанию или из-за опции --tags. Однако если теги извлекаются из-за явной спецификации ссылки (либо в командной строке, либо в удаленной конфигурации, например, если удаленный сервер был клонирован с помощью опции --mirror), то они также подлежат сокращению. Предоставление --prune-tags — это сокращение для предоставления тега refspec.

--no-tags

По умолчанию теги, указывающие на объекты, загруженные из удаленного репозитория, извлекаются и сохраняются локально. Эта опция отключает автоматическое отслеживание тегов. Поведение по умолчанию для удаленного устройства можно указать с помощью параметра Remote.<name>.tagOpt. См. git-config[1] .

--refmap=<refspec>

При получении ссылок, перечисленных в командной строке, используйте указанную спецификацию ссылки (может быть задана более одного раза) для сопоставления ссылок с ветвями удаленного отслеживания вместо значений переменных конфигурации remote.*.fetch для удаленного репозитория. Если указать пустой <refspec> в параметре --refmap , Git будет игнорировать настроенные спецификации ссылок и полностью полагаться на спецификации ссылок, предоставленные в качестве аргументов командной строки. Подробности см. в разделе "Configured Remote-tracking Branches" .

-t
--tags

Извлеките все теги с удаленного устройства (т. е. извлеките удаленные теги refs/tags/* в локальные теги с тем же именем), а также все остальное, что в противном случае было бы получено. Использование только этой опции не приводит к сокращению тегов, даже если используется --prune (хотя теги в любом случае могут быть сокращены, если они также являются местом назначения явной спецификации ссылки; см. --prune ).

-j
--jobs=<n>

Количество параллельных дочерних элементов, которые будут использоваться для всех форм выборки.

Если была указана опция --multiple , разные пульты будут подключаться параллельно. Если выбрано несколько подмодулей, они будут выбраны параллельно. Для независимого управления ими используйте настройки конфига fetch.parallel и submodule.fetchJobs (см. git-config[1] ).

Обычно параллельная рекурсивная и многоудаленная выборка выполняется быстрее. По умолчанию выборка выполняется последовательно, а не параллельно.

--set-upstream

Если удаленное устройство получено успешно, добавьте ссылку на восходящий поток (отслеживание), используемую git-pull[1] без аргументов и другими командами. Для получения дополнительной информации см. branch.<name>.merge и branch.<name>.remote в git-config[1] .

--upload-pack <upload-pack>

Если этот параметр задан и репозиторий, из которого осуществляется выборка, обрабатывается git fetch-pack , --exec=<upload-pack> передается команде, чтобы указать нестандартный путь для команды, запускаемой на другом конце.

--progress

Статус выполнения сообщается в стандартном потоке ошибок по умолчанию, когда он подключен к терминалу, если не указан параметр -q. Этот флаг обеспечивает статус хода выполнения, даже если стандартный поток ошибок не направляется на терминал.

-o <option>
--server-option=<option>

Передайте данную строку на сервер при обмене данными по протоколу версии 2. Данная строка не должна содержать символ NUL или LF. Обработка сервером параметров сервера, включая неизвестные, зависит от сервера. Если задано несколько --server-option=<option> , все они отправляются другой стороне в порядке, указанном в командной строке.

--show-forced-updates

По умолчанию git проверяет, обновляется ли ветка принудительно во время выборки. Это можно отключить с помощью fetch.showForcedUpdates, но опция --show-forced-updates гарантирует выполнение этой проверки. См. git-config[1] .

--no-show-forced-updates

По умолчанию git проверяет, обновляется ли ветка принудительно во время выборки. Передайте --no-show-forced-updates или установите для fetch.showForcedUpdates значение false , чтобы пропустить эту проверку из соображений производительности. Если параметр --ff-only используется во время git-pull , он по-прежнему будет проверять наличие принудительных обновлений перед попыткой быстрого обновления. См. git-config[1] .

-4
--ipv4

Используйте только адреса IPv4, игнорируя адреса IPv6.

-6
--ipv6

Используйте только адреса IPv6, игнорируя адреса IPv4.

<repository>

Репозиторий "remote" , который является источником операции выборки или извлечения. Этот параметр может быть либо URL-адресом (см. раздел GIT URLS ниже), либо именем удаленного устройства (см. раздел REMOTES ниже).

<refspec>

Указывает, какие ссылки следует получить, а какие локальные ссылки обновить. Если в командной строке не отображаются <refspec>, ссылки для выборки вместо этого считываются из переменных remote.<repository>.fetch (см. раздел "CONFIGURED REMOTE-TRACKING BRANCHES" в git-fetch[1] ).

Формат параметра <refspec> является необязательным, плюс + , за которым следует исходный <src>, затем двоеточие : , за которым следует ссылка назначения <dst>. Двоеточие можно опустить, если <dst> пуст. <src> обычно представляет собой ссылку, но также может быть полностью написанным шестнадцатеричным именем объекта.

<refspec> может содержать * в своем <src>, чтобы указать простое совпадение с шаблоном. Такая спецификация ссылки действует как glob, который соответствует любой ссылке с тем же префиксом. Шаблон <refspec> должен иметь * как в <src>, так и в <dst>. Он сопоставит ссылки с пунктом назначения, заменив * содержимым, совпадающим с источником.

Если перед спецификацией ссылки стоит ^ , она будет интерпретироваться как отрицательная спецификация. Вместо того, чтобы указывать, какие ссылки следует получить или какие локальные ссылки обновлять, такая спецификация вместо этого будет указывать ссылки, которые следует исключить. Ссылка будет считаться соответствующей, если она соответствует хотя бы одной положительной спецификации и не соответствует ни одной отрицательной спецификации. Отрицательные спецификации ссылок могут быть полезны для ограничения области применения спецификации шаблона, чтобы она не включала определенные ссылки. Отрицательные ссылки сами по себе могут быть ссылками на шаблоны. Однако они могут содержать только <src> и не указывать <dst>. Полностью написанные шестнадцатеричные имена объектов также не поддерживаются.

tag <tag> означает то же, что и refs/tags/<tag>:refs/tags/<tag> ; он запрашивает получение всего до данного тега.

Выбирается удаленная ссылка, соответствующая <src>, и если <dst> не является пустой строкой, предпринимается попытка обновить соответствующую ей локальную ссылку.

Разрешено ли это обновление без --force , зависит от пространства имен ref, в которое оно извлекается, типа извлекаемого объекта и того, считается ли обновление перемоткой вперед. Как правило, к выборке применяются те же правила, что и при отправке, см. раздел <refspec>... статьи git-push[1] , чтобы узнать, что это такое. Исключения из этих правил, характерные для git fetch , указаны ниже.

До Git версии 2.20, в отличие от git-push[1] , любые обновления refs/tags/* будут приниматься без + в refspec (или --force ). При загрузке мы беспорядочно считали все обновления тегов с удаленного устройства принудительной выборкой. Начиная с Git версии 2.20, получение обновления refs/tags/* работает так же, как и при принудительной отправке. Т.е. любые обновления будут отклонены без + в refspec (или --force ).

В отличие от отправки с помощью git-push[1] , любые обновления за пределами refs/{tags,heads}/* будут приняты без + в refspec (или --force ), независимо от того, заменяется ли это, например, объект дерева на большой двоичный объект или фиксация на другую фиксацию, которая не имеет предыдущей фиксации как предок и т. д.

В отличие от отправки с помощью git-push[1] , не существует конфигурации, которая бы изменила эти правила, и нет ничего похожего на перехватчик pre-fetch , аналогичный перехватчику pre-receive .

Как и в случае с git-push[1] , все описанные выше правила о том, что не разрешено в качестве обновления, можно переопределить, добавив необязательный ведущий + в refspec (или используя параметр командной строки --force ). Единственным исключением из этого правила является то, что никакие принудительные действия не заставят пространство имен refs/heads/* принять нефиксированный объект.

Note
 Когда известно, что удаленная ветвь, которую вы хотите получить, регулярно перематывается и перебазируется, ожидается, что ее новая подсказка не будет потомком предыдущей подсказки (которая хранится в вашей ветке удаленного отслеживания при последнем извлечении). Вы можете использовать знак + , чтобы указать, что для таких ветвей потребуются обновления без перемотки вперед. Невозможно определить или объявить, что ветка с таким поведением будет доступна в репозитории; извлекающий пользователь просто должен знать, что это ожидаемый шаблон использования ветки.
Note
 Существует разница между перечислением нескольких <refspec> непосредственно в командной строке запроса git и наличием нескольких записей remote.<repository>.fetch в вашей конфигурации для <repository> и запуском команды извлечения git без каких-либо явных параметров <refspec>. <refspec>, явно указанный в командной строке, всегда добавляется в текущую ветку после получения. Другими словами, если вы укажете более одной удаленной ссылки, git pull создаст объединение Octopus. С другой стороны, если вы не укажете какой-либо явный параметр <refspec> в командной строке, git pull выберет все <refspec>, которые он найдет в конфигурации remote.<repository>.fetch , и объединит только первый найденный <refspec> в текущую ветку. . Это связано с тем, что создание Octopus из удаленных ссылок выполняется редко, в то время как отслеживание нескольких удаленных голов за один раз путем извлечения более одной часто бывает полезно.

Git urls

Обычно URL-адреса содержат информацию о транспортном протоколе, адресе удаленного сервера и пути к репозиторию. В зависимости от транспортного протокола некоторая часть этой информации может отсутствовать.

Git поддерживает протоколы ssh, git, http и https (кроме того, для выборки можно использовать ftp и ftps, но это неэффективно и не рекомендуется; не используйте их).

Собственный транспорт (например, URL-адрес git://) не выполняет аутентификацию, и его следует использовать с осторожностью в незащищенных сетях.

С ними можно использовать следующие синтаксисы:

  • ssh://[user@]host.xz[:port]/path/to/repo.git/

  • git://host.xz[:port]/path/to/repo.git/

  • http[s]://host.xz[:port]/path/to/repo.git/

  • ftp[s]://host.xz[:port]/path/to/repo.git/

Альтернативный синтаксис, подобный scp, также можно использовать с протоколом ssh:

  • [user@]host.xz:path/to/repo.git/

Этот синтаксис распознается только в том случае, если перед первым двоеточием нет косой черты. Это помогает отличить локальный путь, содержащий двоеточие. Например, локальный путь foo:bar можно указать как абсолютный путь или ./foo:bar , чтобы его не интерпретировали как URL-адрес ssh.

Протоколы ssh и git дополнительно поддерживают расширение ~username:

  • ssh://[user@]host.xz[:port]/~[user]/path/to/repo.git/

  • git://host.xz[:port]/~[user]/path/to/repo.git/

  • [user@]host.xz:/~[user]/path/to/repo.git/

Для локальных репозиториев, которые также поддерживаются Git , могут использоваться следующие синтаксисы:

  • /path/to/repo.git/

  • file:///path/to/repo.git/

Эти два синтаксиса в основном эквивалентны, за исключением случаев клонирования, когда первый подразумевает параметр --local. Подробности см. в git-clone[1] .

git clone , git fetch и git pull , но не git push , также принимают подходящий файл пакета. См. git-bundle[1] .

Когда Git не знает, как обрабатывать определенный транспортный протокол, он пытается использовать удаленный помощник remote-<transport> , если таковой существует. Чтобы явно запросить удаленного помощника, можно использовать следующий синтаксис:

  • <transport>::<address>

где <адрес> может быть путем, сервером и путем или произвольной строкой типа URL-адреса, распознаваемой конкретным вызываемым удаленным помощником. Подробности см. в gitremote-helpers[7] .

Если существует большое количество удаленных репозиториев с одинаковыми именами и вы хотите использовать для них другой формат (например, чтобы используемые вами URL-адреса были перезаписаны в работающие URL-адреса), вы можете создать раздел конфигурации вида:

        [url "<actual url base>"]
                insteadOf = <other url base>

Например, с этим:

        [url "git://git.host.xz/"]
                insteadOf = host.xz:/path/to/
                insteadOf = work:

URL-адрес типа "work:repo.git" или "host.xz:/path/to/repo.git" будет перезаписан в любом контексте, который принимает URL-адрес "git://git.host.xz/repo.git" .

Если вы хотите переписать URL-адреса только для push-уведомлений, вы можете создать раздел конфигурации формы:

        [url "<actual url base>"]
                pushInsteadOf = <other url base>

Например, с этим:

        [url "ssh://example.org/"]
                pushInsteadOf = git://example.org/

URL-адрес типа "git://example.org/path/to/repo.git" будет перезаписан в "ssh://example.org/path/to/repo.git" для push-уведомлений, но при запросе по-прежнему будет использоваться исходный URL-адрес.

Remotes

Вместо URL-адреса в качестве аргумента <repository> можно использовать имя одного из следующих элементов:

  • пульт в файле конфигурации Git : $GIT_DIR/config ,

  • файл в каталоге $GIT_DIR/remotes или

  • файл в каталоге $GIT_DIR/branches .

Все они также позволяют опустить спецификацию ссылки в командной строке, поскольку каждая из них содержит спецификацию ссылки, которую git будет использовать по умолчанию.

Именованный удаленный компьютер в файле конфигурации

Вы можете указать имя пульта, который вы ранее настроили с помощью git-remote[1] , git-config[1] или даже вручную отредактировав файл $GIT_DIR/config . URL-адрес этого пульта будет использоваться для доступа к хранилищу. Спецификация этого пульта будет использоваться по умолчанию, если вы не укажете спецификацию в командной строке. Запись в файле конфигурации будет выглядеть так:

        [remote "<name>"]
                url = <URL>
                pushurl = <pushurl>
                push = <refspec>
                fetch = <refspec>

<pushurl> используется только для push-уведомлений. Это необязательно и по умолчанию — <URL> . Отправка на удаленный компьютер влияет на все определенные pushurl-адреса или на все определенные URL-адреса, если pushurl не определен. Однако Fetch будет выполнять выборку только с первого определенного URL-адреса, если определено несколько URL-адресов.

Именованный файл в $GIT_DIR/remotes

Вы можете указать имя файла в формате $GIT_DIR/remotes . URL-адрес в этом файле будет использоваться для доступа к репозиторию. Спецификация ссылки в этом файле будет использоваться по умолчанию, если вы не укажете спецификацию ссылки в командной строке. Этот файл должен иметь следующий формат:

        URL: one of the above URL formats
        Push: <refspec>
        Pull: <refspec>

Линии Push: используются git push , а линии Pull: используются git pull и git fetch . Для дополнительных сопоставлений ветвей можно указать несколько линий Push: и Pull: .

Именованный файл в $GIT_DIR/branches

Вы можете указать имя файла в формате $GIT_DIR/branches . URL-адрес в этом файле будет использоваться для доступа к репозиторию. Этот файл должен иметь следующий формат:

        <URL>#<head>

Требуется <URL> ; #<head> не является обязательным.

В зависимости от операции git будет использовать одну из следующих спецификаций, если вы не указали ее в командной строке. <branch> — это имя этого файла в $GIT_DIR/branches , а <head> по умолчанию — master .

Для выборки git используются:

        refs/heads/<head>:refs/heads/<branch>

git push использует:

        HEAD:refs/heads/<head>

Merge strategies

Механизм слияния (команды git merge и git pull ) позволяет выбирать серверную часть merge strategies с опцией -s . Некоторые стратегии также могут иметь свои собственные параметры, которые можно передать, передав аргументы -X<option> в git merge и/или git pull .

ort

Это стратегия слияния по умолчанию при извлечении или объединении одной ветки. Эта стратегия может разрешить только две головы, используя алгоритм трехстороннего слияния. Если существует более одного общего предка, который можно использовать для трехстороннего слияния, создается объединенное дерево общих предков и используется в качестве эталонного дерева для трехстороннего слияния. Сообщается, что это приводит к меньшему количеству конфликтов слияния, не вызывая неправильных слияний, согласно тестам, выполненным на реальных коммитах слияния, взятых из истории разработки ядра Linux 2.6. Кроме того, эта стратегия может обнаруживать и обрабатывать слияния, связанные с переименованиями. Он не использует обнаруженные копии. Название этого алгоритма является аббревиатурой ( "Ostensibly Recursive’s Twin" ) и произошло от того, что он был написан в качестве замены предыдущего алгоритма по умолчанию recursive .

Стратегия ort может принимать следующие варианты:

ours

Эта опция заставляет конфликтующие фрагменты автоматически разрешаться, отдавая предпочтение версии our . Изменения из другого дерева, которые не конфликтуют с нашей стороной, отражаются в результате слияния. Для бинарного файла все содержимое берется с нашей стороны.

Это не следует путать со стратегией слияния ours , которая вообще не рассматривает содержимое другого дерева. Он отбрасывает все, что сделало другое дерево, объявляя, что история our содержит все, что в ней произошло.

theirs

Это противоположность ours ; обратите внимание, что, в отличие от ours , не существует стратегии слияния theirs , с которой можно было бы спутать этот вариант слияния.

ignore-space-change
ignore-all-space
ignore-space-at-eol
ignore-cr-at-eol

Считает строки с указанным типом изменения пробелов неизмененными для трехстороннего слияния. Изменения пробелов, смешанные с другими изменениями в строке, не игнорируются. См. также git-diff[1] , -b , -w , --ignore-space-at-eol и --ignore-cr-at-eol .

  • Если версия their вводит в строку только изменения пробелов, используется версия our ;

  • Если версия our содержит изменения в пробелах, а версия their включает существенные изменения, используется версия their ;

  • В противном случае слияние происходит обычным способом.

renormalize

При этом выполняется виртуальное извлечение и возврат всех трех этапов файла при разрешении трехстороннего слияния. Эта опция предназначена для использования при объединении ветвей с различными чистыми фильтрами или правилами нормализации конца строки. Подробности см. в "Merging branches with differing checkin/checkout attributes" и gitattributes[5] .

no-renormalize

Отключает опцию renormalize . Это переопределяет переменную конфигурации merge.renormalize .

find-renames[=<n>]

Включите обнаружение переименования, при необходимости установив порог сходства. Это значение по умолчанию. Это переопределяет переменную конфигурации merge.renames . См. также git-diff[1] --find-renames .

rename-threshold=<n>

Устаревший синоним find-renames=<n> .

subtree[=<path>]

Этот вариант представляет собой более продвинутую форму стратегии subtree , где стратегия предполагает, как два дерева должны быть сдвинуты, чтобы они совпадали друг с другом при слиянии. Вместо этого к указанному пути добавляется префикс (или удаляется с самого начала), чтобы форма двух деревьев совпадала.

recursive

Это может разрешить только две головы, используя алгоритм трехстороннего слияния. Если существует более одного общего предка, который можно использовать для трехстороннего слияния, создается объединенное дерево общих предков и используется в качестве эталонного дерева для трехстороннего слияния. Сообщается, что это приводит к меньшему количеству конфликтов слияния, не вызывая неправильных слияний, согласно тестам, выполненным на реальных коммитах слияния, взятых из истории разработки ядра Linux 2.6. Кроме того, это может обнаруживать и обрабатывать слияния, связанные с переименованиями. Он не использует обнаруженные копии. Это была стратегия по умолчанию для разрешения двух головок от Git v0.99.9k до v2.33.0.

Стратегия recursive использует те же параметры, что и ort . Однако есть три дополнительных параметра, которые ort игнорирует (не описаны выше), которые потенциально полезны для стратегии recursive :

patience

Устаревший синоним diff-algorithm=patience .

diff-algorithm=[patience|minimal|histogram|myers]

Используйте другой алгоритм сравнения при слиянии, что поможет избежать ошибок слияния, возникающих из-за неважных совпадающих строк (например, фигурных скобок из разных функций). См. также git-diff[1] --diff-algorithm . Обратите внимание, что ort специально использует diff-algorithm=histogram , а recursive по умолчанию использует настройку конфигурации diff.algorithm .

no-renames

Отключите обнаружение переименования. Это переопределяет переменную конфигурации merge.renames . См. также git-diff[1] --no-renames .

resolve

Это может разрешить только две головы (т. е. текущую ветвь и другую ветвь, из которой вы извлекли) с использованием алгоритма трехстороннего слияния. Он пытается тщательно обнаружить неоднозначности перекрестного слияния. Он не обрабатывает переименования.

octopus

Это разрешает случаи с более чем двумя головками, но отказывается выполнять сложное слияние, требующее разрешения вручную. В первую очередь он предназначен для объединения заголовков тематических ветвей. Это стратегия слияния по умолчанию при извлечении или объединении более чем одной ветки.

ours

При этом разрешается любое количество голов, но результирующее дерево слияния всегда является деревом текущей головы ветки, эффективно игнорируя все изменения из всех других ветвей. Он предназначен для замены старой истории развития побочных ветвей. Обратите внимание, что это отличается от опции -Xours стратегии слияния recursive .

subtree

Это модифицированная стратегия ort . При слиянии деревьев A и B, если B соответствует поддереву A, B сначала настраивается так, чтобы оно соответствовало древовидной структуре A, вместо чтения деревьев на одном уровне. Эта корректировка также вносится в дерево общего предка.

В стратегиях, использующих трехстороннее слияние (включая стандартную ort ), если изменение внесено в обе ветви, но позже отменено в одной из ветвей, это изменение будет присутствовать в объединенном результате; некоторых людей такое поведение сбивает с толку. Это происходит потому, что при выполнении слияния учитываются только головы и база слияния, а не отдельные фиксации. Поэтому алгоритм слияния считает отмененное изменение вообще отсутствием изменений и вместо этого заменяет измененную версию.

Default behaviour

Часто люди используют git pull без указания каких-либо параметров. Традиционно это эквивалентно слову git pull origin . Однако если конфигурация branch.<name>.remote присутствует в ветви <name> , это значение используется вместо origin .

Чтобы определить, какой URL-адрес использовать для выборки, проверяется значение конфигурации remote.<origin>.url , и если такой переменной нет, используется значение в строке URL: в $GIT_DIR/remotes/<origin> .

Чтобы определить, какие удаленные ветки следует извлечь (и, при необходимости, сохранить в ветках удаленного отслеживания), когда команда запускается без каких-либо параметров refspec в командной строке, проверяются значения переменной конфигурации remote.<origin>.fetch , и, если они отсутствуют, , используется $GIT_DIR/remotes/<origin> и используются его линии Pull: . В дополнение к форматам спецификаций ссылок, описанным в разделе ОПЦИИ, вы можете использовать подстановочную спецификацию ссылок, которая выглядит следующим образом:

refs/heads/*:refs/remotes/origin/*

Спецификация подстановочной ссылки должна иметь непустую RHS (т. е. должна хранить то, что было получено в ветвях удаленного отслеживания), а ее LHS и RHS должны заканчиваться на /* . Вышеупомянутое указывает, что все удаленные ветки отслеживаются с использованием ветвей удаленного отслеживания в иерархии refs/remotes/origin/ под тем же именем.

Правило определения того, какую удаленную ветку объединять после выборки, немного усложнено, чтобы не нарушать обратную совместимость.

Если в командной строке git pull были указаны явные ссылки, все они объединяются.

Если в командной строке не указана refspec, git pull использует refspec из конфигурации или $GIT_DIR/remotes/<origin> . В таких случаях применяются следующие правила:

  1. Если конфигурация branch.<name>.merge для текущей ветви <name> существует, это имя объединяемой ветви на удаленном сайте.

  2. Если спецификация является подстановочной, ничего не объединяется.

  3. В противном случае удаленная ветка первой спецификации будет объединена.

Examples

  • Обновите ветки удаленного отслеживания для репозитория, из которого вы клонировали, а затем объедините одну из них с текущей веткой:

    $ git pull
$ git pull origin

    Обычно объединяемая ветка является HEAD удаленного репозитория, но выбор определяется параметрами ветка.<имя>.remote и ветка.<имя>.merge; подробности см. в git-config[1] .

  • Объединить с текущей веткой удаленную ветку next :

    $ git pull origin next

    При этом копия next временно остается в FETCH_HEAD и обновляется ветвь удаленного отслеживания origin/next . То же самое можно сделать, вызвав fetch и merge:

    $ git fetch origin
$ git merge origin/next

Если вы попытались получить данные, что привело к сложным конфликтам и вам захотелось начать заново, вы можете восстановиться с помощью git reset .

Security

Протоколы выборки и отправки не предназначены для предотвращения кражи одной стороной данных из другого репозитория, которые не предназначены для совместного использования. Если у вас есть данные private , которые необходимо защитить от злонамеренного узла, лучший вариант — сохранить их в другом репозитории. Это касается как клиентов, так и серверов. В частности, пространства имен на сервере неэффективны для управления доступом для чтения; вам следует предоставлять доступ на чтение к пространству имен только тем клиентам, которым вы доверяете доступ на чтение ко всему репозиторию.

Известные векторы атак следующие:

  1. Жертва отправляет строки "have" , объявляющие идентификаторы имеющихся у нее объектов, которые явно не предназначены для совместного использования, но могут использоваться для оптимизации передачи, если они также есть у узла. Злоумышленник выбирает идентификатор объекта X для кражи и отправляет ссылку на X, но не обязан отправлять содержимое X, поскольку оно уже есть у жертвы. Теперь жертва считает, что у злоумышленника есть X, и позже отправляет содержимое X обратно злоумышленнику. (Эту атаку наиболее просто выполнить клиент на сервере, создав ссылку на X в пространстве имен, к которому клиент имеет доступ, а затем извлекая ее. Наиболее вероятный способ, которым сервер может выполнить эту атаку на клиенте, — это "merge" . X в ветку public и надеяться, что пользователь проделает дополнительную работу над этой веткой и отправит ее обратно на сервер, не заметив слияния.)

  2. Как и в № 1, злоумышленник выбирает объект с идентификатором X для кражи. Жертва отправляет объект Y, который уже есть у злоумышленника, и злоумышленник ложно утверждает, что у него есть X, а не Y, поэтому жертва отправляет Y как дельту против X. Дельта показывает злоумышленнику области X, похожие на Y.

Bugs

Использование --recurse-submodules позволяет прямо сейчас получать новые коммиты только в уже извлеченных подмодулях. Когда, например, восходящий поток добавил новый подмодуль в только что полученные коммиты суперпроекта, сам подмодуль не может быть получен, что делает невозможным извлечение этого подмодуля позже без необходимости повторной выборки. Ожидается, что это будет исправлено в будущей версии Git .
