gitdiffcore

Name

gitdiffcore — настройка вывода различий

Synopsis

git diff *

Description

Командам diff git diff-index , git diff-files и git diff-tree можно указать манипулировать найденными различиями нетрадиционными способами перед отображением вывода diff . Манипуляция имеет общее название "diffcore transformation" . В этой короткой заметке описывается, что это такое и как их использовать для получения выходных данных diff , которые легче понять, чем обычные.

Цепочка операций

Семейство git diff-* работает, сначала сравнивая два набора файлов:

  • git diff-index сравнивает содержимое объекта "tree" и рабочего каталога (когда флаг --cached не используется) или объекта "tree" и индексного файла (когда используется флаг --cached );

  • git diff-files сравнивает содержимое индексного файла и рабочего каталога;

  • git diff-tree сравнивает содержимое двух объектов "tree" ;

Во всех этих случаях сами команды сначала опционально ограничивают два набора файлов любыми путями, указанными в их командных строках, и сравнивают соответствующие пути в двух результирующих наборах файлов.

Спецификации пути используются для ограничения мира, в котором работает diff. Они удаляют пары файлов за пределами указанных наборов имен путей. Например, если входной набор пар файлов включал:

:100644 100644 bcd1234... 0123456... M junkfile

но вызов команды был git diff-files myfile , то запись ненужного файла будет удалена из списка, поскольку рассматривается только "myfile" .

Результат сравнения передается из этих команд в то, что внутри называется "diffcore" , в формате, аналогичном тому, который выводится, когда опция -p не используется. Например

in-place edit  :100644 100644 bcd1234... 0123456... M file0
create         :000000 100644 0000000... 1234567... A file4
delete         :100644 000000 1234567... 0000000... D file5
unmerged       :000000 000000 0000000... 0000000... U file6

Механизму diffcore подается список таких результатов сравнения (каждый из которых называется "filepair" , хотя на данный момент каждый из них говорит об одном файле) и преобразует такой список в другой список. На данный момент существует 5 таких преобразований:

  • diffcore-break

  • diffcore-rename

  • diffcore-merge-broken

  • diffcore-pickaxe

  • diffcore-order

  • diffcore-rotate

Они применяются последовательно. Набор пар файлов git diff-* с командами find используется в качестве входных данных для команды diffcore-break, а выходные данные команды diffcore-break используются в качестве входных данных для следующего преобразования. Конечный результат затем передается в процедуру вывода и генерирует либо формат diff-raw (см. разделы «Формат вывода» руководства для команд git diff-* ), либо формат diff-patch.

Diffcore-break: для разделения полных перезаписей

Второе преобразование в цепочке — diffcore-break и управляется опцией -B команд git diff-* . Это используется для обнаружения пары файлов, представляющей "complete rewrite" , и разбиения такой пары файлов на две пары файлов, которые представляют собой удаление и создание. Например, если вход содержал эту пару файлов:

:100644 100644 bcd1234... 0123456... M file0

и если он обнаруживает, что файл "file0" полностью перезаписан, он меняет его на:

:100644 000000 bcd1234... 0000000... D file0
:000000 100644 0000000... 0123456... A file0

С целью разрыва пары файлов команда diffcore-break проверяет степень изменений между содержимым файлов до и после модификации (т. е. содержимым, которое имеет "bcd1234…​" и "0123456…​" в качестве идентификатора контента SHA-1, в приведенном выше примере). Количество удалений исходного содержимого и вставки нового материала суммируется, и если оно превышает "break score" , пара файлов разбивается на две части. Оценка разрыва по умолчанию равна 50 % от размера меньшего размера оригинала и результата (т. е. если редактирование сжимает файл, используется размер результата; если редактирование удлиняет файл, используется размер оригинала). ), и его можно настроить, указав число после параметра "-B" (например, "-B75" , чтобы указать использовать 75%).

Diffcore-rename: для обнаружения переименований и копий.

Это преобразование используется для обнаружения переименований и копий и контролируется опцией -M (для обнаружения переименований) и опцией -C (также для обнаружения копий) команд git diff-* . Если входные данные содержали эти пары файлов:

:100644 000000 0123456... 0000000... D fileX
:000000 100644 0000000... 0123456... A file0

и содержимое удаленного файла fileX достаточно похоже на содержимое созданного файла file0, затем обнаружение переименования объединяет эти пары файлов и создает:

:100644 100644 0123456... 0123456... R100 fileX file0

При использовании опции "-C" исходное содержимое измененных файлов и удаленных файлов (а также немодифицированных файлов, если используется опция "--find-copies-harder" ) рассматриваются как кандидаты исходных файлов в операции переименования/копирования. Если входные данные были похожи на эти пары файлов, это говорит об измененном файле fileY и вновь созданном файле file0:

:100644 100644 0123456... 1234567... M fileY
:000000 100644 0000000... bcd3456... A file0

исходное содержимое fileY и результирующее содержимое file0 сравниваются, и если они достаточно похожи, они заменяются на:

:100644 100644 0123456... 1234567... M fileY
:100644 100644 0123456... bcd3456... C100 fileY file0

Как при переименовании, так и при обнаружении копирования один и тот же алгоритм "extent of changes" , используемый в diffcore-break, используется для определения того, являются ли два файла "similar enough" , и его можно настроить для использования показателя сходства, отличного от значения по умолчанию, равного 50 %, путем указания числа после "-M" или Опция "-C" (например, "-M8" , чтобы указать использовать 8/10 = 80%).

Обратите внимание: если обнаружение переименования включено, но обнаружение копирования и разрыва отключено, обнаружение переименования добавляет предварительный шаг, который сначала проверяет, перемещаются ли файлы между каталогами, сохраняя при этом их имя. Если в каталог добавлен файл, содержимое которого достаточно похоже на файл с тем же именем, который был удален из другого каталога, он пометит их как переименованные и исключит их из последующего квадратичного шага (того, который попарно сравнивает все несовпадающие файлы для поиска совпадений "best" , определяемых по наибольшему сходству контента). Так, например, если удаленный docs/ext.txt и добавленный docs/config/ext.txt достаточно похожи, они будут помечены как переименованные и предотвратят добавление docs/ext.md, которое может быть еще более похоже на удаленный файл docs/ext.txt не будет считаться местом назначения переименования на более позднем этапе. По этой причине на предварительном этапе "match same filename" используется немного более высокий порог, чтобы пометить пару файлов как переименованную и прекратить рассмотрение других кандидатов на лучшее совпадение. На этом предварительном проходе для каждого файла выполняется не более одного сравнения; поэтому, если после точного обнаружения переименования в иерархии каталогов осталось несколько файлов ext.txt, этот предварительный шаг можно пропустить для этих файлов.

Примечание. Когда опция "-C" используется с опцией --find-copies-harder , команды git diff-* передают в механизм diffcore немодифицированные пары файлов, а также измененные. Это позволяет детектору копирования рассматривать немодифицированные файлы как кандидаты на источники копирования, но это замедляет работу. Без --find-copies-harder команды git diff-* могут обнаруживать копии только в том случае, если скопированный файл был изменен в том же наборе изменений.

Diffcore-merge-broken: для объединения полных перезаписей

Это преобразование используется для объединения пар файлов, разбитых с помощью diffcore-break и не преобразованных в переименование/копирование с помощью diffcore-rename, обратно в одну модификацию. Это всегда выполняется, когда используется diffcore-break.

Для объединения поврежденных пар файлов используются вычисления "extent of changes" , отличные от тех, которые используются командами diffcore-break и diffcore-rename. Он учитывает только удаление из оригинала и не учитывает вставку. Если вы удалили только 10 строк из 100-строчного документа, даже если вы добавили 910 новых строк, чтобы создать новый 1000-строчный документ, вы не сделали полной переписывания. diffcore-break прерывает такой случай, чтобы помочь diffcore-rename рассматривать такие пары файлов как кандидаты на обнаружение переименования/копирования, но если сломанные таким образом пары файлов не сопоставлялись с другими парами файлов для создания переименования/копирования, то это преобразование объединяет их обратно в исходный "modification" .

Параметр "extent of changes" можно изменить со значения 80% по умолчанию (то есть, если не будет удалено более 80% исходного материала, сломанные пары объединяются обратно в одну модификацию), задав второе число для опции -B, как показано ниже. :

  • -B50/60 (дайте 50% "break score" для diffcore-break, используйте 60% для diffcore-merge-broken).

  • -B/60 (то же, что и выше, поскольку значение diffcore-break по умолчанию равно 50%).

Обратите внимание, что в более ранней реализации неработающая пара оставалась в виде отдельных патчей создания и удаления. Это был ненужный взлом, и последняя реализация всегда объединяет все сломанные пары обратно в модификации, но результирующий вывод патча форматируется по-другому для облегчения просмотра в случае такой полной перезаписи, показывая все содержимое старой версии с префиксом - . , за которым следует все содержимое новой версии с префиксом + .

Diffcore-кирка: для обнаружения добавления/удаления указанной строки.

Это преобразование ограничивает набор пар файлов теми, которые определенным образом изменяют указанные строки между прообразом и постизображением. Параметры -S<блок текста> и -G<регулярное выражение> используются для указания различных способов поиска этих строк.

"-S<block of text>" обнаруживает пары файлов, в которых прообраз и постизображение имеют разное количество вхождений указанного блока текста. По определению, он не обнаружит перемещения внутри файла. Кроме того, когда набор изменений перемещает файл целиком, не затрагивая интересующую строку, diffcore-rename срабатывает как обычно, и -S опускает пару файлов (поскольку количество вхождений этой строки не изменилось в этой обнаруженной при переименовании паре файлов). При использовании с --pickaxe-regex рассматривайте <блок текста> как расширенное регулярное выражение POSIX, а не как литеральную строку.

"-G<regular expression>" (мнемоника: grep) обнаруживает пары файлов, в текстовых различиях которых есть добавленная или удаленная строка, соответствующая заданному регулярному выражению. Это означает, что он будет обнаруживать перемещения внутри файла (или то, что при обнаружении переименования учитывает тот же файл), что является шумом. Реализация дважды запускает diff и grep, а это может быть довольно затратно. Для ускорения работы двоичные файлы без фильтров textconf будут игнорироваться.

Когда -S или -G используются без --pickaxe-all , в выходных данных сохраняются только пары файлов, соответствующие их соответствующему критерию. При использовании --pickaxe-all , если хотя бы одна пара файлов соответствует соответствующему критерию в наборе изменений, весь набор изменений сохраняется. Такое поведение предназначено для облегчения просмотра изменений в контексте всего набора изменений.

Diffcore-order: для сортировки вывода по именам файлов.

Это используется для изменения порядка пар файлов по вкусу пользователя (или проекта) и контролируется опцией -O команд git diff-* .

Для этого требуется текстовый файл, каждая строка которого представляет собой шаблон оболочки. Пары файлов, соответствующие шаблону подстановки в более ранней строке файла, выводятся перед теми, которые соответствуют более поздней строке, а пары файлов, не соответствующие ни одному шаблону подстановки, выводятся последними.

Например, типичный файл заказа для ядра Git , вероятно, будет выглядеть так:

README
Makefile
Documentation
*.h
*.c
t

Diffcore-rotate: для изменения начала вывода пути.

Это преобразование принимает один путь и вращает набор пар файлов так, чтобы пара файлов для данного пути была первой, при необходимости отбрасывая пути, которые идут перед ней. Это используется для реализации опций --skip-to и --rotate-to . Это ошибка, когда указанный путь отсутствует в наборе пар файлов, но выдавать ошибку при использовании с семейством команд "git log" бесполезно, поскольку неразумно ожидать, что данный путь будет изменяться при каждом коммите. показано командой "git log" . По этой причине при использовании с "git log" пара файлов, которая сортируется так же, как или первая, которая сортируется после заданного пути, является местом начала вывода.

Использование этого преобразования в сочетании с diffcore-order приведет к неожиданным результатам, поскольку входные данные этого преобразования, скорее всего, не сортируются, когда действует diffcore-order.
