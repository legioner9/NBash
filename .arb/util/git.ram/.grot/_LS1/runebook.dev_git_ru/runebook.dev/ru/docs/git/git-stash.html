git-stash

Name

git-stash — спрятать изменения в грязном рабочем каталоге

Synopsis

git stash list [<log-options>]
git stash show [-u | --include-untracked | --only-untracked] [<diff-options>] [<stash>]
git stash drop [-q | --quiet] [<stash>]
git stash pop [--index] [-q | --quiet] [<stash>]
git stash apply [--index] [-q | --quiet] [<stash>]
git stash branch <branchname> [<stash>]
git stash [push [-p | --patch] [-S | --staged] [-k | --[no-]keep-index] [-q | --quiet]
             [-u | --include-untracked] [-a | --all] [(-m | --message) <message>]
             [--pathspec-from-file=<file> [--pathspec-file-nul]]
             [--] [<pathspec>…​]]
git stash save [-p | --patch] [-S | --staged] [-k | --[no-]keep-index] [-q | --quiet]
             [-u | --include-untracked] [-a | --all] [<message>]
git stash clear
git stash create [<message>]
git stash store [(-m | --message) <message>] [-q | --quiet] <commit>

Description

Используйте git stash , если вы хотите записать текущее состояние рабочего каталога и индекса, но хотите вернуться в чистый рабочий каталог. Команда сохраняет ваши локальные изменения и возвращает рабочий каталог в соответствии с коммитом HEAD .

Изменения, спрятанные этой командой, могут быть перечислены с помощью git stash list , проверены с помощью git stash show и восстановлены (возможно, поверх другого коммита) с помощью git stash apply . Вызов git stash без каких-либо аргументов эквивалентен git stash push . По умолчанию тайник указан как "WIP on branchname …​" , но при его создании вы можете указать более подробное сообщение в командной строке.

Последний созданный вами тайник хранится в refs/stash ; более старые тайники находятся в рефлоге этой ссылки и могут быть названы с использованием обычного синтаксиса рефлога (например, stash@{0} — самый последний созданный тайник, stash@{1} — предыдущий, stash@{2.hours.ago} также возможен). На тайники также можно ссылаться, указав только индекс тайника (например, целое число n эквивалентно stash@{n} ).

Commands

push [-p|--patch] [-S|--staged] [-k|--[no-]keep-index] [-u|--include-untracked] [-a|--all] [ -q|--quiet] [(-m|--message) <сообщение>] [--pathspec-from-file=<file> [--pathspec-file-nul]] [--] [<pathspec> …​]

Сохраните локальные изменения в новый stash entry и откатите их обратно в HEAD (в рабочем дереве и в индексе). Часть <message> не является обязательной и содержит описание вместе с спрятанным состоянием.

Для быстрого создания снимка можно опустить "push" . В этом режиме аргументы, не являющиеся параметрами, не допускаются, чтобы предотвратить создание нежелательной записи в тайнике подкомандой с ошибкой. Двумя исключениями из этого правила являются stash -p , который действует как псевдоним для stash push -p , и элементы pathspec, которые разрешены после двойного дефиса -- для устранения неоднозначности.

save [-p|--patch] [-S|--staged] [-k|--[no-]keep-index] [-u|--include-untracked] [-a|--all] [ -q|--тихо] [<сообщение>]

Эта опция устарела в пользу git stash push . Он отличается от "stash push" тем, что не может принимать pathspec. Вместо этого все аргументы, не являющиеся параметрами, объединяются для формирования тайного сообщения.

list [<log-options>]

Перечислите записи тайника, которые у вас есть на данный момент. Каждый stash entry указан с его именем (например, stash@{0} — последняя запись, stash@{1} — предыдущая и т. д.), именем ветки, которая была текущей на момент создания записи, и кратким описанием фиксации, на основе которой была сделана запись. на.

stash@{0}: WIP on submit: 6ebd0e2... Update git-stash documentation
stash@{1}: On master: 9cc0589... Add git-stash

Команда использует параметры, применимые к команде git log , для управления тем, что и как отображается. См. git-log[1] .

show [-u|--include-untracked|--only-untracked] [<diff-options>] [<stash>]

Покажите изменения, записанные в записи тайника, как разницу между спрятанным содержимым и фиксацией, когда запись тайника была впервые создана. По умолчанию команда показывает diffstat, но она принимает любой формат, известный git diff (например, git stash show -p stash@{1} для просмотра второй последней записи в форме исправления). Если <diff-option> не указан, поведение по умолчанию будет задано переменными конфигурации stash.showStat и stash.showPatch . Вы также можете использовать stash.showIncludeUntracked , чтобы установить, включен ли --include-untracked по умолчанию.

pop [--index] [-q|--quiet] [<тайник>]

Удалите одно спрятанное состояние из списка спрятанного и примените его поверх текущего состояния рабочего дерева, т. е. выполните операцию, обратную git stash push . Рабочий каталог должен соответствовать индексу.

Применение состояния может потерпеть неудачу из-за конфликтов; в этом случае он не удаляется из списка тайника. Вам необходимо разрешить конфликты вручную, а затем вручную позвонить на git stash drop .

применить [--index] [-q|--quiet] [<stash>]

Как pop , но не удаляйте состояние из списка тайника. В отличие от pop , <stash> может быть любым коммитом, похожим на коммит, созданный stash push или stash create .

ветка <имя ветки> [<тайник>]

Создает и извлекает новую ветку с именем <branchname> , начиная с фиксации, в которой изначально был создан <stash> , применяет изменения, записанные в <stash> , к новому рабочему дереву и индексу. Если это удалось и <stash> является ссылкой в ​​форме stash@{<revision>} , то <stash> удаляется.

Это полезно, если ветка, в которой вы запускали git stash push , изменилась настолько, что git stash apply выходит из строя из-за конфликтов. Поскольку запись в кэше применяется поверх коммита, который был HEAD на момент запуска git stash , она восстанавливает исходное спрятанное состояние без конфликтов.

clear

Удалите все записи тайника. Обратите внимание, что эти записи затем будут удалены, и их будет невозможно восстановить (возможную стратегию см. в Examples ниже).

drop [-q|--quiet] [<тайник>]

Удалите одну запись тайника из списка записей тайника.

create

Создайте запись в хранилище (которая является обычным объектом фиксации) и верните имя его объекта, не сохраняя его где-либо в пространстве имен ref. Это предназначено для использования в сценариях. Вероятно, это не та команда, которую вы хотите использовать; см. "push" выше.

store

Сохраните заданный тайник, созданный с помощью git stash create (который представляет собой висячую фиксацию слияния), в ссылке на тайник, обновив журнал хранилища. Это предназначено для использования в сценариях. Вероятно, это не та команда, которую вы хотите использовать; см. "push" выше.

Options

-a
--all

Эта опция действительна только для команд push и save .

Все игнорируемые и неотслеживаемые файлы также сохраняются, а затем очищаются с помощью git clean .

-u
--include-untracked
--no-include-untracked

При использовании с командами push и save все неотслеживаемые файлы также сохраняются, а затем очищаются с помощью git clean .

При использовании с командой show отображать неотслеживаемые файлы в записи хранилища как часть различий.

--only-untracked

Эта опция действительна только для команды show .

Показывать только неотслеживаемые файлы в записи тайника как часть различий.

--index

Эта опция действительна только для команд pop и apply .

Пытается восстановить изменения не только рабочего дерева, но и индекса. Однако это может потерпеть неудачу, если у вас есть конфликты (которые хранятся в индексе, и поэтому вы больше не можете применять изменения в том виде, в каком они были изначально).

-k
--keep-index
--no-keep-index

Эта опция действительна только для команд push и save .

Все изменения, уже добавленные в индекс, остаются нетронутыми.

-p
--patch

Эта опция действительна только для команд push и save .

Интерактивно выбирайте фрагменты из различий между HEAD и рабочим деревом, которые нужно спрятать. Запись тайника построена таким образом, что ее состояние индекса совпадает с состоянием индекса вашего репозитория, а ее рабочее дерево содержит только те изменения, которые вы выбрали в интерактивном режиме. Выбранные изменения затем откатываются из вашего рабочего дерева. См. раздел «Интерактивный режим» git-add[1] , чтобы узнать, как работать в режиме --patch .

Вариант --patch подразумевает --keep-index . Вы можете использовать --no-keep-index , чтобы переопределить это.

-S
--staged

Эта опция действительна только для команд push и save .

Сохраняйте только те изменения, которые в данный момент подготовлены. Это похоже на базовый git commit , за исключением того, что состояние фиксируется в тайнике, а не в текущей ветке.

Опция --patch имеет приоритет над этой.

--pathspec-from-file=<file>

Эта опция действительна только для команды push .

Спецификация пути передается в <file> вместо аргументов командной строки. Если <file> — это точно - , то используется стандартный ввод. Элементы Pathspec разделяются LF или CR/LF. Элементы Pathspec можно заключать в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). См. также --pathspec-file-nul и глобальный --literal-pathspecs .

--pathspec-file-nul

Эта опция действительна только для команды push .

Имеет смысл только с --pathspec-from-file . Элементы Pathspec разделяются символом NUL, а все остальные символы воспринимаются буквально (включая символы новой строки и кавычки).

-q
--quiet

Эта опция действительна только для команд apply , drop , pop , push , save , store .

Тихо, подавляйте сообщения обратной связи.

--

Эта опция действительна только для команды push .

Отделяет спецификацию пути от опций в целях устранения неоднозначности.

<pathspec>…

Эта опция действительна только для команды push .

Новая запись в хранилище записывает измененные состояния только для файлов, соответствующих указанному пути. Записи индекса и файлы рабочего дерева затем также откатываются до состояния в HEAD только для этих файлов, оставляя файлы, которые не соответствуют спецификации пути, нетронутыми.

Дополнительные сведения см. в записи pathspec в gitglossary[7] .

<stash>

Эта опция действительна только для команд apply , branch , drop , pop , show .

Ссылка вида stash@{<revision>} . Если <stash> не указан, предполагается самый последний тайник (то есть stash@{0} ).

Discussion

Запись в тайнике представлена ​​как фиксация, дерево которой записывает состояние рабочего каталога, а ее первым родителем является фиксация по адресу HEAD , когда запись была создана. Дерево второго родителя записывает состояние индекса при создании записи и становится дочерним элементом фиксации HEAD . Граф предков выглядит следующим образом:

       .----W
      /    /
-----H----I

где H — это фиксация HEAD , I — фиксация, записывающая состояние индекса, а W — фиксация, записывающая состояние рабочего дерева.

Examples

Въезжая в грязное дерево

Когда вы находитесь в середине чего-то, вы узнаете, что есть изменения, которые, возможно, имеют отношение к тому, что вы делаете. Если ваши локальные изменения не конфликтуют с изменениями в восходящем направлении, простой git pull позволит вам двигаться вперед.

Однако бывают случаи, когда ваши локальные изменения конфликтуют с изменениями вышестоящего уровня, и git pull отказывается перезаписывать ваши изменения. В таком случае вы можете спрятать изменения, выполнить извлечение, а затем разархивировать, вот так:

$ git pull
 ...
file foobar not up to date, cannot merge.
$ git stash
$ git pull
$ git stash pop
Interrupted workflow

Когда вы чем-то заняты, приходит начальник и требует, чтобы вы немедленно что-то исправили. Традиционно вы делаете фиксацию во временной ветке, чтобы сохранить изменения, и возвращаетесь в исходную ветку, чтобы внести экстренное исправление, например:

# ... hack hack hack ...
$ git switch -c my_wip
$ git commit -a -m "WIP"
$ git switch master
$ edit emergency fix
$ git commit -a -m "Fix in a hurry"
$ git switch my_wip
$ git reset --soft HEAD^
# ... continue hacking ...

Вы можете использовать git stash , чтобы упростить вышеизложенное, например:

# ... hack hack hack ...
$ git stash
$ edit emergency fix
$ git commit -a -m "Fix in a hurry"
$ git stash pop
# ...продолжаем взлом...
Тестирование частичных коммитов

Вы можете использовать git stash push --keep-index , если хотите сделать две или более фиксации изменений в рабочем дереве и протестировать каждое изменение перед фиксацией:

# ... hack hack hack ...
$ git add --patch foo            # добавляем в индекс только первую часть
$ git stash push --keep-index    # сохраняем все остальные изменения в тайнике
$ edit/build/test first part
$ git commit -m 'First part'     # зафиксировать полностью протестированное изменение
$ git stash pop                  # подготовьтесь к работе над всеми остальными изменениями
# ... повторяйте вышеуказанные пять шагов, пока не останется один коммит...
$ edit/build/test remaining parts
$ git commit foo -m 'Remaining parts'
Сохранение несвязанных изменений для использования в будущем.

Когда вы находитесь в процессе масштабных изменений и обнаруживаете какую-то несвязанную проблему, которую не хотите забыть исправить, вы можете внести изменения, подготовить их и использовать git stash push --staged , чтобы спрятать их для будущего использования. Это похоже на фиксацию поэтапных изменений, только фиксация оказывается в тайнике, а не в текущей ветке.

# ... хак хак хак ...
$ git add --patch foo           #добавляем несвязанные изменения в индекс
$ git stash push --staged       # сохраняем эти изменения в тайник
# ... hack hack hack, завершить текущие изменения...
$ git commit -m 'Massive'       # зафиксировать полностью протестированные изменения
$ git switch fixup-branch       # переключиться на другую ветку
$ git stash pop                 # для завершения работы над сохраненными изменениями
Восстановление записей тайника, которые были очищены/удалены по ошибке

Если вы по ошибке удалите или очистите записи тайника, их невозможно будет восстановить с помощью обычных механизмов безопасности. Однако вы можете попробовать следующее заклинание, чтобы получить список записей тайника, которые все еще находятся в вашем репозитории, но больше не доступны:

git fsck --unreachable |
grep commit | cut -d\  -f3 |
xargs git log --merges --no-walk --grep=WIP

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

stash.showIncludeUntracked

Если для этого параметра установлено значение true, команда git stash show покажет неотслеживаемые файлы записи тайника. По умолчанию ложь. См. описание команды show в git-stash[1] .

stash.showPatch

Если для этого параметра установлено значение true, команда git stash show без опции покажет запись тайника в форме патча. По умолчанию ложь. См. описание команды show в git-stash[1] .

stash.showStat

Если для этого параметра установлено значение true, команда git stash show без опции покажет diffstat записи тайника. По умолчанию истинно. См. описание команды show в git-stash[1] .
