git-describe

Name

git-describe — дать объекту удобочитаемое имя на основе доступной ссылки.

Synopsis

git describe [--all] [--tags] [--contains] [--abbrev=<n>] [<commit-ish>…​]
git describe [--all] [--tags] [--contains] [--abbrev=<n>] --dirty[=<mark>]
git describe <blob>

Description

Команда находит самый последний тег, доступный из фиксации. Если тег указывает на фиксацию, то отображается только тег. В противном случае к имени тега добавляется количество дополнительных коммитов поверх помеченного объекта и сокращенное имя объекта самого последнего коммита. Результатом является имя объекта "human-readable" , которое также можно использовать для идентификации фиксации других команд git .

По умолчанию (без --all или --tags) git describe отображает только аннотированные теги. Для получения дополнительной информации о создании аннотированных тегов см. параметры -a и -s для git-tag[1] .

Если данный объект ссылается на большой двоичный объект, он будет описан как <commit-ish>:<path> , так что большой двоичный объект можно найти по адресу <path> в <commit-ish> , который сам описывает первую фиксацию, в которой этот большой двоичный объект встречается при обратном обходе ревизий из HEAD.

Options

<commit-ish>…

Зафиксируйте имена объектов для описания. По умолчанию — HEAD, если опущено.

--dirty[=<mark>]
--broken[=<mark>]

Опишите состояние рабочего дерева. Когда рабочее дерево соответствует HEAD, выходные данные такие же, как у "git describe HEAD" . Если рабочее дерево имеет локальную модификацию, к нему добавляется "-dirty" . Если репозиторий поврежден и Git не может определить наличие локальных изменений, Git выдаст ошибку, если только не задано '--broken', к которому вместо этого добавляется суффикс "-broken" .

--all

Вместо использования только аннотированных тегов используйте любую ссылку, найденную в пространстве имен refs/ . Эта опция позволяет сопоставить любую известную ветвь, ветвь удаленного отслеживания или облегченный тег.

--tags

Вместо использования только аннотированных тегов используйте любой тег, найденный в пространстве имен refs/tags . Эта опция позволяет сопоставить облегченный (неаннотированный) тег.

--contains

Вместо поиска тега, который предшествует фиксации, найдите тег, который идет после фиксации и, таким образом, содержит ее. Автоматически подразумевает --tags.

--abbrev=<n>

Вместо использования количества шестнадцатеричных цифр по умолчанию (которое будет варьироваться в зависимости от количества объектов в репозитории со значением по умолчанию 7) сокращенного имени объекта, используйте <n> цифр или столько цифр, сколько необходимо для формирования уникального имени объекта. имя объекта. Значение <n>, равное 0, подавляет длинный формат, отображая только ближайший тег.

--candidates=<n>

Вместо того, чтобы рассматривать только 10 самых последних тегов в качестве кандидатов для описания входной фиксации, рассмотрите до <n> кандидатов. Увеличение <n> выше 10 займет немного больше времени, но может дать более точный результат. Если <n> равно 0, будут выводиться только точные совпадения.

--exact-match

Выводите только точные совпадения (тег напрямую ссылается на предоставленный коммит). Это синоним --candidates=0.

--debug

Подробно отобразить информацию об используемой стратегии поиска с учетом стандартной ошибки. Имя тега по-прежнему будет печататься на стандартном выходе.

--long

Всегда выводите длинный формат (тег, количество коммитов и сокращенное имя коммита), даже если он соответствует тегу. Это полезно, если вы хотите увидеть части имени объекта фиксации в выходных данных "describe" , даже если рассматриваемая фиксация является версией с тегами. Вместо того, чтобы просто выдавать имя тега, он будет описывать такой коммит как v1.2-0-gdeadbee (0-й коммит, начиная с тега v1.2, который указывает на объект DeadBee…​.).

--match <pattern>

Учитывайте только теги, соответствующие заданному шаблону glob(7) , за исключением префикса "refs/tags/" . При использовании с --all он также учитывает локальные ветки и ссылки удаленного отслеживания, соответствующие шаблону, исключая соответственно префиксы "refs/heads/" и "refs/remotes/" ; ссылки других типов никогда не учитываются. Если задано несколько раз, список шаблонов будет накапливаться, и будут учитываться теги, соответствующие любому из шаблонов. Используйте --no-match , чтобы очистить и сбросить список шаблонов.

--exclude <pattern>

Не учитывайте теги, соответствующие данному шаблону glob(7) , за исключением префикса "refs/tags/" . При использовании с --all он также не учитывает локальные ветки и ссылки удаленного отслеживания, соответствующие шаблону, исключая соответственно префикс "refs/heads/" и "refs/remotes/" ; ссылки других типов никогда не учитываются. Если задано несколько раз, список шаблонов будет накоплен, а теги, соответствующие любому из шаблонов, будут исключены. В сочетании с --match тег будет учитываться, если он соответствует хотя бы одному шаблону --match и не соответствует ни одному из шаблонов --exclude. Используйте --no-exclude , чтобы очистить и сбросить список шаблонов.

--always

Показывать уникальный сокращенный объект фиксации в качестве резервного варианта.

--first-parent

Следуйте только первому родительскому коммиту, увидев коммит слияния. Это полезно, если вы не хотите сопоставлять теги в ветвях, объединенных в истории целевого коммита.

Examples

С чем-то вроде текущего дерева git.git я получаю:

[torvalds@g5 git]$ git describe parent
v1.0.4-14-g2414721

т.е. текущая глава моей ветки "parent" основана на версии 1.0.4, но, поскольку в ней есть несколько коммитов, в описание добавлено количество дополнительных коммитов ( "14" ) и сокращенное имя объекта для самого коммита ( "2414721" ) в конце.

Количество дополнительных коммитов — это количество коммитов, которое будет отображаться "git log v1.0.4..parent" . Хэш-суффикс — "-g" + однозначное сокращение для подсказки родительского коммита (который был 2414721b194453f058079d897d13c4e377f92dc6 ). Длина аббревиатуры масштабируется по мере роста репозитория, используя приблизительное количество объектов в репозитории и немного математики вокруг парадокса дня рождения, и по умолчанию составляет минимум 7. Префикс "g" обозначает "git" и используется для описания версия программного обеспечения в зависимости от SCM, с помощью которого оно управляется. Это полезно в среде, где люди могут использовать разные SCM.

Выполнение git describe для имени тега покажет только имя тега:

[torvalds@g5 git]$ git describe v1.0.4
v1.0.4

С --all команда может использовать заголовки ветвей в качестве ссылок, поэтому в выводе также отображается путь к ссылке:

[torvalds@g5 git]$ git describe --all --abbrev=4 v1.0.5^2
tags/v1.0.0-21-g975b
[torvalds@g5 git]$ git describe --all --abbrev=4 ГОЛОВКА^
heads/lt/describe-7-g975b

Если для --abbrev установлено значение 0, команду можно использовать для поиска ближайшего тэга без какого-либо суффикса:

[torvalds@g5 git]$ git describe --abbrev=0 v1.0.5^2
tags/v1.0.0

Обратите внимание, что суффикс, который вы получите, если наберете эти команды сегодня, может быть длиннее, чем тот, который Линус видел выше, когда запускал эти команды, поскольку в вашем репозитории Git могут быть новые коммиты, имена объектов которых начинаются с 975b, которых тогда не существовало, и суффикс "-g975b" . одного может быть недостаточно, чтобы устранить неоднозначность этих коммитов.

Search strategy

Для каждого предоставленного коммита git describe сначала будет искать тег, который помечает именно этот коммит. Теги с аннотациями всегда будут предпочтительнее облегченных тегов, а теги с более новыми датами всегда будут предпочтительнее тегов с более старыми датами. Если будет найдено точное совпадение, будет выведено его имя и поиск будет остановлен.

Если точное совпадение не найдено, git describe вернется к истории коммитов, чтобы найти родительский коммит, который был помечен. Тег предка будет выведен вместе с сокращением SHA-1 входного коммита. Если был указан --first-parent , то при обходе будет учитываться только первый родительский элемент каждого коммита.

Если во время обхода было обнаружено несколько тегов, то тег, который имеет наименьшее количество коммитов, отличных от входного коммита, будет выбран и выведен. Здесь наименьшее количество различных коммитов определяется как количество коммитов, которое будет показано git log tag..input , будет наименьшим возможным количеством коммитов.

Bugs

Объекты дерева, а также объекты-теги, не указывающие на коммиты, не могут быть описаны. При описании больших двоичных объектов облегченные теги, указывающие на большие двоичные объекты, игнорируются, но большой двоичный объект по-прежнему описывается как <commit-ish>:<path>, несмотря на то, что облегченный тег является предпочтительным.
