git-tag

Name

git-tag — Создать, просмотреть, удалить или проверить объект тега, подписанный с помощью GPG.

Synopsis

git tag [-a | -s | -u <key-id>] [-f] [-m <msg> | -F <file>] [-e]
        <tagname> [<commit> | <object>]
git tag -d <tagname>…​
git tag [-n[<num>]] -l [--contains <commit>] [--no-contains <commit>]
        [--points-at <object>] [--column[=<options>] | --no-column]
        [--create-reflog] [--sort=<key>] [--format=<format>]
        [--merged <commit>] [--no-merged <commit>] [<pattern>…​]
git tag -v [--format=<format>] <tagname>…​

Description

Добавьте ссылку на тег в refs/tags/ , если не указан -d/-l/-v для удаления, перечисления или проверки тегов.

Если не указан -f , именованный тег еще не должен существовать.

Если передается один из -a , -s или -u <key-id> , команда создает объект tag и требует сообщения тега. Если не указаны -m <msg> или -F <file> , запускается редактор, позволяющий пользователю ввести сообщение тега.

Если указаны -m <msg> или -F <file> и отсутствуют -a , -s и -u <key-id> , подразумевается -a .

В противном случае создается ссылка на тег, указывающая непосредственно на данный объект (т. е. облегченный тег).

Объект тега со знаком GnuPG будет создан при использовании -s или -u <key-id> . Если -u <key-id> не используется, идентификатор коммиттера текущего пользователя используется для поиска ключа GnuPG для подписи. Переменная конфигурации gpg.program используется для указания пользовательского двоичного файла GnuPG.

Объекты тегов (созданные с помощью -a , -s или -u ) называются тегами "annotated" ; они содержат дату создания, имя и адрес электронной почты тегера, сообщение о теге и необязательную подпись GnuPG. Тогда как тег "lightweight" — это просто имя объекта (обычно объекта фиксации).

Теги с аннотациями предназначены для выпуска, а облегченные теги предназначены для private или меток временных объектов. По этой причине некоторые команды git для именования объектов (например, git describe ) по умолчанию игнорируют облегченные теги.

Options

-a
--annotate

Создайте неподписанный объект-тег с аннотациями.

-s
--sign

Создайте тег, подписанный GPG, используя ключ адреса электронной почты по умолчанию. Поведение подписи тега GPG по умолчанию контролируется конфигурационной переменной tag.gpgSign , если она существует, или отключено в противном случае. См. git-config[1] .

--no-sign

Переопределить переменную конфигурации tag.gpgSign , которая настроена на принудительное подписание каждого тега.

-u <key-id>
--local-user=<key-id>

Создайте тег, подписанный GPG, используя данный ключ.

-f
--force

Заменить существующий тег заданным именем (вместо сбоя)

-d
--delete

Удалите существующие теги с заданными именами.

-v
--verify

Проверьте подпись GPG для указанных имен тегов.

-n<num>

<num> указывает, сколько строк аннотации, если таковые имеются, печатаются при использовании -l. Подразумевается --list .

По умолчанию строки аннотаций не печатаются. Если для -n не указан номер, печатается только первая строка. Если тег не аннотирован, вместо него отображается сообщение о фиксации.

-l
--list

Список тегов. При использовании дополнительного <pattern>... , например git tag --list 'v-*' , перечисляйте только теги, соответствующие шаблону(ам).

Запуск "git tag" без аргументов также выводит список всех тегов. Шаблон представляет собой подстановочный знак оболочки (т. е. сопоставляется с помощью fnmatch(3)). Может быть задано несколько шаблонов; если какой-либо из них совпадает, отображается тег.

Эта опция указывается неявно, если предоставляется любая другая опция в виде списка, например --contains . Подробности смотрите в документации по каждому из этих вариантов.

--sort=<key>

Сортировка по указанному ключу. Префикс - для сортировки в порядке убывания значения. Вы можете использовать опцию --sort=<key> несколько раз, и в этом случае последний ключ становится первичным ключом. Также поддерживается "version:refname" или "v:refname" (имена тегов считаются версиями). На порядок сортировки "version:refname" также может влиять переменная конфигурации "versionsort.suffix" . Поддерживаемые клавиши такие же, как и в git for-each-ref . Порядок сортировки по умолчанию соответствует значению, настроенному для переменной tag.sort , если она существует, или лексикографическому порядку в противном случае. См. git-config[1] .

--color[=<when>]

Соблюдайте любые цвета, указанные в опции --format . Поле <when> должно быть одним из always , never или auto (если <when> отсутствует, ведите себя так, как если бы always был указан).

-i
--ignore-case

Теги сортировки и фильтрации нечувствительны к регистру.

--omit-empty

Не печатайте новую строку после форматированных ссылок, где формат расширяется до пустой строки.

--column[=<options>]
--no-column

Отображение списка тегов в столбцах. Синтаксис опции см. в переменной конфигурации column.tag . --column и --no-column без дополнительных опций эквивалентны always и never соответственно.

Эта опция применима только при перечислении тегов без линий аннотации.

--contains [<commit>]

Перечислите только теги, содержащие указанный коммит (HEAD, если не указан). Подразумевается --list .

--no-contains [<commit>]

Перечислите только теги, которые не содержат указанный коммит (HEAD, если не указан). Подразумевается --list .

--merged [<commit>]

Перечислите только теги, чьи коммиты доступны из указанного коммита ( HEAD , если не указано).

--no-merged [<commit>]

Перечислите только теги, чьи фиксации недоступны из указанного коммита ( HEAD , если не указано).

--points-at <object>

Перечисляет только теги данного объекта (HEAD, если не указано). Подразумевается --list .

-m <msg>
--message=<msg>

Используйте данное сообщение тега (вместо запроса). Если указано несколько параметров -m , их значения объединяются в отдельные абзацы. Подразумевается -a , если задан none из -a , -s или -u <key-id> .

-F <file>
--file=<file>

Возьмите сообщение тега из данного файла. Используйте - , чтобы прочитать сообщение со стандартного ввода. Подразумевается -a , если задан none из -a , -s или -u <key-id> .

-e
--edit

Сообщение, взятое из файла с -F и командной строки с -m , обычно используется в качестве неизмененного сообщения тега. Эта опция позволяет дополнительно редактировать сообщение, полученное из этих источников.

--cleanup=<mode>

Эта опция определяет способ очистки сообщения тега. <mode> может быть одним из verbatim , whitespace и strip . Режим strip установлен по умолчанию. Режим verbatim вообще не меняет сообщение, whitespace удаляет только начальные/конечные пробельные строки, а strip удаляет как пробелы, так и комментарии.

--create-reflog

Создайте рефлог для тега. Чтобы глобально включить журналы обновлений для тегов, см. core.logAllRefUpdates в git-config[1] . Отрицательная форма --no-create-reflog только переопределяет более раннюю версию --create-reflog , но в настоящее время не отменяет настройку core.logAllRefUpdates .

--format=<format>

Строка, которая интерполирует %(fieldname) на основе отображаемой ссылки тега и объекта, на который она указывает. Формат такой же, как у git-for-each-ref[1] . Если не указано, по умолчанию используется %(refname:strip=2) .

<tagname>

Имя тега, который необходимо создать, удалить или описать. Новое имя тега должно пройти все проверки, определенные git-check-ref-format[1] . Некоторые из этих проверок могут ограничивать количество символов, разрешенных в имени тега.

<commit>
<object>

Объект, на который будет ссылаться новый тег, обычно это фиксация. По умолчанию HEAD.

Configuration

По умолчанию git tag в режиме подписи по умолчанию (-s) будет использовать ваш идентификатор коммиттера (в форме Your Name <[email protected]> ) для поиска ключа. Если вы хотите использовать другой ключ по умолчанию, вы можете указать его в конфигурации репозитория следующим образом:

[user]
    signingKey = <gpg-key_id>

pager.tag учитывается только при перечислении тегов, т. е. когда -l используется или подразумевается. По умолчанию используется пейджер. См. git-config[1] .

Discussion

On Re-tagging

Что делать, если вы пометили неправильный коммит и хотите изменить его?

Если вы никогда ничего не выбрасывали, просто пометьте это заново. Используйте "-f" для замены старого. И вы сделали.

Но если вы вытащили что-то (или другие могли просто прочитать ваш репозиторий напрямую), тогда другие уже увидят старый тег. В этом случае вы можете сделать одно из двух:

  1. Разумная вещь. Просто признай, что ты облажался, и возьми другое имя. Другие уже видели одно имя тега, и если вы сохраните то же имя, вы можете оказаться в ситуации, когда у двух людей есть "version X" , но на самом деле у них есть different и "X" . Так что просто назовите его "X.1" и покончим с этим.

  2. Безумная вещь. Новую версию очень хочется тоже назвать "X" , even though другие уже видели старую. Так что просто используйте git tag -f еще раз, как будто вы еще не опубликовали старый.

Однако Git не меняет (и не должен) теги за спиной пользователя. Поэтому, если кто-то уже получил старый тег, использование git pull в вашем дереве не должно просто заставлять его перезаписывать старый.

Если кто-то получил от вас тег выпуска, вы не можете просто изменить для него тег, обновив свой собственный. Это большая проблема безопасности, поскольку люди ДОЛЖНЫ иметь возможность доверять своим именам тегов. Если вы действительно хотите совершить безумный поступок, вам нужно просто признаться в этом и сказать людям, что вы напортачили. Вы можете сделать это, сделав объявление типа public , в котором говорится:

Ok, I messed up, and I pushed out an earlier version tagged as X. I
then fixed something, and retagged the *fixed* tree as X again.

If you got the wrong tag, and want the new one, please delete
the old one and fetch the new one by doing:

        git tag -d X
        git fetch origin tag X

to get my updated tag.

You can test which tag you have by doing

        git rev-parse X

which should return 0123456789abcdef.. if you have the new version.

Sorry for the inconvenience.

Это кажется немного сложным? Должен быть. Невозможно было бы правильно просто использовать "fix" автоматически. Люди должны знать, что их теги могли быть изменены.

Включено автоматическое отслеживание

Если вы следите за чужим деревом, вы, скорее всего, используете ветки удаленного отслеживания (например, refs/remotes/origin/master ). Обычно вам нужны теги с другого конца.

С другой стороны, если вы выполняете выборку, потому что хотите получить одноразовое слияние от кого-то другого, вам обычно не нужно получать оттуда теги. Чаще это случается с людьми, близкими к высшему уровню, но не ограничиваясь ими. Простые смертные при перетаскивании друг у друга не обязательно хотят автоматически получать метки точек привязки private от другого человека.

Часто сообщения "please pull" в списке рассылки содержат только две части информации: URL-адрес репозитория и имя ветки; это предназначено для легкого вырезания и вставки в конец командной строки git fetch :

Linus, please pull from

        git://git..../proj.git master

to get the following updates...

becomes:

$ git pull git://git..../proj.git master

В таком случае вы не хотите автоматически подписываться на теги другого человека.

Одним из важных аспектов Git является его распределенный характер, что в значительной степени означает, что в системе нет встроенных "upstream" или "downstream" . На первый взгляд может показаться, что приведенный выше пример указывает на то, что пространство имен тегов принадлежит высшему эшелону людей и что теги текут только вниз, но это не так. Это только показывает, что шаблон использования определяет, кто чьи теги заинтересованы.

Однократное извлечение является признаком того, что история коммитов теперь пересекает границу между одним кругом людей (например, "people who are primarily interested in the networking part of the kernel" ), которые могут иметь свой собственный набор тегов (например, "this is the third release candidate from the networking group to be proposed for general consumption with 2.6.21 release" ), и другим кругом людей (например, "people who integrate various subsystem improvements" ). Последних обычно не интересуют подробные теги, используемые внутри первой группы (именно это означает "internal" ). Поэтому в этом случае желательно не переходить по тегам автоматически.

Вполне возможно, что среди сетевых людей они захотят обменяться тегами внутри своей группы, но в этом рабочем процессе они, скорее всего, отслеживают прогресс друг друга, имея ветки удаленного отслеживания. Опять же, эвристика автоматического отслеживания таких тегов — это хорошо.

О тегах задним числом

Если вы импортировали некоторые изменения из другой системы контроля версий и хотите добавить теги для основных выпусков вашей работы, полезно иметь возможность указать дату для встраивания внутри объекта тега; такие данные в объекте тега влияют, например, на порядок тегов в интерфейсе gitweb.

Чтобы установить дату, используемую в будущих объектах тегов, установите переменную среды GIT_COMMITTER_DATE (см. дальнейшее обсуждение возможных значений; наиболее распространенная форма — "YYYY-MM-DD HH:MM" ).

For example:

$ GIT_COMMITTER_DATE="2006-10-02 10:31" git tag -s v1.0.1

Date formats

Переменные среды GIT_AUTHOR_DATE и GIT_COMMITTER_DATE поддерживают следующие форматы даты:

Внутренний формат Git

Это <unix-timestamp> <time-zone-offset> , где <unix-timestamp> — количество секунд, прошедших с эпохи UNIX. <time-zone-offset> — это положительное или отрицательное смещение от UTC. Например, CET (который на 1 час опережает UTC) — +0100 .

RFC 2822

Стандартный формат электронной почты, описанный в RFC 2822, например Thu, 07 Apr 2005 22:13:13 +0200 .

ISO 8601

Время и дата указаны стандартом ISO 8601, например 2005-04-07T22:13:13 . Анализатор также принимает пробел вместо символа T . Дробные доли секунды будут игнорироваться, например 2005-04-07T22:13:13.019 будет рассматриваться как 2005-04-07T22:13:13 .

Note
 Кроме того, часть даты принимается в следующих форматах: YYYY.MM.DD , MM/DD/YYYY и DD.MM.YYYY .

Files

$GIT_DIR/TAG_EDITMSG

Этот файл содержит сообщение текущего аннотированного тега. Если git tag завершает работу из-за ошибки до создания аннотированного тега, то сообщение тега, предоставленное пользователем в сеансе редактирования, будет доступно в этом файле, но может быть перезаписано при следующем вызове git tag .

Notes

При объединении нескольких фильтров --contains и --no-contains отображаются только ссылки, которые содержат хотя бы один из коммитов --contains и содержат none из коммитов --no-contains .

При объединении нескольких фильтров --merged и --no-merged отображаются только ссылки, доступные хотя бы из одного из коммитов --merged и из none из коммитов --no-merged .
