git-cherry-pick

Name

git-cherry-pick — применить изменения, внесенные некоторыми существующими коммитами.

Synopsis

git cherry-pick [--edit] [-n] [-m <parent-number>] [-s] [-x] [--ff]
                  [-S[<keyid>]] <commit>…​
git cherry-pick (--continue | --skip | --abort | --quit)

Description

Учитывая один или несколько существующих коммитов, примените изменения, вносимые каждым из них, записав для каждого новый коммит. Для этого необходимо, чтобы ваше рабочее дерево было чистым (без изменений после фиксации HEAD).

Когда неочевидно, как применить изменение, происходит следующее:

  1. Текущая ветвь и указатель HEAD остаются на последней успешно выполненной фиксации.

  2. Ссылка CHERRY_PICK_HEAD указывает на фиксацию, внесшую изменение, которое трудно применить.

  3. Пути, в которых изменения были применены без ошибок, обновляются как в индексном файле, так и в рабочем дереве.

  4. Для конфликтующих путей индексный файл записывает до трех версий, как описано в разделе "TRUE MERGE" документа git-merge[1] . Файлы рабочего дерева будут включать описание конфликта, заключенное в скобки обычными маркерами конфликта <<<<<<< и >>>>>>> .

  5. Никаких других модификаций не производится.

См. git-merge[1] для получения некоторых советов по разрешению таких конфликтов.

Options

<commit>…

Обязуется выбирать вишенку. Более полный список способов написания коммитов см. в gitrevisions[7] . Наборы коммитов могут передаваться, но по умолчанию обход не выполняется, как если бы была указана опция --no-walk , см. git-rev-list[1] . Обратите внимание, что при указании диапазона все аргументы <commit>… будут переданы в одну ревизию (см. более поздний пример, в котором используется maint master..next ).

-e
--edit

С помощью этой опции git cherry-pick позволит вам редактировать сообщение о фиксации перед фиксацией.

--cleanup=<mode>

Этот параметр определяет, как будет очищаться сообщение о фиксации перед передачей механизму фиксации. Дополнительную информацию см. в git-commit[1] . В частности, если <mode> присвоено значение scissors , ножницы будут добавлены к MERGE_MSG перед передачей дальше в случае конфликта.

-x

При записи фиксации добавьте строку с надписью "(cherry picked from commit …​)" к исходному сообщению о фиксации, чтобы указать, из какой фиксации было выбрано это изменение. Это делается только для выбора вишни без конфликтов. Не используйте эту опцию, если вы выбираете информацию из филиала private , поскольку информация бесполезна для получателя. С другой стороны, если вы выбираете между двумя общедоступными ветвями (например, переносите исправление в ветку обслуживания для более старой версии из ветки разработки), добавление этой информации может оказаться полезным.

-r

Раньше команда по умолчанию делала описанный выше -x , а -r его отключала. Теперь по умолчанию -x не используется, поэтому этот параметр не используется.

-m <parent-number>
--mainline <parent-number>

Обычно вы не можете выбрать слияние заранее, потому что не знаете, какую сторону слияния следует считать основной линией. Эта опция указывает родительский номер (начиная с 1) основной линии и позволяет Cherry-Pick воспроизвести изменение относительно указанного родительского элемента.

-n
--no-commit

Обычно команда автоматически создает последовательность коммитов. Этот флаг применяет изменения, необходимые для выбора каждого именованного коммита в ваше рабочее дерево и индекс, без внесения каких-либо коммитов. Кроме того, при использовании этой опции ваш индекс не обязательно должен соответствовать фиксации HEAD. Выбор вишни выполняется по начальному состоянию вашего индекса.

Это полезно при выборе эффекта более чем одного коммита в ваш индекс подряд.

-s
--signoff

Добавьте трейлер Signed-off-by в конце сообщения о фиксации. Дополнительную информацию см. в разделе «Выписка» в git-commit[1] .

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

GPG-знак фиксируется. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

--ff

Если текущий HEAD совпадает с родительским элементом выбранного коммита, то будет выполнена быстрая перемотка вперед к этому коммиту.

--allow-empty

По умолчанию выбор пустого коммита завершится неудачей, что указывает на то, что требуется явный вызов git commit --allow-empty . Эта опция переопределяет это поведение, позволяя автоматически сохранять пустые коммиты при выборе. Обратите внимание: когда действует "--ff" , пустые фиксации, соответствующие требованию "fast-forward" , будут сохраняться даже без этой опции. Также обратите внимание, что при использовании этой опции сохраняются только те коммиты, которые изначально были пустыми (т. е. коммит записывал то же дерево, что и его родительский элемент). Коммиты, которые стали пустыми из-за предыдущего коммита, удаляются. Чтобы принудительно включить эти коммиты, используйте --keep-redundant-commits .

--allow-empty-message

По умолчанию выбор коммита с пустым сообщением завершится неудачно. Эта опция переопределяет это поведение, позволяя выбирать коммиты с пустыми сообщениями.

--keep-redundant-commits

Если выбранный коммит дублирует коммит, уже находящийся в текущей истории, он станет пустым. По умолчанию эти избыточные фиксации приводят к остановке cherry-pick , чтобы пользователь мог проверить фиксацию. Эта опция переопределяет это поведение и создает пустой объект фиксации. Подразумевается --allow-empty .

--strategy=<strategy>

Используйте данную стратегию слияния. Следует использовать только один раз. Подробности см. в разделе «СТРАТЕГИИ СЛИЯНИЯ» в git-merge[1] .

-X<option>
--strategy-option=<option>

Передайте параметр, специфичный для стратегии слияния, в стратегию слияния. Подробности см. в git-merge[1] .

--rerere-autoupdate
--no-rerere-autoupdate

После того, как механизм rerere повторно использует записанное разрешение текущего конфликта для обновления файлов в рабочем дереве, разрешите ему также обновить индекс с результатом разрешения. --no-rerere-autoupdate — хороший способ перепроверить то, что сделал rerere , и выявить возможные ошибки слияния, прежде чем заносить результат в индекс с помощью отдельного git add .

Sequencer subcommands

--continue

Продолжите выполняемую операцию, используя информацию в .git/sequencer . Может использоваться для продолжения после разрешения конфликтов при неудачном выборе или возврата.

--skip

Пропустите текущую фиксацию и продолжите выполнение остальной части последовательности.

--quit

Забудьте о текущей выполняемой операции. Может использоваться для очистки состояния секвенсора после неудачного выбора вишни или возврата.

--abort

Отмените операцию и вернитесь в состояние до последовательности.

Examples

git cherry-pick master

Примените изменение, внесенное коммитом на кончике главной ветки, и создайте новый коммит с этим изменением.

git cherry-pick ..master
git cherry-pick ^HEAD master

Примените изменения, внесенные всеми коммитами, которые являются предками master, но не HEAD, для создания новых коммитов.

git cherry-pick maint next ^master
git cherry-pick maint master..next

Примените изменения, внесенные всеми коммитами, которые являются предками maint или next, но не master или любого из его предков. Обратите внимание, что последнее не означает maint и все, что находится между master и next ; в частности, maint не будет использоваться, если он включен в состав master .

git cherry-pick master~4 master~2

Примените изменения, внесенные пятым и третьим последним коммитами, указанными мастером, и создайте 2 новых коммита с этими изменениями.

git cherry-pick -n master~1 next

Примените к рабочему дереву и индексу изменения, внесенные вторым последним коммитом, на который указывает мастер, и последним коммитом, на который указывает следующий, но не создавайте никаких коммитов с этими изменениями.

git cherry-pick --ff ..next

Если история линейна и HEAD является предком следующего, обновите рабочее дерево и переместите указатель HEAD, чтобы он соответствовал следующему. В противном случае примените изменения, внесенные теми коммитами, которые находятся в следующей, но не в HEAD, к текущей ветке, создавая новый коммит для каждого нового изменения.

git rev-list --reverse master -- README | git cherry-pick -n --stdin

Примените изменения, внесенные всеми коммитами в основной ветке, которые касались README, к рабочему дереву и индексу, чтобы результат можно было проверить и при необходимости превратить в один новый коммит.

Следующая последовательность пытается выполнить резервное копирование исправления, но выходит из строя, поскольку код, к которому применяется исправление, слишком сильно изменился, а затем пытается снова, на этот раз уделяя больше внимания сопоставлению строк контекста.

$ git cherry-pick topic^             (1)
$ git diff                           (2)
$ git cherry-pick --abort            (3)
$ git cherry-pick -Xpatience topic^  (4)

  1. примените изменение, которое будет показано git show topic^ . В этом примере исправление не применяется полностью, поэтому информация о конфликте записывается в индекс и рабочее дерево, и никаких новых результатов фиксации не происходит.

  2. обобщить изменения, подлежащие согласованию

  3. отменить выбор вишни. Другими словами, вернитесь в состояние до выбора вишни, сохранив все локальные изменения, внесенные в рабочее дерево.

  4. попробуйте применить изменение, внесенное topic^ , еще раз, потратив дополнительное время, чтобы избежать ошибок, связанных с неправильным соответствием строк контекста.
