git-pack-refs

Name

git-pack-refs — заголовки пакетов и теги для эффективного доступа к репозиторию.

Synopsis

git pack-refs [--all] [--no-prune] [--include <pattern>] [--exclude <pattern>]

Description

Традиционно кончики ветвей и теги (известные под общим названием refs ) хранились по одному файлу для каждой ссылки в (под)каталоге в каталоге $GIT_DIR/refs . Хотя многие подсказки ветвей имеют тенденцию часто обновляться, большинство тегов и некоторые подсказки ветвей никогда не обновляются. Когда репозиторий имеет сотни или тысячи тегов, такой формат «один файл на ссылку» приводит к нерациональному использованию памяти и снижению производительности.

Эта команда используется для решения проблем с хранилищем и производительностью путем хранения ссылок в одном файле $GIT_DIR/packed-refs . Если ссылка отсутствует в традиционной иерархии каталогов $GIT_DIR/refs , она ищется в этом файле и используется, если найдена.

Последующие обновления ветвей всегда создают новые файлы в иерархии каталогов $GIT_DIR/refs .

Рекомендуемая практика работы с репозиторием со слишком большим количеством ссылок — один раз упаковать его ссылки с помощью --all и время от времени запускать git pack-refs . Теги по определению стационарны и не должны меняться. Заголовки ветвей будут упакованы исходным pack-refs --all , но распакованными станут только активные в данный момент заголовки ветвей, а следующий pack-refs (без --all ) оставит их распакованными.

Options

--all

По умолчанию команда упаковывает все теги и ссылки, которые уже упакованы, и оставляет другие ссылки без изменений. Это связано с тем, что ожидается активное развитие ветвей, а упаковка их советов не способствует повышению производительности. Эта опция также приводит к упаковке всех ссылок, за исключением скрытых, неработающих и символических ссылок. Полезно для хранилища со многими ветвями исторических интересов.

--no-prune

Команда обычно удаляет свободные ссылки в иерархии $GIT_DIR/refs после их упаковки. Эта опция запрещает это делать.

--include <pattern>

Рефы пакета основаны на паттерне glob(7) . Повторения этого варианта накапливают закономерности включения. Если ссылка включена в --include и --exclude , --exclude имеет приоритет. Использование --include исключит включение всех тегов по умолчанию. Символические и неработающие ссылки никогда не будут упакованы. При использовании с --all это будет пустяк. Используйте --no-include , чтобы очистить и сбросить список шаблонов.

--exclude <pattern>

Не упаковывайте ссылки, соответствующие заданному шаблону glob(7) . При повторении этой опции накапливаются шаблоны исключений. Используйте --no-exclude для очистки и сброса списка шаблонов. Если реф уже запакован, включение его в --exclude не распакует его.

При использовании с --all упаковывайте только отдельные ссылки, которые не соответствуют ни одному из предоставленных шаблонов --exclude .

При использовании с --include будут упакованы ссылки, предоставленные --include , за вычетом ссылок, предоставленных --exclude .

Bugs

В более старой документации, написанной до того, как был введен механизм упакованных ссылок, все еще может быть указано что-то вроде ".git/refs/heads/<branch> file exists" , хотя это означает "branch <branch> exists" .
