gitformat-pack

Name

gitformat-pack — формат пакета Git

Synopsis

$GIT_DIR/objects/pack/pack-.{pack,idx}
$GIT_DIR/objects/pack/pack-.rev
$GIT_DIR/objects/pack/pack-*.mtimes
$GIT_DIR/objects/pack/multi-pack-index

Description

Формат пакета Git — это то, как Git хранит большую часть данных своего основного репозитория. В течение срока существования репозитория отдельные объекты (если таковые имеются) и более мелкие пакеты объединяются в более крупные пакеты. См. git-gc[1] и git-pack-objects[1] .

Формат пакета также используется по беспроводной сети, см., например, gitprotocol-v2[5] , а также является частью других форматов контейнеров в случае gitformat-bundle[5] .

Контрольные суммы и идентификаторы объектов

В репозитории, использующем традиционный SHA-1, контрольные суммы пакетов, контрольные суммы индексов и идентификаторы объектов (имена объектов), упомянутые ниже, вычисляются с использованием SHA-1. Аналогично, в репозиториях SHA-256 эти значения вычисляются с использованием SHA-256.

Файлы Pack-*.pack имеют следующий формат:

  • Заголовок появляется в начале и состоит из следующего:

    4-byte signature:
    The signature is: {'P', 'A', 'C', 'K'}
       4-byte version number (network byte order):
Git currently accepts version number 2 or 3 but
       generates version 2 only.
    4-byte number of objects contained in the pack (network byte order)
    Observation: we cannot have more than 4G versions ;-) and
more than 4G objects in a pack.

  • За заголовком следует несколько записей объектов, каждая из которых выглядит следующим образом:

    (undeltified representation)
n-byte type and length (3-bit type, (n-1)*7+4-bit length)
compressed data
       (deltified representation)
   n-byte type and length (3-bit type, (n-1)*7+4-bit length)
   base object name if OBJ_REF_DELTA or a negative relative
offset from the delta object's position in the pack if this
is an OBJ_OFS_DELTA object
   compressed delta data
    Observation: the length of each object is encoded in a variable
length format and is not constrained to 32-bit or anything.

  • Трейлер записывает контрольную сумму пакета всего вышеперечисленного.

Object types

Допустимые типы объектов:

  • OBJ_COMMIT (1)

  • OBJ_TREE (2)

  • OBJ_BLOB (3)

  • OBJ_TAG (4)

  • OBJ_OFS_DELTA (6)

  • OBJ_REF_DELTA (7)

Тип 5 зарезервирован для будущего расширения. Тип 0 недействителен.

Size encoding

В этом документе используется следующий "size encoding" неотрицательных целых чисел: семь младших битов каждого байта используются для формирования результирующего целого числа. Пока старший бит равен 1, этот процесс продолжается; байт со старшим битом 0 содержит последние семь битов. Семибитные фрагменты объединяются. Более поздние значения более значимы.

Эту кодировку размера не следует путать с "offset encoding" , которая также используется в этом документе.

Deltified representation

Концептуально существует только четыре типа объектов: фиксация, дерево, тег и большой двоичный объект. Однако для экономии места объект можно сохранить как "delta" другого объекта "base" . Этим представлениям назначаются новые типы s-delta и ref-delta, которые действительны только в пакетном файле.

И ofs-delta, и ref-delta сохраняют "delta" для применения к другому объекту (называемому base object ) для его реконструкции. Разница между ними в том, что ref-delta напрямую кодирует имя базового объекта. Если базовый объект находится в том же пакете, ofs-delta вместо этого кодирует смещение базового объекта в пакете.

Базовый объект также может быть разделен, если он находится в том же пакете. Ref-delta также может относиться к объекту вне пакета (т.е. к так называемому "thin pack" ). Однако при хранении на диске пакет должен быть автономным, чтобы избежать циклической зависимости.

Дельта-данные начинаются с размера базового объекта и размера объекта, подлежащего реконструкции. Эти размеры кодируются с использованием приведенной выше кодировки размера. Остальная часть дельта-данных представляет собой последовательность инструкций по восстановлению объекта из базового объекта. Если базовый объект разделен, его необходимо сначала преобразовать в каноническую форму. Каждая инструкция добавляет к целевому объекту все больше и больше данных, пока он не будет завершен. На данный момент поддерживаются две инструкции: одна для копирования диапазона байтов из исходного объекта и одна для вставки новых данных, встроенных в саму инструкцию.

Каждая инструкция имеет переменную длину. Тип инструкции определяется седьмым битом первого октета. Следующие диаграммы соответствуют соглашению RFC 1951 (формат сжатых данных Deflate).

Инструкция по копированию из базового объекта

+----------+---------+---------+---------+---------+-------+-------+-------+
| 1xxxxxxx | offset1 | offset2 | offset3 | offset4 | size1 | size2 | size3 |
+----------+---------+---------+---------+---------+-------+-------+-------+

Это формат инструкции для копирования диапазона байтов из исходного объекта. Он кодирует смещение для копирования и количество байтов для копирования. Смещение и размер указаны в порядке с прямым порядком байтов.

Все байты смещения и размера являются необязательными. Это сделано для уменьшения размера инструкции при кодировании небольших смещений или размеров. Первые семь битов первого октета определяют, какой из следующих семи октетов присутствует. Если установлен нулевой бит, присутствует смещение1. Если установлен первый бит, присутствует смещение 2 и так далее.

Обратите внимание, что более компактная инструкция не меняет кодировку смещения и размера. Например, если опущено только смещение2, как показано ниже, смещение3 все равно будет содержать биты 16–23. Оно не становится смещением 2 и содержит биты 8–15, даже если оно находится рядом со смещением 1.

+----------+---------+---------+
| 10000101 | offset1 | offset3 |
+----------+---------+---------+

В своей наиболее компактной форме эта инструкция занимает только один байт (0x80) без смещения и размера, которые по умолчанию будут иметь нулевое значение. Есть еще одно исключение: нулевой размер автоматически преобразуется в 0x10000.

Инструкция по добавлению новых данных

+----------+============+
| 0xxxxxxx |    data    |
+----------+============+

Это инструкция для создания целевого объекта без базового объекта. Следующие данные добавляются к целевому объекту. Первые семь бит первого октета определяют размер данных в байтах. Размер должен быть ненулевым.

Reserved instruction

+----------+============
| 00000000 |
+----------+============

Эта инструкция зарезервирована для будущего расширения.

Исходные файлы package-*.idx (версия 1) имеют следующий формат:

  • Заголовок состоит из 256 4-байтовых целых чисел сетевого порядка байтов. N-я запись этой таблицы записывает количество объектов в соответствующем пакете, первый байт имени объекта которых меньше или равен N. Это называется таблицей first-level fan-out .

  • За заголовком следуют отсортированные 24-байтовые записи, по одной записи на каждый объект в пакете. Каждая запись представляет собой:

    4-byte network byte order integer, recording where the
object is stored in the packfile as the offset from the
beginning.
    one object name of the appropriate size.

  • Завершает файл трейлер:

    A copy of the pack checksum at the end of the corresponding
packfile.
    Index checksum of all of the above.

Файлы пакета версии 2 — *.idx поддерживают пакеты размером более 4 ГБ и

have some other reorganizations.  They have the format:

  • 4-байтовое магическое число \377tOc , которое представляет собой необоснованное значение разветвления[0].

  • 4-байтовый номер версии (= 2).

  • Разветвленная таблица на 256 записей, как и в версии v1.

  • Таблица отсортированных имен объектов. Они упакованы вместе без значений смещения, чтобы уменьшить объем кэша при двоичном поиске определенного имени объекта.

  • Таблица 4-байтовых значений CRC32 упакованных данных объекта. Это новое в версии 2, поэтому сжатые данные можно копировать непосредственно из пакета в пакет во время переупаковки без незаметного повреждения данных.

  • Таблица 4-байтовых значений смещения (в сетевом порядке байтов). Обычно это 31-битные смещения пакетного файла, но большие смещения кодируются как индекс в следующую таблицу с установленным мсбитом.

  • Таблица 8-байтовых записей смещения (пустая для файлов пакета размером менее 2 ГиБ). Файлы пакетов организованы так, что часто используемые объекты расположены в начале, поэтому большинству ссылок на объекты не обязательно ссылаться на эту таблицу.

  • Тот же трейлер, что и пак-файл v1:

    A copy of the pack checksum at the end of the
corresponding packfile.
    Index checksum of all of the above.

Файлы Pack-*.rev имеют формат:

  • 4-байтовое магическое число 0x52494458 ( RIDX ).

  • 4-байтовый идентификатор версии (= 1).

  • 4-байтовый идентификатор хэш-функции (= 1 для SHA-1, 2 для SHA-256).

  • Таблица позиций индексов (по одному на упакованный объект, всего num_objects, каждая из которых представляет собой 4-байтовое целое число без знака в сетевом порядке), отсортированная по соответствующим смещениям в пакетном файле.

  • Трейлер, содержащий:

    checksum of the corresponding packfile, and
    a checksum of all of the above.

Все 4-байтовые числа расположены в сетевом порядке.

Файлы Pack-*.mtimes имеют формат:

Все 4-байтовые числа расположены в сетевом порядке байтов.

  • 4-байтовое магическое число 0x4d544d45 ( MTME ).

  • 4-байтовый идентификатор версии (= 1).

  • 4-байтовый идентификатор хэш-функции (= 1 для SHA-1, 2 для SHA-256).

  • Таблица 4-байтовых целых чисел без знака. i-е значение представляет собой время модификации (mtime) i-го объекта в соответствующем пакете в лексикографическом (индексном) порядке. mtimes отсчитывают стандартные секунды эпохи.

  • Трейлер, содержащий контрольную сумму соответствующего пак-файла и контрольную сумму всего вышеперечисленного (каждая из которых имеет длину в соответствии с указанной хэш-функцией).

Многоиндексные обратные индексы

Подобно обратному индексу на основе упаковки, индекс нескольких упаковок также можно использовать для создания обратного индекса.

Вместо сопоставления между положением смещения, упаковки и индекса этот обратный индекс сопоставляет позицию объекта в MIDX и позицию этого объекта в псевдопакете, который описывает MIDX (т. е. i-я запись обратного мультипака). индекс содержит позицию MIDX i-го объекта в псевдопакетном порядке).

Чтобы прояснить разницу между этими порядками, рассмотрим растровое изображение доступности нескольких пакетов (которого еще не существует, но именно к этому мы здесь стремимся). Каждый бит должен соответствовать объекту в MIDX, поэтому нам необходимо эффективное сопоставление позиции бита с позицией MIDX.

Одно из решений состоит в том, чтобы позволить битам занимать одну и ту же позицию в индексе, отсортированном по oid, хранящемся в MIDX. Но поскольку оиды фактически случайны, их результирующие растровые изображения достижимости не будут иметь локальности и, следовательно, будут плохо сжиматься. (Это причина того, что растровые изображения с одним пакетом используют для той же цели порядок пакетов, а не порядок .idx.)

Поэтому мы хотели бы определить порядок для всего MIDX, основанный на групповом порядке, который имеет гораздо лучшую локальность (и, следовательно, более эффективно сжимает). Мы можем представить себе псевдопакет, созданный путем объединения всех пакетов в MIDX. Например, если бы у нас был MIDX с тремя пакетами (a, b, c) с 10, 15 и 20 объектами соответственно, мы можем представить себе такой порядок объектов:

|a,0|a,1|...|a,9|b,0|b,1|...|b,14|c,0|c,1|...|c,19|

где порядок пакетов определяется списком пакетов MIDX, а затем порядок объектов внутри каждого пакета такой же, как порядок в фактическом файле пакета.

Учитывая список пакетов и количество объектов в них, вы можете наивно реконструировать этот псевдопакетный порядок (например, объект в позиции 27 должен быть (c,1), поскольку пакеты "a" и "b" заняли 25 слотов). Но есть одна загвоздка. Объекты могут дублироваться между пакетами, и в этом случае MIDX хранит только один указатель на объект (и, следовательно, нам нужен только один слот в растровом изображении).

Вызывающие программы могли бы обрабатывать дубликаты самостоятельно, читая объекты в порядке их битовой позиции, но это линейно по количеству объектов и слишком дорого для обычного поиска по растровым изображениям. Эту проблему решает создание обратного индекса, поскольку он является логическим обратным индексом и этот индекс уже удалил дубликаты. Но построение обратного индекса на лету может оказаться дорогостоящим. Поскольку у нас уже есть формат на диске для пакетных обратных индексов, давайте повторно использовать его и для псевдопакета MIDX.

Объекты из MIDX упорядочиваются следующим образом, чтобы объединить псевдопакет. Пусть pack(o) вернет пакет, из которого MIDX выбрал o , и определит порядок пакетов на основе их числового идентификатора (который хранится в MIDX). Пусть offset(o) вернет смещение объекта o внутри pack(o) . Затем сравните o1 и o2 следующим образом:

  • Если один из pack(o1) и pack(o2) является предпочтительным, а другой нет, то предпочтительный сортируется первым.

    (Это деталь, которая позволяет битовой карте MIDX определить, какой пакет должен использоваться механизмом повторного использования пакета, поскольку он может запросить у MIDX пакет, содержащий объект в битовой позиции 0).

  • Если pack(o1) ≠ pack(o2) , отсортируйте два объекта в порядке убывания на основе идентификатора пакета.

  • В противном случае pack(o1) = pack(o2) и объекты сортируются в порядке упаковки (т. е. o1 сортируется впереди o2 именно тогда, когда offset(o1) < offset(o2) ).

Короче говоря, псевдопакет MIDX — это дедуплицированная конкатенация объектов в пакетах, хранящихся в MIDX, расположенных в порядке пакетов, и пакетов, расположенных в порядке MIDX (при этом предпочтительный пакет идет первым).

Обратный индекс MIDX хранится в дополнительном фрагменте RIDX внутри самого MIDX.

Cruft packs

Функция пакетов Cruft предлагает альтернативу традиционному механизму Git удаления недоступных объектов. В этом документе представлен обзор механизма сокращения Git и того, как вместо него можно использовать пакет Cruft Pack.

Background

Для удаления недоступных объектов из вашего репозитория Git предлагает git repack -Ad (см. git-repack[1] ). Цитата из документации:

[...] unreachable objects in a previous pack become loose, unpacked objects,
instead of being left in the old pack. [...] loose unreachable objects will be
pruned according to normal expiry rules with the next 'git gc' invocation.

Недоступные объекты не удаляются немедленно, так как это может привести к соперничеству с входящим push-уведомлением, которое может ссылаться на объект, который собирается быть удален. Вместо этого эти недоступные объекты сохраняются как незакрепленные объекты и остаются такими до тех пор, пока они не станут старше окна истечения срока действия, после чего они удаляются git-prune[1] .

Git должен хранить эти недоступные объекты отдельно, чтобы отслеживать их время для каждого объекта. Если бы эти недоступные объекты были записаны в один большой пакет, то либо обновление этого пакета (поскольку содержащийся в нем объект был перезаписан), либо создание нового пакета недоступных объектов привело бы к обновлению времени пакета и объектов внутри него. никогда не покинет окно истечения срока действия. Вместо этого объекты хранятся отдельно, чтобы отслеживать время каждого отдельного объекта и избегать ситуации, когда все ненужные объекты обновляются одновременно.

Это может привести к нежелательным ситуациям, когда в репозитории содержится много недоступных объектов, которые еще не вышли из льготного периода. Наличие больших каталогов в сегментах .git/objects может привести к снижению производительности репозитория. Но при наличии большого количества недоступных объектов это может привести к нехватке индексных дескрипторов и снижению производительности всей системы. Поскольку мы никогда не сможем упаковать эти объекты, эти репозитории часто занимают большой объем дискового пространства, поскольку мы можем только сжимать их с помощью zlib, но не хранить в дельта-цепочках.

Cruft packs

Пакет Cruft устраняет необходимость хранения недоступных объектов в незакрепленном состоянии, включая mtime для каждого объекта в отдельный файл вместе с одним пакетом, содержащим все незакрепленные объекты.

Крафт-пак записывается git repack --cruft при создании нового пакета. Вариант --cruft для git-pack-objects[1] . Обратите внимание, что git repack --cruft — это классический репак «все в одном», а это означает, что все в полученном пакете доступно, а все остальное — недоступно. После написания опция --cruft предписывает git repack сгенерировать еще один пакет, содержащий только объекты, не упакованные на предыдущем шаге (что соответствует упаковке всех недоступных объектов вместе). Это происходит следующим образом:

  1. Перечислите каждый объект, отметив любой объект, который (а) не содержится в сохраненном пакете и (б) чье время находится в пределах льготного периода, в качестве подсказки по обходу.

  2. Выполните обход достижимости на основе советов, собранных на предыдущем шаге, добавляя в пакет каждый объект на этом пути.

  3. Запишите пакет вместе с файлом .mtimes , в котором записаны временные метки для каждого объекта.

Этот режим вызывается внутренне git-repack[1] , когда ему дано указание написать пакет Cruft. Важно отметить, что набор пакетов, хранящихся в ядре, — это именно тот набор пакетов, который не будет удален при перепаковке; другими словами, они содержат все доступные объекты репозитория.

Если в репозитории уже есть пакет Cruft Pack, git repack --cruft обычно только добавляет в него объекты. Исключением является случай, когда git repack предоставляется опция --cruft-expiration , которая позволяет сгенерированному пакету пропускать объекты с истекшим сроком действия вместо того, чтобы ждать, пока git-gc[1] истечет срок действия этих объектов позже.

За удаление просроченных недоступных объектов обычно отвечает git-gc[1] .

Alternatives

Известные альтернативы этой конструкции включают:

  • Расположение данных mtime для каждого объекта.

По расположению данных mtime был выбран новый вспомогательный файл, привязанный к паку, чтобы не усложнять формат .idx . Если бы формат .idx когда-либо получил поддержку дополнительных фрагментов данных, возможно, имело бы смысл объединить формат .mtimes в сам .idx .
