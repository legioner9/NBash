git-branch

Name

git-branch — список, создание или удаление ветвей

Synopsis

git branch [--color[=<when>] | --no-color] [--show-current]
        [-v [--abbrev=<n> | --no-abbrev]]
        [--column[=<options>] | --no-column] [--sort=<key>]
        [--merged [<commit>]] [--no-merged [<commit>]]
        [--contains [<commit>]] [--no-contains [<commit>]]
        [--points-at <object>] [--format=<format>]
        [(-r | --remotes) | (-a | --all)]
        [--list] [<pattern>…​]
git branch [--track[=(direct|inherit)] | --no-track] [-f]
        [--recurse-submodules] <branchname> [<start-point>]
git branch (--set-upstream-to=<upstream> | -u <upstream>) [<branchname>]
git branch --unset-upstream [<branchname>]
git branch (-m | -M) [<oldbranch>] <newbranch>
git branch (-c | -C) [<oldbranch>] <newbranch>
git branch (-d | -D) [-r] <branchname>…​
git branch --edit-description [<branchname>]

Description

Если задан --list или нет аргументов, не являющихся опциями, отображаются существующие ветки; текущая ветка будет выделена зеленым цветом и отмечена звездочкой. Любые ветки, извлеченные в связанных рабочих деревьях, будут выделены голубым цветом и отмечены знаком плюс. Опция -r приводит к отображению ветвей удаленного отслеживания, а опция -a показывает как локальные, так и удаленные ветки.

Если указан <pattern> , он используется в качестве подстановочного знака оболочки, чтобы ограничить вывод соответствующими ветвями. Если задано несколько шаблонов, ветвь отображается, если она соответствует какому-либо из шаблонов.

Обратите внимание, что при предоставлении <pattern> необходимо использовать --list ; в противном случае команду можно интерпретировать как создание ветки.

При использовании --contains отображаются только ветки, содержащие именованный коммит (другими словами, ветки, чьи верхние коммиты являются потомками именованного коммита), --no-contains инвертирует его. При использовании --merged будут перечислены только ветки, объединенные в именованный коммит (т. е. ветки, чьи верхние коммиты доступны из именованного коммита). В --no-merged будут перечислены только ветки, не объединенные в указанный коммит. Если аргумент <commit> отсутствует, по умолчанию используется значение HEAD (т. е. кончик текущей ветки).

Вторая форма команды создает новый заголовок ветки с именем <branchname>, который указывает на текущий HEAD или <start-point>, если он указан. В частном случае для <start-point> вы можете использовать "A...B" в качестве ярлыка для базы слияния A и B , если существует ровно одна база слияния. Вы можете исключить не более одного из A и B , в этом случае по умолчанию используется HEAD .

Обратите внимание, что это создаст новую ветку, но не переключит на нее рабочее дерево; используйте "git switch <newbranch>" для переключения на новую ветку.

Когда локальная ветвь запускается из ветки удаленного отслеживания, Git настраивает ветвь (в частности, записи конфигурации branch.<name>.remote и branch.<name>.merge ) так, чтобы git pull соответствующим образом объединялся с веткой удаленного отслеживания. Это поведение можно изменить с помощью глобального флага конфигурации branch.autoSetupMerge . Эту настройку можно переопределить с помощью параметров --track и --no-track , а затем изменить с помощью git branch --set-upstream-to .

При использовании опции -m или -M <oldbranch> будет переименован в <newbranch>. Если <oldbranch> имеет соответствующий журнал обновлений, он переименовывается в соответствии с <newbranch>, и создается запись в журнале обновлений, чтобы запомнить переименование ветки. Если <newbranch> существует, необходимо использовать -M, чтобы принудительно выполнить переименование.

Параметры -c и -C имеют ту же семантику, что и -m и -M , за исключением того, что вместо переименования ветки она будет скопирована под новое имя вместе с ее конфигурацией и журналом обновления.

При использовании опции -d или -D <branchname> будет удален. Вы можете указать более одной ветки для удаления. Если ветка в настоящее время имеет журнал обновлений, он также будет удален.

Используйте -r вместе с -d для удаления ветвей удаленного отслеживания. Обратите внимание, что имеет смысл удалять ветки удаленного отслеживания только в том случае, если они больше не существуют в удаленном репозитории или если git fetch настроен так, чтобы не получать их снова. См. также подкоманду prune команды git-remote[1] , чтобы узнать, как очистить все устаревшие ветки удаленного отслеживания.

Options

-d
--delete

Удалить ветку. Ветка должна быть полностью объединена со своей восходящей ветвью или с HEAD , если восходящая ветвь не была установлена ​​с помощью --track или --set-upstream-to .

-D

Ярлык для --delete --force .

--create-reflog

Создайте рефлог ветки. Это активирует запись всех изменений, внесенных в ссылку ветки, что позволяет использовать выражения sha1 на основе даты, такие как "<branchname>@{yesterday}" . Обратите внимание, что в непустых репозиториях журналы обновлений обычно включаются по умолчанию с помощью параметра конфигурации core.logAllRefUpdates . Отрицательная форма --no-create-reflog только переопределяет более раннюю версию --create-reflog , но в настоящее время не отменяет настройку core.logAllRefUpdates .

-f
--force

Сбросьте <имя ветки> до <начальной точки>, даже если <имя ветки> уже существует. Без -f git branch отказывается изменять существующую ветку. В сочетании с -d (или --delete ) позволяет удалять ветку независимо от ее статуса объединения или даже от того, указывает ли она на действительную фиксацию. В сочетании с -m (или --move ) разрешите переименование ветки, даже если новое имя ветки уже существует. То же самое относится и к -c (или --copy ).

Обратите внимание, что git branch -f <branchname> [<start-point>] , даже с -f , отказывается изменять существующую ветвь <branchname> , которая извлечена в другом рабочем дереве, связанном с тем же репозиторием.

-m
--move

Переместите/переименуйте ветку вместе с ее конфигурацией и обновите журнал.

-M

Ярлык для --move --force .

-c
--copy

Скопируйте ветку вместе с ее конфигурацией и обновите журнал.

-C

Ярлык для --copy --force .

--color[=<when>]

Цветные ветки, чтобы выделить текущие, локальные и удаленные ветки. Значение должно быть всегда (по умолчанию), никогда или автоматически.

--no-color

Отключите цвета ветвей, даже если в файле конфигурации по умолчанию указан цветной вывод. То же, что --color=never .

-i
--ignore-case

Ветви сортировки и фильтрации не чувствительны к регистру.

--omit-empty

Не печатайте новую строку после форматированных ссылок, где формат расширяется до пустой строки.

--column[=<options>]
--no-column

Отображение списка филиалов в столбцах. Синтаксис опции см. в переменной конфигурации column.branch . --column и --no-column без дополнительных опций эквивалентны always и never соответственно.

Эта опция применима только в неподробном режиме.

-r
--remotes

Перечислите или удалите (если используется с -d) ветки удаленного отслеживания. Комбинируйте с --list , чтобы соответствовать дополнительному рисунку(ям).

-a
--all

Перечислите как ветки удаленного отслеживания, так и локальные ветки. Комбинируйте с --list для получения дополнительных рисунков.

-l
--list

Перечислите ветки. При использовании дополнительного <pattern>... , например git branch --list 'maint-*' , перечислите только те ветви, которые соответствуют шаблону(ам).

--show-current

Выведите имя текущей ветки. В состоянии отсоединенного HEAD ничего не печатается.

-v
-vv
--verbose

В режиме списка покажите sha1 и зафиксируйте строку темы для каждой главы, а также связь с восходящей веткой (если таковая имеется). Если задано дважды, выведите путь к связанному рабочему дереву (если есть) и имя восходящей ветки (см. также git remote show <remote> ). Обратите внимание, что в HEAD текущего рабочего дерева не будет указан путь (это всегда будет ваш текущий каталог).

-q
--quiet

Будьте тише при создании или удалении ветки, подавляя сообщения об ошибках.

--abbrev=<n>

В подробном списке, показывающем имя объекта фиксации, укажите кратчайший префикс длиной не менее шестнадцатеричных цифр <n> , который однозначно ссылается на объект. Значение по умолчанию — 7, и его можно переопределить с помощью параметра конфигурации core.abbrev .

--no-abbrev

Отобразите полные sha1 в выходном листинге, а не сокращайте их.

-t
--track[=(direct|inherit)]

При создании новой ветки настройте записи конфигурации branch.<name>.remote и branch.<name>.merge , чтобы настроить конфигурацию отслеживания "upstream" для новой ветки. Эта конфигурация заставит git показать связь между двумя ветвями в git status и git branch -v . Более того, он предписывает git pull без аргументов извлечь данные из восходящего потока при извлечении новой ветки.

Точная восходящая ветвь выбирается в зависимости от необязательного аргумента: -t , --track или --track=direct означает использование самой ветви начальной точки в качестве восходящей; --track=inherit означает копирование восходящей конфигурации ветви начальной точки.

Переменная конфигурации Branch.autoSetupMerge определяет, как должны вести себя git switch , git checkout и git branch , если не указаны ни --track , ни --no-track :

Опция по умолчанию true ведет себя так, как будто --track=direct задается всякий раз, когда начальной точкой является ветвь удаленного отслеживания. false ведет себя так, как если бы --no-track был задан. always ведет себя так, как если бы был задан --track=direct . inherit ведет себя так, как если бы --track=inherit был задан. simple ведет себя так, как если бы --track=direct был задан только в том случае, если начальной точкой является ветвь удаленного отслеживания, а новая ветвь имеет то же имя, что и удаленная ветвь.

См. git-pull[1] и git-config[1] для получения дополнительной информации о том, как используются опции branch.<name>.remote и branch.<name>.merge .

--no-track

Не настраивайте конфигурацию "upstream" , даже если установлена ​​переменная конфигурации Branch.autoSetupMerge.

--recurse-submodules

ЭТОТ ВАРИАНТ ЭКСПЕРИМЕНТАЛЬНЫЙ! Заставляет текущую команду рекурсивно переходить к подмодулям, если submodule.propagateBranches включен. См. submodule.propagateBranches в git-config[1] . В настоящее время поддерживается только создание веток.

При использовании при создании ветки новая ветвь <имя ветки> будет создана в суперпроекте и всех подмодулях в <начальной точке> суперпроекта. В подмодулях ветвь будет указывать на фиксацию подмодуля в <start-point> суперпроекта, но информация отслеживания ветки будет настроена на основе ветвей и удаленных модулей подмодуля, например, git branch --recurse-submodules topic origin/main создаст ветвь подмодуля "topic" , которая указывает на фиксацию подмодуля в "origin/main" суперпроекта, но отслеживает "origin/main" субмодуля.

--set-upstream

Поскольку у этой опции запутанный синтаксис, она больше не поддерживается. Вместо этого используйте --track или --set-upstream-to .

-u <upstream>
--set-upstream-to=<upstream>

Настройте информацию отслеживания <branchname>, чтобы <upstream> считался восходящей ветвью <branchname>. Если <имя_ветви> не указано, то по умолчанию используется текущая ветвь.

--unset-upstream

Удалите исходную информацию для <имя ветки>. Если ветка не указана, по умолчанию используется текущая ветка.

--edit-description

Откройте редактор и отредактируйте текст, чтобы объяснить, для чего предназначена эта ветвь, чтобы ее можно было использовать различными другими командами (например, format-patch , request-pull и merge (если включено)). Могут использоваться многострочные пояснения.

--contains [<commit>]

Отобразить только те ветки, которые содержат указанный коммит (HEAD, если не указан). Подразумевается --list .

--no-contains [<commit>]

Перечислите только ветки, которые не содержат указанный коммит (HEAD, если не указан). Подразумевается --list .

--merged [<commit>]

Перечислите только ветки, кончики которых доступны из указанного коммита (HEAD, если не указано). Подразумевается --list .

--no-merged [<commit>]

Перечислите только ветки, кончики которых недоступны из указанного коммита (HEAD, если не указано). Подразумевается --list .

<branchname>

Имя ветки, которую нужно создать или удалить. Новое имя ветки должно пройти все проверки, определенные git-check-ref-format[1] . Некоторые из этих проверок могут ограничивать количество символов, разрешенных в имени ветки.

<start-point>

Новый руководитель ветки укажет на этот коммит. Его можно указать в виде имени ветки, идентификатора фиксации или тега. Если этот параметр опущен, вместо него будет использоваться текущий HEAD.

<oldbranch>

Имя существующей ветки, которую нужно переименовать.

<newbranch>

Новое имя существующей ветки. Применяются те же ограничения, что и для <branchname>.

--sort=<key>

Сортировка по указанному ключу. Префикс - для сортировки в порядке убывания значения. Вы можете использовать опцию --sort=<key> несколько раз, и в этом случае последний ключ становится первичным ключом. Поддерживаемые клавиши такие же, как и в git for-each-ref . Порядок сортировки по умолчанию соответствует значению, настроенному для переменной branch.sort , если она существует, или сортировке на основе полного имени ссылки (включая префикс refs/... ). Сначала перечисляются отсоединенные HEAD (если они есть), затем локальные ветки и, наконец, ветки удаленного отслеживания. См. git-config[1] .

--points-at <object>

Только список ветвей данного объекта.

--format <format>

Строка, которая интерполирует %(fieldname) из отображаемой ссылки на ветку и объекта, на который она указывает. Формат такой же, как у git-for-each-ref[1] .

Configuration

pager.branch учитывается только при перечислении ветвей, т. е. когда --list используется или подразумевается. По умолчанию используется пейджер. См. git-config[1] .

Все, что находится выше этой строки в этом разделе, не включено в документацию git-config[1] . Следующее содержимое такое же, как и найденное там:

branch.autoSetupMerge

Сообщает git branch , git switch и git checkout о необходимости создания новых ветвей, чтобы git-pull[1] соответствующим образом объединился с веткой начальной точки. Обратите внимание, что даже если этот параметр не установлен, такое поведение можно выбрать для каждой ветви с помощью параметров --track и --no-track . Допустимые настройки: false — автоматическая настройка не производится; true — автоматическая настройка производится, когда отправной точкой является ветка удаленного сопровождения; always — ​​автоматическая настройка выполняется, когда отправной точкой является локальная ветка или ветка удаленного отслеживания; inherit — если у начальной точки есть конфигурация отслеживания, она копируется в новую ветку; simple — автоматическая настройка выполняется только в том случае, если начальной точкой является ветка удаленного отслеживания, а новая ветка имеет то же имя, что и удаленная ветка. По умолчанию этот параметр имеет значение true.

branch.autoSetupRebase

Когда с помощью git branch , git switch или git checkout создается новая ветвь, которая отслеживает другую ветвь, эта переменная сообщает Git настроить pull для перебазирования вместо слияния (см. "branch.<name>.rebase" ). При использовании never для rebase никогда автоматически не устанавливается значение true . Если local , для отслеживаемых ветвей других локальных ветвей устанавливается значение rebase true . При remote для rebase устанавливается значение true для отслеживаемых ветвей ветвей с удаленным отслеживанием. При always для всех ветвей отслеживания будет установлено значение true . См. "branch.autoSetupMerge" для получения подробной информации о том, как настроить ветку для отслеживания другой ветки. По умолчанию для этого параметра установлено значение «никогда».

branch.sort

Эта переменная управляет порядком сортировки ветвей при отображении git-branch[1] . Без предоставленной опции "--sort=<value>" значение этой переменной будет использоваться по умолчанию. Допустимые значения см. в именах полей git-for-each-ref[1] .

branch.<name>.remote

Находясь в ветке <имя>, он сообщает git fetch и git push , с какого удаленного устройства следует получить или передать данные. Пульт дистанционного управления, на который осуществляется нажатие, может быть переопределен с помощью remote.pushDefault (для всех филиалов). Пульт дистанционного управления для текущей ветки может быть дополнительно переопределен branch.<name>.pushRemote . Если удаленный доступ не настроен или если вы не находитесь ни в одной ветке и в репозитории определено более одного удаленного устройства, по умолчанию используется origin для выборки и remote.pushDefault для отправки. Кроме того, . (точка) является текущим локальным репозиторием (точечным репозиторием), см. последнее примечание branch.<name>.merge ниже.

branch.<name>.pushRemote

В ветке <имя> он переопределяет branch.<name>.remote для отправки. Он также переопределяет remote.pushDefault для отправки из ветки <имя>. Когда вы извлекаете данные из одного места (например, из восходящего потока) и отправляете их в другое место (например, из собственного репозитория публикации), вам нужно настроить remote.pushDefault , чтобы указать удаленный компьютер для отправки для всех ветвей, и использовать эту опцию, чтобы переопределить ее для конкретная ветка.

branch.<name>.merge

Определяет вместе с Branch.<name>.remote вышестоящую ветвь для данной ветки. Он сообщает git fetch / git pull / git rebase , какую ветку объединять, а также может влиять на git push (см. push.default). Находясь в ветке <имя>, он сообщает git fetch спецификацию ссылки по умолчанию, которая должна быть помечена для слияния в FETCH_HEAD. Значение обрабатывается как удаленная часть спецификации ссылки и должно соответствовать ссылке, полученной с удаленного устройства, заданного "branch.<name>.remote" . Информация о слиянии используется git pull (который сначала вызывает git fetch ) для поиска ветки по умолчанию для слияния. Без этой опции git pull по умолчанию объединяет первую полученную спецификацию ссылки. Укажите несколько значений, чтобы получить слияние осьминога. Если вы хотите настроить git pull так, чтобы он объединялся с <name> из другой ветки в локальном репозитории, вы можете указать ветку.<name>.merge на нужную ветку и использовать настройку относительного пути . (точка) для ветки. .<имя>.удаленный.

branch.<name>.mergeOptions

Устанавливает параметры по умолчанию для слияния с веткой <имя>. Синтаксис и поддерживаемые параметры такие же, как у git-merge[1] , но значения параметров, содержащие символы пробелов, в настоящее время не поддерживаются.

branch.<name>.rebase

Если установлено значение true, перебазируйте ветку <name> поверх выбранной ветки вместо объединения ветки по умолчанию с удаленной веткой по умолчанию при запуске "git pull" . См. "pull.rebase" , чтобы сделать это не зависящим от отрасли способом.

Если merges (или просто m ), передайте параметр --rebase-merges в git rebase , чтобы локальные фиксации слияния были включены в перебазирование (подробности см. в git-rebase[1] ).

Если значение равно interactive (или просто i ), перебазирование выполняется в интерактивном режиме.

ПРИМЕЧАНИЕ. Это потенциально опасная операция; не используйте его, если не понимаете последствий (подробности см. в git-rebase[1] ).

branch.<name>.description

Описание филиала можно редактировать с помощью git branch --edit-description . Описание ветки автоматически добавляется в сопроводительное письмо к исправлению формата или в сводку запроса.

Examples

Начать разработку с известного тега
$ git clone git://git.kernel.org/pub/scm/.../linux-2.6 my2.6
$ cd my2.6
$ git branch my2.6.14 v2.6.14   (1)
$ git switch my2.6.14

  1. Этот и следующий шаг можно объединить в один шаг с помощью "checkout -b my2.6.14 v2.6.14" .

Удалить ненужную ветку
$ git clone git://git.kernel.org/.../git.git my.git
$ cd my.git
$ git branch -d -r origin/todo origin/html origin/man   (1)
$ git branch -D test                                    (2)

  1. Удалите ветки удаленного отслеживания "todo" , "html" и "man" . Следующий fetch или pull создаст их снова, если вы не настроите их не делать этого. См. git-fetch[1] .

  2. Удалите ветку "test" , даже если ветка "master" (или любая другая ветка, извлеченная в данный момент) не имеет всех коммитов из тестовой ветки.

Перечисление ветвей с определенного пульта
$ git branch -r -l '<remote>/<pattern>'                 (1)
$ git for-each-ref 'refs/remotes/<remote>/<pattern>'    (2)

  1. Использование -a приведет к объединению <remote> с любыми локальными ветвями, которые имеют тот же префикс <remote>.

  2. for-each-ref может иметь широкий спектр опций. См. git-for-each-ref[1] .

Шаблоны обычно требуют цитирования.

Notes

Если вы создаете ветку, на которую хотите немедленно переключиться, проще использовать команду "git switch" с ее опцией -c , чтобы сделать то же самое с помощью одной команды.

Опции --contains , --no-contains , --merged и --no-merged служат четырем связанным, но различным целям:

  • --contains <commit> используется для поиска всех ветвей, которые потребуют особого внимания, если <commit> будет перебазирован или изменен, поскольку эти ветки содержат указанный <commit>.

  • --no-contains <commit> является обратным этому, то есть ветвями, которые не содержат указанный <commit>.

  • --merged используется для поиска всех ветвей, которые можно безопасно удалить, поскольку эти ветки полностью содержатся в HEAD.

  • --no-merged используется для поиска ветвей, которые являются кандидатами на слияние с HEAD, поскольку эти ветки не полностью содержатся в HEAD.

При объединении нескольких фильтров --contains и --no-contains отображаются только ссылки, которые содержат хотя бы один из коммитов --contains и содержат none из коммитов --no-contains .

При объединении нескольких фильтров --merged и --no-merged отображаются только ссылки, доступные хотя бы из одного из коммитов --merged и из none из коммитов --no-merged .
