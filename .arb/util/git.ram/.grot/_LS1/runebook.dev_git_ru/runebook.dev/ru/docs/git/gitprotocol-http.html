gitprotocol-http

Name

gitprotocol-http — Git HTTP-протоколы

Synopsis

<over-the-wire-protocol>

Description

Git поддерживает два протокола передачи данных на основе HTTP. Протокол "dumb" , для которого требуется только стандартный HTTP-сервер на стороне сервера соединения, и протокол "smart" , для которого требуется CGI (или серверный модуль) с поддержкой Git . В этом документе описаны оба протокола.

В качестве конструктивной особенности интеллектуальные клиенты могут автоматически обновлять URL-адреса протокола "dumb" до интеллектуальных URL-адресов. Это позволяет всем пользователям иметь один и тот же опубликованный URL-адрес, а узлы автоматически выбирают наиболее эффективный доступный им транспорт.

Url format

URL-адреса для репозиториев Git , к которым осуществляется доступ по HTTP, используют стандартный синтаксис URL-адресов HTTP, описанный в RFC 1738, поэтому они имеют вид:

http://<host>:<port>/<path>?<searchpart>

В этой документации заполнитель $GIT_URL будет обозначать URL-адрес репозитория http://, введенный конечным пользователем.

Серверам СЛЕДУЕТ обрабатывать все запросы к местоположениям, соответствующим $GIT_URL , поскольку протоколы HTTP "smart" и "dumb" , используемые Git , работают путем добавления дополнительных компонентов пути в конец предоставленной пользователем строки $GIT_URL .

Пример тупого клиента, запрашивающего свободный объект:

$GIT_URL:     http://example.com:8080/git/repo.git
URL request:  http://example.com:8080/git/repo.git/objects/d0/49f6c27a2244e12041955e262a404c7faba355

Пример интеллектуального запроса к универсальному шлюзу:

$GIT_URL:     http://example.com/daemon.cgi?svc=git&q=
URL request:  http://example.com/daemon.cgi?svc=git&q=/info/refs&service=git-receive-pack

Пример запроса к подмодулю:

$GIT_URL:     http://example.com/git/repo.git/path/submodule.git
URL request:  http://example.com/git/repo.git/path/submodule.git/info/refs

Клиенты ДОЛЖНЫ удалить завершающий / , если он присутствует, из предоставленной пользователем строки $GIT_URL , чтобы предотвратить появление пустых токенов пути ( // ) в любом URL-адресе, отправляемом на сервер. Совместимые клиенты ДОЛЖНЫ расширять $GIT_URL/info/refs как foo/info/refs , а не foo//info/refs .

Authentication

Стандартная аутентификация HTTP используется, если аутентификация требуется для доступа к хранилищу, и МОЖЕТ быть настроена и применена программным обеспечением HTTP-сервера.

Поскольку доступ к репозиториям Git осуществляется по стандартному пути, администраторы серверов МОГУТ использовать разрешения на основе каталогов на своем HTTP-сервере для управления доступом к репозиторию.

Клиентам СЛЕДУЕТ поддерживать базовую аутентификацию, как описано в RFC 2617. Серверам СЛЕДУЕТ поддерживать базовую аутентификацию, полагаясь на HTTP-сервер, расположенный перед серверным программным обеспечением Git .

Серверам НЕ ДОЛЖНО требоваться использование файлов cookie HTTP для целей аутентификации или контроля доступа.

Клиенты и серверы МОГУТ поддерживать другие распространенные формы аутентификации на основе HTTP, такие как дайджест-аутентификация.

Ssl

Клиенты и серверы ДОЛЖНЫ поддерживать SSL, особенно для защиты паролей при использовании базовой HTTP-аутентификации.

Session state

Протокол Git over HTTP (как и сам HTTP) не имеет состояния с точки зрения стороны HTTP-сервера. Все состояние ДОЛЖНО сохраняться и управляться клиентским процессом. Это позволяет выполнять простую циклическую балансировку нагрузки на стороне сервера, не беспокоясь об управлении состоянием.

Клиенты НЕ ДОЛЖНЫ требовать управления состоянием на стороне сервера для правильной работы.

Серверы НЕ ДОЛЖНЫ требовать HTTP-куки для правильной работы. Клиенты МОГУТ хранить и пересылать файлы cookie HTTP во время обработки запроса, как описано в RFC 2616 (HTTP/1.1). Серверам СЛЕДУЕТ игнорировать любые файлы cookie, отправленные клиентом.

Общая обработка запросов

Если не указано иное, все стандартное поведение HTTP ДОЛЖНО быть принято как клиентом, так и сервером. Это включает (но не обязательно ограничивается):

Если в $GIT_URL нет репозитория или ресурс, на который указывает местоположение, соответствующее $GIT_URL , не существует, сервер НЕ ДОЛЖЕН отвечать ответом 200 OK . Сервер ДОЛЖЕН ответить 404 Not Found , 410 Gone или любым другим подходящим кодом состояния HTTP, который не означает, что ресурс существует по запросу.

Если по адресу $GIT_URL существует репозиторий, но доступ в настоящее время не разрешен, сервер ДОЛЖЕН ответить кодом состояния HTTP 403 Forbidden .

Серверы ДОЛЖНЫ поддерживать как HTTP 1.0, так и HTTP 1.1. Серверы ДОЛЖНЫ поддерживать фрагментированное кодирование как для тела запроса, так и для тела ответа.

Клиенты ДОЛЖНЫ поддерживать как HTTP 1.0, так и HTTP 1.1. Клиентам СЛЕДУЕТ поддерживать фрагментированное кодирование как для тела запроса, так и для тела ответа.

Серверы МОГУТ возвращать заголовки ETag и/или Last-Modified.

Клиенты МОГУТ повторно проверять кэшированные объекты, включая заголовки запроса If-Modified-Since и/или If-None-Match.

Серверы МОГУТ вернуть 304 Not Modified , если соответствующие заголовки присутствуют в запросе и объект не изменился. Клиенты ДОЛЖНЫ обрабатывать 304 Not Modified идентично 200 OK , повторно используя кэшированный объект.

Клиенты МОГУТ повторно использовать кэшированный объект без повторной проверки, если заголовок Cache-Control и/или Expires разрешает кэширование. Клиенты и серверы ДОЛЖНЫ следовать RFC 2616 для управления кэшем.

Discovering references

Все HTTP-клиенты ДОЛЖНЫ начать либо выборку, либо push-обмен, обнаружив ссылки, доступные в удаленном репозитории.

Dumb Clients

Клиенты HTTP, поддерживающие только протокол "dumb" , ДОЛЖНЫ обнаруживать ссылки, отправляя запрос на специальный файл info/refs репозитория.

Глупые HTTP-клиенты ДОЛЖНЫ отправить запрос GET на $GIT_URL/info/refs без каких-либо параметров поиска/запроса.

C: GET $GIT_URL/info/refs HTTP/1.0
S: 200 OK
S:
S: 95dcfa3633004da0049d3d0fa03f80589cbcaf31        refs/heads/maint
S: d049f6c27a2244e12041955e262a404c7faba355        refs/heads/master
S: 2cb58b79488a98d2721cea644875a8dd0026b115        refs/tags/v1.0
S: a3c2e2402b99163d1d59756e5f207ae21cccba4c        refs/tags/v1.0^{}

Content-Type возвращаемого объекта info/refs ДОЛЖЕН быть text/plain; charset=utf-8 , но МОЖЕТ быть любым типом контента. Клиенты НЕ ДОЛЖНЫ пытаться проверить возвращенный Content-Type. Глупые серверы НЕ ДОЛЖНЫ возвращать тип возвращаемого значения, начинающийся с application/x-git- .

Заголовки Cache-Control МОГУТ быть возвращены, чтобы отключить кеширование возвращаемого объекта.

При проверке ответа клиентам СЛЕДУЕТ проверять только код состояния HTTP. Допустимые ответы: 200 OK или 304 Not Modified .

Возвращенное содержимое представляет собой текстовый файл в формате UNIX, описывающий каждую ссылку и ее известное значение. Файл ДОЛЖЕН быть отсортирован по имени в соответствии с порядком локали C. Файл НЕ ДОЛЖЕН включать ссылку по умолчанию с именем HEAD .

info_refs   =  *( ref_record )
ref_record  =  any_ref / peeled_ref
any_ref     =  obj-id HTAB refname LF
peeled_ref  =  obj-id HTAB refname LF
 obj-id HTAB refname "^{}" LF

Smart Clients

Клиенты HTTP, поддерживающие протокол "smart" (или оба протокола "smart" и "dumb" ), ДОЛЖНЫ обнаруживать ссылки, делая параметризованный запрос к файлу info/refs репозитория.

Запрос ДОЛЖЕН содержать ровно один параметр запроса, service=$servicename , где $servicename ДОЛЖЕН быть именем службы, к которой клиент желает обратиться для завершения операции. Запрос НЕ ДОЛЖЕН содержать дополнительные параметры запроса.

C: GET $GIT_URL/info/refs?service=git-upload-pack HTTP/1.0

тупой ответ сервера:

S: 200 OK
S:
S: 95dcfa3633004da0049d3d0fa03f80589cbcaf31        refs/heads/maint
S: d049f6c27a2244e12041955e262a404c7faba355        refs/heads/master
S: 2cb58b79488a98d2721cea644875a8dd0026b115        refs/tags/v1.0
S: a3c2e2402b99163d1d59756e5f207ae21cccba4c        refs/tags/v1.0^{}

умный ответ сервера:

S: 200 OK
S: Content-Type: application/x-git-upload-pack-advertisement
S: Cache-Control: no-cache
S:
S: 001e# service=git-upload-pack\n
S: 0000
S: 004895dcfa3633004da0049d3d0fa03f80589cbcaf31 refs/heads/maint\0multi_ack\n
S: 003fd049f6c27a2244e12041955e262a404c7faba355 refs/heads/master\n
S: 003c2cb58b79488a98d2721cea644875a8dd0026b115 refs/tags/v1.0\n
S: 003fa3c2e2402b99163d1d59756e5f207ae21cccba4c refs/tags/v1.0^{}\n
S: 0000

Клиент может отправлять дополнительные параметры (см. gitprotocol-pack[5] ) в виде строки, разделенной двоеточиями, в HTTP-заголовке Git-Protocol.

Использует опцию --http-backend-info-refs для git-upload-pack[1] .

Тупой ответ сервера

Глупые серверы ДОЛЖНЫ отвечать в формате ответа тупого сервера.

Более подробное описание ответа тупого сервера см. в предыдущем разделе, посвященном тупым клиентам.

Умный ответ сервера

Если сервер не распознает запрошенное имя службы или запрошенное имя службы отключено администратором сервера, сервер ДОЛЖЕН ответить кодом состояния HTTP 403 Forbidden .

В противном случае интеллектуальные серверы ДОЛЖНЫ ответить в формате ответа интеллектуального сервера для запрошенного имени службы.

Заголовки Cache-Control СЛЕДУЕТ использовать для отключения кэширования возвращаемого объекта.

Тип контента ДОЛЖЕН быть application/x-$servicename-advertisement . Клиенты ДОЛЖНЫ вернуться к тупому протоколу, если возвращается другой тип контента. При возврате к тупому протоколу клиенты НЕ ДОЛЖНЫ делать дополнительный запрос к $GIT_URL/info/refs , а вместо этого ДОЛЖНЫ использовать уже имеющийся ответ. Клиенты НЕ ДОЛЖНЫ продолжать работу, если они не поддерживают тупой протокол.

Клиенты ДОЛЖНЫ подтвердить, что код состояния — 200 OK или 304 Not Modified .

Клиенты ДОЛЖНЫ проверить, что первые пять байтов объекта ответа соответствуют регулярному выражению ^[0-9a-f]{4}# . Если этот тест не пройден, клиенты НЕ ДОЛЖНЫ продолжать работу.

Клиенты ДОЛЖНЫ анализировать весь ответ как последовательность записей pkt-line.

Клиенты ДОЛЖНЫ проверить, что первая строка pkt — # service=$servicename . Серверы ДОЛЖНЫ установить $servicename в качестве значения параметра запроса. Серверам СЛЕДУЕТ включать LF в конце этой строки. Клиенты ДОЛЖНЫ игнорировать LF в конце строки.

Серверы ДОЛЖНЫ завершить ответ магическим маркером конца строки pkt 0000 .

Возвращенный ответ представляет собой поток строк pkt, описывающий каждую ссылку и ее известное значение. Поток СЛЕДУЕТ сортировать по имени в соответствии с порядком локали C. Поток ДОЛЖЕН включать ссылку по умолчанию с именем HEAD в качестве первой ссылки. Поток ДОЛЖЕН включать объявления возможностей после NUL в первой ссылке.

Возвращенный ответ содержит "version 1" , если "version=1" был отправлен в качестве дополнительного параметра.

smart_reply     =  PKT-LINE("# service=$servicename" LF)
     "0000"
     *1("version 1")
     ref_list
     "0000"
ref_list        =  empty_list / non_empty_list
empty_list      =  PKT-LINE(zero-id SP "capabilities^{}" NUL cap-list LF)
non_empty_list  =  PKT-LINE(obj-id SP name NUL cap_list LF)
     *ref_record
cap-list        =  capability *(SP capability)
capability      =  1*(LC_ALPHA / DIGIT / "-" / "_")
LC_ALPHA        =  %x61-7A
ref_record      =  any_ref / peeled_ref
any_ref         =  PKT-LINE(obj-id SP name LF)
peeled_ref      =  PKT-LINE(obj-id SP name LF)
     PKT-LINE(obj-id SP name "^{}" LF

Умный сервис git-upload-pack

Эта служба читает данные из репозитория, на который указывает $GIT_URL .

Клиенты ДОЛЖНЫ сначала выполнить обнаружение ссылок с помощью $GIT_URL/info/refs?service=git-upload-pack .

C: POST $GIT_URL/git-upload-pack HTTP/1.0
C: Content-Type: application/x-git-upload-pack-request
C:
C: 0032want 0a53e9ddeaddad63ad106860237bbf53411d11a7\n
C: 0032have 441b40d833fdfa93eb2908e52742248faf0ee993\n
C: 0000
S: 200 OK
S: Content-Type: application/x-git-upload-pack-result
S: Cache-Control: no-cache
S:
S: ....ACK %s, continue
S: ....NAK

Клиенты НЕ ДОЛЖНЫ повторно использовать или повторно проверять кэшированный ответ. Серверы ДОЛЖНЫ включать достаточное количество заголовков Cache-Control, чтобы предотвратить кеширование ответа.

Серверы ДОЛЖНЫ поддерживать все возможности, определенные здесь.

Клиенты ДОЛЖНЫ отправить хотя бы одну команду "want" в теле запроса. Клиенты НЕ ДОЛЖНЫ ссылаться на идентификатор в команде "want" , который не появился в ответе, полученном посредством обнаружения ссылок, если только сервер не объявляет возможность allow-tip-sha1-in-want или allow-reachable-sha1-in-want .

compute_request   =  want_list
       have_list
       request_end
request_end       =  "0000" / "done"
want_list         =  PKT-LINE(want SP cap_list LF)
       *(want_pkt)
want_pkt          =  PKT-LINE(want LF)
want              =  "want" SP id
cap_list          =  capability *(SP capability)
have_list         =  *PKT-LINE("have" SP id LF)

Алгоритм переговоров

Вычисление для выбора минимального пакета происходит следующим образом (C = клиент, S = сервер):

init step:

C: Используйте обнаружение ссылок для получения рекламируемых ссылок.

C: Поместите любой увиденный предмет в набор advertised .

C: Создайте пустой набор common для хранения объектов, которые позже будут определены как находящиеся на обоих концах.

C: Создайте набор want объектов из advertised , которые клиент хочет получить, на основе того, что он увидел во время обнаружения ссылок.

C: Запустить очередь c_pending в порядке времени фиксации (сначала появляются самые новые). Добавьте все ссылки на клиентов. Когда коммит извлекается из очереди, его родители ДОЛЖНЫ быть автоматически вставлены обратно. Коммиты ДОЛЖНЫ входить в очередь только один раз.

one compute step:

C: Отправьте один запрос $GIT_URL/git-upload-pack :

C: 0032want <want #1>...............................
C: 0032want <want #2>...............................
....
C: 0032have <common #1>.............................
C: 0032have <common #2>.............................
....
C: 0032have <have #1>...............................
C: 0032have <have #2>...............................
....
C: 0000

Поток организован в "commands" , где каждая команда отображается отдельно в строке pkt. В командной строке текст до первого пробела — это имя команды, а оставшаяся часть строки до первого LF — это значение. Командные строки завершаются LF в качестве последнего байта значения pkt-line.

Команды ДОЛЖНЫ появляться в следующем порядке, если они вообще появляются в потоке запросов:

  • "want"

  • "have"

Поток завершается сбросом строки pkt ( 0000 ).

Одна команда "want" или "have" ДОЛЖНА иметь в качестве значения одно имя объекта в шестнадцатеричном формате. Несколько имен объектов ДОЛЖНЫ быть отправлены путем отправки нескольких команд. Имена объектов ДОЛЖНЫ задаваться в формате объекта, согласованном с помощью возможности object-format (SHA-1 по умолчанию).

Список have создается путем извлечения первых 32 коммитов из c_pending . Меньшее количество может быть поставлено, если c_pending опорожняется.

Если клиент отправил 256 коммитов "have" и еще не получил ни одного из них обратно от s_common или клиент очистил c_pending , ему СЛЕДУЕТ включить команду "done" , чтобы сообщить серверу, что он не будет продолжен:

C: 0009done

S: Разберите запрос git-upload-pack:

Убедитесь, что все объекты в want доступны напрямую по ссылкам.

Сервер МОЖЕТ пройти назад по истории или по журналу обновлений, чтобы разрешить слегка устаревшие запросы.

Если объекты "want" не получены, отправьте сообщение об ошибке: TODO: Определить ошибку, если строки "want" не запрошены.

Если какой-либо объект "want" недоступен, отправьте сообщение об ошибке: TODO: Определить ошибку, если запрошен недопустимый "want" .

Создайте пустой список s_common .

Если был отправлен "have" :

Перебирайте объекты в порядке, указанном клиентом.

Для каждого объекта, если на сервере есть объект, доступный по ссылке, добавьте его в s_common . Если фиксация добавлена ​​в s_common , не добавляйте никаких предков, даже если они также присутствуют в have .

S: Отправьте ответ git-upload-pack:

Если сервер обнаружил закрытый набор объектов для упаковки или запрос заканчивается на "done" , он отвечает пакетом. ЗАДАЧА: Задокументировать ответ на основе пакета

S: PACK...

Возвращенный поток — это протокол боковой полосы-64k, поддерживаемый службой git-upload-pack, и пакет встроен в поток 1. Сообщения о ходе выполнения со стороны сервера МОГУТ появляться в потоке 2.

Здесь "closed set of objects" определяется как имеющий хотя бы один путь от каждого "want" хотя бы к одному объекту "common" .

Если серверу требуется дополнительная информация, он отвечает сообщением о продолжении статуса: TODO: Задокументируйте ответ, не относящийся к упаковке.

C: Разобрать ответ на пакет загрузки: TODO: Ответ на анализ документа

Умный сервис git-receive-pack

Эта служба читает данные из репозитория, на который указывает $GIT_URL .

Клиенты ДОЛЖНЫ сначала выполнить обнаружение ссылок с помощью $GIT_URL/info/refs?service=git-receive-pack .

C: POST $GIT_URL/git-receive-pack HTTP/1.0
C: Content-Type: application/x-git-receive-pack-request
C:
C: ....0a53e9ddeaddad63ad106860237bbf53411d11a7 441b40d833fdfa93eb2908e52742248faf0ee993 refs/heads/maint\0 report-status
C: 0000
C: PACK....
S: 200 OK
S: Content-Type: application/x-git-receive-pack-result
S: Cache-Control: no-cache
S:
S: ....

Клиенты НЕ ДОЛЖНЫ повторно использовать или повторно проверять кэшированный ответ. Серверы ДОЛЖНЫ включать достаточное количество заголовков Cache-Control, чтобы предотвратить кеширование ответа.

Серверы ДОЛЖНЫ поддерживать все возможности, определенные здесь.

Клиенты ДОЛЖНЫ отправить хотя бы одну команду в теле запроса. В командной части тела запроса клиенты ДОЛЖНЫ отправлять идентификатор, полученный посредством обнаружения ссылок, как old_id.

update_request  =  command_list
     "PACK" <binary data>
command_list    =  PKT-LINE(command NUL cap_list LF)
     *(command_pkt)
command_pkt     =  PKT-LINE(command LF)
cap_list        =  *(SP capability) SP
command         =  create / delete / update
create          =  zero-id SP new_id SP name
delete          =  old_id SP zero-id SP name
update          =  old_id SP new_id SP name
