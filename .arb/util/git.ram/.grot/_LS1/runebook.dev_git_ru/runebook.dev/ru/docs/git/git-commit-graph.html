git-commit-graph

Name

git-commit-graph — запись и проверка файлов графиков фиксации Git .

Synopsis

git commit-graph verify [--object-dir <dir>] [--shallow] [--[no-]progress]
git commit-graph write [--object-dir <dir>] [--append]
                        [--split[=<strategy>]] [--reachable | --stdin-packs | --stdin-commits]
                        [--changed-paths] [--[no-]max-new-filters <n>] [--[no-]progress]
                        <split options>

Description

Управляйте сериализованным файлом графа фиксации.

Options

--object-dir

Используйте указанный каталог для размещения пакфайлов и файла графа фиксации. Этот параметр существует для указания местоположения альтернативы, которая имеет только каталог объектов, а не полный каталог .git . Ожидается, что файл графа фиксации будет находиться в каталоге <dir>/info , а файлы пакета — в <dir>/pack . Если каталог не может быть преобразован в абсолютный путь или не соответствует ни одному известному каталогу объекта, git commit-graph ... завершит работу с ненулевым статусом.

--[no-]progress

Включите/выключите прогресс явно. Если ни один из них не указан, прогресс отображается, если к терминалу подключена стандартная ошибка.

Commands

write

Напишите файл графа коммитов на основе коммитов, найденных в пакфайлах. Если опция конфигурации core.commitGraph отключена, эта команда выведет предупреждение, а затем вернет успех без записи файла графа фиксации.

С опцией --stdin-packs сгенерируйте новый граф фиксации, просматривая объекты только в указанных индексах пакетов. (Невозможно объединить с --stdin-commits или --reachable .)

С помощью опции --stdin-commits сгенерируйте новый граф коммитов, просматривая коммиты, начиная с коммитов, указанных в стандартном вводе, в виде списка OID в шестнадцатеричном формате, по одному OID в строке. OID, которые разрешаются без фиксации (либо напрямую, либо путем очистки тегов), молча игнорируются. OID, которые имеют неправильный формат или не существуют, вызывают ошибку. (Невозможно объединить с --stdin-packs или --reachable .)

С помощью опции --reachable сгенерируйте новый граф коммитов, пройдя коммиты, начиная со всех ссылок. (Невозможно объединить с --stdin-commits или --stdin-packs .)

С помощью опции --append включите все фиксации, которые присутствуют в существующем файле графа фиксации.

С помощью опции --changed-paths вычисляйте и записывайте информацию о путях, измененных между фиксацией и ее первым родителем. Эта операция может занять некоторое время в больших репозиториях. Это обеспечивает значительный прирост производительности при получении истории каталога или файла с помощью git log -- <path> . Если указана эта опция, будущие записи графа фиксации будут автоматически предполагать, что эта опция была задумана. Используйте --no-changed-paths , чтобы прекратить сохранение этих данных.

С помощью опции --max-new-filters=<n> сгенерируйте новые фильтры Блума не более n (если указан --changed-paths ). Если n равен -1 , ограничение не применяется. В счет этого ограничения засчитываются только фиксации, присутствующие на новом слое. Чтобы задним числом вычислить фильтры Блума по более ранним слоям, рекомендуется использовать --split=replace . Переопределяет конфигурацию commitGraph.maxNewFilters .

С опцией --split[=<strategy>] запишите граф фиксации как цепочку нескольких файлов графа фиксации, хранящихся в <dir>/info/commit-graphs . Слои графа фиксации объединяются на основе стратегии и других вариантов разделения. Новые коммиты, которых еще нет в графе коммитов, добавляются в новый файл "tip" . Этот файл объединяется с существующим файлом, если выполняются следующие условия слияния:

  • Если указан --split=no-merge , слияние никогда не выполняется, а остальные параметры игнорируются. --split=replace перезаписывает существующую цепочку новой. Голый --split соответствует остальным вариантам. (Обратите внимание, что при слиянии цепочки графов фиксации существующая цепочка заменяется цепочкой длиной 1, где первый и единственный инкрементальный элемент содержит весь граф).

  • Если --size-multiple=<X> не указан, пусть X равен 2. Если новый файл подсказки будет иметь фиксации N , а предыдущий совет будет иметь фиксации M и X , умноженное на N , чем M , вместо этого объедините два файла в один файл.

  • Если --max-commits=<M> указан с M — положительным целым числом, а новый файл подсказки будет содержать больше коммитов M , то вместо этого объедините новую подсказку с предыдущей подсказкой.

    Наконец, если --expire-time=<datetime> не указан, пусть datetime будет текущим временем. После записи разделенного графа фиксации удалите все неиспользуемые графы фиксации, время изменения которых старше datetime .

verify

Прочтите файл графа фиксации и сверьте его содержимое с базой данных объектов. Используется для проверки поврежденных данных.

С опцией --shallow проверяйте только файл графа фиксации наконечника в цепочке разделенных графов фиксации.

Examples

  • Напишите файл графа фиксации для упакованных коммитов в локальном каталоге .git .

    $ git commit-graph write

  • Напишите файл графа фиксации, расширяя текущий файл графа фиксации с помощью коммитов в <pack-index> .

    $ echo <pack-index> | git commit-graph write --stdin-packs

  • Напишите файл графа коммитов, содержащий все доступные коммиты.

    $ git show-ref -s | git commit-graph write --stdin-commits

  • Напишите файл графа фиксации, содержащий все фиксации в текущем файле графа фиксации, а также те, которые доступны из HEAD .

    $ git rev-parse HEAD | git commit-graph write --stdin-commits --append

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

commitGraph.generationVersion

Указывает тип версии номера поколения, который будет использоваться при записи или чтении файла графа фиксации. Если указана версия 1, то исправленные даты фиксации не будут записаны или прочитаны. По умолчанию 2.

commitGraph.maxNewFilters

Указывает значение по умолчанию для опции --max-new-filters в git commit-graph write (см. git-commit-graph[1] ).

commitGraph.readChangedPaths

Если это правда, то git будет использовать фильтры Блума измененного пути в файле графа фиксации (если он существует и они присутствуют). По умолчанию истинно. См. git-commit-graph[1] для получения дополнительной информации.
