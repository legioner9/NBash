gitrepository-layout

Name

gitrepository-layout — Макет репозитория Git

Synopsis

$GIT_DIR/*

Description

Репозиторий Git поставляется в двух вариантах:

  • каталог .git в корне рабочего дерева;

  • каталог <project>.git , который представляет собой репозиторий bare (т. е. без собственного рабочего дерева), который обычно используется для обмена историями с другими путем ввода в него и извлечения из него.

Примечание. Также вы можете иметь простой текстовый файл .git в корне вашего рабочего дерева, содержащий gitdir: <path> , чтобы указать на реальный каталог, в котором находится репозиторий. Этот механизм называется gitfile и обычно управляется с помощью команд git submodule и git worktree . Он часто используется для рабочего дерева проверки подмодуля, чтобы разрешить вам в содержащем суперпроекте git checkout ветку, в которой нет подмодуля. checkout должен удалить все рабочее дерево субмодуля, не теряя при этом репозиторий субмодуля.

Эти вещи могут существовать в репозитории Git .

objects

Хранилище объектов, связанное с этим репозиторием. Обычно хранилище объектов самодостаточно (т.е. все объекты, на которые ссылается найденный в нем объект, также находятся в нем), но есть несколько способов его нарушить.

  1. Вы можете получить неполный, но пригодный для локального использования репозиторий, создав поверхностный клон. См. git-clone[1] .

  2. Вы можете использовать механизмы objects/info/alternates или $GIT_ALTERNATE_OBJECT_DIRECTORIES для объектов borrow из других хранилищ объектов. Репозиторий с таким неполным хранилищем объектов не подходит для публикации для использования с простыми транспортами, но в остальном все в порядке, если objects/info/alternates указывает на хранилища объектов, из которых он заимствует.

    Этот каталог игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/objects" .

objects/[0-9a-f][0-9a-f]

Вновь созданный объект сохраняется в отдельном файле. Объекты распределяются по 256 подкаталогам с использованием первых двух символов имени объекта sha1, чтобы сохранить управляемое количество записей каталога в самом objects . Найденные здесь объекты часто называют объектами unpacked (или loose ).

objects/pack

В этом каталоге находятся пакеты (файлы, в которых хранится множество объектов в сжатой форме, а также индексные файлы, обеспечивающие произвольный доступ к ним).

objects/info

В этот каталог записывается дополнительная информация о хранилище объектов.

objects/info/packs

Этот файл предназначен для того, чтобы помочь тупым транспортам узнать, какие пакеты доступны в этом хранилище объектов. Всякий раз, когда пакет добавляется или удаляется, следует запускать git update-server-info , чтобы поддерживать этот файл в актуальном состоянии, если репозиторий опубликован для простых транспортировок. git repack делает это по умолчанию.

objects/info/alternates

В этом файле записываются пути к альтернативным хранилищам объектов, из которых это хранилище объектов заимствует объекты, по одному пути в каждой строке. Обратите внимание, что не только встроенные инструменты Git используют его локально, но и сборщик HTTP также пытается использовать его удаленно; Обычно это работает, если у вас есть относительные пути (относительно базы данных объектов, а не репозитория!) в вашем альтернативном файле, но это не будет работать, если вы используете абсолютные пути, если только абсолютный путь в файловой системе и веб-URL не совпадают. См. также objects/info/http-alternates .

objects/info/http-alternates

В этом файле записаны URL-адреса альтернативных хранилищ объектов, из которых это хранилище объектов заимствует объекты, которые будут использоваться при загрузке репозитория через HTTP.

refs

Ссылки хранятся в подкаталогах этого каталога. Команда git prune умеет сохранять объекты, доступные по ссылкам, найденным в этом каталоге и его подкаталогах. Этот каталог игнорируется (кроме refs/bisect, refs/rewrite и refs/worktree), если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/refs" .

refs/heads/name

записывает объекты фиксации на вершине дерева ветки name

refs/tags/name

записывает любое имя объекта (не обязательно объект фиксации или объект-тег, указывающий на объект фиксации).

refs/remotes/name

записывает объекты коммитов на вершине дерева ветвей, скопированных из удаленного репозитория.

refs/replace/<obj-sha1>

записывает SHA-1 объекта, который заменяет <obj-sha1> . Это похоже на info/grafts и используется и поддерживается git-replace[1] внутри компании. Такие ссылки могут обмениваться между репозиториями, а графты — нет.

packed-refs

записывает ту же информацию, что и refs/heads/, refs/tags/, и запись друзей, более эффективным способом. См. git-pack-refs[1] . Этот файл игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/packed-refs" .

HEAD

Символьная ссылка (см. глоссарий) на пространство имен refs/heads/ , описывающая текущую активную ветвь. Это не имеет большого значения, если репозиторий не связан ни с каким рабочим деревом (например, репозиторий bare ), но действительный репозиторий Git должен иметь файл HEAD; некоторые фарфоры могут использовать его, чтобы угадать назначенную ветвь репозитория "default" (обычно master ). Это законно, если указанная ветка name (пока) не существует. В некоторых устаревших конфигурациях это символическая ссылка, а не символьная ссылка, указывающая на текущую ветку.

HEAD также может напрямую записывать конкретный коммит, а не быть символической ссылкой, указывающей на текущую ветку. Такое состояние часто называют detached HEAD. . Подробности см. в git-checkout[1] .

config

Файл конфигурации, специфичный для репозитория. Этот файл игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/config" .

config.worktree

Файл конфигурации, специфичный для рабочего каталога, для основного рабочего каталога при настройке нескольких рабочих каталогов (см. git-worktree[1] ).

branches

Немного устаревший способ хранения сокращений, которые будут использоваться для указания URL-адресов git fetch , git pull и git push . Файл можно сохранить как branches/<name> , а затем этим командам можно передать name вместо аргумента repository . Подробную информацию см. в разделе ПУЛЬТЫ ДИСТАНЦИОННОГО ДИСТАНЦИОННОГО УПРАВЛЕНИЯ в git-fetch[1] . Этот механизм является устаревшим, и его вряд ли можно найти в современных репозиториях. Этот каталог игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/branches" .

hooks

Хуки — это сценарии настройки, используемые различными командами Git . При запуске git init устанавливается несколько примеров перехватчиков, но все они по умолчанию отключены. Для включения суффикс .sample необходимо удалить из имени файла путем переименования. Прочтите githooks[5] для получения более подробной информации о каждом крючке. Этот каталог игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/hooks" .

common

Когда используется несколько рабочих деревьев, большинство файлов в $GIT_DIR относятся к каждому рабочему дереву, за некоторыми известными исключениями. Однако все файлы под common будут использоваться всеми рабочими деревьями.

index

Текущий индексный файл репозитория. Обычно его не найти в пустом репозитории.

sharedindex.<SHA-1>

Часть общего индекса, на которую будут ссылаться $GIT_DIR/index и другие временные индексные файлы. Действует только в режиме разделенного индекса.

info

В этом каталоге записывается дополнительная информация о репозитории. Этот каталог игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/info" .

info/refs

Этот файл помогает тупым транспортам узнать, какие ссылки доступны в этом репозитории. Если репозиторий опубликован для простых транспортировок, этот файл должен создаваться заново git update-server-info каждый раз, когда создается или изменяется тег или ветвь. Обычно это делается с помощью перехватчика hooks/update , который запускается командой git-receive-pack при входе git push в репозиторий.

info/grafts

Этот файл записывает информацию о происхождении фальшивого коммита, чтобы сделать вид, что набор родителей коммита отличается от того, как коммит был фактически создан. Одна запись в строке описывает фиксацию и ее поддельных родителей путем перечисления их 40-байтовых шестнадцатеричных имен объектов, разделенных пробелом и заканчивающихся новой строкой.

Обратите внимание, что механизм трансплантации устарел и может привести к проблемам при переносе объектов между репозиториями; см. git-replace[1] , где представлена ​​более гибкая и надежная система, позволяющая делать то же самое.

info/exclude

В этом файле по соглашению Porcelains хранится список исключенных шаблонов. .gitignore — это файл игнорирования для каждого каталога. git status , git add , git rm и git clean смотрят на него, но основные команды Git на него не смотрят. См. также: gitignore[5] .

info/attributes

Определяет, какие атрибуты присваиваются пути, аналогично файлам .gitattributes для каждого каталога. См. также: gitattributes[5] .

info/sparse-checkout

В этом файле хранятся редкие шаблоны оформления заказа. См. также: git-read-tree[1] .

remotes

Сохраняет сокращения URL-адресов и имена ссылок по умолчанию для использования при взаимодействии с удаленными репозиториями с помощью команд git fetch , git pull и git push . Подробную информацию см. в разделе ПУЛЬТЫ ДИСТАНЦИОННОГО УПРАВЛЕНИЯ в git-fetch[1] . Этот механизм является устаревшим, и его вряд ли можно найти в современных репозиториях. Этот каталог игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/remotes" .

logs

В этом каталоге хранятся записи об изменениях, внесенных в ссылки. См. git-update-ref[1] для получения дополнительной информации. Этот каталог игнорируется (кроме logs/HEAD), если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/logs" .

logs/refs/heads/name

Записывает все изменения, внесенные в кончик ветки с именем name .

logs/refs/tags/name

Записывает все изменения, внесенные в тег с именем name .

shallow

Он похож на info/grafts , но используется и поддерживается внутри страны с помощью механизма поверхностного клонирования. См. опцию --depth для git-clone[1] и git-fetch[1] . Этот файл игнорируется, если установлен $GIT_COMMON_DIR, и вместо него будет использоваться "$GIT_COMMON_DIR/shallow" .

commondir

Если этот файл существует, в $GIT_COMMON_DIR (см. git[1] ) будет установлен путь, указанный в этом файле, если он не задан явно. Если указанный путь относительный, он относится к $GIT_DIR. Репозиторий с commondir будет неполным без репозитория, на который указывает "commondir" .

modules

Содержит git-репозитории подмодулей.

worktrees

Содержит административные данные для связанных рабочих деревьев. Каждый подкаталог содержит связанную с рабочим деревом часть связанного рабочего дерева. Этот каталог игнорируется, если установлен $GIT_COMMON_DIR, и в этом случае вместо него будет использоваться "$GIT_COMMON_DIR/worktrees" .

worktrees/<id>/gitdir

Текстовый файл, содержащий абсолютный обратный путь к файлу .git, который указывает сюда. Это используется для проверки того, был ли связанный репозиторий удален вручную и нет необходимости больше сохранять этот каталог. Время этого файла должно обновляться каждый раз при доступе к связанному репозиторию.

worktrees/<id>/locked

Если этот файл существует, связанное рабочее дерево может находиться на портативном устройстве и недоступно. Наличие этого файла предотвращает автоматическое или ручное удаление worktrees/<id> с помощью git worktree prune . Файл может содержать строку, объясняющую, почему репозиторий заблокирован.

worktrees/<id>/config.worktree

Файл конфигурации, специфичный для рабочего каталога.

Версии формата репозитория Git

Каждый репозиторий git помечен числовой версией в ключе core.repositoryformatversion его файла config . Эта версия определяет правила работы с данными хранилища на диске. Реализация git , которая не понимает конкретную версию, рекламируемую дисковым репозиторием, НЕ ДОЛЖНА работать с этим репозиторием; это рискует не только дать неверные результаты, но и фактически привести к потере данных.

Из-за этого правила изменения версий должны быть сведены к абсолютному минимуму. Вместо этого мы обычно предпочитаем следующие стратегии:

  • изменение номеров версий формата отдельных файлов данных (например, индекса, паковочных файлов и т. д.). Это ограничивает несовместимости только этими файлами.

  • введение новых данных, качество которых корректно ухудшается при использовании старыми клиентами (например, файлы растровых изображений игнорируются старыми клиентами, которые просто не используют преимущества предоставляемой ими оптимизации).

Обновление версии формата всего репозитория должно быть только частью изменения, которое не может быть независимо версионировано. Например, если бы кто-то изменил правила доступности для объектов или правила блокировки ссылок, это потребовало бы изменения версии формата хранилища.

Обратите внимание, что это относится только к прямому доступу к содержимому диска репозитория. Более старый клиент, который понимает только формат 0 , может по-прежнему подключаться через git:// к репозиторию, используя формат 1 , если серверный процесс понимает формат 1 .

Предпочтительная стратегия развертывания обновления версии (будь то весь репозиторий или отдельный файл) состоит в том, чтобы научить git читать новый формат и разрешить запись нового формата с помощью переключателя конфигурации или параметра командной строки (для экспериментов или для тех, кто не заботьтесь об обратной совместимости со старыми git ). Затем, после длительного периода, когда возможность чтения станет обычным явлением, мы можем переключиться на запись нового формата по умолчанию.

В настоящее время определены следующие версии формата:

Version 0

Это формат, определенный начальной версией git , включая, помимо прочего, формат каталога репозитория, файла конфигурации репозитория, а также хранилища объектов и ссылок. Описание полного поведения git выходит за рамки этого документа.

Version 1

Этот формат идентичен версии 0 , за следующими исключениями:

  1. При чтении переменной core.repositoryformatversion реализация git , поддерживающая версию 1, ДОЛЖНА также прочитать все ключи конфигурации, находящиеся в разделе extensions файла конфигурации.

  2. Если в репозитории версии 1 указаны какие-либо ключи extensions.* , которые не реализованы в работающем git , операция НЕ ДОЛЖНА продолжаться. Аналогично, если значение какого-либо известного ключа не понимается реализацией, операция НЕ ДОЛЖНА продолжаться.

Обратите внимание: если в файле конфигурации не указаны расширения, то для core.repositoryformatversion СЛЕДУЕТ установить значение 0 (установка 1 не дает никаких преимуществ и делает репозиторий несовместимым со старыми реализациями git).

Этот документ будет служить основным списком расширений. Любая реализация, желающая определить новое расширение, должна сделать это здесь, чтобы получить имя.

Определенные расширения:

noop

Это расширение вообще не меняет поведение git. Это полезно только для проверки совместимости формата 1.

preciousObjects

Если для конфигурационного ключа extensions.preciousObjects установлено значение true , объекты в репозитории НЕ ДОЛЖНЫ удаляться (например, с помощью git-prune или git repack -d ).

partialClone

Когда установлен ключ конфигурации extensions.partialClone , это указывает, что репо было создано с частичным клонированием (или позже выполнено частичное извлечение) и что удаленный сервер мог пропустить отправку определенных нежелательных объектов. Такой пульт называется "promisor remote" и promises , поэтому все такие пропущенные объекты можно будет извлечь из него в будущем.

Значением этого ключа является имя удаленного поставщика.

worktreeConfig

Если установлено, по умолчанию "git config" считывает файлы "config" и "config.worktree" из GIT_DIR в указанном порядке. В режиме нескольких рабочих каталогов файл "config" является общим, а "config.worktree" — отдельным рабочим каталогом (т. е. он находится в GIT_COMMON_DIR/worktrees/<id>/config.worktree).
