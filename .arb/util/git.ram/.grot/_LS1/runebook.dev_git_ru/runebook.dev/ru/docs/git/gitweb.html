gitweb

Name

gitweb — веб-интерфейс Git (веб-интерфейс для репозиториев Git )

Synopsis

Чтобы начать работу с gitweb, запустите git-instaweb[1] из репозитория Git . Это позволит настроить и запустить ваш веб-сервер, а также запустить веб-браузер, указывающий на gitweb.

Description

Git web предоставляет веб-интерфейс для репозиториев Git . Его особенности включают в себя:

  • Просмотр нескольких репозиториев Git с общим корнем.

  • Просмотр каждой ревизии репозитория.

  • Просмотр содержимого файлов в репозитории любой ревизии.

  • Просмотр журнала изменений веток, истории файлов и каталогов, просмотр того, что, когда и кем было изменено.

  • Просмотр сведений об обвинениях/аннотациях любого файла (если включено).

  • Генерация RSS и Atom-каналов коммитов для любой ветки. Ленты автоматически обнаруживаются в современных веб-браузерах.

  • Просмотр всего, что было изменено в ревизии, и пошаговое перебор ревизий, просмотр истории репозитория.

  • Поиск коммитов, чьи сообщения о коммитах соответствуют заданному поисковому запросу.

См. исходный код gitweb в https://repo.or.cz/w/git.git/tree/HEAD:/gitweb/ , просмотр которого осуществляется с помощью самого gitweb.

Configuration

Различные аспекты поведения gitweb можно контролировать через файл конфигурации gitweb_config.perl или /etc/gitweb.conf . Подробности см. в gitweb.conf[5] .

Repositories

Веб-сайт Git может отображать информацию из одного или нескольких репозиториев Git . Все эти репозитории должны находиться в локальной файловой системе и иметь общий корень репозитория, т.е. все они должны находиться в одном родительском репозитории (но см. также раздел "Advanced web server setup" , подраздел "Webserver configuration with multiple projects' root" ).

our $projectroot = '/path/to/parent/directory';

Значение по умолчанию для $projectroot — /pub/git . Вы можете изменить его во время сборки gitweb с помощью переменной конфигурации сборки GITWEB_PROJECTROOT .

По умолчанию все репозитории Git под $projectroot видимы и доступны для gitweb. Список проектов генерируется по умолчанию путем сканирования каталога $projectroot на наличие репозиториев Git (точнее, для объектных баз данных; gitweb не интересуется рабочей областью и лучше всего подходит для отображения репозиториев "bare" ).

Имя репозитория в gitweb — это путь к его $GIT_DIR (его объектной базе данных) относительно $projectroot . Поэтому репозиторий $repo можно найти по адресу "$projectroot/$repo" .

Формат файла списка проектов

Вместо того, чтобы gitweb находил репозитории путем сканирования файловой системы, начиная с $projectroot, вы можете предоставить предварительно созданный список видимых проектов, настроив $projects_list так, чтобы он указывал на обычный текстовый файл со списком проектов (с некоторой дополнительной информацией).

Этот файл использует следующий формат:

  • Одна запись (для проекта/репозитория) в каждой строке; не поддерживает продолжение строки (экранирование новой строки).

  • Ведущие и конечные пробелы игнорируются.

  • Поля, разделенные пробелами; любой пробел может использоваться в качестве разделителя полей (правила для Perl " split(" ", $line) " ).

  • Поля используют модифицированную кодировку URI, определенную в RFC 3986, раздел 2.1 (Процентное кодирование), или, скорее, "Query string encoding" (см. https://en.wikipedia.org/wiki/Query_string#URL_encoding ), разница в том, что SP ( " " ) может быть закодирован как "+" (и, следовательно, "+" также должен быть процентным). закодировано).

    Зарезервированные символы: "%" (используется для кодирования), "+" (может использоваться для кодирования ПРОБЕЛА), все символы пробелов, определенные в Perl, включая SP, TAB и LF (используются для разделения полей в записи).

  • В настоящее время признаны следующие поля:

    <repository path>

    путь к репозиторию GIT_DIR относительно $projectroot

    <repository owner>

    отображается как владелец репозитория, желательно полное имя или адрес электронной почты, или и то, и другое.

Вы можете создать индексный файл списка проектов с помощью действия project_index (ссылка TXT на странице списка проектов) непосредственно из gitweb; см. также раздел "Generating projects list using gitweb" ниже.

Example contents:

foo.git       Joe+R+Hacker+<joe@example.com>
foo/bar.git   O+W+Ner+<owner@example.org>

По умолчанию этот файл контролирует только то, какие проекты видны на странице списка проектов (обратите внимание, что записи, которые не указывают на правильно распознанные репозитории Git , не будут отображаться gitweb). Даже если проект не отображается на странице списка проектов, вы, тем не менее, можете просмотреть его, создав вручную URL-адрес gitweb. Установив для переменной конфигурации $strict_export (см. gitweb.conf[5] ) значение true , вы можете разрешить просмотр только тех репозиториев, которые также показаны на странице обзора (т.е. будут доступны только проекты, явно перечисленные в файле списка проектов).

Создание списка проектов с помощью gitweb

Мы предполагаем, что GITWEB_CONFIG имеет значение Makefile по умолчанию, а именно gitweb_config.perl . Поместите следующее в файл gitweb_make_index.perl :

read_config_file("gitweb_config.perl");
$projects_list = $projectroot;

Затем создайте следующий скрипт, чтобы получить список проектов в формате, подходящем для переменной конфигурации сборки GITWEB_LIST (или переменной $projects_list в конфигурации gitweb):

#!/bin/sh

export GITWEB_CONFIG="gitweb_make_index.perl"
export GATEWAY_INTERFACE="CGI/1.1"
export HTTP_ACCEPT="*/*"
export REQUEST_METHOD="GET"
export QUERY_STRING="a=project_index"

perl -- /var/www/cgi-bin/gitweb.cgi

Запустите этот сценарий и сохраните его вывод в файл. Этот файл затем можно использовать в качестве файла списка проектов, а это означает, что вы можете установить $projects_list в качестве его имени файла.

Управление доступом к репозиториям Git

По умолчанию все репозитории Git под $projectroot видимы и доступны для gitweb. Однако вы можете настроить, как gitweb контролирует доступ к репозиториям.

  • Как описано в разделе "Projects list file format" , вы можете контролировать, какие проекты будут видимы, выборочно включая репозитории в файл списка проектов и устанавливая переменную конфигурации gitweb $projects_list , чтобы она указывала на него. Если установлен $strict_export , файл списка проектов также можно использовать для управления доступностью репозиториев.

  • Вы можете настроить gitweb только для просмотра и разрешения просмотра явно экспортированных репозиториев с помощью переменной $export_ok в файле конфигурации gitweb; см. справочную страницу gitweb.conf[5] . Если он равен true, gitweb показывает репозитории только в том случае, если этот файл с именем $export_ok существует в его базе данных объектов (если в каталоге есть волшебный файл с именем $export_ok ).

    Например, git-daemon[1] по умолчанию (если не используется опция --export-all ) позволяет извлекать только те репозитории, в которых есть файл git-daemon-export-ok . Добавление

    our $export_ok = "git-daemon-export-ok";

    заставляет gitweb показывать и разрешать доступ только к тем репозиториям, из которых можно получить данные по протоколу git:// .

  • Наконец, можно указать произвольную процедуру Perl, которая будет вызываться для каждого репозитория, чтобы определить, можно ли его экспортировать. Подпрограмма получает в качестве единственного параметра абсолютный путь к проекту (репозиторию) (т. е. "$projectroot/$project" ).

    Например, если вы используете mod_perl для запуска сценария и для ваших репозиториев настроена простая аутентификация по протоколу HTTP, вы можете использовать следующий хук, чтобы разрешить доступ только в том случае, если пользователь имеет право читать файлы:

    $export_auth_hook = sub {
        use Apache2::SubRequest ();
        use Apache2::Const -compile => qw(HTTP_OK);
        my $path = "$_[0]/HEAD";
        my $r    = Apache2::RequestUtil->request;
        my $sub  = $r->lookup_file($path);
        return $sub->filename eq $path
            && $sub->status == Apache2::Const::HTTP_OK;
};

Конфигурация gitweb для каждого репозитория

Вы можете настроить отдельные репозитории, показанные в gitweb, создав файл в репозитории GIT_DIR или Git или установив некоторую переменную конфигурации репозитория (в GIT_DIR/config , см. git-config[1] ).

В репозитории вы можете использовать следующие файлы:

README.html

HTML-файл (фрагмент HTML), который включен на страницу проекта gitweb "summary" внутри блочного элемента <div> . Вы можете использовать его для более подробного описания проекта, для предоставления ссылок (например, на домашнюю страницу проекта) и т. д. Это распознается только в том случае, если предотвращение XSS отключено ( $prevent_xss имеет значение false, см. gitweb.conf[5] ); способ безопасного включения README при включенной защите XSS может быть разработан в будущем.

описание (или gitweb.description )

Краткое (сокращенное до $projects_list_description_width на странице списка проектов, длина которого по умолчанию составляет 25 символов; см. gitweb.conf[5] ) однострочное описание проекта (репозитория). Обычный текстовый файл; HTML будет экранирован. По умолчанию установлено значение

Unnamed repository; edit this file to name it for gitweb.

из шаблона при создании репозитория, обычно устанавливается в /usr/share/git-core/templates/ . Вы можете использовать переменную конфигурации репозитория gitweb.description , но файл имеет приоритет.

категория (или gitweb.category )

Категория одной строки проекта, используемая для группировки проектов, если включен $projects_list_group_categories . По умолчанию (файл и переменная конфигурации отсутствуют) проекты без категорий помещаются в категорию $project_list_default_category . Вы можете использовать переменную конфигурации репозитория gitweb.category , но файл имеет приоритет.

Переменные конфигурации $projects_list_group_categories и $project_list_default_category описаны в gitweb.conf[5] .

cloneurl (или многозначный gitweb.url )

Файл с URL-адресом репозитория (используется для клонирования и извлечения), по одному в строке. Отображается на странице сводки проекта. Для этого вы можете использовать многозначную переменную конфигурации репозитория gitweb.url , но файл имеет приоритет.

Это улучшение/версия глобальной переменной конфигурации gitweb @git_base_url_list на основе префикса для каждого репозитория (см. gitweb.conf[5] ).

gitweb.owner

Вы можете использовать переменную конфигурации репозитория gitweb.owner , чтобы установить владельца репозитория. Он отображается в списке проектов и на странице сводки.

Если он не установлен, используется владелец каталога файловой системы (через поле GECOS, т.е. поле реального имени из getpwuid(3)), если $projects_list не установлен (gitweb сканирует $projectroot в поисках репозиториев); если $projects_list указывает на файл со списком репозиториев, то владелец проекта по умолчанию использует значение из этого файла для данного репозитория.

различные переменные конфигурации gitweb.* (в конфигурации)

Подробный список и описания читайте в описании хеша %feature . См. также раздел "Configuring gitweb features" в gitweb.conf[5] .

Действия и URL-адреса

Gitweb может использовать URL-адреса на основе path_info (компонента) или передавать всю необходимую информацию через параметры запроса. Типичные URL-адреса gitweb разбиты на пять компонентов:

.../gitweb.cgi/<repo>/<action>/<revision>:/<path>?<arguments>
repo

Репозиторий, над которым будет выполнено действие.

Этот параметр требуется для всех действий, за исключением тех, в которых перечислены все доступные проекты в любой форме.

action

Действие, которое будет запущено. По умолчанию используется projects_list , если репо не установлено, и summary в противном случае.

revision

Показана ревизия. По умолчанию HEAD.

path

Путь в <репозитории>, над которым выполняется действие, для тех действий, которые этого требуют.

arguments

Любые аргументы, управляющие поведением действия.

Некоторые действия требуют или позволяют указать две ревизии, а иногда даже два пути. В наиболее общей форме такой URL-адрес gitweb на основе path_info (компонента) выглядит следующим образом:

.../gitweb.cgi/<repo>/<action>/<revision_from>:/<path_from>..<revision_to>:/<path_to>?<arguments>

Каждое действие реализуется как подпрограмма и должно присутствовать в хеше %actions. Некоторые действия отключены по умолчанию и должны быть включены с помощью функционального механизма. Например, чтобы включить представление blame , добавьте в файл конфигурации gitweb следующее:

$feature{'blame'}{'default'} = [1];

Actions:

Стандартные действия:

project_list

Перечисляет доступные репозитории Git . Это команда по умолчанию, если в URL-адресе не указан репозиторий.

summary

Отображает сводную информацию о данном репозитории. Это команда по умолчанию, если в URL-адресе не указано действие и указан только репозиторий.

heads
remotes

Перечисляет все локальные или все ветки удаленного отслеживания в данном репозитории.

Последний недоступен по умолчанию, если не настроен.

tags

Перечислите все теги (облегченные и аннотированные) в данном репозитории.

blob
tree

Показывает файлы и каталоги по заданному пути к репозиторию в данной ревизии. Это команда по умолчанию, если в URL-адресе не указано действие и указан путь.

blob_plain

Возвращает необработанные данные для файла в заданном репозитории по заданному пути и версии. Ссылки на это действие имеют пометку raw .

blobdiff

Показывает разницу между двумя редакциями одного и того же файла.

blame
blame_incremental

Показывает информацию об ошибке (также называемую аннотацией) для файла. Для каждой строки отображается ревизия, в которой эта строка была изменена в последний раз, и пользователь, внесший это изменение. Инкрементальная версия (которая, если она настроена, используется автоматически при включении JavaScript ) использует Ajax для постепенного добавления информации о виновности к содержимому данного файла.

Это действие отключено по умолчанию из соображений производительности.

commit
commitdiff

Показывает информацию о конкретном коммите в репозитории. Представление commit отображает более подробную информацию о коммите, действие commitdiff показывает набор изменений для данного коммита.

patch

Возвращает фиксацию в текстовом почтовом формате, подходящем для применения с git-am[1] .

tag

Отображение определенного аннотированного тега (объекта тега).

log
shortlog

Показывает информацию журнала (сообщение о фиксации или просто тему фиксации) для данной ветки (начиная с данной ревизии).

Вид shortlog более компактен; он показывает один коммит в каждой строке.

history

Показывает историю файла или каталога в заданном пути к репозиторию, начиная с заданной ревизии (по умолчанию HEAD, т.е. ветка по умолчанию).

Этот вид похож на вид shortlog .

rss
atom

Создает RSS (или Atom) канал изменений в репозитории.

Webserver configuration

В этом разделе объясняется, как настроить некоторые распространенные веб-серверы для запуска gitweb. Во всех случаях /path/to/gitweb в примерах — это каталог, в котором вы запустили установленный gitweb, и он содержит gitweb_config.perl .

Если вы настроили для gitweb веб-сервер, которого нет в списке, отправьте инструкции, чтобы их можно было включить в будущий выпуск.

Апач как CGI

Apache должен быть настроен для поддержки сценариев CGI в каталоге, в котором установлен gitweb. Предположим, что это каталог /var/www/cgi-bin .

ScriptAlias /cgi-bin/ "/var/www/cgi-bin/"

<Directory "/var/www/cgi-bin">
    Options Indexes FollowSymlinks ExecCGI
    AllowOverride None
    Order allow,deny
    Allow from all
</Directory>

В этой конфигурации полный путь для просмотра репозиториев будет таким:

http://server/cgi-bin/gitweb.cgi

Apache с mod_perl через ModPerl::Registry

Вы можете использовать mod_perl с gitweb. Вы должны установить Apache::Registry (для mod_perl 1.x) или ModPerl::Registry (для mod_perl 2.x), чтобы включить эту поддержку.

Предполагая, что gitweb установлен на /var/www/perl , подойдет следующая конфигурация Apache (для mod_perl 2.x).

Alias /perl "/var/www/perl"

<Directory "/var/www/perl">
    SetHandler perl-script
    PerlResponseHandler ModPerl::Registry
    PerlOptions +ParseHeaders
    Options Indexes FollowSymlinks +ExecCGI
    AllowOverride None
    Order allow,deny
    Allow from all
</Directory>

В этой конфигурации полный путь для просмотра репозиториев будет таким:

http://server/perl/gitweb.cgi

Apache с FastCGI

Gitweb работает с Apache и FastCGI. Сначала вам нужно переименовать, скопировать или создать символическую ссылку gitweb.cgi на gitweb.fcgi. Предположим, что gitweb установлен в каталоге /usr/share/gitweb . Подходит следующая конфигурация Apache (НЕПРОВЕРЕНО!)

FastCgiServer /usr/share/gitweb/gitweb.cgi
ScriptAlias /gitweb /usr/share/gitweb/gitweb.cgi

Alias /gitweb/static /usr/share/gitweb/static
<Directory /usr/share/gitweb/static>
    SetHandler default-handler
</Directory>

В этой конфигурации полный путь для просмотра репозиториев будет таким:

http://server/gitweb

Расширенная настройка веб-сервера

Во всех этих примерах используется перезапись запросов и требуется mod_rewrite (или эквивалент; примеры ниже написаны для Apache).

Единый URL-адрес для gitweb и для загрузки

Если вы хотите иметь один URL-адрес для репозиториев gitweb и http:// , вы можете настроить Apache следующим образом:

<VirtualHost *:80>
    ServerName    git.example.org
    DocumentRoot  /pub/git
    SetEnv        GITWEB_CONFIG   /etc/gitweb.conf

    # turning on mod rewrite
    RewriteEngine on

    # make the front page an internal rewrite to the gitweb script
    RewriteRule ^/$  /cgi-bin/gitweb.cgi

    # make access for "dumb clients" work
    RewriteRule ^/(.*\.git/(?!/?(HEAD|info|objects|refs)).*)?$ \
                /cgi-bin/gitweb.cgi%{REQUEST_URI}  [L,PT]
</VirtualHost>

В приведенной выше конфигурации предполагается, что ваши репозитории public будут находиться под /pub/git и будут служить им как http://git.domain.org/dir-under-pub-git как клонируемый URL-адрес Git , так и как доступный для просмотра интерфейс gitweb. Если вы затем запустите свой git-daemon[1] с помощью --base-path=/pub/git --export-all , вы даже сможете использовать URL-адрес git:// с точно таким же путем.

Установка переменной среды GITWEB_CONFIG сообщит gitweb использовать указанный файл (т. е. в этом примере /etc/gitweb.conf ) в качестве конфигурации для gitweb. В приведенном выше примере вам это действительно не нужно; это требуется только в том случае, если ваш файл конфигурации находится в другом месте, чем встроенный (во время компиляции gitweb) gitweb_config.perl или /etc/gitweb.conf . Подробности, особенно информацию о правилах приоритета, см. в gitweb.conf[5] .

Если вы используете правила перезаписи из примера, вам также может понадобиться что-то вроде следующего в файле конфигурации gitweb (следующий пример /etc/gitweb.conf ):

@stylesheets = ("/some/absolute/path/gitweb.css");
$my_uri    = "/";
$home_link = "/";
$per_request_config = 1;

Однако в настоящее время gitweb должен при необходимости создавать базовый тег HTML (чтобы установить базовый URI для относительных ссылок), поэтому он должен работать автоматически.

Конфигурация веб-сервера с корнем нескольких проектов

Если вы хотите использовать gitweb с несколькими корнями проектов, вы можете отредактировать виртуальный хост Apache и файлы конфигурации gitweb следующим образом.

Конфигурация виртуального хоста (в файле конфигурации Apache) должна выглядеть следующим образом:

<VirtualHost *:80>
    ServerName    git.example.org
    DocumentRoot  /pub/git
    SetEnv        GITWEB_CONFIG  /etc/gitweb.conf

    # включаем перезапись мода
    RewriteEngine on

    # сделать главную страницу внутренней перезаписью скрипта gitweb
    RewriteRule ^/$  /cgi-bin/gitweb.cgi  [QSA,L,PT]

    # найдите каталог public_git в доме пользователя unix
    # http://git.example.org/~<user>/
    RewriteRule ^/\~([^\/]+)(/|/gitweb.cgi)?$        /cgi-bin/gitweb.cgi \
                [QSA,E=GITWEB_PROJECTROOT:/home/$1/public_git/,L,PT]

    # http://git.example.org/+<user>/
    #RewriteRule ^/\+([^\/]+)(/|/gitweb.cgi)?$ /cgi-bin/gitweb.cgi \
                 [QSA,E=GITWEB_PROJECTROOT:/home/$1/public_git/,L,PT]

    # http://git.example.org/user/<user>/
    #RewriteRule ^/user/([^\/]+)/(gitweb.cgi)?$ /cgi-bin/gitweb.cgi \
                 [QSA,E=GITWEB_PROJECTROOT:/home/$1/public_git/,L,PT]

    # определенный список корней проекта
    RewriteRule ^/scm(/|/gitweb.cgi)?$ /cgi-bin/gitweb.cgi \
                [QSA,E=GITWEB_PROJECTROOT:/pub/scm/,L,PT]
    RewriteRule ^/var(/|/gitweb.cgi)?$ /cgi-bin/gitweb.cgi \
                [QSA,E=GITWEB_PROJECTROOT:/var/git/,L,PT]

    # сделать доступ для работы  "dumb clients"
    RewriteRule ^/(.*\.git/(?!/?(HEAD|info|objects|refs)).*)?$ \
                /cgi-bin/gitweb.cgi%{REQUEST_URI}  [L,PT]
</VirtualHost>

Здесь фактический корень проекта передается в gitweb через переменную среды GITWEB_PROJECT_ROOT с веб-сервера, поэтому вам необходимо поместить следующую строку в файл конфигурации gitweb ( /etc/gitweb.conf в приведенном выше примере):

$projectroot = $ENV{'GITWEB_PROJECTROOT'} || "/pub/git";

Обратите внимание, что это необходимо устанавливать для каждого запроса, поэтому либо $per_request_config должен быть ложным, либо приведенное выше должно быть помещено в код, на который ссылается $per_request_config ;

Эти конфигурации позволяют сделать две вещи. Во-первых, каждый пользователь unix ( <user> ) сервера сможет просматривать репозитории gitweb Git , найденные в ~/public_git/ , по следующему URL-адресу:

http://git.example.org/~<user>/

Если вы не хотите, чтобы эта функция была на вашем сервере, просто удалите второе правило перезаписи.

Если вы уже используете mod_userdir на своем виртуальном хосте или не хотите использовать «~» в качестве первого символа, просто закомментируйте или удалите второе правило перезаписи и раскомментируйте одно из следующих значений в соответствии с вашими предпочтениями.

Во-вторых, репозитории, найденные в /pub/scm/ и /var/git/ , будут доступны через http://git.example.org/scm/ и http://git.example.org/var/ . Вы можете добавить столько корней проекта, сколько захотите, добавив правила перезаписи, такие как третий и четвертый.

PATH_INFO usage

Если вы включите использование PATH_INFO в gitweb, поставив

$feature{'pathinfo'}{'default'} = [1];

в вашем файле конфигурации gitweb можно настроить сервер так, чтобы он потреблял и создавал URL-адреса в форме

http://git.example.com/project.git/shortlog/sometag

т.е. без детали gitweb.cgi , используя следующую конфигурацию. Эта конфигурация предполагает, что /var/www/gitweb является DocumentRoot вашего веб-сервера, содержит сценарий gitweb.cgi и дополнительные файлы static (таблица стилей, значок значка, JavaScript):

<VirtualHost *:80>
        ServerAlias git.example.com

        DocumentRoot /var/www/gitweb

        <Directory /var/www/gitweb>
                Options ExecCGI
                AddHandler cgi-script cgi

                DirectoryIndex gitweb.cgi

                RewriteEngine On
                RewriteCond %{REQUEST_FILENAME} !-f
                RewriteCond %{REQUEST_FILENAME} !-d
                RewriteRule ^.* /gitweb.cgi/$0 [L,PT]
        </Directory>
</VirtualHost>

Правило перезаписи гарантирует, что существующие файлы static будут правильно обработаны, тогда как любой другой URL-адрес будет передан в gitweb как параметр PATH_INFO.

Обратите внимание, что в этом случае вам не нужны специальные настройки для @stylesheets , $my_uri и $home_link , но вы потеряете доступ "dumb client" к каталогам .git вашего проекта (описанным в разделе "Single URL for gitweb and for fetching" ). Возможный обходной путь для последнего: в корневом каталоге вашего проекта (например, /pub/git ) поместите проекты без расширения .git (например, /pub/git/project вместо /pub/git/project.git ) и настройте Apache следующим образом:

<VirtualHost *:80>
        ServerAlias git.example.com

        DocumentRoot /var/www/gitweb

        AliasMatch ^(/.*?)(\.git)(/.*)?$ /pub/git$1$3
        <Directory /var/www/gitweb>
                Options ExecCGI
                AddHandler cgi-script cgi

                DirectoryIndex gitweb.cgi

                RewriteEngine On
                RewriteCond %{REQUEST_FILENAME} !-f
                RewriteCond %{REQUEST_FILENAME} !-d
                RewriteRule ^.* /gitweb.cgi/$0 [L,PT]
        </Directory>
</VirtualHost>

Дополнительный AliasMatch позволяет

http://git.example.com/project.git

предоставит необработанный доступ к каталогу Git проекта (чтобы проект можно было клонировать), в то время как

http://git.example.com/project

обеспечит удобный доступ к gitweb.

Это решение не является на 100% пуленепробиваемым в том смысле, что если какой-то проект имеет именованный ref (ветвь, тег), начинающийся с git/ , то такие пути, как

http://git.example.com/project/command/abranch..git/abranch

завершится ошибкой 404.
