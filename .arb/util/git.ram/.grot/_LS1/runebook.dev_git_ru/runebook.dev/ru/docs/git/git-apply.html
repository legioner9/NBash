git-apply

Name

git-apply — применить патч к файлам и/или индексу.

Synopsis

git apply [--stat] [--numstat] [--summary] [--check] [--index | --intent-to-add] [--3way]
          [--apply] [--no-add] [--build-fake-ancestor=<file>] [-R | --reverse]
          [--allow-binary-replacement | --binary] [--reject] [-z]
          [-p<n>] [-C<n>] [--inaccurate-eof] [--recount] [--cached]
          [--ignore-space-change | --ignore-whitespace]
          [--whitespace=(nowarn|warn|fix|error|error-all)]
          [--exclude=<path>] [--include=<path>] [--directory=<root>]
          [--verbose | --quiet] [--unsafe-paths] [--allow-empty] [<patch>…​]

Description

Считывает предоставленные выходные данные различий (т. е. "a patch" ) и применяет их к файлам. При запуске из подкаталога репозитория исправленные пути за пределами каталога игнорируются. С опцией --index патч также применяется к индексу, а с опцией --cached патч применяется только к индексу. Без этих параметров команда применяет исправление только к файлам и не требует, чтобы они находились в репозитории Git .

Эта команда применяет исправление, но не создает фиксацию. Используйте git-am[1] для создания коммитов из исправлений, созданных git-format-patch[1] и/или полученных по электронной почте.

Options

<patch>…

Файлы, из которых нужно прочитать патч. - можно использовать для чтения со стандартного ввода.

--stat

Вместо применения патча выведите diffstat на вход. Выключается "apply" .

--numstat

Похож на --stat , но показывает количество добавленных и удаленных строк в десятичной записи и путь без сокращений, чтобы сделать его более удобным для машины. Для двоичных файлов выводит два - вместо 0 0 . Выключает "apply" .

--summary

Вместо применения исправления выведите сжатую сводку информации, полученной из расширенных заголовков различий git , например создания, переименования и изменения режима. Выключается "apply" .

--check

Вместо применения исправления проверьте, применимо ли исправление к текущему рабочему дереву и/или индексному файлу и обнаруживает ли оно ошибки. Выключается "apply" .

--index

Примените исправление как к индексу, так и к рабочему дереву (или просто убедитесь, что оно будет корректно применяться к обоим, если действует --check ). Обратите внимание, что --index ожидает, что записи индекса и копии рабочего дерева для соответствующих путей будут идентичными (их содержимое и метаданные, такие как файловый режим, должны совпадать), и выдаст ошибку, если это не так, даже если исправление будет применено как к индексу, так и к файлу. и рабочее дерево изолированно.

--cached

Примените патч только к индексу, не затрагивая рабочее дерево. Если действует --check , просто убедитесь, что он правильно применяется к записи индекса.

--intent-to-add

При применении патча только к рабочему дереву отмечайте новые файлы, которые будут добавлены в индекс позже (см. опцию --intent-to-add в git-add[1] ). Этот параметр игнорируется, если он не работает в репозитории Git и не указан --index . Обратите внимание, что --index может подразумеваться другими параметрами, такими как --cached или --3way .

-3
--3way

Попытайтесь трехстороннего слияния, если исправление записывает идентификаторы больших двоичных объектов, к которым оно должно применяться, и эти большие двоичные объекты доступны локально, возможно, оставив маркеры конфликта в файлах в рабочем дереве, чтобы пользователь мог их разрешить. Эта опция подразумевает опцию --index , если не используется опция --cached , и несовместима с опцией --reject . При использовании с опцией --cached любые конфликты остаются на более высоких уровнях кэша.

--build-fake-ancestor=<file>

В более новых выходных данных git diff для каждого большого двоичного объекта встроен index information , чтобы помочь определить исходную версию, к которой применяется исправление. Если задан этот флаг и если исходные версии больших двоичных объектов доступны локально, создается временный индекс, содержащий эти большие двоичные объекты.

Когда происходит чистое изменение режима (которое не имеет информации об индексе), вместо этого информация считывается из текущего индекса.

-R
--reverse

Примените пластырь в обратном порядке.

--reject

Что касается атомарности, git apply по умолчанию отклоняет весь патч и не затрагивает рабочее дерево, когда некоторые фрагменты не применяются. Эта опция позволяет применять применимые части патча и оставлять отклоненные фрагменты в соответствующих файлах *.rej.

-z

Если указан --numstat , не меняйте имена путей, а используйте машиночитаемый формат с завершающим NUL.

Без этой опции имена путей с символами "unusual" заключаются в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ).

-p<n>

Удалите <n> компонентов начального пути (разделенных косой чертой) из традиционных путей сравнения. Например, в случае с -p2 исправление против a/dir/file будет применено непосредственно к file . Значение по умолчанию — 1.

-C<n>

Убедитесь, что не менее <n> строк окружающего контекста совпадают до и после каждого изменения. Когда существует меньше строк окружающего контекста, все они должны совпадать. По умолчанию ни один контекст не игнорируется.

--unidiff-zero

По умолчанию git apply ожидает, что применяемое исправление представляет собой унифицированный файл различий хотя бы с одной строкой контекста. Это обеспечивает хорошие меры безопасности, но не работает при применении разницы, сгенерированной с помощью --unified=0 . Чтобы обойти эти проверки, используйте --unidiff-zero .

Обратите внимание: по причинам, указанным выше, использование контекстно-свободных патчей не рекомендуется.

--apply

Если вы используете любую из опций, отмеченных выше "Turns off apply " , git apply считывает и выводит запрошенную информацию без фактического применения исправления. Установите этот флаг после этих флагов, чтобы также применить исправление.

--no-add

Применяя патч, игнорируйте дополнения, внесенные им. Это можно использовать для извлечения общей части между двумя файлами, сначала запустив к ним diff и применив результат с этой опцией, которая будет применять часть удаления, но не часть добавления.

--allow-binary-replacement
--binary

Исторически мы не разрешали применение двоичных исправлений без явного разрешения пользователя, и этот флаг был способом сделать это. В настоящее время мы всегда разрешаем применение двоичных исправлений, поэтому это недопустимо.

--exclude=<path-pattern>

Не применять изменения к файлам, соответствующим заданному шаблону пути. Это может быть полезно при импорте наборов обновлений, когда вы хотите исключить определенные файлы или каталоги.

--include=<path-pattern>

Применить изменения к файлам, соответствующим заданному шаблону пути. Это может быть полезно при импорте наборов обновлений, в которые вы хотите включить определенные файлы или каталоги.

При использовании шаблонов --exclude и --include они проверяются в том порядке, в котором они появляются в командной строке, и первое совпадение определяет, используется ли исправление для каждого пути. Исправление для пути, который не соответствует ни одному шаблону включения/исключения, используется по умолчанию, если в командной строке нет шаблона включения, и игнорируется, если какой-либо шаблон включения имеется.

--ignore-space-change
--ignore-whitespace

При применении патча игнорируйте изменения в пробелах в контекстных строках, если это необходимо. Строки контекста сохранят свои пробелы и не будут подвергаться исправлению пробелов независимо от значения опции --whitespace . Однако новые строки все равно будут исправлены.

--whitespace=<action>

При применении исправления обнаруживайте новую или измененную строку, содержащую ошибки пробелов. То, что считается ошибкой пробелов, контролируется конфигурацией core.whitespace . По умолчанию конечные пробелы (включая строки, состоящие исключительно из пробелов) и символ пробела, за которым сразу следует символ табуляции внутри начального отступа строки, считаются ошибками пробелов.

По умолчанию команда выводит предупреждающие сообщения, но применяет исправление. Когда git-apply используется для статистики без применения исправлений, по умолчанию используется nowarn .

Вы можете использовать разные значения <action> для управления этим поведением:

  • nowarn отключает предупреждение о конце пробелов.

  • warn выводит предупреждения о некоторых таких ошибках, но применяет исправление «как есть» (по умолчанию).

  • fix выводит предупреждения для нескольких таких ошибок и применяет исправление после их исправления ( strip — это синоним — инструмент, используемый для рассмотрения только конечных пробельных символов как ошибок, и исправление включало их stripping , но современные Gits делают больше).

  • error выводит предупреждения о нескольких таких ошибках и отказывается применять исправление.

  • error-all аналогичен error , но показывает все ошибки.

--inaccurate-eof

При определенных обстоятельствах некоторые версии diff неправильно определяют отсутствие новой строки в конце файла. В результате патчи, созданные такими программами diff , некорректно записывают неполные строки. Этот параметр добавляет поддержку применения таких исправлений путем обхода этой ошибки.

-v
--verbose

Сообщайте о ходе выполнения в stderr. По умолчанию будет распечатано только сообщение о применении текущего патча. Эта опция приведет к сообщению дополнительной информации.

-q
--quiet

Подавить вывод stderr. Сообщения о статусе и прогрессе исправления не будут распечатаны.

--recount

Не доверяйте количеству строк в заголовках фрагментов, а делайте выводы, проверяя патч (например, после редактирования патча без соответствующей настройки заголовков фрагментов).

--directory=<root>

Добавляйте <root> ко всем именам файлов. Если также был передан аргумент "-p" , он применяется перед добавлением нового корня.

Например, патч, в котором говорится об обновлении a/git-gui.sh до b/git-gui.sh , можно применить к файлу в рабочем дереве modules/git-gui/git-gui.sh , запустив git apply --directory=modules/git-gui .

--unsafe-paths

По умолчанию патч, который влияет за пределы рабочей области (либо рабочее дерево, контролируемое Git , либо текущий рабочий каталог, когда "git apply" используется в качестве замены патча GNU), отклоняется как ошибка (или вред).

Когда git apply используется в качестве "better GNU patch" , пользователь может передать опцию --unsafe-paths , чтобы отменить эту проверку безопасности. Эта опция не действует при использовании --index или --cached .

--allow-empty

Не возвращайте ошибку для патчей, не содержащих различий. Сюда входят пустые патчи и патчи только с текстом фиксации.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

apply.ignoreWhitespace

Если установлено значение change , git apply сообщает git apply игнорировать изменения в пробелах так же, как опция --ignore-space-change . Если установлено одно из следующих значений: нет, нет, никогда, ложь, git apply сообщает, что нужно учитывать все различия в пробелах. См. git-apply[1] .

apply.whitespace

Сообщает git apply , как обрабатывать пробелы, так же, как опция --whitespace . См. git-apply[1] .

Submodules

Если патч содержит какие-либо изменения в субмодулях, то git apply обрабатывает эти изменения следующим образом.

Если указан --index (явно или неявно), то коммиты подмодуля должны точно соответствовать индексу, чтобы исправление можно было применить. Если какие-либо из подмодулей извлечены, то эти извлечения полностью игнорируются, т. е. они не обязаны быть актуальными или чистыми и не обновляются.

Если --index не указан, то коммиты подмодуля в патче игнорируются и проверяется и (если возможно) обновляется только отсутствие или наличие соответствующего подкаталога.
