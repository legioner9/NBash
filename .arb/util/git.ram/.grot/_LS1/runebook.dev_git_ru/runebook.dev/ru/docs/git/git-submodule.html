git-submodule

Name

git-submodule — Инициализировать, обновить или проверить подмодули

Synopsis

git submodule [--quiet] [--cached]
git submodule [--quiet] add [<options>] [--] <repository> [<path>]
git submodule [--quiet] status [--cached] [--recursive] [--] [<path>…​]
git submodule [--quiet] init [--] [<path>…​]
git submodule [--quiet] deinit [-f|--force] (--all|[--] <path>…​)
git submodule [--quiet] update [<options>] [--] [<path>…​]
git submodule [--quiet] set-branch [<options>] [--] <path>
git submodule [--quiet] set-url [--] <path> <newurl>
git submodule [--quiet] summary [<options>] [--] [<path>…​]
git submodule [--quiet] foreach [--recursive] <command>
git submodule [--quiet] sync [--recursive] [--] [<path>…​]
git submodule [--quiet] absorbgitdirs [--] [<path>…​]

Description

Проверяет, обновляет и управляет подмодулями.

Дополнительную информацию о субмодулях см. в gitsubmodules[7] .

Commands

Без аргументов показывает состояние существующих подмодулей. Для выполнения операций над подмодулями доступно несколько подкоманд.

add [-b <ветвь>] [-f|--force] [--name <имя>] [--reference <репозиторий>] [--глубина <глубина>] [--] <репозиторий> [<путь >]

Добавьте данный репозиторий в качестве подмодуля по заданному пути к набору изменений, который будет зафиксирован рядом с текущим проектом: текущий проект называется "superproject" .

<repository> — это URL-адрес исходного репозитория нового подмодуля. Это может быть либо абсолютный URL-адрес, либо (если он начинается с ./ или ../) местоположение относительно удаленного репозитория суперпроекта по умолчанию (обратите внимание, что для указания репозитория foo.git , который расположен рядом с суперпроектом bar.git , вам придется использовать ../foo.git вместо ./foo.git — как и следовало ожидать, следуя правилам для относительных URL-адресов, — поскольку оценка относительных URL-адресов в Git идентична оценке относительных каталогов).

Пультом по умолчанию является пульт ветки удаленного отслеживания текущей ветки. Если такой ветки удаленного отслеживания не существует или HEAD отключен, "origin" считается удаленным по умолчанию. Если в суперпроекте не настроен удаленный сервер по умолчанию, суперпроект является собственным авторитетным восходящим сервером, и вместо него используется текущий рабочий каталог.

Необязательный аргумент <путь> — это относительное расположение клонированного подмодуля в суперпроекте. Если <путь> не указан, используется каноническая часть репозитория исходного кода ( "repo" для "/path/to/repo.git" и "foo" для "host.xz:foo/.git" ). Если <path> существует и уже является действительным репозиторием Git , он готовится к фиксации без клонирования. <path> также используется в качестве логического имени субмодуля в его записях конфигурации, если только --name не используется для указания логического имени.

Данный URL-адрес записывается в .gitmodules для использования последующими пользователями, клонирующими суперпроект. Если URL-адрес указан относительно репозитория суперпроекта, предполагается, что репозитории суперпроекта и подмодуля будут храниться вместе в одном относительном месте, и необходимо указать только URL-адрес суперпроекта. git-submodule правильно найдет подмодуль, используя относительный URL-адрес в .gitmodules .

статус [--cached] [--recursive] [--] [<путь>…​]

Показать статус субмодулей. При этом будет распечатан SHA-1 текущего извлеченного коммита для каждого подмодуля, а также путь к подмодулю и выходные данные git describe для SHA-1. Каждый SHA-1, возможно, будет иметь префикс - , если подмодуль не инициализирован, + , если текущий извлеченный коммит подмодуля не соответствует SHA-1, найденному в индексе содержащего его репозитория, и U , если подмодуль имеет конфликты слияния.

Если указан --cached , эта команда вместо этого распечатает SHA-1, записанный в суперпроекте для каждого подмодуля.

Если указан --recursive , эта команда будет рекурсивно обращаться к вложенным подмодулям, а также показывать их статус.

Если вас интересуют только изменения инициализированных в данный момент подмодулей относительно фиксации, записанной в индексе или HEAD, git-status[1] и git-diff[1] также предоставят эту информацию (а также могут сообщать об изменениях в рабочем дереве подмодуля).

init [--] [<путь>…​]

Инициализируйте подмодули, записанные в индексе (которые были добавлены и зафиксированы в другом месте), установив submodule.$name.url в .git/config , используя в качестве шаблона ту же настройку из .gitmodules . Если URL-адрес является относительным, он будет разрешен с использованием удаленного устройства по умолчанию. Если удаленного хранилища по умолчанию нет, текущий репозиторий будет считаться вышестоящим.

Необязательные аргументы <path> ограничивают количество инициализируемых подмодулей. Если путь не указан и submodule.active настроен, подмодули, настроенные как активные, будут инициализированы, в противном случае инициализируются все подмодули.

Она также скопирует значение submodule.$name.update , если оно присутствует в файле .gitmodules , в .git/config , но (1) эта команда не изменяет существующую информацию в .git/config , и (2) submodule.$name.update , для которого задана пользовательская команда, не копируется в целях безопасности. причины.

Затем вы можете настроить URL-адреса клонирования подмодуля в .git/config для вашей локальной настройки и перейти к git submodule update ; вы также можете просто использовать git submodule update --init без явного шага init , если вы не собираетесь настраивать расположение каких-либо подмодулей.

См. подкоманду add для определения удаленного устройства по умолчанию.

deinit [-f|--force] (--all|[--] <путь>…​)

Отмените регистрацию данных подмодулей, т.е. удалите весь раздел submodule.$name из .git/config вместе с их рабочим деревом. Дальнейшие вызовы git submodule update , git submodule foreach и git submodule sync будут пропускать любые незарегистрированные подмодули до тех пор, пока они не будут инициализированы снова, поэтому используйте эту команду, если вы больше не хотите иметь локальную проверку подмодуля в вашем рабочем дереве.

Когда команда запускается без указания пути, она выдает ошибку вместо того, чтобы отменить все, чтобы предотвратить ошибки.

Если указан --force , рабочее дерево субмодуля будет удалено, даже если оно содержит локальные модификации.

Если вы действительно хотите удалить подмодуль из репозитория и зафиксировать его, используйте вместо него git-rm[1] . См. gitsubmodules[7] для вариантов удаления.

update [--init] [--remote] [-N|--no-fetch] [--[no-]recommend-shallow] [-f|--force] [--checkout|--rebase|- -merge] [--reference <репозиторий>] [--глубина <глубина>] [--recursive] [--jobs <n>] [--[no-]single-branch] [--filter <спецификация фильтра >] [--] [<путь>…​]

Обновите зарегистрированные подмодули, чтобы они соответствовали ожиданиям суперпроекта, клонируя отсутствующие подмодули, извлекая недостающие коммиты в подмодулях и обновляя рабочее дерево подмодулей. "updating" можно сделать несколькими способами в зависимости от параметров командной строки и значения переменной конфигурации submodule.<name>.update . Параметр командной строки имеет приоритет над переменной конфигурации. Если ни один из них не указан, выполняется checkout . (примечание: содержимое файла .gitmodules на данный момент не имеет значения; см. git submodule init выше, чтобы узнать, как используется .gitmodules ). Процедуры update , поддерживаемые как из командной строки, так и через конфигурацию submodule.<name>.update :

checkout

коммит, записанный в суперпроекте, будет извлечен в подмодуле на отдельном HEAD.

Если указан --force , подмодуль будет извлечен (с использованием git checkout --force ), даже если фиксация, указанная в индексе содержащего репозитория, уже соответствует фиксации, извлеченной в подмодуле.

rebase

текущая ветка подмодуля будет перебазирована на коммит, записанный в суперпроекте.

merge

коммит, записанный в суперпроекте, будет объединен с текущей веткой подмодуля.

Следующие процедуры обновления имеют дополнительные ограничения:

custom command

механизм запуска произвольных команд с идентификатором фиксации в качестве аргумента. В частности, если для переменной конфигурации submodule.<name>.update установлено значение !custom command , имя объекта фиксации, записанное в суперпроекте для подмодуля, добавляется к строке custom command и выполняется. Обратите внимание, что этот механизм не поддерживается ни в файле .gitmodules , ни в командной строке.

none

субмодуль не обновляется. Эту процедуру обновления нельзя использовать в командной строке.

Если субмодуль еще не инициализирован, и вы просто хотите использовать настройку, хранящуюся в .gitmodules , вы можете автоматически инициализировать субмодуль с помощью опции --init .

Если указан --recursive , эта команда будет рекурсивно обращаться к зарегистрированным подмодулям и обновлять все вложенные подмодули внутри.

Если указан --filter <filter spec> , к субмодулю будет применен данный фильтр частичного клонирования. Подробную информацию о характеристиках фильтра см. в git-rev-list[1] .

set-branch (-b|--branch) <ветвь> [--] <путь>
set-branch (-d|--default) [--] <путь>

Устанавливает ветку удаленного отслеживания по умолчанию для подмодуля. Опция --branch позволяет указать удаленную ветвь. Опция --default удаляет конфигурационный ключ submodule.<name>.branch, в результате чего в качестве ветки отслеживания по умолчанию используется удаленный HEAD .

set-url [--] <путь> <новый URL>

Устанавливает URL-адрес указанного подмодуля в <newurl>. Затем он автоматически синхронизирует новую конфигурацию удаленного URL-адреса подмодуля.

summary [--cached|--files] [(-n|--summary-limit) <n>] [commit] [--] [<путь>…​]

Показать сводку коммитов между данным коммитом (по умолчанию HEAD) и рабочим деревом/индексом. Для рассматриваемого подмодуля отображается серия коммитов в подмодуле между данным коммитом суперпроекта и индексом или рабочим деревом (переключаемым --cached ). Если указана опция --files , показывает серию коммитов в подмодуле между индексом суперпроекта и рабочим деревом подмодуля (эта опция не позволяет использовать опцию --cached или предоставлять явную фиксацию).

Использование опции --submodule=log с git-diff[1] также предоставит эту информацию.

foreach [--recursive] <команда>

Вычисляет произвольную команду оболочки в каждом извлеченном подмодуле. Команда имеет доступ к переменным $name, $sm_path, $displaypath, $sha1 и $toplevel: $name — имя соответствующего раздела подмодуля в .gitmodules , $sm_path — путь к подмодулю, записанный в непосредственном суперпроекте, $displaypath содержит относительный путь от текущего рабочего каталога до корневого каталога подмодулей, $sha1 — это фиксация, записанная в непосредственном суперпроекте, а $toplevel — это абсолютный путь к верхнему уровню непосредственного суперпроекта. Обратите внимание: во избежание конфликтов с $PATH в Windows переменная $path теперь является устаревшим синонимом переменной $sm_path . Любые подмодули, определенные в суперпроекте, но не извлеченные, игнорируются этой командой. Если не указано --quiet , foreach печатает имя каждого подмодуля перед оценкой команды. Если задан --recursive , подмодули просматриваются рекурсивно (т.е. данная команда оболочки также оценивается во вложенных подмодулях). Ненулевой возврат команды в любом подмодуле приводит к прекращению обработки. Это можно обойти, добавив || : в конец команды.

В качестве примера команда ниже покажет путь и текущий зафиксированный коммит для каждого подмодуля:

git submodule foreach 'echo $sm_path `git rev-parse HEAD`'
sync [--recursive] [--] [<path>…​]

Синхронизирует настройку конфигурации удаленного URL-адреса подмодулей со значением, указанным в .gitmodules . Это повлияет только на те подмодули, у которых уже есть запись URL в .git/config (это тот случай, когда они инициализированы или добавлены недавно). Это полезно, когда URL-адреса подмодулей изменяются в восходящем направлении и вам необходимо соответствующим образом обновить локальные репозитории.

git submodule sync синхронизирует все субмодули, а git submodule sync -- A синхронизирует только субмодуль "A" .

Если указан --recursive , эта команда будет рекурсивно обращаться к зарегистрированным подмодулям и синхронизировать все вложенные подмодули внутри.

absorbgitdirs

Если каталог git субмодуля находится внутри субмодуля, переместите каталог git субмодуля в путь $GIT_DIR/modules его суперпроекта, а затем соедините каталог git и его рабочий каталог, установив core.worktree и добавив расширение . Файл git , указывающий на каталог git , встроенный в каталог git суперпроекта.

Репозиторий, который был клонирован независимо и позже добавлен как подмодуль или старые настройки, имеет каталог подмодулей git внутри подмодуля, а не встроен в каталог суперпроектов git .

По умолчанию эта команда является рекурсивной.

Options

-q
--quiet

Печатайте только сообщения об ошибках.

--progress

Эта опция действительна только для команд добавления и обновления. Статус выполнения сообщается в стандартном потоке ошибок по умолчанию, когда он подключен к терминалу, если не указан параметр -q. Этот флаг обеспечивает статус хода выполнения, даже если стандартный поток ошибок не направляется на терминал.

--all

Эта опция действительна только для команды deinit. Отмените регистрацию всех подмодулей в рабочем дереве.

-b <branch>
--branch <branch>

Ветка репозитория, которую нужно добавить как подмодуль. Имя ветки записывается как submodule.<name>.branch в .gitmodules для update --remote . Специальное значение . используется для указания того, что имя ветки в подмодуле должно совпадать с именем текущей ветки в текущем репозитории. Если опция не указана, по умолчанию используется удаленный HEAD .

-f
--force

Эта опция действительна только для команд добавления, деинитизации и обновления. При запуске добавления разрешите добавление игнорируемого в противном случае пути подмодуля. При запуске deinit рабочие деревья подмодуля будут удалены, даже если они содержат локальные изменения. При запуске обновления (эффективно только при процедуре извлечения) локальные изменения в подмодулях отбрасываются при переключении на другой коммит; и всегда запускайте операцию извлечения в подмодуле, даже если фиксация, указанная в индексе содержащего репозитория, соответствует фиксации, извлеченной в подмодуле.

--cached

Эта опция действительна только для команд состояния и сводки. Эти команды обычно используют фиксацию, найденную в подмодуле HEAD, но с этой опцией вместо этого используется фиксация, хранящаяся в индексе.

--files

Эта опция действительна только для сводной команды. Эта команда сравнивает фиксацию в индексе с фиксацией в подмодуле HEAD, когда используется эта опция.

-n
--summary-limit

Эта опция действительна только для сводной команды. Ограничьте размер сводки (общее количество коммитов). Установка 0 отключит сводку; отрицательное число означает неограниченное (по умолчанию). Это ограничение применяется только к измененным подмодулям. Размер всегда ограничен 1 для добавленных/удаленных/измененных подмодулей.

--remote

Эта опция действительна только для команды обновления. Вместо использования записанного SHA-1 суперпроекта для обновления подмодуля используйте статус ветки удаленного отслеживания подмодуля. Используемый пульт — это пульт филиала ( branch.<name>.remote ), по умолчанию — origin . По умолчанию используется удаленная ветвь HEAD , но имя ветки можно переопределить, установив параметр submodule.<name>.branch либо в .gitmodules , либо в .git/config (при этом .git/config имеет приоритет).

Это работает для любой из поддерживаемых процедур обновления ( --checkout , --rebase и т. д.). Единственное изменение — это источник целевого SHA-1. Например, submodule update --remote --merge объединит изменения исходного подмодуля с подмодулями, а submodule update --merge объединит изменения gitlink суперпроекта с подмодулями.

Чтобы обеспечить текущее состояние ветки отслеживания, update --remote извлекает удаленный репозиторий подмодуля перед вычислением SHA-1. Если вы не хотите получать данные, вам следует использовать submodule update --remote --no-fetch .

Используйте эту опцию, чтобы интегрировать изменения из вышестоящего подпроекта с текущим HEAD вашего подмодуля. В качестве альтернативы вы можете запустить git pull из подмодуля, что эквивалентно, за исключением имени удаленной ветки: update --remote использует восходящий репозиторий по умолчанию и submodule.<name>.branch , а git pull использует branch.<name>.merge подмодуля. Предпочитайте submodule.<name>.branch , если вы хотите распространять восходящую ветвь по умолчанию с суперпроектом, и branch.<name>.merge , если вы хотите более естественное ощущение при работе с самим подмодулем.

-N
--no-fetch

Эта опция действительна только для команды обновления. Не извлекайте новые объекты с удаленного сайта.

--checkout

Эта опция действительна только для команды обновления. Извлеките коммит, записанный в суперпроекте на отдельном HEAD в подмодуле. Это поведение по умолчанию. Основное использование этой опции — переопределить submodule.$name.update , если для него установлено значение, отличное от checkout . Если ключ submodule.$name.update не задан явно или имеет значение checkout , этот параметр является неявным.

--merge

Эта опция действительна только для команды обновления. Объедините коммит, записанный в суперпроекте, с текущей веткой подмодуля. Если указана эта опция, HEAD подмодуля не будет отсоединен. Если сбой слияния препятствует этому процессу, вам придется разрешать возникшие конфликты внутри подмодуля с помощью обычных инструментов разрешения конфликтов. Если для ключа submodule.$name.update установлено значение merge , эта опция является неявной.

--rebase

Эта опция действительна только для команды обновления. Перебазируйте текущую ветку на фиксацию, записанную в суперпроекте. Если указана эта опция, HEAD подмодуля не будет отсоединен. Если ошибка слияния препятствует этому процессу, вам придется устранить эти ошибки с помощью git-rebase[1] . Если для ключа submodule.$name.update установлено значение rebase , этот параметр является неявным.

--init

Эта опция действительна только для команды обновления. Перед обновлением инициализируйте все субмодули, для которых "git submodule init" еще не вызывался.

--name

Эта опция действительна только для команды добавления. Он устанавливает имя подмодуля в заданную строку вместо значения по умолчанию его пути. Имя должно быть допустимым как имя каталога и не может заканчиваться на / .

--reference <repository>

Эта опция действительна только для команд добавления и обновления. Этим командам иногда необходимо клонировать удаленный репозиторий. В этом случае эта опция будет передана команде git-clone[1] .

ПРИМЕЧАНИЕ. Не используйте эту опцию, если вы внимательно не прочитали примечания к опциям --reference , --shared и --dissociate для git-clone[1] .

--dissociate

Эта опция действительна только для команд добавления и обновления. Этим командам иногда необходимо клонировать удаленный репозиторий. В этом случае эта опция будет передана команде git-clone[1] .

ПРИМЕЧАНИЕ: см. ПРИМЕЧАНИЕ для опции --reference .

--recursive

Этот параметр действителен только для команд foreach, update, status и sync. Обход подмодулей рекурсивно. Операция выполняется не только в подмодулях текущего репозитория, но также во всех вложенных подмодулях внутри этих подмодулей (и так далее).

--depth

Эта опция действительна для команд добавления и обновления. Создайте клон shallow с историей, усеченной до указанного количества ревизий. См. git-clone[1] .

--[no-]recommend-shallow

Эта опция действительна только для команды обновления. Первоначальный клон субмодуля будет использовать рекомендованный submodule.<name>.shallow , предоставленный файлом .gitmodules по умолчанию. Чтобы игнорировать предложения, используйте --no-recommend-shallow .

-j <n>
--jobs <n>

Эта опция действительна только для команды обновления. Клонируйте новые подмодули параллельно с таким же количеством заданий. По умолчанию используется опция submodule.fetchJobs .

--[no-]single-branch

Эта опция действительна только для команды обновления. Клонируйте только одну ветку во время обновления: HEAD или ту, которая указана в --branch.

<path>…

Пути к подмодулям. Если указано это, команда будет работать только с подмодулями, найденными по указанным путям. (Этот аргумент необходим при добавлении).

Files

При инициализации подмодулей файл .gitmodules в каталоге верхнего уровня содержащего репозитория используется для поиска URL-адреса каждого подмодуля. Этот файл должен быть отформатирован так же, как $GIT_DIR/config . Ключ к URL-адресу каждого подмодуля — "submodule.$name.url" . Подробности см. в gitmodules[5] .
