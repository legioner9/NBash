git-range-diff

Name

git-range-diff — сравнить два диапазона фиксации (например, две версии ветки)

Synopsis

git range-diff [--color=[<when>]] [--no-color] [<diff-options>]
        [--no-dual-color] [--creation-factor=<factor>]
        [--left-only | --right-only]
        ( <range1> <range2> | <rev1>…​<rev2> | <base> <rev1> <rev2> )
        [[--] <path>…​]

Description

Эта команда показывает различия между двумя версиями серии патчей или, в более общем смысле, двумя диапазонами коммитов (игнорируя коммиты слияния).

При наличии аргументов <path> эти диапазоны фиксации соответственно ограничиваются.

Для этого он сначала находит пары коммитов из обоих диапазонов коммитов, которые соответствуют друг другу. Говорят, что два коммита соответствуют, когда разница между их патчами (т.е. информация об авторе, сообщение о коммите и разница коммита) достаточно мала по сравнению с размером патчей. Подробности см. в разделе «Алгоритм» ниже.

Наконец, список совпадающих коммитов отображается в порядке второго диапазона коммитов, при этом несовпадающие коммиты вставляются сразу после того, как показаны все их предки.

Существует три способа указать диапазоны фиксации:

  • <range1> <range2> : Любой диапазон фиксации может иметь форму <base>..<rev> , <rev>^! или <rev>^-<n> . Более подробную информацию см. в SPECIFYING RANGES в gitrevisions[7] .

  • <rev1>...<rev2> . Это эквивалентно <rev2>..<rev1> <rev1>..<rev2> .

  • <base> <rev1> <rev2> : эквивалент <base>..<rev1> <base>..<rev2> .

Options

--no-dual-color

Если различия коммитов различаются, git range-diff воссоздает цвет исходных различий и добавляет внешние маркеры различий -/+ с красным/зеленым фоном, чтобы было легче увидеть, например, когда произошло изменение в том, какие именно строки были добавлены.

Кроме того, строки различий фиксации, которые присутствуют только в первом диапазоне фиксации, отображаются "dimmed" (это можно переопределить с помощью параметра конфигурации color.diff.<slot> , где <slot> — один из contextDimmed , oldDimmed и newDimmed ), а строки различий фиксации, которые присутствуют только в второй диапазон фиксации выделен жирным шрифтом (который можно переопределить с помощью настроек конфигурации color.diff.<slot> , где <slot> является одним из contextBold , oldBold или newBold ).

Он известен range-diff как "dual coloring" . Используйте --no-dual-color , чтобы вернуться к раскрашиванию всех линий в соответствии с внешними маркерами различий (и полностью игнорировать внутренние различия, когда дело касается цвета).

--creation-factor=<percent>

Установите коэффициент неточности стоимости создания/удаления на <percent> . По умолчанию — 60. Попробуйте использовать большее значение, если git range-diff ошибочно считает большое изменение полной перезаписью (удаление одного коммита и добавление другого), и меньшее — в обратном случае. См. раздел «Алгоритм» ниже, где объясняется, почему это необходимо.

--left-only

Подавлять фиксации, отсутствующие в первом указанном диапазоне (или "left range" при использовании формата <rev1>...<rev2> ).

--right-only

Подавлять фиксации, отсутствующие во втором указанном диапазоне (или "right range" при использовании формата <rev1>...<rev2> ).

--[no-]notes[=<ref>]

Этот флаг передается программе git log (см. git-log[1] ), которая генерирует исправления.

<range1> <range2>

Сравните фиксации, указанные в двух диапазонах, где <range1> считается более старой версией <range2> .

<rev1>…​<rev2>

Эквивалент прохождения <rev2>..<rev1> и <rev1>..<rev2> .

<база> <версия1> <версия2>

Эквивалент прохождения <base>..<rev1> и <base>..<rev2> . Обратите внимание, что <base> не обязательно должен быть точной точкой ветвления. Пример: после перебазирования ветки my-topic git range-diff my-topic@{u} my-topic@{1} my-topic покажет различия, возникшие в результате перебазирования.

git range-diff также принимает обычные параметры дифференциала (см. git-diff[1] ), особенно параметры --color=[<when>] и --no-color . Эти параметры используются при генерации "diff between patches" , т. е. для сравнения автора, сообщения коммита и различий соответствующих старых/новых коммитов. В настоящее время нет возможности настроить большинство параметров различий, передаваемых в git log при создании этих патчей.

Output stability

Вывод команды range-diff может быть изменен. Он предназначен для удобочитаемого фарфорового вывода, а не для использования в разных версиях Git для получения стабильного текста range-diff (в отличие от чего-то вроде опции --stable для git-patch-id[1] ). Для range-diff также не существует эквивалента git-apply[1] , выходные данные не предназначены для машинного считывания.

Особенно это касается true при передаче опций сравнения. В настоящее время некоторые опции, такие как --stat , могут в качестве возникающего эффекта выдавать выходные данные, которые совершенно бесполезны в контексте range-diff . Будущие версии range-diff могут научиться интерпретировать такие параметры способом, специфичным для range-diff (например, для --stat , выдающего удобочитаемый вывод, который суммирует изменения в diffstat).

Configuration

Эта команда использует настройки diff.color.* и pager.range-diff (последний включен по умолчанию). См. git-config[1] .

Examples

Если при перебазировании необходимо разрешить конфликты слияния, сразу после этого сравните изменения, внесенные при перебазировании, используя:

$ git range-diff @{u} @{1} @

Типичный вывод git range-diff будет выглядеть так:

-:  ------- > 1:0ddba11 Готовьтесь к неизбежному!
1:  c0debee = 2:  cab005e Add a helpful message at the start
2:  f00dbal ! 3:  decafe1 Describe a bug
    @@ -1,3 +1,3 @@
     Author: A U Thor <author@example.com>

    -TODO: Describe a bug
    +Describe a bug
    @@ -324,5 +324,6
      This is expected.

    -+What is unexpected is that it will also crash.
    ++Unexpectedly, it also crashes. This is a bug, and the jury is
    ++still out there how to fix it best. See ticket #314 for details.

      Contact
3:  bedead < -:  ------- TO-UNDO

В этом примере есть 3 старых и 3 новых коммита, причем разработчик удалил третий, добавил новый перед первыми двумя и изменил сообщение о коммите второго коммита, а также его разницу.

Когда вывод поступает на терминал, он по умолчанию имеет цветовую маркировку, как и обычный вывод git diff . Кроме того, первая строка (добавление фиксации) — зеленая, последняя строка (удаление фиксации) — красная, вторая строка (с идеальным совпадением) — желтая, как заголовок фиксации выходных данных git show , а третья строка окрашивает старый коммит в красный цвет, новый — в зеленый, а остальное — как в заголовок фиксации git show .

Однако наивную цветовую кодировку различий на самом деле немного сложно читать, поскольку она окрашивает все строки в красный или зеленый цвет. Например, строка, добавляющая "What is unexpected" в старый коммит, полностью красная, даже если целью старого коммита было что-то добавить.

Чтобы помочь в этом, range по умолчанию использует режим --dual-color . В этом режиме разница между различиями сохранит исходные цвета различий и будет снабжена префиксами строк с маркерами -/+ с красным или зеленым фоном, чтобы было более очевидно, что они описывают, как изменился сам разницу.

Algorithm

Общая идея такова: мы генерируем матрицу стоимости между коммитами в обоих диапазонах коммитов, а затем решаем назначение с наименьшей стоимостью.

Матрица стоимости заполняется следующим образом: для каждой пары коммитов генерируются оба различия и генерируется "diff of diffs" с 3 контекстными строками, затем количество строк в этом различии используется в качестве стоимости.

Чтобы избежать положительных результатов false (например, когда исправление было удалено, а несвязанное исправление было добавлено между двумя итерациями одной и той же серии исправлений), матрица затрат расширяется, чтобы учесть это, путем добавления записей с фиксированной стоимостью для оптовых удалений. добавляет.

Пример: пусть коммиты 1--2 будут первой итерацией серии патчей, а A--C — второй итерацией. Предположим, что A — это вариант 2, , а C — вариант 1 , но с небольшой модификацией (скажем, исправленной опечаткой). Визуализируйте коммиты в виде двудольного графа:

    1            A

    2            B

                 C

Ищем "best" объяснение новой серии на языке старой. Мы можем представить "explanation" как ребро на графике:

    1            A
               /
    2 --------' Б

                 C

Это объяснение относится к "free" , поскольку изменений не произошло. Подобным образом C можно объяснить с помощью 1 , но это потребует некоторых затрат c>0 из-за модификации:

    1 ----.      A
          |    /
    2 ----+---'  B
          |
          `----- C
          c>0

С математической точки зрения, мы ищем своего рода двустороннее сопоставление с минимальной стоимостью; 1 сопоставляется с C за определенную плату и т. д. Базовый граф на самом деле представляет собой полный двудольный граф; стоимость, которую мы связываем с каждым ребром, — это размер разницы между патчами двух коммитов. Чтобы объяснить новые коммиты, мы вводим фиктивные узлы с обеих сторон:

    1 ----.      A
          |    /
    2 ----+---'  B
          |
    o     `----- C
          c>0
    o            o

    o            o

Стоимость ребра o--C — это размер разницы C , модифицированный коэффициентом неточности, который должен быть меньше 100%. Стоимость лезвия o--o бесплатно. Коэффициент выдумки необходим, потому что даже если 1 и C не имеют ничего общего, они все равно могут иметь несколько пустых строк и тому подобное, что, возможно, делает назначение 1--C , o--o немного дешевле, чем 1--o , o--C , даже если 1 и C не имеют ничего общего. При использовании фактора выдумки нам требуется гораздо большая общая часть, чтобы считать патчи соответствующими.

Общее время, необходимое для вычисления этого алгоритма, равно времени, необходимому для вычисления n+m различий фиксации, а затем n*m различий патчей, плюс время, необходимое для вычисления назначения с наименьшей стоимостью между n и m различиями. Git использует реализацию алгоритма Джонкера-Вольгенанта для решения задачи назначения, сложность выполнения которой имеет кубическую сложность. Соответствие, найденное в этом случае, будет выглядеть следующим образом:

    1 ----. А
          |    /
    2 ----+---' Б
       .--+-----'
    o -'  `----- C
          c>0
    o ---------- o

    o ---------- o
