git-fetch

Name

git-fetch — загрузить объекты и ссылки из другого репозитория.

Synopsis

git fetch [<options>] [<repository> [<refspec>…​]]
git fetch [<options>] <group>
git fetch --multiple [<options>] [(<repository> | <group>)…​]
git fetch --all [<options>]

Description

Извлекайте ветки и/или теги (в совокупности "refs" ) из одного или нескольких других репозиториев вместе с объектами, необходимыми для завершения их истории. Обновлены ветки удаленного отслеживания (способы управления этим поведением см. в описании <refspec> ниже).

По умолчанию любой тег, указывающий на извлекаемую историю, также извлекается; эффект заключается в получении тегов, указывающих на интересующие вас ветки. Это поведение по умолчанию можно изменить, используя параметры --tags или --no-tags или настроив Remote.<name>.tagOpt. Используя спецификацию ссылок, которая явно извлекает теги, вы также можете получать теги, которые не указывают на интересующие вас ветки.

git fetch может выполнять выборку либо из одного именованного репозитория или URL-адреса, либо из нескольких репозиториев одновременно, если задан <group> и в файле конфигурации есть запись Remotes.<group>. (См. git-config[1] ).

Если удаленный доступ не указан, по умолчанию будет использоваться удаленный origin , если только для текущей ветви не настроена восходящая ветвь.

Имена извлекаемых ссылок вместе с именами объектов, на которые они указывают, записываются в .git/FETCH_HEAD . Эта информация может использоваться сценариями или другими командами git , такими как git-pull[1] .

Options

--all

Соберите все пульты.

-a
--append

Добавьте имена ссылок и имена объектов полученных ссылок к существующему содержимому .git/FETCH_HEAD . Без этой опции старые данные в .git/FETCH_HEAD будут перезаписаны.

--atomic

Используйте атомарную транзакцию для обновления локальных ссылок. Либо все ссылки обновляются, либо в случае ошибки ни одна ссылка не обновляется.

--depth=<depth>

Ограничьте выборку указанным количеством коммитов из кончика истории каждой удаленной ветки. При извлечении в репозиторий shallow , созданный git clone с опцией --depth=<depth> (см. git-clone[1] ), углубите или сократите историю до указанного количества коммитов. Теги для углубленных коммитов не извлекаются.

--deepen=<depth>

Аналогично --Deep, за исключением того, что он определяет количество коммитов от текущей мелкой границы, а не от кончика истории каждой удаленной ветки.

--shallow-since=<date>

Углубите или сократите историю мелкого репозитория, чтобы включить все доступные коммиты после <дата>.

--shallow-exclude=<revision>

Углубите или сократите историю мелкого репозитория, чтобы исключить коммиты, доступные из указанной удаленной ветки или тега. Эту опцию можно указать несколько раз.

--unshallow

Если исходный репозиторий завершен, преобразуйте поверхностный репозиторий в полный, устранив все ограничения, налагаемые мелкими репозиториями.

Если исходный репозиторий неглубокий, извлеките как можно больше данных, чтобы текущий репозиторий имел ту же историю, что и исходный репозиторий.

--update-shallow

По умолчанию при загрузке из мелкого репозитория git fetch отклоняет ссылки, требующие обновления .git/shallow. Эта опция обновляет .git/shallow и принимает такие ссылки.

--negotiation-tip=<commit|glob>

По умолчанию Git будет сообщать серверу о коммитах, доступных со всех локальных ссылок, чтобы найти общие коммиты и попытаться уменьшить размер получаемого пакетного файла. Если указано, Git будет сообщать только о коммитах, доступных из данных советов. Это полезно для ускорения выборки, когда пользователь знает, какая локальная ссылка, вероятно, будет иметь общие фиксации с извлекаемой исходной ссылкой.

Эту опцию можно указать более одного раза; если да, то Git сообщит о коммитах, доступных из любого из заданных коммитов.

Аргументом этой опции может быть обобщение имен ссылок, ссылка или (возможно, сокращенный) SHA-1 коммита. Указание glob эквивалентно указанию этой опции несколько раз, по одному для каждого соответствующего имени ссылки.

См. также переменные конфигурации fetch.negotiationAlgorithm и push.negotiate , описанные в git-config[1] , и опцию --negotiate-only ниже.

--negotiate-only

Ничего не извлекайте с сервера, а вместо этого печатайте предки предоставленных аргументов --negotiation-tip=* , которые у нас есть общие с сервером.

Это несовместимо с --recurse-submodules=[yes|on-demand] . Внутри это используется для реализации опции push.negotiate , см. git-config[1] .

--dry-run

Покажите, что будет сделано, не внося никаких изменений.

--porcelain

Выведите выходные данные на стандартный вывод в удобном для анализа формате для сценариев. Подробности см. в разделе ВЫХОД в git-fetch[1] .

Это несовместимо с --recurse-submodules=[yes|on-demand] и имеет приоритет над параметром конфигурации fetch.output .

--[no-]write-fetch-head

Запишите список удаленных ссылок, полученных в файле FETCH_HEAD , непосредственно под $GIT_DIR . Это значение по умолчанию. Передача --no-write-fetch-head из командной строки сообщает Git не записывать файл. При использовании опции --dry-run файл никогда не записывается.

-f
--force

Когда git fetch используется со спецификацией <src>:<dst> , он может отказаться обновлять локальную ветку, как описано в разделе <refspec> ниже. Эта опция отменяет эту проверку.

-k
--keep

Сохраните скачанный пакет.

--multiple

Разрешить указывать несколько аргументов <repository> и <group>. Нельзя указывать <refspec>.

--[no-]auto-maintenance
--[no-]auto-gc

В конце запустите git maintenance run --auto , чтобы при необходимости выполнить автоматическое обслуживание репозитория. ( --[no-]auto-gc — синоним.) По умолчанию включено.

--[no-]write-commit-graph

Напишите граф фиксации после извлечения. Это переопределяет настройку конфигурации fetch.writeCommitGraph .

--prefetch

Измените настроенную спецификацию ссылок, чтобы поместить все ссылки в пространство имен refs/prefetch/ . См. задачу prefetch в git-maintenance[1] .

-p
--prune

Перед извлечением удалите все ссылки на удаленное отслеживание, которые больше не существуют на удаленном компьютере. Теги не подлежат сокращению, если они извлекаются только из-за автоматического следования тегам по умолчанию или из-за опции --tags. Однако если теги извлекаются из-за явной спецификации ссылки (либо в командной строке, либо в удаленной конфигурации, например, если удаленный сервер был клонирован с помощью опции --mirror), то они также подлежат сокращению. Предоставление --prune-tags — это сокращение для предоставления тега refspec.

Более подробную информацию смотрите в разделе ОБРЕЗКА ниже.

-P
--prune-tags

Перед извлечением удалите все локальные теги, которых больше нет на удаленном компьютере, если включен --prune . Эту опцию следует использовать более осторожно, поскольку в отличие от --prune она удалит все созданные локальные ссылки (локальные теги). Эта опция является сокращением для предоставления явной спецификации тега refspec вместе с --prune , см. обсуждение этого в документации.

Более подробную информацию смотрите в разделе ОБРЕЗКА ниже.

-n
--no-tags

По умолчанию теги, указывающие на объекты, загруженные из удаленного репозитория, извлекаются и сохраняются локально. Эта опция отключает автоматическое отслеживание тегов. Поведение по умолчанию для удаленного устройства можно указать с помощью параметра Remote.<name>.tagOpt. См. git-config[1] .

--refetch

Вместо согласования с сервером во избежание передачи коммитов и связанных объектов, которые уже присутствуют локально, этот параметр извлекает все объекты, как это сделал бы новый клон. Используйте это, чтобы повторно применить фильтр частичного клонирования из конфигурации или использовать --filter= , когда определение фильтра изменилось. Автоматическое обслуживание после выборки выполнит консолидацию пакетов базы данных объектов для удаления любых дубликатов объектов.

--refmap=<refspec>

При получении ссылок, перечисленных в командной строке, используйте указанную спецификацию ссылки (может быть задана более одного раза) для сопоставления ссылок с ветвями удаленного отслеживания вместо значений переменных конфигурации remote.*.fetch для удаленного репозитория. Если указать пустой <refspec> в параметре --refmap , Git будет игнорировать настроенные спецификации ссылок и полностью полагаться на спецификации ссылок, предоставленные в качестве аргументов командной строки. Подробности см. в разделе "Configured Remote-tracking Branches" .

-t
--tags

Извлеките все теги с удаленного устройства (т. е. извлеките удаленные теги refs/tags/* в локальные теги с тем же именем), а также все остальное, что в противном случае было бы получено. Использование только этой опции не приводит к сокращению тегов, даже если используется --prune (хотя теги в любом случае могут быть сокращены, если они также являются местом назначения явной спецификации ссылки; см. --prune ).

--recurse-submodules[=yes|on-demand|no]

Эта опция определяет, следует ли и при каких условиях получать новые коммиты подмодулей. При рекурсии через подмодули git fetch всегда пытается получить подмодули "changed" , то есть подмодуль, имеющий фиксации, на которые ссылается вновь полученная фиксация суперпроекта, но которые отсутствуют в локальном клоне подмодуля. Измененный субмодуль можно получить, пока он присутствует локально, например, в $GIT_DIR/modules/ (см. gitsubmodules[7] ); если восходящий поток добавляет новый субмодуль, этот субмодуль не может быть получен до тех пор, пока он не будет клонирован, например, с помощью git submodule update .

Если установлено значение on-demand , извлекаются только измененные подмодули. Если установлено значение yes , извлекаются все заполненные подмодули, а также незаполненные и измененные подмодули. Если установлено значение no , подмодули никогда не извлекаются.

Если не указано, используется значение fetch.recurseSubmodules , если оно установлено (см. git-config[1] ), по умолчанию используется значение on-demand , если оно не установлено. Если этот параметр используется без какого-либо значения, по умолчанию используется значение yes .

-j
--jobs=<n>

Количество параллельных дочерних элементов, которые будут использоваться для всех форм выборки.

Если была указана опция --multiple , разные пульты будут подключаться параллельно. Если выбрано несколько подмодулей, они будут выбраны параллельно. Для независимого управления ими используйте настройки конфига fetch.parallel и submodule.fetchJobs (см. git-config[1] ).

Обычно параллельная рекурсивная и многоудаленная выборка выполняется быстрее. По умолчанию выборка выполняется последовательно, а не параллельно.

--no-recurse-submodules

Отключите рекурсивную выборку подмодулей (это имеет тот же эффект, что и использование опции --recurse-submodules=no ).

--set-upstream

Если удаленное устройство получено успешно, добавьте ссылку на восходящий поток (отслеживание), используемую git-pull[1] без аргументов и другими командами. Для получения дополнительной информации см. branch.<name>.merge и branch.<name>.remote в git-config[1] .

--submodule-prefix=<path>

Добавляйте <path> к путям, напечатанным в информационных сообщениях, например "Fetching submodule foo" . Эта опция используется внутренне при рекурсии по подмодулям.

--recurse-submodules-default=[yes|on-demand]

Эта опция используется внутри, чтобы временно предоставить неотрицательное значение по умолчанию для опции --recurse-submodules. Все другие методы настройки рекурсии подмодуля fetch (например, настройки в gitmodules[5] и git-config[1] ) переопределяют эту опцию, как и непосредственное указание --[no-]recurse-submodules.

-u
--update-head-ok

По умолчанию git fetch отказывается обновлять заголовок, соответствующий текущей ветке. Этот флаг отключает проверку. Это предназначено исключительно для внутреннего использования git pull для связи с git fetch , и если вы не реализуете свой собственный Porcelain, вы не должны его использовать.

--upload-pack <upload-pack>

Если этот параметр задан и репозиторий, из которого осуществляется выборка, обрабатывается git fetch-pack , --exec=<upload-pack> передается команде, чтобы указать нестандартный путь для команды, запускаемой на другом конце.

-q
--quiet

Передайте --quiet в git -fetch-pack и отключите любые другие внутренние команды git . О ходе выполнения не сообщается в стандартный поток ошибок.

-v
--verbose

Be verbose.

--progress

Статус выполнения сообщается в стандартном потоке ошибок по умолчанию, когда он подключен к терминалу, если не указан параметр -q. Этот флаг обеспечивает статус хода выполнения, даже если стандартный поток ошибок не направляется на терминал.

-o <option>
--server-option=<option>

Передайте данную строку на сервер при обмене данными по протоколу версии 2. Данная строка не должна содержать символ NUL или LF. Обработка сервером параметров сервера, включая неизвестные, зависит от сервера. Если задано несколько --server-option=<option> , все они отправляются другой стороне в порядке, указанном в командной строке.

--show-forced-updates

По умолчанию git проверяет, обновляется ли ветка принудительно во время выборки. Это можно отключить с помощью fetch.showForcedUpdates, но опция --show-forced-updates гарантирует выполнение этой проверки. См. git-config[1] .

--no-show-forced-updates

По умолчанию git проверяет, обновляется ли ветка принудительно во время выборки. Передайте --no-show-forced-updates или установите для fetch.showForcedUpdates значение false , чтобы пропустить эту проверку из соображений производительности. Если параметр --ff-only используется во время git-pull , он по-прежнему будет проверять наличие принудительных обновлений перед попыткой быстрого обновления. См. git-config[1] .

-4
--ipv4

Используйте только адреса IPv4, игнорируя адреса IPv6.

-6
--ipv6

Используйте только адреса IPv6, игнорируя адреса IPv4.

<repository>

Репозиторий "remote" , который является источником операции выборки или извлечения. Этот параметр может быть либо URL-адресом (см. раздел GIT URLS ниже), либо именем удаленного устройства (см. раздел REMOTES ниже).

<group>

Имя, ссылающееся на список репозиториев как значение Remotes.<group> в файле конфигурации. (См. git-config[1] ).

<refspec>

Указывает, какие ссылки следует получить, а какие локальные ссылки обновить. Если в командной строке не отображаются <refspec>, ссылки для выборки считываются из переменных remote.<repository>.fetch (см. CONFIGURED REMOTE-TRACKING BRANCHES ниже).

Формат параметра <refspec> является необязательным, плюс + , за которым следует исходный <src>, затем двоеточие : , за которым следует ссылка назначения <dst>. Двоеточие можно опустить, если <dst> пуст. <src> обычно представляет собой ссылку, но также может быть полностью написанным шестнадцатеричным именем объекта.

<refspec> может содержать * в своем <src>, чтобы указать простое совпадение с шаблоном. Такая спецификация ссылки действует как glob, который соответствует любой ссылке с тем же префиксом. Шаблон <refspec> должен иметь * как в <src>, так и в <dst>. Он сопоставит ссылки с пунктом назначения, заменив * содержимым, совпадающим с источником.

Если перед спецификацией ссылки стоит ^ , она будет интерпретироваться как отрицательная спецификация. Вместо того, чтобы указывать, какие ссылки следует получить или какие локальные ссылки обновлять, такая спецификация вместо этого будет указывать ссылки, которые следует исключить. Ссылка будет считаться соответствующей, если она соответствует хотя бы одной положительной спецификации и не соответствует ни одной отрицательной спецификации. Отрицательные спецификации ссылок могут быть полезны для ограничения области применения спецификации шаблона, чтобы она не включала определенные ссылки. Отрицательные ссылки сами по себе могут быть ссылками на шаблоны. Однако они могут содержать только <src> и не указывать <dst>. Полностью написанные шестнадцатеричные имена объектов также не поддерживаются.

tag <tag> означает то же, что и refs/tags/<tag>:refs/tags/<tag> ; он запрашивает получение всего до данного тега.

Выбирается удаленная ссылка, соответствующая <src>, и если <dst> не является пустой строкой, предпринимается попытка обновить соответствующую ей локальную ссылку.

Разрешено ли это обновление без --force , зависит от пространства имен ref, в которое оно извлекается, типа извлекаемого объекта и того, считается ли обновление перемоткой вперед. Как правило, к выборке применяются те же правила, что и при отправке, см. раздел <refspec>... статьи git-push[1] , чтобы узнать, что это такое. Исключения из этих правил, характерные для git fetch , указаны ниже.

До Git версии 2.20, в отличие от git-push[1] , любые обновления refs/tags/* будут приниматься без + в refspec (или --force ). При загрузке мы беспорядочно считали все обновления тегов с удаленного устройства принудительной выборкой. Начиная с Git версии 2.20, получение обновления refs/tags/* работает так же, как и при принудительной отправке. Т.е. любые обновления будут отклонены без + в refspec (или --force ).

В отличие от отправки с помощью git-push[1] , любые обновления за пределами refs/{tags,heads}/* будут приняты без + в refspec (или --force ), независимо от того, заменяется ли это, например, объект дерева на большой двоичный объект или фиксация на другую фиксацию, которая не имеет предыдущей фиксации как предок и т. д.

В отличие от отправки с помощью git-push[1] , не существует конфигурации, которая бы изменила эти правила, и нет ничего похожего на перехватчик pre-fetch , аналогичный перехватчику pre-receive .

Как и в случае с git-push[1] , все описанные выше правила о том, что не разрешено в качестве обновления, можно переопределить, добавив необязательный ведущий + в refspec (или используя параметр командной строки --force ). Единственным исключением из этого правила является то, что никакие принудительные действия не заставят пространство имен refs/heads/* принять нефиксированный объект.

Note
 Когда известно, что удаленная ветвь, которую вы хотите получить, регулярно перематывается и перебазируется, ожидается, что ее новая подсказка не будет потомком предыдущей подсказки (которая хранится в вашей ветке удаленного отслеживания при последнем извлечении). Вы можете использовать знак + , чтобы указать, что для таких ветвей потребуются обновления без перемотки вперед. Невозможно определить или объявить, что ветка с таким поведением будет доступна в репозитории; извлекающий пользователь просто должен знать, что это ожидаемый шаблон использования ветки.
--stdin

Прочитайте спецификации ссылок, по одной на строку, из стандартного ввода в дополнение к тем, которые предоставлены в качестве аргументов. Формат "tag <name>" не поддерживается.

Git urls

Обычно URL-адреса содержат информацию о транспортном протоколе, адресе удаленного сервера и пути к репозиторию. В зависимости от транспортного протокола некоторая часть этой информации может отсутствовать.

Git поддерживает протоколы ssh, git, http и https (кроме того, для выборки можно использовать ftp и ftps, но это неэффективно и не рекомендуется; не используйте их).

Собственный транспорт (например, URL-адрес git://) не выполняет аутентификацию, и его следует использовать с осторожностью в незащищенных сетях.

С ними можно использовать следующие синтаксисы:

  • ssh://[user@]host.xz[:port]/path/to/repo.git/

  • git://host.xz[:port]/path/to/repo.git/

  • http[s]://host.xz[:port]/path/to/repo.git/

  • ftp[s]://host.xz[:port]/path/to/repo.git/

Альтернативный синтаксис, подобный scp, также можно использовать с протоколом ssh:

  • [user@]host.xz:path/to/repo.git/

Этот синтаксис распознается только в том случае, если перед первым двоеточием нет косой черты. Это помогает отличить локальный путь, содержащий двоеточие. Например, локальный путь foo:bar можно указать как абсолютный путь или ./foo:bar , чтобы его не интерпретировали как URL-адрес ssh.

Протоколы ssh и git дополнительно поддерживают расширение ~username:

  • ssh://[user@]host.xz[:port]/~[user]/path/to/repo.git/

  • git://host.xz[:port]/~[user]/path/to/repo.git/

  • [user@]host.xz:/~[user]/path/to/repo.git/

Для локальных репозиториев, которые также поддерживаются Git , могут использоваться следующие синтаксисы:

  • /path/to/repo.git/

  • file:///path/to/repo.git/

Эти два синтаксиса в основном эквивалентны, за исключением случаев клонирования, когда первый подразумевает параметр --local. Подробности см. в git-clone[1] .

git clone , git fetch и git pull , но не git push , также принимают подходящий файл пакета. См. git-bundle[1] .

Когда Git не знает, как обрабатывать определенный транспортный протокол, он пытается использовать удаленный помощник remote-<transport> , если таковой существует. Чтобы явно запросить удаленного помощника, можно использовать следующий синтаксис:

  • <transport>::<address>

где <адрес> может быть путем, сервером и путем или произвольной строкой типа URL-адреса, распознаваемой конкретным вызываемым удаленным помощником. Подробности см. в gitremote-helpers[7] .

Если существует большое количество удаленных репозиториев с одинаковыми именами и вы хотите использовать для них другой формат (например, чтобы используемые вами URL-адреса были перезаписаны в работающие URL-адреса), вы можете создать раздел конфигурации вида:

        [url "<actual url base>"]
                insteadOf = <other url base>

Например, с этим:

        [url "git://git.host.xz/"]
                insteadOf = host.xz:/path/to/
                insteadOf = work:

URL-адрес типа "work:repo.git" или "host.xz:/path/to/repo.git" будет перезаписан в любом контексте, который принимает URL-адрес "git://git.host.xz/repo.git" .

Если вы хотите переписать URL-адреса только для push-уведомлений, вы можете создать раздел конфигурации формы:

        [url "<actual url base>"]
                pushInsteadOf = <other url base>

Например, с этим:

        [url "ssh://example.org/"]
                pushInsteadOf = git://example.org/

URL-адрес типа "git://example.org/path/to/repo.git" будет перезаписан в "ssh://example.org/path/to/repo.git" для push-уведомлений, но при запросе по-прежнему будет использоваться исходный URL-адрес.

Remotes

Вместо URL-адреса в качестве аргумента <repository> можно использовать имя одного из следующих элементов:

  • пульт в файле конфигурации Git : $GIT_DIR/config ,

  • файл в каталоге $GIT_DIR/remotes или

  • файл в каталоге $GIT_DIR/branches .

Все они также позволяют опустить спецификацию ссылки в командной строке, поскольку каждая из них содержит спецификацию ссылки, которую git будет использовать по умолчанию.

Именованный удаленный компьютер в файле конфигурации

Вы можете указать имя пульта, который вы ранее настроили с помощью git-remote[1] , git-config[1] или даже вручную отредактировав файл $GIT_DIR/config . URL-адрес этого пульта будет использоваться для доступа к хранилищу. Спецификация этого пульта будет использоваться по умолчанию, если вы не укажете спецификацию в командной строке. Запись в файле конфигурации будет выглядеть так:

        [remote "<name>"]
                url = <URL>
                pushurl = <pushurl>
                push = <refspec>
                fetch = <refspec>

<pushurl> используется только для push-уведомлений. Это необязательно и по умолчанию — <URL> . Отправка на удаленный компьютер влияет на все определенные pushurl-адреса или на все определенные URL-адреса, если pushurl не определен. Однако Fetch будет выполнять выборку только с первого определенного URL-адреса, если определено несколько URL-адресов.

Именованный файл в $GIT_DIR/remotes

Вы можете указать имя файла в формате $GIT_DIR/remotes . URL-адрес в этом файле будет использоваться для доступа к репозиторию. Спецификация ссылки в этом файле будет использоваться по умолчанию, если вы не укажете спецификацию ссылки в командной строке. Этот файл должен иметь следующий формат:

        URL: one of the above URL formats
        Push: <refspec>
        Pull: <refspec>

Линии Push: используются git push , а линии Pull: используются git pull и git fetch . Для дополнительных сопоставлений ветвей можно указать несколько линий Push: и Pull: .

Именованный файл в $GIT_DIR/branches

Вы можете указать имя файла в формате $GIT_DIR/branches . URL-адрес в этом файле будет использоваться для доступа к репозиторию. Этот файл должен иметь следующий формат:

        <URL>#<head>

Требуется <URL> ; #<head> не является обязательным.

В зависимости от операции git будет использовать одну из следующих спецификаций, если вы не указали ее в командной строке. <branch> — это имя этого файла в $GIT_DIR/branches , а <head> по умолчанию — master .

Для выборки git используются:

        refs/heads/<head>:refs/heads/<branch>

git push использует:

        HEAD:refs/heads/<head>

Настроены ветки удаленного отслеживания

Вы часто взаимодействуете с одним и тем же удаленным репозиторием, регулярно и неоднократно извлекая данные из него. Чтобы отслеживать ход работы такого удаленного репозитория, git fetch позволяет настраивать переменные конфигурации remote.<repository>.fetch .

Обычно такая переменная может выглядеть так:

[remote "origin"]
        fetch = +refs/heads/*:refs/remotes/origin/*

Эта конфигурация используется двумя способами:

  • Когда git fetch запускается без указания в командной строке, какие ветки и/или теги следует извлечь, например git fetch origin или git fetch , значения remote.<repository>.fetch используются в качестве спецификаций ссылок — они определяют, какие ссылки следует получать, а какие локальные ссылки обновлять. В приведенном выше примере будут выбраны все ветки, существующие в origin (т. е. любая ссылка, соответствующая левой части значения, refs/heads/* ), и будут обновлены соответствующие ветки удаленного отслеживания в иерархии refs/remotes/origin/* .

  • Когда git fetch запускается с явными ветвями и/или тегами для выборки в командной строке, например git fetch origin master , <refspec>, заданный в командной строке, определяет, что должно быть извлечено (например, master в примере, который является сокращенным обозначением для master: , что, в свою очередь, означает "fetch the master branch but I do not explicitly say what remote-tracking branch to update with it from the command line" ), и команда примера выберет only ветку master . Значения remote.<repository>.fetch определяют, какая ветвь удаленного отслеживания (если таковая имеется) обновляется. При таком использовании значения remote.<repository>.fetch не влияют на выборку what (т. е. значения не используются в качестве спецификаций ссылок, когда в командной строке перечислены спецификации ссылок); они используются только для определения where , выбранные ссылки сохраняются, действуя как сопоставление.

Последнее использование значений remote.<repository>.fetch можно отменить, указав параметр(ы) --refmap=<refspec> в командной строке.

Pruning

Git по умолчанию сохраняет данные, если они явно не выброшены; это распространяется на сохранение локальных ссылок на ветки на удаленных устройствах, которые сами удалили эти ветки.

Если оставить их накапливаться, эти устаревшие ссылки могут ухудшить производительность в больших и загруженных репозиториях с большим количеством ветвей и, например, сделать вывод таких команд, как git branch -a --contains <commit> , излишне многословным, а также повлиять на все остальное, что будет работать с полной версией. набор известных ссылок.

Эти ссылки на удаленное отслеживание можно удалить единовременно одним из следующих способов:

# Во время получения
$ git fetch --prune <name>

# Только обрезка, а не выборка
$ git remote prune <name>

Чтобы сократить ссылки в рамках обычного рабочего процесса без необходимости помнить об этом, установите fetch.prune глобально или remote.<name>.prune для каждого удаленного устройства в конфигурации. См. git-config[1] .

Здесь все становится сложнее и конкретнее. Функция сокращения на самом деле не заботится о ветках, вместо этого она удаляет локальные ←→ удаленные ссылки в зависимости от спецификации удаленного устройства (см. <refspec> и CONFIGURED REMOTE-TRACKING BRANCHES выше).

Поэтому, если refspec для удаленного устройства включает, например, refs/tags/*:refs/tags/* , или вы запускаете вручную, например, git fetch --prune <name> "refs/tags/*:refs/tags/*" , удаляются не устаревшие ветки удаленного отслеживания, а любой локальный тег, которого нет на удаленном устройстве.

Это может быть не то, что вы ожидаете, т. е. вы хотите удалить удаленный <name> , но также явно извлечь из него теги, поэтому при извлечении из него вы удаляете все свои локальные теги, большинство из которых, возможно, не пришли с удаленного <name> в первое место.

Поэтому будьте осторожны при использовании этого со спецификацией ссылки, такой как refs/tags/*:refs/tags/* , или любой другой спецификацией ссылки, которая может сопоставлять ссылки с нескольких удаленных устройств в одно и то же локальное пространство имен.

Поскольку поддержание актуальности как ветвей, так и тегов на удаленном компьютере является распространенным вариантом использования, опция --prune-tags может поставляться вместе с --prune для удаления локальных тегов, которые не существуют на удаленном компьютере, и принудительного обновления тех тегов, которые различаются. Очистку тегов также можно включить с помощью fetch.pruneTags или remote.<name>.pruneTags в конфигурации. См. git-config[1] .

Опция --prune-tags эквивалентна объявлению refs/tags/*:refs/tags/* в характеристиках пульта дистанционного управления. Это может привести к некоторым, казалось бы, странным взаимодействиям:

# These both fetch tags
$ git fetch --no-tags origin 'refs/tags/*:refs/tags/*'
$ git fetch --no-tags --prune-tags происхождение

Причина, по которой он не выдает ошибку при предоставлении без --prune или его версий конфигурации, заключается в гибкости настроенных версий и поддержании сопоставления 1 = 1 между тем, что делают флаги командной строки, и тем, что делают версии конфигурации.

Разумно, например, настроить fetch.pruneTags=true в ~/.gitconfig так, чтобы теги обрезались при каждом запуске git fetch --prune , не делая при этом каждый вызов git fetch без --prune ошибкой.

Удаление тегов с помощью --prune-tags также работает при получении URL-адреса вместо именованного удаленного устройства. Они удалят все теги, не найденные в источнике:

$ git fetch origin --prune --prune-tags
$ git fetch origin --prune 'refs/tags/*:refs/tags/*'
$ git fetch <url of origin> --prune --prune-tags
$ git fetch <url of origin> --prune 'refs/tags/*:refs/tags/*'

Output

Выходные данные "git fetch" зависят от используемого метода транспортировки; в этом разделе описываются выходные данные при выборке по протоколу Git (локально или через ssh) и протоколу Smart HTTP.

Статус выборки выводится в табличной форме, где каждая строка представляет статус одной ссылки. Каждая строка имеет вид:

 <flag> <summary> <from> -> <to> [<reason>]

При использовании --porcelain выходной формат предназначен для машинного анализа. В отличие от удобочитаемых форматов вывода, он печатает в стандартный вывод вместо стандартных ошибок. Каждая строка имеет вид:

<flag> <old-object-id> <new-object-id> <local-reference>

Статус актуальных ссылок отображается только в том случае, если используется опция --verbose.

В режиме компактного вывода, указанном с помощью переменной конфигурации fetch.output, если в другой строке найден весь <from> или <to> , он будет заменен на * в другой строке. Например, master -> origin/master становится master -> origin/* .

flag

Один символ, обозначающий статус ссылки:

(space)

за успешно выполненную перемотку вперед;

+

для успешного принудительного обновления;

-

за успешно обрезанный реф;

t

для успешного обновления тега;

*

за успешно полученную новую ссылку;

!

для ссылки, которая была отклонена или не была обновлена; и

=

для ссылки, которая была актуальной и не нуждалась в загрузке.

summary

Для успешно полученной ссылки в сводке отображаются старые и новые значения ссылки в форме, подходящей для использования в качестве аргумента для git log (в большинстве случаев это <old>..<new> и <old>...<new> для принудительного обновления без перемотки вперед).

from

Имя удаленной ссылки, из которой извлекается информация, за вычетом префикса refs/<type>/ . В случае удаления имя удаленной ссылки — "(none)" .

to

Имя обновляемой локальной ссылки за вычетом префикса refs/<type>/ .

reason

Человекочитаемое объяснение. В случае успешного получения ссылок объяснение не требуется. Для неудачной ссылки описывается причина неудачи.

Examples

  • Обновите ветки удаленного отслеживания:

    $ git fetch origin

    Приведенная выше команда копирует все ветки из удаленного пространства имен refs/heads/ и сохраняет их в локальном пространстве имен refs/remotes/origin/ , если только опция remote.<repository>.fetch не используется для указания спецификации ссылки, отличной от умолчанию.

  • Явное использование refspecs:

    $ git fetch origin +seen:seen maint:tmp

    При этом обновляются (или создаются, при необходимости) ветки seen и tmp в локальном репозитории путем выборки из ветвей (соответственно) seen и maint из удаленного репозитория.

    Ветка seen будет обновлена, даже если не будет выполняться перемотка вперед, поскольку перед ней стоит знак плюса; tmp не будет.

  • Просмотрите ветку удаленного устройства, не настраивая его в локальном репозитории:

    $ git fetch git://git.kernel.org/pub/scm/git/git.git maint
$ git log FETCH_HEAD

    Первая команда извлекает ветку maint из репозитория по адресу git://git.kernel.org/pub/scm/git/git.git , а вторая команда использует FETCH_HEAD для проверки ветки с git-log[1] . Извлеченные объекты в конечном итоге будут удалены встроенной функцией git (см. git-gc[1] ).

Security

Протоколы выборки и отправки не предназначены для предотвращения кражи одной стороной данных из другого репозитория, которые не предназначены для совместного использования. Если у вас есть данные private , которые необходимо защитить от злонамеренного узла, лучший вариант — сохранить их в другом репозитории. Это касается как клиентов, так и серверов. В частности, пространства имен на сервере неэффективны для управления доступом для чтения; вам следует предоставлять доступ на чтение к пространству имен только тем клиентам, которым вы доверяете доступ на чтение ко всему репозиторию.

Известные векторы атак следующие:

  1. Жертва отправляет строки "have" , объявляющие идентификаторы имеющихся у нее объектов, которые явно не предназначены для совместного использования, но могут использоваться для оптимизации передачи, если они также есть у узла. Злоумышленник выбирает идентификатор объекта X для кражи и отправляет ссылку на X, но не обязан отправлять содержимое X, поскольку оно уже есть у жертвы. Теперь жертва считает, что у злоумышленника есть X, и позже отправляет содержимое X обратно злоумышленнику. (Эту атаку наиболее просто выполнить клиент на сервере, создав ссылку на X в пространстве имен, к которому клиент имеет доступ, а затем извлекая ее. Наиболее вероятный способ, которым сервер может выполнить эту атаку на клиенте, — это "merge" . X в ветку public и надеяться, что пользователь проделает дополнительную работу над этой веткой и отправит ее обратно на сервер, не заметив слияния.)

  2. Как и в № 1, злоумышленник выбирает объект с идентификатором X для кражи. Жертва отправляет объект Y, который уже есть у злоумышленника, и злоумышленник ложно утверждает, что у него есть X, а не Y, поэтому жертва отправляет Y как дельту против X. Дельта показывает злоумышленнику области X, похожие на Y.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

fetch.recurseSubmodules

Этот параметр определяет, будет ли git fetch (и базовая выборка в git pull ) рекурсивно выбираться в заполненные подмодули. Для этой опции можно установить либо значение boolean , либо значение on-demand . Установка значения boolean изменяет поведение операций выборки и извлечения: он безоговорочно рекурсивно переходит в подмодули, если установлено значение true , или вообще не выполняет рекурсию, если установлено значение false. Если установлено значение on-demand , выборка и извлечение будут рекурсивно обращаться к заполненному подмодулю только тогда, когда его суперпроект получает фиксацию, которая обновляет ссылку на подмодуль. По умолчанию on-demand или значение submodule.recurse , если оно установлено.

fetch.fsckObjects

Если для него установлено значение true, git-fetch-pack будет проверять все извлеченные объекты. См. transfer.fsckObjects , чтобы узнать, что проверено. По умолчанию ложь. Если не установлено, вместо него используется значение transfer.fsckObjects .

fetch.fsck.<msg-id>

Действует как fsck.<msg-id> , но используется git-fetch-pack[1] вместо git-fsck[1] . Подробности см. в документации fsck.<msg-id> .

fetch.fsck.skipList

Действует как fsck.skipList , но используется git-fetch-pack[1] вместо git-fsck[1] . Подробности см. в документации fsck.skipList .

fetch.unpackLimit

Если количество объектов, полученных посредством встроенной передачи Git , ниже этого предела, объекты будут распакованы в отдельные объектные файлы. Однако если количество полученных объектов равно или превышает этот предел, полученный пакет будет сохранен как пакет после добавления недостающих дельта-баз. Сохранение пакета из принудительной рассылки может ускорить ее выполнение, особенно на медленных файловых системах. Если не установлено, вместо него используется значение transfer.unpackLimit .

fetch.prune

Если это правда, выборка будет автоматически вести себя так, как если бы опция --prune была указана в командной строке. См. также remote.<name>.prune и раздел ОБРЕЗКА git-fetch[1] .

fetch.pruneTags

Если это правда, выборка будет автоматически вести себя так, как если бы при сокращении была указана refspec refs/tags/*:refs/tags/* , если она еще не установлена. Это позволяет установить как эту опцию, так и fetch.prune для поддержания сопоставления 1=1 с восходящими ссылками. См. также remote.<name>.pruneTags и раздел ОБРЕЗКА git-fetch[1] .

fetch.output

Управляйте тем, как печатается статус обновления ссылки. Допустимые значения: full и compact . Значение по умолчанию — full . Подробности см. в разделе ВЫХОД в git-fetch[1] .

fetch.negotiationAlgorithm

Управляйте отправкой информации о коммитах в локальном репозитории при согласовании содержимого пакетного файла, который будет отправлен сервером. Установите значение "consecutive" , чтобы использовать алгоритм, который проходит последовательные фиксации и проверяет каждую из них. Установите значение "skipping" , чтобы использовать алгоритм, который пропускает фиксации для более быстрой сходимости, но может привести к увеличению размера пакетного файла; или установите значение "noop" , чтобы вообще не отправлять никакой информации, что почти наверняка приведет к увеличению размера пакетного файла, но приведет к пропуску этапа согласования. Установите значение "default" , чтобы отменить ранее выполненные настройки и использовать поведение по умолчанию. Обычно значением по умолчанию является "consecutive" , но если feature.experimental истинно, то значением по умолчанию является "skipping" . Неизвестные значения приведут к ошибке git fetch .

См. также варианты --negotiate-only и --negotiation-tip для git-fetch[1] .

fetch.showForcedUpdates

Установите значение false , чтобы включить --no-show-forced-updates в командах git-fetch[1] и git-pull[1] . По умолчанию истинно.

fetch.parallel

Указывает максимальное количество операций выборки, которые могут выполняться параллельно одновременно (подмодули или удаленные устройства, если действует опция --multiple для git-fetch[1] ).

Значение 0 даст некоторое разумное значение по умолчанию. Если не установлено, по умолчанию установлено значение 1.

Для субмодулей этот параметр можно переопределить с помощью параметра конфигурации submodule.fetchJobs .

fetch.writeCommitGraph

Установите значение true , чтобы записывать граф фиксации после каждой команды git fetch , которая загружает пакетный файл с удаленного устройства. При использовании опции --split в большинстве случаев создается очень маленький файл графа фиксации поверх существующих файлов графа фиксации. Иногда эти файлы объединяются, и запись может занять больше времени. Наличие обновленного файла графа фиксации помогает выполнять многие команды Git , включая git merge-base , git push -f и git log --graph . По умолчанию ложь.

fetch.bundleURI

В этом значении хранится URI для загрузки данных объекта Git из URI пакета перед выполнением добавочной выборки с исходного сервера Git . Это похоже на то, как ведет себя опция --bundle-uri в git-clone[1] . git clone --bundle-uri установит значение fetch.bundleURI , если предоставленный URI пакета содержит список пакетов, организованный для добавочной выборки.

Если вы измените это значение и в вашем репозитории есть значение fetch.bundleCreationToken , удалите это значение fetch.bundleCreationToken перед извлечением из нового URI пакета.

fetch.bundleCreationToken

При использовании fetch.bundleURI для постепенной выборки из списка пакетов, использующего эвристику "creationToken" , в этом значении конфигурации сохраняется максимальное значение creationToken для загруженных пакетов. Это значение используется для предотвращения загрузки пакетов в будущем, если объявленный creationToken строго не превышает это значение.

Значения токена создания выбираются поставщиком, обслуживающим конкретный URI пакета. Если вы измените URI в fetch.bundleURI , обязательно удалите значение fetch.bundleCreationToken перед извлечением.

Bugs

Использование --recurse-submodules позволяет получать новые коммиты только в подмодулях, которые присутствуют локально, например, в $GIT_DIR/modules/ . Если восходящий поток добавляет новый субмодуль, этот субмодуль не может быть получен до тех пор, пока он не будет клонирован, например, с помощью git submodule update . Ожидается, что это будет исправлено в будущей версии Git .
