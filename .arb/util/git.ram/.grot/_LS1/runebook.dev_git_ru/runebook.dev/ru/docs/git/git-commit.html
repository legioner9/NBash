git-commit

Name

git-commit — Записать изменения в репозиторий

Synopsis

git commit [-a | --interactive | --patch] [-s] [-v] [-u<mode>] [--amend]
           [--dry-run] [(-c | -C | --squash) <commit> | --fixup [(amend|reword):]<commit>)]
           [-F <file> | -m <msg>] [--reset-author] [--allow-empty]
           [--allow-empty-message] [--no-verify] [-e] [--author=<author>]
           [--date=<date>] [--cleanup=<mode>] [--[no-]status]
           [-i | -o] [--pathspec-from-file=<file> [--pathspec-file-nul]]
           [(--trailer <token>[(=|:)<value>])…​] [-S[<keyid>]]
           [--] [<pathspec>…​]

Description

Создайте новый коммит, содержащий текущее содержимое индекса и данное сообщение журнала, описывающее изменения. Новый коммит является прямым дочерним элементом HEAD, обычно это вершина текущей ветки, и ветка обновляется, чтобы указывать на нее (если только ни одна ветвь не связана с рабочим деревом, в этом случае HEAD — это "detached" , как описано в git-checkout[1] ).

Содержимое, которое необходимо зафиксировать, можно указать несколькими способами:

  1. с помощью git-add[1] для постепенного изменения индекса "add" перед использованием команды commit (Примечание: даже измененные файлы должны быть "added" );

  2. с помощью git-rm[1] удалить файлы из рабочего дерева и индекса, опять же перед использованием команды commit ;

  3. путем перечисления файлов в качестве аргументов команды commit (без переключателя --interactive или --patch), и в этом случае фиксация будет игнорировать изменения, размещенные в индексе, и вместо этого запишет текущее содержимое перечисленных файлов (которое уже должно быть известно). в Git);

  4. с помощью ключа -a с командой commit для автоматического изменения "add" из всех известных файлов (т. е. всех файлов, которые уже указаны в индексе) и для автоматического изменения файлов "rm" в индексе, которые были удалены из рабочего дерева, а затем выполнения фактическая фиксация;

  5. используя ключи --interactive или --patch с командой commit , чтобы один за другим решить, какие файлы или фрагменты должны быть частью фиксации в дополнение к содержимому в индексе, прежде чем завершить операцию. См. раздел «Интерактивный режим» git-add[1] , чтобы узнать, как работать с этими режимами.

Опцию --dry-run можно использовать для получения сводной информации о том, что включается в любой из вышеперечисленных элементов для следующего коммита, задав тот же набор параметров (опций и путей).

Если вы сделали коммит, а затем сразу после этого обнаружили ошибку, вы можете восстановить ее с помощью git reset .

Options

-a
--all

Укажите команде, чтобы она автоматически помещала файлы, которые были изменены и удалены, но новые файлы, о которых вы не сообщили Git , не затрагиваются.

-p
--patch

Используйте интерактивный интерфейс выбора патчей, чтобы выбрать, какие изменения следует зафиксировать. Подробности см. в git-add[1] .

-C <commit>
--reuse-message=<commit>

Возьмите существующий объект фиксации и повторно используйте сообщение журнала и информацию об авторстве (включая метку времени) при создании фиксации.

-c <commit>
--reedit-message=<commit>

Подобно -C , но с -c вызывается редактор, чтобы пользователь мог дополнительно редактировать сообщение фиксации.

--fixup=[(amend|reword):]<commit>

Создайте новую фиксацию "fixes up" <commit> при применении с git rebase --autosquash . Обычный --fixup=<commit> создает фиксацию "fixup!" , которая изменяет содержимое <commit> , но оставляет сообщение журнала нетронутым. --fixup=amend:<commit> аналогичен, но создает фиксацию "amend!" , которая также заменяет сообщение журнала <commit> сообщением журнала фиксации "amend!" . --fixup=reword:<commit> создает фиксацию "amend!" , которая заменяет сообщение журнала <commit> собственным сообщением журнала, но не вносит изменений в содержимое <commit> .

Коммит, созданный простым --fixup=<commit> , имеет тему, состоящую из "fixup!" , за которой следует строка темы из <commit>, и специально распознается git rebase --autosquash . Опцию -m можно использовать для дополнения сообщения журнала о созданном коммите, но дополнительный комментарий будет удален, как только коммит "fixup!" будет сжат в <commit> с помощью git rebase --autosquash .

Коммит, созданный --fixup=amend:<commit> , аналогичен, но его субъект имеет префикс "amend!" . Сообщение журнала <commit> копируется в сообщение журнала фиксации "amend!" и открывается в редакторе для уточнения. Когда git rebase --autosquash сжимает фиксацию "amend!" в <commit> , сообщение журнала <commit> заменяется уточненным сообщением журнала из фиксации "amend!" . Если сообщение журнала фиксации "amend!" не указано, это ошибка.

--fixup=reword:<commit> — это сокращение от --fixup=amend:<commit> --only . Он создает фиксацию "amend!" только с сообщением журнала (игнорируя любые изменения, внесенные в индекс). При сжатии git rebase --autosquash оно заменяет сообщение журнала <commit> без внесения каких-либо других изменений.

Ни "fixup!" , ни "amend!" не фиксируют изменение авторства <commit> при применении git rebase --autosquash . Подробности см. в git-rebase[1] .

--squash=<commit>

Создайте сообщение фиксации для использования с rebase --autosquash . Строка темы сообщения о фиксации берется из указанной фиксации с префиксом "squash! " . Может использоваться с дополнительными опциями сообщения о фиксации ( -m / -c / -C / -F ). Подробности см. в git-rebase[1] .

--reset-author

При использовании с опциями -C/-c/--amend или при фиксации после конфликтного выбора вишни заявите, что авторство результирующего коммита теперь принадлежит коммиттеру. Это также обновляет временную метку автора.

--short

При пробном прогоне дайте результат в коротком формате. Подробности см. в git-status[1] . Подразумевается --dry-run .

--branch

Покажите филиал и информацию об отслеживании даже в коротком формате.

--porcelain

При пробном прогоне дайте результат в готовом для фарфора формате. Подробности см. в git-status[1] . Подразумевается --dry-run .

--long

При пробном прогоне выводите результат в длинном формате. Подразумевается --dry-run .

-z
--null

При отображении вывода состояния short или porcelain печатайте имя файла дословно и завершайте записи NUL вместо LF. Если формат не указан, подразумевается выходной формат --porcelain . Без опции -z имена файлов с символами "unusual" заключаются в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ).

-F <file>
--file=<file>

Возьмите сообщение фиксации из данного файла. Используйте - , чтобы прочитать сообщение со стандартного ввода.

--author=<author>

Переопределить автора коммита. Укажите явного автора, используя стандартный формат A U Thor <[email protected]> . В противном случае предполагается, что <author> является шаблоном и используется для поиска существующего коммита этого автора (т. е. rev-list --all -i --author=<author>); затем автор коммита копируется из первого найденного коммита.

--date=<date>

Переопределить дату автора, используемую в коммите.

-m <msg>
--message=<msg>

Используйте данный <msg> в качестве сообщения фиксации. Если указано несколько параметров -m , их значения объединяются в отдельные абзацы.

Опция -m является взаимоисключающей с -c , -C и -F .

-t <file>
--template=<file>

При редактировании сообщения фиксации запустите редактор с содержимым данного файла. Переменная конфигурации commit.template часто используется для неявного указания этой опции команде. Этот механизм может использоваться проектами, которые хотят дать участникам подсказки о том, что и в каком порядке писать в сообщении. Если пользователь выходит из редактора, не отредактировав сообщение, фиксация прерывается. Это не имеет никакого эффекта, если сообщение передается другими способами, например, с помощью опций -m или -F .

-s
--signoff
--no-signoff

Добавьте трейлер Signed-off-by от коммиттера в конце сообщения журнала фиксации. Значение подписания зависит от проекта, за который вы беретесь. Например, он может удостоверить, что коммиттер имеет права представить работу по лицензии проекта или согласен на какое-либо представление участника, такое как Сертификат происхождения разработчика. (См. https://developercertificate.org , который используется в ядре Linux и проектах Git .) Обратитесь к документации или руководству проекта, в который вы вносите свой вклад, чтобы понять, как подписи используются в этом проекте.

Опцию --no-signoff можно использовать для отмены более ранней опции --signoff в командной строке.

--trailer <token>[(=|:)<value>]

Укажите пару (<токен>, <значение>), которую следует применять в качестве трейлера. (например, git commit --trailer "Signed-off-by:C O Mitter \ <[email protected]>" --trailer "Helped-by:C O Mitter \ <[email protected]>" добавит трейлер "Signed-off-by" и трейлер "Helped-by" к сообщению о фиксации.) Переменные конфигурации trailer.* ( git-interpret-trailers[1] ) можно использовать для определения того, опущен ли дублированный трейлер, где при запуске трейлеров будет появляться каждый трейлер и другие подробности.

-n
--[no-]verify

По умолчанию запускаются перехватчики pre-commit и commit-msg. Если задан любой из --no-verify или -n , они игнорируются. См. также githooks[5] .

--allow-empty

Обычно запись коммита, имеющего то же дерево, что и его единственный родительский коммит, является ошибкой, и команда не позволяет вам сделать такой коммит. Эта опция обходит систему безопасности и в первую очередь предназначена для использования внешними сценариями интерфейса SCM.

--allow-empty-message

Как и --allow-empty, эта команда в первую очередь предназначена для использования внешними сценариями интерфейса SCM. Это позволяет вам создать фиксацию с пустым сообщением о фиксации без использования сантехнических команд, таких как git-commit-tree[1] .

--cleanup=<mode>

Эта опция определяет, как должно быть очищено предоставленное сообщение о фиксации перед фиксацией. <mode> может быть strip , whitespace , verbatim , scissors или default .

strip

Удалить начальные и конечные пустые строки, конечные пробелы, комментарии и свернуть последовательные пустые строки.

whitespace

То же, что и strip , за исключением того, что #commentary не удален.

verbatim

Ни в коем случае не меняйте сообщение.

scissors

То же, что и whitespace , за исключением того, что все, начиная со строки, указанной ниже (включительно), обрезается, если сообщение подлежит редактированию. " # " можно настроить с помощью core.commentChar.

# ------------------------ >8 ------------------------
default

То же, что и strip , если сообщение нужно редактировать. В противном случае whitespace .

Значение по умолчанию можно изменить с помощью переменной конфигурации commit.cleanup (см. git-config[1] ).

-e
--edit

Сообщение, взятое из файла с -F , командной строки с -m и из объекта фиксации с -C , обычно используется в качестве неизмененного сообщения журнала фиксации. Эта опция позволяет дополнительно редактировать сообщение, полученное из этих источников.

--no-edit

Используйте выбранное сообщение фиксации, не запуская редактор. Например, git commit --amend --no-edit вносит изменения в фиксацию, не меняя сообщения фиксации.

--amend

Замените кончик текущей ветки, создав новый коммит. Записанное дерево подготавливается как обычно (включая влияние опций -i и -o и явную спецификацию пути), а в качестве отправной точки используется сообщение из исходного коммита, а не пустое сообщение, если в файле не указано другое сообщение. командной строки с помощью таких параметров, как -m , -F , -c и т. д. Новый коммит имеет тех же родителей и автора, что и текущий (опция --reset-author может отменить это).

Это грубый эквивалент:

        $ git reset --soft HEAD^
        $ ... do something else to come up with the right tree ...
        $ git commit -c ORIG_HEAD

но может использоваться для изменения коммита слияния.

Вы должны понимать последствия переписывания истории, если вы вносите изменения в уже опубликованный коммит. (См. раздел "RECOVERING FROM UPSTREAM REBASE" в git-rebase[1] .)

--no-post-rewrite

Обход хука после перезаписи.

-i
--include

Прежде чем делать фиксацию промежуточного содержимого, также проиндексируйте содержимое путей, указанных в командной строке. Обычно это не то, что вам нужно, если только вы не завершаете конфликтное слияние.

-o
--only

Сделайте фиксацию, взяв обновленное содержимое рабочего дерева путей, указанных в командной строке, игнорируя любое содержимое, подготовленное для других путей. Это режим работы git commit по умолчанию, если в командной строке указаны какие-либо пути; в этом случае эту опцию можно опустить. Если эта опция указана вместе с --amend , то не нужно указывать пути, которые можно использовать для изменения последней фиксации без фиксации уже подготовленных изменений. При использовании вместе с --allow-empty пути также не требуются, и будет создан пустой коммит.

--pathspec-from-file=<file>

Спецификация пути передается в <file> вместо аргументов командной строки. Если <file> — это точно - , то используется стандартный ввод. Элементы Pathspec разделяются LF или CR/LF. Элементы Pathspec можно заключать в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). См. также --pathspec-file-nul и глобальный --literal-pathspecs .

--pathspec-file-nul

Имеет смысл только с --pathspec-from-file . Элементы Pathspec разделяются символом NUL, а все остальные символы воспринимаются буквально (включая символы новой строки и кавычки).

-u[<mode>]
--untracked-files[=<mode>]

Показать неотслеживаемые файлы.

Параметр mode является необязательным (по умолчанию all ) и используется для указания обработки неотслеживаемых файлов; если -u не используется, по умолчанию используется normal , т.е. отображаются неотслеживаемые файлы и каталоги.

Возможные варианты:

  • no - Не показывать неотслеживаемые файлы

  • normal — показывает неотслеживаемые файлы и каталоги.

  • all — также показывает отдельные файлы в неотслеживаемых каталогах.

Значение по умолчанию можно изменить с помощью переменной конфигурации status.showUntrackedFiles, описанной в git-config[1] .

-v
--verbose

Покажите унифицированную разницу между фиксацией HEAD и тем, что будет зафиксировано, в нижней части шаблона сообщения о фиксации, чтобы помочь пользователю описать фиксацию, напоминая, какие изменения она содержит. Обратите внимание, что строки этого вывода diff не имеют префикса # . Этот diff не будет частью сообщения фиксации. См. переменную конфигурации commit.verbose в git-config[1] .

Если указано дважды, дополнительно показывается унифицированная разница между тем, что будет зафиксировано, и файлами рабочего дерева, т. е. неиндексированные изменения в отслеживаемых файлах.

-q
--quiet

Подавить сводное сообщение о фиксации.

--dry-run

Не создавайте фиксацию, а покажите список путей, которые должны быть зафиксированы, путей с локальными изменениями, которые останутся незафиксированными, и путей, которые не отслеживаются.

--status

Включите выходные данные git-status[1] в шаблон сообщения о фиксации при использовании редактора для подготовки сообщения о фиксации. По умолчанию включено, но может использоваться для переопределения переменной конфигурации commit.status.

--no-status

Не включайте выходные данные git-status[1] в шаблон сообщения о фиксации при использовании редактора для подготовки сообщения о фиксации по умолчанию.

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

GPG-знак фиксируется. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

--

Не интерпретируйте дополнительные аргументы как варианты.

<pathspec>…

Если в командной строке указана спецификация пути, зафиксируйте содержимое файлов, соответствующих спецификации пути, без записи изменений, уже добавленных в индекс. Содержимое этих файлов также готовится для следующего коммита поверх того, что было подготовлено ранее.

Дополнительные сведения см. в записи pathspec в gitglossary[7] .

Examples

При записи вашей собственной работы содержимое измененных файлов в вашем рабочем дереве временно сохраняется в промежуточной области, называемой "index" с git add . Файл можно вернуть обратно только в индексе, но не в рабочем дереве, к файлу последней фиксации с помощью git restore --staged <file> , что эффективно возвращает git add и предотвращает участие изменений в этом файле в следующей фиксации. После создания состояния, которое будет постепенно зафиксировано с помощью этих команд, git commit (без какого-либо параметра пути) используется для записи того, что уже было подготовлено. Это самая основная форма команды. Пример:

$ edit hello.c
$ git rm goodbye.c
$ git add hello.c
$ git commit

Вместо того, чтобы размещать файлы после каждого отдельного изменения, вы можете указать git commit , чтобы он замечал изменения в файлах, содержимое которых отслеживается в вашем рабочем дереве, и выполнял за вас соответствующие действия git add и git rm . То есть этот пример делает то же самое, что и предыдущий пример, если в вашем рабочем дереве нет других изменений:

$ edit hello.c
$ rm goodbye.c
$ git commit -a

Команда git commit -a сначала просматривает ваше рабочее дерево, замечает, что вы изменили hello.c и удалили Goodbye.c, и выполняет необходимые действия git add и git rm .

После внесения изменений во многие файлы вы можете изменить порядок записи изменений, указав пути к git commit . Когда указаны пути, команда выполняет фиксацию, в которой записываются только изменения, внесенные в именованные пути:

$ edit hello.c hello.h
$ git add hello.c hello.h
$ edit Makefile
$ git commit Makefile

При этом выполняется фиксация, записывающая изменение Makefile . Изменения, подготовленные для hello.c и hello.h , не включены в итоговую фиксацию. Однако их изменения не теряются — они по-прежнему инсценируются и просто сдерживаются. Если после описанной выше последовательности:

$ git commit

этот второй коммит запишет изменения в hello.c и hello.h , как и ожидалось.

После того как слияние (инициированное git merge или git pull ) прекращается из-за конфликтов, полностью объединенные пути уже подготовлены для фиксации, а конфликтующие пути остаются в неслитом состоянии. Вам нужно будет сначала проверить, какие пути конфликтуют с git status , и после исправления их вручную в рабочем дереве вы, как обычно, обработаете результат с помощью git add :

$ git status | grep unmerged
unmerged: hello.c
$ edit hello.c
$ git add hello.c

После разрешения конфликтов и подготовки результата git ls-files -u переставал упоминать конфликтный путь. Когда вы закончите, запустите git commit , чтобы наконец записать слияние:

$ git commit

Как и в случае записи собственных изменений, вы можете использовать опцию -a , чтобы не вводить текст. Единственное отличие состоит в том, что во время разрешения слияния вы не можете использовать git commit с именами путей для изменения порядка фиксации изменений, поскольку слияние должно быть записано как одна фиксация. Фактически, команда отказывается запускаться при задании пути (но см. опцию -i ).

Commit information

Информация об авторе и коммиттере берется из следующих переменных среды, если они установлены:

GIT_AUTHOR_NAME
GIT_AUTHOR_EMAIL
GIT_AUTHOR_DATE
GIT_COMMITTER_NAME
GIT_COMMITTER_EMAIL
GIT_COMMITTER_DATE

(Примечание: "<" , ">" и "\n" удалены)

Имена автора и коммиттера по соглашению являются некоторой формой личного имени (то есть имени, под которым другие люди обращаются к вам), хотя Git не требует какой-либо конкретной формы. Можно использовать произвольный Unicode с учетом ограничений, перечисленных выше. Это имя не влияет на аутентификацию; для этого см. переменную credential.username в git-config[1] .

В случае, если (некоторые из) этих переменных среды не установлены, информация берется из элементов конфигурации user.name и user.email или, если она отсутствует, из переменной среды EMAIL или, если она не установлена, из имени системного пользователя и имени хоста. используется для исходящей почты (берётся из /etc/mailname и возвращается к полному имени хоста, когда этот файл не существует).

author.name и committer.name и соответствующие им параметры электронной почты переопределяют user.name и user.email , если они установлены, и переопределяются самими переменными среды.

Обычно используется установка только переменных user.name и user.email ; другие параметры предназначены для более сложных случаев использования.

Date formats

Переменные среды GIT_AUTHOR_DATE и GIT_COMMITTER_DATE поддерживают следующие форматы даты:

Внутренний формат Git

Это <unix-timestamp> <time-zone-offset> , где <unix-timestamp> — количество секунд, прошедших с эпохи UNIX. <time-zone-offset> — это положительное или отрицательное смещение от UTC. Например, CET (который на 1 час опережает UTC) — +0100 .

RFC 2822

Стандартный формат электронной почты, описанный в RFC 2822, например Thu, 07 Apr 2005 22:13:13 +0200 .

ISO 8601

Время и дата указаны стандартом ISO 8601, например 2005-04-07T22:13:13 . Анализатор также принимает пробел вместо символа T . Дробные доли секунды будут игнорироваться, например 2005-04-07T22:13:13.019 будет рассматриваться как 2005-04-07T22:13:13 .

Note
 Кроме того, часть даты принимается в следующих форматах: YYYY.MM.DD , MM/DD/YYYY и DD.MM.YYYY .

Помимо распознавания всех вышеуказанных форматов даты, опция --date также пытается разобраться в других, более ориентированных на человека форматах даты, таких как относительные даты, такие как "yesterday" или "last Friday at noon" .

Discussion

Хотя это и не обязательно, рекомендуется начинать сообщение о фиксации с одной короткой (не более 50 символов) строки, в которой суммируется изменение, за которой следует пустая строка, а затем более подробное описание. Текст до первой пустой строки в сообщении о фиксации рассматривается как заголовок фиксации, и этот заголовок используется во всем Git. Например, git-format-patch[1] преобразует коммит в электронное письмо и использует заголовок в строке «Тема», а остальную часть коммита — в теле.

Git в некоторой степени не зависит от кодировки символов.

  • Содержимое объектов blob представляет собой неинтерпретированные последовательности байтов. На уровне ядра нет трансляции кодировки.

  • Имена путей кодируются в форме нормализации UTF-8 C. Это относится к древовидным объектам, индексному файлу, именам ссылок, а также именам путей в аргументах командной строки, переменных среды и файлах конфигурации ( .git/config (см. git-config[1] ), gitignore[5] , gitattributes[5] ). и gitmodules[5] ).

    Обратите внимание, что Git на уровне ядра обрабатывает имена путей просто как последовательности байтов, отличных от NUL, преобразования кодировок имен путей не выполняются (кроме Mac и Windows). Таким образом, использование имен путей, отличных от ASCII, в большинстве случаев будет работать даже на платформах и файловых системах, использующих устаревшие расширенные кодировки ASCII. Однако репозитории, созданные в таких системах, не будут корректно работать в системах на базе UTF-8 (например, Linux, Mac, Windows) и наоборот. Кроме того, многие инструменты на основе Git просто предполагают, что имена путей имеют формат UTF-8, и не могут правильно отображать другие кодировки.

  • Сообщения журнала фиксации обычно кодируются в UTF-8, но поддерживаются и другие расширенные кодировки ASCII. Сюда входят ISO-8859-x, CP125x и многие другие, а также многобайтовые кодировки not UTF-16/32, EBCDIC и CJK (GBK, Shift-JIS, Big5, EUC-x, CP9xx и т. д.).

Хотя мы рекомендуем, чтобы сообщения журнала фиксации были закодированы в UTF-8, ядро ​​и Git Porcelain не предназначены для принудительного использования UTF-8 в проектах. Если всем участникам конкретного проекта удобнее использовать устаревшие кодировки, Git не запрещает этого. Однако есть несколько вещей, которые следует иметь в виду.

  1. git commit и git commit-tree выдают предупреждение, если переданное ему сообщение журнала фиксации не похоже на допустимую строку UTF-8, если только вы явно не укажете, что ваш проект использует устаревшую кодировку. Это можно сказать так: i18n.commitEncoding в файле .git/config , например:

    [i18n]
        commitEncoding = ISO-8859-1

    Объекты фиксации, созданные с указанным выше параметром, записывают значение i18n.commitEncoding в свой заголовок encoding . Это сделано для того, чтобы помочь другим людям, которые посмотрят на них позже. Отсутствие этого заголовка означает, что сообщение журнала фиксации закодировано в UTF-8.

  2. git log , git show , git blame и его друзья просматривают заголовок encoding объекта фиксации и пытаются перекодировать сообщение журнала в UTF-8, если не указано иное. Вы можете указать желаемую выходную кодировку с помощью i18n.logOutputEncoding в файле .git/config , например:

    [i18n]
        logOutputEncoding = ISO-8859-1

    Если у вас нет этой переменной конфигурации, вместо нее используется значение i18n.commitEncoding .

Обратите внимание, что мы намеренно решили не перекодировать сообщение журнала фиксации при выполнении фиксации, чтобы принудительно использовать UTF-8 на уровне объекта фиксации, поскольку перекодирование в UTF-8 не обязательно является обратимой операцией.

Переменные среды и конфигурации

Редактор, используемый для редактирования сообщения журнала фиксации, будет выбран из переменной среды GIT_EDITOR , переменной конфигурации core.editor, переменной среды VISUAL или переменной среды EDITOR (в указанном порядке). Подробности см. в git-var[1] .

Все, что находится выше этой строки в этом разделе, не включено в документацию git-config[1] . Следующее содержимое такое же, как и найденное там:

commit.cleanup

Этот параметр переопределяет параметр --cleanup по умолчанию в git commit . Подробности см. в git-commit[1] . Изменение значения по умолчанию может быть полезно, если вы всегда хотите сохранить в сообщении журнала строки, начинающиеся с символа комментария # , и в этом случае вы должны сделать git config commit.cleanup whitespace (обратите внимание, что вам придется удалить вспомогательные строки, начинающиеся с # в коммите). шаблон журнала самостоятельно, если будете это делать).

commit.gpgSign

boolean , чтобы указать, должны ли все фиксации быть подписаны GPG. Использование этой опции при выполнении таких операций, как перебазирование, может привести к подписанию большого количества коммитов. Возможно, будет удобно использовать агент, чтобы не вводить парольную фразу GPG несколько раз.

commit.status

boolean для включения/отключения включения информации о состоянии в шаблон сообщения о фиксации при использовании редактора для подготовки сообщения о фиксации. По умолчанию истинно.

commit.template

Укажите путь к файлу, который будет использоваться в качестве шаблона для новых сообщений о фиксации.

commit.verbose

boolean или int, чтобы указать уровень детализации с помощью git commit . См. git-commit[1] .

Hooks

Эта команда может запускать перехватчики commit-msg , prepare-commit-msg , pre-commit , post-commit и post-rewrite . См. githooks[5] для получения дополнительной информации.

Files

$GIT_DIR/COMMIT_EDITMSG

Этот файл содержит сообщение о текущей фиксации. Если git commit завершает работу из-за ошибки перед созданием фиксации, любое сообщение о фиксации, предоставленное пользователем (например, в сеансе редактирования), будет доступно в этом файле, но будет перезаписано при следующем вызове git commit .
