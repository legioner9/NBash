gitprotocol-pack

Name

gitprotocol-pack — Как пакеты передаются по сети

Synopsis

<over-the-wire-protocol>

Description

Git поддерживает передачу данных в пакетных файлах по протоколам ssh://, git://, http:// и file://. Существует два набора протоколов: один для передачи данных от клиента на сервер, а другой для передачи данных с сервера на клиент. Три транспорта (ssh, git, file) используют один и тот же протокол для передачи данных. http документирован в gitprotocol-http[5] .

Процессы, вызываемые в канонической реализации Git : upload-pack на стороне сервера и fetch-pack на стороне клиента для получения данных; затем receive-pack на сервере и send-pack на клиенте для отправки данных. Функции протокола заключаются в том, чтобы сервер сообщал клиенту, что в данный момент находится на сервере, а затем оба согласовывали наименьший объем данных для отправки, чтобы полностью обновить один или другой.

Pkt-line format

Описания ниже основаны на формате pkt-line, описанном в gitprotocol-common[5] . Когда в грамматике указано PKT-LINE(...) , если не указано иное, применяются обычные правила LF для pkt-линии: отправитель ДОЛЖЕН включать LF, но получатель НЕ ДОЛЖЕН жаловаться, если он отсутствует.

Пакет ошибок — это специальная строка pkt, содержащая строку ошибки.

  error-line     =  PKT-LINE("ERR" SP explanation-text)

По всему протоколу, где ожидается PKT-LINE(...) , МОЖЕТ быть отправлен пакет ошибки. Как только этот пакет отправляется клиентом или сервером, процесс передачи данных, определенный в этом протоколе, завершается.

Transports

Существует три транспорта, по которым инициируется протокол Packfile. Транспорт Git — это простой сервер без аутентификации, который принимает команду (почти всегда upload-pack , хотя серверы Git могут быть настроены на глобальную запись, в которой также разрешена инициация receive- pack ), с которой клиент желает связаться, выполняет ее и подключает ее. к запрашивающему процессу.

При транспортировке SSH клиент просто запускает процесс upload-pack или receive-pack на сервере по протоколу SSH, а затем взаимодействует с этим вызванным процессом через соединение SSH.

Транспорт file:// запускает процесс upload-pack или receive-pack локально и взаимодействует с ним через канал.

Extra parameters

Протокол предоставляет механизм, с помощью которого клиенты могут отправлять дополнительную информацию в своем первом сообщении на сервер. Они называются "Extra Parameters" и поддерживаются протоколами Git, SSH и HTTP.

Каждый дополнительный параметр имеет форму <key>=<value> или <key> .

Серверы, которые получают любые такие дополнительные параметры, ДОЛЖНЫ игнорировать все нераспознанные ключи. В настоящее время единственным распознаваемым дополнительным параметром является "version" со значением 1 или 2 . См. gitprotocol-v2[5] для получения дополнительной информации о версии протокола 2.

Git transport

Транспорт Git начинается с отправки команды и репозитория по сети в формате pkt-line, за которым следует нулевой байт и параметр имени хоста, завершающийся нулевым байтом.

0033git-upload-pack /project.git\0host=myserver.com\0

Транспорт может отправлять дополнительные параметры, добавляя дополнительный NUL-байт, а затем добавляя одну или несколько строк, завершающихся NUL:

003egit-upload-pack /project.git\0host=myserver.com\0\0version=1\0
git-proto-request = request-command SP pathname NUL
      [ host-parameter NUL ] [ NUL extra-parameters ]
request-command   = "git-upload-pack" / "git-receive-pack" /
      "git-upload-archive"   ; case sensitive
pathname          = *( %x01-ff ) ; exclude NUL
host-parameter    = "host=" hostname [ ":" port ]
extra-parameters  = 1*extra-parameter
extra-parameter   = 1*( %x01-ff ) NUL

Параметр хоста используется для виртуального хостинга git -daemon на основе имени. См. параметр --interpolated-path для демона git с символами формата %H/%CH.

По сути, клиент Git делает следующее для подключения к процессу upload-pack на стороне сервера по протоколу Git :

$ echo -e -n \
  "003agit-upload-pack /schacon/gitbook.git\0host=example.com\0" |
  nc -v example.com 9418

Ssh transport

Инициирование процессов загрузки или получения пакетов через SSH означает выполнение двоичного файла на сервере посредством удаленного выполнения SSH. По сути, это эквивалентно запуску этого:

$ ssh git.example.com "git-upload-pack '/project.git'"

Чтобы сервер поддерживал отправку и получение Git для данного пользователя через SSH, этот пользователь должен иметь возможность выполнить одну или обе эти команды через оболочку SSH, которая ему предоставляется при входе в систему. В некоторых системах доступ к оболочке ограничен возможностью запуска только этих двух команд или даже одной из них.

В URI формата ssh:// он является абсолютным в URI, поэтому / после имени хоста (или номера порта) отправляется в качестве аргумента, который затем считывается удаленным пакетом git-upload-pack точно так же, как есть, поэтому по сути, это абсолютный путь в удаленной файловой системе.

   git clone ssh://user@example.com/project.git
  |
  v
ssh user@example.com "git-upload-pack '/project.git'"

В URI формата "user@host:path" он относится к домашнему каталогу пользователя, поскольку клиент Git будет работать:

   git clone user@example.com:project.git
    |
    v
ssh user@example.com "git-upload-pack 'project.git'"

Исключение составляет случай, когда используется ~ , и в этом случае мы выполняем его без ведущего / .

   ssh://user@example.com/~alice/project.git,
    |
    v
ssh user@example.com "git-upload-pack '~alice/project.git'"

В зависимости от значения переменной конфигурации protocol.version Git может попытаться отправить дополнительные параметры в виде строки, разделенной двоеточиями, в переменной среды GIT_PROTOCOL. Это делается только в том случае, если переменная конфигурации ssh.variant указывает, что команда ssh поддерживает передачу переменных среды в качестве аргумента.

Здесь следует запомнить несколько вещей:

  • "command name" пишется через тире (например, git-upload-pack), но клиент может это изменить;

  • Путь к репозиторию всегда указывается в одинарных кавычках.

Получение данных с сервера

Когда один репозиторий Git хочет получить данные, которые есть у второго репозитория, первый может fetch получить данные из второго. Эта операция определяет, какие данные есть на сервере и которые клиент не передает, а затем передает эти данные клиенту в формате пакетного файла.

Reference discovery

Когда клиент первоначально подключается, сервер немедленно ответит номером версии (если "version=1" отправлен как дополнительный параметр) и списком каждой имеющейся у него ссылки (все ветки и теги), а также именем объекта, на который в данный момент указывает каждая ссылка. .

 $ echo -e -n "0045git-upload-pack /schacon/gitbook.git\0host=example.com\0\0version=1\0" |
    nc -v example.com 9418
 000eversion 1
 00887217a7c7e582c46cec22a130adf4b9d7d950fba0 HEAD\0multi_ack thin-pack
side-band side-band-64k ofs-delta shallow no-progress include-tag
 00441d3fcd5ced445d1abc402225c0b8a1299641f497 refs/heads/integration
 003f7217a7c7e582c46cec22a130adf4b9d7d950fba0 refs/heads/master
 003cb88d2441cac0977faf98efc80305012112238d9d refs/tags/v0.9
 003c525128480b96c89e6418b1e40909bf6c5b2d580f refs/tags/v1.0
 003fe92df48743b7bc7d26bcaabfddde0a1e20cae47c refs/tags/v1.0^{}
 0000

Возвращенный ответ представляет собой поток строк pkt, описывающий каждую ссылку и ее текущее значение. Поток ДОЛЖЕН быть отсортирован по имени в соответствии с порядком локали C.

Если HEAD является допустимой ссылкой, HEAD ДОЛЖЕН появиться как первая объявленная ссылка. Если HEAD не является допустимой ссылкой, HEAD НЕ ДОЛЖЕН вообще появляться в списке объявлений, но другие ссылки все равно могут появляться.

Поток ДОЛЖЕН включать объявления возможностей после NUL в первой ссылке. Очищенное значение ссылки (то есть "ref^{}" ) ДОЛЖНО быть сразу после самой ссылки, если она представлена. Соответствующий сервер ДОЛЖЕН очистить ссылку, если это аннотированный тег.

  advertised-refs  =  *1("version 1")
                      (no-refs / list-of-refs)
                      *shallow
                      flush-pkt

  no-refs          =  PKT-LINE(zero-id SP "capabilities^{}"
                      NUL capability-list)

  list-of-refs     =  first-ref *other-ref
  first-ref        =  PKT-LINE(obj-id SP refname
                      NUL capability-list)

  other-ref        =  PKT-LINE(other-tip / other-peeled)
  other-tip        =  obj-id SP refname
  other-peeled     =  obj-id SP refname "^{}"

  shallow          =  PKT-LINE("shallow" SP obj-id)

  capability-list  =  capability *(SP capability)
  capability       =  1*(LC_ALPHA / DIGIT / "-" / "_")
  LC_ALPHA         =  %x61-7A

Сервер и клиент ДОЛЖНЫ использовать строчные буквы для obj-id, оба ДОЛЖНЫ рассматривать obj-id как регистронезависимый.

См. протокол-capabilities.txt для получения списка разрешенных возможностей сервера и их описаний.

Packfile negotiation

После обнаружения ссылок и возможностей клиент может принять решение о прекращении соединения, отправив флеш-pkt, сообщая серверу, что теперь он может корректно завершить соединение и отключиться, когда ему не нужны какие-либо пакетные данные. Это может произойти с помощью команды ls-remote, а также может произойти, когда клиент уже обновлен.

В противном случае он вступает в фазу переговоров, где клиент и сервер определяют минимальный пакетный файл, необходимый для транспортировки, сообщая серверу, какие объекты ему нужны, его мелкие объекты (если есть) и максимальную глубину фиксации, которую он хочет (если таковая имеется). ). Клиент также отправит список возможностей, которые он хочет использовать, исходя из того, что сервер сказал, что он может сделать с первой строкой want .

  upload-request    =  want-list
                       *shallow-line
                       *1depth-request
                       [filter-request]
                       flush-pkt

  want-list         =  first-want
                       *additional-want

  shallow-line      =  PKT-LINE("shallow" SP obj-id)

  depth-request     =  PKT-LINE("deepen" SP depth) /
                       PKT-LINE("deepen-since" SP timestamp) /
                       PKT-LINE("deepen-not" SP ref)

  first-want        =  PKT-LINE("want" SP obj-id SP capability-list)
  additional-want   =  PKT-LINE("want" SP obj-id)

  depth             =  1*DIGIT

  filter-request    =  PKT-LINE("filter" SP filter-spec)

Клиенты ДОЛЖНЫ отправлять все нужные им obj-id на этапе обнаружения ссылок в виде строк want . Клиенты ДОЛЖНЫ отправить хотя бы одну команду want в теле запроса. Клиенты НЕ ДОЛЖНЫ упоминать obj-id в команде want , который не появился в ответе, полученном посредством обнаружения ссылки.

Клиент ДОЛЖЕН записать все obj-id, которые у него есть только поверхностные копии (это означает, что у него нет родителей коммита) в виде строк shallow , чтобы сервер знал об ограничениях истории клиента.

Теперь клиент отправляет максимальную глубину истории коммитов, которую он хочет для этой транзакции, то есть количество коммитов, которые он хочет, из кончика истории, если таковые имеются, в виде строки deepen . Глубина 0 означает отсутствие запроса глубины. Клиент не хочет получать какие-либо коммиты сверх этой глубины, а также ему не нужны объекты, необходимые только для завершения этих коммитов. Коммиты, родители которых в результате не получены, определяются как мелкие и помечаются как таковые на сервере. Эта информация отправляется обратно клиенту на следующем этапе.

Клиент может дополнительно запросить, чтобы объекты пакета исключали различные объекты из файла пакета, используя один из нескольких методов фильтрации. Они предназначены для использования с операциями частичного клонирования и частичной выборки. Объект, который не соответствует значению спецификации фильтра, опускается, если это явно не запрошено в строке want . См. rev-list для возможных значений характеристик фильтра.

После того как все want’s and 'shallow’s (and optional 'deepen передаются, клиенты ДОЛЖНЫ отправить flash-pkt, чтобы сообщить серверной стороне, что отправка списка завершена.

В противном случае, если клиент отправил положительный запрос глубины, сервер определит, какие коммиты будут, а какие нет, и отправит эту информацию клиенту. Если клиент не запрашивал положительную глубину, этот шаг пропускается.

  shallow-update   =  *shallow-line
                      *unshallow-line
                      flush-pkt

  shallow-line     =  PKT-LINE("shallow" SP obj-id)

  unshallow-line   =  PKT-LINE("unshallow" SP obj-id)

Если клиент запросил положительную глубину, сервер вычислит набор коммитов, глубина которых не превышает желаемой. Набор коммитов начинается по желанию клиента.

Сервер записывает строки shallow для каждого коммита, родители которого в результате не будут отправлены. Сервер записывает строку unshallow для каждого коммита, который, по мнению клиента, является поверхностным, но больше не является поверхностным на текущей запрошенной глубине (то есть теперь будут отправлены его родительские элементы). Сервер НЕ ДОЛЖЕН помечать как неглубокий все, что клиент не указал как мелкое.

Теперь клиент отправит список имеющихся у него obj-id, используя строки have , чтобы сервер мог создать пакетный файл, содержащий только те объекты, которые нужны клиенту. В режиме multi_ack каноническая реализация отправит до 32 таких сообщений одновременно, а затем отправит флеш-пкт. Каноническая реализация пропустит вперед и немедленно отправит следующие 32, так что одновременно всегда будет блок из 32 "in-flight on the wire" .

  upload-haves      =  have-list
                       compute-end

  have-list         =  *have-line
  have-line         =  PKT-LINE("have" SP obj-id)
  compute-end       =  flush-pkt / PKT-LINE("done")

Если сервер читает строки have , он затем ответит подтверждением любого из obj-id, которые, по словам клиента, имеются и у сервера. Сервер будет подтверждать объекты по-разному в зависимости от того, какой режим подтверждения выбран клиентом.

В режиме multi_ack:

  • сервер ответит ACK obj-id continue для любых распространенных коммитов.

  • Как только сервер найдет приемлемую общую базовую фиксацию и будет готов создать пакетный файл, он вслепую подтвердит все идентификаторы объектов have обратно клиенту.

  • затем сервер отправит NAK , а затем дождется другого ответа от клиента — либо done , либо другого списка строк have .

В режиме multi_ack_detailed:

  • сервер будет различать ACK, где он сигнализирует о готовности отправлять данные с помощью строк ACK obj-id ready , и сигнализирует об обнаруженных общих фиксации с помощью строк ACK obj-id common .

Без multi_ack или multi_ack_detailed:

  • upload-pack отправляет "ACK obj-id" на первый найденный общий объект. После этого он ничего не говорит, пока клиент не предоставит ему "done" .

  • upload-pack отправляет "NAK" на флеш-пакет, если общий объект еще не найден. Если он был найден и, следовательно, ACK уже был отправлен, на флеш-пкте ничего не происходит.

После того, как клиент получил достаточно ответов ACK, он может определить, что сервер имеет достаточно информации для отправки эффективного паковочного файла (в канонической реализации это определяется, когда он получил достаточно ACK, чтобы он мог раскрасить все, что осталось в --date- очередь заказов как общая с сервером, или очередь --date-order пуста), или клиент определяет, что хочет сдаться (в канонической реализации это определяется, когда клиент отправляет 256 строк have , не получив ни одной из они подтверждены сервером - это означает, что нет ничего общего, и сервер должен просто отправить все свои объекты), тогда клиент отправит команду done . Команда done сигнализирует серверу, что клиент готов получить данные своего пакетного файла.

Однако ограничение в 256 включается только в реализации канонического клиента, если мы получили хотя бы один "ACK %s continue" в течение предыдущего раунда. Это помогает гарантировать, что будет найден хотя бы один общий предок, прежде чем мы полностью сдадимся.

Как только строка done будет прочитана от клиента, сервер либо отправит окончательный ACK obj-id , либо отправит NAK . obj-id — это имя объекта последнего коммита, определенного как общий. Сервер отправляет ACK после done только в том случае, если существует хотя бы одна общая база и включен multi_ack или multi_ack_detailed. Сервер всегда отправляет NAK после done , если не найдена общая база.

Вместо ACK или NAK сервер может отправить сообщение об ошибке (например, если он не распознает объект в строке want , полученной от клиента).

Затем сервер начнет отправлять данные своего паковочного файла.

  server-response = *ack_multi ack / nak
  ack_multi       = PKT-LINE("ACK" SP obj-id ack_status)
  ack_status      = "continue" / "common" / "ready"
  ack             = PKT-LINE("ACK" SP obj-id)
  nak             = PKT-LINE("NAK")

   S: 003aACK 7e47fe2bd8d01d481f44d7af0531bd93d3b21c01 continue\n
   S: 003aACK 74730d410fcb6603ace96f1dc55ea6196122532d continue\n
   S: 0008NAK\n

   C: 0009done\n

   S: 0031ACK 74730d410fcb6603ace96f1dc55ea6196122532d\n
   S: [PACKFILE]

Packfile data

Теперь, когда клиент и сервер завершили согласование минимального объема данных, которые необходимо отправить клиенту, сервер создаст и отправит необходимые данные в формате пакетного файла.

См. gitformat-pack[5] , чтобы узнать, как на самом деле выглядит сам пакетный файл.

Если возможности side-band или side-band-64k были указаны клиентом, сервер отправит данные пакетного файла в мультиплексированном виде.

Каждый пакет начинается с длины строки пакета, соответствующей объему следующих за ним данных, за которым следует один байт, определяющий боковую полосу, по которой поступают следующие данные.

В режиме side-band он отправит до 999 байт данных плюс 1 управляющий код, всего до 1000 байт в pkt-линии. В режиме side-band-64k он отправит до 65519 байт данных плюс 1 управляющий код, всего до 65520 байт в pkt-линии.

Байт боковой полосы будет 1 , 2 или 3 . Боковая полоса 1 будет содержать данные пакетного файла, боковая полоса 2 будет использоваться для информации о ходе выполнения, которую клиент обычно выводит на стандартный поток ошибок, а боковая полоса 3 используется для информации об ошибках.

Если возможность side-band не указана, сервер будет передавать весь пакетный файл без мультиплексирования.

Передача данных на сервер

Отправка данных на сервер вызовет процесс receive-pack на сервере, который позволит клиенту сообщить ему, какие ссылки ему следует обновить, а затем отправить все данные, которые потребуются серверу для завершения этих новых ссылок. Как только все данные будут получены и проверены, сервер обновит свои ссылки в соответствии с тем, что указал клиент.

Authentication

Сам протокол не содержит механизмов аутентификации. Это должно быть обработано транспортом, например SSH, до вызова процесса receive-pack . Если receive-pack настроен на транспорт Git , эти репозитории будут доступны для записи любому, кто имеет доступ к этому порту (9418), поскольку этот транспорт не аутентифицирован.

Reference discovery

Этап обнаружения ссылок выполняется почти так же, как и в протоколе выборки. Каждый ссылочный идентификатор объекта и имя на сервере отправляются клиенту в формате пакетной строки, за которым следует флеш-пкт. Единственное реальное отличие состоит в том, что список возможностей отличается — единственные возможные значения — report-status , report-status-v2 , delete-refs , ofs-delta , atomic и push-options .

Запрос эталонного обновления и передача пакфайла

Как только клиент узнает, по каким ссылкам находится сервер, он может отправить список запросов на обновление ссылок. Для каждой ссылки на сервере, которую он хочет обновить, он отправляет строку со списком obj-id, который в данный момент находится на сервере, obj-id, который клиент хотел бы обновить, и имя ссылки.

За этим списком следует флеш-пкт.

  update-requests   =  *shallow ( command-list | push-cert )

  shallow           =  PKT-LINE("shallow" SP obj-id)

  command-list      =  PKT-LINE(command NUL capability-list)
                       *PKT-LINE(command)
                       flush-pkt

  command           =  create / delete / update
  create            =  zero-id SP new-id  SP name
  delete            =  old-id  SP zero-id SP name
  update            =  old-id  SP new-id  SP name

  old-id            =  obj-id
  new-id            =  obj-id

  push-cert         = PKT-LINE("push-cert" NUL capability-list LF)
                      PKT-LINE("certificate version 0.1" LF)
                      PKT-LINE("pusher" SP ident LF)
                      PKT-LINE("pushee" SP url LF)
                      PKT-LINE("nonce" SP nonce LF)
                      *PKT-LINE("push-option" SP push-option LF)
                      PKT-LINE(LF)
                      *PKT-LINE(command LF)
                      *PKT-LINE(gpg-signature-lines LF)
                      PKT-LINE("push-cert-end" LF)

  push-option       =  1*( VCHAR | SP )

Если сервер объявил о возможности push-options , а клиент указал push-options как часть приведенного выше списка возможностей, клиент затем отправляет свои параметры push, за которыми следует флеш-pkt.

  push-options      =  *PKT-LINE(push-option) flush-pkt

Для обратной совместимости со старыми серверами Git , если клиент отправляет push-сертификат и параметры push-уведомлений, он ДОЛЖЕН отправить свои параметры push-уведомлений как встроенные в push-сертификат, так и после него. (Обратите внимание, что параметры push в сертификате имеют префикс, а параметры push после сертификата — нет.) Оба этих списка ДОЛЖНЫ быть одинаковыми по модулю префикса.

После этого будет отправлен пакетный файл, который должен содержать все объекты, которые понадобятся серверу для завершения новых ссылок.

  packfile          =  "PACK" 28*(OCTET)

Если принимающая сторона не поддерживает ссылки на удаление, отправляющая сторона НЕ ДОЛЖНА запрашивать команду удаления.

Если принимающая сторона не поддерживает push-cert, отправляющая сторона НЕ ДОЛЖНА отправлять команду push-cert. При отправке команды push-cert НЕ ДОЛЖЕН отправляться список команд; Вместо этого используются команды, записанные в push-сертификате.

Пакетный файл НЕ ДОЛЖЕН отправляться, если используется единственная команда — delete .

Пакетный файл ДОЛЖЕН быть отправлен, если используется команда создания или обновления, даже если на сервере уже есть все необходимые объекты. В этом случае клиент ДОЛЖЕН отправить пустой пакетный файл. Это может произойти только в том случае, если клиент создает новую ветку или тег, указывающий на существующий идентификатор объекта.

Сервер получит пакетный файл, распакует его, затем проверит каждую обновляемую ссылку на предмет того, что она не изменилась во время обработки запроса (obj-id остается таким же, как и старый-id), и он запустится. любые перехватчики обновлений, чтобы убедиться, что обновление приемлемо. Если все в порядке, сервер обновит ссылки.

Push certificate

Push-сертификат начинается с набора строк заголовка. После заголовка и пустой строки следуют команды протокола, по одной в строке. Обратите внимание, что завершающий LF в PKT-LINE push-cert является необязательным not ; оно должно присутствовать.

В настоящее время определены следующие поля заголовка:

pusher ident

Определите ключ GPG в формате "Human Readable Name <email@address>" .

pushee url

URL-адрес репозитория (анонимный, если URL-адрес содержит материалы для аутентификации), в который намеревался ввести пользователь, запустивший git push .

nonce nonce

Строку nonce принимающий репозиторий попросил отправляющего пользователя включить в сертификат, чтобы предотвратить атаки повторного воспроизведения.

Строки подписи GPG представляют собой отдельную подпись для содержимого, записанного в push-сертификате до начала блока подписи. Отдельная подпись используется для подтверждения того, что команды были даны отправителем, который должен быть подписывающим лицом.

Report status

После получения пакетных данных от отправителя получатель отправляет отчет, действует ли возможность report-status или report-status-v2 . Это краткий список того, что произошло в этом обновлении. Сначала он укажет статус распаковки паковочного файла как unpack ok или unpack [error] . Затем он отобразит статус каждой ссылки, которую он пытался обновить. Каждая строка имеет код ok [refname] , если обновление прошло успешно, или ng [refname] [error] , если обновление не прошло.

  report-status     = unpack-status
                      1*(command-status)
                      flush-pkt

  unpack-status     = PKT-LINE("unpack" SP unpack-result)
  unpack-result     = "ok" / error-msg

  command-status    = command-ok / command-fail
  command-ok        = PKT-LINE("ok" SP refname)
  command-fail      = PKT-LINE("ng" SP refname SP error-msg)

  error-msg         = 1*(OCTET) ; where not "ok"

Возможность report-status-v2 расширяет протокол за счет добавления новых строк опций для поддержки отчетов о ссылках, переписанных перехватчиком proc-receive . Перехватчик proc-receive может обрабатывать команду псевдоссылки, которая может создавать или обновлять одну или несколько ссылок, и каждая ссылка может иметь разное имя, разные новые идентификаторы и разные старые идентификаторы.

  report-status-v2  = unpack-status
                      1*(command-status-v2)
                      flush-pkt

  unpack-status     = PKT-LINE("unpack" SP unpack-result)
  unpack-result     = "ok" / error-msg

  command-status-v2 = command-ok-v2 / command-fail
  command-ok-v2     = command-ok
                      *option-line

  command-ok        = PKT-LINE("ok" SP refname)
  command-fail      = PKT-LINE("ng" SP refname SP error-msg)

  error-msg         = 1*(OCTET) ; where not "ok"

  option-line       = *1(option-refname)
                      *1(option-old-oid)
                      *1(option-new-oid)
                      *1(option-forced-update)

  option-refname    = PKT-LINE("option" SP "refname" SP refname)
  option-old-oid    = PKT-LINE("option" SP "old-oid" SP obj-id)
  option-new-oid    = PKT-LINE("option" SP "new-oid" SP obj-id)
  option-force      = PKT-LINE("option" SP "forced-update")

Обновления могут быть неудачными по ряду причин. Ссылка могла измениться с момента первоначальной отправки фазы обнаружения ссылки, что означает, что кто-то тем временем нажал. Пересылаемая ссылка может быть ссылкой без перемотки вперед, и перехватчики обновления или конфигурация могут быть настроены так, чтобы не допускать этого, и т. д. Кроме того, некоторые ссылки могут быть обновлены, а другие могут быть отклонены.
