git-receive-pack

Name

git-receive-pack — получить то, что помещено в репозиторий

Synopsis

git receive-pack <git-dir>

Description

Вызывается git send-pack и обновляет репозиторий информацией, поступающей с удаленного конца.

Эта команда обычно не вызывается конечным пользователем напрямую. Пользовательский интерфейс протокола находится на стороне git send-pack , и эта пара программ предназначена для отправки обновлений в удаленный репозиторий. Информацию об операциях извлечения см. в git-fetch-pack[1] .

Команда позволяет создавать и быстро пересылать ссылки sha1 (заголовки/теги) на удаленном конце (строго говоря, это локальный конец, на котором работает git-receive-pack , но для пользователя, который сидит на конце пакета отправки, это обновление пульта. Запутались?)

Существуют и другие реальные примеры использования обработчиков обновления и постобновления, которые можно найти в каталоге Documentation/howto.

git-receive-pack учитывает параметр конфигурации получения.denyNonFastForwards, который сообщает ему, следует ли запрещать обновления ссылки, если они не перемотаны вперед.

Доступен ряд других параметров конфигурации получения.* для настройки его поведения, см. git-config[1] .

Options

<git-dir>

Репозиторий для синхронизации.

--http-backend-info-refs

Используется git-http-backend[1] для обслуживания запросов $GIT_URL/info/refs?service=git-receive-pack . См. --http-backend-info-refs в git-upload-pack[1] .

Pre-receive hook

Прежде чем какая-либо ссылка будет обновлена, если файл $GIT_DIR/hooks/pre-receive существует и является исполняемым, он будет вызван один раз без параметров. Стандартный ввод перехватчика будет состоять из одной строки для каждой обновляемой ссылки:

sha1-old SP sha1-new SP refname LF

Значение refname относится к $GIT_DIR; например, для главной головки это "refs/heads/master" . Два значения sha1 перед каждым именем ссылки — это имена объектов для имени ссылки до и после обновления. Создаваемые ссылки будут иметь sha1-old, равный 0{40}, а удаляемые ссылки будут иметь sha1-new, равный 0{40}, в противном случае sha1-old и sha1-new должны быть допустимыми объектами в репозитории.

При принятии подписанного push-уведомления (см. git-push[1] ) подписанный push-сертификат сохраняется в большом двоичном объекте, а имя его объекта можно узнать в переменной среды GIT_PUSH_CERT . Пример см. в описании крючка post-receive . Кроме того, сертификат проверяется с помощью GPG, а результат экспортируется со следующими переменными среды:

GIT_PUSH_CERT_SIGNER

Имя и адрес электронной почты владельца ключа, подписавшего push-сертификат.

GIT_PUSH_CERT_KEY

Идентификатор ключа GPG ключа, которым был подписан push-сертификат.

GIT_PUSH_CERT_STATUS

Статус проверки GPG push-сертификата с использованием той же мнемоники, что и в формате %G? семейства команд git log (см. git-log[1] ).

GIT_PUSH_CERT_NONCE

Строку nonce, которую процесс попросил подписавшую включить в push-сертификат. Если оно не соответствует значению, записанному в заголовке "nonce" в push-сертификате, это может указывать на то, что сертификат является действительным и воспроизводится из отдельного сеанса "git push" .

GIT_PUSH_CERT_NONCE_STATUS
UNSOLICITED

"git push --signed" отправил одноразовый номер, хотя мы не просили его об этом.

MISSING

"git push --signed" не отправил ни одного заголовка nonce.

BAD

"git push --signed" отправил поддельный одноразовый номер.

OK

"git push --signed" отправил одноразовый номер, который мы просили отправить.

SLOP

"git push --signed" отправил одноразовый номер, отличный от того, который мы просили отправить сейчас, но в предыдущем сеансе. См. переменную среды GIT_PUSH_CERT_NONCE_SLOP .

GIT_PUSH_CERT_NONCE_SLOP

"git push --signed" отправил одноразовый номер, отличный от того, который мы просили отправить сейчас, но в другом сеансе, время начала которого отличается на это количество секунд от текущего сеанса. Имеет смысл только тогда, когда GIT_PUSH_CERT_NONCE_STATUS говорит SLOP . Также прочтите о переменной receive.certNonceSlop в git-config[1] .

Этот хук вызывается перед обновлением любого имени ссылки и перед выполнением каких-либо проверок вперед.

Если перехватчик предварительного получения завершается с ненулевым статусом выхода, обновления выполняться не будут, а перехватчики обновления, пост-получения и пост-обновления также не будут вызываться. Это может быть полезно для быстрого выхода из ситуации, если обновление не поддерживается.

См. примечания о карантинной среде ниже.

Update hook

Перед обновлением каждой ссылки, если файл $GIT_DIR/hooks/update существует и является исполняемым, он вызывается один раз для каждой ссылки с тремя параметрами:

$GIT_DIR/hooks/update refname sha1-old sha1-new

Параметр refname относится к $GIT_DIR; например, для главной головки это "refs/heads/master" . Два аргумента sha1 — это имена объектов для ссылки до и после обновления. Обратите внимание, что перехват вызывается до обновления имени ссылки, поэтому либо sha1-old равен 0{40} (то есть такой ссылки еще нет), либо он должен соответствовать тому, что записано в имени ссылки.

Хук должен выйти с ненулевым статусом, если он хочет запретить обновление именованной ссылки. В противном случае он должен выйти с нулем.

Успешное выполнение (нулевой статус выхода) этого перехватчика не гарантирует, что ссылка действительно будет обновлена, это всего лишь предварительное условие. По этой причине отправлять уведомления (например, по электронной почте) с помощью этого крючка не рекомендуется. Вместо этого рассмотрите возможность использования перехватчика post-receive.

Post-receive hook

После обновления всех ссылок (или попыток обновления), если какое-либо обновление ссылок прошло успешно и если файл $GIT_DIR/hooks/post-receive существует и является исполняемым, он будет вызван один раз без параметров. Стандартный ввод перехватчика будет представлять собой одну строку для каждой успешно обновленной ссылки:

sha1-old SP sha1-new SP refname LF

Значение refname относится к $GIT_DIR; например, для главной головки это "refs/heads/master" . Два значения sha1 перед каждым именем ссылки — это имена объектов для имени ссылки до и после обновления. Созданные ссылки будут иметь sha1-old, равный 0{40}, а удаленные ссылки будут иметь sha1-new, равный 0{40}, в противном случае sha1-old и sha1-new должны быть допустимыми объектами в репозитории.

Переменные среды GIT_PUSH_CERT* можно проверить, как и в ловушке pre-receive , после принятия подписанного push-уведомления.

Используя этот хук, легко генерировать письма с описанием обновлений репозитория. Этот пример сценария отправляет одно почтовое сообщение на каждую ссылку со списком коммитов, отправленных в репозиторий, и регистрирует push-сертификаты подписанных push-уведомлений с хорошими подписями в службу регистрации:

#!/bin/sh
# рассылка информации об обновлении фиксации.
while read oval nval ref
do
        if expr "$oval" : '0*$' >/dev/null
        then
                echo "Created a new ref, with the following commits:"
                git rev-list --pretty "$nval"
        else
                echo "New commits:"
                git rev-list --pretty "$nval" "^$oval"
        fi |
        mail -s "Changes to ref $ref" commit-list@mydomain
done
# журнал подписанного push-сертификата, если таковой имеется
if test -n "${GIT_PUSH_CERT-}" && test ${GIT_PUSH_CERT_STATUS} = G
then
        (
                echo expected nonce is ${GIT_PUSH_NONCE}
                git cat-file blob ${GIT_PUSH_CERT}
        ) | mail -s "push certificate from $GIT_PUSH_CERT_SIGNER" push-log@mydomain
fi
exit 0

Код выхода из этого вызова ловушки игнорируется, однако код выхода, отличный от нуля, выдаст сообщение об ошибке.

Обратите внимание, что при запуске этого перехватчика в refname может не быть sha1-new. Это может легко произойти, если другой пользователь изменит ссылку после того, как она была обновлена ​git-receive-pack , но до того, как перехватчик смог ее оценить. Рекомендуется, чтобы перехватчики полагались на sha1-new, а не на текущее значение refname.

Post-update hook

После всей остальной обработки, если хотя бы одна ссылка была обновлена ​​и если файл $GIT_DIR/hooks/post-update существует и является исполняемым, то post-update будет вызван со списком обновленных ссылок. Это можно использовать для реализации любых задач очистки всего репозитория.

Код выхода из этого вызова ловушки игнорируется; единственное, что остается сделать git-receive-pack в этот момент, — это все равно выйти из себя.

Этот хук можно использовать, например, для запуска git update-server-info , если репозиторий упакован и обслуживается через тупой транспорт.

#!/bin/sh
exec git update-server-info

Quarantine environment

Когда receive-pack принимает объекты, они помещаются во временный каталог "quarantine" внутри каталога $GIT_DIR/objects и переносятся в основное хранилище объектов только после завершения перехвата pre-receive . Если передача не удалась до этого момента, временный каталог полностью удаляется.

Это имеет несколько видимых пользователем эффектов и предостережений:

  1. Push-передачи, которые завершились неудачно из-за проблем с входящим пакетом, отсутствия объектов или из-за перехватчика pre-receive , не оставят никаких данных на диске. Обычно это помогает предотвратить заполнение диска повторными неудачными попытками отправки, но может усложнить отладку.

  2. Любые объекты, созданные перехватчиком pre-receive , будут созданы в каталоге карантина (и перенесены только в случае успеха).

  3. Перехватчик pre-receive НЕ ДОЛЖЕН обновлять какие-либо ссылки, чтобы они указывали на объекты, помещенные в карантин. Другие программы, обращающиеся к репозиторию, не смогут видеть объекты (и если перехват предварительного получения не удастся, эти ссылки будут повреждены). В целях безопасности любые обновления ссылок изнутри pre-receive автоматически отклоняются.
