git-revert

Name

git-revert — отменить некоторые существующие коммиты

Synopsis

git revert [--[no-]edit] [-n] [-m <parent-number>] [-s] [-S[<keyid>]] <commit>…​
git revert (--continue | --skip | --abort | --quit)

Description

Учитывая один или несколько существующих коммитов, отмените изменения, внесенные соответствующими патчами, и запишите несколько новых коммитов, которые их записывают. Для этого необходимо, чтобы ваше рабочее дерево было чистым (без изменений после фиксации HEAD).

Примечание. git revert используется для записи некоторых новых коммитов, чтобы отменить эффект некоторых предыдущих коммитов (часто только ошибочных). Если вы хотите удалить все незафиксированные изменения в вашем рабочем каталоге, вы должны увидеть git-reset[1] , особенно опцию --hard . Если вы хотите извлечь определенные файлы в том виде, в каком они были в другом коммите, вы должны увидеть git-restore[1] , а именно опцию --source . Будьте осторожны с этими альтернативами, поскольку оба будут отбрасывать незафиксированные изменения в вашем рабочем каталоге.

См. "Reset, restore and revert" в git[1] , чтобы узнать о различиях между тремя командами.

Options

<commit>…

Обязуется вернуться. Более полный список способов написания имен коммитов см. в gitrevisions[7] . Также могут быть заданы наборы коммитов, но по умолчанию обход не выполняется, см. git-rev-list[1] и его опцию --no-walk .

-e
--edit

С помощью этой опции git revert позволит вам редактировать сообщение о фиксации перед фиксацией возврата. Это значение по умолчанию, если вы запускаете команду с терминала.

-m parent-number
--mainline parent-number

Обычно вы не можете отменить слияние, поскольку не знаете, какую сторону слияния следует считать основной линией. Эта опция указывает родительский номер (начиная с 1) основной линии и позволяет отменить изменение относительно указанного родительского элемента.

Отмена фиксации слияния означает, что вы никогда не захотите вносить изменения в дерево, внесенные слиянием. В результате более поздние слияния будут приносить только изменения в дереве, внесенные коммитами, которые не являются предками ранее отмененного слияния. Это может быть, а может и не быть тем, чего вы хотите.

Более подробную информацию см. в revert-a-faulty-merge How-To .

--no-edit

При использовании этой опции git revert не будет запускать редактор сообщений о фиксации.

--cleanup=<mode>

Этот параметр определяет, как будет очищаться сообщение о фиксации перед передачей механизму фиксации. Дополнительную информацию см. в git-commit[1] . В частности, если <mode> присвоено значение scissors , ножницы будут добавлены к MERGE_MSG перед передачей дальше в случае конфликта.

-n
--no-commit

Обычно команда автоматически создает несколько коммитов с сообщениями журнала коммитов, в которых указывается, какие коммиты были отменены. Этот флаг применяет изменения, необходимые для возврата именованных коммитов в ваше рабочее дерево и индекс, но не выполняет коммиты. Кроме того, при использовании этой опции ваш индекс не обязательно должен соответствовать фиксации HEAD. Возврат выполняется к начальному состоянию вашего индекса.

Это полезно при возврате эффекта более чем одного коммита к вашему индексу подряд.

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

GPG-знак фиксируется. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

-s
--signoff

Добавьте трейлер Signed-off-by в конце сообщения о фиксации. Дополнительную информацию см. в разделе «Выписка» в git-commit[1] .

--strategy=<strategy>

Используйте данную стратегию слияния. Следует использовать только один раз. Подробности см. в разделе «СТРАТЕГИИ СЛИЯНИЯ» в git-merge[1] .

-X<option>
--strategy-option=<option>

Передайте параметр, специфичный для стратегии слияния, в стратегию слияния. Подробности см. в git-merge[1] .

--rerere-autoupdate
--no-rerere-autoupdate

После того, как механизм rerere повторно использует записанное разрешение текущего конфликта для обновления файлов в рабочем дереве, разрешите ему также обновить индекс с результатом разрешения. --no-rerere-autoupdate — хороший способ перепроверить то, что сделал rerere , и выявить возможные ошибки слияния, прежде чем заносить результат в индекс с помощью отдельного git add .

--reference

Вместо того, чтобы начинать тело сообщения журнала с "This reverts <full object name of the commit being reverted>." , обратитесь к фиксации, используя формат "--pretty=reference" (см. git-log[1] ). Переменную конфигурации revert.reference можно использовать для включения этой опции по умолчанию.

Sequencer subcommands

--continue

Продолжите выполняемую операцию, используя информацию в .git/sequencer . Может использоваться для продолжения после разрешения конфликтов при неудачном выборе или возврата.

--skip

Пропустите текущую фиксацию и продолжите выполнение остальной части последовательности.

--quit

Забудьте о текущей выполняемой операции. Может использоваться для очистки состояния секвенсора после неудачного выбора вишни или возврата.

--abort

Отмените операцию и вернитесь в состояние до последовательности.

Examples

git revert HEAD~3

Отмените изменения, указанные в четвертом последнем коммите в HEAD, и создайте новый коммит с отмененными изменениями.

git revert -n master~5..master~2

Отмените изменения, внесенные коммитами с пятого последнего коммита в master (включительно) до третьего последнего коммита в master (включительно), но не создавайте коммитов с отмененными изменениями. Возврат изменяет только рабочее дерево и индекс.

Discussion

Хотя git автоматически создает базовое сообщение о фиксации, strongly рекомендуется объяснить, почему исходная фиксация отменяется. Кроме того, многократное откат приведет к созданию более громоздких строк темы, например Reapply "Reapply "<original subject>"" . Пожалуйста, рассмотрите возможность их изменения, чтобы они были короче и более уникальными.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

revert.reference

Установка для этой переменной значения true приводит к тому, что git revert ведет себя так, как если бы была указана опция --reference .
