git-cherry

Name

git-cherry — найти коммиты, которые еще не были применены к восходящему потоку

Synopsis

git cherry [-v] [<upstream> [<head> [<limit>]]]

Description

Определите, есть ли в <head>..<upstream> коммиты, эквивалентные коммитам в диапазоне <limit>..<head> .

Проверка эквивалентности основана на разнице после удаления пробелов и номеров строк. Поэтому git-cherry определяет, когда коммиты были "copied" , с помощью git-cherry-pick[1] , git-am[1] или git-rebase[1] .

Выводит SHA1 каждого коммита в <limit>..<head> с префиксом - для коммитов, имеющих эквивалент в <upstream>, и + для коммитов, у которых его нет.

Options

-v

Покажите темы коммитов рядом с SHA1.

<upstream>

Восходящая ветвь для поиска эквивалентных коммитов. По умолчанию используется восходящая ветвь HEAD.

<head>

Рабочая ветка; по умолчанию — HEAD.

<limit>

Не сообщайте о коммитах до предела (включительно).

Examples

Patch workflows

git-cherry часто используется в рабочих процессах на основе исправлений (см. gitworkflows[7] ), чтобы определить, была ли применена серия исправлений вышестоящим сопровождающим. В таком рабочем процессе вы можете создать и отправить ветку темы следующим образом:

$ git checkout -b topic origin/master
# work and create some commits
$ git format-patch origin/master
$ git send-email ... 00*

Позже вы сможете увидеть, были ли применены ваши изменения, сказав (все еще на topic ):

$ git fetch  # обновите ваше представление о происхождении/хозяине
$ git cherry -v

Concrete example

В ситуации, когда тема состояла из трех коммитов, а мейнтейнер применил два из них, ситуация может выглядеть так:

$ git log --graph --oneline --decorate --boundary origin/master...topic
* 7654321 (origin/master) upstream tip commit
[... snip some other commits ...]
* cccc111 cherry-pick of C
* aaaa111 cherry-pick of A
[... snip a lot more that has happened ...]
| * cccc000 (topic) commit C
| * bbbb000 commit B
| * aaaa000 commit A
|/
o 1234567 branch point

В таких случаях git-cherry показывает краткое описание того, что еще предстоит применить:

$ git cherry origin/master topic
- cccc000... commit C
+ bbbb000... commit B
- aaaa000... commit A

Здесь мы видим, что коммиты A и C (отмеченные - ) могут быть удалены из вашей ветки topic при перебазировании ее поверх origin/master , в то время как коммит B (отмеченный + ) все еще необходимо сохранить, чтобы он был отправлено для применения в origin/master .

Использование лимита

Необязательный <limit> полезен в тех случаях, когда ваша тема основана на другой работе, не входящей в состав основной ветки разработки. Если расширить предыдущий пример, это может выглядеть так:

$ git log --graph --oneline --decorate --boundary origin/master...topic
* 7654321 (origin/master) upstream tip commit
[... snip some other commits ...]
* cccc111 cherry-pick of C
* aaaa111 cherry-pick of A
[... snip a lot more that has happened ...]
| * cccc000 (topic) commit C
| * bbbb000 commit B
| * aaaa000 commit A
| * 0000fff (base) unpublished stuff F
[... snip ...]
| * 0000aaa unpublished stuff A
|/
o 1234567 merge-base between upstream and topic

Указав base в качестве ограничения, вы можете избежать перечисления коммитов между base и topic :

$ git cherry origin/master topic base
- cccc000... commit C
+ bbbb000... commit B
- aaaa000... commit A
