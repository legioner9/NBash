git-merge-file

Name

git-merge-file — запустить трехстороннее слияние файлов

Synopsis

git merge-file [-L <current-name> [-L <base-name> [-L <other-name>]]]
        [--ours|--theirs|--union] [-p|--stdout] [-q|--quiet] [--marker-size=<n>]
        [--[no-]diff3] [--object-id] <current> <base> <other>

Description

Учитывая три файла <current> , <base> и <other> , git merge-file включает в себя все изменения, которые ведут от <base> к <other> , в <current> . Результат обычно попадает в <current> . git merge-file полезен для объединения отдельных изменений оригинала. Предположим, <base> — оригинал, а <current> и <other> — модификации <base> , тогда git merge-file объединяет оба изменения.

Конфликт возникает, если и <current> , и <other> имеют изменения в общем сегменте строк. Если обнаружен конфликт, git merge-file обычно выводит предупреждение и заключает конфликт в скобки со строками, содержащими маркеры <<<<<<< и >>>>>>>. Типичный конфликт будет выглядеть так:

<<<<<<< A
lines in file A
=======
lines in file B
>>>>>>> B

Если есть конфликты, пользователю следует отредактировать результат и удалить одну из альтернатив. Однако, когда действует опция --ours , --theirs или --union , эти конфликты разрешаются в пользу строк из <current> , строк из <other> или строк из обоих соответственно. Длину маркеров конфликта можно задать с помощью опции --marker-size .

Если указан --object-id , происходит точно такое же поведение, за исключением того, что вместо указания того, что объединять в виде файлов, указывается список идентификаторов объектов, ссылающихся на большие двоичные объекты.

Значение выхода этой программы является отрицательным в случае ошибки, а в противном случае — число конфликтов (усекается до 127, если конфликтов больше). Если слияние было чистым, значение выхода равно 0.

git merge-file представляет собой минимальный клон RCS merge ; то есть он реализует все функции RCS merge , необходимые для git[1] .

Options

--object-id

Укажите содержимое, которое нужно объединить в виде больших двоичных объектов в текущем репозитории, а не в виде файлов. В этом случае операция должна выполняться в допустимом репозитории.

Если указана опция -p , объединенный файл (включая конфликты, если таковые имеются) отправляется на стандартный вывод как обычно; в противном случае объединенный файл записывается в хранилище объектов, а идентификатор объекта его большого двоичного объекта записывается в стандартный вывод.

-L <label>

Эту опцию можно использовать до трех раз, и она определяет метки, которые будут использоваться вместо соответствующих имен файлов в отчетах о конфликтах. То есть git merge-file -L x -L y -L z a b c генерирует выходные данные, которые выглядят так, как будто они получены из файлов x, y и z, а не из файлов a, b и c.

-p

Отправьте результаты на стандартный вывод вместо перезаписи <current> .

-q

Тихий; не предупреждать о конфликтах.

--diff3

Показывать конфликты в стиле "diff3" .

--zdiff3

Показывать конфликты в стиле "zdiff3" .

--ours
--theirs
--union

Вместо того, чтобы оставлять конфликты в файле, разрешайте конфликты в пользу нашей (или их, или обеих) сторон строк.

Examples

git merge-file README.my README README.upstream

объединяет изменения README.my и README.upstream с момента README, пытается объединить их и записывает результат в README.my.

git merge-file -L a -L b -L c tmp/a123 tmp/b234 tmp/c345

объединяет tmp/a123 и tmp/c345 с базовым tmp/b234, но использует метки a и c вместо tmp/a123 и tmp/c345 .

git merge-file -p --object-id abc1234 def567 890abcd

объединяет изменения blob abc1234 и 890abcd начиная с def567, пытается объединить их и записывает результат в стандартный вывод
