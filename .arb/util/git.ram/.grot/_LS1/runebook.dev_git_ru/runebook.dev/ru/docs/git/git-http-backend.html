git-http-backend

Name

git-http-backend — реализация Git на стороне сервера через HTTP.

Synopsis

git http-backend

Description

Простая программа CGI для предоставления содержимого репозитория Git клиентам Git , обращающимся к репозиторию по протоколам http:// и https://. Программа поддерживает выборку клиентов с использованием как интеллектуального протокола HTTP, так и обратно совместимого тупого протокола HTTP, а также отправку клиентов с использованием интеллектуального протокола HTTP. Он также поддерживает более эффективный протокол "v2" Git , если он правильно настроен; см. обсуждение GIT_PROTOCOL в разделе ОКРУЖАЮЩАЯ СРЕДА ниже.

Он проверяет, что в каталоге есть волшебный файл "git-daemon-export-ok" , и отказывается экспортировать любой каталог Git , который не был явно помечен для экспорта таким образом (если не установлена ​​переменная среды GIT_HTTP_EXPORT_ALL ).

По умолчанию включена только служба upload-pack , которая обслуживает клиентов git fetch-pack и git ls-remote , которые вызываются из git fetch , git pull и git clone . Если клиент аутентифицирован, включается служба receive-pack , которая обслуживает клиентов git send-pack , вызываемая из git push .

Services

Эти службы можно включить/отключить с помощью файла конфигурации для каждого репозитория:

http.getanyfile

Это обслуживает клиентов Git старше версии 1.6.6, которые не могут использовать службу загрузки пакетов. Если этот параметр включен, клиенты могут читать любой файл в репозитории, включая объекты, которые больше не доступны из ветки, но все еще присутствуют. Он включен по умолчанию, но репозиторий может отключить его, установив для этого значения конфигурации значение false .

http.uploadpack

Он обслуживает клиентов git fetch-pack и git ls-remote . Он включен по умолчанию, но репозиторий может отключить его, установив для этого значения конфигурации значение false .

http.receivepack

Это обслуживает клиентов git send-pack , позволяя осуществлять push. По умолчанию он отключен для анонимных пользователей и включен по умолчанию для пользователей, аутентифицированных веб-сервером. Его можно отключить, установив для этого элемента значение false , или включить для всех пользователей, включая анонимных пользователей, установив для него значение true .

Url translation

Чтобы определить расположение репозитория на диске, git http-backend объединяет переменные среды PATH_INFO, которые устанавливаются автоматически веб-сервером, и GIT_PROJECT_ROOT, которые необходимо установить вручную в конфигурации веб-сервера. Если GIT_PROJECT_ROOT не установлен, git http-backend считывает PATH_TRANSLATED, который также автоматически устанавливается веб-сервером.

Examples

Все следующие примеры сопоставляют http://$hostname/git/foo/bar.git с /var/www/git/foo/bar.git .

Apache 2.x

Убедитесь, что mod_cgi, mod_alias и mod_env включены, установите GIT_PROJECT_ROOT (или DocumentRoot) соответствующим образом и создайте ScriptAlias ​​для CGI:

SetEnv GIT_PROJECT_ROOT /var/www/git
SetEnv GIT_HTTP_EXPORT_ALL
ScriptAlias /git/ /usr/libexec/git-core/git-http-backend/

# Это не является строго необходимым при использовании Apache и современной версии
# git-http-backend, так как веб-сервер передаст заголовок в
# среду как HTTP_GIT_PROTOCOL, и http-backend скопирует это в
# GIT_ПРОТОКОЛ. Но вам может понадобиться эта строка (или что-то подобное, если вы
# используете другой веб-сервер), или если вы хотите поддерживать более старую версию Git
# версии, в которых это копирование не выполнялось.
#
# Настроить GIT_PROTOCOL на веб-сервере вполне нормально, даже если
# современные версии (и будут иметь приоритет над HTTP_GIT_PROTOCOL,
# что означает, что его можно использовать для отмены запроса клиента).
SetEnvIf Git-Protocol ".*" GIT_PROTOCOL=$0

Чтобы включить анонимный доступ для чтения, но аутентифицированный доступ для записи, потребуется авторизация как для первоначального объявления ref (которое мы определяем как push-уведомление через параметр службы в строке запроса), так и для самого вызова пакета получения:

RewriteCond %{QUERY_STRING} service=git-receive-pack [OR]
RewriteCond %{REQUEST_URI} /git-receive-pack$
RewriteRule ^/git/ - [E=AUTHREQUIRED:yes]

<LocationMatch "^/git/">
        Order Deny,Allow
        Deny from env=AUTHREQUIRED

        AuthType Basic
        AuthName "Git Access"
        Require group committers
        Satisfy Any
        ...
</LocationMatch>

Если у вас нет mod_rewrite , доступного для сопоставления со строкой запроса, достаточно просто защитить сам git-receive-pack , например:

<LocationMatch "^/git/.*/git-receive-pack$">
        AuthType Basic
        AuthName "Git Access"
        Require group committers
        ...
</LocationMatch>

В этом режиме сервер не будет запрашивать аутентификацию до тех пор, пока клиент фактически не начнет фазу согласования объекта push-уведомления, а не во время первоначального контакта. По этой причине вы также должны включить опцию конфигурации http.receivepack во всех репозиториях, которые должны принимать push-уведомления. Поведение по умолчанию, если http.receivepack не установлен, заключается в отклонении любых запросов от неаутентифицированных пользователей; Таким образом, первоначальный запрос сообщит клиенту 403 Forbidden , даже не давая возможности для аутентификации.

Чтобы требовать аутентификацию как для чтения, так и для записи, используйте директиву Location для репозитория или одного из его родительских каталогов:

<Location /git/private>
        AuthType Basic
        AuthName "Private Git Access"
        Require group committers
        ...
</Location>

Чтобы обслуживать gitweb по тому же URL-адресу, используйте ScriptAliasMatch только для тех URL-адресов, которые может обрабатывать git http-backend , а остальные перенаправьте в gitweb:

ScriptAliasMatch \
        "(?x)^/git/(.*/(HEAD | \
                        info/refs | \
                        objects/(info/[^/]+ | \
                                 [0-9a-f]{2}/[0-9a-f]{38} | \
                                 pack/pack-[0-9a-f]{40}\.(pack|idx)) | \
                        git-(upload|receive)-pack))$" \
        /usr/libexec/git-core/git-http-backend/$1

ScriptAlias /git/ /var/www/cgi-bin/gitweb.cgi/

Чтобы обслуживать несколько репозиториев из разных gitnamespaces[7] в одном репозитории:

SetEnvIf Request_URI "^/git/([^/]*)" GIT_NAMESPACE=$1
ScriptAliasMatch ^/git/[^/]*(.*) /usr/libexec/git-core/git-http-backend/storage.git$1
Ускоренный static Apache 2.x

Аналогично описанному выше, но Apache можно использовать для возврата файлов static , хранящихся на диске. Во многих системах это может быть более эффективно, поскольку Apache может попросить ядро ​​скопировать содержимое файла из файловой системы непосредственно в сеть:

SetEnv GIT_PROJECT_ROOT /var/www/git

AliasMatch ^/git/(.*/objects/[0-9a-f]{2}/[0-9a-f]{38})$          /var/www/git/$1
AliasMatch ^/git/(.*/objects/pack/pack-[0-9a-f]{40}.(pack|idx))$ /var/www/git/$1
ScriptAlias /git/ /usr/libexec/git-core/git-http-backend/

Это можно объединить с конфигурацией gitweb:

SetEnv GIT_PROJECT_ROOT /var/www/git

AliasMatch ^/git/(.*/objects/[0-9a-f]{2}/[0-9a-f]{38})$          /var/www/git/$1
AliasMatch ^/git/(.*/objects/pack/pack-[0-9a-f]{40}.(pack|idx))$ /var/www/git/$1
ScriptAliasMatch \
        "(?x)^/git/(.*/(HEAD | \
                        info/refs | \
                        objects/info/[^/]+ | \
                        git-(upload|receive)-pack))$" \
        /usr/libexec/git-core/git-http-backend/$1
ScriptAlias /git/ /var/www/cgi-bin/gitweb.cgi/
Lighttpd

Убедитесь, что mod_cgi , mod_alias , mod_auth , mod_setenv загружены, затем установите GIT_PROJECT_ROOT соответствующим образом и перенаправьте все запросы в CGI:

alias.url += ( "/git" => "/usr/lib/git-core/git-http-backend" )
$HTTP["url"] =~ "^/git" {
        cgi.assign = ("" => "")
        setenv.add-environment = (
                "GIT_PROJECT_ROOT" => "/var/www/git",
                "GIT_HTTP_EXPORT_ALL" => ""
        )
}

Чтобы включить анонимный доступ для чтения, но доступ для записи с аутентификацией:

$HTTP["querystring"] =~ "service=git-receive-pack" {
        include "git-auth.conf"
}
$HTTP["url"] =~ "^/git/.*/git-receive-pack$" {
        include "git-auth.conf"
}

где git-auth.conf выглядит примерно так:

auth.require = (
        "/" => (
                "method" => "basic",
                "realm" => "Git Access",
                "require" => "valid-user"
               )
)
# ...и настройте здесь auth.backend

Чтобы потребовать аутентификацию как для чтения, так и для записи:

$HTTP["url"] =~ "^/git/private" {
        include "git-auth.conf"
}

Environment

git http-backend использует переменные среды CGI , установленные вызывающим веб-сервером, в том числе:

  • PATH_INFO (если установлен GIT_PROJECT_ROOT, в противном случае PATH_TRANSLATED)

  • REMOTE_USER

  • REMOTE_ADDR

  • CONTENT_TYPE

  • QUERY_STRING

  • REQUEST_METHOD

Переменную среды GIT_HTTP_EXPORT_ALL можно передать в git-http-backend , чтобы обойти проверку файла "git-daemon-export-ok" в каждом репозитории перед разрешением экспорта этого репозитория.

Переменная среды GIT_HTTP_MAX_REQUEST_BUFFER (или опция конфигурации http.maxRequestBuffer ) может быть установлена ​​для изменения самого большого запроса на согласование ссылки, который git будет обрабатывать во время выборки; любая выборка, требующая большего буфера, не будет успешной. Обычно это значение не требуется изменять, но оно может оказаться полезным, если вы извлекаете данные из репозитория с чрезвычайно большим количеством ссылок. Значение можно указать в единицах измерения (например, 100M для 100 мегабайт). По умолчанию — 10 мегабайт.

Клиенты могут проверять возможности дополнительных протоколов (например, протокола v2), используя HTTP-заголовок Git-Protocol . Для их поддержки содержимое этого заголовка должно появиться в переменной среды GIT_PROTOCOL . Большинство веб-серверов передадут этот заголовок в CGI через переменную HTTP_GIT_PROTOCOL , а git-http-backend автоматически скопирует его в GIT_PROTOCOL . Однако некоторые веб-серверы могут более избирательно выбирать заголовки, которые они будут передавать, и в этом случае их необходимо настроить явно (см. упоминание Git-Protocol в конфигурации Apache из предыдущего раздела ПРИМЕРЫ).

Внутренний процесс устанавливает GIT_COMMITTER_NAME в $REMOTE_USER и GIT_COMMITTER_EMAIL в ${REMOTE_USER}@http.${REMOTE_ADDR} , гарантируя, что любые журналы обновлений, созданные git-receive-pack , содержат некоторую идентифицирующую информацию удаленного пользователя, выполнившего отправку.

Все переменные среды CGI доступны для каждого из перехватчиков, вызываемых git-receive-pack .
