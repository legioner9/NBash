api-trace2

API Trace2 можно использовать для вывода информации об отладке, производительности и телеметрии в стандартный поток ошибок или в файл. Функция Trace2 неактивна, если она явно не включена путем включения одной или нескольких целей Trace2.

API Trace2 предназначен для замены существующей трассировки в стиле printf() (Trace1), предоставляемой существующими средствами GIT_TRACE и GIT_TRACE_PERFORMANCE . Во время первоначальной реализации Trace2 и Trace1 могут работать параллельно.

API Trace2 определяет набор сообщений высокого уровня с известными полями, например ( start : argv ) и ( exit : { exit-code , elapsed-time }).

Инструменты Trace2 по всей базе кода Git отправляют сообщения Trace2 включенным целям Trace2. Цели преобразуют содержимое этих сообщений в форматы, специфичные для конкретной цели, и записывают события в свои потоки данных. Таким образом, API Trace2 может выполнять множество различных типов анализа.

Цели определяются с помощью VTable, что позволяет в будущем легко расширять их до других форматов. Это можно использовать, например, для определения двоичного формата.

Trace2 управляется с помощью значений конфигурации trace2.* в системных и глобальных файлах конфигурации, а также переменных среды GIT_TRACE2* . Trace2 не считывает файлы конфигурации локального репозитория или рабочего дерева, а также не учитывает параметры конфигурации командной строки -c .

Trace2 targets

Trace2 определяет следующий набор целей Trace2. Подробности формата приведены в следующем разделе.

Цель обычного формата

Целевым нормальным форматом является традиционный формат printf() , аналогичный формату GIT_TRACE . Этот формат включается с помощью переменной среды GIT_TRACE2 или системного или глобального параметра конфигурации trace2.normalTarget .

For example

$ export GIT_TRACE2=~/log.normal
$ git version
git version 2.20.1.155.g426c96fcdb

or

$ git config --global trace2.normalTarget ~/log.normal
$ git version
git version 2.20.1.155.g426c96fcdb

yields

$ cat ~/log.normal
12:28:42.620009 common-main.c:38                  version 2.20.1.155.g426c96fcdb
12:28:42.620989 common-main.c:39                  start git version
12:28:42.621101 git.c:432                         cmd_name version (version)
12:28:42.621215 git.c:662                         exit elapsed:0.001227 code:0
12:28:42.621250 trace2/tr2_tgt_normal.c:124       atexit elapsed:0.001265 code:0

Цель формата производительности

Целевой формат производительности (PERF) — это формат на основе столбцов, который заменяет GIT_TRACE_PERFORMANCE и подходит для разработки и тестирования, возможно, в качестве дополнения к таким инструментам, как gprof . Этот формат включается с помощью переменной среды GIT_TRACE2_PERF или системного или глобального параметра конфигурации trace2.perfTarget .

For example

$ export GIT_TRACE2_PERF=~/log.perf
$ git version
git version 2.20.1.155.g426c96fcdb

or

$ git config --global trace2.perfTarget ~/log.perf
$ git version
git version 2.20.1.155.g426c96fcdb

yields

$ cat ~/log.perf
12:28:42.620675 common-main.c:38                  | d0 | main                     | version      |     |           |           |            | 2.20.1.155.g426c96fcdb
12:28:42.621001 common-main.c:39                  | d0 | main                     | start        |     |  0.001173 |           |            | git version
12:28:42.621111 git.c:432                         | d0 | main                     | cmd_name     |     |           |           |            | version (version)
12:28:42.621225 git.c:662                         | d0 | main                     | exit         |     |  0.001227 |           |            | code:0
12:28:42.621259 trace2/tr2_tgt_perf.c:211         | d0 | main                     | atexit       |     |  0.001265 |           |            | code:0

Целевой формат мероприятия

Целевой формат событий — это формат данных событий на основе JSON, подходящий для анализа телеметрии. Этот формат включается с помощью переменной среды GIT_TRACE2_EVENT , системного или глобального параметра конфигурации trace2.eventTarget .

For example

$ export GIT_TRACE2_EVENT=~/log.event
$ git version
git version 2.20.1.155.g426c96fcdb

or

$ git config --global trace2.eventTarget ~/log.event
$ git version
git version 2.20.1.155.g426c96fcdb

yields

$ cat ~/log.event
{"event":"version","sid":"20190408T191610.507018Z-H9b68c35f-P000059a8","thread":"main","time":"2019-01-16T17:28:42.620713Z","file":"common-main.c","line":38,"evt":"3","exe":"2.20.1.155.g426c96fcdb"}
{"event":"start","sid":"20190408T191610.507018Z-H9b68c35f-P000059a8","thread":"main","time":"2019-01-16T17:28:42.621027Z","file":"common-main.c","line":39,"t_abs":0.001173,"argv":["git","version"]}
{"event":"cmd_name","sid":"20190408T191610.507018Z-H9b68c35f-P000059a8","thread":"main","time":"2019-01-16T17:28:42.621122Z","file":"git.c","line":432,"name":"version","hierarchy":"version"}
{"event":"exit","sid":"20190408T191610.507018Z-H9b68c35f-P000059a8","thread":"main","time":"2019-01-16T17:28:42.621236Z","file":"git.c","line":662,"t_abs":0.001227,"code":0}
{"event":"atexit","sid":"20190408T191610.507018Z-H9b68c35f-P000059a8","thread":"main","time":"2019-01-16T17:28:42.621268Z","file":"trace2/tr2_tgt_event.c","line":163,"t_abs":0.001265,"code":0}

Включение цели

Чтобы включить цель, установите для соответствующей переменной среды или системного или глобального значения конфигурации одно из следующих значений:

  • 0 или false — отключает цель.

  • 1 или true — записывает в STDERR .

  • [2-9] — записывает в уже открытый дескриптор файла.

  • <absolute-pathname> — записывает в файл в режиме добавления. Если цель уже существует и является каталогом, трассировки будут записаны в файлы (по одному на каждый процесс) под данным каталогом.

  • af_unix:[<socket_type>:]<absolute-pathname> — запись в доменный сокет Unix (на платформах, которые их поддерживают). Тип сокета может быть stream или dgram ; если этот параметр опущен, Git попробует оба варианта.

Когда файлы трассировки записываются в целевой каталог, они будут названы в соответствии с последним компонентом SID (за которым необязательно следует счетчик, чтобы избежать конфликтов имен файлов).

Trace2 api

API Trace2 public определен и документирован в trace2.h ; обратитесь к нему для получения дополнительной информации. Все функции и макросы public имеют префикс trace2_ и реализованы в trace2.c .

Структуры данных public Trace2 отсутствуют.

Код Trace2 также определяет набор функций и типов данных private в каталоге trace2/ . Эти символы имеют префикс tr2_ и должны использоваться только функциями в trace2.c (или другими исходными файлами private в trace2/ ).

Соглашения о функциях и макросах Public

Некоторые функции имеют суффикс _fl() , указывающий, что они принимают аргументы file и line-number .

Некоторые функции имеют суффикс _va_fl() , указывающий на то, что они также принимают аргумент va_list .

Некоторые функции имеют суффикс _printf_fl() , указывающий на то, что они также принимают формат стиля printf() с переменным количеством аргументов.

Макросы-оболочки CPP определены так, чтобы скрыть большую часть этих деталей.

Целевые форматы Trace2

NORMAL Format

События записываются строками вида:

[<time> SP <filename>:<line> SP+] <event-name> [[SP] <event-message>] LF
<event-name>

это название события.

<event-message>

представляет собой сообщение printf() в свободной форме, предназначенное для потребления человеком.

Обратите внимание, что оно может содержать встроенные символы LF или CRLF, которые не экранированы, поэтому событие может распространяться на несколько строк.

Если GIT_TRACE2_BRIEF или trace2.normalBrief имеет значение true, поля time , filename и line опускаются.

Эта цель должна быть скорее сводной (например, GIT_TRACE) и менее подробной, чем другие цели. Например, он игнорирует поток, регион и сообщения с данными.

PERF Format

События записываются строками вида:

[<time> SP <filename>:<line> SP+
    BAR SP] d<depth> SP
    BAR SP <thread-name> SP+
    BAR SP <event-name> SP+
    BAR SP [r<repo-id>] SP+
    BAR SP [<t_abs>] SP+
    BAR SP [<t_rel>] SP+
    BAR SP [<category>] SP+
    BAR SP DOTS* <perf-event-message>
    LF
<depth>

— это глубина процесса git . Это количество родительских процессов git . Команда git верхнего уровня имеет значение глубины "d0" . Дочерний элемент имеет значение глубины "d1" . Дочерний элемент второго уровня имеет значение глубины "d2" и так далее.

<thread-name>

— уникальное имя потока. Основной поток называется "main" . Другие имена потоков имеют вид "th%d:%s" и включают уникальный номер и имя процесса потока.

<event-name>

это название события.

<repo-id>

если он присутствует, это число, указывающее используемый репозиторий. Событие def_repo генерируется при открытии репозитория. Это определяет идентификатор репо и связанное с ним рабочее дерево. Последующие события, специфичные для репо, будут ссылаться на этот идентификатор репо.

В настоящее время для основного репозитория это всегда "r1" . Это поле находится в ожидании подмодулей внутри процесса в будущем.

<t_abs>

если он присутствует, это абсолютное время в секундах с момента запуска программы.

<t_rel>

если он присутствует, это время в секундах относительно начала текущего региона. Для события выхода из потока это время, прошедшее в потоке.

<category>

присутствует в событиях региона и данных и используется для обозначения широкой категории, например "index" или "status" .

<perf-event-message>

представляет собой сообщение printf() в свободной форме, предназначенное для потребления человеком.

15:33:33.532712 wt-status.c:2310                  | d0 | main                     | region_enter | r1  |  0.126064 |           | status     | label:print
15:33:33.532712 wt-status.c:2331                  | d0 | main                     | region_leave | r1  |  0.127568 |  0.001504 | status     | label:print

Если GIT_TRACE2_PERF_BRIEF или trace2.perfBrief имеет значение true, поля time , file и line опускаются.

d0 | main                     | region_leave | r1  |  0.011717 |  0.009122 | index      | label:preload

Цель PERF предназначена для интерактивного анализа производительности во время разработки и довольно шумна.

EVENT Format

Каждое событие представляет собой объект JSON, содержащий несколько пар ключ/значение, записанных в виде одной строки, за которой следует LF.

'{' <key> ':' <value> [',' <key> ':' <value>]* '}' LF

Некоторые пары ключ/значение являются общими для всех событий, а некоторые зависят от события.

Общие пары ключ/значение

Следующие пары ключ/значение являются общими для всех событий:

{
        "event":"version",
        "sid":"20190408T191827.272759Z-H9b68c35f-P00003510",
        "thread":"main",
        "time":"2019-04-08T19:18:27.282761Z",
        "file":"common-main.c",
        "line":42,
        ...
}
"event":<event>

это название события.

"sid":<sid>

это идентификатор сеанса. Это уникальная строка, идентифицирующая экземпляр процесса, позволяющая идентифицировать все события, исходящие от процесса. Идентификатор сеанса используется вместо PID, поскольку PID перерабатываются операционной системой. Для дочерних процессов git к идентификатору сеанса добавляется идентификатор сеанса родительского процесса git , чтобы можно было идентифицировать отношения родитель-потомок во время постобработки.

"thread":<thread>

это имя потока.

"time":<time>

— время события по всемирному координированному времени.

"file":<filename>

— исходный файл, генерирующий событие.

"line":<line-number>

— это целочисленный номер строки источника, генерирующей событие.

"repo":<repo-id>

если он присутствует, это целочисленный идентификатор репо, как описано ранее.

Если GIT_TRACE2_EVENT_BRIEF или trace2.eventBrief имеет значение true, поля file и line исключаются из всех событий, а поле time присутствует только в событиях "start" и "atexit" .

Пары ключ/значение, специфичные для события

"version"

Это событие предоставляет версию исполняемого файла и формат EVENT. Это всегда должно быть первое событие в сеансе трассировки. Версия формата EVENT будет увеличиваться, если добавляются новые типы событий, удаляются существующие поля или происходят существенные изменения в интерпретации существующих событий или полей. Небольшие изменения, такие как добавление нового поля к существующему событию, не потребуют увеличения версии формата EVENT.

{
        "event":"version",
        ...
        "evt":"3",                       # версия формата EVENT
        "exe":"2.20.1.155.g426c96fcdb" # git-версия
}
"too_many_files"

Это событие записывается в дозорный файл git-trace2-discard, если в целевом каталоге трассировки слишком много файлов (см. параметр конфигурации трассировки2.maxFiles).

{
        "event":"too_many_files",
        ...
}
"start"

Это событие содержит полный argv, полученный функцией main().

{
        "event":"start",
        ...
        "t_abs":0.001227, # elapsed time in seconds
        "argv":["git","version"]
}
"exit"

Это событие генерируется, когда git вызывает exit() .

{
        "event":"exit",
        ...
        "t_abs":0.001227, # elapsed time in seconds
        "code":0          # exit code
}
"atexit"

Это событие генерируется процедурой Trace2 atexit во время окончательного завершения работы. Это должно быть последнее событие, созданное процессом.

(Затраченное здесь время больше, чем время, указанное в событии "exit" , поскольку оно запускается после завершения всех других задач atexit.)

{
        "event":"atexit",
        ...
        "t_abs":0.001227, # elapsed time in seconds
        "code":0          # код выхода
}
"signal"

Это событие генерируется, когда программа завершается по сигналу пользователя. В зависимости от платформы событие сигнала может препятствовать созданию события "atexit" .

{
        "event":"signal",
        ...
        "t_abs":0.001227,  # прошедшее время в секундах
        "signo":13         # SIGTERM, SIGINT и т. д.
}
"error"

Это событие генерируется при вызове одной из функций BUG() , bug() , error() , die() , warning() или usage() .

{
        "event":"error",
        ...
        "msg":"invalid option: --cahced", # formatted error message
        "fmt":"invalid option: %s"          # error format string
}

Событие ошибки может генерироваться более одного раза. Строка формата позволяет постпроцессорам группировать ошибки по типу, не беспокоясь о конкретных аргументах ошибки.

"cmd_path"

Это событие содержит обнаруженный полный путь к исполняемому файлу git (на платформах, которые настроены для его разрешения).

{
        "event":"cmd_path",
        ...
        "path":"C:/work/gfw/git.exe"
}
"cmd_ancestry"

Это событие содержит имя текстовой команды для родителя (и предыдущих поколений родителей) текущего процесса в массиве, упорядоченном от ближайшего родителя к самому дальнему прародителю. Это может быть реализовано не на всех платформах.

{
        "event":"cmd_ancestry",
        ...
        "ancestry":["bash","tmux: server","systemd"]
}
"cmd_name"

Это событие содержит имя команды для этого процесса git и иерархию команд родительских процессов git .

{
        "event":"cmd_name",
        ...
        "name":"pack-objects",
        "hierarchy":"push/pack-objects"
}

Обычно поле "name" содержит каноническое имя команды. Если каноническое имя недоступно, используется одно из следующих специальных значений:

"_query_"            # "git --html-path"
"_run_dashed_"       # when "git foo" tries to run "git-foo"
"_run_shell_alias_"  # alias expansion to a shell command
"_run_git_alias_"    # alias expansion to a git command
"_usage_"            # usage error
"cmd_mode"

Это событие, если оно присутствует, описывает вариант команды. Это событие может быть создано более одного раза.

{
        "event":"cmd_mode",
        ...
        "name":"branch"
}

Поле "name" представляет собой произвольную строку, описывающую командный режим. Например, checkout может извлечь ветку или отдельный файл. И эти вариации обычно имеют разные, несопоставимые ТТХ.

"alias"

Это событие присутствует при расширении псевдонима.

{
        "event":"alias",
        ...
        "alias":"l",                 # зарегистрированный псевдоним
        "argv":["log","--graph"] # расширение псевдонима
}
"child_start"

Это событие описывает дочерний процесс, который вот-вот будет запущен.

{
        "event":"child_start",
        ...
        "child_id":2,
        "child_class":"?",
        "use_shell":false,
        "argv":["git","rev-list","--objects","--stdin","--not","--all","--quiet"]

        "hook_name":"<hook_name>"  # present when child_class is "hook"
        "cd":"<path>"                   # present when cd is required
}

Поле "child_id" можно использовать для сопоставления этого child_start с соответствующим событием child_exit.

Поле "child_class" представляет собой приблизительную классификацию, например "editor" , "pager" , "transport/*" и "hook" . Неклассифицированные дети классифицируются как "?" .

"child_exit"

Это событие генерируется после того, как текущий процесс вернулся из waitpid() и получил информацию о выходе от дочернего процесса.

{
        "event":"child_exit",
        ...
        "child_id":2,
        "pid":14708,         # child PID
        "code":0,         # дочерний код выхода
        "t_rel":0.110605 # observed run-time of child process
}

Обратите внимание, что идентификатор сеанса дочернего процесса недоступен для текущего/порожденного процесса, поэтому PID дочернего процесса указывается здесь как подсказка для постобработки. (Но это всего лишь подсказка, поскольку дочерний процесс может быть сценарием оболочки, у которого нет идентификатора сеанса.)

Обратите внимание, что поле t_rel содержит наблюдаемое время выполнения дочернего процесса в секундах (начиная до разветвления/выполнения/порождения и заканчивая после waitpid() , а также включает служебные данные создания процесса ОС). Таким образом, это время будет немного больше, чем время выхода, сообщаемое самим дочерним процессом.

"child_ready"

Это событие генерируется после того, как текущий процесс запустил фоновый процесс и освободил все его дескрипторы.

{
        "event":"child_ready",
        ...
        "child_id":2,
        "pid":14708,         # дочерний PID
        "ready":"ready", # состояние готовности к дочернему процессу
        "t_rel":0.110605 # наблюдаемое время выполнения дочернего процесса
}

Обратите внимание, что идентификатор сеанса дочернего процесса недоступен для текущего/порожденного процесса, поэтому PID дочернего процесса указывается здесь как подсказка для постобработки. (Но это всего лишь подсказка, поскольку дочерний процесс может быть сценарием оболочки, у которого нет идентификатора сеанса.)

Это событие генерируется после того, как дочерний элемент запускается в фоновом режиме и ему дается немного времени для загрузки и начала работы. Если дочерний процесс запускается нормально, пока родительский все еще ожидает, поле "ready" будет иметь значение "ready" . Если дочерний процесс запускается слишком медленно, а у родительского истекло время ожидания, поле будет иметь значение "timeout" . Если дочерний процесс запускается, но родительский элемент не может его проверить, поле будет иметь значение "error" .

После того как родительский процесс выдаст это событие, он передаст все свои дескрипторы дочернему процессу и будет рассматривать дочерний процесс как фоновый демон. Таким образом, даже если дочерний процесс в конечном итоге завершит загрузку, родительский элемент не отправит обновленное событие.

Обратите внимание, что поле t_rel содержит наблюдаемое время выполнения в секундах, когда родительский процесс перевел дочерний процесс в фоновый режим. Предполагается, что дочерний процесс является длительно работающим демоническим процессом и может пережить родительский процесс. Таким образом, время дочернего события родительского элемента не следует сравнивать со временем выхода дочернего элемента.

"exec"

Это событие генерируется до того, как git попытается выполнить exec() другую команду вместо запуска дочернего процесса.

{
        "event":"exec",
        ...
        "exec_id":0,
        "exe":"git",
        "argv":["foo", "bar"]
}

Поле "exec_id" представляет собой уникальный идентификатор команды и полезно только в случае сбоя exec() и создания соответствующего события exec_result.

"exec_result"

Это событие генерируется, если exec() выходит из строя и управление возвращается к текущей команде git .

{
        "event":"exec_result",
        ...
        "exec_id":0,
        "code":1      # error code (errno) from exec()
}
"thread_start"

Это событие генерируется при запуске потока. Он генерируется внутри процесса потока нового потока (поскольку ему необходим доступ к данным в локальном хранилище потока).

{
        "event":"thread_start",
        ...
        "thread":"th02:preload_thread" # thread name
}
"thread_exit"

Это событие генерируется при выходе из потока. Он генерируется внутри потока-proc.

{
        "event":"thread_exit",
        ...
        "thread":"th02:preload_thread", # имя потока
        "t_rel":0.007328                # прошедшее время потока
}
"def_param"

Это событие генерируется для регистрации глобального параметра, такого как параметр конфигурации, флаг командной строки или переменная среды.

{
        "event":"def_param",
        ...
        "scope":"global",
        "param":"core.abbrev",
        "value":"7"
}
"def_repo"

Это событие определяет идентификатор репо и связывает его с корнем рабочего дерева.

{
        "event":"def_repo",
        ...
        "repo":1,
        "worktree":"/Users/jeffhost/work/gfw"
}

Как говорилось ранее, идентификатор репо в настоящее время всегда равен 1, поэтому будет только одно событие def_repo. Позже, если поддерживаются подмодули внутри процесса, событие def_repo должно создаваться для каждого посещенного подмодуля.

"region_enter"

Это событие генерируется при входе в регион.

{
        "event":"region_enter",
        ...
        "repo":1,                # optional
        "nesting":1,             # глубина стека текущего региона
        "category":"index",      # optional
        "label":"do_read_index", # optional
        "msg":".git/index"       # optional
}

Поле category может использоваться в будущих улучшениях для фильтрации по категориям.

GIT_TRACE2_EVENT_NESTING или trace2.eventNesting можно использовать для фильтрации глубоко вложенных регионов и событий данных. По умолчанию используется "2" .

"region_leave"

Это событие генерируется при выходе из региона.

{
        "event":"region_leave",
        ...
        "repo":1,                # optional
        "t_rel":0.002876,        # time spent in region in seconds
        "nesting":1,             # region stack depth
        "category":"index",      # optional
        "label":"do_read_index", # optional
        "msg":".git/index"       # optional
}
"data"

Это событие генерируется для регистрации пары ключ/значение локального потока и региона.

{
        "event":"data",
        ...
        "repo":1,              # optional
        "t_abs":0.024107,      # absolute elapsed time
        "t_rel":0.001031,      # elapsed time in region/thread
        "nesting":2,           # region stack depth
        "category":"index",
        "key":"read/cache_nr",
        "value":"3552"
}

Поле "value" может быть целым числом или строкой.

"data-json"

Это событие генерируется для регистрации предварительно отформатированной строки JSON, содержащей структурированные данные.

{
        "event":"data_json",
        ...
        "repo":1,              # optional
        "t_abs":0.015905,
        "t_rel":0.015905,
        "nesting":1,
        "category":"process",
        "key":"windows/ancestry",
        "value":["bash.exe","bash.exe"]
}
"th_timer"

Это событие регистрирует время, в течение которого в потоке работал таймер секундомера. Это событие генерируется, когда поток завершает работу таймеров, запрашивающих события для каждого потока.

{
        "event":"th_timer",
        ...
        "category":"my_category",
        "name":"my_timer",
        "intervals":5,         # number of time it was started/stopped
        "t_total":0.052741,    # total time in seconds it was running
        "t_min":0.010061,      # shortest interval
        "t_max":0.011648       # longest interval
}
"timer"

Это событие регистрирует время, в течение которого работал таймер секундомера, агрегированное по всем потокам. Это событие генерируется при выходе из процесса.

{
        "event":"timer",
        ...
        "category":"my_category",
        "name":"my_timer",
        "intervals":5,         # количество раз, когда он был запущен/остановлен
        "t_total":0.052741,    # общее время работы в секундах
        "t_min":0.010061,      # кратчайший интервал
        "t_max":0.011648       # самый длинный интервал
}
"th_counter"

Это событие регистрирует значение переменной-счетчика в потоке. Это событие генерируется, когда поток завершает работу счетчиков, запрашивающих события для каждого потока.

{
        "event":"th_counter",
        ...
        "category":"my_category",
        "name":"my_counter",
        "count":23
}
"counter"

Это событие регистрирует значение переменной счетчика во всех потоках. Это событие генерируется при выходе из процесса. Общее значение, указанное здесь, представляет собой сумму по всем потокам.

{
        "event":"counter",
        ...
        "category":"my_category",
        "name":"my_counter",
        "count":23
}

Пример использования API трассировки2

Вот гипотетическое использование API Trace2, показывающее предполагаемое использование (не беспокоясь о фактических деталях Git ).

Initialization

Инициализация происходит в main() . За кулисами регистрируются обработчики atexit и signal .

int main(int argc, const char **argv)

        int exit_code;

        trace2_initialize();
        trace2_cmd_start(argv);

        exit_code = cmd_main(argc, argv);

        trace2_cmd_exit(exit_code);

        return exit_code;
}

        trace2_cmd_mode("branch");

static void *preload_thread(void *_data)
{
        // запускаем часы для каждого потока и выдаем сообщение.
        trace2_thread_start("preload_thread");

        // сообщаем, какой кусок массива был назначен этому потоку.
        trace2_data_intmax("index", the_repository, "offset", p->offset);
        trace2_data_intmax("index", the_repository, "count", nr);

        do {
            ...
        } while (--nr > 0);
        ...

        // сообщаем о времени, затраченном этим потоком.
        trace2_thread_exit();
        return NULL;
}

void preload_index(struct index_state *index,
        const struct pathspec *pathspec,
        unsigned int refresh_flags)
{
        trace2_region_enter("index", "preload", the_repository);

        for (i = 0; i < threads; i++) {
            ... /* создать тему */
        }

        for (i = 0; i < threads; i++) {
            ... /* присоединяемся к теме */
        }

        trace2_region_leave("index", "preload", the_repository);
}

В этом примере preload_index() был выполнен потоком main и запустил регион preload . Было запущено семь потоков с именами от th01:preload_thread до th07:preload_thread . Конфигурация (параметр определения) События

Сохраните значения конфигурации "interesting" в журнал трассировки2.

При желании мы можем генерировать события конфигурации, см. trace2.configparams в git-config[1] , чтобы узнать, как это включить.

$ git config --system color.ui никогда
$ git config --global color.ui всегда
$ git config --local color.ui авто
$ git config --list --show-scope | grep 'color.ui'
system  color.ui=never
global  color.ui=always
local   color.ui=auto

Затем отметьте конфигурацию color.ui как конфигурацию "interesting" с помощью GIT_TRACE2_CONFIG_PARAMS :

$ export GIT_TRACE2_PERF_BRIEF=1
$ export GIT_TRACE2_PERF=~/log.perf
$ export GIT_TRACE2_CONFIG_PARAMS=color.ui
$ git version
...
$ cat ~/log.perf

События таймера секундомера

Измерьте время, затрачиваемое на вызов функции или фрагмента кода, который может быть вызван из многих мест кода на протяжении всего процесса.

static void expensive_function(void)
{
        trace2_timer_start(TRACE2_TIMER_ID_TEST1);
        ...
        sleep_millisec(1000); // Делаем что-то дорогое
        ...
        trace2_timer_stop(TRACE2_TIMER_ID_TEST1);
}

static int ut_100timer(int argc, const char **argv)
{
        ...

        expensive_function();

        // Делаем что-нибудь еще 1...

        expensive_function();

        // Делаем что-нибудь еще 2...

        expensive_function();

        return 0;
}

В этом примере мы измеряем общее время, затраченное на expensive_function() , независимо от того, когда он вызывается в общем потоке программы.

$ export GIT_TRACE2_PERF_BRIEF=1
$ export GIT_TRACE2_PERF=~/log.perf
$ t/helper/test-tool trace2 100timer 3 1000
...
$ cat ~/log.perf
d0 | main                     | version      |     |           |           |              | ...

d0 | main                     | cmd_name     |     |           |           |              | trace2 (trace2)
d0 | main                     | exit         |     |  3.003667 |           |              | code:0
d0 | main                     | timer        |     |           |           | test         | name:test1 intervals:3 total:3.001686 min:1.000254 max:1.000929
d0 | main                     | atexit       |     |  3.003796 |           |              | code:0
