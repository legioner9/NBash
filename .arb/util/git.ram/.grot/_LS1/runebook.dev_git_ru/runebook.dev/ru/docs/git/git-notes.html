git-notes

Name

git-notes — Добавить или проверить заметки к объекту

Synopsis

git notes [list [<object>]]
git notes add [-f] [--allow-empty] [--[no-]separator | --separator=<paragraph-break>] [--[no-]stripspace] [-F <file> | -m <msg> | (-c | -C) <object>] [<object>]
git notes copy [-f] ( --stdin | <from-object> [<to-object>] )
git notes append [--allow-empty] [--[no-]separator | --separator=<paragraph-break>] [--[no-]stripspace] [-F <file> | -m <msg> | (-c | -C) <object>] [<object>]
git notes edit [--allow-empty] [<object>] [--[no-]stripspace]
git notes show [<object>]
git notes merge [-v | -q] [-s <strategy> ] <notes-ref>
git notes merge --commit [-v | -q]
git notes merge --abort [-v | -q]
git notes remove [--ignore-missing] [--stdin] [<object>…​]
git notes prune [-n] [-v]
git notes get-ref

Description

Добавляет, удаляет или читает примечания, прикрепленные к объектам, не касаясь самих объектов.

По умолчанию заметки сохраняются и считываются из refs/notes/commits , но это значение по умолчанию можно переопределить. См. разделы ОПЦИИ, КОНФИГУРАЦИЯ и СРЕДА ниже. Если эта ссылка не существует, она будет автоматически создана, когда впервые понадобится сохранить заметку.

Типичное использование примечаний — дополнение сообщения о коммите без изменения самого коммита. Примечания могут отображаться с помощью git log вместе с исходным сообщением о фиксации. Чтобы отличить эти примечания от сообщения, хранящегося в объекте фиксации, примечания имеют отступ, как и сообщение, после строки без отступа с надписью "Notes (<refname>):" (или "Notes:" для refs/notes/commits ).

Примечания также можно добавлять к патчам, подготовленным с помощью git format-patch , используя опцию --notes . Такие примечания добавляются в виде комментария к патчу после разделительной линии из трех тире.

Чтобы изменить примечания, отображаемые git log , см. обсуждение "notes.displayRef" в CONFIGURATION .

См. конфигурацию "notes.rewrite.<command>" , чтобы узнать о способе переноса примечаний между командами, которые перезаписывают фиксации.

Subcommands

list

Перечислите объект примечаний для данного объекта. Если объект не указан, покажите список всех объектов примечаний и объектов, которые они аннотируют (в формате "<note object> <annotated object>" ). Это подкоманда по умолчанию, если подкоманда не указана.

add

Добавить примечания к данному объекту (по умолчанию HEAD). Прервите, если у объекта уже есть примечания (используйте -f , чтобы перезаписать существующие примечания). Однако если вы используете add в интерактивном режиме (используя редактор для предоставления содержимого заметок), то вместо прерывания существующие заметки будут открыты в редакторе (например, подкоманда edit ). Если вы укажете несколько -m и -F , между сообщениями будет вставлена ​​пустая строка. Используйте опцию --separator для вставки других разделителей.

copy

Скопируйте примечания для первого объекта на второй объект (по умолчанию HEAD). Прервите, если у второго объекта уже есть примечания или если у первого объекта есть none (используйте -f, чтобы перезаписать существующие примечания на второй объект). Эта подкоманда эквивалентна: git notes add [-f] -C $(git notes list <from-object>) <to-object> .

В режиме --stdin брать строки в формате

<from-object> SP <to-object> [ SP <rest> ] LF

на стандартный ввод и скопируйте примечания из каждого <from-object> в соответствующий ему <to-object>. (Необязательный <rest> игнорируется, чтобы команда могла прочитать входные данные, переданные перехватчику post-rewrite .)

append

Добавьте новые сообщения, заданные параметрами -m или -F , к существующей заметке или добавьте их в качестве новой заметки, если таковая не существует, для объекта (по умолчанию HEAD). При добавлении к существующей заметке перед каждым новым сообщением добавляется пустая строка в качестве разделителя между абзацами. Сепаратор можно настроить с помощью опции --separator .

edit

Редактировать заметки для данного объекта (по умолчанию HEAD).

show

Показать примечания к данному объекту (по умолчанию HEAD).

merge

Объединить данную ссылку на заметку с текущей ссылкой на заметку. Будет предпринята попытка объединить изменения, внесенные с помощью данной ссылки на заметку (называемой "remote" ), начиная с базы слияния (если таковая имеется), с текущей ссылкой на заметку (называемой "local" ).

Если возникают конфликты и не указана стратегия автоматического разрешения конфликтующих нот (см. раздел "NOTES MERGE STRATEGIES" ), используется преобразователь "manual" . Этот преобразователь проверяет конфликтующие заметки в специальном рабочем дереве ( .git/NOTES_MERGE_WORKTREE ) и дает пользователю указание вручную разрешить конфликты там. По завершении пользователь может либо завершить слияние с помощью git notes merge --commit , либо прервать слияние с помощью git notes merge --abort .

remove

Удалить примечания к заданным объектам (по умолчанию HEAD). При указании нуля или одного объекта из командной строки это эквивалентно указанию пустого сообщения-заметки для подкоманды edit .

prune

Удалите все примечания к несуществующим/недоступным объектам.

get-ref

Распечатайте текущие заметки по ссылке. Это обеспечивает простой способ получения ссылки на текущую заметку (например, из сценариев).

Options

-f
--force

При добавлении примечаний к объекту, у которого уже есть примечания, перезапишите существующие примечания (вместо прерывания).

-m <msg>
--message=<msg>

Используйте данное примечание (вместо подсказки). Если указано несколько параметров -m , их значения объединяются в отдельные абзацы. Строки, начинающиеся с # , и пустые строки, кроме одной строки между абзацами, будут удалены. Если вы хотите сохранить их дословно, используйте --no-stripspace .

-F <file>
--file=<file>

Возьмите заметку-сообщение из данного файла. Используйте - , чтобы прочитать сообщение-заметку со стандартного ввода. Строки, начинающиеся с # , и пустые строки, кроме одной строки между абзацами, будут удалены. Если вы хотите сохранить их дословно, используйте --no-stripspace .

-C <object>
--reuse-message=<object>

Возьмите данный объект blob (например, другую заметку) в качестве сообщения заметки. (Вместо этого используйте git notes copy <object> для копирования заметок между объектами.). По умолчанию сообщение будет скопировано дословно, но если вы хотите удалить строки, начинающиеся с # , и пустые строки, кроме одной строки между абзацами, используйте опцию with`--stripspace`.

-c <object>
--reedit-message=<object>

Подобно -C , но с -c вызывается редактор, чтобы пользователь мог дополнительно редактировать сообщение примечания.

--allow-empty

Разрешить сохранение пустого объекта заметки. Поведение по умолчанию — автоматическое удаление пустых заметок.

--[no-]separator, --separator=<paragraph-break>

Укажите строку, используемую в качестве пользовательского разделителя между абзацами (при необходимости в конце добавляется новая строка). Если --no-separator , разделители между абзацами добавляться не будут. По умолчанию пустая строка.

--[no-]stripspace

Удалите начальные и конечные пробелы из сообщения примечания. Также удалите пустые строки, кроме одной строки между абзацами. Строки, начинающиеся с # , будут удалены в случаях без редактора, таких как -m , -F и -C , но не в случае редактора, например git notes edit , -c и т. д.

--ref <ref>

Управляйте деревом заметок в <ref>. Это переопределяет конфигурацию GIT_NOTES_REF и "core.notesRef" . Ссылка указывает полное имя ссылки, если оно начинается с refs/notes/ ; когда он начинается с notes/ , refs/ и в противном случае refs/notes/ имеет префикс, образующий полное имя ссылки.

--ignore-missing

Не считайте ошибкой требование удалить примечания из объекта, к которому не прикреплены примечания.

--stdin

Также прочитайте имена объектов, чтобы удалить примечания из стандартного ввода (нет причин, по которым вы не можете объединить это с именами объектов из командной строки).

-n
--dry-run

Ничего не удаляйте; просто сообщите имена объектов, примечания к которым будут удалены.

-s <strategy>
--strategy=<strategy>

При объединении заметок разрешайте конфликты заметок, используя данную стратегию. Распознаются следующие стратегии: "manual" (по умолчанию), "ours" , "theirs" , "union" и "cat_sort_uniq" . Эта опция переопределяет настройку конфигурации "notes.mergeStrategy" . См. раздел "NOTES MERGE STRATEGIES" ниже для получения дополнительной информации о каждой стратегии объединения заметок.

--commit

Завершить незавершенный git notes merge . Используйте эту опцию, если вы разрешили конфликты, которые git notes merge сохранил в .git/NOTES_MERGE_WORKTREE. Это изменяет частичную фиксацию слияния, созданную git notes merge (хранящуюся в .git/NOTES_MERGE_PARTIAL), путем добавления примечаний в .git/NOTES_MERGE_WORKTREE. Ссылка на заметки, хранящаяся в символической ссылке .git/NOTES_MERGE_REF, обновляется в соответствии с полученным коммитом.

--abort

Прервите/сбросьте текущий git notes merge , т.е. примечания объединяются с конфликтами. Это просто удалит все файлы, связанные с объединением заметок.

-q
--quiet

При объединении заметок действуйте тихо.

-v
--verbose

При объединении заметок будьте более подробными. При сокращении примечаний сообщайте обо всех именах объектов, примечания к которым удалены.

Discussion

Заметки о коммите — это большие двоичные объекты, содержащие дополнительную информацию об объекте (обычно информацию, дополняющую сообщение о коммите). Эти капли взяты из ссылок на заметки. Ссылка на примечания обычно представляет собой ветку, содержащую "files" , чьи пути представляют собой имена объектов, которые они описывают, с некоторыми разделителями каталогов, включенными из соображений производительности [ 1 ].

Каждое изменение заметок создает новую фиксацию по указанной ссылке заметки. Таким образом, вы можете просмотреть историю заметок, вызвав, например, git log -p notes/commits . В настоящее время в сообщении о фиксации указывается только, какая операция вызвала обновление, а авторство фиксации определяется в соответствии с обычными правилами (см. git-commit[1] ). Эти детали могут измениться в будущем.

Также разрешено, чтобы ссылка на примечания указывала непосредственно на объект дерева, и в этом случае историю примечаний можно прочитать с помощью git log -p -g <refname> .

Стратегии объединения заметок

Стратегией объединения примечаний по умолчанию является "manual" , которая проверяет конфликтующие примечания в специальном рабочем дереве для разрешения конфликтов примечаний ( .git/NOTES_MERGE_WORKTREE ) и дает пользователю указание разрешить конфликты в этом рабочем дереве. По завершении пользователь может либо завершить слияние с помощью git notes merge --commit , либо прервать слияние с помощью git notes merge --abort .

Пользователи могут выбрать стратегию автоматического слияния из числа следующих, используя опцию -s/--strategy или соответствующим образом настроив Notes.mergeStrategy:

"ours" автоматически разрешает конфликтующие примечания в пользу локальной версии (т. е. текущей ссылки примечаний).

"theirs" автоматически разрешает конфликты примечаний в пользу удаленной версии (т. е. данная ссылка примечаний объединяется с текущей ссылкой примечаний).

"union" автоматически разрешает конфликты заметок путем объединения локальной и удаленной версий.

"cat_sort_uniq" аналогичен "union" , но помимо объединения локальной и удаленной версий эта стратегия также сортирует полученные строки и удаляет из них повторяющиеся строки. Это эквивалентно применению конвейера оболочки "cat | sort | uniq" к локальной и удаленной версиям. Эта стратегия полезна, если примечания имеют построчный формат и требуется избежать дублирования строк в результате слияния. Обратите внимание: если локальная или удаленная версия содержит повторяющиеся строки до слияния, они также будут удалены с помощью этой стратегии слияния заметок.

Examples

Вы можете использовать заметки для добавления аннотаций с информацией, которая не была доступна на момент написания коммита.

$ git notes add -m 'Tested-by: Johannes Sixt <[email protected]>' 72a144e2
$ git show -s 72a144e
[...]
    Signed-off-by: Junio C Hamano <gitster@pobox.com>

Notes:
    Tested-by: Johannes Sixt <j6t@kdbg.org>

В принципе, заметка представляет собой обычный объект Git , и принимается любой (не)формат. Вы можете безопасно создавать заметки из произвольных файлов в двоичном формате, используя git hash-object :

$ cc *.c
$ blob=$(git hash-object -w a.out)
$ git notes --ref=built add --allow-empty -C "$blob" HEAD

(Вы не можете просто использовать git notes --ref=built add -F a.out HEAD , поскольку он не является бинарно-безопасным.) Конечно, нет особого смысла отображать заметки нетекстового формата с помощью git log , поэтому, если вы используете такие заметки, вам, вероятно, придется написать некоторые специальные инструменты, чтобы сделать с ними что-нибудь полезное.

Configuration

core.notesRef

Примечания относятся к чтению и манипулированию вместо refs/notes/commits . Должно быть несокращенное имя ссылки. Этот параметр можно переопределить через среду и командную строку.

Все, что находится выше этой строки в этом разделе, не включено в документацию git-config[1] . Следующее содержимое такое же, как и найденное там:

notes.mergeStrategy

Какую стратегию слияния выбрать по умолчанию при разрешении конфликтов заметок. Должен быть один из manual , ours , theirs , union или cat_sort_uniq . По умолчанию manual . Дополнительную информацию о каждой стратегии см. в разделе "NOTES MERGE STRATEGIES" статьи git-notes[1] .

Этот параметр можно переопределить, передав параметр --strategy в git-notes[1] .

notes.<name>.mergeStrategy

Какую стратегию слияния выбрать при объединении заметок в refs/notes/<name>. Это отменяет более общий "notes.mergeStrategy" . Дополнительную информацию о доступных стратегиях см. в разделе "NOTES MERGE STRATEGIES" в git-notes[1] .

notes.displayRef

Какую ссылку (или ссылки, если она является общей или указана более одного раза), в дополнение к значению по умолчанию, установленному core.notesRef или GIT_NOTES_REF , для чтения примечаний при отображении сообщений о фиксации с помощью команд семейства git log .

Этот параметр можно переопределить с помощью переменной среды GIT_NOTES_DISPLAY_REF , которая должна представлять собой список ссылок или glob, разделенных двоеточиями.

Предупреждение будет выдано для ссылок, которые не существуют, но glob, который не соответствует ни одной ссылке, игнорируется.

Этот параметр можно отключить с помощью опции --no-notes семейства команд git log или опции --notes=<ref> , принимаемой этими командами.

Действующее значение "core.notesRef" (возможно, переопределяемое GIT_NOTES_REF) также неявно добавляется в список отображаемых ссылок.

notes.rewrite.<command>

При перезаписи фиксации с помощью <command> (в настоящее время amend или rebase ), если эта переменная равна false , git не будет копировать примечания из оригинала в переписанную фиксацию. По умолчанию true . См. также " notes.rewriteRef " ниже.

Этот параметр можно переопределить с помощью переменной среды GIT_NOTES_REWRITE_REF , которая должна представлять собой список ссылок или glob, разделенных двоеточиями.

notes.rewriteMode

При копировании примечаний во время перезаписи (см. параметр "notes.rewrite.<command>" ) определяет, что делать, если в целевом коммите уже есть примечание. Должен быть один из overwrite , concatenate , cat_sort_uniq или ignore . По умолчанию concatenate .

Этот параметр можно переопределить с помощью переменной среды GIT_NOTES_REWRITE_MODE .

notes.rewriteRef

При копировании заметок во время перезаписи указывает (полную) ссылку, чьи заметки следует скопировать. Может быть glob, и в этом случае будут скопированы примечания во всех соответствующих ссылках. Вы также можете указать эту конфигурацию несколько раз.

Не имеет значения по умолчанию; вы должны настроить эту переменную, чтобы включить перезапись заметок. Установите значение refs/notes/commits , чтобы включить перезапись примечаний к фиксации по умолчанию.

Может быть переопределено с помощью переменной среды GIT_NOTES_REWRITE_REF . См. notes.rewrite.<command> выше для дальнейшего описания его формата.

Environment

GIT_NOTES_REF

Какую ссылку использовать для управления заметками вместо refs/notes/commits . Это отменяет настройку core.notesRef .

GIT_NOTES_DISPLAY_REF

Список ссылок или glob, разделенных двоеточиями, указывающий, какие ссылки, помимо ссылок по умолчанию из core.notesRef или GIT_NOTES_REF , читать примечания при отображении сообщений о фиксации. Это отменяет настройку notes.displayRef .

Предупреждение будет выдано для ссылок, которые не существуют, но glob, который не соответствует ни одной ссылке, игнорируется.

GIT_NOTES_REWRITE_MODE

Что делать при копировании заметок во время перезаписи, если в целевом коммите уже есть заметка. Должен быть один из overwrite , concatenate , cat_sort_uniq или ignore . Это отменяет настройку core.rewriteMode .

GIT_NOTES_REWRITE_REF

При переписывании коммитов какие примечания копировать из оригинала в переписанный коммит. Должен представлять собой список ссылок или glob, разделенных двоеточиями.

Если этот параметр не установлен в среде, список копируемых заметок зависит от настроек notes.rewrite.<command> и notes.rewriteRef .

1 . Разрешенные имена путей имеют вид bf / fe / 30 / … / 680d5a…​: последовательность имен каталогов, состоящая из двух шестнадцатеричных цифр, за каждой из которых следует имя файла с остальной частью идентификатора объекта.
