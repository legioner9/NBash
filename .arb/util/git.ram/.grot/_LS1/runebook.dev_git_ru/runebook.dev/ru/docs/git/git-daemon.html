git-daemon

Name

git-daemon — действительно простой сервер для репозиториев Git .

Synopsis

git daemon [--verbose] [--syslog] [--export-all]
             [--timeout=<n>] [--init-timeout=<n>] [--max-connections=<n>]
             [--strict-paths] [--base-path=<path>] [--base-path-relaxed]
             [--user-path | --user-path=<path>]
             [--interpolated-path=<pathtemplate>]
             [--reuseaddr] [--detach] [--pid-file=<file>]
             [--enable=<service>] [--disable=<service>]
             [--allow-override=<service>] [--forbid-override=<service>]
             [--access-hook=<path>] [--[no-]informative-errors]
             [--inetd |
              [--listen=<host_or_ipaddr>] [--port=<n>]
              [--user=<user> [--group=<group>]]]
             [--log-destination=(stderr|syslog|none)]
             [<directory>…​]

Description

Действительно простой демон TCP Git , который обычно прослушивает порт "DEFAULT_GIT_PORT" , также известный как 9418. Он ожидает соединения, запрашивающего услугу, и будет обслуживать эту услугу, если она включена.

Он проверяет, что в каталоге есть волшебный файл "git-daemon-export-ok" , и откажется экспортировать любой каталог Git , который явно не помечен для экспорта таким образом (если не указан параметр --export-all ). Если вы передадите некоторые пути к каталогам в качестве аргументов git daemon , предложения будут ограничены репозиториями внутри этих каталогов.

По умолчанию включена только служба upload-pack , которая обслуживает клиентов git fetch-pack и git ls-remote , которые вызываются из git fetch , git pull и git clone .

Это идеально подходит для обновлений только для чтения, т. е. извлечения из репозиториев Git .

upload-archive также существует для обслуживания git archive .

Options

--strict-paths

Точно сопоставьте пути (т. е. не допускайте использования "/foo/repo" , если реальный путь — "/foo/repo.git" или "/foo/repo/.git" ) и не используйте пути, относящиеся к пользователю. git daemon откажется запускаться, если эта опция включена и аргументы каталога не указаны.

--base-path=<path>

Пересопоставьте все запросы пути относительно заданного пути. Это своего рода "Git root" — если вы запустите git daemon с --base-path=/srv/git на сайте example.com, то если вы позже попытаетесь вытащить git://example.com/hello.git , git daemon интерпретирует путь как /srv/git/hello.git .

--base-path-relaxed

Если опция --base-path включена и поиск репозитория не удался, с помощью этой опции git daemon попытается выполнить поиск без префикса базового пути. Это полезно для переключения на использование --base-path, сохраняя при этом возможность использования старых путей.

--interpolated-path=<pathtemplate>

Для поддержки виртуального хостинга можно использовать шаблон интерполированного пути для динамического создания альтернативных путей. Шаблон поддерживает %H для целевого имени хоста, предоставленного клиентом, но преобразованного в нижний регистр, %CH для канонического имени хоста, %IP для IP-адреса сервера, %P для номера порта и %D для абсолютного пути к названный репозиторий. После интерполяции путь проверяется по списку каталогов.

--export-all

Разрешить извлечение из всех каталогов, похожих на репозитории Git (имеющих подкаталоги objects и refs ), даже если в них нет файла git-daemon-export-ok .

--inetd

Запустите сервер как службу inetd. Подразумевается --syslog (может быть переопределен с помощью --log-destination= ). Несовместимо с параметрами --detach, --port, --listen, --user и --group.

--listen=<host_or_ipaddr>

Прослушивайте определенный IP-адрес или имя хоста. IP-адреса могут быть адресами IPv4 или IPv6, если они поддерживаются. Если IPv6 не поддерживается, то --listen=имя_хоста также не поддерживается, и --listen должен быть присвоен адрес IPv4. Можно давать более одного раза. Несовместим с опцией --inetd .

--port=<n>

Слушайте альтернативный порт. Несовместим с опцией --inetd .

--init-timeout=<n>

Тайм-аут (в секундах) между моментом установления соединения и получением клиентского запроса (обычно довольно низкое значение, поскольку он должен быть в основном немедленным).

--timeout=<n>

Тайм-аут (в секундах) для определенных клиентских подзапросов. Сюда входит время, необходимое серверу для обработки подзапроса, и время, потраченное на ожидание следующего запроса клиента.

--max-connections=<n>

Максимальное количество одновременных клиентов, по умолчанию — 32. Установите его на ноль, чтобы не было ограничений.

--syslog

Сокращение от --log-destination=syslog .

--log-destination=<destination>

Отправлять сообщения журнала в указанное место назначения. Обратите внимание, что эта опция не подразумевает --verbose, поэтому по умолчанию будут регистрироваться только ошибки. <пункт назначения> должен быть одним из:

stderr

Напишите стандартную ошибку. Обратите внимание: если указан --detach , процесс отключается от реальной стандартной ошибки, что делает этот пункт назначения фактически эквивалентным none .

syslog

Запишите в системный журнал, используя идентификатор git-daemon .

none

Отключите все журналирование.

Назначением по умолчанию является syslog , если указан --inetd или --detach , в противном случае stderr .

--user-path
--user-path=<path>

Разрешить использование обозначения ~user в запросах. Если указано без параметра, запрос к git://host/~alice/foo воспринимается как запрос на доступ к репозиторию foo в домашнем каталоге пользователя alice . Если указан --user-path=path , тот же запрос воспринимается как запрос на доступ к репозиторию path/foo в домашнем каталоге пользователя alice .

--verbose

Записывайте подробности о входящих соединениях и запрошенных файлах.

--reuseaddr

Используйте SO_REUSEADDR при привязке сокета прослушивания. Это позволяет серверу перезапуститься, не дожидаясь истечения времени ожидания старых подключений.

--detach

Отделите от скорлупы. Подразумевается --syslog.

--pid-file=<file>

Сохраните идентификатор процесса в file . Игнорируется, когда демон запускается под управлением --inetd .

--user=<user>
--group=<group>

Измените uid и gid демона перед входом в сервисный цикл. Если указан только --user без --group , используется идентификатор основной группы пользователя. Значения параметра указаны для getpwnam(3) и getgrnam(3) , а числовые идентификаторы не поддерживаются.

Указание этих параметров является ошибкой при использовании с --inetd ; используйте возможности демона inet, чтобы добиться того же, прежде чем создавать git daemon , если это необходимо.

Как и многие программы, которые меняют идентификатор пользователя, демон не сбрасывает переменные среды, такие как $HOME , при запуске программ git , например upload-pack и receive-pack . При использовании этой опции вы также можете установить и экспортировать HOME так, чтобы он указывал на домашний каталог <user> перед запуском демона, и убедиться, что все файлы конфигурации Git в этом каталоге доступны для чтения <user> .

--enable=<service>
--disable=<service>

Включите/отключите службу для всего сайта по умолчанию. Обратите внимание, что службу, отключенную для всего сайта, все равно можно включить для каждого репозитория, если она помечена как переопределяемая, а репозиторий включает службу с помощью элемента конфигурации.

--allow-override=<service>
--forbid-override=<service>

Разрешить/запретить переопределение настроек по умолчанию для всего сайта с помощью конфигурации каждого репозитория. По умолчанию все службы могут быть переопределены.

--[no-]informative-errors

Когда информативные ошибки включены, git-daemon будет сообщать клиенту более подробные ошибки, отличая такие условия, как "no such repository" , от "repository not exported" . Это удобнее для клиентов, но может привести к утечке информации о существовании неэкспортированных репозиториев. Если информативные ошибки отключены, обо всех ошибках клиенту сообщается "access denied" . По умолчанию используется --no-informative-errors.

--access-hook=<path>

Каждый раз, когда клиент подключается, сначала запускайте внешнюю команду, указанную в <путь> с именем службы (например, "upload-pack" ), путем к репозиторию, именем хоста (%H), каноническим именем хоста (%CH), IP-адресом (%IP), и TCP-порт (%P) в качестве аргументов командной строки. Внешняя команда может принять решение об отказе от услуги, выйдя с ненулевым статусом (или разрешить ее, выйдя с нулевым статусом). Он также может просмотреть переменные среды $REMOTE_ADDR и $REMOTE_PORT , чтобы узнать об отправителе запроса при принятии этого решения.

Внешняя команда может опционально записать одну строку в свой стандартный вывод, которая будет отправлена ​​запрашивающей стороне в качестве сообщения об ошибке, когда она отклонит услугу.

<directory>

Остальные аргументы предоставляют список каталогов. Если указаны какие-либо каталоги, то процесс git-daemon будет обслуживать запрошенный каталог, только если он содержится в одном из этих каталогов. Если указан --strict-paths , то запрошенный каталог должен точно соответствовать одному из этих каталогов.

Services

Эти службы можно глобально включать/отключать с помощью параметров командной строки этой команды. Если требуется более детальный контроль (например, чтобы разрешить запуск git archive только в нескольких выбранных репозиториях, которые обслуживает демон), для их включения или отключения можно использовать файл конфигурации для каждого репозитория.

upload-pack

Он обслуживает клиентов git fetch-pack и git ls-remote . Он включен по умолчанию, но репозиторий может отключить его, установив для элемента конфигурации daemon.uploadpack значение false .

upload-archive

Это обслуживает git archive --remote . По умолчанию он отключен, но репозиторий может включить его, установив для элемента конфигурации daemon.uploadarch значение true .

receive-pack

Он обслуживает клиентов git send-pack , обеспечивая анонимную отправку сообщений. По умолчанию он отключен, так как в протоколе есть аутентификация no (другими словами, кто угодно может запихнуть в репозиторий что угодно, включая удаление рефов). Это предназначено исключительно для закрытой локальной сети, где все дружелюбны. Эту службу можно включить, установив для элемента конфигурации daemon.receivepack значение true .

Examples

Мы предполагаем следующее в /etc/services
$ grep 9418 /etc/services
git                9418/tcp                # Система контроля версий Git
Демон git в качестве сервера inetd

Чтобы настроить git daemon как службу inetd, которая обрабатывает любой репозиторий внутри /pub/foo или /pub/bar , поместите в /etc/inetd запись, подобную следующей, в одну строку:

        git stream tcp nowait nobody  /usr/bin/git
                git daemon --inetd --verbose --export-all
                /pub/foo /pub/bar
Демон git в качестве сервера inetd для виртуальных хостов

Чтобы настроить git daemon в качестве службы inetd, которая обрабатывает репозитории для разных виртуальных хостов, www.example.com и www.example.org , поместите в /etc/inetd запись, подобную следующей, в одну строку:

        git stream tcp nowait nobody /usr/bin/git
                git daemon --inetd --verbose --export-all
                --interpolated-path=/pub/%H%D
                /pub/www.example.org/software
                /pub/www.example.com/software
                /software

В этом примере каталог корневого уровня /pub будет содержать подкаталог для каждого поддерживаемого имени виртуального хоста. Далее оба хоста рекламируют репозитории просто как git://www.example.com/software/repo.git . Для клиентов версий до 1.4.0 также можно создать символическую ссылку из /software в соответствующий репозиторий по умолчанию.

Демон git как обычный демон для виртуальных хостов

Чтобы настроить git daemon как обычную службу, не связанную с inetd, которая обрабатывает репозитории для нескольких виртуальных хостов на основе их IP-адресов, запустите демон следующим образом:

        git daemon --verbose --export-all
                --interpolated-path=/pub/%IP/%D
                /pub/192.168.1.200/software
                /pub/10.10.220.23/software

В этом примере каталог корневого уровня /pub будет содержать подкаталог для каждого поддерживаемого IP-адреса виртуального хоста. Однако доступ к репозиториям по-прежнему возможен по имени хоста, если они соответствуют этим IP-адресам.

выборочно включать/отключать сервисы для каждого репозитория

Чтобы включить git archive --remote и отключить git fetch для репозитория, добавьте следующее в файл конфигурации репозитория (то есть файл config рядом с HEAD , refs и objects ).

        [daemon]
                uploadpack = false
                uploadarch = true

Environment

git daemon установит REMOTE_ADDR на IP-адрес клиента, который подключился к нему, если IP-адрес доступен. REMOTE_ADDR будет доступен в среде перехватчиков, вызываемых при выполнении сервисов.
