git-read-tree

Name

git-read-tree — считывает информацию о дереве в индекс

Synopsis

git read-tree [(-m [--trivial] [--aggressive] | --reset | --prefix=<prefix>)
                [-u | -i]] [--index-output=<file>] [--no-sparse-checkout]
                (--empty | <tree-ish1> [<tree-ish2> [<tree-ish3>]])

Description

Считывает информацию о дереве, заданную <tree-ish>, в индекс, но фактически не обновляет ни один из файлов. "caches" . (см.: git-checkout-index[1] )

При желании он может объединить дерево с индексом, выполнить ускоренное (т. е. двухстороннее) или трехстороннее слияние с помощью флага -m . При использовании с -m флаг -u заставляет его также обновлять файлы в рабочем дереве с результатом слияния.

Только тривиальные слияния выполняются самим git read-tree . Только конфликтующие пути будут в несвязанном состоянии при возвращении git read-tree .

Options

-m

Выполните слияние, а не просто чтение. Команда откажется запускаться, если в вашем индексном файле есть неслитые записи, что указывает на то, что вы не завершили предыдущее начатое слияние.

--reset

То же, что и -m, за исключением того, что необъединённые записи отбрасываются, а не выдаются неудачно. При использовании с -u обновления, приводящие к потере изменений рабочего дерева или неотслеживаемым файлам или каталогам, не прерывают операцию.

-u

После успешного слияния обновите файлы в рабочем дереве с результатом слияния.

-i

Обычно для слияния требуется, чтобы индексный файл, а также файлы в рабочем дереве были обновлены до текущей головной фиксации, чтобы не потерять локальные изменения. Этот флаг отключает проверку с рабочим деревом и предназначен для использования при создании слияния деревьев, не связанных напрямую с текущим состоянием рабочего дерева, во временный индексный файл.

-n
--dry-run

Проверьте, не выдаст ли команда ошибку без реального обновления индекса или файлов в рабочем дереве.

-v

Показывать ход извлечения файлов.

--trivial

Ограничьте трехстороннее слияние с помощью git read-tree только в том случае, если слияние на уровне файлов не требуется, вместо того, чтобы разрешать слияние для тривиальных случаев и оставлять конфликтующие файлы неразрешенными в индексе.

--aggressive

Обычно трехстороннее слияние с помощью git read-tree разрешает слияние для действительно тривиальных случаев и оставляет другие случаи неразрешенными в индексе, так что фарфоры могут реализовать разные политики слияния. Этот флаг заставляет команду разрешать еще несколько случаев внутри себя:

  • когда одна сторона удаляет путь, а другая сторона оставляет путь неизмененным. Решение состоит в том, чтобы удалить этот путь.

  • когда обе стороны удаляют путь. Решение состоит в том, чтобы удалить этот путь.

  • когда обе стороны добавляют путь одинаково. Решение состоит в том, чтобы добавить этот путь.

--prefix=<prefix>

Сохраните текущее содержимое индекса и прочитайте содержимое именованного дерева в каталоге <prefix> . Команда откажется перезаписывать записи, которые уже существовали в исходном индексном файле.

--index-output=<file>

Вместо записи результатов в $GIT_INDEX_FILE запишите полученный индекс в указанный файл. Пока команда работает, исходный индексный файл блокируется тем же механизмом, что и обычно. Файл должен допускать переименование(2) из ​​временного файла, который создается рядом с обычным индексным файлом; обычно это означает, что он должен находиться в той же файловой системе, что и сам индексный файл, и вам необходимо разрешение на запись в каталоги, в которых расположены индексный файл и выходной файл индекса.

--[no-]recurse-submodules

Использование --recurse-submodules обновит содержимое всех активных подмодулей в соответствии с фиксацией, записанной в суперпроекте, путем рекурсивного вызова дерева чтения, а также установки HEAD подмодулей для отсоединения при этой фиксации.

--no-sparse-checkout

Отключите поддержку разреженных проверок, даже если core.sparseCheckout имеет значение true.

--empty

Вместо чтения объектов дерева в индекс просто очистите его.

-q
--quiet

Тихо, подавляйте сообщения обратной связи.

<tree-ish#>

Идентификатор объекта(ов) дерева, который необходимо прочитать/объединить.

Merging

Если указан -m , git read-tree может выполнять 3 вида слияния: слияние одного дерева, если задано только 1 дерево, ускоренное слияние с двумя деревьями или трехстороннее слияние, если задано 3 или более деревьев.

Слияние одного дерева

Если указано только одно дерево, git read-tree работает так, как если бы пользователь не указал -m , за исключением того, что если в исходном индексе есть запись для данного пути и содержимое пути совпадает с считываемым деревом, статистическая информация из используется индекс. (Другими словами, stat() индекса имеют приоритет над объединенным деревом).

Это означает, что если вы выполняете git read-tree -m <newtree> , а затем git checkout-index -f -u -a , git checkout-index проверяет только то, что действительно изменилось.

Это используется, чтобы избежать ненужных попаданий false , когда git diff-files запускается после git read-tree .

Слияние двух деревьев

Обычно это вызывается как git read-tree -m $H $M , где $H — это голова фиксации текущего репозитория, а $M — это голова внешнего дерева, которое просто опережает $H (т. е. мы находимся в ситуации перемотки вперед).

Если указаны два дерева, пользователь сообщает git read-tree следующее:

  1. Текущий индекс и рабочее дерево получены из $H, но пользователь может иметь в них локальные изменения, начиная с $H.

  2. Пользователь хочет перейти к $M.

В этом случае команда git read-tree -m $H $M гарантирует, что никакие локальные изменения не будут потеряны в результате действия "merge" . Вот правила "carry forward" , где "I" обозначает индекс, "clean" означает, что индекс и рабочее дерево совпадают, а "exists" / "nothing" относятся к наличию пути в указанном коммите:

        I                   H        M        Result
       -------------------------------------------------------
     0  nothing             nothing  nothing  (does not happen)
     1  nothing             nothing  exists   use M
     2  nothing             exists   nothing  remove path from index
     3  nothing             exists   exists,  use M if "initial checkout",
                                     H == M   keep index otherwise
                                     exists,  fail
                                     H != M

        clean I==H  I==M
       ------------------
     4  yes   N/A   N/A     nothing  nothing  keep index
     5  no    N/A   N/A     nothing  nothing  keep index

     6  yes   N/A   yes     nothing  exists   keep index
     7  no    N/A   yes     nothing  exists   keep index
     8  yes   N/A   no      nothing  exists   fail
     9  no    N/A   no      nothing  exists   fail

     10 yes   yes   N/A     exists   nothing  remove path from index
     11 no    yes   N/A     exists   nothing  fail
     12 yes   no    N/A     exists   nothing  fail
     13 no    no    N/A     exists   nothing  fail

        clean (H==M)
       ------
     14 yes                 exists   exists   keep index
     15 no                  exists   exists   keep index

        clean I==H  I==M (H!=M)
       ------------------
     16 yes   no    no      exists   exists   fail
     17 no    no    no      exists   exists   fail
     18 yes   no    yes     exists   exists   keep index
     19 no    no    yes     exists   exists   keep index
     20 yes   yes   no      exists   exists   use M
     21 no    yes   no      exists   exists   fail

Во всех случаях "keep index" индексная запись остается такой же, как в исходном индексном файле. Если запись устарела, git read-tree сохраняет копию в рабочем дереве неповрежденной при работе с флагом -u.

Когда эта форма git read-tree успешно возвращается, вы можете увидеть, какой из созданных вами "local changes" был перенесен, запустив git diff-index --cached $M . Обратите внимание, что это не обязательно соответствует тому, что git diff-index --cached $H создал бы до такого слияния двух деревьев. Это связано со случаями 18 и 19 — если бы у вас уже были изменения в $M (например, вы получили их по электронной почте в форме исправления), git diff-index --cached $H сообщил бы вам об изменении до этого слияния, но это не отображаться в выводе git diff-index --cached $M после слияния двух деревьев.

Случай 3 немного сложен и требует пояснения. Логически результатом этого правила должно быть удаление пути, если пользователь инсценировал удаление пути, а затем переключился на новую ветку. Однако это предотвратит первоначальную проверку, поэтому правило изменено на использование M (новое дерево) только тогда, когда содержимое индекса пусто. В противном случае удаление пути сохраняется до тех пор, пока $H и $M совпадают.

3-Way Merge

Каждая запись "index" имеет два бита состояния "stage" . стадия 0 является нормальной и единственной, которую вы можете увидеть при обычном использовании.

Однако когда вы выполняете git read-tree с тремя деревьями, "stage" начинается с 1.

Это означает, что вы можете сделать

$ git read-tree -m <tree1> <tree2> <tree3>

и в итоге вы получите индекс со всеми записями <tree1> в "stage1" , всеми записями <tree2> в "stage2" и всеми записями <tree3> в "stage3" . При слиянии другой ветки с текущей ветвью мы используем общее предковое дерево как <дерево1>, текущий заголовок ветки как <дерево2> и другой заголовок ветки как <дерево3>.

Кроме того, git read-tree имеет логику особого случая, которая гласит: если вы видите файл, который соответствует во всех отношениях в следующих состояниях, он "collapses" возвращается к "stage0" :

  • 2 и 3 этапы одинаковы; взять то или другое (без разницы - на нашей ветке на 2 этапе и на их ветке на 3 этапе проделана одна и та же работа)

  • этап 1 и этап 2 одинаковы, а этап 3 отличается; взять этап 3 (наша ветка на этапе 2 ничего не делала, так как предок на этапе 1 пока над ним работал их филиал на этапе 3)

  • этап 1 и этап 3 одинаковы, а этап 2 отличается, возьмем этап 2 (мы что-то сделали, а они ничего не сделали)

Команда git write-tree отказывается записывать бессмысленное дерево и будет жаловаться на необъединённые записи, если увидит одну запись, которая не находится на этапе 0.

Хорошо, все это звучит как набор совершенно бессмысленных правил, но на самом деле это именно то, что вам нужно для быстрого слияния. Различные этапы представляют собой "result tree" (этап 0, он же "merged" ), исходное дерево (этап 1, он же "orig" ) и два дерева, которые вы пытаетесь объединить (этапы 2 и 3 соответственно).

Порядок этапов 1, 2 и 3 (следовательно, порядок трех аргументов командной строки <tree-ish>) важен, когда вы запускаете трехстороннее слияние с уже заполненным индексным файлом. Вот краткое описание того, как работает алгоритм:

  • если файл существует в одинаковом формате во всех трех деревьях, он автоматически свернется до состояния "merged" с помощью git read-tree .

  • файл, который имеет разницу any в трех деревьях, останется в индексе как отдельные записи. "porcelain policy" должен определить, как удалить этапы, отличные от 0, и вставить объединенную версию.

  • индексный файл сохраняет и восстанавливает всю эту информацию, поэтому вы можете объединять элементы постепенно, но пока в нем есть записи на этапах 1/2/3 (т. е. "unmerged entries" ), вы не можете записать результат. Итак, теперь алгоритм слияния становится очень простым:

    • вы проходите индекс по порядку и игнорируете все записи этапа 0, поскольку они уже были сделаны.

    • если вы нашли "stage1" , но не нашли подходящего "stage2" или "stage3" , вы знаете, что он был удален из обоих деревьев (он существовал только в исходном дереве), и вы удаляете эту запись.

    • если вы найдете совпадающее дерево "stage2" и "stage3" , вы удалите одно из них и превратите другое в запись "stage0" . Удалите любую соответствующую запись "stage1" , если она также существует. ..все нормальные тривиальные правила..

Для выполнения последнего шага обычно используется git merge-index с поставляемым git merge-one-file . Скрипт обновляет файлы в рабочем дереве при слиянии каждого пути и в конце успешного слияния.

Когда вы запускаете трехстороннее слияние с уже заполненным индексным файлом, предполагается, что он представляет состояние файлов в вашем рабочем дереве, и вы даже можете иметь файлы с изменениями, не записанными в индексном файле. Далее предполагается, что это состояние "derived" из дерева этапа 2. Трехстороннее слияние отказывается выполняться, если в исходном индексном файле обнаруживается запись, не соответствующая этапу 2.

Это сделано для того, чтобы вы не потеряли незавершенные изменения и не смешали случайные изменения в несвязанном коммите слияния. Для иллюстрации предположим, что вы начинаете с того, что было зафиксировано в вашем репозитории последним:

$ JC=`git rev-parse --verify "HEAD^0"`
$ git checkout-index -f -u -a $JC

Вы делаете случайные изменения, не запуская git update-index . И тут вы замечаете, что верхушка вашего дерева "upstream" выдвинулась вперед с тех пор, как вы его выдернули:

$ git fetch git://.... linus
$ LT=`git rev-parse FETCH_HEAD`

Ваше рабочее дерево по-прежнему основано на HEAD ($JC), но с тех пор вы внесли некоторые изменения. Трехстороннее слияние гарантирует, что вы не добавляли и не изменяли записи индекса после $JC, а если вы этого не сделали, то поступает правильно. Итак, в следующей последовательности:

$ git read-tree -m -u `git merge-base $JC $LT` $JC $LT
$ git merge-index git-merge-one-file -a
$ echo "Merge with Linus" | \
  git commit-tree `git write-tree` -p $JC -p $LT

то, что вы бы зафиксировали, - это чистое слияние $JC и $LT без изменений незавершенной работы, и ваше рабочее дерево будет обновлено в соответствии с результатом слияния.

Однако если у вас есть локальные изменения в рабочем дереве, которые будут перезаписаны в результате этого слияния, git read-tree откажется запускаться, чтобы предотвратить потерю ваших изменений.

Другими словами, не нужно беспокоиться о том, что существует только в рабочем дереве. Если у вас есть локальные изменения в части проекта, которая не участвует в слиянии, ваши изменения не мешают слиянию и остаются неизменными. Когда они мешают, слияние даже не начинается ( git read-tree громко жалуется и выходит из строя, ничего не модифицируя). В таком случае вы можете просто продолжать делать то, что вы делали в середине, и когда ваше рабочее дерево будет готово (т. е. вы закончите незавершенную работу), попытайтесь выполнить слияние еще раз.

Sparse checkout

Примечание. Возможности пропуска рабочего дерева в git-update-index[1] и read-tree появились до появления git-sparse-checkout[1] . Пользователям рекомендуется использовать команду sparse-checkout вместо этих сантехнических команд для нужд, связанных с разреженной проверкой/пропуском рабочего дерева. Однако информация ниже может быть полезна пользователям, пытающимся понять стиль узора, используемый в неконусном режиме команды sparse-checkout .

"Sparse checkout" позволяет разреженно заполнять рабочий каталог. Он использует бит Skip-worktree (см. git-update-index[1] ), чтобы сообщить Git , стоит ли просматривать файл в рабочем каталоге.

git read-tree и другие команды на основе слияния ( git merge , git checkout …​) могут помочь поддерживать растровое изображение пропуска рабочего дерева и обновление рабочего каталога. $GIT_DIR/info/sparse-checkout используется для определения эталонного растрового изображения рабочего дерева пропуска. Когда git read-tree необходимо обновить рабочий каталог, он сбрасывает бит Skip-worktree в индексе на основе этого файла, который использует тот же синтаксис, что и файлы .gitignore. Если запись соответствует шаблону в этом файле или запись соответствует файлу, присутствующему в рабочем дереве, то Skip-worktree не будет установлен для этой записи. В противном случае будет установлено Skip-Worktree.

Затем он сравнивает новое значение Skip-Worktree с предыдущим. Если параметр Skip-worktree переключится из установленного в неустановленное состояние, соответствующий файл будет добавлен обратно. Если он изменится с неустановленного на установленное, этот файл будет удален.

Хотя $GIT_DIR/info/sparse-checkout обычно используется для указания того, в каких файлах находится not , вы также можете указать, в каких файлах находится not , используя шаблоны отрицания. Например, чтобы удалить файл unwanted :

/*
!unwanted

Еще одна сложная вещь — полное повторное заполнение рабочего каталога, когда вам больше не нужна разреженная проверка. Вы не можете просто отключить "sparse checkout" , поскольку биты пропуска рабочего дерева все еще находятся в индексе, а ваш рабочий каталог все еще малонаселен. Вам следует повторно заполнить рабочий каталог содержимым файла $GIT_DIR/info/sparse-checkout следующим образом:

/*

Затем вы можете отключить разреженную проверку. Поддержка разреженной проверки в git read-tree и подобных командах отключена по умолчанию. Вам необходимо включить core.sparseCheckout , чтобы иметь редкую поддержку оформления заказа.
