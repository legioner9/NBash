git-log

Name

git-log — Показать журналы коммитов

Synopsis

git log [<options>] [<revision-range>] [[--] <path>…​]

Description

Показывает журналы коммитов.

Перечислите коммиты, доступные по ссылкам parent из заданных коммитов, но исключите коммиты, доступные из коммитов, указанных с ^ перед ними. По умолчанию выходные данные выдаются в обратном хронологическом порядке.

Вы можете думать об этом как о заданной операции. Коммиты, доступные из любого коммита, заданного в командной строке, образуют набор, а затем из этого набора вычитаются коммиты, доступные из любого из коммитов, указанных с ^ впереди. Остальные коммиты — это то, что выводится на выходе команды. Для дальнейшего ограничения результата можно использовать различные другие параметры и параметры путей.

Таким образом, следующая команда:

$ git log foo bar ^baz

означает "list all the commits which are reachable from foo or bar , but not from baz " .

Специальное обозначение " <commit1> .. <commit2> " может использоваться как сокращение для "^ <commit1> <commit2> " . Например, любое из следующих значений может использоваться взаимозаменяемо:

$ git log origin..HEAD
$ git log HEAD ^origin

Еще одно специальное обозначение — " <commit1> … <commit2> " , которое полезно для слияний. Результирующий набор коммитов представляет собой симметричную разницу между двумя операндами. Следующие две команды эквивалентны:

$ git log A B --not $(git merge-base --all A B)
$ git log A...B

Команда принимает параметры, применимые к команде git-rev-list[1] , для управления тем, что и как отображается, а также параметры, применимые к команде git-diff[1] , для управления тем, как отображаются изменения, вносимые каждой фиксацией.

Options

--follow

Продолжить просмотр истории файла без переименований (работает только для одного файла).

--no-decorate
--decorate[=short|full|auto|no]

Распечатайте имена ссылок всех отображаемых коммитов. Если указан short , префиксы имен refs/heads/ , refs/tags/ и refs/remotes/ не будут напечатаны. Если указан full , будет напечатано полное имя ссылки (включая префикс). Если указан auto , то если выход идет на терминал, имена ссылок отображаются так, как если бы были заданы short , в противном случае имена ссылок не отображаются. Опция --decorate является сокращением от --decorate=short . По умолчанию используется значение конфигурации log.decorate , если оно настроено, в противном случае — auto .

--decorate-refs=<pattern>
--decorate-refs-exclude=<pattern>

Не используйте для украшения каждую ссылку-кандидат, если она соответствует каким-либо шаблонам, заданным для --decorate-refs-exclude , или если она не соответствует ни одному из шаблонов, заданных для --decorate-refs . Опция конфигурации log.excludeDecoration позволяет исключать ссылки из украшений, но явный шаблон --decorate-refs переопределит совпадение в log.excludeDecoration .

Если указаны none этих опций или настроек конфигурации, то в качестве украшения используются ссылки, если они соответствуют HEAD , refs/heads/ , refs/remotes/ , refs/stash/ или refs/tags/ .

--clear-decorations

Если этот параметр указан, он очищает все предыдущие параметры --decorate-refs или --decorate-refs-exclude и ослабляет фильтр оформления по умолчанию, чтобы включить все ссылки. Этот параметр предполагается, если для параметра конфигурации log.initialDecorationSet установлено значение all .

--source

Распечатайте имя ссылки, указанное в командной строке, с помощью которого был достигнут каждый коммит.

--[no-]mailmap
--[no-]use-mailmap

Используйте файл mailmap, чтобы сопоставить имена и адреса электронной почты авторов и коммиттеров с каноническими настоящими именами и адресами электронной почты. См. git-shortlog[1] .

--full-diff

Без этого флага git log -p <path>... показывает коммиты, затрагивающие указанные пути, и различия по тем же указанным путям. При этом полный diff отображается для коммитов, которые касаются указанных путей; это означает, что "<path>…​" ограничивает только коммиты и не ограничивает различия для этих коммитов.

Обратите внимание, что это влияет на все типы вывода на основе различий, например, созданные --stat и т. д.

--log-size

Включите строку «размер журнала <число>» в выходные данные для каждого коммита, где <номер> — это длина сообщения этого коммита в байтах. Предназначен для ускорения инструментов, которые читают сообщения журнала из вывода git log , позволяя им заранее выделять пространство.

-L<start>,<end>:<file>
-L:<funcname>:<file>

Проследите эволюцию диапазона строк, заданного <start>,<end> или именем функции regex <funcname> , внутри <file> . Вы не можете указывать какие-либо ограничители пути. В настоящее время это ограничено обходом, начиная с одной ревизии, т. е. вы можете указать только ноль или один положительный аргумент ревизии, а <start> и <end> (или <funcname> ) должны существовать в начальной ревизии. Эту опцию можно указать несколько раз. Подразумевается --patch . Вывод патчей можно подавить с помощью --no-patch , но другие форматы различий (а именно --raw , --numstat , --shortstat , --dirstat , --summary , --name-only , --name-status , --check ) в настоящее время не реализованы.

<start> и <end> могут принимать одну из следующих форм:

  • number

    Если <start> или <end> — число, оно указывает абсолютный номер строки (строки отсчитываются с 1).

  • /regex/

    В этой форме будет использоваться первая строка, соответствующая данному регулярному выражению POSIX. Если <start> является регулярным выражением, поиск будет осуществляться с конца предыдущего диапазона -L , если таковой имеется, в противном случае — с начала файла. Если <start> равен ^/regex/ , поиск будет начинаться с начала файла. Если <end> является регулярным выражением, поиск будет начинаться со строки, заданной <start> .

  • +смещение или -смещение

    Это справедливо только для <end> и будет указывать количество строк до или после строки, заданной <start> .

Если вместо <start> и <end> указан :<funcname> , это регулярное выражение, обозначающее диапазон от первой строки имени функции, соответствующей <funcname> , до следующей строки имени функции. :<funcname> выполняет поиск с конца предыдущего диапазона -L , если таковой имеется, в противном случае — с начала файла. ^:<funcname> выполняет поиск с начала файла. Имена функций определяются так же, как git diff обрабатывает заголовки фрагментов патчей (см. Defining a custom hunk-header в gitattributes[5] ).

<revision-range>

Показывать только коммиты в указанном диапазоне ревизий. Если <revision-range> не указан, по умолчанию используется HEAD (т. е. вся история, ведущая к текущему коммиту). origin..HEAD указывает все коммиты, доступные из текущего коммита (т. е. HEAD ), но не из origin . Полный список способов написания <revision-range> см. в разделе Specifying Ranges документа gitrevisions[7] .

[--] <path>…

Показывать только те коммиты, которых достаточно, чтобы объяснить, как появились файлы, соответствующие указанным путям. Подробности и другие режимы упрощения см. в History Simplification ниже.

В случае возникновения путаницы в путях может потребоваться префикс -- , чтобы отделить их от опций или диапазона ревизий.

Commit Limiting

Помимо указания диапазона коммитов, которые должны быть перечислены с использованием специальных обозначений, объясненных в описании, может быть применено дополнительное ограничение коммитов.

Использование большего количества параметров обычно дополнительно ограничивает вывод (например, --since=<date1> ограничивает коммиты новее, чем <date1> , а использование его с --grep=<pattern> дополнительно ограничивает коммиты, сообщение журнала которых имеет строку, соответствующую <pattern> ), если не указано иное.

Обратите внимание, что они применяются перед фиксацией параметров порядка и форматирования, таких как --reverse .

-<number>
-n <number>
--max-count=<number>

Ограничьте количество коммитов для вывода.

--skip=<number>

Пропустите фиксации number перед отображением результатов фиксации.

--since=<date>
--after=<date>

Показывать коммиты, более поздние, чем определенная дата.

--since-as-filter=<date>

Показать все коммиты, сделанные позднее определенной даты. При этом выполняются все коммиты в диапазоне, а не останавливаются на первом коммите, который старше определенной даты.

--until=<date>
--before=<date>

Показать коммиты старше определенной даты.

--author=<pattern>
--committer=<pattern>

Ограничьте вывод коммитов теми, у которых строки заголовка автора/коммиттера соответствуют указанному шаблону (регулярному выражению). При наличии более одного --author=<pattern> выбираются коммиты, автор которых соответствует любому из заданных шаблонов (аналогично для нескольких --committer=<pattern> ).

--grep-reflog=<pattern>

Ограничьте вывод коммитов теми, в которых записи журнала обновлений соответствуют указанному шаблону (регулярному выражению). При наличии более чем одного --grep-reflog выбираются коммиты, сообщение рефлога которых соответствует любому из заданных шаблонов. Использование этой опции, если не используется --walk-reflogs , является ошибкой.

--grep=<pattern>

Ограничьте вывод коммитов теми, в которых сообщение журнала соответствует указанному шаблону (регулярному выражению). При наличии более чем одного --grep=<pattern> выбираются коммиты, сообщение которых соответствует любому из заданных шаблонов (но см. --all-match ).

Когда действует --notes , сообщение из примечаний сопоставляется, как если бы оно было частью сообщения журнала.

--all-match

Ограничьте вывод коммитов теми, которые соответствуют всем заданным --grep , вместо тех, которые соответствуют хотя бы одному.

--invert-grep

Ограничьте вывод коммитов теми, в которых сообщение журнала не соответствует шаблону, указанному в --grep=<pattern> .

-i
--regexp-ignore-case

Сопоставьте шаблоны ограничения регулярных выражений без учета регистра букв.

--basic-regexp

Считайте, что ограничивающие шаблоны — это базовые регулярные выражения; это значение по умолчанию.

-E
--extended-regexp

Рассматривайте ограничивающие шаблоны как расширенные регулярные выражения вместо базовых регулярных выражений по умолчанию.

-F
--fixed-strings

Считайте, что ограничивающие шаблоны представляют собой фиксированные строки (не интерпретируйте шаблон как регулярное выражение).

-P
--perl-regexp

Считайте, что ограничивающие шаблоны — это регулярные выражения, совместимые с Perl.

Поддержка этих типов регулярных выражений является дополнительной зависимостью времени компиляции. Если Git не был скомпилирован с их поддержкой, предоставление этой опции приведет к его прекращению.

--remove-empty

Остановитесь, когда заданный путь исчезнет из дерева.

--merges

Печатать только коммиты слияния. Это точно так же, как --min-parents=2 .

--no-merges

Не печатайте коммиты с более чем одним родителем. Это точно так же, как --max-parents=1 .

--min-parents=<number>
--max-parents=<number>
--no-min-parents
--no-max-parents

Показывать только те коммиты, которые имеют как минимум (или максимум) такое же количество родительских коммитов. В частности, --max-parents=1 аналогичен --no-merges , --min-parents=2 аналогичен --merges . --max-parents=0 дает все корневые коммиты, а --min-parents=3 все слияния осьминога.

--no-min-parents и --no-max-parents снова сбрасывают эти ограничения (на отсутствие ограничений). Эквивалентными формами являются --min-parents=0 (любая фиксация имеет 0 или более родительских элементов) и --max-parents=-1 (отрицательные числа означают отсутствие верхнего предела).

--first-parent

При поиске коммитов для включения следуйте только первому родительскому коммиту после просмотра коммита слияния. Этот параметр может дать лучший обзор при просмотре развития конкретной тематической ветки, поскольку слияния с тематической веткой, как правило, сводятся только к корректировке время от времени обновлений исходной ветки, и этот параметр позволяет игнорировать отдельные коммиты, внесенные в ваша история от такого слияния.

Эта опция также меняет формат различий по умолчанию для коммитов слияния на first-parent , подробности см. в --diff-merges=first-parent .

--exclude-first-parent-only

При поиске коммитов для исключения (с ^ ) следуйте только первому родительскому коммиту после просмотра коммита слияния. Это можно использовать для поиска набора изменений в тематической ветке с той точки, где он расходится с удаленной веткой, учитывая, что произвольные слияния могут быть действительными изменениями тематической ветки.

--not

Меняет значение префикса ^ (или его отсутствия) для всех последующих спецификаторов версии, вплоть до следующего --not . При использовании в командной строке перед --stdin изменения, передаваемые через стандартный ввод, не будут затронуты. И наоборот, при передаче через стандартный ввод изменения, переданные в командной строке, не будут затронуты.

--all

Представьте, что все ссылки в refs/ вместе с HEAD указаны в командной строке как <commit> .

--branches[=<pattern>]

Представьте, что все ссылки в refs/heads указаны в командной строке как <commit> . Если задан <pattern> , ограничьте ветки теми, которые соответствуют данному glob оболочки. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--tags[=<pattern>]

Представьте, что все ссылки в refs/tags указаны в командной строке как <commit> . Если задан <pattern> , ограничьте теги теми, которые соответствуют данному glob оболочки. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--remotes[=<pattern>]

Представьте, что все ссылки в refs/remotes указаны в командной строке как <commit> . Если задан <pattern> , ограничьте ветки удаленного отслеживания теми, которые соответствуют заданному glob оболочки. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--glob=<glob-pattern>

Представьте, что все ссылки, соответствующие glob оболочки <glob-pattern> , указаны в командной строке как <commit> . Ведущий refs/ автоматически добавляется в начало, если отсутствует. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--exclude=<glob-pattern>

Не включайте ссылки, соответствующие <glob-pattern> , которые в противном случае учитывались бы следующими --all , --branches , --tags , --remotes или --glob . При повторении этого параметра накапливаются шаблоны исключений до следующего параметра --all , --branches , --tags , --remotes или --glob (другие параметры или аргументы не удаляют накопленные шаблоны).

Приведенные шаблоны не должны начинаться с refs/heads , refs/tags или refs/remotes при применении к --branches , --tags или --remotes соответственно, и они должны начинаться с refs/ при применении к --glob или --all . Если предполагается завершающий /* , его необходимо указать явно.

--exclude-hidden=[fetch|receive|uploadpack]

Не включайте ссылки, которые могут быть скрыты git-fetch , git-receive-pack или git-upload-pack , сверившись с соответствующей конфигурацией fetch.hideRefs , receive.hideRefs или uploadpack.hideRefs вместе с transfer.hideRefs (см. git-config[1] ). Эта опция влияет на следующую опцию псевдо-ссылки --all или --glob и очищается после их обработки.

--reflog

Представьте, что все объекты, упомянутые в рефлогах, указаны в командной строке как <commit> .

--alternate-refs

Представьте, что все объекты, упомянутые в качестве справочных подсказок альтернативных репозиториев, были перечислены в командной строке. Альтернативный репозиторий — это любой репозиторий, каталог объектов которого указан в objects/info/alternates . Набор включенных объектов может быть изменен с помощью core.alternateRefsCommand и т. д. См. git-config[1] .

--single-worktree

По умолчанию все рабочие деревья будут проверяться по следующим параметрам, если их больше одного (см. git-worktree[1] ): --all , --reflog и --indexed-objects . Эта опция заставляет их проверять только текущее рабочее дерево.

--ignore-missing

Увидев во входных данных недопустимое имя объекта, притворитесь, будто неверных входных данных не было.

--bisect

Представьте, что в списке указан неправильный номер деления пополам refs/bisect/bad , а за ним в командной строке следуют --not и хорошие ссылки пополам refs/bisect/good-* .

--stdin

Помимо получения аргументов из командной строки, также читайте их из стандартного ввода. При этом принимаются коммиты и псевдоопции, такие как --all и --glob= . Когда виден разделитель -- , следующие входные данные обрабатываются как пути и используются для ограничения результата. Флаги типа --not , которые считываются через стандартный ввод, учитываются только для аргументов, передаваемых таким же образом, и не влияют на последующие аргументы командной строки.

--cherry-mark

Как --cherry-pick (см. ниже), но помечайте эквивалентные коммиты с помощью = , а не опускайте их, а неэквивалентные — с помощью + .

--cherry-pick

Опускайте любой коммит, который вносит те же изменения, что и другой коммит на «другой стороне», если набор коммитов ограничен симметричной разницей.

Например, если у вас есть две ветки, A и B , обычный способ перечислить все коммиты только на одной стороне из них — это --left-right (см. пример ниже в описании опции --left-right ). Однако он показывает коммиты, которые были выбраны из другой ветки (например, «3rd on b» может быть выбран из ветки A). При использовании этой опции такие пары коммитов исключаются из вывода.

--left-only
--right-only

В списке фиксируются только соответствующие стороны симметричной разницы, т. е. только те, которые будут отмечены < соответственно. > от --left-right .

Например, --cherry-pick --right-only A...B пропускает те фиксации из B , которые находятся в A или эквивалентны исправлениям фиксации в A . Другими словами, здесь перечислены коммиты + из git cherry A B . Точнее, --cherry-pick --right-only --no-merges дает точный список.

--cherry

Синоним --right-only --cherry-mark --no-merges ; полезно ограничить вывод коммитами на нашей стороне и пометить те, которые были применены к другой стороне разветвленной истории, с помощью git log --cherry upstream...mybranch , аналогично git cherry upstream mybranch .

-g
--walk-reflogs

Вместо того, чтобы идти по цепочке предков коммита, пройдитесь по записям релога от самой последней к более старой. При использовании этой опции вы не можете указать исключаемые фиксации (то есть нельзя использовать нотации ^commit , commit1..commit2 и commit1...commit2 ).

Если формат --pretty отличается от oneline и reference (по очевидным причинам), это приводит к тому, что в выводе появляются две дополнительные строки информации, взятые из журнала ссылок. Обозначение журнала обновлений в выходных данных может отображаться как ref@{Nth} (где Nth — индекс обратной хронологии в журнале обновлений) или как ref@{timestamp} (с ​​отметкой времени для этой записи), в зависимости от нескольких правил:

  1. Если отправная точка указана как ref@{Nth} , покажите формат индекса.

  2. Если отправная точка была указана как ref@{now} , покажите формат метки времени.

  3. Если ни один из них не использовался, но в командной строке был указан --date , покажите временную метку в формате, запрошенном --date .

  4. В противном случае покажите формат индекса.

В --pretty=oneline сообщение о фиксации предваряется этой информацией в той же строке. Эту опцию нельзя комбинировать с --reverse . См. также git-reflog[1] .

Под --pretty=reference эта информация вообще не будет отображаться.

--merge

После неудачного слияния покажите ссылки, которые касаются файлов, имеющих конфликт, и не существуют на всех головках для слияния.

--boundary

Выходные данные исключают граничные фиксации. Граничные коммиты имеют префикс - .

History Simplification

Иногда вас интересуют только части истории, например, коммиты, изменяющие определенный <path>. Но есть две части History Simplification : одна часть выбирает коммиты, а другая — как это сделать, поскольку существуют различные стратегии для упрощения истории.

Следующие параметры позволяют выбрать коммиты для отображения:

<paths>

Выбраны коммиты, изменяющие заданные <пути>.

--simplify-by-decoration

Выбираются коммиты, на которые ссылается какая-либо ветка или тег.

Обратите внимание, что дополнительные коммиты могут быть показаны для получения значимой истории.

Следующие параметры влияют на способ выполнения упрощения:

Default mode

Упрощает историю до простейшей истории, объясняющей конечное состояние дерева. Самый простой, потому что он удаляет некоторые боковые ветки, если конечный результат тот же (т. е. объединяет ветки с одинаковым содержимым).

--show-pulls

Включите все коммиты из режима по умолчанию, а также любые коммиты слияния, которые не являются TREESAME для первого родителя, но являются TREESAME для более позднего родителя. Этот режим полезен для отображения коммитов слияния, которые "first introduced" являются изменением ветки.

--full-history

То же, что и режим по умолчанию, но не удаляет часть истории.

--dense

Показаны только выбранные коммиты, а также некоторые из них, имеющие значимую историю.

--sparse

Показаны все коммиты в упрощенной истории.

--simplify-merges

Дополнительная опция для --full-history , позволяющая удалить некоторые ненужные слияния из результирующей истории, поскольку в это слияние не входят выбранные коммиты.

--ancestry-path[=<commit>]

Если задан диапазон отображаемых коммитов (например, commit1..commit2 или commit2 ^commit1 ), в этом диапазоне отображаются только те коммиты, которые являются предками <commit>, потомками <commit> или самого <commit>. Если фиксация не указана, используйте commit1 (исключенная часть диапазона) в качестве <фиксации>. Можно проходить несколько раз; в этом случае коммит включается, если он является одним из данных коммитов или является предком или потомком одного из них.

Далее следует более подробное объяснение.

Предположим, вы указали foo в качестве <paths>. Мы будем называть коммиты, изменяющие foo , !TREESAME, а остальные TREESAME. (В diff, отфильтрованном для foo , они выглядят разными и равными соответственно.)

В дальнейшем мы всегда будем обращаться к одному и тому же примеру истории, чтобы проиллюстрировать различия между настройками упрощения. Мы предполагаем, что вы фильтруете файл foo в этом графе коммитов:

          .-A---M---N---O---P---Q
         /     /   /   /   /   /
        I     B   C   D   E   Y
         \   /   /   /   /   /
          `-------------'   X

Горизонтальная линия истории A---Q считается первым родителем каждого слияния. Коммиты:

  • I — это первоначальный коммит, в котором существует foo с содержимым «asdf», а файл quux существует с содержимым «quux». Первоначальные коммиты сравниваются с пустым деревом, поэтому I имеет значение !TREESAME.

  • В A foo содержит только «foo».

  • B содержит те же изменения, что и A . Его слияние M тривиально и, следовательно, TREESAME для всех родителей.

  • C не меняет foo , но его слияние N меняет его на «foobar», поэтому он не является TREESAME ни для одного родителя.

  • D устанавливает для foo значение «baz». Его слияние O объединяет строки из N и D в «foobarbaz»; т. е. это не TREESAME ни для одного родителя.

  • E изменяет quux на «xyzzy», а его слияние P объединяет строки в «quux xyzzy». P TREESAME O , но не E .

  • X — это независимый корневой коммит, который добавил новый файл side , а Y изменил его. Y — это ДЕРЕВО, аналогичное X . Его слияние Q добавило side к P , а Q является TREESAME к P , но не к Y .

rev-list просматривает историю назад, включая или исключая фиксации в зависимости от того, используется ли --full-history и/или родительская перезапись (через --parents или --children ). Доступны следующие настройки.

Default mode

Коммиты включаются, если они не являются TREESAME ни для одного родителя (хотя это можно изменить, см. --sparse ниже). Если фиксация была слиянием и была TREESAME для одного родителя, следуйте только этому родителю. (Даже если существует несколько родителей TREESAME, подписывайтесь только на одного из них.) В противном случае следуйте за всеми родителями.

Это приводит к:

          .-A---N---O
         /     /   /
        I---------D

Обратите внимание, что правило следовать только родительскому элементу TREESAME, если таковой имеется, полностью исключило B из рассмотрения. C рассматривался через N , но это TREESAME. Корневые коммиты сравниваются с пустым деревом, поэтому I — это !TREESAME.

Отношения родитель/потомок видны только с помощью --parents , но это не влияет на коммиты, выбранные в режиме по умолчанию, поэтому мы показали родительские строки.

--полная история без перезаписи родительских элементов

Этот режим отличается от режима по умолчанию в одном пункте: всегда следует за всеми родительскими элементами слияния, даже если для одного из них это TREESAME. Даже если более чем одна сторона слияния имеет включенные коммиты, это не означает, что само слияние включено! В примере мы получаем

        I  A  B  N  D  O  P  Q

M был исключен, поскольку он является TREESAME для обоих родителей. E , C и B были пройдены, но только B был !TREESAME, поэтому остальные не отображаются.

Обратите внимание, что без перезаписи родительских элементов невозможно говорить об отношениях родитель/потомок между коммитами, поэтому мы показываем их несвязанными.

--полная история с перезаписью родительских элементов

Обычные коммиты включаются только в том случае, если они имеют формат !TREESAME (хотя это можно изменить, см. --sparse ниже).

Слияния всегда включены. Однако их родительский список переписан: вдоль каждого родителя удаляются коммиты, которые не включены сами. Это приводит к

          .-A---M---N---O---P---Q
         /     /   /   /   /
        I     B   /   D   /
         \   /   /   /   /
          `-------------'

Сравните с --full-history не переписывая выше. Обратите внимание, что E был удален, поскольку он TREESAME, но родительский список P был переписан, чтобы содержать родительский I E . То же самое произошло с C и N , а также X , Y и Q .

Помимо вышеуказанных настроек, вы можете изменить, влияет ли TREESAME на включение:

--dense

Проходящие коммиты включаются, если они не являются TREESAME ни для одного родителя.

--sparse

Включаются все пройденные коммиты.

Обратите внимание, что без --full-history это по-прежнему упрощает слияния: если один из родительских элементов TREESAME, мы следуем только этому, поэтому другие стороны слияния никогда не обрабатываются.

--simplify-merges

Сначала построим график истории так же, как это делает --full-history с перезаписью родителя (см. выше).

Затем упростите каждую фиксацию C до замены C' в итоговой истории по следующим правилам:

  • Установите C' на C .

  • Замените каждый родительский P C' его упрощением P' . В процессе удаления родителей, которые являются предками других родителей или корневых коммитов TREESAME в пустое дерево, и удаляйте дубликаты, но будьте осторожны, чтобы никогда не удалять всех родителей, к которым мы TREESAME.

  • Если после перезаписи родительского объекта C' является корневым коммитом или фиксацией слияния (имеет ноль или >1 родительских элементов), граничным коммитом или !TREESAME, он остается. В противном случае он заменяется единственным родителем.

Эффект от этого лучше всего проявляется в сравнении с --full-history с перезаписью родительского элемента. Пример превращается в:

          .-A---M---N---O
         /     /       /
        I     B       D
         \   /       /
          `---------'

Обратите внимание на основные отличия N , P и Q от --full-history :

  • Из родительского списка N был удален I , поскольку он является предком другого родительского списка M . Тем не менее, N остался, потому что это !TREESAME.

  • Из родительского списка P аналогичным образом был удален I . Затем P был полностью удален, поскольку у него был один родительский элемент и он был TREESAME.

  • Родительский список Q имел Y , упрощенный до X . Затем X был удален, поскольку это был корень TREESAME. Затем Q был полностью удален, поскольку у него был один родительский элемент и он был TREESAME.

Доступен еще один режим упрощения:

--ancestry-path[=<commit>]

Ограничьте отображаемые коммиты теми, которые являются предками <commit>, потомками <commit> или самими <commit>.

В качестве примера использования рассмотрим следующую историю коммитов:

            D---E-------F
           /     \       \
          B---C---G---H---I---J
         /                     \
        A-------K---------------L--M

Обычный D..M вычисляет набор коммитов, которые являются предками M , но исключает те, которые являются предками D . Это полезно для того, чтобы увидеть, что произошло с историей, ведущей к M , начиная с D , в том смысле, что «что есть в M такого, чего не было в D ». Результатом в этом примере будут все коммиты, кроме A и B (и, конечно, самого D ).

Однако, когда мы хотим выяснить, какие коммиты в M заражены ошибкой, представленной D , и нуждаются в исправлении, мы можем захотеть просмотреть только подмножество D..M , которое на самом деле является потомками D , то есть исключая C и K . Именно это и делает опция --ancestry-path . Применительно к линейке D..M это приводит к:

                E-------F
                 \       \
                  G---H---I---J
                               \
                                L--M

Мы также можем использовать --ancestry-path=D вместо --ancestry-path , что означает то же самое применительно к диапазону D..M , но это более явно.

Если вместо этого нас интересует данная тема в этом диапазоне и все коммиты, затронутые этой темой, мы можем захотеть просмотреть только подмножество D..M , которое содержит эту тему в своем исходном пути. Итак, использование --ancestry-path=H D..M , например, приведет к:

                E
                 \
                  G---H---I---J
                               \
                                L--M

Тогда как --ancestry-path=K D..M приведет к

                K---------------L--M

Прежде чем обсуждать другой вариант, --show-pulls , нам нужно создать новый пример истории.

Распространенная проблема, с которой сталкиваются пользователи при просмотре упрощенной истории, заключается в том, что известная им фиксация, изменившая файл, каким-то образом не отображается в упрощенной истории файла. Продемонстрируем новый пример и покажем, как в таком случае работают такие варианты, как --full-history и --simplify-merges :

          .-A---M-----C--N---O---P
         /     / \  \  \/   /   /
        I     B   \  R-'`-Z'   /
         \   /     \/         /
          \ /      /\        /
           `---X--'  `---Y--'

В этом примере предположим, что I создал file.txt , который был модифицирован A , B и X по-разному. Однородительские фиксации C , Z и Y не изменяют file.txt . Коммит слияния M был создан путем разрешения конфликта слияния и включил в себя оба изменения из A и B и, следовательно, не является TREESAME ни для одного из них. Однако коммит слияния R был создан путем игнорирования содержимого file.txt по адресу M и взятия только содержимого file.txt по адресу X . Следовательно, R является TREESAME X , но не M . Наконец, естественным решением слияния для создания N является использование содержимого file.txt по адресу R , поэтому N является TREESAME для R , но не для C . Коммиты слияния O и P являются TREESAME для своих первых родительских элементов, но не для вторых родительских, Z и Y соответственно.

При использовании режима по умолчанию N и R имеют родительского элемента TREESAME, поэтому эти ребра обходят, а остальные игнорируются. Итоговый график истории:

        I---X

При использовании --full-history Git проходит каждое ребро. Это позволит обнаружить коммиты A и B и слияние M , а также выявить коммиты слияния O и P . При переписывании родительских элементов результирующий график выглядит следующим образом:

          .-A---M--------N---O---P
         /     / \  \  \/   /   /
        I     B   \  R-'`--'   /
         \   /     \/         /
          \ /      /\        /
           `---X--'  `------'

Здесь коммиты слияния O и P вносят дополнительный шум, поскольку на самом деле они не вносят изменений в file.txt . Они лишь объединили тему, основанную на более старой версии file.txt . Это распространенная проблема в репозиториях, использующих рабочий процесс, в котором многие участники работают параллельно и объединяют свои тематические ветки в одну магистраль: в результатах --full-history появляется много несвязанных слияний.

При использовании опции --simplify-merges коммиты O и P исчезают из результатов. Это связано с тем, что переписанные вторые родительские элементы O и P доступны из их первых родителей. Эти ребра удаляются, и тогда коммиты выглядят как коммиты с одним родителем, которые являются TREESAME для своего родителя. Это также происходит с коммитом N , в результате чего история выглядит следующим образом:

          .-A---M--.
         /     /    \
        I     B      R
         \   /      /
          \ /      /
           `---X--'

В этом представлении мы видим все важные изменения одного родителя из A , B и X . Мы также видим тщательно решенное слияние M и не очень тщательно разрешенное слияние R . Обычно этой информации достаточно, чтобы определить, почему A и B "disappeared" фиксируются из истории в представлении по умолчанию. Однако у этого подхода есть несколько проблем.

Первая проблема — производительность. В отличие от любого предыдущего варианта, параметр --simplify-merges требует пройти всю историю коммитов, прежде чем возвращать единственный результат. Это может затруднить использование этой опции для очень больших репозиториев.

Второй вопрос касается аудита. Когда над одним и тем же репозиторием работает много участников, важно, какие коммиты слияния внесли изменения в важную ветку. Вышеупомянутое проблемное слияние R вряд ли является фиксацией слияния, которая использовалась для слияния с важной веткой. Вместо этого было использовано слияние N для объединения R и X в важную ветвь. Этот коммит может содержать информацию о том, почему изменение X переопределяет изменения из A и B в его сообщении о фиксации.

--show-pulls

В дополнение к коммитам, отображаемым в истории по умолчанию, покажите каждый коммит слияния, который не является TREESAME для своего первого родителя, но является TREESAME для более позднего родителя.

Когда фиксация слияния включена в --show-pulls , слияние обрабатывается так, как если бы это "pulled" было изменением из другой ветки. При использовании --show-pulls в этом примере (и никаких других опций) результирующий график выглядит следующим образом:

        I---X---R---N

Здесь включены коммиты слияния R и N , поскольку они перенесли коммиты X и R в базовую ветку соответственно. Эти слияния являются причиной того, что коммиты A и B не отображаются в истории по умолчанию.

Когда --show-pulls работает в паре с --simplify-merges , график включает всю необходимую информацию:

          .-A---M--.   N
         /     /    \ /
        I     B      R
         \   /      /
          \ /      /
           `---X--'

Обратите внимание: поскольку M доступен из R , граница от N до M была упрощена. Однако N по-прежнему фигурирует в истории как важный коммит, поскольку это "pulled" — изменение R в основной ветке.

Опция --simplify-by-decoration позволяет просматривать только общую картину топологии истории, опуская коммиты, на которые не ссылаются теги. Коммиты помечаются как !TREESAME (другими словами, сохраняются после описанных выше правил упрощения истории), если (1) на них ссылаются теги или (2) они изменяют содержимое путей, указанных в командной строке. Все остальные коммиты помечены как TREESAME (можно упростить).

Commit Ordering

По умолчанию коммиты отображаются в обратном хронологическом порядке.

--date-order

Не показывать родителей до того, как будут показаны все его дочерние элементы, но в противном случае показывать фиксации в порядке временных меток фиксации.

--author-date-order

Не показывать родителей до того, как будут показаны все его дочерние элементы, но в противном случае показывать фиксации в порядке временных меток автора.

--topo-order

Не показывать родительских элементов до того, как будут показаны все его дочерние элементы, и избегайте отображения коммитов в нескольких перемешанных строках истории.

Например, в истории коммитов так:

    ---1----2----4----7
        \               \
         3----5----6----8---

где цифры обозначают порядок временных меток коммитов, git rev-list и его друзья с --date-order показывают коммиты в порядке временных меток: 8 7 6 5 4 3 2 1.

С --topo-order они будут показывать 8 6 5 3 7 4 2 1 (или 8 7 4 2 6 5 3 1); некоторые старые коммиты отображаются перед новыми, чтобы избежать смешивания коммитов из двух параллельных треков разработки.

--reverse

Выведите коммиты, выбранные для отображения (см. раздел «Ограничение коммитов» выше), в обратном порядке. Невозможно объединить с --walk-reflogs .

Object Traversal

Эти опции в основном предназначены для упаковки репозиториев Git .

--no-walk[=(sorted|unsorted)]

Показывайте только данные коммиты, но не просматривайте их предков. Это не имеет никакого эффекта, если указан диапазон. Если указан аргумент unsorted , фиксации отображаются в том порядке, в котором они были указаны в командной строке. В противном случае (если sorted или аргумент не указан), фиксации отображаются в обратном хронологическом порядке по времени фиксации. Невозможно объединить с --graph .

--do-walk

Переопределяет предыдущий --no-walk .

Commit Formatting

--pretty[=<format>]
--format=<format>

Довольно распечатать содержимое журналов коммитов в заданном формате, где <format> может быть одним из oneline , short , medium , full , fuller , reference , email , raw , format:<string> и tformat:<string> . Когда <format> является none из вышеперечисленных и содержит %placeholder , он действует так, как если бы был задан --pretty=tformat:<format> .

Дополнительные сведения о каждом формате см. в разделе "PRETTY FORMATS" . Если часть =<format> опущена, по умолчанию используется medium .

Примечание: вы можете указать красивый формат по умолчанию в конфигурации репозитория (см. git-config[1] ).

--abbrev-commit

Вместо отображения полного 40-байтового шестнадцатеричного имени объекта фиксации покажите префикс, который дает уникальное имя объекту. Опцию "--abbrev=<n>" (которая также изменяет вывод различий, если она отображается) можно использовать для указания минимальной длины префикса.

Это должно сделать "--pretty=oneline" намного более читабельным для людей, использующих 80-колоночные терминалы.

--no-abbrev-commit

Показать полное 40-байтовое шестнадцатеричное имя объекта фиксации. Это отменяет --abbrev-commit , явный или подразумеваемый другими параметрами, такими как "--oneline" . Он также переопределяет переменную log.abbrevCommit .

--oneline

Это сокращение от "--pretty=oneline --abbrev-commit" , используемое вместе.

--encoding=<encoding>

Объекты фиксации записывают кодировку символов, используемую для сообщения журнала, в свой заголовок кодировки; эту опцию можно использовать, чтобы указать команде перекодировать сообщение журнала фиксации в кодировке, предпочитаемой пользователем. Для команд, не связанных с сантехникой, по умолчанию используется UTF-8. Обратите внимание: если объект утверждает, что он закодирован в X , а мы выводим в X , мы выведем объект дословно; это означает, что недопустимые последовательности в исходном коммите могут быть скопированы в выходные данные. Аналогично, если iconv(3) не сможет преобразовать фиксацию, мы спокойно выведем исходный объект дословно.

--expand-tabs=<n>
--expand-tabs
--no-expand-tabs

Выполните расширение табуляции (замените каждую табуляцию достаточным количеством пробелов для заполнения до следующего столбца отображения, кратного <n> ) в сообщении журнала, прежде чем отображать его в выходных данных. --expand-tabs — это сокращение от --expand-tabs=8 , а --no-expand-tabs — сокращение от --expand-tabs=0 , которое отключает раскрытие вкладок.

По умолчанию вкладки раскрываются в красивых форматах с отступом в сообщении журнала на 4 пробела (например, medium , который используется по умолчанию, full и fuller ).

--notes[=<ref>]

Показывать примечания (см. git-notes[1] ), аннотирующие фиксацию, при отображении сообщения журнала фиксации. Это значение по умолчанию для команд git log , git show и git whatchanged , когда в командной строке не указана опция --pretty , --format или --oneline .

По умолчанию отображаемые примечания взяты из ссылок на примечания, перечисленных в переменных core.notesRef и notes.displayRef (или в соответствующих переопределениях среды). Дополнительную информацию см. в git-config[1] .

С дополнительным аргументом <ref> используйте ссылку, чтобы найти примечания для отображения. Ссылка может указать полное имя ссылки, если оно начинается с refs/notes/ ; когда он начинается с notes/ , refs/ и в противном случае refs/notes/ имеет префикс, образующий полное имя ссылки.

Несколько опций --notes можно комбинировать, чтобы контролировать, какие заметки будут отображаться. Примеры: "--notes=foo" отображает только примечания "refs/notes/foo" ; "--notes=foo --notes" будет отображать как примечания из "refs/notes/foo" , так и ссылки по умолчанию.

--no-notes

Не показывать заметки. Это отменяет указанную выше опцию --notes путем сброса списка ссылок на заметки, из которых отображаются заметки. Параметры анализируются в порядке, указанном в командной строке, поэтому, например, "--notes --notes=foo --no-notes --notes=bar" будет отображать только примечания из "refs/notes/bar" .

--show-notes-by-default

Показывать заметки по умолчанию, если не указаны параметры отображения конкретных заметок.

--show-notes[=<ref>]
--[no-]standard-notes

Эти параметры устарели. Вместо этого используйте приведенные выше параметры --notes/--no-notes.

--show-signature

Проверьте достоверность подписанного объекта фиксации, передав подпись gpg --verify , и покажите выходные данные.

--relative-date

Синоним --date=relative .

--date=<format>

Действует только для дат, отображаемых в удобочитаемом формате, например, при использовании --pretty . Переменная конфигурации log.date устанавливает значение по умолчанию для параметра --date команды журнала. По умолчанию даты отображаются в исходном часовом поясе (коммиттере или авторе). Если к формату добавляется -local (например, iso-local ), вместо него используется местный часовой пояс пользователя.

--date=relative показывает даты относительно текущего времени, например «2 часа назад». Опция -local не влияет на --date=relative .

--date=local — это псевдоним --date=default-local .

--date=iso (или --date=iso8601 ) отображает временные метки в формате, подобном ISO 8601. Отличия от строгого формата ISO 8601:

  • пробел вместо разделителя даты и времени T

  • пространство между временем и часовым поясом

  • нет двоеточия между часами и минутами часового пояса

--date=iso-strict (или --date=iso8601-strict ) отображает временные метки в строгом формате ISO 8601.

--date=rfc (или --date=rfc2822 ) показывает временные метки в формате RFC 2822, часто встречающиеся в сообщениях электронной почты.

--date=short показывает только дату, но не время, в формате YYYY-MM-DD .

--date=raw показывает дату в секундах с начала эпохи (1970-01-01 00:00:00 UTC), за которой следует пробел, а затем часовой пояс как смещение от UTC ( + или - с четырьмя цифрами; первые две — это часы, а вторые две — минуты). Т.е. как если бы временная метка была отформатирована с помощью strftime("%s %z") ). Обратите внимание, что опция -local не влияет на значение секунд с начала эпохи (которое всегда измеряется в формате UTC), но переключает соответствующее значение часового пояса.

--date=human показывает часовой пояс, если часовой пояс не соответствует текущему часовому поясу, и не печатает всю дату, если он совпадает (т. е. пропускает печать года для дат, которые являются "this year" , но также пропускает всю дату, если она находится в последнем несколько дней, и мы можем просто сказать, какой это был будний день). Для более старых дат также опускаются час и минута.

--date=unix отображает дату как временную метку эпохи Unix (в секундах с 1970 года). Как и в случае с --raw , здесь всегда используется формат UTC, поэтому -local не имеет никакого эффекта.

--date=format:... передает формат ... в вашу систему strftime , за исключением %s, %z и %Z, которые обрабатываются внутри. Используйте --date=format:%c , чтобы отобразить дату в предпочтительном формате вашей системы. Полный список заполнителей формата см. в руководстве strftime . При использовании -local правильный синтаксис — --date=format-local:... .

--date=default — это формат по умолчанию, основанный на выводе ctime(3). Он показывает одну строку с трехбуквенным днем ​​недели, трехбуквенным месяцем, днем ​​месяца, часаминутами-секундами в формате "HH:MM:SS" , за которым следует 4-значный год, а также информация о часовом поясе, если не указан местный часовой пояс. используется, например, Thu Jan 1 00:00:00 1970 +0000 .

--parents

Выведите также родителей коммита (в виде "commit parent…​" ). Также включает перезапись родительских элементов, см. History Simplification выше.

--children

Выведите также дочерние элементы коммита (в виде "commit child…​" ). Также включает перезапись родительских элементов, см. History Simplification выше.

--left-right

Отметьте, с какой стороны симметричного различия доступен коммит. Коммиты слева имеют префикс < , а коммиты справа — > . В сочетании с --boundary эти коммиты имеют префикс - .

Например, если у вас есть такая топология:

             y---b---b  branch B
            / \ /
           /   .
          /   / \
         o---x---a---a  branch A

вы получите такой вывод:

        $ git rev-list --left-right --boundary --pretty=oneline A...B

        >bbbbbbb... 3rd on b
        >bbbbbbb... 2nd on b
        <aaaaaaa... 3rd on a
        <aaaaaaa... 2nd on a
        -yyyyyyy... 1st on b
        -xxxxxxx... 1st on a
--graph

Нарисуйте текстовое графическое представление истории коммитов в левой части выходных данных. Это может привести к печати дополнительных строк между коммитами, чтобы история графика отображалась правильно. Невозможно объединить с --no-walk .

Это позволяет перезаписать родительский элемент, см. History Simplification выше.

По умолчанию это подразумевает опцию --topo-order , но можно также указать опцию --date-order .

--show-linear-break[=<barrier>]

Когда --graph не используется, все ветки истории сглаживаются, из-за чего может быть трудно увидеть, что два последовательных коммита не принадлежат линейной ветке. В этом случае этот вариант ставит барьер между ними. Если указан <barrier> , именно эта строка будет отображаться вместо строки по умолчанию.

Pretty formats

Если фиксация представляет собой слияние и если симпатичный формат не oneline , email или raw , перед строкой Author: вставляется дополнительная строка. Эта строка начинается с "Merge: " , и хэши наследственных коммитов печатаются через пробелы. Обратите внимание, что перечисленные коммиты не обязательно могут быть списком прямых родительских коммитов, если вы ограничили просмотр истории: например, если вас интересуют только изменения, относящиеся к определенному каталогу или файлу.

Существует несколько встроенных форматов, и вы можете определить дополнительные форматы, установив для параметра конфигурации Pretty.<name> либо другое имя формата, либо строку format: , как описано ниже (см. git-config[1] ). Вот подробности о встроенных форматах:

  • oneline

    <hash> <title-line>

    Это сделано для того, чтобы быть максимально компактным.

  • short

    commit <hash>
Author: <author>
    <title-line>

  • medium

    commit <hash>
Author: <author>
Date:   <author-date>
    <title-line>
    <full-commit-message>

  • full

    commit <hash>
Author: <author>
Commit: <committer>
    <title-line>
    <full-commit-message>

  • fuller

    commit <hash>
Author:     <author>
AuthorDate: <author-date>
Commit:     <committer>
CommitDate: <committer-date>
    <title-line>
    <full-commit-message>

  • reference

    <abbrev-hash> (<title-line>, <short-author-date>)

    Этот формат используется для ссылки на другую фиксацию в сообщении о фиксации и аналогичен --pretty='format:%C(auto)%h (%s, %ad)' . По умолчанию дата форматируется с помощью --date=short , если явно не указан другой параметр --date . Как и в случае с любым format: с заполнителями формата, на его вывод не влияют другие параметры, такие как --decorate и --walk-reflogs .

  • email

    From <hash> <date>
From: <author>
Date: <author-date>
Subject: [PATCH] <title-line>
    <full-commit-message>

  • mboxrd

    Аналогично email , но строки в сообщении о фиксации, начинающиеся с "From " (перед которыми стоит ноль или более ">" ), заключаются в кавычки с ">" , поэтому их не путают с началом новой фиксации.

  • raw

    Формат raw показывает всю фиксацию точно так, как она хранится в объекте фиксации. Примечательно, что хэши отображаются полностью, независимо от того, используется ли --abbrev или --no-abbrev, а информация parents показывает родительские коммиты true , без учета трансплантаций или упрощения истории. Обратите внимание, что этот формат влияет на способ отображения коммитов, но не на способ отображения различий, например, с помощью git log --raw . Чтобы получить полные имена объектов в формате необработанных различий, используйте --no-abbrev .

  • format:<format-string>

    Формат format:<format-string> позволяет указать, какую информацию вы хотите отображать. Он работает немного похоже на формат printf, за исключением того, что вы получаете новую строку с %n вместо \n .

    Например, format:"The author of %h was %an, %ar%nThe title was >>%s<<%n" покажет что-то вроде этого:

    The author of fe6e0ee was Junio C Hamano, 23 hours ago
The title was >>t4119: test autocomputing -p<n> for traditional diff input.<<

    Заполнители:

    • Заполнители, которые расширяются до одного буквального символа:

      %n

      newline

      %%

      сырой %

      %x00

      %x , за которым следуют две шестнадцатеричные цифры, заменяется байтом со значением шестнадцатеричных цифр (в остальной части документа мы будем называть его "literal formatting code" ).

    • Заполнители, влияющие на форматирование последующих заполнителей:

      %Cred

      переключить цвет на красный

      %Cgreen

      переключить цвет на зеленый

      %Cblue

      переключить цвет на синий

      %Creset

      reset color

      %C(…​)

      спецификация цвета, как описано в разделе «Значения» раздела "CONFIGURATION FILE" документа git-config[1] . По умолчанию цвета отображаются только при включении вывода журнала (с помощью color.diff , color.ui или --color и с учетом настроек auto первого, если мы идем к терминалу). %C(auto,...) принят как исторический синоним значения по умолчанию (например, %C(auto,red) ). При указании %C(always,...) цвета будут отображаться, даже если цвет не включен иным образом (хотя рассмотрите возможность просто использовать --color=always , чтобы включить цвет для всего вывода, включая этот формат и все остальное, что может окрашивать git ). Только auto (т.е. %C(auto) ) включит автоматическую раскраску для следующих заполнителей до тех пор, пока цвет не будет переключен снова.

      %m

      левая ( < ), правая ( > ) или граничная ( - ) метка

      %w([<w>[,<i1>[,<i2>]]])

      переключение переноса строк, например, опция -w git-shortlog[1] .

      %<( <N> [,trunc|ltrunc|mtrunc])

      сделайте так, чтобы следующий заполнитель занимал не менее N ширины столбца, при необходимости заполняя пробелы справа. При необходимости обрежьте (с многоточием .. ) слева (ltrunc) ..ft , в середине (mtrunc) mi..le или в конце (trunc) rig.. , если выходные данные длиннее N столбцов. Примечание 1: усечение работает правильно только при N >= 2. Примечание 2: пробелы вокруг значений N и M (см. ниже) не являются обязательными. Примечание 3. Эмодзи и другие широкие символы занимают два столбца отображения, что может выходить за границы столбцов. Примечание 4: метки объединения разложенных символов могут быть смещены на границах заполнения.

      %<|( <M> )

      сделайте так, чтобы следующий заполнитель занимал как минимум до M-го столбца отображения, при необходимости заполняя пробелы справа. Используйте отрицательные значения M для позиций столбцов, измеренных от правого края окна терминала.

      %>( <N>), %>|( <M>)

      аналогично %<( <N> ) и %<|( <M> ) соответственно, но с пробелами слева

      %>>( <N>), %>>|( <M>)

    %(trailers[:<options>])

    отобразить прицепы кузова, интерпретированные git-interpret-trailers[1] . За строкой trailers может следовать двоеточие и ноль или более параметров, разделенных запятыми. Если какой-либо вариант указан несколько раз, побеждает последний вариант.

    • key=<key> : показывать только трейлеры с указанным <ключом>. Сопоставление выполняется без учета регистра, двоеточие в конце не является обязательным. Если опция задана несколько раз, отображаются строки концевика, соответствующие любому из ключей. Эта опция автоматически включает опцию only , поэтому линии, не относящиеся к концевому блоку, скрываются. Если это нежелательно, его можно отключить с помощью only=false . Например, %(trailers:key=Reviewed-by) показывает строки трейлера с ключом Reviewed-by .

    • only[=<bool>] : выберите, следует ли включать неконцевые линии из трейлерного блока.

    • separator=<sep> : укажите разделитель, вставленный между завершающими строками. Если эта опция не задана, каждая завершающая строка завершается символом перевода строки. Строка <sep> может содержать буквальные коды форматирования, описанные выше. Чтобы использовать запятую в качестве разделителя, необходимо использовать %x2C , так как в противном случае она будет анализироваться как следующая опция. Например, %(trailers:key=Ticket,separator=%x2C ) показывает все строки концевика, ключ которых "Ticket" , разделенный запятой и пробелом.

    • unfold[=<bool>] : заставить его вести себя так, как если бы была указана опция --unfold интерпретатора-трейлера. Например, %(trailers:only,unfold=true) разворачивается и показывает все строки трейлера.

    • keyonly[=<bool>] : показывать только ключевую часть трейлера.

    • valueonly[=<bool>] : показывать только ценную часть трейлера.

    • key_value_separator=<sep> : укажите разделитель, вставленный между завершающими строками. Если этот параметр не задан, каждая пара ключ-значение трейлера разделяется ": " . В остальном он имеет ту же семантику, что и separator=<sep> выше.

Note
 Некоторые заполнители могут зависеть от других параметров, заданных механизму обхода ревизий. Например, параметры журнала ссылок %g* будут вставлять пустую строку, если только мы не просматриваем записи журнала ссылок (например, с помощью git log -g ). Заполнители %d и %D будут использовать формат оформления "short" , если --decorate еще не был указан в командной строке.

Опции boolean принимают необязательное значение [=<bool-value>] . Все значения true , false , on , off и т. д. принимаются. См. подраздел "boolean" в "EXAMPLES" в git-config[1] . Если параметр boolean указан без значения, он включен.

Если вы добавите + (знак плюса) после % заполнителя, перевод строки вставляется непосредственно перед расширением тогда и только тогда, когда заполнитель расширяется до непустой строки.

Если вы добавите - (знак минус) после % заполнителя, все последовательные переводы строк, непосредственно предшествующие расширению, удаляются тогда и только тогда, когда заполнитель расширяется до пустой строки.

Если вы добавите ` ` (пробел) после % заполнителя, пробел вставляется непосредственно перед раскрытием тогда и только тогда, когда заполнитель расширяется до непустой строки.

  • tformat:

    Формат tformat: работает точно так же, как format: , за исключением того, что он предоставляет семантику "terminator" вместо семантики "separator" . Другими словами, к каждому коммиту добавляется символ завершения сообщения (обычно новая строка), а не разделитель, помещаемый между записями. Это означает, что окончательная запись однострочного формата будет корректно завершаться новой строкой, как это происходит в формате "oneline" . Например:

    $ git log -2 --pretty=формат:%h 4da45bef \
  | perl -pe '$_ .= " -- NO NEWLINE\n" unless /\n/'
4da45be
7134973 -- НЕТ НОВОЙ СТРОКИ

$ git log -2 --pretty=tformat:%h 4da45bef \
  | perl -pe '$_ .= " -- NO NEWLINE\n" unless /\n/'
4da45be
7134973

    Кроме того, любая нераспознанная строка, содержащая % , интерпретируется так, как будто перед ней стоит tformat: . Например, эти два эквивалентны:

    $ git log -2 --pretty=tformat:%h 4da45bef
$ git log -2 --pretty=%h 4da45bef

Diff formatting

По умолчанию git log не генерирует никаких результатов сравнения. Приведенные ниже параметры можно использовать для отображения изменений, внесенных каждым коммитом.

Обратите внимание: если явно не указан один из вариантов --diff-merges (включая короткие параметры -m , -c , --cc и --dd ), коммиты слияния не будут отображать различия, даже если выбран формат различий, такой как --patch , и они не будут соответствовать параметрам поиска, таким как -S . . Исключением является использование --first-parent , и в этом случае first-parent является форматом по умолчанию для коммитов слияния.

-p
-u
--patch

Сгенерируйте патч (см. Generating patch text with -p ).

-s
--no-patch

Подавить весь вывод механизма сравнения. Полезно для таких команд, как git show , которые по умолчанию показывают патч, чтобы подавить их вывод или отменить эффект таких параметров, как --patch , --stat ранее в командной строке в псевдониме.

-m

Показывать различия для коммитов слияния в формате по умолчанию. Это похоже на --diff-merges=on , за исключением того, что -m не будет выдавать никаких результатов, если также не указан -p .

-c

Создание комбинированного вывода различий для коммитов слияния. Ярлык для --diff-merges=combined -p .

--cc

Создание плотного комбинированного вывода различий для коммитов слияния. Ярлык для --diff-merges=dense-combined -p .

--dd

Создайте разницу относительно первого родителя как для слияния, так и для обычных коммитов. Ярлык для --diff-merges=first-parent -p .

--remerge-diff

Создайте выходные данные remerge-diff для коммитов слияния. Ярлык для --diff-merges=remerge -p .

--no-diff-merges

Синоним --diff-merges=off .

--diff-merges=<format>

Укажите формат различий, который будет использоваться для коммитов слияния. По умолчанию установлено значение «выключено», если не используется --first-parent , в этом случае значением по умолчанию является first-parent .

Поддерживаются следующие форматы:

off, none

Отключите вывод различий для коммитов слияния. Полезно для переопределения подразумеваемого значения.

on, m

Сделайте вывод различий для коммитов слияния, чтобы они отображались в формате по умолчанию. Формат по умолчанию можно изменить с помощью переменной конфигурации log.diffMerges , значением по умолчанию которой является separate .

first-parent, 1

Показать полную разницу относительно первого родителя. Это тот же формат, который --patch создает для коммитов без слияния.

separate

Показать полную разницу по отношению к каждому из родителей. Для каждого родителя создается отдельная запись в журнале и разница.

combined, c

Покажите различия от каждого из родительских элементов к результату слияния одновременно вместо отображения попарных различий между родительским элементом и результатом по одному. Более того, в нем перечислены только файлы, которые были изменены всеми родительскими файлами.

dense-combined, cc

Дальнейшее сжатие вывода, создаваемое --diff-merges=combined , путем исключения неинтересных блоков, содержимое которых в родительских элементах имеет только два варианта, и результат слияния выбирает один из них без изменений.

remerge, r

Повторно объединить коммиты слияния двух родителей для создания временного объекта дерева, который потенциально может содержать файлы с маркерами конфликтов и тому подобное. Затем отображается разница между этим временным деревом и фактическим коммитом слияния.

Вывод, выдаваемый при использовании этой опции, может быть изменен, как и его взаимодействие с другими опциями (если это явно не задокументировано).

--combined-all-paths

Этот флаг заставляет комбинированные различия (используемые для коммитов слияния) отображать имя файла от всех родителей. Таким образом, он действует только тогда, когда используется --diff-merges=[dense-]combined , и, вероятно, полезен только в том случае, если обнаружены изменения имени файла (т. е. когда запрошено обнаружение переименования или копирования).

Например, если вы настроили для переменной diff.algorithm значение, отличное от значения по умолчанию, и хотите использовать значение по умолчанию, вам необходимо использовать опцию --diff-algorithm=default .

--stat[=<width>[,<name-width>[,<count>]]]

Создайте диффстат. По умолчанию столько места, сколько необходимо, будет использоваться для части имени файла, а остальное — для части графика. Максимальная ширина по умолчанию равна ширине терминала или 80 столбцам, если он не подключен к терминалу, и может быть переопределена <width> . Ширину части имени файла можно ограничить, указав другую ширину <name-width> после запятой или установив diff.statNameWidth=<width> . Ширину части графика можно ограничить, используя --stat-graph-width=<width> или установив diff.statGraphWidth=<width> . Использование --stat или --stat-graph-width влияет на все команды, генерирующие график статистики, а установка diff.statNameWidth или diff.statGraphWidth не влияет на git format-patch . Указав третий параметр <count> , вы можете ограничить вывод первыми строками <count> , а затем ... , если их больше.

Эти параметры также можно установить индивидуально с помощью --stat-width=<width> , --stat-name-width=<name-width> и --stat-count=<count> .

-X[<param1,param2,…​>]
--dirstat[=<param1,param2,…​>]

Выведите распределение относительного количества изменений для каждого подкаталога. Поведение --dirstat можно настроить, передав ему список параметров, разделенных запятыми. Значения по умолчанию контролируются переменной конфигурации diff.dirstat (см. git-config[1] ). Доступны следующие параметры:

changes

Вычислите номера dirstat, подсчитав строки, которые были удалены из источника или добавлены в место назначения. При этом игнорируется количество чистых движений кода внутри файла. Другими словами, перестановка строк в файле засчитывается не так сильно, как другие изменения. Это поведение по умолчанию, когда параметр не указан.

lines

Вычислите числа dirstat, выполнив обычный анализ различий на основе строк и суммируя количество удаленных/добавленных строк. (Для двоичных файлов вместо этого считайте куски по 64 байта, поскольку в двоичных файлах нет естественного понятия строк). Это более затратное поведение --dirstat , чем поведение changes , но оно учитывает перестановку строк в файле так же, как и другие изменения. Полученный результат соответствует тому, что вы получаете от других вариантов --*stat .

files

Вычислите числа dirstat, подсчитав количество измененных файлов. Каждый измененный файл одинаково учитывается при анализе dirstat. Это самое дешевое в вычислительном отношении поведение --dirstat , поскольку ему вообще не нужно просматривать содержимое файла.

<limit>

Целочисленный параметр определяет процент отсечки (по умолчанию 3%). Каталоги, на долю которых приходится меньше этого процента изменений, не отображаются в выводе.

Пример: Следующий код будет подсчитывать измененные файлы, игнорируя каталоги с менее чем 10% от общего количества измененных файлов и накапливая счетчики дочерних каталогов в родительских каталогах: --dirstat=files,10,cumulative .

--submodule[=<format>]

Укажите, как отображаются различия в подмодулях. При указании --submodule=short используется формат short . Этот формат просто показывает имена коммитов в начале и конце диапазона. Если указан --submodule или --submodule=log , используется формат log . В этом формате перечислены коммиты в диапазоне, как в git-submodule[1] и summary . Если указан --submodule=diff , используется формат diff . Этот формат показывает встроенную разницу изменений в содержимом подмодуля между диапазоном фиксации. По умолчанию используется формат diff.submodule или short , если параметр конфигурации не установлен.

--color[=<when>]

Показать цветные различия. --color (т.е. без =<when> ) аналогичен --color=always . <when> может быть одним из always , never или auto .

--no-color

Отключить цветной дифф. Это то же самое, что и --color=never .

--color-moved[=<mode>]

Перемещенные строки кода окрашиваются по-другому. По умолчанию <mode> имеет значение no , если опция не указана, и zebra , если указана опция без режима. Режим должен быть одним из:

no

Перемещенные строки не выделяются.

default

Является синонимом zebra . В будущем этот режим может измениться на более разумный.

plain

Любая линия, добавленная в одном месте и удаленная в другом, будет окрашена в цвет color.diff.newMoved . Аналогично color.diff.oldMoved будет использоваться для удаленных строк, которые добавляются где-то еще в файле diff. Этот режим улавливает любую перемещенную строку, но при просмотре он не очень полезен для определения того, был ли блок кода перемещен без перестановки.

blocks

Блоки перемещенного текста длиной не менее 20 буквенно-цифровых символов обнаруживаются жадно. Обнаруженные блоки окрашиваются в цвет color.diff.{old,new}Moved . Соседние блоки невозможно отличить друг от друга.

zebra

Блоки перемещенного текста обнаруживаются как в режиме blocks . Блоки окрашены либо в цвет color.diff.{old,new}Moved , либо в цвет color.diff.{old,new}MovedAlternative . Изменение двух цветов указывает на то, что был обнаружен новый блок.

dimmed-zebra

Аналогичен zebra , но выполняется дополнительное затемнение неинтересных частей перемещаемого кода. Интересными считаются граничащие линии двух соседних блоков, остальное неинтересно. dimmed_zebra — устаревший синоним.

--no-color-moved

Отключите обнаружение движения. Это можно использовать для переопределения настроек конфигурации. Это то же самое, что и --color-moved=no .

--color-moved-ws=<modes>

Это определяет, как пробелы игнорируются при обнаружении перемещения для --color-moved . Эти режимы можно указать в виде списка, разделенного запятыми:

no

Не игнорируйте пробелы при обнаружении перемещения.

ignore-space-at-eol

Игнорировать изменения в пробелах в конце EOL.

ignore-space-change

Игнорировать изменения в количестве пробелов. При этом пробелы в конце строки игнорируются, а все остальные последовательности из одного или нескольких символов пробелов считаются эквивалентными.

ignore-all-space

Игнорируйте пробелы при сравнении строк. При этом различия игнорируются, даже если в одной строке есть пробелы, а в другой их нет.

allow-indentation-change

Первоначально игнорируйте любые пробелы при обнаружении перемещения, затем группируйте перемещенные блоки кода в блок только в том случае, если изменение пробелов одинаково для каждой строки. Это несовместимо с другими режимами.

--no-color-moved-ws

Не игнорируйте пробелы при обнаружении перемещения. Это можно использовать для переопределения настроек конфигурации. Это то же самое, что и --color-moved-ws=no .

--word-diff[=<mode>]

Покажите разницу слов, используя <mode> для разделения измененных слов. По умолчанию слова разделяются пробелами; см. --word-diff-regex ниже. По умолчанию <mode> имеет значение plain и должно быть одним из:

color

Выделите измененные слова, используя только цвета. Подразумевается --color .

plain

Показывать слова как [-removed-] и {+added+} . Не предпринимает попыток избежать разделителей, если они появляются во входных данных, поэтому выходные данные могут быть неоднозначными.

porcelain

Используйте специальный построчный формат, предназначенный для использования в сценариях. Добавленные/удаленные/неизмененные прогоны печатаются в обычном унифицированном формате различий, начиная с символа + / - /`` в начале строки и продолжаясь до конца строки. Новые строки во входных данных обозначаются тильдой ~ в отдельной строке.

none

Снова отключите разницу слов.

Обратите внимание, что несмотря на название первого режима, цвет используется для выделения измененных частей во всех режимах, если они включены.

--word-diff-regex=<regex>

Используйте <regex>, чтобы определить, что такое слово, вместо того, чтобы считать словом ряды непробельных символов. Также подразумевается --word-diff , если он еще не был включен.

Каждое непересекающееся совпадение <regex> считается словом. Все, что находится между этими совпадениями, считается пробелом и игнорируется (!) в целях поиска различий. Возможно, вы захотите добавить |[^[:space:]] к своему регулярному выражению, чтобы убедиться, что оно соответствует всем символам, не содержащим пробелов. Соответствие, содержащее новую строку, автоматически обрезается(!) на новой строке.

Например, --word-diff-regex=. будет трактовать каждый символ как слово и, соответственно, посимвольно показывать различия.

Регулярное выражение также можно установить с помощью драйвера различий или параметра конфигурации, см. gitattributes[5] или git-config[1] . Его явное указание переопределяет любой драйвер сравнения или настройку конфигурации. Драйверы различий переопределяют параметры конфигурации.

--color-words[=<regex>]

Эквивалент --word-diff=color плюс (если было указано регулярное выражение) --word-diff-regex=<regex> .

--no-renames

Отключите обнаружение переименования, даже если в файле конфигурации указано значение по умолчанию.

--[no-]rename-empty

Использовать ли пустые BLOB-объекты в качестве источника переименования.

--check

Предупреждать, если изменения приводят к появлению маркеров конфликта или ошибок в виде пробелов. То, что считается ошибкой пробелов, контролируется конфигурацией core.whitespace . По умолчанию конечные пробелы (включая строки, состоящие исключительно из пробелов) и символ пробела, за которым сразу следует символ табуляции внутри начального отступа строки, считаются ошибками пробелов. Выход с ненулевым статусом, если обнаружены проблемы. Несовместимо с --exit-code.

--ws-error-highlight=<kind>

Выделите ошибки пробелов в строках context , old или new файла diff. Несколько значений разделяются запятой, none сбрасывает предыдущие значения, default сбрасывает список на new , а all — это сокращение от old,new,context . Если эта опция не задана и переменная конфигурации diff.wsErrorHighlight не установлена, выделяются только ошибки пробелов в строках new . Ошибки пробелов окрашены color.diff.whitespace .

--full-index

Вместо первых нескольких символов отображайте полные имена объектов BLOB-объектов до и после изображения в строке "index" при создании выходных данных формата исправления.

--binary

--abbrev[=<n>]

-B[<n>][/<m>]
--break-rewrites[=[<n>][/<m>]]

Разбейте полные изменения перезаписи на пары удаления и создания. Это служит двум целям:

Это влияет на то, как изменение, которое представляет собой полную перезапись файла, происходит не как серия удалений и вставок, смешанных с очень немногими строками, которые текстуально совпадают с контекстом, а как однократное удаление всего старого с последующим однократная вставка всего нового, и число m управляет этим аспектом опции -B (по умолчанию 60%). -B/70% указывает, что в результате должно оставаться менее 30% оригинала, чтобы Git считал его полной перезаписью (т. е. в противном случае результирующий патч будет представлять собой серию удалений и вставок, смешанных вместе с контекстными строками).

При использовании с -M полностью переписанный файл также считается источником переименования (обычно -M считает источником переименования только исчезнувший файл), а число n управляет этим аспектом опции -B. (по умолчанию 50%). -B20% указывает, что изменение с добавлением и удалением, составляющее 20% или более от размера файла, может быть выбрано в качестве возможного источника переименования в другой файл.

-M[<n>]
--find-renames[=<n>]

При создании различий обнаруживайте переименования и сообщайте о них для каждого коммита. Информацию о переименовании файлов при просмотре истории см. в --follow . Если указан n , это порог индекса сходства (т.е. количество добавлений/удалений по сравнению с размером файла). Например, -M90% означает, что Git должен рассматривать пару удаления/добавления как переименование, если более 90% файла не изменилось. Без знака % число следует читать как дробь с десятичной точкой перед ней. Т.е. -M5 становится равным 0,5 и, таким образом, совпадает с -M50% . Аналогично, -M05 аналогичен -M5% . Чтобы ограничить обнаружение точными переименованиями, используйте -M100% . Индекс сходства по умолчанию составляет 50%.

-C[<n>]
--find-copies[=<n>]

Обнаружение копий, а также переименований. См. также --find-copies-harder . Если указан n , он имеет то же значение, что и -M<n> .

--find-copies-harder

По соображениям производительности опция -C по умолчанию находит копии только в том случае, если исходный файл копии был изменен в том же наборе изменений. Этот флаг заставляет команду проверять неизмененные файлы как кандидатов на источник копирования. Это очень дорогостоящая операция для больших проектов, поэтому используйте ее с осторожностью. Предоставление более одной опции -C имеет тот же эффект.

-D
--irreversible-delete

При удалении пропустите прообраз, т.е. печатайте только заголовок, но не разницу между прообразом и /dev/null . Полученный патч не предназначен для применения с patch или git apply ; это исключительно для людей, которые хотят просто сконцентрироваться на просмотре текста после внесения изменений. Кроме того, в выводе явно недостаточно информации, чтобы применить такой патч в обратном порядке, даже вручную, отсюда и название опции.

При использовании вместе с -B опустите также прообраз в части удаления пары удаление/создание.

-l<num>

Опции -M и -C включают в себя некоторые предварительные шаги, которые позволяют дешево обнаружить подмножества переименований/копий, за которыми следует исчерпывающая запасная часть, которая сравнивает все оставшиеся непарные адресаты со всеми соответствующими источниками. (Для переименований актуальны только оставшиеся непарные источники; для копий актуальны все исходные источники.) Для N источников и мест назначения эта исчерпывающая проверка равна O(N^2). Этот параметр предотвращает запуск исчерпывающей части обнаружения переименования/копирования, если количество задействованных файлов-источников/назначений превышает указанное число. По умолчанию — diff.renameLimit. Обратите внимание, что значение 0 считается неограниченным.

--diff-filter=[(A|C|D|M|R|T|U|X|B)…​[*]]

Выбирайте только те файлы, которые добавлены ( A ), скопированы ( C ), удалены ( D ), изменены ( M ), переименованы ( R ), имеют измененный тип (т. е. обычный файл, символическая ссылка, подмодуль и т. д.) ( T ), не объединены ( U ), неизвестны ( X ) или соединение разорвано ( B ). Можно использовать любую комбинацию символов фильтра (в том числе отсутствие). Когда к комбинации добавляется * (все или ничего), все пути выбираются, если есть какой-либо файл, соответствующий другим критериям сравнения; если нет файла, соответствующего другим критериям, ничего не выбрано.

Кроме того, эти прописные буквы можно уменьшить, чтобы исключить их. Например, --diff-filter=ad исключает добавленные и удаленные пути.

Обратите внимание, что не все различия могут включать все типы. Например, скопированные и переименованные записи не могут появиться, если обнаружение этих типов отключено.

-S<string>

Ищите различия, которые изменяют количество вхождений указанной строки (т. е. добавление/удаление) в файле. Предназначен для использования сценаристами.

Это полезно, когда вы ищете точный блок кода (например, структуру) и хотите узнать историю этого блока с момента его первого появления: используйте эту функцию итеративно, чтобы вернуть интересный блок в прообразе обратно в -S и продолжайте, пока не получите самую первую версию блока.

Бинарные файлы также ищутся.

-G<regex>

Найдите различия, текст исправления которых содержит добавленные/удаленные строки, соответствующие <regex>.

Чтобы проиллюстрировать разницу между -S<regex> --pickaxe-regex и -G<regex> , рассмотрим фиксацию со следующей разницей в одном и том же файле:

+    return frotz(nitfol, two->ptr, 1, 0);
...
-    hit = frotz(nitfol, mf2.ptr, 1, 0);

Хотя git log -G"frotz\(nitfol" покажет эту фиксацию, git log -S"frotz\(nitfol" --pickaxe-regex — нет (поскольку количество вхождений этой строки не изменилось).

Если не поставляется --text , исправления двоичных файлов без фильтра textconv будут игнорироваться.

Дополнительную информацию см. в записи pickaxe в gitdiffcore[7] .

--find-object=<object-id>

Ищите различия, которые изменяют количество вхождений указанного объекта. Подобно -S , аргумент отличается тем, что он ищет не конкретную строку, а определенный идентификатор объекта.

Объект может быть большим двоичным объектом или фиксацией подмодуля. Это подразумевает наличие опции -t в git-log для поиска деревьев.

--pickaxe-all

Когда -S или -G обнаружит изменение, отобразите все изменения в этом наборе изменений, а не только файлы, содержащие изменение в <string>.

--pickaxe-regex

Считайте <строку>, переданную -S , как расширенное регулярное выражение POSIX для соответствия.

-O<orderfile>

Управляйте порядком появления файлов в выходных данных. Это переопределяет переменную конфигурации diff.orderFile (см. git-config[1] ). Чтобы отменить diff.orderFile , используйте -O/dev/null .

Порядок вывода определяется порядком шаблонов glob в <orderfile>. Все файлы с путями, соответствующими первому шаблону, выводятся первыми, все файлы с путями, соответствующими второму шаблону (но не первому), выводятся затем и так далее. Все файлы, имена путей которых не соответствуют никакому шаблону, выводятся последними, как если бы в конце файла был неявный шаблон соответствия всем. Если несколько имен путей имеют одинаковый ранг (они соответствуют одному и тому же шаблону, но не более ранним шаблонам), их порядок вывода относительно друг друга является обычным порядком.

<orderfile> анализируется следующим образом:

  • Пустые строки игнорируются, поэтому их можно использовать в качестве разделителей для удобства чтения.

  • Строки, начинающиеся с хеша ( " # " ), игнорируются, поэтому их можно использовать для комментариев. Добавьте обратную косую черту ( " \ " ) в начало шаблона, если он начинается с решетки.

  • Каждая вторая строка содержит один шаблон.

Шаблоны имеют тот же синтаксис и семантику, что и шаблоны, используемые для fnmatch(3) без флага FNM_PATHNAME, за исключением того, что путь также соответствует шаблону, если удаление любого количества окончательных компонентов пути соответствует шаблону. Например, шаблон " foo*bar " соответствует " fooasdfbar " и " foo/bar/baz/asdf " , но не соответствует " foobarx " .

--skip-to=<file>
--rotate-to=<file>

Удалите файлы перед указанным <file> из вывода (т. е. skip to ) или переместите их в конец вывода (т. е. rotate to ). Эти параметры были придуманы в первую очередь для использования команды git difftool и в противном случае могут оказаться бесполезными.

-R

Поменяйте местами два входа; то есть показывать различия между индексом или файлом на диске и содержимым дерева.

--relative[=<path>]
--no-relative

При запуске из подкаталога проекта с помощью этой опции можно указать исключить изменения за пределами каталога и показать пути относительно него. Если вы не находитесь в подкаталоге (например, в пустом репозитории), вы можете указать, относительно какого подкаталога следует выполнять выходные данные, указав <path> в качестве аргумента. --no-relative можно использовать для отмены как опции конфигурации diff.relative , так и предыдущей версии --relative .

-a
--text

Рассматривать все файлы как текст.

--ignore-cr-at-eol

При сравнении игнорируйте возврат каретки в конце строки.

--ignore-space-at-eol

Игнорировать изменения в пробелах в конце EOL.

-b
--ignore-space-change

Игнорировать изменения в количестве пробелов. При этом пробелы в конце строки игнорируются, а все остальные последовательности из одного или нескольких символов пробелов считаются эквивалентными.

-w
--ignore-all-space

Игнорируйте пробелы при сравнении строк. При этом различия игнорируются, даже если в одной строке есть пробелы, а в другой их нет.

--ignore-blank-lines

Игнорировать изменения, все строки которых пусты.

-I<regex>
--ignore-matching-lines=<regex>

Игнорировать изменения, все строки которых соответствуют <regex>. Эту опцию можно указать более одного раза.

--inter-hunk-context=<lines>

Показать контекст между разными фрагментами до указанного количества строк, тем самым объединяя фрагменты, расположенные близко друг к другу. По умолчанию diff.interHunkContext или 0, если опция конфигурации не установлена.

-W
--function-context

Показывать всю функцию в виде контекстных строк для каждого изменения. Имена функций определяются так же, как git diff обрабатывает заголовки фрагментов патчей (см. Defining a custom hunk-header в gitattributes[5] ).

--ext-diff

Разрешить выполнение внешнего помощника сравнения. Если вы установили внешний драйвер различий с gitattributes[5] , вам нужно использовать эту опцию с git-log[1] и его друзьями.

--no-ext-diff

Запретить внешние драйверы различий.

--textconv
--no-textconv

Разрешить (или запретить) запуск внешних фильтров преобразования текста при сравнении двоичных файлов. Подробности см. в gitattributes[5] . Поскольку фильтры textconv обычно представляют собой одностороннее преобразование, полученный результат пригоден для восприятия человеком, но не может быть применен. По этой причине фильтры textconv включены по умолчанию только для git-diff[1] и git-log[1] , но не для git-format-patch[1] или команд diff сантехники.

--ignore-submodules[=<when>]

Игнорировать изменения в подмодулях при генерации различий. <When> может быть "none" , "untracked" , "dirty" или "all" , который используется по умолчанию. При использовании "none" субмодуль будет считаться измененным, если он либо содержит неотслеживаемые или измененные файлы, либо его HEAD отличается от фиксации, записанной в суперпроекте, и может использоваться для переопределения любых настроек параметра ignore в git-config[1] или gitmodules[5] . При использовании "untracked" субмодули не считаются «грязными», если они содержат только неотслеживаемый контент (но они все равно сканируются на предмет измененного контента). Использование "dirty" игнорирует все изменения в рабочем дереве подмодулей, отображаются только изменения коммитов, хранящихся в суперпроекте (так было до версии 1.7.0). Использование "all" скрывает все изменения в субмодулях.

--src-prefix=<prefix>

Показывать данный префикс источника вместо "a/" .

--dst-prefix=<prefix>

Показывать указанный префикс пункта назначения вместо "b/" .

--no-prefix

Не показывать префикс источника или назначения.

--default-prefix

Используйте префиксы источника и назначения по умолчанию ( "a/" и "b/" ). Обычно это значение по умолчанию, но его можно использовать для переопределения конфигурации, например diff.noprefix .

--line-prefix=<prefix>

Добавьте дополнительный префикс к каждой строке вывода.

--ita-invisible-in-index

По умолчанию записи, добавленные "git add -N" , отображаются как существующий пустой файл в "git diff" и новый файл в "git diff --cached" . Эта опция делает запись видимой как новый файл в "git diff" и несуществующей в "git diff --cached" . Эту опцию можно вернуть с помощью --ita-visible-in-index . Оба варианта являются экспериментальными и могут быть удалены в будущем.

Более подробное объяснение этих распространенных опций см. также в gitdiffcore[7] .

Генерация текста патча с помощью -p

Запуск git-diff[1] , git-log[1] , git-show[1] , git-diff-index[1] , git-diff-tree[1] или git-diff-files[1] с опцией -p создает текст исправления. Вы можете настроить создание текста патча с помощью переменных среды GIT_EXTERNAL_DIFF и GIT_DIFF_OPTS (см. git[1] ), а также атрибута diff (см. gitattributes[5] ).

Результат использования опции -p немного отличается от традиционного формата diff:

  1. Ему предшествует заголовок "git diff" , который выглядит следующим образом:

    diff --git a/file1 b/file2

    Имена файлов a/ и b/ одинаковы, если не требуется переименование/копирование. В частности, даже при создании или удалении вместо имен файлов a/ или b/ используется /dev/null .

    При переименовании/копировании file1 и file2 показывают имя исходного файла переименования/копирования и имя файла, созданного в результате переименования/копирования, соответственно.

  2. За ним следует одна или несколько расширенных строк заголовка:

    old mode <mode>
new mode <mode>
deleted file mode <mode>
new file mode <mode>
copy from <path>
copy to <path>
rename from <path>
rename to <path>
similarity index <number>
dissimilarity index <number>
index <hash>..<hash> <mode>

    Режимы файлов печатаются как 6-значные восьмеричные числа, включая тип файла и биты разрешения файла.

    Имена путей в расширенных заголовках не включают префиксы a/ и b/ .

    Индекс сходства — это процент неизмененных строк, а индекс несходства — процент измененных строк. Это округленное целое число, за которым следует знак процента. Таким образом, значение индекса сходства 100% зарезервировано для двух одинаковых файлов, а 100%-ное различие означает, что ни одна строка из старого файла не попала в новый.

    Индексная строка включает имена объектов больших двоичных объектов до и после изменения. <mode> включается, если режим файла не изменяется; в противном случае отдельные строки обозначают старый и новый режим.

  3. Имена путей с символами "unusual" заключаются в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ).

  4. Все файлы file1 в выходных данных относятся к файлам до фиксации, а все файлы file2 относятся к файлам после фиксации. Неправильно применять каждое изменение к каждому файлу последовательно. Например, этот патч поменяет местами a и b:

    diff --git a/a b/b
rename from a
rename to b
diff --git a/b b/a
rename from b
rename to a

  5. В заголовках ханков упоминается имя функции, к которой применяется ханк. См. "Defining a custom hunk-header" в gitattributes[5] для получения подробной информации о том, как адаптировать это к конкретным языкам.

Комбинированный формат различий

Любая команда, генерирующая различия, может использовать опцию -c или --cc для создания combined diff при отображении слияния. Это формат по умолчанию при отображении слияний с git-diff[1] или git-show[1] . Также обратите внимание, что вы можете указать подходящую опцию --diff-merges для любой из этих команд, чтобы принудительно генерировать различия в определенном формате.

Формат "combined diff" выглядит следующим образом:

diff --combined describe.c
index fabadb8,cc95eb0..4866510
--- a/describe.c
+++ b/describe.c
@@@ -98,20 -98,12 +98,20 @@@
        return (a_date > b_date) ? -1 : (a_date == b_date) ? 0 : 1;
  }

- static void describe(char *arg)
 -static void describe(struct commit *cmit, int last_one)
++static void describe(char *arg, int last_one)
  {
 +        unsigned char sha1[20];
 +        struct commit *cmit;
        struct commit_list *list;
        static int initialized = 0;
        struct commit_name *n;

 +        if (get_sha1(arg, sha1) < 0)
 +                usage(describe_usage);
 +        cmit = lookup_commit_reference(sha1);
 +        if (!cmit)
 +                usage(describe_usage);
 +
        if (!initialized) {
                initialized = 1;
                for_each_ref(get_name);

  1. Ему предшествует заголовок "git diff" , который выглядит следующим образом (при использовании опции -c ):

    diff --combined file

    или вот так (при использовании опции --cc ):

    diff --cc file

  2. За ним следует одна или несколько строк расширенного заголовка (в этом примере показано слияние с двумя родительскими строками):

    index <hash>,<hash>..<hash>
mode <mode>,<mode>..<mode>
new file mode <mode>
deleted file mode <mode>,<mode>

    Строка mode <mode>,<mode>..<mode> появляется только в том случае, если хотя бы один из <mode> отличается от остальных. Расширенные заголовки с информацией об обнаруженном перемещении контента (обнаружение переименований и копирования) предназначены для работы с различиями двух <tree-ish> и не используются в комбинированном формате различий.

  3. За ним следует двухстрочный заголовок «из файла/в файл»:

    --- a/file
+++ b/file

    Подобно двухстрочному заголовку традиционного формата различий unified , /dev/null используется для обозначения созданных или удаленных файлов.

    Однако если указана опция --combined-all-paths, вместо двухстрочного заголовка «из файла/в файл» вы получите заголовок «из файла/в файл» из N+1 строки, где N — номер. родителей в коммите слияния:

    --- a/file
--- a/file
--- a/file
+++ b/file

    Этот расширенный формат может быть полезен, если активно обнаружение переименования или копирования, чтобы вы могли видеть исходное имя файла у разных родителей.

  4. Формат заголовка чанка изменен, чтобы люди не могли случайно передать его в patch -p1 . Комбинированный формат различий был создан для проверки изменений коммитов слияния и не предназначался для применения. Изменение аналогично изменению расширенного заголовка index :

    @@@ <from-file-range> <from-file-range> <to-file-range> @@@

    В заголовке чанка для комбинированного формата различий есть (количество родительских элементов + 1) символы @ .

В отличие от традиционного формата различий unified , который показывает два файла A и B с одним столбцом, имеющим - (минус — появляется в A, но удаляется в B), + (плюс — отсутствует в A, но добавляется в B) или " " (пробел — неизмененный) префикс, этот формат сравнивает два или более файлов file1, file2,… с одним файлом X и показывает, чем X отличается от каждого из файлов N. К выходной строке добавляется один столбец для каждого файла fileN, чтобы отметить, чем строка X отличается от нее.

Символ - в столбце N означает, что строка появляется в файле N, но не отображается в результате. Символ + в столбце N означает, что строка появляется в результате, а в файле N этой строки нет (другими словами, строка была добавлена ​​с точки зрения этого родителя).

В приведенном выше примере сигнатура функции была изменена в обоих файлах (следовательно, два удаления - из файла1 и файла2 плюс ++ , чтобы означать, что одна добавленная строка не отображается ни в файле1, ни в файле2). Кроме того, восемь других строк такие же, как в файле 1, но не появляются в файле 2 (следовательно, с префиксом + ).

Когда отображается git diff-tree -c , он сравнивает родителей коммита слияния с результатом слияния (т. е. file1..fileN являются родителями). Когда отображается git diff-files -c , он сравнивает двух неразрешенных родительских элементов слияния с файлом рабочего дерева (т. е. файл1 — это этап 2, он же "our version" , файл2 — это этап 3, он же "their version" ).

Examples

git log --no-merges

Показать всю историю коммитов, но пропустить любые слияния.

git log v2.6.12.. include/scsi drivers/scsi

Показать все коммиты, начиная с версии v2.6.12 , которые изменили любой файл в подкаталогах include/scsi или drivers/scsi .

git log --since="2 weeks ago" -- gitk

Показать изменения за последние две недели в файле gitk . -- необходим, чтобы не путать с веткой gitk .

git log --name-status release..test

Покажите коммиты, которые находятся в ветке "test" , но еще не в ветке "release" , а также список путей, которые изменяет каждый коммит.

git log --follow builtin/rev-list.c

Показывает коммиты, изменившие builtin/rev-list.c , включая те коммиты, которые произошли до того, как файлу было присвоено его нынешнее имя.

git log --branches --not --remotes=origin

Показывает все коммиты, которые есть в любой из локальных веток, но не в одной из веток удаленного отслеживания для origin (то, что у вас есть, этого источника нет).

git log master --not --remotes=*/master

Показывает все коммиты, которые находятся в локальном главном репозитории, но отсутствуют в главных ветках удаленного репозитория.

git log -p -m --first-parent

Показывает историю, включая различия в изменениях, но только с точки зрения «основной ветки», пропуская коммиты, поступающие из объединенных ветвей, и показывая полные различия изменений, внесенных в результате слияний. Это имеет смысл только при соблюдении строгой политики объединения всех тематических веток при сохранении одной ветки интеграции.

git log -L '/int main/',/^}/:main.c

Показывает, как функция main() в файле main.c развивалась с течением времени.

git log -3

Ограничивает количество отображаемых коммитов до 3.

Discussion

Git в некоторой степени не зависит от кодировки символов.

  • Содержимое объектов blob представляет собой неинтерпретированные последовательности байтов. На уровне ядра нет трансляции кодировки.

  • Имена путей кодируются в форме нормализации UTF-8 C. Это относится к древовидным объектам, индексному файлу, именам ссылок, а также именам путей в аргументах командной строки, переменных среды и файлах конфигурации ( .git/config (см. git-config[1] ), gitignore[5] , gitattributes[5] ). и gitmodules[5] ).

    Обратите внимание, что Git на уровне ядра обрабатывает имена путей просто как последовательности байтов, отличных от NUL, преобразования кодировок имен путей не выполняются (кроме Mac и Windows). Таким образом, использование имен путей, отличных от ASCII, в большинстве случаев будет работать даже на платформах и файловых системах, использующих устаревшие расширенные кодировки ASCII. Однако репозитории, созданные в таких системах, не будут корректно работать в системах на базе UTF-8 (например, Linux, Mac, Windows) и наоборот. Кроме того, многие инструменты на основе Git просто предполагают, что имена путей имеют формат UTF-8, и не могут правильно отображать другие кодировки.

  • Сообщения журнала фиксации обычно кодируются в UTF-8, но поддерживаются и другие расширенные кодировки ASCII. Сюда входят ISO-8859-x, CP125x и многие другие, а также многобайтовые кодировки not UTF-16/32, EBCDIC и CJK (GBK, Shift-JIS, Big5, EUC-x, CP9xx и т. д.).

Хотя мы рекомендуем, чтобы сообщения журнала фиксации были закодированы в UTF-8, ядро ​​и Git Porcelain не предназначены для принудительного использования UTF-8 в проектах. Если всем участникам конкретного проекта удобнее использовать устаревшие кодировки, Git не запрещает этого. Однако есть несколько вещей, которые следует иметь в виду.

  1. git commit и git commit-tree выдают предупреждение, если переданное ему сообщение журнала фиксации не похоже на допустимую строку UTF-8, если только вы явно не укажете, что ваш проект использует устаревшую кодировку. Это можно сказать так: i18n.commitEncoding в файле .git/config , например:

    [i18n]
        commitEncoding = ISO-8859-1

    Объекты фиксации, созданные с указанным выше параметром, записывают значение i18n.commitEncoding в свой заголовок encoding . Это сделано для того, чтобы помочь другим людям, которые посмотрят на них позже. Отсутствие этого заголовка означает, что сообщение журнала фиксации закодировано в UTF-8.

  2. git log , git show , git blame и его друзья просматривают заголовок encoding объекта фиксации и пытаются перекодировать сообщение журнала в UTF-8, если не указано иное. Вы можете указать желаемую выходную кодировку с помощью i18n.logOutputEncoding в файле .git/config , например:

    [i18n]
        logOutputEncoding = ISO-8859-1

    Если у вас нет этой переменной конфигурации, вместо нее используется значение i18n.commitEncoding .

Обратите внимание, что мы намеренно решили не перекодировать сообщение журнала фиксации при выполнении фиксации, чтобы принудительно использовать UTF-8 на уровне объекта фиксации, поскольку перекодирование в UTF-8 не обязательно является обратимой операцией.

Configuration

См. git-config[1] для основных переменных и git-diff[1] для настроек, связанных с генерацией различий.

format.pretty

По умолчанию для опции --format . (См. Pretty Formats выше.) По умолчанию — medium .

i18n.logOutputEncoding

Кодировка, используемая при отображении журналов. (См. Discussion выше.) По умолчанию используется значение i18n.commitEncoding , если оно установлено, и UTF-8 в противном случае.

Все, что находится выше этой строки в этом разделе, не включено в документацию git-config[1] . Следующее содержимое такое же, как и найденное там:

log.abbrevCommit

Если это правда, git-log[1] , git-show[1] и git-whatchanged[1] предполагают --abbrev-commit . Вы можете переопределить эту опцию с помощью --no-abbrev-commit .

log.date

Установите режим даты и времени по умолчанию для команды log . Установка значения log.date аналогична использованию опции --date git log . Подробности см. в git-log[1] .

Если установлен формат "auto:foo" и используется пейджер, в качестве формата даты будет использоваться формат "foo" . В противном случае будет использоваться "default" .

log.decorate

Распечатайте имена ссылок всех коммитов, отображаемых командой log. Если указан short , префиксы имен refs/heads/ , refs/tags/ и refs/remotes/ не будут напечатаны. Если указан full , будет напечатано полное имя ссылки (включая префикс). Если указан auto , то если выход идет на терминал, имена ссылок отображаются так, как если бы были заданы short , в противном случае имена ссылок не отображаются. Это то же самое, что вариант --decorate для git log .

log.initialDecorationSet

По умолчанию git log отображает украшения только для определенных известных пространств имен ссылок. Если указан all , то все ссылки будут отображаться как украшения.

log.excludeDecoration

Исключить указанные узоры из украшений бревна. Это похоже на параметр командной строки --decorate-refs-exclude , но параметр config может быть переопределен параметром --decorate-refs .

log.diffMerges

Установите формат различий, который будет использоваться, если указан --diff-merges=on , подробности см. в --diff-merges в git-log[1] . По умолчанию separate .

log.follow

Если true , git log будет действовать так, как если бы была использована опция --follow , когда указан одиночный <path>. Он имеет те же ограничения, что и --follow , то есть его нельзя использовать для отслеживания нескольких файлов и он плохо работает с нелинейной историей.

log.graphColors

Список цветов, разделенных запятыми, которые можно использовать для рисования линий истории в git log --graph .

log.showRoot

Если это правда, первоначальная фиксация будет отображаться как большое событие создания. Это эквивалентно сравнению с пустым деревом. Такие инструменты, как git-log[1] или git-whatchanged[1] , которые обычно скрывают корневой коммит, теперь его покажут. True по умолчанию.

log.showSignature

Если это правда, git-log[1] , git-show[1] и git-whatchanged[1] предполагают --show-signature .

log.mailmap

Если это правда, git-log[1] , git-show[1] и git-whatchanged[1] предполагают --use-mailmap , в противном случае предполагают --no-use-mailmap . True по умолчанию.

notes.mergeStrategy

Какую стратегию слияния выбрать по умолчанию при разрешении конфликтов заметок. Должен быть один из manual , ours , theirs , union или cat_sort_uniq . По умолчанию manual . Дополнительную информацию о каждой стратегии см. в разделе "NOTES MERGE STRATEGIES" статьи git-notes[1] .

Этот параметр можно переопределить, передав параметр --strategy в git-notes[1] .

notes.<name>.mergeStrategy

Какую стратегию слияния выбрать при объединении заметок в refs/notes/<name>. Это отменяет более общий "notes.mergeStrategy" . Дополнительную информацию о доступных стратегиях см. в разделе "NOTES MERGE STRATEGIES" в git-notes[1] .

notes.displayRef

Какую ссылку (или ссылки, если она является общей или указана более одного раза), в дополнение к значению по умолчанию, установленному core.notesRef или GIT_NOTES_REF , для чтения примечаний при отображении сообщений о фиксации с помощью команд семейства git log .

Этот параметр можно переопределить с помощью переменной среды GIT_NOTES_DISPLAY_REF , которая должна представлять собой список ссылок или glob, разделенных двоеточиями.

Предупреждение будет выдано для ссылок, которые не существуют, но glob, который не соответствует ни одной ссылке, игнорируется.

Этот параметр можно отключить с помощью опции --no-notes семейства команд git log или опции --notes=<ref> , принимаемой этими командами.

Действующее значение "core.notesRef" (возможно, переопределяемое GIT_NOTES_REF) также неявно добавляется в список отображаемых ссылок.

notes.rewrite.<command>

При перезаписи фиксации с помощью <command> (в настоящее время amend или rebase ), если эта переменная равна false , git не будет копировать примечания из оригинала в переписанную фиксацию. По умолчанию true . См. также " notes.rewriteRef " ниже.

Этот параметр можно переопределить с помощью переменной среды GIT_NOTES_REWRITE_REF , которая должна представлять собой список ссылок или glob, разделенных двоеточиями.

notes.rewriteMode

При копировании примечаний во время перезаписи (см. параметр "notes.rewrite.<command>" ) определяет, что делать, если в целевом коммите уже есть примечание. Должен быть один из overwrite , concatenate , cat_sort_uniq или ignore . По умолчанию concatenate .

Этот параметр можно переопределить с помощью переменной среды GIT_NOTES_REWRITE_MODE .

notes.rewriteRef

При копировании заметок во время перезаписи указывает (полную) ссылку, чьи заметки следует скопировать. Может быть glob, и в этом случае будут скопированы примечания во всех соответствующих ссылках. Вы также можете указать эту конфигурацию несколько раз.

Не имеет значения по умолчанию; вы должны настроить эту переменную, чтобы включить перезапись заметок. Установите значение refs/notes/commits , чтобы включить перезапись примечаний к фиксации по умолчанию.

Может быть переопределено с помощью переменной среды GIT_NOTES_REWRITE_REF . См. notes.rewrite.<command> выше для дальнейшего описания его формата.
