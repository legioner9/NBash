gitcli

Name

gitcli — Git интерфейс командной строки и соглашения

Synopsis

gitcli

Description

В этом руководстве описываются соглашения, используемые в интерфейсе командной строки Git .

Многие команды принимают в качестве аргументов версии (чаще всего "commits" , но иногда "tree-ish" , в зависимости от контекста и команды) и пути. Вот правила:

  • Сначала идут опции, а затем аргументы. Подкоманда может принимать пунктирные параметры (которые могут иметь собственные аргументы, например "--max-parents 2" ) и аргументы. Вы ДОЛЖНЫ сначала указать пунктирные варианты, а затем аргументы. Некоторые команды могут принимать пунктирные параметры после того, как вы уже указали аргументы, не являющиеся параметрами (что может сделать команду неоднозначной), но вам не следует полагаться на них (потому что в конечном итоге мы можем найти способ исправить эту неоднозначность, применяя правило "options then args" ).

  • Сначала идут изменения, а затем пути. Например, в git diff v1.0 v2.0 arch/x86 include/asm-x86 v1.0 и v2.0 — это версии, а arch/x86 и include/asm-x86 — это пути.

  • Если аргумент может быть ошибочно воспринят как ревизия или путь, их можно устранить, поместив между ними -- . Например, git diff -- HEAD — это "I have a file called HEAD in my work tree. Please show changes between the version I staged in the index and what I have in the work tree for that file" , а не "show the difference between the HEAD commit and the work tree as a whole" . Вы можете сказать git diff HEAD -- , чтобы попросить последнее.

  • Не устраняя неоднозначность -- , Git делает разумное предположение, но выдает ошибку и просит вас устранить неоднозначность, если она неоднозначна. Например, если в вашем рабочем дереве есть файл с именем HEAD, git diff HEAD неоднозначен, и вам нужно сказать либо git diff HEAD -- , либо git diff -- HEAD , чтобы устранить неоднозначность.

  • Поскольку -- устраняет неоднозначность версий и путей в некоторых командах, его нельзя использовать для этих команд для разделения параметров и версий. Для этого можно использовать --end-of-options (он также работает для команд, которые не различают ревизии в путях, и в этом случае это просто псевдоним для -- ).

    При написании сценария, который, как ожидается, будет обрабатывать случайный пользовательский ввод, рекомендуется четко указывать, какие аргументы являются какими, помещая устраняющую неоднозначность -- в соответствующих местах.

  • Многие команды допускают использование подстановочных знаков в путях, но вам необходимо защитить их от подстановки оболочкой. Эти два означают разные вещи:

    $ git restore *.c
$ git restore \*.c

    Первый позволяет вашей оболочке расширить файловый массив, и вы запрашиваете перезапись файлов dot-C в вашем рабочем дереве версией из индекса. Последний передает *.c в Git, и вы запрашиваете пути в индексе, соответствующие шаблону, для извлечения в ваше рабочее дерево. После запуска git add hello.c; rm hello.c вы увидите not в рабочем дереве hello.c с первым, но со вторым вы увидите.

  • Так же, как файловая система . (точка) относится к текущему каталогу, использование . в качестве имени репозитория в Git (точечный репозиторий) является относительным путем и означает ваш текущий репозиторий.

Вот правила, касающиеся "flags" , которым следует следовать при написании сценариев Git:

  • Предпочтительно использовать непунктирную форму команд Git , что означает, что вам следует предпочесть git foo git-foo .

  • Разбиение коротких опций на отдельные слова (предпочитайте git foo -a -b git foo -ab , последний может даже не работать).

  • Если параметр командной строки принимает аргумент, используйте форму stuck . Другими словами, для коротких опционов пишите git foo -oArg вместо git foo -o Arg и для длинных опционов git foo --long-opt=Arg вместо git foo --long-opt Arg . Опция, которая принимает необязательный аргумент-опцию, должна быть записана в форме stuck .

  • Когда вы задаете команде параметр редакции, убедитесь, что этот параметр не является двусмысленным с именем файла в рабочем дереве. Например, не пишите git log -1 HEAD , а пишите git log -1 HEAD -- ; первый не будет работать, если в рабочем дереве есть файл с именем HEAD .

  • Многие команды допускают сокращение длинного параметра --option только до его уникального префикса (например, если нет другого параметра, имя которого начинается с opt , вы можете написать --opt для вызова флага --option ), но вам следует прописать их полностью. при написании своих сценариев; в более поздних версиях Git может появиться новая опция, имя которой имеет тот же префикс, например --optimize , чтобы сделать короткий префикс, который раньше был уникальным, больше не уникальным.

Расширенный парсер опций

Начиная с серии Git 1.5.4 и более поздних версий, многие команды Git (хотя не все из них на момент написания статьи) поставляются с расширенным анализатором параметров.

Вот список возможностей, предоставляемых этим анализатором опций.

Magic Options

Все команды, у которых активирован расширенный анализатор параметров, понимают пару волшебных параметров командной строки:

-h

дает довольно печатное использование команды.

$ git describe -h
usage: git describe [<options>] <commit-ish>*
   or: git describe [<options>] --dirty

    --contains найти тег, который идет после фиксации
    --debug стратегия поиска отладки на stderr
    --all использовать любую ссылку
    --tags использовать любой тег, даже без аннотаций
    --long всегда использовать длинный формат
    --abbrev[=<n>] использовать <n> цифр для отображения SHA-1

Обратите внимание, что некоторые подкоманды (например, git grep ) могут вести себя по-другому, если в командной строке есть что-то, кроме -h , но git subcmd -h без чего-либо еще в командной строке предназначен для последовательного указания использования.

--help-all

Некоторые команды Git принимают параметры, которые используются только для водопровода или являются устаревшими, и такие параметры скрыты от использования по умолчанию. Эта опция дает полный список опций.

Negating options

Опции с длинными именами можно отменить, добавив префикс --no- . Например, git branch имеет опцию --track , которая по умолчанию — on . Вы можете использовать --no-track , чтобы изменить это поведение. То же самое касается --color и --no-color .

Агрегирование коротких опционов

Команды, поддерживающие расширенный анализатор параметров, позволяют объединять короткие параметры. Это означает, что вы можете использовать, например, git rm -rf или git clean -fdx .

Сокращение длинных опций

Команды, поддерживающие расширенный анализатор параметров, принимают уникальный префикс длинного параметра, как если бы он был полностью написан, но используйте его с осторожностью. Например, git commit --amen ведет себя так, как если бы вы набрали git commit --amend , но это будет true только до тех пор, пока в более поздней версии Git не появится другая опция с тем же префиксом, например опция git commit --amenity .

Отделение аргумента от опции

Вы можете записать обязательный параметр опции в виде отдельного слова в командной строке. Это означает, что все следующие варианты использования работают:

$ git foo --long-opt=Arg
$ git foo --long-opt Arg
$ git foo -oArg
$ git foo -o Arg

Однако это НЕ разрешено для переключателей с необязательным значением, где необходимо использовать форму stuck :

$ git describe --abbrev HEAD # правильно
$ git describe --abbrev=10 HEAD # правильно
$ git describe --abbrev 10 HEAD # НЕ ЧТО ВЫ имели в виду

Примечания к опциям, которые часто путают

Многие команды, которые могут работать с файлами в рабочем дереве и/или индексе, могут принимать параметры --cached и/или --index . Иногда люди ошибочно полагают, что, поскольку индекс изначально назывался кешем, эти два слова являются синонимами. Это не так — эти два варианта означают совершенно разные вещи.

  • Опция --cached используется для запроса команды, которая обычно работает с файлами в рабочем дереве, работать только с индексом. Например, git grep при использовании без фиксации для указания, в какой фиксации искать строки, обычно работает с файлами в рабочем дереве, но с опцией --cached она ищет строки в индексе.

  • Опция --index используется, чтобы попросить команду, которая обычно работает с файлами в рабочем дереве, также повлиять на индекс. Например, git stash apply обычно объединяет изменения, записанные в записи тайника, в рабочее дерево, но с помощью опции --index он также объединяет изменения и с индексом.

Команда git apply может использоваться с --cached и --index (но не одновременно). Обычно команда влияет только на файлы в рабочем дереве, но с --index она исправляет как файлы, так и их записи индекса, а с --cached она изменяет только записи индекса.

Дополнительную информацию см. также в https://lore.kernel.org/git/[email protected]/ и https://lore.kernel.org/git/[email protected]/ .

Некоторые другие команды, которые также работают с файлами в рабочем дереве и/или индексе, могут использовать --staged и/или --worktree .

  • --staged точно такой же, как --cached , который используется для запроса команды работать только с индексом, а не с рабочим деревом.

  • --worktree — наоборот: попросить команду работать только с рабочим деревом, а не с индексом.

  • Эти две опции можно указать вместе, чтобы команда работала как с индексом, так и с рабочим деревом.
