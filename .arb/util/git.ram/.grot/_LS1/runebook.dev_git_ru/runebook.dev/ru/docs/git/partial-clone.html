partial-clone

Функция "Partial Clone" — это оптимизация производительности Git , которая позволяет Git работать без полной копии репозитория. Цель этой работы — позволить Git лучше обрабатывать чрезвычайно большие репозитории.

Во время операций клонирования и выборки Git загружает полное содержимое и историю репозитория. Сюда входят все коммиты, деревья и BLOB-объекты за весь срок существования репозитория. Для очень больших репозиториев клонирование может занять часы (или дни) и занять более 100 ГиБ дискового пространства.

Часто в этих репозиториях имеется множество ненужных пользователю блобов и деревьев, таких как:

  1. файлы за пределами рабочей области пользователя в дереве. Например, в репозитории с 500 тыс. каталогов и 3,5 млн файлов в каждом коммите мы можем избежать загрузки большого количества объектов, если пользователю нужен только узкий "cone" дерева исходников.

  2. крупные бинарные активы. Например, в репозитории, где большие артефакты сборки регистрируются в дереве, мы можем избежать загрузки всех предыдущих версий этих необъединяемых двоичных ресурсов и загружать только те версии, на которые действительно есть ссылки.

Частичное клонирование позволяет нам избежать предварительной загрузки таких ненужных объектов во время операций клонирования и выборки и тем самым сократить время загрузки и использование диска. Отсутствующие объекты позже могут быть "demand fetched" , если/когда это необходимо.

Пульт, который может позже предоставить недостающие объекты, называется удаленным промисором, поскольку он promises отправляет объекты по запросу. Первоначально Git поддерживал только один удаленный промисор, исходный удаленный компьютер, с которого клонировался пользователь, и который был настроен в параметре конфигурации "extensions.partialClone" . Позже была реализована поддержка более чем одного пульта Promisor.

Для использования частичного клонирования необходимо, чтобы пользователь был в сети, а исходный удаленный компьютер или другие удаленные устройства-гаранты были доступны для извлечения отсутствующих объектов по требованию. Это может быть или не быть проблематичным для пользователя. Например, если пользователь может оставаться в пределах заранее выбранного подмножества исходного дерева, он может не столкнуться с недостающими объектами. В качестве альтернативы пользователь может попытаться предварительно загрузить различные объекты, если он знает, что они переходят в автономный режим.

Non-goals

Частичное клонирование — это механизм ограничения количества BLOB-объектов и деревьев, загружаемых в пределах заданного диапазона коммитов, и поэтому он независим и не предназначен для конфликта с существующими механизмами уровня DAG для ограничения набора запрошенных коммитов (т. е. поверхностного клонирования, одиночного клонирования). ветку или получите <refspec> ).

Design overview

Частичный клон логически состоит из следующих частей:

  • Механизм, позволяющий клиенту описывать серверу ненужные или нежелательные объекты.

  • Механизм, позволяющий серверу исключать такие нежелательные объекты из пакетных файлов, отправляемых клиенту.

  • Механизм, позволяющий клиенту корректно обрабатывать отсутствующие объекты (которые ранее были пропущены сервером).

  • Механизм, позволяющий клиенту заполнять недостающие объекты по мере необходимости.

Design details

  • В согласование fetch-pack и upload-pack добавлена ​​новая возможность пакетного протокола "filter" .

    При этом используется существующий механизм обнаружения возможностей. См. "filter" в gitprotocol-pack[5] .

  • Клиенты передают "filter-spec" для клонирования и выборки, который передается на сервер для запроса фильтрации во время создания пакетного файла.

    Существуют различные фильтры для разных ситуаций. См. "--filter=<filter-spec>" в Documentation/rev-list-options.txt.

  • К объектам пакета сервера применяется запрошенная спецификация фильтра при создании файлов пакета "filtered" для клиента.

    Эти отфильтрованные пакетные файлы являются неполными в традиционном смысле, поскольку они могут содержать объекты, ссылающиеся на объекты, не содержащиеся в пакетном файле и которых еще нет у клиента. Например, отфильтрованный пакетный файл может содержать деревья или теги, ссылающиеся на отсутствующие BLOB-объекты, или фиксации, ссылающиеся на отсутствующие деревья.

  • На клиенте эти неполные файлы пакетов помечаются как "promisor packfiles" и обрабатываются различными командами по-разному.

  • На клиенте в локальную конфигурацию добавляется расширение репозитория, чтобы предотвратить сбой старых версий git во время работы из-за отсутствия объектов, с которыми они не могут справиться. См. "extensions.partialClone" в Documentation/technical/repository-version.txt».

Обработка недостающих объектов

  • Объект может отсутствовать из-за частичного клонирования или выборки или из-за повреждения репозитория. Чтобы различать эти случаи, в локальном репозитории такие фильтрованные пакфайлы, полученные с удаленных промисоров, специально обозначаются как "promisor packfiles" .

    Эти файлы пакетов обещания состоят из файла "<name>.promisor" с произвольным содержимым (например, файлы "<name>.keep" ) в дополнение к файлам "<name>.pack" и "<name>.idx" .

  • Локальный репозиторий считает "promisor object" объектом, о котором он знает (в меру своих возможностей), который удаленные поставщики обещали иметь, либо потому, что локальный репозиторий имеет этот объект в одном из своих пакетных файлов, либо потому, что другой объект-промисор ссылается на это.

    Когда Git обнаруживает отсутствующий объект, Git может определить, является ли это обещанным объектом, и обработать его соответствующим образом. В противном случае Git может сообщить о повреждении.

    Это означает, что клиенту нет необходимости явно поддерживать дорогостоящий для изменения список отсутствующих объектов.[a]

  • Поскольку почти весь код Git в настоящее время ожидает, что любой объект, на который ссылаются, будет присутствовать локально, и поскольку мы не хотим заставлять каждую команду сначала выполнять пробный прогон, добавлен резервный механизм, позволяющий Git пытаться динамически получать недостающие объекты с удаленных устройств-промисоров. .

    Когда обычный поиск объекта не может найти объект, Git вызывает promisor_remote_get_direct(), чтобы попытаться получить объект от удаленного промизора, а затем повторить поиск объекта. Это позволяет объектам быть "faulted in" без сложных алгоритмов прогнозирования.

    По соображениям эффективности проверка того, является ли отсутствующий объект действительно объектом-обещателем, не выполняется.

    Извлечение динамических объектов обычно происходит медленно, поскольку объекты извлекаются по одному.

  • checkout (и любая другая команда, использующая unpack-trees ) обучена предварительной выборке всех необходимых недостающих больших двоичных объектов в одном пакете.

  • rev-list обучен печатать недостающие объекты.

    Это может использоваться другими командами для предварительной выборки объектов. Например, "git log -p A..B" может внутренне захотеть сначала сделать что-то вроде "git rev-list --objects --quiet --missing=print A..B" и выполнить предварительную выборку этих объектов в большом объеме.

  • fsck был обновлен, чтобы полностью учитывать объекты-обещатели.

  • repack в GC был обновлен, чтобы вообще не трогать файлы пакетов обещаний и переупаковывать только другие объекты.

  • Глобальная переменная "fetch_if_missing" используется для управления тем, будет ли поиск объекта пытаться динамически получить отсутствующий объект или сообщить об ошибке.

    Нас не устраивает эта глобальная переменная, и мы хотели бы ее удалить, но это требует существенного рефакторинга объектного кода для передачи дополнительного флага.

Получение недостающих объектов

  • Извлечение объектов осуществляется путем вызова подпроцесса "git fetch" .

  • Локальный репозиторий отправляет запрос с хешами всех запрошенных объектов и не выполняет никаких согласований с пакетными файлами. Затем он получает пакетный файл.

  • Поскольку мы повторно используем существующий механизм выборки, в настоящее время выборка извлекает все объекты, на которые ссылаются запрошенные объекты, даже если они не являются необходимыми.

  • При извлечении с помощью --refetch с удаленного устройства будет запрошен полностью новый отфильтрованный пакетный файл, который можно использовать для изменения фильтра без необходимости динамического извлечения отсутствующих объектов.

Использование многих пультов Promisor

Многие пульты Promisor можно настроить и использовать.

Это позволяет, например, пользователю иметь несколько географически близких кэш-серверов для извлечения недостающих больших двоичных объектов, продолжая при этом выполнять отфильтрованные команды git-fetch с центрального сервера.

При извлечении объектов удаленные промисоры опробуются один за другим, пока все объекты не будут извлечены.

Пульты дистанционного управления, которые считаются пультами "promisor" , определяются следующими переменными конфигурации:

  • extensions.partialClone = <name>

  • remote.<name>.promisor = true

  • remote.<name>.partialCloneFilter = ...

С помощью переменной конфигурации extensions.partialClone можно настроить только один пульт дистанционного управления Promisor. Этот удаленный промисор будет использоваться последним при извлечении объектов.

Мы решили сделать его последним, что попробуем, потому что вполне вероятно, что кто-то, использующий много пультов-промисоров, делает это потому, что другие пульты-промисоры по какой-то причине лучше (возможно, они ближе или быстрее для каких-то объектов), чем исходный , а источником, скорее всего, будет удаленный компьютер, указанный в Extensions.partialClone.

Это обоснование не очень сильное, но один выбор нужно было сделать, и в любом случае долгосрочный план должен заключаться в том, чтобы сделать заказ каким-то образом полностью настраиваемым.

На данный момент другие пульты Promisor будут опробованы в том порядке, в котором они указаны в файле конфигурации.

Current limitations

  • Невозможно указать порядок, в котором пульты-промисоры опробуются другими способами, кроме порядка, в котором они указаны в файле конфигурации.

    Также невозможно указать порядок использования при выборке с одного пульта и другой порядок при выборке с другого пульта.

  • Невозможно отправить на удаленный промисор только определенные объекты.

    Невозможно одновременно отправить сообщение нескольким удаленным обещаниям в определенном порядке.

  • При динамической выборке объектов будет запрашиваться только удаленный доступ к отсутствующим объектам. Мы предполагаем, что удаленные промисоры имеют полное представление о репозитории и могут удовлетворить все такие запросы.

  • Repack по сути рассматривает гарантирующие и не гарантирующие пакетные файлы как два отдельных раздела и не смешивает их.

  • При динамической выборке объекта fetch-pack вызывается один раз для каждого элемента, поскольку большинство алгоритмов натыкаются на отсутствующий объект и должны разрешить его, прежде чем продолжить свою работу. Это может привести к значительным накладным расходам и множественным запросам аутентификации, если требуется много объектов.

  • Для динамической выборки объектов в настоящее время используется существующий пакетный протокол V0, что означает, что каждый объект запрашивается через fetch-pack. Сервер отправит полный набор информации/ссылки, когда соединение будет установлено. Если имеется большое количество ссылок, это может повлечь за собой значительные накладные расходы.

Future work

  • Улучшите способ указания порядка, в котором опробуются пульты-промисоры.

    Например, это может позволить явно указать что-то вроде: "When fetching from this remote, I want to use these promisor remotes in this order, though, when pushing or fetching to that remote, I want to use those promisor remotes in that order."

  • Разрешить отправку на пульты-промисоры.

    Пользователь может захотеть работать в треугольном рабочем процессе с несколькими удаленными устройствами Promisor, каждый из которых имеет неполное представление о репозитории.

  • Разрешить фильтрам, не основанным на имени пути, использовать растровые изображения пакфайлов (если они есть). Это было просто упущением во время первоначальной реализации.

  • Исследуйте использование длительного процесса для динамического извлечения серии объектов, например, предложенного в [5,6], чтобы уменьшить затраты на запуск процесса и накладные расходы.

    Было бы неплохо, если бы пакетный протокол V2 позволял этому длительному процессу выполнять серию запросов через одно длительное соединение.

  • Изучите пакетный протокол V2, чтобы избежать широковещательной передачи информации и ссылок при каждом соединении с сервером для динамического получения недостающих объектов.

  • Изучите необходимость обращения с незакрепленными обещанными объектами.

    Объектам в пакетных файлах promisor разрешено ссылаться на отсутствующие объекты, которые можно динамически получать с сервера. Было сделано предположение, что свободные объекты создаются только локально и поэтому не должны ссылаться на отсутствующий объект. Возможно, нам придется вернуться к этому предположению, если, например, мы динамически извлекаем отсутствующее дерево и сохраняем его как отдельный объект, а не как отдельный пакетный файл объекта.

    Это не обязательно означает, что нам нужно помечать свободные объекты как обещающие; этого может быть достаточно, чтобы ослабить функции поиска объекта или функции is-promiser.

Non-tasks

  • Каждый раз, когда всплывает тема "demand loading blobs" , создается впечатление, что кто-то предлагает разрешить серверу "guess" и отправлять дополнительные объекты, которые могут быть связаны с запрошенными объектами.

    На самом деле для этого не было сделано никакой работы; мы просто документируем, что это распространенное предложение. Мы не уверены, как это будет работать, и не планируем над этим работать.

    Сервер может отправлять больше объектов, чем запрошено (даже при динамической выборке объектов), но мы на этом не основываемся.

Footnotes

[a] Дорогостоящий для изменения список отсутствующих объектов. Ранее при разработке частичного клонирования мы обсуждали необходимость единого списка отсутствующих объектов. По сути, это будет отсортированный линейный список OID, которые были пропущены сервером во время клонирования или последующих выборок.

Этот файл необходимо будет загружать в память при каждом поиске объекта. Его нужно будет читать, обновлять и перезаписывать (как .git/index) при каждой явной команде "git fetch" и при любой выборке динамического объекта.

Затраты на чтение, обновление и запись этого файла могут привести к значительным издержкам для каждой команды, если в нем много недостающих объектов. Например, если пропущено 100 миллионов больших двоичных объектов, размер этого файла на диске будет не менее 2 ГиБ.

С помощью концепции "promisor" мы делаем вывод об отсутствующем объекте на основе типа паковочного файла, который ссылается на него.
