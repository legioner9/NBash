git-clone

Name

git-clone — клонировать репозиторий в новый каталог.

Synopsis

git clone [--template=<template-directory>]
          [-l] [-s] [--no-hardlinks] [-q] [-n] [--bare] [--mirror]
          [-o <name>] [-b <name>] [-u <upload-pack>] [--reference <repository>]
          [--dissociate] [--separate-git-dir <git-dir>]
          [--depth <depth>] [--[no-]single-branch] [--no-tags]
          [--recurse-submodules[=<pathspec>]] [--[no-]shallow-submodules]
          [--[no-]remote-submodules] [--jobs <n>] [--sparse] [--[no-]reject-shallow]
          [--filter=<filter> [--also-filter-submodules]] [--] <repository>
          [<directory>]

Description

Клонирует репозиторий во вновь созданный каталог, создает ветки удаленного отслеживания для каждой ветки в клонированном репозитории (отображается с помощью git branch --remotes ), а также создает и извлекает исходную ветвь, которая является ответвлением от текущей активной ветки клонированного репозитория.

После клонирования простой git fetch без аргументов обновит все ветки удаленного отслеживания, а git pull без аргументов дополнительно объединит удаленную главную ветвь с текущей главной веткой, если таковая имеется (это неверно, если задан "--single-branch" ; см. ниже). ).

Эта конфигурация по умолчанию достигается путем создания ссылок на заголовки удаленных ветвей в refs/remotes/origin и инициализации переменных конфигурации remote.origin.url и remote.origin.fetch .

Options

-l
--local

Когда репозиторий, из которого необходимо клонировать, находится на локальном компьютере, этот флаг обходит обычный механизм транспорта "Git aware" и клонирует репозиторий, создавая копию HEAD и всего, что находится в каталогах объектов и ссылок. Файлы в каталоге .git/objects/ жестко связаны для экономии места, когда это возможно.

Если репозиторий указан как локальный путь (например, /path/to/repo ), это значение по умолчанию, а --local по сути является пустым. Если репозиторий указан как URL, то этот флаг игнорируется (и мы никогда не используем локальные оптимизации). Указание --no-local переопределит значение по умолчанию, если задан /path/to/repo , вместо этого будет использоваться обычный транспорт Git .

Если $GIT_DIR/objects репозитория имеет символические ссылки или является символической ссылкой, клонирование завершится неудачно. Это мера безопасности, позволяющая предотвратить непреднамеренное копирование файлов путем разыменования символических ссылок.

ПРИМЕЧАНИЕ. Эта операция может сопровождаться одновременным изменением исходного репозитория, аналогично запуску cp -r src dst при изменении src .

--no-hardlinks

Заставьте процесс клонирования из репозитория в локальной файловой системе скопировать файлы в каталог .git/objects вместо использования жестких ссылок. Это может быть желательно, если вы пытаетесь сделать резервную копию своего репозитория.

-s
--shared

Если клонируемый репозиторий находится на локальном компьютере, вместо использования жестких ссылок автоматически настройте .git/objects/info/alternates для совместного использования объектов с исходным репозиторием. Полученный репозиторий начинается без какого-либо собственного объекта.

ПРИМЕЧАНИЕ. Это потенциально опасная операция; не используйте его, если не понимаете, что он делает. Если вы клонируете свой репозиторий с помощью этой опции, а затем удаляете ветки (или используете любую другую команду Git , которая делает любую существующую фиксацию без ссылки) в исходном репозитории, некоторые объекты могут стать ненужными (или висеть). Эти объекты могут быть удалены обычными операциями Git (такими как git commit ), которые автоматически вызывают git maintenance run --auto . (См. git-maintenance[1] .) Если эти объекты удалены и на них ссылается клонированный репозиторий, то клонированный репозиторий станет поврежденным.

Обратите внимание, что запуск git repack без опции --local в репозитории, клонированном с помощью --shared , приведет к копированию объектов из исходного репозитория в пакет в клонированном репозитории, устраняя экономию дискового пространства, которую обеспечивает clone --shared . Однако безопасно запускать git gc , который по умолчанию использует опцию --local .

Если вы хотите разорвать зависимость репозитория, клонированного с помощью --shared , от его исходного репозитория, вы можете просто запустить git repack -a , чтобы скопировать все объекты из исходного репозитория в пакет в клонированном репозитории.

--reference[-if-able] <repository>

Если эталонный репозиторий находится на локальном компьютере, автоматически настройте .git/objects/info/alternates для получения объектов из эталонного репозитория. Использование уже существующего репозитория в качестве альтернативы потребует копирования меньшего количества объектов из клонируемого репозитория, что снижает затраты на сетевое и локальное хранилище. При использовании --reference-if-able несуществующий каталог пропускается с предупреждением вместо прерывания клонирования.

ПРИМЕЧАНИЕ. См. ПРИМЕЧАНИЕ для опции --shared , а также для опции --dissociate .

--dissociate

Заимствуйте объекты из эталонных репозиториев, указанных в параметрах --reference , только для уменьшения передачи по сети и прекратите заимствование из них после создания клона, создав необходимые локальные копии заимствованных объектов. Эту опцию также можно использовать при локальном клонировании из репозитория, который уже заимствует объекты из другого репозитория — новый репозиторий будет заимствовать объекты из того же репозитория, и эту опцию можно использовать для остановки заимствования.

-q
--quiet

Работайте тихо. О ходе выполнения не сообщается в стандартный поток ошибок.

-v
--verbose

Бегите многословно. Не влияет на отчет о статусе выполнения в стандартном потоке ошибок.

--progress

Статус выполнения по умолчанию сообщается в стандартном потоке ошибок, когда он подключен к терминалу, если не указан --quiet . Этот флаг обеспечивает статус хода выполнения, даже если стандартный поток ошибок не направляется на терминал.

--server-option=<option>

Передайте данную строку на сервер при обмене данными по протоколу версии 2. Данная строка не должна содержать символ NUL или LF. Обработка сервером параметров сервера, включая неизвестные, зависит от сервера. Если задано несколько --server-option=<option> , все они отправляются другой стороне в порядке, указанном в командной строке.

-n
--no-checkout

После завершения клонирования проверка HEAD не выполняется.

--[no-]reject-shallow

Ошибка, если исходный репозиторий является неглубоким репозиторием. Переменную конфигурации clone.rejectShallow можно использовать для указания значения по умолчанию.

--bare

Создайте репозиторий bare Git . То есть вместо того, чтобы создавать <directory> и помещать административные файлы в <directory>/.git , сделать сам <directory> $GIT_DIR . Очевидно, имеется в виду --no-checkout , поскольку проверить рабочее дерево негде. Кроме того, заголовки удаленных ветвей копируются непосредственно в соответствующие заголовки локальных ветвей, без сопоставления их с refs/remotes/origin/ . При использовании этой опции не создаются ни ветки удаленного отслеживания, ни связанные переменные конфигурации.

--sparse

Используйте разреженную проверку, при которой изначально присутствуют только файлы в каталоге верхнего уровня. Команду git-sparse-checkout[1] можно использовать для увеличения рабочего каталога по мере необходимости.

--filter=<filter-spec>

Используйте функцию частичного клонирования и запросите, чтобы сервер отправлял подмножество доступных объектов в соответствии с заданным фильтром объектов. При использовании --filter входящий в комплект поставки <filter-spec> используется для фильтра частичного клонирования. Например, --filter=blob:none будет отфильтровывать все BLOB-объекты (содержимое файла), пока они не понадобятся Git. Кроме того, --filter=blob:limit=<size> отфильтрует все капли размером не ниже <size> . Более подробную информацию о характеристиках фильтра см. в опции --filter в git-rev-list[1] .

--also-filter-submodules

Также примените фильтр частичного клонирования ко всем подмодулям в репозитории. Требуется --filter и --recurse-submodules . Это можно включить по умолчанию, установив параметр конфигурации clone.filterSubmodules .

--mirror

Настройте зеркало исходного репозитория. Это подразумевает --bare . По сравнению с --bare , --mirror не только сопоставляет локальные ветки источника с локальными ветвями цели, он сопоставляет все ссылки (включая ветки удаленного отслеживания, заметки и т. д.) и устанавливает конфигурацию спецификации ссылок, при которой все эти ссылки перезаписываются git remote update в целевом репозитории.

-o <name>
--origin <name>

Вместо использования удаленного имени origin для отслеживания восходящего репозитория используйте <name> . Переопределяет clone.defaultRemoteName из конфигурации.

-b <name>
--branch <name>

Вместо того, чтобы указывать вновь созданный HEAD на ветку, на которую указывает HEAD клонированного репозитория, вместо этого укажите на ветку <name> . В непустом репозитории эта ветка будет извлечена. --branch также может принимать теги и отсоединять HEAD при этом коммите в результирующем репозитории.

-u <upload-pack>
--upload-pack <upload-pack>

Если этот параметр задан и доступ к репозиторию для клонирования осуществляется через ssh, это указывает нестандартный путь для команды, запускаемой на другом конце.

--template=<template-directory>

Укажите каталог, из которого будут использоваться шаблоны; (См. раздел "TEMPLATE DIRECTORY" в git-init[1] .)

-c <key>=<value>
--config <key>=<value>

Установите переменную конфигурации во вновь созданном репозитории; это вступает в силу сразу после инициализации репозитория, но до получения удаленной истории или извлечения каких-либо файлов. Ключ имеет тот же формат, что и git-config[1] (например, core.eol=true ). Если для одного и того же ключа задано несколько значений, каждое значение будет записано в файл конфигурации. Это делает безопасным, например, добавление дополнительных ссылок на выборку к удаленному источнику.

Из-за ограничений текущей реализации некоторые переменные конфигурации вступают в силу только после первоначальной выборки и извлечения. Известно, что переменные конфигурации не вступают в силу: remote.<name>.mirror и remote.<name>.tagOpt . Вместо этого используйте соответствующие опции --mirror и --no-tags .

--depth <depth>

Создайте клон shallow с историей, усеченной до указанного количества коммитов. Подразумевается --single-branch , если не задан --no-single-branch для получения истории вблизи кончиков всех ветвей. Если вы хотите поверхностно клонировать подмодули, также передайте --shallow-submodules .

--shallow-since=<date>

Создайте поверхностный клон с историей после указанного времени.

--shallow-exclude=<revision>

Создайте поверхностный клон с историей, исключая коммиты, доступные из указанной удаленной ветки или тега. Эту опцию можно указать несколько раз.

--[no-]single-branch

Клонируйте только историю, ведущую к вершине одной ветки, указанной либо опцией --branch , либо на которую указывает HEAD удаленной основной ветки. Дальнейшие выборки в результирующий репозиторий будут обновлять только ветку удаленного отслеживания для той ветки, которая использовалась для первоначального клонирования. Если HEAD на удаленном компьютере не указывал ни на одну ветку при создании клона --single-branch , ветка удаленного отслеживания не создается.

--no-tags

Не клонируйте никакие теги и установите remote.<remote>.tagOpt=--no-tags в конфигурации, чтобы будущие операции git pull и git fetch не следовали никаким тегам. Последующие явные выборки тегов по-прежнему будут работать (см. git-fetch[1] ).

Может использоваться вместе с --single-branch для клонирования и поддержки ветки без каких-либо ссылок, кроме одной клонированной ветки. Это полезно, например, для поддержки минимальных клонов ветки по умолчанию некоторого репозитория для индексации поиска.

--recurse-submodules[=<pathspec>]

После создания клона инициализируйте и клонируйте подмодули внутри него на основе предоставленной спецификации пути. Если спецификация пути не указана, все подмодули инициализируются и клонируются. Эту опцию можно указать несколько раз для спецификаций пути, состоящих из нескольких записей. В полученном клоне submodule.active установлен на указанную спецификацию пути или "." (то есть все подмодули), если спецификация пути не указана.

Субмодули инициализируются и клонируются с использованием настроек по умолчанию. Это эквивалентно запуску git submodule update --init --recursive <pathspec> сразу после завершения клонирования. Эта опция игнорируется, если клонированный репозиторий не имеет рабочего дерева/проверки (т. е. если указан какой-либо из --no-checkout / -n , --bare или --mirror ).

--[no-]shallow-submodules

Все клонированные подмодули будут неглубокими с глубиной 1.

--[no-]remote-submodules

Все клонированные подмодули будут использовать статус ветки удаленного отслеживания подмодуля для обновления подмодуля, а не записанный суперпроектом SHA-1. Эквивалентно передаче --remote в git submodule update .

--separate-git-dir=<git-dir>

Вместо размещения клонированного репозитория там, где он должен быть, поместите клонированный репозиторий в указанный каталог, а затем создайте туда символическую ссылку Git , не зависящую от файловой системы. В результате репозиторий Git можно отделить от рабочего дерева.

-j <n>
--jobs <n>

Количество подмодулей, полученных одновременно. По умолчанию используется опция submodule.fetchJobs .

<repository>

Репозиторий (возможно, удаленный), из которого выполняется клонирование. Дополнительную информацию об указании репозиториев см. в разделе GIT URLS ниже.

<directory>

Имя нового каталога для клонирования. Часть "humanish" исходного репозитория используется, если каталог явно не указан ( repo для /path/to/repo.git и foo для host.xz:foo/.git ). Клонирование в существующий каталог разрешено только в том случае, если каталог пуст.

--bundle-uri=<uri>

Перед извлечением с удаленного устройства извлеките пакет из данного <uri> и разпакуйте данные в локальный репозиторий. Ссылки в пакете будут храниться в скрытом пространстве имен refs/bundle/* . Эта опция несовместима с --depth , --shallow-since и --shallow-exclude .

Git urls

Обычно URL-адреса содержат информацию о транспортном протоколе, адресе удаленного сервера и пути к репозиторию. В зависимости от транспортного протокола некоторая часть этой информации может отсутствовать.

Git поддерживает протоколы ssh, git, http и https (кроме того, для выборки можно использовать ftp и ftps, но это неэффективно и не рекомендуется; не используйте их).

Собственный транспорт (например, URL-адрес git://) не выполняет аутентификацию, и его следует использовать с осторожностью в незащищенных сетях.

С ними можно использовать следующие синтаксисы:

  • ssh://[user@]host.xz[:port]/path/to/repo.git/

  • git://host.xz[:port]/path/to/repo.git/

  • http[s]://host.xz[:port]/path/to/repo.git/

  • ftp[s]://host.xz[:port]/path/to/repo.git/

Альтернативный синтаксис, подобный scp, также можно использовать с протоколом ssh:

  • [user@]host.xz:path/to/repo.git/

Этот синтаксис распознается только в том случае, если перед первым двоеточием нет косой черты. Это помогает отличить локальный путь, содержащий двоеточие. Например, локальный путь foo:bar можно указать как абсолютный путь или ./foo:bar , чтобы его не интерпретировали как URL-адрес ssh.

Протоколы ssh и git дополнительно поддерживают расширение ~username:

  • ssh://[user@]host.xz[:port]/~[user]/path/to/repo.git/

  • git://host.xz[:port]/~[user]/path/to/repo.git/

  • [user@]host.xz:/~[user]/path/to/repo.git/

Для локальных репозиториев, которые также поддерживаются Git , могут использоваться следующие синтаксисы:

  • /path/to/repo.git/

  • file:///path/to/repo.git/

Эти два синтаксиса в основном эквивалентны, за исключением того, что первый подразумевает опцию --local.

git clone , git fetch и git pull , но не git push , также принимают подходящий файл пакета. См. git-bundle[1] .

Когда Git не знает, как обрабатывать определенный транспортный протокол, он пытается использовать удаленный помощник remote-<transport> , если таковой существует. Чтобы явно запросить удаленного помощника, можно использовать следующий синтаксис:

  • <transport>::<address>

где <адрес> может быть путем, сервером и путем или произвольной строкой типа URL-адреса, распознаваемой конкретным вызываемым удаленным помощником. Подробности см. в gitremote-helpers[7] .

Если существует большое количество удаленных репозиториев с одинаковыми именами и вы хотите использовать для них другой формат (например, чтобы используемые вами URL-адреса были перезаписаны в работающие URL-адреса), вы можете создать раздел конфигурации вида:

        [url "<actual url base>"]
                insteadOf = <other url base>

Например, с этим:

        [url "git://git.host.xz/"]
                insteadOf = host.xz:/path/to/
                insteadOf = work:

URL-адрес типа "work:repo.git" или "host.xz:/path/to/repo.git" будет перезаписан в любом контексте, который принимает URL-адрес "git://git.host.xz/repo.git" .

Если вы хотите переписать URL-адреса только для push-уведомлений, вы можете создать раздел конфигурации формы:

        [url "<actual url base>"]
                pushInsteadOf = <other url base>

Например, с этим:

        [url "ssh://example.org/"]
                pushInsteadOf = git://example.org/

URL-адрес типа "git://example.org/path/to/repo.git" будет перезаписан в "ssh://example.org/path/to/repo.git" для push-уведомлений, но при запросе по-прежнему будет использоваться исходный URL-адрес.

Examples

  • Клонировать из восходящего потока:

    $ git clone git://git.kernel.org/pub/scm/.../linux.git my-linux
$ cd my-linux
$ make

  • Создайте локальный клон, который заимствует данные из текущего каталога, не проверяя его:

    $ git clone -l -s -n . ../copy
$ cd ../copy
$ git show-branch

  • Клонирование из исходного кода при заимствовании из существующего локального каталога:

    $ git clone --reference /git/linux.git \
        git://git.kernel.org/pub/scm/.../linux.git \
        my-linux
$ cd my-linux

  • Создайте пустой репозиторий для публичной публикации ваших изменений:

    $ git clone --bare -l /home/proj/.git /pub/scm/proj.git

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

init.templateDir

Укажите каталог, из которого будут копироваться шаблоны. (См. раздел "TEMPLATE DIRECTORY" в git-init[1] .)

init.defaultBranch

Позволяет переопределить имя ветки по умолчанию, например, при инициализации нового репозитория.

clone.defaultRemoteName

Имя удаленного устройства, которое будет создано при клонировании репозитория. По умолчанию используется origin , и его можно переопределить, передав параметр командной строки --origin в git-clone[1] .

clone.rejectShallow

Отклонить клонирование репозитория, если он неглубокий; это можно переопределить, передав параметр --reject-shallow в командной строке. См. git-clone[1] .

clone.filterSubmodules

Если предусмотрен фильтр частичного клонирования (см. --filter в git-rev-list[1] ) и используется --recurse-submodules , также примените фильтр к подмодулям.
