git-merge-tree

Name

git-merge-tree — выполнить слияние, не касаясь индекса или рабочего дерева.

Synopsis

git merge-tree [--write-tree] [<options>] <branch1> <branch2>
git merge-tree [--trivial-merge] <base-tree> <branch1> <branch2> (deprecated)

Description

Эта команда имеет современный режим --write-tree и устаревший режим --trivial-merge . За исключением раздела DEPRECATED DESCRIPTION в конце, остальная часть документации описывает современный режим --write-tree .

Выполняет слияние, но не делает новых фиксаций и не читает и не записывает ни в рабочее дерево, ни в индекс.

Выполненное объединение будет использовать те же функции, что и "real" git-merge[1] , в том числе:

  • трехстороннее объединение содержимого отдельных файлов

  • rename detection

  • правильная обработка конфликтов каталогов/файлов

  • рекурсивная консолидация предков (т. е. при наличии более одной базы слияния создание виртуальной базы слияния путем слияния баз слияния)

  • etc.

После завершения слияния создается новый объект дерева верхнего уровня. Подробности см. в OUTPUT ниже.

Options

-z

Не заключайте имена файлов в кавычки в разделе <Информация о конфликтующем файле> и заканчивайте каждое имя файла символом NUL, а не символом новой строки. Также начните раздел сообщений с символа NUL вместо новой строки. Для получения дополнительной информации см. OUTPUT ниже.

--name-only

В разделе «Информация о конфликтующем файле» вместо записи списка кортежей (режим, oid, стадия, путь) для вывода для конфликтующих файлов просто укажите список имен файлов с конфликтами (и не перечисляйте имена файлов несколько раз, если у них есть несколько конфликтующих файлов). этапы).

--[no-]messages

Запишите любые информационные сообщения, такие как "Auto-merging <path>" или уведомления о конфликте, в конец стандартного вывода. Если не указано, по умолчанию эти сообщения включаются в случае конфликтов слияния и опускаются в противном случае.

--allow-unrelated-histories

merge-tree по умолчанию выдаст ошибку, если две указанные ветки не имеют общей истории. Этот флаг может быть задан для отмены этой проверки и продолжения слияния в любом случае.

--merge-base=<commit>

Вместо поиска баз слияния для <branch1> и <branch2> укажите базу слияния для слияния, а указание нескольких баз в настоящее время не поддерживается. Эта опция несовместима с --stdin .

Output

Для успешного слияния вывод git-merge-tree представляет собой просто одну строку:

<OID of toplevel tree>

В то время как в случае конфликтного слияния выходные данные по умолчанию имеют вид:

<OID of toplevel tree>
<Conflicted file info>
<Informational messages>

Они обсуждаются индивидуально ниже.

Однако есть исключение. Если передан --stdin , то в начале появляется дополнительный раздел, в конце — символ NUL, а затем все разделы повторяются для каждой строки ввода. Таким образом, если первое слияние конфликтное, а второе чистое, выходные данные будут иметь вид:

<Merge status>
<OID of toplevel tree>
<Conflicted file info>
<Informational messages>
NUL
<Merge status>
<OID of toplevel tree>
NUL

Merge status

Это целое число, за которым следует символ NUL. Целочисленный статус:

   0: merge had conflicts
   1: merge was clean
   <0: something prevented the merge from running (e.g. access to repository
objects denied by filesystem)

OID дерева верхнего уровня

Это объект дерева, который представляет то, что будет извлечено из рабочего дерева в конце git merge . Если были конфликты, то файлы в этом дереве могут иметь встроенные маркеры конфликта. За этим разделом всегда следует новая строка (или NUL, если передан -z ).

Конфликт информации о файле

Это последовательность строк формата

<mode> <object> <stage> <filename>

Имя файла будет заключено в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). Однако если указана опция --name-only , режим, объект и стадия будут опущены. Если передается -z , "lines" завершается символом NUL вместо символа новой строки.

Informational messages

В этом разделе представлены информационные сообщения, обычно о конфликтах. Формат раздела существенно меняется в зависимости от того, пропущен ли -z .

Если передан -z :

Выходной формат — ноль или более записей с информацией о конфликтах, каждая из которых имеет вид:

<list-of-paths><conflict-type>NUL<conflict-message>NUL

где <список путей> имеет форму

<number-of-paths>NUL<path1>NUL<path2>NUL...<pathN>NUL

и включает пути (или имена ветвей), затронутые конфликтом или информационным сообщением, в <conflict-message>. Кроме того, <conflict-type> — это стабильная строка, объясняющая тип конфликта, например:

  • "Auto-merging"

  • "CONFLICT (rename/delete)"

  • "CONFLICT (submodule lacks merge base)"

  • "CONFLICT (binary)"

а <conflict-message> — это более подробное сообщение о конфликте, которое часто (но не всегда) включает в себя <stable-short-type-description>. Эти строки могут измениться в будущих версиях Git . Некоторые примеры:

  • "Auto-merging <file>"

  • "CONFLICT (rename/delete): <oldfile> renamed…​but deleted in…​"

  • "Failed to merge submodule <submodule> (no merge base)"

  • "Warning: cannot merge binary files: <filename>"

Если -z НЕ пройден:

Этот раздел начинается с пустой строки, чтобы отделить его от предыдущих разделов, а затем содержит только информацию <conflict-message> из предыдущего раздела (разделенную символами новой строки). Это нестабильные строки, которые не должны анализироваться скриптами и предназначены только для потребления человеком. Также обратите внимание, что хотя строки <conflict-message> обычно не содержат встроенных символов новой строки, иногда они содержат. (Однако сообщения в свободной форме никогда не будут содержать встроенный NUL-символ). Итак, весь блок информации предназначен для читателей-людей как совокупность всех конфликтных сообщений.

Exit status

Для успешного слияния без конфликтов статус выхода равен 0. Если слияние имеет конфликты, статус выхода равен 1. Если слияние не может быть завершено (или запущено) из-за какой-либо ошибки, статус выхода равен 1. что-то отличное от 0 или 1 (и результат не указан). Когда передается --stdin, статус возврата равен 0 как для успешных, так и для конфликтных слияний, а также для других значений, кроме 0 или 1, если не удается завершить все запрошенные слияния.

Usage notes

Эта команда предназначена для низкоуровневой сантехники, аналогично git-hash-object[1] , git-mktree[1] , git-commit-tree[1] , git-write-tree[1] , git-update-ref[1] и git-mktag[1] . Таким образом, его можно использовать как часть ряда шагов, таких как:

NEWTREE=$(git merge-tree --write-tree $BRANCH1 $BRANCH2)
test $? -eq 0 || die "There were conflicts..."
NEWCOMMIT=$(git commit-tree $NEWTREE -p $BRANCH1 -p $BRANCH2)
git update-ref $BRANCH1 $NEWCOMMIT

Обратите внимание: когда статус выхода не равен нулю, NEWTREE в этой последовательности будет содержать гораздо больше выходных данных, чем просто дерево.

В случае конфликтов выходные данные включают ту же информацию, что и git-merge[1] :

  • что будет записано в рабочее дерево ( OID of toplevel tree )

  • этапы более высокого порядка, которые будут записаны в индекс ( Conflicted file info )

  • любые сообщения, которые были бы выведены на стандартный вывод ( Informational messages )

Input format

Формат ввода git merge-tree --stdin полностью текстовый. Каждая строка имеет следующий формат:

[<base-commit> -- ]<branch1> <branch2>

Если одна строка разделена -- , строка перед разделителем используется для указания базы слияния для слияния, а строка после разделителя описывает ветки, подлежащие слиянию.

Ошибки, которых следует избегать

НЕ просматривайте полученное дерево верхнего уровня, чтобы попытаться определить, какие файлы конфликтуют; вместо этого проанализируйте раздел Conflicted file info . Мало того, что анализ всего дерева будет ужасно медленным в больших репозиториях, существует множество типов конфликтов, которые не могут быть представлены маркерами конфликта (изменение/удаление, конфликт режимов, двоичный файл изменен с обеих сторон, конфликты файлов/каталогов, различные варианты конфликтов переименования, и т. д.)

НЕ интерпретируйте пустой список Conflicted file info как чистое слияние; проверьте статус выхода. Слияние может иметь конфликты без конфликтов отдельных файлов (есть несколько типов конфликтов переименования каталогов, которые попадают в эту категорию, и другие также могут быть добавлены в будущем).

НЕ пытайтесь угадать или заставить пользователя угадать типы конфликтов из списка Conflicted file info . Информации для этого недостаточно. Например: конфликты переименования/переименования (1to2) (обе стороны переименовывают один и тот же файл по-разному) приведут к тому, что три разных файла будут иметь стадии более высокого порядка (но каждый имеет только одну ступень более высокого порядка), без возможности (за исключением раздела Informational messages ) чтобы определить, какие три файла связаны. Конфликты файлов/каталогов также приводят к тому, что файл имеет ровно одну ступень более высокого порядка. Возможные конфликты при переименовании каталога (когда "merge.directoryRenames" не установлен или установлен на "conflicts" ) также приводят к созданию файла ровно с одним этапом более высокого порядка. Во всех случаях раздел Informational messages содержит необходимую информацию, хотя он не предназначен для машинного анализа.

НЕ предполагайте, что каждый путь от Conflicted file info и логические конфликты в Informational messages имеют сопоставление «один к одному», а также что не существует сопоставления «один ко многим» или сопоставления «многие к одному». Существуют сопоставления «многие ко многим», что означает, что каждый путь может иметь множество типов логических конфликтов в одном слиянии, и каждый тип логического конфликта может влиять на множество путей.

НЕ предполагайте, что все имена файлов, перечисленные в разделе Informational messages , имеют конфликты. Сообщения могут быть включены для файлов, у которых нет конфликтов, например "Auto-merging <file>" .

ИЗБЕГАЙТЕ брать OIDS из Conflicted file info и повторно объединять их, чтобы представить пользователю конфликты. Это приведет к потере информации. Вместо этого найдите версию файла, найденную в OID of toplevel tree , и покажите ее. В частности, последний будет иметь маркеры конфликта, аннотированные исходной объединяемой веткой/фиксацией, и, если были задействованы переименования, исходным именем файла. Хотя вы можете включить исходную ветвь/фиксацию в аннотации маркера конфликта при повторном объединении, исходное имя файла недоступно в Conflicted file info , и, таким образом, вы потеряете информацию, которая может помочь пользователю разрешить конфликт.

Deprecated description

Для DESCRIPTION , в отличие от остальной части документации, в этом разделе описан устаревший режим --trivial-merge .

За исключением дополнительного --trivial-merge , этот режим не допускает никаких опций.

Этот режим считывает три древовидных структуры и выводит тривиальные результаты слияния и конфликтующие этапы на стандартный вывод в полуразностном формате. Поскольку это было разработано для сценариев более высокого уровня, которые могли использовать и объединять результаты обратно в индекс, он опускает записи, соответствующие <branch1>. Результат этой второй формы аналогичен тому, что делает трехсторонний git read-tree -m , но вместо сохранения результатов в индексе команда выводит записи на стандартный вывод.

Эта форма не только имеет ограниченную применимость (тривиальное слияние не может обрабатывать слияния содержимого отдельных файлов, обнаружение переименования, правильную обработку конфликтов каталогов/файлов и т. д.), с выходным форматом также сложно работать, и, как правило, он будет менее производительным. чем первая форма даже при успешных слияниях (особенно при работе в больших репозиториях).
