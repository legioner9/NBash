git-shortlog

Name

git-shortlog — суммировать выходные данные git log

Synopsis

git shortlog [<options>] [<revision-range>] [[--] <path>…​]
git log --pretty=short | git shortlog [<options>]

Description

Обобщает выходные данные git log в формате, подходящем для включения в объявления о выпуске. Каждый коммит будет сгруппирован по автору и названию.

Кроме того, "[PATCH]" будет удален из описания фиксации.

Если в командной строке не передаются никакие версии и стандартный ввод не является терминалом или текущая ветвь отсутствует, git shortlog выведет сводку журнала, прочитанного из стандартного ввода, без ссылки на текущий репозиторий.

Options

-n
--numbered

Сортируйте выходные данные по количеству коммитов на каждого автора, а не по алфавитному порядку авторов.

-s
--summary

Скрыть описание фиксации и предоставить только сводку количества коммитов.

-e
--email

Покажите адрес электронной почты каждого автора.

--format[=<format>]

Вместо темы коммита используйте другую информацию для описания каждого коммита. <format> может быть любой строкой, принимаемой опцией --format git log , например * [%h] %s . (См. раздел "PRETTY FORMATS" git-log[1] .)

Each pretty-printed commit will be rewrapped before it is shown.
--date=<format>

Показать даты, отформатированные в соответствии с заданной строкой даты. (См. опцию --date в разделе "Commit Formatting" статьи git-log[1] ). Полезно с --group=format:<format> .

--group=<type>

Групповые коммиты на основе <type> . Если опция --group не указана, значением по умолчанию является author . <type> является одним из:

  • author , коммиты сгруппированы по авторам

  • committer , коммиты группируются по коммиттерам (так же, как -c )

  • trailer:<field> , <field> интерпретируется как трейлер сообщения фиксации без учета регистра (см. git-interpret-trailers[1] ). Например, если в вашем проекте используются прицепы Reviewed-by , возможно, вам захочется узнать, кто писал отзывы о git shortlog -ns --group=trailer:reviewed-by .

  • format:<format> — любая строка, принимаемая опцией --format в git log . (См. раздел "PRETTY FORMATS" в git-log[1] .)

    Обратите внимание, что коммиты, не включающие трейлер, засчитываться не будут. Аналогично, коммиты с несколькими трейлерами (например, с несколькими подписаниями) могут учитываться более одного раза (но только один раз для каждого уникального значения трейлера в этом коммите).

    Shortlog попытается проанализировать каждое значение трейлера как идентификатор name <email> . В случае успеха применяется карта почты, а адрес электронной почты опускается, если не указана опция --email . Если значение невозможно проанализировать как идентификатор, оно будет воспринято буквально и полностью.

Если --group указан несколько раз, фиксации учитываются для каждого значения (но опять же, только один раз для каждого уникального значения в этом коммите). Например, git shortlog --group=author --group=trailer:co-authored-by учитывает как авторов, так и соавторов.

-c
--committer

Это псевдоним --group=committer .

-w[<width>[,<indent1>[,<indent2>]]]

Перенесите вывод, перенеся каждую строку в width . Первая строка каждой записи имеет отступ indent1 , а вторая и последующие строки — пробелы indent2 . width , indent1 и indent2 по умолчанию равны 76, 6 и 9 соответственно.

Если ширина равна 0 (ноль), сделайте отступ для строк вывода, не перенося их.

<revision-range>

Показывать только коммиты в указанном диапазоне ревизий. Если <revision-range> не указан, по умолчанию используется HEAD (т. е. вся история, ведущая к текущему коммиту). origin..HEAD указывает все коммиты, доступные из текущего коммита (т. е. HEAD ), но не из origin . Полный список способов написания <revision-range> см. в разделе "Specifying Ranges" документа gitrevisions[7] .

[--] <path>…

Рассматривайте только те коммиты, которых достаточно, чтобы объяснить, как появились файлы, соответствующие указанным путям.

В случае возникновения путаницы в путях может потребоваться префикс -- , чтобы отделить их от опций или диапазона ревизий.

Commit Limiting

Помимо указания диапазона коммитов, которые должны быть перечислены с использованием специальных обозначений, объясненных в описании, может быть применено дополнительное ограничение коммитов.

Использование большего количества параметров обычно дополнительно ограничивает вывод (например, --since=<date1> ограничивает коммиты новее, чем <date1> , а использование его с --grep=<pattern> дополнительно ограничивает коммиты, сообщение журнала которых имеет строку, соответствующую <pattern> ), если не указано иное.

Обратите внимание, что они применяются перед фиксацией параметров порядка и форматирования, таких как --reverse .

-<number>
-n <number>
--max-count=<number>

Ограничьте количество коммитов для вывода.

--skip=<number>

Пропустите фиксации number перед отображением результатов фиксации.

--since=<date>
--after=<date>

Показывать коммиты, более поздние, чем определенная дата.

--since-as-filter=<date>

Показать все коммиты, сделанные позднее определенной даты. При этом выполняются все коммиты в диапазоне, а не останавливаются на первом коммите, который старше определенной даты.

--until=<date>
--before=<date>

Показать коммиты старше определенной даты.

--author=<pattern>
--committer=<pattern>

Ограничьте вывод коммитов теми, у которых строки заголовка автора/коммиттера соответствуют указанному шаблону (регулярному выражению). При наличии более одного --author=<pattern> выбираются коммиты, автор которых соответствует любому из заданных шаблонов (аналогично для нескольких --committer=<pattern> ).

--grep-reflog=<pattern>

Ограничьте вывод коммитов теми, в которых записи журнала обновлений соответствуют указанному шаблону (регулярному выражению). При наличии более чем одного --grep-reflog выбираются коммиты, сообщение рефлога которых соответствует любому из заданных шаблонов. Использование этой опции, если не используется --walk-reflogs , является ошибкой.

--grep=<pattern>

Ограничьте вывод коммитов теми, в которых сообщение журнала соответствует указанному шаблону (регулярному выражению). При наличии более чем одного --grep=<pattern> выбираются коммиты, сообщение которых соответствует любому из заданных шаблонов (но см. --all-match ).

Когда действует --notes , сообщение из примечаний сопоставляется, как если бы оно было частью сообщения журнала.

--all-match

Ограничьте вывод коммитов теми, которые соответствуют всем заданным --grep , вместо тех, которые соответствуют хотя бы одному.

--invert-grep

Ограничьте вывод коммитов теми, в которых сообщение журнала не соответствует шаблону, указанному в --grep=<pattern> .

-i
--regexp-ignore-case

Сопоставьте шаблоны ограничения регулярных выражений без учета регистра букв.

--basic-regexp

Считайте, что ограничивающие шаблоны — это базовые регулярные выражения; это значение по умолчанию.

-E
--extended-regexp

Рассматривайте ограничивающие шаблоны как расширенные регулярные выражения вместо базовых регулярных выражений по умолчанию.

-F
--fixed-strings

Считайте, что ограничивающие шаблоны представляют собой фиксированные строки (не интерпретируйте шаблон как регулярное выражение).

-P
--perl-regexp

Считайте, что ограничивающие шаблоны — это регулярные выражения, совместимые с Perl.

Поддержка этих типов регулярных выражений является дополнительной зависимостью времени компиляции. Если Git не был скомпилирован с их поддержкой, предоставление этой опции приведет к его прекращению.

--remove-empty

Остановитесь, когда заданный путь исчезнет из дерева.

--merges

Печатать только коммиты слияния. Это точно так же, как --min-parents=2 .

--no-merges

Не печатайте коммиты с более чем одним родителем. Это точно так же, как --max-parents=1 .

--min-parents=<number>
--max-parents=<number>
--no-min-parents
--no-max-parents

Показывать только те коммиты, которые имеют как минимум (или максимум) такое же количество родительских коммитов. В частности, --max-parents=1 аналогичен --no-merges , --min-parents=2 аналогичен --merges . --max-parents=0 дает все корневые коммиты, а --min-parents=3 все слияния осьминога.

--no-min-parents и --no-max-parents снова сбрасывают эти ограничения (на отсутствие ограничений). Эквивалентными формами являются --min-parents=0 (любая фиксация имеет 0 или более родительских элементов) и --max-parents=-1 (отрицательные числа означают отсутствие верхнего предела).

--first-parent

При поиске коммитов для включения следуйте только первому родительскому коммиту после просмотра коммита слияния. Этот параметр может дать лучший обзор при просмотре развития конкретной тематической ветки, поскольку слияния с тематической веткой, как правило, сводятся только к корректировке время от времени обновлений исходной ветки, и этот параметр позволяет игнорировать отдельные коммиты, внесенные в ваша история от такого слияния.

--exclude-first-parent-only

При поиске коммитов для исключения (с ^ ) следуйте только первому родительскому коммиту после просмотра коммита слияния. Это можно использовать для поиска набора изменений в тематической ветке с той точки, где он расходится с удаленной веткой, учитывая, что произвольные слияния могут быть действительными изменениями тематической ветки.

--not

Меняет значение префикса ^ (или его отсутствия) для всех последующих спецификаторов версии, вплоть до следующего --not . При использовании в командной строке перед --stdin изменения, передаваемые через стандартный ввод, не будут затронуты. И наоборот, при передаче через стандартный ввод изменения, переданные в командной строке, не будут затронуты.

--all

Представьте, что все ссылки в refs/ вместе с HEAD указаны в командной строке как <commit> .

--branches[=<pattern>]

Представьте, что все ссылки в refs/heads указаны в командной строке как <commit> . Если задан <pattern> , ограничьте ветки теми, которые соответствуют данному glob оболочки. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--tags[=<pattern>]

Представьте, что все ссылки в refs/tags указаны в командной строке как <commit> . Если задан <pattern> , ограничьте теги теми, которые соответствуют данному glob оболочки. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--remotes[=<pattern>]

Представьте, что все ссылки в refs/remotes указаны в командной строке как <commit> . Если задан <pattern> , ограничьте ветки удаленного отслеживания теми, которые соответствуют заданному glob оболочки. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--glob=<glob-pattern>

Представьте, что все ссылки, соответствующие glob оболочки <glob-pattern> , указаны в командной строке как <commit> . Ведущий refs/ автоматически добавляется в начало, если отсутствует. Если в шаблоне отсутствуют ? , * или [ , в конце подразумевается /* .

--exclude=<glob-pattern>

Не включайте ссылки, соответствующие <glob-pattern> , которые в противном случае учитывались бы следующими --all , --branches , --tags , --remotes или --glob . При повторении этого параметра накапливаются шаблоны исключений до следующего параметра --all , --branches , --tags , --remotes или --glob (другие параметры или аргументы не удаляют накопленные шаблоны).

Приведенные шаблоны не должны начинаться с refs/heads , refs/tags или refs/remotes при применении к --branches , --tags или --remotes соответственно, и они должны начинаться с refs/ при применении к --glob или --all . Если предполагается завершающий /* , его необходимо указать явно.

--exclude-hidden=[fetch|receive|uploadpack]

Не включайте ссылки, которые могут быть скрыты git-fetch , git-receive-pack или git-upload-pack , сверившись с соответствующей конфигурацией fetch.hideRefs , receive.hideRefs или uploadpack.hideRefs вместе с transfer.hideRefs (см. git-config[1] ). Эта опция влияет на следующую опцию псевдо-ссылки --all или --glob и очищается после их обработки.

--reflog

Представьте, что все объекты, упомянутые в рефлогах, указаны в командной строке как <commit> .

--alternate-refs

Представьте, что все объекты, упомянутые в качестве справочных подсказок альтернативных репозиториев, были перечислены в командной строке. Альтернативный репозиторий — это любой репозиторий, каталог объектов которого указан в objects/info/alternates . Набор включенных объектов может быть изменен с помощью core.alternateRefsCommand и т. д. См. git-config[1] .

--single-worktree

По умолчанию все рабочие деревья будут проверяться по следующим параметрам, если их больше одного (см. git-worktree[1] ): --all , --reflog и --indexed-objects . Эта опция заставляет их проверять только текущее рабочее дерево.

--ignore-missing

Увидев во входных данных недопустимое имя объекта, притворитесь, будто неверных входных данных не было.

--bisect

Представьте, что в списке указан неправильный номер деления пополам refs/bisect/bad , а за ним в командной строке следуют --not и хорошие ссылки пополам refs/bisect/good-* .

--stdin

Помимо получения аргументов из командной строки, также читайте их из стандартного ввода. При этом принимаются коммиты и псевдоопции, такие как --all и --glob= . Когда виден разделитель -- , следующие входные данные обрабатываются как пути и используются для ограничения результата. Флаги типа --not , которые считываются через стандартный ввод, учитываются только для аргументов, передаваемых таким же образом, и не влияют на последующие аргументы командной строки.

--cherry-mark

Как --cherry-pick (см. ниже), но помечайте эквивалентные коммиты с помощью = , а не опускайте их, а неэквивалентные — с помощью + .

--cherry-pick

Опускайте любой коммит, который вносит те же изменения, что и другой коммит на «другой стороне», если набор коммитов ограничен симметричной разницей.

Например, если у вас есть две ветки, A и B , обычный способ перечислить все коммиты только на одной стороне из них — это --left-right (см. пример ниже в описании опции --left-right ). Однако он показывает коммиты, которые были выбраны из другой ветки (например, «3rd on b» может быть выбран из ветки A). При использовании этой опции такие пары коммитов исключаются из вывода.

--left-only
--right-only

В списке фиксируются только соответствующие стороны симметричной разницы, т. е. только те, которые будут отмечены < соответственно. > от --left-right .

Например, --cherry-pick --right-only A...B пропускает те фиксации из B , которые находятся в A или эквивалентны исправлениям фиксации в A . Другими словами, здесь перечислены коммиты + из git cherry A B . Точнее, --cherry-pick --right-only --no-merges дает точный список.

--cherry

Синоним --right-only --cherry-mark --no-merges ; полезно ограничить вывод коммитами на нашей стороне и пометить те, которые были применены к другой стороне разветвленной истории, с помощью git log --cherry upstream...mybranch , аналогично git cherry upstream mybranch .

-g
--walk-reflogs

Вместо того, чтобы идти по цепочке предков коммита, пройдитесь по записям релога от самой последней к более старой. При использовании этой опции вы не можете указать исключаемые фиксации (то есть нельзя использовать нотации ^commit , commit1..commit2 и commit1...commit2 ).

Если формат --pretty отличается от oneline и reference (по очевидным причинам), это приводит к тому, что в выводе появляются две дополнительные строки информации, взятые из журнала ссылок. Обозначение журнала обновлений в выходных данных может отображаться как ref@{Nth} (где Nth — индекс обратной хронологии в журнале обновлений) или как ref@{timestamp} (с ​​отметкой времени для этой записи), в зависимости от нескольких правил:

  1. Если отправная точка указана как ref@{Nth} , покажите формат индекса.

  2. Если отправная точка была указана как ref@{now} , покажите формат метки времени.

  3. Если ни один из них не использовался, но в командной строке был указан --date , покажите временную метку в формате, запрошенном --date .

  4. В противном случае покажите формат индекса.

В --pretty=oneline сообщение о фиксации предваряется этой информацией в той же строке. Эту опцию нельзя комбинировать с --reverse . См. также git-reflog[1] .

Под --pretty=reference эта информация вообще не будет отображаться.

--merge

После неудачного слияния покажите ссылки, которые касаются файлов, имеющих конфликт, и не существуют на всех головках для слияния.

--boundary

Выходные данные исключают граничные фиксации. Граничные коммиты имеют префикс - .

History Simplification

Иногда вас интересуют только части истории, например, коммиты, изменяющие определенный <path>. Но есть две части History Simplification : одна часть выбирает коммиты, а другая — как это сделать, поскольку существуют различные стратегии для упрощения истории.

Следующие параметры позволяют выбрать коммиты для отображения:

<paths>

Выбраны коммиты, изменяющие заданные <пути>.

--simplify-by-decoration

Выбираются коммиты, на которые ссылается какая-либо ветка или тег.

Обратите внимание, что дополнительные коммиты могут быть показаны для получения значимой истории.

Следующие параметры влияют на способ выполнения упрощения:

Default mode

Упрощает историю до простейшей истории, объясняющей конечное состояние дерева. Самый простой, потому что он удаляет некоторые боковые ветки, если конечный результат тот же (т. е. объединяет ветки с одинаковым содержимым).

--show-pulls

Включите все коммиты из режима по умолчанию, а также любые коммиты слияния, которые не являются TREESAME для первого родителя, но являются TREESAME для более позднего родителя. Этот режим полезен для отображения коммитов слияния, которые "first introduced" являются изменением ветки.

--full-history

То же, что и режим по умолчанию, но не удаляет часть истории.

--dense

Показаны только выбранные коммиты, а также некоторые из них, имеющие значимую историю.

--sparse

Показаны все коммиты в упрощенной истории.

--simplify-merges

Дополнительная опция для --full-history , позволяющая удалить некоторые ненужные слияния из результирующей истории, поскольку в это слияние не входят выбранные коммиты.

--ancestry-path[=<commit>]

Если задан диапазон отображаемых коммитов (например, commit1..commit2 или commit2 ^commit1 ), в этом диапазоне отображаются только те коммиты, которые являются предками <commit>, потомками <commit> или самого <commit>. Если фиксация не указана, используйте commit1 (исключенная часть диапазона) в качестве <фиксации>. Можно проходить несколько раз; в этом случае коммит включается, если он является одним из данных коммитов или является предком или потомком одного из них.

Далее следует более подробное объяснение.

Предположим, вы указали foo в качестве <paths>. Мы будем называть коммиты, изменяющие foo , !TREESAME, а остальные TREESAME. (В diff, отфильтрованном для foo , они выглядят разными и равными соответственно.)

В дальнейшем мы всегда будем обращаться к одному и тому же примеру истории, чтобы проиллюстрировать различия между настройками упрощения. Мы предполагаем, что вы фильтруете файл foo в этом графе коммитов:

          .-A---M---N---O---P---Q
         /     /   /   /   /   /
        I     B   C   D   E   Y
         \   /   /   /   /   /
          `-------------'   X

Горизонтальная линия истории A---Q считается первым родителем каждого слияния. Коммиты:

  • I — это первоначальный коммит, в котором существует foo с содержимым «asdf», а файл quux существует с содержимым «quux». Первоначальные коммиты сравниваются с пустым деревом, поэтому I имеет значение !TREESAME.

  • В A foo содержит только «foo».

  • B содержит те же изменения, что и A . Его слияние M тривиально и, следовательно, TREESAME для всех родителей.

  • C не меняет foo , но его слияние N меняет его на «foobar», поэтому он не является TREESAME ни для одного родителя.

  • D устанавливает для foo значение «baz». Его слияние O объединяет строки из N и D в «foobarbaz»; т. е. это не TREESAME ни для одного родителя.

  • E изменяет quux на «xyzzy», а его слияние P объединяет строки в «quux xyzzy». P TREESAME O , но не E .

  • X — это независимый корневой коммит, который добавил новый файл side , а Y изменил его. Y — это ДЕРЕВО, аналогичное X . Его слияние Q добавило side к P , а Q является TREESAME к P , но не к Y .

rev-list просматривает историю назад, включая или исключая фиксации в зависимости от того, используется ли --full-history и/или родительская перезапись (через --parents или --children ). Доступны следующие настройки.

Default mode

Коммиты включаются, если они не являются TREESAME ни для одного родителя (хотя это можно изменить, см. --sparse ниже). Если фиксация была слиянием и была TREESAME для одного родителя, следуйте только этому родителю. (Даже если существует несколько родителей TREESAME, подписывайтесь только на одного из них.) В противном случае следуйте за всеми родителями.

Это приводит к:

          .-A---N---O
         /     /   /
        I---------D

Обратите внимание, что правило следовать только родительскому элементу TREESAME, если таковой имеется, полностью исключило B из рассмотрения. C рассматривался через N , но это TREESAME. Корневые коммиты сравниваются с пустым деревом, поэтому I — это !TREESAME.

Отношения родитель/потомок видны только с помощью --parents , но это не влияет на коммиты, выбранные в режиме по умолчанию, поэтому мы показали родительские строки.

--полная история без перезаписи родительских элементов

Этот режим отличается от режима по умолчанию в одном пункте: всегда следует за всеми родительскими элементами слияния, даже если для одного из них это TREESAME. Даже если более чем одна сторона слияния имеет включенные коммиты, это не означает, что само слияние включено! В примере мы получаем

        I  A  B  N  D  O  P  Q

M был исключен, поскольку он является TREESAME для обоих родителей. E , C и B были пройдены, но только B был !TREESAME, поэтому остальные не отображаются.

Обратите внимание, что без перезаписи родительских элементов невозможно говорить об отношениях родитель/потомок между коммитами, поэтому мы показываем их несвязанными.

--полная история с перезаписью родительских элементов

Обычные коммиты включаются только в том случае, если они имеют формат !TREESAME (хотя это можно изменить, см. --sparse ниже).

Слияния всегда включены. Однако их родительский список переписан: вдоль каждого родителя удаляются коммиты, которые не включены сами. Это приводит к

          .-A---M---N---O---P---Q
         /     /   /   /   /
        I     B   /   D   /
         \   /   /   /   /
          `-------------'

Сравните с --full-history не переписывая выше. Обратите внимание, что E был удален, поскольку он TREESAME, но родительский список P был переписан, чтобы содержать родительский I E . То же самое произошло с C и N , а также X , Y и Q .

Помимо вышеуказанных настроек, вы можете изменить, влияет ли TREESAME на включение:

--dense

Проходящие коммиты включаются, если они не являются TREESAME ни для одного родителя.

--sparse

Включаются все пройденные коммиты.

Обратите внимание, что без --full-history это по-прежнему упрощает слияния: если один из родительских элементов TREESAME, мы следуем только этому, поэтому другие стороны слияния никогда не обрабатываются.

--simplify-merges

Сначала построим график истории так же, как это делает --full-history с перезаписью родителя (см. выше).

Затем упростите каждую фиксацию C до замены C' в итоговой истории по следующим правилам:

  • Установите C' на C .

  • Замените каждый родительский P C' его упрощением P' . В процессе удаления родителей, которые являются предками других родителей или корневых коммитов TREESAME в пустое дерево, и удаляйте дубликаты, но будьте осторожны, чтобы никогда не удалять всех родителей, к которым мы TREESAME.

  • Если после перезаписи родительского объекта C' является корневым коммитом или фиксацией слияния (имеет ноль или >1 родительских элементов), граничным коммитом или !TREESAME, он остается. В противном случае он заменяется единственным родителем.

Эффект от этого лучше всего проявляется в сравнении с --full-history с перезаписью родительского элемента. Пример превращается в:

          .-A---M---N---O
         /     /       /
        I     B       D
         \   /       /
          `---------'

Обратите внимание на основные отличия N , P и Q от --full-history :

  • Из родительского списка N был удален I , поскольку он является предком другого родительского списка M . Тем не менее, N остался, потому что это !TREESAME.

  • Из родительского списка P аналогичным образом был удален I . Затем P был полностью удален, поскольку у него был один родительский элемент и он был TREESAME.

  • Родительский список Q имел Y , упрощенный до X . Затем X был удален, поскольку это был корень TREESAME. Затем Q был полностью удален, поскольку у него был один родительский элемент и он был TREESAME.

Доступен еще один режим упрощения:

--ancestry-path[=<commit>]

Ограничьте отображаемые коммиты теми, которые являются предками <commit>, потомками <commit> или самими <commit>.

В качестве примера использования рассмотрим следующую историю коммитов:

            D---E-------F
           /     \       \
          B---C---G---H---I---J
         /                     \
        A-------K---------------L--M

Обычный D..M вычисляет набор коммитов, которые являются предками M , но исключает те, которые являются предками D . Это полезно для того, чтобы увидеть, что произошло с историей, ведущей к M , начиная с D , в том смысле, что «что есть в M такого, чего не было в D ». Результатом в этом примере будут все коммиты, кроме A и B (и, конечно, самого D ).

Однако, когда мы хотим выяснить, какие коммиты в M заражены ошибкой, представленной D , и нуждаются в исправлении, мы можем захотеть просмотреть только подмножество D..M , которое на самом деле является потомками D , то есть исключая C и K . Именно это и делает опция --ancestry-path . Применительно к линейке D..M это приводит к:

                E-------F
                 \       \
                  G---H---I---J
                               \
                                L--M

Мы также можем использовать --ancestry-path=D вместо --ancestry-path , что означает то же самое применительно к диапазону D..M , но это более явно.

Если вместо этого нас интересует данная тема в этом диапазоне и все коммиты, затронутые этой темой, мы можем захотеть просмотреть только подмножество D..M , которое содержит эту тему в своем исходном пути. Итак, использование --ancestry-path=H D..M , например, приведет к:

                E
                 \
                  G---H---I---J
                               \
                                L--M

Тогда как --ancestry-path=K D..M приведет к

                K---------------L--M

Прежде чем обсуждать другой вариант, --show-pulls , нам нужно создать новый пример истории.

Распространенная проблема, с которой сталкиваются пользователи при просмотре упрощенной истории, заключается в том, что известная им фиксация, изменившая файл, каким-то образом не отображается в упрощенной истории файла. Продемонстрируем новый пример и покажем, как в таком случае работают такие варианты, как --full-history и --simplify-merges :

          .-A---M-----C--N---O---P
         /     / \  \  \/   /   /
        I     B   \  R-'`-Z'   /
         \   /     \/         /
          \ /      /\        /
           `---X--'  `---Y--'

В этом примере предположим, что I создал file.txt , который был модифицирован A , B и X по-разному. Однородительские фиксации C , Z и Y не изменяют file.txt . Коммит слияния M был создан путем разрешения конфликта слияния и включил в себя оба изменения из A и B и, следовательно, не является TREESAME ни для одного из них. Однако коммит слияния R был создан путем игнорирования содержимого file.txt по адресу M и взятия только содержимого file.txt по адресу X . Следовательно, R является TREESAME X , но не M . Наконец, естественным решением слияния для создания N является использование содержимого file.txt по адресу R , поэтому N является TREESAME для R , но не для C . Коммиты слияния O и P являются TREESAME для своих первых родительских элементов, но не для вторых родительских, Z и Y соответственно.

При использовании режима по умолчанию N и R имеют родительского элемента TREESAME, поэтому эти ребра обходят, а остальные игнорируются. Итоговый график истории:

        I---X

При использовании --full-history Git проходит каждое ребро. Это позволит обнаружить коммиты A и B и слияние M , а также выявить коммиты слияния O и P . При переписывании родительских элементов результирующий график выглядит следующим образом:

          .-A---M--------N---O---P
         /     / \  \  \/   /   /
        I     B   \  R-'`--'   /
         \   /     \/         /
          \ /      /\        /
           `---X--'  `------'

Здесь коммиты слияния O и P вносят дополнительный шум, поскольку на самом деле они не вносят изменений в file.txt . Они лишь объединили тему, основанную на более старой версии file.txt . Это распространенная проблема в репозиториях, использующих рабочий процесс, в котором многие участники работают параллельно и объединяют свои тематические ветки в одну магистраль: в результатах --full-history появляется много несвязанных слияний.

При использовании опции --simplify-merges коммиты O и P исчезают из результатов. Это связано с тем, что переписанные вторые родительские элементы O и P доступны из их первых родителей. Эти ребра удаляются, и тогда коммиты выглядят как коммиты с одним родителем, которые являются TREESAME для своего родителя. Это также происходит с коммитом N , в результате чего история выглядит следующим образом:

          .-A---M--.
         /     /    \
        I     B      R
         \   /      /
          \ /      /
           `---X--'

В этом представлении мы видим все важные изменения одного родителя из A , B и X . Мы также видим тщательно решенное слияние M и не очень тщательно разрешенное слияние R . Обычно этой информации достаточно, чтобы определить, почему A и B "disappeared" фиксируются из истории в представлении по умолчанию. Однако у этого подхода есть несколько проблем.

Первая проблема — производительность. В отличие от любого предыдущего варианта, параметр --simplify-merges требует пройти всю историю коммитов, прежде чем возвращать единственный результат. Это может затруднить использование этой опции для очень больших репозиториев.

Второй вопрос касается аудита. Когда над одним и тем же репозиторием работает много участников, важно, какие коммиты слияния внесли изменения в важную ветку. Вышеупомянутое проблемное слияние R вряд ли является фиксацией слияния, которая использовалась для слияния с важной веткой. Вместо этого было использовано слияние N для объединения R и X в важную ветвь. Этот коммит может содержать информацию о том, почему изменение X переопределяет изменения из A и B в его сообщении о фиксации.

--show-pulls

В дополнение к коммитам, отображаемым в истории по умолчанию, покажите каждый коммит слияния, который не является TREESAME для своего первого родителя, но является TREESAME для более позднего родителя.

Когда фиксация слияния включена в --show-pulls , слияние обрабатывается так, как если бы это "pulled" было изменением из другой ветки. При использовании --show-pulls в этом примере (и никаких других опций) результирующий график выглядит следующим образом:

        I---X---R---N

Здесь включены коммиты слияния R и N , поскольку они перенесли коммиты X и R в базовую ветку соответственно. Эти слияния являются причиной того, что коммиты A и B не отображаются в истории по умолчанию.

Когда --show-pulls работает в паре с --simplify-merges , график включает всю необходимую информацию:

          .-A---M--.   N
         /     /    \ /
        I     B      R
         \   /      /
          \ /      /
           `---X--'

Обратите внимание: поскольку M доступен из R , граница от N до M была упрощена. Однако N по-прежнему фигурирует в истории как важный коммит, поскольку это "pulled" — изменение R в основной ветке.

Опция --simplify-by-decoration позволяет просматривать только общую картину топологии истории, опуская коммиты, на которые не ссылаются теги. Коммиты помечаются как !TREESAME (другими словами, сохраняются после описанных выше правил упрощения истории), если (1) на них ссылаются теги или (2) они изменяют содержимое путей, указанных в командной строке. Все остальные коммиты помечены как TREESAME (можно упростить).
