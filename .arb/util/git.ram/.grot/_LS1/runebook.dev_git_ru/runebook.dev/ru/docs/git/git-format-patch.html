git-format-patch

Name

git-format-patch — Подготавливает патчи для отправки по электронной почте.

Synopsis

git format-patch [-k] [(-o|--output-directory) <dir> | --stdout]
                   [--no-thread | --thread[=<style>]]
                   [(--attach|--inline)[=<boundary>] | --no-attach]
                   [-s | --signoff]
                   [--signature=<signature> | --no-signature]
                   [--signature-file=<file>]
                   [-n | --numbered | -N | --no-numbered]
                   [--start-number <n>] [--numbered-files]
                   [--in-reply-to=<message id>] [--suffix=.<sfx>]
                   [--ignore-if-in-upstream] [--always]
                   [--cover-from-description=<mode>]
                   [--rfc] [--subject-prefix=<subject prefix>]
                   [(--reroll-count|-v) <n>]
                   [--to=<email>] [--cc=<email>]
                   [--[no-]cover-letter] [--quiet]
                   [--[no-]encode-email-headers]
                   [--no-notes | --notes[=<ref>]]
                   [--interdiff=<previous>]
                   [--range-diff=<previous> [--creation-factor=<percent>]]
                   [--filename-max-length=<n>]
                   [--progress]
                   [<common diff options>]
                   [ <since> | <revision range> ]

Description

Подготовьте каждую фиксацию без слияния с ее "patch" в одной "message" для каждой фиксации, отформатированной так, чтобы она напоминала почтовый ящик UNIX. Вывод этой команды удобен для отправки по электронной почте или для использования с git am .

"message" , созданный командой, состоит из трех частей:

  • Краткий заголовок метаданных, который начинается с From <commit> с фиксированной отметкой даты Mon Sep 17 00:00:00 2001 , чтобы помочь программам, таким как "file(1)" , распознать, что файл является результатом этой команды, поля, в которых записываются личность автора, дата автора и название изменения (взято из первый абзац сообщения журнала фиксации).

  • Второй и последующие абзацы сообщения журнала фиксации.

  • "patch" , который является выходом "diff -p --stat" (см. git-diff[1] ) между фиксацией и ее родителем.

Сообщение журнала и патч разделены линией из трех пунктиров.

Есть два способа указать, какие коммиты будут обрабатываться.

  1. Одиночный коммит <since> указывает, что коммиты, ведущие к кончику текущей ветки, которых нет в истории, приводят к выводу <since>.

  2. Общее выражение <диапазон версий> (см. раздел "SPECIFYING REVISIONS" в gitrevisions[7] ) означает фиксации в указанном диапазоне.

Первое правило имеет приоритет в случае одного <commit>. Чтобы применить второе правило, т. е. отформатировать все с начала истории до <commit>, используйте опцию --root : git format-patch --root <commit> . Если вы хотите отформатировать только сам <commit>, вы можете сделать это с помощью git format-patch -1 <commit> .

По умолчанию каждый выходной файл нумеруется последовательно, начиная с 1, и в качестве имени файла используется первая строка сообщения фиксации (обработанная для безопасности пути). При использовании опции --numbered-files имена выходных файлов будут только числами без добавления первой строки коммита. Имена выходных файлов выводятся на стандартный вывод, если не указана опция --stdout .

Если указан -o , выходные файлы создаются в <dir>. В противном случае они создаются в текущем рабочем каталоге. Путь по умолчанию можно установить с помощью опции конфигурации format.outputDirectory . Опция -o имеет приоритет над format.outputDirectory . Чтобы хранить исправления в текущем рабочем каталоге, даже если format.outputDirectory указывает на другое место, используйте -o . . Все компоненты каталога будут созданы.

По умолчанию предметом одного патча является "[PATCH] " , за которым следует объединение строк от сообщения о фиксации до первой пустой строки (см. раздел ОБСУЖДЕНИЕ git-commit[1] ).

При выводе нескольких патчей префикс субъекта будет "[PATCH n/m] " . Чтобы принудительно добавить 1/1 для одного патча, используйте -n . Чтобы исключить номера патчей из темы, используйте -N .

Если задан --thread , git-format-patch будет генерировать заголовки In-Reply-To и References , чтобы второе и последующие сообщения с исправлениями отображались как ответы на первое письмо; это также создает заголовок Message-ID для ссылки.

Options

-p
--no-stat

Генерируйте простые патчи без каких-либо различий.

-U<n>
--unified=<n>

Генерируйте различия с <n> строк контекста вместо обычных трех.

--output=<file>

Вывод в конкретный файл вместо стандартного вывода.

--output-indicator-new=<char>
--output-indicator-old=<char>
--output-indicator-context=<char>

Укажите символ, используемый для обозначения новых, старых или контекстных строк в сгенерированном патче. Обычно это + , - и '' соответственно.

--indent-heuristic

Включите эвристику, которая смещает границы различных фрагментов, чтобы патчи было легче читать. Это значение по умолчанию.

--no-indent-heuristic

Отключите эвристику отступов.

--minimal

Потратьте дополнительное время, чтобы убедиться, что создается минимально возможная разница.

--patience

Создайте разницу, используя алгоритм "patience diff" .

--histogram

Создайте разницу, используя алгоритм "histogram diff" .

--anchored=<text>

Создайте разницу, используя алгоритм "anchored diff" .

Эту опцию можно указать более одного раза.

Если строка существует как в источнике, так и в месте назначения, существует только один раз и начинается с этого текста, этот алгоритм пытается предотвратить ее появление в виде удаления или добавления в выходных данных. Он использует внутренний алгоритм "patience diff" .

--diff-algorithm={patience|minimal|histogram|myers}

Выберите алгоритм сравнения. Варианты следующие:

default, myers

Базовый жадный алгоритм сравнения. В настоящее время это значение по умолчанию.

minimal

Потратьте дополнительное время, чтобы убедиться, что создается минимально возможная разница.

patience

Используйте алгоритм "patience diff" при создании патчей.

histogram

Этот алгоритм расширяет алгоритм терпения до "support low-occurrence common elements" .

Например, если вы настроили для переменной diff.algorithm значение, отличное от значения по умолчанию, и хотите использовать значение по умолчанию, вам необходимо использовать опцию --diff-algorithm=default .

--stat[=<width>[,<name-width>[,<count>]]]

Создайте диффстат. По умолчанию столько места, сколько необходимо, будет использоваться для части имени файла, а остальное — для части графика. Максимальная ширина по умолчанию равна ширине терминала или 80 столбцам, если он не подключен к терминалу, и может быть переопределена <width> . Ширину части имени файла можно ограничить, указав другую ширину <name-width> после запятой или установив diff.statNameWidth=<width> . Ширину части графика можно ограничить, используя --stat-graph-width=<width> или установив diff.statGraphWidth=<width> . Использование --stat или --stat-graph-width влияет на все команды, генерирующие график статистики, а установка diff.statNameWidth или diff.statGraphWidth не влияет на git format-patch . Указав третий параметр <count> , вы можете ограничить вывод первыми строками <count> , а затем ... , если их больше.

Эти параметры также можно установить индивидуально с помощью --stat-width=<width> , --stat-name-width=<name-width> и --stat-count=<count> .

--compact-summary

Выведите сжатую сводку расширенной информации заголовка, такой как создание или удаление файлов ( "new" или "gone" , опционально "+l" , если это символическая ссылка) и изменения режима ( "+x" или "-x" для добавления или удаления исполняемого бита соответственно) в diffstat. Информация помещается между частью имени файла и частью графика. Подразумевается --stat .

--numstat

Похож на --stat , но показывает количество добавленных и удаленных строк в десятичной записи и путь без сокращений, чтобы сделать его более удобным для машины. Для двоичных файлов выводит два - вместо 0 0 .

--shortstat

Выведите только последнюю строку формата --stat , содержащую общее количество измененных файлов, а также количество добавленных и удаленных строк.

-X[<param1,param2,…​>]
--dirstat[=<param1,param2,…​>]

Выведите распределение относительного количества изменений для каждого подкаталога. Поведение --dirstat можно настроить, передав ему список параметров, разделенных запятыми. Значения по умолчанию контролируются переменной конфигурации diff.dirstat (см. git-config[1] ). Доступны следующие параметры:

changes

Вычислите номера dirstat, подсчитав строки, которые были удалены из источника или добавлены в место назначения. При этом игнорируется количество чистых движений кода внутри файла. Другими словами, перестановка строк в файле засчитывается не так сильно, как другие изменения. Это поведение по умолчанию, когда параметр не указан.

lines

Вычислите числа dirstat, выполнив обычный анализ различий на основе строк и суммируя количество удаленных/добавленных строк. (Для двоичных файлов вместо этого считайте куски по 64 байта, поскольку в двоичных файлах нет естественного понятия строк). Это более затратное поведение --dirstat , чем поведение changes , но оно учитывает перестановку строк в файле так же, как и другие изменения. Полученный результат соответствует тому, что вы получаете от других вариантов --*stat .

files

Вычислите числа dirstat, подсчитав количество измененных файлов. Каждый измененный файл одинаково учитывается при анализе dirstat. Это самое дешевое в вычислительном отношении поведение --dirstat , поскольку ему вообще не нужно просматривать содержимое файла.

cumulative

Подсчитайте также изменения в дочернем каталоге для родительского каталога. Обратите внимание, что при использовании cumulative сумма сообщаемых процентов может превышать 100%. Поведение по умолчанию (некумулятивное) можно указать с помощью параметра noncumulative .

<limit>

Целочисленный параметр определяет процент отсечки (по умолчанию 3%). Каталоги, на долю которых приходится меньше этого процента изменений, не отображаются в выводе.

Пример: Следующий код будет подсчитывать измененные файлы, игнорируя каталоги с менее чем 10% от общего количества измененных файлов и накапливая счетчики дочерних каталогов в родительских каталогах: --dirstat=files,10,cumulative .

--cumulative

Синоним --dirstat=кумулятивный

--dirstat-by-file[=<param1,param2>…​]

Синоним --dirstat=files,param1,param2…​

--summary

Выведите сжатую сводку расширенной информации заголовка, такой как создание, переименование и изменение режима.

--no-renames

Отключите обнаружение переименования, даже если в файле конфигурации указано значение по умолчанию.

--[no-]rename-empty

Использовать ли пустые BLOB-объекты в качестве источника переименования.

--full-index

Вместо первых нескольких символов отображайте полные имена объектов BLOB-объектов до и после изображения в строке "index" при создании выходных данных формата исправления.

--binary

В дополнение к --full-index выведите двоичный результат сравнения, который можно применить с помощью git-apply .

--abbrev[=<n>]

Вместо отображения полного 40-байтового шестнадцатеричного имени объекта в выходных данных формата diff-raw и строках заголовка diff-tree покажите самый короткий префикс длиной не менее шестнадцатеричных цифр <n> , который однозначно ссылается на объект. В формате вывода diff-patch --full-index имеет более высокий приоритет, т. е. если указан --full-index , полные имена больших двоичных объектов будут показаны независимо от --abbrev . Нестандартное количество цифр можно указать с помощью --abbrev=<n> .

-B[<n>][/<m>]
--break-rewrites[=[<n>][/<m>]]

Разбейте полные изменения перезаписи на пары удаления и создания. Это служит двум целям:

Это влияет на то, как изменение, которое представляет собой полную перезапись файла, происходит не как серия удалений и вставок, смешанных с очень немногими строками, которые текстуально совпадают с контекстом, а как однократное удаление всего старого с последующим однократная вставка всего нового, и число m управляет этим аспектом опции -B (по умолчанию 60%). -B/70% указывает, что в результате должно оставаться менее 30% оригинала, чтобы Git считал его полной перезаписью (т. е. в противном случае результирующий патч будет представлять собой серию удалений и вставок, смешанных вместе с контекстными строками).

При использовании с -M полностью переписанный файл также считается источником переименования (обычно -M считает источником переименования только исчезнувший файл), а число n управляет этим аспектом опции -B. (по умолчанию 50%). -B20% указывает, что изменение с добавлением и удалением, составляющее 20% или более от размера файла, может быть выбрано в качестве возможного источника переименования в другой файл.

-M[<n>]
--find-renames[=<n>]

Обнаружение переименований. Если указан n , это порог индекса сходства (т.е. количество добавлений/удалений по сравнению с размером файла). Например, -M90% означает, что Git должен рассматривать пару удаления/добавления как переименование, если более 90% файла не изменилось. Без знака % число следует читать как дробь с десятичной точкой перед ней. Т.е. -M5 становится равным 0,5 и, таким образом, совпадает с -M50% . Аналогично, -M05 — это то же самое, что и -M5% . Чтобы ограничить обнаружение точными переименованиями, используйте -M100% . Индекс сходства по умолчанию составляет 50%.

-C[<n>]
--find-copies[=<n>]

Обнаружение копий, а также переименований. См. также --find-copies-harder . Если указан n , он имеет то же значение, что и -M<n> .

--find-copies-harder

По соображениям производительности опция -C по умолчанию находит копии только в том случае, если исходный файл копии был изменен в том же наборе изменений. Этот флаг заставляет команду проверять неизмененные файлы как кандидатов на источник копирования. Это очень дорогостоящая операция для больших проектов, поэтому используйте ее с осторожностью. Предоставление более одной опции -C имеет тот же эффект.

-D
--irreversible-delete

При удалении пропустите прообраз, т.е. печатайте только заголовок, но не разницу между прообразом и /dev/null . Полученный патч не предназначен для применения с patch или git apply ; это исключительно для людей, которые хотят просто сконцентрироваться на просмотре текста после внесения изменений. Кроме того, в выводе явно недостаточно информации, чтобы применить такой патч в обратном порядке, даже вручную, отсюда и название опции.

При использовании вместе с -B опустите также прообраз в части удаления пары удаление/создание.

-l<num>

Опции -M и -C включают в себя некоторые предварительные шаги, которые позволяют дешево обнаружить подмножества переименований/копий, за которыми следует исчерпывающая запасная часть, которая сравнивает все оставшиеся непарные адресаты со всеми соответствующими источниками. (Для переименований актуальны только оставшиеся непарные источники; для копий актуальны все исходные источники.) Для N источников и мест назначения эта исчерпывающая проверка равна O(N^2). Этот параметр предотвращает запуск исчерпывающей части обнаружения переименования/копирования, если количество задействованных файлов-источников/назначений превышает указанное число. По умолчанию — diff.renameLimit. Обратите внимание, что значение 0 считается неограниченным.

-O<orderfile>

Управляйте порядком появления файлов в выходных данных. Это переопределяет переменную конфигурации diff.orderFile (см. git-config[1] ). Чтобы отменить diff.orderFile , используйте -O/dev/null .

Порядок вывода определяется порядком шаблонов glob в <orderfile>. Все файлы с путями, соответствующими первому шаблону, выводятся первыми, все файлы с путями, соответствующими второму шаблону (но не первому), выводятся затем и так далее. Все файлы, имена путей которых не соответствуют никакому шаблону, выводятся последними, как если бы в конце файла был неявный шаблон соответствия всем. Если несколько имен путей имеют одинаковый ранг (они соответствуют одному и тому же шаблону, но не более ранним шаблонам), их порядок вывода относительно друг друга является обычным порядком.

<orderfile> анализируется следующим образом:

  • Пустые строки игнорируются, поэтому их можно использовать в качестве разделителей для удобства чтения.

  • Строки, начинающиеся с хеша ( " # " ), игнорируются, поэтому их можно использовать для комментариев. Добавьте обратную косую черту ( " \ " ) в начало шаблона, если он начинается с решетки.

  • Каждая вторая строка содержит один шаблон.

Шаблоны имеют тот же синтаксис и семантику, что и шаблоны, используемые для fnmatch(3) без флага FNM_PATHNAME, за исключением того, что путь также соответствует шаблону, если удаление любого количества окончательных компонентов пути соответствует шаблону. Например, шаблон " foo*bar " соответствует " fooasdfbar " и " foo/bar/baz/asdf " , но не соответствует " foobarx " .

--skip-to=<file>
--rotate-to=<file>

Удалите файлы перед указанным <file> из вывода (т. е. skip to ) или переместите их в конец вывода (т. е. rotate to ). Эти параметры были придуманы в первую очередь для использования команды git difftool и в противном случае могут оказаться бесполезными.

--relative[=<path>]
--no-relative

При запуске из подкаталога проекта с помощью этой опции можно указать исключить изменения за пределами каталога и показать пути относительно него. Если вы не находитесь в подкаталоге (например, в пустом репозитории), вы можете указать, относительно какого подкаталога следует выполнять выходные данные, указав <path> в качестве аргумента. --no-relative можно использовать для отмены как опции конфигурации diff.relative , так и предыдущей версии --relative .

-a
--text

Рассматривать все файлы как текст.

--ignore-cr-at-eol

При сравнении игнорируйте возврат каретки в конце строки.

--ignore-space-at-eol

Игнорировать изменения в пробелах в конце EOL.

-b
--ignore-space-change

Игнорировать изменения в количестве пробелов. При этом пробелы в конце строки игнорируются, а все остальные последовательности из одного или нескольких символов пробелов считаются эквивалентными.

-w
--ignore-all-space

Игнорируйте пробелы при сравнении строк. При этом различия игнорируются, даже если в одной строке есть пробелы, а в другой их нет.

--ignore-blank-lines

Игнорировать изменения, все строки которых пусты.

-I<regex>
--ignore-matching-lines=<regex>

Игнорировать изменения, все строки которых соответствуют <regex>. Эту опцию можно указать более одного раза.

--inter-hunk-context=<lines>

Показать контекст между разными фрагментами до указанного количества строк, тем самым объединяя фрагменты, расположенные близко друг к другу. По умолчанию diff.interHunkContext или 0, если опция конфигурации не установлена.

-W
--function-context

Показывать всю функцию в виде контекстных строк для каждого изменения. Имена функций определяются так же, как git diff обрабатывает заголовки фрагментов патчей (см. Defining a custom hunk-header в gitattributes[5] ).

--ext-diff

Разрешить выполнение внешнего помощника сравнения. Если вы установили внешний драйвер различий с gitattributes[5] , вам нужно использовать эту опцию с git-log[1] и его друзьями.

--no-ext-diff

Запретить внешние драйверы различий.

--textconv
--no-textconv

Разрешить (или запретить) запуск внешних фильтров преобразования текста при сравнении двоичных файлов. Подробности см. в gitattributes[5] . Поскольку фильтры textconv обычно представляют собой одностороннее преобразование, полученный результат пригоден для восприятия человеком, но не может быть применен. По этой причине фильтры textconv включены по умолчанию только для git-diff[1] и git-log[1] , но не для git-format-patch[1] или команд diff сантехники.

--ignore-submodules[=<when>]

Игнорировать изменения в подмодулях при генерации различий. <When> может быть "none" , "untracked" , "dirty" или "all" , который используется по умолчанию. При использовании "none" субмодуль будет считаться измененным, если он либо содержит неотслеживаемые или измененные файлы, либо его HEAD отличается от фиксации, записанной в суперпроекте, и может использоваться для переопределения любых настроек параметра ignore в git-config[1] или gitmodules[5] . При использовании "untracked" субмодули не считаются «грязными», если они содержат только неотслеживаемый контент (но они все равно сканируются на предмет измененного контента). Использование "dirty" игнорирует все изменения в рабочем дереве подмодулей, отображаются только изменения коммитов, хранящихся в суперпроекте (так было до версии 1.7.0). Использование "all" скрывает все изменения в субмодулях.

--src-prefix=<prefix>

Показывать данный префикс источника вместо "a/" .

--dst-prefix=<prefix>

Показывать указанный префикс пункта назначения вместо "b/" .

--no-prefix

Не показывать префикс источника или назначения.

--default-prefix

Используйте префиксы источника и назначения по умолчанию ( "a/" и "b/" ). Обычно это значение по умолчанию, но его можно использовать для переопределения конфигурации, например diff.noprefix .

--line-prefix=<prefix>

Добавьте дополнительный префикс к каждой строке вывода.

--ita-invisible-in-index

По умолчанию записи, добавленные "git add -N" , отображаются как существующий пустой файл в "git diff" и новый файл в "git diff --cached" . Эта опция делает запись видимой как новый файл в "git diff" и несуществующей в "git diff --cached" . Эту опцию можно вернуть с помощью --ita-visible-in-index . Оба варианта являются экспериментальными и могут быть удалены в будущем.

Более подробное объяснение этих распространенных опций см. также в gitdiffcore[7] .

-<n>

Подготовьте патчи из самых верхних коммитов <n>.

-o <dir>
--output-directory <dir>

Используйте <dir> для хранения полученных файлов вместо текущего рабочего каталога.

-n
--numbered

Вывод имени в формате [PATCH n/m] , даже с одним патчем.

-N
--no-numbered

Вывод имени в формате [PATCH] .

--start-number <n>

Начните нумерацию патчей с <n> вместо 1.

--numbered-files

Имена выходных файлов будут представлять собой простую числовую последовательность без добавления первой строки коммита по умолчанию.

-k
--keep-subject

Не удаляйте и не добавляйте [PATCH] из первой строки сообщения журнала фиксации.

-s
--signoff

Добавьте трейлер Signed-off-by к сообщению о фиксации, используя свою личность коммиттера. Дополнительную информацию см. в разделе «Выписка» в git-commit[1] .

--stdout

Выводите все коммиты на стандартный вывод в формате mbox вместо создания файла для каждого.

--attach[=<boundary>]

Создайте многочастное/смешанное вложение, первой частью которого является сообщение о фиксации, а во второй части — сам патч, с помощью Content-Disposition: attachment .

--no-attach

Отключите создание вложения, переопределив параметр конфигурации.

--inline[=<boundary>]

Создайте многочастное/смешанное вложение, первой частью которого является сообщение о фиксации, а во второй части — сам патч, с помощью Content-Disposition: inline .

--thread[=<style>]
--no-thread

Управляет добавлением заголовков In-Reply-To и References , чтобы второе и последующие сообщения отображались как ответы на первое. Также управляет генерацией заголовка Message-ID для ссылки.

Необязательный аргумент <style> может быть shallow или deep . Потоковая обработка shallow делает каждое письмо ответом на заголовок серии, где заголовок выбирается из сопроводительного письма, --in-reply-to и первого исправленного письма в этом порядке. Поточность deep делает каждое письмо ответом на предыдущее.

По умолчанию используется --no-thread , если не установлена ​​конфигурация format.thread . --thread без аргумента эквивалентен --thread=shallow .

Имейте в виду, что по умолчанию git send-email автоматически отправляет электронные письма. Если вы хотите, чтобы git format-patch позаботился о многопоточной обработке, вам нужно убедиться, что потоковая обработка отключена для git send-email .

--in-reply-to=<message id>

Сделайте так, чтобы первое письмо (или все письма с --no-thread ) отображалось как ответ на заданный <идентификатор сообщения>, что позволяет избежать прерывания потоков для предоставления новой серии исправлений.

--ignore-if-in-upstream

Не включайте патч, соответствующий коммиту в <until>..<since>. При этом будут проверены все патчи, доступные из <since>, но не из <until>, и сравнены их с генерируемыми патчами, при этом любой соответствующий патч будет игнорироваться.

--always

Включите исправления для коммитов, которые не вносят никаких изменений и которые по умолчанию опущены.

--cover-from-description=<mode>

Определяет, какие части сопроводительного письма будут автоматически заполнены с использованием описания филиала.

Если <mode> — message или default , тема сопроводительного письма будет заполнена текстом-заполнителем. В тексте сопроводительного письма будет указано описание филиала. Это режим по умолчанию, когда не указаны ни конфигурация, ни параметр командной строки.

Если <mode> — subject , первый абзац описания ветки будет заполнен темой сопроводительного письма. Оставшаяся часть описания будет заполнена основной частью сопроводительного письма.

Если <mode> — auto , если первый абзац описания ветки больше 100 байт, то режим будет message , иначе будет использоваться subject .

Если <mode> — none , и тема, и текст сопроводительного письма будут заполнены текстом-заполнителем.

--description-file=<file>

Используйте содержимое <file> вместо описания ветки для создания сопроводительного письма.

--subject-prefix=<subject prefix>

Вместо стандартного префикса [PATCH] в теме письма используйте [<subject prefix>] . Его можно использовать для названия серии патчей и комбинировать с опцией --numbered .

Переменную конфигурации format.subjectPrefix также можно использовать для настройки префикса субъекта для применения к данному репозиторию для всех исправлений. Это часто бывает полезно в списках рассылки, которые получают исправления для нескольких репозиториев, и может использоваться для устранения неоднозначности исправлений (например, со значением "PATCH my-project" ).

--filename-max-length=<n>

Вместо стандартных 64 байтов сократите сгенерированные имена выходных файлов примерно до <n> байт (слишком короткое значение будет автоматически увеличено до разумной длины). По умолчанию используется значение переменной конфигурации format.filenameMaxLength или 64, если она не настроена.

--rfc

Добавляет "RFC" к префиксу субъекта (по умолчанию создается "RFC PATCH" ). RFC означает "Request For Comments" ; используйте это при отправке экспериментального патча для обсуждения, а не для применения.

-v <n>
--reroll-count=<n>

Отметить серию как <n>-ю итерацию темы. К именам выходных файлов добавляется v<n> , а к префиксу субъекта (по умолчанию "PATCH" , но настраивается с помощью опции --subject-prefix ) добавляется ` v<n>`. Например, --reroll-count=4 может создать файл v4-0001-add-makefile.patch , содержащий "Subject: [PATCH v4 1/20] Add makefile" . <n> не обязательно должен быть целым числом (например, "--reroll-count=4.4" или "--reroll-count=4rev2" разрешены), но недостатком использования такого счетчика повторного чередования является то, что диапазон-diff/interdiff с предыдущей версией не указывает, какая именно версия является новой итерацией. по сравнению с.

--to=<email>

Добавьте заголовок To: к заголовкам электронной почты. Это дополнение к любым настроенным заголовкам и может использоваться несколько раз. Инвертированная форма --no-to отбрасывает все добавленные на данный момент заголовки To: (из конфигурации или командной строки).

--cc=<email>

Добавьте заголовок Cc: к заголовкам электронной почты. Это дополнение к любым настроенным заголовкам и может использоваться несколько раз. Инвертированная форма --no-cc отбрасывает все добавленные на данный момент заголовки Cc: (из конфигурации или командной строки).

--from
--from=<ident>

Используйте ident в заголовке From: каждого электронного письма о фиксации. Если идентификатор автора фиксации не идентичен по тексту предоставленному ident , поместите заголовок From: в тело сообщения с указанием исходного автора. Если ident не указан, используйте идентификатор коммиттера.

Обратите внимание, что эта опция полезна только в том случае, если вы действительно отправляете электронные письма и хотите идентифицировать себя как отправителя, но сохранить исходного автора (и git am правильно определит заголовок в теле письма). Также обратите внимание, что git send-email уже выполняет это преобразование за вас, и эту опцию не следует использовать, если вы передаете результат в git send-email .

--[no-]force-in-body-from

Если отправитель электронного письма указан с помощью параметра --from , по умолчанию в тексте сообщения "From:" для идентификации реального автора фиксации добавляется вверху сообщения журнала фиксации, если отправитель отличается от автора. При использовании этой опции внутренний "From:" добавляется, даже если отправитель и автор имеют одно и то же имя и адрес, что может помочь, если программное обеспечение списка рассылки искажает личность отправителя. По умолчанию используется значение переменной конфигурации format.forceInBodyFrom .

--add-header=<header>

Добавьте произвольный заголовок в заголовки электронной почты. Это дополнение к любым настроенным заголовкам и может использоваться несколько раз. Например, --add-header="Organization: git-foo" . Отрицательная форма --no-add-header отбрасывает все заголовки ( To: , Cc: и пользовательские), добавленные на данный момент из конфигурации или командной строки.

--[no-]cover-letter

В дополнение к патчам создайте файл сопроводительного письма, содержащий описание ветки, краткий журнал и общую статистику различий. Вы можете заполнить описание в файле перед его отправкой.

--encode-email-headers
--no-encode-email-headers

Кодируйте заголовки электронной почты, содержащие символы, отличные от ASCII, с помощью "Q-encoding" (описано в RFC 2047) вместо дословного вывода заголовков. По умолчанию используется значение переменной конфигурации format.encodeEmailHeaders .

--interdiff=<previous>

В качестве помощи рецензенту вставьте в сопроводительное письмо вставку interdiff или в качестве комментария к единственному патчу из серии из 1 патча, показывающую различия между предыдущей версией серии патчей и серией, форматируемой в данный момент. previous — это отдельная версия, называющая наконечник предыдущей серии, который имеет общую основу с форматируемой серией (например, git format-patch --cover-letter --interdiff=feature/v1 -3 feature/v2 ).

--range-diff=<previous>

В качестве помощи рецензенту вставьте разницу диапазона (см. git-range-diff[1] ) в сопроводительное письмо или в качестве комментария к единственному патчу из серии из 1 патча, показывая различия между предыдущей версией серии патчей и серией, форматируемой в данный момент. . previous может представлять собой одну редакцию, обозначающую кончик предыдущей серии, если она имеет общую основу с форматируемой серией (например, git format-patch --cover-letter --range-diff=feature/v1 -3 feature/v2 ), или диапазон ревизий, если две версии серии не пересекаются (например, git format-patch --cover-letter --range-diff=feature/v1~3..feature/v1 -3 feature/v2 ).

Обратите внимание, что параметры diff, передаваемые команде, влияют на то, как создается основной продукт format-patch , и они не передаются базовому оборудованию range-diff , используемому для создания материала сопроводительного письма (это может измениться в будущем).

--creation-factor=<percent>

При использовании с --range-diff настройте эвристику, которая сопоставляет фиксации между предыдущей и текущей сериями исправлений, путем настройки коэффициента неточности стоимости создания/удаления. Подробности см. в git-range-diff[1] ).

--notes[=<ref>]
--no-notes

Добавьте примечания (см. git-notes[1] ) для фиксации после трехпунктирной линии.

Ожидаемый вариант использования этого — написать вспомогательное объяснение для фиксации, которое не принадлежит собственно сообщению журнала фиксации, и включить его в отправку исправления. Хотя можно просто написать эти пояснения после запуска format-patch , но перед отправкой, сохранение их в виде примечаний Git позволяет сохранять их между версиями серии патчей (но для использования этого рабочего процесса см. обсуждение параметров конфигурации notes.rewrite в git-notes[1] ).

По умолчанию используется --no-notes , если не установлена ​​конфигурация format.notes .

--[no-]signature=<signature>

Добавьте подпись к каждому созданному сообщению. Согласно RFC 3676 подпись отделяется от тела линией с символом '--'. Если опция подписи опущена, по умолчанию для подписи используется номер версии Git .

--signature-file=<file>

Работает так же, как --signature, за исключением того, что подпись считывается из файла.

--suffix=.<sfx>

Вместо использования .patch в качестве суффикса для сгенерированных имен файлов используйте указанный суффикс. Распространенной альтернативой является --suffix=.txt . Если оставить это поле пустым, суффикс .patch будет удален.

Обратите внимание, что ведущим символом не обязательно должна быть точка; например, вы можете использовать --suffix=-patch , чтобы получить 0001-description-of-my-change-patch .

-q
--quiet

Не выводите имена сгенерированных файлов в стандартный вывод.

--no-binary

Не выводить содержимое изменений в двоичных файлах, вместо этого отображать уведомление об изменении этих файлов. Патчи, созданные с использованием этой опции, не могут быть применены должным образом, но они по-прежнему полезны для проверки кода.

--zero-commit

Выведите нулевой хэш в заголовке From каждого патча вместо хеша коммита.

--[no-]base[=<commit>]

Запишите информацию базового дерева, чтобы определить состояние, к которому относится серия исправлений. Подробности см. в разделе «ИНФОРМАЦИЯ О БАЗОВОМ ДЕРЕВЕ» ниже. Если <commit> — "auto" , базовая фиксация выбирается автоматически. Опция --no-base имеет приоритет перед конфигурацией format.useAutoBase .

--root

Рассматривайте аргумент ревизии как <диапазон ревизий>, даже если это всего лишь одна фиксация (которая обычно рассматривается как <since>). Обратите внимание, что корневые коммиты, включенные в указанный диапазон, всегда форматируются как патчи создания, независимо от этого флага.

--progress

Показывать отчеты о ходе выполнения stderr по мере создания исправлений.

Configuration

Вы можете указать дополнительные строки заголовков почты, которые будут добавляться к каждому сообщению, значения по умолчанию для префикса темы и суффикса файла, нумеровать исправления при выводе более одного исправления, добавить заголовки "To:" или "Cc:" , настроить вложения, изменить каталог вывода исправлений и выйти из системы. патчи с переменными конфигурации.

[format]
        headers = "Organization: git-foo\n"
        subjectPrefix = CHANGE
        suffix = .txt
        numbered = auto
        to = <email>
        cc = <email>
        attach [ = mime-boundary-string ]
        signOff = true
        outputDirectory = <directory>
        coverLetter = auto
        coverFromDescription = auto

Discussion

Патч, созданный git format-patch , имеет формат почтового ящика UNIX с фиксированной отметкой времени "magic" , указывающей, что файл выводится из format-patch, а не из реального почтового ящика, например:

From 8f72bad1baf19a53459661343e21d6491c3908d3 Mon Sep 17 00:00:00 2001
From: Tony Luck <[email protected]>
Date: Tue, 13 Jul 2010 11:42:54 -0700
Subject: [PATCH] =?UTF-8?q?[IA64]=20Put=20ia64=20config=20files=20on=20the=20?=
 =?UTF-8?q?Uwe=20Kleine-K=C3=B6nig=20diet?=
MIME-Version: 1.0
Content-Type: text/plain; charset=UTF-8
Content-Transfer-Encoding: 8bit

arch/arm config files were slimmed down using a python script
(See commit c2330e286f68f1c408b4aa6515ba49d57f05beae comment)

Do the same for ia64 so we can have sleek & trim looking
...

Обычно оно помещается в папку черновиков MUA, редактируется для добавления своевременных комментариев, которые не должны появляться в журнале изменений после трех тире, а затем отправляется как сообщение, тело которого в нашем примере начинается с "arch/arm config files were…​" . На принимающей стороне читатели могут сохранять интересные патчи в почтовом ящике UNIX и применять их с помощью git-am[1] .

Если патч является частью текущего обсуждения, патч, созданный git format-patch , можно настроить, чтобы использовать преимущества функции git am --scissors . После вашего ответа на обсуждение идет строка, состоящая исключительно из " -- >8 -- " (ножницы и перфорация), за которой следует патч с удаленными ненужными полями заголовка:

...
> So we should do such-and-such.

Makes sense to me.  How about this patch?

-- >8 --
Subject: [IA64] Put ia64 config files on the Uwe Kleine-König diet

arch/arm config files were slimmed down using a python script
...

При отправке патча таким способом чаще всего вы отправляете собственный патч, поэтому помимо маркера " From $SHA1 $magic_timestamp " вам следует опустить строки From: и Date: из файла патча. Название патча, скорее всего, будет отличаться от темы обсуждения, на которое патч является ответом, поэтому вполне вероятно, что вы захотите сохранить строку «Тема:», как в примере выше.

Проверка наличия повреждений патча

Многие почтовые программы, если они не настроены должным образом, повреждают пробелы. Вот два распространенных типа коррупции:

  • Пустые контекстные строки, не содержащие пробелов any .

  • Непустые строки контекста, в начале которых имеется один лишний пробел.

Один из способов проверить правильность настройки MUA:

  • Отправьте патч себе точно так же, как вы это делаете, за исключением строк «Кому:» и «Копия:», которые не содержат список и адрес сопровождающего.

  • Сохраните этот патч в файл в формате почтового ящика UNIX. Назовите это, скажем, патчем.

  • Apply it:

    $ git fetch <project> master:test-apply
$ git switch test-apply
$ git restore --source=HEAD --staged --worktree :/
$ git am a.patch

Если он применяется неправильно, причины могут быть разные.

  • Сам патч наносится неаккуратно. Это bad , но он не имеет особого отношения к вашему MUA. В этом случае вы можете перебазировать патч с помощью git-rebase[1] перед его регенерацией.

  • MUA повредил ваш патч; "am" будет жаловаться, что патч не применяется. Загляните в подкаталог .git/rebase-apply/ и посмотрите, что содержит файл patch , и проверьте наличие распространенных шаблонов повреждения, упомянутых выше.

  • При этом проверьте также файлы info и final-commit . Если то, что находится в final-commit , не совсем то, что вы хотели бы видеть в сообщении журнала фиксации, весьма вероятно, что получателю придется вручную редактировать сообщение журнала при применении вашего патча. Такие вещи, как "Hi, this is my first patch.\n" , в электронном письме с исправлением должны идти после трехпунктирной линии, которая сигнализирует об окончании сообщения о фиксации.

Mua-specific hints

Вот несколько советов о том, как успешно отправлять патчи в режиме онлайн с помощью различных почтовых программ.

GMail

В GMail нет возможности отключить перенос строк в веб-интерфейсе, поэтому он будет искажать любые отправляемые вами электронные письма. Однако вы можете использовать "git send-email" и отправлять свои исправления через SMTP-сервер GMail или использовать любой почтовый клиент IMAP для подключения к серверу IMAP Google и пересылки электронных писем через него.

Советы по использованию git send-email для отправки исправлений через SMTP-сервер GMail см. в разделе ПРИМЕР документа git-send-email[1] .

Советы по отправке с использованием интерфейса IMAP см. в разделе ПРИМЕР документа git-imap-send[1] .

Thunderbird

По умолчанию Thunderbird как упаковывает электронные письма, так и помечает их как format=flowed , что делает полученное электронное письмо непригодным для использования Git.

Существует три разных подхода: использовать надстройку, чтобы отключить перенос строк, настроить Thunderbird так, чтобы не искажать патчи, или использовать внешний редактор, чтобы Thunderbird не искажал патчи.

Подход №1 (дополнение)

Установите надстройку Toggle Word Wrap, доступную на https://addons.mozilla.org/thunderbird/addon/toggle-word-wrap/ . Она добавляет пункт меню "Enable Word Wrap" в меню композитора "Options" , который вы можете отметить галочкой. Теперь вы можете составить сообщение, как обычно (вырезать + вставить, git format-patch | git imap-send и т. д.), но вам придется вручную вставлять разрывы строк в любой вводимый вами текст.

Подход №2 (конфигурация)

Three steps:

  1. Настройте состав своего почтового сервера в виде обычного текста: «Редактировать… Настройки учетной записи… Состав и адресация», снимите флажок "Compose Messages in HTML" .

  2. Настройте общее окно композиции так, чтобы оно не переносилось.

    В Thunderbird 2: Edit..Preferences..Composition перенесите текстовые сообщения в 0.

    В Thunderbird 3: Редактировать..Настройки..Дополнительно..Редактор конфигурации. Найдите "mail.wrap_long_lines" . Переключите его, чтобы убедиться, что для него установлено значение false . Также найдите "mailnews.wraplength" и установите значение 0.

  3. Отключите использование format=flowed: Edit..Preferences..Advanced..Config Editor. Найдите "mailnews.send_plaintext_flowed" . Переключите его, чтобы убедиться, что для него установлено значение false .

После этого вы сможете составить электронное письмо так, как вы это делаете в противном случае (вырезать + вставить, git format-patch | git imap-send и т. д.), и исправления не будут повреждены.

Подход №3 (внешний редактор)

Необходимы следующие расширения Thunderbird: AboutConfig от https://mjg.github.io/AboutConfig/ и Внешний редактор от https://globs.org/articles.php?lng=en&pg=8 .

  1. Подготовьте патч в виде текстового файла, используя выбранный вами метод.

  2. Прежде чем открыть окно создания сообщения, используйте «Правка» → «Настройки учетной записи», чтобы снять флажок с настройки "Compose messages in HTML format" на панели "Composition & Addressing" учетной записи, которая будет использоваться для отправки патча.

  3. В главном окне Thunderbird before вы открываете окно создания патча, используете «Инструменты» → «about:config», чтобы установить следующие указанные значения:

            mailnews.send_plaintext_flowed  => false
        mailnews.wraplength             => 0

  4. Откройте окно создания сообщения и щелкните значок внешнего редактора.

  5. В окне внешнего редактора прочитайте файл патча и закройте редактор обычным способом.

Примечание: возможно, можно выполнить шаг 2 с помощью about:config и следующих настроек, но никто еще не пробовал.

        mail.html_compose                       => false
        mail.identity.default.compose_html      => false
        mail.identity.id?.compose_html          => false

В contrib/thunderbird-patch-inline есть скрипт, который поможет вам легко включать патчи в Thunderbird. Чтобы использовать его, выполните описанные выше шаги, а затем используйте скрипт в качестве внешнего редактора.

KMail

Это должно помочь вам отправлять исправления прямо с помощью KMail.

  1. Подготовьте патч в виде текстового файла.

  2. Нажмите «Новая почта».

  3. Перейдите под "Options" в окне Composer и убедитесь, что "Word wrap" не установлен.

  4. Используйте Сообщение → Вставить файл… и вставьте патч.

  5. Вернитесь в окно создания сообщения: добавьте в сообщение любой другой текст, заполните поля адреса и темы и нажмите «Отправить».

Информация о базовом дереве

Информационный блок базового дерева используется сопровождающими или сторонними тестировщиками, чтобы знать точное состояние, к которому относится серия исправлений. Он состоит из base commit , который является хорошо известным коммитом, который является частью стабильной части истории проекта, над которой работают все остальные, и нуля или более prerequisite patches , которые представляют собой хорошо известные патчи в работе, которые еще не являются частью base commit , которые необходимо применить поверх base commit в топологическом порядке, прежде чем можно будет применить исправления.

base commit отображается как "base-commit: " , за которым следует 40-значное имя объекта фиксации. prerequisite patch отображается как "prerequisite-patch-id: " , за которым следует 40-шестнадцатеричный patch id , который можно получить, передав патч через команду git patch-id --stable .

Представьте, что поверх коммита P public вы применили известные патчи X, Y и Z от кого-то другого, а затем создали серию из трех патчей A, B, C, история будет такой:

---P---X---Y---Z---A---B---C

В случае git format-patch --base=P -3 C (или его вариантов, например, --cover-letter или использования Z..C вместо -3 C для указания диапазона) информационный блок базового дерева отображается в конце первого сообщения, выводимого командой (либо первого патча, либо сопроводительного письма). , так:

base-commit: P
prerequisite-patch-id: X
prerequisite-patch-id: Y
prerequisite-patch-id: Z

Для нелинейной топологии, такой как

---P---X---A---M---C
    \         /
     Y---Z---B

Вы также можете использовать git format-patch --base=P -3 C для создания патчей для A, B и C, а идентификаторы P, X, Y, Z добавляются в конце первого сообщения.

Если установить --base=auto в cmdline, он автоматически вычислит базовую фиксацию как базу слияния подсказки ветви удаленного отслеживания и диапазона ревизий, указанного в cmdline. Для локального филиала вам необходимо настроить отслеживание удаленного филиала с помощью git branch --set-upstream-to , прежде чем использовать эту опцию.

Examples

  • Извлеките коммиты между ревизиями R1 и R2 и примените их поверх текущей ветки, используя git am для их отбора:

    $ git format-patch -k --stdout R1..R2 | мерзавец утра -3 -к

  • Извлеките все коммиты, которые находятся в текущей ветке, но не в исходной ветке:

    $ git format-patch origin

    Для каждого коммита в текущем каталоге создается отдельный файл.

  • Извлеките все коммиты, которые привели к origin с момента создания проекта:

    $ git format-patch --root origin

  • То же, что и предыдущий:

    $ git format-patch -M -B origin

    Кроме того, он обнаруживает и обрабатывает переименования, а также интеллектуально выполняет перезапись для создания патча переименования. Патч переименования уменьшает объем выводимого текста и, как правило, упрощает его просмотр. Обратите внимание, что программы, отличные от Git и "patch" , не поддерживают переименование исправлений, поэтому используйте его только в том случае, если вы знаете, что получатель использует Git для применения вашего исправления.

  • Извлеките три самых верхних коммита из текущей ветки и отформатируйте их как патчи для отправки по электронной почте:

    $ git format-patch -3

Caveats

Обратите внимание, что format-patch будет пропускать коммиты слияния из выходных данных, даже если они являются частью запрошенного диапазона. Простой "patch" не содержит достаточно информации, чтобы принимающая сторона могла воспроизвести ту же фиксацию слияния.
