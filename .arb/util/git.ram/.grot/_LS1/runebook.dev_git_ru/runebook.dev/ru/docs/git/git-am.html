git-am

Name

git-am — применить серию патчей из почтового ящика

Synopsis

git am [--signoff] [--keep] [--[no-]keep-cr] [--[no-]utf8] [--no-verify]
         [--[no-]3way] [--interactive] [--committer-date-is-author-date]
         [--ignore-date] [--ignore-space-change | --ignore-whitespace]
         [--whitespace=<action>] [-C<n>] [-p<n>] [--directory=<dir>]
         [--exclude=<path>] [--include=<path>] [--reject] [-q | --quiet]
         [--[no-]scissors] [-S[<keyid>]] [--patch-format=<format>]
         [--quoted-cr=<action>]
         [--empty=(stop|drop|keep)]
         [(<mbox> | <Maildir>)…​]
git am (--continue | --skip | --abort | --quit | --show-current-patch[=(diff|raw)] | --allow-empty)

Description

Разделяет почтовые сообщения в почтовом ящике на сообщения журнала фиксации, информацию об авторстве и исправления и применяет их к текущей ветке. Вы можете думать об этом как об обратной операции git-format-patch[1] , запущенной на ветке с прямой историей без слияний.

Options

(<mbox>|<Maildir>)…

Список файлов почтового ящика для чтения патчей. Если вы не укажете этот аргумент, команда считывает данные со стандартного ввода. Если вы предоставите каталоги, они будут рассматриваться как Maildirs.

-s
--signoff

Добавьте трейлер Signed-off-by к сообщению о фиксации, используя свою личность коммиттера. Дополнительную информацию см. в разделе «Выписка» в git-commit[1] .

-k
--keep

Передайте флаг -k в git mailinfo (см. git-mailinfo[1] ).

--keep-non-patch

Передайте флаг -b в git mailinfo (см. git-mailinfo[1] ).

--[no-]keep-cr

При использовании --keep-cr вызовите git mailsplit (см. git-mailsplit[1] ) с той же опцией, чтобы предотвратить удаление CR в конце строк. Переменную конфигурации am.keepcr можно использовать для указания поведения по умолчанию. --no-keep-cr полезен для переопределения am.keepcr .

-c
--scissors

Удалите все внутри корпуса перед ножницеобразной линией (см. git-mailinfo[1] ). Может быть активирован по умолчанию с помощью переменной конфигурации mailinfo.scissors .

--no-scissors

Не обращайте внимания на ножничные линии (см. git-mailinfo[1] ).

--quoted-cr=<action>

Этот флаг будет передан git mailinfo (см. git-mailinfo[1] ).

--empty=(stop|drop|keep)

По умолчанию или если для этого параметра установлено значение stop , команда выдает ошибку при входящем сообщении электронной почты, в котором отсутствует исправление, и останавливается в середине текущего сеанса утра. Если для этого параметра установлено значение drop , вместо этого пропустите такое сообщение электронной почты. Если для этого параметра установлено значение keep , создайте пустую фиксацию, записывая содержимое сообщения электронной почты в журнал.

-m
--message-id

Передайте флаг -m в git mailinfo (см. git-mailinfo[1] ), чтобы заголовок Message-ID был добавлен в сообщение фиксации. Переменную конфигурации am.messageid можно использовать для указания поведения по умолчанию.

--no-message-id

Не добавляйте заголовок Message-ID в сообщение фиксации. no-message-id полезен для замены am.messageid .

-q
--quiet

Будь спокоен. Печатайте только сообщения об ошибках.

-u
--utf8

Передайте флаг -u в git mailinfo (см. git-mailinfo[1] ). Предлагаемое сообщение журнала фиксации, взятое из электронного письма, перекодируется в кодировку UTF-8 (переменная конфигурации i18n.commitEncoding может использоваться для указания предпочтительной кодировки проекта, если она не UTF-8).

В предыдущих версиях git это было необязательно, но теперь это значение по умолчанию. Вы можете использовать --no-utf8 , чтобы переопределить это.

--no-utf8

Передайте флаг -n в git mailinfo (см. git-mailinfo[1] ).

-3
--3way
--no-3way

Если исправление не применяется корректно, вернитесь к трехстороннему слиянию, если исправление записывает идентификаторы больших двоичных объектов, к которым оно должно применяться, и эти большие двоичные объекты доступны локально. --no-3way можно использовать для переопределения переменной конфигурации am.threeWay. Для получения дополнительной информации см. am.threeWay в git-config[1] .

--rerere-autoupdate
--no-rerere-autoupdate

После того, как механизм rerere повторно использует записанное разрешение текущего конфликта для обновления файлов в рабочем дереве, разрешите ему также обновить индекс с результатом разрешения. --no-rerere-autoupdate — хороший способ перепроверить то, что сделал rerere , и выявить возможные ошибки слияния, прежде чем заносить результат в индекс с помощью отдельного git add .

--ignore-space-change
--ignore-whitespace
--whitespace=<action>
-C<n>
-p<n>
--directory=<dir>
--exclude=<path>
--include=<path>
--reject

Эти флаги передаются программе git apply (см. git-apply[1] ), которая применяет исправление.

--patch-format

По умолчанию команда попытается автоматически определить формат патча. Эта опция позволяет пользователю обойти автоматическое обнаружение и указать формат исправления, в котором следует интерпретировать исправления. Допустимые форматы: mbox, mboxrd, stgit, stgit-series и hg.

-i
--interactive

Run interactively.

-n
--no-verify

По умолчанию запускаются перехватчики pre-applypatch и applypatch-msg. Если задан любой из --no-verify или -n , они игнорируются. См. также githooks[5] .

--committer-date-is-author-date

По умолчанию команда записывает дату из сообщения электронной почты как дату автора фиксации и использует время создания фиксации в качестве даты коммиттера. Это позволяет пользователю лгать о дате коммиттера, используя то же значение, что и дата автора.

--ignore-date

По умолчанию команда записывает дату из сообщения электронной почты как дату автора фиксации и использует время создания фиксации в качестве даты коммиттера. Это позволяет пользователю лгать о дате автора, используя то же значение, что и дата коммиттера.

--skip

Пропустить текущий патч. Это имеет смысл только при перезапуске прерванного патча.

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

GPG-знак фиксируется. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

--continue
-r
--resolved

После сбоя исправления (например, при попытке применить конфликтующее исправление) пользователь применил его вручную, и в индексном файле сохраняется результат применения. Сделайте фиксацию, используя авторство и журнал фиксации, извлеченные из сообщения электронной почты и текущего индексного файла, и продолжайте.

--resolvemsg=<msg>

В случае сбоя исправления перед выходом на экран будет выведено <msg>. Это переопределяет стандартное сообщение, информирующее вас об использовании --continue или --skip для устранения сбоя. Это исключительно для внутреннего использования между git rebase и git am .

--abort

Восстановите исходную ветку и прервите операцию исправления. Верните содержимое файлов, участвующих в операции am, в состояние до их состояния.

--quit

Прервите операцию исправления, но оставьте HEAD и индекс нетронутыми.

--show-current-patch[=(diff|raw)]

Показать сообщение, на котором git am остановился из-за конфликта. Если указан raw , показать необработанное содержимое сообщения электронной почты; если diff , показывать только часть различий. По умолчанию raw .

--allow-empty

После сбоя исправления входного сообщения электронной почты, в котором отсутствует исправление, создайте пустую фиксацию с содержимым сообщения электронной почты в качестве сообщения журнала.

Discussion

Имя автора фиксации берется из строки "From: " сообщения, а дата автора фиксации берется из строки "Date: " сообщения. Строка "Subject: " используется в качестве заголовка коммита после удаления общего префикса "[PATCH <anything>]" . Строка "Subject: " должна кратко описывать суть коммита в одной строке текста.

Строки "From: " , "Date: " и "Subject: " , начинающие тело, переопределяют соответствующие имя автора коммита и значения заголовка, взятые из заголовков.

Сообщение о фиксации состоит из заголовка, взятого из "Subject: " , пустой строки и тела сообщения до начала патча. Лишние пробелы в конце каждой строки автоматически удаляются.

Ожидается, что патч будет встроенным сразу после сообщения. Любая строка вида:

  • три тире и конец строки или

  • строка, начинающаяся с "diff -" , или

  • строка, начинающаяся с "Index: "

считается началом патча, и сообщение журнала фиксации завершается до первого появления такой строки.

При первоначальном вызове git am вы указываете имена почтовых ящиков для обработки. При обнаружении первого патча, который не применяется, он прерывается посередине. Вы можете восстановиться после этого одним из двух способов:

  1. пропустите текущий патч, повторно запустив команду с опцией --skip .

  2. вручную разрешите конфликт в рабочем каталоге и обновите индексный файл, чтобы привести его в состояние, которое должно было создать исправление. Затем запустите команду с опцией --continue .

Команда отказывается обрабатывать новые почтовые ящики до тех пор, пока не завершится текущая операция, поэтому, если вы решите начать все сначала, запустите git am --abort перед запуском команды с именами почтовых ящиков.

Перед применением каких-либо исправлений ORIG_HEAD устанавливается на кончик текущей ветки. Это полезно, если у вас есть проблемы с несколькими коммитами, например, запуск git am не в той ветке или ошибка в коммитах, которую легче исправить, изменив почтовый ящик (например, ошибки в строках "From:" ).

Hooks

Эта команда может запускать перехватчики applypatch-msg , pre-applypatch и post-applypatch . См. githooks[5] для получения дополнительной информации.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

am.keepcr

Если это правда, git-am будет вызывать git-mailsplit для исправлений в формате mbox с параметром --keep-cr . В этом случае git-mailsplit не удалит \r из строк, заканчивающихся на \r\n . Можно переопределить, указав --no-keep-cr из командной строки. См. git-am[1] , git-mailsplit[1] .

am.threeWay

По умолчанию git am выйдет из строя, если исправление не будет применено корректно. Если установлено значение true, этот параметр указывает git am вернуться к трехстороннему слиянию, если исправление записывает идентификаторы больших двоичных объектов, к которым оно должно применяться, и у нас есть эти большие двоичные объекты, доступные локально (эквивалентно указанию опции --3way из командной строки). . По умолчанию false . См. git-am[1] .
