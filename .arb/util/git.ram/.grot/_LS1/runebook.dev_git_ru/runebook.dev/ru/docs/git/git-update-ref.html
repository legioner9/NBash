git-update-ref

Name

git-update-ref — безопасно обновить имя объекта, хранящееся в ссылке.

Synopsis

git update-ref [-m <reason>] [--no-deref] (-d <ref> [<oldvalue>] | [--create-reflog] <ref> <newvalue> [<oldvalue>] | --stdin [-z])

Description

Учитывая два аргумента, сохраняет <newvalue> в <ref>, возможно, разыменовывая символические ссылки. Например, git update-ref HEAD <newvalue> обновляет заголовок текущей ветки новым объектом.

Учитывая три аргумента, сохраняет <newvalue> в <ref>, возможно, разыменовывая символические ссылки, после проверки того, что текущее значение <ref> соответствует <oldvalue>. Например, git update-ref refs/heads/master <newvalue> <oldvalue> обновляет заголовок главной ветки до <newvalue>, только если его текущее значение равно <oldvalue>. Вы можете указать 40 "0" или пустую строку в качестве <oldvalue>, чтобы убедиться, что создаваемая вами ссылка не существует.

Это также позволяет файлу "ref" быть символическим указателем на другой файл ссылки, начиная с четырехбайтовой последовательности заголовка "ref:" .

Что еще более важно, это позволяет обновлять ref-файл по этим символическим указателям, будь то символические ссылки или эти "regular file symbolic refs" . Он следует за настоящими символическими ссылками, только если они начинаются с "refs/" : в противном случае он просто попытается прочитать их и обновить как обычный файл (т. е. он позволит файловой системе следовать за ними, но перезапишет такую ​​символическую ссылку в другое место обычным именем файла). ).

Если указан --no-deref, перезаписывается сам <ref>, а не результат следования символическим указателям.

В общем, используя

git update-ref HEAD "$head"

должен быть lot безопаснее, чем делать

echo "$head" > "$GIT_DIR/HEAD"

как с точки зрения следования символическим ссылкам, так и с точки зрения проверки ошибок. Правило "refs/" для символических ссылок означает, что символические ссылки, указывающие на дерево "outside" , безопасны: они будут использоваться для чтения, но не для записи (поэтому мы никогда не будем писать через символическую ссылку ref на какое-либо другое дерево, если вы скопировали целое дерево). архив путем создания дерева символических ссылок).

С флагом -d он удаляет указанный <ref> после проверки того, что он все еще содержит <oldvalue>.

В --stdin update-ref считывает инструкции со стандартного ввода и выполняет все изменения вместе. Укажите команды вида:

update SP <ref> SP <newvalue> [SP <oldvalue>] LF
create SP <ref> SP <newvalue> LF
delete SP <ref> [SP <oldvalue>] LF
verify SP <ref> [SP <oldvalue>] LF
option SP <opt> LF
start LF
prepare LF
commit LF
abort LF

С --create-reflog update-ref создаст журнал обновлений для каждой ссылки, даже если обычно он не создается.

Поля, содержащие пробелы, заключаются в кавычки, как если бы они были строками в исходном коде C; т. е. окружен двойными кавычками и с обратной косой чертой. Используйте 40 символов "0" или пустую строку, чтобы указать нулевое значение. Чтобы указать отсутствующее значение, полностью опустите это значение и предшествующий ему SP.

Альтернативно, используйте -z для указания в формате с завершающим NUL, без кавычек:

update SP <ref> NUL <newvalue> NUL [<oldvalue>] NUL
create SP <ref> NUL <newvalue> NUL
delete SP <ref> NUL [<oldvalue>] NUL
verify SP <ref> NUL [<oldvalue>] NUL
option SP <opt> NUL
start NUL
prepare NUL
commit NUL
abort NUL

В этом формате используйте 40 "0" , чтобы указать нулевое значение, и используйте пустую строку, чтобы указать отсутствующее значение.

В любом формате значения могут быть указаны в любой форме, которую Git распознает как имя объекта. Команды в любом другом формате или повторяющийся <ref> приводят к ошибке. Значения команды:

update

Установите для <ref> значение <newvalue> после проверки <oldvalue>, если оно указано. Укажите нулевое <newvalue>, чтобы гарантировать, что ссылка не существует после обновления, и/или нулевое <oldvalue>, чтобы убедиться, что ссылка не существует до обновления.

create

Создайте <ref> с <newvalue> после проверки его отсутствия. Данное <новое значение> не может быть нулевым.

delete

Удалите <ref> после проверки его существования с помощью <oldvalue>, если он указан. Если задано, <oldvalue> не может быть нулевым.

verify

Проверьте <ref> на соответствие <oldvalue>, но не изменяйте его. Если <oldvalue> равно нулю или отсутствует, ссылка не должна существовать.

option

Измените поведение следующей команды, называющей <ref>. Единственная допустимая опция — no-deref , чтобы избежать разыменования символической ссылки.

start

Начать транзакцию. В отличие от нетранзакционного сеанса, транзакция автоматически прерывается, если сеанс завершается без явного подтверждения. Эта команда может создать новую пустую транзакцию, если текущая уже зафиксирована или прервана.

prepare

Подготовьтесь к совершению транзакции. Это создаст файлы блокировки для всех обновлений ссылок, находящихся в очереди. Если одну ссылку не удалось заблокировать, транзакция будет прервана.

commit

Зафиксируйте все обновления ссылок, поставленные в очередь для транзакции, завершив транзакцию.

abort

Прервите транзакцию, сняв все блокировки, если транзакция находится в подготовленном состоянии.

Если все <ref> можно одновременно заблокировать совпадающими <oldvalue>, все изменения выполняются. В противном случае никакие изменения не выполняются. Обратите внимание, что хотя каждый отдельный <ref> обновляется или удаляется атомарно, одновременный читатель все равно может видеть подмножество изменений.

Logging updates

Если параметр конфигурации "core.logAllRefUpdates" — true , а ссылка — одна из "refs/heads/" , "refs/remotes/" , "refs/notes/" или псевдоссылка, например HEAD или ORIG_HEAD; или файл "$GIT_DIR/logs/<ref>" существует, то git update-ref добавит строку к файлу журнала "$GIT_DIR/logs/<ref>" (разыменовывая все символические ссылки перед созданием имени журнала), описывающую изменение значения ссылки. Строки журнала форматируются как:

oldsha1 SP newsha1 SP committer LF

Где "oldsha1" — это 40-значное шестнадцатеричное значение, ранее сохраненное в <ref>, "newsha1" — это 40-значное шестнадцатеричное значение <newvalue>, а "committer" — это имя, адрес электронной почты и дата коммиттера в стандартном формате идентификатора коммиттера Git .

Необязательно с -m:

oldsha1 SP newsha1 SP committer TAB message LF

Где все поля такие, как описано выше, а "message" — это значение, указанное в опции -m.

Обновление завершится неудачей (без изменения <ref>), если текущий пользователь не может создать новый файл журнала, добавить его к существующему файлу журнала или не имеет доступной информации о коммиттере.
