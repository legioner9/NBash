git-rerere

Name

git-rerere — повторно использовать записанное разрешение конфликтных слияний

Synopsis

git rerere [clear | forget <pathspec>…| diff | status | remaining | gc]

Description

В рабочем процессе, использующем относительно долгоживущие ветки тем, разработчику иногда приходится разрешать одни и те же конфликты снова и снова, пока ветки тем не будут готовы (либо объединены с веткой "release" , либо отправлены и приняты вверх по течению).

Эта команда помогает разработчику в этом процессе, записывая противоречивые результаты автоматического слияния и соответствующие результаты ручного разрешения при первоначальном ручном слиянии, а также применяя ранее записанные ручные разрешения к соответствующим результатам автоматического слияния.

Note
 Вам необходимо установить переменную конфигурации rerere.enabled , чтобы включить эту команду.

Commands

Обычно git rerere запускается без аргументов или вмешательства пользователя. Однако у него есть несколько команд, которые позволяют ему взаимодействовать с его рабочим состоянием.

clear

Сбросьте метаданные, используемые rerere, если разрешение слияния должно быть прервано. Вызов git am [--skip|--abort] или git rebase [--skip|--abort] автоматически вызовет эту команду.

forget <pathspec>

Сбросьте разрешения конфликтов, записанные rerere для текущего конфликта в <pathspec>.

diff

Отображение различий для текущего состояния разрешения. Это полезно для отслеживания того, что изменилось, пока пользователь разрешает конфликты. Дополнительные аргументы передаются непосредственно системной команде diff , установленной в PATH.

status

Вывести пути с конфликтами, разрешение слияния которых будет записываться rerere.

remaining

Вывести пути с конфликтами, которые не были автоматически разрешены с помощью rerere. Сюда входят пути, разрешение которых невозможно отследить с помощью rerere, например конфликтующие подмодули.

gc

Сократите записи о конфликтующих слияниях, произошедших давным-давно. По умолчанию удаляются неразрешенные конфликты старше 15 дней и разрешенные конфликты старше 60 дней. Эти значения по умолчанию контролируются с помощью переменных конфигурации gc.rerereUnresolved и gc.rerereResolved соответственно.

Discussion

Когда ваша тематическая ветка изменяет перекрывающуюся область, которую затронула ваша главная ветка (или вышестоящая ветка) с тех пор, как ваша тематическая ветка отделилась от нее, вы можете захотеть протестировать ее с помощью последней главной ветки, даже до того, как ваша тематическая ветка будет готова к отправке вверх по ветке:

              o---*---o topic
             /
    o---o---o---*---o---o master

Для такого теста нужно как-то объединить мастер и тему. Один из способов сделать это — перетащить master в ветку темы:

        $ git switch topic
        $ git merge master

              o---*---o---+ topic
             /           /
    o---o---o---*---o---o master

Коммиты, отмеченные * , касаются одной и той же области в одном файле; вам необходимо разрешить конфликты при создании коммита, отмеченного + . Затем вы можете проверить результат, чтобы убедиться, что ваша незавершенная работа по-прежнему работает с последней версией мастера.

После этого тестового слияния есть два способа продолжить работу над этой темой. Самый простой способ — построить поверх тестового коммита слияния + , и когда ваша работа в тематической ветке, наконец, будет готова, перенесите тематическую ветку в master и/или попросите восходящий поток получить от вас. Однако к тому времени главный или восходящий поток могли быть продвинуты после тестового слияния + , и в этом случае окончательный график фиксации будет выглядеть следующим образом:

        $ git switch topic
        $ git merge master
        $ ... work on both topic and master branches
        $ git switch master
        $ git merge topic

              o---*---o---+---o---o topic
             /           /         \
    o---o---o---*---o---o---o---o---+ master

Однако, если ваша тематическая ветка долгоживущая, в вашей тематической ветке будет много таких коммитов "Merge from master" , что будет излишне загромождать историю разработки. Читатели списка рассылки ядра Linux, возможно, помнят, что Линус жаловался на слишком частые слияния тестов, когда специалист по сопровождению подсистемы попросил извлечь данные из ветки, полной "useless merges" .

В качестве альтернативы, чтобы сохранить ветку темы чистой от тестовых слияний, вы можете убрать тестовое слияние и продолжать дополнять подсказку до тестового слияния:

        $ git switch topic
        $ git merge master
        $ git reset --hard HEAD^ ;# перематываем тестовое слияние
        $ ... work on both topic and master branches
        $ git switch master
        $ git merge topic

              o---*---o-------o---o topic
             /                     \
    o---o---o---*---o---o---o---o---+ master

В результате останется только один коммит слияния, когда ваша тематическая ветка будет наконец готова и объединена с основной веткой. Это слияние потребует от вас разрешения конфликта, вызванного фиксациями, отмеченными * . Однако этот конфликт часто является тем же конфликтом, который вы разрешили, когда создали тестовое слияние, которое вы уничтожили. git rerere поможет вам разрешить это окончательное конфликтное слияние, используя информацию, полученную вами ранее.

Выполнение команды git rerere сразу после конфликтного автообъединения записывает конфликтующие файлы рабочего дерева с обычными маркерами конфликта <<<<<<< , ======= и >>>>>>> в них. Позже, после того как вы закончите разрешение конфликтов, повторный запуск git rerere запишет разрешенное состояние этих файлов. Предположим, вы сделали это, когда создавали тестовое слияние master с веткой темы.

В следующий раз, увидев то же самое конфликтное автоматическое слияние, запуск git rerere выполнит трехстороннее слияние между предыдущим конфликтным автоматическим объединением, предыдущим разрешением вручную и текущим конфликтным автоматическим объединением. Если это трехстороннее слияние разрешается правильно, результат записывается в файл рабочего дерева, поэтому вам не придется разрешать его вручную. Обратите внимание, что git rerere оставляет индексный файл в покое, поэтому вам все равно придется выполнить окончательную проверку работоспособности с помощью git diff (или git diff -c ) и git add , когда вы будете удовлетворены.

В целях удобства git merge автоматически вызывает git rerere при выходе из-за неудачного автоматического объединения, а git rerere записывает решение руки, когда возникает новый конфликт, или повторно использует предыдущее решение руки, когда это не так. git commit также вызывает git rerere при фиксации результата слияния. Это означает, что вам не нужно делать ничего особенного самостоятельно (кроме включения переменной конфигурации rerere.enabled).

В нашем примере, когда вы выполняете тестовое слияние, записанное вручную разрешение записывается и будет повторно использовано при дальнейшем фактическом слиянии с обновленной основной веткой и темой, пока записанное разрешение все еще применимо.

Информационные записи git rerere также используются при запуске git rebase . После завершения тестового слияния и продолжения разработки ветки темы:

              o---*---o-------o---o topic
             /
    o---o---o---*---o---o---o---o   master

        $ git rebase master topic

                                  o---*---o-------o---o topic
                                 /
    o---о---о---*---о---о---о---о хозяин

вы можете запустить git rebase master topic , чтобы быть в курсе событий, прежде чем ваша тема будет готова к отправке в восходящий поток. Это приведет к возврату к трехстороннему слиянию и будет конфликтовать так же, как и тестовое слияние, которое вы разрешили ранее. git rerere будет запускаться git rebase , чтобы помочь вам разрешить этот конфликт.

[ПРИМЕЧАНИЕ] git rerere использует маркеры конфликта в файле для обнаружения конфликта. Если файл уже содержит строки, которые выглядят так же, как строки с маркерами конфликта, git rerere может не записать решение конфликта. Чтобы обойти эту проблему, можно использовать настройку conflict-marker-size в gitattributes[5] .
