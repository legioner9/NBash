gitsubmodules

Name

gitsubmodules — монтирование одного репозитория внутри другого

Synopsis

.gitmodules, $GIT_DIR/config
git submodule
git <command> --recurse-submodules

Description

Субмодуль — это репозиторий, встроенный в другой репозиторий. Субмодуль имеет свою историю; репозиторий, в который он встроен, называется суперпроектом.

В файловой системе подмодуль обычно (но не всегда — см. FORMS ниже) состоит из (i) каталога Git , расположенного в каталоге $GIT_DIR/modules/ его суперпроекта, (ii) рабочего каталога внутри рабочего каталога суперпроекта и файла .git по адресу корень рабочего каталога подмодуля, указывающий на (i).

Предполагая, что подмодуль имеет каталог Git по адресу $GIT_DIR/modules/foo/ и рабочий каталог по адресу path/to/bar/ , суперпроект отслеживает подмодуль через запись gitlink в дереве по адресу path/to/bar и запись в его файле .gitmodules (см. gitmodules[5] ) формы submodule.foo.path = path/to/bar .

Запись gitlink содержит имя объекта фиксации, в котором суперпроект ожидает находиться в рабочем каталоге подмодуля.

Раздел submodule.foo.* в файле .gitmodules дает дополнительные сведения о фарфоровом слое Git. Например, параметр submodule.foo.url указывает, где получить субмодуль.

Субмодули можно использовать как минимум для двух разных случаев:

  1. Использование другого проекта при сохранении независимой истории. Подмодули позволяют вам содержать рабочее дерево другого проекта в вашем собственном рабочем дереве, сохраняя при этом историю обоих проектов отдельно. Кроме того, поскольку подмодули привязаны к произвольной версии, другой проект может разрабатываться независимо, не затрагивая суперпроект, что позволяет проекту суперпроекта приспосабливаться к новым версиям только при желании.

  2. Разделение (логически единого) проекта на несколько репозиториев и их обратное связывание. Это можно использовать для преодоления текущих ограничений реализации Git и обеспечения более детального доступа:

    • Размер репозитория Git . В своей текущей форме Git плохо масштабируется для больших репозиториев, содержащих контент, который не сжимается путем дельта-вычислений между деревьями. Например, вы можете использовать подмодули для хранения больших двоичных ресурсов, и эти репозитории можно неглубоко клонировать, чтобы у вас не было большой локальной истории.

    • Размер передачи: в своей текущей форме Git требует наличия всего рабочего дерева. Он не позволяет переносить частичные деревья при выборке или клонировании. Если проект, над которым вы работаете, состоит из нескольких репозиториев, связанных вместе как подмодули в суперпроекте, вы можете избежать получения рабочих деревьев репозиториев, которые вас не интересуют.

    • Контроль доступа: Ограничивая доступ пользователей к подмодулям, можно использовать для реализации политик чтения/записи для разных пользователей.

Конфигурация субмодулей

Операции подмодуля можно настроить с использованием следующих механизмов (от высшего к низшему приоритету):

  • Командная строка для тех команд, которые поддерживают использование подмодулей как часть их спецификаций пути. Большинство команд имеют флаг boolean --recurse-submodules , который указывает, следует ли рекурсивно переходить к подмодулям. Примеры: grep и checkout . Некоторые команды принимают перечисления, например fetch и push , где вы можете указать, как будут затронуты подмодули.

  • Конфигурация внутри субмодуля. Сюда входит $GIT_DIR/config в субмодуле, а также настройки в дереве, такие как файлы .gitattributes или .gitignore , которые определяют поведение команд внутри субмодуля.

    Например, эффект от файла .gitignore подмодуля будет наблюдаться при запуске git status --ignore-submodules=none в суперпроекте. При этом информация из рабочего каталога субмодуля собирается путем запуска status в субмодуле, при этом обращая внимание на файл .gitignore субмодуля.

    Файл $GIT_DIR/config субмодуля будет задействован при запуске git push --recurse-submodules=check в суперпроекте, так как это проверит, есть ли в субмодуле какие-либо изменения, не опубликованные ни на одном удаленном компьютере. Пульты настраиваются в субмодуле как обычно в файле $GIT_DIR/config .

  • Конфигурационный файл $GIT_DIR/config в суперпроекте. Git рекурсивно переходит только в активные подмодули (см. раздел "ACTIVE SUBMODULES" ниже).

    Если субмодуль еще не инициализирован, то конфигурации внутри субмодуля еще не существует, поэтому здесь настраивается, например, откуда взять субмодуль.

  • Файл .gitmodules внутри суперпроекта. Проект обычно использует этот файл, чтобы предложить значения по умолчанию для вышестоящей коллекции репозиториев для требуемого сопоставления между именем подмодуля и его путем.

    Этот файл в основном служит для сопоставления имени и пути подмодулей в суперпроекте, так что можно найти каталог Git подмодуля.

    Если субмодуль никогда не инициализировался, это единственное место, где можно найти конфигурацию субмодуля. Он служит последним запасным вариантом, позволяющим указать, откуда получить подмодуль.

Forms

Субмодули могут принимать следующие формы:

  • Базовая форма, описанная в ОПИСАНИЕ, с каталогом Git , рабочим каталогом, записью gitlink и записью .gitmodules .

  • Субмодуль "Old-form" : рабочий каталог со встроенным каталогом .git и записью отслеживания gitlink и .gitmodules в суперпроекте. Обычно это встречается в репозиториях, созданных с использованием старых версий Git.

    Эти старые репозитории форм можно создать вручную.

    При деинициализации или удалении (см. ниже) каталог Git подмодуля автоматически перемещается в $GIT_DIR/modules/<name>/ суперпроекта.

  • Деинициализированный подмодуль: gitlink и запись .gitmodules , но нет рабочего каталога подмодуля. Каталог Git субмодуля может находиться там, поскольку после деинициализации каталог Git сохраняется. Вместо этого каталог, который должен быть рабочим, пуст.

    Субмодуль можно деинициализировать, запустив git submodule deinit . Помимо очистки рабочего каталога, эта команда изменяет только файл $GIT_DIR/config суперпроекта, поэтому история суперпроекта не затрагивается. Это можно отменить с помощью git submodule init .

  • Удаленный субмодуль: субмодуль можно удалить, запустив git rm <submodule path> && git commit . Это можно отменить с помощью git revert .

    При удалении удаляются данные отслеживания суперпроекта, которые представляют собой как запись gitlink , так и раздел в файле .gitmodules . Рабочий каталог подмодуля удаляется из файловой системы, но каталог Git сохраняется, чтобы можно было извлекать предыдущие коммиты без необходимости извлечения из другого репозитория.

    Чтобы полностью удалить субмодуль, удалите $GIT_DIR/modules/<name>/ вручную.

Active submodules

Субмодуль считается активным,

  1. если для submodule.<name>.active установлено значение true

    or

  2. если путь подмодуля соответствует спецификации пути в submodule.active

    or

  3. если установлен submodule.<name>.url .

и они оцениваются в этом порядке.

For example:

[submodule "foo"]
  active = false
  url = https://example.org/foo
[submodule "bar"]
  active = true
  url = https://example.org/bar
[submodule "baz"]
  url = https://example.org/baz

В приведенном выше конфиге активны только субмодули bar и baz , bar из-за (1) и baz из-за (3). foo неактивен, поскольку (1) имеет приоритет над (3).

Обратите внимание, что (3) является историческим артефактом и будет игнорироваться, если (1) и (2) указывают, что субмодуль не активен. Другими словами, если для submodule.<name>.active установлено значение false или если путь к подмодулю исключен из спецификации пути в submodule.active , URL-адрес не имеет значения, присутствует он или нет. Это показано в следующем примере.

[submodule "foo"]
  active = true
  url = https://example.org/foo
[submodule "bar"]
  url = https://example.org/bar
[submodule "baz"]
  url = https://example.org/baz
[submodule "bob"]
  ignore = true
[submodule]
  active = b*
  active = :(exclude) baz

Здесь активны все субмодули, кроме baz (foo, bar, bob). foo из-за собственного флага активности, а все остальные из-за спецификации активного пути субмодуля, которая указывает, что любой субмодуль, начинающийся с b , кроме baz , также активен, независимо от наличия поля .url.

Рабочий процесс для стороннего library

# Add a submodule
git submodule add <URL> <path>
# Occasionally update the submodule to a new version:
git -C <path> checkout <new version>
git add <path>
git commit -m "update submodule to new version"
# Посмотреть список подмодулей в суперпроекте
git submodule status
# См. FORMS об удалении подмодулей.

Рабочий процесс для искусственно разделенного репо

# Включите рекурсию для соответствующих команд, например
# обычные команды по умолчанию рекурсивно переходят в подмодули
git config --global submodule.recurse true
# В отличие от большинства других команд, приведенных ниже, клонированию по-прежнему требуется
# свой собственный флаг рекурсии:
git clone --recurse <URL> <directory>
cd <directory>
# Знакомимся с кодом:
git grep foo
git ls-files --recurse-submodules
Note
 Для git ls-files также требуется собственный флаг --recurse-submodules . 
# Get new code
git fetch
git pull --rebase
# Изменить рабочее дерево
git checkout
git reset

Implementation details

При клонировании или извлечении репозитория, содержащего подмодули, подмодули не будут извлечены по умолчанию; вы можете поручить clone рекурсивно перейти к подмодулям. Субкоманды init и update команды git submodule будут поддерживать подмодули извлеченными и иметь соответствующую редакцию в вашем рабочем дереве. Альтернативно вы можете настроить submodule.recurse на рекурсию checkout в подмодули (обратите внимание, что submodule.recurse также влияет на другие команды Git , полный список см. в git-config[1] ).
