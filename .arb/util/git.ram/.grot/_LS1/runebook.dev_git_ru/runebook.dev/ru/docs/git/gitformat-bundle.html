gitformat-bundle

Name

gitformat-bundle — формат файла пакета.

Synopsis

*.bundle
*.bdl

Description

Формат пакета Git — это формат, который представляет как ссылки, так и объекты Git . Пакет — это заголовок в формате, аналогичном git-show-ref[1] , за которым следует пакет в формате *.pack.

Формат создается и считывается командой git-bundle[1] и поддерживается, например, git-fetch[1] и git-clone[1] .

Format

Мы будем использовать нотацию ABNF для определения формата пакета Git . Подробности см. в gitprotocol-common[5] .

Пакет v2 выглядит так:

bundle    = signature *prerequisite *reference LF pack
signature = "# v2 git bundle" LF

prerequisite = "-" obj-id SP comment LF
comment      = *CHAR
reference    = obj-id SP refname LF

pack         = ... ; packfile

Пакет v3 выглядит так:

bundle    = signature *capability *prerequisite *reference LF pack
signature = "# v3 git bundle" LF

capability   = "@" key ["=" value] LF
prerequisite = "-" obj-id SP comment LF
comment      = *CHAR
reference    = obj-id SP refname LF
key          = 1*(ALPHA / DIGIT / "-")
value        = *(%01-09 / %0b-FF)

pack         = ... ; packfile

Semantics

Комплектация Git состоит из нескольких частей.

  • "Capabilities" , которые представлены только в формате v3, указывают на функциональные возможности, необходимые для правильного чтения пакета.

  • "Prerequisites" перечисляет объекты, которые НЕ включены в пакет и которые ДОЛЖНЫ уже иметься в считывателе пакета, чтобы использовать данные в пакете. Объекты, хранящиеся в пакете, могут ссылаться на обязательные объекты и все, что достижимо из них (например, древовидный объект в пакете может ссылаться на большой двоичный объект, доступный из обязательных объектов) и/или выражаться как дельта от обязательных объектов.

  • "References" записывает подсказки графика истории, то, что МОЖЕТ из него ридер комплектации "git fetch" .

  • "Pack" — это поток пакетных данных, который "git fetch" отправит, если вы извлекаете из репозитория, в котором есть ссылки, записанные в "References" выше, в репозиторий, в котором есть ссылки, указывающие на объекты, перечисленные в "Prerequisites" выше.

В формате пакета после обязательного obj-id может быть комментарий. Это комментарий и он не имеет конкретного значения. Создатель пакета МОЖЕТ поместить сюда любую строку. Читатель пакета ДОЛЖЕН игнорировать комментарий.

Примечание о мелких клонах и связках Git .

Обратите внимание, что предварительные условия не представляют собой границу мелкого клонирования. Семантика предварительных требований и границ мелкого клонирования различны, и формат пакета Git версии 2 не может представлять репозиторий мелкого клонирования.

Capabilities

Поскольку возможность согласования отсутствует, неизвестные возможности приводят к аварийному завершению работы git bundle .

  • object-format определяет используемый алгоритм хэширования и может принимать те же значения, что и значение конфигурации extensions.objectFormat .

  • filter задает фильтр объектов, как в опции --filter в git-rev-list[1] . Полученный пакетный файл после разделения должен быть помечен как пакетный файл .promisor .
