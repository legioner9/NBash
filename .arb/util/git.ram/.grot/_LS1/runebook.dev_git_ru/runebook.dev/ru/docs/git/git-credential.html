git-credential

Name

git-credential — Получить и сохранить учетные данные пользователя

Synopsis

'git credential' (fill|approve|reject)

Description

Git имеет внутренний интерфейс для хранения и получения учетных данных от системных помощников, а также для запроса у пользователя имен пользователей и паролей. Команда git-credential предоставляет этот интерфейс сценариям, которые могут захотеть получить, сохранить или запросить учетные данные таким же образом, как Git . Дизайн этого скриптового интерфейса моделирует внутренний C API; см. credential.h для получения дополнительной информации об этих концепциях.

git-credential принимает параметр "action" в командной строке (один из fill , approve или reject ) и считывает описание учетных данных на стандартном вводе (см. INPUT/OUTPUT FORMAT ).

Если действие — fill , git-credential попытается добавить атрибуты "username" и "password" в описание, читая файлы конфигурации, обращаясь к любым настроенным помощникам по учетным данным или запрашивая пользователя. Атрибуты имени пользователя и пароля описания учетных данных затем выводятся на стандартный вывод вместе с уже предоставленными атрибутами.

Если действие — approve , git-credential отправит описание всем настроенным помощникам по учетным данным, которые могут сохранить учетные данные для дальнейшего использования.

Если действие — reject , git-credential отправит описание всем настроенным помощникам по учетным данным, которые могут удалить все сохраненные учетные данные, соответствующие описанию.

Если действие — approve или reject , выходные данные выдаваться не должны.

Типичное использование учетных данных git

Приложение, использующее git-credential, обычно использует git credential , выполнив следующие действия:

  1. Создайте описание учетных данных на основе контекста.

    Например, если нам нужен пароль для https://example.com/foo.git , мы можем сгенерировать следующее описание учетных данных (не забудьте пустую строку в конце; она сообщает git credential , что приложение завершило передачу всей имеющейся у него информации):

    protocol=https
host=example.com
path=foo.git

  2. Попросите git-credential предоставить нам имя пользователя и пароль для этого описания. Это делается путем запуска git credential fill и подачи описания из шага (1) на стандартный ввод. Полное описание учетных данных (включая учетные данные как таковые, т. е. логин и пароль) будет выведено на стандартный вывод, например:

    protocol=https
host=example.com
username=bob
password=secr3t

    В большинстве случаев это означает, что атрибуты, указанные во входных данных, будут повторяться в выходных данных, но Git также может изменить описание учетных данных, например, удалив атрибут path , если протокол HTTP(s), а credential.useHttpPath имеет значение false.

    Если бы git credential знал о пароле, на этом этапе пользователь мог бы не вводить этот пароль (вместо этого пользователь мог ввести пароль для разблокировки связки ключей или не было никакого взаимодействия с пользователем, если связка ключей уже была разблокирована) до его возврата. password=secr3t .

  3. Используйте учетные данные (например, получите доступ к URL-адресу с именем пользователя и паролем из шага (2)) и проверьте, принято ли оно.

  4. Сообщите об успешном или неудачном вводе пароля. Если учетные данные позволили успешно завершить операцию, их можно пометить действием "approve" , чтобы указать git credential повторно использовать их при следующем вызове. Если учетные данные были отклонены во время операции, используйте действие "reject" , чтобы git credential запросил новый пароль при следующем вызове. В любом случае на git credential должно быть передано описание учетных данных, полученное на шаге (2) (которое также содержит поля, указанные на шаге (1)).

Input/output format

git credential считывает и/или записывает (в зависимости от используемого действия) учетную информацию в свой стандартный ввод/вывод. Эта информация может соответствовать либо ключам, для которых git credential получит информацию для входа (например, хост, протокол, путь), либо фактическим учетным данным, которые необходимо получить (имя пользователя/пароль).

Учетные данные разделены на набор именованных атрибутов, по одному атрибуту в каждой строке. Каждый атрибут задается парой ключ-значение, разделенной знаком = (равно), за которым следует новая строка.

Ключ может содержать любые байты, кроме = , новой строки или NUL. Значение может содержать любые байты, кроме новой строки или NUL.

Атрибуты, ключи которых заканчиваются скобками массива в стиле C [] , могут иметь несколько значений. Каждый экземпляр многозначного атрибута образует упорядоченный список значений — порядок повторяющихся атрибутов определяет порядок значений. Пустой многозначный атрибут ( key[]=\n ) удаляет все предыдущие записи и сбрасывает список.

Во всех случаях все байты обрабатываются как есть (т. е. кавычки отсутствуют, и невозможно передать значение с символом новой строки или NUL). Список атрибутов завершается пустой строкой или концом файла.

Git понимает следующие атрибуты:

protocol

Протокол, по которому будут использоваться учетные данные (например, https ).

host

Имя удаленного хоста для сетевых учетных данных. Сюда входит номер порта, если он был указан (например, "example.com:8088" ).

path

Путь, по которому будут использоваться учетные данные. Например, для доступа к удаленному репозиторию https это будет путь к репозиторию на сервере.

username

Имя пользователя учетных данных, если оно у нас уже есть (например, из URL-адреса, конфигурации, пользователя или из ранее запущенного помощника).

password

Пароль учетных данных, если мы просим его сохранить.

password_expiry_utc

Созданные пароли, такие как токен доступа OAuth, могут иметь срок действия. При чтении учетных данных от помощников git credential fill игнорирует пароли с истекшим сроком действия. Представлено как время Unix UTC, секунды с 1970 года.

oauth_refresh_token

Токен обновления OAuth может сопровождать пароль, который является токеном доступа OAuth. Помощники должны обращаться с этим атрибутом как с конфиденциальным, как и с атрибутом пароля. Сам Git не имеет особого поведения для этого атрибута.

url

Когда git credential считывает этот специальный атрибут, значение анализируется как URL-адрес и обрабатывается так, как если бы были прочитаны его составные части (например, url=https://example.com будет вести себя так, как если бы были предоставлены protocol=https и host=example.com ). Это может помочь вызывающим абонентам избежать самостоятельного анализа URL-адресов.

Обратите внимание, что указание протокола является обязательным, и если URL-адрес не указывает имя хоста (например, "cert:///path/to/file" ), учетные данные будут содержать атрибут имени хоста, значение которого представляет собой пустую строку.

Компоненты, отсутствующие в URL-адресе (например, в приведенном выше примере нет имени пользователя), останутся неустановленными.

wwwauth[]

Когда Git получает ответ HTTP, который включает один или несколько заголовков аутентификации WWW-Authenticate , они будут переданы Git помощникам по учетным данным.

Каждое значение заголовка WWW-Authenticate передается как многозначный атрибут wwwauth[] , где порядок атрибутов такой же, как они появляются в ответе HTTP. Этот атрибут — one-way из Git для передачи дополнительной информации помощникам по учетным данным.

Нераспознанные атрибуты автоматически отбрасываются.
