gitprotocol-common

Name

gitprotocol-common — общие сведения для различных протоколов.

Synopsis

<over-the-wire-protocol>

Description

Этот документ определяет общие черты различных протоколов беспроводной связи и форматов файлов, используемых в Git.

Abnf notation

Нотация ABNF, описанная в RFC 5234, используется в документах протокола, за исключением следующих основных правил замены:

  HEXDIG    =  DIGIT / "a" / "b" / "c" / "d" / "e" / "f"

Мы также определяем следующие общие правила:

  NUL       =  %x00
  zero-id   =  40*"0"
  obj-id    =  40*(HEXDIGIT)

  refname  =  "HEAD"
  refname /=  "refs/" <see discussion below>

Имя ссылки — это иерархическая строка октетов, начинающаяся с "refs/" и не нарушающая правила проверки команды git-check-ref-format . Более конкретно, они:

  1. Они могут включать косую черту / для иерархической группировки (каталогов), но ни один компонент, разделенный косой чертой, не может начинаться с точки . .

  2. Они должны содержать хотя бы один / . Это обеспечивает наличие такой категории, как heads/ , tags/ и т. д., но фактические имена не ограничиваются.

  3. У них нигде не может быть двух последовательных точек .. .

  4. Они не могут содержать управляющие символы ASCII (т. е. байты, значения которых меньше \040 или \177 DEL ), пробел, тильду ~ , курсор ^ , двоеточие : , вопросительный знак ? , звездочку * или открывающую скобку [ где угодно.

  5. Они не могут заканчиваться косой чертой / или точкой . .

  6. Они не могут заканчиваться последовательностью .lock .

  7. Они не могут содержать последовательность @{ .

  8. Они не могут содержать \\ .

Pkt-line format

Большая часть (но не вся) полезной нагрузки описывается вокруг pkt-lines.

Pkt-line — это двоичная строка переменной длины. Первые четыре байта строки, pkt-len, указывают общую длину строки в шестнадцатеричном формате. Pkt-len включает в себя 4 байта, используемые для хранения шестнадцатеричного представления длины.

Строка pkt МОЖЕТ содержать двоичные данные, поэтому разработчики ДОЛЖНЫ обеспечить 8-битную чистоту процедур анализа/форматирования строки pkt.

Недвоичная линия ДОЛЖНА БЫТЬ заканчиваться LF, который, если он присутствует, ДОЛЖЕН быть включен в общую длину. Получатели ДОЛЖНЫ обрабатывать строки pkt с недвоичными данными одинаково, независимо от того, содержат ли они конечный LF (удаляя LF, если он присутствует, и не жалуясь, когда он отсутствует).

Максимальная длина компонента данных pkt-линии составляет 65516 байт. Реализации НЕ ДОЛЖНЫ отправлять pkt-строку, длина которой превышает 65520 (65516 байт полезной нагрузки + 4 байта данных длины).

Реализации НЕ ДОЛЖНЫ отправлять пустую строку pkt ( "0004" ).

Строка pkt с полем длины 0 ( "0000" ), называемая флеш-pkt, является особым случаем и ДОЛЖНА обрабатываться иначе, чем пустая строка pkt ( "0004" ).

  pkt-line     =  data-pkt / flush-pkt

  data-pkt     =  pkt-len pkt-payload
  pkt-len      =  4*(HEXDIG)
  pkt-payload  =  (pkt-len - 4)*(OCTET)

  flush-pkt    = "0000"

Примеры (как строки в стиле C):

  pkt-line          actual value
  ---------------------------------
  "0006a\n"         "a\n"
  "0005a"           "a"
  "000bfoobar\n"    "foobar\n"
  "0004"            ""
