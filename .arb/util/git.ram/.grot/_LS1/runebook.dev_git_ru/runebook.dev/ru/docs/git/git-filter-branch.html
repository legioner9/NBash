git-filter-branch

Name

git-filter-branch — Переписать ветки

Synopsis

git filter-branch [--setup <command>] [--subdirectory-filter <directory>]
        [--env-filter <command>] [--tree-filter <command>]
        [--index-filter <command>] [--parent-filter <command>]
        [--msg-filter <command>] [--commit-filter <command>]
        [--tag-name-filter <command>] [--prune-empty]
        [--original <namespace>] [-d <directory>] [-f | --force]
        [--state-branch <branch>] [--] [<rev-list options>…​]

Warning

git filter-branch имеет множество ошибок, которые могут привести к неочевидным искажениям запланированной перезаписи истории (и могут оставить вам мало времени для исследования таких проблем, поскольку у него такая ужасная производительность). Эти проблемы безопасности и производительности не могут быть устранены с помощью обратной совместимости, поэтому их использование не рекомендуется. Используйте альтернативный инструмент фильтрации истории, например git filter-repo . Если вам все еще нужно использовать git filter-branch , пожалуйста, внимательно прочитайте SAFETY (и PERFORMANCE ), чтобы узнать о минах ветки фильтра, а затем бдительно избегайте как можно большего количества перечисленных там опасностей.

Description

Позволяет перезаписать историю версий Git , переписав ветки, упомянутые в <параметрах списка версий>, применяя пользовательские фильтры к каждой ревизии. Эти фильтры могут изменять каждое дерево (например, удалять файл или запускать перезапись Perl для всех файлов) или информацию о каждом коммите. В противном случае вся информация (включая исходное время фиксации или информацию о слиянии) будет сохранена.

Команда перезапишет только ссылки positive , упомянутые в командной строке (например, если вы укажете a..b , будет переписан только b ). Если вы не укажете фильтры, коммиты будут повторно зафиксированы без каких-либо изменений, которые обычно не имеют никакого эффекта. Тем не менее, это может быть полезно в будущем для компенсации некоторых ошибок Git или тому подобного, поэтому такое использование разрешено.

ПРИМЕЧАНИЕ. Эта команда учитывает файл .git/info/grafts и ссылки в пространстве имен refs/replace/ . Если у вас определены какие-либо трансплантаты или замещающие ссылки, запуск этой команды сделает их постоянными.

ПРЕДУПРЕЖДЕНИЕ! Переписанная история будет иметь разные имена для всех объектов и не будет сходиться с исходной ветвью. Вы не сможете легко отправить и распространить переписанную ветку поверх исходной ветки. Пожалуйста, не используйте эту команду, если вы не знаете всех последствий, и в любом случае избегайте ее использования, если для решения вашей проблемы достаточно простой единственной фиксации. (Дополнительную информацию о перезаписи опубликованной истории см. в разделе "RECOVERING FROM UPSTREAM REBASE" в git-rebase[1] .)

Всегда проверяйте правильность переписанной версии: исходные ссылки, если они отличаются от переписанных, будут храниться в пространстве имен refs/original/ .

Обратите внимание: поскольку эта операция требует очень больших затрат ввода-вывода, может быть хорошей идеей перенаправить временный каталог за пределы диска с помощью опции -d , например, в tmpfs. Сообщается, что ускорение очень заметно.

Filters

Фильтры применяются в порядке, указанном ниже. Аргумент <command> всегда оценивается в контексте оболочки с помощью команды eval (за заметным исключением фильтра фиксации по техническим причинам). До этого переменная среды $GIT_COMMIT будет содержать идентификатор перезаписываемого коммита. Кроме того, GIT_AUTHOR_NAME, GIT_AUTHOR_EMAIL, GIT_AUTHOR_DATE, GIT_COMMITTER_NAME, GIT_COMMITTER_EMAIL и GIT_COMMITTER_DATE берутся из текущего коммита и экспортируются в среду, чтобы повлиять на личности автора и коммиттера замещающего коммита, созданного git-commit-tree[1] после работы фильтров.

Если какая-либо оценка <команды> вернет ненулевой статус выхода, вся операция будет прервана.

Доступна функция map , которая принимает аргумент "original sha1 id" и выводит "rewritten sha1 id" , если фиксация уже была переписана, и "original sha1 id" в противном случае; функция map может возвращать несколько идентификаторов в отдельных строках, если ваш фильтр фиксации выдал несколько коммитов.

Options

--setup <command>

Это не настоящий фильтр, выполняемый для каждого коммита, а однократная настройка непосредственно перед циклом. Поэтому переменные, специфичные для фиксации, еще не определены. Определенные здесь функции или переменные могут использоваться или изменяться на следующих этапах фильтра, за исключением фильтра фиксации, по техническим причинам.

--subdirectory-filter <directory>

Посмотрите только историю, которая касается данного подкаталога. Результат будет содержать этот каталог (и только он) в качестве корня проекта. Подразумевается Remap to ancestor .

--env-filter <command>

Этот фильтр можно использовать, если вам нужно изменить только среду, в которой будет выполняться фиксация. В частности, вы можете переписать переменные среды автора/коммиттора/адреса электронной почты/времени (подробности см. в git-commit-tree[1] ).

--tree-filter <command>

Это фильтр для перезаписи дерева и его содержимого. Аргумент оценивается в оболочке с рабочим каталогом, установленным в корень извлеченного дерева. Затем новое дерево используется как есть (новые файлы добавляются автоматически, исчезнувшие файлы автоматически удаляются - ни файлы .gitignore, ни какие-либо другие правила игнорирования НЕ ИМЕЮТ НИКАКОГО ЭФФЕКТА!).

--index-filter <command>

Это фильтр для перезаписи индекса. Он похож на фильтр дерева, но не проверяет дерево, что делает его намного быстрее. Часто используется с git rm --cached --ignore-unmatch ... , см. ПРИМЕРЫ ниже. Для волосатых случаев см. git-update-index[1] .

--parent-filter <command>

Это фильтр для перезаписи родительского списка коммита. Он получит родительскую строку на стандартный ввод и выведет новую родительскую строку на стандартный вывод. Родительская строка имеет формат, описанный в git-commit-tree[1] : пустая для первоначальной фиксации, "-p parent" для обычной фиксации и "-p parent1 -p parent2 -p parent3 …​" для фиксации слиянием.

--msg-filter <command>

Это фильтр для перезаписи сообщений о фиксации. Аргумент оценивается в оболочке с исходным сообщением фиксации на стандартном вводе; его стандартный вывод используется в качестве нового сообщения фиксации.

--commit-filter <command>

Это фильтр для выполнения фиксации. Если этот фильтр указан, он будет вызываться вместо команды git commit-tree с аргументами формы "<TREE_ID> [(-p <PARENT_COMMIT_ID>)…​]" и сообщением журнала на стандартном вводе. Идентификатор фиксации ожидается на стандартном выводе.

В качестве специального расширения фильтр фиксации может выдавать несколько идентификаторов фиксации; в этом случае все они будут родителями для переписанных дочерних элементов исходного коммита.

В этом фильтре вы можете использовать удобную функцию map , а также другие удобные функции. Например, вызов skip_commit "$@" приведет к исключению текущего коммита (но не его изменений! Если вы этого хотите, используйте вместо этого git rebase ).

Вы также можете использовать git_commit_non_empty_tree "$@" вместо git commit-tree "$@" , если вы не хотите сохранять коммиты с одним родителем и это не вносит изменений в дерево.

--tag-name-filter <command>

Это фильтр для перезаписи имен тегов. При передаче он будет вызываться для каждой ссылки на тег, указывающей на перезаписанный объект (или на объект-тег, указывающий на перезаписанный объект). Исходное имя тега передается через стандартный ввод, а новое имя тега ожидается на стандартном выходе.

Исходные теги не удаляются, но могут быть перезаписаны; используйте "--tag-name-filter cat" , чтобы просто обновить теги. В этом случае будьте очень осторожны и убедитесь, что у вас есть резервная копия старых тегов на случай, если преобразование не удастся.

Поддерживается почти правильное перезапись объектов тегов. Если к тегу прикреплено сообщение, будет создан новый объект тега с тем же сообщением, автором и отметкой времени. Если к тегу прикреплена подпись, подпись будет удалена. Сохранить подписи невозможно по определению. Причина, по которой это собственно "nearly" , заключается в том, что в идеале, если тег не изменился (указывает на тот же объект, имеет то же имя и т. д.), он должен сохранить любую подпись. Это не так, подписи всегда будут удалены, будьте осторожны. Также нет поддержки изменения автора или временной метки (или сообщения тега, если на то пошло). Теги, которые указывают на другие теги, будут переписаны, чтобы указывать на базовый коммит.

--prune-empty

Некоторые фильтры генерируют пустые коммиты, оставляющие дерево нетронутым. Эта опция предписывает git-filter-branch удалять такие коммиты, если у них есть ровно один или ноль необрезанных родительских элементов; Таким образом, коммиты слияния останутся нетронутыми. Эту опцию нельзя использовать вместе с --commit-filter , хотя того же эффекта можно добиться, используя предоставленную функцию git_commit_non_empty_tree в фильтре фиксации.

--original <namespace>

Используйте эту опцию, чтобы установить пространство имен, в котором будут храниться исходные коммиты. Значение по умолчанию — refs/original .

-d <directory>

Используйте эту опцию, чтобы установить путь к временному каталогу, используемому для перезаписи. При применении фильтра дерева команде необходимо временно извлечь дерево в какой-либо каталог, что может занимать значительное пространство в случае больших проектов. По умолчанию это делается в каталоге .git-rewrite/ , но вы можете переопределить этот выбор с помощью этого параметра.

-f
--force

git filter-branch отказывается запускаться с существующим временным каталогом или когда уже есть ссылки, начинающиеся с refs/original/ , если это не принудительно.

--state-branch <branch>

Эта опция приведет к тому, что сопоставление старых и новых объектов будет загружено из именованной ветки при запуске и сохранено как новая фиксация в этой ветке при выходе, что позволит увеличивать большие деревья. Если <branch> не существует, он будет создан.

<rev-list options>…

Аргументы в пользу git rev-list . Все положительные рефы, включенные в эти опции, переписываются. Вы также можете указать такие параметры, как --all , но вы должны использовать -- , чтобы отделить их от параметров git filter-branch . Подразумевается Remap to ancestor .

Переназначить предку

Используя аргументы git-rev-list[1] , например ограничители пути, вы можете ограничить набор перезаписываемых ревизий. Однако положительные ссылки в командной строке выделяются: мы не позволяем исключать их такими ограничителями. Для этой цели они вместо этого переписываются так, чтобы указывать на ближайшего предка, который не был исключен.

Exit status

В случае успеха статус выхода — 0 . Если фильтр не может найти никаких коммитов для перезаписи, статус выхода — 2 . В случае любой другой ошибки статус выхода может иметь любое другое ненулевое значение.

Examples

Предположим, вы хотите удалить файл (содержащий конфиденциальную информацию или нарушение авторских прав) из всех коммитов:

git filter-branch --tree-filter 'rm filename' HEAD

Однако если файл отсутствует в дереве какого-либо коммита, простой rm filename завершится неудачей для этого дерева и коммита. Таким образом, вместо этого вы можете использовать rm -f filename в качестве сценария.

Использование --index-filter с git rm дает значительно более быструю версию. Как и при использовании rm filename , git rm --cached filename завершится ошибкой, если файл отсутствует в дереве коммита. Если вы хотите использовать файл "completely forget" , не имеет значения, когда он вошел в историю, поэтому мы также добавляем --ignore-unmatch :

git filter-branch --index-filter 'git rm --cached --ignore-unmatch filename' HEAD

Теперь вы получите переписанную историю, сохраненную в HEAD.

Чтобы переписать репозиторий так, чтобы он выглядел так, как если бы foodir/ был корнем его проекта, и отбросить всю остальную историю:

git filter-branch --subdirectory-filter foodir -- --all

Таким образом, вы можете, например, превратить подкаталог library в собственный репозиторий. Обратите внимание на -- , который отделяет параметры filter-branch от параметров версии, и --all для перезаписи всех ветвей и тегов.

Чтобы установить фиксацию (которая обычно находится в конце другой истории) в качестве родительской для текущей первоначальной фиксации, чтобы вставить другую историю позади текущей истории:

git filter-branch --parent-filter 'sed "s/^\$/-p <graft-id>/"' HEAD

(если родительская строка пуста — что происходит, когда мы имеем дело с первоначальным коммитом — добавьте Graftcommit в качестве родителя). Обратите внимание, что это предполагает историю с одним корнем (то есть слияния без общих предков не произошло). Если это не так, используйте:

git filter-branch --parent-filter \
        'test $GIT_COMMIT = <commit-id> && echo "-p <graft-id>" || cat' HEAD

или еще проще:

git replace --graft $commit-id $graft-id
git filter-branch $graft-id..HEAD

Чтобы удалить коммиты, созданные "Darl McBribe" , из истории:

git filter-branch --commit-filter '
        if [ "$GIT_AUTHOR_NAME" = "Darl McBribe" ];
        then
                skip_commit "$@";
        else
                git commit-tree "$@";
        fi' HEAD

Функция skip_commit определяется следующим образом:

skip_commit()
{
        shift;
        while [ -n "$1" ];
        do
                shift;
                map "$1";
                shift;
        done;
}

Магия сдвига сначала отбрасывает идентификатор дерева, а затем параметры -p. Обратите внимание, что эти дескрипторы объединяются правильно! В случае, если Дарл совершил слияние между P1 и P2, оно будет распространяться правильно, и все дочерние элементы слияния станут коммитами слияния с P1, P2 в качестве своих родителей вместо коммита слияния.

ПРИМЕЧАНИЕ. Изменения, внесенные коммитами и не отмененные последующими коммитами, все равно останутся в переписанной ветке. Если вы хотите выкинуть changes вместе с коммитами, вам следует использовать интерактивный режим git rebase .

Вы можете переписать сообщения журнала фиксации, используя --msg-filter . Например, строки git svn-id в репозитории, созданном git svn , можно удалить следующим образом:

git filter-branch --msg-filter '
        sed -e "/^git-svn-id:/d"
'

Если вам нужно добавить строки Acked-by , скажем, к последним 10 коммитам (ни один из которых не является слиянием), используйте эту команду:

git filter-branch --msg-filter '
        cat &&
        echo "Acked-by: Bugs Bunny <[email protected]>"
' HEAD~10..HEAD

Опцию --env-filter можно использовать для изменения личности коммиттера и/или автора. Например, если вы обнаружили, что ваши коммиты имеют неправильную идентификацию из-за неправильно настроенного пользователя user.email, вы можете внести исправления перед публикацией проекта следующим образом:

git filter-branch --env-filter '
        if test "$GIT_AUTHOR_EMAIL" = "root@localhost"
        then
                GIT_AUTHOR_EMAIL=john@example.com
        fi
        if test "$GIT_COMMITTER_EMAIL" = "root@localhost"
        then
                GIT_COMMITTER_EMAIL=john@example.com
        fi
' -- --all

Чтобы ограничить перезапись только частью истории, укажите диапазон редакций в дополнение к новому имени ветки. Новое имя ветки будет указывать на самую верхнюю версию, которую будет печатать git rev-list этого диапазона.

Рассмотрим эту историю:

     D--E--F--G--H
    /     /
A--B-----C

Чтобы переписать только коммиты D,E,F,G,H, но оставить A, B и C в покое, используйте:

git filter-branch ... C..H

Чтобы переписать коммиты E,F,G,H, используйте один из них:

git filter-branch ... C..H --not D
git filter-branch ... D..H --not C

Чтобы переместить все дерево в подкаталог или удалить его оттуда:

git filter-branch --index-filter \
        'git ls-files -s | sed "s-\t\"*-&newsubdir/-" |
                GIT_INDEX_FILE=$GIT_INDEX_FILE.new \
                        git update-index --index-info &&
         mv "$GIT_INDEX_FILE.new" "$GIT_INDEX_FILE"' HEAD

Контрольный список для сокращения репозитория

git-filter-branch можно использовать для удаления подмножества файлов, обычно с помощью некоторой комбинации --index-filter и --subdirectory-filter . Люди ожидают, что полученный репозиторий будет меньше исходного, но вам нужно сделать еще несколько шагов, чтобы действительно уменьшить его, потому что Git изо всех сил старается не потерять ваши объекты, пока вы не прикажете ему это сделать. Сначала убедитесь, что:

  • Вы действительно удалили все варианты имени файла, если большой объект был перемещен за время его существования. git log --name-only --follow --all -- filename может помочь вам найти переименования.

  • Вы действительно отфильтровали все ссылки: используйте --tag-name-filter cat -- --all при вызове git-filter-branch.

Тогда есть два способа получить меньший репозиторий. Более безопасный способ — клонировать, сохраняя оригинал нетронутым.

  • Клонируйте его с помощью git clone file:///path/to/repo . В клоне удаленных объектов не будет. См. git-clone[1] . (Обратите внимание, что клонирование с использованием простого пути просто создает жесткие ссылки на все!)

Если вы действительно не хотите его клонировать по каким-либо причинам, вместо этого проверьте следующие пункты (в этом порядке). Это очень разрушительный подход, поэтому сделайте резервную копию или вернитесь к клонированию. Вы были предупреждены.

  • Удалите исходные ссылки, резервные копии которых созданы с помощью git-filter-branch: скажем, git for-each-ref --format="%(refname)" refs/original/ | xargs -n 1 git update-ref -d .

  • Завершите все повторные журналы с помощью git reflog expire --expire=now --all .

  • Соберите мусор со всех объектов, на которые нет ссылок, с помощью git gc --prune=now (или, если ваш git-gc недостаточно новый, чтобы поддерживать аргументы --prune , вместо этого используйте git repack -ad; git prune ).

Performance

Производительность git-filter-branch крайне медленная; его конструкция делает невозможным, чтобы обратно совместимая реализация была быстра:

  • При редактировании файлов git-filter-branch по замыслу проверяет каждый коммит в том виде, в котором он существовал в исходном репозитории. Если в вашем репозитории есть файлы 10^5 и коммиты 10^5 , но каждый коммит изменяет только пять файлов, то git-filter-branch заставит вас вносить изменения в 10^10 , несмотря на наличие (максимум) уникальных блоков 5*10^5 .

  • Если вы попытаетесь обмануть и попытаться заставить git-filter-branch работать только с файлами, измененными в коммите, произойдет две вещи.

    • вы сталкиваетесь с проблемами удаления всякий раз, когда пользователь просто пытается переименовать файлы (потому что попытка удалить файлы, которые не существуют, выглядит как пустая операция; требуется некоторая хитрость, чтобы переназначить удаления при переименовании файлов, когда переименования происходят через произвольного пользователя -предоставленная оболочка)

    • даже если вы преуспеете в хитрости «удаление карты ради переименования», вы все равно технически нарушаете обратную совместимость, поскольку пользователям разрешено фильтровать файлы способами, зависящими от топологии коммитов, вместо фильтрации исключительно на основе содержимого или имен файлов (хотя это имеет в природе не наблюдалось).

  • Даже если вам не нужно редактировать файлы, а вы хотите, например, переименовать или удалить некоторые из них и, таким образом, можете избежать извлечения каждого файла (т. е. вы можете использовать --index-filter), вы все равно передаете фрагменты оболочки для своих фильтров. Это означает, что для каждого коммита у вас должен быть подготовленный репозиторий git , в котором можно запускать эти фильтры. Это важная установка.

  • Кроме того, при каждом коммите с помощью git-filter-branch создается или обновляется несколько дополнительных файлов. Некоторые из них предназначены для поддержки удобных функций, предоставляемых git-filter-branch (например, map()), тогда как другие предназначены для отслеживания внутреннего состояния (но к ним также могут быть доступны пользовательские фильтры; один из git-filter- регрессионные тесты ветки делают это). По сути, это сводится к использованию файловой системы в качестве механизма IPC между git-filter-branch и фильтрами, предоставляемыми пользователем. Диски, как правило, являются медленным механизмом IPC, и запись этих файлов также фактически представляет собой точку принудительной синхронизации между отдельными процессами, которые мы затрагиваем при каждом коммите.

  • Команды оболочки, предоставляемые пользователем, скорее всего, будут включать в себя конвейер команд, что приведет к созданию множества процессов для каждого коммита. Создание и запуск другого процесса в разных операционных системах занимает разное количество времени, но на любой платформе это очень медленно по сравнению с вызовом функции.

  • Сам git-filter-branch написан в оболочке, что довольно медленно. Это единственная проблема с производительностью, которую можно устранить с помощью обратной совместимости, но по сравнению с вышеупомянутыми проблемами, присущими конструкции git-filter-branch, язык самого инструмента является относительно незначительной проблемой.

    • Примечание: к сожалению, люди склонны зацикливаться на аспекте написания в оболочке и периодически спрашивают, можно ли переписать git-filter-branch на другом языке, чтобы исправить проблемы с производительностью. Это не только игнорирует более серьезные внутренние проблемы дизайна, но и поможет меньше, чем вы ожидаете: если бы сам git-filter-branch не был оболочкой, то удобные функции (map(),skip_commit() и т. д.) и аргумент --setup больше не может быть выполнен один раз в начале программы, а вместо этого его нужно будет добавлять к каждому пользовательскому фильтру (и, таким образом, повторно выполнять при каждом коммите).

Инструмент git filter-repo является альтернативой git-filter-branch, которая не страдает от этих проблем с производительностью или проблемами безопасности (упомянутыми ниже). Для тех, у кого уже есть инструментарий, основанный на git-filter-branch, git filter-repo также предоставляет filter-lamely , замену git-filter-branch (с некоторыми оговорками). Хотя filter-lamely страдает от тех же проблем с безопасностью, что и git-filter-branch, он, по крайней мере, немного улучшает проблемы с производительностью.

Safety

git-filter-branch изобилует ошибками, которые приводят к различным способам легкого повреждения репозиториев или к еще худшему беспорядку, чем тот, с которого вы начали:

  • У кого-то может быть набор "working and tested filters" , который он документирует или предоставляет коллеге, который затем запускает его в другой ОС, где те же команды не работают/не тестируются (это также влияет на некоторые примеры на странице руководства git-filter-branch). . Различия в пользовательском пространстве BSD и GNU могут оказаться весьма ощутимыми. Если повезет, выдаются сообщения об ошибках. Но столь же вероятно, что команды либо не выполняют запрошенную фильтрацию, либо незаметно искажают, внося какие-то нежелательные изменения. Нежелательное изменение может затронуть только несколько коммитов, поэтому оно также не обязательно очевидно. (Тот факт, что проблемы не обязательно будут очевидными, означает, что они, скорее всего, останутся незамеченными до тех пор, пока переписанная история не будет использоваться в течение некоторого времени, и в этот момент действительно трудно оправдать еще один день флага для еще одной перезаписи.)

  • Имена файлов с пробелами часто неправильно обрабатываются фрагментами оболочки, поскольку они вызывают проблемы в конвейерах оболочки. Не все знакомы с find -print0, xargs -0, git-ls-files -z и т. д. Даже люди, знакомые с ними, могут предположить, что такие флаги не имеют значения, потому что кто-то другой переименовал любые такие файлы в своем репозитории еще до того, как человек, выполняющий фильтрацию, присоединился к проекту. И часто даже те, кто знаком с обработкой аргументов с помощью пробелов, могут не делать этого только потому, что они не думают обо всем, что может пойти не так.

  • Имена файлов, отличных от ascii, можно удалить автоматически, несмотря на то, что они находятся в нужном каталоге. Сохранение только нужных путей часто осуществляется с помощью таких конвейеров, как git ls-files | grep -v ^WANTED_DIR/ | xargs git rm . ls-files будет заключать имена файлов только в кавычки, если это необходимо, поэтому люди могут не заметить, что один из файлов не соответствует регулярному выражению (по крайней мере, пока не станет слишком поздно). Да, тот, кто знает о core.quotePath, может избежать этого (если у него нет других специальных символов, таких как \t, \n или "), и люди, которые используют ls-files -z с чем-то отличным от grep, могут избежать этого, но это это не значит, что они будут.

  • Аналогично, при перемещении файлов можно обнаружить, что имена файлов, содержащие не-ascii или специальные символы, оказываются в другом каталоге, в котором есть символ двойной кавычки. (Технически это та же проблема, что и вышеописанная с цитированием, но, возможно, это интересный другой способ, которым она может проявиться и уже проявилась как проблема.)

  • Слишком легко случайно перепутать старую и новую историю. Это все еще возможно с помощью любого инструмента, но git-filter-branch почти предлагает это. Если повезет, единственным недостатком будет то, что пользователи разочаруются из-за того, что они не знают, как сжать свой репозиторий и удалить старый материал. Если повезет, они объединят старую и новую историю и в итоге получат несколько "copies" каждого коммита, некоторые из которых содержат нежелательные или конфиденциальные файлы, а другие — нет. Это происходит разными способами:

    • по умолчанию выполняется только частичная перезапись истории ( --all не используется по умолчанию, и это показано в нескольких примерах)

    • тот факт, что нет автоматической очистки после запуска

    • тот факт, что --tag-name-filter (при переименовании тегов) не удаляет старые теги, а просто добавляет новые с новым именем

    • тот факт, что предоставляется мало образовательной информации, чтобы информировать пользователей о последствиях переписывания и о том, как избежать смешивания старой и новой истории. Например, на этой странице руководства обсуждается, как пользователи должны понимать, что им необходимо перебазировать свои изменения для всех своих ветвей поверх новой истории (или удалить и повторно клонировать), но это только одна из множества проблем, которые следует учитывать. Дополнительную информацию см. в разделе "DISCUSSION" на странице руководства по репозиторию фильтров git .

  • Теги с аннотациями могут быть случайно преобразованы в упрощенные теги по одной из двух причин:

    • Кто-то может переписать историю, осознать, что что-то напутал, восстановить из резервных копий в refs/original/, а затем повторить команду git-filter-branch. (Резервная копия в refs/original/ не является настоящей резервной копией; сначала она разыменовывает теги.)

    • Запуск git-filter-branch с --tags или --all в ваших <rev-list options>. Чтобы сохранить аннотированные теги как аннотированные, вы должны использовать --tag-name-filter (и не должны восстанавливаться из refs/original/ при ранее неудачной перезаписи).

  • Любые сообщения фиксации, в которых указана кодировка, будут повреждены при перезаписи; git-filter-branch игнорирует кодировку, берет исходные байты и передает их в дерево фиксации, не сообщая ему правильную кодировку. (Это происходит независимо от того, используется --msg-filter или нет.)

  • Сообщения коммитов (даже если все они в формате UTF-8) по умолчанию повреждаются из-за того, что не обновляются — любые ссылки на другие хэши коммитов в сообщениях о коммитах теперь будут относиться к уже не существующим коммитам.

  • Нет никаких средств, которые могли бы помочь пользователям найти ненужную грязь, которую им следует удалить, а это означает, что у них гораздо больше шансов получить неполную или частичную очистку, что иногда приводит к путанице, и люди тратят время, пытаясь понять. (Например, люди склонны просто искать большие файлы для удаления вместо больших каталогов или расширений, и как только они это сделают, через некоторое время люди, использующие новый репозиторий и просматривающие историю, заметят каталог артефакта сборки, в котором есть несколько файлов. но не другие, или кеш зависимостей (node_modules или аналогичный), который никогда не мог работать, поскольку в нем отсутствуют некоторые файлы.)

  • Если --prune-empty не указан, то процесс фильтрации может создать кучу запутанных пустых коммитов.

  • Если указан --prune-empty, то намеренно помещенные пустые коммиты, сделанные до операции фильтрации, также удаляются, а не просто удаляются коммиты, которые стали пустыми из-за правил фильтрации.

  • Если указан --prune-empty, иногда пустые коммиты пропускаются и все равно остаются (довольно редкая ошибка, но случается…​)

  • Незначительная проблема, но пользователи, которые хотят обновить все имена и адреса электронной почты в репозитории, могут быть перенаправлены на --env-filter, который будет обновлять только авторов и коммиттеров, а также отсутствующие теги.

  • Если пользователь предоставляет --tag-name-filter, который сопоставляет несколько тегов с одним и тем же именем, предупреждение или ошибка не выдаются; git-filter-branch просто перезаписывает каждый тег в каком-то недокументированном, заранее определенном порядке, в результате чего в конце остается только один тег. (Регрессионный тест git-filter-branch требует такого удивительного поведения.)

Кроме того, низкая производительность git-filter-branch часто приводит к проблемам с безопасностью:

  • Придумать правильный фрагмент оболочки для выполнения нужной фильтрации иногда бывает сложно, если только вы не выполняете тривиальную модификацию, например удаление пары файлов. К сожалению, люди часто узнают, правильный или неправильный фрагмент, проверив его, но правильность или неправильность могут варьироваться в зависимости от особых обстоятельств (пробелы в именах файлов, имена файлов, отличные от ASCII, забавные имена авторов или адреса электронной почты, недопустимые часовые пояса, наличие привязок). или заменять объекты и т. д.), то есть им, возможно, придется долго ждать, обнаружить ошибку, а затем перезапустить. Производительность git-filter-branch настолько плоха, что этот цикл является болезненным, сокращая время, доступное для тщательной повторной проверки (не говоря уже о том, что это делает с терпением человека, выполняющего переписывание, даже если у него технически есть больше возможностей). свободное время). Эта проблема усугубляется тем, что ошибки из-за сломанных фильтров могут не отображаться в течение длительного времени и/или теряться в море выходных данных. Хуже того, сломанные фильтры часто просто приводят к бесшумной некорректной перезаписи.

  • В довершение всего, даже когда пользователи наконец находят работающие команды, они, естественно, хотят ими поделиться. Но они могут не знать, что в их репозитории нет особых случаев, которые есть у кого-то другого. Таким образом, когда кто-то другой с другим репозиторием выполняет те же команды, он сталкивается с описанными выше проблемами. Или пользователь просто запускает команды, которые действительно были проверены для особых случаев, но запускает их в другой ОС, где они не работают, как отмечалось выше.
