gitcore-tutorial

Name

gitcore-tutorial — основное руководство по Git для разработчиков.

Synopsis

git *

Description

В этом руководстве объясняется, как использовать команды "core" Git для настройки репозитория Git и работы с ним.

Если вам просто нужно использовать Git в качестве системы контроля версий, вы можете начать с "A Tutorial Introduction to Git" ( gittutorial[7] ) или the Git User Manual .

Однако понимание этих низкоуровневых инструментов может оказаться полезным, если вы хотите понять внутренности Git.

Ядро Git часто называют "plumbing" , а более красивый пользовательский интерфейс поверх него называется "porcelain" . Возможно, вы не захотите часто пользоваться водопроводом, но полезно знать, что делает водопровод, когда фарфор не смывается.

Когда этот документ был первоначально написан, многие фарфоровые команды представляли собой сценарии оболочки. Для простоты они по-прежнему используются в качестве примеров, чтобы проиллюстрировать, как сантехника сочетается друг с другом, образуя фарфоровые композиции. Дерево исходного кода включает некоторые из этих сценариев в contrib/examples/ для справки. Хотя они больше не реализованы в виде сценариев оболочки, описание того, что делают команды сантехнического уровня, по-прежнему актуально.

Note
 Более глубокие технические подробности часто помечаются как примечания, которые вы можете пропустить при первом чтении.

Создание репозитория git

Создание нового репозитория Git не может быть проще: все репозитории Git изначально пусты, и единственное, что вам нужно сделать, это найти себе подкаталог, который вы хотите использовать в качестве рабочего дерева — либо пустой для совершенно нового проекта. или существующее рабочее дерево, которое вы хотите импортировать в Git .

В нашем первом примере мы собираемся создать совершенно новый репозиторий с нуля, без ранее существовавших файлов, и назовем его git-tutorial . Для запуска создайте для него подкаталог, перейдите в этот подкаталог и инициализируйте инфраструктуру Git с помощью git init :

$ mkdir git-tutorial
$ cd git-tutorial
$ git init

на что Git ответит

Initialized empty Git repository in .git/

это всего лишь способ Git сказать, что вы не делали ничего странного и что он создаст локальную настройку каталога .git для вашего нового проекта. Теперь у вас будет каталог .git , и вы можете проверить его с помощью ls . Для вашего нового пустого проекта он должен показать вам, среди прочего, три записи:

  • файл с именем HEAD , в котором есть ref: refs/heads/master . Это похоже на символическую ссылку и указывает на refs/heads/master относительно файла HEAD .

    Не беспокойтесь о том, что файл, на который указывает ссылка HEAD , еще даже не существует — вы еще не создали коммит, который запустит вашу ветку разработки HEAD .

  • подкаталог objects , который будет содержать все объекты вашего проекта. У вас никогда не должно быть реальной причины смотреть на объекты напрямую, но вам может быть интересно знать, что именно эти объекты содержат весь настоящий data в вашем репозитории.

  • подкаталог refs , содержащий ссылки на объекты.

В частности, подкаталог refs будет содержать два других подкаталога с именами heads и tags соответственно. Они делают именно то, что подразумевают их названия: они содержат ссылки на любое количество различных разрабатываемых heads (также известных как branches ) и на любой tags , который вы создали для обозначения конкретных версий в вашем репозитории.

Одно замечание: специальная голова master является веткой по умолчанию, поэтому созданный файл .git/HEAD указывает на нее, даже если она еще не существует. По сути, ссылка HEAD всегда должна указывать на ветку, над которой вы сейчас работаете, и вы всегда начинаете с ожидания работы над веткой master .

Однако это всего лишь соглашение, и вы можете называть свои ветки как угодно, и вам даже не обязательно использовать have как ветку master . Однако ряд инструментов Git предполагают, что .git/HEAD действителен.

Note
 Объект идентифицируется по 160-битному хешу SHA-1, также известному как имя объекта, а ссылка на объект всегда представляет собой 40-байтовое шестнадцатеричное представление этого имени SHA-1. Ожидается, что файлы в подкаталоге refs будут содержать эти шестнадцатеричные ссылки (обычно с последним \n в конце), и поэтому вы должны ожидать увидеть несколько 41-байтовых файлов, содержащих эти ссылки, в этих подкаталогах refs , когда вы действительно начнете заполнять твое дерево.
Note
 Опытный пользователь может захотеть взглянуть на gitrepository-layout[5] после завершения этого руководства.

Теперь вы создали свой первый репозиторий Git . Конечно, поскольку он пуст, это не очень полезно, поэтому давайте начнем заполнять его данными.

Заполнение репозитория git

Мы оставим это простым и глупым, поэтому начнем с заполнения нескольких тривиальных файлов, просто чтобы почувствовать это.

Начните с создания случайных файлов, которые вы хотите сохранить в своем репозитории Git . Мы начнем с нескольких плохих примеров, просто чтобы понять, как это работает:

$ echo "Hello World" >hello
$ echo "Silly example" >example

Теперь вы создали два файла в своем рабочем дереве (также известном как working directory ), но чтобы действительно проверить свою тяжелую работу, вам придется выполнить два шага:

  • заполните файл index (он же cache ) информацией о состоянии вашего рабочего дерева.

  • зафиксируйте этот индексный файл как объект.

Первый шаг тривиален: когда вы хотите сообщить Git о любых изменениях в вашем рабочем дереве, вы используете программу git update-index . Эта программа обычно просто берет список имен файлов, которые вы хотите обновить, но, чтобы избежать тривиальных ошибок, она отказывается добавлять новые записи в индекс (или удалять существующие), если вы явно не сообщите ей, что добавляете новую запись с помощью Флаг --add (или удаление записи с флагом --remove ).

Итак, чтобы заполнить индекс двумя только что созданными файлами, вы можете сделать

$ git update-index --добавить пример приветствия

и теперь вы сказали Git отслеживать эти два файла.

Фактически, когда вы это сделали, если вы теперь заглянете в свой каталог объектов, вы заметите, что Git добавил два новых объекта в базу данных объектов. Если вы выполнили именно описанные выше шаги, теперь вы сможете сделать

$ ls .git/objects/??/*

и видим два файла:

.git/objects/55/7db03de997c86a4a028e1ebd3a1ceb225be238
.git/objects/f2/4c74a2e500f5ee1332c86b94199f52b1d1d962

которые соответствуют объектам с именами 557db... и f24c7... соответственно.

Если вы хотите, вы можете использовать git cat-file для просмотра этих объектов, но вам придется использовать имя объекта, а не имя файла объекта:

$ git cat-file -t 557db03de997c86a4a028e1ebd3a1ceb225be238

где -t сообщает git cat-file , что такое "type" объекта. Git сообщит вам, что у вас есть объект "blob" (т. е. обычный файл), и вы можете увидеть его содержимое с помощью

$ git cat-file blob 557db03

который распечатает "Hello World" . Объект 557db03 — это не что иное, как содержимое вашего файла hello .

Note
 Не путайте этот объект с самим файлом hello . Объект буквально представляет собой конкретное содержимое файла, и сколько бы вы позже ни изменили содержимое файла hello , объект, который мы только что рассмотрели, никогда не изменится. Объекты неизменяемы.
Note
 Второй пример демонстрирует, что в большинстве мест имя объекта можно сократить до нескольких первых шестнадцатеричных цифр.

В любом случае, как мы упоминали ранее, вы обычно никогда не смотрите на сами объекты, а ввод длинных 40-символьных шестнадцатеричных имен — это не то, что вам обычно хотелось бы делать. Вышеупомянутое отступление было сделано только для того, чтобы показать, что git update-index сделал что-то волшебное и фактически сохранил содержимое ваших файлов в объектную базу данных Git .

Обновление индекса привело и к еще одному результату: был создан файл .git/index . Это индекс, описывающий ваше текущее рабочее дерево, и вам следует об этом хорошо знать. Опять же, вы обычно никогда не беспокоитесь о самом индексном файле, но вы должны осознавать тот факт, что на самом деле вы еще не превратили свои файлы в Git , вы только сообщили о них Git .

Однако, поскольку Git знает о них, теперь вы можете начать использовать некоторые из самых основных команд Git для управления файлами или просмотра их статуса.

В частности, давайте пока даже не будем регистрировать эти два файла в Git , мы начнем с добавления еще одной строки в hello :

$ echo "It's a new day for git" >>hello

и теперь вы можете, поскольку вы рассказали Git о предыдущем состоянии hello , спросить Git , что изменилось в дереве по сравнению с вашим старым индексом, с помощью команды git diff-files :

$ git diff-files

Упс. Это было не очень читабельно. Он просто выдает свою собственную внутреннюю версию diff , но эта внутренняя версия на самом деле просто сообщает вам, что он заметил, что "hello" был изменен, и что содержимое старого объекта было заменено чем-то другим.

Чтобы сделать его читабельным, мы можем указать git diff-files выводить различия в виде патча, используя флаг -p :

$ git diff-files -p
diff --git a/hello b/hello
index 557db03..263414f 100644
--- a/hello
+++ b/hello
@@ -1 +1,2 @@
 Hello World
+It's a new day for git

т.е. разница изменений, которые мы вызвали добавлением еще одной строки в hello .

Другими словами, git diff-files всегда показывает нам разницу между тем, что записано в индексе, и тем, что сейчас находится в рабочем дереве. Это очень полезно.

Обычное сокращение git diff-files -p — просто написать git diff , что делает то же самое.

$ git diff
diff --git а/привет б/привет
индекс 557db03..263414f 100644
--- a/hello
+++ b/hello
@@ -1 +1,2 @@
 Hello World
+It's a new day for git

Фиксация состояния git

Теперь мы хотим перейти к следующему этапу в Git : взять файлы, о которых Git знает в индексе, и зафиксировать их как настоящее дерево. Мы делаем это в два этапа: создаем объект tree и фиксируем этот объект tree как объект commit вместе с объяснением того, что представляет собой дерево, а также информацией о том, как мы пришли к этому состоянию.

Создание объекта дерева тривиально и выполняется с помощью git write-tree . Никаких опций или других входных данных нет: git write-tree возьмет текущее состояние индекса и напишет объект, описывающий весь этот индекс. Другими словами, теперь мы связываем вместе все различные имена файлов с их содержимым (и разрешениями) и создаем эквивалент объекта Git "directory" :

$ git write-tree

и это просто выведет имя полученного дерева, в этом случае (если вы сделали именно так, как я описал) оно должно быть

8988da15d077d4829fc51d8544c097def6644dbb

это еще одно непонятное имя объекта. Опять же, если вы хотите, вы можете использовать git cat-file -t 8988d... , чтобы увидеть, что на этот раз объект является не объектом "blob" , а объектом "tree" (вы также можете использовать git cat-file для фактического вывода необработанного содержимого объекта, но вы увидите в основном двоичный файл). беспорядок, так что это менее интересно).

Однако обычно вы никогда не будете использовать git write-tree отдельно, потому что обычно вы всегда фиксируете дерево в объект фиксации с помощью команды git commit-tree . На самом деле, проще вообще не использовать git write-tree отдельно, а просто передать его результат в качестве аргумента git commit-tree .

git commit-tree обычно принимает несколько аргументов — он хочет знать, каким был parent коммита, но поскольку это первый коммит в этом новом репозитории и у него нет родителей, нам нужно передать только имя объекта дерева. Однако git commit-tree также хочет получить сообщение о фиксации на свой стандартный ввод, и он запишет полученное имя объекта для фиксации на свой стандартный вывод.

И здесь мы создаем файл .git/refs/heads/master , на который указывает HEAD . Предполагается, что этот файл содержит ссылку на вершину дерева главной ветки, и поскольку git commit-tree выдает именно это, мы можем сделать все это с помощью последовательности простых команд оболочки:

$ tree=$(git write-tree)
$ commit=$(echo 'Initial commit' | git commit-tree $tree)
$ git update-ref HEAD $commit

В данном случае создается совершенно новый коммит, не связанный ни с чем другим. Обычно вы делаете это только один раз для проекта, и все последующие коммиты будут родительскими поверх более ранних коммитов.

Опять же, обычно вы никогда не сделаете это вручную. Существует полезный скрипт git commit , который сделает все это за вас. Таким образом, вместо этого вы могли бы просто написать git commit , и он выполнил бы за вас описанный выше волшебный сценарий.

Внесение изменений

Помните, как мы создали git update-index в файле hello , а затем изменили hello и смогли сравнить новое состояние hello с состоянием, которое мы сохранили в индексном файле?

Далее, помните, как я говорил, что git write-tree записывает в дерево содержимое индексного файла, и таким образом то, что мы только что зафиксировали, на самом деле было исходным содержимым файла hello , а не новым. Мы сделали это специально, чтобы показать разницу между состоянием индекса и состоянием рабочего дерева, а также то, что они не обязательно должны совпадать, даже когда мы что-то фиксируем.

Как и прежде, если мы воспользуемся git diff-files -p в нашем проекте git-tutorial, мы все равно увидим ту же разницу, что и в прошлый раз: индексный файл не изменился в процессе фиксации чего-либо. Однако теперь, когда мы что-то зафиксировали, мы также можем научиться использовать новую команду: git diff-index .

В отличие от git diff-files , который показывал разницу между индексным файлом и рабочим деревом, git diff-index показывает различия между зафиксированным деревом и индексным файлом или рабочим деревом. Другими словами, git diff-index хочет, чтобы дерево сравнивалось, но до того, как мы сделали фиксацию, мы не могли этого сделать, потому что нам не с чем сравнивать.

Но теперь мы можем сделать

$ git diff-index -p HEAD

(где -p имеет то же значение, что и в git diff-files ), и он покажет нам ту же разницу, но по совершенно другой причине. Теперь мы сравниваем рабочее дерево не с индексным файлом, а с деревом, которое мы только что написали. Так уж получилось, что эти два, очевидно, одинаковы, поэтому мы получаем один и тот же результат.

Опять же, поскольку это обычная операция, вы также можете просто сократить ее с помощью

$ git diff HEAD

который в конечном итоге сделает для вас все вышеперечисленное.

Другими словами, git diff-index обычно сравнивает дерево с рабочим деревом, но когда ему задан флаг --cached , ему предлагается вместо этого сравнивать только содержимое индексного кэша и полностью игнорировать текущее состояние рабочего дерева. Поскольку мы только что записали индексный файл в HEAD, выполнение git diff-index --cached -p HEAD должно вернуть пустой набор различий, и это именно то, что он делает.

Note

git diff-index действительно всегда использует индекс для своих сравнений, и поэтому утверждение, что он сравнивает дерево с рабочим деревом, не совсем точно. В частности, список файлов для сравнения ( "meta-data" ) всегда берется из индексного файла, независимо от того, используется флаг --cached или нет. Флаг --cached на самом деле определяет только то, поступает ли сравниваемое содержимое файла из рабочего дерева или нет.

Это нетрудно понять, как только вы поймете, что Git просто никогда не знает (или не заботится) о файлах, о которых ему не указано явно. Git никогда не будет искать файлы для сравнения, он ожидает, что вы сообщите ему, что это за файлы, и для этого и нужен индекс.

Однако наш следующий шаг — зафиксировать внесенное нами изменение, и опять же, чтобы понять, что происходит, имейте в виду разницу между "working tree contents" , "index file" и "committed tree" . У нас есть изменения в рабочем дереве, которые мы хотим закоммитить, и нам всегда приходится работать через индексный файл, поэтому первое, что нам нужно сделать, это обновить индексный кеш:

$ git update-index hello

(обратите внимание, что на этот раз нам не нужен был флаг --add , поскольку Git уже знал об этом файле).

Обратите внимание, что здесь происходит с различными версиями git diff-* . После того, как мы обновили индекс hello , git diff-files -p теперь не показывает никаких различий, но git diff-index -p HEAD по-прежнему показывает, что текущее состояние отличается от зафиксированного нами состояния. Фактически, теперь git diff-index показывает ту же разницу, используем ли мы флаг --cached или нет, поскольку теперь индекс согласован с рабочим деревом.

Теперь, поскольку мы обновили индекс hello , мы можем зафиксировать новую версию. Мы могли бы сделать это, снова написав дерево вручную и зафиксировав его (на этот раз нам придется использовать флаг -p HEAD , чтобы сообщить коммиту, что HEAD является родителем нового коммита, и что это не начальный коммит). зафиксируйте больше), но вы уже сделали это однажды, поэтому давайте на этот раз просто воспользуемся полезным скриптом:

$ git commit

который запускает редактор, в котором вы можете написать сообщение о фиксации, и немного рассказывает вам о том, что вы сделали.

Напишите любое сообщение, которое вы хотите, и все строки, начинающиеся с # , будут удалены, а остальные будут использоваться как сообщение о фиксации изменения. Если вы решите, что на этом этапе вы вообще ничего не хотите коммитировать (вы можете продолжать редактировать что-то и обновлять индекс), вы можете просто оставить пустое сообщение. В противном случае git commit зафиксирует изменение за вас.

Теперь вы сделали свой первый настоящий коммит Git . И если вам интересно посмотреть, что на самом деле делает git commit , не стесняйтесь исследовать: это несколько очень простых сценариев оболочки для создания полезных (?) заголовков сообщений о фиксации, и несколько однострочных строк, которые фактически выполняют саму фиксацию ( git commit ).

Inspecting changes

Вносить изменения полезно, но еще полезнее, если вы сможете позже сказать, что изменилось. Самая полезная для этого команда — другая из семейства diff , а именно git diff-tree .

git diff-tree можно задать два произвольных дерева, и он сообщит вам различия между ними. Однако, возможно, даже чаще вы можете передать ему только один объект коммита, и он сам вычислит родителя этого коммита и напрямую покажет разницу. Таким образом, чтобы получить тот же дифференциал, который мы уже видели несколько раз, теперь мы можем сделать

$ git diff-tree -p HEAD

(опять же, -p означает показать разницу в виде удобочитаемого патча), и он покажет, что на самом деле изменил последний коммит (в HEAD ).

Что еще интереснее, вы также можете назначить git diff-tree флаг --pretty , который предписывает ему также отображать сообщение о коммите, автора и дату коммита, и вы можете указать ему показывать целую серию различий. В качестве альтернативы вы можете указать, что это "silent" , и вообще не показывать различия, а просто показывать фактическое сообщение о фиксации.

Фактически, вместе с программой git rev-list (генерирующей список ревизий), git diff-tree оказывается настоящим кладезем изменений. Вы можете эмулировать git log , git log -p и т. д. с помощью тривиального сценария, который передает выходные данные git rev-list в git diff-tree --stdin , именно так были реализованы ранние версии git log .

Маркировка версии

В Git существует два типа тегов: "light" и "annotated tag" .

Тег "light" технически является не чем иным, как ветвью, за исключением того, что мы помещаем его в подкаталог .git/refs/tags/ вместо того, чтобы называть его head . Таким образом, простейшая форма тега включает в себя не что иное, как

$ git tag my-first-tag

который просто записывает текущий HEAD в файл .git/refs/tags/my-first-tag , после чего вы можете использовать это символическое имя для этого конкретного состояния. Вы можете, например, сделать

$ git diff my-first-tag

чтобы сравнить ваше текущее состояние с этим тегом, который на данный момент, очевидно, будет пустым различием, но если вы продолжите разрабатывать и фиксировать что-то, вы можете использовать свой тег как "anchor-point" , чтобы увидеть, что изменилось с тех пор, как вы его пометили.

"annotated tag" на самом деле является реальным объектом Git и содержит не только указатель на состояние, которое вы хотите пометить, но также небольшое имя и сообщение тега, а также, при необходимости, подпись PGP, которая говорит, что да, вы действительно сделали этот тег. Вы создаете эти аннотированные теги с флагом -a или -s для git tag :

$ git tag -s <tagname>

который будет подписывать текущую точку HEAD (но вы также можете передать ему другой аргумент, указывающий объект, который нужно пометить, например, вы могли бы пометить текущую точку mybranch с помощью git tag <tagname> mybranch ).

Обычно вы используете подписанные теги только для основных выпусков или тому подобного, в то время как облегченные теги полезны для любой маркировки, которую вы хотите сделать — каждый раз, когда вы решите, что хотите запомнить определенный момент, просто создайте для него тег private . , и у вас есть красивое символическое имя для состояния на данный момент.

Copying repositories

Репозитории Git обычно полностью самодостаточны и могут перемещаться. В отличие, например, от CVS, здесь нет отдельного понятия "repository" и "working tree" . Репозиторий Git обычно является рабочим деревом, а локальная информация Git скрыта в подкаталоге .git . Больше ничего нет. То, что вы видите, это то, что вы получили.

Note
 Вы можете указать Git отделить внутреннюю информацию Git от каталога, который он отслеживает, но пока мы проигнорируем это: обычные проекты работают не так, и на самом деле они предназначены только для специального использования. Таким образом, мысленная модель "the Git information is always tied directly to the working tree that it describes" , возможно, не является технически на 100% точной, но это хорошая модель для любого обычного использования.

Это имеет два значения:

  • если вам наскучил созданный вами репозиторий учебных пособий (или вы допустили ошибку и хотите начать все сначала), вы можете просто сделать

    $ rm -rf git-tutorial

    и оно исчезнет. Никакого внешнего репозитория и истории за пределами созданного вами проекта нет.

  • если вы хотите переместить или скопировать репозиторий Git , вы можете это сделать. Существует команда git clone , но если все, что вам нужно, — это просто создать копию вашего репозитория (со всей сопутствующей ей историей), вы можете сделать это с помощью обычного cp -a git-tutorial new-git-tutorial .

    Обратите внимание: когда вы переместили или скопировали репозиторий Git , ваш индексный файл Git (который кэширует различную информацию, в частности некоторую информацию "stat" для задействованных файлов), вероятно, потребуется обновить. Итак, после того, как вы создадите новую копию cp -a , вам нужно будет сделать

    $ git update-index --refresh

    в новом репозитории, чтобы убедиться, что индексный файл обновлен.

Обратите внимание, что второй пункт — true даже на разных машинах. Вы можете дублировать удаленный репозиторий Git с помощью любого обычного механизма копирования, будь то scp , rsync или wget .

При копировании удаленного репозитория вам нужно как минимум обновить кеш индекса, когда вы это делаете, и особенно в случае с репозиториями других людей вы часто хотите убедиться, что кеш индекса находится в каком-то известном состоянии (вы не знают, что они сделали и еще не зарегистрировались), поэтому обычно перед git update-index ставится

$ git read-tree --reset HEAD
$ git update-index --refresh

что приведет к перестроению общего индекса из дерева, на которое указывает HEAD . Он сбрасывает содержимое индекса на HEAD , а затем git update-index проверяет соответствие всех записей индекса извлеченным файлам. Если в рабочем дереве исходного репозитория были незафиксированные изменения, git update-index --refresh заметит их и сообщит, что их необходимо обновить.

Вышеупомянутое также можно записать просто

$ git reset

и на самом деле многие из распространенных комбинаций команд Git могут быть написаны с помощью интерфейсов git xyz . Вы можете многому научиться, просто взглянув на то, что делают различные сценарии git . Например, git reset раньше представлял собой две вышеуказанные строки, реализованные в git reset , но некоторые вещи, такие как git status и git commit , представляют собой немного более сложные сценарии вокруг основных команд Git .

Многие (большинство?) удаленных репозиториев public не будут содержать ни одного из извлеченных файлов или даже индексного файла, а будут содержать только фактические основные файлы Git . В таком репозитории обычно даже нет подкаталога .git , но все файлы Git находятся непосредственно в репозитории.

Чтобы создать собственную локальную действующую копию такого репозитория "raw" Git , сначала необходимо создать собственный подкаталог для проекта, а затем скопировать необработанное содержимое репозитория в каталог .git . Например, чтобы создать собственную копию репозитория Git , вам нужно сделать следующее:

$ mkdir my-git
$ cd my-git
$ rsync -rL rsync://rsync.kernel.org/pub/scm/git/git.git/ .git

followed by

$ git read-tree HEAD

для заполнения индекса. Однако теперь вы заполнили индекс и у вас есть все внутренние файлы Git , но вы заметите, что на самом деле у вас нет ни одного файла рабочего дерева, над которым можно было бы работать. Чтобы получить их, вы должны проверить их с помощью

$ git checkout-index -u -a

где флаг -u означает, что вы хотите, чтобы при оформлении заказа постоянно обновлялся индекс (чтобы вам не приходилось его обновлять впоследствии), а флаг -a означает "check out all files" (если у вас есть устаревшая копия или более старая версия извлеченное дерево, вам также может потребоваться сначала добавить флаг -f , чтобы указать git checkout-index принудительно перезаписывать любые старые файлы).

Опять же, все это можно упростить с помощью

$ git clone git://git.kernel.org/pub/scm/git/git.git/ my-git
$ cd my-git
$ git checkout

который в конечном итоге сделает все вышеперечисленное за вас.

Теперь вы успешно скопировали чужой (мой) удаленный репозиторий и проверили его.

Создание новой ветки

Ветви в Git на самом деле являются не чем иным, как указателями на базу данных объектов Git из подкаталога .git/refs/ , и, как мы уже обсуждали, ветвь HEAD представляет собой не что иное, как символическую ссылку на один из этих указателей объектов.

Вы можете в любой момент создать новую ветку, просто выбрав произвольную точку в истории проекта и просто записав имя SHA-1 этого объекта в файл под именем .git/refs/heads/ . Вы можете использовать любое имя файла (и даже подкаталоги), но по соглашению ветка "normal" называется master . Однако это всего лишь условность, и ничто не обеспечивает ее соблюдение.

Чтобы продемонстрировать это в качестве примера, давайте вернемся к репозиторию git-tutorial, который мы использовали ранее, и создадим в нем ветку. Вы делаете это, просто говоря, что хотите проверить новую ветку:

$ git switch -c mybranch

создаст новую ветку на основе текущей позиции HEAD и переключится на нее.

Note

Если вы примете решение начать новую ветку в какой-то момент истории, отличный от текущего HEAD , вы можете сделать это, просто указав git switch , какой будет база оформления заказа. Другими словами, если у вас есть более ранний тег или ветка, вам просто нужно сделать

$ git switch -c mybranch earlier-commit

и он создаст новую ветку mybranch при предыдущем коммите и проверит состояние на тот момент.

Вы всегда можете просто вернуться к исходной ветке master , выполнив

$ git switch master

(или любое другое имя ветки, если уж на то пошло), и если вы забудете, в какой ветке вы находитесь, просто

$ cat .git/HEAD

скажет вам, куда он указывает. Чтобы получить список имеющихся у вас ветвей, вы можете сказать

$ git branch

который раньше был не чем иным, как простым скриптом на основе ls .git/refs/heads . Перед веткой, в которой вы сейчас находитесь, будет отмечена звездочка.

Иногда вам может потребоваться создать новую ветку without , фактически проверив ее и переключившись на нее. Если да, просто используйте команду

$ git branch <branchname> [startingpoint]

который просто введет ветку create , но дальше ничего делать не будет. Позже вы сможете — как только вы решите, что действительно хотите разрабатывать эту ветку — переключиться на эту ветку с помощью обычного git switch с именем ветки в качестве аргумента.

Объединение двух ветвей

Одна из идей создания ветки заключается в том, что вы выполняете в ней некоторую (возможно, экспериментальную) работу и в конечном итоге объединяете ее обратно с основной веткой. Итак, предположим, что вы создали вышеупомянутую ветку mybranch , которая изначально была такой же, как исходная ветка master , давайте убедимся, что мы находимся в этой ветке, и проделаем там некоторую работу.

$ git switch mybranch
$ echo "Work, work, work" >>hello
$ git commit -m "Some work." -i hello

Здесь мы только что добавили еще одну строку в hello и использовали сокращение для git update-index hello и git commit , просто задав имя файла непосредственно git commit с флагом -i (он сообщает Git include об этом файле в дополнение к тому, что вы сделали для индексный файл на данный момент при совершении фиксации). Флаг -m предназначен для передачи сообщения журнала фиксации из командной строки.

Теперь, чтобы было немного интереснее, давайте предположим, что кто-то другой выполняет какую-то работу в исходной ветке, и смоделируем это, вернувшись в главную ветку и отредактировав там тот же файл по-другому:

$ git switch master

Здесь, взгляните на содержимое hello и обратите внимание, что оно не содержит работы, которую мы только что проделали в mybranch — потому что в ветке master этой работы вообще не было. Тогда делай

$ echo "Play, play, play" >>hello
$ echo "Lots of fun" >>example
$ git commit -m "Some fun." -i hello example

поскольку главная ветка явно в гораздо лучшем настроении.

Теперь у вас есть две ветки, и вы решаете объединить проделанную работу. Прежде чем мы это сделаем, давайте представим классный графический инструмент, который поможет вам увидеть, что происходит:

$ gitk --all

графически покажет вам обе ваши ветки (это то, что означает --all : обычно он просто показывает вам ваш текущий HEAD ) и их историю. Вы также можете увидеть, как именно они возникли из общего источника.

В любом случае, давайте выйдем из gitk ( ^Q или меню «Файл») и решим, что мы хотим объединить работу, которую мы проделали над веткой mybranch , с веткой master (которая в настоящее время также является нашей HEAD ). Для этого есть хороший скрипт под названием git merge , который хочет знать, какие ветки вы хотите разрешить и в чем суть слияния:

$ git merge -m "Merge work in mybranch" mybranch

где первый аргумент будет использоваться в качестве сообщения о фиксации, если слияние может быть разрешено автоматически.

Однако в этом случае мы намеренно создали ситуацию, когда слияние необходимо будет исправить вручную, поэтому Git сделает все возможное автоматически (в данном случае это просто слияние файла example , который не имел отличий в ветке mybranch ), и скажите:

        Auto-merging hello
        CONFLICT (content): Merge conflict in hello
        Automatic merge failed; fix conflicts and then commit the result.

В нем сообщается, что он выполнил "Automatic merge" , который вышел из строя из-за конфликтов в hello .

Не беспокоиться. Это оставило (тривиальный) конфликт в hello в той же форме, к которой вы уже должны были хорошо привыкнуть, если когда-либо использовали CVS, поэтому давайте просто откроем hello в нашем редакторе (каким бы он ни был) и исправим его как-нибудь. Я бы предложил просто сделать так, чтобы hello содержал все четыре строки:

Hello World
It's a new day for git
Play, play, play
Work, work, work

и как только вы будете довольны объединением вручную, просто выполните

$ git commit -i hello

который очень громко предупредит вас о том, что вы сейчас совершаете слияние (это правильно, так что неважно), и вы можете написать небольшое сообщение о слиянии о своих приключениях в git merge -land.

После завершения запустите gitk --all , чтобы графически увидеть, как выглядит история. Обратите внимание, что mybranch все еще существует, и вы можете переключиться на него и продолжить с ним работу, если захотите. Ветка mybranch не будет содержать слияние, но в следующий раз, когда вы объедините ее с веткой master , Git будет знать, как вы ее объединили, поэтому вам не придется выполнять слияние that снова.

Еще один полезный инструмент, особенно если вы не всегда работаете в среде X-Window, — git show-branch .

$ git show-branch --topo-order --more=1 мастер моей ветки
* [master] Merge work in mybranch
 ! [mybranch] Some work.
--
-  [master] Merge work in mybranch
*+ [mybranch] Some work.
*  [master^] Some fun.

Первые две строки указывают, что показаны две ветви с названиями их верхних коммитов, в настоящее время вы находитесь в ветке master (обратите внимание на символ звездочки * ), а первый столбец для последующих строк вывода: используется для отображения коммитов, содержащихся в ветке master , и второго столбца для ветки mybranch . Три коммита показаны вместе с их названиями. Все они имеют непустые символы в первом столбце ( * показывает обычный коммит в текущей ветке, - — коммит слияния), что означает, что теперь они являются частью ветки master . Только коммит "Some work" имеет символ плюс + во втором столбце, поскольку mybranch не был объединен для включения этих коммитов из основной ветки. Строка внутри скобок перед сообщением журнала фиксации — это короткое имя, которое можно использовать для обозначения фиксации. В приведенном выше примере master и mybranch являются головками ветвей. master^ является первым родителем главы ветки master . Если вы хотите увидеть более сложные случаи, см. gitrevisions[7] .

Note
 Без опции --more=1 git show-branch не будет выводить коммит [master^], поскольку коммит [mybranch] является общим предком советов master и mybranch. Подробную информацию см. в git-show-branch[1] .
Note
 Если бы после слияния в основной ветке было больше коммитов, сам коммит слияния не отображался бы в ветке git по умолчанию. В этом случае вам нужно будет предоставить опцию --sparse , чтобы сделать фиксацию слияния видимой.

Теперь давайте представим, что вы — тот, кто проделал всю работу над mybranch , и плод вашей тяжелой работы наконец-то был объединен с веткой master . Давайте вернемся к mybranch и запустим git merge , чтобы вернуть "upstream changes" в вашу ветку.

$ git switch mybranch
$ git merge -m "Merge upstream changes." master

Это выводит что-то вроде этого (фактические имена объектов фиксации будут другими)

Updating from ae3a2da... to a80b4aa....
Fast-forward (no commit created; -m option ignored)
 example | 1 +
 hello   | 1 +
 2 files changed, 2 insertions(+)

Поскольку ваша ветка не содержала ничего, кроме того, что уже было объединено с веткой master , операция слияния фактически не выполняла слияние. Вместо этого он просто обновил верхнюю часть дерева вашей ветки до ветки master . Это часто называют слиянием fast-forward .

Вы можете снова запустить gitk --all , чтобы увидеть, как выглядит предок фиксации, или запустить show-branch , который сообщит вам об этом.

$ git show-branch master mybranch
! [master] Merge work in mybranch
 * [mybranch] Merge work in mybranch
--
-- [master] Merge work in mybranch

Объединение внешней работы

Обычно гораздо чаще происходит слияние с кем-то другим, чем слияние со своими собственными ветвями, поэтому стоит отметить, что Git тоже делает это очень легко, и на самом деле это не сильно отличается от выполнения git merge . Фактически, удаленное слияние в конечном итоге представляет собой не что иное, как "fetch the work from a remote repository into a temporary tag" , за которым следует git merge .

Загрузка из удаленного репозитория, что неудивительно, осуществляется с помощью git fetch :

$ git fetch <remote-repository>

Для указания репозитория для загрузки можно использовать один из следующих транспортов:

SSH

remote.machine:/path/to/repo.git/ or

ssh://remote.machine/path/to/repo.git/

Этот транспорт можно использовать как для выгрузки, так и для скачивания, и он требует от вас привилегий входа в систему через ssh на удаленном компьютере. Он определяет набор объектов, которых не хватает другой стороне, путем обмена головными фиксациями, имеющимися на обоих концах, и передает (близкий к) минимальный набор объектов. На сегодняшний день это самый эффективный способ обмена объектами Git между репозиториями.

Local directory

/path/to/repo.git/

Этот транспорт аналогичен транспорту SSH, но использует sh для запуска обоих концов на локальном компьютере вместо запуска другого конца на удаленном компьютере через ssh .

Git Native

git://remote.machine/path/to/repo.git/

Этот транспорт был разработан для анонимного скачивания. Как и транспорт SSH, он определяет набор объектов, которых не хватает нисходящей стороне, и передает (близкий) минимальный набор объектов.

HTTP(S)

http://remote.machine/path/to/repo.git/

Загрузчик по URL-адресам http и https сначала получает имя самого верхнего объекта фиксации с удаленного сайта, просматривая указанное имя ссылки в каталоге repo.git/refs/ , а затем пытается получить объект фиксации, загружая из repo.git/objects/xx/xxx... , используя имя объекта этого объекта фиксации. Затем он считывает объект фиксации, чтобы узнать его родительские фиксации и связанный объект дерева; он повторяет этот процесс до тех пор, пока не получит все необходимые объекты. Из-за такого поведения их иногда еще называют commit walkers .

commit walkers иногда также называют dumb transports , поскольку для них не требуется какой-либо интеллектуальный сервер с поддержкой Git , в отличие от Git Native Transport. Подойдет любой стандартный HTTP-сервер, который даже не поддерживает индекс каталогов. Но вы должны подготовить свой репозиторий с помощью git update-server-info , чтобы помочь тупым загрузчикам транспорта.

Как только вы получите данные из удаленного репозитория, вы получите merge это с вашей текущей веткой.

Однако — это настолько обычное дело для fetch и тут же merge , что он называется git pull , и можно просто сделать

$ git pull <remote-repository>

и, при необходимости, укажите имя ветки для удаленного конца в качестве второго аргумента.

Note
 Вы можете обойтись вообще без использования каких-либо ветвей, сохранив столько локальных репозиториев, сколько хотите, и выполнить слияние между ними с помощью git pull, точно так же, как вы объединяете между собой ветки. Преимущество этого подхода заключается в том, что он позволяет вам сохранять набор файлов для каждого извлеченного branch , и вам может быть проще переключаться туда и обратно, если вы одновременно манипулируете несколькими линиями разработки. Конечно, вам придется заплатить больше за использование дискового пространства для хранения нескольких рабочих деревьев, но в наши дни дисковое пространство стоит дешево.

Вполне вероятно, что вы будете время от времени извлекать данные из одного и того же удаленного репозитория. Вкратце, вы можете сохранить URL-адрес удаленного репозитория в файле конфигурации локального репозитория следующим образом:

$ git config remote.linus.url https://git.kernel.org/pub/scm/git/git.git/

и используйте ключевое слово "linus" с git pull вместо полного URL-адреса.

Examples.

  1. git pull linus

  2. git pull linus tag v0.99.1

вышеперечисленное эквивалентно:

  1. git pull http://www.kernel.org/pub/scm/git/git.git/ HEAD

  2. git pull http://www.kernel.org/pub/scm/git/git.git/ tag v0.99.1

Как происходит слияние?

Мы сказали, что в этом уроке показано, как сантехника помогает вам справиться с несмываемым фарфором, но мы до сих пор не говорили о том, как на самом деле работает слияние. Если вы читаете это руководство впервые, я бы посоветовал перейти к разделу "Publishing your work" и вернуться сюда позже.

Хорошо, ты еще со мной? Чтобы дать нам пример, давайте вернемся к предыдущему репозиторию с файлами "hello" и "example" и вернемся в состояние перед слиянием:

$ git show-branch --more=2 мастер моей ветки
! [master] Merge work in mybranch
 * [mybranch] Merge work in mybranch
--
-- [мастер] Объединить работу в моей ветке
+* [master^2] Some work.
+* [master^] Some fun.

Помните, что перед запуском git merge наша головка master находилась в состоянии фиксации "Some fun." , а головка mybranch — в состоянии фиксации "Some work." .

$ git switch -C mybranch master^2
$ git switch master
$ git reset --hard master^

После перемотки структура коммита должна выглядеть так:

$ git show-branch
* [master] Some fun.
 ! [mybranch] Some work.
--
*  [master] Some fun.
 + [mybranch] Some work.
*+ [master^] Initial commit

Теперь мы готовы поэкспериментировать со слиянием вручную.

Команда git merge при объединении двух ветвей использует алгоритм трехстороннего слияния. Сначала он находит общего предка между ними. Используемая команда — git merge-base :

$ mb=$(git merge-base HEAD mybranch)

Команда записывает имя объекта фиксации общего предка в стандартный вывод, поэтому мы зафиксировали его вывод в переменной, поскольку мы будем использовать его на следующем шаге. Кстати, в данном случае общим предком является коммит "Initial commit" . Вы можете сказать это по:

$ git name-rev --name-only --tags $mb
my-first-tag

После выяснения общего предка, второй шаг заключается в следующем:

$ git read-tree -m -u $mb HEAD mybranch

Это та же самая команда git read-tree , которую мы уже видели, но она использует три дерева, в отличие от предыдущих примеров. При этом содержимое каждого дерева считывается в разные stage индексного файла (первое дерево переходит на этап 1, второе — на этап 2 и т. д.). После считывания трех деревьев на три этапа пути, одинаковые на всех трех этапах, переходят collapsed на этап 0. Кроме того, пути, одинаковые на двух из трех этапов, сворачиваются в этап 0, принимая SHA-1 из любого этапа 2. или стадия 3, в зависимости от того, какая из них отличается от стадии 1 (т.е. от общего предка изменилась только одна сторона).

После работы collapsing пути, разные в трёх деревьях, остаются на ненулевых стадиях. На этом этапе вы можете проверить индексный файл с помощью этой команды:

$ git ls-files --stage
100644 7f8b141b65fdcee47321e399a2598a235a032422 0        example
100644 557db03de997c86a4a028e1ebd3a1ceb225be238 1        hello
100644 ba42a2a96e3027f3333e13ede4ccf4498c3ae942 2        hello
100644 cc44c73eb783565da5831b4d820c962954019b69 3        hello

В нашем примере, состоящем всего из двух файлов, неизмененных файлов не было, поэтому схлопнулся только example . Но в реальных больших проектах, когда за один коммит изменяется лишь небольшое количество файлов, этот collapsing имеет тенденцию довольно быстро тривиально объединять большинство путей, оставляя лишь несколько реальных изменений на ненулевых стадиях.

Чтобы просмотреть только ненулевые этапы, используйте флаг --unmerged :

$ git ls-files --unmerged
100644 557db03de997c86a4a028e1ebd3a1ceb225be238 1        hello
100644 ba42a2a96e3027f3333e13ede4ccf4498c3ae942 2        hello
100644 cc44c73eb783565da5831b4d820c962954019b69 3        hello

Следующим шагом слияния является объединение этих трех версий файла с использованием трехстороннего слияния. Это делается путем указания команды git merge-one-file в качестве одного из аргументов команды git merge-index :

$ git merge-index git-merge-one-file hello
Auto-merging hello
ERROR: Merge conflict in hello
fatal: merge program failed

Сценарий git merge-one-file вызывается с параметрами для описания этих трех версий и отвечает за сохранение результатов слияния в рабочем дереве. Это довольно простой сценарий оболочки, который в конечном итоге вызывает программу merge из пакета RCS для выполнения трехстороннего слияния на уровне файла. В этом случае merge обнаруживает конфликты, а результат слияния с пометками конфликта остается в рабочем дереве. В этом можно убедиться, если еще раз запустить ls-files --stage на этом этапе:

$ git ls-files --stage
100644 7f8b141b65fdcee47321e399a2598a235a032422 0        example
100644 557db03de997c86a4a028e1ebd3a1ceb225be238 1        hello
100644 ba42a2a96e3027f3333e13ede4ccf4498c3ae942 2        hello
100644 cc44c73eb783565da5831b4d820c962954019b69 3        hello

Это состояние индексного файла и рабочего файла после того, как git merge возвращает вам управление, оставляя конфликтующее слияние на ваше усмотрение. Обратите внимание, что путь hello все еще не объединен, и то, что вы видите с git diff на этом этапе, — это различия со стадии 2 (т. е. вашей версии).

Публикация вашей работы

Итак, мы можем использовать чужую работу из удаленного репозитория, но как подготовить репозиторий, чтобы другие люди могли извлекать из него данные?

Вы выполняете свою настоящую работу в своем рабочем дереве, под которым в качестве подкаталога .git висит ваш основной репозиторий. Вы могли бы сделать этот репозиторий доступным удаленно и попросить людей извлекать из него данные, но на практике обычно это делается не так. Рекомендуемый способ — создать репозиторий public , сделать его доступным для других людей, а когда изменения, внесенные вами в основное рабочее дерево, будут в хорошем состоянии, обновить репозиторий public из него. Его часто называют pushing .

Note
 Этот репозиторий public можно дополнительно зеркалировать, и именно так управляются репозитории Git в kernel.org .

Для публикации изменений из вашего локального (частного) репозитория в удаленный (публичный) репозиторий требуются права записи на удаленном компьютере. Вам понадобится учетная запись SSH для запуска одной команды git-receive-pack .

Во-первых, вам нужно создать пустой репозиторий на удаленном компьютере, на котором будет размещен ваш репозиторий public . Этот пустой репозиторий будет заполнен и поддерживаться в актуальном состоянии путем добавления в него данных позже. Очевидно, что создание этого репозитория необходимо выполнить только один раз.

Note
 git push использует пару команд: git send-pack на локальном компьютере и git -receive-pack на удаленном компьютере. Для внутренней связи между ними по сети используется соединение SSH.

Каталог Git вашего репозитория private обычно называется .git , но ваш репозиторий public часто называется по имени проекта, т. е. <project>.git . Давайте создадим такой репозиторий public для проекта my-git . После входа на удаленный компьютер создайте пустой каталог:

$ mkdir my-git.git

Затем превратите этот каталог в репозиторий Git , запустив git init , но на этот раз, поскольку его имя отличается от обычного .git , мы поступим немного по-другому:

$ GIT_DIR=my-git.git git init

Убедитесь, что этот каталог доступен для других пользователей, которым вы хотите, чтобы ваши изменения были перенесены через выбранный вами транспорт. Также необходимо убедиться, что у вас на $PATH установлена ​​программа git-receive-pack .

Note
 Многие установки sshd не вызывают вашу оболочку в качестве оболочки входа в систему при непосредственном запуске программ; это означает, что если ваша оболочка входа — bash, читается только .bashrc , а не .bash_profile . В качестве обходного пути убедитесь, что .bashrc настроил $PATH , чтобы вы могли запускать программу git-receive-pack.
Note
 Если вы планируете опубликовать этот репозиторий для доступа через http, на этом этапе вам следует выполнить mv my-git.git/hooks/post-update.sample my-git.git/hooks/post-update . Это гарантирует, что каждый раз, когда вы вносите данные в этот репозиторий, запускается git update-server-info .

Теперь ваш "public repository" готов принять ваши изменения. Вернитесь к машине, у вас есть репозиторий private . Оттуда запустите эту команду:

$ git push <public-host>:/path/to/my-git.git master

Это синхронизирует ваш репозиторий public , чтобы он соответствовал названному заголовку ветки (т.е. в данном случае master ) и объектам, доступным из них в вашем текущем репозитории.

В качестве реального примера я обновляю свой репозиторий public Git . Зеркальная сеть Kernel.org заботится о распространении на другие видимые машины public :

$ git push master.kernel.org:/pub/scm/git/git.git/

Упаковка вашего репозитория

Ранее мы видели, что для каждого создаваемого вами объекта Git хранится один файл в каталоге .git/objects/??/ . Такое представление эффективно создавать атомарно и безопасно, но не так удобно транспортировать по сети. Поскольку объекты Git являются неизменяемыми после их создания, существует способ оптимизировать хранилище с помощью "packing them together" . Команда

$ git repack

сделаю это за тебя. Если бы вы следовали примерам руководства, к настоящему моменту у вас накопилось бы около 17 объектов в каталогах .git/objects/??/ . git repack сообщает вам, сколько объектов он упаковал, и сохраняет упакованный файл в каталоге .git/objects/pack .

Note
 Вы увидите два файла pack-*.pack и pack-*.idx в каталоге .git/objects/pack . Они тесно связаны друг с другом, и если вы по какой-либо причине когда-либо скопируете их вручную в другой репозиторий, обязательно скопируйте их вместе. Первый содержит все данные объектов в пакете, а второй — индекс для произвольного доступа.

Если вы параноик, запуск команды git verify-pack определит, есть ли у вас поврежденный пакет, но не беспокойтесь слишком сильно. Наши программы всегда идеальны ;-).

После того как вы запаковали объекты, вам больше не нужно оставлять распакованные объекты, содержащиеся в файле упаковки.

$ git prune-packed

удалил бы их для вас.

Если вам интересно, вы можете попробовать запустить find .git/objects -type f до и после запуска git prune-packed . Также git count-objects сообщит вам, сколько распакованных объектов находится в вашем репозитории и сколько места они занимают.

Note
 git pull немного громоздок для транспорта HTTP, поскольку упакованный репозиторий может содержать относительно мало объектов в относительно большом пакете. Если вы ожидаете, что из вашего репозитория public будет много запросов HTTP, возможно, вам захочется переупаковывать и сокращать его часто или никогда.

Если вы снова запустите git repack на этом этапе, он скажет "Nothing new to pack." . Как только вы продолжите разработку и накопите изменения, повторный запуск git repack создаст новый пакет, содержащий объекты, созданные с момента последней упаковки репозитория. Мы рекомендуем вам упаковать проект вскоре после первоначального импорта (если только вы не начинаете проект с нуля), а затем запускать git repack время от времени, в зависимости от того, насколько активен ваш проект.

Когда репозиторий синхронизируется через git push и git pull , объекты, упакованные в исходном репозитории, обычно хранятся в распакованном виде в месте назначения. Хотя это позволяет вам использовать разные стратегии упаковки на обоих концах, это также означает, что вам может потребоваться время от времени переупаковывать оба репозитория.

Работа с другими

Хотя Git — действительно распределенная система, часто бывает удобно организовать проект с помощью неформальной иерархии разработчиков. Разработка ядра Linux осуществляется таким образом. Есть хорошая иллюстрация (стр. 17, "Merges to Mainline" ) у Randy Dunlap’s presentation .

Следует подчеркнуть, что эта иерархия носит чисто неформальный характер. В Git нет ничего фундаментального, что навязывало бы "chain of patch flow" , подразумеваемую этой иерархией. Вам не обязательно получать данные только из одного удаленного репозитория.

Рекомендуемый рабочий процесс для "project lead" выглядит следующим образом:

  1. Подготовьте основной репозиторий на локальном компьютере. Ваша работа там закончена.

  2. Подготовьте репозиторий public , доступный другим.

    Если другие люди извлекают данные из вашего репозитория по тупым транспортным протоколам (HTTP), вам необходимо сохранить этот репозиторий dumb transport friendly . После git init $GIT_DIR/hooks/post-update.sample , скопированный из стандартных шаблонов, будет содержать вызов git update-server-info , но вам необходимо вручную включить перехватчик с помощью mv post-update.sample post-update . Это гарантирует, что git update-server-info будет поддерживать необходимые файлы в актуальном состоянии.

  3. Загрузите в репозиторий public из вашего основного репозитория.

  4. git repack репозиторий public . При этом создается большой пакет, содержащий исходный набор объектов в качестве базового и, возможно, git prune , если транспорт, используемый для извлечения из вашего репозитория, поддерживает упакованные репозитории.

  5. Продолжайте работать в своем основном репозитории. Ваши изменения включают в себя ваши собственные модификации, исправления, которые вы получаете по электронной почте, а также слияния, полученные в результате извлечения репозиториев "public" вашего "subsystem maintainers" .

    Вы можете переупаковать этот репозиторий private , когда захотите.

  6. Отправьте свои изменения в репозиторий public и сообщите об этом public .

  7. Время от времени git repack — репозиторий public . Вернитесь к шагу 5. и продолжайте работу.

Рекомендуемый рабочий цикл для "subsystem maintainer" , который работает над этим проектом и имеет собственный "public repository" , выглядит следующим образом:

  1. Подготовьте рабочий репозиторий, запустив git clone в репозитории public устройства "project lead" . URL-адрес, используемый для первоначального клонирования, хранится в переменной конфигурации Remote.origin.url.

  2. Подготовьте репозиторий public , доступный другим, как это делает пользователь "project lead" .

  3. Скопируйте упакованные файлы из репозитория "project lead" public в репозиторий public , если только репозиторий "project lead" не находится на том же компьютере, что и ваш. В последнем случае вы можете использовать файл objects/info/alternates , чтобы указать репозиторий, из которого вы заимствуете.

  4. Загрузите в репозиторий public из вашего основного репозитория. Запустите git repack и, возможно, git prune , если транспорт, используемый для извлечения из вашего репозитория, поддерживает упакованные репозитории.

  5. Продолжайте работать в своем основном репозитории. Ваши изменения включают в себя ваши собственные модификации, исправления, которые вы получаете по электронной почте, а также слияния, полученные в результате извлечения репозиториев "public" вашего "project lead" и, возможно, вашего "sub-subsystem maintainers" .

    Вы можете переупаковать этот репозиторий private , когда захотите.

  6. Отправьте изменения в репозиторий public и попросите "project lead" и, возможно, "sub-subsystem maintainers" извлечь из него изменения.

  7. Время от времени git repack — репозиторий public . Вернитесь к шагу 5. и продолжайте работу.

Рекомендуемый рабочий цикл для "individual developer" , у которого нет репозитория "public" , несколько отличается. Это выглядит следующим образом:

  1. Подготовьте свой рабочий репозиторий: git clone — репозиторий public для "project lead" (или "subsystem maintainer" , если вы работаете с подсистемой). URL-адрес, используемый для первоначального клонирования, хранится в переменной конфигурации Remote.origin.url.

  2. Делайте свою работу в своем репозитории в ветке master .

  3. Время от времени запускайте git fetch origin из репозитория public вашего восходящего сервера. Это делает только первую половину git pull но не сливает. Голова репозитория public хранится в .git/refs/remotes/origin/master .

  4. Используйте git cherry origin , чтобы узнать, какие из ваших исправлений были приняты, и/или используйте git rebase origin для переноса несвязанных изменений в обновленный восходящий поток.

  5. Используйте git format-patch origin для подготовки исправлений для отправки по электронной почте в вышестоящий отдел и отправки их. Вернитесь к шагу 2 и продолжайте.

Работа с другими, стиль общего репозитория

Если у вас есть опыт работы с CVS, стиль сотрудничества, предложенный в предыдущем разделе, может быть для вас новым. Вы не должны беспокоиться. Git поддерживает стиль взаимодействия "shared public repository" , с которым вы, вероятно, также более знакомы.

Подробности см. в gitcvs-migration[7] .

Объединение вашей работы вместе

Вполне вероятно, что вы будете работать над несколькими делами одновременно. Этими более или менее независимыми задачами легко управлять, используя ветки Git.

Ранее мы уже видели, как работают ветки, на примере "fun and work" , использующего две ветки. Идея та же, если ветвей больше двух. Допустим, вы начали с головы "master" и у вас есть новый код в ветке "master" и два независимых исправления в ветках "commit-fix" и "diff-fix" :

$ git show-branch
! [commit-fix] Fix commit message normalization.
 ! [diff-fix] Fix rename detection.
  * [master] Release candidate #1
---
 +  [diff-fix] Fix rename detection.
 +  [diff-fix~1] Better common substring algorithm.
+   [commit-fix] Fix commit message normalization.
  * [master] Release candidate #1
++* [diff-fix~2] Pretty-print messages.

Оба исправления хорошо протестированы, и на данный момент вы хотите объединить их оба. Вы можете сначала объединить diff-fix , а затем commit-fix , вот так:

$ git merge -m "Merge fix in diff-fix" diff-fix
$ git merge -m "Merge fix in commit-fix" commit-fix

Что приведет к:

$ git show-branch
! [commit-fix] Fix commit message normalization.
 ! [diff-fix] Fix rename detection.
  * [master] Merge fix in commit-fix
---
  - [master] Merge fix in commit-fix
+ * [commit-fix] Fix commit message normalization.
  - [master~1] Merge fix in diff-fix
 +* [diff-fix] Fix rename detection.
 +* [diff-fix~1] Better common substring algorithm.
  * [master~2] Release candidate #1
++* [master~3] Pretty-print messages.

Однако нет особой причины сначала объединять одну ветку, а затем другую, когда у вас есть набор действительно независимых изменений (если порядок имеет значение, то они не являются независимыми по определению). Вместо этого вы можете объединить эти две ветки в текущую ветку одновременно. Сначала давайте отменим то, что мы только что сделали, и начнем заново. Мы хотели бы получить главную ветку до того, как эти две объединятся, сбросив ее на master~2 :

$ git reset --hard master~2

Вы можете убедиться, что git show-branch соответствует состоянию перед теми двумя git merge , которые вы только что сделали. Затем вместо запуска двух команд git merge подряд вы должны объединить эти две ветки (это известно как making an Octopus ):

$ git merge commit-fix diff-fix
$ git show-branch
! [commit-fix] Fix commit message normalization.
 ! [diff-fix] Fix rename detection.
  * [master] Octopus merge of branches 'diff-fix' and 'commit-fix'
---
  - [master] Octopus merge of branches 'diff-fix' and 'commit-fix'
+ * [commit-fix] Fix commit message normalization.
 +* [diff-fix] Fix rename detection.
 +* [diff-fix~1] Better common substring algorithm.
  * [master~1] Release candidate #1
++* [master~2] Pretty-print messages.

Обратите внимание, что вам не следует использовать Octopus только потому, что вы можете. Осьминог — это допустимая вещь, и он часто упрощает просмотр истории коммитов, если вы одновременно объединяете более двух независимых изменений. Однако, если у вас есть конфликты слияния с какой-либо из веток, в которые вы объединяете, и вам нужно разрешить их вручную, это указывает на то, что разработка произошла в этих ветвях, в конце концов, не были независимыми, и вам следует объединить две одновременно, документируя, как вы разрешили конфликты, а также причину, по которой вы предпочли изменения, внесенные с одной стороны, а не с другой. В противном случае следить за историей проекта будет труднее, а не проще.
