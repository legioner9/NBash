githooks

Name

githooks — хуки, используемые Git

Synopsis

$GIT_DIR/hooks/* (или `git config core.hooksPath`/*)

Description

Хуки — это программы, которые вы можете поместить в каталог хуков для запуска действий в определенных точках выполнения git. Перехватчики, у которых не установлен бит исполняемого файла, игнорируются.

По умолчанию каталог перехватчиков — $GIT_DIR/hooks , но его можно изменить с помощью переменной конфигурации core.hooksPath (см. git-config[1] ).

Прежде чем Git вызовет перехват, он меняет свой рабочий каталог либо на $GIT_DIR в пустом репозитории, либо на корень рабочего дерева в непустом репозитории. Исключением являются перехваты, срабатывающие во время push ( pre-receive , update , post-receive , post-update , push-to-checkout ), которые всегда выполняются в $GIT_DIR.

Переменные среды, такие как GIT_DIR , GIT_WORK_TREE и т. д., экспортируются, чтобы команды Git , запускаемые перехватчиком, могли правильно найти репозиторий. Если вашему перехватчику необходимо вызывать команды Git во внешнем репозитории или в другом рабочем дереве того же репозитория, ему следует очистить эти переменные среды, чтобы они не мешали операциям Git во внешнем хранилище. Например:

local_desc=$(git describe)
foreign_desc=$(unset $(git rev-parse --local-env-vars); git -C ../foreign-repo описать)

Хуки могут получать свои аргументы через среду, аргументы командной строки и стандартный ввод. Подробности смотрите в документации по каждому хуку ниже.

git init может копировать перехватчики в новый репозиторий, в зависимости от его конфигурации. Подробности см. в разделе "TEMPLATE DIRECTORY" в git-init[1] . Когда остальная часть этого документа ссылается на "default hooks" , речь идет о шаблоне по умолчанию, поставляемом с Git.

Поддерживаемые в настоящее время перехватчики описаны ниже.

Hooks

applypatch-msg

Этот хук вызывается git-am[1] . Он принимает один параметр — имя файла, содержащего предлагаемое сообщение журнала фиксации. Выход с ненулевым статусом приводит к аварийному завершению работы git am перед применением исправления.

Хук позволяет редактировать файл сообщения на месте и может использоваться для нормализации сообщения в некотором стандартном формате проекта. Его также можно использовать для отказа от фиксации после проверки файла сообщения.

Перехватчик applypatch-msg по умолчанию, если он включен, запускает перехватчик commit-msg , если последний включен.

pre-applypatch

Этот хук вызывается git-am[1] . Он не принимает параметров и вызывается после применения исправления, но до выполнения фиксации.

Если он завершается с ненулевым статусом, то рабочее дерево не будет зафиксировано после применения патча.

Его можно использовать для проверки текущего рабочего дерева и отказа от фиксации, если оно не проходит определенные тесты.

Перехватчик pre-applypatch по умолчанию, если он включен, запускает перехватчик pre-commit , если последний включен.

post-applypatch

Этот хук вызывается git-am[1] . Он не принимает параметров и вызывается после применения исправления и выполнения фиксации.

Этот перехват предназначен в первую очередь для уведомления и не может повлиять на результат git am .

pre-commit

Этот хук вызывается git-commit[1] , и его можно обойти с помощью опции --no-verify . Он не принимает никаких параметров и вызывается перед получением предложенного сообщения журнала фиксации и совершением фиксации. Выход из этого сценария с ненулевым статусом приводит к прерыванию выполнения команды git commit перед созданием фиксации.

Перехватчик pre-commit по умолчанию, если он включен, перехватывает строки с конечными пробелами и прерывает фиксацию, когда такая строка найдена.

Все перехватчики git commit вызываются с переменной среды GIT_EDITOR=: , если команда не вызывает редактор для изменения сообщения фиксации.

Перехватчик pre-commit по умолчанию, когда он включен, а также если параметр конфигурации hooks.allownonascii не установлен или установлен в значение false, предотвращает использование имен файлов, отличных от ASCII.

pre-merge-commit

Этот хук вызывается git-merge[1] , и его можно обойти с помощью опции --no-verify . Он не принимает никаких параметров и вызывается после успешного выполнения слияния и до получения предлагаемого сообщения журнала фиксации для выполнения фиксации. Выход из этого сценария с ненулевым статусом приводит к прерыванию выполнения команды git merge перед созданием фиксации.

Перехватчик pre-merge-commit по умолчанию, если он включен, запускает перехватчик pre-commit , если последний включен.

Этот хук вызывается с переменной среды GIT_EDITOR=: , если команда не вызывает редактор для изменения сообщения фиксации.

Если слияние не может быть выполнено автоматически, конфликты необходимо разрешить и результат зафиксировать отдельно (см. git-merge[1] ). В этот момент этот перехватчик не будет выполнен, а перехватчик pre-commit будет выполнен, если он включен.

prepare-commit-msg

Этот хук вызывается git-commit[1] сразу после подготовки сообщения журнала по умолчанию и перед запуском редактора.

Требуется от одного до трех параметров. Первый — это имя файла, содержащего сообщение журнала фиксации. Второй является источником сообщения о фиксации и может быть: message (если была указана опция -m или -F ); template (если была указана опция -t или установлена ​​опция конфигурации commit.template ); merge (если коммит представляет собой слияние или существует файл .git/MERGE_MSG ); squash (если файл .git/SQUASH_MSG существует); или commit , за которым следует имя объекта фиксации (если была указана опция -c , -C или --amend ).

Если статус выхода не равен нулю, git commit прервется.

Целью перехватчика является редактирование файла сообщения на месте, и оно не подавляется опцией --no-verify . Ненулевой выход означает сбой перехватчика и прерывает фиксацию. Его не следует использовать в качестве замены хука предварительной фиксации.

Пример перехватчика prepare-commit-msg , поставляемый с Git , удаляет справочное сообщение, найденное в закомментированной части шаблона фиксации.

commit-msg

Этот хук вызывается git-commit[1] и git-merge[1] , и его можно обойти с помощью опции --no-verify . Он принимает один параметр — имя файла, содержащего предлагаемое сообщение журнала фиксации. Выход с ненулевым статусом приводит к прерыванию команды.

Хук позволяет редактировать файл сообщения на месте и может использоваться для нормализации сообщения в некотором стандартном формате проекта. Его также можно использовать для отказа от фиксации после проверки файла сообщения.

Перехватчик commit-msg по умолчанию, если он включен, обнаруживает повторяющиеся трейлеры Signed-off-by и прерывает фиксацию, если таковой обнаружен.

post-commit

Этот хук вызывается git-commit[1] . Он не принимает параметров и вызывается после выполнения фиксации.

Этот перехват предназначен в первую очередь для уведомления и не может повлиять на результат git commit .

pre-rebase

Этот хук вызывается git-rebase[1] и может использоваться для предотвращения перебазирования ветки. Хук может быть вызван с одним или двумя параметрами. Первый параметр — это восходящий поток, из которого была создана серия. Второй параметр — это перебазируемая ветвь, и он не задается при перебазировании текущей ветки.

post-checkout

Этот перехват вызывается, когда git-checkout[1] или git-switch[1] запускается после обновления рабочего дерева. Перехватчику даются три параметра: ссылка на предыдущую HEAD, ссылка на новую HEAD (которая могла измениться или не измениться) и флаг, указывающий, была ли проверка проверкой ветки (изменение ветвей, флаг = 1) или извлечение файла (извлечение файла из индекса, флаг=0). Этот перехватчик не может повлиять на результат git switch или git checkout , за исключением того, что статус выхода перехватчика становится статусом завершения этих двух команд.

Он также запускается после git-clone[1] , если не используется опция --no-checkout ( -n ). Первый параметр, передаваемый перехватчику, — это нулевая ссылка, второй — ссылка на новый HEAD, а флаг всегда равен 1. Аналогично для git worktree add , если не используется --no-checkout .

Этот хук можно использовать для проверки достоверности репозитория, автоматического отображения отличий от предыдущего HEAD, если он отличается, или установки свойств метаданных рабочего каталога.

post-merge

Этот хук вызывается git-merge[1] , что происходит, когда git pull выполняется в локальном репозитории. Перехватчик принимает единственный параметр — флаг состояния, определяющий, было ли выполненное слияние сквош-слиянием или нет. Этот перехватчик не может повлиять на результат git merge и не выполняется, если слияние не удалось из-за конфликтов.

Этот крючок можно использовать в сочетании с соответствующим хуком предварительной фиксации для сохранения и восстановления любой формы метаданных, связанных с рабочим деревом (например: разрешения/владение, ACLS и т. д.). См. contrib/hooks/setgitperms.perl для примера того, как это сделать.

pre-push

Этот хук вызывается git-push[1] и может использоваться для предотвращения push-уведомлений. Перехватчик вызывается с двумя параметрами, которые предоставляют имя и местоположение удаленного устройства назначения. Если именованный удаленный компьютер не используется, оба значения будут одинаковыми.

Информация о том, что должно быть отправлено, предоставляется на стандартном вводе перехватчика в виде строк вида:

<local ref> SP <local object name> SP <remote ref> SP <remote object name> LF

Например, если бы была запущена команда git push origin master:foreign , перехватчик получил бы строку, подобную следующей:

refs/heads/master 67890 refs/heads/foreign 12345

хотя будет указано полное имя объекта. Если внешняя ссылка еще не существует, <remote object name> будет именем объекта, состоящим из всех нулей. Если ссылку необходимо удалить, <local ref> будет предоставлен как (delete) , а <local object name> будет именем объекта, состоящим из всех нулей. Если локальная фиксация была указана чем-то отличным от имени, которое можно было бы расширить (например, HEAD~ или имя объекта), оно будет предоставлено в том виде, в каком оно было изначально задано.

Если этот перехватчик завершается с ненулевым статусом, git push прервется, ничего не отправляя. Информация о том, почему push-уведомление отклонено, может быть отправлено пользователю путем записи в стандартную ошибку.

pre-receive

Этот хук вызывается git-receive-pack[1] , когда он реагирует на git push и обновляет ссылки в своем репозитории. Непосредственно перед началом обновления ссылок в удаленном репозитории вызывается перехватчик предварительного получения. Его статус завершения определяет успех или неудачу обновления.

Этот хук выполняется один раз для операции приема. Он не принимает аргументов, но для каждой обновляемой ссылки он получает на стандартный ввод строку формата:

<old-value> SP <new-value> SP <ref-name> LF

где <old-value> — это старое имя объекта, хранящееся в ссылке, <new-value> — новое имя объекта, которое будет храниться в ссылке, а <ref-name> — полное имя ссылки. При создании новой ссылки <old-value> представляет собой имя объекта, состоящее из всех нулей.

Если перехват завершается с ненулевым статусом, none ссылок будет обновлен. Если перехват завершается с нулем, обновление отдельных ссылок все равно можно предотвратить с помощью перехватчика update .

Как стандартный вывод, так и стандартный вывод ошибок перенаправляются на git send-pack на другом конце, поэтому вы можете просто отправлять сообщения echo для пользователя.

Количество опций push, заданных в командной строке git push --push-option=... , можно прочитать из переменной среды GIT_PUSH_OPTION_COUNT , а сами параметры находятся в GIT_PUSH_OPTION_0 , GIT_PUSH_OPTION_1 ,… Если согласовано не использовать фазу опций push, переменные среды не будут быть установлен. Если клиент выбирает использование опций push, но не передает их, переменная count будет установлена ​​в ноль, GIT_PUSH_OPTION_COUNT=0 .

Некоторые предостережения см. в разделе "Quarantine Environment" в git-receive-pack[1] .

update

Этот хук вызывается git-receive-pack[1] , когда он реагирует на git push и обновляет ссылки в своем репозитории. Непосредственно перед обновлением ссылки в удаленном репозитории вызывается перехватчик обновления. Его статус выхода определяет успех или неудачу обновления ссылки.

Хук выполняется один раз для каждой обновляемой ссылки и принимает три параметра:

  • имя обновляемой ссылки,

  • старое имя объекта, хранящееся в ссылке,

  • и новое имя объекта, которое будет сохранено в ссылке.

Нулевой выход из хука обновления позволяет обновить ссылку. Выход с ненулевым статусом не позволяет git receive-pack обновить эту ссылку.

Этот крючок можно использовать для предотвращения обновления forced для определенных ссылок, убедившись, что имя объекта является объектом фиксации, который является потомком объекта фиксации, названного по старому имени объекта. То есть для обеспечения соблюдения политики "fast-forward only" .

Его также можно использовать для регистрации статуса «старый…новый». Однако он не знает всего набора ветвей, поэтому при наивном использовании он будет отправлять одно электронное письмо на каждую ссылку. Крючок post-receive для этого больше подходит.

В среде, которая ограничивает доступ пользователей только к командам git по проводной сети, этот перехватчик можно использовать для реализации контроля доступа, не полагаясь на владение файловой системой и членство в группах. См. git-shell[1] , чтобы узнать, как можно использовать оболочку входа в систему, чтобы ограничить доступ пользователя только к командам git .

Как стандартный вывод, так и стандартный вывод ошибок перенаправляются на git send-pack на другом конце, поэтому вы можете просто отправлять сообщения echo для пользователя.

Перехватчик update по умолчанию, когда он включен, а также если параметр конфигурации hooks.allowunannotated не установлен или установлен в значение false, предотвращает отправку неаннотированных тегов.

proc-receive

Этот хук вызывается git-receive-pack[1] . Если сервер установил многозначную переменную конфигурации receive.procReceiveRefs и команды, отправленные в receive-pack , имеют совпадающие ссылочные имена, эти команды будут выполняться этим перехватчиком, а не внутренней функцией execute_commands() . Этот хук отвечает за обновление соответствующих ссылок и передачу результатов обратно в receive-pack .

Этот хук выполняется один раз для операции приема. Он не принимает аргументов, но использует протокол формата pkt-line для связи с receive-pack для чтения команд, опций push-уведомлений и отправки результатов. В следующем примере протокола буква S обозначает receive-pack , а буква H — этот крючок.

# Обсуждение версии и функций.
S: PKT-LINE(version=1\0push-options atomic...)
S: flush-pkt
H: PKT-LINE(version=1\0push-options...)
H: flush-pkt
# Send commands from server to the hook.
S: PKT-LINE(<old-oid> <new-oid> <ref>)
S: ... ...
S: flush-pkt
# Send push-options only if the Функция «push-options» включена.
S: PKT-LINE(push-option)
S: ... ...
S: flush-pkt
# Получаем результаты от хука.
# ОК, успешно выполните эту команду.
H: PKT-LINE(ok <ref>)
# НЕТ, я отвергаю это.
H: PKT-LINE(ng <ref> <reason>)
# Провалиться, пусть 'receive-pack' выполнит его.
H: PKT-LINE(ok <ref>)
H: PKT-LINE(option fall-through)
# Хорошо, но есть альтернативная ссылка. Альтернативное имя ссылки
# и другой статус можно указать в директивах опций.
H: PKT-LINE(ok <ref>)
H: PKT-LINE(option refname <refname>)
H: PKT-LINE(option old-oid <old-oid>)
H: PKT-LINE(option new-oid <new-oid>)
H: PKT-LINE(option forced-update)
H: ... ...
H: flush-pkt

Каждая команда для перехватчика proc-receive может указывать на псевдоссылку и всегда имеет нулевой старый идентификатор в качестве старого идентификатора, в то время как перехватчик proc-receive может обновлять альтернативную ссылку, и альтернативная ссылка может уже существовать с ненулевым старым идентификатором. . В этом случае этот перехватчик будет использовать директивы "option" для сообщения о расширенных атрибутах для ссылки, заданной ведущей директивой "ok" .

Отчет о командах этого хука должен иметь тот же порядок, что и входные данные. Статус выхода перехватчика proc-receive определяет только успех или неудачу отправленной ему группы команд, если только не используется атомарная отправка.

post-receive

Этот хук вызывается git-receive-pack[1] , когда он реагирует на git push и обновляет ссылки в своем репозитории. Он выполняется в удаленном репозитории один раз после обновления всех ссылок.

Этот хук выполняется один раз для операции приема. Он не принимает аргументов, но получает ту же информацию, что и перехватчик pre-receive на свой стандартный ввод.

Этот хук не влияет на результат git receive-pack , поскольку он вызывается после выполнения реальной работы.

Это заменяет перехватчик post-update , поскольку он получает как старые, так и новые значения всех ссылок в дополнение к их именам.

Как стандартный вывод, так и стандартный вывод ошибок перенаправляются на git send-pack на другом конце, поэтому вы можете просто отправлять сообщения echo для пользователя.

Перехватчик post-receive по умолчанию пуст, но в каталоге contrib/hooks дистрибутива Git имеется образец сценария post-receive-email , который реализует отправку сообщений о фиксации по электронной почте.

Количество опций push, заданных в командной строке git push --push-option=... , можно прочитать из переменной среды GIT_PUSH_OPTION_COUNT , а сами параметры находятся в GIT_PUSH_OPTION_0 , GIT_PUSH_OPTION_1 ,… Если согласовано не использовать фазу опций push, переменные среды не будут быть установлен. Если клиент выбирает использование опций push, но не передает их, переменная count будет установлена ​​в ноль, GIT_PUSH_OPTION_COUNT=0 .

post-update

Этот хук вызывается git-receive-pack[1] , когда он реагирует на git push и обновляет ссылки в своем репозитории. Он выполняется в удаленном репозитории один раз после обновления всех ссылок.

Он принимает переменное количество параметров, каждый из которых представляет собой имя ссылки, которая была фактически обновлена.

Этот перехват предназначен в первую очередь для уведомления и не может повлиять на результат git receive-pack .

Перехватчик post-update может определить, какие головки были отправлены, но он не знает, каковы их исходные и обновленные значения, поэтому это плохое место для ведения журнала «старое...новое». Хук post-receive получает как исходные, так и обновленные значения ссылок. Вместо этого вы можете рассмотреть это, если они вам нужны.

Если этот параметр включен, перехватчик post-update по умолчанию запускает git update-server-info , чтобы поддерживать актуальность информации, используемой простыми транспортами (например, HTTP). Если вы публикуете репозиторий Git , доступный через HTTP, вам, вероятно, следует включить этот перехват.

Как стандартный вывод, так и стандартный вывод ошибок перенаправляются на git send-pack на другом конце, поэтому вы можете просто отправлять сообщения echo для пользователя.

reference-transaction

Этот перехват вызывается любой командой Git , выполняющей эталонные обновления. Он выполняется всякий раз, когда эталонная транзакция готовится, фиксируется или прерывается, и поэтому может вызываться несколько раз. Хук не распространяется на символические ссылки (но это может измениться в будущем).

Хук принимает ровно один аргумент, который представляет собой текущее состояние, в котором находится данная ссылочная транзакция:

  • "prepared" : Все обновления ссылок были поставлены в очередь транзакции, а ссылки заблокированы на диске.

  • "committed" : Ссылочная транзакция была зафиксирована, и все ссылки теперь имеют соответствующее новое значение.

  • "aborted" : Эталонная транзакция была прервана, никаких изменений не было выполнено, блокировки сняты.

Для каждого обновления ссылки, которое было добавлено в транзакцию, перехватчик получает на стандартный ввод строку формата:

<old-value> SP <new-value> SP <ref-name> LF

где <old-value> — имя старого объекта, переданное в ссылочную транзакцию, <new-value> — новое имя объекта, которое будет сохранено в ссылке, а <ref-name> — полное имя ссылки. При принудительном обновлении ссылки независимо от ее текущего значения или когда ссылка должна быть создана заново, <old-value> — это имя объекта, состоящее из всех нулей. Чтобы отличить эти случаи, вы можете проверить текущее значение <ref-name> через git rev-parse .

Статус выхода перехватчика игнорируется для любого состояния, кроме состояния "prepared" . В состоянии "prepared" ненулевой статус выхода приведет к прерыванию транзакции. В этом случае перехватчик не будет вызываться с состоянием "aborted" .

push-to-checkout

Этот хук вызывается git-receive-pack[1] , когда он реагирует на git push и обновляет ссылки в своем репозитории, а также когда push пытается обновить ветку, которая в данный момент извлечена, и для переменной конфигурации receive.denyCurrentBranch установлено значение updateInstead . Такая передача по умолчанию отклоняется, если рабочее дерево и индекс удаленного репозитория отличаются от текущего извлеченного коммита; когда и рабочее дерево, и индекс соответствуют текущему коммиту, они обновляются, чтобы соответствовать вновь отправленному кончику ветки. Этот хук должен использоваться для переопределения поведения по умолчанию.

Хук получает коммит, с помощью которого будет обновлен кончик текущей ветки. Он может выйти с ненулевым статусом, чтобы отказаться от отправки (при этом он не должен изменять индекс или рабочее дерево). Либо он может внести любые необходимые изменения в рабочее дерево и индекс, чтобы привести их в нужное состояние, когда верхушка текущей ветки обновится до нового коммита, и выйти с нулевым статусом.

Например, перехватчик может просто запустить git read-tree -u -m HEAD "$1" , чтобы эмулировать git fetch , который запускается в обратном направлении с git push , поскольку двухдеревьевая форма git read-tree -u -m по сути такая же, как git switch или git checkout , который переключает ветви, сохраняя при этом локальные изменения в рабочее дерево, не мешающее разнице между ветвями.

pre-auto-gc

Этот хук вызывается git gc --auto (см. git-gc[1] ). Он не принимает никаких параметров, и выход из этого сценария с ненулевым статусом приводит к аварийному завершению работы git gc --auto .

post-rewrite

Этот хук вызывается командами, которые перезаписывают фиксации ( git-commit[1] при вызове с --amend и git-rebase[1] ; однако инструменты перезаписи полной истории, такие как git-fast-import[1] или git-filter-repo , обычно не вызывают его!). Его первый аргумент обозначает команду, которой он был вызван: в настоящее время это одна из amend или rebase . В будущем могут быть переданы дополнительные аргументы, зависящие от команды.

Хук получает на стандартный ввод список переписанных коммитов в формате

<old-object-name> SP <new-object-name> [ SP <extra-info> ] LF

extra-info снова зависит от команды. Если он пуст, предыдущий SP также опускается. В настоящее время ни одна команда не передает extra-info .

Перехватчик всегда запускается после того, как произошло автоматическое копирование заметок (см. "notes.rewrite.<command>" в git-config[1] ), и, таким образом, имеет доступ к этим заметкам.

Применяются следующие комментарии, специфичные для команды:

rebase

Для операций squash и fixup все сжатые фиксации отображаются как перезаписанные в сжатую фиксацию. Это означает, что будет несколько линий, использующих один и тот же new-object-name .

Коммиты гарантированно будут перечислены в том порядке, в котором они были обработаны при перебазировании.

sendemail-validate

Этот хук вызывается git-send-email[1] .

Он принимает эти аргументы командной строки. Это: 1. Имя файла, содержащего содержимое отправляемого электронного письма. 2. Имя файла, содержащего SMTP-заголовки электронного письма.

Заголовки SMTP передаются точно так же, как они передаются агенту транспорта почты (MTA) пользователя. По сути, электронное письмо, передаваемое MTA пользователя, представляет собой содержимое $2, за которым следует содержимое $1.

Ниже показан пример нескольких распространенных заголовков. Обратите внимание на использование заглавных букв и многострочную структуру вкладок.

From: Example <from@example.com>
To: to@example.com
Cc: cc@example.com,
 A <author@example.com>,
 One <one@example.com>,
 two@example.com
Subject: PATCH-STRING

Выход с ненулевым статусом приводит к прерыванию работы git send-email перед отправкой каких-либо электронных писем.

Следующие переменные среды устанавливаются при выполнении перехватчика.

GIT_SENDEMAIL_FILE_COUNTER

Счетчик, отсчитываемый от 1, увеличивается на единицу для каждого файла, содержащего отправляемое электронное письмо (исключая любые FIFO). Этот счетчик не соответствует схеме счетчика серии исправлений. Он всегда начинается с 1 и заканчивается в GIT_SENDEMAIL_FILE_TOTAL.

GIT_SENDEMAIL_FILE_TOTAL

Общее количество файлов, которые будут отправлены (исключая любые FIFO). Этот счетчик не соответствует схеме счетчика серии исправлений. Оно всегда будет равно количеству отправляемых файлов, независимо от того, есть сопроводительное письмо или нет.

Эти переменные могут, например, использоваться для проверки серии исправлений.

Пример перехватчика sendemail-validate , поставляемый с Git , проверяет, что все отправленные исправления (за исключением сопроводительного письма) могут быть применены поверх ветки по умолчанию вышестоящего репозитория без конфликтов. Некоторые заполнители оставлены для дополнительных шагов проверки, которые необходимо выполнить после применения всех исправлений данной серии.

fsmonitor-watchman

Этот перехват вызывается, когда для параметра конфигурации core.fsmonitor установлено значение .git/hooks/fsmonitor-watchman или .git/hooks/fsmonitor-watchmanv2 , в зависимости от используемой версии перехватчика.

Версия 1 принимает два аргумента: версию (1) и время в наносекундах, прошедшее с полуночи 1 января 1970 года.

Версия 2 принимает два аргумента: версию (2) и токен, который используется для идентификации изменений с момента появления токена. Для сторожа это будет идентификатор часов. Эта версия должна выводить на стандартный вывод новый токен, за которым следует NUL перед списком файлов.

Хук должен вывести на стандартный вывод список всех файлов в рабочем каталоге, которые могли измениться с запрошенного времени. Логика должна быть всеобъемлющей, чтобы не упустить ни одного потенциального изменения. Пути должны указываться относительно корня рабочего каталога и разделяться одним NUL.

Можно включать файлы, которые фактически не изменились. Все изменения, включая вновь созданные и удаленные файлы, должны быть включены. При переименовании файлов следует указывать как старое, так и новое имя.

Git будет ограничивать, какие файлы он проверяет на наличие изменений, а также какие каталоги проверяются на наличие неотслеживаемых файлов на основе заданных имен путей.

Оптимизированный способ сообщить git о "all files have changed" — вернуть имя файла / .

Статус выхода определяет, будет ли git использовать данные перехватчика для ограничения поиска. В случае ошибки произойдет возврат к проверке всех файлов и папок.

p4-changelist

Этот хук вызывается git-p4 submit .

Перехват p4-changelist выполняется после того, как сообщение списка изменений было отредактировано пользователем. Его можно обойти с помощью опции --no-verify . Он принимает единственный параметр — имя файла, содержащего предлагаемый текст списка изменений. Выход с ненулевым статусом приводит к прерыванию команды.

Хук позволяет редактировать файл списка изменений и может использоваться для нормализации текста в каком-либо стандартном формате проекта. Его также можно использовать для отказа в отправке после проверки файла сообщения.

Запустите git-p4 submit --help для получения подробной информации.

p4-prepare-changelist

Этот хук вызывается git-p4 submit .

Перехватчик p4-prepare-changelist выполняется сразу после подготовки сообщения списка изменений по умолчанию и перед запуском редактора. Он принимает один параметр — имя файла, содержащего текст списка изменений. Выход из сценария с ненулевым статусом приведет к прерыванию процесса.

Целью перехватчика является редактирование файла сообщения на месте, и оно не подавляется опцией --no-verify . Этот хук вызывается, даже если установлен --prepare-p4-only .

Запустите git-p4 submit --help для получения подробной информации.

p4-post-changelist

Этот хук вызывается git-p4 submit .

Перехватчик p4-post-changelist вызывается после успешной отправки в P4. Он не принимает никаких параметров и предназначен в первую очередь для уведомления и не может повлиять на результат действия отправки git p4.

Запустите git-p4 submit --help для получения подробной информации.

p4-pre-submit

Этот хук вызывается git-p4 submit . Он не принимает никаких параметров и ничего из стандартного ввода. Выход из этого сценария с ненулевым статусом предотвращает запуск git-p4 submit . Его можно обойти с помощью параметра командной строки --no-verify . Запустите git-p4 submit --help для получения подробной информации.

post-index-change

Этот хук вызывается, когда индекс записывается в read-cache.c do_write_locked_index.

Первый параметр, передаваемый перехватчику, — это индикатор обновляемого рабочего каталога. "1" означает, что рабочий каталог был обновлен, или "0" , когда рабочий каталог не был обновлен.

Второй параметр, передаваемый перехватчику, — это индикатор того, был ли обновлен индекс и мог ли измениться бит пропуска рабочего дерева. "1" означает, что биты пропуска рабочего дерева могли быть обновлены, а "0" означает, что они не были обновлены.

При запуске перехвата только один параметр должен быть установлен на "1" . Крюк, проходящий мимо "1" , "1" , не должен быть возможен.
