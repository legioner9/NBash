git-cvsserver

Name

git-cvsserver — эмулятор CVS-сервера для Git .

Synopsis

SSH:

export CVS_SERVER="git cvsserver"
cvs -d :ext:user@server/path/repo.git co <HEAD_name>

pserver (/etc/inetd.conf):

cvspserver stream tcp nowait nobody /usr/bin/git-cvsserver git-cvsserver pserver

Usage:

git-cvsserver [<options>] [pserver|server] [<directory> …​]

Description

Это приложение представляет собой уровень эмуляции CVS для Git.

Он очень функционален. Однако не все методы реализованы, а для тех методов, которые реализованы, реализованы не все переключатели.

Тестирование проводилось с использованием как клиента CLI CVS, так и плагина Eclipse CVS. Большая часть функций отлично работает с обоими этими клиентами.

Options

Очевидно, что все эти параметры имеют смысл только в том случае, если они реализованы на стороне сервера. Они были реализованы так, чтобы максимально напоминать варианты git-daemon[1] .

--base-path <path>

Добавьте path к запрошенному CVSROOT.

--strict-paths

Запретить рекурсию в подкаталоги

--export-all

Не проверяйте gitcvs.enabled в конфигурации. Вам также необходимо указать список разрешенных каталогов (см. ниже), если вы хотите использовать эту опцию.

-V
--version

Распечатать информацию о версии и выйти

-h
-H
--help

Распечатать информацию об использовании и выйти

<directory>

Остальные аргументы предоставляют список каталогов. Если каталоги не указаны, то разрешены все. Репозиториям в этих каталогах по-прежнему требуется опция конфигурации gitcvs.enabled , если не указан --export-all .

Limitations

Клиенты CVS не могут помечать, разветвлять или выполнять слияния Git .

git-cvsserver отображает ветви Git на модули CVS. Это сильно отличается от того, что ожидает большинство пользователей CVS, поскольку модули CVS обычно представляют собой один или несколько каталогов.

Installation

  1. Если вы собираетесь предоставить доступ к CVS через pserver, добавьте строку в /etc/inetd.conf, например

       cvspserver stream tcp nowait nobody git-cvsserver pserver

    Примечание. Некоторые серверы inetd позволяют указать имя исполняемого файла независимо от значения argv[0] (т. е. имени, под которым программа предполагает, что он был выполнен). В этом случае правильная строка в файле /etc/inetd.conf выглядит так:

       cvspserver stream tcp nowait nobody /usr/bin/git-cvsserver git-cvsserver pserver

    По умолчанию pserver предоставляет только анонимный доступ. Для фиксации вам нужно будет создать учетные записи pserver, просто добавьте параметр gitcvs.authdb в файл конфигурации репозиториев, в которые вы хотите, чтобы cvsserver разрешил запись, например:

       [gitcvs]
        authdb = /etc/cvsserver/passwd

    Формат этих файлов — имя пользователя, за которым следует зашифрованный пароль, например:

       myuser:sqkNi8zPf01HI
   myuser:$1$9K7FzU28$VfF6EoPYCJEYcVQwATgOP/
   myuser:$5$.NqmNH1vwfzGpV8B$znZIcumu1tNLATgV2l6e1/mY8RzhUDHMOaVOeL1cxV3

    Для создания этих файлов вы можете использовать средство htpasswd , поставляемое с Apache, но только с опцией -d (или -B, если ваша система поддерживает ее).

    Предпочтительно использовать системную утилиту, которая управляет созданием хэша паролей на вашей платформе (например, mkpasswd в Linux, шифровать в OpenBSD или pwhash в NetBSD) и вставить ее в нужное место.

    Затем укажите свой пароль с помощью метода pserver, например:

       cvs -d:pserver:someuser:somepassword@server:/path/repo.git co <HEAD_name>

    Для доступа по SSH не требуется никакой специальной настройки, кроме наличия инструментов Git в PATH. Если у вас есть клиенты, которые не принимают переменную среды CVS_SERVER, вы можете переименовать git-cvsserver в cvs .

    Примечание. Более новые версии CVS (>= 1.12.11) также поддерживают указание CVS_SERVER непосредственно в CVSROOT, например

       cvs -d ":ext;CVS_SERVER=git cvsserver:user@server/path/repo.git" co <HEAD_name>

    Преимущество этого подхода заключается в том, что он будет сохранен в ваших файлах CVS/Root , и вам не нужно беспокоиться о том, чтобы всегда устанавливать правильную переменную среды. Пользователям SSH, ограниченным git-shell , не нужно переопределять значение по умолчанию с помощью CVS_SERVER (и не следует этого делать), поскольку git-shell понимает, что cvs означает git-cvsserver , и делает вид, что на другом конце лучше работает настоящий cvs .

  2. Для каждого репозитория, доступ к которому вы хотите получить из CVS, вам необходимо отредактировать конфигурацию в репозитории и добавить следующий раздел.

       [gitcvs]
        enabled=1
        # необязательно для отладки
        logFile=/path/to/logfile

    Примечание: вам необходимо убедиться, что каждый пользователь, который собирается вызвать git-cvsserver , имеет доступ на запись к файлу журнала и базе данных (см. Database Backend ). Если вы хотите предложить доступ на запись через SSH, пользователям, конечно, также потребуется доступ на запись к Git . сам репозиторий.

    Вам также необходимо убедиться, что каждый репозиторий имеет тип "bare" (без индексного файла Git ), чтобы cvs commit работал. См. gitcvs-migration[7] .

    Все переменные конфигурации также могут быть переопределены для определенного метода доступа. Допустимые имена методов: "ext" (для доступа по SSH) и "pserver" . В следующем примере конфигурации отключается доступ к pserver, но при этом разрешается доступ через SSH.

       [gitcvs]
        enabled=0

   [gitcvs "ext"]
        enabled=1

  3. Если вы не указали CVSROOT/CVS_SERVER непосредственно в команде оформления заказа, автоматически сохраняя его в файлах CVS/Root , вам необходимо явно установить их в своей среде. CVSROOT должен быть установлен как обычно, но каталог должен указывать на соответствующий репозиторий Git . Как указано выше, для SSH-клиентов not , ограниченных git-shell , CVS_SERVER должен быть установлен в git-cvsserver .

       export CVSROOT=:ext:user@server:/var/git/project.git
   export CVS_SERVER="git cvsserver"

  4. Для клиентов SSH, которые будут выполнять коммиты, убедитесь, что их серверные файлы .ssh/environment (или .bashrc и т. д., в зависимости от их конкретной оболочки) экспортируют соответствующие значения для GIT_AUTHOR_NAME, GIT_AUTHOR_EMAIL, GIT_COMMITTER_NAME и GIT_COMMITTER_EMAIL. Для SSH-клиентов, чья оболочка входа — bash, разумной альтернативой может быть .bashrc.

  5. Теперь клиенты смогут оценить проект. Используйте имя CVS module , чтобы указать, какой Git head вы хотите проверить. Это также устанавливает имя вашего вновь извлеченного каталога, если вы не укажете иное с помощью -d <dir_name> . Например, это проверяет ветвь master к каталогу project-master :

       cvs co -d project-master master

Database backend

git-cvsserver использует одну базу данных на головку Git (т. е. модуль CVS) для хранения информации о репозитории и поддержания согласованных номеров версий CVS. Базу данных необходимо обновлять (т. е. записывать в нее) после каждой фиксации.

Если фиксация выполняется напрямую с помощью git (в отличие от использования git-cvsserver ), обновление должно произойти при следующем доступе к репозиторию с помощью git-cvsserver , независимо от метода доступа и запрошенной операции.

Это означает, что даже если вы предлагаете только доступ для чтения (например, используя метод pserver), git-cvsserver должен иметь доступ для записи в базу данных для надежной работы (в противном случае вам необходимо убедиться, что база данных обновлена ​​каждый раз, когда выполняется git-cvsserver ). .

По умолчанию он использует базы данных SQLite в каталоге Git с именем gitcvs.<module_name>.sqlite . Обратите внимание, что серверная часть SQLite создает временные файлы в том же каталоге, что и файл базы данных при записи, поэтому может быть недостаточно предоставить пользователям, использующим git-cvsserver , доступ на запись к файлу базы данных, не предоставив им также доступ на запись в каталог.

Базу данных невозможно надежно восстановить в согласованной форме после изменения отслеживаемой ветки. Пример: для объединенных ветвей git-cvsserver отслеживает только одну ветвь разработки, а после git merge постепенно обновляемая база данных может отслеживать другую ветвь, чем база данных, восстановленная с нуля, что приводит к несогласованности номеров версий CVS. git-cvsserver не имеет возможности узнать, какую ветвь он бы выбрал, если бы он выполнялся постепенно перед слиянием. Поэтому, если вам необходимо полностью или частично (из старой резервной копии) восстановить базу данных, вам следует с подозрением относиться к уже существовавшим песочницам CVS.

Вы можете настроить серверную часть базы данных с помощью следующих переменных конфигурации:

Настройка серверной части базы данных

git-cvsserver использует модуль Perl DBI. Пожалуйста, также прочитайте документацию при изменении этих переменных, особенно о DBI->connect() .

gitcvs.dbName

Имя базы данных. Точное значение зависит от выбранного драйвера базы данных, для SQLite это имя файла. Поддерживает замену переменных (см. ниже). Не может содержать точки с запятой ( ; ). По умолчанию: %Ggitcvs.%m.sqlite

gitcvs.dbDriver

Используемый драйвер DBI. Здесь вы можете указать любой доступный драйвер, но он может не работать. cvsserver протестирован с DBD::SQLite , сообщается о работе с DBD::Pg и о неработоспособности с DBD::mysql . Пожалуйста, расценивайте это как экспериментальную функцию. Не может содержать двоеточий ( : ). По умолчанию: SQLite

gitcvs.dbuser

Пользователь базы данных. Полезно только при установке dbDriver , поскольку в SQLite нет понятия пользователей базы данных. Поддерживает замену переменных (см. ниже).

gitcvs.dbPass

Пароль базы данных. Полезно только при установке dbDriver , поскольку SQLite не имеет понятия паролей к базе данных.

gitcvs.dbTableNamePrefix

Префикс имени таблицы базы данных. Поддерживает замену переменных (см. ниже). Любые неалфавитные символы будут заменены символами подчеркивания.

Все переменные также могут быть установлены для каждого метода доступа, см. above .

Variable substitution

В dbDriver и dbUser вы можете использовать следующие переменные:

%G

Имя каталога Git

%g

Имя каталога Git , где все символы, кроме буквенно-цифровых, . и - , заменены на _ (это должно облегчить использование имени каталога в имени файла, если это необходимо)

%m

Модуль CVS/имя заголовка Git

%a

метод доступа (один из "ext" или "pserver" )

%u

Имя пользователя, работающего с git-cvsserver . Если имя невозможно определить, используется числовой идентификатор uid.

Environment

Эти переменные в некоторых случаях устраняют необходимость в параметрах командной строки, что упрощает ограниченное использование через git-shell.

GIT_CVSSERVER_BASE_PATH

Эта переменная заменяет аргумент --base-path.

GIT_CVSSERVER_ROOT

Эта переменная указывает один каталог, заменяя список аргументов <directory>... . Репозиторию по-прежнему требуется опция конфигурации gitcvs.enabled , если не указан --export-all .

Если эти переменные среды установлены, соответствующие аргументы командной строки использовать нельзя.

Заметки о клиенте Eclipse cvs

Чтобы оформить заказ с помощью клиента Eclipse CVS:

  1. Выберите "Create a new project → From CVS checkout"

  2. Создайте новое местоположение. Подробную информацию о том, как выбрать правильный протокол, см. в примечаниях ниже.

  3. Просмотрите доступные модели modules . Он предоставит вам список голов в репозитории. Отсюда вы не сможете просматривать дерево. Только головы.

  4. Выберите HEAD , когда он спросит, какую ветку/тег проверить. Снимите флажок "launch commit wizard" , чтобы избежать фиксации файла .project.

Примечания к протоколу: Если вы используете анонимный доступ через pserver, просто выберите это. Тем, кто использует доступ по SSH, следует выбрать протокол ext и настроить доступ ext на панели «Настройки» → «Команда» → «CVS» → «ExtConnection». Установите для CVS_SERVER значение " git cvsserver " . Обратите внимание, что поддержка паролей не очень хороша при использовании ext , вам обязательно понадобится настроить ключи SSH.

Альтернативно вы можете просто использовать нестандартный протокол extssh, предлагаемый Eclipse. В этом случае CVS_SERVER игнорируется, и вам придется заменить утилиту cvs на сервере на git-cvsserver или манипулировать вашим .bashrc так, чтобы вызов cvs эффективно вызывал git-cvsserver .

Известно, что клиенты работают

  • CVS 1.12.9 на Debian

  • CVS 1.11.17 на MacOSX (из пакета Fink)

  • Eclipse 3.0, 3.1.2 на MacOSX (см. Примечания к клиенту Eclipse CVS)

  • TortoiseCVS

Operations supported

Поддерживаются все операции, необходимые для нормального использования, включая проверку, сравнение, статус, обновление, журнал, добавление, удаление, фиксацию.

Большинство аргументов команды CVS, которые читают теги CVS или номера ревизий (обычно -r), работают, а также поддерживают любую спецификацию ссылки git (тег, ветку, идентификатор фиксации и т. д.). Однако номера ревизий CVS для нестандартных ветвей эмулируются плохо, а журнал cvs вообще не показывает теги или ветки. (Номера ревизий CVS, не относящихся к основной ветке, внешне напоминают номера ревизий CVS, но на самом деле они напрямую кодируют идентификатор фиксации git , а не представляют количество ревизий с момента точки ветвления.)

Обратите внимание, что есть два способа оформить заказ в конкретной ветке. Как описано в другом месте на этой странице, параметр "module" проверки cvs интерпретируется как имя ветки и становится основной веткой. Она остается основной веткой для данной песочницы, даже если вы временно закрепите другую ветку с помощью cvs update -r. Альтернативно, аргумент -r может указывать на другую ветвь для фактического извлечения, даже если модуль по-прежнему является ветвью "main" . Компромиссы (в текущей реализации): каждый новый "module" создает на диске новую базу данных с историей данного модуля, и после создания базы данных операции с этой основной ветвью выполняются быстро. Или, альтернативно, -r не занимает дополнительного дискового пространства, но может выполняться значительно медленнее для многих операций, таких как обновление cvs.

Если вы хотите сослаться на ссылку git , содержащую символы, не разрешенные CVS, у вас есть два варианта. Во-первых, можно просто передать спецификацию ссылки git непосредственно в соответствующий аргумент CVS -r; некоторые клиенты CVS, похоже, не особо проверяют аргументы. Во-вторых, если это не помогло, вы можете использовать специальный механизм escape-символов, который использует только символы, допустимые в тегах CVS. Последовательность из 4 или 5 символов вида (подчеркивание ( "_" ), тире ( "-" ), один или два символа и тире ( "-" )) может кодировать различные символы на основе одной или двух букв: "s" для косой черты ( "/" ). , "p" для точки ( "." ), "u" для подчеркивания ( "_" ) или два шестнадцатеричных di git для любого байтового значения (обычно это число ASCII или, возможно, часть символа в кодировке UTF-8).

Устаревшие операции мониторинга не поддерживаются (редактирование, просмотр и т.п.). Экспорт и тегирование (теги и ветки) на данном этапе не поддерживаются.

Преобразования окончания строки CRLF

По умолчанию сервер оставляет режим -k пустым для всех файлов, в результате чего клиент CVS обрабатывает их как текстовые файлы, подлежащие преобразованию конца строки на некоторых платформах.

Вы можете заставить сервер использовать атрибуты преобразования конца строки для установки режимов -k для файлов, задав переменную конфигурации gitcvs.usecrlfattr . См. gitattributes[5] для получения дополнительной информации о конечном преобразовании.

В качестве альтернативы, если конфигурация gitcvs.usecrlfattr не включена или атрибуты не позволяют автоматически определять имя файла, сервер использует конфигурацию gitcvs.allBinary для настройки по умолчанию. Если установлен gitcvs.allBinary , то для файла, не указанного иначе, по умолчанию будет использоваться режим -kb . В противном случае режим -k остается пустым. Но если для gitcvs.allBinary установлено значение "guess" , то правильный режим -k будет угадываться на основе содержимого файла.

Для лучшей совместимости с cvs , вероятно, лучше переопределить значения по умолчанию, установив для gitcvs.usecrlfattr значение true, а для gitcvs.allBinary — значение "guess" .

Dependencies

git-cvsserver зависит от DBD::SQLite.
