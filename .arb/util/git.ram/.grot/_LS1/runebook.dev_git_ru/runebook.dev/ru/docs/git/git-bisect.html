git-bisect

Name

git-bisect — используйте двоичный поиск, чтобы найти коммит, в котором возникла ошибка.

Synopsis

git bisect <subcommand> <options>

Description

Команда принимает различные подкоманды и разные параметры в зависимости от подкоманды:

git bisect start [--term-(new|bad)=<term-new> --term-(old|good)=<term-old>]
          [--no-checkout] [--first-parent] [<bad> [<good>...]] [--] [<paths>...]
git bisect (bad|new|<term-new>) [<rev>]
git bisect (good|old|<term-old>) [<rev>...]
git bisect terms [--term-good | --term-bad]
git bisect skip [(<rev>|<range>)...]
git bisect reset [<commit>]
git bisect (visualize|view)
git bisect replay <logfile>
git bisect log
git bisect run <cmd> [<arg>...]
git bisect help

Эта команда использует алгоритм двоичного поиска, чтобы найти, какой коммит в истории вашего проекта привел к ошибке. Вы используете его, сначала сообщая ему коммит "bad" , который, как известно, содержит ошибку, и коммит "good" , который, как известно, был сделан до появления ошибки. Затем git bisect выбирает фиксацию между этими двумя конечными точками и спрашивает, является ли выбранная фиксация "good" или "bad" . Он продолжает сужать диапазон, пока не найдет точный коммит, который внес изменение.

Фактически, git bisect можно использовать для поиска фиксации, которая изменила какое-либо свойство вашего проекта; например, коммит, исправивший ошибку, или коммит, который привел к улучшению производительности теста. Чтобы поддержать это более общее использование, вместо "good" и "bad" можно использовать термины "old" и "new" , или вы можете выбрать свои собственные термины. Дополнительную информацию см. в разделе "Alternate terms" ниже.

Основные команды по биссектрисе: начало, плохо, хорошо.

В качестве примера предположим, что вы пытаетесь найти коммит, который нарушил функцию, которая, как известно, работала в версии v2.6.13-rc2 вашего проекта. Вы начинаете сеанс пополам следующим образом:

$ git bisect start
$ git bisect bad                 # Текущая версия плохая
$ git bisect good v2.6.13-rc2    # v2.6.13-rc2 известен как хорошая

После того как вы указали хотя бы один плохой и один хороший коммит, git bisect выбирает коммит в середине этого диапазона истории, проверяет его и выводит что-то похожее на следующее:

Bisecting: 675 revisions left to test after this (roughly 10 steps)

Теперь вам следует скомпилировать извлеченную версию и протестировать ее. Если эта версия работает правильно, введите

$ git bisect good

Если эта версия не работает, введите

$ git bisect bad

Тогда git bisect ответит что-то вроде

Bisecting: 337 revisions left to test after this (roughly 9 steps)

Продолжайте повторять процесс: скомпилируйте дерево, протестируйте его и, в зависимости от того, хорошее оно или плохое, запустите git bisect good или git bisect bad , чтобы запросить следующий коммит, требующий тестирования.

В конце концов ревизий для проверки больше не останется, и команда распечатает описание первого неудачного коммита. Ссылка refs/bisect/bad останется указывающей на этот коммит.

Bisect reset

После сеанса разделения пополам, чтобы очистить состояние разделения пополам и вернуться к исходному HEAD, введите следующую команду:

$ git bisect reset

По умолчанию это вернет ваше дерево к коммиту, который был извлечен до git bisect start . (Новый git bisect start также будет делать это, поскольку он очищает старое состояние деления пополам.)

С необязательным аргументом вы можете вместо этого вернуться к другому коммиту:

$ git bisect reset <commit>

Например, git bisect reset bisect/bad проверит первую плохую ревизию, а git bisect reset HEAD оставит вас на текущем коммите пополам и вообще избежит переключения коммитов.

Alternate terms

Иногда вы ищете не фиксацию, которая привела к поломке, а скорее фиксацию, которая вызвала изменение между каким-либо другим состоянием "old" и состоянием "new" . Например, вы можете искать коммит, содержащий определенное исправление. Или вы можете искать первый коммит, в котором имена файлов исходного кода были наконец преобразованы в стандарт именования вашей компании. Или что-то еще.

В таких случаях может быть очень запутанно использовать термины "good" и "bad" для обозначения "the state before the change" и "the state after the change" . Поэтому вместо "good" и "bad" вы можете использовать термины "old" и "new" соответственно. (Но учтите, что вы не можете совмещать "good" и "bad" с "old" и "new" в одном сеансе.)

В этом более общем использовании вы предоставляете git bisect фиксацию "new" , имеющую некоторое свойство, и фиксацию "old" , не имеющую этого свойства. Каждый раз, когда git bisect проверяет фиксацию, вы проверяете, имеет ли эта фиксация свойство. Если да, отметьте коммит как "new" ; в противном случае отметьте его как "old" . Когда деление пополам будет завершено, git bisect сообщит, какой коммит ввел это свойство.

Чтобы использовать "old" и "new" вместо "good" и bad, вы должны запустить git bisect start без коммитов в качестве аргумента, а затем выполнить следующие команды, чтобы добавить коммиты:

git bisect old [<rev>]

чтобы указать, что фиксация произошла до искомого изменения, или

git bisect new [<rev>...]

чтобы указать, что это было после.

Чтобы получить напоминание об используемых в данный момент терминах, используйте

git bisect terms

Получить просто старый (соответственно новый) термин можно с помощью git bisect terms --term-old или git bisect terms --term-good .

Если вы хотите использовать свои собственные термины вместо "bad" / "good" или "new" / "old" , вы можете выбрать любые имена, которые вам нравятся (кроме существующих подкоманд пополам, таких как reset , start , …), начав деление пополам с помощью

git bisect start --term-old <term-old> --term-new <term-new>

Например, если вы ищете коммит, который привел к снижению производительности, вы можете использовать

git bisect start --term-old fast --term-new slow

Или, если вы ищете коммит, исправивший ошибку, вы можете использовать

git bisect start --term-new fixed --term-old broken

Затем используйте git bisect <term-old> и git bisect <term-new> вместо git bisect good и git bisect bad для маркировки коммитов.

Bisect visualize/view

Чтобы увидеть оставшихся на данный момент подозреваемых в gitk , введите следующую команду во время процесса деления пополам (подкоманду view можно использовать в качестве альтернативы visualize ):

$ git bisect visualize

Git определяет графическую среду с помощью различных переменных среды: DISPLAY , которая задается в средах X Window System в системах Unix. SESSIONNAME , который устанавливается в Cygwin в сеансах интерактивного рабочего стола. MSYSTEM , который устанавливается в Msys2 и Git для Windows. SECURITYSESSIONID , который можно установить в macOS в сеансах интерактивного рабочего стола.

Если для этих переменных среды установлена ​none , вместо нее используется git log . Вы также можете указать параметры командной строки, такие как -p и --stat .

$ git bisect visualize --stat

Журнал пополам и повтор пополам

После того, как ревизии отмечены как хорошие или плохие, введите следующую команду, чтобы показать, что было сделано на данный момент:

$ git bisect log

Если вы обнаружите, что допустили ошибку при указании статуса редакции, вы можете сохранить вывод этой команды в файл, отредактировать его, чтобы удалить неправильные записи, а затем выполнить следующие команды, чтобы вернуться к исправленному состоянию:

$ git bisect reset
$ git bisect replay that-file

Как избежать тестирования коммита

Если в середине сеанса пополам вы знаете, что предложенная ревизия не подходит для тестирования (например, ее не удается собрать, и вы знаете, что ошибка не имеет ничего общего с ошибкой, которую вы преследуете), вы может вручную выбрать ближайший коммит и протестировать его.

For example:

$ git bisect good/bad                        # предыдущий раунд был хорошим или плохим.
Bisecting: 337 revisions left to test after this (roughly 9 steps)
$ git bisect visualize                        # ой, это неинтересно.
$ git reset --hard HEAD~3                # попробуй 3 ревизии перед чем
                                        # было предложено

Затем скомпилируйте и протестируйте выбранную версию, а затем отметьте ее как хорошую или плохую обычным способом.

Bisect skip

Вместо того, чтобы самостоятельно выбирать ближайший коммит, вы можете попросить Git сделать это за вас, введя команду:

$ git bisect skip                 # Текущая версия не может быть протестирована

Однако если вы пропустите коммит, соседний с тем, который вы ищете, Git не сможет точно определить, какой из этих коммитов был первым плохим.

Вы также можете пропустить диапазон коммитов вместо одного, используя обозначение диапазона. Например:

$ git bisect skip v2.5..v2.6

Это сообщает процессу bisect, что никакие фиксации после v2.5 , вплоть до v2.6 включительно, не должны проверяться.

Обратите внимание: если вы также хотите пропустить первую фиксацию диапазона, вам следует ввести команду:

$ git bisect skip v2.5 v2.5..v2.6

Это сообщает процессу bisect, что фиксации между v2.5 и v2.6 (включительно) следует пропустить.

Сокращение деления пополам за счет предоставления большего количества параметров для начала разделения пополам.

Вы можете еще больше сократить количество попыток, если знаете, какая часть дерева задействована в проблеме, которую вы отслеживаете, указав параметры пути при вводе команды bisect start :

$ git bisect start -- arch/i386 include/asm-i386

Если вам заранее известно более одного хорошего коммита, вы можете сузить пространство пополам, указав все хорошие коммиты сразу после плохого коммита при вводе команды bisect start :

$ git bisect start v2.6.20-rc6 v2.6.20-rc4 v2.6.20-rc1 --
                   # v2.6.20-rc6 плохой
                   # v2.6.20-rc4 и v2.6.20-rc1 хороши

Bisect run

Если у вас есть сценарий, который может определить, хороший или плохой текущий исходный код, вы можете разделить его пополам, введя команду:

$ git bisect run my_script arguments

Обратите внимание, что сценарий ( my_script в приведенном выше примере) должен завершиться с кодом 0, если текущий исходный код хороший/старый, и завершиться с кодом от 1 до 127 (включительно), за исключением 125, если текущий исходный код плохой/ новый.

Любой другой код выхода прервет процесс разделения пополам. Следует отметить, что программа, завершающаяся через exit(-1) , оставляет $? = 255 (см. страницу руководства exit(3)), так как значение обрезается с помощью & 0377 .

Специальный код завершения 125 следует использовать, когда текущий исходный код не может быть проверен. Если сценарий завершается с этим кодом, текущая версия будет пропущена (см. git bisect skip выше). 125 было выбрано как максимальное разумное значение для этой цели, поскольку 126 и 127 используются оболочками POSIX для сигнализации конкретного состояния ошибки (127 — для команды, которая не найдена, 126 — для команды, найденной, но невыполнимой — эти детали не не имеет значения, поскольку это обычные ошибки в сценарии, что касается bisect run ).

Вы часто можете обнаружить, что во время сеанса пополам вам нужно, чтобы временные изменения (например, s/#define DEBUG 0/#define DEBUG 1/ в файле заголовка или "revision that does not have this commit needs this patch applied to work around another problem this bisection is not interested in" ) применялись к тестируемой версии.

Чтобы справиться с такой ситуацией, после того, как внутренний git bisect найдет следующую ревизию для тестирования, сценарий может применить исправление перед компиляцией, запустить реальный тест, а затем решить, прошла ли ревизия (возможно, с необходимым исправлением) тест, а затем вернуть дерево в первозданное состояние. Наконец, сценарий должен завершиться со статусом реального теста, чтобы позволить командному циклу git bisect run определить конечный результат сеанса пополам.

Options

--no-checkout

Не проверяйте новое рабочее дерево на каждой итерации процесса деления пополам. Вместо этого просто обновите ссылку с именем BISECT_HEAD , чтобы она указывала на коммит, который необходимо протестировать.

Этот параметр может быть полезен, если тест, который вы будете выполнять на каждом этапе, не требует извлеченного дерева.

Если репозиторий пустой, предполагается --no-checkout .

--first-parent

Следуйте только первому родительскому коммиту, увидев коммит слияния.

При обнаружении регрессий, вызванных слиянием ветки, фиксация слияния будет идентифицирована как появление ошибки, а ее предки будут игнорироваться.

Эта опция особенно полезна для предотвращения срабатываний false , когда объединенная ветка содержала неработающие или не подлежащие сборке коммиты, но само слияние прошло нормально.

Examples

  • Автоматически разделить сломанную сборку между v1.2 и HEAD:

    $ git bisect start HEAD v1.2 --      # HEAD — плохо, v1.2 — хорошо
$ git bisect run make                #  "make"  собирает приложение
$ git bisect reset                   # quit the bisect session

  • Автоматически разделить отказ теста пополам между источником и HEAD:

    $ git bisect start HEAD origin --    # HEAD плохой, origin хороший
$ git bisect run make test           #  "make test"  строит и тестирует
$ git bisect reset                   # quit the bisect session

  • Автоматически разделить сломанный тестовый пример пополам:

    $ cat ~/test.sh
#!/bin/sh
make || exit 125                     # это пропускает неработающие сборки
~/check_test_case.sh                 # Пройден ли тестовый пример?
$ git bisect start HEAD HEAD~10 --   # culprit is among the last 10
$ git bisect run ~/test.sh
$ git bisect reset                   # quit the bisect session

    Здесь мы используем собственный скрипт test.sh . В этом сценарии в случае сбоя make мы пропускаем текущую фиксацию. check_test_case.sh должен exit 0 , если тестовый пример пройден, и exit 1 в противном случае.

    Безопаснее, если и test.sh , и check_test_case.sh находятся за пределами репозитория, чтобы предотвратить взаимодействие между процессами разделения, создания и тестирования и сценариями.

  • Автоматически разделить пополам с временными изменениями (исправление):

    $ cat ~/test.sh
#!/bin/sh

# настраиваем рабочее дерево, объединяя ветку исправлений
# и затем попытаемся выполнить сборку
if        git merge --no-commit --no-ff hot-fix &&
        make
then
        # запустить тест конкретного проекта и сообщить о его статусе
        ~/check_test_case.sh
        status=$?
else
        # сообщаем вызывающему абоненту, что это невозможно проверить
        status=125
fi

# отмените настройку, чтобы разрешить чистый переход к следующему коммиту
git reset --hard

# возврат управления
exit $status

    Это применяет изменения из ветки исправлений перед каждым запуском теста, например, в случае, если ваша среда сборки или тестирования изменилась так, что более старые версии могут нуждаться в исправлении, которое уже есть в более новых. (Убедитесь, что ветвь исправления основана на фиксации, которая содержится во всех ревизиях, которые вы делите пополам, чтобы слияние не занимало слишком много времени, или используйте git cherry-pick вместо git merge .)

  • Автоматически разделить сломанный тестовый пример пополам:

    $ git bisect start HEAD HEAD~10 --   # виновник входит в число последних 10
$ git bisect run sh -c "make || exit 125; ~/check_test_case.sh"
$ git bisect reset                   # quit the bisect session

    Это показывает, что можно обойтись и без скрипта запуска, если писать тест в одну строку.

  • Найдите хорошую область графа объекта в поврежденном репозитории.

    $ git bisect start HEAD <known-good-commit> [ <boundary-commit> ... ] --no-checkout
$ git bisect run sh -c '
        GOOD=$(git for-each-ref "--format=%(objectname)" refs/bisect/good-*) &&
        git rev-list --objects BISECT_HEAD --not $GOOD >tmp.$$ &&
        git pack-objects --stdout >/dev/null <tmp.$$
        rc=$?
        rm -f tmp.$$
        test $rc = 0'

$ git bisect reset                   # выходим из сеанса bisect

    В этом случае, когда git bisect run завершится, bisect/bad будет относиться к коммиту, у которого есть хотя бы один родительский элемент, чей достижимый граф полностью проходим в смысле, требуемом git pack objects .

  • Ищите исправление, а не регрессию в коде.

    $ git bisect start
$ git bisect new HEAD    # current commit is marked as new
$ git bisect old HEAD~10 # the tenth commit from now is marked as old

    or:

$ git bisect start --term-old broken --term-new fixed
$ git bisect fixed
$ git bisect broken HEAD~10

Getting help

Используйте git bisect , чтобы получить краткое описание использования, и git bisect help или git bisect -h , чтобы получить подробное описание использования.
