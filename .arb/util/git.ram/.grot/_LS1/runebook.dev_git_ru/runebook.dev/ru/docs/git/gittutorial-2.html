gittutorial-2

Name

gittutorial-2 — Введение в Git: часть вторая

Synopsis

git *

Description

Прежде чем читать это руководство, вам следует поработать с gittutorial[7] .

Цель этого руководства — представить две фундаментальные части архитектуры Git — объектную базу данных и индексный файл — и предоставить читателю все необходимое для понимания остальной части документации Git .

Объектная база данных git

Давайте начнем новый проект и создадим небольшой объем истории:

$ mkdir test-project
$ cd test-project
$ git init
Initialized empty Git repository in .git/
$ echo 'hello world' > file.txt
$ git add .
$ git commit -a -m "initial commit"
[master (root-commit) 54196cc] initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 file.txt
$ echo 'hello world!' >file.txt
$ git commit -a -m "add emphasis"
[master c4d59f3] add emphasis
 1 file changed, 1 insertion(+), 1 deletion(-)

Какие 7 шестнадцатеричных цифр Git ответил на фиксацию?

В первой части руководства мы видели, что коммиты имеют такие имена. Оказывается, каждый объект в истории Git хранится под 40-значным шестнадцатеричным именем. Это имя представляет собой хэш SHA-1 содержимого объекта; среди прочего, это гарантирует, что Git никогда не будет хранить одни и те же данные дважды (поскольку идентичным данным присваивается идентичное имя SHA-1) и что содержимое объекта Git никогда не изменится (поскольку это также приведет к изменению имени объекта). ). 7 шестнадцатеричных строк char здесь являются просто сокращением таких 40 длинных строк char . Аббревиатуры можно использовать везде, где могут использоваться 40 актерских строк char , при условии, что они однозначны.

Ожидается, что содержимое объекта фиксации, созданного вами в соответствии с приведенным выше примером, генерирует хэш SHA-1, отличный от показанного выше, поскольку объект фиксации записывает время его создания и имя человека, выполнившего фиксацию.

Мы можем спросить Git об этом конкретном объекте с помощью команды cat-file . Не копируйте 40 шестнадцатеричных цифр из этого примера, а используйте их из своей версии. Обратите внимание, что вы можете сократить его до нескольких символов, чтобы не вводить все 40 шестнадцатеричных цифр:

$ git cat-file -t 54196cc2
commit
$ git cat-file commit 54196cc2
tree 92b8b694ffb1675e5975148e1121810081dbdffe
author J. Bruce Fields <bfields@puzzle.fieldses.org> 1143414668 -0500
committer J. Bruce Fields <bfields@puzzle.fieldses.org> 1143414668 -0500

initial commit

Дерево может ссылаться на один или несколько объектов "blob" , каждый из которых соответствует файлу. Кроме того, дерево может также ссылаться на другие объекты дерева, создавая таким образом иерархию каталогов. Вы можете просмотреть содержимое любого дерева с помощью ls-tree (помните, что достаточно длинная начальная часть SHA-1 также подойдет):

$ git ls-tree 92b8b694
100644 blob 3b18e512dba79e4c8300dd08aeb37f8e728b8dad    file.txt

Таким образом, мы видим, что в этом дереве есть один файл. Хэш SHA-1 является ссылкой на данные этого файла:

$ git cat-file -t 3b18e512
blob

"blob" — это просто данные файла, которые мы также можем просмотреть с помощью cat-файла:

$ git cat-file blob 3b18e512
hello world

Обратите внимание, что это старые данные файла; поэтому объект, который Git назвал в своем ответе на исходное дерево, был деревом со снимком состояния каталога, записанным при первом коммите.

Все эти объекты хранятся под своими именами SHA-1 в каталоге Git :

$ find .git/objects/
.git/objects/
.git/objects/pack
.git/objects/info
.git/objects/3b
.git/objects/3b/18e512dba79e4c8300dd08aeb37f8e728b8dad
.git/objects/92
.git/objects/92/b8b694ffb1675e5975148e1121810081dbdffe
.git/objects/54
.git/objects/54/196cc2703dc165cbd373a65a4dcf22d50ae7f7
.git/objects/a0
.git/objects/a0/423896973644771497bdc03eb99d5281615b51
.git/objects/d0
.git/objects/d0/492b368b66bdabf2ac1fd8c92b39d3db916e59
.git/objects/c4
.git/objects/c4/d59f390b9cfd4318117afde11d601c1085f241

и содержимое этих файлов — это просто сжатые данные плюс заголовок, определяющий их длину и тип. Типом может быть большой двоичный объект, дерево, фиксация или тег.

Самый простой коммит, который можно найти, — это коммит HEAD, который мы можем найти в .git/HEAD:

$ cat .git/HEAD
ref: refs/heads/master

Как вы можете видеть, это сообщает нам, в какой ветке мы сейчас находимся, и сообщает нам об этом, называя файл в каталоге .git, который сам содержит имя SHA-1, ссылающееся на объект фиксации, который мы можем проверить с помощью кошачий-файл:

$ cat .git/refs/heads/master
c4d59f390b9cfd4318117afde11d601c1085f241
$ git cat-file -t c4d59f39
commit
$ git cat-file commit c4d59f39
tree d0492b368b66bdabf2ac1fd8c92b39d3db916e59
parent 54196cc2703dc165cbd373a65a4dcf22d50ae7f7
author J. Bruce Fields <bfields@puzzle.fieldses.org> 1143418702 -0500
committer J. Bruce Fields <bfields@puzzle.fieldses.org> 1143418702 -0500

add emphasis

Объект "tree" здесь относится к новому состоянию дерева:

$ git ls-tree d0492b36
100644 blob a0423896973644771497bdc03eb99d5281615b51    file.txt
$ git cat-file blob a0423896
hello world!

а объект "parent" ссылается на предыдущий коммит:

$ git cat-file commit 54196cc2
tree 92b8b694ffb1675e5975148e1121810081dbdffe
author J. Bruce Fields <bfields@puzzle.fieldses.org> 1143414668 -0500
committer J. Bruce Fields <bfields@puzzle.fieldses.org> 1143414668 -0500

initial commit

Объект дерева — это дерево, которое мы исследовали первым, и этот коммит необычен тем, что у него нет родителя.

Большинство коммитов имеют только одного родителя, но коммит также может иметь несколько родителей. В этом случае фиксация представляет собой слияние, при этом родительские ссылки указывают на заголовки объединенных ветвей.

Помимо больших двоичных объектов, деревьев и коммитов, единственным оставшимся типом объектов является "tag" , который мы здесь обсуждать не будем; подробности см. в git-tag[1] .

Итак, теперь мы знаем, как Git использует базу данных объектов для представления истории проекта:

  • Объекты "commit" ссылаются на объекты "tree" , представляющие снимок дерева каталогов в определенный момент истории, и ссылаются на фиксации "parent" , чтобы показать, как они связаны с историей проекта.

  • Объекты "tree" представляют состояние одного каталога, связывая имена каталогов с объектами "blob" , содержащими данные файла, и объектами "tree" , содержащими информацию о подкаталогах.

  • Объекты "blob" содержат данные файла без какой-либо другой структуры.

  • Ссылки на объекты фиксации в начале каждой ветки хранятся в файлах .git/refs/heads/.

  • Имя текущей ветки хранится в .git/HEAD.

Кстати, обратите внимание, что многие команды принимают в качестве аргумента дерево. Но, как мы видим выше, на дерево можно ссылаться по-разному: по имени SHA-1 этого дерева, по имени коммита, который ссылается на дерево, по имени ветки, заголовок которой ссылается на дерево. к этому дереву и т. д. — и большинство таких команд могут принимать любое из этих имен.

В описаниях команд для обозначения такого аргумента иногда используется слово "tree-ish" .

Индексный файл

Основным инструментом, который мы использовали для создания коммитов, является git-commit -a , который создает коммит, включающий все изменения, внесенные вами в рабочее дерево. Но что, если вы хотите зафиксировать изменения только в определенных файлах? Или только определенные изменения в определенных файлах?

Если мы посмотрим на то, как создаются коммиты, мы увидим, что существуют более гибкие способы создания коммитов.

Продолжая наш тестовый проект, давайте еще раз изменим файл file.txt:

$ echo "hello world, again" >>file.txt

но на этот раз вместо того, чтобы сразу делать коммит, давайте сделаем промежуточный шаг и запросим различия по пути, чтобы отслеживать, что происходит:

$ git diff
--- a/file.txt
+++ b/file.txt
@@ -1 +1,2 @@
 hello world!
+hello world, again
$ git add file.txt
$ git diff

Последний дифф пуст, но новых коммитов сделано не было, а в заголовке по-прежнему нет новой строки:

$ git diff HEAD
diff --git a/file.txt b/file.txt
index a042389..513feba 100644
--- a/file.txt
+++ b/file.txt
@@ -1 +1,2 @@
 hello world!
+hello world, again

Так что git diff сравнивается с чем-то другим, кроме головы. То, с чем он сравнивается, на самом деле является индексным файлом, который хранится в .git/index в двоичном формате, но содержимое которого мы можем проверить с помощью ls-файлов:

$ git ls-files --stage
100644 513feba2e53ebbd2532419ded848ba19de88ba00 0       file.txt
$ git cat-file -t 513feba2
blob
$ git cat-file blob 513feba2
hello world!
hello world, again

Итак, наш git add сохранил новый большой двоичный объект, а затем поместил ссылку на него в индексный файл. Если мы изменим файл еще раз, мы увидим, что новые изменения отражены в выводе git diff :

$ echo 'again?' >>file.txt
$ git diff
index 513feba..ba3da7b 100644
--- a/file.txt
+++ b/file.txt
@@ -1,2 +1,3 @@
 hello world!
 hello world, again
+again?

При правильных аргументах git diff также может показать нам разницу между рабочим каталогом и последним коммитом или между индексом и последним коммитом:

$ git diff HEAD
diff --git a/file.txt b/file.txt
индекс a042389..ba3da7b 100644
--- a/file.txt
+++ b/file.txt
@@ -1 +1,3 @@
 hello world!
+hello world, again
+again?
$ git diff --cached
diff --git a/file.txt b/file.txt
индекс a042389..513feba 100644
--- a/file.txt
+++ b/file.txt
@@ -1 +1,2 @@
 hello world!
+hello world, again

В любой момент мы можем создать новую фиксацию, используя git commit (без опции "-a" ), и убедиться, что зафиксированное состояние включает только изменения, хранящиеся в индексном файле, а не дополнительные изменения, которые все еще есть только в нашем рабочем дереве:

$ git commit -m "repeat"
$ git diff HEAD
diff --git a/file.txt b/file.txt
индекс 513feba..ba3da7b 100644
--- a/file.txt
+++ b/file.txt
@@ -1,2 +1,3 @@
 hello world!
 hello world, again
+again?

Таким образом, по умолчанию git commit использует индекс для создания фиксации, а не рабочего дерева; опция фиксации "-a" указывает ему сначала обновить индекс со всеми изменениями в рабочем дереве.

Наконец, стоит посмотреть на влияние git add на индексный файл:

$ echo "goodbye, world" >closing.txt
$ git add closing.txt

Эффектом git add было добавление одной записи в индексный файл:

$ git ls-files --stage
100644 8b9743b20d4b15be3955fc8d5cd2b09cd2336138 0       closing.txt
100644 513feba2e53ebbd2532419ded848ba19de88ba00 0       file.txt

И, как вы можете видеть на примере cat-file, эта новая запись относится к текущему содержимому файла:

$ git cat-file blob 8b9743b2
goodbye, world

Команда "status" — полезный способ получить краткую информацию о ситуации:

$ git status
On branch master
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)

        new file:   closing.txt

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)

        modified:   file.txt

Поскольку текущее состояние файла close.txt кэшируется в индексном файле, оно указано как "Changes to be committed" . Поскольку файл file.txt содержит изменения в рабочем каталоге, которые не отражены в индексе, он помечен как "changed but not updated" . На этом этапе запуск "git commit" создаст коммит, который добавит файл close.txt (с его новым содержимым), но не изменит файл file.txt.

Также обратите внимание, что пустой git diff показывает изменения в файле file.txt, но не добавление файла close.txt, поскольку версия файла close.txt в индексном файле идентична версии в рабочем каталоге.

Помимо того, что индексный файл является промежуточной областью для новых коммитов, он также заполняется из объектной базы данных при извлечении ветки и используется для хранения деревьев, участвующих в операции слияния. Подробности см. в gitcore-tutorial[7] и на соответствующих страницах руководства.

What next?

На этом этапе вы должны знать все необходимое для чтения страниц руководства по любой из команд git ; Лучше всего начать с команд, упомянутых в giteveryday[7] . Вы сможете найти любой неизвестный жаргон в gitglossary[7] .

Git User’s Manual обеспечивает более полное введение в Git.

gitcvs-migration[7] объясняет, как импортировать репозиторий CVS в Git , и показывает, как использовать Git аналогично CVS.

Некоторые интересные примеры использования Git см. в howtos .

Для разработчиков Git статья gitcore-tutorial[7] подробно описывает механизмы Git нижнего уровня, задействованные, например, при создании новой фиксации.
