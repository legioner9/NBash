git-rev-parse

Name

git-rev-parse — выбрать и обработать параметры

Synopsis

git rev-parse [<options>] <args>…​

Description

Многие фарфоровые команды Git принимают смесь флагов (т.е. параметров, которые начинаются с тире - ) и параметров, предназначенных для базовой команды git rev-list , которую они используют внутри, а также флагов и параметров для других команд, которые они используют после git rev-list . Эта команда используется, чтобы различать их.

Options

Operation Modes

Каждый из этих параметров должен появиться первым в командной строке.

--parseopt

Используйте git rev-parse в режиме анализа опций (см. раздел PARSEOPT ниже).

--sq-quote

Используйте git rev-parse в режиме цитирования оболочки (см. раздел SQ-QUOTE ниже). В отличие от опции --sq , представленной ниже, этот режим выполняет только цитирование. Больше ничего не делается для ввода команды.

Параметры --parseopt

--keep-dashdash

Имеет смысл только в режиме --parseopt . Приказывает анализатору параметров выводить первый встреченный -- , а не пропускать его.

--stop-at-non-option

Имеет смысл только в режиме --parseopt . Позволяет анализатору опций остановиться на первом аргументе, не являющемся опцией. Это можно использовать для анализа подкоманд, которые сами принимают параметры.

--stuck-long

Имеет смысл только в режиме --parseopt . Выведите варианты в их полной форме, если они доступны, и с их аргументами.

Опции фильтрации

--revs-only

Не выводите флаги и параметры, не предназначенные для команды git rev-list .

--no-revs

Не выводить флаги и параметры, предназначенные для команды git rev-list .

--flags

Не выводите параметры без флагов.

--no-flags

Не выводить параметры флага.

Варианты вывода

--default <arg>

Если пользователь не указал параметр, используйте вместо него <arg> .

--prefix <arg>

Ведите себя так, как если бы git rev-parse был вызван из подкаталога <arg> рабочего дерева. Любые относительные имена файлов разрешаются так, как если бы они имели префикс <arg> , и будут печататься в этой форме.

Это можно использовать для преобразования аргументов в команду, выполняемую в подкаталоге, чтобы их можно было использовать после перехода на верхний уровень репозитория. Например:

prefix=$(git rev-parse --show-prefix)
cd "$(git rev-parse --show-toplevel)"
# rev-parse предоставляет -- необходимые для 'set'
eval "set $(git rev-parse --sq --prefix "$prefix" -- "$@")"
--verify

Убедитесь, что указан ровно один параметр и что его можно преобразовать в необработанный 20-байтовый SHA-1, который можно использовать для доступа к базе данных объектов. Если да, выведите его на стандартный вывод; в противном случае выдается ошибка.

Если вы хотите убедиться, что выходные данные действительно именуют объект в вашей базе данных объектов и/или могут использоваться в качестве объекта определенного типа, который вам нужен, вы можете добавить к параметру оператор очистки ^{type} . Например, git rev-parse "$VAR^{commit}" будет следить за тем, чтобы $VAR давал имя существующему объекту, который является коммитом (т. е. коммитом или аннотированным тегом, указывающим на коммит). Чтобы убедиться, что $VAR называет существующий объект любого типа, можно использовать git rev-parse "$VAR^{object}" .

Обратите внимание: если вы проверяете имя из ненадежного источника, разумно использовать --end-of-options , чтобы аргумент имени не был ошибочно принят за другой параметр.

-q
--quiet

Имеет смысл только в режиме --verify . Не выводить сообщение об ошибке, если первый аргумент не является допустимым именем объекта; вместо этого выйдите с ненулевым статусом молча. SHA-1 для допустимых имен объектов выводятся на стандартный вывод в случае успеха.

--sq

Обычно вывод осуществляется по одной строке на каждый флаг и параметр. Эта опция делает вывод одной строкой, правильно заключенной в кавычки для использования оболочкой. Полезно, если вы ожидаете, что ваш параметр будет содержать пробелы и новые строки (например, при использовании кирки -S с git diff-* ). В отличие от опции --sq-quote , ввод команды по-прежнему интерпретируется как обычно.

--short[=length]

То же, что и --verify , но имя объекта сокращается до уникального префикса, содержащего как минимум символы length . Минимальная длина — 4, по умолчанию — эффективное значение переменной конфигурации core.abbrev (см. git-config[1] ).

--not

При отображении имен объектов добавляйте к ним префикс ^ и удаляйте префикс ^ из имен объектов, у которых он уже есть.

--abbrev-ref[=(strict|loose)]

Однозначное короткое имя имени объекта. Опция core.warnAmbigousRefs используется для выбора строгого режима сокращений.

--symbolic

Обычно имена объектов выводятся в форме SHA-1 (с возможным префиксом ^ ); эта опция заставляет их выводиться в форме, максимально приближенной к исходному вводу.

--symbolic-full-name

Это похоже на --symbolic, но пропускает входные данные, которые не являются ссылками (т. е. имена ветвей или тегов; или, более явно, устраняющую неоднозначность форму "heads/master" , когда вы хотите назвать ветвь "master" , когда есть неудачно названный тег "master" ), и показывает их как полные имена (например, "refs/heads/master" ).

Опции для объектов

--all

Показать все ссылки, найденные на refs/ .

--branches[=pattern]
--tags[=pattern]
--remotes[=pattern]

Показать все ветки, теги или ветки удаленного отслеживания соответственно (т. е. ссылки, найденные в refs/heads , refs/tags или refs/remotes соответственно).

Если указан pattern , отображаются только ссылки, соответствующие данному glob оболочки. Если шаблон не содержит подстановочного символа ( ? , * или [ ), он преобразуется в совпадение префикса путем добавления /* .

--glob=pattern

Показать все ссылки, соответствующие шаблону pattern оболочки. Если шаблон не начинается с refs/ , он автоматически добавляется в начало. Если шаблон не содержит подстановочного символа ( ? , * или [ ), он преобразуется в совпадение префикса путем добавления /* .

--exclude=<glob-pattern>

Не включайте ссылки, соответствующие <glob-pattern> , которые в противном случае учитывались бы следующими --all , --branches , --tags , --remotes или --glob . При повторении этого параметра накапливаются шаблоны исключений до следующего параметра --all , --branches , --tags , --remotes или --glob (другие параметры или аргументы не удаляют накопленные шаблоны).

Приведенные шаблоны не должны начинаться с refs/heads , refs/tags или refs/remotes при применении к --branches , --tags или --remotes соответственно, и они должны начинаться с refs/ при применении к --glob или --all . Если предполагается завершающий /* , его необходимо указать явно.

--exclude-hidden=[fetch|receive|uploadpack]

Не включайте ссылки, которые могут быть скрыты git-fetch , git-receive-pack или git-upload-pack , сверившись с соответствующей конфигурацией fetch.hideRefs , receive.hideRefs или uploadpack.hideRefs вместе с transfer.hideRefs (см. git-config[1] ). Эта опция влияет на следующую опцию псевдо-ссылки --all или --glob и очищается после их обработки.

--disambiguate=<prefix>

Показать все объекты, имя которых начинается с данного префикса. Длина <префикса> должна быть не менее 4 шестнадцатеричных цифр, чтобы избежать ошибочного перечисления всех объектов в репозитории.

Опции для файлов

--local-env-vars

Перечислите переменные среды GIT_*, которые являются локальными для репозитория (например, GIT_DIR или GIT_WORK_TREE, но не GIT_EDITOR). В списке указаны только имена переменных, а не их значения, даже если они установлены.

--path-format=(absolute|relative)

Управляет поведением некоторых других опций. Если указано как абсолютное, пути, напечатанные этими параметрами, будут абсолютными и каноническими. Если указано относительное значение, пути будут указываться относительно текущего рабочего каталога, если это возможно. Значение по умолчанию зависит от опции.

Этот параметр может быть указан несколько раз и влияет только на аргументы, следующие за ним в командной строке, либо до конца командной строки, либо до следующего экземпляра этого параметра.

Следующие параметры модифицируются --path-format :

--git-dir

Покажите $GIT_DIR , если он определен. В противном случае укажите путь к каталогу .git. Отображаемый путь, если он относительный, относится к текущему рабочему каталогу.

Если $GIT_DIR не определен и текущий каталог не находится в репозитории Git или рабочем дереве, выведите сообщение в stderr и завершите работу с ненулевым статусом.

--git-common-dir

Покажите $GIT_COMMON_DIR , если определено, иначе $GIT_DIR .

--resolve-git-dir <path>

Проверьте, является ли <path> действительным репозиторием или git-файлом, указывающим на действительный репозиторий, и распечатайте местоположение репозитория. Если <path> — это git-файл, то выводится разрешенный путь к реальному репозиторию.

--git-path <path>

Разрешите "$GIT_DIR/<path>" и примите во внимание другие переменные перемещения пути, такие как $GIT_OBJECT_DIRECTORY, $GIT_INDEX_FILE…. Например, если для $GIT_OBJECT_DIRECTORY установлено значение /foo/bar, то "git rev-parse --git-path objects/abc" возвращает /foo/bar/abc.

--show-toplevel

Показать путь (по умолчанию абсолютный) к каталогу верхнего уровня рабочего дерева. Если рабочего дерева нет, сообщите об ошибке.

--show-superproject-working-tree

Покажите абсолютный путь к корню рабочего дерева суперпроекта (если оно существует), которое использует текущий репозиторий в качестве своего подмодуля. Ничего не выводит, если текущий репозиторий не используется в качестве подмодуля каким-либо проектом.

--shared-index-path

Покажите путь к общему индексному файлу в режиме разделенного индекса или пустой, если не в режиме разделенного индекса.

--path-format не влияет на следующие параметры:

--absolute-git-dir

Подобно --git-dir , но его выходные данные всегда представляют собой канонизированный абсолютный путь.

--is-inside-git-dir

Если текущий рабочий каталог находится ниже каталога репозитория, выведите "true" , в противном случае "false" .

--is-inside-work-tree

Если текущий рабочий каталог находится внутри рабочего дерева репозитория, выведите "true" , в противном случае "false" .

--is-bare-repository

Если репозиторий пуст, выведите "true" , иначе "false" .

--is-shallow-repository

Если репозиторий неглубокий, выведите "true" , иначе "false" .

--show-cdup

Когда команда вызывается из подкаталога, укажите путь к каталогу верхнего уровня относительно текущего каталога (обычно это последовательность "../" или пустая строка).

--show-prefix

Когда команда вызывается из подкаталога, показывает путь к текущему каталогу относительно каталога верхнего уровня.

--show-object-format[=(storage|input|output)]

Покажите формат объекта (алгоритм хеширования), используемый для хранилища для хранения внутри каталога .git , ввода или вывода. Для ввода можно вывести несколько алгоритмов, разделенных пробелами. Если не указано, значение по умолчанию — "storage" .

Other Options

--since=datestring
--after=datestring

Проанализируйте строку даты и выведите соответствующий параметр --max-age= для git rev-list .

--until=datestring
--before=datestring

Проанализируйте строку даты и выведите соответствующий параметр --min-age= для git rev-list .

<args>…

Флаги и параметры для анализа.

Specifying revisions

Параметр редакции <rev> обычно, но не обязательно, называет объект фиксации. Он использует так называемый синтаксис extended SHA-1 . Вот различные способы написания названий объектов. Те, что перечислены в конце этого списка, называют деревья и объекты, содержащиеся в коммите.

Note
 В этом документе показан синтаксис "raw" , как его видит git. Оболочке и другим пользовательским интерфейсам могут потребоваться дополнительные кавычки для защиты специальных символов и во избежание разделения слов.
<sha1>, например dae86e1950b1277e545cee180551750029cfe735, dae86e

Полное имя объекта SHA-1 (40-байтовая шестнадцатеричная строка) или ведущая подстрока, уникальная в пределах репозитория. Например, dae86e1950b1277e545cee180551750029cfe735 и dae86e называют один и тот же объект фиксации, если в вашем репозитории нет другого объекта, имя которого начинается с dae86e.

<describeOutput>, например v1.7.4.2-679-g3bee7fb

Выход из git describe ; т. е. ближайший тег, за которым необязательно следует тире и несколько коммитов, за которыми следует тире, g и сокращенное имя объекта.

<имя ссылки>, например master, heads/master, refs/heads/master

Символическое имя ссылки. Например, master обычно означает объект фиксации, на который ссылается refs/heads/master . Если у вас есть и heads/master , и tags/master , вы можете явно сказать heads/master , чтобы сообщить Git , какой из них вы имеете в виду. В случае неоднозначности <refname> устраняется путем поиска первого совпадения по следующим правилам:

  1. Если $GIT_DIR/<refname> существует, вы имеете в виду именно это (обычно это полезно только для HEAD , FETCH_HEAD , ORIG_HEAD , MERGE_HEAD , REBASE_HEAD , REVERT_HEAD , CHERRY_PICK_HEAD , BISECT_HEAD и AUTO_MERGE );

  2. в противном случае refs/<refname> , если он существует;

  3. в противном случае refs/tags/<refname> , если он существует;

  4. в противном случае refs/heads/<refname> , если он существует;

  5. в противном случае refs/remotes/<refname> , если он существует;

  6. в противном случае — refs/remotes/<refname>/HEAD , если он существует.

    HEAD

    называет коммит, на котором вы основывали изменения в рабочем дереве.

    FETCH_HEAD

    записывает ветку, которую вы получили из удаленного репозитория при последнем вызове git fetch .

    ORIG_HEAD

    создается командами, которые резко перемещают ваш HEAD ( git am , git merge , git rebase , git reset ), чтобы записать положение HEAD перед его работой, чтобы вы могли легко вернуть кончик ветки в состояние до того, как вы управлял ими.

    MERGE_HEAD

    записывает коммиты, которые вы объединяете в свою ветку при запуске git merge .

    REBASE_HEAD

    во время перебазирования записывает фиксацию, на которой операция в данный момент остановлена ​​либо из-за конфликтов, либо из-за команды edit при интерактивном перебазировании.

    REVERT_HEAD

    записывает фиксацию, которую вы отменяете при запуске git revert .

    CHERRY_PICK_HEAD

    записывает коммит, который вы выбираете при запуске git cherry-pick .

    BISECT_HEAD

    записывает текущий коммит для проверки при запуске git bisect --no-checkout .

    AUTO_MERGE

    записывает объект дерева, соответствующий состоянию, которое стратегия слияния ort записала в рабочее дерево, когда операция слияния привела к конфликтам.

Обратите внимание, что любой из приведенных выше случаев refs/* может быть получен либо из каталога $GIT_DIR/refs , либо из файла $GIT_DIR/packed-refs . Хотя кодировка имени ссылки не указана, предпочтительнее использовать UTF-8, поскольку при некоторой обработке вывода могут использоваться имена ссылок в UTF-8.

@

@ сам по себе является сокращением для HEAD .

[<refname>]@{<date>}, например master@{вчера}, HEAD@{5 минут назад}

Ссылка, за которой следует суффикс @ со спецификацией даты, заключенной в пару фигурных скобок (например, {yesterday} , {1 month 2 weeks 3 days 1 hour 1 second ago} или {1979-02-26 18:30:00} ), указывает значение ссылки в предыдущий момент времени. Этот суффикс можно использовать только сразу после имени ссылки, и ссылка должна иметь существующий журнал ( $GIT_DIR/logs/<ref> ). Обратите внимание, что при этом проверяется состояние вашей локальной ссылки в данный момент времени; например, что было в вашем местном филиале master на прошлой неделе. Если вы хотите просмотреть коммиты, сделанные в определенное время, см. --since и --until .

<refname>@{<n>}, например master@{1}

Ссылка, за которой следует суффикс @ с порядковым номером, заключенным в пару фигурных скобок (например, {1} , {15} ), указывает n-е предшествующее значение этой ссылки. Например, master@{1} — это непосредственное предшествующее значение master , а master@{5} — пятое предшествующее значение master . Этот суффикс можно использовать только сразу после имени ссылки, и ссылка должна иметь существующий журнал ( $GIT_DIR/logs/<refname> ).

@{<n>}, например @{1}

Вы можете использовать конструкцию @ с пустой частью ссылки, чтобы получить запись журнала обновления текущей ветки. Например, если вы находитесь в ветке blabla , то @{1} означает то же самое, что и blabla@{1} .

@{-<n>}, например @{-1}

Конструкция @{-<n>} означает, что <n>-я ветвь/фиксация извлечена перед текущей.

[<имя ветки>]@{upstream}, например master@{upstream}, @{u}

Ветвь B может быть настроена поверх ветки X (настроенной с помощью branch.<name>.merge ) на удаленном R (настроенной с помощью branch.<name>.remote ). B@{u} относится к ветке удаленного отслеживания для ветки X, взятой из удаленного R, обычно находящейся по адресу refs/remotes/R/X .

[<имя ветки>]@{push}, например master@{push}, @{push}

Суффикс @{push} сообщает о ветке "where we would push to" , если git push был запущен во время извлечения branchname (или текущий HEAD , если имя ветки не указано). Как и в случае с @{upstream} , мы сообщаем о ветке удаленного отслеживания, которая соответствует этой ветке на удаленном компьютере.

Вот пример, чтобы было более понятно:

$ git config push.default current
$ git config remote.pushdefault myfork
$ git switch -c mybranch origin/master

$ git rev-parse --symbolic-full-name @{upstream}
refs/remotes/origin/master

$ git rev-parse --symbolic-full-name @{push}
refs/remotes/myfork/mybranch

Обратите внимание, что в примере мы настроили треугольный рабочий процесс, в котором мы извлекаем данные из одного места и передаем их в другое. В нетреугольном рабочем процессе @{push} — то же самое, что @{upstream} , и необходимости в нем нет.

Этот суффикс также принимается, если он написан заглавными буквами, и означает одно и то же, независимо от регистра.

<rev>^[<n>], например HEAD^, v1.5.1^0

Суффикс ^ к параметру версии означает первого родительского объекта этого объекта фиксации. ^<n> означает <n>-го родительского элемента (т. е. <rev>^ эквивалентен <rev>^1 ). Как особое правило, <rev>^0 означает саму фиксацию и используется, когда <rev> — это имя объекта тега, который ссылается на объект фиксации.

<rev>~[<n>], например HEAD~, master~3

Суффикс ~ к параметру версии означает первого родительского объекта этого объекта фиксации. Суффикс ~<n> к параметру редакции означает объект фиксации, который является предком <n>-го поколения именованного объекта фиксации, следующим только за первыми родительскими объектами. Т.е. <rev>~3 эквивалентен <rev>^^^ , который эквивалентен <rev>^1^1^1 . Ниже приведена иллюстрация использования этой формы.

<rev>^{<type>}, например v0.99.8^{commit}

Суффикс ^ , за которым следует имя типа объекта, заключенное в пару фигурных скобок, означает рекурсивное разыменование объекта по адресу <rev> до тех пор, пока не будет найден объект типа <type> или пока объект больше не сможет быть разыменован (в этом случае — barf). Например, если <rev> является коммитом, <rev>^{commit} описывает соответствующий объект фиксации. Аналогично, если <rev> является древовидным, <rev>^{tree} описывает соответствующий объект дерева. <rev>^0 — это сокращение от <rev>^{commit} .

<rev>^{object} можно использовать, чтобы убедиться, что <rev> называет существующий объект, не требуя, чтобы <rev> был тегом, и без разыменования <rev> ; поскольку тег уже является объектом, его не нужно разыменовывать ни разу, чтобы добраться до объекта.

<rev>^{tag} можно использовать, чтобы гарантировать, что <rev> идентифицирует существующий объект-тег.

<rev>^{}, например v0.99.8^{}

Суффикс ^ , за которым следует пустая пара фигурных скобок, означает, что объект может быть тегом и разыменовывать тег рекурсивно, пока не будет найден объект, не являющийся тегом.

<rev>^{/<text>}, например HEAD^{/исправить неприятную ошибку}

Суффикс ^ к параметру версии, за которым следует пара фигурных скобок, содержащая текст, возглавляемый косой чертой, аналогичен синтаксису :/fix nasty bug ниже, за исключением того, что он возвращает самый молодой соответствующий коммит, доступный из <rev> до ^ .

:/<текст>, например:/исправить неприятную ошибку

Двоеточие, за которым следует косая черта, а затем текст, называют коммит, сообщение о коммите которого соответствует указанному регулярному выражению. Это имя возвращает самый молодой соответствующий коммит, доступный по любой ссылке, включая HEAD. Регулярное выражение может соответствовать любой части сообщения фиксации. Чтобы сопоставить сообщения, начинающиеся со строки, можно использовать, например, :/^foo . Специальная последовательность :/! зарезервирована для модификаторов того, что соответствует. :/!-foo выполняет отрицательное сопоставление, а :/!!foo соответствует буквальному символу ! , за которым следует foo . Любая другая последовательность, начинающаяся с :/! , пока зарезервирована. В зависимости от данного текста правила разделения слов оболочки могут потребовать дополнительных кавычек.

<rev>:<путь>, например HEAD:README, master:./README

Суффикс : , за которым следует путь, называет большой объект или дерево по заданному пути в древовидном объекте, имя которого указано в части перед двоеточием. Путь, начинающийся с ./ или ../ , относится к текущему рабочему каталогу. Указанный путь будет преобразован относительно корневого каталога рабочего дерева. Это наиболее полезно для обращения к BLOB-объекту или дереву из коммита или дерева, которое имеет ту же древовидную структуру, что и рабочее дерево.

:[<n>:]<путь>, например :0:README, :README

Двоеточие, за которым необязательно следует номер стадии (от 0 до 3), и двоеточие, за которым следует путь, называют объект BLOB-объекта в индексе по заданному пути. Отсутствующий номер этапа (и двоеточие, которое следует за ним) обозначает запись этапа 0. Во время слияния этап 1 — это общий предок, этап 2 — версия целевой ветки (обычно текущая ветвь), а этап 3 — версия объединяемой ветки.

Вот иллюстрация Джона Лелигера. Оба узла фиксации B и C являются родительскими узлами фиксации A. Родительские узлы упорядочены слева направо.

G   H   I   J
 \ /     \ /
  D   E   F
   \  |  / \
    \ | /   |
     \|/    |
      B     C
       \   /
        \ /
         A
A =      = A^0
B = A^   = A^1     = A~1
C =      = A^2
D = A^^  = A^1^1   = A~2
E = B^2  = A^^2
F = B^3  = A^^3
G = A^^^ = A^1^1^1 = A~3
H = D^2  = B^^2    = A^^^2  = A~2^2
I = F^   = B^3^    = A^^3^
J = F^2  = B^3^2   = A^^3^2

Specifying ranges

Команды просмотра истории, такие как git log , работают с набором коммитов, а не с одним коммитом.

Для этих команд указание одной ревизии с использованием обозначений, описанных в предыдущем разделе, означает набор коммитов reachable из данного коммита.

Указание нескольких ревизий означает набор коммитов, доступных из любого из данных коммитов.

Достижимый набор коммита — это сам коммит и коммиты в его родительской цепочке.

Существует несколько обозначений для указания набора связанных коммитов (называемых "revision range" ), как показано ниже.

Commit Exclusions

^<rev> (курсор) Обозначение

Чтобы исключить коммиты, доступные из коммита, используется нотация префикса ^ . Например, ^r1 r2 означает коммиты, доступные из r2 , но исключающие те, которые доступны из r1 (т. е. r1 и его предки).

Обозначения точечного диапазона

Обозначение диапазона .. (две точки)

Операция установки ^r1 r2 встречается так часто, что для нее существует сокращение. Если у вас есть два коммита r1 и r2 (названные в соответствии с синтаксисом, описанным выше в разделе «УКАЗАНИЕ ВЕРСИЙ»), вы можете запросить коммиты, доступные из r2, за исключением тех, которые доступны из r1 для ^r1 r2 , и их можно записать как r1..r2 .

... (три точки) Обозначение симметричных разностей

Подобное обозначение r1...r2 называется симметричной разностью r1 и r2 и определяется как r1 r2 --not $(git merge-base --all r1 r2) . Это набор коммитов, которые доступны либо из r1 (левая сторона), либо из r2 (правая сторона), но не из обоих.

В этих двух сокращенных обозначениях вы можете опустить один конец и оставить его по умолчанию HEAD. Например, origin.. является сокращением для origin..HEAD и запрашивает "What did I do since I forked from the origin branch?" . Аналогично, ..origin является сокращением для HEAD..origin и запрашивает "What did the origin do since I forked from them?" . Обратите внимание, что .. будет означать HEAD..HEAD , который представляет собой пустой диапазон, который одновременно достижим и недостижим из HEAD.

Команды, специально предназначенные для работы с двумя разными диапазонами (например, "git range-diff R1 R2" для сравнения двух диапазонов), существуют, но это исключения. Если не указано иное, все команды "git" , которые работают с набором коммитов, работают в одном диапазоне ревизий. Другими словами, написание двух "two-dot range notation" рядом друг с другом, например

$ git log A..B C..D

не указывает два диапазона версий для большинства команд. Вместо этого он будет называть один связанный набор коммитов, то есть те, которые доступны из B или D, но недоступны ни из A, ни из C. В такой линейной истории:

---A---B---o---o---C---D

поскольку A и B доступны из C, диапазон ревизий, указанный этими двумя пунктирными диапазонами, представляет собой одну фиксацию D.

Другие <rev>^ родительские сокращенные обозначения

Существуют еще три сокращения, особенно полезные для коммитов слияния, для обозначения набора, образованного коммитом и его родительскими коммитами.

Обозначение r1^@ означает всех родителей r1 .

Нотация r1^! включает фиксацию r1 , но исключает всех ее родительских элементов. Сама по себе эта запись обозначает одиночный коммит r1 .

Нотация <rev>^-[<n>] включает <rev> , но исключает <n>-го родительского элемента (т. е. сокращение для <rev>^<n>..<rev> ), при этом <n> = 1, если он не указан. Обычно это полезно для коммитов слияния, когда вы можете просто передать <commit>^- , чтобы получить все коммиты в ветке, которая была объединена в коммит слияния <commit> (включая сам <commit> ).

Хотя в <rev>^<n> речь шла об указании одного родителя фиксации, эти три обозначения также учитывают и его родителей. Например, вы можете сказать HEAD^2^@ , но не можете сказать HEAD^@^2 .

Сводка диапазона редакций

<rev>

Включите коммиты, доступные из <rev> (т. е. <rev> и его предков).

^<rev>

Исключите коммиты, доступные из <rev> (т. е. <rev> и его предков).

<rev1>..<rev2>

Включите коммиты, доступные из <rev2>, но исключите те, которые доступны из <rev1>. Если <rev1> или <rev2> опущены, по умолчанию используется HEAD .

<rev1>...<rev2>

Включите коммиты, доступные из <rev1> или <rev2>, но исключите те, которые доступны из обоих. Если <rev1> или <rev2> опущены, по умолчанию используется HEAD .

<rev>^@, например HEAD^@

Суффикс ^ , за которым следует знак at, аналогичен списку всех родителей <rev> (то есть включает все, что доступно от его родителей, но не сам коммит).

<rev>^!, например HEAD^!

Суффикс ^ , за которым следует восклицательный знак, аналогичен коммиту <rev> и всем его родительским элементам с префиксом ^ , чтобы исключить их (и их предков).

<rev>^-<n>, например HEAD^-, HEAD^-2

Эквивалент <rev>^<n>..<rev> , с <n> = 1, если не указано.

   Args   Expanded arguments    Selected commits
   D                            G H D
   D F                          G H I J D F
   ^G D                         H D
   ^D B                         E I J F B
   ^D B C                       E I J F B C
   C                            I J F C
   B..C   = ^B C                C
   B...C  = B ^F C              G H D E B C
   B^-    = B^..B
          = ^B^1 B              E I J F B
   C^@    = C^1
          = F                   I J F
   B^@    = B^1 B^2 B^3
          = D E F               D G H E F I J
   C^!    = C ^C^@
          = C ^C^1
          = C ^F                C
   B^!    = B ^B^@
          = B ^B^1 ^B^2 ^B^3
          = B ^D ^E ^F          B

Parseopt

В режиме --parseopt git rev-parse помогает массировать параметры, чтобы придать сценариям оболочки те же возможности, которые есть у встроенных C. Он работает как нормализатор опций (например, разделяет совокупные значения отдельных переключателей), примерно так же, как это делает getopt(1) .

Он принимает на стандартный ввод спецификацию параметров для анализа и понимания и отображает на стандартном выходе строку, подходящую для sh(1) eval для замены аргументов нормализованными. В случае ошибки он выводит информацию об использовании в стандартный поток ошибок и завершает работу с кодом 129.

Примечание. Обязательно укажите результат при передаче его в eval . См. пример ниже.

Input Format

Формат ввода git rev-parse --parseopt полностью текстовый. Он состоит из двух частей, разделенных строкой, содержащей только -- . Строки перед разделителем (должно быть одна или несколько) используются для использования. Строки после разделителя описывают параметры.

Каждая строка параметров имеет следующий формат:

<opt-spec><flags>*<arg-hint>? SP+ help LF
<opt-spec>

его формат — короткий символ параметра, затем длинное имя параметра, разделенное запятой. Обе части не обязательны, хотя необходима хотя бы одна. Не может содержать символы <flags> . h,help , dry-run и f являются примерами правильного <opt-spec> .

<flags>

<flags> — это * , = , ? или ! .

  • Используйте = , если опция принимает аргумент.

  • Используйте ? , чтобы указать, что опция принимает необязательный аргумент. Вероятно, вы захотите использовать режим --stuck-long , чтобы иметь возможность однозначно анализировать необязательный аргумент.

  • Используйте * , чтобы указать, что этот параметр не должен быть указан в использовании, созданном для аргумента -h . Это показано для --help-all , как описано в gitcli[7] .

  • Используйте ! , чтобы не делать доступной соответствующую отрицательную длинную опцию.

<arg-hint>

<arg-hint> , если он указан, используется в качестве имени аргумента в выводе справки для параметров, принимающих аргументы. <arg-hint> заканчивается первым пробелом. В подсказках, состоящих из нескольких слов, для разделения слов принято использовать тире.

Оставшаяся часть строки после удаления пробелов используется в качестве подсказки, связанной с опцией.

Пустые строки игнорируются, а строки, не соответствующие этой спецификации, используются в качестве заголовков групп опций (начинайте строку с пробела, чтобы создавать такие строки специально).

Example

OPTS_SPEC="\
some-command [<options>] <args>...

some-command does foo and bar!
--
h,help!   show the help

foo       some nifty option --foo
bar=      some cool option --bar with an argument
baz=arg   another cool option --baz with a named argument
qux?path  qux may take a path argument but has meaning by itself

  An option group Header
C?        option C with an optional argument"

eval "$(echo "$OPTS_SPEC" | git rev-parse --parseopt -- "$@" || echo exit $?)"

Usage text

Если "$@" — это -h или --help в приведенном выше примере, будет показан следующий текст использования:

usage: some-command [<options>] <args>...

    some-command does foo and bar!

    -h, --help показать помощь
    --[no-]foo, отличный вариант --foo
    --[no-]bar ... какой-нибудь классный вариант --bar с аргументом
    --[no-]baz <arg> еще один интересный вариант --baz с именованным аргументом
    --[no-]qux[=<путь>] qux может принимать аргумент пути, но имеет смысл сам по себе

An option group Header
    -C[...]               option C with an optional argument

Sq-quote

В режиме --sq-quote git rev-parse отображает на стандартном выходе одну строку, подходящую для sh(1) eval . Эта строка создается путем нормализации аргументов, следующих за --sq-quote . Ничего, кроме цитирования аргументов, не делается.

Если вы хотите, чтобы ввод команды по-прежнему интерпретировался git rev-parse как обычно до того, как вывод будет заключен в кавычки оболочки, см. опцию --sq .

Example

$ cat >your-git-script.sh <<\EOF
#!/bin/sh
args=$(git rev-parse --sq-quote "$@")   # цитируем аргументы, предоставленные пользователем
command="git frotz -n24 $args"          # и использовать его внутри созданного вручную
                                        # командная строка
eval "$command"
EOF

$ sh your-git-script.sh "a b'c"

Examples

  • Выведите имя объекта текущего коммита:

    $ git rev-parse --verify HEAD

  • Напечатайте имя объекта фиксации из ревизии в переменной оболочки $REV:

    $ git rev-parse --verify --end-of-options $REV^{commit}

    Это приведет к ошибке, если $REV пуста или не является допустимой ревизией.

  • Аналогично приведенному выше:

    $ git rev-parse --default master --verify --end-of-options $REV

    но если $REV пуст, будет напечатано имя объекта фиксации из мастера.
