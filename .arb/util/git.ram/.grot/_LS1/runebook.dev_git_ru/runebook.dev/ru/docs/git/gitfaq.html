gitfaq

Name

gitfaq — Часто задаваемые вопросы об использовании Git

Synopsis

gitfaq

Description

В примерах этого FAQ предполагается использование стандартной оболочки POSIX, например bash или dash , и пользователя AU Thor, имеющего учетную запись author у хостинг-провайдера git.example.org .

Configuration

Что мне поставить в user.name ?

Вы должны указать свое личное имя, обычно это форма, в которой используются имя и фамилия. Например, текущий сопровождающий Git использует "Junio C Hamano" . Это будет часть имени, которая сохраняется в каждом сделанном вами коммите.

Эта конфигурация не влияет на аутентификацию в удаленных службах; для этого см. credential.username в git-config[1] .

Что на самом деле делает http.postBuffer ?

Этот параметр изменяет размер буфера, который Git использует при отправке данных на удаленный компьютер через HTTP или HTTPS. Если данные превышают этот размер, libcurl, который обеспечивает поддержку HTTP для Git , будет использовать фрагментированное кодирование передачи, поскольку заранее неизвестно, каким будет размер отправленных данных.

Оставить это значение в размере по умолчанию можно, если только вы не знаете, что удаленный сервер или прокси-сервер в середине не поддерживает HTTP/1.1 (который ввел кодировку фрагментированной передачи) или известно, что он не работает с фрагментированными данными. Это часто (ошибочно) предлагается в качестве решения общих проблем push-уведомлений, но поскольку почти каждый сервер и прокси-сервер поддерживают как минимум HTTP/1.1, повышение этого значения обычно не решает большинство проблем push-уведомлений. Сервер или прокси-сервер, которые неправильно поддерживают HTTP/1.1 и фрагментированное кодирование передачи, сегодня не будут так полезны в Интернете, поскольку это приведет к нарушению большого количества трафика.

Обратите внимание, что увеличение этого значения приведет к увеличению памяти, используемой при каждой соответствующей отправке данных, которую Git выполняет через HTTP или HTTPS, поскольку весь буфер выделяется независимо от того, используется ли он весь. Таким образом, лучше оставить значение по умолчанию, если вы не уверены, что вам нужно другое значение.

Как мне настроить другой редактор?

Если вы не указали редактор специально для Git, по умолчанию он будет использовать редактор, который вы настроили с помощью переменных среды VISUAL или EDITOR , или, если ни один из них не указан, системный редактор по умолчанию (обычно это vi ). Поскольку некоторые люди находят vi трудным в использовании или предпочитают другой редактор, может оказаться желательным сменить используемый редактор.

Если вы хотите настроить общий редактор для большинства программ, которым он нужен, вы можете отредактировать конфигурацию оболочки (например, ~/.bashrc или ~/.zshenv ), чтобы она содержала строку, устанавливающую для переменной среды EDITOR или VISUAL подходящее значение. Например, если вы предпочитаете редактор nano , то вы можете написать следующее:

export VISUAL=nano

Если вы хотите настроить редактор специально для Git, вы можете установить значение конфигурации core.editor или переменную среды GIT_EDITOR . Подробную информацию о порядке обращения к этим параметрам можно найти в git-var[1] .

Обратите внимание, что во всех случаях значение редактора будет передано в оболочку, поэтому любые аргументы, содержащие пробелы, должны быть заключены в кавычки. Кроме того, если ваш редактор обычно отключается от терминала при вызове, вам следует указать его с аргументом, который запрещает это делать, иначе Git не увидит никаких изменений. Примером конфигурации, решающей обе эти проблемы в Windows, может быть конфигурация "C:\Program Files\Vim\gvim.exe" --nofork , которая заключает имя файла в кавычки с пробелами и указывает параметр --nofork , чтобы избежать фонового процесса.

Credentials

Как указать свои учетные данные при отправке через HTTP?

Самый простой способ сделать это — использовать помощник по учетным данным через конфигурацию credential.helper . Большинство систем предоставляют стандартный вариант интеграции с системным диспетчером учетных данных. Например, Git для Windows предоставляет диспетчер учетных данных wincred , macOS имеет диспетчер учетных данных osxkeychain , а системы Unix со стандартной средой рабочего стола могут использовать диспетчер учетных данных libsecret . Все эти учетные данные хранятся в зашифрованном хранилище, чтобы обеспечить безопасность ваших паролей или токенов.

Кроме того, вы можете использовать диспетчер учетных данных store , который хранится в файле в вашем домашнем каталоге, или диспетчер учетных данных cache , который не хранит ваши учетные данные постоянно, но не позволяет вам запрашивать их в течение определенного периода времени.

Вы также можете просто ввести свой пароль при появлении соответствующего запроса. Хотя можно поместить пароль (который должен быть закодирован в процентах) в URL-адрес, это не особенно безопасно и может привести к случайному раскрытию учетных данных, поэтому не рекомендуется.

Как прочитать пароль или токен из переменной среды?

Параметр конфигурации credential.helper также может принимать произвольную команду оболочки, которая создает протокол учетных данных на стандартном выходе. Это полезно, например, при передаче учетных данных в контейнер.

Такую команду оболочки можно указать, начиная значение параметра с восклицательного знака. Если ваш пароль или токен хранились в GIT_TOKEN , вы можете запустить следующую команду, чтобы установить помощника по учетным данным:

$ git config credential.helper \
        '!f() { echo username=author; echo "password=$GIT_TOKEN"; };f'
Как изменить пароль или токен, сохраненный в диспетчере учетных данных?

Обычно, если пароль или токен недействительны, Git сотрет их и предложит ввести новый. Однако бывают случаи, когда это происходит не всегда. Чтобы изменить пароль или токен, вы можете стереть существующие учетные данные, а затем Git предложит ввести новые. Чтобы стереть учетные данные, используйте следующий синтаксис (заменив свое имя пользователя и имя хоста):

$ echo url=https://[email protected] | git credential reject
Как использовать несколько учетных записей у одного и того же хостинг-провайдера с использованием HTTP?

Обычно самый простой способ отличить эти учетные записи — использовать имя пользователя в URL-адресе. Например, если у вас есть учетные записи author и committer на git.example.org , вы можете использовать URL-адреса https://[email protected]/org1/project1.git и https://[email protected]/org2/project2.git . Таким образом, когда вы используете помощник по учетным данным, он автоматически попытается найти правильные учетные данные для вашей учетной записи. Если у вас уже есть удаленная настройка, вы можете изменить URL-адрес, например, на git remote set-url origin https://[email protected]/org1/project1.git (подробнее см. git-remote[1] ).

Как мне использовать несколько учетных записей у одного и того же хостинг-провайдера с помощью SSH?

У большинства хостинг-провайдеров, поддерживающих SSH, одна пара ключей однозначно идентифицирует пользователя. Поэтому, чтобы использовать несколько учетных записей, необходимо создать пару ключей для каждой учетной записи. Если вы используете достаточно современную версию OpenSSH, вы можете создать новую пару ключей, например, ssh-keygen -t ed25519 -f ~/.ssh/id_committer . Затем вы можете зарегистрировать ключ public (в данном случае ~/.ssh/id_committer.pub ; обратите внимание на .pub ) у хостинг-провайдера.

Большинство хостинг-провайдеров используют для отправки одну учетную запись SSH; то есть все пользователи переходят на учетную запись git (например, [email protected] ). Если это касается вашего провайдера, вы можете настроить несколько псевдонимов в SSH, чтобы было понятно, какую пару ключей использовать. Например, вы можете написать в ~/.ssh/config что-то вроде следующего, заменив правильный файл ключа private :

# Это учетная запись автора на git.example.org.
Host example_author
        HostName git.example.org
        User git
        # Это пара ключей, зарегистрированная для автора на git.example.org.
        IdentityFile ~/.ssh/id_author
        IdentitiesOnly yes
# Это учетная запись коммиттера на git.example.org.
Host example_committer
        HostName git.example.org
        User git
        # Это пара ключей, зарегистрированная для коммиттера на git.example.org.
        IdentityFile ~/.ssh/id_committer
        IdentitiesOnly yes

Затем вы можете настроить URL-адрес push-уведомления, чтобы использовать git@example_author или git@example_committer вместо [email protected] (например, git remote set-url git@example_author:org1/project1.git ).

Common issues

Я допустил ошибку в последнем коммите. Как мне это изменить?

Вы можете внести соответствующие изменения в свое рабочее дерево, запустить git add <file> или git rm <file> , в зависимости от ситуации, для его подготовки, а затем git commit --amend . Ваше изменение будет включено в фиксацию, и вам будет предложено снова отредактировать сообщение о фиксации; Если вы хотите использовать исходное сообщение дословно, вы можете дополнительно использовать опцию --no-edit для git commit или просто сохранить и выйти, когда откроется редактор.

Я внес изменение с ошибкой, и оно было включено в основную ветку. Как мне это отменить?

Обычный способ справиться с этим — использовать git revert . Это сохраняет историю того, что исходное изменение было внесено и было ценным вкладом, но также вводится новый коммит, который отменяет эти изменения, поскольку в исходном была проблема. Сообщение о фиксации возврата указывает на фиксацию, которая была отменена, и обычно редактируется, чтобы включить объяснение того, почему был сделан возврат.

Как игнорировать изменения в отслеживаемом файле?

Git не предоставляет возможности сделать это. Причина в том, что если Git необходимо перезаписать этот файл, например, во время проверки, он не знает, являются ли изменения в файле ценными и их следует сохранить, или они не имеют значения и могут быть безопасно уничтожены. Поэтому ему следует выбирать безопасный маршрут и всегда сохранять его.

Соблазнительно попытаться использовать определенные функции git update-index , а именно биты предположения неизмененного и пропуска рабочего дерева, но они не работают должным образом для этой цели и не должны использоваться таким образом.

Если ваша цель — изменить файл конфигурации, часто бывает полезно зарегистрировать в репозитории файл, который представляет собой шаблон или набор значений по умолчанию, который затем можно скопировать и изменить соответствующим образом. Этот второй измененный файл обычно игнорируется, чтобы предотвратить его случайное сохранение.

Я попросил Git игнорировать различные файлы, но они все равно отслеживаются.

Файл gitignore гарантирует, что определенные файлы, которые не отслеживаются Git , остаются неотслеживаемыми. Однако иногда определенные файлы могут быть отслежены до их добавления в .gitignore , поэтому они все равно остаются отслеживаемыми. Чтобы отменить отслеживание и игнорировать файлы/шаблоны, используйте git rm --cached <file/pattern> и добавьте в .gitignore шаблон, соответствующий <file>. Подробности см. в gitignore[5] .

Как мне узнать, хочу ли я выполнить выборку или вытягивание?

При выборке сохраняется копия последних изменений из удаленного репозитория без изменения рабочего дерева или текущей ветки. Затем вы можете на досуге проверить, объединить, перебазировать или проигнорировать вышестоящие изменения. Извлечение состоит из выборки, за которой немедленно следует слияние или перебазирование. См. git-pull[1] .

Слияние и перебазирование

Какие проблемы могут возникнуть при объединении долгоживущих ветвей с помощью сквош-слияний?

В общем, существует множество проблем, которые могут возникнуть при использовании сквош-слияний для многократного объединения двух ветвей. Они могут включать в себя просмотр дополнительных коммитов в выходных данных git log , с помощью графического интерфейса или при использовании нотации ... для выражения диапазона, а также возможность необходимости снова и снова разрешать конфликты.

Когда Git выполняет обычное слияние между двумя ветвями, он учитывает ровно три точки: две ветки и третий коммит, называемый merge base , который обычно является общим предком коммитов. Результатом слияния является сумма изменений между базой слияния и каждой головкой. Когда вы объединяете две ветки с помощью обычного коммита слияния, это приводит к созданию нового коммита, который в конечном итоге станет базой слияния при повторном слиянии, поскольку теперь есть новый общий предок. Git не нужно учитывать изменения, произошедшие до объединения базы, поэтому вам не придется повторно разрешать конфликты, которые вы разрешили ранее.

Когда вы выполняете сжатое слияние, фиксация слияния не создается; вместо этого изменения с одной стороны применяются как обычная фиксация на другой стороне. Это означает, что база слияния для этих ветвей не изменится, и поэтому, когда Git приступает к следующему слиянию, он учитывает все изменения, которые он рассматривал в последний раз, а также новые изменения. Это означает, что любые конфликты, возможно, придется решать заново. Аналогично, все, что использует нотацию ... в git diff , git log или графическом интерфейсе, приведет к отображению всех изменений с момента исходной базы слияния.

Как следствие, если вы хотите повторно объединить две долгоживущие ветки, лучше всегда использовать обычную фиксацию слияния.

Если я вношу изменение в две ветки, но отменяю его в одной, почему слияние этих ветвей включает в себя это изменение?

По умолчанию, когда Git выполняет слияние, он использует стратегию, называемую стратегией ort , которая выполняет необычное трехстороннее слияние. В таком случае, когда Git выполняет слияние, он учитывает ровно три точки: две головы и третью точку, называемую merge base , которая обычно является общим предком этих коммитов. Git вообще не учитывает историю или отдельные фиксации, произошедшие в этих ветвях.

В результате, если обе стороны имеют изменение, а одна сторона отменила это изменение, результатом будет включение изменения. Это связано с тем, что код изменился с одной стороны, а с другой нет чистых изменений, и в этом сценарии Git принимает изменения.

Если для вас это проблема, вы можете вместо этого выполнить перебазирование, перебазировав ветку с возвратом на другую ветку. Перебазирование в этом сценарии приведет к отмене изменения, поскольку при перебазировании применяется каждая отдельная фиксация, включая возврат. Обратите внимание, что перебазирование перезаписывает историю, поэтому вам следует избегать перебазирования опубликованных ветвей, если вы не уверены, что вас это устраивает. Дополнительную информацию см. в разделе «ПРИМЕЧАНИЯ» в git-rebase[1] .

Hooks

Как использовать перехватчики, чтобы пользователи не могли вносить определенные изменения?

Единственное безопасное место для внесения этих изменений — это удаленный репозиторий (т. е. сервер Git ), обычно в перехватчике pre-receive или в системе непрерывной интеграции (CI). Это те места, где политика может эффективно применяться.

Обычно для проверки этих вещей пытаются использовать перехватчики pre-commit (или, для сообщений о фиксации, перехватчики commit-msg ), и это здорово, если вы работаете в качестве индивидуального разработчика и хотите, чтобы этот инструментарий вам помог. Однако использование перехватчиков на компьютере разработчика неэффективно в качестве контроля политики, поскольку пользователь может незаметно обойти эти перехватчики с помощью --no-verify (среди других способов). Git предполагает, что пользователь контролирует свои локальные репозитории, и не пытается предотвратить это или сплетничать о пользователе.

Кроме того, некоторые опытные пользователи считают, что перехватчики pre-commit являются препятствием для рабочих процессов, которые используют временные фиксации для этапа незавершенной работы или создают корректирующие фиксации, поэтому в любом случае лучше отправлять такие проверки на сервер.

Cross-platform issues

Я использую Windows, и мои текстовые файлы определяются как двоичные.

Git лучше всего работает при хранении текстовых файлов в формате UTF-8. Многие программы в Windows поддерживают UTF-8, но некоторые нет и используют только формат UTF-16 с прямым порядком байтов, который Git распознает как двоичный. Если вы не можете использовать UTF-8 в своих программах, вы можете указать кодировку рабочего дерева, которая указывает, с какой кодировкой следует извлекать ваши файлы, сохраняя при этом эти файлы в репозитории как UTF-8. Это позволяет таким инструментам, как git-diff[1] , работать должным образом, сохраняя при этом работоспособность ваших инструментов.

Для этого вы можете указать шаблон gitattributes[5] с атрибутом working-tree-encoding . Например, следующий шаблон устанавливает для всех файлов C использование UTF-16LE-BOM, которая является распространенной кодировкой в ​​Windows:

*.c        working-tree-encoding=UTF-16LE-BOM

Вам нужно будет запустить git add --renormalize , чтобы это вступило в силу. Обратите внимание: если вы вносите эти изменения в проект, который используется на разных платформах, вам, вероятно, захочется внести их в файл конфигурации для каждого пользователя или в файл $GIT_DIR/info/attributes , поскольку внесение его в файл .gitattributes в репозитории приведет к применить ко всем пользователям репозитория.

Дополнительную информацию о нормализации концов строк см. в следующей записи, а дополнительную информацию о файлах атрибутов см. в gitattributes[5] .

Я использую Windows, и разница git показывает, что мои файлы имеют ^M в конце.

По умолчанию Git ожидает, что файлы будут храниться с окончаниями строк Unix. Таким образом, отображается возврат каретки ( ^M ), который является частью окончания строки Windows, поскольку он считается конечным пробелом. Git по умолчанию показывает конечные пробелы только в новых строках, а не в существующих.

Вы можете хранить файлы в репозитории с окончаниями строк Unix и автоматически конвертировать их в окончания строк вашей платформы. Для этого установите для параметра конфигурации core.eol значение native и просмотрите следующую запись для получения информации о том, как настроить файлы как текстовые или двоичные.

Вы также можете контролировать это поведение с помощью настройки core.whitespace , если не хотите удалять возврат каретки из концов строки.

Почему у меня есть файл, который всегда изменяется?

Внутри Git всегда сохраняет имена файлов в виде последовательностей байтов и не выполняет никакого кодирования или свертывания регистра. Однако Windows и macOS по умолчанию выполняют свертывание регистра в именах файлов. В результате можно получить несколько файлов или каталогов, имена которых различаются только по регистру. Git прекрасно с этим справляется, но файловая система может хранить только один из этих файлов, поэтому, когда Git читает другой файл, чтобы увидеть его содержимое, он выглядит измененным.

Лучше всего удалить один из файлов, чтобы у вас остался только один файл. Вы можете сделать это с помощью следующих команд (при условии, что есть два файла AFile.txt и afile.txt ) в чистом рабочем дереве:

$ git rm --cached AFile.txt
$ git commit -m 'Remove files conflicting in case'
$ git checkout .

Это позволяет избежать прикосновения к диску, но удаляет дополнительный файл. Ваш проект может предпочесть принять соглашение об именах, например, имена в нижнем регистре, чтобы избежать повторного возникновения этой проблемы; такое соглашение можно проверить с помощью перехватчика pre-receive или как часть системы непрерывной интеграции (CI).

Также возможно, что постоянно измененные файлы появятся на любой платформе, если в вашей системе используется фильтр очистки или очистки, но файл был ранее зафиксирован без запуска фильтра очистки или очистки. Чтобы это исправить, запустите следующую команду на чистом рабочем дереве:

$ git add --renormalize .
Какой рекомендуемый способ хранить файлы в Git?

Хотя Git может хранить и обрабатывать любые файлы любого типа, некоторые настройки работают лучше других. В общем, мы рекомендуем хранить текстовые файлы в формате UTF-8 без метки порядка байтов (BOM) с окончаниями LF (в стиле Unix). Мы также рекомендуем использовать UTF-8 (опять же, без спецификации) в сообщениях о фиксации. Эти настройки лучше всего работают на разных платформах и с такими инструментами, как git diff и git merge .

Кроме того, если у вас есть выбор между текстовыми и нетекстовыми форматами хранения, мы рекомендуем хранить файлы в текстовом формате и, при необходимости, преобразовывать их в другой формат. Например, текстовый дамп SQL с одной записью в строке будет работать гораздо лучше для сравнения и слияния, чем реальный файл базы данных. Аналогично, текстовые форматы, такие как Markdown и AsciiDoc, будут работать лучше, чем двоичные форматы, такие как Microsoft Word и PDF.

Аналогично, обычно не рекомендуется хранить двоичные зависимости (например, общие файлы libraries или JAR) или продукты сборки в репозитории. Зависимости и продукты сборки лучше всего хранить на сервере артефактов или пакетов, где в репозитории хранятся только ссылки, URL-адреса и хэши.

Мы также рекомендуем настроить файл gitattributes[5] , чтобы явно указать, какие файлы являются текстовыми, а какие — двоичными. Если вы хотите, чтобы Git угадал, вы можете установить атрибут text=auto . Например, в некоторых проектах может быть уместно следующее:

# By default, guess.
*        text=auto
# Mark all C files as text.
*.c        text
# Mark all JPEG files as binary.
*.jpg        binary

Эти настройки помогают инструментам выбирать правильный формат вывода, например исправлений, и приводят к извлечению файлов с соответствующим окончанием строки для платформы.
