git-rebase

Name

git-rebase — повторно применить коммиты поверх другой базовой подсказки

Synopsis

git rebase [-i | --interactive] [<options>] [--exec <cmd>]
        [--onto <newbase> | --keep-base] [<upstream> [<branch>]]
git rebase [-i | --interactive] [<options>] [--exec <cmd>] [--onto <newbase>]
        --root [<branch>]
git rebase (--continue | --skip | --abort | --quit | --edit-todo | --show-current-patch)

Description

Если указан <branch> , git rebase автоматически выполнит git switch <branch> , прежде чем делать что-либо еще. В противном случае он остается в текущей ветке.

Если <upstream> не указан, будет использоваться восходящий канал, настроенный в опциях branch.<name>.remote и branch.<name>.merge (подробности см. в git-config[1] ), и предполагается вариант --fork-point . Если вы в данный момент не находитесь ни в одной ветке или если текущая ветка не имеет настроенного восходящего потока, перебазирование будет прервано.

Все изменения, внесенные коммитами в текущей ветке, но не в <upstream> , сохраняются во временной области. Это тот же набор коммитов, который будет показан git log <upstream>..HEAD ; или git log 'fork_point'..HEAD , если --fork-point активен (см. описание --fork-point ниже); или git log HEAD , если указана опция --root .

Текущая ветвь сбрасывается на <upstream> или <newbase> , если была указана опция --onto . Это имеет тот же эффект, что и git reset --hard <upstream> (или <newbase> ). Перед сбросом ORIG_HEAD настроен на кончик ветки.

Note
 ORIG_HEAD не гарантирует, что он по-прежнему будет указывать на предыдущую вершину ветки в конце перебазирования, если во время перебазирования используются другие команды, записывающие эту псевдо-ссылку (например, git reset ). Однако доступ к предыдущей подсказке ветки можно получить, используя журнал текущей ветки (т. е. @{1} , см. gitrevisions[7] ).

Коммиты, которые ранее были сохранены во временной области, затем по порядку повторно применяются к текущей ветке. Обратите внимание, что любые коммиты в HEAD , которые вносят те же текстовые изменения, что и коммиты в HEAD..<upstream> , опускаются (т. е. патч, уже принятый в исходном коде с другим сообщением о коммите или отметкой времени, будет пропущен).

Вполне возможно, что ошибка слияния помешает этому процессу стать полностью автоматическим. Вам придется устранить любую такую ​​ошибку слияния и запустить git rebase --continue . Другой вариант — обойти фиксацию, вызвавшую сбой слияния с git rebase --skip . Чтобы извлечь исходный <branch> и удалить рабочие файлы .git/rebase-apply , используйте вместо этого команду git rebase --abort .

Предположим, существует следующая история и текущая ветка — "topic" :

          A---B---C topic
         /
    D---E---F---G master

С этого момента результат любой из следующих команд:

git rebase master
git rebase master topic

would be:

                  A'--B'--C' topic
                 /
    D---E---F---G master

ПРИМЕЧАНИЕ. Последняя форма — это сокращение от git checkout topic , за которым следует git rebase master . После завершения перебазирования topic останется извлеченной веткой.

Если восходящая ветвь уже содержит внесенные вами изменения (например, потому что вы отправили по почте патч, который был применен к вышестоящей ветке), то этот коммит будет пропущен и будут выданы предупреждения (если используется серверная часть merge ). Например, запуск git rebase master в следующей истории (в которой A' и A вносят один и тот же набор изменений, но имеют разную информацию о коммиттере):

          A---B---C topic
         /
    D---E---A'---F master

приведет к:

                   B'---C' topic
                  /
    D---E---A'---F master

Вот как можно пересадить ветку темы, основанную на одной ветке, в другую, чтобы притвориться, что вы отделили ветку темы от последней ветки, используя rebase --onto .

Сначала предположим, что ваш topic основан на ветке next . Например, функция, разработанная в topic , зависит от некоторой функциональности, присутствующей в next .

    o---o---o---o---o  master
         \
          o---o---o---o---o  next
                           \
                            o---o---o  topic

Мы хотим сделать topic ответвлением от ветки master ; например, потому что функциональность, от которой зависит topic , была объединена в более стабильную ветку master . Мы хотим, чтобы наше дерево выглядело так:

    o---о---о---о---о хозяин
        |            \
        |             o'--o'--o' тема
         \
          o---о---о---о---о следующий

Мы можем получить это, используя следующую команду:

git rebase --onto master next topic

Другой пример опции --onto — перебазирование части ветки. Если у нас следующая ситуация:

                            H---I---J topicB
                           /
                  E---F---G  topicA
                 /
    A---B---C---D  master

затем команда

git rebase --onto master topicA topicB

приведет к:

                 H'--I'--J' темаБ
                /
                | E---F---G темаА
                |/
    A---B---C---D мастер

Это полезно, когда тема B не зависит от темы A.

Ряд коммитов также можно удалить с помощью rebase. Если у нас следующая ситуация:

    E---F---G---H---I---J  topicA

затем команда

git rebase --onto topicA~5 topicA~3 topicA

приведет к удалению коммитов F и G:

    E---H'---I'---J'  topicA

Это полезно, если F и G были каким-то образом ошибочны или не должны быть частью темы A. Обратите внимание, что аргументом --onto и параметром <upstream> может быть любая допустимая фиксация.

В случае конфликта git rebase остановится на первой проблемной фиксации и оставит маркеры конфликта в дереве. Вы можете использовать git diff , чтобы найти маркеры (<<<<<<) и внести изменения для разрешения конфликта. Для каждого файла, который вы редактируете, вам необходимо сообщить Git , что конфликт разрешен. Обычно это делается с помощью

git add <filename>

После разрешения конфликта вручную и обновления индекса до желаемого разрешения вы можете продолжить процесс перебазирования с помощью

git rebase --continue

Альтернативно вы можете отменить git rebase с помощью

git rebase --abort

Mode options

Параметры в этом разделе нельзя использовать ни с каким другим параметром, в том числе друг с другом:

--continue

Перезапустите процесс перебазирования после разрешения конфликта слияния.

--skip

Перезапустите процесс перебазирования, пропустив текущий патч.

--abort

Прервите операцию перебазирования и сбросьте HEAD в исходную ветку. Если <branch> был указан при запуске операции перебазирования, то HEAD будет сброшен до <branch> . В противном случае HEAD будет сброшен туда, где он был при запуске операции перебазирования.

--quit

Прервать операцию перебазирования, но HEAD не возвращается к исходной ветке. В результате индекс и рабочее дерево также остаются неизменными. Если запись временного тайника была создана с помощью --autostash , она будет сохранена в списке тайника.

--edit-todo

Отредактируйте список дел во время интерактивного перебазирования.

--show-current-patch

Показывать текущий патч при интерактивном перебазировании или когда перебазирование остановлено из-за конфликтов. Это эквивалент git show REBASE_HEAD .

Options

--onto <newbase>

Начальная точка для создания новых коммитов. Если опция --onto не указана, начальной точкой является <upstream> . Это может быть любой допустимый коммит, а не просто существующее имя ветки.

В частном случае вы можете использовать "A...B" в качестве ярлыка для базы слияния A и B, если существует ровно одна база слияния. Вы можете исключить не более одного из A и B, и в этом случае по умолчанию используется HEAD.

--keep-base

Установите отправную точку для создания новых коммитов для базы слияния <upstream> и <branch> . Запуск git rebase --keep-base <upstream> <branch> эквивалентен запуску git rebase --reapply-cherry-picks --no-fork-point --onto <upstream>...<branch> <upstream> <branch> .

Эта опция полезна в случае, когда функция разрабатывается поверх вышестоящей ветки. Пока эта функция находится в разработке, восходящая ветвь может развиваться, и, возможно, продолжать перебазирование поверх восходящей ветки может быть не лучшей идеей, а сохранять базовую фиксацию как есть. Поскольку базовый коммит не изменился, эта опция подразумевает --reapply-cherry-picks , чтобы избежать потери коммитов.

Хотя и этот параметр, и --fork-point находят базу слияния между <upstream> и <branch> , этот параметр использует базу слияния в качестве starting point , на которой будут созданы новые фиксации, тогда как --fork-point использует базу слияния для определения set of commits , который будет перебазирован.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

<upstream>

Восходящая ветвь для сравнения. Это может быть любой действительный коммит, а не просто существующее имя ветки. По умолчанию используется настроенный восходящий поток для текущей ветки.

<branch>

Рабочая ветка; по умолчанию HEAD .

--apply

Используйте стратегии применения для перебазирования (внутренний вызов git-am ). В будущем эта опция может стать неактивной, как только серверная часть слияния будет обрабатывать все, что делает применяемая.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--empty=(drop|keep|ask)

Как обрабатывать коммиты, которые не являются пустыми при запуске и не являются чистыми выборками любого восходящего коммита, но которые становятся пустыми после перебазирования (поскольку они содержат подмножество уже вышестоящих изменений). При использовании drop (по умолчанию) коммиты, которые становятся пустыми, удаляются. При использовании Keep такие коммиты сохраняются. С помощью Ask (подразумевается --interactive ) перебазирование остановится, когда будет применена пустая фиксация, что позволит вам выбрать, удалить ли ее, редактировать файлы дальше или просто зафиксировать пустые изменения. Другие параметры, такие как --exec , будут использовать значение удаления по умолчанию, если не указано явно -i / --interactive .

Обратите внимание, что коммиты, которые начинаются пустыми, сохраняются (если не указан --no-keep-empty ), а коммиты, которые являются чистыми выборками (как определено git log --cherry-mark ... ), обнаруживаются и удаляются в качестве предварительного шага (если не переданы --reapply-cherry-picks или --keep-base ).

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--no-keep-empty
--keep-empty

Не оставляйте в результате коммиты, которые начинаются пустыми до перебазирования (т. е. которые ничего не меняют по сравнению с родительским объектом). По умолчанию коммиты, которые начинаются пустыми, сохраняются, поскольку для создания таких коммитов требуется передать флаг переопределения --allow-empty в git commit , что означает, что пользователь намеренно создает такой коммит и поэтому хочет его сохранить.

Использование этого флага, вероятно, будет редким, поскольку вы можете избавиться от коммитов, которые начинаются пустыми, просто запустив интерактивную перебазировку и удалив строки, соответствующие коммитам, которые вам не нужны. Этот флаг существует как удобный ярлык, например, в случаях, когда внешние инструменты генерируют много пустых коммитов, и вы хотите, чтобы все они были удалены.

Для коммитов, которые не начинаются пустыми, но становятся пустыми после перебазирования, см. флаг --empty .

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--reapply-cherry-picks
--no-reapply-cherry-picks

Повторно примените все чистые выборки любого восходящего коммита вместо их упреждающего удаления. (Если после перебазирования эти коммиты становятся пустыми, поскольку они содержат подмножество уже вышестоящих изменений, поведение по отношению к ним контролируется флагом --empty .)

При отсутствии --keep-base (или если указан --no-reapply-cherry-picks ) эти коммиты будут автоматически удалены. Поскольку для этого необходимо прочитать все восходящие коммиты, это может быть дорогостоящим в репозиториях с большим количеством восходящих коммитов, которые необходимо прочитать. При использовании серверной части merge предупреждения будут выдаваться для каждого отброшенного коммита (если не указан --quiet ). Рекомендации также будут выдаваться, если для advice.skippedCherryPicks не установлено значение false (см. git-config[1] ).

--reapply-cherry-picks позволяет выполнить rebase, чтобы отказаться от чтения всех восходящих коммитов, что потенциально повышает производительность.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--allow-empty-message

Нет-оп. Перебазирование коммитов с пустым сообщением раньше приводило к сбою, и эта опция переопределяет это поведение, позволяя перебазировать коммиты с пустыми сообщениями. Теперь коммиты с пустым сообщением не приводят к остановке перебазирования.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

-m
--merge

Использование стратегий слияния для перебазирования (по умолчанию).

Обратите внимание, что слияние с перебазированием работает путем воспроизведения каждого коммита из рабочей ветки поверх ветки <upstream> . По этой причине, когда происходит конфликт слияния, сторона, обозначенная как ours , является перебазированной серией, начиная с <upstream> , а theirs — рабочей ветвью. Другими словами, стороны меняются местами.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

-s <strategy>
--strategy=<strategy>

Используйте данную стратегию слияния вместо ort по умолчанию. Это подразумевает --merge .

Поскольку git rebase воспроизводит каждый коммит из рабочей ветки поверх ветки <upstream> с использованием заданной стратегии, использование стратегии ours просто удаляет все патчи из <branch> , что не имеет особого смысла.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

-X <strategy-option>
--strategy-option=<strategy-option>

Передайте <strategy-option> в стратегию слияния. Это подразумевает --merge и, если стратегия не указана, -s ort . Обратите внимание на перестановку ours и theirs , как отмечено выше для опции -m .

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--rerere-autoupdate
--no-rerere-autoupdate

После того, как механизм rerere повторно использует записанное разрешение текущего конфликта для обновления файлов в рабочем дереве, разрешите ему также обновить индекс с результатом разрешения. --no-rerere-autoupdate — хороший способ перепроверить то, что сделал rerere , и выявить возможные ошибки слияния, прежде чем заносить результат в индекс с помощью отдельного git add .

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

GPG-знак фиксируется. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

-q
--quiet

Будь спокоен. Подразумевается --no-stat .

-v
--verbose

Будьте многословны. Подразумевается --stat .

--stat

Покажите разницу того, что изменилось в исходном коде с момента последней перебазировки. diffstat также контролируется параметром конфигурации rebase.stat.

-n
--no-stat

Не показывайте diffstat как часть процесса перебазирования.

--no-verify

Эта опция обходит перехватчик перед перебазированием. См. также githooks[5] .

--verify

Разрешает запуск перехватчика перед перебазированием (по умолчанию). Эту опцию можно использовать для переопределения --no-verify . См. также githooks[5] .

-C<n>

Убедитесь, что по крайней мере строки <n> окружающего контекста совпадают до и после каждого изменения. Когда существует меньше строк окружающего контекста, все они должны совпадать. По умолчанию ни один контекст не игнорируется. Подразумевается --apply .

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--no-ff
--force-rebase
-f

Индивидуально воспроизводите все перебазированные коммиты вместо быстрой перемотки вперед по неизмененным. Это гарантирует, что вся история перебазированной ветки состоит из новых коммитов.

Это может оказаться полезным после отмены слияния ветки темы, поскольку эта опция воссоздает ветку темы со свежими коммитами, поэтому ее можно успешно повторно объединить без необходимости использования "revert the reversion" (подробности см. в revert-a-faulty-merge How-To ).

--fork-point
--no-fork-point

Используйте reflog, чтобы найти лучшего общего предка между <upstream> и <branch> при расчете того, какие коммиты были введены <branch> .

Когда --fork-point активен, fork_point будет использоваться вместо <upstream> для расчета набора коммитов для перебазирования, где fork_point — результат команды git merge-base --fork-point <upstream> <branch> (см. git-merge-base[1] ). Если fork_point окажется пустым, <upstream> будет использоваться в качестве резервного варианта.

Если в командной строке заданы <upstream> или --keep-base , то значением по умолчанию является --no-fork-point , в противном случае значением по умолчанию является --fork-point . См. также rebase.forkpoint в git-config[1] .

Если ваша ветка была основана на <upstream> , но <upstream> была перемотана и ваша ветка содержит коммиты, которые были удалены, эту опцию можно использовать с --keep-base , чтобы удалить эти коммиты из вашей ветки.

См. также НЕСОВМЕСТИМЫЕ ОПЦИИ ниже.

--ignore-whitespace

Игнорируйте различия в пробелах при попытке согласовать различия. В настоящее время каждый бэкэнд реализует примерно такое поведение:

apply backend

При применении патча игнорируйте изменения пробелов в строках контекста. К сожалению, это означает, что если строки "old" , заменяемые патчем, отличаются от существующего файла только пробелами, вместо успешного применения патча вы получите конфликт слияния.

merge backend

Считайте строки, в которых изменены только пробелы, неизмененными при слиянии. К сожалению, это означает, что любые фрагменты патчей, предназначенные для изменения пробелов и ничего другого, будут удалены, даже если на другой стороне не было никаких конфликтующих изменений.

Этот флаг передается программе git apply (см. git-apply[1] ), которая применяет исправление. Подразумевается --apply .

-r
--rebase-merges[=(rebase-cousins|no-rebase-cousins)]
--no-rebase-merges

По умолчанию при перебазировании коммиты слияния просто удаляются из списка задач и помещаются в одну линейную ветвь. При использовании --rebase-merges перебазирование вместо этого попытается сохранить структуру ветвления внутри коммитов, которые должны быть перебазированы, путем воссоздания коммитов слияния. Любые разрешенные конфликты слияния или ручные поправки в этих коммитах слияния придется разрешать/повторно применять вручную. --no-rebase-merges можно использовать для отмены как опции конфигурации rebase.rebaseMerges , так и предыдущей версии --rebase-merges .

При ребазировании слияний есть два режима: rebase-cousins и no-rebase-cousins . Если режим не указан, по умолчанию используется no-rebase-cousins . В режиме no-rebase-cousins коммиты, у которых нет <upstream> в качестве прямого предка, сохранят свою исходную точку ветвления, т. е. коммиты, которые были бы исключены опцией --ancestry-path git-log[1] , сохранят свое исходное происхождение по умолчанию. В режиме rebase-cousins такие фиксации вместо этого перебазируются на <upstream> (или <onto> , если указано).

В настоящее время воссоздать коммиты слияния можно только с помощью стратегии слияния ort ; различные стратегии слияния можно использовать только с помощью явных команд exec git merge -s <strategy> [...] .

Behavioral differences

git rebase имеет два основных серверных модуля: apply и merge . (Раньше серверная часть apply была известна как серверная часть am , но это название привело к путанице, поскольку оно выглядело как глагол, а не существительное. Кроме того, серверная часть merge раньше была известна как интерактивная серверная часть, но теперь она используется для а также неинтерактивные случаи. Оба были переименованы на основе функциональности нижнего уровня, лежащей в основе каждого.) Есть некоторые тонкие различия в поведении этих двух бэкэндов:

Empty commits

К сожалению, серверная часть apply намеренно удаляет пустые коммиты, то есть коммиты, которые начались пустыми, хотя на практике это случается редко. Он также удаляет коммиты, которые становятся пустыми, и не имеет возможности контролировать такое поведение.

Серверная часть merge по умолчанию сохраняет намеренно пустые коммиты (хотя в -i они помечаются как пустые в редакторе списка задач или их можно автоматически удалить с помощью --no-keep-empty ).

Подобно бэкэнду apply, по умолчанию бэкенд слияния удаляет коммиты, которые становятся пустыми, если не указан -i / --interactive (в этом случае он останавливается и спрашивает пользователя, что делать). Серверная часть слияния также имеет опцию --empty=(drop|keep|ask) для изменения поведения обработки коммитов, которые становятся пустыми.

Обнаружение переименования каталога

Из-за отсутствия точной информации о дереве (возникающей из-за создания поддельных предков с ограниченной информацией, доступной в исправлениях), обнаружение переименования каталогов отключено в серверной части apply . Отключенное обнаружение переименования каталога означает, что если одна сторона истории переименовывает каталог, а другая добавляет новые файлы в старый каталог, то новые файлы останутся в старом каталоге без какого-либо предупреждения во время перебазирования, которое вы, возможно, захотите переместите эти файлы в новый каталог.

Обнаружение переименования каталога работает с серверной частью merge и предупреждает вас в таких случаях.

Context

Серверная часть apply работает путем создания последовательности исправлений (путем внутреннего вызова format-patch ), а затем последовательного применения исправлений (внутреннего вызова am ). Патчи состоят из нескольких фрагментов, каждый из которых имеет номера строк, контекстную область и фактические изменения. Номера строк должны быть взяты с некоторой размытостью, поскольку другая сторона, скорее всего, вставит или удалит строки ранее в файле. Контекстная область предназначена для того, чтобы помочь найти, как настроить номера строк, чтобы применить изменения к нужным строкам. Однако если несколько областей кода имеют одинаковые окружающие строки контекста, можно выбрать не ту. Существуют реальные случаи, когда это приводило к неправильному повторному применению коммитов без каких-либо сообщений о конфликтах. Установка большего значения diff.context может предотвратить проблемы такого типа, но увеличивает вероятность ложных конфликтов (поскольку для применения потребуется больше строк соответствующего контекста).

Серверная часть merge работает с полной копией каждого соответствующего файла, изолируя его от подобных проблем.

Маркировка маркеров конфликтов

При возникновении конфликтов контента механизм слияния пытается пометить маркеры конфликта каждой стороны коммитами, откуда поступил контент. Поскольку серверная часть apply удаляет исходную информацию о перебазированных коммитах и ​​их родителях (и вместо этого генерирует новые поддельные коммиты на основе ограниченной информации в сгенерированных патчах), эти коммиты невозможно идентифицировать; вместо этого он должен вернуться к сводке коммитов. Кроме того, когда для merge.conflictStyle установлено значение diff3 или zdiff3 , серверная часть apply будет использовать "constructed merge base" для маркировки содержимого из базы слияния и, таким образом, не предоставлять никакой информации о фиксации базы слияния.

Серверная часть merge работает с полными коммитами с обеих сторон истории и поэтому не имеет таких ограничений.

Hooks

Серверная часть apply традиционно не называется перехватчиком после фиксации, в отличие от серверной части merge . Оба вызвали перехватчик после оформления заказа, хотя серверная часть merge подавила его вывод. Кроме того, оба бэкэнда вызывают перехватчик post-checkout только с начальной точкой фиксации перебазирования, а не с промежуточной фиксацией или окончательной фиксацией. В каждом случае вызов этих перехватчиков был случайностью реализации, а не намерением (оба бэкэнда изначально были реализованы как сценарии оболочки и вызывали другие команды, такие как git checkout или git commit , которые вызывали перехватчики). Оба бэкэнда должны вести себя одинаково, хотя не совсем ясно, какое из них правильное. Скорее всего, в будущем мы прекратим вызов любого из этих перехватчиков.

Interruptability

Серверная часть apply имеет проблемы с безопасностью из-за несвоевременного прерывания; если пользователь нажимает Ctrl-C в неподходящее время, чтобы попытаться прервать перебазирование, перебазирование может перейти в состояние, в котором его нельзя прервать с помощью последующего git rebase --abort . Серверная часть merge , похоже, не страдает этим недостатком. (Подробнее см. в https://lore.kernel.org/git/[email protected]/ .)

Commit Rewording

Если во время перебазирования возникает конфликт, перебазирование останавливается и просит пользователя разрешить его. Поскольку пользователю может потребоваться внести заметные изменения при разрешении конфликтов, после того, как конфликты разрешены и пользователь запустил git rebase --continue , при перебазировании следует открыть редактор и попросить пользователя обновить сообщение о фиксации. Это делает серверная часть merge , а серверная часть apply слепо применяет исходное сообщение о фиксации.

Miscellaneous differences

Есть еще несколько поведенческих различий, которые большинство людей, вероятно, посчитали бы несущественными, но которые упомянуты для полноты картины:

  • Reflog: два бэкэнда будут использовать разные формулировки при описании изменений, внесенных в журнал ссылок, хотя оба будут использовать слово "rebase" .

  • Сообщения о ходе выполнения, информационные сообщения и сообщения об ошибках. Два серверных модуля предоставляют немного разные сообщения о ходе выполнения и информационные сообщения. Кроме того, серверная часть применения записывает сообщения об ошибках (например, "Your files would be overwritten…​" ) в стандартный вывод, а серверная часть слияния записывает их в стандартный вывод.

  • Каталоги состояний: два серверных сервера хранят свое состояние в разных каталогах под .git/ .

Merge strategies

Механизм слияния (команды git merge и git pull ) позволяет выбирать серверную часть merge strategies с опцией -s . Некоторые стратегии также могут иметь свои собственные параметры, которые можно передать, передав аргументы -X<option> в git merge и/или git pull .

ort

Это стратегия слияния по умолчанию при извлечении или объединении одной ветки. Эта стратегия может разрешить только две головы, используя алгоритм трехстороннего слияния. Если существует более одного общего предка, который можно использовать для трехстороннего слияния, создается объединенное дерево общих предков и используется в качестве эталонного дерева для трехстороннего слияния. Сообщается, что это приводит к меньшему количеству конфликтов слияния, не вызывая неправильных слияний, согласно тестам, выполненным на реальных коммитах слияния, взятых из истории разработки ядра Linux 2.6. Кроме того, эта стратегия может обнаруживать и обрабатывать слияния, связанные с переименованиями. Он не использует обнаруженные копии. Название этого алгоритма является аббревиатурой ( "Ostensibly Recursive’s Twin" ) и произошло от того, что он был написан в качестве замены предыдущего алгоритма по умолчанию recursive .

Стратегия ort может принимать следующие варианты:

ours

Эта опция заставляет конфликтующие фрагменты автоматически разрешаться, отдавая предпочтение версии our . Изменения из другого дерева, которые не конфликтуют с нашей стороной, отражаются в результате слияния. Для бинарного файла все содержимое берется с нашей стороны.

Это не следует путать со стратегией слияния ours , которая вообще не рассматривает содержимое другого дерева. Он отбрасывает все, что сделало другое дерево, объявляя, что история our содержит все, что в ней произошло.

theirs

Это противоположность ours ; обратите внимание, что, в отличие от ours , не существует стратегии слияния theirs , с которой можно было бы спутать этот вариант слияния.

ignore-space-change
ignore-all-space
ignore-space-at-eol
ignore-cr-at-eol

Считает строки с указанным типом изменения пробелов неизмененными для трехстороннего слияния. Изменения пробелов, смешанные с другими изменениями в строке, не игнорируются. См. также git-diff[1] , -b , -w , --ignore-space-at-eol и --ignore-cr-at-eol .

  • Если версия their вводит в строку только изменения пробелов, используется версия our ;

  • Если версия our содержит изменения в пробелах, а версия their включает существенные изменения, используется версия their ;

  • В противном случае слияние происходит обычным способом.

renormalize

При этом выполняется виртуальное извлечение и возврат всех трех этапов файла при разрешении трехстороннего слияния. Эта опция предназначена для использования при объединении ветвей с различными чистыми фильтрами или правилами нормализации конца строки. Подробности см. в "Merging branches with differing checkin/checkout attributes" и gitattributes[5] .

no-renormalize

Отключает опцию renormalize . Это переопределяет переменную конфигурации merge.renormalize .

find-renames[=<n>]

Включите обнаружение переименования, при необходимости установив порог сходства. Это значение по умолчанию. Это переопределяет переменную конфигурации merge.renames . См. также git-diff[1] --find-renames .

rename-threshold=<n>

Устаревший синоним find-renames=<n> .

subtree[=<path>]

Этот вариант представляет собой более продвинутую форму стратегии subtree , где стратегия предполагает, как два дерева должны быть сдвинуты, чтобы они совпадали друг с другом при слиянии. Вместо этого к указанному пути добавляется префикс (или удаляется с самого начала), чтобы форма двух деревьев совпадала.

recursive

Это может разрешить только две головы, используя алгоритм трехстороннего слияния. Если существует более одного общего предка, который можно использовать для трехстороннего слияния, создается объединенное дерево общих предков и используется в качестве эталонного дерева для трехстороннего слияния. Сообщается, что это приводит к меньшему количеству конфликтов слияния, не вызывая неправильных слияний, согласно тестам, выполненным на реальных коммитах слияния, взятых из истории разработки ядра Linux 2.6. Кроме того, это может обнаруживать и обрабатывать слияния, связанные с переименованиями. Он не использует обнаруженные копии. Это была стратегия по умолчанию для разрешения двух головок от Git v0.99.9k до v2.33.0.

Стратегия recursive использует те же параметры, что и ort . Однако есть три дополнительных параметра, которые ort игнорирует (не описаны выше), которые потенциально полезны для стратегии recursive :

patience

Устаревший синоним diff-algorithm=patience .

diff-algorithm=[patience|minimal|histogram|myers]

Используйте другой алгоритм сравнения при слиянии, что поможет избежать ошибок слияния, возникающих из-за неважных совпадающих строк (например, фигурных скобок из разных функций). См. также git-diff[1] --diff-algorithm . Обратите внимание, что ort специально использует diff-algorithm=histogram , а recursive по умолчанию использует настройку конфигурации diff.algorithm .

no-renames

Отключите обнаружение переименования. Это переопределяет переменную конфигурации merge.renames . См. также git-diff[1] --no-renames .

resolve

Это может разрешить только две головы (т. е. текущую ветвь и другую ветвь, из которой вы извлекли) с использованием алгоритма трехстороннего слияния. Он пытается тщательно обнаружить неоднозначности перекрестного слияния. Он не обрабатывает переименования.

octopus

Это разрешает случаи с более чем двумя головками, но отказывается выполнять сложное слияние, требующее разрешения вручную. В первую очередь он предназначен для объединения заголовков тематических ветвей. Это стратегия слияния по умолчанию при извлечении или объединении более чем одной ветки.

ours

При этом разрешается любое количество голов, но результирующее дерево слияния всегда является деревом текущей головы ветки, эффективно игнорируя все изменения из всех других ветвей. Он предназначен для замены старой истории развития побочных ветвей. Обратите внимание, что это отличается от опции -Xours стратегии слияния recursive .

subtree

Это модифицированная стратегия ort . При слиянии деревьев A и B, если B соответствует поддереву A, B сначала настраивается так, чтобы оно соответствовало древовидной структуре A, вместо чтения деревьев на одном уровне. Эта корректировка также вносится в дерево общего предка.

В стратегиях, использующих трехстороннее слияние (включая стандартную ort ), если изменение внесено в обе ветви, но позже отменено в одной из ветвей, это изменение будет присутствовать в объединенном результате; некоторых людей такое поведение сбивает с толку. Это происходит потому, что при выполнении слияния учитываются только головы и база слияния, а не отдельные фиксации. Поэтому алгоритм слияния считает отмененное изменение вообще отсутствием изменений и вместо этого заменяет измененную версию.

Notes

Вы должны понимать последствия использования git rebase в общем репозитории. См. также ВОССТАНОВЛЕНИЕ ИЗ UpSTREAM REBASE ниже.

При запуске перебазирования сначала выполняется перехват pre-rebase , если он существует. Вы можете использовать этот крючок для проверки работоспособности и отклонения перебазирования, если оно не подходит. Пример см. в шаблоне сценария перехвата pre-rebase .

После завершения текущей веткой станет <branch> .

Interactive mode

Интерактивное перебазирование означает, что у вас есть возможность редактировать перебазированные коммиты. Вы можете изменить порядок коммитов и удалить их (удаляя плохие или иным образом нежелательные исправления).

Интерактивный режим предназначен для такого типа рабочего процесса:

  1. есть замечательная идея

  2. взломать код

  3. подготовить серию к отправке

  4. submit

где пункт 2. состоит из нескольких экземпляров

а) регулярное употребление

  1. закончить что-то, достойное обязательства

  2. commit

б) самостоятельный ремонт

  1. осознать, что что-то не работает

  2. fix that

  3. commit it

Иногда то, что исправлено в б.2. не может быть изменен в не совсем идеальный коммит, который он исправляет, потому что этот коммит глубоко спрятан в серии патчей. Именно для этого и нужна интерактивная перебазировка: используйте ее после большого количества "a" и "b" , переставляя и редактируя коммиты, а также объединяя несколько коммитов в один.

Начните с последнего коммита, который вы хотите сохранить как есть:

git rebase -i <after-this-commit>

Редактор будет запущен со всеми коммитами в вашей текущей ветке (игнорируя коммиты слияния), которые следуют после данного коммита. Вы можете изменить порядок коммитов в этом списке по своему усмотрению или удалить их. Список выглядит примерно так:

pick deadbee The oneline of this commit
pick fa1afe1 The oneline of the next commit
...

Однострочные описания предназначены исключительно для вашего удовольствия; git rebase будет просматривать не их, а имена коммитов (в этом примере "deadbee" и "fa1afe1" ), поэтому не удаляйте и не редактируйте имена.

Заменив команду "pick" на команду "edit" , вы можете указать git rebase остановиться после применения этой фиксации, чтобы вы могли редактировать файлы и/или сообщение о фиксации, изменить фиксацию и продолжить перебазирование.

Чтобы прервать перебазирование (так же, как это делает команда "edit" , но без предварительного выбора каких-либо коммитов), используйте команду "break" .

Если вы просто хотите отредактировать сообщение о фиксации, замените команду "pick" на команду "reword" .

Чтобы отменить фиксацию, замените команду "pick" на "drop" или просто удалите соответствующую строку.

Если вы хотите объединить два или более коммита в один, замените команду "pick" для второго и последующих коммитов на "squash" или "fixup" . Если у коммитов были разные авторы, свернутый коммит будет приписан автору первого коммита. Предлагаемое сообщение о фиксации для свернутой фиксации представляет собой объединение сообщений первой фиксации с сообщениями, указанными командами "squash" , без учета сообщений о фиксации, идентифицированных командами "fixup" , если не используется "fixup -c" . В этом случае предлагаемое сообщение о фиксации является только сообщением фиксации "fixup -c" , и открывается редактор, позволяющий редактировать сообщение. Содержимое (исправление) коммита "fixup -c" по-прежнему включено в свернутый коммит. Если существует более одного коммита "fixup -c" , используется сообщение из последнего. Вы также можете использовать "fixup -C" , чтобы получить то же поведение, что и "fixup -c" , но без открытия редактора.

git rebase остановится, когда "pick" будет заменен на "edit" или когда команда не будет выполнена из-за ошибок слияния. Когда вы закончите редактирование и/или разрешение конфликтов, вы можете продолжить работу с git rebase --continue .

Например, если вы хотите изменить порядок последних 5 коммитов, чтобы то, что было HEAD~4 , стало новым HEAD . Для этого вы должны вызвать git rebase следующим образом:

$ git rebase -i HEAD~5

И переместите первый патч в конец списка.

Возможно, вы захотите воссоздать коммиты слияния, например, если у вас есть такая история:

           X
            \
         A---M---B
        /
---o---O---P---Q

Предположим, вы хотите перебазировать боковую ветвь, начиная с "A" , на "Q" . Убедитесь, что текущий HEAD — "B" , и позвоните

$ git rebase -i -r --onto QO

Изменение порядка и редактирование коммитов обычно приводит к созданию непроверенных промежуточных шагов. Вы можете проверить, что редактирование истории ничего не нарушило, запустив тест или, по крайней мере, перекомпилировав в промежуточных точках истории с помощью команды "exec" (ярлык "x" ). Вы можете сделать это, создав список задач, подобный этому:

pick deadbee Implement feature XXX
fixup f1a5c00 Fix to feature XXX
exec make
pick c0ffeee The oneline of the next commit
edit deadbab The oneline of the commit after
exec cd subdir; make test
...

Интерактивное перебазирование остановится в случае сбоя команды (т. е. выхода со статусом, отличным от 0), чтобы дать вам возможность устранить проблему. Вы можете продолжить с git rebase --continue .

Команда "exec" запускает команду в оболочке (той, которая указана в $SHELL или оболочке по умолчанию, если $SHELL не установлен), поэтому вы можете использовать функции оболочки (например, "cd" , ">" , ";" …​). Команда запускается из корня рабочего дерева.

$ git rebase -i --exec "make test"

Эта команда позволяет вам проверить возможность компиляции промежуточных коммитов. Список дел становится таким:

pick 5928aea one
exec make test
pick 04d0fda two
exec make test
pick ba46169 three
exec make test
pick f4593f9 four
exec make test

Splitting commits

В интерактивном режиме вы можете пометить фиксации с помощью действия "edit" . Однако это не обязательно означает, что git rebase ожидает, что результатом этого редактирования будет ровно одна фиксация. Действительно, вы можете отменить коммит или добавить другие коммиты. Это можно использовать для разделения коммита на две части:

  • Запустите интерактивное перебазирование с помощью git rebase -i <commit>^ , где <commit> — это коммит, который вы хотите разделить. Фактически подойдет любой диапазон коммитов, если он содержит этот коммит.

  • Отметьте коммит, который хотите разделить, действием "edit" .

  • Когда дело доходит до редактирования этого коммита, выполните git reset HEAD^ . В результате HEAD перематывается на единицу, и индекс следует этому примеру. Однако рабочее дерево остается прежним.

  • Теперь добавьте изменения в индекс, который вы хотите иметь в первом коммите. Для этого вы можете использовать git add (возможно, в интерактивном режиме) или git gui (или оба).

  • Зафиксируйте текущий индекс с помощью любого сообщения фиксации, которое сейчас подходит.

  • Повторяйте последние два шага, пока ваше рабочее дерево не станет чистым.

  • Продолжите перебазирование с помощью git rebase --continue .

Если вы не совсем уверены в том, что промежуточные версии согласованы (они компилируются, передают набор тестов и т. д.), вам следует использовать git stash для хранения еще не зафиксированных изменений после каждого коммита, тестирования и внесения исправлений в коммит. необходимый.

Восстановление после перебазирования восходящего потока

Перебазирование (или любая другая форма переписывания) ветки, на которой основывались работы других, — плохая идея: любой, кто находится ниже по течению, вынужден вручную исправлять свою историю. В этом разделе объясняется, как выполнить исправление с точки зрения нижестоящей стороны. Реальное решение, однако, заключалось бы в том, чтобы вообще избежать перебазирования восходящего потока.

Для иллюстрации предположим, что вы находитесь в ситуации, когда кто-то разрабатывает ветку subsystem , а вы работаете над topic , который зависит от этого subsystem . В итоге у вас может получиться следующая история:

    o---o---o---o---o---o---o---o  master
         \
          o---о---о---о---о подсистема
                           \
                            *---*---*  topic

Если subsystem перебазируется на master , происходит следующее:

    o---o---o---o---o---o---o---o  master
         \                         \
          o---о---о---о---о о'--о'--о'--о'--о' подсистема
                           \
                            *---*---*  topic

Если вы продолжите разработку как обычно и в конечном итоге объедините topic с subsystem , коммиты из subsystem останутся дублированными навсегда:

    o---o---o---o---o---o---o---o  master
         \                         \
          o---o---o---o---o o'--o'--o'--o'--o'--M подсистема
                           \                             /
                            *---*---*-..........-*--* тема

Такие дубликаты обычно не одобряются, поскольку они засоряют историю, затрудняя ее отслеживание. Чтобы навести порядок, нужно пересадить коммиты с topic на новый наконечник subsystem , т.е. перебазировать topic . Это становится волновым эффектом: любой, кто находится ниже по течению от topic , тоже вынужден перебазироваться и так далее!

Существует два типа исправлений, которые обсуждаются в следующих подразделах:

Простой случай: изменения буквально такие же.

Это происходит, если перебазирование subsystem было простым перебазированием и не имело конфликтов.

Сложный случай: изменения не одинаковы.

Это происходит, если при перебазировании subsystem возникли конфликты или если --interactive использовался для пропуска, редактирования, сжатия или исправления коммитов; или если восходящий канал использовал одну из команд commit --amend , reset или команду полной перезаписи истории, например filter-repo .

Простой случай

Работает только в том случае, если изменения (идентификаторы исправлений, основанные на содержимом различий) на subsystem буквально одинаковы до и после перебазирования subsystem .

В этом случае исправить это легко, поскольку git rebase умеет пропускать изменения, которые уже присутствуют в новом восходящем канале (если не указан --reapply-cherry-picks ). Итак, если вы скажете (при условии, что вы используете topic )

    $ git rebase subsystem

вы получите фиксированную историю

    o---о---о---о---о---о---о---о хозяин
                                 \
                                  o'--o'--o'--o'--o' подсистема
                                                   \
                                                    *---*---* тема

Тяжелый случай

Ситуация усложняется, если изменения subsystem не совсем соответствуют тем, что были до перебазирования.

Note
 Хотя "easy case recovery" иногда кажется успешным даже в сложных случаях, это может иметь непредвиденные последствия. Например, коммит, удаленный с помощью git rebase --interactive , будет воскрешен!

Идея состоит в том, чтобы вручную сообщить git rebase "where the old subsystem ended and your topic began" , то есть, какая была старая база слияния между ними. Вам нужно будет найти способ назвать последний коммит старого subsystem , например:

  • С рефлогом subsystem : после git fetch старая подсказка subsystem находится на subsystem@{1} . Последующие выборки увеличат это число. (См. git-reflog[1] .)

  • Относительно кончика topic : зная, что ваш topic имеет три фиксации, старый совет subsystem должен быть topic~3 .

Затем вы можете пересадить старый subsystem..topic на новый наконечник, сказав (для случая повторной регистрации и при условии, что вы уже используете topic ):

    $ git rebase --onto subsystem subsystem@{1}

Волновой эффект восстановления "hard case" особенно плох: everyone после topic теперь также должен будет выполнять восстановление "hard case" !

Rebasing merges

Команда интерактивного перебазирования изначально была разработана для обработки отдельных серий исправлений. Таким образом, имеет смысл исключить коммиты слияния из списка задач, поскольку разработчик мог объединить текущий на тот момент master во время работы над веткой только для того, чтобы в конечном итоге перебазировать все коммиты на master (пропустив коммиты слияния).

Однако есть законные причины, по которым разработчик может захотеть воссоздать коммиты слияния: сохранить структуру ветвей (или "commit topology" ) при работе с несколькими взаимосвязанными ветвями.

В следующем примере разработчик работает над ветвью темы, которая выполняет рефакторинг способа определения кнопок, и над другой ветвью темы, которая использует этот рефакторинг для реализации кнопки "Report a bug" . Вывод git log --graph --format=%s -5 может выглядеть так:

*   Merge branch 'report-a-bug'
|\
| * Add the feedback button
* | Merge branch 'refactor-button'
|\ \
| |/
| * Use the Button class for all buttons
| * Extract a generic Button class from the DownloadButton one

Разработчик может захотеть перенести эти коммиты на более новую версию master , сохранив при этом топологию ветки, например, когда ожидается, что первая тематическая ветка будет интегрирована в master намного раньше, чем вторая, скажем, для разрешения конфликтов слияния с изменениями в DownloadButton. класс, который вошёл в master .

Эту перезагрузку можно выполнить с помощью опции --rebase-merges . Он сгенерирует список задач, похожий на этот:

label onto

# Branch: refactor-button
reset onto
pick 123456 Extract a generic Button class from the DownloadButton one
pick 654321 Use the Button class for all buttons
label refactor-button

# Branch: report-a-bug
reset refactor-button # Use the Button class for all buttons
pick abcdef Add the feedback button
label report-a-bug

reset onto
merge -C a1b2c3 refactor-button # Merge 'refactor-button'
merge -C 6f5e4d report-a-bug # Merge 'report-a-bug'

В отличие от обычного интерактивного ребаза, помимо pick есть команды label , reset и merge .

Команда label связывает метку с текущим HEAD при выполнении этой команды. Эти метки создаются как локальные ссылки рабочего дерева ( refs/rewritten/<label> ), которые будут удалены после завершения перебазирования. Таким образом, операции перебазирования в нескольких рабочих деревьях, связанных с одним и тем же репозиторием, не мешают друг другу. Если команда label не удалась, она немедленно перепланируется с полезным сообщением о том, как действовать.

Команда reset сбрасывает HEAD, индекс и рабочее дерево до указанной версии. Он похож на exec git reset --hard <label> , но отказывается перезаписывать неотслеживаемые файлы. Если команда reset завершается неудачно, она немедленно переносится с полезным сообщением о том, как редактировать список задач (обычно это происходит, когда команда reset была вставлена ​​в список задач вручную и содержит опечатку).

Команда merge объединит указанные ревизии с тем, что в данный момент является HEAD. При использовании -C <original-commit> будет использоваться сообщение фиксации указанного слияния. Когда -C изменяется на -c в нижнем регистре, сообщение откроется в редакторе после успешного слияния, чтобы пользователь мог редактировать сообщение.

Если команда merge завершается неудачно по какой-либо причине, кроме конфликтов слияния (т. е. когда операция слияния даже не началась), она немедленно перепланируется.

По умолчанию команда merge будет использовать стратегию слияния ort для обычных слияний и octopus для слияний Octopus. Можно указать стратегию по умолчанию для всех слияний, используя аргумент --strategy при вызове перебазирования, или можно переопределить определенные слияния в интерактивном списке команд, используя команду exec для явного вызова git merge с аргументом --strategy . Обратите внимание, что при явном вызове git merge таким образом вы можете использовать тот факт, что метки являются локальными ссылками рабочего дерева (например, ссылка refs/rewritten/onto будет соответствовать метке onto ), чтобы ссылаться на ветки, которые вы хотите объединить.

Примечание: первая команда ( label onto ) отмечает ревизию, на которую перебазируются фиксации; Название onto — это всего лишь условность, отсылка к варианту --onto .

Также можно ввести совершенно новые коммиты слияния с нуля, добавив команду вида merge <merge-head> . Эта форма генерирует предварительное сообщение о фиксации и всегда открывает редактор, позволяющий пользователю редактировать его. Это может быть полезно, например, когда тематическая ветвь решает более чем одну проблему и ее необходимо разделить на две или даже более тематических ветвей. Рассмотрим этот список задач:

pick 192837 Switch from GNU Makefiles to CMake
pick 5a6c7e Document the switch to CMake
pick 918273 Fix detection of OpenSSL in CMake
pick afbecd http: add support for TLS v1.3
pick fdbaec Fix detection of cURL in CMake on Windows

Один коммит в этом списке, не связанный с CMake, вполне мог быть мотивирован работой над исправлением всех ошибок, возникших при переходе на CMake, но он решает другую проблему. Чтобы разделить эту ветку на две тематические ветки, список дел можно отредактировать следующим образом:

label onto

pick afbecd http: add support for TLS v1.3
label tlsv1.3

reset onto
pick 192837 Switch from GNU Makefiles to CMake
pick 918273 Fix detection of OpenSSL in CMake
pick fdbaec Fix detection of cURL in CMake on Windows
pick 5a6c7e Document the switch to CMake
label cmake

reset onto
merge tlsv1.3
merge cmake

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

rebase.backend

Серверная часть по умолчанию, используемая для перебазирования. Возможные варианты: apply или merge . В будущем, если серверная часть слияния получит все оставшиеся возможности серверной части применения, этот параметр может стать неиспользованным.

rebase.stat

Показывать ли статистику того, что изменилось в исходном коде с момента последней перебазировки. False по умолчанию.

rebase.autoSquash

Если установлено значение true , включите опцию --autosquash по умолчанию.

rebase.autoStash

Если установлено значение true, автоматически создается запись временного тайника перед началом операции и применяется ее после завершения операции. Это означает, что вы можете запустить rebase на «грязном» рабочем дереве. Однако будьте осторожны: приложение окончательного кэша после успешного перебазирования может привести к нетривиальным конфликтам. Эту опцию можно переопределить опциями --no-autostash и --autostash для git-rebase[1] . По умолчанию ложь.

rebase.updateRefs

Если установлено значение true , включите опцию --update-refs по умолчанию.

rebase.missingCommitsCheck

Если установлено значение "warn" , git rebase -i выведет предупреждение, если некоторые фиксации будут удалены (например, строка была удалена), однако перебазирование все равно продолжится. Если установлено значение "error" , будет напечатано предыдущее предупреждение и остановлено перебазирование. Затем можно использовать git rebase --edit-todo для исправления ошибки. Если установлено значение "ignore" , проверка не производится. Чтобы отменить фиксацию без предупреждения или ошибки, используйте команду drop в списке задач. По умолчанию "ignore" .

rebase.instructionFormat

Строка формата, указанная в git-log[1] , которая будет использоваться для списка задач во время интерактивного перебазирования. К формату автоматически добавляется хеш фиксации.

rebase.abbreviateCommands

Если установлено значение true, git rebase будет использовать сокращенные имена команд в списке задач, что приведет к чему-то вроде этого:

        p deadbee The oneline of the commit
        p fa1afe1 The oneline of the next commit
        ...

instead of:

        pick deadbee The oneline of the commit
        pick fa1afe1 The oneline of the next commit
        ...

По умолчанию ложь.

rebase.rescheduleFailedExec

Автоматически перепланировать неудачные команды exec . Это имеет смысл только в интерактивном режиме (или когда была предоставлена ​​опция --exec ). Это то же самое, что указать опцию --reschedule-failed-exec .

rebase.forkPoint

Если установлено значение false , установите параметр --no-fork-point по умолчанию.

rebase.rebaseMerges

Можно ли и как установить опцию --rebase-merges по умолчанию. Может быть rebase-cousins , no-rebase-cousins или логическое значение. Установка true или no-rebase-cousins эквивалентна --rebase-merges=no-rebase-cousins , установка rebase-cousins эквивалентна --rebase-merges=rebase-cousins , а установка false эквивалентна --no-rebase-merges . Передача --rebase-merges в командной строке с аргументом или без него отменяет любую конфигурацию rebase.rebaseMerges .

rebase.maxLabelLength

При создании имен меток из субъектов фиксации усекайте имена до этой длины. По умолчанию имена усекаются до значения, немного меньшего, чем NAME_MAX (чтобы, например, можно было записывать файлы .lock для соответствующих свободных ссылок).

sequence.editor

Текстовый редактор, используемый git rebase -i для редактирования файла инструкций перебазирования. Значение предназначено для интерпретации оболочкой при его использовании. Его можно переопределить с помощью переменной среды GIT_SEQUENCE_EDITOR . Если этот параметр не настроен, вместо него используется редактор сообщений о фиксации по умолчанию.
