git-blame

Name

git-blame — показать, какая редакция и автор последним изменили каждую строку файла.

Synopsis

git blame [-c] [-b] [-l] [--root] [-t] [-f] [-n] [-s] [-e] [-p] [-w] [--incremental]
            [-L <range>] [-S <revs-file>] [-M] [-C] [-C] [-C] [--since=<date>]
            [--ignore-rev <rev>] [--ignore-revs-file <file>]
            [--color-lines] [--color-by-age] [--progress] [--abbrev=<n>]
            [ --contents <file> ] [<rev> | --reverse <rev>..<rev>] [--] <file>

Description

Аннотирует каждую строку в данном файле информацией из редакции, которая последней модифицировала строку. При желании начните аннотирование с данной ревизии.

Если указано один или несколько раз, -L ограничивает аннотацию запрошенными строками.

Начало строк автоматически отслеживается при переименовании всего файла (в настоящее время нет возможности отключить отслеживание переименования). Чтобы следовать за строками, перемещенными из одного файла в другой, или следовать за строками, которые были скопированы и вставлены из другого файла и т. д., см. параметры -C и -M .

В отчете ничего не говорится об удаленных или замененных строках; вам необходимо использовать такой инструмент, как git diff или интерфейс "pickaxe" , кратко упомянутый в следующем параграфе.

Помимо поддержки аннотаций файлов, Git также поддерживает поиск в истории разработки того, когда в фрагменте кода произошло изменение. Это позволяет отслеживать, когда фрагмент кода был добавлен в файл, перемещен или скопирован между файлами и, в конечном итоге, удален или заменен. Он работает путем поиска текстовой строки в файле diff. Небольшой пример интерфейса кирки, который ищет blame_usage :

$ git log --pretty=oneline -S'blame_usage'
5040f17eba15504bad66b14a645bddd9b015ebb7 blame -S <ancestry-file>
ea4c7f9bf69e781dd0cd88d2bccb2bf5cc15c9a7 git-blame: Make the output

Options

-b

Покажите пустой SHA-1 для граничных коммитов. Этим также можно управлять с помощью опции конфигурации blame.blankBoundary .

--root

Не рассматривайте корневые коммиты как границы. Этим также можно управлять с помощью опции конфигурации blame.showRoot .

--show-stats

Включите дополнительную статистику в конце вывода об обвинениях.

-L <start>,<end>
-L :<funcname>

Аннотируйте только диапазон строк, заданный <start>,<end> или именем функции regex <funcname> . Может быть указано несколько раз. Допускаются перекрывающиеся диапазоны.

<start> и <end> являются дополнительными. -L <start> или -L <start>, простираются от <start> до конца файла. -L ,<end> простирается от начала файла до <end> .

<start> и <end> могут принимать одну из следующих форм:

  • number

    Если <start> или <end> — число, оно указывает абсолютный номер строки (строки отсчитываются с 1).

  • /regex/

    В этой форме будет использоваться первая строка, соответствующая данному регулярному выражению POSIX. Если <start> является регулярным выражением, поиск будет осуществляться с конца предыдущего диапазона -L , если таковой имеется, в противном случае — с начала файла. Если <start> равен ^/regex/ , поиск будет начинаться с начала файла. Если <end> является регулярным выражением, поиск будет начинаться со строки, заданной <start> .

  • +смещение или -смещение

    Это справедливо только для <end> и будет указывать количество строк до или после строки, заданной <start> .

Если вместо <start> и <end> указан :<funcname> , это регулярное выражение, обозначающее диапазон от первой строки имени функции, соответствующей <funcname> , до следующей строки имени функции. :<funcname> выполняет поиск с конца предыдущего диапазона -L , если таковой имеется, в противном случае — с начала файла. ^:<funcname> выполняет поиск с начала файла. Имена функций определяются так же, как git diff обрабатывает заголовки фрагментов патчей (см. Defining a custom hunk-header в gitattributes[5] ).

-l

Показывать длинные обороты (по умолчанию: выключено).

-t

Показать необработанную временную метку (по умолчанию: выключено).

-S <revs-file>

Используйте версии из файла revs вместо вызова git-rev-list[1] .

--reverse <rev>..<rev>

Идите по истории вперед, а не назад. Вместо отображения ревизии, в которой появилась строка, отображается последняя ревизия, в которой существовала строка. Это требует ряда изменений, таких как START..END, где виновный путь находится в START. git blame --reverse START для удобства принят за git blame --reverse START..HEAD .

--first-parent

Следуйте только первому родительскому коммиту, увидев коммит слияния. Эту опцию можно использовать для определения того, когда строка была добавлена ​​в конкретную ветку интеграции, а не когда она была введена в историю в целом.

-p
--porcelain

Показать в формате, предназначенном для машинного потребления.

--line-porcelain

Покажите фарфоровый формат, но выведите информацию о фиксации для каждой строки, а не только при первом обращении к фиксации. Подразумевается --фарфор.

--incremental

Покажите результат постепенно в формате, предназначенном для машинного использования.

--encoding=<encoding>

Указывает кодировку, используемую для вывода имен авторов и сводок коммитов. Установка значения none приводит к выводу непреобразованных данных. Для получения дополнительной информации см. обсуждение кодирования на странице руководства git-log[1] .

--contents <file>

Аннотируйте, используя содержимое именованного файла, начиная с <rev>, если он указан, и HEAD в противном случае. Вы можете указать - , чтобы команда считывала содержимое файла из стандартного ввода.

--date <format>

Указывает формат, используемый для вывода дат. Если параметр --date не указан, используется значение переменной конфигурации Assessment.date. Если переменная конфигурации обвиняемого.дата также не установлена, используется формат iso. Поддерживаемые значения см. в обсуждении параметра --date на git-log[1] .

--[no-]progress

Статус выполнения по умолчанию сообщается в стандартном потоке ошибок, когда он подключен к терминалу. Этот флаг включает отчеты о ходе выполнения, даже если он не подключен к терминалу. Невозможно использовать --progress вместе с --porcelain или --incremental .

-M[<num>]

Обнаружение перемещенных или скопированных строк в файле. Когда фиксация перемещает или копирует блок строк (например, в исходном файле есть A, а затем B, а фиксация меняет его на B, а затем на A), традиционный алгоритм blame замечает только половину перемещения и обычно винит строки, которые были перемещен вверх (т.е. B) к родительскому коммиту и возлагает вину на строки, которые были перемещены вниз (т.е. A) к дочернему коммиту. При использовании этой опции ответственность за обе группы линий возлагается на родительскую группу за счет выполнения дополнительных проходов проверки.

<num> является необязательным, но это нижняя граница количества буквенно-цифровых символов, которые Git должен обнаружить как перемещение/копирование в файле, чтобы связать эти строки с родительским коммитом. Значение по умолчанию — 20.

-C[<num>]

Помимо -M , обнаруживает строки, перемещенные или скопированные из других файлов, которые были изменены в том же коммите. Это полезно, когда вы реорганизуете свою программу и перемещаете код по файлам. Если эта опция задана дважды, команда дополнительно ищет копии других файлов в фиксации, создающей файл. Если эта опция задана три раза, команда дополнительно ищет копии других файлов в любом коммите.

<num> является необязательным, но это нижняя граница количества буквенно-цифровых символов, которые Git должен обнаружить как перемещение/копирование между файлами, чтобы связать эти строки с родительским коммитом. Значение по умолчанию — 40. Если задано более одного параметра -C , вступит в силу аргумент <num> последнего -C .

--ignore-rev <rev>

Игнорируйте изменения, внесенные редакцией при назначении вины, как будто изменений никогда не было. Строки, которые были изменены или добавлены игнорируемым коммитом, будут связаны с предыдущим коммитом, который изменил эту строку или близлежащие строки. Эту опцию можно указать несколько раз, чтобы игнорировать более одной ревизии. Если установлена ​​опция конфигурации blame.markIgnoredLines , то строки, которые были изменены игнорируемым коммитом и приписаны другому коммиту, будут помечены ? в выводе об обвинениях. Если установлена ​​опция конфигурации blame.markUnblamableLines , то строки, затронутые игнорируемым коммитом, которые мы не могли отнести к другой ревизии, помечаются * .

--ignore-revs-file <file>

Игнорируйте версии, перечисленные в file , которые должны быть в том же формате, что и fsck.skipList . Эту опцию можно повторить, и эти файлы будут обрабатываться после любых файлов, указанных в опции конфигурации blame.ignoreRevsFile . Пустое имя файла "" очистит список оборотов из ранее обработанных файлов.

--color-lines

Цветовые аннотации строк в формате по умолчанию различаются, если они взяты из того же коммита, что и предыдущая строка. Это облегчает различение блоков кода, введенных разными коммитами. Цвет по умолчанию — голубой, и его можно настроить с помощью параметра конфигурации color.blame.repeatedLines .

--color-by-age

Цвет аннотаций линий в зависимости от возраста линии в формате по умолчанию. Параметр конфигурации color.blame.highlightRecent определяет, какой цвет используется для каждого возрастного диапазона.

-h

Показать справочное сообщение.

-c

Используйте тот же режим вывода, что и git-annotate[1] (по умолчанию: выключено).

--score-debug

Включите отладочную информацию, связанную с перемещением строк между файлами (см. -C ) и строк, перемещаемых внутри файла (см. -M ). Первое число в списке — это счет. Это количество буквенно-цифровых символов, обнаруженных при перемещении между файлами или внутри них. Это значение должно быть выше определенного порога, чтобы git blame мог считать эти строки кода перемещенными.

-f
--show-name

Показать имя файла в исходном коммите. По умолчанию имя файла отображается, если есть какая-либо строка из файла с другим именем из-за обнаружения переименования.

-n
--show-number

Показывать номер строки в исходном коммите (по умолчанию: выключено).

-s

Подавить имя автора и временную метку в выводе.

-e
--show-email

Показывать адрес электронной почты автора вместо имени автора (по умолчанию: выключено). Этим также можно управлять с помощью опции конфигурации blame.showEmail .

-w

Игнорируйте пробелы при сравнении родительской и дочерней версий, чтобы определить, откуда взялись строки.

--abbrev=<n>

Вместо использования стандартных 7+1 шестнадцатеричных цифр в качестве сокращенного имени объекта используйте цифры <m>+1, где <m> равно как минимум <n>, но гарантирует уникальность имен объектов фиксации. Обратите внимание, что один столбец используется для курсора, обозначающего граничную фиксацию.

Формат по умолчанию

Если ни опция --porcelain , ни опция --incremental не указаны, git blame выводит аннотацию для каждой строки с помощью:

  • сокращенное имя объекта коммита, из которого взята строка;

  • идентификатор автора (по умолчанию имя и дата автора, если не указано -s или -e ); и

  • line number

перед содержимым строки.

Формат фарфора.

В этом формате каждая строка выводится после заголовка; заголовок как минимум имеет первую строку, которая имеет:

  • 40-байтовый SHA-1 коммита, которому присвоена строка;

  • номер строки исходного файла;

  • номер строки в конечном файле;

  • в строке, которая начинает группу строк из коммита, отличного от предыдущего, — количество строк в этой группе. В последующих строках это поле отсутствует.

За этой строкой заголовка следует следующая информация хотя бы один раз для каждого коммита:

  • имя автора ( "author" ), адрес электронной почты ( "author-mail" ), время ( "author-time" ) и часовой пояс ( "author-tz" ); аналогично для коммиттера.

  • имя файла в коммите, к которому приписана строка.

  • первая строка сообщения журнала фиксации ( "summary" ).

Содержимое фактической строки выводится после приведенного выше заголовка с префиксом TAB. Это сделано для того, чтобы позже можно было добавить больше элементов заголовка.

Формат фарфора обычно подавляет информацию о фиксации, которая уже была просмотрена. Например, будут показаны две строки, в которых виноват один и тот же коммит, но сведения об этом коммите будут показаны только один раз. Это более эффективно, но может потребовать, чтобы читатель сохранял больше состояния. Опцию --line-porcelain можно использовать для вывода полной информации о фиксации для каждой строки, что обеспечивает более простое (но менее эффективное) использование, например:

# подсчитаем количество строк, приписываемых каждому автору
git blame --line-porcelain file |
sed -n 's/^author //p' |
sort | uniq -c | sort -rn

Specifying ranges

В отличие от git blame и git annotate в старых версиях git, размер аннотации может быть ограничен как диапазонами строк, так и диапазонами редакций. Опция -L , ограничивающая аннотацию диапазоном строк, может быть указана несколько раз.

Если вы хотите найти происхождение строк 40-60 для файла foo , вы можете использовать опцию -L следующим образом (они означают одно и то же — оба запрашивают 21 строку, начиная со строки 40):

git blame -L 40,60 foo
git blame -L 40,+21 foo

Также вы можете использовать регулярное выражение для указания диапазона строк:

git blame -L '/^sub hello {/,/^}$/' foo

что ограничивает аннотацию телом подпрограммы hello .

Если вас не интересуют изменения старше версии v2.6.18 или изменения старше 3 недель, вы можете использовать спецификаторы диапазона ревизий, аналогичные git rev-list :

git blame v2.6.18.. -- foo
git blame --since=3.weeks -- foo

Когда для ограничения аннотации используются спецификаторы диапазона ревизий, строки, которые не изменились с момента границы диапазона (либо фиксация v2.6.18, либо самая последняя фиксация, возраст которой превышает 3 недели в приведенном выше примере), считаются причиной этой границы диапазона. совершить.

Особенно полезный способ — проверить, есть ли в добавленном файле строки, созданные путем копирования и вставки из существующих файлов. Иногда это указывает на то, что разработчик поступил небрежно и не провел рефакторинг кода должным образом. Сначала вы можете найти коммит, который представил файл:

git log --diff-filter=A --pretty=short -- foo

а затем прокомментируйте изменение между коммитом и его родителями, используя нотацию commit^! :

git blame -C -C -f $commit^! -- foo

Incremental output

При вызове с опцией --incremental команда выводит результат в том виде, в котором он построен. В выводе обычно сначала говорится о строках, затронутых более поздними коммитами (т. е. строки будут аннотированы не по порядку), и они предназначены для использования интерактивными средствами просмотра.

Выходной формат аналогичен формату Porcelain, но он не содержит фактических строк из аннотируемого файла.

  1. Каждая запись об обвинении всегда начинается со строки:

    <40-byte hex sha1> <sourceline> <resultline> <num_lines>

    Номера строк считаются от 1.

  2. Когда коммит впервые появляется в потоке, он распечатывает различную другую информацию о нем с тегом из одного слова в начале каждой строки, описывающим дополнительную информацию о коммите (автор, адрес электронной почты, коммиттер, даты, сводка и т. д.). .).

  3. В отличие от формата Porcelain, информация об имени файла указывается всегда и завершает запись:

    "filename" <whitespace-quoted-filename-goes-here>

    и поэтому его действительно довольно легко разобрать с помощью какого-либо синтаксического анализатора, ориентированного на строки и слова (что должно быть вполне естественно для большинства языков сценариев).

    Note
    		 Для людей, которые выполняют синтаксический анализ: чтобы сделать его более надежным, просто игнорируйте любые строки между первой и последней строками (строки "<sha1>" и "filename" ), в которых вы не распознаете слова-теги (или не заботитесь об этом конкретном слове) в начале "extended information" . линии. Таким образом, если когда-либо будет добавлена ​​информация (например, кодировка коммита или расширенный комментарий к коммиту), просмотрщику обвинений будет все равно.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

blame.blankBoundary

Показывать пустое имя объекта фиксации для граничных коммитов в git-blame[1] . По умолчанию эта опция имеет значение false.

blame.coloring

Это определяет цветовую схему, которая будет применяться к выводу обвинений. Это может быть repeatedLines , highlightRecent или none , который используется по умолчанию.

blame.date

Указывает формат, используемый для вывода дат в git-blame[1] . Если значение не установлено, используется формат ISO. Поддерживаемые значения см. в обсуждении опции --date на странице git-log[1] .

blame.showEmail

Показывать адрес электронной почты автора вместо имени автора в git-blame[1] . По умолчанию эта опция имеет значение false.

blame.showRoot

Не рассматривайте корневые коммиты как границы в git-blame[1] . По умолчанию эта опция имеет значение false.

blame.ignoreRevsFile

Игнорировать версии, перечисленные в файле, по одному несокращенному имени объекта в строке, в git-blame[1] . Пробелы и комментарии, начинающиеся с # , игнорируются. Эту опцию можно повторить несколько раз. Пустые имена файлов сбросят список игнорируемых ревизий. Этот параметр будет обработан перед параметром командной строки --ignore-revs-file .

blame.markUnblamableLines

Отметьте строки, которые были изменены игнорируемой ревизией, которую мы не могли отнести к другому коммиту, с помощью * в выходных данных git-blame[1] .

blame.markIgnoredLines

Отметьте строки, которые были изменены игнорируемой ревизией, которую мы приписали другому коммиту, с помощью ? в выходных данных git-blame[1] .
