git-repack

Name

git-repack — Упаковать распакованные объекты в репозиторий.

Synopsis

git repack [-a] [-A] [-d] [-f] [-F] [-l] [-n] [-q] [-b] [-m] [--window=<n>] [--depth=<n>] [--threads=<n>] [--keep-pack=<pack-name>] [--write-midx]

Description

Эта команда используется для объединения всех объектов, которые в данный момент не находятся в "pack" , в пакет. Его также можно использовать для реорганизации существующих пакетов в один, более эффективный пакет.

Пакет — это набор объектов, индивидуально сжатых с применением дельта-сжатия, хранящихся в одном файле со связанным индексным файлом.

Пакеты используются для снижения нагрузки на зеркальные системы, механизмы резервного копирования, дисковые хранилища и т. д.

Options

-a

Вместо постепенной упаковки распакованных объектов упакуйте все, на что есть ссылки, в один пакет. Особенно полезно при упаковке репозитория, который используется для разработки private . Используйте с -d . Это очистит объекты, оставленные git prune , но git fsck --full --dangling отображается как висячий.

Обратите внимание, что пользователям, осуществляющим выборку по тупым протоколам, придется получить весь новый пакет, чтобы получить любой содержащийся объект, независимо от того, сколько других объектов в этом пакете у них уже есть локально.

Пакетные файлы Promisor перепаковываются отдельно: если существуют пакеты пакетов, у которых есть связанный файл ".promisor" , эти файлы пакетов будут переупакованы в другой отдельный пакет, и будет записан пустой файл ".promisor" , соответствующий новому отдельному пакету.

-A

То же, что и -a , если не используется -d . Тогда любые недоступные объекты в предыдущем пакете становятся свободными, распакованными объектами, а не остаются в старом пакете. Недоступные объекты никогда намеренно не добавляются в пакет, даже при переупаковке. Эта опция предотвращает немедленное удаление недоступных объектов, поскольку они остаются в старом пакете и затем удаляются. Вместо этого свободные недоступные объекты будут удалены в соответствии с обычными правилами истечения срока действия при следующем вызове git gc . См. git-gc[1] .

-d

Если после упаковки вновь созданные пакеты делают некоторые существующие пакеты избыточными, удалите избыточные пакеты. Также запустите git prune-packed , чтобы удалить лишние свободные объектные файлы.

--cruft

То же, что и -a , если не используется -d . Затем все недоступные объекты упаковываются в отдельный крафт-пак. Недостижимые объекты могут быть удалены с использованием обычных правил срока действия при следующем вызове git gc (см. git-gc[1] ). Несовместим с -k .

--cruft-expiration=<approxidate>

Срок действия недоступных объектов старше <approxidate> истекает немедленно, вместо ожидания следующего вызова git gc . Полезно только с --cruft -d .

--max-cruft-size=<n>

Переупаковывайте объекты Cruft в пакеты размером до <n> байт перед созданием новых пакетов. Пока имеется достаточное количество пакетов меньшего размера, чем <n> , переупаковка приведет к созданию нового пакета, содержащего объекты из любых объединенных пакетов, а также любые новые недоступные объекты. Упаковки Cruft размером более <n> не будут модифицированы. Когда новый пакет Cruft превышает размер <n> байт, он будет разделен на несколько пакетов, каждый из которых гарантированно будет иметь размер не более <n> байт. Полезно только с --cruft -d .

--expire-to=<dir>

Запишите пакет, содержащий обрезанные объекты (если таковые имеются), в каталог <dir> . Эта опция полезна для хранения копий любых удаленных объектов в отдельном каталоге в качестве резервной копии. Полезно только с --cruft -d .

-l

Передайте опцию --local в git pack-objects . См. git-pack-objects[1] .

-f

Передайте опцию --no-reuse-delta в git-pack-objects , см. git-pack-objects[1] .

-F

Передайте опцию --no-reuse-object в git-pack-objects , см. git-pack-objects[1] .

-q
--quiet

Покажите отсутствие прогресса в стандартном потоке ошибок и передайте параметр -q в git pack-objects . См. git-pack-objects[1] .

-n

Не обновляйте информацию о сервере с помощью git update-server-info . Эта опция пропускает обновление файлов локального каталога, необходимых для публикации этого репозитория (или его прямой копии) через HTTP или FTP. См. git-update-server-info[1] .

--window=<n>
--depth=<n>

Эти два параметра влияют на то, как объекты, содержащиеся в пакете, сохраняются с использованием дельта-сжатия. Объекты сначала сортируются внутри по типу, размеру и, при необходимости, именам, а затем сравниваются с другими объектами в --window , чтобы увидеть, экономит ли пространство использование дельта-сжатия. --depth ограничивает максимальную глубину отклонения; слишком глубокая обработка влияет на производительность распаковщика, поскольку дельта-данные необходимо применять столько раз, чтобы добраться до нужного объекта.

Значение по умолчанию для --window — 10, а —eep — 50. Максимальная глубина — 4095.

--threads=<n>

Эта опция передается на git pack-objects .

--window-memory=<n>

Эта опция обеспечивает дополнительный лимит сверх --window ; размер окна будет динамически уменьшаться, чтобы не занимать в памяти более <n> байт. Это полезно в репозиториях со смесью больших и маленьких объектов, чтобы не исчерпать память при большом окне, но при этом иметь возможность использовать большое окно для меньших объектов. К размеру можно добавить "k" , "m" или "g" . --window-memory=0 делает использование памяти неограниченным. Значение по умолчанию берется из переменной конфигурации pack.windowMemory . Обратите внимание, что фактическое использование памяти будет равно пределу, умноженному на количество потоков, используемых git-pack-objects[1] .

--max-pack-size=<n>

Максимальный размер каждого выходного пакетного файла. К размеру можно добавить суффикс "k" , "m" или "g" . Минимально допустимый размер ограничен 1 МБ. Если указано, может быть создано несколько паковочных файлов, что также предотвращает создание растрового индекса. Значение по умолчанию не ограничено, если не установлена ​​переменная конфигурации pack.packSizeLimit . Обратите внимание, что этот вариант может привести к увеличению размера и замедлению репозитория; см. обсуждение в pack.packSizeLimit .

--filter=<filter-spec>

Удалите объекты, соответствующие спецификации фильтра, из полученного файла пакета и поместите их в отдельный файл пакета. Обратите внимание, что объекты, используемые в рабочем каталоге, не отфильтровываются. Поэтому, чтобы разделение полностью работало, лучше всего выполнить его в чистом репозитории и использовать параметры -a и -d вместе с этим параметром. Также следует использовать --no-write-bitmap-index (или параметр конфигурации repack.writebitmaps , установленный на false ), иначе запись растрового индекса не удастся, поскольку предполагается, что один пакетный файл содержит все объекты. См. git-rev-list[1] для получения действительных форм <filter-spec> .

--filter-to=<dir>

Запишите пакет, содержащий отфильтрованные объекты, в каталог <dir> . Полезно только с --filter . Это можно использовать для помещения пакета в отдельный каталог объектов, доступ к которому осуществляется через механизм альтернатив Git . ВНИМАНИЕ: Если пакетный файл, содержащий отфильтрованные объекты, недоступен, репозиторий может быть поврежден, поскольку доступ к объектам в этом пакетном файле может оказаться невозможным. См. разделы objects и objects/info/alternates в gitrepository-layout[5] .

-b
--write-bitmap-index

Напишите растровый индекс достижимости как часть репака. Это имеет смысл только при использовании с -a , -A или -m , поскольку растровые изображения должны иметь возможность ссылаться на все доступные объекты. Эта опция отменяет настройку repack.writeBitmaps . Эта опция не имеет эффекта, если создается несколько файлов упаковки, за исключением случаев записи MIDX (в этом случае создается растровое изображение с несколькими упаковками).

--pack-kept-objects

Включайте объекты в файлы .keep при переупаковке. Обратите внимание, что мы по-прежнему не удаляем пакеты .keep после завершения pack-objects . Это означает, что мы можем дублировать объекты, но это делает эту опцию безопасной для использования при одновременной отправке или выборке. Этот параметр обычно полезен только в том случае, если вы пишете растровые изображения с помощью -b или repack.writeBitmaps , поскольку он гарантирует, что растровый пакетный файл содержит необходимые объекты.

--keep-pack=<pack-name>

Исключить данную упаковку из переупаковки. Это эквивалент наличия файла .keep на упаковке. <pack-name> — это имя файла пакета без ведущего каталога (например, pack-123.pack ). Эту опцию можно указать несколько раз, чтобы сохранить несколько пакетов.

--unpack-unreachable=<when>

При освобождении недоступных объектов не беспокойтесь об освобождении объектов старше <when> . Это можно использовать для оптимизации записи любых объектов, которые будут немедленно удалены последующим git prune .

-k
--keep-unreachable

При использовании с -ad все недоступные объекты из существующих пакетов будут добавлены в конец файла пакета, а не удалены. Кроме того, все недоступные незакрепленные предметы будут упакованы (а их незакрепленные аналоги удалены).

-i
--delta-islands

Передайте опцию --delta-islands в git-pack-objects , см. git-pack-objects[1] .

-g<factor>
--geometric=<factor>

Организуйте полученную структуру пакета так, чтобы каждый последующий пакет содержал как минимум <factor> раз больше объектов, чем следующий по величине пакет.

git repack обеспечивает это, определяя "cut" пакетов файлов, которые необходимо переупаковать в один, чтобы обеспечить геометрическую прогрессию. Он выбирает наименьший набор файлов пакетов, чтобы как можно больше файлов пакетов большего размера (по количеству объектов, содержащихся в этом пакете) можно было оставить нетронутыми.

В отличие от других режимов переупаковки, набор объектов для упаковки определяется однозначно набором пакетов "rolled-up" ; другими словами, колоды, которые необходимо объединить, чтобы восстановить геометрическую прогрессию.

Свободные объекты неявно включены в этот "roll-up" , независимо от их достижимости. Это может измениться в будущем.

При записи растрового изображения с несколькими пакетами git repack выбирает самый большой полученный пакет в качестве предпочтительного пакета для выбора объекта с помощью MIDX (см. git-multi-pack-index[1] ).

-m
--write-midx

Напишите индекс мультипакета (см. git-multi-pack-index[1] ), содержащий неизбыточные пакеты.

Configuration

Различные переменные конфигурации влияют на упаковку, см. git-config[1] (найдите "pack" и "delta" ).

По умолчанию команда передает параметр --delta-base-offset в git pack-objects ; Обычно это приводит к получению пакетов немного меньшего размера, но созданные пакеты несовместимы с версиями Git старше версии 1.4.4. Если вам нужно поделиться своим репозиторием с такими древними версиями Git , напрямую или через тупой протокол http, вам нужно установить для переменной конфигурации repack.UseDeltaBaseOffset значение "false" и перепаковать. Эта опция не влияет на доступ из старых версий Git по собственному протоколу, поскольку в этом случае преобразование выполняется «на лету» по мере необходимости.

Дельта-сжатие не используется для объектов, размер которых превышает переменную конфигурации core.bigFileThreshold , а также для файлов с атрибутом delta , для которого установлено значение false.
