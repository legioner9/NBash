gitcredentials

Name

gitcredentials — предоставление имен пользователей и паролей для Git

Synopsis

git config credential.https://example.com.username myusername
git config credential.helper "$helper $options"

Description

Git иногда требуются учетные данные пользователя для выполнения операций; например, ему может потребоваться запросить имя пользователя и пароль для доступа к удаленному репозиторию через HTTP. Некоторые пульты принимают в качестве пароля личный токен доступа или токен доступа OAuth. В этом руководстве описаны механизмы, которые Git использует для запроса этих учетных данных, а также некоторые функции, позволяющие избежать повторного ввода этих учетных данных.

Requesting credentials

Без определения каких-либо помощников по учетным данным Git будет пытаться использовать следующие стратегии, чтобы запросить у пользователя имена пользователей и пароли:

  1. Если установлена ​​переменная среды GIT_ASKPASS , запускается программа, указанная в этой переменной. Подходящее приглашение предоставляется программе в командной строке, а ввод пользователя считывается из ее стандартного вывода.

  2. В противном случае, если установлена ​​переменная конфигурации core.askPass , ее значение используется, как указано выше.

  3. В противном случае, если установлена ​​переменная среды SSH_ASKPASS , ее значение используется, как указано выше.

  4. В противном случае пользователю будет предложено это сделать на терминале.

Avoiding repetition

Вводить одни и те же учетные данные снова и снова может быть затруднительно. Git предоставляет два способа уменьшить это раздражение:

  1. Static конфигурация имен пользователей для данного контекста аутентификации.

  2. Помощники по учетным данным для кэширования или хранения паролей, а также для взаимодействия с системным кошельком паролей или связкой ключей.

Первый вариант прост и удобен, если у вас нет безопасного хранилища для пароля. Обычно это настраивается путем добавления этого в вашу конфигурацию:

[credential "https://example.com"]
        username = me

С другой стороны, помощники по учетным данным — это внешние программы, из которых Git может запрашивать как имена пользователей, так и пароли; они обычно взаимодействуют с безопасным хранилищем, предоставляемым ОС или другими программами. Альтернативно, помощник по созданию учетных данных может генерировать учетные данные для определенных серверов через некоторый API.

Чтобы использовать помощника, вы должны сначала выбрать его для использования. Git в настоящее время включает в себя следующие помощники:

cache

Кэшируйте учетные данные в памяти на короткий период времени. Подробности см. в git-credential-cache[1] .

store

Храните учетные данные на диске на неопределенный срок. Подробности см. в git-credential-store[1] .

У вас также могут быть установлены сторонние помощники; найдите credential-* в выходных данных git help -a и обратитесь к документации отдельных помощников. После того как вы выбрали помощника, вы можете указать Git использовать его, поместив его имя в переменную credential.helper.

  1. Найдите помощника.

    $ git help -a | grep credential-
credential-foo

  2. Прочтите его описание.

    $ git help credential-foo

  3. Скажите Git , чтобы он использовал его.

    $ git config --global credential.helper foo

Available helpers

Сообщество ведет полный список помощников по учетным данным Git по адресу https://git-scm.com/doc/credential-helpers .

OAuth

Альтернативой вводу паролей или токенов личного доступа является использование помощника по учетным данным OAuth. Первоначальная аутентификация открывает окно браузера для хоста. Последующая аутентификация происходит в фоновом режиме. Многие популярные хосты Git поддерживают OAuth.

Credential contexts

Git считает, что каждые учетные данные имеют контекст, определенный URL-адресом. Этот контекст используется для поиска конфигурации, зависящей от контекста, и передается любым помощникам, которые могут использовать его в качестве индекса в безопасном хранилище.

Например, представьте, что мы обращаемся к https://example.com/foo.git . Когда Git просматривает файл конфигурации, чтобы определить, соответствует ли раздел этому контексту, он будет считать эти два раздела совпадающими, если контекст является более конкретным подмножеством шаблона в файле конфигурации. Например, если в вашем конфигурационном файле есть следующее:

[credential "https://example.com"]
        username = foo

тогда мы совпадем: оба протокола одинаковы, оба хоста одинаковы, а URL-адрес "pattern" вообще не заботится о компоненте пути. Однако этот контекст не будет соответствовать:

[credential "https://kernel.org"]
        username = foo

потому что имена хостов различаются. И при этом он не будет соответствовать foo.example.com ; Git точно сравнивает имена хостов, не учитывая, являются ли два хоста частью одного и того же домена. Аналогично, запись конфигурации для http://example.com не будет совпадать: Git точно сравнивает протоколы. Однако вы можете использовать подстановочные знаки в имени домена и другие методы сопоставления с образцом, как в случае с параметрами http.<URL>.* .

Если URL-адрес "pattern" включает компонент пути, то он также должен точно совпадать: контекст https://example.com/bar/baz.git будет соответствовать записи конфигурации для https://example.com/bar/baz.git (в дополнение к записи конфигурации для https://example.com ), но не будет соответствовать записи конфигурации для https://example.com/bar .

Configuration options

Параметры контекста учетных данных можно настроить либо в credential.* (который применяется ко всем учетным данным), либо в credential.<URL>.* , где <URL> соответствует контексту, как описано выше.

Следующие параметры доступны в любом месте:

helper

Имя внешнего помощника по учетным данным и все связанные параметры. Если имя помощника не является абсолютным путем, то перед ним добавляется строка git credential- . Результирующая строка выполняется оболочкой (например, установка foo --option=bar приведет к выполнению git credential-foo --option=bar через оболочку. Примеры их использования см. в руководстве по конкретным помощникам.

Если существует несколько экземпляров переменной конфигурации credential.helper , каждый помощник будет проверен по очереди и может предоставить имя пользователя, пароль или ничего не предоставить. Как только Git получит и имя пользователя, и пароль с неистёкшим сроком действия, больше не будет использоваться помощников.

Если credential.helper настроен на пустую строку, это сбрасывает список помощников на пустое значение (поэтому вы можете переопределить помощник, установленный файлом конфигурации с более низким приоритетом, настроив помощник с пустой строкой, а затем любой набор помощников, который вам нужен).

username

Имя пользователя по умолчанию, если оно не указано в URL-адресе.

useHttpPath

По умолчанию Git не считает компонент "path" URL-адреса http достойным сопоставления с помощью внешних помощников. Это означает, что учетные данные, сохраненные для https://example.com/foo.git , будут также использоваться для https://example.com/bar.git . Если вы хотите различать эти случаи, установите для этой опции значение true .

Custom helpers

Вы можете написать свои собственные помощники для взаимодействия с любой системой, в которой вы храните учетные данные.

Помощники по учетным данным — это программы, выполняемые Git для извлечения или сохранения учетных данных из и в долговременное хранилище (где "long-term" просто длиннее, чем один процесс Git ; например, учетные данные могут храниться в памяти в течение нескольких минут или на неопределенный срок на диске). .

Каждый помощник задается одной строкой в ​​переменной конфигурации credential.helper (и других, см. git-config[1] ). Строка преобразуется Git в команду, которая будет выполнена с использованием следующих правил:

  1. Если вспомогательная строка начинается с "!" , она считается фрагментом оболочки, а все, что находится после "!" , становится командой.

  2. В противном случае, если вспомогательная строка начинается с абсолютного пути, дословная вспомогательная строка становится командой.

  3. В противном случае строка "git credential-" добавляется к вспомогательной строке, и результат становится командой.

К полученной команде затем добавляется аргумент "operation" (подробности см. ниже), и результат выполняется оболочкой.

Вот несколько примеров характеристик:

# запустить  "git credential-foo"
[credential]
        helper = foo

# то же, что и выше, но передаем аргумент помощнику
[credential]
        helper = "foo --bar=baz"

# аргументы анализируются оболочкой, поэтому используйте оболочку
# цитируем, если необходимо
[credential]
        helper = "foo --bar='whitespace arg'"

# вы также можете использовать абсолютный путь, который не будет использовать оболочку git
[credential]
        helper = "/path/to/my/helper --with-arguments"

# или вы можете указать свой собственный фрагмент оболочки
[credential "https://example.com"]
        username = your_user
        helper = "!f() { test \"$1\" = get && echo \"password=$(cat $HOME/.secret)\"; }; f"

Вообще говоря, правило (3), приведенное выше, проще всего указать пользователям. Авторы помощников по учетным данным должны приложить усилия, чтобы помочь своим пользователям, назвав свою программу "git-credential-$NAME" и поместив ее в $PATH или $GIT_EXEC_PATH во время установки, что позволит пользователю включить ее с помощью git config credential.helper $NAME .

Когда помощник выполняется, к его командной строке добавляется один аргумент "operation" , который является одним из:

get

Верните соответствующие учетные данные, если таковые существуют.

store

Сохраните учетные данные, если они применимы к помощнику.

erase

Удалите соответствующие учетные данные, если таковые имеются, из хранилища помощника.

Подробности учетных данных будут предоставлены в потоке stdin помощника. Точный формат такой же, как формат ввода/вывода сантехнической команды git credential (подробную спецификацию см. в разделе INPUT/OUTPUT FORMAT в git-credential[1] ).

Для операции get помощник должен создать список атрибутов на стандартном выводе в том же формате (общие атрибуты см. в git-credential[1] ). Помощник может создавать подмножество или даже вообще не создавать значений, если ему нечего предоставить. Любые предоставленные атрибуты перезапишут те, которые уже известны подсистеме учетных данных Git. Нераспознанные атрибуты автоматически отбрасываются.

Хотя можно переопределить все атрибуты, хорошо ведущие себя помощники должны воздерживаться от этого для любого атрибута, кроме имени пользователя и пароля.

Если помощник выводит атрибут quit со значением true или 1 , к дальнейшим помощникам обращаться не будут, а пользователю не будет предлагаться (если учетные данные не были предоставлены, операция завершится неудачей).

Аналогичным образом, после указания имени пользователя и пароля больше не будут обращаться к помощникам.

Для операции store или erase выходные данные помощника игнорируются.

Если помощнику не удается выполнить запрошенную операцию или ему необходимо уведомить пользователя о потенциальной проблеме, он может выполнить запись в stderr.

Если он не поддерживает запрошенную операцию (например, хранилище или генератор только для чтения), он должен молча игнорировать запрос.

Если помощник получает какую-либо другую операцию, он должен молча игнорировать запрос. Это оставляет место для добавления будущих операций (старые помощники просто игнорируют новые запросы).
