git-sparse-checkout

Name

git-sparse-checkout — Сократите рабочее дерево до подмножества отслеживаемых файлов.

Synopsis

git sparse-checkout (init | list | set | add | reapply | disable | check-rules) [<options>]

Description

Эта команда используется для создания разреженных проверок, которые меняют рабочее дерево: вместо присутствия всех отслеживаемых файлов на наличие только подмножества этих файлов. Он также может переключать, какое подмножество файлов присутствует, или отменить операцию и вернуться к наличию всех отслеживаемых файлов в рабочей копии.

Подмножество файлов выбирается путем предоставления списка каталогов в конусном режиме (по умолчанию) или путем предоставления списка шаблонов в неконусном режиме.

При разреженной проверке другие команды Git ведут себя немного по-другому. Например, переключение ветвей не будет обновлять пути за пределами каталогов/шаблонов разреженной проверки, а git commit -a не будет записывать пути за пределами каталогов/шаблонов разреженной проверки как удаленные.

ЭТА КОМАНДА ЭКСПЕРИМЕНТАЛЬНАЯ. ЕЕ ПОВЕДЕНИЕ И ПОВЕДЕНИЕ ДРУГИХ КОМАНД ПРИ НАЛИЧИИ SSPARSE-CHECKOUTS, ВЕРОЯТНО, ИЗМЕНИТСЯ В БУДУЩЕМ.

Commands

list

Опишите каталоги или шаблоны в файле разреженной проверки.

set

Включите необходимые параметры конфигурации разреженной проверки ( core.sparseCheckout , core.sparseCheckoutCone и index.sparse ), если для них еще не установлены нужные значения, заполните файл разреженной проверки из списка аргументов, следующих за подкомандой set , и обновите рабочий каталог, чтобы он соответствовал .

Чтобы гарантировать, что настройка параметров разреженной проверки в рабочем дереве не изменяет настройки разреженной проверки в других рабочих деревьях, подкоманда set обновит конфигурацию вашего репозитория, чтобы использовать конфигурацию, специфичную для рабочего дерева, если она еще не присутствует. Разреженность, определенная аргументами подкоманды set , сохраняется в файле разреженной проверки, специфичном для рабочего дерева. Более подробную информацию см. в git-worktree[1] и документации extensions.worktreeConfig в git-config[1] .

Если указана опция --stdin , каталоги или шаблоны считываются из стандарта в виде списка, разделенного новой строкой, а не из аргументов.

По умолчанию входной список считается списком каталогов, соответствующим выходным данным git ls-tree -d --name-only . Это включает в себя интерпретацию имен путей, которые начинаются с двойной кавычки (""), как строк в кавычках в стиле C. Обратите внимание, что все файлы в указанных каталогах (на любой глубине) будут включены в разреженную проверку, а также файлы, которые являются братьями и сестрами любого из них. данный каталог или любой из его предков (подробнее см. CONE PATTERN SET ниже). Раньше это не было значением по умолчанию, и нужно было указать --cone или включить core.sparseCheckoutCone .

При передаче --no-cone входной список считается списком шаблонов. Этот режим имеет ряд недостатков, в том числе не работает с некоторыми опциями типа --sparse-index . Как поясняется в разделе "Non-cone Problems" ниже, мы не рекомендуем его использовать.

Используйте параметр --[no-]sparse-index , чтобы использовать разреженный индекс (по умолчанию он не используется). Разреженный индекс уменьшает размер индекса, чтобы он был более точно согласован с вашим определением разреженной проверки. Это может дать значительные преимущества в производительности для таких команд, как git status или git add . Эта функция все еще является экспериментальной. Некоторые команды могут работать медленнее из-за разреженного индекса, пока они не будут должным образом интегрированы с этой функцией.

ВНИМАНИЕ: Использование разреженного индекса требует изменения индекса таким образом, который не совсем понятен внешним инструментам. Если у вас возникли проблемы с этой совместимостью, запустите git sparse-checkout init --no-sparse-index , чтобы переписать индекс, чтобы он не был разреженным. Более старые версии Git не будут понимать расширение индекса разреженных записей каталога и могут не взаимодействовать с вашим репозиторием, пока оно не будет отключено.

add

Обновите файл разреженной проверки, включив в него дополнительные каталоги (в режиме конуса) или шаблоны (в режиме без конуса). По умолчанию эти каталоги или шаблоны считываются из аргументов командной строки, но их можно прочитать из стандартного ввода с помощью опции --stdin .

reapply

Повторно примените правила шаблона разреженности к путям в рабочем дереве. Такие команды, как слияние или перебазирование, могут материализовать пути для выполнения своей работы (например, чтобы показать вам конфликт), а другие команды разреженной проверки могут не выполнить разрежение отдельного файла (например, из-за того, что в нем есть непроиндексированные изменения или конфликты). В таких случаях имеет смысл запустить git sparse-checkout reapply позже, после очистки затронутых путей (например, разрешения конфликтов, отмены или фиксации изменений и т. д.).

Команда reapply также может принимать флаги --[no-]cone и --[no-]sparse-index с тем же значением, что и флаги из команды set , чтобы изменить используемый режим разреженности без необходимости переопределения всех путей разреженности.

disable

Отключите настройку конфигурации core.sparseCheckout и восстановите рабочий каталог, включив в него все файлы.

init

Устаревшая команда, ведущая себя как set , без указания путей. Возможно, в будущем будут удалены.

Исторически сложилось так, что set не обрабатывал все необходимые настройки конфигурации, а это означало, что приходилось вызывать как init , так и set . Вызов обоих означал, что шаг init сначала удалит почти все отслеживаемые файлы (а в режиме конуса и игнорируемые файлы), затем шаг set добавит обратно многие отслеживаемые файлы (но не игнорируемые файлы). Помимо потерянных файлов, производительность и пользовательский интерфейс этой комбинации были плохими.

Кроме того, исторически init фактически не инициализировал файл разреженной проверки, если он уже существовал. Это означало, что можно было вернуться к разреженной проверке, не запоминая, какие пути передать последующей команде set или add . Однако параметры --cone и --sparse-index не будут запоминаться при команде отключения, поэтому простое восстановление вызова простого init теряет свою полезность.

check-rules

Проверьте, соответствуют ли правила разреженности одному или нескольким путям.

По умолчанию check-rules считывает список путей из стандартного ввода и выводит только те, которые соответствуют текущим правилам разреженности. Ожидается, что входные данные будут состоять из одного пути в строке, что соответствует выходным данным git ls-tree --name-only , включая то, что имена путей, начинающиеся с двойной кавычки (""), интерпретируются как строки в кавычках в стиле C.

При вызове с флагом --rules-file <file> входные файлы сопоставляются с разреженными правилами извлечения, найденными в <file> , а не с текущими. Ожидается, что правила в файлах будут иметь ту же форму, что и принятая git sparse-checkout set --stdin (в частности, они должны быть разделены символом новой строки).

По умолчанию правила, передаваемые опции --rules-file , интерпретируются как каталоги режима конуса. Чтобы передать шаблоны неконусного режима с помощью --rules-file , объедините эту опцию с опцией --no-cone .

При вызове с флагом -z формат путей, вводимых на стандартный ввод, а также выходных путей завершается \0 и не заключен в кавычки. Обратите внимание, что это не относится к формату правил, передаваемых с опцией --rules-file .

Examples

git sparse-checkout set MY/DIR1 SUB/DIR2

Перейдите к разреженной проверке со всеми файлами (на любой глубине) в каталогах MY/DIR1/ и SUB/DIR2/, присутствующими в рабочей копии (плюс все файлы непосредственно в каталогах MY/ и SUB/ и в каталоге верхнего уровня). Если вы уже находитесь в разреженной выдаче, измените файлы, присутствующие в рабочей копии, на этот новый выбор. Обратите внимание, что эта команда также удалит все игнорируемые файлы в любом каталоге, в котором больше нет отслеживаемых или неигнорируемых-неотслеживаемых файлов.

git sparse-checkout disable

Заново заполните рабочий каталог всеми файлами, отключив редкие извлечения.

git sparse-checkout add SOME/DIR/ECTORY

Добавьте все файлы в каталоге SOME/DIR/ECTORY/ (на любой глубине) в разреженную проверку, а также все файлы непосредственно в каталоге SOME/DIR/ и сразу в каталоге SOME/. Прежде чем использовать эту команду, он уже должен находиться в разреженной проверке.

git sparse-checkout reapply

Команды могут обновлять рабочее дерево таким образом, чтобы не учитывать выбранные каталоги разреженности. Это может происходить из-за инструментов, внешних по отношению к записи файлов Git , или даже влиять на команды Git либо из-за особых случаев (например, возникновения конфликтов при слиянии/перебазировании), либо из-за того, что некоторые команды не полностью поддерживали разреженные извлечения (например, старый сервер слияния recursive ). имели лишь ограниченную поддержку). Эта команда повторно применяет существующие спецификации разреженного каталога, чтобы обеспечить соответствие рабочего каталога.

Внутреннее устройство — редкая проверка

"Sparse checkout" позволяет разреженно заполнять рабочий каталог. Он использует бит Skip-worktree (см. git-update-index[1] ), чтобы сообщить Git , стоит ли просматривать файл в рабочем каталоге. Если бит Skip-worktree установлен, а файла нет в рабочем дереве, то его отсутствие игнорируется. Git не будет заполнять содержимое этих файлов, что делает разреженное извлечение полезным при работе в репозитории с большим количеством файлов, но только некоторые из них важны для текущего пользователя.

Файл $GIT_DIR/info/sparse-checkout используется для определения эталонного растрового изображения рабочего дерева пропуска. Когда Git обновляет рабочий каталог, он обновляет биты пропуска рабочего дерева в индексе на основе этого файла. Файлы, соответствующие шаблонам в файле, появятся в рабочем каталоге, а остальные — нет.

Внутренности — проблемы, не связанные с конусом

Файл $GIT_DIR/info/sparse-checkout , заполняемый подкомандами set и add , определяется как набор шаблонов (по одному на строку), использующих тот же синтаксис, что и файлы .gitignore . В режиме конуса эти шаблоны ограничены соответствующими каталогами (и пользователям нужно только указывать или просматривать имена каталогов), тогда как в режиме без конуса разрешен любой шаблон в стиле gitignore. Использование полных шаблонов в стиле gitignore в неконусном режиме имеет ряд недостатков:

  • По сути, это приводит к тому, что различные процессы обновления рабочего дерева (извлечение, объединение, перебазирование, переключение, сброс, извлечение и т. д.) требуют сопоставления шаблонов O(N*M), где N — количество шаблонов, а M — количество путей в индекс. Это плохо масштабируется.

  • Чтобы избежать проблемы масштабирования, необходимо ограничить количество шаблонов, указав ведущее имя каталога или glob.

  • Передача glob в командной строке подвержена ошибкам, поскольку пользователи могут забыть заключить его в кавычки, в результате чего оболочка развернет его во все соответствующие файлы и передаст их все по отдельности в set/add разреженной проверки. Хотя это также может быть проблемой, например, для "git grep — *.c" , в непосредственном выводе появляются ошибки с grep/log/status. При использовании разреженной проверки ошибка фиксируется во время выполнения команды разреженной проверки и может не вызывать проблем до тех пор, пока пользователь позже не переключит ветки, не перебазирует или не слит, тем самым создавая задержку между ошибкой пользователя и моментом, когда у него появится возможность поймать/заметить это.

  • Как и в предыдущем пункте, разреженная проверка имеет подкоманду add , но не имеет подкоманды remove . Даже если была добавлена ​​подкоманда remove , отмена случайного glob без кавычек сопряжена с риском "removing too much" , поскольку может удалить записи, которые были включены до случайного добавления.

  • В режиме без конуса используются шаблоны в стиле gitignore для выбора того, что включать (за исключением отрицаемых шаблонов), тогда как в файлах .gitignore используются шаблоны в стиле gitignore для выбора того, что следует исключить (за исключением отрицаемых шаблонов). В документации по шаблонам в стиле gitignore обычно говорится не о совпадении или несоответствии, а о том, что пользователь хочет от "exclude" . Это может вызвать путаницу у пользователей, пытающихся научиться задавать шаблоны разреженной проверки для достижения желаемого поведения.

  • Любая другая подкоманда git , которая хочет предоставить какой-либо тип "special path pattern matching" , использует спецификации пути, но неконусный режим для разреженной проверки использует шаблоны игнорирования git , что кажется противоречивым.

  • Есть крайние случаи, когда поведение "right" неясно. Два примера:

    First, two users are in a subdirectory, and the first runs
   git sparse-checkout set '/toplevel-dir/*.c'
while the second runs
   git sparse-checkout set relative-dir
Should those arguments be transliterated into
   current/subdirectory/toplevel-dir/*.c и current/subdirectory/relative-dir перед вставкой в ​​файл разреженной проверки? Пользователь, набравший первую команду, вероятно, знает, что аргументы для установки/добавления должны быть шаблонами в неконусном режиме, и, вероятно, не будет доволен такой транслитерацией. Однако многие шаблоны в стиле gitignore — это просто пути, о чем мог думать пользователь, набравший вторую команду, и он был бы расстроен, если бы его аргумент не был транслитерирован.
    Second, what should bash-completion complete on for set/add commands
for non-cone users?  If it suggests paths, is it exacerbating the
problem above?  Also, if it suggests paths, what if the user has a
file or directory that begins with either a '!' or '#' or has a '*',
'\', '?', '[', or ']' in its name?  And if it suggests paths, will
it complete "/pro" to "/proc" (in the root filesystem) rather than to
"/progress.txt" in the current directory?  (Note that users are
likely to want to start paths with a leading '/' in non-cone mode,
for the same reason that .gitignore files often have one.)
Completing on files or directories might give nasty surprises in
all these cases.

  • Излишняя гибкость сделала другие расширения практически непрактичными. --sparse-index , скорее всего, невозможен в неконусном режиме; даже если это каким-то образом осуществимо, реализация потребовала бы гораздо больше усилий и, возможно, на практике была бы слишком медленной. Некоторые идеи по добавлению связи между частичными клонами и разреженными проверками практичны только при более ограниченном наборе путей.

По всем этим причинам неконусный режим устарел. Пожалуйста, переключитесь в режим конуса.

Внутреннее устройство — обработка режима конуса

"cone mode" , используемый по умолчанию, позволяет указать только те каталоги, которые необходимо включить. Для любого указанного каталога будут включены все пути ниже этого каталога, а также все пути непосредственно под ведущими каталогами (включая каталог верхнего уровня). Таким образом, если вы указали каталог Documentation/technical/, то ваша разреженная проверка будет содержать:

  • все файлы в каталоге верхнего уровня

  • все файлы сразу в разделе Документация/

  • все файлы на любой глубине в разделе Documentation/technical/

Кроме того, в режиме конуса, даже если каталоги не указаны, будут включены файлы из каталога верхнего уровня.

При изменении шаблонов разреженной проверки в режиме конуса Git будет проверять каждый отслеживаемый каталог, который находится за пределами конуса разреженной проверки, чтобы определить, содержит ли он какие-либо неотслеживаемые файлы. Если все эти файлы игнорируются из-за шаблонов .gitignore , каталог будет удален. Если какой-либо из неотслеживаемых файлов в этом каталоге не игнорируется, то удаления в этом каталоге не произойдет, и появится предупреждающее сообщение. Если эти файлы важны, сбросьте определение разреженной проверки, чтобы они были включены, используйте git add и git commit для их хранения, затем удалите все оставшиеся файлы вручную, чтобы обеспечить оптимальное поведение Git .

См. также раздел "Internals — Cone Pattern Set" , чтобы узнать, как каталоги преобразуются «под капотом» в подмножество полного набора шаблонов разреженной проверки.

Внутренности — полный набор шаблонов

Полный набор шаблонов допускает произвольные совпадения с шаблонами и сложные правила включения/исключения. Это может привести к совпадению шаблонов O(N*M) при обновлении индекса, где N — количество шаблонов, а M — количество путей в индексе. Чтобы решить эту проблему с производительностью, при включенном core.sparseCheckoutCone допускается более ограниченный набор шаблонов.

Файл разреженной проверки использует тот же синтаксис, что и файлы .gitignore ; подробности см. в gitignore[5] . Однако здесь шаблоны обычно используются для выбора того, какие файлы включать, а не какие файлы исключать. (Однако это может немного сбить с толку, поскольку шаблоны в стиле gitignore имеют отрицания, определенные шаблонами, начинающимися с ! , поэтому вы также можете выбирать файлы для включения not .)

Например, чтобы выбрать все, а затем удалить файл unwanted (чтобы в вашем рабочем дереве появился каждый файл, кроме файла с именем unwanted ):

git sparse-checkout set --no-cone '/*' '!unwanted'

Эти шаблоны просто помещаются в $GIT_DIR/info/sparse-checkout как есть, поэтому содержимое этого файла на данный момент будет

/*
!unwanted

См. также раздел "Sparse Checkout" статьи git-read-tree[1] , чтобы узнать больше о шаблонах в стиле gitignore, используемых в разреженных проверках.

Внутренние детали — набор конусных узоров

В режиме конуса принимаются только каталоги, но они преобразуются в те же шаблоны в стиле gitignore, которые используются в полном наборе шаблонов. Мы относим к конкретным шаблонам, используемым в этом режиме, один из двух типов:

  1. Рекурсивный: включаются все пути внутри каталога.

  2. Родительский: включаются все файлы непосредственно внутри каталога.

Поскольку режим конуса всегда включает файлы верхнего уровня, при запуске git sparse-checkout set без указания каталогов каталог верхнего уровня добавляется в качестве родительского шаблона. На данный момент файл разреженной проверки содержит следующие шаблоны:

/*
!/*/

Здесь написано "include everything immediately under the toplevel directory, but nothing at any level below that."

В режиме конуса субкоманда git sparse-checkout set принимает список каталогов. Команда git sparse-checkout set A/B/C устанавливает каталог A/B/C как рекурсивный шаблон, каталоги A и A/B добавляются в качестве родительских шаблонов. Результирующий файл разреженной проверки теперь имеет вид

/*
!/*/
/A/
!/A/*/
/A/B/
!/A/B/*/
/A/B/C/

Здесь порядок имеет значение, поэтому отрицательные шаблоны переопределяются положительными шаблонами, которые появляются ниже в файле.

Если для core.sparseCheckoutCone явно не установлено значение false , Git будет анализировать файл разреженной проверки, ожидая шаблоны этих типов. Git предупредит, если шаблоны не совпадают. Если шаблоны соответствуют ожидаемому формату, то Git будет использовать более быстрые алгоритмы на основе хеш-функции для вычисления включения в разреженную проверку. Если они не совпадают, git будет вести себя так, как если бы core.sparseCheckoutCone был ложным, независимо от его настройки.

В случае режима конуса, несмотря на то, что полные шаблоны записываются в файл $GIT_DIR/info/sparse-checkout, подкоманда git sparse-checkout list выводит список каталогов, определяющих рекурсивные шаблоны. Для приведенного выше примера файла разреженной проверки вывод выглядит следующим образом:

$ git sparse-checkout list
A/B/C

Если core.ignoreCase=true , то алгоритм сопоставления с образцом будет использовать проверку без учета регистра. Это исправляет несоответствие регистра имен файлов в команде git sparse-checkout set , чтобы отразить ожидаемый конус в рабочем каталоге.

Внутренности — субмодули

Если ваш репозиторий содержит один или несколько подмодулей, то подмодули заполняются на основе взаимодействия с командой git submodule . В частности, git submodule init -- <path> обеспечит наличие субмодуля <path> , а git submodule deinit [-f] -- <path> удалит файлы субмодуля <path> (включая все неотслеживаемые файлы, незафиксированные изменения и неотправленную историю). Подобно тому, как разреженная проверка удаляет файлы из рабочего дерева, но по-прежнему оставляет записи в индексе, деинициализированные подмодули удаляются из рабочего каталога, но все еще имеют запись в индексе.

Поскольку подмодули могут содержать неотправленные изменения или неотслеживаемые файлы, их удаление может привести к потере данных. Таким образом, изменение разреженных правил включения/исключения не приведет к удалению уже извлеченного подмодуля из рабочей копии. Другими словами, так же, как checkout не приведет к автоматическому удалению или инициализации подмодулей даже при переключении между ветвями, которые удаляют или добавляют подмодули, использование sparse-checkout для уменьшения или расширения области файлов "interesting" не приведет к автоматической деинициализации или инициализации подмодулей. .

Кроме того, приведенные выше факты означают, что существует несколько причин, по которым файлы "tracked" могут отсутствовать в рабочей копии: приложение шаблона разреженности из разреженной проверки и состояние инициализации подмодуля. Таким образом, такие команды, как git grep , которые работают с отслеживаемыми файлами в рабочей копии, могут возвращать результаты, ограниченные одним или обоими этими ограничениями.
