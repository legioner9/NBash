git-rm

Name

git-rm — Удалить файлы из рабочего дерева и из индекса

Synopsis

git rm [-f | --force] [-n] [-r] [--cached] [--ignore-unmatch]
          [--quiet] [--pathspec-from-file=<file> [--pathspec-file-nul]]
          [--] [<pathspec>…​]

Description

Удалите файлы, соответствующие указателю пути, из индекса или из рабочего дерева и индекса. git rm не удалит файл только из вашего рабочего каталога. (Невозможно удалить файл только из рабочего дерева и сохранить его в индексе; используйте /bin/rm , если вы хотите это сделать.) Удаляемые файлы должны быть идентичны кончику ветки, и никаких обновлений. к их содержимому можно разместить в индексе, хотя это поведение по умолчанию можно переопределить с помощью параметра -f . Если указан --cached , промежуточный контент должен соответствовать либо кончику ветки, либо файлу на диске, что позволяет удалить файл только из индекса. Когда используются разреженные проверки (см. git-sparse-checkout[1] ), git rm будет удалять только пути внутри шаблонов разреженных проверок.

Options

<pathspec>…

Файлы для удаления. Для удаления всех файлов в каталоге и рекурсивно всех подкаталогов можно указать ведущее имя каталога (например, dir для удаления dir/file1 и dir/file2 ), но для этого необходимо явно указать параметр -r .

Команда удаляет только те пути, которые известны Git.

Подстановка файлов соответствует границам каталогов. Таким образом, учитывая два каталога d и d2 , существует разница между использованием git rm 'd*' и git rm 'd/*' , поскольку первый также удалит весь каталог d2 .

Дополнительные сведения см. в записи pathspec в gitglossary[7] .

-f
--force

Переопределить проверку актуальности.

-n
--dry-run

На самом деле не удаляйте файлы. Вместо этого просто покажите, существуют ли они в индексе и в противном случае они были бы удалены командой.

-r

Разрешить рекурсивное удаление, если указано ведущее имя каталога.

--

Эту опцию можно использовать для отделения параметров командной строки от списка файлов (полезно, когда имена файлов могут быть ошибочно приняты за параметры командной строки).

--cached

Используйте эту опцию, чтобы отключить и удалить пути только из индекса. Файлы рабочего дерева, независимо от того, изменены они или нет, останутся в покое.

--ignore-unmatch

Выйти с нулевым статусом, даже если ни один файл не соответствует.

--sparse

Разрешить обновление записей индекса за пределами конуса разреженной проверки. Обычно git rm отказывается обновлять записи индекса, пути которых не укладываются в конус разреженной проверки. Дополнительную информацию см. в git-sparse-checkout[1] .

-q
--quiet

git rm обычно выводит одну строку (в виде команды rm ) для каждого удаленного файла. Эта опция подавляет этот вывод.

--pathspec-from-file=<file>

Спецификация пути передается в <file> вместо аргументов командной строки. Если <file> — это точно - , то используется стандартный ввод. Элементы Pathspec разделяются LF или CR/LF. Элементы Pathspec можно заключать в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). См. также --pathspec-file-nul и глобальный --literal-pathspecs .

--pathspec-file-nul

Имеет смысл только с --pathspec-from-file . Элементы Pathspec разделяются символом NUL, а все остальные символы воспринимаются буквально (включая символы новой строки и кавычки).

Удаление файлов, которые исчезли из файловой системы

У git rm нет возможности удалять из индекса только те пути, которые исчезли из файловой системы. Однако в зависимости от варианта использования можно сделать несколько способов.

Использование «git commit -a»

Если вы предполагаете, что ваш следующий коммит должен записывать все изменения отслеживаемых файлов в рабочем дереве и записывать все удаления файлов, которые были удалены из рабочего дерева с помощью rm (в отличие от git rm ), используйте git commit -a , поскольку он автоматически заметит и записывать все удаления. Вы также можете получить аналогичный эффект без каких-либо коммитов, используя git add -u .

Использование «git add -A»

Принимая новый выпуск кода для ветки поставщика, вы, вероятно, захотите записать как удаление путей и добавление новых путей, так и изменения существующих путей.

Обычно вы сначала удаляете все отслеживаемые файлы из рабочего дерева с помощью этой команды:

git ls-files -z | xargs -0 rm -f

а затем распакуйте новый код в рабочем дереве. Альтернативно вы можете rsync внести изменения в рабочее дерево.

После этого самый простой способ зафиксировать все удаления, дополнения и изменения в рабочем дереве:

git add -A

See git-add[1].

Other ways

Если все, что вам действительно нужно, — это удалить из индекса файлы, которых больше нет в рабочем дереве (возможно, потому, что ваше рабочее дерево загрязнено и вы не можете использовать git commit -a ), используйте следующую команду:

git diff --name-only --diff-filter=D -z | xargs -0 git rm --cached

Submodules

Только подмодули, использующие файл git (что означает, что они были клонированы с помощью Git версии 1.7.8 или новее), будут удалены из рабочего дерева, поскольку их репозиторий находится внутри файла . git каталог суперпроекта. Если подмодуль (или один из вложенных в него) по-прежнему использует файл . git , git rm переместит каталог подмодулей git в каталог суперпроектов git , чтобы защитить историю подмодуля. Если он существует, раздел submodule.<name> в файле gitmodules[5] также будет удален, и этот файл будет проиндексирован (если не используются --cached или -n).

Субмодуль считается обновленным, если HEAD совпадает с записанным в индексе, отслеживаемые файлы не изменяются и в рабочем дереве подмодуля отсутствуют неотслеживаемые файлы, которые не игнорируются. Игнорируемые файлы считаются расходуемыми и не предотвращают удаление рабочего дерева подмодуля.

Если вы хотите удалить только локальное извлечение подмодуля из вашего рабочего дерева, не фиксируя удаление, вместо этого используйте git-submodule[1] deinit . Также см. gitsubmodules[7] для получения подробной информации об удалении субмодуля.

Examples

git rm Documentation/\*.txt

Удаляет из индекса все файлы *.txt , находящиеся в каталоге Documentation и любом из его подкаталогов.

Обратите внимание, что в этом примере звездочка * цитируется из оболочки; это позволяет Git, а не оболочке, расширять пути к файлам и подкаталогам в каталоге Documentation/ .

git rm -f git-*.sh

Поскольку этот пример позволяет оболочке расширить звездочку (т. е. вы явно указываете файлы), он не удаляет subdir/git-foo.sh .

Bugs

Каждый раз, когда обновление суперпроекта удаляет заполненный подмодуль (например, при переключении между фиксациями до и после удаления), извлечение устаревшего подмодуля останется в старом месте. Удаление старого каталога безопасно только в том случае, если он использует git-файл, иначе история подмодуля также будет удалена. Этот шаг станет устаревшим, когда будет реализовано рекурсивное обновление подмодуля.
