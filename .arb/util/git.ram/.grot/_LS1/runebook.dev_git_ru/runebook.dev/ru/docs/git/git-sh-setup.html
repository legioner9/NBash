git-sh-setup

Name

git-sh-setup — общий код установки сценария оболочки Git

Synopsis

. "$(git --exec-path)/git-sh-setup"

Description

Это не та команда, которую хотел бы выполнить конечный пользователь. Всегда. Эта документация предназначена для людей, которые изучают сценарии Porcelain и/или пишут новые.

Сценарий git sh-setup предназначен для использования (с использованием . ) из других сценариев оболочки для настройки некоторых переменных, указывающих на обычные каталоги Git , и нескольких вспомогательных функций оболочки.

Прежде чем использовать его, ваш скрипт должен настроить несколько переменных; USAGE (и LONG_USAGE , если таковой имеется) используется для определения сообщения, выдаваемого функцией оболочки usage() . SUBDIRECTORY_OK можно установить, если сценарий может запускаться из подкаталога рабочего дерева (некоторые команды этого не делают).

Скриптлет устанавливает переменные оболочки GIT_DIR и GIT_OBJECT_DIRECTORY , но не экспортирует их в среду.

Functions

die

выйти после отправки предоставленного сообщения об ошибке в стандартный поток ошибок.

usage

умереть с сообщением об использовании.

set_reflog_action

Установите для среды GIT_REFLOG_ACTION заданную строку (обычно имя программы), если она еще не установлена. Всякий раз, когда сценарий запускает команду git , обновляющую ссылки, создается запись в журнале ссылок, использующая значение этой строки, чтобы оставить запись о том, какая команда обновила ссылку.

git_editor

запускает редактор по выбору пользователя (GIT_EDITOR, core.editor, VISUAL или EDITOR) для данного файла, но выдает ошибку, если редактор не указан и терминал тупой.

is_bare_repository

выводит true или false в стандартный поток вывода, чтобы указать, является ли репозиторий пустым (т. е. без связанного рабочего дерева).

cd_to_toplevel

запускает chdir на верхний уровень рабочего дерева.

require_work_tree

проверяет, находится ли текущий каталог в рабочем дереве репозитория, и в противном случае умирает.

require_work_tree_exists

проверяет, существует ли рабочее дерево, связанное с репозиторием, и в противном случае умирает. Часто делается перед вызовом cd_to_toplevel, что невозможно сделать, если нет рабочего дерева.

require_clean_work_tree <действие> [<подсказка>]

проверяет, что рабочее дерево и индекс, связанные с репозиторием, не содержат незафиксированных изменений в отслеживаемых файлах. В противном случае он выдает сообщение об ошибке вида Cannot <action>: <reason>. <hint> и умирает. Пример:

require_clean_work_tree rebase "Please commit or stash them."
get_author_ident_from_commit

выводит код для использования с eval для установки переменных GIT_AUTHOR_NAME, GIT_AUTHOR_EMAIL и GIT_AUTHOR_DATE для данного коммита.

create_virtual_base

изменяет первый файл так, что остаются только общие строки со вторым файлом. Если общего материала недостаточно, то первый файл остается пустым. Результат подходит в качестве виртуального базового входа для трехстороннего слияния.
