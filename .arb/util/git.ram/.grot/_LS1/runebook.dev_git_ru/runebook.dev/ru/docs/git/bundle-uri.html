bundle-uri

Пакеты Git — это файлы, в которых хранится пакетный файл вместе с некоторыми дополнительными метаданными, включая набор ссылок и (возможно, пустой) набор необходимых коммитов. Дополнительную информацию см. в git-bundle[1] и gitformat-bundle[5] .

URI пакета — это места, где Git может загрузить один или несколько пакетов, чтобы загрузить базу данных объектов перед получением остальных объектов с удаленного устройства.

Одна из целей — ускорить клонирование и выборку для пользователей с плохим сетевым подключением к исходному серверу. Еще одним преимуществом является предоставление возможности активным пользователям, таким как фермы CI-сборки, использовать локальные ресурсы для большинства данных Git и тем самым снизить нагрузку на исходный сервер.

Чтобы включить функцию URI пакета, пользователи могут указать URI пакета с помощью параметров командной строки, или исходный сервер может объявить один или несколько URI с помощью возможности протокола v2.

Design goals

Стандарт URI пакета стремится быть достаточно гибким, чтобы удовлетворить несколько рабочих нагрузок. Поставщик пакета и клиент Git имеют несколько вариантов создания и использования URI пакета.

  • Пакеты могут иметь любое имя, какое пожелает сервер. Это имя может относиться к неизменяемым данным, используя хэш содержимого пакета. Однако это означает, что новый URI потребуется после каждого обновления контента. Это может быть приемлемо, если сервер объявляет URI (и сервер знает о создании новых пакетов), но не будет эргономично для пользователей, использующих параметр командной строки.

  • Пакеты могут быть организованы специально для начальной загрузки полных клонов, но также могут быть организованы с целью начальной начальной загрузки дополнительных выборок. Поставщик пакетов должен выбрать одну из нескольких схем организации, чтобы минимизировать загрузку клиента во время добавочной выборки, но клиент Git также может выбрать, использовать ли пакеты для любой из этих операций.

  • Поставщик пакета может выбрать поддержку полных клонов, частичных клонов или того и другого. Клиент может определить, какие пакеты подходят для фильтра частичного клонирования репозитория, если таковой имеется.

  • Поставщик пакетов может использовать один пакет (только для клонов) или список пакетов. При использовании списка пакетов поставщик может указать, нужен ли клиенту all URI пакета для полного клонирования или any достаточно одного из URI пакета. Это позволяет поставщику пакетов использовать разные URI для разных географических регионов.

  • Поставщик пакетов может организовывать пакеты с помощью эвристики, например токенов создания, чтобы помочь клиенту предотвратить загрузку пакетов, которые ему не нужны. Если поставщик пакета не предоставляет эту эвристику, клиент может использовать оптимизацию, чтобы минимизировать объем загружаемых данных.

  • Поставщик пакета не обязательно должен быть связан с сервером Git . Клиент может использовать поставщика пакетов без его объявления сервером Git .

  • Клиент может выбрать поставщиков пакетов, рекламируемых сервером Git . Это могло произойти во время git clone , во время git fetch , в обоих случаях или ни в одном из них. Пользователь может выбрать, какая комбинация лучше всего ему подходит.

  • Клиент может в любое время настроить поставщика пакетов вручную. Клиент также может указать поставщика пакетов вручную в качестве параметра командной строки git clone .

Каждый репозиторий уникален, и у каждого сервера Git разные потребности. Надеемся, что функция URI пакета достаточно гибкая, чтобы удовлетворить все потребности. Если нет, то эту функцию можно расширить с помощью механизма управления версиями.

Server requirements

Для реализации серверной реализации пакетных серверов никакие другие части протокола Git не требуются. Это позволяет специалистам по обслуживанию серверов использовать контент-решения static , такие как CDN, для обслуживания файлов пакетов.

В текущей области действия функции URI пакета ожидается, что все URI будут URL-адресами HTTP(S), где контент загружается в локальный файл с использованием запроса GET к этому URL-адресу. Сервер может включать требования аутентификации в эти запросы с целью запуска настроенного помощника по учетным данным для безопасного доступа. (В будущих расширениях могут использоваться URI "file://" или URI SSH.)

Предполагая ответ 200 OK от сервера, проверяется содержимое URL-адреса. Сначала Git пытается проанализировать файл как файл пакета версии 2 или выше. Если файл не является пакетом, он анализируется как обычный текстовый файл с помощью анализатора конфигурации Git . Ожидается, что пары ключ-значение в этом файле конфигурации будут описывать список URI пакета. Если ни одна из этих попыток анализа не увенчалась успехом, Git сообщит пользователю об ошибке, что URI пакета предоставил ошибочные данные.

Любые другие данные, предоставленные сервером, считаются ошибочными.

Bundle lists

Сервер Git может объявлять URI пакета, используя набор пар key=value . URI пакета также может обслуживать простой текстовый файл в формате конфигурации Git , содержащий те же самые пары key=value . В обоих случаях мы считаем это bundle list . Пары определяют информацию о пакетах, которую клиент может использовать для принятия решения, какие пакеты загружать, а какие игнорировать.

Несколько ключей посвящены свойствам самого списка.

bundle.version

(Обязательно) Это значение предоставляет номер версии списка пакетов. Если будущее изменение Git активирует функцию, требующую, чтобы клиент Git реагировал на новый ключ в файле списка пакетов, то эта версия будет увеличиваться. Единственный текущий номер версии — 1, и если указано любое другое значение, Git не сможет использовать этот файл.

bundle.mode

(Обязательно) Это значение имеет одно из двух значений: all и any . Если указан all , клиенту следует ожидать, что ему потребуются все перечисленные URI пакета, соответствующие требованиям его репозитория. Если указан any , клиент должен ожидать, что будет достаточно любого из URI пакета, соответствующего требованиям его репозитория. Обычно опция any используется для перечисления нескольких различных серверов пакета, расположенных в разных регионах.

bundle.heuristic

Если этот ключ со строковым значением существует, то список пакетов предназначен для хорошей работы с инкрементными командами git fetch . Эвристика сигнализирует о том, что для каждого пакета доступны дополнительные ключи, которые помогают определить, какое подмножество пакетов следует загрузить клиенту. Единственная запланированная на данный момент эвристика — creationToken .

Остальные ключи включают сегмент <id> , который представляет собой назначенное сервером имя для каждого доступного пакета. <id> должен содержать только буквенно-цифровые символы и символы - .

bundle.<id>.uri

(Обязательно) Это строковое значение является URI для загрузки пакета <id> . Если URI начинается с протокола ( http:// или https:// ), тогда URI является абсолютным. В противном случае URI интерпретируется как относительный URI, используемый для списка пакетов. Если URI начинается с / , то этот относительный путь относится к имени домена, используемому для списка пакетов. (Такое использование относительных путей предназначено для упрощения распространения набора пакетов по большому количеству серверов или CDN с разными доменными именами.)

bundle.<id>.filter

Это строковое значение представляет фильтр объектов, который также должен появиться в заголовке этого пакета. Сервер использует это значение для различения различных типов пакетов, из которых клиент может выбирать те, которые соответствуют его фильтрам объектов.

bundle.<id>.creationToken

Это значение представляет собой неотрицательное 64-битное целое число, используемое для сортировки списка пакетов. Это используется для загрузки подмножества пакетов во время выборки, когда bundle.heuristic=creationToken .

bundle.<id>.location

Это строковое значение объявляет реальное местоположение, откуда обслуживается URI пакета. Это можно использовать, чтобы предоставить пользователю возможность использовать URI пакета или просто в качестве информативного индикатора того, какой URI пакета был выбран Git. Это полезно только в том случае, если bundle.mode равен any .

Вот пример списка пакетов с использованием формата конфигурации Git :

[bundle]
        version = 1
        mode = all
        heuristic = creationToken
[bundle "2022-02-09-1644442601-daily"]
        uri = https://bundles.example.com/git/git/2022-02-09-1644442601-daily.bundle
        creationToken = 1644442601
[bundle "2022-02-02-1643842562"]
        uri = https://bundles.example.com/git/git/2022-02-02-1643842562.bundle
        creationToken = 1643842562
[bundle "2022-02-09-1644442631-daily-blobless"]
        uri = 2022-02-09-1644442631-daily-blobless.bundle
        creationToken = 1644442631
        filter = blob:none
[bundle "2022-02-02-1643842568-blobless"]
        uri = /git/git/2022-02-02-1643842568-blobless.bundle
        creationToken = 1643842568
        filter = blob:none

В этом примере используется bundle.mode=all , а также эвристика bundle.<id>.creationToken . Он также использует параметры bundle.<id>.filter для представления двух параллельных наборов пакетов: один для полных клонов, а другой для частичных клонов без больших двоичных объектов.

Предположим, что этот список пакетов был найден по URI https://bundles.example.com/git/git/ , и поэтому два пакета без больших двоичных объектов имеют следующие полностью развернутые URI:

  • https://bundles.example.com/git/git/2022-02-09-1644442631-daily-blobless.bundle

  • https://bundles.example.com/git/git/2022-02-02-1643842568-blobless.bundle

Рекламный пакет uris

Если пользователь знает URI пакета для клонируемого репозитория, он может указать этот URI вручную с помощью параметра командной строки. Однако хост Git может захотеть объявить URI пакета во время операции клонирования, помогая пользователям, не знающим об этой функции.

Единственное, что требуется для этой функции, — это то, чтобы сервер мог объявлять один или несколько URI пакета. Это объявление принимает форму новой возможности протокола v2, специально предназначенной для обнаружения URI пакетов.

Клиент может выбрать произвольный URI пакета в качестве опции. or выбирает URI с наилучшей производительностью путем некоторых предварительных проверок. Поставщик пакета должен решить, предпочтительнее ли наличие нескольких URI по сравнению с одним URI, который геораспределяется через инфраструктуру на стороне сервера.

Клонирование с помощью пакета uris

Основная потребность в пакетных URI — ускорение клонирования. Клиент Git будет взаимодействовать с URI пакета согласно следующему алгоритму:

  1. Пользователь указывает URI пакета с помощью параметра командной строки --bundle-uri or , клиент обнаруживает список пакетов, объявленный сервером Git .

  2. Если загруженные данные из URI пакета являются пакетом, клиент проверяет заголовки пакета, чтобы убедиться, что необходимые OID фиксации присутствуют в репозитории клиента. Если некоторые из них отсутствуют, клиент откладывает разделение до тех пор, пока не будут разделены другие пакеты, в результате чего эти OID появятся. Когда присутствуют все необходимые OID, клиент разделяет эти данные с помощью refspec. Спецификация по умолчанию — +refs/heads/*:refs/bundles/* , но это можно настроить. Эти ссылки сохраняются, чтобы при последующих согласованиях git fetch можно было передавать каждую объединенную ссылку как have , уменьшая размер выборки по протоколу Git . Чтобы разрешить удаление ссылок из этого пространства имен ссылок, Git может ввести пронумерованное пространство имен (например, refs/bundles/<i>/* ), чтобы можно было удалить устаревшие ссылки на пакеты.

  3. Если вместо этого файл представляет собой список пакетов, клиент проверяет bundle.mode , чтобы определить, имеет ли список форму all или any .

    1. Если bundle.mode=all , клиент учитывает все URI пакета. Список сокращается на основе параметров bundle.<id>.filter , соответствующих фильтру частичного клонирования клиентского репозитория. Затем запрашиваются все URI пакета. Если предоставлена ​​эвристика bundle.<id>.creationToken , пакеты загружаются в порядке убывания токена создания и останавливаются, когда пакет имеет все необходимые OID. Затем пакеты можно разделить в порядке увеличения порядка создания токенов. Клиент сохраняет последний токен создания в качестве эвристики, чтобы избежать будущих загрузок, если в списке пакетов не рекламируются пакеты с более крупными токенами создания.

    2. Если bundle.mode=any , клиент может выбрать для проверки любой из URI пакета. Клиент может использовать различные способы выбора среди этих URI. Клиент также может вернуться к другому URI, если первоначальный выбор не возвращает результат.

Обратите внимание, что во время клонирования мы ожидаем, что потребуются все пакеты, и эвристики, такие как bundle.<uri>.creationToken , можно использовать для загрузки пакетов в хронологическом порядке или параллельно.

Если данный URI пакета представляет собой список пакетов со значением bundle.heuristic , то клиент может выбрать сохранение этого URI в качестве выбранного URI пакета. Затем клиент может перейти непосредственно к этому URI во время последующих вызовов git fetch .

При загрузке URI пакета клиент может проверить исходный контент, прежде чем приступить к загрузке всего контента. Это может предоставить достаточно информации, чтобы определить, является ли URI списком пакетов или пакетом. В случае пакета клиент может проверить заголовок пакета, чтобы определить, что все рекламируемые советы уже находятся в репозитории клиента, и отменить оставшуюся загрузку.

Извлечение с помощью пакета URI

Когда клиент извлекает новые данные, он может решить получить их с серверов пакетов, прежде чем получать их с удаленного источника. Это можно сделать с помощью параметра командной строки, но более вероятно будет полезно использовать значение конфигурации, например, указанное во время клонирования.

Операция выборки следует той же процедуре, что и загрузка пакетов из списка пакетов (хотя мы хотим, чтобы not использовал здесь параллельную загрузку). Мы ожидаем, что процесс завершится, когда все необходимые OID фиксации в тонком пакете уже будут в базе данных объектов.

При использовании эвристики creationToken клиент может избежать загрузки каких-либо пакетов, если их токены создания не больше, чем сохраненный токен создания. После получения новых пакетов Git обновляет этот локальный токен создания.

Если поставщик пакета не предоставляет эвристику, то клиент должен попытаться проверить заголовки пакета перед загрузкой полных данных пакета на случай, если подсказки пакета уже существуют в репозитории клиента.

Error conditions

Если клиент Git обнаруживает что-то неожиданное при загрузке информации в соответствии с URI пакета или списком пакетов, найденным в этом месте, то Git может игнорировать эти данные и продолжить работу, как если бы ему не был предоставлен URI пакета. Удаленный сервер Git является окончательным источником достоверной информации, а не URI пакета.

Вот несколько примеров ошибок:

  • Клиенту не удается подключиться к серверу по заданному URI или соединение теряется без возможности восстановления.

  • Клиент получает ответ уровня 400 (например, 404 Not Found или 401 Not Authorized ). Клиент должен использовать помощник по учетным данным, чтобы найти и предоставить учетные данные для URI, но при этом соответствовать семантике других HTTP-протоколов Git с точки зрения обработки конкретных ошибок уровня 400.

  • Сервер сообщает о любом другом ответе об ошибке.

  • Клиент получает данные, которые невозможно проанализировать как пакет или список пакетов.

  • В комплект входит фильтр, который не соответствует ожиданиям.

  • Клиент не может разделить пакеты, поскольку необходимые OID фиксации отсутствуют в базе данных объектов и больше нет пакетов для загрузки.

Существуют также ситуации, которые можно рассматривать как расточительность, но они не являются ошибками:

  • Загруженные пакеты содержат больше информации, чем запрашивается запросом клонирования или выборки. Основной пример: пользователь запрашивает клон с помощью --single-branch , но загружает пакеты, в которых хранятся все доступные фиксации из всех ссылок refs/heads/* . Поначалу это может быть расточительно, но, возможно, эти объекты станут доступными при более позднем обновлении ссылки, которое волнует клиента.

  • Загрузка пакета во время git fetch содержит объекты, уже находящиеся в базе данных объектов. Это, вероятно, неизбежно, если мы используем пакеты для выборки, поскольку клиент почти всегда будет немного опережать серверы пакетов после выполнения выборки "catch-up" на удаленный сервер. Эта дополнительная работа наиболее расточительна, когда клиент выполняет выборку гораздо чаще, чем сервер обрабатывает пакеты, например, если клиент использует ежечасную предварительную выборку с фоновым обслуживанием, а сервер обрабатывает пакеты еженедельно. По этой причине клиент не должен использовать URI пакета для выборки, если сервер явно не рекомендовал это через значение bundle.heuristic .

Пример организации поставщика пакетов

Функция URI пакета специально разработана так, чтобы обеспечить гибкость в зависимости от того, как поставщик пакета хочет организовать данные объекта. Однако может оказаться полезным иметь описанную здесь полную модель организации, чтобы поставщики могли начать с этой основы.

Этот пример организации представляет собой упрощенную модель того, что используется кэш-серверами GVFS (см. раздел в конце этого документа), которые помогают ускорить клонирование и выборку для очень больших репозиториев, хотя и используют дополнительное программное обеспечение вне Git.

Поставщик пакетов развертывает серверы в нескольких регионах. Каждый сервер управляет своим собственным набором пакетов. Сервер может отслеживать несколько репозиториев Git , но предоставляет список пакетов для каждого на основе шаблона. Например, при зеркалировании репозитория по адресу https://<domain>/<org>/<repo> сервер пакетов может иметь список пакетов, доступный по адресу https://<server-url>/<domain>/<org>/<repo> . Исходный сервер Git может перечислить все эти серверы в режиме "any" :

[bundle]
        version = 1
        mode = any
[bundle "eastus"]
        uri = https://eastus.example.com/<domain>/<org>/<repo>
[bundle "europe"]
        uri = https://europe.example.com/<domain>/<org>/<repo>
[bundle "apac"]
        uri = https://apac.example.com/<domain>/<org>/<repo>

Этот "list of lists" является static и изменяется только в том случае, если добавляется или удаляется пакетный сервер.

Каждый сервер пакетов управляет своим собственным набором пакетов. Исходный список пакетов содержит только один пакет, содержащий все объекты, полученные в результате клонирования репозитория с исходного сервера. В списке используется эвристика creationToken , а creationToken создается для пакета на основе временной метки сервера.

Сервер пакетов регулярно выполняет обновления списка пакетов, например, раз в день. Во время этой задачи сервер извлекает последнее содержимое с исходного сервера и генерирует пакет, содержащий объекты, доступные из последних ссылок на источник, но не содержащиеся в ранее вычисленном пакете. Эта связка добавлена ​​в список с учетом того, чтобы creationToken был строго больше предыдущего максимума creationToken .

Когда список пакетов становится слишком большим, скажем, более 30 пакетов, самые старые пакеты " N minus 30" объединяются в один пакет. creationToken этого пакета равен максимальному creationToken среди объединенных пакетов.

Здесь приведен пример списка пакетов, хотя в нем есть только два ежедневных пакета, а не полный список из 30:

[bundle]
        version = 1
        mode = all
        heuristic = creationToken
[bundle "2022-02-13-1644770820-daily"]
        uri = https://eastus.example.com/<domain>/<org>/<repo>/2022-02-09-1644770820-daily.bundle
        creationToken = 1644770820
[bundle "2022-02-09-1644442601-daily"]
        uri = https://eastus.example.com/<domain>/<org>/<repo>/2022-02-09-1644442601-daily.bundle
        creationToken = 1644442601
[bundle "2022-02-02-1643842562"]
        uri = https://eastus.example.com/<domain>/<org>/<repo>/2022-02-02-1643842562.bundle
        creationToken = 1643842562

Чтобы избежать постоянного хранения и обслуживания данных объекта, даже если они становятся недоступными на исходном сервере, это объединение пакетов может быть более осторожным. Вместо абсолютного объединения старых пакетов можно создать пакет, просматривая новые пакеты и гарантируя, что все их необходимые коммиты доступны в этом объединенном пакете (или в другом из более новых пакетов). Это позволяет использовать данные объекта "expiring" , которые не используются новыми коммитами в этом окне времени. Эти данные могут быть повторно представлены позже.

Цель такой организации данных преследует две основные цели. Во-первых, первоначальные клоны репозитория работают быстрее за счет загрузки предварительно вычисленных данных объекта из более близкого источника. Во-вторых, команды git fetch могут выполняться быстрее, особенно если клиент не получал данные в течение нескольких дней. Однако если клиент не получает данные в течение 30 дней, то организация списка пакетов приведет к повторной загрузке большого объема данных объекта.

Один из способов сделать эту организацию более полезной для пользователей, которые часто получают данные, — это чаще создавать пакеты. Например, пакеты можно создавать каждый час, а затем раз в день эти пакеты "hourly" можно объединять в пакет "daily" . Ежедневные пакеты объединяются с самым старым пакетом через 30 дней.

Рекомендуется повторить эту стратегию пакета с фильтром blob:none , если клиенты этого репозитория планируют использовать частичные клоны без больших двоичных объектов. Этот список пакетов без больших двоичных объектов остается в том же списке, что и полные пакеты, но для разделения двух групп используется ключ bundle.<id>.filter . Для очень больших репозиториев поставщик пакетов может захотеть only предоставить пакеты без больших двоичных объектов.

Implementation plan

Этот проектный документ подается отдельно как желательный документ с целью реализации всех упомянутых клиентских функций в течение нескольких серий патчей. Вот потенциальный план отправки этих функций:

  1. Интегрируйте URI пакета в git clone с помощью опции --bundle-uri . Это будет включать новый режим git fetch --bundle-uri , который будет использоваться в качестве реализации под git clone . Первоначальная версия здесь ожидает один пакет по данному URI.

  2. Реализовать возможность анализа списка пакетов из URI пакета и обновить логику git fetch --bundle-uri , чтобы правильно различать параметры bundle.mode . Специально спроектируйте эту функцию так, чтобы анализ формата конфигурации передавал список пар ключ-значение в логику списка пакетов.

  3. Создайте команду протокола bundle-uri v2, чтобы серверы Git могли объявлять URI пакета с использованием пар ключ-значение. Подключите существующие входные данные «ключ-значение» к логике списка пакетов. Разрешите git clone обнаружить эти URI пакета и загрузить клиентский репозиторий на основе данных пакета. (Этот выбор осуществляется с помощью параметра конфигурации и параметра командной строки.)

  4. Позвольте клиенту понять ключ конфигурации bundle.heuristic и эвристику bundle.<id>.creationToken . Когда git clone обнаруживает URI пакета с помощью bundle.heuristic , он настраивает клиентский репозиторий для проверки этого URI пакета во время последующих команд git fetch <remote> .

  5. Разрешить клиентам обнаруживать URI пакета во время git fetch и настраивать URI пакета для последующих выборок, если установлен bundle.heuristic .

  6. Внедрите эвристику "inspect headers" , чтобы уменьшить загрузку данных, когда эвристика bundle.<id>.creationToken недоступна.

По мере проверки этих функций этот план может быть обновлен. Мы также ожидаем, что новые разработки будут обнаружены и реализованы по мере того, как эта функция будет развиваться и использоваться в реальных сценариях.

Сопутствующая работа: пакетный файл uris

Протокол Git уже имеет возможность, позволяющую серверу Git перечислять набор URL-адресов вместе с ответом на пакетный файл при обслуживании запроса клиента. Затем ожидается, что клиент загрузит пак-файлы в этих местах, чтобы получить полное представление об ответе.

Этот механизм используется сервером Gerrit (реализован с помощью JGit) и эффективен для снижения нагрузки на процессор и повышения производительности пользователей при работе с клонами.

Основным недостатком этого механизма является то, что исходный сервер должен знать exactly , что находится в этих пакетных файлах, а пакетные файлы должны быть доступны пользователю в течение некоторого времени после ответа сервера. Эту связь между источником и данными пакфайла трудно контролировать.

Кроме того, эту реализацию чрезвычайно сложно заставить работать с выборками.

Сопутствующая работа: серверы кэширования gvfs

Протокол GVFS [2] представляет собой набор конечных точек HTTP, разработанных независимо от проекта Git до создания частичного клона Git . Одной из особенностей этого протокола является идея "cache server" , который можно разместить рядом со сборочными машинами или офисами разработчиков для передачи данных Git без перегрузки центрального сервера.

Конечная точка, которой славится VFS для Git , — это конечная точка GET /gvfs/objects/{oid} , которая позволяет загружать объект по требованию. Это важная часть виртуализации файловой системы этого продукта.

Однако более тонкая потребность — это конечная точка GET /gvfs/prefetch?lastPackTimestamp=<t> . При наличии необязательной временной метки кэш-сервер отвечает списком предварительно вычисленных пакетных файлов, содержащих коммиты и деревья, которые были введены в эти интервалы времени.

Кэш-сервер вычисляет эти пак-файлы "prefetch" , используя следующую стратегию:

  1. Каждый час создается пакет "hourly" с заданной меткой времени.

  2. Каждую ночь предыдущие 24-часовые пакеты объединяются в пакет "daily" .

  3. Каждую ночь все пакеты предварительной выборки старше 30 дней объединяются в один пакет.

Когда пользователь запускает gvfs clone или scalar clone в репозитории с кэш-серверами, клиент запрашивает все пакетные файлы предварительной выборки, то есть максимум пакетные файлы 24 + 30 + 1 загружают только коммиты и деревья. Затем клиент отправляет запрос на исходный сервер для получения ссылок и пытается получить эту ссылку на подсказку. (Существует дополнительная конечная точка, которая помогает получить все доступные деревья из данного коммита, если этот коммит еще не был в пакетном файле предварительной выборки.)

Во время git fetch перехватчик запрашивает конечную точку предварительной выборки, используя самую последнюю временную метку из ранее загруженного пакетного файла предварительной выборки. Загружается только список паковочных файлов с более поздними временными метками. Большинство пользователей получают данные ежечасно, поэтому они получают не более одного пакета предварительной загрузки каждый час. Пользователи, чьи компьютеры были выключены или по какой-либо другой причине не загружались в течение более 30 дней, могут повторно загрузить все файлы пакетов предварительной выборки. Это редкость.

Важно отметить, что клиенты всегда обращаются к исходному серверу за объявлением ссылок, поэтому ссылки часто представляют собой "ahead" предварительно выбранных данных пакета. Недостающие объекты загружаются по требованию с помощью запросов GET gvfs/objects/{oid} , когда это необходимо такой командой, как git checkout или git log . Некоторые оптимизации Git отключают проверки, которые могут привести к слишком агрессивной загрузке по требованию.
