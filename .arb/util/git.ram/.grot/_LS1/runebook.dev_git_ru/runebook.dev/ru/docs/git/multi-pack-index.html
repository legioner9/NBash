multi-pack-index

Каталог объектов Git содержит каталог pack , содержащий файлы пакетов (с суффиксом ".pack" ) и индексы пакетов (с суффиксом ".idx" ). Индексы пакета предоставляют возможность поиска объектов и перехода к их смещению внутри пакета, но они должны идти в паре с файлами пакета. Это соединение зависит от имен файлов, поскольку индекс пакета отличается от его файла пакета только суффиксом. Хотя индексы пакетов обеспечивают быстрый поиск каждого файла пакета, эта производительность снижается по мере увеличения количества файлов пакета, поскольку сокращениям необходимо проверять каждый файл пакета, и мы с большей вероятностью пропустим наш последний использованный файл пакета. Для некоторых больших репозиториев переупаковка в один пакетный файл невозможна из-за нехватки места для хранения или чрезмерного времени переупаковки.

Мультипак-индекс (сокращенно MIDX) хранит список объектов и их смещений в нескольких паковочных файлах. Это содержит:

  • Список имен пакфайлов.

  • Отсортированный список идентификаторов объектов.

  • Список метаданных для i-го идентификатора объекта, включая:

    • Значение j, относящееся к j-му пакетному файлу.

    • Смещение в j-м пакетном файле объекта.

  • Если требуются большие смещения, мы используем другой список больших смещений, аналогичный индексам пакетов версии 2.

    • Необязательный список объектов в псевдопакетном порядке (используется с растровыми изображениями MIDX).

Таким образом, мы можем обеспечить время поиска O(log N) для любого количества паковочных файлов.

Design details

  • MIDX хранится в файле с именем multi-pack-index в каталоге .git/objects/pack. Это может быть сохранено в каталоге пакета альтернативного пакета. Это относится только к паковочным файлам в том же каталоге.

  • Для использования файлов MIDX параметр конфигурации core.multiPackIndex должен быть включен (это значение по умолчанию). Установка значения false не позволяет Git читать файл MIDX, даже если он существует.

  • Формат файла включает параметры для хэш-функции идентификатора объекта, поэтому будущее изменение алгоритма хэширования не требует изменения формата.

  • MIDX хранит только одну запись для каждого идентификатора объекта. Если объект присутствует в нескольких пакетных файлах, то MIDX выбирает копию в предпочитаемом пакетном файле, в противном случае — из последнего измененного пакетного файла.

  • Если в каталоге пакета существуют файлы пакетов, не зарегистрированные в MIDX, то эти файлы пакетов загружаются в список packed_git и кэш packed_git_mru .

  • Индексы пакетов (файлы .idx) остаются в каталоге пакета, поэтому мы можем удалить файл MIDX, установить для core.midx значение false или перейти на более раннюю версию без потери информации.

  • Формат файла MIDX использует подход на основе фрагментов (аналогичный файлу графа фиксации), который позволяет добавлять дополнительные данные.

Future work

  • Индекс multi-pack позволяет использовать множество пак-файлов, особенно в контексте, где переупаковка обходится дорого (например, при очень большом репозитории) или когда неожиданное время обслуживания неприемлемо (например, на машине сборки с высоким спросом). Однако мультипак-индекс необходимо каждый раз переписывать полностью. Мы можем расширить формат, сделав его инкрементным, чтобы запись выполнялась быстро. Сохраняя небольшой индекс нескольких пакетов "tip" , который указывает на большие файлы MIDX "base" , мы можем обеспечить быструю запись, одновременно сокращая количество двоичных поисков, необходимых для поиска объектов.

  • Если индекс мульти-пакета расширен для хранения "stable object order" (функция Order(hash) = целое число, которое является постоянным для данного хэша, даже при обновлении индекса мульти-пакета), то растровые изображения MIDX могут обновляться независимо от МИДКС.

  • Пакетные файлы можно пометить как "special" , используя пустые файлы с одинаковым исходным именем, но заменяя ".pack" на ".keep" или ".promisor" . Мы можем добавить необязательный фрагмент данных в multi-pack-index, который записывает флаги информации о пак-файлах. Это позволяет использовать новые состояния, такие как repacked или redeltified , которые могут помочь в обслуживании упаковки в среде с несколькими упаковками. Также может оказаться полезным организовать пакеты файлов по типам объектов (фиксация, дерево, большой двоичный объект и т. д.) и использовать эти метаданные для облегчения обслуживания.
