git-status

Name

git-status — Показать статус рабочего дерева

Synopsis

git status [<options>] [--] [<pathspec>…​]

Description

Отображает пути, которые имеют различия между файлом индекса и текущей фиксацией HEAD, пути, которые имеют различия между рабочим деревом и файлом индекса, а также пути в рабочем дереве, которые не отслеживаются Git (и не игнорируются gitignore[5] ). Первое — это то, что вы фиксируете would , запустив git commit ; второе и третье — это то, что вы фиксируете could , запустив git add перед запуском git commit .

Options

-s
--short

Дайте вывод в коротком формате.

-b
--branch

Покажите филиал и информацию об отслеживании даже в коротком формате.

--show-stash

Показать количество записей, спрятанных в данный момент.

--porcelain[=<version>]

Предоставьте вывод в удобном для анализа формате для сценариев. Это похоже на короткий вывод, но останется стабильным во всех версиях Git и независимо от пользовательской конфигурации. Подробности см. ниже.

Параметр версии используется для указания версии формата. Это необязательно и по умолчанию соответствует формату исходной версии v1 .

--long

Дайте вывод в длинном формате. Это значение по умолчанию.

-v
--verbose

В дополнение к именам файлов, которые были изменены, также покажите текстовые изменения, которые подготовлены для фиксации (т. е. как выходные данные git diff --cached ). Если -v указан дважды, то также отображаются изменения в рабочем дереве, которые еще не были проиндексированы (т. е. как выходные данные git diff ).

-u[<mode>]
--untracked-files[=<mode>]

Показать неотслеживаемые файлы.

Параметр mode используется для указания обработки неотслеживаемых файлов. Это необязательно: по умолчанию используется all , и если он указан, он должен быть привязан к этой опции (например, -uno , но не -u no ).

Возможные варианты:

  • no — Не показывать неотслеживаемые файлы.

  • normal — Показывает неотслеживаемые файлы и каталоги.

  • all — также показывает отдельные файлы в неотслеживаемых каталогах.

Когда опция -u не используется, отображаются неотслеживаемые файлы и каталоги (т. е. так же, как при указании normal ), чтобы вы не забыли добавить вновь созданные файлы. Поскольку поиск неотслеживаемых файлов в файловой системе требует дополнительной работы, этот режим может занять некоторое время в большом рабочем дереве. Рассмотрите возможность включения неотслеживаемого кэша и разделенного индекса, если они поддерживаются (см. git update-index --untracked-cache и git update-index --split-index ). В противном случае вы можете использовать no , чтобы git status возвращался быстрее, не показывая неотслеживаемые файлы.

Значение по умолчанию можно изменить с помощью переменной конфигурации status.showUntrackedFiles, описанной в git-config[1] .

--ignore-submodules[=<when>]

Игнорируйте изменения в подмодулях при поиске изменений. <When> может быть "none" , "untracked" , "dirty" или "all" , который используется по умолчанию. При использовании "none" подмодуль будет считаться измененным, если он либо содержит неотслеживаемые или измененные файлы, либо его HEAD отличается от фиксации, записанной в суперпроекте, и может использоваться для переопределения любых настроек параметра ignore в git-config[1] или gitmodules[5] . При использовании "untracked" субмодули не считаются «грязными», если они содержат только неотслеживаемый контент (но они все равно сканируются на наличие измененного контента). Использование "dirty" игнорирует все изменения в рабочем дереве подмодулей, отображаются только изменения коммитов, хранящихся в суперпроекте (так было до версии 1.7.0). Использование "all" скрывает все изменения в подмодулях (и подавляет вывод сводок подмодулей, если установлен параметр конфигурации status.submoduleSummary ).

--ignored[=<mode>]

Также показывать игнорируемые файлы.

Параметр mode используется для указания обработки игнорируемых файлов. Это необязательно: по умолчанию используется traditional .

Возможные варианты:

  • traditional — показывает игнорируемые файлы и каталоги, если не указан параметр --untracked-files=all; в этом случае отображаются отдельные файлы в игнорируемых каталогах.

  • no — Не показывать игнорируемые файлы.

  • matching — показывает игнорируемые файлы и каталоги, соответствующие шаблону игнорирования.

Если указан режим matching , отображаются пути, явно соответствующие игнорируемому шаблону. Если каталог соответствует шаблону игнорирования, то отображается он, но не пути, содержащиеся в игнорируемом каталоге. Если каталог не соответствует шаблону игнорирования, но все его содержимое игнорируется, то каталог не отображается, но отображается все содержимое.

-z

Завершайте записи с помощью NUL вместо LF. Это подразумевает выходной формат --porcelain=v1 , если не указан другой формат.

--column[=<options>]
--no-column

Отображать неотслеживаемые файлы в столбцах. Синтаксис опции см. в переменной конфигурации column.status . --column и --no-column без дополнительных опций эквивалентны always и never соответственно.

--ahead-behind
--no-ahead-behind

Отображать или не отображать подробные счетчики опережения/отставания для ветви относительно ее восходящей ветви. По умолчанию истинно.

--renames
--no-renames

Включите/выключите обнаружение переименования независимо от конфигурации пользователя. См. также git-diff[1] --no-renames .

--find-renames[=<n>]

Включите обнаружение переименования, при необходимости установив порог сходства. См. также git-diff[1] --find-renames .

<pathspec>…

См. запись pathspec в gitglossary[7] .

Output

Вывод этой команды предназначен для использования в качестве комментария к шаблону фиксации. Длинный формат по умолчанию предназначен для удобочитаемости, многословия и описательности. Его содержание и формат могут быть изменены в любое время.

Пути, упомянутые в выводе, в отличие от многих других команд Git , создаются относительно текущего каталога, если вы работаете в подкаталоге (это сделано специально, чтобы облегчить вырезание и вставку). См. параметр конфигурации status.relativePaths ниже.

Short Format

В кратком формате статус каждого пути отображается в одной из следующих форм:

XY PATH
XY ORIG_PATH -> PATH

где ORIG_PATH — источник переименованного/скопированного содержимого. ORIG_PATH отображается только при переименовании или копировании записи. XY — это двухбуквенный код состояния.

Поля (включая -> ) отделяются друг от друга одним пробелом. Если имя файла содержит пробелы или другие непечатаемые символы, это поле будет заключено в кавычки в виде строкового литерала C: в двойных кавычках ASCII (34) и с внутренними специальными символами, экранированными обратной косой чертой.

Существует три различных типа состояний, которые отображаются в этом формате, и каждый из них использует синтаксис XY по-разному:

  • Когда происходит слияние и оно прошло успешно, или вне ситуации слияния, X показывает состояние индекса, а Y показывает состояние рабочего дерева.

  • Если конфликт слияния произошел, но еще не разрешен, X и Y показывают состояние, введенное каждой головкой слияния, относительно общего предка. Эти пути называются unmerged .

  • Когда путь не отслеживается, X и Y всегда одинаковы, поскольку они неизвестны индексу. ?? используется для неотслеживаемых путей. Игнорируемые файлы не отображаются в списке, если не используется --ignored ; если это так, игнорируемые файлы обозначаются !! .

Обратите внимание, что термин merge здесь также включает в себя перебазирование с использованием стратегии --merge по умолчанию, выборку и все остальное, используя механизм слияния.

В следующей таблице эти три класса показаны в отдельных разделах, и эти символы используются в полях X и Y первых двух разделов, в которых показаны отслеживаемые пути:

  • '' = неизмененный

  • M = модифицированный

  • T = тип файла изменен (обычный файл, символическая ссылка или подмодуль)

  • A = добавлено

  • D = удалено

  • R = переименован

  • C = скопировано (если для параметра конфигурации status.renames установлено значение "copies" )

  • U = обновлено, но не объединено

X          Y     Meaning
-------------------------------------------------
         [AMD]   not updated
M        [ MTD]  updated in index
T        [ MTD]  type changed in index
A        [ MTD]  added to index
D                deleted from index
R        [ MTD]  renamed in index
C        [ MTD]  copied in index
[MTARC]          index and work tree matches
[ MTARC]    M    work tree changed since index
[ MTARC]    T    type changed in work tree since index
[ MTARC]    D    deleted in work tree
            R    renamed in work tree
            C    copied in work tree
-------------------------------------------------
D           D    unmerged, both deleted
A           U    unmerged, added by us
U           D    unmerged, deleted by them
U           A    unmerged, added by them
D           U    unmerged, deleted by us
A           A    unmerged, both added
U           U    unmerged, both modified
-------------------------------------------------
?           ?    untracked
!           !    ignored
-------------------------------------------------

Подмодули имеют больше состояний и вместо этого сообщают

  • M = субмодуль имеет другой HEAD, чем записанный в индексе

  • m = содержимое субмодуля изменено

  • ? = в субмодуле есть неотслеживаемые файлы

Это связано с тем, что измененный контент или неотслеживаемые файлы в подмодуле нельзя добавить через git add в суперпроект для подготовки фиксации.

m и ? применяются рекурсивно. Например, если вложенный подмодуль в подмодуле содержит неотслеживаемый файл, об этом также сообщается как ? .

Если используется -b, статусу в кратком формате предшествует строка

## информация для отслеживания названия филиала

Фарфоровый формат, версия 1

Формат фарфора версии 1 аналогичен короткому формату, но гарантированно не будет изменяться обратно несовместимым образом между версиями Git или в зависимости от конфигурации пользователя. Это делает его идеальным для анализа скриптами. Приведенное выше описание краткого формата также описывает формат фарфора, за некоторыми исключениями:

  1. Конфигурация пользователя color.status не учитывается; цвет всегда будет выключен.

  2. Конфигурация пользователя status.relativePaths не учитывается; Показанные пути всегда будут относиться к корню репозитория.

Для машинного анализа также рекомендуется использовать альтернативный формат -z. В этом формате поле статуса такое же, но некоторые другие вещи меняются. Во-первых, -> исключается из записей переименования, и порядок полей меняется на обратный (например, from -> to становится to from ). Во-вторых, за каждым именем файла следует NUL (ASCII 0), заменяющий пробел в качестве разделителя полей и завершающую новую строку (но пробел по-прежнему отделяет поле состояния от первого имени файла). В-третьих, имена файлов, содержащие специальные символы, не форматируются особым образом; кавычки или экранирование обратной косой черты не выполняются.

О любых изменениях субмодуля сообщается как модифицированный M вместо m или одиночный ? .

Фарфоровый формат, версия 2

Формат версии 2 добавляет более подробную информацию о состоянии рабочего дерева и измененных элементах. Версия 2 также определяет расширяемый набор простых для анализа дополнительных заголовков.

Строки заголовка начинаются с "#" и добавляются в ответ на определенные аргументы командной строки. Парсеры должны игнорировать заголовки, которые они не распознают.

Branch Headers

Если указан --branch , выводится ряд строк заголовка с информацией о текущей ветке.

Line                                     Notes
------------------------------------------------------------
# branch.oid <commit> | (initial)        Current commit.
# branch.head <branch> | (detached)      Current branch.
# branch.upstream <upstream_branch>      If upstream is set.
# branch.ab +<ahead> -<behind>           If upstream is set and
                                         the commit is present.
------------------------------------------------------------

Stash Information

Если задан --show-stash , выводится одна строка, показывающая количество записей тайника, если оно не равно нулю:

# stash <N>

Измененные отслеживаемые записи

После заголовков печатается ряд строк для отслеживаемых записей. Для описания записи может использоваться один из трех различных форматов строк в зависимости от типа изменения. Отслеживаемые записи печатаются в неопределенном порядке; парсеры должны позволять смешивать три типа строк в любом порядке.

Обычные измененные записи имеют следующий формат:

1 <XY> <sub> <mH> <mI> <mW> <hH> <hI> <path>

Переименованные или скопированные записи имеют следующий формат:

2 <XY> <sub> <mH> <mI> <mW> <hH> <hI> <X><score> <path><sep><origPath>
Field       Meaning
--------------------------------------------------------
<XY>        A 2 character field containing the staged and
            unstaged XY values described in the short format,
            with unchanged indicated by a "." rather than
            a space.
<sub>       A 4 character field describing the submodule state.
            "N..." when the entry is not a submodule.
            "S<c><m><u>" when the entry is a submodule.
            <c> is "C" if the commit changed; otherwise ".".
            <m> is "M" if it has tracked changes; otherwise ".".
            <u> is "U" if there are untracked changes; otherwise ".".
<mH>        The octal file mode in HEAD.
<mI>        The octal file mode in the index.
<mW>        The octal file mode in the worktree.
<hH>        The object name in HEAD.
<hI>        The object name in the index.
<X><score>  The rename or copy score (denoting the percentage
            of similarity between the source and target of the
            move or copy). For example "R100" or "C75".
<path>      The pathname.  In a renamed/copied entry, this
            is the target path.
<sep>       When the `-z` option is used, the 2 pathnames are separated
            with a NUL (ASCII 0x00) byte; otherwise, a tab (ASCII 0x09)
            byte separates them.
<origPath>  The pathname in the commit at HEAD or in the index.
            This is only present in a renamed/copied entry, and
            tells where the renamed/copied contents came from.
--------------------------------------------------------

Необъединенные записи имеют следующий формат; первый символ — "u" , чтобы отличать его от обычных измененных записей.

u <XY> <sub> <m1> <m2> <m3> <mW> <h1> <h2> <h3> <path>
Field       Meaning
--------------------------------------------------------
<XY>        A 2 character field describing the conflict type
            as described in the short format.
<sub>       A 4 character field describing the submodule state
            as described above.
<m1>        The octal file mode in stage 1.
<m2>        The octal file mode in stage 2.
<m3>        The octal file mode in stage 3.
<mW>        The octal file mode in the worktree.
<h1>        The object name in stage 1.
<h2>        The object name in stage 2.
<h3>        The object name in stage 3.
<path>      The pathname.
--------------------------------------------------------

Other Items

После отслеживаемых записей (и по запросу) будет напечатан ряд строк для неотслеживаемых, а затем проигнорированных элементов, найденных в рабочем дереве.

Неотслеживаемые элементы имеют следующий формат:

? <path>

Игнорируемые элементы имеют следующий формат:

! <path>

Примечания к формату пути и -z

Если задана опция -z , имена путей печатаются как есть, без каких-либо кавычек, а строки завершаются нулевым байтом (ASCII 0x00).

Без опции -z имена путей с символами "unusual" заключаются в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ).

Configuration

Команда учитывает color.status (или status.color — они означают одно и то же, а последнее сохранено для обратной совместимости) и переменные конфигурации color.status.<slot> для раскрашивания вывода.

Если для переменной конфигурации status.relativePaths установлено значение false, то все отображаемые пути относятся к корню репозитория, а не к текущему каталогу.

Если для status.submoduleSummary установлено ненулевое число или true (идентичное -1 или неограниченное число), сводка подмодуля будет включена для длинного формата и будет показана сводка коммитов для измененных подмодулей (см. --summary-limit вариант git-submodule[1] ). Обратите внимание, что сводный вывод команды status будет подавлен для всех субмодулей, если для diff.ignoreSubmodules установлено значение all , или только для тех субмодулей, где submodule.<name>.ignore=all . Чтобы также просмотреть сводку для игнорируемых подмодулей, вы можете использовать параметр командной строки --ignore-submodules=dirty или команду git submodule summary , которая показывает аналогичный вывод, но не учитывает эти настройки.

Background refresh

По умолчанию git status автоматически обновляет индекс, обновляя кэшированную информацию о статистике из рабочего дерева и записывая результат. Запись обновленного индекса — это оптимизация, которая не является строго необходимой ( status вычисляет значения самостоятельно, но запись их предназначена только для того, чтобы уберечь последующие программы от повторения наших вычислений). Когда status работает в фоновом режиме, блокировка, удерживаемая во время записи, может конфликтовать с другими одновременными процессами, что приводит к их сбою. В сценариях, выполняющих status в фоновом режиме, следует рассмотреть возможность использования git --no-optional-locks status (подробнее см. git[1] ).

Неотслеживаемые файлы и производительность

git status может работать очень медленно в больших рабочих деревьях, если/когда ему необходимо искать неотслеживаемые файлы и каталоги. Существует множество вариантов конфигурации, позволяющих ускорить этот процесс, либо избегая этой работы, либо используя кэшированные результаты предыдущих команд Git . Не существует единого оптимального набора настроек, подходящего для всех. Мы перечислим краткий список соответствующих опций, чтобы помочь вам, но прежде чем перейти к списку, вы можете захотеть снова запустить git status , поскольку ваша конфигурация, возможно, уже кэширует результаты git status , поэтому при последующих запусках он может работать быстрее.

  • Флаг --untracked-files=no или конфигурация status.showUntrackedfiles=false (оба см. выше): указывают, что git status не должен сообщать о неотслеживаемых файлах. Это самый быстрый вариант. git status не будет отображать неотслеживаемые файлы, поэтому вам нужно быть осторожным и помнить, если вы создаете какие-либо новые файлы и вручную git add их создаете.

  • advice.statusUoption=false (см. git-config[1] ): установка для этой переменной значения false отключает предупреждающее сообщение, появляющееся, когда перечисление неотслеживаемых файлов занимает более 2 секунд. В большом проекте это может занять больше времени, и пользователь, возможно, уже принял компромисс (например, использование "-uno" может быть неприемлемым вариантом для пользователя), и в этом случае нет смысла выдавать предупреждающее сообщение, и в таком случае В этом случае лучше всего отключить предупреждение.

  • core.untrackedCache=true (см. git-update-index[1] ): включите функцию неотслеживаемого кэша и выполните поиск только в каталогах, которые были изменены со времени предыдущей команды git status . Git запоминает набор неотслеживаемых файлов в каждом каталоге и предполагает, что если каталог не был изменен, то и набор неотслеживаемых файлов внутри каждого каталога не изменился. Это намного быстрее, чем перебор содержимого каждого каталога, но все же не без затрат, поскольку Git все равно придется искать набор измененных каталогов. Неотслеживаемый кэш хранится в файле .git/index . Снижение затрат на поиск неотслеживаемых файлов немного компенсируется увеличением размера индекса и стоимостью его поддержания в актуальном состоянии. Это сокращение времени поиска обычно стоит дополнительных размеров.

  • core.untrackedCache=true и core.fsmonitor=true или core.fsmonitor=<hook_command_pathname> (см. git-update-index[1] ): включите как неотслеживаемый кэш, так и функции FSMonitor, а также выполните поиск только в каталогах, которые были изменены со времени предыдущей команды git status . Это быстрее, чем использование только неотслеживаемого кэша, поскольку Git также может избежать поиска измененных каталогов. Git нужно только перечислить точный набор каталогов, которые недавно изменились. Хотя функцию FSMonitor можно включить и без неотслеживаемого кэша, в этом случае преимущества значительно уменьшаются.

Обратите внимание, что после включения неотслеживаемого кэша и/или функций FSMonitor может потребоваться несколько команд git status для прогрева различных кэшей, прежде чем вы увидите улучшенное время выполнения команд. Это нормально.
