git-maintenance

Name

git-maintenance — запуск задач по оптимизации данных репозитория Git .

Synopsis

git maintenance run [<options>]
git maintenance start [--scheduler=<scheduler>]
git maintenance (stop|register|unregister) [<options>]

Description

Запускайте задачи по оптимизации данных репозитория Git , ускоряя выполнение других команд Git и снижая требования к объему хранилища для репозитория.

Команды Git , добавляющие данные репозитория, такие как git add или git fetch , оптимизированы для быстрого реагирования пользователей. Эти команды не требуют времени для оптимизации данных Git , поскольку такая оптимизация масштабируется в соответствии с полным размером репозитория, в то время как каждая из этих пользовательских команд выполняет относительно небольшое действие.

Команда git maintenance обеспечивает гибкость оптимизации репозитория Git .

Subcommands

run

Запустите одну или несколько задач обслуживания. Если указаны один или несколько параметров --task , эти задачи выполняются в указанном порядке. В противном случае задачи определяются тем, какие параметры конфигурации maintenance.<task>.enabled верны. По умолчанию верен только maintenance.gc.enabled .

start

Начните обслуживание текущего репозитория. При этом выполняются те же обновления конфигурации, что и подкомандой register , а затем обновляется фоновый планировщик для ежечасного запуска git maintenance run --scheduled .

stop

Остановите график фонового обслуживания. Текущий репозиторий не удаляется из списка обслуживаемых репозиториев, если фоновое обслуживание будет перезапущено позже.

register

Инициализируйте значения конфигурации Git , чтобы в этом репозитории началось любое плановое обслуживание. Это добавит репозиторий в переменную конфигурации maintenance.repo в глобальной конфигурации текущего пользователя или в конфигурацию, указанную параметром --config-file, и активирует некоторые рекомендуемые значения конфигурации для maintenance.<task>.schedule . Включенные задачи можно безопасно выполнять в фоновом режиме, не нарушая приоритетные процессы.

Субкоманда register также установит значение конфигурации maintenance.strategy на incremental , если это значение не было установлено ранее. Стратегия incremental использует следующий график для каждой задачи обслуживания:

  • gc: disabled.

  • commit-graph: hourly.

  • prefetch: hourly.

  • loose-objects: daily.

  • incremental-repack: daily.

git maintenance register также отключит приоритетное обслуживание, установив maintenance.auto = false в текущем репозитории. Этот параметр конфигурации сохранится после команды git maintenance unregister .

unregister

Удалите текущий репозиторий из режима фонового обслуживания. Это только удалит репозиторий из настроенного списка. Это не останавливает выполнение процессов фонового обслуживания.

Подкоманда unregister сообщит об ошибке, если текущий репозиторий еще не зарегистрирован. Используйте опцию --force , чтобы вернуть успех, даже если текущий репозиторий не зарегистрирован.

Tasks

commit-graph

Задание commit-graph постепенно обновляет файлы commit-graph , а затем проверяет правильность записанных данных. Инкрементальную запись можно безопасно выполнять вместе с параллельными процессами Git , поскольку она не истечет срока действия файлов .graph , которые были в предыдущем файле commit-graph-chain . Они будут удалены при более позднем запуске в зависимости от задержки истечения срока действия.

prefetch

Задача prefetch обновляет каталог объектов новейшими объектами со всех зарегистрированных удаленных устройств. Для каждого пульта запускается команда git fetch . Настроенная спецификация ссылок изменяется для размещения всех запрошенных ссылок в refs/prefetch/ . Также теги не обновляются.

Это сделано для того, чтобы не нарушать работу ветвей удаленного отслеживания. Конечные пользователи ожидают, что эти ссылки останутся неподвижными, если они не инициируют выборку. Однако при выполнении задачи предварительной выборки объекты, необходимые для завершения последующей реальной выборки, уже будут получены, что ускоряет реальную выборку. В идеальном случае это будет просто обновление кучи веток удаленного отслеживания без какой-либо передачи объектов.

gc

Очистите ненужные файлы и оптимизируйте локальный репозиторий. "GC" означает "garbage collection," , но эта задача выполняет множество более мелких задач. Эта задача может быть дорогостоящей для больших репозиториев, так как она переупаковывает все объекты Git в один пакетный файл. В некоторых ситуациях это также может быть разрушительным, поскольку удаляет устаревшие данные. Дополнительные сведения о сборке мусора в Git см. в git-gc[1] .

loose-objects

Задание loose-objects очищает незакрепленные объекты и помещает их в пак-файлы. Чтобы предотвратить условия гонки с одновременными командами Git , выполняется двухэтапный процесс. Во-первых, он удаляет все незакрепленные объекты, которые уже существуют в пакетном файле; параллельные процессы Git будут проверять пакетный файл на предмет данных объекта, а не отдельного объекта. Во-вторых, он создает новый пакетный файл (начиная с "loose-" ), содержащий пакет незакрепленных объектов. Размер пакета ограничен 50 тысячами объектов, чтобы задание не занимало слишком много времени в репозитории с большим количеством незакрепленных объектов. Задача gc записывает недоступные объекты как незакрепленные объекты, подлежащие очистке на более позднем этапе, только если они не добавлены повторно в пакетный файл; по этой причине не рекомендуется одновременно включать задачи loose-objects и gc .

incremental-repack

Задание incremental-repack переупаковывает каталог объектов с помощью функции multi-pack-index . Чтобы предотвратить условия гонки с одновременными командами Git , выполняется двухэтапный процесс. Сначала он вызывает git multi-pack-index expire для удаления файлов пакета, на которые не ссылается файл multi-pack-index . Во-вторых, он вызывает git multi-pack-index repack , чтобы выбрать несколько небольших файлов-пакетов и переупаковать их в один больший, а затем обновить записи multi-pack-index , которые ссылаются на небольшие файлы-пакеты, чтобы они ссылались на новый файл-пакет. Это подготавливает эти небольшие файлы пакета к удалению при следующем запуске git multi-pack-index expire . Выбор маленьких файлов упаковки таков, что ожидаемый размер большого файла упаковки равен, по меньшей мере, размеру пакета; см. опцию --batch-size для подкоманды repack в git-multi-pack-index[1] . Размер пакета по умолчанию равен нулю, что является особым случаем, когда все файлы пакета пытаются переупаковать в один файл пакета.

pack-refs

Задача pack-refs собирает отдельные справочные файлы и объединяет их в один файл. Это ускоряет операции, которые необходимо повторять по множеству ссылок. См. git-pack-refs[1] для получения дополнительной информации.

Options

--auto

В сочетании с подкомандой run запускайте задачи обслуживания только при соблюдении определенных пороговых значений. Например, задача gc запускается, когда количество незакрепленных объектов превышает число, хранящееся в настройке конфигурации gc.auto , или когда количество файлов пакета превышает настройку конфигурации gc.autoPackLimit . Несовместимо с опцией --schedule .

--schedule

В сочетании с подкомандой run запускайте задачи обслуживания только при соблюдении определенных временных условий, как указано в значении конфигурации maintenance.<task>.schedule для каждого <task> . Это значение конфигурации указывает количество секунд с момента последнего запуска этой задачи в соответствии со значением конфигурации maintenance.<task>.lastRun . Тестируются задачи, предоставляемые опциями --task=<task> , или задачи, для которых maintenance.<task>.enabled установлено значение true.

--quiet

Не сообщайте о ходе работы или другую информацию по stderr .

--task=<task>

Если этот параметр указан один или несколько раз, указанные задачи выполняются только в указанном порядке. Если аргументы --task=<task> не указаны, то рассматриваются только задачи, в которых maintenance.<task>.enabled настроен как true . Список допустимых значений <task> см. в разделе TASKS .

--scheduler=auto|crontab|systemd-timer|launchctl|schtasks

В сочетании с подкомандой start укажите планировщик для ежечасного, ежедневного и еженедельного выполнения git maintenance run . Возможные значения <scheduler> : auto , crontab (POSIX), systemd-timer (Linux), launchctl (macOS) и schtasks (Windows). Если указан auto , используется соответствующий планировщик для конкретной платформы; в Linux используется systemd-timer , если доступен, в противном случае crontab . По умолчанию — auto .

Troubleshooting

Команда git maintenance предназначена для упрощения шаблонов обслуживания репозитория и минимизации времени ожидания пользователя во время выполнения команд Git . Доступны различные параметры конфигурации, позволяющие настроить этот процесс. Параметры обслуживания по умолчанию ориентированы на операции, которые выполняются быстро, даже в больших репозиториях.

В некоторых случаях пользователи могут столкнуться с случаями, когда запланированные задачи обслуживания выполняются не так часто, как предполагалось. Каждая команда git maintenance run блокирует объектную базу данных репозитория, и это предотвращает параллельный запуск других команд git maintenance run в том же репозитории. Без этой защиты конкурирующие процессы могут привести репозиторий в непредсказуемое состояние.

График фонового обслуживания запускает процессы git maintenance run ежечасно. При каждом запуске выполняются задачи "hourly" . В полночь этот процесс также выполняет задачи "daily" . В полночь первого дня недели этот процесс также выполняет задачи "weekly" . Один процесс перебирает каждый зарегистрированный репозиторий, выполняя запланированные задачи с этой частотой. В зависимости от количества зарегистрированных репозиториев и их размеров этот процесс может занять больше часа. В этом случае несколько команд git maintenance run могут одновременно выполняться в одном и том же репозитории, что приводит к конфликту при блокировке базы данных объектов. Это приводит к тому, что одна из двух задач не выполняется.

Если вы обнаружите, что некоторые окна обслуживания занимают больше часа, рассмотрите возможность снижения сложности задач обслуживания. Например, задача gc выполняется намного медленнее, чем задача incremental-repack . Однако за это приходится платить немного большей базой данных объектов. Рассмотрите возможность переноса более ресурсоемких задач на менее частое выполнение.

Опытные пользователи могут рассмотреть возможность планирования собственных задач по техническому обслуживанию, используя расписание, отличное от того, которое доступно в параметрах конфигурации git maintenance start и Git . Эти пользователи должны знать о блокировке базы данных объектов и о том, как ведут себя одновременные команды git maintenance run . Кроме того, команду git gc не следует объединять с командами git maintenance run . git gc изменяет базу данных объектов, но не принимает блокировку так же, как git maintenance run . Если возможно, используйте git maintenance run --task=gc вместо git gc .

В следующих разделах описываются механизмы, используемые для запуска фонового обслуживания с помощью git maintenance start , и способы их настройки.

Фоновое обслуживание систем posix

Стандартным механизмом планирования фоновых задач в системах POSIX является cron(8). Этот инструмент выполняет команды по заданному расписанию. Текущий список запланированных пользователем задач можно найти, запустив crontab -l . Расписание, написанное git maintenance start , похоже на это:

# НАЧАТЬ ГРАФИК ОБСЛУЖИВАНИЯ GIT
# Следующее расписание было создано Git
# Любые изменения, сделанные в этом регионе, могут быть
# заменен в будущем командой Git.

0 1-23 * * * "/<path>/git" --exec-path="/<path>" for-each-repo --config=maintenance.repo maintenance run --schedule=hourly
0 0 * * 1-6 "/<path>/git" --exec-path="/<path>" for-each-repo --config=maintenance.repo maintenance run --schedule=daily
0 0 * * 0 "/<path>/git" --exec-path="/<path>" for-each-repo --config=maintenance.repo maintenance run --schedule=weekly

# КОНЕЦ ГРАФИКА ОБСЛУЖИВАНИЯ GIT

Комментарии используются в качестве региона для обозначения расписания, написанного Git. Любые изменения в этом регионе будут полностью удалены git maintenance stop или перезаписаны git maintenance start .

Запись crontab указывает полный путь к исполняемому файлу git , чтобы гарантировать, что выполненная команда git совпадает с той, с которой git maintenance start был выполнен независимо от PATH . Если один и тот же пользователь запускает git maintenance start с несколькими исполняемыми файлами Git , то используется только последний исполняемый файл.

Эти команды используют git for-each-repo --config=maintenance.repo для запуска git maintenance run --schedule=<frequency> в каждом репозитории, указанном в многозначном параметре конфигурации maintenance.repo . Обычно они загружаются из глобальной конфигурации пользователя. Затем процесс git maintenance определяет, какие задачи обслуживания настроены для выполнения в каждом репозитории с каждым <frequency> , используя параметры конфигурации maintenance.<task>.schedule . Эти значения загружаются из глобальных значений конфигурации или значений конфигурации репозитория.

Если значений конфигурации недостаточно для достижения желаемого расписания фонового обслуживания, вы можете создать собственное расписание. Если вы запустите crontab -e , то загрузится редактор с вашим пользовательским расписанием cron . В этом редакторе вы можете добавлять свои собственные строки графика. Вы можете начать с адаптации расписания по умолчанию, указанного ранее, или прочитать документацию crontab(5), чтобы узнать о расширенных методах планирования. Пожалуйста, используйте полный путь и методы --exec-path из расписания по умолчанию, чтобы убедиться, что вы выполняете правильные двоичные файлы в своем расписании.

Фоновое обслуживание систем Linux systemd

Хотя Linux поддерживает cron , в зависимости от дистрибутива cron может быть дополнительным пакетом, который не обязательно должен быть установлен. В современных дистрибутивах Linux его заменяют таймеры systemd.

Если доступны пользовательские системные таймеры, они будут использоваться в качестве замены cron .

В этом случае git maintenance start создаст пользовательские таймеры systemd и запустит таймеры. Текущий список запланированных пользователем задач можно найти, запустив systemctl --user list-timers . Таймеры, написанные git maintenance start , аналогичны этому:

$ systemctl --user list-timers
NEXT                         LEFT          LAST                         PASSED     UNIT                         ACTIVATES
Thu 2021-04-29 19:00:00 CEST 42min left    Thu 2021-04-29 18:00:11 CEST 17min ago  git-maintenance@hourly.timer git-maintenance@hourly.service
Fri 2021-04-30 00:00:00 CEST 5h 42min left Thu 2021-04-29 00:00:11 CEST 18h ago    git-maintenance@daily.timer  git-maintenance@daily.service
Mon 2021-05-03 00:00:00 CEST 3 days left   Mon 2021-04-26 00:00:11 CEST 3 days ago git-maintenance@weekly.timer git-maintenance@weekly.service

Для каждой опции --schedule=<frequency> зарегистрирован один таймер.

Определение модулей systemd можно просмотреть в следующих файлах:

~/.config/systemd/user/git[email protected]
~/.config/systemd/user/git[email protected]
~/.config/systemd/user/timers.target.wants/git-maintenance@hourly.timer
~/.config/systemd/user/timers.target.wants/git-maintenance@daily.timer
~/.config/systemd/user/timers.target.wants/git-maintenance@weekly.timer

git maintenance start перезапишет эти файлы и снова запустит таймер с помощью systemctl --user , поэтому любую настройку следует выполнять путем создания вставного файла, то есть файла с суффиксом .conf в каталоге ~/.config/systemd/user/[email protected] .

git maintenance stop остановит системные таймеры пользователя и удалит вышеупомянутые файлы.

Подробнее см. systemd.timer(5) .

Фоновое обслуживание систем MacOS

Хотя macOS технически поддерживает cron , для использования crontab -e требуются повышенные привилегии, и исполняемый процесс не имеет полного пользовательского контекста. Без полного пользовательского контекста Git и его помощники по учетным данным не могут получить доступ к сохраненным учетным данным, поэтому некоторые задачи обслуживания не работают.

Вместо этого git maintenance start взаимодействует с инструментом launchctl , который является рекомендуемым способом планирования заданий по времени в macOS. Для планирования обслуживания через git maintenance (start|stop) требуются некоторые функции launchctl , доступные только в macOS 10.11 или более поздней версии.

Запланированные задачи, специфичные для пользователя, хранятся в виде файлов .plist в формате XML в ~/Library/LaunchAgents/ . Посмотреть зарегистрированные в данный момент задачи можно с помощью следующей команды:

$ ls ~/Library/LaunchAgents/org.git-scm.git*
org.git-scm.git.daily.plist
org.git-scm.git.hourly.plist
org.git-scm.git.weekly.plist

Для каждой опции --schedule=<frequency> регистрируется одна задача. Чтобы проверить, как формат XML описывает каждое расписание, откройте один из этих файлов .plist в редакторе и проверьте элемент <array> , следующий за элементом <key>StartCalendarInterval</key> .

git maintenance start перезапишет эти файлы и снова зарегистрирует задачи в launchctl , поэтому любые настройки следует выполнять путем создания собственных файлов .plist с разными именами. Аналогичным образом команда git maintenance stop отменит регистрацию задач в launchctl и удалит файлы .plist .

Чтобы создать более сложные настройки для ваших фоновых задач, см. дополнительную информацию в файле launchctl.plist(5).

Фоновое обслуживание систем Windows

Windows не поддерживает cron и вместо этого имеет собственную систему планирования фоновых задач. Команда git maintenance start использует команду schtasks для отправки задач в эту систему. Вы можете проверить все фоновые задачи с помощью приложения «Планировщик заданий». Задачи, добавляемые Git , имеют имена вида Git Maintenance (<frequency>) . В графическом интерфейсе планировщика задач есть способы проверки этих задач, но вы также можете экспортировать задачи в файлы XML и просматривать подробности там.

Обратите внимание: поскольку Git является консольным приложением, эти фоновые задачи создают окно консоли, видимое текущему пользователю. Это можно изменить вручную, выбрав опцию "Run whether user is logged in or not" в планировщике задач. Это изменение требует ввода пароля, поэтому git maintenance start не выбирает его по умолчанию.

Если вы хотите настроить фоновые задачи, переименуйте их, чтобы будущие вызовы git maintenance (start|stop) не перезаписывали ваши пользовательские задачи.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

maintenance.auto

Этот параметр конфигурации boolean определяет, будут ли некоторые команды запускать git maintenance run --auto после выполнения своей обычной работы. По умолчанию истинно.

maintenance.strategy

Этот параметр конфигурации строки позволяет указать одно из нескольких рекомендуемых расписаний фонового обслуживания. Это влияет только на то, какие задачи выполняются во время команд git maintenance run --schedule=X , при условии, что аргументы --task=<task> не указаны. Кроме того, если установлено значение конфигурации maintenance.<task>.schedule , то это значение используется вместо значения, предоставленного maintenance.strategy . Возможные строки стратегии:

  • none : этот параметр по умолчанию подразумевает, что никакие задачи не выполняются ни по одному расписанию.

  • incremental : этот параметр оптимизирует выполнение небольших действий по обслуживанию, не приводящих к удалению данных. При этом задача gc не планируется, а задачи prefetch и commit-graph выполняются ежечасно, задачи loose-objects и incremental-repack — ежедневно, а задача pack-refs — еженедельно.

maintenance.<task>.enabled

Этот параметр конфигурации boolean определяет, будет ли выполняться задача обслуживания с именем <task> , если для git maintenance run не указан параметр --task . Эти значения конфигурации игнорируются, если существует опция --task . По умолчанию верен только maintenance.gc.enabled .

maintenance.<task>.schedule

Этот параметр конфигурации определяет, запускается ли данный <task> во время команды git maintenance run --schedule=<frequency> . Значение должно быть одним из "hourly" , "daily" или "weekly" .

maintenance.commit-graph.auto

Этот целочисленный параметр конфигурации определяет, как часто задача commit-graph должна запускаться как часть git maintenance run --auto . Если ноль, то задача commit-graph не будет запускаться с опцией --auto . Отрицательное значение заставит задачу запускаться каждый раз. В противном случае положительное значение означает, что команда должна выполняться, когда количество доступных коммитов, которых нет в файле графа коммитов, равно как минимум значению maintenance.commit-graph.auto . Значение по умолчанию — 100.

maintenance.loose-objects.auto

Этот целочисленный параметр конфигурации определяет, как часто задача loose-objects должна запускаться как часть git maintenance run --auto . Если ноль, то задача loose-objects не будет запускаться с опцией --auto . Отрицательное значение заставит задачу запускаться каждый раз. В противном случае положительное значение означает, что команда должна выполняться, когда количество незакрепленных объектов не превышает значения maintenance.loose-objects.auto . Значение по умолчанию — 100.

maintenance.incremental-repack.auto

Этот целочисленный параметр конфигурации определяет, как часто задача incremental-repack должна запускаться как часть git maintenance run --auto . Если ноль, то задача incremental-repack не будет запускаться с опцией --auto . Отрицательное значение заставит задачу запускаться каждый раз. В противном случае положительное значение означает, что команда должна выполняться, когда количество файлов пакета, не входящих в индекс мультипакета, равно как минимум значению maintenance.incremental-repack.auto . Значение по умолчанию — 10.
