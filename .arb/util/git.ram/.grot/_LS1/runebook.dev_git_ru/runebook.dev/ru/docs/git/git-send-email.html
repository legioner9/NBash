git-send-email

Name

git-send-email — отправить набор патчей по электронной почте.

Synopsis

git send-email [<options>] <file|directory>…​
git send-email [<options>] <format-patch options>
git send-email --dump-aliases

Description

Принимает исправления, указанные в командной строке, и отправляет их по электронной почте. Патчи можно указать в виде файлов, каталогов (в которые будут отправлены все файлы в каталоге) или непосредственно в виде списка ревизий. В последнем случае в git для отправки электронной почты можно передать любой формат, принимаемый git-format-patch[1] , а также параметры, понятные git-format-patch[1] .

Заголовок электронного письма настраивается с помощью параметров командной строки. Если это не указано в командной строке, пользователю будет предложено использовать интерфейс с поддержкой ReadLine для предоставления необходимой информации.

Для файлов исправлений принимаются два формата:

  1. файлы формата mbox

    Это то, что генерирует git-format-patch[1] . Большинство заголовков и форматирование MIME игнорируются.

  2. Исходный формат, используемый сценарием send_lots_of_email.pl Грега Кроа-Хартмана.

    Этот формат предполагает, что первая строка файла будет содержать значение "Cc:" , а во второй строке — "Subject:" сообщения.

Options

Composing

--annotate

Просмотрите и отредактируйте каждый патч, который вы собираетесь отправить. По умолчанию используется значение sendemail.annotate . См. раздел КОНФИГУРАЦИЯ для sendemail.multiEdit .

--bcc=<address>,…

Укажите значение "Bcc:" для каждого электронного письма. По умолчанию используется значение sendemail.bcc .

Эту опцию можно указать несколько раз.

--cc=<address>,…

Укажите начальное значение "Cc:" для каждого электронного письма. По умолчанию используется значение sendemail.cc .

Эту опцию можно указать несколько раз.

--compose

Вызовите текстовый редактор (см. GIT_EDITOR в git-var[1] ), чтобы отредактировать вступительное сообщение для серии патчей.

При использовании --compose при отправке электронного письма git будут использоваться заголовки From, To, Cc, Bcc, subject, Reply-To и In-Reply-To, указанные в сообщении. Если тело сообщения (то, что вы вводите после заголовков и пустой строки) содержит только пустые строки (или Git: с префиксом), сводка не будет отправлена, но будут использоваться упомянутые выше заголовки, если они не будут удалены.

Будет предложено ввести заголовки «Отсутствует» или «В ответе».

См. раздел КОНФИГУРАЦИЯ для sendemail.multiEdit .

--from=<address>

Укажите отправителя писем. Если не указано в командной строке, используется значение параметра конфигурации sendemail.from . Если ни параметр командной строки, ни sendemail.from не установлены, пользователю будет предложено ввести значение. По умолчанию для приглашения будет значение GIT_AUTHOR_IDENT или GIT_COMMITTER_IDENT, если оно не установлено, как возвращается "git var -l" .

--reply-to=<address>

Укажите адрес, на который должны поступать ответы от получателей. Используйте это, если ответы на сообщения должны отправляться на другой адрес, отличный от того, который указан в параметре --from.

--in-reply-to=<identifier>

Сделайте так, чтобы первое письмо (или все письма с --no-thread ) отображалось в качестве ответа на данный идентификатор сообщения, что позволяет избежать прерывания потоков для предоставления новой серии исправлений. Второе и последующие электронные письма будут отправлены в виде ответов в соответствии с настройкой --[no-]chain-reply-to .

Например, если указаны --thread и --no-chain-reply-to , второй и последующие исправления будут ответами на первый, как показано на рисунке ниже, где [PATCH v2 0/3] является ответом на [PATCH 0/2] :

[PATCH 0/2] Here is what I did...
  [PATCH 1/2] Clean up and tests
  [PATCH 2/2] Implementation
  [PATCH v2 0/3] Here is a reroll
    [PATCH v2 1/3] Clean up
    [PATCH v2 2/3] New tests
    [PATCH v2 3/3] Implementation

Необходимо только в том случае, если также установлен --compose. Если --compose не установлен, это будет предложено.

--subject=<string>

Укажите начальную тему ветки электронной почты. Необходимо только в том случае, если также установлен --compose. Если --compose не установлен, это будет предложено.

--to=<address>,…

Укажите основного получателя созданных электронных писем. Как правило, это будет вышестоящий специалист по сопровождению соответствующего проекта. По умолчанию — это значение конфигурации sendemail.to ; если это значение не указано и параметр --to-cmd не указан, это будет предложено.

Эту опцию можно указать несколько раз.

--8bit-encoding=<encoding>

При обнаружении сообщения или темы, отличной от ASCII, в которой не указана его кодировка, добавьте заголовки/кавычки, чтобы указать, что оно закодировано в <кодировке>. По умолчанию используется значение sendemail.assume8bitEncoding ; если это значение не указано, это будет предложено при обнаружении каких-либо файлов, отличных от ASCII.

Обратите внимание, что не предпринимается никаких попыток проверить кодировку.

--compose-encoding=<encoding>

Укажите кодировку сообщения создания. По умолчанию используется значение sendemail.composeencoding ; если это не указано, предполагается UTF-8.

--transfer-encoding=(7bit|8bit|quoted-printable|base64|auto)

Укажите кодировку передачи, которая будет использоваться для отправки сообщения по SMTP. 7bit завершится сбоем при обнаружении сообщения, отличного от ASCII. Quote-printable может быть полезен, когда репозиторий содержит файлы, содержащие возвраты каретки, но делает необработанный файл электронной почты исправления (сохраненный из MUA) гораздо сложнее проверить вручную. base64 еще более надежен, но и еще более непрозрачен. auto будет использовать 8 бит, если это возможно, а в противном случае — для печати в кавычках.

По умолчанию — это значение конфигурации sendemail.transferEncoding ; если это не указано, по умолчанию используется auto .

--xmailer
--no-xmailer

Добавьте (или запретите добавление) заголовок "X-Mailer:" . По умолчанию заголовок добавляется, но его можно отключить, задав для переменной конфигурации sendemail.xmailer значение false .

Sending

--envelope-sender=<address>

Укажите отправителя конверта, используемого для отправки электронных писем. Это полезно, если ваш адрес по умолчанию не является адресом, подписанным на список. Чтобы использовать адрес From , установите значение "auto" . Если вы используете двоичный файл sendmail, у вас должны быть соответствующие права для параметра -f. По умолчанию — это значение переменной конфигурации sendemail.envelopeSender ; если это не указано, выбор отправителя конверта остается на усмотрение вашего MTA.

--sendmail-cmd=<command>

Укажите команду, которую нужно выполнить для отправки электронного письма. Команда должна быть похожа на sendmail; в частности, он должен поддерживать опцию -i . При необходимости команда будет выполнена в оболочке. По умолчанию используется значение sendemail.sendmailcmd . Если не указано и --smtp-server также не указано, git-send-email будет искать sendmail в /usr/sbin , /usr/lib и $PATH.

--smtp-encryption=<encryption>

Укажите, каким образом начинается шифрование SMTP-соединения. Допустимые значения: ssl и tls . Любое другое значение возвращается к простому (незашифрованному) SMTP, для которого по умолчанию используется порт 25. Несмотря на имена, оба значения будут использовать одну и ту же новую версию TLS, но по историческим причинам имеют эти имена. ssl относится к шифрованию "implicit" (иногда называемому SMTPS), которое по умолчанию использует порт 465. tls относится к шифрованию "explicit" (часто известному как STARTTLS), которое по умолчанию использует порт 25. SMTP-сервером могут использоваться другие порты, которые не являются портами по умолчанию. Обычно альтернативный незашифрованный порт для tls — 587. Вам необходимо проверить документацию вашего провайдера или конфигурацию вашего сервера, чтобы убедиться в своем собственном случае. По умолчанию используется значение sendemail.smtpEncryption .

--smtp-domain=<FQDN>

Указывает полное доменное имя (FQDN), используемое в команде HELO/EHLO для SMTP-сервера. Некоторые серверы требуют, чтобы полное доменное имя соответствовало вашему IP-адресу. Если этот параметр не установлен, git при отправке электронной почты пытается автоматически определить ваше полное доменное имя. По умолчанию используется значение sendemail.smtpDomain .

--smtp-auth=<mechanisms>

Список разрешенных механизмов SMTP-AUTH, разделенный пробелами. Эта настройка заставляет использовать только перечисленные механизмы. Пример:

$ git send-email --smtp-auth="PLAIN LOGIN GSSAPI" ...

Если хотя бы один из указанных механизмов соответствует механизмам, объявленным SMTP-сервером, и если он поддерживается используемым SASL library , этот механизм используется для аутентификации. Если не указан ни sendemail.smtpAuth , ни --smtp-auth , можно использовать все механизмы, поддерживаемые SASL library . Специальное значение none может быть указано для полного отключения аутентификации независимо от --smtp-user .

--smtp-pass[=<password>]

Пароль для SMTP-AUTH. Аргумент не является обязательным: если аргумент не указан, в качестве пароля используется пустая строка. По умолчанию используется значение sendemail.smtpPass , однако --smtp-pass всегда переопределяет это значение.

Более того, пароли не обязательно указывать в файлах конфигурации или в командной строке. Если указано имя пользователя (с --smtp-user или sendemail.smtpUser ), но не указан пароль (с --smtp-pass или sendemail.smtpPass ), то пароль получается с помощью git-credential .

--no-smtp-auth

Отключите аутентификацию SMTP. Короткая рука для --smtp-auth=none

--smtp-server=<host>

Если установлено, указывает используемый исходящий SMTP-сервер (например, smtp.example.com или необработанный IP-адрес). Если не указано и --sendmail-cmd также не указано, по умолчанию выполняется поиск sendmail в /usr/sbin , /usr/lib и $PATH, если такая программа доступна, в противном случае возвращается к localhost .

Для обратной совместимости этот параметр также может указывать полный путь к программе, похожей на sendmail; программа должна поддерживать опцию -i . Этот метод не поддерживает передачу аргументов или использование простых имен команд. В таких случаях рассмотрите возможность использования --sendmail-cmd .

--smtp-server-port=<port>

Указывает порт, отличный от порта по умолчанию (SMTP-серверы обычно прослушивают порт smtp 25, но могут также прослушивать порт отправки 587 или общий порт SSL smtp 465); символические имена портов (например, "submission" вместо 587) также принимаются. Порт также можно настроить с помощью переменной конфигурации sendemail.smtpServerPort .

--smtp-server-option=<option>

Если установлено, указывает используемый параметр исходящего SMTP-сервера. Значение по умолчанию можно указать с помощью параметра конфигурации sendemail.smtpServerOption .

Опцию --smtp-server-option необходимо повторять для каждой опции, которую вы хотите передать на сервер. Аналогично, для каждой опции необходимо использовать разные строки в файлах конфигурации.

--smtp-ssl

Устаревший псевдоним для --smtp-encryption ssl .

--smtp-ssl-cert-path

Путь к хранилищу доверенных сертификатов CA для проверки сертификатов SMTP SSL/TLS (либо каталог, обработанный c_rehash , либо отдельный файл, содержащий один или несколько объединенных вместе сертификатов формата PEM: см.verify(1) -CAfile и -CApath). дополнительную информацию об этом). Установите пустую строку, чтобы отключить проверку сертификата. По умолчанию используется значение переменной конфигурации sendemail.smtpsslcertpath , если оно установлено, или значение по умолчанию, скомпилированное в поддерживающей библиотеке SSL (что должно быть лучшим выбором на большинстве платформ).

--smtp-user=<user>

Имя пользователя для SMTP-AUTH. По умолчанию используется значение sendemail.smtpUser ; если имя пользователя не указано (с --smtp-user или sendemail.smtpUser ), попытка аутентификации не выполняется.

--smtp-debug=0|1

Включить (1) или отключить (0) вывод отладки. Если этот параметр включен, команды и ответы SMTP будут распечатаны. Полезно для отладки TLS-соединения и проблем с аутентификацией.

--batch-size=<num>

Некоторые почтовые серверы (например, smtp.163.com) ограничивают количество писем, отправляемых за сеанс (соединение), и это приведет к сбою при отправке большого количества сообщений. При использовании этой опции send-email отключится после отправки $<num> сообщений, подождет несколько секунд (см. --relogin-delay) и снова подключится, чтобы обойти такое ограничение. Возможно, вы захотите использовать какой-либо помощник по учетным данным, чтобы избежать необходимости каждый раз повторно вводить пароль. По умолчанию используется переменная конфигурации sendemail.smtpBatchSize .

--relogin-delay=<int>

Ожидание $<int> секунд перед повторным подключением к SMTP-серверу. Используется вместе с опцией --batch-size. По умолчанию используется переменная конфигурации sendemail.smtpReloginDelay .

Automating

--no-[to|cc|bcc]

Очищает любой список адресов "To:" , "Cc:" , "Bcc:" , ранее установленных в конфигурации.

--no-identity

Очищает ранее прочитанное значение sendemail.identity , установленное в конфигурации, если таковое имеется.

--to-cmd=<command>

Укажите команду, которая будет выполняться один раз для каждого файла исправления, которая должна генерировать записи "To:" , специфичные для файла исправления. Вывод этой команды должен состоять из одного адреса электронной почты в каждой строке. По умолчанию используется значение конфигурации sendemail.tocmd .

--cc-cmd=<command>

Укажите команду, которая будет выполняться один раз для каждого файла исправления, которая должна генерировать записи "Cc:" , специфичные для файла исправления. Вывод этой команды должен состоять из одного адреса электронной почты в каждой строке. По умолчанию используется значение конфигурации sendemail.ccCmd .

--header-cmd=<command>

Укажите команду, которая выполняется один раз для каждого исходящего сообщения, и выведите строки заголовка в стиле RFC 2822, которые будут вставлены в них. Если установлена ​​переменная конфигурации sendemail.headerCmd , всегда используется ее значение. Если в командной строке указан параметр --header-cmd, его значение имеет приоритет над переменной конфигурации sendemail.headerCmd .

--no-header-cmd

Отключите любую используемую команду заголовка.

--[no-]chain-reply-to

Если этот параметр установлен, каждое электронное письмо будет отправляться как ответ на предыдущее отправленное электронное письмо. Если отключено с помощью "--no-chain-reply-to" , все электронные письма после первого будут отправляться как ответы на первое отправленное электронное письмо. При его использовании рекомендуется, чтобы первый файл представлял собой обзор всей серии патчей. По умолчанию отключено, но для его включения можно использовать переменную конфигурации sendemail.chainReplyTo .

--identity=<identity>

Идентификатор конфигурации. Если задано, значения в подразделе sendemail.<identity> имеют приоритет над значениями в разделе sendemail . Идентификатор по умолчанию — это значение sendemail.identity .

--[no-]signed-off-by-cc

Если этот параметр установлен, добавьте электронные письма, найденные в трейлере Signed-off-by или строках «Копия:», в список копий. По умолчанию используется значение конфигурации sendemail.signedoffbycc ; если это не указано, по умолчанию используется --signed-off-by-cc.

--[no-]cc-cover

Если этот параметр установлен, электронные письма, найденные в заголовках «Копия:» в первом патче серии (обычно сопроводительное письмо), добавляются в список копий для каждого набора адресов электронной почты. По умолчанию используется значение конфигурации sendemail.cccover ; если это не указано, по умолчанию используется --no-cc-cover.

--[no-]to-cover

Если этот параметр установлен, электронные письма, найденные в заголовках «Кому:» в первом патче серии (обычно сопроводительное письмо), добавляются в список «В список» для каждого набора адресов электронной почты. По умолчанию используется значение конфигурации sendemail.tocover ; если это не указано, по умолчанию используется --no-to-cover.

--suppress-cc=<category>

Укажите дополнительную категорию получателей, чтобы запретить автоматическую копию:

  • author не будет включать автора патча.

  • self не будет включать отправителя.

  • cc не будет включать в заголовок патча кого-либо, упомянутого в строках Cc, за исключением себя (для этого используйте self ).

  • bodycc не будет включать в тело патча кого-либо, упомянутого в строках Cc (сообщение о фиксации), за исключением себя (для этого используйте self ).

  • sob не будет включать кого-либо, упомянутого в трейлерах «Подписано», за исключением себя (для этого используйте self ).

  • misc-by не будет включать в тело патча кого-либо, упомянутого в строках «Подтверждено», «Проверено», «Протестировано» и других строках "-by" , за исключением «Подписано» (для этого используйте sob ).

  • cccmd не будет запускать --cc-cmd.

  • body эквивалентен sob + bodycc + misc-by .

  • all подавляет все значения автоматической копии.

По умолчанию используется значение конфигурации sendemail.suppresscc ; если он не указан, по умолчанию используется self , если указан --suppress-from, а также body , если указан --no-signed-off-cc.

--[no-]suppress-from

Если этот параметр установлен, не добавляйте адрес «От:» в список «Копия:». По умолчанию используется значение конфигурации sendemail.suppressFrom ; если это не указано, по умолчанию используется --no-suppress-from.

--[no-]thread

Если этот параметр установлен, заголовки In-Reply-To и References будут добавлены к каждому отправленному электронному письму. Ссылается ли каждое письмо на предыдущее электронное письмо (последовательность deep согласно формулировке git format-patch ) или на первое электронное письмо (последовательность shallow ), определяется "--[no-]chain-reply-to" .

Если отключено с помощью "--no-thread" , эти заголовки не будут добавлены (если не указано с помощью --in-reply-to). По умолчанию — это значение конфигурации sendemail.thread ; если это не указано, по умолчанию используется --thread.

Пользователь должен убедиться, что заголовок In-Reply-To еще не существует, когда git send-email попросят его добавить (особенно обратите внимание, что git format-patch можно настроить для выполнения самой потоковой передачи). Невыполнение этого требования может не привести к ожидаемому результату в MUA получателя.

Administering

--confirm=<mode>

Подтвердите непосредственно перед отправкой:

  • always всегда подтвердит перед отправкой

  • never никогда не подтвердит перед отправкой

  • cc подтвердит перед отправкой, когда send-email автоматически добавит адреса из патча в список копий.

  • compose подтвердит перед отправкой первого сообщения при использовании --compose.

  • auto эквивалентен cc + compose .

По умолчанию используется значение конфигурации sendemail.confirm ; если он не указан, по умолчанию используется auto , если не указана какая-либо опция подавления, и в этом случае по умолчанию используется compose .

--dry-run

Делайте все, кроме фактической отправки электронных писем.

--[no-]format-patch

Если аргумент можно понимать либо как ссылку, либо как имя файла, выберите его как аргумент исправления формата ( --format-patch ) или как имя файла ( --no-format-patch ). По умолчанию при возникновении такого конфликта отправить электронное письмо git не удастся.

--quiet

Сделайте git-send-email менее многословным. Одна строка для каждого электронного письма должна быть всем, что выводится.

--[no-]validate

Выполните проверку исправности исправлений. В настоящее время валидация означает следующее:

  • Вызовите перехватчик sendemail-validate, если он присутствует (см. githooks[5] ).

  • Предупреждать об исправлениях, содержащих строки длиной более 998 символов, если не используется подходящая кодировка передачи ( auto , base64 или quoted-printable ); это связано с ограничениями SMTP, описанными в https://www.ietf.org/rfc/rfc5322.txt .

По умолчанию используется значение sendemail.validate ; если он не установлен, по умолчанию используется --validate .

--force

Отправляйте электронные письма, даже если проверки безопасности препятствуют этому.

Information

--dump-aliases

Вместо обычной операции выгрузите сокращенные имена псевдонимов из настроенных файлов псевдонимов, по одному на строку в алфавитном порядке. Обратите внимание, что сюда входит только псевдоним, а не расширенные адреса электронной почты. См. sendemail.aliasesfile для получения дополнительной информации о псевдонимах.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

sendemail.identity

Идентификатор конфигурации. Если задано, значения в подразделе sendemail.<identity> имеют приоритет над значениями в разделе sendemail . Идентификатор по умолчанию — это значение sendemail.identity .

sendemail.smtpEncryption

См. описание git-send-email[1] . Обратите внимание, что этот параметр не зависит от механизма identity .

sendemail.smtpsslcertpath

Путь к сертификатам CA (каталог или отдельный файл). Установите пустую строку, чтобы отключить проверку сертификата.

sendemail.<identity>.*

Версии параметров sendemail.* , специфичные для удостоверения, указанные ниже, имеют приоритет над теми, когда это удостоверение выбрано либо через командную строку, либо через sendemail.identity .

sendemail.multiEdit

Если true (по умолчанию), будет создан один экземпляр редактора для редактирования файлов, которые вам нужно редактировать (патчи при использовании --annotate и сводка при использовании --compose ). Если установлено значение false, файлы будут редактироваться один за другим, каждый раз создавая новый редактор.

sendemail.confirm

Устанавливает значение по умолчанию для подтверждения перед отправкой. Должен быть один из always , never , cc , compose или auto . Значение этих значений см. в --confirm в документации git-send-email[1] .

sendemail.aliasesFile

Чтобы не вводить длинные адреса электронной почты, укажите один или несколько файлов псевдонимов электронной почты. Вы также должны предоставить sendemail.aliasFileType .

sendemail.aliasFileType

Формат файла(ов), указанный в sendemail.aliasesFile. Должен быть один из mutt , mailrc , pine , elm , gnus или sendmail .

Как выглядит файл псевдонима в каждом формате, можно узнать из документации одноименной почтовой программы. Отличия и ограничения от стандартных форматов описаны ниже:

sendmail

  • Псевдонимы в кавычках и адреса в кавычках не поддерживаются: строки, содержащие символ " , игнорируются.

  • Перенаправление в файл ( /path/name ) или канал ( |command ) не поддерживается.

  • Включение файлов ( :include: /path/name ) не поддерживается.

  • Предупреждения выводятся в стандартном выводе ошибок для любых явно неподдерживаемых конструкций и любых других строк, которые не распознаются анализатором.

Examples

Используйте Gmail в качестве SMTP-сервера

Чтобы использовать git send-email для отправки исправлений через SMTP-сервер GMail, отредактируйте ~/.gitconfig, указав настройки своей учетной записи:

[sendemail]
        smtpEncryption = tls
        smtpServer = smtp.gmail.com
        smtpUser = [email protected]
        smtpServerPort = 587

Если в вашей учетной записи Gmail настроена многофакторная аутентификация, вы можете создать пароль для конкретного приложения для использования с git send-email . Посетите https://security.google.com/settings/security/apppasswords , чтобы создать его.

Как только ваши коммиты будут готовы к отправке в список рассылки, выполните следующие команды:

$ git format-patch --cover-letter -M origin/master -o outgoing/
$ edit outgoing/0000-*
$ git send-email outgoing/*

При первом запуске вам будет предложено ввести учетные данные. Введите соответствующий пароль приложения или обычный пароль. Если у вас настроен помощник по учетным данным (см. git-credential[1] ), пароль будет сохранен в хранилище учетных данных, поэтому вам не придется вводить его в следующий раз.

Примечание. Требуются следующие основные модули Perl, которые могут быть установлены вместе с вашим дистрибутивом Perl: MIME::Base64, MIME::QuotedPrint, Net::Domain и Net::SMTP. Также необходимы следующие дополнительные модули Perl: Authen::SASL и Mail::Address.
