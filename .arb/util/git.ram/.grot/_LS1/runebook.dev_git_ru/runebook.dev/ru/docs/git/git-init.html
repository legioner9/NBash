git-init

Name

git-init — создать пустой репозиторий Git или повторно инициализировать существующий.

Synopsis

git init [-q | --quiet] [--bare] [--template=<template-directory>]
          [--separate-git-dir <git-dir>] [--object-format=<format>]
          [-b <branch-name> | --initial-branch=<branch-name>]
          [--shared[=<permissions>]] [<directory>]

Description

Эта команда создает пустой репозиторий Git — по сути, каталог .git с подкаталогами для objects , refs/heads , refs/tags и файлов шаблонов. Будет создана начальная ветка без каких-либо коммитов (ее имя см. в опции --initial-branch ниже).

Если установлена ​​переменная среды $GIT_DIR , она указывает путь, который будет использоваться вместо ./.git для базы репозитория.

Если каталог хранения объектов указан через переменную среды $GIT_OBJECT_DIRECTORY , то под ним создаются каталоги sha1; в противном случае используется каталог $GIT_DIR/objects по умолчанию.

Запуск git init в существующем репозитории безопасен. Он не будет перезаписывать уже существующие вещи. Основная причина повторного запуска git init — получение новых добавленных шаблонов (или перемещение репозитория в другое место, если указан --separate-git-dir).

Options

-q
--quiet

Печатать только сообщения об ошибках и предупреждения; все остальные выходные данные будут подавлены.

--bare

Создайте пустой репозиторий. Если среда GIT_DIR не установлена, устанавливается текущий рабочий каталог.

--object-format=<format>

Укажите данный формат объекта (алгоритм хеширования) для репозитория. Допустимые значения: sha1 и (если включено) sha256 . sha1 используется по умолчанию.

Примечание. В настоящее время между репозиториями SHA-256 и SHA-1 отсутствует совместимость.

Исторически мы предупреждали, что репозиториям SHA-256 впоследствии могут потребоваться изменения, несовместимые с предыдущими версиями, когда мы введем такие функции взаимодействия. Сегодня мы ожидаем только совместимых изменений. Более того, если такие изменения окажутся необходимыми, можно ожидать, что репозитории SHA-256, созданные с помощью сегодняшнего Git , можно будет использовать в будущих версиях Git без потери данных.

--template=<template-directory>

Укажите каталог, из которого будут использоваться шаблоны. (См. раздел "TEMPLATE DIRECTORY" ниже.)

--separate-git-dir=<git-dir>

Вместо инициализации репозитория как каталога $GIT_DIR или ./.git/ создайте там текстовый файл, содержащий путь к фактическому репозиторию. Этот файл действует как независимая от файловой системы символическая ссылка Git на репозиторий.

Если это повторная инициализация, репозиторий будет перемещен по указанному пути.

-b <branch-name>
--initial-branch=<branch-name>

Используйте указанное имя для начальной ветки во вновь созданном репозитории. Если не указано, вернитесь к имени по умолчанию (в настоящее время master , но оно может быть изменено в будущем; имя можно настроить с помощью переменной конфигурации init.defaultBranch ).

--shared[=(false|true|umask|group|all|world|everybody|<perm>)]

Укажите, что репозиторий Git будет доступен нескольким пользователям. Это позволяет пользователям, принадлежащим к одной группе, отправлять данные в этот репозиторий. Если указано, переменная конфигурации "core.sharedRepository" задается таким образом, что файлы и каталоги под $GIT_DIR создаются с запрошенными разрешениями. Если не указано иное, Git будет использовать разрешения, сообщаемые umask(2).

Опция может иметь следующие значения, по умолчанию используется group , если значение не указано:

маска (или ложь)

Разрешения на использование, предоставленные umask(2). Значение по умолчанию, если не указан --shared .

группа (или правда)

Сделайте группу репозитория доступной для записи (и g+sx, поскольку группа git может не быть основной группой для всех пользователей). Это используется для ослабления разрешений безопасного значения umask(2). Обратите внимание, что umask по-прежнему применяется к другим битам разрешений (например, если umask имеет значение 0022 , использование group не приведет к удалению привилегий чтения у других (не входящих в группу) пользователей). См. 0xxx , чтобы узнать, как точно указать разрешения репозитория.

все (или мир или все)

То же, что и group , но сделайте репозиторий доступным для чтения всем пользователям.

<perm>

<perm> — это трехзначное восьмеричное число с префиксом 0 , и каждый файл будет иметь режим <perm> . <perm> переопределяет значение umask(2) пользователя (а не только ослабляет разрешения, как это делают group и all ). 0640 создаст репозиторий, доступный для группового чтения, но не доступный для групповой записи или доступный другим. 0660 создаст репозиторий, доступный для чтения и записи текущему пользователю и группе, но недоступный для других (каталоги и исполняемые файлы получают бит x из бита r для соответствующих классов пользователей).

По умолчанию флаг конфигурации receive.denyNonFastForwards включен в общих репозиториях, поэтому вы не можете принудительно выполнить в него передачу без быстрой пересылки.

Если вы предоставляете directory , команда запускается внутри него. Если этот каталог не существует, он будет создан.

Template directory

Файлы и каталоги в каталоге шаблона, имена которых не начинаются с точки, будут скопированы в $GIT_DIR после его создания.

Каталог шаблона будет одним из следующих (по порядку):

  • аргумент, заданный опцией --template ;

  • содержимое переменной среды $GIT_TEMPLATE_DIR ;

  • переменная конфигурации init.templateDir ; или

  • каталог шаблонов по умолчанию: /usr/share/git-core/templates .

Каталог шаблонов по умолчанию включает некоторую структуру каталогов, предложенную "exclude patterns" (см. gitignore[5] ), и примеры файлов ловушек.

По умолчанию все хуки сэмплов отключены. Чтобы включить один из примеров перехватчиков, переименуйте его, удалив суффикс .sample .

См. githooks[5] для получения более общей информации о выполнении перехвата.

Examples

Запустите новый репозиторий Git для существующей базы кода.
$ cd /path/to/my/codebase
$ git init      (1)
$ git add .     (2)
$ git commit    (3)

  1. Создайте каталог /path/to/my/codebase/.git.

  2. Добавьте все существующие файлы в индекс.

  3. Запишите исходное состояние как первый коммит в истории.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

init.templateDir

Укажите каталог, из которого будут копироваться шаблоны. (См. раздел "TEMPLATE DIRECTORY" в git-init[1] .)

init.defaultBranch

Позволяет переопределить имя ветки по умолчанию, например, при инициализации нового репозитория.
