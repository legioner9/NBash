git-checkout

Name

git-checkout — переключить ветки или восстановить файлы рабочего дерева

Synopsis

git checkout [-q] [-f] [-m] [<branch>]
git checkout [-q] [-f] [-m] --detach [<branch>]
git checkout [-q] [-f] [-m] [--detach] <commit>
git checkout [-q] [-f] [-m] [[-b|-B|--orphan] <new-branch>] [<start-point>]
git checkout [-f] <tree-ish> [--] <pathspec>…​
git checkout [-f] <tree-ish> --pathspec-from-file=<file> [--pathspec-file-nul]
git checkout [-f|--ours|--theirs|-m|--conflict=<style>] [--] <pathspec>…​
git checkout [-f|--ours|--theirs|-m|--conflict=<style>] --pathspec-from-file=<file> [--pathspec-file-nul]
git checkout (-p|--patch) [<tree-ish>] [--] [<pathspec>…​]

Description

Обновляет файлы в рабочем дереве, чтобы они соответствовали версии в индексе или указанном дереве. Если спецификация пути не указана, git checkout также обновит HEAD , чтобы установить указанную ветвь в качестве текущей.

git касса [<филиал>]

Чтобы подготовиться к работе над <branch> , переключитесь на него, обновив индекс и файлы в рабочем дереве, и указав HEAD на ветку. Локальные изменения файлов в рабочем дереве сохраняются, поэтому их можно зафиксировать в <branch> .

Если <branch> не найден, но существует ветвь отслеживания ровно на одном удаленном компьютере (назовем его <remote> ) с совпадающим именем и --no-guess не указан, рассматривайте его как эквивалент

$ git checkout -b <branch> --track <remote>/<branch>

Вы можете опустить <branch> , и в этом случае команда вырождается в "check out the current branch" , который представляет собой прославленную пустую операцию с довольно дорогостоящими побочными эффектами, позволяющую отображать только информацию отслеживания, если она существует, для текущей ветки.

git checkout -b|-B <новая-ветвь> [<начальная точка>]

Указание -b приводит к созданию новой ветки, как если бы git-branch[1] был вызван, а затем извлечен. В этом случае вы можете использовать опции --track или --no-track , которые будут переданы git branch . Для удобства --track без -b подразумевает создание ветвей; см. описание --track ниже.

Если задан -B , создается <new-branch> , если он не существует; в противном случае он сбрасывается. Это транзакционный эквивалент

$ git branch -f <branch> [<start-point>]
$ git checkout <branch>

то есть ветвь не сбрасывается/создается, пока "git checkout" не завершится успешно.

git оформление заказа --detach [<ветвь>]
git оформление заказа [--detach] <фиксация>

Подготовьтесь к работе поверх <commit> , отсоединив от него HEAD (см. раздел "DETACHED HEAD" ) и обновив индекс и файлы в рабочем дереве. Локальные изменения файлов в рабочем дереве сохраняются, так что результирующее рабочее дерево будет иметь состояние, записанное в коммите, плюс локальные изменения.

Если аргумент <commit> является именем ветки, параметр --detach можно использовать для отсоединения HEAD на кончике ветки ( git checkout <branch> проверит эту ветвь, не отключая HEAD ).

Если пропустить <branch> , HEAD отсоединится на кончике текущей ветви.

git checkout [-f|--наши|--их|-m|--conflict=<стиль>] [<дерево>] [--] <pathspec>…
git checkout [-f|--наши|--их|-m|--conflict=<стиль>] [<дерево-иш>] --pathspec-from-file=<file> [--pathspec-file- ноль]

Перезапишите содержимое файлов, соответствующих указанному пути. Если <tree-ish> (чаще всего фиксация) не указан, перезапишите рабочее дерево содержимым индекса. Когда указан <tree-ish> , перезапишите индекс и рабочее дерево содержимым <tree-ish> .

Индекс может содержать неслитые записи из-за предыдущего неудачного слияния. По умолчанию, если вы попытаетесь извлечь такую ​​запись из индекса, операция извлечения завершится неудачно, и ничего не будет извлечено. Использование -f будет игнорировать эти несвязанные записи. Содержимое определенной стороны слияния можно извлечь из индекса с помощью --ours или --theirs . С помощью -m изменения, внесенные в файл рабочего дерева, можно отменить, чтобы воссоздать исходный конфликтный результат слияния.

git checkout (-p|--patch) [<tree-ish>] [--] [<pathspec>…​]

Это похоже на предыдущий режим, но позволяет вам использовать интерактивный интерфейс, чтобы показать выходные данные "diff" и выбрать, какие фрагменты использовать в результате. Ниже приведено описание опции --patch .

Options

-q
--quiet

Тихо, подавляйте сообщения обратной связи.

--progress
--no-progress

Статус выполнения по умолчанию сообщается в стандартном потоке ошибок, когда он подключен к терминалу, если не указан --quiet . Этот флаг включает отчет о ходе выполнения, даже если он не подключен к терминалу, независимо от --quiet .

-f
--force

При переключении веток действуйте, даже если индекс или рабочее дерево отличаются от HEAD , и даже если на пути есть неотслеживаемые файлы. Это используется для удаления локальных изменений и любых неотслеживаемых файлов или каталогов, которые мешают.

При проверке путей из индекса не допускайте сбоев при неслитых записях; вместо этого несвязанные записи игнорируются.

--ours
--theirs

При проверке путей из индекса проверьте этап № 2 ( ours ) или № 3 ( theirs ) на наличие несвязанных путей.

Обратите внимание, что во время git rebase и git pull --rebase ours и theirs могут отображаться перепутанными; --ours предоставляет версию из ветки, на которую перебазируются изменения, а --theirs предоставляет версию из ветки, в которой хранится ваша работа, которая перебазируется.

Это связано с тем, что rebase используется в рабочем процессе, который рассматривает историю на удаленном компьютере как общую каноническую, а работу, выполненную в ветке, которую вы перебазируете, рассматривает как стороннюю работу, подлежащую интеграции, и вы временно принимаете на себя эту роль. хранителя канонической истории во время перебазирования. Как хранителю канонической истории, вам необходимо просматривать историю с пульта как ours (т.е. "our shared canonical history" ), а то, что вы делали на боковой ветке, как theirs (т.е. "one contributor’s work on top of it" ).

-b <new-branch>

Создайте новую ветку с именем <new-branch> , запустите ее с <start-point> и проверьте полученную ветку; подробности см. в git-branch[1] .

-B <new-branch>

Создает ветку <new-branch> , запускает ее по адресу <start-point> ; если он уже существует, то сбросьте его на <start-point> . А затем проверьте полученную ветку. Это эквивалентно запуску "git branch" с "-f" , за которым следует "git checkout" этой ветки; подробности см. в git-branch[1] .

-t
--track[=(direct|inherit)]

При создании новой ветки настройте конфигурацию "upstream" . Подробности см. в "--track" и git-branch[1] .

Если опция -b не указана, имя новой ветки будет получено из ветки удаленного отслеживания путем просмотра локальной части спецификации ссылки, настроенной для соответствующего удаленного устройства, а затем удаления начальной части до "*" . Это подскажет нам использовать hack в качестве локальной ветви при ответвлении от origin/hack (или remotes/origin/hack , или даже refs/remotes/origin/hack ). Если данное имя не имеет косой черты или в результате приведенного выше угадывания имя остается пустым, угадывание прерывается. В таком случае вы можете явно указать имя с помощью -b .

--no-track

Не настраивайте конфигурацию "upstream" , даже если переменная конфигурации branch.autoSetupMerge имеет значение true.

--guess
--no-guess

Если <branch> не найден, но существует ветвь отслеживания ровно на одном удаленном компьютере (назовем его <remote> ) с совпадающим именем, рассматривайте его как эквивалент

$ git checkout -b <branch> --track <remote>/<branch>

Если ветвь существует на нескольких удаленных устройствах и один из них назван конфигурационной переменной checkout.defaultRemote , мы будем использовать ее в целях устранения неоднозначности, даже если <branch> не уникален для всех удаленных устройств. Установите его, например, на checkout.defaultRemote=origin , чтобы всегда получать оттуда удаленные ветки, если <branch> неоднозначен, но существует на удаленном origin . См. также checkout.defaultRemote в git-config[1] .

--guess — поведение по умолчанию. Используйте --no-guess , чтобы отключить его.

Поведение по умолчанию можно установить с помощью переменной конфигурации checkout.guess .

-l

Создайте рефлог новой ветки; подробности см. в git-branch[1] .

-d
--detach

Вместо того, чтобы проверять ветку для работы над ней, проверьте коммит для проверки и отбрасываемых экспериментов. Это поведение git checkout <commit> по умолчанию, когда <commit> не является именем ветки. Подробности см. в разделе "DETACHED HEAD" ниже.

--orphan <new-branch>

Создайте новую неродившуюся ветку с именем <new-branch> , начав с <start-point> , и переключитесь на нее. Первый коммит, сделанный в этой новой ветке, не будет иметь родителей и станет корнем новой истории, полностью оторванной от всех других ветвей и коммитов.

Индекс и рабочее дерево настраиваются так, как если бы вы ранее запускали git checkout <start-point> . Это позволяет вам начать новую историю, в которой записывается набор путей, аналогичных <start-point> , путем простого запуска git commit -a для выполнения корневого коммита.

Это может быть полезно, если вы хотите опубликовать дерево из коммита, не раскрывая его полную историю. Возможно, вы захотите сделать это, чтобы опубликовать ветку проекта с открытым исходным кодом, текущее дерево которого — "clean" , но полная история которого содержит проприетарные или иным образом обремененные фрагменты кода.

Если вы хотите запустить отключенную историю, записывающую набор путей, полностью отличающийся от пути <start-point> , то вам следует очистить индекс и рабочее дерево сразу после создания потерянной ветки, запустив git rm -rf . с верхнего уровня рабочего дерева. дерево. После этого вы будете готовы подготовить новые файлы, заново заполнить рабочее дерево, скопировать их из другого места, распаковать архив и т. д.

--ignore-skip-worktree-bits

В режиме разреженной проверки git checkout -- <paths> будет обновлять только записи, соответствующие <paths> , и разреженные шаблоны в $GIT_DIR/info/sparse-checkout . Этот параметр игнорирует разреженные шаблоны и добавляет обратно все файлы в <paths> .

-m
--merge

При переключении веток, если у вас есть локальные изменения в одном или нескольких файлах, которые отличаются в текущей ветке и ветке, на которую вы переключаетесь, команда отказывается переключать ветки, чтобы сохранить ваши изменения в контексте. Однако при использовании этой опции трехстороннее слияние текущей ветки, содержимого вашего рабочего дерева и новой ветки выполнено, и вы окажетесь в новой ветке.

Когда происходит конфликт слияния, записи индекса для конфликтующих путей остаются несвязанными, и вам необходимо разрешить конфликты и пометить разрешенные пути с помощью git add (или git rm , если слияние должно привести к удалению пути).

При извлечении путей из индекса этот параметр позволяет воссоздать конфликтующее слияние в указанных путях. Эту опцию нельзя использовать при проверке путей из дерева.

При переключении ветвей с помощью --merge поэтапные изменения могут быть потеряны.

--conflict=<style>

То же, что и опция --merge выше, но меняет способ представления конфликтующих фрагментов, переопределяя переменную конфигурации merge.conflictStyle . Возможные значения: "merge" (по умолчанию), "diff3" и "zdiff3" .

-p
--patch

В интерактивном режиме выберите фрагменты разницы между <tree-ish> (или индексом, если он не указан) и рабочим деревом. Выбранные фрагменты затем применяются в обратном порядке к рабочему дереву (и, если был указан <tree-ish> , к индексу).

Это означает, что вы можете использовать git checkout -p для выборочного отклонения изменений из текущего рабочего дерева. См. раздел «Интерактивный режим» git-add[1] , чтобы узнать, как работать в режиме --patch .

Обратите внимание, что этот параметр по умолчанию использует режим без наложения (см. также --overlay ) и в настоящее время не поддерживает режим наложения.

--ignore-other-worktrees

git checkout отказывается, когда требуемая ссылка уже извлечена другим рабочим деревом. Эта опция в любом случае заставляет его проверять ссылку. Другими словами, ссылка может храниться более чем в одном рабочем дереве.

--overwrite-ignore
--no-overwrite-ignore

Тихо перезаписывать игнорируемые файлы при переключении ветвей. Это поведение по умолчанию. Используйте --no-overwrite-ignore , чтобы прервать операцию, если новая ветвь содержит игнорируемые файлы.

--recurse-submodules
--no-recurse-submodules

Использование --recurse-submodules обновит содержимое всех активных подмодулей в соответствии с коммитом, записанным в суперпроекте. Если локальные изменения в подмодуле будут перезаписаны, проверка завершится неудачей, если не используется -f . Если ничего (или --no-recurse-submodules ) не используется, рабочие деревья подмодулей обновляться не будут. Как и в случае с git-submodule[1] , это позволит отсоединить HEAD от субмодуля.

--overlay
--no-overlay

В режиме наложения по умолчанию git checkout никогда не удаляет файлы из индекса или рабочего дерева. При указании --no-overlay файлы, которые появляются в индексе и рабочем дереве, но не в <tree-ish> , удаляются, чтобы они точно соответствовали <tree-ish> .

--pathspec-from-file=<file>

Спецификация пути передается в <file> вместо аргументов командной строки. Если <file> — это точно - , то используется стандартный ввод. Элементы Pathspec разделяются LF или CR/LF. Элементы Pathspec можно заключать в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). См. также --pathspec-file-nul и глобальный --literal-pathspecs .

--pathspec-file-nul

Имеет смысл только с --pathspec-from-file . Элементы Pathspec разделяются символом NUL, а все остальные символы воспринимаются буквально (включая символы новой строки и кавычки).

<branch>

Отделение для оформления заказа; если оно ссылается на ветку (т. е. имя, которое, если добавить к нему "refs/heads/" , является допустимой ссылкой), то эта ветка извлекается. В противном случае, если это относится к действительному коммиту, ваш HEAD становится "detached" , и вы больше не находитесь ни в одной ветке (подробности см. ниже).

Вы можете использовать синтаксис @{-N} для ссылки на N-ю последнюю ветвь/фиксацию, полученную с помощью операции "git checkout" . Вы также можете указать - , который является синонимом @{-1} .

В частном случае вы можете использовать A...B в качестве ярлыка для базы слияния A и B , если существует ровно одна база слияния. Вы можете исключить не более одного из A и B , в этом случае по умолчанию используется HEAD .

<new-branch>

Название новой ветки.

<start-point>

Имя коммита, с которого начинается новая ветка; подробности см. в git-branch[1] . По умолчанию HEAD .

В частном случае вы можете использовать "A...B" в качестве ярлыка для базы слияния A и B , если существует ровно одна база слияния. Вы можете исключить не более одного из A и B , в этом случае по умолчанию используется HEAD .

<tree-ish>

Дерево для оформления заказа (если заданы пути). Если не указано, будет использоваться индекс.

В частном случае вы можете использовать "A...B" в качестве ярлыка для базы слияния A и B , если существует ровно одна база слияния. Вы можете исключить не более одного из A и B , в этом случае по умолчанию используется HEAD .

--

Не интерпретируйте дополнительные аргументы как варианты.

<pathspec>…

Ограничивает пути, на которые влияет операция.

Дополнительные сведения см. в записи pathspec в gitglossary[7] .

Detached head

HEAD обычно относится к именованной ветке (например, master ). При этом каждая ветка относится к определенному коммиту. Давайте посмотрим на репозиторий с тремя коммитами, один из которых помечен, и с извлеченной веткой master :

           HEAD (refers to branch 'master')
            |
            v
a---b---c  branch 'master' (refers to commit 'c')
    ^
    |
  tag 'v2.0' (refers to commit 'b')

Когда в этом состоянии создается коммит, ветка обновляется и ссылается на новый коммит. В частности, git commit создает новый коммит d , родителем которого является коммит c , а затем обновляет ветвь master , чтобы она ссылалась на новый коммит d . HEAD по-прежнему ссылается на ветку master и, следовательно, теперь косвенно ссылается на коммит d :

$ edit; git add; git commit

               HEAD (refers to branch 'master')
                |
                v
a---b---c---d  branch 'master' (refers to commit 'd')
    ^
    |
  tag 'v2.0' (refers to commit 'b')

Иногда полезно иметь возможность извлечь коммит, который не находится на вершине какой-либо именованной ветки, или даже создать новый коммит, на который не ссылается именованная ветвь. Давайте посмотрим, что происходит, когда мы извлекаем фиксацию b (здесь мы показываем два способа это сделать):

$ git checkout v2.0  # or
$ git checkout master^^

   HEAD (refers to commit 'b')
    |
    v
a---b---c---d  branch 'master' (refers to commit 'd')
    ^
    |
  tag 'v2.0' (refers to commit 'b')

Обратите внимание: независимо от того, какую команду оформления заказа мы используем, HEAD теперь напрямую ссылается на фиксацию b . Это называется состоянием отсоединенного HEAD . Это просто означает, что HEAD ссылается на конкретный коммит, а не на именованную ветку. Давайте посмотрим, что происходит, когда мы создаем коммит:

$ edit; git add; git commit

     HEAD (refers to commit 'e')
      |
      v
      e
     /
a---b---c---d  branch 'master' (refers to commit 'd')
    ^
    |
  tag 'v2.0' (refers to commit 'b')

Теперь есть новый коммит e , но на него ссылается только HEAD . Мы, конечно, можем добавить еще один коммит в этом состоянии:

$ edit; git add; git commit

         HEAD (refers to commit 'f')
          |
          v
      e---f
     /
a---b---c---d ветка 'master' (относится к фиксации 'd')
    ^
    |
  tag 'v2.0' (refers to commit 'b')

Фактически, мы можем выполнять все обычные операции Git . Но давайте посмотрим, что произойдет, когда мы затем проверим master :

$ git checkout master

               HEAD (refers to branch 'master')
      e---f     |
     /          v
a---b---c---d  branch 'master' (refers to commit 'd')
    ^
    |
  tag 'v2.0' (refers to commit 'b')

Важно понимать, что на данный момент ничего не относится к коммиту f . В конечном итоге коммит f (и, как следствие, коммит e ) будет удален обычным процессом сборки мусора Git , если только мы не создадим ссылку до того, как это произойдет. Если мы еще не отошли от коммита f , любой из них создаст ссылку на него:

$ git checkout -b foo  # или  "git switch -c foo"  (1)
$ git branch foo                                 (2)
$ git tag foo                                    (3)

  1. создает новую ветвь foo , которая ссылается на фиксацию f , а затем обновляет HEAD , чтобы она ссылалась на ветвь foo . Другими словами, после этой команды мы больше не будем находиться в состоянии отсоединенного HEAD .

  2. аналогичным образом создаётся новая ветка foo , которая ссылается на коммит f , но оставляет HEAD отсоединённой.

  3. создает новый тег foo , который ссылается на фиксацию f , оставляя HEAD отключенным.

Если мы отошли от фиксации f , то сначала мы должны восстановить имя его объекта (обычно с помощью рефлога git ), а затем мы можем создать ссылку на него. Например, чтобы увидеть два последних коммита, на которые ссылается HEAD , мы можем использовать любую из этих команд:

$ git reflog -2 HEAD # or
$ git log -g -2 HEAD

Argument disambiguation

Когда задан только один аргумент, и это не -- (например, git checkout abc ), и когда аргумент является одновременно допустимым <tree-ish> (например, существует ветвь abc ) и допустимым <pathspec> (например, существует файл или каталог с именем "abc" ) , Git обычно просит вас устранить неоднозначность. Однако, поскольку проверка ветки является довольно распространенной операцией, git checkout abc в такой ситуации воспринимает "abc" как <tree-ish> . Используйте git checkout -- <pathspec> , если вы хотите извлечь эти пути из индекса.

Examples

1. Paths

Следующая последовательность проверяет ветвь master , возвращает Makefile на две ревизии назад, по ошибке удаляет hello.c и возвращает его из индекса.

$ git checkout master             (1)
$ git checkout master~2 Makefile  (2)
$ rm -f hello.c
$ git checkout hello.c            (3)

  1. switch branch

  2. взять файл из другого коммита

  3. восстановить hello.c из индекса

Если вы хотите получить исходные файлы all C из индекса, вы можете сказать

$ git checkout -- '*.c'

Обратите внимание на кавычки вокруг *.c . Файл hello.c также будет извлечен, хотя его больше нет в рабочем дереве, поскольку подстановка файлов используется для сопоставления записей в индексе (а не в рабочем дереве оболочки).

Если у вас есть неудачная ветка с именем hello.c , этот шаг будет воспринят как инструкция по переключению на эту ветку. Вместо этого вам следует написать:

$ git checkout -- hello.c

2. Merge

После работы не в той ветке переключение на правильную ветку будет осуществляться с помощью:

$ git checkout mytopic

Однако ваша ветка "wrong" и правильная ветка mytopic могут отличаться в файлах, которые вы изменили локально, и в этом случае приведенная выше проверка завершится неудачей следующим образом:

$ git checkout mytopic
error: You have local changes to 'frotz'; not switching branches.

Вы можете указать флаг -m команде, которая попытается выполнить трехстороннее слияние:

$ git checkout -m mytopic
Auto-merging frotz

После этого трехстороннего слияния локальные изменения not регистрируются в вашем индексном файле, поэтому git diff покажет вам, какие изменения вы внесли с момента начала новой ветки.

3. Объединение конфликта

Когда конфликт слияния происходит во время переключения ветвей с опцией -m , вы увидите что-то вроде этого:

$ git checkout -m mytopic
Auto-merging frotz
ERROR: Merge conflict in frotz
fatal: merge program failed

На этом этапе git diff показывает чисто объединенные изменения, как в предыдущем примере, а также изменения в конфликтующих файлах. Отредактируйте и разрешите конфликт и отметьте его как разрешенный с помощью git add , как обычно:

$ edit frotz
$ git add frotz

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

checkout.defaultRemote

Если вы используете git checkout <something> или git switch <something> и у вас есть только один пульт дистанционного управления, он может неявно вернуться к проверке и отслеживанию, например origin/<something> . Это перестанет работать, как только у вас будет более одного пульта с кодом <something> . Этот параметр позволяет указать имя предпочтительного пульта, который всегда должен выигрывать, когда дело доходит до устранения неоднозначности. Типичный вариант использования — установить для этого параметра значение origin .

В настоящее время это используется git-switch[1] и git-checkout[1] , когда git checkout <something> или git switch <something> проверяет ветку <something> на другом удаленном устройстве, и git-worktree[1] , когда git worktree add ссылается на удаленную ветку. Этот параметр может использоваться для других команд или функций, подобных оформлению заказа, в будущем.

checkout.guess

Предоставляет значение по умолчанию для параметра --guess или --no-guess в git checkout и git switch . См. git-switch[1] и git-checkout[1] .

checkout.workers

Количество параллельных рабочих процессов, используемых при обновлении рабочего дерева. По умолчанию установлено одно, т.е. последовательное выполнение. Если установлено значение меньше единицы, Git будет использовать столько рабочих процессов, сколько доступно логических ядер. Этот параметр и checkout.thresholdForParallelism влияют на все команды, выполняющие проверку. Например, проверка, клонирование, сброс, разреженная проверка и т. д.

Примечание. Параллельная проверка обычно обеспечивает более высокую производительность для репозиториев, расположенных на твердотельных накопителях или через NFS. Для репозиториев на вращающихся дисках и/или машинах с небольшим количеством ядер последовательная проверка по умолчанию часто работает лучше. Размер и уровень сжатия репозитория также могут влиять на производительность параллельной версии.

checkout.thresholdForParallelism

При параллельном извлечении небольшого количества файлов затраты на создание подпроцессов и межпроцессное взаимодействие могут перевесить выгоды от распараллеливания. Этот параметр позволяет определить минимальное количество файлов, для которых следует попытаться выполнить параллельное извлечение. По умолчанию — 100.
