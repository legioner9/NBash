gitmodules

Name

gitmodules — Определение свойств подмодуля

Synopsis

$GIT_WORK_TREE/.gitmodules

Description

Файл .gitmodules , расположенный в каталоге верхнего уровня рабочего дерева Git , представляет собой текстовый файл с синтаксисом, соответствующим требованиям git-config[1] .

Файл содержит один подраздел для каждого подмодуля, а значение подраздела — это имя подмодуля. В качестве имени указывается путь, по которому был добавлен подмодуль, если он не был настроен с помощью опции --name в git submodule add . Каждый раздел подмодуля также содержит следующие обязательные ключи:

submodule.<name>.path

Определяет путь относительно каталога верхнего уровня рабочего дерева Git , куда предполагается извлечь подмодуль. Имя пути не должно заканчиваться на / . Все пути к субмодулям должны быть уникальными в файле .gitmodules .

submodule.<name>.url

Определяет URL-адрес, с которого можно клонировать репозиторий подмодуля. Это может быть либо абсолютный URL-адрес, готовый для передачи в git-clone[1] , либо (если он начинается с ./ или ../ ) местоположение относительно исходного репозитория суперпроекта.

Кроме того, имеется ряд дополнительных клавиш:

submodule.<name>.update

Определяет процедуру обновления по умолчанию для именованного субмодуля, т. е. то, как субмодуль обновляется командой git submodule update в суперпроекте. Это используется только git submodule init для инициализации одноименной переменной конфигурации. Допустимые значения здесь: checkout , rebase , merge или none , но не !command (по соображениям безопасности). Дополнительные сведения см. в описании команды update в git-submodule[1] .

submodule.<name>.branch

Имя удаленной ветки для отслеживания обновлений в вышестоящем подмодуле. Если опция не указана, по умолчанию используется удаленный HEAD . Специальное значение . используется для указания того, что имя ветки в подмодуле должно совпадать с именем текущей ветки в текущем репозитории. Подробности см. в документации --remote в git-submodule[1] .

submodule.<name>.fetchRecurseSubmodules

Эту опцию можно использовать для управления рекурсивной выборкой этого подмодуля. Если эта опция также присутствует в записи подмодуля в .git/config суперпроекта, настройка там будет переопределять настройку, найденную в .gitmodules . Обе настройки можно переопределить в командной строке, используя параметр --[no-]recurse-submodules для git fetch и git pull .

submodule.<name>.ignore

Определяет, при каких обстоятельствах git status и семейство diff отображают подмодуль как измененный. Поддерживаются следующие значения:

all

Субмодуль никогда не будет считаться измененным (но, тем не менее, он будет отображаться в выводе статуса и фиксации после его подготовки).

dirty

Все изменения в дереве работы субмодуля будут игнорироваться, учитываются только зафиксированные различия между HEAD субмодуля и его записанным состоянием в суперпроекте.

untracked

Будут игнорироваться только неотслеживаемые файлы в подмодулях. Отобразятся зафиксированные различия и изменения в отслеживаемых файлах.

none

Никакие изменения в подмодулях не игнорируются, отображаются все зафиксированные различия, а также изменения в отслеживаемых и неотслеживаемых файлах. Это опция по умолчанию.

Если эта опция также присутствует в записи подмодуля в .git/config суперпроекта, настройка там будет переопределять настройку, найденную в .gitmodules .

Обе настройки можно переопределить в командной строке с помощью параметра --ignore-submodules . Эта настройка не влияет на команды git submodule .

submodule.<name>.shallow

Если установлено значение true, клон этого подмодуля будет выполняться как поверхностное клонирование (с глубиной истории 1), если только пользователь явно не запросит немелкое клонирование.

Notes

Git не позволяет файлу .gitmodules в рабочем дереве быть символической ссылкой и отказывается извлекать такую ​​запись дерева. Это обеспечивает единообразие поведения при доступе к файлу из индекса или дерева, а не из файловой системы, и помогает Git надежно обеспечивать проверку безопасности содержимого файла.

Examples

Рассмотрим следующий файл .gitmodules :

[submodule "libfoo"]
        path = include/foo
        url = git://foo.com/git/lib.git

[submodule "libbar"]
        path = include/bar
        url = git://bar.com/git/lib.git

Это определяет два субмодуля: libfoo и libbar . Ожидается, что они будут извлечены по путям include/foo и include/bar , и для обоих подмодулей указан URL-адрес, который можно использовать для клонирования подмодулей.
