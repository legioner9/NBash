git-index-pack

Name

git-index-pack — создать индексный файл пакета для существующего упакованного архива.

Synopsis

git index-pack [-v] [-o <index-file>] [--[no-]rev-index] <pack-file>
git index-pack --stdin [--fix-thin] [--keep] [-v] [-o <index-file>]
                  [--[no-]rev-index] [<pack-file>]

Description

Считывает упакованный архив (.pack) из указанного файла, создает для него индексный файл пакета (.idx) и при необходимости записывает обратный индекс (.rev) для указанного пакета. Упакованный архив вместе с индексом пакета можно затем поместить в каталог объектов/pack/ репозитория Git .

Options

-v

Подробно рассказывайте о том, что происходит, включая статус прогресса.

-o <index-file>

Запишите сгенерированный индекс пакета в указанный файл. Без этой опции имя индексного файла пакета создается из имени упакованного файла архива путем замены .pack на .idx (и программа не работает, если имя упакованного архива не заканчивается на .pack).

--[no-]rev-index

Если указан этот флаг, сгенерируйте обратный индекс (файл .rev ), соответствующий данному пакету. Если указан --verify , убедитесь, что существующий обратный индекс верен. Имеет приоритет над pack.writeReverseIndex .

--stdin

Если указан этот флаг, вместо этого пакет считывается со стандартного ввода, а затем копия записывается в <pack-file>. Если <pack-file> не указан, пакет записывается в каталог Objects/pack/ текущего репозитория Git с именем по умолчанию, определенным из содержимого пакета. Если <pack-file> не указан, рассмотрите возможность использования --keep, чтобы предотвратить состояние гонки между этим процессом и git repack .

--fix-thin

Исправьте пакет "thin" , созданный git pack-objects --thin (подробнее см. в git-pack-objects[1] ), добавив в пакет исключенные объекты, на которых основаны дельтифицированные объекты. Эта опция имеет смысл только в сочетании с --stdin.

--keep

Прежде чем перемещать индекс в конечный пункт назначения, создайте пустой файл .keep для соответствующего файла пакета. Эта опция обычно необходима вместе с --stdin, чтобы предотвратить одновременное удаление вновь созданного пакета и индекса процессом git repack до того, как ссылки смогут быть обновлены для использования объектов, содержащихся в пакете.

--keep=<msg>

Как и --keep, создайте файл .keep перед перемещением индекса в конечный пункт назначения. Однако вместо создания пустого файла поместите <msg> , за которым следует LF, в файл .keep. Сообщение <msg> позже можно будет найти во всех файлах .keep, чтобы найти те, которые устарели.

--index-version=<version>[,<offset>]

Это предназначено для использования только в наборе тестов. Это позволяет принудительно использовать версию сгенерированного индекса пакета и принудительно использовать 64-битные записи индекса для объектов, расположенных выше заданного смещения.

--strict

Умрет, если в паке есть сломанные предметы или ссылки.

--progress-title

Только для внутреннего использования.

Установите заголовок индикатора выполнения. По умолчанию заголовок — "Receiving objects" , а если указан --stdin , — "Indexing objects" .

--check-self-contained-and-connected

Умрет, если в паке есть неработающие ссылки. Только для внутреннего использования.

--fsck-objects

Только для внутреннего использования.

Умрите, если в пакете есть сломанные предметы. Если пакет содержит дерево, указывающее на несуществующий большой двоичный объект .gitmodules, печатает хэш этого большого двоичного объекта (для проверки вызывающей стороной) после хэша, который входит в имя файла пакета/idx (см. "Notes" ).

--threads=<n>

Указывает количество потоков, которые будут создаваться при разрешении дельт. Для этого необходимо, чтобы index-pack был скомпилирован с pthreads, в противном случае эта опция игнорируется с предупреждением. Это предназначено для сокращения времени упаковки на многопроцессорных машинах. Однако необходимый объем памяти для окна дельта-поиска умножается на количество потоков. Если указать 0, Git автоматически определит количество процессоров и будет использовать максимум 3 потока.

--max-input-size=<size>

Умрет, если упаковка больше <размера>.

--object-format=<hash-algorithm>

Укажите данный формат объекта (алгоритм хеширования) для пакета. Допустимые значения: sha1 и (если включено) sha256 . По умолчанию используется алгоритм для текущего репозитория (устанавливается extensions.objectFormat ) или sha1 , если значение не установлено или находится за пределами репозитория.

Эту опцию нельзя использовать с --stdin.

Примечание. В настоящее время между репозиториями SHA-256 и SHA-1 отсутствует совместимость.

Исторически мы предупреждали, что репозиториям SHA-256 впоследствии могут потребоваться изменения, несовместимые с предыдущими версиями, когда мы введем такие функции взаимодействия. Сегодня мы ожидаем только совместимых изменений. Более того, если такие изменения окажутся необходимыми, можно ожидать, что репозитории SHA-256, созданные с помощью сегодняшнего Git , можно будет использовать в будущих версиях Git без потери данных.

--promisor[=<message>]

Прежде чем зафиксировать индекс пакета, создайте файл .promisor для этого пакета. Особенно полезно при написании пакета обещаний с помощью --fix-thin, поскольку имя пакета не является окончательным, пока пакет не будет полностью написан. Если указан <message> , то это содержимое будет записано в файл .promisor для дальнейшего использования. См. partial clone для получения дополнительной информации.

Notes

После создания индекса хэш, который входит в имя файла package/idx, выводится на стандартный вывод. Если также использовался --stdin, то перед ним ставится префикс "pack\t" или "keep\t" , если новый файл .keep был успешно создан. Это полезно для удаления файла .keep, используемого в качестве блокировки, чтобы предотвратить гонку с git repack , упомянутую выше.
