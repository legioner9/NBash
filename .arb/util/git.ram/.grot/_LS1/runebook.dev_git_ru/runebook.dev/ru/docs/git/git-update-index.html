git-update-index

Name

git-update-index — зарегистрировать содержимое файла в рабочем дереве в индексе

Synopsis

git update-index
             [--add] [--remove | --force-remove] [--replace]
             [--refresh] [-q] [--unmerged] [--ignore-missing]
             [(--cacheinfo <mode>,<object>,<file>)…​]
             [--chmod=(+|-)x]
             [--[no-]assume-unchanged]
             [--[no-]skip-worktree]
             [--[no-]ignore-skip-worktree-entries]
             [--[no-]fsmonitor-valid]
             [--ignore-submodules]
             [--[no-]split-index]
             [--[no-|test-|force-]untracked-cache]
             [--[no-]fsmonitor]
             [--really-refresh] [--unresolve] [--again | -g]
             [--info-only] [--index-info]
             [-z] [--stdin] [--index-version <n>]
             [--verbose]
             [--] [<file>…​]

Description

Изменяет индекс. Каждый упомянутый файл обновляется в индексе, и любое состояние unmerged или needs updating очищается.

См. также git-add[1] , чтобы узнать о более удобном для пользователя способе выполнения некоторых наиболее распространенных операций с индексом.

Способ обработки git update-index файлов, о которых он сообщает, можно изменить с помощью различных опций:

Options

--add

Если указанного файла еще нет в индексе, он добавляется. Поведение по умолчанию — игнорировать новые файлы.

--remove

Если указанный файл присутствует в индексе, но отсутствует, он удаляется. Поведение по умолчанию — игнорировать удаленные файлы.

--refresh

Просматривает текущий индекс и проверяет, необходимы ли слияния или обновления, проверяя информацию stat().

-q

Тихий. Если --refresh обнаружит, что индекс нуждается в обновлении, по умолчанию выдается ошибка. Эта опция все равно заставляет git update-index продолжать работу.

--ignore-submodules

Не пытайтесь обновлять субмодули. Эта опция учитывается только в том случае, если она передана до --refresh.

--unmerged

Если --refresh обнаруживает неслитые изменения в индексе, по умолчанию выдается ошибка. Эта опция все равно заставляет git update-index продолжать работу.

--ignore-missing

Игнорирует отсутствующие файлы во время --refresh

--cacheinfo <mode>,<object>,<path>
--cacheinfo <режим> <объект> <путь>

Непосредственно вставьте указанную информацию в индекс. Для обратной совместимости вы также можете указать эти три аргумента как три отдельных параметра, но новым пользователям рекомендуется использовать форму с одним параметром.

--index-info

Считайте информацию об индексе из стандартного ввода.

--chmod=(+|-)x

Установите права на выполнение обновленных файлов.

--[no-]assume-unchanged

Если указан этот флаг, имена объектов, записанные для путей, не обновляются. Вместо этого эта опция устанавливает/сбрасывает бит "assume unchanged" для путей. Когда бит "assume unchanged" включен, пользователь promises не может изменять файл и позволяет Git предположить, что файл рабочего дерева соответствует тому, что записано в индексе. Если вы хотите изменить файл рабочего дерева, вам нужно сбросить бит, чтобы сообщить Git . Иногда это полезно при работе с большим проектом в файловой системе с очень медленным системным вызовом lstat(2) (например, cifs).

Git потерпит неудачу (корректно) в случае, если ему необходимо изменить этот файл в индексе, например, при слиянии в коммите; таким образом, если предполагаемый неотслеживаемый файл будет изменен в исходном коде, вам придется обрабатывать ситуацию вручную.

--really-refresh

Аналогичен --refresh , но проверяет статистическую информацию безоговорочно, без учета настройки "assume unchanged" .

--[no-]skip-worktree

Если указан один из этих флагов, имена объектов, записанные для путей, не обновляются. Вместо этого эти параметры устанавливают и снимают бит "skip-worktree" для путей. Дополнительную информацию см. в разделе "Skip-worktree bit" ниже.

--[no-]ignore-skip-worktree-entries

Не удаляйте записи Skip-Worktree (AKA "index-only" ), даже если была указана опция --remove .

--[no-]fsmonitor-valid

Если указан один из этих флагов, имена объектов, записанные для путей, не обновляются. Вместо этого эти параметры устанавливают и снимают бит "fsmonitor valid" для путей. Дополнительную информацию см. в разделе "File System Monitor" ниже.

-g
--again

Запускает git update-index по путям, индексные записи которых отличаются от записей фиксации HEAD .

--unresolve

Восстанавливает состояние unmerged или needs updating файла во время слияния, если оно было очищено случайно.

--info-only

Не создавайте объекты в базе данных объектов для всех аргументов <file>, следующих за этим флагом; просто вставьте идентификаторы их объектов в индекс.

--force-remove

Удалите файл из индекса, даже если в рабочем каталоге еще есть такой файл. (Подразумевается --remove.)

--replace

По умолчанию, когда в индексе существует файл path , git update-index отклоняет попытку добавить path/file . Аналогично, если файл path/file существует, файл path добавить невозможно. При использовании флага --replace существующие записи, конфликтующие с добавляемой записью, автоматически удаляются с предупреждающими сообщениями.

--stdin

Вместо того, чтобы брать список путей из командной строки, прочитайте список путей из стандартного ввода. По умолчанию пути разделяются LF (т.е. один путь в строке).

--verbose

Сообщайте, что добавляется и удаляется из индекса.

--index-version <n>

Запишите полученный индекс в названной версии формата на диске. Поддерживаемые версии: 2, 3 и 4. Текущая версия по умолчанию — 2 или 3, в зависимости от того, используются ли дополнительные функции, например git add -N . С помощью --verbose также сообщите версию, которую индексный файл использует до и после этой команды.

Версия 4 выполняет простое сжатие имен путей, которое уменьшает размер индекса на 30–50 % в больших репозиториях, что приводит к ускорению загрузки. Git поддерживает его, начиная с версии 1.8.0, выпущенной в октябре 2012 года, а поддержка была добавлена ​​в libgit2 в 2016 году и в J Git в 2020 году. В более старых версиях этой страницы руководства она называлась "relatively young" , но в наши дни ее следует считать зрелой технологией. .

--show-index-version

Сообщите версию формата индекса, используемую индексным файлом на диске. См. --index-version выше.

-z

Имеет смысл только для --stdin или --index-info ; пути разделяются символом NUL вместо LF.

--split-index
--no-split-index

Включите или отключите режим разделенного индекса. Если режим разделения индекса уже включен и --split-index задан снова, все изменения в $GIT_DIR/index передаются обратно в общий индексный файл.

Эти параметры вступают в силу независимо от значения переменной конфигурации core.splitIndex (см. git-config[1] ). Но если изменение противоречит настроенному значению, выдается предупреждение, поскольку настроенное значение вступит в силу при следующем чтении индекса, и это устранит предполагаемый эффект этой опции.

--untracked-cache
--no-untracked-cache

Включите или отключите функцию неотслеживаемого кэша. Пожалуйста, используйте --test-untracked-cache перед его включением.

Эти параметры вступают в силу независимо от значения переменной конфигурации core.untrackedCache (см. git-config[1] ). Но если изменение противоречит настроенному значению, выдается предупреждение, поскольку настроенное значение вступит в силу при следующем чтении индекса, и это устранит предполагаемый эффект этой опции.

--test-untracked-cache

Выполняйте тесты только в рабочем каталоге, чтобы убедиться, что можно использовать неотслеживаемый кэш. Вам придется вручную включить неотслеживаемый кэш, используя --untracked-cache или --force-untracked-cache или переменную конфигурации core.untrackedCache , если вы действительно хотите его использовать. Если тест не пройден, код выхода равен 1, и в сообщении объясняется, что не работает должным образом, в противном случае код выхода равен 0 и выводится сообщение ОК.

--force-untracked-cache

То же, что --untracked-cache . Предусмотрено для обратной совместимости со старыми версиями Git , где --untracked-cache раньше подразумевало --test-untracked-cache , но этот параметр безоговорочно активирует расширение.

--fsmonitor
--no-fsmonitor

Включите или отключите функцию мониторинга файловой системы. Эти параметры вступают в силу независимо от значения переменной конфигурации core.fsmonitor (см. git-config[1] ). Но если изменение противоречит настроенному значению, выдается предупреждение, поскольку настроенное значение вступит в силу при следующем чтении индекса, и это устранит предполагаемый эффект этой опции.

--

Не интерпретируйте дополнительные аргументы как варианты.

<file>

Файлы для действий. Обратите внимание, что файлы, начинающиеся с . , отбрасываются. Сюда входят ./file и dir/./file . Если вы этого не хотите, используйте более чистые имена. То же самое относится к каталогам, заканчивающимся / , и путям на // .

Using --refresh

--refresh не вычисляет новый файл sha1 и не обновляет индекс для изменений режима/содержимого. Но что он делает, так это передает "re-match" статистическую информацию файла с индексом, чтобы вы могли обновить индекс для файла, который не был изменен, но в котором запись статистики устарела.

Например, вы захотите сделать это после выполнения git read-tree , чтобы связать детали индекса статистики с соответствующими файлами.

Использование --cacheinfo или --info-only

--cacheinfo используется для регистрации файла, которого нет в текущем рабочем каталоге. Это полезно для слияния с минимальной проверкой.

Чтобы представить, что у вас есть файл по пути с режимом и sha1, скажите:

$ git update-index --add --cacheinfo <mode>,<sha1>,<path>

--info-only используется для регистрации файлов без помещения их в объектную базу данных. Это полезно для репозиториев только со статусом.

И --cacheinfo , и --info-only ведут себя одинаково: индекс обновляется, а база данных объектов — нет. --cacheinfo полезен, когда объект находится в базе данных, но файл недоступен локально. --info-only полезен, когда файл доступен, но вы не хотите обновлять базу данных объектов.

Using --index-info

--index-info — это более мощный механизм, позволяющий вводить несколько определений записей из стандартного ввода и разработанный специально для сценариев. Он может принимать входные данные трех форматов:

  1. режим SP тип SP sha1 TAB путь

    Этот формат предназначен для помещения вывода git ls-tree в индекс.

  2. режим SP sha1 SP этап TAB путь

    Этот формат предназначен для помещения стадий более высокого порядка в индексный файл и соответствует выходным данным git ls-files --stage .

  3. режим SP sha1 TAB путь

    Этот формат больше не создается какой-либо командой Git , но поддерживается и будет поддерживаться update-index --index-info .

Чтобы поместить запись более высокого уровня в индекс, путь сначала следует удалить, введя для пути запись mode=0, а затем введя необходимые входные строки в третьем формате.

Например, начиная с этого индекса:

$ git ls-files -s
100644 8a1218a1024a212bb3db30becd860315f9f3ac52 0       frotz

вы можете подать следующий ввод в --index-info :

$ git update-index --index-info
0 0000000000000000000000000000000000000000        frotz
100644 8a1218a1024a212bb3db30becd860315f9f3ac52 1        frotz
100755 8a1218a1024a212bb3db30becd860315f9f3ac52 2        frotz

В первой строке входных данных указывается 0 как режим удаления пути; SHA-1 не имеет значения, если он хорошо отформатирован. Затем вторая и третья строки переносят записи этапов 1 и 2 для этого пути. После вышесказанного мы получим следующее:

$ git ls-files -s
100644 8a1218a1024a212bb3db30becd860315f9f3ac52 1        frotz
100755 8a1218a1024a212bb3db30becd860315f9f3ac52 2        frotz

Использование бита «предполагать неизменным»

Многие операции в Git зависят от эффективной реализации lstat(2) в вашей файловой системе, поэтому информацию st_mtime для рабочих файлов дерева можно легко проверить, чтобы увидеть, изменилось ли содержимое файла по сравнению с версией, записанной в индексном файле. К сожалению, некоторые файловые системы имеют неэффективный lstat(2) . Если ваша файловая система входит в их число, вы можете установить бит "assume unchanged" для путей, которые вы не меняли, чтобы Git не выполнял эту проверку. Обратите внимание, что установка этого бита в пути не означает, что Git будет проверять содержимое файла, чтобы увидеть, изменилось ли оно — это заставляет Git пропустить любую проверку и предположить, что оно не изменилось. Когда вы вносите изменения в рабочие файлы дерева, вы должны явно сообщить об этом Git , отбросив бит "assume unchanged" либо до, либо после их изменения.

Чтобы установить бит "assume unchanged" , используйте опцию --assume-unchanged . Для сброса используйте --no-assume-unchanged . Чтобы узнать, в каких файлах установлен бит "assume unchanged" , используйте git ls-files -v (см. git-ls-files[1] ).

Команда просматривает переменную конфигурации core.ignorestat . Если это так, пути, обновленные с помощью git update-index paths... , и пути, обновленные с помощью других команд Git , которые обновляют как индекс, так и рабочее дерево (например, git apply --index , git checkout-index -u и git read-tree -u ), автоматически помечаются как "assume unchanged" . Обратите внимание, что бит "assume unchanged" не устанавливается, если git update-index --refresh обнаруживает, что файл рабочего дерева соответствует индексу (используйте git update-index --really-refresh , если вы хотите пометить их как "assume unchanged" ).

Иногда пользователи путают бит предположения неизменности с битом пропуска рабочего дерева. Объяснение различий см. в последнем абзаце раздела "Skip-worktree bit" ниже.

Examples

Чтобы обновить только уже извлеченные файлы:

$ git checkout-index -n -f -a && git update-index --ignore-missing --refresh
На неэффективной файловой системе с установленным core.ignorestat
$ git update-index --really-refresh              (1)
$ git update-index --no-assume-unchanged foo.c   (2)
$ git diff --name-only                           (3)
$ edit foo.c
$ git diff --name-only                           (4)
M foo.c
$ git update-index foo.c                         (5)
$ git diff --name-only                           (6)
$ edit foo.c
$ git diff --name-only                           (7)
$ git update-index --no-assume-unchanged foo.c   (8)
$ git diff --name-only                           (9)
M foo.c

  1. заставляет lstat(2) устанавливать биты "assume unchanged" для путей, соответствующих индексу.

  2. отметьте путь для редактирования.

  3. это выполняет lstat(2) и находит индекс, соответствующий пути.

  4. это срабатывает lstat(2) и обнаруживает, что индекс не соответствует пути.

  5. регистрация новой версии для набора индексов бита "assume unchanged" .

  6. и предполагается неизменным.

  7. даже после того, как вы его отредактируете.

  8. Вы можете сказать об изменении постфактум.

  9. теперь он проверяет с помощью lstat(2) и обнаруживает, что он был изменен.

Skip-worktree bit

Бит пропуска рабочего дерева можно определить в одном (длинном) предложении: Скажите git избегать записи файла в рабочий каталог, когда это разумно возможно, и обрабатывать файл как неизмененный, если он отсутствует в рабочем каталоге.

Обратите внимание, что не все команды git обращают внимание на этот бит, а некоторые поддерживают его лишь частично.

Флаги индекса обновления и возможности чтения дерева, относящиеся к биту пропуска рабочего дерева, появились до появления команды git-sparse-checkout[1] , которая обеспечивает гораздо более простой способ настройки и обработки битов пропуска рабочего дерева. Если вы хотите сократить рабочее дерево, чтобы оно работало только с подмножеством файлов в репозитории, мы настоятельно рекомендуем использовать git-sparse-checkout[1] вместо низкоуровневых примитивов индекса обновления и дерева чтения.

Основная цель бита пропуска рабочего дерева — разрешить разреженные проверки, т. е. иметь рабочие каталоги, в которых присутствует только подмножество путей. Если установлен бит пропуска рабочего дерева, команды Git (такие как switch , pull , merge ) будут избегать записи этих файлов. Однако эти команды иногда все равно записывают эти файлы в важных случаях, таких как конфликты во время слияния или перебазирования. Команды Git также не будут рассматривать отсутствие таких файлов как преднамеренное удаление; например, git add -u не будет выполнять удаление этих файлов, а git commit -a также не будет выполнять фиксацию их удаления.

Хотя этот бит похож на бит предположения неизменяемого, его цель другая. Бит «предполагать неизменность» предназначен для того, чтобы оставить файл в рабочем дереве, но Git не проверяет его на наличие изменений и предполагает, что файл не был изменен (хотя, если он может определить, не указывая файл, что он изменился, это бесплатно фиксировать изменения). Skip-worktree сообщает Git игнорировать отсутствие файла, избегать его обновления, когда это возможно, с помощью команд, которые обычно обновляют большую часть рабочего каталога (например, checkout , switch , pull и т. д.), и не записывать его отсутствие в коммитах. Обратите внимание, что при разреженных извлечениях (настройка с помощью git sparse-checkout или настройка core.sparseCheckout на true), если файл помечен в индексе как Skip-Worktree, но найден в рабочем дереве, Git очистит бит Skip-Worktree для этого файла. .

Split index

Этот режим предназначен для репозиториев с очень большими индексами и направлен на сокращение времени, необходимого для многократной записи этих индексов.

В этом режиме индекс разделен на два файла: $GIT_DIR/index и $GIT_DIR/sharedindex.<SHA-1>. Изменения накапливаются в $GIT_DIR/index, разделенном индексе, в то время как общий индексный файл содержит все записи индекса и остается неизменным.

Все изменения в индексе разделения передаются обратно в файл общего индекса, когда количество записей в индексе разделения достигает уровня, указанного в переменной конфигурации SplitIndex.maxPercentChange (см. git-config[1] ).

Каждый раз, когда создается новый файл общего индекса, старые файлы общего индекса удаляются, если время их изменения старше, чем указано в переменной конфигурации SplitIndex.sharedIndexExpire (см. git-config[1] ).

Чтобы избежать удаления файла общего индекса, который все еще используется, время его изменения обновляется до текущего времени каждый раз, когда создается или считывается новый разделенный индекс на основе файла общего индекса.

Untracked cache

Этот кеш предназначен для ускорения выполнения команд, предполагающих определение неотслеживаемых файлов, таких как git status .

Эта функция работает путем записи времени mtime каталогов рабочего дерева, а затем исключения каталогов чтения и вызовов статистики для файлов в тех каталогах, время mtime которых не изменилось. Чтобы это работало, базовая операционная система и файловая система должны изменить поле каталогов st_mtime , если файлы в каталоге добавляются, изменяются или удаляются.

Вы можете проверить, поддерживает ли это файловая система, с помощью опции --test-untracked-cache . Опция --untracked-cache использовалась для неявного выполнения этого теста в старых версиях Git, но это уже не так.

Если вы хотите включить (или отключить) эту функцию, проще использовать переменную конфигурации core.untrackedCache (см. git-config[1] ), чем использовать опцию --untracked-cache для git update-index в каждом репозитории, особенно если вы хотите сделать это для всех используемых вами репозиториев, поскольку вы можете установить переменную конфигурации true (или false ) в вашем $HOME/.gitconfig только один раз, и она повлияет на все репозитории, к которым вы обращаетесь.

При изменении переменной конфигурации core.untrackedCache неотслеживаемый кэш добавляется в индекс или удаляется из него при следующем чтении индекса командой; тогда как при использовании --[no-|force-]untracked-cache неотслеживаемый кеш немедленно добавляется в индекс или удаляется из него.

До версии 2.17 в неотслеживаемом кеше была ошибка, из-за которой замена каталога символической ссылкой на другой каталог могла привести к неправильному отображению файлов, отслеживаемых git , как неотслеживаемых. См. коммит "status: add a failing test showing a core.untrackedCache bug" в git . git . Обходной путь для этого (и это может сработать для других необнаруженных ошибок в будущем):

$ git -c core.untrackedCache=false status

Также было показано, что эта ошибка затрагивает случаи замены каталога файлом, не связанные с символическими ссылками, когда речь идет о внутренних структурах неотслеживаемого кэша, но не было зарегистрировано ни одного случая, когда это приводило к неправильному выводу "git status" .

Также бывают случаи, когда существующие индексы, написанные версиями git до 2.17, будут ссылаться на каталоги, которые больше не существуют, что потенциально может привести к печати множества предупреждений "could not open directory" на "git status" . Это новые предупреждения о существующих проблемах, которые ранее молча отбрасывались.

Как и в случае с ошибкой, описанной выше, решение состоит в том, чтобы единоразово запустить "git status" с core.untrackedCache=false , чтобы удалить оставшиеся неверные данные.

Монитор файловой системы

Эта функция предназначена для ускорения операций git для репозиториев с большими рабочими каталогами.

Это позволяет git работать вместе с монитором файловой системы (см. git-fsmonitor--daemon[1] и раздел "fsmonitor-watchman" в githooks[5] ), который может информировать его о том, какие файлы были изменены. Это позволяет git избежать необходимости использовать lstat() для каждого файла для поиска измененных файлов.

При использовании в сочетании с неотслеживаемым кешем он может еще больше повысить производительность, избегая затрат на сканирование всего рабочего каталога в поисках новых файлов.

Если вы хотите включить (или отключить) эту функцию, проще использовать переменную конфигурации core.fsmonitor (см. git-config[1] ), чем использовать опцию --fsmonitor для git update-index в каждом репозитории, особенно если вы хотите сделать это для всех используемых вами репозиториев, поскольку вы можете установить переменную конфигурации в своем $HOME/.gitconfig только один раз, и она повлияет на все репозитории, к которым вы обращаетесь.

При изменении переменной конфигурации core.fsmonitor монитор файловой системы добавляется в индекс или удаляется из него при следующем чтении индекса командой. При использовании --[no-]fsmonitor монитор файловой системы немедленно добавляется в индекс или удаляется из него.

Configuration

Команда учитывает переменную конфигурации core.filemode . Если ваш репозиторий находится в файловой системе, исполняемые биты которой ненадежны, для этого параметра следует установить значение false (см. git-config[1] ). Это приводит к тому, что команда игнорирует различия в файловых режимах, записанных в индексе, и файловом режиме в файловой системе, если они различаются только исполняемым битом. В такой неудачной файловой системе вам может потребоваться использовать git update-index --chmod= .

Аналогично, если для переменной конфигурации core.symlinks установлено значение false (см. git-config[1] ), символические ссылки извлекаются как простые файлы, и эта команда не изменяет режим записанного файла с символической ссылки на обычный файл.

Команда просматривает переменную конфигурации core.ignorestat . См. раздел Using "assume unchanged" bit выше.

Команда также проверяет переменную конфигурации core.trustctime . Это может быть полезно, когда время смены индексного дескриптора регулярно изменяется чем-то вне Git (сканеры файловых систем и системы резервного копирования используют ctime для маркировки файлов как обработанных) (см. git-config[1] ).

Расширение неотслеживаемого кэша можно включить с помощью переменной конфигурации core.untrackedCache (см. git-config[1] ).

Notes

Пользователи часто пытаются использовать биты «предполагать неизменность» и «пропустить рабочее дерево», чтобы указать Git игнорировать изменения в отслеживаемых файлах. Это не работает должным образом, поскольку Git все равно может проверять файлы рабочего дерева по индексу при выполнении определенных операций. В общем, Git не предоставляет возможности игнорировать изменения в отслеживаемых файлах, поэтому рекомендуется использовать альтернативные решения.

Например, если файл, который вы хотите изменить, представляет собой какой-то файл конфигурации, репозиторий может включать образец файла конфигурации, который затем можно скопировать в игнорируемое имя и изменить. Репозиторий может даже включать сценарий, который будет рассматривать файл примера как шаблон, автоматически изменяя и копируя его.
