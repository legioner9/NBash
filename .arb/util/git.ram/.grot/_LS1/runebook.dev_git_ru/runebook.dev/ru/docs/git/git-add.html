git-add

Name

git-add — добавить содержимое файла в индекс

Synopsis

git add [--verbose | -v] [--dry-run | -n] [--force | -f] [--interactive | -i] [--patch | -p]
          [--edit | -e] [--[no-]all | -A | --[no-]ignore-removal | [--update | -u]] [--sparse]
          [--intent-to-add | -N] [--refresh] [--ignore-errors] [--ignore-missing] [--renormalize]
          [--chmod=(+|-)x] [--pathspec-from-file=<file> [--pathspec-file-nul]]
          [--] [<pathspec>…​]

Description

Эта команда обновляет индекс, используя текущий контент, найденный в рабочем дереве, чтобы подготовить контент для следующей фиксации. Обычно он добавляет текущее содержимое существующих путей в целом, но с некоторыми опциями его также можно использовать для добавления содержимого с применением только части изменений, внесенных в файлы рабочего дерева, или удаления путей, которые не существуют в рабочем дереве. больше.

"index" хранит снимок содержимого рабочего дерева, и именно этот снимок используется в качестве содержимого следующего коммита. Таким образом, после внесения каких-либо изменений в рабочее дерево и перед запуском команды фиксации вы должны использовать команду add для добавления в индекс любых новых или измененных файлов.

Эту команду можно выполнить несколько раз перед фиксацией. Он добавляет содержимое указанных файлов только во время выполнения команды добавления; если вы хотите, чтобы последующие изменения были включены в следующую фиксацию, вам необходимо снова запустить git add , чтобы добавить новое содержимое в индекс.

Команду git status можно использовать для получения сводной информации о том, какие файлы содержат изменения, подготовленные для следующей фиксации.

Команда git add по умолчанию не добавляет игнорируемые файлы. Если какие-либо игнорируемые файлы были явно указаны в командной строке, git add завершится ошибкой со списком игнорируемых файлов. Игнорируемые файлы, полученные в результате рекурсии каталогов или подстановки имен файлов, выполняемой Git (закавычивайте свои подстановки перед оболочкой), будут игнорироваться без уведомления. Команду git add можно использовать для добавления игнорируемых файлов с помощью опции -f (force).

См. git-commit[1] , чтобы узнать об альтернативных способах добавления содержимого в коммит.

Options

<pathspec>…

Файлы, из которых нужно добавить контент. Fileglobs (например, *.c ) можно указать для добавления всех соответствующих файлов. Также можно указать ведущее имя каталога (например, dir для добавления dir/file1 и dir/file2 ), чтобы обновить индекс, чтобы он соответствовал текущему состоянию каталога в целом (например, указание dir будет записывать не только файл dir/file1 , измененный в рабочем дереве, файл dir/file2 добавлен в рабочее дерево, а также файл dir/file3 удален из рабочего дерева). Обратите внимание, что более старые версии Git игнорировали удаленные файлы; используйте опцию --no-all , если вы хотите добавить измененные или новые файлы, но игнорировать удаленные.

Дополнительные сведения о синтаксисе <pathspec> см. в записи pathspec в gitglossary[7] .

-n
--dry-run

На самом деле не добавляйте файлы, просто покажите, существуют ли они и/или будут ли они проигнорированы.

-v
--verbose

Be verbose.

-f
--force

Разрешить добавление игнорируемых иначе файлов.

--sparse

Разрешить обновление записей индекса за пределами конуса разреженной проверки. Обычно git add отказывается обновлять записи индекса, пути которых не попадают в конус разреженной проверки, поскольку эти файлы могут быть удалены из рабочего дерева без предупреждения. Более подробную информацию см. в git-sparse-checkout[1] .

-i
--interactive

Добавляйте измененное содержимое рабочего дерева в индекс в интерактивном режиме. Могут быть предоставлены дополнительные аргументы пути, чтобы ограничить работу подмножеством рабочего дерева. Подробности см. в разделе «Интерактивный режим».

-p
--patch

В интерактивном режиме выбирайте фрагменты патчей между индексом и рабочим деревом и добавляйте их в индекс. Это дает пользователю возможность просмотреть разницу перед добавлением измененного содержимого в индекс.

Это эффективно запускает add --interactive , но пропускает начальное меню команд и сразу переходит к подкоманде patch . Подробности см. в разделе «Интерактивный режим».

-e
--edit

Откройте разницу и индекс в редакторе и позвольте пользователю редактировать ее. После закрытия редактора настройте заголовки ханков и примените патч к индексу.

Целью этой опции является выбор строк патча для применения или даже изменение содержимого строк, которые будут подготовлены. Это может быть быстрее и гибче, чем использование интерактивного выбора фрагментов. Однако легко запутаться и создать патч, не применимый к индексу. См. РЕДАКТИРОВАНИЕ ПАТЧЕЙ ниже.

-u
--update

Обновите индекс там, где уже есть запись, соответствующая <pathspec>. При этом удаляются и изменяются записи индекса в соответствии с рабочим деревом, но новые файлы не добавляются.

Если при использовании параметра -u не указан <pathspec>, обновляются все отслеживаемые файлы во всем рабочем дереве (старые версии Git использовались для ограничения обновления текущим каталогом и его подкаталогами).

-A
--all
--no-ignore-removal

Обновите индекс не только там, где в рабочем дереве есть файл, соответствующий <pathspec>, но и там, где в индексе уже есть запись. Это добавляет, изменяет и удаляет записи индекса в соответствии с рабочим деревом.

Если при использовании опции -A не указан <pathspec>, обновляются все файлы во всем рабочем дереве (старые версии Git использовались для ограничения обновления текущим каталогом и его подкаталогами).

--no-all
--ignore-removal

Обновите индекс, добавив новые файлы, неизвестные индексу, и файлы, измененные в рабочем дереве, но игнорируйте файлы, удаленные из рабочего дерева. Эта опция не используется, если не используется <pathspec>.

Эта опция предназначена в первую очередь для помощи пользователям, привыкшим к более старым версиям Git, чей "git add <pathspec>…​" был синонимом "git add --no-all <pathspec>…​" , т.е. игнорировал удаленные файлы.

-N
--intent-to-add

Запишите только тот факт, что путь будет добавлен позже. Запись о пути помещается в индекс без содержимого. Это полезно, помимо прочего, для отображения неподготовленного содержимого таких файлов с помощью git diff и их фиксации с помощью git commit -a .

--refresh

Не добавляйте файлы, а только обновляйте их информацию stat() в индексе.

--ignore-errors

Если некоторые файлы не удалось добавить из-за ошибок их индексации, не прерывайте операцию, а продолжайте добавлять остальные. Команда по-прежнему должна завершиться с ненулевым статусом. Для переменной конфигурации add.ignoreErrors можно установить значение true , чтобы сделать это поведением по умолчанию.

--ignore-missing

Эту опцию можно использовать только вместе с --dry-run. Используя эту опцию, пользователь может проверить, будет ли игнорироваться какой-либо из заданных файлов, независимо от того, присутствуют они уже в рабочем дереве или нет.

--no-warn-embedded-repo

По умолчанию git add выдает предупреждение при добавлении встроенного репозитория в индекс без использования git submodule add для создания записи в .gitmodules . Эта опция подавляет предупреждение (например, если вы вручную выполняете операции над подмодулями).

--renormalize

Примените процесс "clean" заново ко всем отслеживаемым файлам, чтобы принудительно добавить их снова в индекс. Это полезно после изменения конфигурации core.autocrlf или атрибута text для исправления файлов, добавленных с неправильными окончаниями строк CRLF/LF. Этот вариант подразумевает -u . Одиночные символы CR остаются нетронутыми, поэтому, хотя CRLF очищается до LF, последовательность CRCRLF очищается до CRLF лишь частично.

--chmod=(+|-)x

Переопределить исполняемый бит добавленных файлов. Меняется только бит исполняемого файла в индексе, файлы на диске остаются без изменений.

--pathspec-from-file=<file>

Спецификация пути передается в <file> вместо аргументов командной строки. Если <file> — это точно - , то используется стандартный ввод. Элементы Pathspec разделяются LF или CR/LF. Элементы Pathspec можно заключать в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). См. также --pathspec-file-nul и глобальный --literal-pathspecs .

--pathspec-file-nul

Имеет смысл только с --pathspec-from-file . Элементы Pathspec разделяются символом NUL, а все остальные символы воспринимаются буквально (включая символы новой строки и кавычки).

--

Эту опцию можно использовать для отделения параметров командной строки от списка файлов (полезно, когда имена файлов могут быть ошибочно приняты за параметры командной строки).

Examples

  • Добавляет содержимое из всех файлов *.txt в каталоге Documentation и его подкаталогах:

    $ git add Documentation/\*.txt

    Обратите внимание, что в этом примере звездочка * цитируется из оболочки; это позволяет команде включать файлы из подкаталогов каталога Documentation/ .

  • Рассматривается возможность добавления контента из всех скриптов git-*.sh:

    $ git add git-*.sh

    Поскольку этот пример позволяет оболочке расширить звездочку (т. е. вы явно указываете файлы), он не учитывает subdir/git-foo.sh .

Interactive mode

Когда команда переходит в интерактивный режим, она отображает выходные данные подкоманды status , а затем переходит в цикл интерактивных команд.

Командный цикл показывает список доступных подкоманд и выдает приглашение "What now> " . В общем, когда приглашение заканчивается одним > , вы можете выбрать только один из предложенных вариантов и ввести return, например:

    *** Commands ***
      1: status       2: update       3: revert       4: add untracked
      5: patch        6: diff         7: quit         8: help
    What now> 1

Вы также можете сказать s , sta или status выше, если выбор уникален.

Основной командный цикл имеет 6 подкоманд (плюс help и quit).

status

Это показывает изменение между HEAD и индексом (т. е. что будет зафиксировано, если вы скажете git commit ), а также между индексом и файлами рабочего дерева (т. е. что вы могли бы выполнить дальше перед git commit , используя git add ) для каждого пути. Пример вывода выглядит следующим образом:

              staged     unstaged path
     1:       binary      nothing foo.png
     2:     +403/-35        +1/-1 add-interactive.c

Он показывает, что foo.png имеет отличия от HEAD (но он двоичный, поэтому количество строк не может быть показано) и нет никакой разницы между индексированной копией и версией рабочего дерева (если бы версия рабочего дерева также была другой, был бы показан binary ). вместо nothing ). В другой файл, add-interactive.c, добавлено 403 строки и удалено 35 строк, если вы зафиксируете то, что находится в индексе, но файл рабочего дерева имеет дополнительные модификации (одно добавление и одно удаление).

update

При этом отображается информация о состоянии и выдается приглашение "Update>>" . Когда приглашение заканчивается двойным >> , вы можете сделать более одного выбора, объединенного пробелами или запятыми. Также вы можете сказать диапазоны. Например, "2-5 7,9" , чтобы выбрать 2,3,4,5,7,9 из списка. Если второе число в диапазоне опущено, будут взяты все оставшиеся патчи. Например, "7-" , чтобы выбрать 7,8,9 из списка. Вы можете сказать * , чтобы выбрать все.

То, что вы выбрали, затем выделяется * , вот так:

           staged     unstaged path
  1:       binary      nothing foo.png
* 2:     +403/-35        +1/-1 add-interactive.c

Чтобы снять выделение, добавьте к вводу префикс - следующим образом:

Update>> -2

Сделав выбор, ответьте пустой строкой, чтобы разместить содержимое файлов рабочего дерева для выбранных путей в индексе.

revert

Пользовательский интерфейс очень похож на update , а промежуточная информация для выбранных путей возвращается к версии HEAD. Возврат новых путей делает их неотслеживаемыми.

add untracked

Он имеет пользовательский интерфейс, очень похожий на update и revert , и позволяет добавлять в индекс неотслеживаемые пути.

patch

Это позволяет вам выбрать один путь из выбора, подобного status . После выбора пути он представляет разницу между индексом и файлом рабочего дерева и спрашивает, хотите ли вы выполнить поэтапное изменение каждого фрагмента. Вы можете выбрать один из следующих вариантов и ввести return:

y - stage this hunk
n - do not stage this hunk
q - quit; do not stage this hunk or any of the remaining ones
a - stage this hunk and all later hunks in the file
d - do not stage this hunk or any of the later hunks in the file
g - select a hunk to go to
/ - search for a hunk matching the given regex
j - leave this hunk undecided, see next undecided hunk
J - leave this hunk undecided, see next hunk
k - leave this hunk undecided, see previous undecided hunk
K - leave this hunk undecided, see previous hunk
s - split the current hunk into smaller hunks
e - manually edit the current hunk
? - print help

После определения судьбы всех ханков, если был выбран какой-либо ханк, индекс обновляется выбранными ханками.

Вы можете не вводить здесь return, установив для переменной конфигурации interactive.singleKey значение true .

diff

Это позволяет вам просмотреть, что будет зафиксировано (т. е. между HEAD и индексом).

Editing patches

Вызов git add -e или выбор e в интерактивном селекторе фрагментов откроет патч в вашем редакторе; после выхода из редактора результат применяется к индексу. Вы можете вносить произвольные изменения в патч, но учтите, что некоторые изменения могут привести к запутанным результатам или даже привести к тому, что патч невозможно будет применить. Если вы хотите полностью прервать операцию (т. е. не добавлять в индекс ничего нового), просто удалите все строки патча. В списке ниже описаны некоторые общие вещи, которые вы можете увидеть в патче, и какие операции редактирования имеют для них смысл.

added content

Добавленный контент представлен строками, начинающимися с "+" . Вы можете предотвратить создание дополнительных строк, удалив их.

removed content

Удаленное содержимое представлено строками, начинающимися с "-" . Вы можете предотвратить их поэтапное удаление, преобразовав "-" в " " (пробел).

modified content

Измененный контент представлен строками "-" (удаление старого контента), за которыми следуют строки "+" (добавление заменяющего контента). Вы можете предотвратить промежуточную модификацию, преобразовав строки "-" в " " и удалив строки "+" . Помните, что изменение только половины пары может привести к запутанным изменениям в индексе.

Можно выполнять и более сложные операции. Но будьте осторожны: поскольку исправление применяется только к индексу, а не к рабочему дереву, рабочее дерево отобразится для "undo" при изменении индекса. Например, добавление в индекс новой строки, которой нет ни в HEAD, ни в рабочем дереве, приведет к тому, что новая строка будет подготовлена ​​для фиксации, но в рабочем дереве эта строка будет выглядеть возвращенной.

Избегайте использования этих конструкций или делайте это с особой осторожностью.

удаление нетронутого контента

Содержимое, которое не различается между индексом и рабочим деревом, может отображаться в строках контекста, начиная с " " (пробел). Вы можете подготовить контекстные строки для удаления, преобразовав пространство в "-" . Появится полученный файл рабочего дерева для повторного добавления содержимого.

изменение существующего контента

Можно также изменить строки контекста, отправив их на удаление (путем преобразования " " в "-" ) и добавив строку "+" с новым содержимым. Аналогичным образом можно изменить линии "+" для существующих дополнений или модификаций. Во всех случаях новая модификация будет отображаться в рабочем дереве в отменённом виде.

new content

Вы также можете добавить новый контент, которого нет в патче; просто добавьте новые строки, каждая из которых начинается с "+" . Дополнение отобразится в рабочем дереве в отменённом виде.

Есть также несколько операций, которых следует полностью избегать, поскольку они сделают невозможным применение патча:

  • добавление контекстных ( " " ) или удаление ( "-" ) строк

  • удаление контекста или удаление строк

  • изменение содержимого строк контекста или удаления

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

add.ignoreErrors
add.ignore-errors (deprecated)

Сообщает git add продолжать добавление файлов, если некоторые файлы не могут быть добавлены из-за ошибок индексации. Эквивалент опции --ignore-errors для git-add[1] . add.ignore-errors устарел, так как не соответствует обычному соглашению об именах переменных конфигурации.

add.interactive.useBuiltin

Неиспользуемая переменная конфигурации. Используется в версиях Git от 2.25.0 до 2.36.0 для включения встроенной версии интерактивного режима git-add[1] , который затем стал режимом по умолчанию в версиях Git от 2.37.0 до 2.39.0.
