api-simple-ipc

API Simple-IPC представляет собой набор подпрограмм library с префиксом ipc_ и базового протокола связи, который позволяет процессу IPC-клиента отправлять специфическое для приложения сообщение запроса IPC процессу IPC-сервера и получать специфичный для приложения IPC-ответ. сообщение.

Связь происходит через именованный канал в Windows и сокет домена Unix на других платформах. IPC-клиенты и IPC-серверы встречаются по заранее согласованному пути для конкретного приложения (который выходит за рамки данного проекта), который является локальным для компьютерной системы.

Подпрограммы IPC-сервера в процессе серверного приложения создают пул потоков для прослушивания соединений и получения сообщений запросов от нескольких одновременно работающих IPC-клиентов. При получении эти сообщения передаются обратным вызовам серверного приложения для обработки. Затем подпрограммы IPC-сервера постепенно передают ответы обратно IPC-клиенту.

Подпрограммы IPC-клиента в процессе клиентского приложения подключаются к IPC-серверу, отправляют сообщение запроса и ждут ответа. При получении ответ возвращается обратно вызывающему абоненту.

Например, функция fsmonitor--daemon будет построена как серверное приложение поверх подпрограмм IPC-сервера library . У него будут потоки, отслеживающие события файловой системы, и пул потоков, ожидающий клиентских подключений. Клиенты, такие как git status , запросят список событий файловой системы с определенного момента времени, а сервер ответит списком измененных файлов и каталогов. Форматы запроса и ответа зависят от приложения; процедуры IPC-клиента и IPC-сервера рассматривают их как непрозрачные потоки байтов.

Сравнение с моделью подпроцесса

Механизм Simple-IPC отличается от существующей модели sub-process.c (Documentation/technical/long-running-process-protocol.txt) и используется такими приложениями, как Git-LFS. В модели подпроцесса в стиле LFS помощник запускается приоритетным процессом, связь происходит через пару файловых дескрипторов, привязанных к стандартному вводу/стандартному выводу подпроцесса, подпроцесс обслуживает только текущий приоритетный процесс. и подпроцесс завершается, когда завершается процесс переднего плана.

В модели Simple-IPC сервер представляет собой очень долго работающую службу. Он может обслуживать множество клиентов одновременно и имеет соединение через сокет private или именованный канал с каждым активным клиентом. Он может быть запущен (по требованию) текущим клиентским процессом или предыдущим клиентом или операционной системой во время загрузки. Серверный процесс не связан с терминалом и сохраняется после завершения работы клиентов. Клиенты не имеют доступа к stdin/stdout серверного процесса и поэтому должны взаимодействовать через сокеты или именованные каналы.

Запуск и завершение работы сервера

То, как запускается сервер приложений на основе IPC-сервера, также выходит за рамки проектирования Simple-IPC и является свойством использующего его приложения. Например, сервер может быть запущен или перезапущен во время плановых операций обслуживания, или он может быть запущен как системная служба во время загрузки системы, или он может быть запущен по требованию командой Git на переднем плане, когда это необходимо.

Аналогичным образом, завершение работы сервера является свойством приложения, использующего процедуры simple-ipc. Например, сервер может решить завершить работу во время простоя или только по явному запросу.

Simple-ipc protocol

Протокол Simple-IPC состоит из одного сообщения запроса от клиента и дополнительного ответного сообщения от сервера. Сообщения клиента и сервера не ограничены по длине и завершаются флеш-пакетом.

Подпрограммы pkt-line ( gitprotocol-common[5] ) используются для упрощения управления буфером во время генерации, передачи и приема сообщений. Пакет сброса используется для обозначения конца сообщения. Это позволяет отправителю постепенно генерировать и передавать сообщение. Это позволяет получателю постепенно получать сообщение частями и знать, когда он получил все сообщение.

Фактический формат байтов сообщений запроса клиента и ответа сервера зависит от приложения. Уровень IPC передает и принимает их в виде непрозрачных байтовых буферов, не заботясь о содержимом внутри. Задача уровня вызывающего приложения — понять содержимое сообщений запроса и ответа.

Summary

Концептуально протокол Simple-IPC аналогичен запросу HTTP REST. Клиенты подключаются, отправляют запрос без сохранения состояния, специфичный для приложения, получают ответ, специфичный для приложения, и отключаются. Это возможность обращения к серверу в один конец. Подпрограммы Simple-IPC скрывают сведения о сокете, именованном канале и пуле потоков и позволяют уровню приложения сосредоточиться на текущей задаче.
