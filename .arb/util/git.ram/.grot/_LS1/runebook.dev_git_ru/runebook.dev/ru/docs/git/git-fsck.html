git-fsck

Name

git-fsck — проверяет подключение и достоверность объектов в базе данных.

Synopsis

git fsck [--tags] [--root] [--unreachable] [--cache] [--no-reflogs]
         [--[no-]full] [--strict] [--verbose] [--lost-found]
         [--[no-]dangling] [--[no-]progress] [--connectivity-only]
         [--[no-]name-objects] [<object>…​]

Description

Проверяет подключение и достоверность объектов в базе данных.

Options

<object>

Объект, который следует рассматривать как начало трассировки недостижимости.

Если объекты не указаны, git fsck по умолчанию использует индексный файл, все ссылки SHA-1 в пространстве имен refs и все журналы ссылок (если не указан параметр --no-reflogs) в качестве заголовков.

--unreachable

Распечатайте объекты, которые существуют, но недоступны ни с одного из ссылочных узлов.

--[no-]dangling

Печать объектов, которые существуют, но никогда не используются directly (по умолчанию). --no-dangling можно использовать для исключения этой информации из вывода.

--root

Сообщить о корневых узлах.

--tags

Report tags.

--cache

Рассматривайте любой объект, записанный в индексе, как головной узел трассировки недоступности.

--no-reflogs

Не считайте коммиты, на которые ссылается только запись в журнале ссылок, доступными. Эта опция предназначена только для поиска коммитов, которые раньше были в ссылке, но теперь их нет, но все еще находятся в соответствующей ссылке.

--full

Проверяйте не только объекты в GIT_OBJECT_DIRECTORY ($GIT_DIR/objects), но и объекты, найденные в альтернативных пулах объектов, перечисленных в GIT_ALTERNATE_OBJECT_DIRECTORIES или $GIT_DIR/objects/info/alternates, а также в упакованных архивах Git , найденных в $GIT_DIR/objects/pack и соответствующих упаковывайте подкаталоги в альтернативные пулы объектов. Теперь это значение по умолчанию; вы можете отключить его с помощью --no-full.

--connectivity-only

Проверяйте только связность доступных объектов, проверяя наличие всех объектов, на которые ссылается тег доступности, фиксация или дерево. Это ускоряет операцию, поскольку полностью исключается чтение больших двоичных объектов (хотя при этом все равно проверяется существование указанных больших двоичных объектов). Это позволит обнаружить повреждения в коммитах и ​​деревьях, но не выполнит никаких семантических проверок (например, ошибок формата). Повреждения BLOB-объектов вообще не будут обнаружены.

Также будет доступен доступ к недоступным тегам, коммитам и деревьям, чтобы найти кончики висячих сегментов истории. Используйте --no-dangling , если вас не волнует этот вывод и вы хотите еще больше его ускорить.

--strict

Включите более строгую проверку, а именно перехват режима файла, записанного с установленным битом g+w, который был создан более старыми версиями Git . Существующие репозитории, включая ядро ​​Linux, сам Git и разреженный репозиторий, содержат старые объекты, которые вызывают эту проверку, но рекомендуется проверять новые проекты с этим флагом.

--verbose

Be chatty.

--lost-found

Записывайте висячие объекты в .git/lost-found/commit/ или .git/lost-found/other/, в зависимости от типа. Если объект является большим двоичным объектом, в файл записывается его содержимое, а не имя объекта.

--name-objects

При отображении имен доступных объектов в дополнение к SHA-1 также отображается имя, описывающее, как они доступны, совместимое с git-rev-parse[1] , например HEAD@{1234567890}~25^2:src/ .

--[no-]progress

Статус выполнения сообщается в стандартном потоке ошибок по умолчанию, когда он подключен к терминалу, если не указано --no-progress или --verbose. --progress принудительно устанавливает статус выполнения, даже если стандартный поток ошибок не направляется на терминал.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

fsck.<msg-id>

Во время fsck git может обнаружить проблемы с устаревшими данными, которые не были бы созданы текущими версиями git и которые не были бы отправлены по сети, если бы был установлен transfer.fsckObjects . Эта функция предназначена для поддержки работы с устаревшими репозиториями, содержащими такие данные.

Настройка fsck.<msg-id> будет подхвачена git-fsck[1] , но вместо этого для приема передачи такого набора данных receive.fsck.<msg-id> или для его клонирования или извлечения установите fetch.fsck.<msg-id> .

В остальной документации для краткости обсуждается fsck.* , но то же самое относится и к соответствующим receive.fsck.* и fetch.fsck.* . переменные.

В отличие от таких переменных, как color.ui и core.editor , переменные receive.fsck.<msg-id> и fetch.fsck.<msg-id> не будут использовать конфигурацию fsck.<msg-id> , если они не установлены. Чтобы единообразно настроить одни и те же параметры fsck в разных обстоятельствах, для всех трех из них должны быть установлены одинаковые значения.

Если установлен fsck.<msg-id> , ошибки можно переключить на предупреждения и наоборот, настроив параметр fsck.<msg-id> , где <msg-id> — это идентификатор сообщения fsck, а значение — одно из error , warn или ignore . Для удобства fsck добавляет к ошибке/предупреждению идентификатор сообщения, например "missingEmail: invalid author/committer line - missing email" означает, что установка fsck.missingEmail = ignore скроет эту проблему.

В общем, лучше перечислить существующие объекты с проблемами с fsck.skipList вместо того, чтобы перечислять виды поломок, которые эти проблемные объекты разделяют, которые следует игнорировать, поскольку в последнем случае новые экземпляры тех же поломок останутся незамеченными.

Установка неизвестного значения fsck.<msg-id> приведет к прекращению работы fsck, но если сделать то же самое для receive.fsck.<msg-id> и fetch.fsck.<msg-id> , git только выдаст предупреждение.

См. раздел Fsck Messages в git-fsck[1] для получения информации о поддерживаемых значениях <msg-id> .

fsck.skipList

Путь к списку имен объектов (т. е. по одному несокращенному SHA-1 на строку), которые, как известно, повреждены нефатально и которые следует игнорировать. В версиях Git 2.20 и более поздних комментарии ( # ), пустые строки и любые начальные и конечные пробелы игнорируются. В более старых версиях все, кроме SHA-1 на строку, будет выдавать ошибку.

Эта функция полезна, когда существующий проект должен быть принят, несмотря на ранние фиксации, содержащие ошибки, которые можно безопасно игнорировать, например неверные адреса электронной почты коммиттера. Примечание. С помощью этой настройки нельзя пропустить поврежденные объекты.

Как и fsck.<msg-id> , эта переменная имеет соответствующие варианты receive.fsck.skipList и fetch.fsck.skipList .

В отличие от таких переменных, как color.ui и core.editor , переменные receive.fsck.skipList и fetch.fsck.skipList не будут использовать конфигурацию fsck.skipList , если они не установлены. Чтобы единообразно настроить одни и те же параметры fsck в разных обстоятельствах, для всех трех из них должны быть установлены одинаковые значения.

В более старых версиях Git (до 2.20) документировано, что список имен объектов необходимо сортировать. Это никогда не было требованием; имена объектов могли появляться в любом порядке, но при чтении списка мы отслеживали, был ли список отсортирован для целей реализации внутреннего двоичного поиска, что могло бы сэкономить некоторую работу с уже отсортированным списком. Если у вас не огромный список, не было причин изо всех сил стараться его предварительно отсортировать. После Git версии 2.20 вместо этого используется реализация хеширования, поэтому нет необходимости предварительно сортировать список.

Discussion

git-fsck проверяет SHA-1 и общую работоспособность объекта, а также полностью отслеживает результирующую достижимость и все остальное. Он распечатывает все найденные повреждения (отсутствующие или поврежденные объекты), а если вы используете флаг --unreachable , он также распечатывает объекты, которые существуют, но недоступны ни для одного из указанных головных узлов (или набора по умолчанию, как уже упоминалось). выше).

Любые поврежденные объекты вам придется найти в резервных копиях или других архивах (т. е. вы можете просто удалить их и выполнить rsync с каким-нибудь другим сайтом в надежде, что у кого-то еще есть поврежденный вами объект).

Если core.commitGraph имеет значение true, файл графа фиксации также будет проверен с использованием git commit-graph verify . См. git-commit-graph[1] .

Extracted diagnostics

недоступен <тип> <объект>

Объект <type> <object> на самом деле не упоминается прямо или косвенно ни в одном из видимых деревьев или коммитов. Это может означать, что существует другой корневой узел, который вы не указали, или что дерево повреждено. Если вы не пропустили корневой узел, вы можете удалить недоступные узлы, поскольку их нельзя использовать.

отсутствует <тип> <объект>

На объект <type> <object> имеется ссылка, но он отсутствует в базе данных.

свисающий <тип> <объект>

Объект <type> <object> присутствует в базе данных, но никогда не используется directly . Висячий коммит может быть корневым узлом.

несоответствие хеша <объект>

В базе данных есть объект, хэш которого не соответствует значению базы данных объекта. Это указывает на серьезную проблему с целостностью данных.

Ниже перечислены типы ошибок, которые обнаруживает git fsck , и что означает каждая ошибка, а также их серьезность по умолчанию. Серьезность ошибки, кроме тех, которые помечены как "(FATAL)" , можно изменить, установив соответствующую переменную конфигурации fsck.<msg-id> .

Environment variables

используется для указания дополнительных корней базы данных объектов (обычно не задано)
