gitremote-helpers

Name

gitremote-helpers — Вспомогательные программы для взаимодействия с удаленными репозиториями.

Synopsis

git remote-<transport> <repository> [<URL>]

Description

Удаленные вспомогательные программы обычно не используются конечными пользователями напрямую, но они вызываются Git , когда ему необходимо взаимодействовать с удаленными репозиториями. Git не поддерживает встроенную поддержку. Данный помощник будет реализовывать подмножество возможностей, описанных здесь. Когда Git необходимо взаимодействовать с репозиторием с помощью удаленного помощника, он порождает помощника как независимый процесс, отправляет команды на стандартный ввод помощника и ожидает результатов от стандартного вывода помощника. Поскольку удаленный помощник запускается как независимый процесс от Git , нет необходимости повторно связывать Git для добавления нового помощника, а также нет необходимости связывать помощник с реализацией Git .

Каждый помощник должен поддерживать команду "capabilities" , которую Git использует для определения того, какие другие команды будет принимать помощник. Эти другие команды можно использовать для обнаружения и обновления удаленных ссылок, транспортировки объектов между базой данных объектов и удаленным репозиторием, а также обновления локального хранилища объектов.

Git поставляется с семейством удаленных помощников "curl" , которые обрабатывают различные транспортные протоколы, такие как git-remote-http , git-remote-https , git-remote-ftp и git-remote-ftps . Они реализуют возможности fetch , option и push .

Invocation

Удаленные вспомогательные программы вызываются с одним или (необязательно) двумя аргументами. Первый аргумент указывает удаленный репозиторий, как в Git ; это либо имя настроенного пульта, либо URL-адрес. Второй аргумент указывает URL-адрес; обычно он имеет вид <transport>://<address> , но возможна любая произвольная строка. Переменная среды GIT_DIR настроена для удаленного помощника и может использоваться для определения, где хранить дополнительные данные или из какого каталога вызывать вспомогательные команды Git .

Когда Git встречает URL-адрес формы <transport>://<address> , где <transport> — это протокол, который он не может обрабатывать в собственном коде, он автоматически вызывает git remote-<transport> с полным URL-адресом в качестве второго аргумента. Если такой URL-адрес встречается непосредственно в командной строке, первый аргумент совпадает со вторым, а если он встречается на настроенном удаленном компьютере, первым аргументом является имя этого удаленного устройства.

URL-адрес формы <transport>::<address> явно указывает Git вызвать git remote-<transport> с <address> в качестве второго аргумента. Если такой URL-адрес встречается непосредственно в командной строке, первым аргументом является <address> , а если он встречается на настроенном удаленном компьютере, первым аргументом является имя этого удаленного устройства.

Кроме того, если на настроенном удаленном устройстве для remote.<name>.vcs установлено значение <transport> , Git явно вызывает git remote-<transport> с <name> в качестве первого аргумента. Если установлено, вторым аргументом будет remote.<name>.url ; в противном случае второй аргумент опускается.

Input format

Git отправляет удаленному помощнику список команд на стандартный ввод, по одной в строке. Первой командой всегда является команда capabilities , в ответ на которую удаленный помощник должен распечатать список поддерживаемых им возможностей (см. ниже), за которым следует пустая строка. Ответ на команду возможностей определяет, какие команды Git использует в оставшейся части потока команд.

Поток команд завершается пустой строкой. В некоторых случаях (указанных в документации соответствующих команд) за этой пустой строкой следует полезная нагрузка в каком-то другом протоколе (например, пакетном протоколе), а в других она указывает на конец ввода.

Capabilities

Ожидается, что каждый удаленный помощник будет поддерживать только подмножество команд. Операции, которые поддерживает помощник, объявляются в Git в ответе на команду capabilities (см. КОМАНДЫ ниже).

Ниже мы перечисляем все определенные возможности и для каждой перечисляем команды, которые должен предоставить помощник с этой возможностью.

Возможности для толкания

connect

Можно попытаться подключиться к git receive-pack (для отправки), git upload-pack и т. д. для связи, используя собственный протокол Packfile git. Для этого требуется двунаправленное полнодуплексное соединение.

Поддерживаемые команды: connect .

stateless-connect

Экспериментальный; только для внутреннего использования. Может попытаться подключиться к удаленному серверу для связи, используя проводной протокол git версии 2. Дополнительную информацию см. в документации по команде stateless-connect.

Поддерживаемые команды: stateless-connect .

push

Может обнаруживать удаленные ссылки и отправлять локальные коммиты и историю, ведущую к ним, в новые или существующие удаленные ссылки.

Поддерживаемые команды: list for-push , push .

export

Может обнаруживать удаленные ссылки и отправлять указанные объекты из потока быстрого импорта на удаленные ссылки.

Поддерживаемые команды: list for-push , export .

Если помощник объявляет connect , Git будет использовать его, если это возможно, и вернется к другой возможности, если помощник запросит это при подключении (см. команду connect в разделе КОМАНДЫ). Выбирая между push и export , Git отдает предпочтение push . Другие интерфейсы могут иметь другой порядок предпочтений.

no-private-update

При использовании возможности refspec git обычно обновляет ссылку private при успешной отправке. Это обновление отключается, когда удаленный помощник объявляет возможность no-private-update .

Возможности для получения

connect

Можно попробовать подключиться к git upload-pack (для получения), git receive-pack и т. д. для связи, используя собственный протокол пакетных файлов Git. Для этого требуется двунаправленное полнодуплексное соединение.

Поддерживаемые команды: connect .

stateless-connect

Экспериментальный; только для внутреннего использования. Может попытаться подключиться к удаленному серверу для связи, используя проводной протокол git версии 2. Дополнительную информацию см. в документации по команде stateless-connect.

Поддерживаемые команды: stateless-connect .

fetch

Может обнаруживать удаленные ссылки и передавать доступные по ним объекты в локальное хранилище объектов.

Поддерживаемые команды: list , fetch .

import

Может обнаруживать удаленные ссылки и выводить объекты, доступные по ним, в виде потока в формате быстрого импорта.

Поддерживаемые команды: list , import .

check-connectivity

Может гарантировать, что при запросе клона полученный пакет является автономным и подключенным.

get

Можно использовать команду get для загрузки файла по заданному URI.

Если помощник объявляет connect , Git будет использовать его, если это возможно, и вернется к другой возможности, если помощник запросит это при подключении (см. команду connect в разделе КОМАНДЫ). Выбирая между fetch и import , Git отдает предпочтение fetch . Другие интерфейсы могут иметь другой порядок предпочтений.

Miscellaneous capabilities

option

Для указания таких параметров, как verbosity (сколько вывода нужно записать в stderr) и depth (сколько истории требуется в случае поверхностного клонирования), которые влияют на выполнение других команд.

refspec <refspec>

Для удаленных помощников, реализующих import или export , эта возможность позволяет ограничить ссылки пространством имен private вместо прямой записи в refs/heads или refs/remotes. Рекомендуется, чтобы все импортеры, предоставляющие возможность import , использовали это. Это обязательно для export .

Помощник, рекламирующий возможность refspec refs/heads/*:refs/svn/origin/branches/* , говорит, что, когда его запрашивают import refs/heads/topic , выдаваемый им поток обновит refs/svn/origin/branches/topic ref.

Эту возможность можно рекламировать несколько раз. Первая применимая спецификация имеет приоритет. Левая часть спецификаций ссылок, объявленная с этой возможностью, должна охватывать все ссылки, о которых сообщает команда list. Если возможности refspec не объявлены, подразумевается refspec *:* .

При написании удаленных помощников для децентрализованных систем контроля версий рекомендуется сохранить локальную копию репозитория для взаимодействия и позволить ссылкам пространства имен private указывать на этот локальный репозиторий, в то время как пространство имен refs/remotes используется для отслеживания удаленный репозиторий.

bidi-import

Это изменяет возможности import . Команды быстрого импорта cat-blob и ls могут использоваться удаленными помощниками для получения информации о больших объектах и ​​деревьях, которые уже существуют в памяти быстрого импорта. Для этого требуется канал от быстрого импорта до удаленного помощника. Если он объявлен в дополнение к "import" , Git устанавливает канал от быстрого импорта к стандартному вводу удаленного помощника. Отсюда следует, что Git и быстрый импорт подключены к стандартному вводу удаленного помощника. Поскольку Git может отправлять несколько команд удаленному помощнику, необходимо, чтобы помощники, использующие bidi-import , буферизовали все команды import в пакете перед отправкой данных в режим быстрого импорта. Это сделано для предотвращения смешивания команд и ответов быстрого импорта на стандартном вводе помощника.

export-marks <file>

Это изменяет возможности export , предписывая Git выгрузить таблицу внутренних меток в <file> после завершения. Подробности читайте о --export-marks=<file> в git-fast-export[1] .

import-marks <file>

Это изменяет возможности export , предписывая Git загружать метки, указанные в <file>, перед обработкой любого ввода. Подробности читайте о --import-marks=<file> в git-fast-export[1] .

signed-tags

Это изменяет возможности export , указывая Git передать --signed-tags=verbatim на git-fast-export[1] . При отсутствии этой возможности Git будет использовать --signed-tags=warn-strip .

object-format

Это указывает на то, что помощник может взаимодействовать с удаленной стороной, используя явное расширение алгоритма хеширования.

Commands

Команды передаются вызывающей стороной на стандартный ввод помощника, по одной в строке.

capabilities

Перечисляет возможности помощника, по одной в строке, заканчиваясь пустой строкой. Каждой возможности может предшествовать * , что делает их обязательными для версий Git , использующих удаленный помощник для понимания. Любая неизвестная обязательная возможность является фатальной ошибкой.

Поддержка этой команды обязательна.

list

Перечисляет ссылки, по одной в строке, в формате "<value> <name> [<attr> …​]" . Значением может быть шестнадцатеричный хэш sha1, "@<dest>" для symref, ":<keyword> <value>" для пары ключ-значение или "?" , чтобы указать, что помощник не смог получить значение ссылки. За именем следует список атрибутов, разделенных пробелами; нераспознанные атрибуты игнорируются. Список заканчивается пустой строкой.

См. REF LIST ATTRIBUTES для получения списка определенных в настоящее время атрибутов. См. КЛЮЧЕВЫЕ СЛОВА СПИСКА ССЫЛОК для получения списка определенных в настоящее время ключевых слов.

Поддерживается, если помощник имеет возможности "fetch" или "import" .

list for-push

Аналогичен list , за исключением того, что он используется тогда и только тогда, когда вызывающая сторона хочет использовать результирующий список ссылок для подготовки push-команд. Помощник, поддерживающий как push, так и fetch, может использовать это, чтобы определить, для какой операции будут использоваться выходные данные list , что, возможно, уменьшит объем работы, которую необходимо выполнить.

Поддерживается, если помощник имеет возможности "push" или "export" .

опция <имя> <значение>

Устанавливает для вспомогательной опции транспорта <имя> значение <значение>. Выводит одну строку, содержащую одно из ok (опция успешно установлена), unsupported (опция не распознана) или error <msg> (опция <имя> поддерживается, но <значение> для нее недопустимо). Параметры должны быть установлены перед другими командами и могут влиять на поведение этих команд.

См. ОПЦИИ для списка определенных в настоящее время параметров.

Поддерживается, если помощник имеет возможность "option" .

принести <sha1> <имя>

Извлекает заданный объект, записывая необходимые объекты в базу данных. Команды выборки отправляются пакетами, по одной на строку, завершаясь пустой строкой. Выводит одну пустую строку, когда все команды выборки в одном пакете завершены. Таким способом можно получить только объекты, о которых сообщалось в выходных данных list с sha1.

При желании можно вывести строку lock <file> , указывающую полный путь к файлу под $GIT_DIR/objects/pack , который сохраняет пакет до тех пор, пока ссылки не будут соответствующим образом обновлены. Путь должен заканчиваться на .keep . Это механизм присвоения имени кортежу <pack,idx,keep> путем указания только компонента Keep. Сохраненный пакет не будет удален при одновременной повторной упаковке, даже если на его объекты нельзя ссылаться до завершения выборки. Файл .keep будет удален по завершении выборки.

Если запрошена опция check-connectivity , помощник должен вывести connectivity-ok , если клон автономен и подключен.

Поддерживается, если помощник имеет возможность "fetch" .

push +<src>:<dst>

Отправляет данную локальную фиксацию или ветку <src> в удаленную ветку, описанную <dst>. Пакетная последовательность одной или нескольких команд push завершается пустой строкой (если для отправки требуется только одна ссылка, за одной командой push следует пустая строка). Например, следующее будет двумя пакетами push : первый просит удаленного помощника передать локальный ref master на удаленный ref master и локальный HEAD на удаленный branch , а второй просит передать ref foo на ref bar . (принудительное обновление, запрошенное + ).

push refs/heads/master:refs/heads/master
push HEAD:refs/heads/branch
\n
push +refs/heads/foo:refs/heads/bar
\n

Ноль или более опций протокола могут быть введены после последней команды push , перед завершающей пустой строкой пакета.

Когда передача завершена, выводит одну или несколько строк ok <dst> или error <dst> <why>? , чтобы указать успех или неудачу каждого отправленного ссылки. Вывод отчета о состоянии завершается пустой строкой. Поле опции <why> может быть заключено в кавычки в строке стиля C, если оно содержит LF.

Поддерживается, если помощник имеет возможность "push" .

import <name>

Создает поток быстрого импорта, который импортирует текущее значение именованной ссылки. При необходимости он может дополнительно импортировать другие ссылки для эффективного построения истории. Сценарий записывает в пространство имен private , специфичное для помощника. Значение именованной ссылки должно быть записано в место в этом пространстве имен, полученное путем применения спецификации ссылки из возможности "refspec" к имени ссылки.

Особенно полезно для взаимодействия с сторонней системой управления версиями.

Как и push , пакетная последовательность одного или нескольких import завершается пустой строкой. Для каждого пакета import удаленный помощник должен создать поток быстрого импорта, завершаемый командой done .

Обратите внимание: если используется возможность bidi-import , вся последовательность пакетов должна быть помещена в буфер перед началом отправки данных в режим быстрого импорта, чтобы предотвратить смешивание команд и ответов быстрого импорта на стандартном вводе помощника.

Поддерживается, если помощник имеет возможность "import" .

export

Указывает удаленному помощнику, что любой последующий ввод является частью потока быстрого импорта (генерируемого git fast-export ), содержащего объекты, которые следует отправить на удаленный компьютер.

Особенно полезно для взаимодействия с сторонней системой управления версиями.

Возможности export-marks и import-marks , если они указаны, влияют на эту команду, поскольку они передаются git fast-export , который затем загружает/сохраняет таблицу меток для локальных объектов. Это можно использовать для реализации дополнительных операций.

Поддерживается, если помощник имеет возможность "export" .

connect <service>

Подключается к данному сервису. Стандартный ввод и стандартный вывод помощника подключены к указанной службе (префикс git включен в имя службы, поэтому, например, при извлечении используется git-upload-pack в качестве службы) на удаленной стороне. Действительными ответами на эту команду являются пустая строка (соединение установлено), fallback (нет поддержки интеллектуального транспорта, возврат к тупому транспорту) и просто выход с выведенным сообщением об ошибке (невозможно подключиться, не пытайтесь вернуться назад). После перевода строки, завершающего положительный (пустой) ответ, начинается вывод услуги. После завершения соединения удаленный помощник завершает работу.

Поддерживается, если помощник имеет возможность "connect" .

stateless-connect <service>

Экспериментальный; только для внутреннего использования. Подключается к данной удаленной службе для связи с использованием проводного протокола git версии 2. Действительными ответами на эту команду являются пустая строка (соединение установлено), fallback (нет поддержки интеллектуального транспорта, возврат к тупому транспорту) и просто выход с напечатанным сообщением об ошибке ( не могу подключиться, не пытайтесь отступить). После перевода строки, завершающего положительный (пустой) ответ, начинается вывод службы. Сообщения (как запрос, так и ответ) должны состоять из нуля или более PKT-LINE, оканчивающихся флеш-пакетом. Ответные сообщения будут иметь пакет завершения ответа после пакета сброса, указывающий на конец ответа. Клиент не должен ожидать, что сервер сохранит какое-либо состояние между парами запрос-ответ. После завершения соединения удаленный помощник завершает работу.

Поддерживается, если помощник имеет возможность "stateless-connect" .

получить <uri> <путь>

Загружает файл с данного <uri> на данный <path> . Если <path>.temp существует, то Git предполагает, что файл .temp представляет собой частичную загрузку с предыдущей попытки, и возобновит загрузку с этой позиции.

Если происходит фатальная ошибка, программа записывает сообщение об ошибке в stderr и завершает работу. Вызывающий должен ожидать, что будет напечатано подходящее сообщение об ошибке, если дочерний элемент закроет соединение, не выполнив действительный ответ на текущую команду.

Могут поддерживаться дополнительные команды, что можно определить по возможностям, сообщенным помощником.

Атрибуты списка ссылок

Команда list создает список ссылок, в котором за каждой ссылкой может следовать список атрибутов. Определены следующие атрибуты списка ссылок.

unchanged

Эта ссылка не изменилась с момента последнего импорта или выборки, хотя помощник не обязательно может определить, какое значение это создало.

Ключевые слова в списке ссылок

Команда list может создать список пар ключ-значение. Определены следующие ключи.

object-format

Ссылки используют данный алгоритм хеширования. Это ключевое слово используется только в том случае, если сервер и клиент поддерживают расширение объектного формата.

Options

Следующие параметры определяются и (при соответствующих обстоятельствах) устанавливаются Git , если удаленный помощник имеет возможности option .

опция подробностей <n>

Изменяет степень детализации сообщений, отображаемых помощником. Значение 0 для <n> означает, что процессы работают спокойно, а помощник выдает только ошибки. 1 — это уровень детализации по умолчанию, а более высокие значения <n> соответствуют количеству флагов -v, передаваемых в командной строке.

прогресс опции {true|false}

Включает (или отключает) сообщения о ходе выполнения, отображаемые помощником транспорта во время команды.

глубина опции <глубина>

Углубляет историю мелкого репозитория.

'option deep-since <метка времени>

Углубляет историю мелкого репозитория на основе времени.

'опция углублять-не <ref>

Углубляет историю мелкого репозитория, исключая ссылку. Несколько вариантов суммируются.

опция deepe-relative {'true|false}

Углубляет историю мелкого репозитория относительно текущей границы. Действительно только при использовании с "option depth" .

опция Followtags {true|false}

Если этот параметр включен, помощник должен автоматически извлекать аннотированные объекты тегов, если объект, на который указывает тег, был передан во время команды выборки. Если помощник не извлекает тег, обычно отправляется вторая команда выборки, чтобы запросить именно тег. Некоторые помощники могут использовать эту опцию, чтобы избежать второго сетевого подключения.

option dry-run { true | false }: Если это правда, притворитесь, что операция завершилась успешно, но фактически не изменяйте данные репозитория. Для большинства помощников это относится только к push , если он поддерживается.

опция servpath <c-style-путь в кавычках>

Устанавливает путь к сервису (--upload-pack, --receive-pack и т. д.) для следующего подключения. Удаленный помощник может поддерживать эту опцию, но не должен полагаться на то, что эта опция будет установлена ​​до того, как будет выполнен запрос на соединение.

опция проверки подключения {true|false}

Попросите помощника проверить подключение клона.

опция принудительного действия {true|false}

Попросите помощника выполнить принудительное обновление. По умолчанию false .

вариант клонирования {true|false}

Сообщите помощнику, что это запрос на клонирование (т. е. текущий репозиторий гарантированно пуст).

опция update-shallow {true|false}

Разрешить расширение .git/shallow, если этого требуют новые ссылки.

опция pushcert {true|false}

Знак GPG толкает.

'option push-option <строка>

Передавайте <строку> как опцию push. Поскольку опция push не должна содержать символы LF или NUL, строка не кодируется.

вариант from-обещателя {true|false}

Укажите, что эти объекты извлекаются из обещающего.

опция без зависимостей {true|false}

Укажите, что необходимо извлечь только нужные объекты, а не их зависимые объекты.

вариант атомарный {true|false}

При отправке запросите удаленный сервер обновить ссылки в одной атомарной транзакции. В случае успеха все ссылки будут обновлены или none . Если удаленная сторона не поддерживает эту возможность, передача не удастся.

вариант формата объекта {true|алгоритм}

Если true , укажите, что вызывающая сторона хочет, чтобы информация об алгоритме хэширования была передана обратно с удаленного устройства. Этот режим используется при получении ссылок.

Если установлен алгоритм, укажите, что вызывающий абонент хочет взаимодействовать с удаленной стороной, используя этот алгоритм.
