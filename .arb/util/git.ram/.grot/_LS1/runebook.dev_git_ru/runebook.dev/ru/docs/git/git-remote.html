git-remote

Name

git-remote — Управление набором отслеживаемых репозиториев

Synopsis

git remote [-v | --verbose]
git remote add [-t <branch>] [-m <master>] [-f] [--[no-]tags] [--mirror=(fetch|push)] <name> <URL>
git remote rename [--[no-]progress] <old> <new>
git remote remove <name>
git remote set-head <name> (-a | --auto | -d | --delete | <branch>)
git remote set-branches [--add] <name> <branch>…​
git remote get-url [--push] [--all] <name>
git remote set-url [--push] <name> <newurl> [<oldurl>]
git remote set-url --add [--push] <name> <newurl>
git remote set-url --delete [--push] <name> <URL>
git remote [-v | --verbose] show [-n] <name>…​
git remote prune [-n | --dry-run] <name>…​
git remote [-v | --verbose] update [-p | --prune] [(<group> | <remote>)…​]

Description

Управляйте набором репозиториев ( "remotes" ), ветки которых вы отслеживаете.

Options

-v
--verbose

Будьте немного более подробными и показывайте удаленный URL-адрес после имени. Для пультов Promisor также укажите, какой фильтр ( blob:none и т. д.) настроен. ПРИМЕЧАНИЕ. Его необходимо поместить между remote и подкомандой.

Commands

Без аргументов показывает список существующих пультов. Для выполнения операций на пультах дистанционного управления доступно несколько подкоманд.

add

Добавьте удаленный сервер с именем <name> для репозитория по адресу <URL>. Затем команду git fetch <name> можно использовать для создания и обновления ветвей удаленного отслеживания <имя>/<ветвь>.

С опцией -f git fetch <name> запускается сразу после настройки удаленной информации.

С опцией --tags git fetch <name> импортирует каждый тег из удаленного репозитория.

С опцией --no-tags git fetch <name> не импортирует теги из удаленного репозитория.

По умолчанию импортируются только теги выбранных ветвей (см. git-fetch[1] ).

При использовании опции -t <branch> вместо стандартной глобальной спецификации для удаленного отслеживания всех ветвей в пространстве имен refs/remotes/<name>/ создается ссылка для отслеживания только <branch> . Вы можете предоставить более одного -t <branch> для отслеживания нескольких ветвей, не захватывая все ветки.

С опцией -m <master> символическая ссылка refs/remotes/<name>/HEAD настраивается так, чтобы указывать на удаленную ветвь <master> . См. также команду set-head.

Когда зеркало выборки создается с помощью --mirror=fetch , ссылки не будут храниться в пространстве имен refs/remotes/ , а все, что находится в refs/ на удаленном компьютере, будет напрямую зеркалировано в refs/ в локальном репозитории. Эта опция имеет смысл только в пустых репозиториях, поскольку выборка перезапишет любые локальные коммиты.

Когда принудительное зеркало создается с помощью --mirror=push , git push всегда будет вести себя так, как если бы был передан --mirror .

rename

Переименуйте пульт с именем <old> в <new>. Все ветки удаленного отслеживания и параметры конфигурации удаленного устройства обновляются.

Если <old> и <new> одинаковы, а <old> — это файл $GIT_DIR/remotes или $GIT_DIR/branches , пульт дистанционного управления преобразуется в формат файла конфигурации.

remove
rm

Удалите пульт с именем <имя>. Все ветки удаленного отслеживания и параметры конфигурации удаленного устройства удалены.

set-head

Устанавливает или удаляет ветку по умолчанию (т. е. цель символической ссылки refs/remotes/<name>/HEAD ) для именованного удаленного устройства. Наличие ветки по умолчанию для удаленного устройства не является обязательным, но позволяет указать имя удаленного устройства вместо конкретной ветки. Например, если ветка по умолчанию для origin установлена ​​на master , то origin можно указать там, где вы обычно указываете origin/master .

В -d или --delete символический номер refs/remotes/<name>/HEAD удаляется.

При использовании -a или --auto удаленному устройству запрашивается определение его HEAD , затем для символической ссылки refs/remotes/<name>/HEAD устанавливается та же ветвь. например, если удаленный HEAD указан на next , git remote set-head origin -a установит символическую ссылку refs/remotes/origin/HEAD на refs/remotes/origin/next . Это будет работать, только если refs/remotes/origin/next уже существует; в противном случае его необходимо сначала получить.

Используйте <branch> , чтобы явно установить символическую ссылку refs/remotes/<name>/HEAD . например, git remote set-head origin master установит для символьной ссылки refs/remotes/origin/HEAD значение refs/remotes/origin/master . Это будет работать, только если refs/remotes/origin/master уже существует; в противном случае его необходимо сначала получить.

set-branches

Изменяет список ветвей, отслеживаемых указанным удаленным устройством. Это можно использовать для отслеживания подмножества доступных удаленных веток после первоначальной настройки удаленного узла.

Именованные ветки будут интерпретироваться так, как если бы они были указаны с помощью параметра -t в командной строке git remote add .

С --add вместо замены списка отслеживаемых в данный момент ветвей происходит добавление к этому списку.

get-url

Получает URL-адреса для удаленного устройства. Здесь расширены конфигурации для insteadOf и pushInsteadOf . По умолчанию отображается только первый URL-адрес.

При использовании --push запрашиваются URL-адреса push, а не извлекаются URL-адреса.

При использовании --all будут перечислены все URL-адреса пульта дистанционного управления.

set-url

Изменяет URL-адреса для удаленного устройства. Устанавливает первый URL-адрес для удаленного <name>, который соответствует регулярному выражению <oldurl> (первый URL-адрес, если <oldurl> не указан), в <newurl>. Если <oldurl> не соответствует ни одному URL-адресу, возникает ошибка и ничего не меняется.

В --push манипулируются URL-адресами push-уведомлений, а не URL-адресами выборки.

В --add вместо изменения существующих URL-адресов добавляется новый URL-адрес.

При использовании --delete вместо изменения существующих URL-адресов все URL-адреса, соответствующие регулярному выражению <URL>, удаляются для удаленного <name>. Попытка удалить все URL-адреса, не предназначенные для push-уведомлений, является ошибкой.

Обратите внимание, что URL-адрес отправки и URL-адрес выборки, даже если они могут быть установлены по-разному, все равно должны ссылаться на одно и то же место. То, что вы отправили на URL-адрес push-уведомления, должно быть тем, что вы бы увидели, если бы сразу получили данные с URL-адреса выборки. Если вы пытаетесь получить данные из одного места (например, из исходного источника) и отправить их в другое (например, из репозитория публикации), используйте два отдельных пульта дистанционного управления.

show

Предоставляет некоторую информацию об удаленном <имя>.

При использовании опции -n удаленные головки не опрашиваются сначала с помощью git ls-remote <name> ; Вместо этого используется кэшированная информация.

prune

Удаляет устаревшие ссылки, связанные с <имя>. По умолчанию устаревшие ветки удаленного отслеживания под <name> удаляются, но в зависимости от глобальной конфигурации и конфигурации удаленного устройства мы можем даже удалить локальные теги, которые не были туда помещены. Эквивалент git fetch --prune <name> , за исключением того, что новые ссылки не извлекаются.

См. раздел «ОБРЕЗКА» git-fetch[1] , чтобы узнать, что он будет сокращать в зависимости от различных конфигураций.

С помощью опции --dry-run можно сообщить, какие ветки будут обрезаны, но не обрезать их на самом деле.

update

Получите обновления для удаленных устройств или удаленных групп в репозитории, как определено remotes.<group> . Если в командной строке не указаны ни группа, ни удаленный доступ, будет использоваться параметр конфигурации Remotes.default; если Remotes.default не определен, все пульты, у которых для параметра конфигурации remote.<name>.skipDefaultUpdate не установлено значение true , будут обновлены. (См. git-config[1] ).

С опцией --prune запустите обрезку для всех обновленных пультов дистанционного управления.

Discussion

Удаленная настройка осуществляется с помощью переменных конфигурации remote.origin.url и remote.origin.fetch . (См. git-config[1] ).

Exit status

В случае успеха статус выхода — 0 .

Если такие подкоманды, как add , rename и remove , не могут найти соответствующий пульт, статус выхода — 2 . Если пульт уже существует, статус выхода — 3 .

В случае любой другой ошибки статус выхода может иметь любое другое ненулевое значение.

Examples

  • Добавьте новый удаленный компьютер, извлеките и извлеките из него ветку.

    $ git remote
origin
$ git branch -r
  origin/HEAD -> origin/master
  origin/master
$ git remote add staging git://git.kernel.org/.../gregkh/staging.git
$ git remote
origin
staging
$ git fetch staging
...
From git://git.kernel.org/pub/scm/linux/kernel/git/gregkh/staging
 * [new branch]      master     -> staging/master
 * [new branch]      staging-linus -> staging/staging-linus
 * [new branch]      staging-next -> staging/staging-next
$ git branch -r
  origin/HEAD -> origin/master
  origin/master
  staging/master
  staging/staging-linus
  staging/staging-next
$ git switch -c staging staging/master
...

  • Имитируйте git clone , но отслеживайте только выбранные ветки.

    $ mkdir project.git
$ cd project.git
$ git init
$ git remote add -f -t master -m master origin git://example.com/git.git/
$ git merge origin
