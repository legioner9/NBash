git-svn

Name

git-svn — двунаправленная операция между репозиторием Subversion и Git .

Synopsis

git svn <command> [<options>] [<arguments>]

Description

git svn — это простой канал для наборов изменений между Subversion и Git . Он обеспечивает двунаправленный поток изменений между Subversion и репозиторием Git .

git svn может отслеживать стандартный репозиторий Subversion, следуя общей схеме "trunk/branches/tags" , с опцией --stdlayout. Он также может следовать за ветвями и тегами в любом макете с помощью параметров -T/-t/-b (см. параметры init ниже, а также команду clone ).

После отслеживания репозитория Subversion (любым из вышеперечисленных методов) репозиторий Git можно обновить из Subversion с помощью команды fetch , а Subversion обновить с Git с помощью команды dcommit .

Commands

init

Инициализирует пустой репозиторий Git с дополнительными каталогами метаданных для git svn . URL-адрес Subversion может быть указан как аргумент командной строки или как полный аргумент URL-адреса для -T/-t/-b. При желании целевой каталог для работы можно указать в качестве второго аргумента. Обычно эта команда инициализирует текущий каталог.

-T<trunk_subdir>
--trunk=<trunk_subdir>
-t<tags_subdir>
--tags=<tags_subdir>
-b<branches_subdir>
--branches=<branches_subdir>
-s
--stdlayout

Это необязательные параметры командной строки для init. Каждый из этих флагов может указывать на относительный путь к репозиторию (--tags=project/tags) или полный URL-адрес (--tags=https://foo.org/project/tags). Вы можете указать более одного параметра --tags и/или --branches, если ваш репозиторий Subversion размещает теги или ветки по нескольким путям. Опция --stdlayout — это сокращенный способ установки ствола, тегов и ветвей в качестве относительных путей, что является значением Subversion по умолчанию. Если также указаны какие-либо другие параметры, они имеют приоритет.

--no-metadata

Установите опцию noMetadata в конфигурации [svn-remote]. Этот параметр не рекомендуется. Прежде чем использовать этот параметр, прочтите раздел svn.noMetadata на этой странице руководства.

--use-svm-props

Установите опцию useSvmProps в конфигурации [svn-remote].

--use-svnsync-props

Установите опцию useSvnsyncProps в конфигурации [svn-remote].

--rewrite-root=<URL>

Установите опцию rewriteRoot в конфигурации [svn-remote].

--rewrite-uuid=<UUID>

Установите опцию rewriteUUID в конфигурации [svn-remote].

--username=<user>

Для транспортов, для которых SVN обрабатывает аутентификацию (http, https и обычный svn), укажите имя пользователя. Для других видов транспорта (например, svn+ssh:// ) вы должны включить имя пользователя в URL-адрес, например svn+ssh://[email protected]/project .

--prefix=<prefix>

Это позволяет указать префикс, который добавляется к именам удаленных устройств, если указаны магистраль/ветви/теги. Префикс не включает автоматически завершающую косую черту, поэтому обязательно включите ее в аргумент, если вы этого хотите. Если указан --branches/-b, префикс должен включать косую черту в конце. Установка префикса (с косой чертой в конце) настоятельно рекомендуется в любом случае, так как тогда ваши ссылки для отслеживания SVN будут расположены по адресу "refs/remotes/$prefix/" , что совместимо с собственным макетом ссылок для удаленного отслеживания Git (refs/remotes/$remote/). Установка префикса также полезна, если вы хотите отслеживать несколько проектов, имеющих общий репозиторий. По умолчанию префикс установлен на origin/ .

Note
 До Git версии 2.0 префиксом по умолчанию был "" (no prefix). This meant that SVN-tracking refs were put at " refs/remotes/*", что несовместимо с тем, как организованы собственные ссылки удаленного отслеживания Git . Если вам все еще нужен старый префикс по умолчанию, вы можете получить его, передав --prefix "" в командной строки ( --prefix="" может не работать, если Getopt::Long вашего Perl < v2.37).
--ignore-refs=<regex>

При передаче в init или clone это регулярное выражение будет сохранено как ключ конфигурации. См. fetch для описания --ignore-refs .

--ignore-paths=<regex>

При передаче в init или clone это регулярное выражение будет сохранено как ключ конфигурации. См. fetch для описания --ignore-paths .

--include-paths=<regex>

При передаче в init или clone это регулярное выражение будет сохранено как ключ конфигурации. См. fetch для описания --include-paths .

--no-minimize-url

При отслеживании нескольких каталогов (с использованием опций --stdlayout, --branches или --tags) git svn попытается подключиться к корню (или наивысшему разрешенному уровню) репозитория Subversion. Это значение по умолчанию позволяет лучше отслеживать историю, если целые проекты перемещаются внутри репозитория, но может вызвать проблемы в репозиториях, где действуют ограничения доступа на чтение. Передача --no-minimize-url позволит svn git принимать URL-адреса как есть, не пытаясь подключиться к каталогу более высокого уровня. Эта опция по умолчанию отключена, если отслеживается только один URL-адрес/ветвь (это принесет мало пользы).

fetch

Получите неизвлеченные версии с удаленного устройства Subversion, которое мы отслеживаем. Имя раздела [svn-remote "…​" ] в файле $GIT_DIR/config можно указать в качестве необязательного аргумента командной строки.

При необходимости это автоматически обновляет rev_map (подробности см. в $GIT_DIR/svn/**/.rev_map.* в разделе «ФАЙЛЫ» ниже).

--localtime

Сохраняйте время фиксации Git в местном часовом поясе, а не в формате UTC. Это заставляет git log (даже без --date=local) показывать то же время, что и svn log в местном часовом поясе.

Это не мешает взаимодействию с репозиторием Subversion, из которого вы клонировали, но если вы хотите, чтобы ваш локальный репозиторий Git мог взаимодействовать с чужим локальным репозиторием Git , либо не используйте эту опцию, либо вам следует использовать ее вместе тот же местный часовой пояс.

--parent

Извлекайте только из родительского SVN текущего HEAD.

--ignore-refs=<regex>

Игнорируйте ссылки на ветки или теги, соответствующие регулярному выражению Perl. "negative look-ahead assertion" , как и ^refs/remotes/origin/(?!tags/wanted-tag|wanted-branch).*$ , можно использовать для разрешения только определенных ссылок.

config key: svn-remote.<name>.ignore-refs

Если задан конфигурационный ключ ignore-refs, а также указан параметр командной строки, будут использоваться оба регулярных выражения.

--ignore-paths=<regex>

Это позволяет указать регулярное выражение Perl, которое будет вызывать пропуск всех совпадающих путей при оформлении заказа из SVN. Опция --ignore-paths должна соответствовать каждому fetch (включая автоматические выборки из-за clone , dcommit , rebase и т. д.) в данном репозитории.

config key: svn-remote.<name>.ignore-paths

Если установлен ключ конфигурации ignore-paths и указана опция командной строки, будут использоваться оба регулярных выражения.

Examples:

Пропускать каталог "doc*" при каждой выборке
--ignore-paths="^doc"
Пропустить "branches" и "tags" каталогов первого уровня.
--ignore-paths="^[^/]+/(?:branches|tags)"
--include-paths=<regex>

Это позволяет указать регулярное выражение Perl, которое будет вызывать включение только совпадающих путей при оформлении заказа из SVN. Опция --include-paths должна соответствовать каждому fetch (включая автоматические выборки из-за clone , dcommit , rebase и т. д.) в данном репозитории. --ignore-paths имеет приоритет над --include-paths .

config key: svn-remote.<name>.include-paths
--log-window-size=<n>

Получить <n> записей журнала по запросу при сканировании истории Subversion. Значение по умолчанию — 100. Для очень больших репозиториев Subversion могут потребоваться большие значения, чтобы clone / fetch завершились в разумные сроки. Но слишком большие значения могут привести к более высокому использованию памяти и тайм-аутам запросов.

clone

Работает с init и fetch . Он автоматически создаст каталог на основе базового имени переданного ему URL-адреса; или если передан второй аргумент; он создаст каталог и будет работать в нем. Он принимает все аргументы, которые принимают команды init и fetch ; за исключением --fetch-all и --parent . После клонирования репозитория команда fetch сможет обновлять версии, не затрагивая рабочее дерево; а команда rebase сможет обновлять рабочее дерево с учетом последних изменений.

--preserve-empty-dirs

Создайте файл-заполнитель в локальном репозитории Git для каждого пустого каталога, полученного из Subversion. Сюда входят каталоги, которые становятся пустыми в результате удаления всех записей в репозитории Subversion (но не самого каталога). Файлы-заполнители также отслеживаются и удаляются, когда в них больше нет необходимости.

--placeholder-filename=<filename>

Задайте имена файлов-заполнителей, созданных с помощью --preserve-empty-dirs. По умолчанию: ".gitignore"

rebase

Это извлекает версии из родительского SVN текущего HEAD и перебазирует текущую (незафиксированную в SVN) работу с ним.

Это работает аналогично svn update или git pull , за исключением того, что он сохраняет линейную историю с git rebase вместо git merge для упрощения фиксации с git svn .

При этом принимаются все параметры, которые поддерживают git svn fetch и git rebase . Однако --fetch-all извлекает данные только из текущего определения [svn-remote], а не из всех определений [svn-remote].

Как git rebase ; для этого необходимо, чтобы рабочее дерево было чистым и не содержало незафиксированных изменений.

При необходимости это автоматически обновляет rev_map (подробности см. в $GIT_DIR/svn/**/.rev_map.* в разделе «ФАЙЛЫ» ниже).

-l
--local

Не скачивайте удаленно; запускайте git rebase только для последнего полученного коммита из восходящего SVN.

dcommit

Фиксируйте каждую разницу из текущей ветки непосредственно в репозиторий SVN, а затем переустанавливайте или сбрасывайте ее (в зависимости от того, существует ли разница между SVN и заголовком). Это создаст ревизию в SVN для каждого коммита в Git.

Если в качестве аргумента указано необязательное имя ветки Git (или имя объекта фиксации Git ), подкоманда работает с указанной веткой, а не с текущей веткой.

Использование dcommit предпочтительнее set-tree (ниже).

--no-rebase

После фиксации не делайте перебазирование или сброс.

--commit-url <URL>

Зафиксируйте этот URL-адрес SVN (полный путь). Это предназначено для того, чтобы позволить повторно использовать существующие репозитории git svn , созданные с помощью одного метода транспорта (например, svn:// или http:// для анонимного чтения), если пользователю позже будет предоставлен доступ к альтернативному методу транспорта (например, svn+ssh:// или https:// ) для фиксации.

config key: svn-remote.<name>.commiturl
config key: svn.commiturl (overwrites all svn-remote.<name>.commiturl options)

Обратите внимание, что URL-адрес SVN конфигурационного ключа commiturl включает ветку SVN. Если вы предпочитаете установить URL-адрес фиксации для всего репозитория SVN, используйте вместо этого svn-remote.<name>.pushurl.

Использование этой опции для каких-либо других целей (не спрашивайте) настоятельно не рекомендуется.

--mergeinfo=<mergeinfo>

Добавьте данную информацию о слиянии во время dcommit (например, --mergeinfo="/branches/foo:1-10" ). Все версии сервера svn могут хранить эту информацию (как свойство), а клиенты svn, начиная с версии 1.5, могут ее использовать. Чтобы указать информацию о слиянии из нескольких ветвей, используйте один пробел между ветвями ( --mergeinfo="/branches/foo:1-10 /branches/bar:3,5-6,8" ).

config key: svn.pushmergeinfo

Эта опция приведет к тому, что git-svn попытается автоматически заполнить свойство svn:mergeinfo в репозитории SVN, если это возможно. В настоящее время это можно сделать только при выполнении слияний без перемотки вперед, когда все родительские элементы, кроме первого, уже были отправлены в SVN.

--interactive

Попросите пользователя подтвердить, что набор исправлений действительно должен быть отправлен в SVN. По каждому патчу можно ответить "yes" (принять этот патч), "no" (отменить этот патч), "all" (принять все патчи) или "quit" .

git svn dcommit возвращается немедленно, если ответ "no" или "quit" , без каких-либо изменений в SVN.

branch

Создайте ветку в репозитории SVN.

-m
--message

Позволяет указать сообщение фиксации.

-t
--tag

Создайте тег, используя tags_subdir вместо Branch_subdir, указанного во время инициализации svn git .

-d<path>
--destination=<path>

Если команде init или clone было присвоено более одной опции --branches (или --tags), вы должны указать расположение ветки (или тега), которую вы хотите создать, в репозитории SVN. <path> указывает, какой путь использовать для создания ветки или тега, и должен соответствовать шаблону в левой части одной из настроенных спецификаций ссылок на ветки или теги. Вы можете просмотреть эти спецификации с помощью команд

git config --get-all svn-remote.<name>.branches
git config --get-all svn-remote.<name>.tags

где <имя> — это имя репозитория SVN, указанное опцией -R для init (или "svn" по умолчанию).

--username

Укажите имя пользователя SVN, от которого будет выполняться фиксация. Этот параметр переопределяет свойство конфигурации username .

--commit-url

Используйте указанный URL-адрес для подключения к целевому репозиторию Subversion. Это полезно в тех случаях, когда исходный репозиторий SVN доступен только для чтения. Эта опция переопределяет свойство конфигурации commiturl .

git config --get-all svn-remote.<name>.commiturl
--parents

Создайте родительские папки. Этот параметр эквивалентен параметру --parents в командах svn cp и полезен для нестандартных макетов репозитория.

tag

Создайте тег в репозитории SVN. Это сокращение от branch -t .

log

Это должно облегчить поиск сообщений журнала svn, когда пользователи svn обращаются к номерам -r/--revision.

Поддерживаются следующие функции из журнала svn:

-r <n>[:<n>]
--revision=<n>[:<n>]

поддерживается, нечисловые аргументы — нет: HEAD, NEXT, BASE, PREV и т. д.

-v
--verbose

он не полностью совместим с выводом --verbose в журнале svn, но достаточно близок.

--limit=<n>

НЕ то же самое, что --max-count, не учитывает объединенные/исключенные коммиты

--incremental

supported

New features:

--show-commit

также показывает коммит Git sha1

--oneline

наша версия --pretty=oneline

Note
 Сам SVN хранит только время в формате UTC и ничего больше. Обычный клиент svn преобразует время UTC в местное время (или на основе среды TZ=). Эта команда имеет такое же поведение.

Любые другие аргументы передаются непосредственно в git log .

blame

Покажите, какая редакция и автор в последний раз изменяли каждую строку файла. Вывод этого режима по умолчанию совместим по формату с выводом svn Assessment. Как и команда SVN обвинять, локальные незафиксированные изменения в рабочем дереве игнорируются; версия файла в редакции HEAD аннотируется. Неизвестные аргументы передаются непосредственно в git blame .

--git-format

Выведите выходные данные в том же формате, что и git blame , но с номерами ревизий SVN вместо хэшей фиксации Git . В этом режиме изменения, которые не были зафиксированы в SVN (включая изменения локальной рабочей копии), отображаются как версия 0.

find-rev

Если задан номер версии SVN в форме rN , возвращается соответствующий хеш фиксации Git (за ним может следовать древовидная структура, чтобы указать, в какой ветке следует искать). Если задано древовидное представление, возвращает соответствующий номер версии SVN.

-B
--before

Не требуйте точного совпадения, если дана ревизия SVN, вместо этого найдите фиксацию, соответствующую состоянию репозитория SVN (в текущей ветке) в указанной ревизии.

-A
--after

Не требовать точного совпадения, если указана версия SVN; если точного совпадения нет, верните ближайшее совпадение в истории поиска.

set-tree

Вам следует рассмотреть возможность использования dcommit вместо этой команды. Зафиксируйте указанные объекты коммита или дерева в SVN. Это зависит от актуальности импортированных данных выборки. Это не делает абсолютно никаких попыток сделать патчи при коммите в SVN, он просто перезаписывает файлы теми, которые указаны в дереве или коммите. Предполагается, что все слияния произошли независимо от функций git svn .

create-ignore

Рекурсивно находит свойство svn:ignore в каталогах и создает соответствующие файлы .gitignore. Полученные файлы готовятся к фиксации, но не фиксируются. Используйте -r/--revision для ссылки на конкретную версию.

show-ignore

Рекурсивно находит и перечисляет свойство svn:ignore в каталогах. Вывод подходит для добавления в файл $GIT_DIR/info/exclude.

mkdirs

Попытки воссоздать пустые каталоги, которые ядро ​Git не может отслеживать, на основе информации в файлах $GIT_DIR/svn/<refname>/unhandled.log. Пустые каталоги автоматически воссоздаются при использовании "git svn clone" и "git svn rebase" , поэтому "mkdirs" предназначен для использования после таких команд, как "git checkout" или "git reset" . (Дополнительную информацию см. в параметре конфигурационного файла svn-remote.<name>.automkdirs.)

commit-diff

Фиксирует разницу двух древовидных аргументов из командной строки. Эта команда не зависит от нахождения внутри репозитория git svn init . Эта команда принимает три аргумента: (а) исходное дерево для сравнения, (б) результат нового дерева, (в) URL-адрес целевого репозитория Subversion. Последний аргумент (URL) можно опустить, если вы работаете с репозиторием, поддерживающим git svn (который был обработан init с помощью git svn ). Для этого необходима опция -r<ревизия>.

Сообщение фиксации предоставляется либо напрямую с опцией -m или -F , либо косвенно из тега или фиксации, когда второе дерево обозначает такой объект, либо оно запрашивается путем вызова редактора (см. опцию --edit ниже).

-m <msg>
--message=<msg>

Используйте данный msg в качестве сообщения фиксации. Эта опция отключает опцию --edit .

-F <filename>
--file=<filename>

Возьмите сообщение фиксации из данного файла. Эта опция отключает опцию --edit .

info

Показывает информацию о файле или каталоге, аналогичную той, что предоставляет «svn info». В настоящее время не поддерживает аргумент -r/--revision. Используйте опцию --url для вывода только значения поля URL: .

proplist

Перечисляет свойства, хранящиеся в репозитории Subversion, для данного файла или каталога. Используйте -r/--revision для ссылки на конкретную версию Subversion.

propget

Получает свойство Subversion, заданное в качестве первого аргумента для файла. Конкретную версию можно указать с помощью -r/--revision.

propset

Устанавливает для свойства Subversion, указанного в качестве первого аргумента, значение, указанное в качестве второго аргумента для файла, указанного в качестве третьего аргумента.

Example:

git svn propset svn:keywords "FreeBSD=%H" devel/py-tipper/Makefile

Это установит для свойства svn:keywords значение FreeBSD=%H для файла devel/py-tipper/Makefile .

show-externals

Показывает внешние элементы Subversion. Используйте -r/--revision, чтобы указать конкретную версию.

gc

Сожмите файлы $GIT_DIR/svn/<refname>/unhandled.log и удалите файлы $GIT_DIR/svn/<refname>/index.

reset

Отменяет эффекты fetch обратно на указанную версию. Это позволит вам повторно использовать версию SVN fetch . Обычно содержимое версии SVN никогда не должно меняться, и reset не требуется. Однако, если разрешения SVN изменяются или если вы измените параметр --ignore-paths, fetch может выйти из строя с "not found in commit" (файл ранее не был виден) или "checksum mismatch" (пропущена модификация). Если проблемный файл нельзя игнорировать навсегда (с помощью --ignore-paths), единственный способ восстановить репозиторий — использовать reset .

Изменяются только rev_map и refs/remotes/git-svn (подробности см. в $GIT_DIR/svn/**/.rev_map.* в разделе ФАЙЛЫ ниже). Следуйте за reset fetch , а затем git reset или git rebase , чтобы переместить локальные ветви в новое дерево.

-r <n>
--revision=<n>

Укажите самую последнюю версию, которую нужно сохранить. Все более поздние версии отбрасываются.

-p
--parent

Также отмените указанную ревизию, сохранив вместо нее ближайшую родительскую версию.

Example:

Предположим, у вас есть локальные изменения в "master" , но вам необходимо обновить "r2" .

    r1---r2---r3 remotes/git-svn
                \
                 A---B master

Исправьте проблему с игнорированием путей или разрешений SVN, из-за которой "r2" изначально был неполным. Затем:

git svn reset -r2 -p
git svn fetch
    r1---r2'--r3' remotes/git-svn
      \
       r2---r3---A---B master

Затем исправьте "master" на git rebase . НЕ используйте git merge , иначе ваша история не будет совместима с будущим dcommit !

git rebase --onto remotes/git-svn A^ master
    r1---r2'--r3' remotes/git-svn
                \
                 A'--B' master

Options

--shared[=(false|true|umask|group|all|world|everybody)]
--template=<template-directory>

Используется только с командой init . Они передаются непосредственно в git init .

-r <arg>
--revision <arg>

Используется с командой fetch .

Это позволяет поддерживать диапазоны ревизий для частичной/прижигаемой истории. Поддерживаются $NUMBER, $NUMBER1:$NUMBER2 (числовые диапазоны), $NUMBER:HEAD и BASE:$NUMBER.

Это может позволить вам создавать частичные зеркала при выполнении выборки; но обычно это не рекомендуется, поскольку история будет пропущена и потеряна.

-
--stdin

Используется только с командой set-tree .

Прочтите список коммитов из стандартного ввода и зафиксируйте их в обратном порядке. Из каждой строки считывается только ведущий sha1, поэтому можно использовать выход git rev-list --pretty=oneline .

--rmdir

Используется только с командами dcommit , set-tree и commit-diff .

Удалите каталоги из дерева SVN, если в нем не осталось файлов. SVN может версионировать пустые каталоги, и они по умолчанию не удаляются, если в них не осталось файлов. Git не может создавать версии для пустых каталогов. Включение этого флага приведет к тому, что фиксация в SVN будет действовать как Git .

config key: svn.rmdir
-e
--edit

Используется только с командами dcommit , set-tree и commit-diff .

Отредактируйте сообщение о фиксации перед фиксацией в SVN. По умолчанию эта функция отключена для объектов, которые фиксируются, и включается принудительно при фиксации объектов дерева.

config key: svn.edit
-l<num>
--find-copies-harder

Используется только с командами dcommit , set-tree и commit-diff .

Оба они передаются непосредственно в git diff-tree ; дополнительную информацию см. в git-diff-tree[1] .

config key: svn.l
config key: svn.findcopiesharder
-A<filename>
--authors-file=<filename>

Синтаксис совместим с файлом, используемым git cvsimport , но вместе с <> можно указать пустой адрес электронной почты:

        loginname = Joe User <user@example.com>

Если указана эта опция и git svn встретит имя коммиттера SVN, которого нет в файле авторов, git svn прервет операцию. Затем пользователю придется добавить соответствующую запись. Повторный запуск предыдущей команды git svn после изменения файла автора должен продолжить работу.

config key: svn.authorsfile
--authors-prog=<filename>

Если указана эта опция, для каждого имени коммиттера SVN, которого нет в файле авторов, данный файл выполняется с именем коммиттера в качестве первого аргумента. Ожидается, что программа вернет одну строку вида "Name <email>" или "Name <>" , которая будет считаться включенной в файл автора.

По историческим причинам относительный filename сначала ищется относительно текущего каталога для init и clone и относительно корня рабочего дерева для fetch . Если filename не найден, он ищется как любая другая команда в $PATH .

config key: svn.authorsProg
-q
--quiet

Сделайте git svn менее многословным. Укажите второй раз, чтобы сделать его еще менее многословным.

-m
--merge
-s<strategy>
--strategy=<strategy>
-p
--rebase-merges

Они используются только с командами dcommit и rebase .

Передается непосредственно в git rebase при использовании dcommit , если невозможно использовать git reset (см. dcommit ).

-n
--dry-run

Это можно использовать с командами dcommit , rebase , branch и tag .

Для dcommit распечатайте серию аргументов Git , которые покажут, какие различия будут переданы в SVN.

Для rebase отобразите локальную ветвь, связанную с вышестоящим репозиторием svn, связанным с текущей ветвью, и URL-адрес репозитория svn, из которого будет получена информация.

Для branch и tag отобразите URL-адреса, которые будут использоваться для копирования при создании ветки или тега.

--use-log-author

При получении фиксации svn в Git (как часть операций fetch , rebase или dcommit ) найдите первую строку From: или трейлер Signed-off-by в сообщении журнала и используйте ее в качестве строки автора.

config key: svn.useLogAuthor
--add-author-from

При фиксации svn из Git (как часть операций set-tree или dcommit ), если существующее сообщение журнала еще не имеет трейлера From: или Signed-off-by , добавьте строку From: на основе строки автора фиксации Git . Если вы используете это, --use-log-author будет получать действительную строку автора для всех коммитов.

config key: svn.addAuthorFrom

Advanced options

-i<GIT_SVN_ID>
--id <GIT_SVN_ID>

Это устанавливает GIT_SVN_ID (вместо использования среды). Это позволяет пользователю переопределить имя ссылки по умолчанию для извлечения при отслеживании одного URL-адреса. Команды log и dcommit больше не требуют этого переключателя в качестве аргумента.

-R<remote name>
--svn-remote <имя удаленного устройства>

Укажите раздел [svn-remote "<remote name>" ], который будет использоваться. Это позволит отслеживать несколько репозиториев SVN. По умолчанию: "svn"

--follow-parent

Эта опция актуальна только в том случае, если мы отслеживаем ветки (используя один из вариантов макета репозитория --trunk, --tags, --branches, --stdlayout). Для каждой отслеживаемой ветки попытайтесь выяснить, откуда была скопирована ее ревизия, и установите подходящий родительский элемент в первом коммите Git для этой ветки. Это особенно полезно, когда мы отслеживаем каталог, который был перемещен внутри репозитория. Если эта функция отключена, все ветки, созданные git svn , будут линейными и не будут иметь общей истории, а это означает, что не будет никакой информации о том, где ветки были разветвлены или объединены. Однако отслеживание длинной/запутанной истории может занять много времени, поэтому отключение этой функции может ускорить процесс клонирования. Эта функция включена по умолчанию, используйте --no-follow-parent, чтобы отключить ее.

config key: svn.followparent

Параметры только для файла конфигурации

svn.noMetadata
svn-remote.<name>.noMetadata

Это избавит от строк git-svn-id: в конце каждого коммита.

Эту опцию можно использовать только для однократного импорта, поскольку git svn не сможет выполнить повторную выборку без метаданных. Кроме того, если вы потеряете файлы $GIT_DIR/svn/**/.rev_map.* , git svn не сможет их восстановить.

Команда git svn log также не будет работать с репозиториями, использующими это. Использование этого параметра противоречит опции useSvmProps по (надеюсь) очевидным причинам.

Этот вариант НЕ рекомендуется, так как он затрудняет отслеживание старых ссылок на номера ревизий SVN в существующей документации, отчетах об ошибках и архивах. Если вы планируете в конечном итоге перейти с SVN на Git и уверены в удалении истории SVN, рассмотрите вместо этого git-filter-repo . filter-repo также позволяет переформатировать метаданные для удобства чтения и перезаписи информации об авторстве для пользователей, не являющихся пользователями "svn.authorsFile" .

svn.useSvmProps
svn-remote.<name>.useSvmProps

Это позволяет git svn пересопоставлять URL-адреса репозитория и UUID из зеркал, созданных с использованием SVN::Mirror (или svk) для метаданных.

Если ревизия SVN имеет свойство "svm:headrev" , вполне вероятно, что ревизия была создана SVN::Mirror (также используемым SVK). Свойство содержит UUID репозитория и ревизию. Мы хотим, чтобы это выглядело так, как будто мы зеркально отображаем исходный URL-адрес, поэтому введите вспомогательную функцию, которая возвращает исходный URL-адрес удостоверения и UUID, и используйте ее при создании метаданных в сообщениях фиксации.

svn.useSvnsyncProps
svn-remote.<name>.useSvnsyncprops

Аналогично параметру useSvmProps; это для пользователей команды svnsync(1), поставляемой с SVN 1.4.x и более поздних версий.

svn-remote.<name>.rewriteRoot

Это позволяет пользователям создавать репозитории из альтернативных URL-адресов. Например, администратор может запустить git svn на сервере локально (доступ через file://), но хочет распространить репозиторий с URL-адресом public http:// или svn:// в метаданных, чтобы пользователи видели public . URL-адрес.

svn-remote.<name>.rewriteUUID

Аналогично параметру useSvmProps; это для пользователей, которым необходимо переназначить UUID вручную. Это может быть полезно в ситуациях, когда исходный UUID недоступен ни через useSvmProps, ни через useSvnsyncProps.

svn-remote.<name>.pushurl

Подобно ключу Git remote.<name>.pushurl , этот ключ предназначен для использования в случаях, когда url указывает на репозиторий SVN через транспорт только для чтения, чтобы обеспечить альтернативный транспорт чтения/записи. Предполагается, что оба ключа указывают на один и тот же репозиторий. В отличие от commiturl , pushurl — это базовый путь. Если можно использовать commiturl или pushurl , commiturl имеет приоритет.

svn.brokenSymlinkWorkaround

Это отключает потенциально дорогостоящие проверки для обхода неработающих символических ссылок, проверенных в SVN сломанными клиентами. Установите для этого параметра значение "false" , если вы отслеживаете репозиторий SVN со множеством пустых больших объектов, которые не являются символическими ссылками. Эту опцию можно изменить во время работы git svn , и она вступит в силу со следующей полученной версией. Если этот параметр не установлен, git svn предполагает, что этот параметр равен "true" .

svn.pathnameencoding

Это дает указание git svn перекодировать имена путей в заданную кодировку. Его могут использовать пользователи Windows и те, кто работает в локалях, отличных от utf8, чтобы избежать повреждения имен файлов с символами, отличными от ASCII. Допустимые кодировки — это те, которые поддерживаются модулем Perl Encode.

svn-remote.<name>.automkdirs

Обычно команды "git svn clone" и "git svn rebase" пытаются воссоздать пустые каталоги, находящиеся в репозитории Subversion. Если для этого параметра установлено значение "false" , пустые каталоги будут создаваться только в том случае, если команда "git svn mkdirs" выполняется явно. Если эта опция не установлена, git svn предполагает, что эта опция — "true" .

Поскольку параметры noMetadata, rewriteRoot, rewriteUUID, useSvnsyncProps и useSvmProps влияют на метаданные, создаваемые и используемые git svn ; они должны быть установлены в файле конфигурации до импорта истории, и эти настройки никогда не следует изменять после их установки.

Кроме того, для каждого раздела svn-remote можно использовать только один из этих параметров, поскольку они влияют на строку метаданных git-svn-id: , за исключением rewriteRoot и rewriteUUID, которые можно использовать вместе.

Basic examples

Отслеживание и участие в стволе проекта, управляемого Subversion (игнорируя теги и ветки):

# Клонировать репозиторий (например, клон git):
        git svn clone http://svn.example.com/project/trunk
# Введите новый клонированный каталог:
        cd trunk
# Вы должны находиться в основной ветке, дважды проверьте с помощью «git Branch».
        git branch
# Проделайте некоторую работу и зафиксируйте локально в Git:
        git commit ...
# Что-то привязано к SVN, перебазируйте ваши локальные изменения в соответствии с
# последние изменения в SVN:
        git svn rebase
# Теперь зафиксируйте ваши изменения (которые ранее были зафиксированы с помощью Git) в SVN,
# а также автоматическое обновление вашего рабочего HEAD:
        git svn dcommit
# Добавляем настройки svn:ignore в файл исключения Git по умолчанию:
        git svn show-ignore >> .git/info/exclude

Отслеживание и участие во всем проекте, управляемом Subversion (вместе со стволом, тегами и ветвями):

# Клонировать репозиторий со стандартной структурой каталога SVN (например, git clone):
        git svn clone http://svn.example.com/project --stdlayout --prefix svn/
# Или, если в репозитории используется нестандартная структура каталога:
        git svn clone http://svn.example.com/project -T tr -b branch -t tag --prefix svn/
# Просмотрите все клонированные вами ветки и теги:
        git branch -r
# Создаем новую ветку в SVN
        git svn branch waldo
# Сбросьте ваш мастер на транк (или любую другую ветку, заменив «транк»
# с соответствующим именем):
        git reset --hard svn/trunk
# Вы можете одновременно использовать только одну ветку/тег/транк. Использование
# dcommit/rebase/show-ignore должен быть таким же, как указано выше.

Первоначальный git svn clone может занять довольно много времени (особенно для больших репозиториев Subversion). Если несколько человек (или один человек с несколькими машинами) хотят использовать git svn для взаимодействия с одним и тем же репозиторием Subversion, вы можете выполнить первоначальный git svn clone для репозитория на сервере и попросить каждого человека клонировать этот репозиторий с помощью git clone :

# Выполняем первоначальный импорт на сервер
        ssh server "cd /pub && git svn clone http://svn.example.com/project [options...]"
# Клонировать локально — убедитесь, что refs/remotes/space соответствует серверу
        mkdir project
        cd project
        git init
        git remote add origin server:/pub/project
        git config --replace-all remote.origin.fetch '+refs/remotes/*:refs/remotes/*'
        git fetch
# Запретить выборку/извлечение данных с удаленного сервера Git в будущем,
# мы хотим использовать git svn только для будущих обновлений
        git config --remove-section remote.origin
# Создаем локальную ветку из одной из только что полученных ветвей
        git checkout -b master FETCH_HEAD
# Инициализируйте 'git svn' локально (обязательно используйте тот же URL-адрес и
# Опции --stdlayout/-T/-b/-t/--prefix, которые использовались на сервере)
        git svn init http://svn.example.com/project [options...]
# Извлекаем последние изменения из Subversion
        git svn rebase

Перебазирование против извлечения/слияния

Предпочитайте использовать git svn rebase или git rebase , а не git pull или git merge для синхронизации неинтегрированных коммитов с веткой git svn . Это позволит сохранить линейную историю неинтегрированных коммитов по отношению к восходящему репозиторию SVN и позволит использовать предпочтительную подкоманду git svn dcommit для отправки неинтегрированных коммитов обратно в SVN.

Первоначально git svn рекомендовалось разработчикам извлечь или объединить ветку git svn . Это произошло потому, что автор предпочитал использовать git svn set-tree B для фиксации одной головы, а не нотацию git svn set-tree A..B для фиксации нескольких коммитов. Использование git pull или git merge с git svn set-tree A..B приведет к сглаживанию нелинейной истории при фиксации в SVN, и это может привести к тому, что коммиты слияния неожиданно отменяют предыдущие фиксации в SVN.

Merge tracking

Хотя git svn может отслеживать историю копирования (включая ветки и теги) для репозиториев, использующих стандартный макет, он пока не может отображать историю слияний, произошедшую внутри git , обратно для пользователей SVN. Поэтому пользователям рекомендуется сохранять историю как можно более линейной внутри Git , чтобы облегчить совместимость с SVN (см. раздел «ПРЕДОСТЕРЕЖЕНИЯ» ниже).

Обработка ветвей SVN

Если git svn настроен на выборку ветвей (и действует --follow-branches), он иногда создает несколько ветвей Git для одной ветки SVN, где дополнительные ветки имеют имена вида branchname@nnn (где nnn — номер версии SVN). Эти дополнительные ветки создаются, если git svn не может найти родительский коммит для первого коммита в ветке SVN, чтобы соединить ветку с историей других ветвей.

Обычно первый коммит в ветке SVN состоит из операции копирования. git svn прочитает этот коммит, чтобы получить версию SVN, из которой была создана ветка. Затем он попытается найти коммит Git , соответствующий этой версии SVN, и использовать его в качестве родительского элемента ветки. Однако возможно, что не существует подходящего коммита Git , который мог бы служить родительским. Это произойдет, среди прочего, если ветвь SVN является копией ревизии, которая не была получена git svn (например, потому что это старая ревизия, которая была пропущена с помощью --revision ), или если в SVN был скопирован каталог, который не отслеживается git svn (например, ветка, которая вообще не отслеживается, или подкаталог отслеживаемой ветки). В этих случаях git svn по-прежнему создаст ветку Git , но вместо использования существующего коммита Git в качестве родительского элемента ветки он прочитает историю SVN каталога, из которого была скопирована ветвь, и создаст соответствующие коммиты Git . Об этом свидетельствует сообщение "Initializing parent: <branchname>" .

Кроме того, будет создана специальная ветка с именем <branchname>@<SVN-Revision> , где <SVN-Revision> — это номер версии SVN, из которой была скопирована ветка. Эта ветка будет указывать на недавно созданный родительский коммит ветки. Если в SVN ветка была удалена, а затем воссоздана из другой версии, таких веток с @ будет несколько.

Обратите внимание, что это может означать, что для одной версии SVN создается несколько коммитов Git .

Пример: в репозитории SVN со стандартной структурой «транк/теги/ветви» в r.100 создается каталог «транк/подразделитель». В версии r.200 транковый/подразделительный узел разветвляется путем копирования его в ветки/. git svn clone -s затем создаст ветку sub . Он также создаст новые коммиты Git для версий с 100 по 199 и будет использовать их в качестве истории ветки sub . Таким образом, для каждой ревизии от версии 100 до версии 199 будет два коммита Git (один содержит Trunk/, другой содержит Trunk/Sub/). Наконец, будет создана ветка sub@200 , указывающая на новый родительский коммит ветки sub (т.е. коммит для r.200 и Trunk/sub/).

Caveats

Для простоты и взаимодействия с Subversion всем пользователям git svn рекомендуется клонировать, извлекать и выполнять dcommit непосредственно с сервера SVN и избегать всех операций git clone / pull / merge / push между репозиториями и ветвями Git . Рекомендуемый метод обмена кодом между ветвями Git и пользователями — git format-patch и git am или просто «dcommit» в репозиторий SVN.

НЕ рекомендуется запускать git merge или git pull в ветке, из которой вы планируете использовать dcommit , поскольку пользователи Subversion не смогут видеть сделанные вами слияния. Более того, если вы объединяете или извлекаете из ветки Git , которая является зеркалом ветки SVN, dcommit может зафиксировать неправильную ветку.

Если вы выполняете слияние, обратите внимание на следующее правило: git svn dcommit попытается выполнить фиксацию поверх фиксации SVN, указанной в

git log --grep=^git-svn-id: --first-parent -1

Поэтому вы, must , гарантируете, что самая последняя фиксация ветки, в которую вы хотите выполнить dcommit, является родительской first для слияния. В противном случае наступит хаос, особенно если первый родительский элемент является более старым коммитом в той же ветке SVN.

git clone не клонирует ветки в иерархии refs/remotes/ или каких-либо метаданных или конфигурации git svn . Таким образом, репозитории, созданные и управляемые с помощью git svn , должны использовать rsync для клонирования, если клонирование вообще должно выполняться.

Поскольку dcommit использует внутреннюю перебазировку, любые ветки Git , к которым вы используете git push до dcommit , потребуют принудительной перезаписи существующей ссылки в удаленном репозитории. Обычно это считается плохой практикой, подробности см. в документации git-push[1] .

Не используйте опцию --amend git-commit[1] для изменений, которые вы уже зафиксировали. Считается плохой практикой --amend коммиты, которые вы уже отправили в удаленный репозиторий для других пользователей, и dcommit с SVN аналогичен этому.

При клонировании репозитория SVN, если используются none из опций описания структуры репозитория (--trunk, --tags, --branches, --stdlayout), git svn clone создаст репозиторий Git с полностью линейной историей, где ветки и теги отображаются как отдельные каталоги в рабочей копии. Хотя это самый простой способ получить копию полного репозитория, для проектов со многими ветками это приведет к тому, что рабочая копия будет во много раз больше, чем просто ствол. Таким образом, для проектов, использующих стандартную структуру каталогов (магистраль/ветви/теги), рекомендуется клонировать с опцией --stdlayout . Если в проекте используется нестандартная структура и/или если ветки и теги не требуются, проще всего клонировать только один каталог (обычно магистральный), не задавая никаких параметров макета репозитория. Если требуется полная история с ветвями и тегами, необходимо использовать опции --trunk / --branches / --tags .

При использовании нескольких --branches или --tags git svn не обрабатывает автоматически конфликты имен (например, если две ветви из разных путей имеют одинаковое имя или если ветвь и тег имеют одинаковое имя). В этих случаях используйте init для настройки репозитория Git , а затем перед первым fetch отредактируйте файл $GIT_DIR/config так, чтобы ветки и теги были связаны с разными пространствами имен. Например:

branches = stable/*:refs/remotes/svn/stable/*
branches = debug/*:refs/remotes/svn/debug/*

Configuration

git svn хранит информацию о конфигурации [svn-remote] в файле репозитория $GIT_DIR/config. Это похоже на основные [удаленные] разделы Git , за исключением того, что ключи fetch не принимают аргументы glob; но вместо этого они обрабатываются клавишами branches и tags . Поскольку некоторые репозитории SVN странно настроены с использованием нескольких расширений glob проектов, такие перечисленные ниже разрешены:

[svn-remote "project-a"]
        url = http://server.org/svn
        fetch = trunk/project-a:refs/remotes/project-a/trunk
        branches = branches/*/project-a:refs/remotes/project-a/branches/*
        branches = branches/release_*:refs/remotes/project-a/branches/release_*
        branches = branches/re*se:refs/remotes/project-a/branches/*
        tags = tags/*/project-a:refs/remotes/project-a/tags/*

Имейте в виду, что подстановочный знак * (звездочка) локальной ссылки (справа от : ) должен быть самым дальним компонентом правого пути; однако удаленный подстановочный знак может находиться где угодно, если он является независимым компонентом пути (окруженным / или EOL). Этот тип конфигурации не создается автоматически init и должен вводиться вручную с помощью текстового редактора или с помощью git config .

Также обратите внимание, что в каждом слове допускается только одна звездочка. Например:

branches = branches/re*se:refs/remotes/project-a/branches/*

однако будет соответствовать ветвям release , rese , re123se .

branches = branches/re*s*e:refs/remotes/project-a/branches/*

выдаст ошибку.

Также возможно получить подмножество ветвей или тегов, используя список имен, разделенных запятыми, в фигурных скобках. Например:

[svn-remote "huge-project"]
        url = http://server.org/svn
        fetch = trunk/src:refs/remotes/trunk
        branches = branches/{red,green}/src:refs/remotes/project-a/branches/*
        tags = tags/{1.0,2.0}/src:refs/remotes/project-a/tags/*

Поддерживаются несколько ключей выборки, ветвей и тегов:

[svn-remote "messy-repo"]
        url = http://server.org/svn
        fetch = trunk/project-a:refs/remotes/project-a/trunk
        fetch = branches/demos/june-project-a-demo:refs/remotes/project-a/demos/june-demo
        branches = branches/server/*:refs/remotes/project-a/branches/*
        branches = branches/demos/2011/*:refs/remotes/project-a/2011-demos/*
        tags = tags/server/*:refs/remotes/project-a/tags/*

Создание ветки в такой конфигурации требует устранения неоднозначности, какое местоположение использовать, с помощью флага -d или --destination:

$ git svn branch -d branches/server release-2-3-0

Обратите внимание, что git-svn отслеживает самую старшую ревизию, в которой появилась ветка или тег. Если подмножество ветвей или тегов изменяется после извлечения, то $GIT_DIR/svn/.metadata необходимо вручную отредактировать, чтобы удалить (или сбросить) ветки-maxRev и/или теги-maxRev, если это необходимо.

Files

$GIT_DIR/svn/**/.rev_map.*

Сопоставление номеров ревизий Subversion и имен коммитов Git . В репозитории, где опция noMetadata не установлена, ее можно перестроить из строк git-svn-id:, которые находятся в конце каждого коммита (подробности см. в разделе svn.noMetadata выше).

git svn fetch и git svn rebase автоматически обновляют rev_map, если он отсутствует или устарел. git svn reset автоматически перематывает его.

Bugs

Мы игнорируем все свойства SVN, кроме svn:executable. Любые необработанные свойства записываются в $GIT_DIR/svn/<refname>/unhandled.log.

Переименованные и скопированные каталоги не обнаруживаются Git и, следовательно, не отслеживаются при фиксации в SVN. Я не планирую добавлять эту поддержку, поскольку заставить работать все возможные крайние случаи довольно сложно и отнимает много времени ( Git тоже этого не делает). Фиксация переименованных и скопированных файлов полностью поддерживается, если они достаточно похожи, чтобы Git мог их обнаружить.

В SVN можно (хотя и не рекомендуется) фиксировать изменения в теге (поскольку тег — это просто копия каталога, то есть технически то же самое, что и ветвь). При клонировании репозитория SVN git svn не может знать, произойдет ли такая фиксация тега в будущем. Таким образом, он действует консервативно и импортирует все теги SVN как ветки, добавляя к имени тега префикс tags/ .
