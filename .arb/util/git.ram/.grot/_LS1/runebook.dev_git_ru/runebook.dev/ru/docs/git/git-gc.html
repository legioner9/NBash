git-gc

Name

git-gc — очистить ненужные файлы и оптимизировать локальный репозиторий.

Synopsis

git gc [--aggressive] [--auto] [--quiet] [--prune=<date> | --no-prune] [--force] [--keep-largest-pack]

Description

Выполняет ряд служебных задач в текущем репозитории, таких как сжатие ревизий файлов (для уменьшения дискового пространства и повышения производительности), удаление недоступных объектов, которые могли быть созданы в результате предыдущих вызовов git add , упаковка ссылок, сокращение журналов обновлений, повторное повторение метаданных или устаревших. рабочие деревья. Также может обновляться вспомогательные индексы, такие как граф фиксации.

При выполнении обычных фарфоровых операций по созданию объектов они проверяют, значительно ли увеличился репозиторий с момента последнего обслуживания, и если да, автоматически запускают git gc . См. gc.auto ниже, чтобы узнать, как отключить это поведение.

Запуск git gc вручную необходим только при добавлении объектов в репозиторий без регулярного запуска таких фарфоровых команд, для выполнения разовой оптимизации репозитория или, например, для очистки неоптимального массового импорта. Дополнительную информацию об импорте см. в разделе "PACKFILE OPTIMIZATION" в git-fast-import[1] .

Options

--aggressive

Обычно git gc работает очень быстро, обеспечивая при этом хорошее использование дискового пространства и производительность. Этот параметр заставит git gc более агрессивно оптимизировать репозиторий, но это потребует гораздо больше времени. Эффекты этой оптимизации в основном постоянны. Подробности см. в разделе "AGGRESSIVE" ниже.

--auto

С помощью этой опции git gc проверяет, требуется ли какая-либо очистка; в противном случае он завершает работу, не выполняя никакой работы.

См. опцию gc.auto в разделе "CONFIGURATION" ниже, чтобы узнать, как работает эта эвристика.

Как только служебное обслуживание запускается при превышении ограничений параметров конфигурации, таких как gc.auto и gc.autoPackLimit , все остальные служебные задачи (например, rerere, рабочие деревья, повторный журнал…​) также будут выполняться.

--[no-]cruft

Недоступные предметы с истекающим сроком годности упаковывайте их отдельно в пакет для мусора, а не храните как незакрепленные предметы. --cruft включен по умолчанию.

--max-cruft-size=<n>

При упаковке недоступных объектов в пакет Cruft ограничьте размер новых пакетов не более <n> байтами. Переопределяет любое значение, указанное в конфигурации gc.maxCruftSize . Дополнительную информацию см. в опции --max-cruft-size для git-repack[1] .

--prune=<date>

Удалить незакрепленные объекты старше даты (по умолчанию — 2 недели назад, это можно переопределить с помощью переменной конфигурации gc.pruneExpire ). --prune=now удаляет незакрепленные объекты независимо от их возраста и увеличивает риск повреждения, если другой процесс одновременно записывает в репозиторий; см. "NOTES" ниже. --prune включен по умолчанию.

--no-prune

Не обрезайте незакрепленные предметы.

--quiet

Подавить все отчеты о ходе работы.

--force

Принудительно запустить git gc , даже если в этом репозитории может быть запущен другой экземпляр git gc .

--keep-largest-pack

Все упаковки, за исключением самой большой упаковки, не являющейся «крафтовой», любых упаковок, помеченных файлом .keep , а также любых упаковок «крафт», объединены в одну упаковку. При использовании этой опции gc.bigPackThreshold игнорируется.

Aggressive

Когда указана опция --aggressive , git-repack[1] будет вызван с флагом -f , который, в свою очередь, передаст --no-reuse-delta в git-pack-objects[1] . Это позволит отбросить все существующие дельты и вычислить их заново, но на переупаковку потратится гораздо больше времени.

Эффекты этого в основном постоянны, например, когда пакеты и отдельные объекты объединяются друг в друга, существующие дельты в этом пакете могут быть повторно использованы, но есть также различные случаи, когда мы можем выбрать неоптимальную дельту из более нового пакета. вместо этого упакуйте.

Кроме того, поставка --aggressive позволит усовершенствовать параметры --depth и --window , переданные в git-repack[1] . См. настройки gc.aggressiveDepth и gc.aggressiveWindow ниже. Используя больший размер окна, мы с большей вероятностью найдем более оптимальные дельты.

Вероятно, не стоит использовать эту опцию в конкретном репозитории, не запустив для него специальные тесты производительности. Это занимает гораздо больше времени, и результирующая оптимизация пространства/дельты может того стоить, а может и не стоить. Не использовать это вообще — правильный компромисс для большинства пользователей и их репозиториев.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

gc.aggressiveDepth

Параметр глубины, используемый в алгоритме дельта-сжатия, используемом git gc --aggressive . По умолчанию это значение 50, которое является значением по умолчанию для опции --depth , когда --aggressive не используется.

Более подробную информацию см. в документации к опции --depth в git-repack[1] .

gc.aggressiveWindow

Параметр размера окна, используемый в алгоритме дельта-сжатия, используемом git gc --aggressive . По умолчанию это значение равно 250, что является гораздо более агрессивным размером окна, чем значение по умолчанию --window , равное 10.

Более подробную информацию см. в документации к опции --window в git-repack[1] .

gc.auto

Когда в репозитории окажется примерно больше этого количества незакрепленных объектов, git gc --auto упакует их. Некоторые команды Porcelain время от времени используют эту команду для облегченной сборки мусора. Значение по умолчанию — 6700.

Установка значения 0 отключает не только автоматическую упаковку на основе количества незакрепленных объектов, но и любую другую эвристику, которую git gc --auto в противном случае будет использовать для определения наличия работы, например gc.autoPackLimit .

gc.autoPackLimit

Если в репозитории имеется больше пакетов, не помеченных файлом *.keep , git gc --auto объединяет их в один больший пакет. Значение по умолчанию — 50. Установка значения 0 отключает его. Установка gc.auto на 0 также отключит это.

См. переменную конфигурации gc.bigPackThreshold ниже. При использовании это повлияет на работу лимита автоупаковки.

gc.autoDetach

Заставьте git gc --auto немедленно вернуться и запустить его в фоновом режиме, если система это поддерживает. По умолчанию верно.

gc.bigPackThreshold

Если значение не равно нулю, все упаковки, не содержащие мусора, размер которых превышает этот предел, сохраняются при запуске git gc . Это очень похоже на --keep-largest-pack , за исключением того, что сохраняются все не крафтовые упаковки, соответствующие пороговому значению, а не только самая большая упаковка. По умолчанию равно нулю. Поддерживаются общие суффиксы единиц измерения k , m или g .

Обратите внимание: если количество сохраняемых пакетов превышает gc.autoPackLimit, эта переменная конфигурации игнорируется, все пакеты, кроме базового, будут переупакованы. После этого количество пакетов должно упасть ниже gc.autoPackLimit и gc.bigPackThreshold снова должно соблюдаться.

Если объем памяти, рассчитанный для бесперебойной работы git repack , недоступен и gc.bigPackThreshold не установлен, самый большой пакет также будет исключен (это эквивалент запуска git gc с --keep-largest-pack ).

gc.writeCommitGraph

Если это правда, то gc перезапишет файл графа фиксации при запуске git-gc[1] . При использовании git gc --auto граф фиксации будет обновляться, если потребуется служебное обслуживание. По умолчанию верно. Подробности см. в git-commit-graph[1] .

gc.logExpiry

Если файл gc.log существует, то git gc --auto распечатает его содержимое и завершит работу с нулевым статусом вместо запуска, если только этот файл не старше gc.logExpiry . По умолчанию — "1.day" . См. gc.pruneExpire , чтобы узнать больше о способах указания его значения.

gc.packRefs

Запуск git pack-refs в репозитории делает его невозможным для клонирования версий Git до 1.5.1.2 через простые транспортные средства, такие как HTTP. Эта переменная определяет, запускает ли git gc git pack-refs . Для этого параметра можно установить значение notbare , чтобы включить его во всех непустых репозиториях, или можно установить значение boolean . По умолчанию — true .

gc.cruftPacks

Храните недоступные предметы в упаковке (см. git-repack[1] ), а не как незакрепленные предметы. По умолчанию — true .

gc.maxCruftSize

Ограничьте размер новых крафт-упаковок при переупаковке. Если указан в дополнение к --max-cruft-size , параметр командной строки имеет приоритет. См. вариант --max-cruft-size для git-repack[1] .

gc.pruneExpire

При запуске git gc он вызывает prune --expire 2.weeks.ago (и repack --cruft --cruft-expiration 2.weeks.ago при использовании пакетов Cruft через gc.cruftPacks или --cruft ). Переопределите льготный период с помощью этой переменной конфигурации. Значение "now" можно использовать для отключения этого льготного периода и всегда немедленно удалять недоступные объекты, либо "never" можно использовать для подавления сокращения. Эта функция помогает предотвратить повреждение, когда git gc выполняется одновременно с другим процессом, записывающим в репозиторий; см. раздел "NOTES" git-gc[1] .

gc.worktreePruneExpire

Когда git gc запускается, он вызывает git worktree prune --expire 3.months.ago . Эту переменную конфигурации можно использовать для установки другого льготного периода. Значение "now" можно использовать для отключения льготного периода и немедленного сокращения $GIT_DIR/worktrees , либо "never" можно использовать для подавления сокращения.

gc.reflogExpire
gc.<pattern>.reflogExpire

git reflog expire удаляет записи журнала обновлений старше этого времени; по умолчанию 90 дней. Значение "now" немедленно уничтожает все записи, а "never" полностью подавляет срок действия. Если "<pattern>" (например, "refs/stash" ) находится посередине, настройка применяется только к ссылкам, соответствующим <шаблону>.

gc.reflogExpireUnreachable
gc.<pattern>.reflogExpireUnreachable

git reflog expire удаляет записи журнала обновлений старше этого времени и недоступны из текущей подсказки; по умолчанию 30 дней. Значение "now" немедленно уничтожает все записи, а "never" полностью подавляет срок действия. Если "<pattern>" (например, "refs/stash" ) посередине, настройка применяется только к ссылкам, соответствующим <шаблону>.

Эти типы записей обычно создаются в результате использования git commit --amend или git rebase и представляют собой фиксации до внесения изменений или перебазирования. Поскольку эти изменения не являются частью текущего проекта, большинство пользователей захотят прекратить их действие раньше, поэтому значение по умолчанию более агрессивно, чем gc.reflogExpire .

gc.recentObjectsHook

При рассмотрении вопроса об удалении объекта (либо при создании пакета мусора, либо при сохранении недоступных объектов как незакрепленных) используйте оболочку для выполнения указанных команд. Интерпретируйте их вывод как идентификаторы объектов, которые Git будет считать "recent" , независимо от их возраста. Если их mtime рассматривать как "now" , любые объекты (и их потомки), упомянутые в выходных данных, будут сохранены независимо от их возраста true .

Вывод должен содержать ровно один идентификатор шестнадцатеричного объекта в каждой строке и ничего больше. Объекты, которые невозможно найти в репозитории, игнорируются. Поддерживается несколько перехватчиков, но все они должны завершиться успешно, иначе операция (либо создание пакета мусора, либо распаковка недоступных объектов) будет остановлена.

gc.repackFilter

При переупаковке используйте указанный фильтр для перемещения определенных объектов в отдельный паковочный файл. См. вариант --filter=<filter-spec> для git-repack[1] .

gc.repackFilterTo

При переупаковке и использовании фильтра, см. gc.repackFilter , указанное местоположение будет использоваться для создания паковочного файла, содержащего отфильтрованные объекты. ВНИМАНИЕ: Указанное местоположение должно быть доступно, например, с использованием альтернативного механизма Git , в противном случае Git может счесть репозиторий поврежденным, поскольку он не сможет получить доступ к объектам в этом пакетном файле. См. опцию --filter-to=<dir> для git-repack[1] и раздел objects/info/alternates для gitrepository-layout[5] .

gc.rerereResolved

Записи о конфликтных слияниях, которые вы разрешили ранее, хранятся в течение указанного количества дней при запуске git rerere gc . Вы также можете использовать более удобочитаемый "1.month.ago" и т. д. Значение по умолчанию — 60 дней. См. git-rerere[1] .

gc.rerereUnresolved

Записи о конфликтных слияниях, которые вы не разрешили, хранятся в течение указанного количества дней при запуске git rerere gc . Вы также можете использовать более удобочитаемый "1.month.ago" и т. д. Значение по умолчанию — 15 дней. См. git-rerere[1] .

Notes

git gc очень старается не удалять объекты, на которые есть ссылки где-либо в вашем репозитории. В частности, он будет хранить не только объекты, на которые ссылается ваш текущий набор ветвей и тегов, но также объекты, на которые ссылается индекс, ветки удаленного отслеживания, журналы обновлений (которые могут ссылаться на коммиты в ветвях, которые были позже изменены или перемотаны) и все остальное. else в пространстве имен refs/*. Обратите внимание, что заметка (например, созданная git notes ), прикрепленная к объекту, не способствует поддержанию активности объекта. Если вы ожидаете, что некоторые объекты будут удалены, а это не так, проверьте все эти места и решите, имеет ли смысл в вашем случае удалять эти ссылки.

С другой стороны, когда git gc работает одновременно с другим процессом, существует риск удаления объекта, который другой процесс использует, но не создал ссылку на него. Это может привести к сбою другого процесса или повреждению репозитория, если другой процесс позже добавит ссылку на удаленный объект. Git имеет две особенности, которые существенно смягчают эту проблему:

  1. Любой объект, время модификации которого позже даты --prune , сохраняется вместе со всем, что можно из него получить.

  2. Большинство операций, добавляющих объект в базу данных, обновляют время модификации объекта, если он уже присутствует, так что применяется № 1.

Однако эти функции не являются полноценным решением, поэтому пользователям, одновременно запускающим команды, приходится сталкиваться с некоторым риском повреждения (который на практике кажется низким).
