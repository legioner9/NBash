gitformat-chunk

Name

gitformat-chunk — форматы файлов на основе фрагментов

Synopsis

Используется форматом gitformat-commit-graph[5] и "MIDX" (см. документацию по формату пакета в gitformat-pack[5] ).

Description

Некоторые форматы файлов в Git используют общую концепцию "chunks" для описания разделов файла. Это обеспечивает структурированный доступ к большому файлу путем сканирования небольшого "table of contents" на наличие оставшихся данных. Этот общий формат используется файлами commit-graph и multi-pack-index . См. формат multi-pack-index в gitformat-pack[5] и формат commit-graph в gitformat-commit-graph[5] , чтобы узнать, как они используют фрагменты для описания структурированных данных.

Формат файла на основе фрагментов начинается с некоторой информации заголовка, адаптированной к этому формату. Этот заголовок должен содержать достаточно информации для идентификации типа файла, версии формата и количества фрагментов в файле. На основе этой информации этот файл может определить начало области на основе фрагментов.

Область на основе фрагментов начинается с оглавления, описывающего, где начинается и заканчивается каждый фрагмент. Он состоит из (C+1) строк по 12 байт каждая, где C — количество фрагментов. Рассмотрим следующую таблицу:

| Chunk ID (4 bytes) | Chunk Offset (8 bytes) |
|--------------------|------------------------|
| ID[0]              | OFFSET[0]              |
| ...                | ...                    |
| ID[C]              | OFFSET[C]              |
| 0x0000             | OFFSET[C+1]            |

Каждая строка состоит из 4-байтового идентификатора фрагмента (ID) и 8-байтового смещения. Каждое целое число хранится в сетевом порядке байтов.

Идентификатор фрагмента ID[i] — это метка данных, хранящихся в этом файле, от OFFSET[i] (включительно) до OFFSET[i+1] (эксклюзивно). Таким образом, размеры i`th chunk is equal to the difference between `OFFSET[i+1] и OFFSET[i] . Для этого необходимо, чтобы данные фрагментов располагались непрерывно в том же порядке, что и таблица содержания.

Последняя запись в оглавлении должна содержать четыре нулевых байта. Это подтверждает, что оглавление заканчивается, и обеспечивает смещение конца данных на основе фрагментов.

Примечание. Формат на основе фрагментов предполагает, что файл содержит at least и конечный хэш после OFFSET[C+1] .

Функции для работы с форматами файлов на основе фрагментов объявлены в chunk-format.h . Использование этих методов обеспечивает дополнительные проверки, которые помогают разработчикам при создании новых форматов файлов.

Написание форматов файлов на основе фрагментов

Чтобы записать формат файла на основе фрагментов, создайте struct chunkfile , вызвав init_chunkfile() и передав указатель struct hashfile . Вызывающая сторона отвечает за открытие hashfile и запись информации заголовка, чтобы формат файла можно было идентифицировать до того, как начнется форматирование на основе фрагментов.

Затем вызовите add_chunk() для каждого фрагмента, предназначенного для записи. При этом chunkfile заполняется информацией о порядке и размере каждого записываемого фрагмента. Предоставьте указатель функции chunk_write_fn для выполнения записи данных фрагмента по запросу.

Вызовите write_chunkfile() , чтобы записать оглавление в hashfile , за которым следуют все фрагменты. Это позволит убедиться, что каждый фрагмент записал ожидаемый объем данных, поэтому оглавление правильное.

Наконец, вызовите free_chunkfile() , чтобы очистить данные struct chunkfile . Вызывающая сторона отвечает за завершение hashfile путем записи конечного хеша и закрытия файла.

Чтение форматов файлов на основе фрагментов

Чтобы прочитать формат файла на основе фрагментов, файл должен быть открыт как область, отображенная в памяти. API формата фрагментов ожидает, что весь файл отображается как непрерывная область памяти.

Инициализируйте указатель struct chunkfile с помощью init_chunkfile(NULL) .

После чтения информации заголовка из начала файла, включая количество фрагментов, вызовите read_table_of_contents() , чтобы заполнить struct chunkfile списком фрагментов, их смещениями и размерами.

Извлеките информацию о данных для каждого фрагмента, используя pair_chunk() или read_chunk() :

  • pair_chunk() назначает заданному указателю местоположение внутри отображенного в памяти файла, соответствующее смещению этого фрагмента. Если чанк не существует, указатель не изменяется.

  • read_chunk() принимает указатель на функцию chunk_read_fn и вызывает его с соответствующим начальным указателем и информацией о размере. Функция не вызывается, если чанк не существует. Используйте этот метод для чтения фрагментов, если вам нужно выполнить немедленный анализ или если вам нужно выполнить логику в зависимости от размера фрагмента.

После вызова этих методов вызовите free_chunkfile() , чтобы очистить данные struct chunkfile . Это не закроет область, отображенную в памяти. Ожидается, что вызывающие объекты будут владеть этими данными за период времени, когда необходимы указатели на регион.

Examples

Эти форматы файлов используют API формата фрагментов и могут использоваться в качестве примеров для будущих форматов:

  • commit-graph: см. write_commit_graph_file() и parse_commit_graph() в commit-graph.c , чтобы узнать, как API формата фрагментов используется для записи и анализа формата файла графика фиксации, документированного в формате файла графика фиксации в gitformat-commit-graph[5] .

  • multi-pack-index: см. write_midx_internal() и load_multi_pack_index() в midx.c , чтобы узнать, как API формата фрагментов используется для записи и анализа формата файла multi-pack-index, описанного в разделе формата файла multi-pack-index в gitformat-pack[5] .
