git-worktree

Name

git-worktree — Управление несколькими рабочими деревьями

Synopsis

git worktree add [-f] [--detach] [--checkout] [--lock [--reason <string>]]
                   [--orphan] [(-b | -B) <new-branch>] <path> [<commit-ish>]
git worktree list [-v | --porcelain [-z]]
git worktree lock [--reason <string>] <worktree>
git worktree move <worktree> <new-path>
git worktree prune [-n] [-v] [--expire <expire>]
git worktree remove [-f] <worktree>
git worktree repair [<path>…​]
git worktree unlock <worktree>

Description

Управляйте несколькими рабочими деревьями, прикрепленными к одному и тому же репозиторию.

Репозиторий git может поддерживать несколько рабочих деревьев, что позволяет вам извлекать более одной ветки одновременно. В git worktree add с репозиторием связано новое рабочее дерево вместе с дополнительными метаданными, которые отличают это рабочее дерево от других в том же репозитории. Рабочее дерево вместе с этими метаданными называется "worktree" .

Это новое рабочее дерево называется "linked worktree" , в отличие от "main worktree" , созданного git-init[1] или git-clone[1] . Репозиторий имеет одно основное рабочее дерево (если это не пустой репозиторий) и ноль или более связанных рабочих деревьев. Когда вы закончите работу со связанным рабочим деревом, удалите его с помощью git worktree remove .

В простейшем виде git worktree add <path> автоматически создает новую ветку, имя которой является конечным компонентом <path> , что удобно, если вы планируете работать над новой темой. Например, git worktree add ../hotfix создает новую ветвь hotfix и проверяет ее по пути ../hotfix . Чтобы вместо этого работать с существующей ветвью в новом рабочем дереве, используйте git worktree add <path> <branch> . С другой стороны, если вы просто планируете внести некоторые экспериментальные изменения или провести тестирование, не нарушая существующую разработку, часто бывает удобно создать рабочее дерево throwaway , не связанное ни с какой ветвью. Например, git worktree add -d <path> создает новое рабочее дерево с отключенным HEAD при той же фиксации, что и текущая ветка.

Если рабочее дерево удалено без использования git worktree remove , то связанные с ним административные файлы, находящиеся в репозитории (см. "DETAILS" ниже), в конечном итоге будут удалены автоматически (см. gc.worktreePruneExpire в git-config[1] ), либо вы можете запустить git worktree prune в основном или любом связанном worktree для очистки устаревших административных файлов.

Если рабочее дерево связанного рабочего дерева хранится на портативном устройстве или в общей сетевой папке, которая не всегда подключена, вы можете предотвратить удаление его административных файлов, введя команду git worktree lock , дополнительно указав --reason , чтобы объяснить, почему рабочее дерево заблокировано.

Commands

добавить <путь> [<фиксация>]

Создайте рабочее дерево <path> и извлеките в него <commit-ish> . Новое рабочее дерево связано с текущим репозиторием, в котором используется все, кроме файлов каждого рабочего дерева, таких как HEAD , index и т. д. Для удобства <commit-ish> может быть простым " - " , что является синонимом @{-1} .

Если <commit-ish> — это имя ветки (назовем ее <branch> ) и не найдено, и ни -b , ни -B , ни --detach не используются, но существует ветвь отслеживания ровно на одном удаленном компьютере (назовем ее <remote> ) с совпадающим именем, рассматривайте как эквивалент к:

$ git worktree add --track -b <branch> <path> <remote>/<branch>

Если ветвь существует на нескольких удаленных устройствах и один из них назван конфигурационной переменной checkout.defaultRemote , мы будем использовать ее в целях устранения неоднозначности, даже если <branch> не уникален для всех удаленных устройств. Установите его, например, на checkout.defaultRemote=origin , чтобы всегда получать оттуда удаленные ветки, если <branch> неоднозначен, но существует на удаленном origin . См. также checkout.defaultRemote в git-config[1] .

Если <commit-ish> опущен и не используются ни -b , ни -B , ни --detach , то для удобства новое рабочее дерево связано с ветвью (назовем ее <branch> ), названной в честь $(basename <path>) . Если <branch> не существует, автоматически создается новая ветвь на основе HEAD , как если бы был задан -b <branch> . Если <branch> существует, он будет извлечен в новом рабочем дереве, если он не извлечен где-либо еще, в противном случае команда откажется создавать рабочее дерево (если не используется --force ).

Если <commit-ish> опущен, ни --detach , ни --orphan не используются, и нет допустимых локальных ветвей (или удаленных ветвей, если указан --guess-remote ), то для удобства новое рабочее дерево связывается с новой нерожденной ветвью с именем <branch> (после $(basename <path>) ). если ни -b , ни -B не используются), как если бы в команду был передан --orphan . Если у репозитория есть удаленный доступ и используется --guess-remote , но удаленных или локальных ветвей не существует, команда завершается с ошибкой с предупреждением, напоминающим пользователю сначала получить данные со своего удаленного компьютера (или переопределить с помощью -f/--force ).

list

Перечислите детали каждого рабочего дерева. Сначала указывается основное рабочее дерево, а затем каждое из связанных рабочих деревьев. Выходные данные включают в себя информацию о том, является ли рабочее дерево пустым, текущая извлеченная ревизия, извлеченная в данный момент ветвь (или "detached HEAD" , если ее нет), "locked" , если рабочее дерево заблокировано, "prunable" , если рабочее дерево может быть сокращено командой prune .

lock

Если рабочее дерево находится на переносном устройстве или в сетевой папке, которая не всегда подключена, заблокируйте его, чтобы предотвратить автоматическое удаление административных файлов. Это также предотвращает его перемещение или удаление. При желании укажите причину блокировки с помощью --reason .

move

Переместите рабочее дерево в новое место. Обратите внимание, что основное рабочее дерево или связанные рабочие деревья, содержащие подмодули, невозможно переместить с помощью этой команды. (Однако команда git worktree repair может восстановить соединение со связанными рабочими деревьями, если вы переместите основное рабочее дерево вручную.)

prune

Сократите информацию о рабочем дереве в $GIT_DIR/worktrees .

remove

Удалить рабочее дерево. Удалить можно только чистые рабочие деревья (без неотслеживаемых файлов и изменений в отслеживаемых файлах). Неочищенные рабочие деревья или деревья с подмодулями можно удалить с помощью --force . Основное рабочее дерево удалить невозможно.

repair [<path>…​]

Если возможно, восстановите административные файлы рабочего дерева, если они были повреждены или устарели из-за внешних факторов.

Например, если основное рабочее дерево (или пустой репозиторий) будет перемещено, связанные рабочие деревья не смогут его найти. Запуск repair в основном рабочем дереве восстановит соединение связанных рабочих деревьев с основным рабочим деревом.

Аналогично, если рабочее дерево связанного рабочего дерева перемещается без использования git worktree move , основное рабочее дерево (или пустой репозиторий) не сможет его найти. Запуск repair в недавно перемещенном рабочем дереве восстановит соединение. Если перемещено несколько связанных рабочих деревьев, запуск repair из любого рабочего дерева с новым <path> каждого дерева в качестве аргумента восстановит соединение со всеми указанными путями.

Если и основное рабочее дерево, и связанные рабочие деревья были перемещены вручную, то запуск repair в главном рабочем дереве и указание нового <path> для каждого связанного рабочего дерева восстановит все соединения в обоих направлениях.

unlock

Разблокируйте рабочее дерево, чтобы его можно было обрезать, перемещать или удалять.

Options

-f
--force

По умолчанию add отказывается создавать новое рабочее дерево, если <commit-ish> является именем ветки и уже извлечено другим рабочим деревом, или если <path> уже назначен какому-либо рабочему дереву, но отсутствует (например, если <path> был удален вручную). Эта опция отменяет эти меры безопасности. Чтобы добавить отсутствующий, но заблокированный путь к рабочему дереву, дважды укажите --force .

move отказывается перемещать заблокированное рабочее дерево, если --force не указан дважды. Если пункт назначения уже назначен какому-либо другому рабочему дереву, но отсутствует (например, если <new-path> был удален вручную), то --force позволяет продолжить перемещение; используйте --force дважды, если пункт назначения заблокирован.

remove отказывается удалять нечистое рабочее дерево, если не используется --force . Чтобы удалить заблокированное рабочее дерево, дважды укажите --force .

-b <new-branch>
-B <new-branch>

С помощью add создайте новую ветвь с именем <new-branch> , начиная с <commit-ish> , и извлеките <new-branch> в новое рабочее дерево. Если <commit-ish> опущен, по умолчанию используется HEAD . По умолчанию -b отказывается создавать новую ветку, если она уже существует. -B отменяет эту защиту, сбрасывая <new-branch> на <commit-ish> .

-d
--detach

С помощью add отсоедините HEAD в новом рабочем дереве. См. "DETACHED HEAD" в git-checkout[1] .

--[no-]checkout

По умолчанию add извлекает <commit-ish> , однако --no-checkout можно использовать для подавления извлечения, чтобы выполнить настройки, такие как настройка разреженной проверки. См. "Sparse checkout" в git-read-tree[1] .

--[no-]guess-remote

С worktree add <path> , без <commit-ish> , вместо создания новой ветки из HEAD , если существует ветвь отслеживания ровно на одном удаленном компьютере, соответствующем базовому имени <path> , создайте новую ветвь на основе ветки удаленного отслеживания и пометьте ветвь удаленного отслеживания как "upstream" из новой ветки.

Это также можно настроить как поведение по умолчанию, используя параметр конфигурации worktree.guessRemote .

--[no-]track

При создании новой ветки, если <commit-ish> является веткой, отметьте ее как "upstream" из новой ветки. Это значение по умолчанию, если <commit-ish> является ветвью удаленного отслеживания. Подробности см. в --track и git-branch[1] .

--lock

Держите рабочее дерево заблокированным после создания. Это эквивалент git worktree lock после git worktree add , но без состояния гонки.

-n
--dry-run

С prune ничего не удаляйте; просто сообщите, что он удалит.

--orphan

С помощью add сделайте новое рабочее дерево и индекс пустыми, связав рабочее дерево с новой нерожденной ветвью с именем <new-branch> .

--porcelain

С list выводите данные в удобном для анализа формате для сценариев. Этот формат останется стабильным во всех версиях Git и независимо от конфигурации пользователя. Рекомендуется комбинировать это с -z . Подробности см. ниже.

-z

Завершите каждую строку NUL, а не новой строкой, если --porcelain указан с помощью list . Это позволяет анализировать выходные данные, когда путь рабочего дерева содержит символ новой строки.

-q
--quiet

С помощью add подавляйте сообщения обратной связи.

-v
--verbose

С помощью prune сообщайте обо всех удалениях.

С помощью list выведите дополнительную информацию о рабочих деревьях (см. ниже).

--expire <time>

При использовании prune истекает срок действия только неиспользуемых рабочих деревьев старше <time> .

С помощью list помечайте отсутствующие рабочие деревья как сокращаемые, если они старше <time> .

--reason <string>

Для lock или add --lock — объяснение, почему рабочее дерево заблокировано.

<worktree>

Рабочие деревья можно идентифицировать по пути, относительному или абсолютному.

Если последние компоненты пути в пути рабочего дерева уникальны среди рабочих деревьев, его можно использовать для идентификации рабочего дерева. Например, если у вас есть только два рабочих дерева: /abc/def/ghi и /abc/def/ggg , то ghi или def/ghi достаточно, чтобы указать на предыдущее рабочее дерево.

Refs

При использовании нескольких рабочих деревьев некоторые ссылки являются общими для всех рабочих деревьев, а другие относятся только к отдельному рабочему дереву. Одним из примеров является HEAD , который различен для каждого рабочего дерева. В этом разделе рассказывается о правилах общего доступа и о том, как получить доступ к ссылкам одного рабочего дерева из другого.

Как правило, все псевдоссылки относятся к каждому рабочему дереву, а все ссылки, начинающиеся с refs/ , являются общими. Псевдоссылки — это такие, как HEAD , которые находятся непосредственно под $GIT_DIR , а не внутри $GIT_DIR/refs . Однако есть исключения: ссылки внутри refs/bisect , refs/worktree и refs/rewritten не являются общими.

Доступ к ссылкам, относящимся к каждому рабочему дереву, по-прежнему можно получить из другого рабочего дерева по двум специальным путям: main-worktree и worktrees . Первый предоставляет доступ к ссылкам на каждое рабочее дерево основного рабочего дерева, а второй — ко всем связанным рабочим деревьям.

Например, main-worktree/HEAD или main-worktree/refs/bisect/good разрешаются в то же значение, что и HEAD и refs/bisect/good основного рабочего дерева соответственно. Аналогично, worktrees/foo/HEAD или worktrees/bar/refs/bisect/bad аналогичны $GIT_COMMON_DIR/worktrees/foo/HEAD и $GIT_COMMON_DIR/worktrees/bar/refs/bisect/bad .

Чтобы получить доступ к ссылкам, лучше не заглядывать внутрь $GIT_DIR напрямую. Вместо этого используйте такие команды, как git-rev-parse[1] или git-update-ref[1] , которые будут правильно обрабатывать ссылки.

Configuration file

По умолчанию файл репозитория config является общим для всех рабочих деревьев. Если переменные конфигурации core.bare или core.worktree присутствуют в общем файле конфигурации и extensions.worktreeConfig отключен, то они будут применяться только к основному рабочему дереву.

Чтобы иметь конфигурацию, специфичную для рабочего дерева, вы можете включить расширение worktreeConfig , например:

$ git config extensions.worktreeConfig true

В этом режиме конкретная конфигурация остается на пути, указанном git rev-parse --git-path config.worktree . Вы можете добавить или обновить конфигурацию в этом файле с помощью git config --worktree . Старые версии Git откажутся от доступа к репозиториям с этим расширением.

Обратите внимание, что в этом файле исключено исключение для core.bare и core.worktree . Если они существуют в $GIT_DIR/config , необходимо переместить их в config.worktree основного рабочего дерева. Вы также можете воспользоваться этой возможностью, чтобы просмотреть и переместить другую конфигурацию, которую вы не хотите использовать во всех рабочих деревьях:

  • core.worktree никогда не следует использовать совместно.

  • core.bare не следует использовать совместно, если значение равно core.bare=true .

  • core.sparseCheckout не следует использовать совместно, если только вы не уверены, что всегда используете разреженную проверку для всех рабочих деревьев.

Более подробную информацию см. в документации extensions.worktreeConfig в git-config[1] .

Details

Каждое связанное рабочее дерево имеет подкаталог private в каталоге $GIT_DIR/worktrees репозитория. Имя подкаталога private обычно представляет собой базовое имя пути связанного рабочего дерева, к которому, возможно, добавляется число, чтобы сделать его уникальным. Например, когда $GIT_DIR=/path/main/.git , команда git worktree add /path/other/test-next next создает связанное рабочее дерево в /path/other/test-next , а также создает каталог $GIT_DIR/worktrees/test-next (или $GIT_DIR/worktrees/test-next1 , если test-next уже занят).

В связанном рабочем дереве $GIT_DIR настроен так, чтобы указывать на этот каталог private (например, /path/main/.git/worktrees/test-next в примере), а $GIT_COMMON_DIR настроен так, чтобы указывать обратно на $GIT_DIR основного рабочего дерева (например, /path/main/.git ). Эти настройки выполняются в файле .git , расположенном в верхнем каталоге связанного рабочего дерева.

Разрешение пути через git rev-parse --git-path использует либо $GIT_DIR , либо $GIT_COMMON_DIR в зависимости от пути. Например, в связанном рабочем дереве git rev-parse --git-path HEAD возвращает /path/main/.git/worktrees/test-next/HEAD (а не /path/other/test-next/.git/HEAD или /path/main/.git/HEAD ), тогда как git rev-parse --git-path refs/heads/master использует $GIT_COMMON_DIR и возвращает /path/main/.git/refs/heads/master , поскольку ссылки являются общими для всех рабочих деревьев, кроме refs/bisect , refs/worktree и refs/rewritten .

См. gitrepository-layout[5] для получения дополнительной информации. Эмпирическое правило: не делайте никаких предположений о том, принадлежит ли путь $GIT_DIR или $GIT_COMMON_DIR , когда вам нужно напрямую получить доступ к чему-то внутри $GIT_DIR . Используйте git rev-parse --git-path , чтобы получить окончательный путь.

Если вы вручную перемещаете связанное рабочее дерево, вам необходимо обновить файл gitdir в каталоге записи. Например, если связанное рабочее дерево перемещается в /newpath/test-next и его файл .git указывает на /path/main/.git/worktrees/test-next , то вместо этого обновите /path/main/.git/worktrees/test-next/gitdir , чтобы он ссылался на /newpath/test-next . А еще лучше запустить git worktree repair , чтобы соединение автоматически восстановилось.

Чтобы предотвратить удаление записи $GIT_DIR/worktrees (что может быть полезно в некоторых ситуациях, например, когда рабочее дерево записи хранится на портативном устройстве), используйте команду git worktree lock , которая добавляет файл с именем locked в каталог записи. Файл содержит причину в виде обычного текста. Например, если файл .git связанного рабочего дерева указывает на /path/main/.git/worktrees/test-next , то файл с именем /path/main/.git/worktrees/test-next/locked предотвратит удаление записи test-next . Подробности см. в gitrepository-layout[5] .

Когда extensions.worktreeConfig включен, файл конфигурации .git/worktrees/<id>/config.worktree считывается после .git/config .

Формат вывода списка

Команда worktree list имеет два формата вывода. Формат по умолчанию показывает подробную информацию в одной строке со столбцами. Например:

$ git worktree list
/path/to/bare-source            (bare)
/path/to/linked-worktree        abcd1234 [master]
/path/to/other-linked-worktree  1234abc  (detached HEAD)

Команда также отображает аннотации для каждого рабочего дерева в соответствии с его состоянием. Эти аннотации:

  • locked , если рабочее дерево заблокировано.

  • prunable , если рабочее дерево можно обрезать с помощью git worktree prune .

$ git worktree list
/path/to/linked-worktree    abcd1234 [master]
/path/to/locked-worktree    acbd5678 (brancha) locked
/path/to/prunable-worktree  5678abc  (detached HEAD) prunable

Для этих аннотаций также может быть доступна причина, которую можно увидеть в подробном режиме. Затем аннотация перемещается на следующую строку с отступом, за которой следует дополнительная информация.

$ git worktree list --verbose
/path/to/linked-worktree              abcd1234 [master]
/path/to/locked-worktree-no-reason    abcd5678 (detached HEAD) locked
/path/to/locked-worktree-with-reason  1234abcd (brancha)
        locked: worktree path is mounted on a portable device
/path/to/prunable-worktree            5678abc1 (detached HEAD)
        prunable: gitdir file points to non-existent location

Обратите внимание, что аннотация перемещается на следующую строку, если доступна дополнительная информация, в противном случае она остается на той же строке, что и само рабочее дерево.

Porcelain Format

Формат фарфора имеет строку для каждого атрибута. Если указан -z , строки завершаются NUL, а не новой строкой. Атрибуты перечислены с меткой и значением, разделенными одним пробелом. Атрибуты Boolean (например, bare и detached ) указаны только как метки и присутствуют только в том случае, если значение истинно. Некоторые атрибуты (например, locked ) могут быть указаны только как метки или со значением в зависимости от того, доступна ли причина. Первым атрибутом рабочего дерева всегда является worktree , пустая строка указывает на конец записи. Например:

$ git worktree list --porcelain
worktree /path/to/bare-source
bare

worktree /path/to/linked-worktree
HEAD abcd1234abcd1234abcd1234abcd1234abcd1234
branch refs/heads/master

worktree /path/to/other-linked-worktree
HEAD 1234abc1234abc1234abc1234abc1234abc1234a
detached

worktree /path/to/linked-worktree-locked-no-reason
HEAD 5678abc5678abc5678abc5678abc5678abc5678c
branch refs/heads/locked-no-reason
locked

worktree /path/to/linked-worktree-locked-with-reason
HEAD 3456def3456def3456def3456def3456def3456b
branch refs/heads/locked-with-reason
locked reason why is locked

worktree /path/to/linked-worktree-prunable
HEAD 1233def1234def1234def1234def1234def1234b
detached
prunable gitdir file points to non-existent location

Если не используется -z , любые символы "unusual" в причине блокировки, такие как символы новой строки, экранируются, и вся причина указывается в кавычках, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). Например:

$ git worktree list --porcelain
...
locked "reason\nwhy is locked"
...

Examples

Вы находитесь в середине сеанса рефакторинга, и приходит ваш начальник и требует, чтобы вы немедленно что-то исправили. Обычно вы можете использовать git-stash[1] для временного хранения изменений, однако ваше рабочее дерево находится в таком беспорядке (с новыми, перемещенными и удаленными файлами, а также другими фрагментами, разбросанными вокруг), что вы не хотите рисковать нарушая что-либо из этого. Вместо этого вы создаете временное связанное рабочее дерево для экстренного исправления, удаляете его по завершении и затем возобновляете предыдущий сеанс рефакторинга.

$ git worktree add -b emergency-fix ../temp master
$ pushd ../temp
# ... hack hack hack ...
$ git commit -a -m 'emergency fix for boss'
$ popd
$ git worktree remove ../temp

Bugs

Множественная проверка в целом все еще является экспериментальной, а поддержка подмодулей неполная. НЕ рекомендуется производить несколько проверок суперпроекта.
