gitglossary

Name

gitglossary — Глоссарий Git

Synopsis

*

Description

альтернативная база данных объектов

С помощью альтернативного механизма repository может наследовать часть своего object database из другой базы данных объектов, которая называется "alternate" .

bare repository

Пустой репозиторий обычно представляет собой directory с соответствующим названием и суффиксом .git , который не имеет локально извлеченной копии какого-либо файла, находящегося под контролем версий. То есть все административные и управляющие файлы Git , которые обычно присутствуют в скрытом подкаталоге .git , вместо этого непосредственно присутствуют в каталоге repository.git , и никакие другие файлы не присутствуют и не извлекаются. Обычно издатели репозиториев public предоставляют доступ к голым репозиториям.

blob object

Нетипизированный object , например содержимое файла.

branch

"branch" — ​​это линия развития. Самый последний commit в ветке называется кончиком этой ветки. Наконечником ветки является referenced от ветки head , которая движется вперед по мере того, как в ветке выполняются дополнительные разработки. Один Git repository может отслеживать произвольное количество ветвей, но ваш working tree связан только с одним из них (ветвью "current" или "checked out" ), а HEAD указывает на эту ветвь.

cache

Устарело для: index .

chain

Список объектов, где каждый object в списке содержит ссылку на своего преемника (например, преемником commit может быть один из его parents ).

changeset

BitKeeper/cvsps говорит о " commit " . Поскольку Git не сохраняет изменения, а констатирует, использовать термин "changesets" с Git действительно не имеет смысла.

checkout

Действие по обновлению всего или части working tree с помощью tree object или blob из object database , а также обновления index и HEAD , если все рабочее дерево было указано на новый branch .

cherry-picking

На жаргоне SCM "cherry pick" означает выбор подмножества изменений из серии изменений (обычно фиксаций) и запись их как новую серию изменений поверх другой кодовой базы. В Git это выполняется командой "git cherry-pick" для извлечения изменения, внесенного существующим commit , и записи его на основе кончика текущего branch в качестве нового коммита.

clean

working tree является чистым, если он соответствует revision , на который ссылается текущий head . См. также " dirty " .

commit

Как существительное: Одна точка в истории Git ; вся история проекта представлена ​​как набор взаимосвязанных коммитов. Слово "commit" часто используется Git в тех же местах, где другие системы контроля версий используют слова "revision" или "version" . Также используется как сокращение для commit object .

В качестве глагола: действие по сохранению нового снимка состояния проекта в истории Git путем создания новой фиксации, представляющей текущее состояние index , и продвижения HEAD , чтобы указать на новую фиксацию.

концепция графа фиксации, представления и использование

Синоним структуры DAG , образованной коммитами в базе данных объектов, referenced - советами ветвей с использованием их chain связанных коммитов. Эта структура представляет собой окончательный граф коммитов. Граф можно представить и другими способами, например "commit-graph" file .

commit-graph file

Файл "commit-graph" (обычно через дефис) является дополнительным представлением commit graph , которое ускоряет обход графа фиксации. Файл "commit-graph" хранится либо в каталоге .git/objects/info, либо в каталоге info альтернативной базы данных объектов.

commit object

object , который содержит информацию о конкретном revision , например parents , коммиттере, авторе, дате и tree object , который соответствует верхнему directory сохраненной версии.

совершить (также совершить)

commit object или object , которые могут быть рекурсивно преобразованы в dereferenced к объекту фиксации. Ниже перечислены все фиксации: объект фиксации, tag object , указывающий на объект фиксации, объект-тег, указывающий на объект-тег, указывающий на объект фиксации, и т. д.

core Git

Фундаментальные структуры данных и утилиты Git. Предоставляет только ограниченные инструменты управления исходным кодом.

DAG

Ориентированный ациклический граф. commit objects образуют направленный ациклический граф, поскольку у них есть родительские (направленные), а граф объектов фиксации является ациклическим (нет chain , который начинается и заканчивается одним и тем же object ).

dangling object

unreachable object , который не является reachable даже из других недоступных объектов; висячий объект не имеет ссылок на него ни из одной ссылки, ни из object в repository .

dereference

Что касается symbolic ref : действие доступа к reference , на которое указывает символическая ссылка. Рекурсивное разыменование включает повторение вышеупомянутого процесса для полученной ссылки до тех пор, пока не будет найдена несимволическая ссылка.

Что касается tag object : действие доступа к object указывает на тег. Теги рекурсивно разыменовываются путем повторения операции над объектом результата до тех пор, пока результат не будет иметь указанный тип объекта object type (где применимо) или любой тип объекта, отличный от "tag" . Синоним "recursive dereference" в контексте тегов — " peel " .

Ссылаясь на commit object : действие доступа к объекту дерева коммита. Коммиты не могут быть разыменованы рекурсивно.

Если не указано иное, "dereferencing" , используемый в контексте команд или протоколов Git , является неявно рекурсивным.

detached HEAD

Обычно HEAD хранит имя branch , а команды, которые работают с историей, которую представляет HEAD, работают с историей, ведущей к кончику ветки, на которую указывает HEAD. Однако Git также позволяет вам check out использовать произвольный commit , который не обязательно является вершиной какой-либо конкретной ветви. ГОЛОВКА в таком состоянии называется "detached" .

Обратите внимание, что команды, которые работают с историей текущей ветки (например, git commit для создания новой истории поверх нее), по-прежнему работают, пока HEAD отключен. Они обновляют HEAD, чтобы он указывал на начало обновленной истории, не затрагивая ни одну ветвь. Команды, которые обновляют или запрашивают информацию about текущей ветки (например, git branch --set-upstream-to , которая устанавливает, с какой веткой удаленного отслеживания интегрируется текущая ветка), очевидно, не работают, поскольку в этом состоянии нет (реальной) текущей ветки, о которой можно было бы спросить.

directory

Список, который вы получаете с "ls" :-)

dirty

working tree называется "dirty" , если он содержит модификации, которые не были committed для текущего branch .

evil merge

Злое слияние — это merge , который вносит изменения, которых нет ни в одном parent .

fast-forward

Перемотка вперед — это особый тип merge , где у вас есть revision , а вы — "merging" , еще одно изменение branch , которое является потомком того, что у вас есть. В таком случае вы не создаете новую merge commit , а просто обновляете свою ветку, чтобы она указывала на ту же ревизию, что и ветка, которую вы объединяете. Это часто случается на remote-tracking branch или удаленном repository .

fetch

Получение branch означает получение head ref ветки от удаленного repository , чтобы выяснить, какие объекты отсутствуют в локальном object database , а также получить их. См. также git-fetch[1] .

file system

Линус Торвальдс изначально разработал Git как файловую систему пользовательского пространства, то есть инфраструктуру для хранения файлов и каталогов. Это обеспечило эффективность и скорость работы Git .

Git archive

Синоним repository (для архилюдей).

gitfile

Простой файл .git в корне рабочего дерева, указывающий на каталог, который является реальным репозиторием. Для правильного использования см. git-worktree[1] или git-submodule[1] . Синтаксис см. в gitrepository-layout[5] .

grafts

Графты позволяют объединить две разные линии разработки, записывая фальшивую информацию о происхождении для коммитов. Таким образом, вы можете заставить Git притворяться, что набор parents , который имеет commit , отличается от того, что было записано при создании коммита. Настраивается через файл .git/info/grafts .

Обратите внимание, что механизм трансплантации устарел и может привести к проблемам при переносе объектов между репозиториями; см. git-replace[1] , где представлена ​​более гибкая и надежная система, позволяющая делать то же самое.

hash

В контексте Git это синоним object name .

head

От named reference до commit на кончике branch . Заголовки хранятся в файле в каталоге $GIT_DIR/refs/heads/ , за исключением случаев использования упакованных ссылок. (См. git-pack-refs[1] .)

HEAD

Текущий branch . Более подробно: ваш working tree обычно получается из состояния дерева, на которое ссылается HEAD. HEAD — это ссылка на один из heads в вашем репозитории, за исключением случая использования detached HEAD , в этом случае он напрямую ссылается на произвольный коммит.

head ref

Синоним head .

hook

Во время обычного выполнения нескольких команд Git выполняются вызовы дополнительных сценариев, которые позволяют разработчику добавлять функциональные возможности или проверять. Обычно перехватчики позволяют предварительно проверить и потенциально прервать команду, а также позволяют отправлять пост-уведомление после завершения операции. Сценарии перехвата находятся в каталоге $GIT_DIR/hooks/ и включаются простым удалением суффикса .sample из имени файла. В более ранних версиях Git вам приходилось делать их исполняемыми.

index

Коллекция файлов со статистической информацией, содержимое которых хранится в виде объектов. Индекс представляет собой сохраненную версию вашего working tree . Правда, он может содержать и вторую, и даже третью версию рабочего дерева, которые используются при merging .

index entry

Информация о конкретном файле хранится в index . Запись индекса можно разделить, если merge был запущен, но еще не завершен (т. е. если индекс содержит несколько версий этого файла).

master

Разработка по умолчанию branch . Всякий раз, когда вы создаете Git repository , создается ветка с именем "master" , которая становится активной веткой. В большинстве случаев это включает в себя локальную разработку, хотя это чисто по соглашению и не является обязательным.

merge

Как глагол: перенести содержимое другого branch (возможно, из внешнего repository ) в текущую ветку. В случае, когда объединенная ветвь взята из другого репозитория, это сначала выполняется путем fetching удаленной ветки, а затем путем слияния результата с текущей веткой. Эта комбинация операций выборки и слияния называется pull . Слияние выполняется автоматическим процессом, который определяет изменения, внесенные после разделения ветвей, а затем применяет все эти изменения вместе. В случаях, когда изменения конфликтуют, для завершения слияния может потребоваться вмешательство вручную.

Как существительное: если это не fast-forward , успешное слияние приводит к созданию нового commit , представляющего результат слияния, и имеющего в качестве parents кончики объединенного branches . Этот коммит называется "merge commit" , а иногда просто "merge" .

object

Единица хранения в Git. Он однозначно идентифицируется по номеру SHA-1 его содержимого. Следовательно, объект не может быть изменен.

object database

Хранит набор "objects" , а отдельный object идентифицируется по его object name . Объекты обычно живут в $GIT_DIR/objects/ .

идентификатор объекта (oid)

Синоним object name .

object name

Уникальный идентификатор object . Имя объекта обычно представляет собой шестнадцатеричную строку из 40 символов. Также в просторечии называется SHA-1 .

object type

Один из идентификаторов " commit " , " tree " , " tag " или " blob " , описывающий тип object .

octopus

К merge больше двух branches .

orphan

Акт получения branch , который еще не существует (т. е. ветка unborn ). После такой операции первый созданный коммит становится коммитом без родителя, начиная новую историю.

origin

Восходящий repository по умолчанию. Большинство проектов имеют по крайней мере один проект, который они отслеживают. По умолчанию для этой цели используется origin . Новые обновления восходящего потока будут загружены в remote-tracking branches с именем origin/name-of-upstream-branch, которое вы можете увидеть с помощью git branch -r .

overlay

Обновляйте и добавляйте файлы только в рабочий каталог, но не удаляйте их, аналогично тому, как cp -R обновляет содержимое в целевом каталоге. Это режим по умолчанию в checkout при извлечении файлов из index или tree-ish . Напротив, режим без наложения также удаляет отслеживаемые файлы, отсутствующие в источнике, аналогично rsync --delete .

pack

Набор объектов, сжатых в один файл (для экономии места или для более эффективной передачи).

pack index

Список идентификаторов и другой информации объектов в pack , помогающий эффективно получить доступ к содержимому пакета.

pathspec

Шаблон, используемый для ограничения путей в командах Git .

Спецификации пути используются в командной строке "git ls-files" , "git ls-tree" , "git add" , "git grep" , "git diff" , "git checkout" и многих других команд для ограничения объема операций некоторым подмножеством дерева или рабочего дерева. См. документацию каждой команды, чтобы узнать, относятся ли пути к текущему каталогу или верхнему уровню. Синтаксис спецификации пути следующий:

  • любой путь совпадает сам с собой

  • спецификация пути до последней косой черты представляет собой префикс каталога. Область действия этой спецификации пути ограничена этим поддеревом.

  • остальная часть спецификации пути является шаблоном для оставшейся части пути. Пути относительно префикса каталога будут сопоставляться с этим шаблоном с помощью fnmatch(3); в частности, * и ? can соответствуют разделителям каталогов.

Например, Documentation/*.jpg будет соответствовать всем файлам .jpg в поддереве Documentation, включая Documentation/chapter_1/figure_1.jpg.

Спецификация пути, начинающаяся с двоеточия : , имеет особое значение. В краткой форме за первым двоеточием : следует ноль или более букв "magic signature" (которое необязательно завершается другим двоеточием : ), а остаток представляет собой шаблон, сопоставляемый с путем. "magic signature" состоит из символов ASCII, которые не являются ни буквенно-цифровыми, ни специальными символами регулярных выражений, ни двоеточиями. Необязательное двоеточие, завершающее "magic signature" , можно опустить, если шаблон начинается с символа, который не принадлежит к набору символов "magic signature" и не является двоеточием.

В длинной форме за двоеточием : следует открывающая скобка ( , список из нуля или более "magic words" , разделенный запятыми, и закрывающие скобки ) , а остаток представляет собой шаблон для сопоставления с путем.

Спецификация пути, содержащая только двоеточие, означает "there is no pathspec" . Эту форму не следует комбинировать с другими путями.

top

Волшебное слово top (магическая подпись: / ) обеспечивает соответствие шаблону из корня рабочего дерева, даже если вы запускаете команду из подкаталога.

literal

Подстановочные знаки в шаблоне, такие как * или ? , рассматриваются как буквальные символы.

icase

Регистронезависимое совпадение.

glob

Git обрабатывает шаблон как glob оболочки, подходящий для использования fnmatch(3) с флагом FNM_PATHNAME: подстановочные знаки в шаблоне не будут соответствовать / в пути. Например, "Documentation/*.html" соответствует "Documentation/git.html" , но не соответствует "Documentation/ppc/ppc.html" или "tools/perf/Documentation/perf.html" .

Две последовательные звездочки ( " ** " ) в шаблонах, соответствующих полному пути, могут иметь особое значение:

  • Ведущий " ** " , за которым следует косая черта, означает совпадение во всех каталогах. Например, " **/foo " соответствует файлу или каталогу " foo " в любом месте, так же, как шаблон " foo " . " **/foo/bar " соответствует файлу или каталогу " bar " в любом месте, находящемся непосредственно под каталогом " foo " .

  • Завершающий " /** " соответствует всему внутри. Например, " abc/** " соответствует всем файлам внутри каталога "abc" относительно местоположения файла .gitignore с бесконечной глубиной.

  • Косая черта, за которой следуют две последовательные звездочки, означает, что косая черта соответствует нулю или более каталогам. Например, " a/**/b " соответствует " a/b " , " a/x/b " , " a/x/y/b " и так далее.

  • Остальные звездочки подряд считаются недействительными.

    Шаровая магия несовместима с буквальной магией.

attr

После attr: следует разделенный пробелами список "attribute requirements" , все из которых должны быть соблюдены, чтобы путь считался совпадающим; это в дополнение к обычному немагическому сопоставлению шаблонов pathspec. См. gitattributes[5] .

Каждое из требований к атрибуту пути принимает одну из следующих форм:

  • " ATTR " требует установки атрибута ATTR .

  • " -ATTR " требует, чтобы атрибут ATTR был отключен.

  • " ATTR=VALUE " требует, чтобы атрибут ATTR был установлен в строку VALUE .

  • " !ATTR " требует, чтобы атрибут ATTR не был указан.

    Обратите внимание, что при сопоставлении с объектом дерева атрибуты по-прежнему получаются из рабочего дерева, а не из данного объекта дерева.

exclude

После того, как путь соответствует любой неисключаемой спецификации пути, он будет пропущен через все исключаемые спецификации пути (магическая сигнатура: ! или ее синоним ^ ). Если он совпадает, путь игнорируется. Если неисключающая спецификация пути отсутствует, исключение применяется к набору результатов, как если бы оно было вызвано без какой-либо спецификации пути.

parent

commit object содержит (возможно, пустой) список логических предшественников в линии разработки, то есть его родителей.

peel

Действие рекурсивно dereferencing и tag object .

pickaxe

Термин pickaxe относится к опции процедур diffcore, которая помогает выбирать изменения, добавляющие или удаляющие данную текстовую строку. С опцией --pickaxe-all ее можно использовать для просмотра полной версии changeset , в которой была введена или удалена, скажем, определенная строка текста. См. git-diff[1] .

plumbing

Симпатичное имя для core Git .

porcelain

Симпатичное название для программ и пакетов программ, зависящих от core Git , обеспечивающее высокий уровень доступа к ядру Git. Фарфор имеет больше интерфейса SCM , чем plumbing .

per-worktree ref

Ссылки, относящиеся к worktree , а не глобальные. В настоящее время это только HEAD и все ссылки, начинающиеся с refs/bisect/ , но позже могут включаться и другие необычные ссылки.

pseudoref

Псевдорефы — это класс файлов под $GIT_DIR , которые ведут себя как ссылки для целей анализа версий, но git обрабатывает их особым образом. Оба псевдоссылки имеют имена, написанные заглавными буквами, и всегда начинаются со строки, состоящей из SHA-1 , за которой следует пробел. Итак, HEAD не является псевдоссылкой, поскольку иногда это символическая ссылка. Они могут опционально содержать некоторые дополнительные данные. Примеры: MERGE_HEAD и CHERRY_PICK_HEAD . В отличие от per-worktree refs , эти файлы не могут быть символическими ссылками и никогда не имеют журналов ссылок. Их также нельзя обновить с помощью обычного механизма обновления ссылок. Вместо этого они обновляются путем прямой записи в файлы. Однако их можно прочитать, как если бы они были ссылками, поэтому git rev-parse MERGE_HEAD будет работать.

pull

Вытащить branch означает fetch и merge . См. также git-pull[1] .

push

Нажатие branch означает получение head ref ветки от удаленного repository , выяснение, является ли он предком локальной ссылки на заголовок ветки, и в этом случае помещение всех объектов, которые являются reachable из локальной ссылки на голову и которые отсутствуют. из удаленного репозитория в удаленный object database и обновление удаленной головки ref. Если удаленный head не является предком локального головного устройства, передача не удастся.

reachable

Говорят, что все предки данного commit — это "reachable" из этого коммита. В более общем смысле, один object доступен из другого, если мы можем достичь одного из другого с помощью chain , который следует за tags к тому, что они помечают, commits к их родителям или деревьям и trees к деревьям или blobs , которые они содержат.

reachability bitmaps

Растровые изображения доступности хранят информацию о reachability выбранного набора коммитов в пакетном файле или индексе нескольких пакетов (MIDX), чтобы ускорить поиск объектов. Растровые изображения хранятся в файле ".bitmap" . В репозитории может использоваться не более одного растрового файла. Растровый файл может принадлежать либо одному пакету, либо индексу нескольких пакетов репозитория (если он существует).

rebase

Чтобы повторно применить серию изменений с branch на другую базу и сбросить head этой ветки к результату.

ref

Имя, начинающееся с refs/ (например, refs/heads/master ), указывающее на object name или другую ссылку (последняя называется symbolic ref ). Для удобства ссылку иногда можно сокращать при использовании в качестве аргумента команды Git ; подробности см. в gitrevisions[7] . Рефы хранятся в repository .

Пространство имен ref является иерархическим. Различные подиерархии используются для разных целей (например, иерархия refs/heads/ используется для представления локальных ветвей).

Есть несколько ссылок специального назначения, которые не начинаются с refs/ . Самый яркий пример — HEAD .

reflog

Журнал рефлогов показывает локальный "history" ссылки. Другими словами, он может сказать вам, какой была третья последняя ревизия в репозитории this и каково было текущее состояние в репозитории this вчера в 21:14. Подробности см. в git-reflog[1] .

refspec

"refspec" используется fetch и push для описания сопоставления между удаленным ref и локальным эталоном.

remote repository

repository , который используется для отслеживания того же проекта, но находится где-то в другом месте. Для связи с пультами см. fetch или push .

remote-tracking branch

ref , который используется для отслеживания изменений из другого repository . Обычно он выглядит как refs/remotes/foo/bar (что указывает на то, что он отслеживает ветвь с именем bar на удаленном устройстве с именем foo ) и соответствует правой части настроенной выборки refspec . Ветка удаленного отслеживания не должна содержать прямых изменений или локальных коммитов.

repository

Коллекция refs вместе с object database , содержащая все объекты reachable из ссылок, возможно, сопровождаемые метаданными из одного или нескольких porcelains . Репозиторий может использовать базу данных объектов совместно с другими репозиториями через alternates mechanism .

resolve

Действия по исправлению вручную того, что оставил после себя вышедший из строя автоматический merge .

revision

Синоним commit (существительное).

rewind

Отбросить часть разработки, т.е. присвоить head более раннему revision .

SCM

Управление исходным кодом (инструмент).

SHA-1

"Secure Hash Algorithm 1" ; криптографическая хэш-функция. В контексте Git используется как синоним object name .

shallow clone

В основном это синоним shallow repository , но эта фраза более явно указывает на то, что он был создан с помощью команды git clone --depth=... .

shallow repository

Неглубокий repository имеет неполную историю, у некоторых из commits parents прижжен (другими словами, Git приказано сделать вид, что у этих коммитов нет родителей, даже если они записаны в commit object ). Иногда это полезно, когда вас интересует только недавняя история проекта, хотя реальная история, записанная в восходящем потоке, намного больше. Неглубокий репозиторий создается путем предоставления опции --depth для git-clone[1] , а его история может быть позже углублена с помощью git-fetch[1] .

stash entry

object используется для временного хранения содержимого рабочего каталога dirty и индекса для будущего повторного использования.

submodule

repository , хранящий историю отдельного проекта внутри другого репозитория (последний из которых называется superproject ).

superproject

repository , который ссылается на репозитории других проектов в своем рабочем дереве как submodules . Суперпроект знает имена (но не хранит копии) объектов фиксации содержащихся подмодулей.

symref

Символическая ссылка: вместо того, чтобы содержать сам идентификатор SHA-1 , он имеет формат ref: refs/some/thing и при ссылке на него рекурсивно dereferences к этой ссылке. HEAD — яркий пример symref. Символьные ссылки управляются командой git-symbolic-ref[1] .

tag

ref в пространстве имен refs/tags/ , указывающий на объект произвольного типа (обычно тег указывает либо на tag , либо на commit object ). В отличие от head , тег не обновляется командой commit . Тег Git не имеет ничего общего с тегом Lisp (который в контексте Git будет называться object type ). Тег чаще всего используется для обозначения определенной точки в предке фиксации chain .

tag object

object , содержащий ref , указывающий на другой объект, который может содержать сообщение, как и commit object . Он также может содержать подпись (PGP), и в этом случае он называется "signed tag object" .

topic branch

Обычный Git branch , используемый разработчиком для определения концептуального направления разработки. Поскольку ветки очень просты и недороги, часто желательно иметь несколько небольших ветвей, каждая из которых содержит очень четко определенные концепции или небольшие постепенные, но связанные между собой изменения.

tree

Либо working tree , либо tree object вместе с зависимым blob и объектами дерева (т. е. сохраненным представлением рабочего дерева).

tree object

object , содержащий список имен файлов и режимов, а также ссылки на связанные объекты blob и/или дерева. tree эквивалентен directory .

древесный (также древесный)

tree object или object , которые могут быть рекурсивно преобразованы в dereferenced к объекту дерева. Разыменование commit object дает объект дерева, соответствующий верхнему directory revision . Все следующие элементы являются древовидными: commit-ish , объект-дерево, tag object , который указывает на объект-дерево, объект-тег, который указывает на объект-тег, который указывает на объект-дерево, и т. д.

unborn

HEAD может указывать на branch , который еще не существует и для которого еще нет никаких коммитов, и такая ветвь называется нерожденной ветвью. Самый типичный способ, с помощью которого пользователи сталкиваются с еще не рожденной веткой, — это создание репозитория заново без клонирования из другого места. HEAD будет указывать на ветку main (или master , в зависимости от вашей конфигурации), которой еще предстоит родиться. Кроме того, некоторые операции могут привести к переходу в еще не рожденную ветвь с помощью опции orphan .

unmerged index

index , который содержит несвязанный index entries .

unreachable object

object , который не является reachable из branch , tag или любой другой ссылки.

upstream branch

branch по умолчанию, который объединяется с рассматриваемой веткой (или на которую перебазируется рассматриваемая ветвь). Он настраивается через ветку.<имя>.remote и ветку.<имя>.merge. Если восходящая ветвь A — это origin/B , иногда мы говорим " A is tracking origin/B " .

working tree

Дерево фактически извлеченных файлов. Рабочее дерево обычно содержит содержимое дерева фиксации HEAD , а также любые локальные изменения, которые вы сделали, но еще не зафиксировали.

worktree

Репозиторий может иметь ноль (т. е. пустой репозиторий) или одно или несколько прикрепленных к нему рабочих деревьев. Один "worktree" состоит из "working tree" и метаданных репозитория, большая часть которых совместно используется другими рабочими деревьями одного репозитория, а некоторые из них поддерживаются отдельно для каждого рабочего дерева (например, индекс, HEAD и псевдоссылки, такие как MERGE_HEAD, ссылки для каждого рабочего дерева и файл конфигурации рабочего дерева).
