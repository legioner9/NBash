git-p4

Name

git-p4 — импортировать из репозиториев Perforce и отправлять их в них.

Synopsis

git p4 clone [<sync-options>] [<clone-options>] <p4-depot-path>…​
git p4 sync [<sync-options>] [<p4-depot-path>…​]
git p4 rebase
git p4 submit [<submit-options>] [<master-branch-name>]

Description

Эта команда предоставляет способ взаимодействия с репозиториями p4 с помощью Git.

Создайте новый репозиторий Git из существующего репозитория p4 с помощью git p4 clone , задав ему один или несколько путей к депо p4. Включите новые коммиты из изменений p4 с помощью git p4 sync . Команда sync также используется для включения новых ветвей из других путей хранилища p4. Отправьте изменения Git обратно на p4, используя git p4 submit . Команда git p4 rebase выполняет синхронизацию и перебазирует текущую ветку на обновленную удаленную ветку p4.

Examples

  • Клонировать репозиторий:

    $ git p4 clone //depot/path/project

  • Поработайте над вновь созданным репозиторием Git :

    $ cd project
$ vi foo.h
$ git commit -a -m "edited foo.h"

  • Обновите репозиторий Git последними изменениями из p4, перебазировав свою работу сверху:

    $ git p4 rebase

  • Отправьте свои коммиты обратно на p4:

    $ git p4 submit

Commands

Clone

Обычно git p4 clone используется для создания нового каталога Git из существующего репозитория p4:

$ git p4 clone //depot/path/project

This:

  1. Создает пустой репозиторий Git в подкаталоге с именем project .

  2. Импортирует полное содержимое головной ревизии из заданного пути хранилища p4 в одну фиксацию в ветке Git refs/remotes/p4/master .

  3. Создает локальную ветку master с этого удаленного устройства и проверяет ее.

Чтобы воспроизвести всю историю p4 в Git, используйте модификатор @all в пути к депо:

$ git p4 clone //depot/path/project@all

Sync

Поскольку разработка в репозитории p4 продолжается, эти изменения можно включить в репозиторий Git , используя:

$ git p4 sync

Эта команда находит новые изменения в p4 и импортирует их по мере фиксации Git .

Репозитории P4 также можно добавить в существующий репозиторий Git с помощью git p4 sync :

$ mkdir repo-git
$ cd repo-git
$ git init
$ git p4 sync //path/in/your/perforce/depot

При этом указанное хранилище импортируется в refs/remotes/p4/master в существующем репозитории Git . Опцию --branch можно использовать для указания другой ветки, которая будет использоваться для содержимого p4.

Если репозиторий Git включает ветки refs/remotes/origin/p4 , они будут выбраны и проверены в первую очередь во время git p4 sync . Поскольку импорт непосредственно из p4 выполняется значительно медленнее, чем получение изменений с удаленного Git , это может быть полезно в среде с несколькими разработчиками.

Если существует несколько ветвей, при выполнении git p4 sync автоматически будет использоваться алгоритм "BRANCH DETECTION" , чтобы попытаться разделить новые изменения на правильную ветвь. Это можно переопределить с помощью опции --branch , чтобы указать только одну ветвь для обновления.

Rebase

Обычная схема работы — получение последних изменений из хранилища p4 и объединение их с локальными незафиксированными изменениями. Часто репозиторий p4 является основным местом хранения всего кода, поэтому имеет смысл перебазировать рабочий процесс. Эта команда выполняет git p4 sync , а затем git rebase , чтобы переместить локальные фиксации поверх обновленных изменений p4.

$ git p4 rebase

Submit

Для отправки изменений из репозитория Git обратно в репозиторий p4 требуется отдельное рабочее пространство клиента p4. Это следует указать с помощью переменной среды P4CLIENT или переменной конфигурации Git git-p4.client . Клиент p4 должен существовать, но корневой каталог клиента будет создан и заполнен, если он еще не существует.

Чтобы отправить все изменения, которые есть в текущей ветке Git , но не в ветке p4/master , используйте:

$ git p4 submit

Чтобы указать ветку, отличную от текущей, используйте:

$ git p4 submit topicbranch

Чтобы указать один коммит или диапазон коммитов, используйте:

$ git p4 submit --commit <sha1>
$ git p4 submit --commit <sha1..sha1>

Восходящая ссылка обычно — refs/remotes/p4/master , но ее можно переопределить с помощью параметра командной строки --origin= .

Изменения p4 будут созданы пользователем, вызывающим git p4 submit . Опция --preserve-user приведет к изменению владельца в соответствии с автором фиксации Git . Для этого параметра требуются права администратора в p4, которые можно предоставить с помощью p4 protect .

Чтобы отложить изменения вместо отправки, используйте --shelve и --update-shelve :

$ git p4 submit --shelve
$ git p4 submit --update-shelve 1234 --update-shelve 2345

Unshelve

Unshelving возьмет отложенный список изменений P4 и создаст эквивалентный коммит git в ветке refs/remotes/p4-unshelved/<changelist>.

Коммит git создается относительно текущей исходной версии (по умолчанию HEAD). Родительский коммит создается на основе источника, а затем на его основе создается unshelve коммит.

Исходную версию можно изменить с помощью опции "--origin" .

Если целевая ветка в refs/remotes/p4-unshelved уже существует, старая будет переименована.

$ git p4 sync
$ git p4 unshelve 12345
$ git show p4-unshelved/12345
<submit more changes via p4 to the same files>
$ git p4 unshelve 12345
<refuses to unshelve until git is in sync with p4 again>

Options

General options

Все команды, кроме клонирования, принимают эти параметры.

--git-dir <dir>

Установите переменную среды GIT_DIR . См. git[1] .

-v
--verbose

Предоставьте больше информации о ходе работы.

Sync options

Эти параметры можно использовать в исходном clone , а также в последующих операциях sync .

--branch <ref>

Импортируйте изменения в <ref> вместо refs/remotes/p4/master. Если <ref> начинается с refs/, он используется как есть. В противном случае, если он не начинается с p4/, добавляется этот префикс.

По умолчанию <ref>, не начинающийся с refs/, рассматривается как имя ветки удаленного отслеживания (в разделе refs/remotes/). Это поведение можно изменить с помощью опции --import-local.

По умолчанию <ref> — "master" .

В этом примере новый удаленный "p4/proj2" импортируется в существующий репозиторий Git :

    $ git init
    $ git p4 sync --branch=refs/remotes/p4/proj2 //depot/proj2
--detect-branches

Используйте алгоритм обнаружения ветвей, чтобы найти новые пути в p4. Это описано ниже в "BRANCH DETECTION" .

--changesfile <file>

Импортируйте именно те номера изменений p4, которые указаны в file , по одному в каждой строке. Обычно git p4 проверяет текущее состояние репозитория p4 и определяет изменения, которые необходимо импортировать.

--silent

Не распечатывайте информацию о ходе выполнения.

--detect-labels

Запросите у p4 метки, связанные с путями к депо, и добавьте их как теги в Git. Ограниченная полезность, поскольку импортируются только метки, связанные с новыми списками изменений. Устарело.

--import-labels

Импортируйте метки из p4 в Git.

--import-local

По умолчанию ветки p4 хранятся в refs/remotes/p4/ , где они будут рассматриваться как ветки удаленного отслеживания с помощью git-branch[1] и других команд. Вместо этого этот параметр помещает ветки p4 в refs/heads/p4/ . Обратите внимание, что будущие операции синхронизации также должны указывать --import-local , чтобы они могли найти ветки p4 в refs/heads.

--max-changes <n>

Импортируйте максимум изменений n , а не весь диапазон изменений, включенных в указанный спецификатор версии. Типичным использованием будет использование @all в качестве спецификатора версии, а затем использование --max-changes 1000 для импорта только последних 1000 версий, а не всей истории версий.

--changes-block-size <n>

Размер внутреннего блока, который будет использоваться при преобразовании спецификатора версии, например @all , в список конкретных номеров изменений. Вместо использования одного вызова p4 changes для поиска полного списка изменений для преобразования существует последовательность вызовов p4 changes -m , каждый из которых запрашивает один блок изменений заданного размера. Размер блока по умолчанию — 500, что обычно вполне подходит.

--keep-path

Сопоставление имен файлов из пути к хранилищу p4 с Git по умолчанию включает удаление всего пути к хранилищу. При этом варианте полный путь депо p4 сохраняется в Git . Например, путь //depot/main/foo/bar.c при импорте из //depot/main/ становится foo/bar.c . В случае --keep-path путь Git вместо depot/main/foo/bar.c .

--use-client-spec

Используйте спецификацию клиента, чтобы найти список интересных файлов в p4. См. раздел "CLIENT SPEC" ниже.

-/ <path>

Исключить выбранные пути к депо при клонировании или синхронизации.

Clone options

Эти опции могут использоваться в исходном clone вместе с опциями sync , описанными выше.

--destination <directory>

Где создать репозиторий Git . Если он не указан, последний компонент в пути к хранилищу p4 используется для создания нового каталога.

--bare

Выполните голый клон. См. git-clone[1] .

Submit options

Эти параметры можно использовать для изменения поведения git p4 submit .

--origin <commit>

Восходящее расположение, из которого определяются фиксации для отправки на p4. По умолчанию это самый последний коммит p4, доступный из HEAD .

-M

Обнаружение переименований. См. git-diff[1] . Переименования будут представлены в p4 с использованием явных операций move . Соответствующей опции для обнаружения копий нет, но есть переменные как для перемещений, так и для копий.

--preserve-user

Перепишите изменения p4 перед отправкой на p4. Для этой опции требуются права администратора p4.

--export-labels

Экспортируйте теги из Git как метки p4. Теги, найденные в Git , применяются к рабочему каталогу Perforce.

-n
--dry-run

Покажите, какие коммиты будут отправлены на p4; не меняйте состояние в Git или p4.

--prepare-p4-only

Примените фиксацию к рабочей области p4, открывая, добавляя и удаляя файлы в p4, как при обычной операции отправки. Не выдавайте окончательный "p4 submit" , а вместо этого распечатайте сообщение о том, как отправить вручную или отменить. Эта опция всегда останавливается после первого (самого старого) коммита. Теги Git не экспортируются в p4.

--shelve

Вместо отправки создайте серию отложенных списков изменений. После создания каждой полки соответствующие файлы восстанавливаются/удаляются. Если у вас есть несколько ожидающих коммитов, будет создано несколько полок.

--update-shelve CHANGELIST

Обновите существующий отложенный список изменений с помощью этого коммита. Подразумевается --shelve. Повторите эти действия для нескольких отложенных списков изменений.

--conflict=(ask|skip|quit)

Конфликты могут возникнуть при применении фиксации к p4. В этом случае поведение по умолчанию ( "ask" ) — запрос на то, пропустить эту фиксацию и продолжить или выйти. Эту опцию можно использовать для обхода запроса, что приведет к автоматическому пропуску конфликтующих коммитов, или для прекращения попыток применения коммитов без запроса.

--branch <branch>

После отправки синхронизируйте эту именованную ветку вместо p4/master по умолчанию. Дополнительную информацию см. в разделе "Sync options" выше.

--commit (<sha1>|<sha1>..<sha1>)

Отправьте только указанный коммит или диапазон коммитов вместо полного списка изменений, находящихся в текущей ветке Git .

--disable-rebase

Отключите автоматическое перебазирование после успешной отправки всех коммитов. Также можно установить с помощью git-p4.disableRebase.

--disable-p4sync

Отключите автоматическую синхронизацию p4/master из Perforce после отправки коммитов. Подразумевается --disable-rebase. Также можно установить с помощью git-p4.disableP4Sync. Синхронизация с источником/мастером по-прежнему продолжается, если это возможно.

Крючки для отправки

p4-pre-submit

Перехватчик p4-pre-submit выполняется, если он существует и является исполняемым. Хук не принимает никаких параметров и ничего из стандартного ввода. Выход из этого сценария с ненулевым статусом предотвращает запуск git-p4 submit . Его можно обойти с помощью параметра командной строки --no-verify .

Один из сценариев использования — запуск модульных тестов в перехватчике.

p4-prepare-changelist

Перехватчик p4-prepare-changelist выполняется сразу после подготовки сообщения списка изменений по умолчанию и перед запуском редактора. Он принимает один параметр — имя файла, содержащего текст списка изменений. Выход из сценария с ненулевым статусом приведет к прерыванию процесса.

Целью перехватчика является редактирование файла сообщения на месте, и оно не подавляется опцией --no-verify . Этот хук вызывается, даже если установлен --prepare-p4-only .

p4-changelist

Перехват p4-changelist выполняется после того, как сообщение списка изменений было отредактировано пользователем. Его можно обойти с помощью опции --no-verify . Он принимает единственный параметр — имя файла, содержащего предлагаемый текст списка изменений. Выход с ненулевым статусом приводит к прерыванию команды.

Хук позволяет редактировать файл списка изменений и может использоваться для нормализации текста в каком-либо стандартном формате проекта. Его также можно использовать для отказа в отправке после проверки файла сообщения.

p4-post-changelist

Перехватчик p4-post-changelist вызывается после успешной отправки в P4. Он не принимает никаких параметров и предназначен в первую очередь для уведомления и не может повлиять на результат действия отправки git p4.

Rebase options

Эти параметры можно использовать для изменения поведения git p4 rebase .

--import-labels

Импортируйте этикетки p4.

Unshelve options

--origin

Устанавливает исходную спецификацию git , с которой сравнивается отложенный список изменений P4. По умолчанию p4/master.

Синтаксис пути к депо

Аргумент пути к хранилищу p4 для git p4 sync и git p4 clone может представлять собой один или несколько путей к хранилищу p4, разделенных пробелами, с необязательным спецификатором версии p4 в конце:

"//depot/my/project"

Импортируйте один коммит со всеми файлами изменения #head в этом дереве.

"//depot/my/project@all"

Импортируйте один коммит для каждого изменения в истории этого пути к депо.

"//depot/my/project@1,6"

Импортируйте только изменения с 1 по 6.

"//depot/proj1@all //depot/proj2@all"

Импортируйте все изменения из обоих именованных путей к хранилищу в один репозиторий. Включаются только файлы, расположенные ниже этих каталогов. В Git нет подкаталога для каждого "proj1" и "proj2" . Вы должны использовать опцию --destination при указании более одного пути к депо. Спецификатор редакции должен быть указан одинаково для каждого пути хранилища. Если в путях хранилища есть файлы с таким же именем, путь с последней обновленной версией файла — это тот, который отображается в Git .

Полный синтаксис спецификаторов версии p4 см. в p4 help revisions .

Client spec

Спецификация клиента p4 поддерживается с помощью команды p4 client и содержит, помимо других полей, представление, которое определяет, как хранилище отображается в репозиторий клиента. Команды clone и sync могут обращаться к спецификации клиента, если задана опция --use-client-spec или когда переменная useClientSpec имеет значение true. После git p4 clone переменная useClientSpec автоматически устанавливается в файле конфигурации репозитория. Это позволит будущим командам git p4 submit работать правильно; команда submit просматривает только переменную и не имеет опции командной строки.

Полный синтаксис представления p4 описан в p4 help views . git p4 знает только подмножество синтаксиса представления. Он понимает многострочные сопоставления, наложения с + , исключения с - и двойные кавычки вокруг пробелов. Из возможных подстановочных знаков git p4 обрабатывает только …​ и только тогда, когда он находится в конце пути. git p4 будет жаловаться, если встретит необработанный подстановочный знак.

Существуют ошибки в реализации сопоставлений перекрытия. Если несколько путей к депо сопоставляются через наложения с одним и тем же местом в репозитории, git p4 может выбрать неправильный путь. Эту проблему трудно решить, не выделив клиентскую спецификацию только для git p4 .

Имя клиента git p4 может быть присвоено несколькими способами. Переменная git-p4.client имеет приоритет, если она существует. В противном случае используются обычные механизмы определения клиента p4: переменная среды P4CLIENT , файл, на который ссылается P4CONFIG , или имя локального хоста.

Branch detection

P4 не имеет той же концепции ветки, что и Git. Вместо этого p4 организует свое содержимое в виде дерева каталогов, где по соглашению разные логические ветви находятся в разных местах дерева. Команда p4 branch используется для поддержания сопоставлений между различными областями дерева и указания связанного контента. git p4 может использовать эти сопоставления для определения связей ветвей.

Если у вас есть репозиторий, в котором все интересующие ветки существуют как подкаталоги одного пути к хранилищу, вы можете использовать --detect-branches при клонировании или синхронизации, чтобы git p4 автоматически находил подкаталоги в p4 и создавал их как ветки в Git.

Например, если структура репозитория P4 следующая:

//depot/main/...
//depot/branch1/...

А "p4 branch -o branch1" показывает строку просмотра, которая выглядит так:

//depot/main/... //depot/branch1/...

Затем эта команда git p4 clone :

git p4 clone --detect-branches //depot@all

создает отдельную ветвь в refs/remotes/p4/ для //depot/main, называемую master , и одну для //depot/branch1, называемую depot/branch1 .

Однако не обязательно создавать ветки в p4, чтобы можно было использовать их как ветки. Поскольку автоматически определить связи ветвей сложно, для явной идентификации связей ветвей можно использовать параметр конфигурации Git git-p4.branchList . Это список пар "source:destination" , похожий на простую спецификацию ветки p4, где "source" и "destination" — это элементы пути в репозитории p4. Приведенный выше пример основывался на наличии ветки p4. Без ветвей p4 тот же результат будет:

git init depot
cd depot
git config git-p4.branchList main:branch1
git p4 clone --detect-branches //depot@all .

Performance

Механизм быстрого импорта, используемый git p4 , создает один пакетный файл для каждого вызова git p4 sync . Обычно сжатие мусора Git ( git-gc[1] ) автоматически сжимает их до меньшего количества файлов пакетов, но явный вызов git repack -adf может повысить производительность.

Configuration variables

Следующие параметры конфигурации можно использовать для изменения поведения git p4 . Все они находятся в разделе git-p4 .

General variables

git-p4.user

Пользователь указан как опция для всех команд p4 с -u <user> . Вместо этого можно использовать переменную среды P4USER .

git-p4.password

Пароль указывается как опция для всех команд p4, с -P <password> . Вместо этого можно использовать переменную среды P4PASS .

git-p4.port

Порт указан как опция для всех команд p4, с -p <port> . Вместо этого можно использовать переменную среды P4PORT .

git-p4.host

Хост указан как опция для всех команд p4 с -h <host> . Вместо этого можно использовать переменную среды P4HOST .

git-p4.client

Клиент указан как опция для всех команд p4 с -c <client> , включая спецификацию клиента.

git-p4.retries

Указывает количество повторов команды p4 (в частности, p4 sync ), если время ожидания сети истекло. Значение по умолчанию — 3. Установите значение 0, чтобы отключить повторные попытки или если ваша версия p4 не поддерживает повторные попытки (до 2012.2).

Клонировать и синхронизировать переменные

git-p4.syncFromOrigin

Поскольку импорт коммитов из других репозиториев Git происходит намного быстрее, чем импорт их из p4, существует механизм, позволяющий сначала найти изменения p4 на удаленных Git . Если под refs/remote/origin/p4 существуют ветки, они будут выбраны и использованы при синхронизации с p4. Для этой переменной можно установить значение false , чтобы отключить это поведение.

git-p4.branchUser

Один из этапов обнаружения ветвей включает в себя просмотр ветвей p4 с целью поиска новых для импорта. По умолчанию проверяются все ветки. Эта опция ограничивает поиск только теми, которые принадлежат одному пользователю, указанному в переменной.

git-p4.branchList

Список ветвей, которые будут импортированы, если включено обнаружение ветвей. Каждая запись должна представлять собой пару имен ветвей, разделенных двоеточием (:). В этом примере объявляется, что и ветка A, и ветка B были созданы из основной:

git config       git-p4.branchList main:branchA
git config --add git-p4.branchList main:branchB
git-p4.ignoredP4Labels

Список ярлыков p4, которые следует игнорировать. Он создается автоматически при обнаружении неимпортируемых меток.

git-p4.importLabels

Импортируйте метки p4 в git, как указано в --import-labels.

git-p4.labelImportRegexp

Будут импортированы только метки p4, соответствующие этому регулярному выражению. Значение по умолчанию — [a-zA-Z0-9_\-.]+$ .

git-p4.useClientSpec

Укажите, что спецификация клиента p4 должна использоваться для определения интересующих путей к депо p4. Это эквивалентно указанию опции --use-client-spec . См. раздел "CLIENT SPEC" выше. Эта переменная является логическим значением, а не именем клиента p4.

git-p4.pathEncoding

Perforce сохраняет кодировку пути, заданную исходной ОС. Git ожидает пути в кодировке UTF-8. Используйте эту конфигурацию, чтобы сообщить git -p4, какую кодировку Perforce использовал для путей. Эта кодировка используется для перекодирования путей в UTF-8. Например, Perforce в Windows часто использует "cp1252" для кодирования имен путей. Если этот параметр передается в запрос на клонирование p4, он сохраняется в полученном новом репозитории git .

git-p4.metadataDecodingStrategy

Perforce сохраняет кодировку описаний списка изменений и полных имен пользователей, хранящуюся клиентом в данной ОС. Клиент p4v использует локальную для ОС кодировку, поэтому разные пользователи могут хранить разные описания списков изменений или полные имена пользователей в разных кодировках в одном и том же хранилище. Git допускает противоречивые/неправильные кодировки в сообщениях фиксации и именах авторов, но ожидает, что они будут указаны в utf-8. git -p4 может использовать три различные стратегии декодирования для обработки неопределенности кодирования в Perforce: passthrough просто передает исходные байты из Perforce в git , создавая пригодные для использования, но неправильно закодированные данные, когда данные Perforce закодированы как угодно, кроме utf-8. strict ожидает, что данные Perforce будут закодированы как utf-8, и не может импортировать, если это не так. fallback пытается интерпретировать данные как utf-8, а в противном случае возвращается к использованию вторичной кодировки — по умолчанию общей кодировки Windows cp-1252 — с экранированием байтов верхнего диапазона, если декодирование с использованием резервной кодировки также завершается неудачей. В Python2 стратегией по умолчанию является passthrough по историческим причинам, а в Python3 — fallback . Если выбран strict и декодирование завершается неудачей, в сообщении об ошибке будет предложено изменить этот параметр конфигурации в качестве обходного пути. Если этот параметр передается в запрос на клонирование p4, он сохраняется в полученном новом репозитории git .

git-p4.metadataFallbackEncoding

Укажите резервную кодировку, которая будет использоваться при декодировании имен авторов Perforce и описаний списков изменений с использованием стратегии fallback (см. git -p4.metadataDecodingStrategy). Резервная кодировка будет использоваться только в случае сбоя декодирования в формате utf-8. По умолчанию для этой опции используется cp1252, стандартная кодировка Windows. Если этот параметр передается в запрос на клонирование p4, он сохраняется в полученном новом репозитории git .

git-p4.largeFileSystem

Укажите систему, которая используется для больших (двоичных) файлов. Обратите внимание, что большие файловые системы не поддерживают команду git p4 submit . На данный момент реализован только Git LFS (дополнительную информацию см. в https://git-lfs.github.com/ ). Загрузите и установите расширение командной строки Git LFS, чтобы использовать эту опцию, и настройте его следующим образом:

git config       git-p4.largeFileSystem GitLFS
git-p4.largeFileExtensions

Все файлы, соответствующие расширению файла в списке, будут обработаны большой файловой системой. Не добавляйте к расширениям префикс . .

git-p4.largeFileThreshold

Все файлы, размер которых в несжатом виде превышает пороговое значение, будут обработаны большой файловой системой. По умолчанию порог определяется в байтах. Добавьте суффикс k, m или g, чтобы изменить единицу измерения.

git-p4.largeFileCompressedThreshold

Все файлы со сжатым размером, превышающим пороговое значение, будут обработаны большой файловой системой. Эта опция может замедлить процесс клонирования/синхронизации. По умолчанию порог определяется в байтах. Добавьте суффикс k, m или g, чтобы изменить единицу измерения.

git-p4.largeFilePush

Переменная Boolean , которая определяет, будут ли большие файлы автоматически отправляться на сервер.

git-p4.keepEmptyCommits

Список изменений, содержащий только исключенные файлы, будет импортирован как пустая фиксация, если для этого параметра boolean установлено значение true.

git-p4.mapUser

Сопоставьте пользователя P4 с именем и адресом электронной почты в Git. Для создания сопоставления используйте строку следующего формата:

git config --add git-p4.mapUser "p4user = First Last <[email protected]>"

Сопоставление переопределяет любую пользовательскую информацию из P4. Можно определить сопоставления для нескольких пользователей P4.

Submit variables

git-p4.detectRenames

Обнаружение переименований. См. git-diff[1] . Это может быть истина, ложь или оценка, ожидаемая git diff -M .

git-p4.detectCopies

Обнаружение копий. См. git-diff[1] . Это может быть истина, ложь или оценка, ожидаемая git diff -C .

git-p4.detectCopiesHarder

Обнаруживать копии сложнее. См. git-diff[1] . Логическое значение.

git-p4.preserveUser

При отправке изменения переписываются, чтобы отразить автора Git , независимо от того, кто вызывает git p4 submit .

git-p4.allowMissingP4Users

Если preserveUser имеет значение true, git p4 обычно умирает, если не может найти автора в карте пользователей p4. Этот параметр сохраняет изменения независимо от этого.

git-p4.skipSubmitEdit

Процесс отправки вызывает редактор перед отправкой каждого изменения p4. Однако если этот параметр имеет значение true, шаг редактирования пропускается.

git-p4.skipSubmitEditCheck

После редактирования сообщения об изменении p4 git p4 проверяет, действительно ли описание было изменено, просматривая время изменения файла. Эта опция отключает этот тест.

git-p4.allowSubmit

По умолчанию любая ветвь может использоваться в качестве источника для операции git p4 submit . Эта переменная конфигурации, если она установлена, позволяет использовать в качестве источников отправки только именованные ветки. Имена ветвей должны быть короткими именами (без "refs/heads/" ) и должны быть разделены запятыми ( "," ) без пробелов.

git-p4.skipUserNameCheck

Если пользователь, выполняющий git p4 submit , не существует в карте пользователей p4, git p4 завершает работу. Эту опцию можно использовать для принудительной отправки в любом случае.

git-p4.attemptRCSCleanup

Если этот параметр включен, git p4 submit попытается очистить ключевые слова RCS ($Header$ и т. д.). В противном случае это могло бы вызвать конфликты слияния и помешать отправке. В настоящее время этот вариант следует считать экспериментальным.

git-p4.exportLabels

Экспортируйте теги Git в метки p4 согласно --export-labels.

git-p4.labelExportRegexp

Будут экспортированы только метки p4, соответствующие этому регулярному выражению. Значение по умолчанию — [a-zA-Z0-9_\-.]+$ .

git-p4.conflict

Укажите поведение отправки при обнаружении конфликта с p4 согласно --conflict. Поведение по умолчанию — ask .

git-p4.disableRebase

Не переустанавливайте дерево на p4/master после отправки.

git-p4.disableP4Sync

Не синхронизируйте p4/master с Perforce после отправки. Подразумевается git-p4.disableRebase.

Implementation details

  • Наборы изменений из p4 импортируются с помощью быстрого импорта Git .

  • Для клонирования или синхронизации не требуется клиент p4; Содержимое файла собирается с помощью p4 print .

  • Для отправки требуется клиент p4, который не находится в том же месте, что и репозиторий Git . Исправления применяются по одному к этому клиенту p4 и отправляются оттуда.

  • Каждый коммит, импортированный git p4 , имеет строку в конце сообщения журнала, указывающую местоположение хранилища p4 и номер изменения. Эта строка используется последующими операциями git p4 sync , чтобы узнать, какие изменения p4 являются новыми.
