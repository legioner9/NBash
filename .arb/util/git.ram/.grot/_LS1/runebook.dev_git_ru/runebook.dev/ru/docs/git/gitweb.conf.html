gitweb.conf

Name

gitweb.conf — файл конфигурации Gitweb (веб-интерфейс Git).

Synopsis

/etc/gitweb.conf, /etc/gitweb-common.conf, $GITWEBDIR/gitweb_config.perl

Description

CGI-скрипт gitweb для просмотра репозиториев Git через Интернет использует фрагмент сценария perl в качестве файла конфигурации. Вы можете устанавливать переменные с помощью " our $variable = value " ; текст от символа "#" до конца строки игнорируется. Подробности смотрите в perlsyn(1).

An example:

# файл конфигурации gitweb для http://git.example.org
#
our $projectroot = "/srv/git"; # рекомендация FHS
our $site_name = 'Example.org >> Repos';

Файл конфигурации используется для переопределения настроек по умолчанию, которые были встроены в gitweb во время создания сценария gitweb.cgi .

Хотя можно было просто изменить параметры конфигурации в самом CGI gitweb, эти изменения будут потеряны при обновлении. Параметры конфигурации также могут быть помещены в файл в том же каталоге, что и сценарий CGI, с именем по умолчанию gitweb_config.perl , что позволяет иметь несколько экземпляров gitweb с разными конфигурациями с помощью символических ссылок.

Обратите внимание, что некоторой конфигурацией можно управлять для каждого репозитория, а не для всего gitweb: см. подраздел "Per-repository gitweb configuration" на странице руководства gitweb[1] .

Discussion

Gitweb считывает данные конфигурации из следующих источников в следующем порядке:

  • встроенные значения (некоторые задаются на этапе сборки),

  • общий общесистемный файл конфигурации (по умолчанию /etc/gitweb-common.conf ),

  • либо файл конфигурации для каждого экземпляра (по умолчанию gitweb_config.perl в том же каталоге, что и установленный gitweb), либо, если он не существует, резервный общесистемный файл конфигурации (по умолчанию /etc/gitweb.conf ).

Значения, полученные в более поздних файлах конфигурации, переопределяют значения, полученные ранее в приведенной выше последовательности.

Расположение общего общесистемного файла конфигурации, резервного общесистемного файла конфигурации и файла конфигурации для каждого экземпляра определяется во время компиляции с использованием переменных конфигурации Makefile времени сборки, соответственно GITWEB_CONFIG_COMMON , GITWEB_CONFIG_SYSTEM и GITWEB_CONFIG .

Вы также можете переопределить расположение файлов конфигурации gitweb во время выполнения, установив для следующих переменных среды: GITWEB_CONFIG_COMMON , GITWEB_CONFIG_SYSTEM и GITWEB_CONFIG непустое значение.

Синтаксис файлов конфигурации аналогичен Perl, поскольку эти файлы обрабатываются как фрагменты кода Perl (язык, на котором написан сам gitweb). Переменные обычно задаются с использованием квалификатора our (как в " our $variable = <value>; " ), чтобы избежать синтаксических ошибок, если новая версия gitweb больше не использует переменную и, следовательно, перестает ее объявлять.

Вы можете включить другой файл конфигурации, используя функцию read_config_file(). Например, можно захотеть поместить конфигурацию gitweb, связанную с контролем доступа для просмотра репозиториев через Git olite (один из инструментов управления репозиториями Git ), в отдельный файл, например, в /etc/gitweb-gitolite.conf . Чтобы включить его, поставьте

read_config_file("/etc/gitweb-gitolite.conf");

где-то в используемом файле конфигурации gitweb, например, в файле конфигурации gitweb для каждой установки. Обратите внимание, что read_config_file() проверяет существование читаемого файла и ничего не делает, если он не найден. Он также обрабатывает ошибки во включенном файле.

Конфигурация по умолчанию без файла конфигурации вообще может отлично работать для некоторых установок. Тем не менее, файл конфигурации полезен для настройки или настройки поведения gitweb во многих отношениях, и некоторые дополнительные функции не будут присутствовать, если они явно не включены с помощью настраиваемой переменной %features (см. также раздел "Configuring gitweb features" ниже).

Configuration variables

Некоторые переменные конфигурации имеют значения по умолчанию (встроенные в CGI-скрипт), установленные во время сборки gitweb — если это так, этот факт указывается в их описании. Инструкции по сборке и установке gitweb см. в файле INSTALL gitweb.

Расположение репозиториев

Переменные конфигурации, описанные ниже, управляют тем, как gitweb находит репозитории Git , а также как репозитории отображаются и доступны к ним.

См. также "Repositories" и последующие подразделы на странице руководства gitweb[1] .

$projectroot

Абсолютный путь к файловой системе, который будет добавлен к пути к проекту; путь к репозиторию — $projectroot/$project . Во время установки установите значение $GITWEB_PROJECTROOT . Эта переменная должна быть установлена ​​правильно, чтобы gitweb мог находить репозитории.

Например, если для $projectroot установлено значение "/srv/git" , вставьте следующее в файл конфигурации gitweb:

our $projectroot = "/srv/git";

then

http://git.example.com/gitweb.cgi?p=foo/bar.git

и его эквивалент на основе path_info

http://git.example.com/gitweb.cgi/foo/bar.git

будет соответствовать пути /srv/git/foo/bar.git в файловой системе.

$projects_list

Имя обычного текстового файла со списком проектов или имя каталога для сканирования проектов.

В файлах списка проектов должен быть указан по одному проекту в каждой строке, причем каждая строка имеет следующий формат.

<URI-encoded filesystem path to repository> SP <URI-encoded repository owner>

Значение этой переменной по умолчанию определяется переменной make-файла GITWEB_LIST во время установки. Если эта переменная пуста, gitweb вернется к сканированию каталога $projectroot на наличие репозиториев.

$project_maxdepth

Если переменная $projects_list не установлена, gitweb будет рекурсивно сканировать файловую систему на наличие репозиториев Git . $project_maxdepth используется для ограничения глубины перемещения относительно $projectroot (начальная точка); это означает, что каталоги, находящиеся дальше от $projectroot , чем от $project_maxdepth , будут пропущены.

Это чисто оптимизация производительности, изначально предназначенная для MacOS X, где рекурсивный обход каталогов медленный. Gitweb следует по символическим ссылкам, но обнаруживает циклы, игнорируя любые повторяющиеся файлы и каталоги.

Значение по умолчанию для этой переменной определяется переменной конфигурации времени сборки GITWEB_PROJECT_MAXDEPTH , которая по умолчанию равна 2007.

$export_ok

Показывать репозиторий, только если этот файл существует (в репозитории). Эффективно только в том случае, если эта переменная имеет значение true. Можно установить при сборке gitweb, установив GITWEB_EXPORT_OK . Этот путь указан относительно GIT_DIR . git-daemon[1] использует git-daemon-export-ok , если не запущен с --export-all . По умолчанию эта переменная не установлена, что означает, что эта функция отключена.

$export_auth_hook

Функция, используемая для определения того, какие репозитории следует отображать. Эта подпрограмма должна принимать один параметр — полный путь к проекту, и если она возвращает true, этот проект будет включен в список проектов, и к нему можно будет получить доступ через gitweb, если он соответствует другим требованиям, описанным в $export_ok, $projects_list. и $projects_maxглубина. Пример:

our $export_auth_hook = sub { return -e "$_[0]/git-daemon-export-ok"; };

хотя вышеописанное можно сделать, используя вместо этого $export_ok

our $export_ok = "git-daemon-export-ok";

Если не установлено (по умолчанию), это означает, что эта функция отключена.

См. также более сложный пример в подразделе "Controlling access to Git repositories" на странице руководства gitweb[1] .

$strict_export

Разрешить просмотр только репозиториев, также показанных на странице обзора. Например, это заставляет файл $export_ok решать, доступен ли репозиторий, а не только то, отображается ли он. Если $projects_list указывает на файл со списком проектов, для gitweb будут доступны только указанные репозитории. Может быть установлен во время сборки gitweb через GITWEB_STRICT_EXPORT . По умолчанию эта переменная не установлена, что означает, что вы можете напрямую получить доступ к тем репозиториям, которые скрыты на странице списка проектов (например, не указаны в файле $projects_list).

Finding files

Следующие переменные конфигурации сообщают gitweb, где искать файлы. Значения этих переменных представляют собой пути в файловой системе.

$GIT

Исполняемый файл Core git для использования. По умолчанию установлено значение $GIT_BINDIR/git , которое, в свою очередь, по умолчанию установлено на $(bindir)/git . Если вы используете Git , установленный из двоичного пакета, обычно вам следует установить значение "/usr/bin/git" . Это может быть просто "git" , если ваш веб-сервер имеет разумный PATH; с точки зрения безопасности лучше использовать абсолютный путь к двоичному файлу git . Если у вас установлено несколько версий Git , с их помощью можно выбрать, какую из них использовать. Для работы git Web необходимо (правильно) настроить его.

$mimetypes_file

Файл, который будет использоваться для угадывания типов MIME (на основе расширения имени файла) перед попыткой /etc/mime.types . ОБРАТИТЕ ВНИМАНИЕ, что этот путь, если он относительный, воспринимается как относительный к текущему репозиторию Git , а не к сценарию CGI. Если не установлено, используется только /etc/mime.types (если присутствует в файловой системе). Если файл mimetypes не найден, угадывание mimetype на основе расширения файла отключается. Не установлено по умолчанию.

$highlight_bin

Путь к используемому исполняемому файлу выделения (это должен быть файл из http://andre-simon.de/zip/download.php из-за предположений о параметрах и выводе). По умолчанию установлено highlight ; установите для него полный путь, чтобы выделить исполняемый файл, если он не установлен в PATH вашего веб-сервера. Обратите внимание, что функция highlight должна быть установлена, чтобы gitweb действительно использовал подсветку синтаксиса.

ПРИМЕЧАНИЕ. Чтобы файл был выделен, необходимо определить его тип синтаксиса и этот синтаксис должен поддерживаться "highlight" . Обнаружение синтаксиса по умолчанию минимально, и существует множество поддерживаемых типов синтаксиса, которые по умолчанию не обнаруживаются. Существует три варианта добавления обнаружения синтаксиса. Первым и вторым приоритетом являются %highlight_basename и %highlight_ext , которые определяют базовое имя (полное имя файла, например "Makefile" ) и расширение (например, "sh" ). Ключами этих хэшей являются базовое имя и расширение соответственно, а значением данного ключа является имя синтаксиса, которое должно быть передано через --syntax <syntax> в "highlight" . Последним приоритетом является настройка "highlight" регулярных выражений Shebang для определения языка на основе первой строки файла (например, соответствия строке "#!/bin/bash" ). Дополнительную информацию см. в документации по выделению и конфигурации по умолчанию в /etc/highlight/filetypes.conf.

Например, если репозитории, которые вы размещаете, используют расширение "phtml" для файлов PHP, и вы хотите иметь правильную подсветку синтаксиса для этих файлов, вы можете добавить следующее в конфигурацию gitweb:

our %highlight_ext;
$highlight_ext{'phtml'} = 'php';

Ссылки и их цели

Переменные конфигурации, описанные ниже, настраивают некоторые ссылки gitweb: их цель и внешний вид (текст или изображение), а также местонахождение предварительных требований страницы (таблица стилей, значок значка, изображения, скрипты). Обычно для них оставляются значения по умолчанию, за возможным исключением переменной @stylesheets .

@stylesheets

Список URI таблиц стилей (относительно базового URI страницы). Вы можете указать более одной таблицы стилей, например, чтобы использовать "gitweb.css" в качестве базы с изменениями для конкретного сайта в отдельной таблице стилей, чтобы упростить обновление gitweb. Например, вы можете добавить таблицу стилей site , поместив

push @stylesheets, "gitweb-site.css";

в файле конфигурации gitweb. Те значения, которые являются относительными путями, относятся к базовому URI gitweb.

Этот список должен содержать URI стандартной таблицы стилей gitweb. URI по умолчанию для таблицы стилей gitweb можно установить во время сборки с помощью переменной make-файла GITWEB_CSS . Его значение по умолчанию — static/gitweb.css (или static/gitweb.min.css , если определена переменная CSSMIN , т. е. если во время сборки используется минификатор CSS).

Примечание. Существует также устаревшая переменная конфигурации $stylesheet , которая использовалась в более старых версиях gitweb. Если определена переменная $stylesheet , gitweb использует только таблицу стилей CSS, заданную этой переменной.

$logo

Указывает на место, где вы разместили git-logo.png на своем веб-сервере, или, если точнее, на общий URI логотипа размером 72x27). Это изображение отображается в правом верхнем углу каждой страницы gitweb и используется в качестве логотипа канала Atom. Относительно базового URI gitweb (как пути). Можно настроить при сборке gitweb с использованием переменной GITWEB_LOGO . По умолчанию установлено значение static/git-logo.png .

$favicon

Указывает на место, где вы разместили git-favicon.png на своем веб-сервере, или, если точнее, на общий URI значка, который будет обслуживаться как тип "image/png" . Веб-браузеры, поддерживающие значки веб-сайтов (значки веб-сайтов), могут отображать их в строке URL-адреса браузера и рядом с названием сайта в закладках. Относительно базового URI gitweb. Можно настроить во время сборки с помощью переменной GITWEB_FAVICON . По умолчанию установлено static/git-favicon.png .

$javascript

Указывает на место, где вы разместили gitweb.js на своем веб-сервере, или, если быть более общим, на URI кода JavaScript , используемого gitweb. Относительно базового URI gitweb. Может быть установлен во время сборки с помощью переменной конфигурации времени сборки GITWEB_JS .

Значение по умолчанию — static/gitweb.js или static/gitweb.min.js , если была определена переменная сборки JSMIN , т. е. если во время сборки использовался минификатор JavaScript . Обратите внимание, что этот единственный файл создается из нескольких отдельных JavaScript "modules" .

$home_link

Цель домашней ссылки вверху всех страниц (первая часть вида "breadcrumbs" ). По умолчанию ему присваивается абсолютный URI текущей страницы (значение переменной $my_uri или "/" , если $my_uri не определен или представляет собой пустую строку).

$home_link_str

Метка "home link" вверху всех страниц, ведущая к $home_link (обычно это главная страница gitweb, содержащая список проектов). Он используется в качестве первого компонента "breadcrumb trail" gitweb: <home link> / <project> / <action> . Может быть установлен во время сборки с помощью переменной GITWEB_HOME_LINK_STR . По умолчанию установлено "projects" , так как эта ссылка ведет на список проектов. Другой популярный вариант — установить его в качестве имени сайта. Обратите внимание, что он рассматривается как необработанный HTML, поэтому его не следует устанавливать из ненадежных источников.

@extra_breadcrumbs

Дополнительные ссылки должны быть добавлены в начало навигационной цепочки перед домашней ссылкой на страницы, которые логически являются "above" списка проектов gitweb, например организация и отдел, на которых размещен сервер gitweb. Каждый элемент списка является ссылкой на массив, в котором элемент 0 — это текст ссылки (эквивалент $home_link_str ), а элемент 1 — целевой URL-адрес (эквивалент $home_link ).

Например, следующая настройка создает навигационную цепочку типа "home / dev / projects / …​" , где "projects" — это домашняя ссылка.

    our @extra_breadcrumbs = (
      [ 'home' => 'https://www.example.org/' ],
      [ 'dev'  => 'https://dev.example.org/' ],
    );
$logo_url
$logo_label

URI и метка (название) для ссылки на логотип Git (или логотип вашего сайта, если вы решили использовать другое изображение логотипа). По умолчанию оба относятся к домашней странице Git , https://git-scm.com ; в прошлом на документацию Git указывали по адресу https://www.kernel.org .

Изменение внешнего вида gitweb

Вы можете настроить внешний вид страниц, созданных gitweb, используя переменные, описанные ниже. Вы можете изменить имя сайта, добавить общие верхние и нижние колонтитулы для всех страниц, а также добавить описание этой установки gitweb на его главную страницу (которая является страницей списка проектов) и т. д.

$site_name

Название вашего сайта или организации, которое будет отображаться в заголовках страниц. Установите для него что-нибудь описательное для более понятных закладок и т. д. Если эта переменная не установлена ​​или установлена, то gitweb использует значение переменной среды SERVER_NAME CGI , устанавливая имя сайта на "$SERVER_NAME Git" или "Untitled Git" , если эта переменная не установлена ​​(например, при запуске gitweb как отдельный скрипт).

Можно настроить с помощью GITWEB_SITENAME во время сборки. Не установлено по умолчанию.

$site_html_head_string

HTML-фрагмент, который будет включен в раздел <head> каждой страницы. Можно установить с помощью GITWEB_SITE_HTML_HEAD_STRING во время сборки. Нет значения по умолчанию.

$site_header

Имя файла с HTML, который будет включен вверху каждой страницы. Относительно каталога, содержащего сценарий gitweb.cgi . Можно установить с помощью GITWEB_SITE_HEADER во время сборки. Нет значения по умолчанию.

$site_footer

Имя файла с HTML, который будет включен внизу каждой страницы. Относительно каталога, содержащего сценарий gitweb.cgi . Можно установить с помощью GITWEB_SITE_FOOTER во время сборки. Нет значения по умолчанию.

$home_text

Имя HTML-файла, который, если он существует, включается на страницу обзора проектов gitweb (представление "projects_list" ). Относительно каталога, содержащего скрипт gitweb.cgi. Значение по умолчанию можно изменить во время сборки с помощью переменной GITWEB_HOMETEXT . По умолчанию установлено indextext.html .

$projects_list_description_width

Ширина (в символах) столбца "Description" списка проектов. Более длинные описания будут обрезаны (попытка обрезаться по границе слов); полное описание доступно в атрибуте title (обычно отображается при наведении курсора мыши). Значение по умолчанию — 25, что может быть слишком мало, если вы используете длинные описания проектов.

$default_projects_order

Значение по умолчанию порядка проектов на странице списка проектов, что означает порядок, используемый, если вы не сортируете список проектов явно (если в URL-адресе нет параметра запроса CGI "o" ). Допустимые значения: "none" (несортированный), "project" (проекты указаны по имени проекта, т. е. пути к репозиторию относительно $projectroot ), "descr" (описание проекта), "owner" и "age" (по дате последней фиксации).

Значение по умолчанию — "project" . Неизвестное значение означает несортированное.

Изменение поведения gitweb

Эти переменные конфигурации управляют поведением internal gitweb.

$default_blob_plain_mimetype

Тип mime по умолчанию для представления blob_plain (raw), если проверка типа mime не приводит к получению какого-либо другого типа; по умолчанию "text/plain" . Gitweb угадывает mimetype файла для отображения на основе расширения его имени файла, используя файлы $mimetypes_file (если он установлен и файл существует) и /etc/mime.types (см. справочную страницу mime.types(5); gitweb поддерживает только правила расширения имени файла).

$default_text_plain_charset

Кодировка по умолчанию для текстовых файлов. Если этот параметр не установлен, будет использоваться конфигурация веб-сервера. Не установлено по умолчанию.

$fallback_encoding

Gitweb предполагает эту кодировку, если строка содержит символы, отличные от UTF-8. Резервное декодирование используется без проверки ошибок, поэтому это может быть даже "utf-8" . Значение должно быть допустимой кодировкой; список см. на странице руководства Encoding::Supported(3pm). По умолчанию стоит "latin1" , он же. "iso-8859-1" .

@diff_opts

Переименуйте параметры обнаружения для git-diff и git-diff-tree. По умолчанию используется ('-M'); установите для него значение ('-C') или ('-C', '-C'), чтобы также обнаруживать копии, или установите его в (), т.е. пустой список, если вы не хотите, чтобы обнаружение переименований.

Обратите внимание, что переименование и особенно обнаружение копирования могут быть весьма ресурсоемкими. Также обратите внимание, что инструменты, отличные от Git , могут иметь проблемы с исправлениями, созданными с помощью параметров, упомянутых выше, особенно когда они включают копирование файлов ('-C') или перекрестное переименование ('-B').

Некоторые дополнительные функции и политики

Большинство функций настраиваются через хэш %feature ; однако некоторые дополнительные функции gitweb можно включить и настроить с помощью переменных, описанных ниже. Этот список, помимо переменных конфигурации, которые контролируют внешний вид gitweb, содержит переменные, настраивающие административную часть gitweb (например, предотвращение межсайтового скриптинга; по общему признанию, это как побочный эффект влияет на внешний вид страниц "summary" или ограничение нагрузки).

@git_base_url_list

Список базовых URL-адресов Git . Эти URL-адреса используются для создания URL-адресов, описывающих, откуда получить проект, которые отображаются на странице сводки проекта. Полный URL-адрес получения — " $git_base_url/$project " для каждого элемента этого списка. Вы можете настроить несколько базовых URL-адресов (например, один для протокола git:// и один для протокола http:// ).

Обратите внимание, что конфигурация каждого репозитория может быть установлена ​​в файле $GIT_DIR/cloneurl или как значения многозначной переменной конфигурации gitweb.url в конфигурации проекта. Конфигурация для каждого репозитория имеет приоритет над значением, состоящим из элементов @git_base_url_list и имени проекта.

Вы можете установить одно значение (одна запись/элемент в этом списке) во время сборки, установив переменную конфигурации времени сборки GITWEB_BASE_URL . По умолчанию установлено значение (), т.е. пустой список. Это означает, что gitweb не будет пытаться создать URL-адрес проекта (для получения) на основе имени проекта.

$projects_list_group_categories

Включить ли группировку проектов по категориям на странице списка проектов. Категория проекта определяется файлом $GIT_DIR/category или переменной gitweb.category в конфигурации каждого репозитория. По умолчанию отключено (установлено значение 0).

$project_list_default_category

Категория по умолчанию для проектов, для которых указан none . Если для этого параметра задана пустая строка, такие проекты останутся без категории и будут перечислены вверху над проектами, отнесенными к категориям. Используется только в том случае, если категории проектов включены, что означает, что $projects_list_group_categories имеет значение true. По умолчанию установлено значение «» (пустая строка).

$prevent_xss

Если true , некоторые функции gitweb отключены, чтобы предотвратить запуск атак межсайтового скриптинга (XSS) содержимым в репозиториях. Установите для этого параметра значение true , если вы не доверяете содержимому своих репозиториев. False по умолчанию (установлен на 0).

$maxload

Используется для установки максимальной нагрузки, на которую мы по-прежнему будем отвечать на запросы gitweb. Если загрузка сервера превышает это значение, gitweb вернет ошибку "503 Service Unavailable" . Нагрузка на сервер принимается равной 0, если gitweb не может определить ее значение. В настоящее время он работает только в Linux, где используется /proc/loadavg ; нагрузка — это количество активных задач в системе — фактически запущенных процессов — усредненное за последнюю минуту.

Установите для $maxload неопределенное значение ( undef ), чтобы отключить эту функцию. Значение по умолчанию — 300.

$omit_age_column

Если это правда, опустите столбец с датой самого последнего коммита на странице списка проектов. Это может сэкономить немного операций ввода-вывода и форка для каждого репозитория.

$omit_owner

Если true запрещает отображение информации о владельце репозитория.

$per_request_config

Если для этого параметра задана ссылка на код, он будет выполняться один раз для каждого запроса. Таким образом вы можете установить части конфигурации, которые изменяются в течение сеанса. Например, можно использовать следующий код в файле конфигурации gitweb

our $per_request_config = sub {
        $ENV{GL_USER} = $cgi->remote_user || "gitweb";
};

Если $per_request_config не является ссылкой на код, он интерпретируется как значение boolean . Если это true , gitweb будет обрабатывать файлы конфигурации один раз за запрос, а если это false , gitweb будет обрабатывать файлы конфигурации только один раз при каждом выполнении. True по умолчанию (установлен на 1).

ПРИМЕЧАНИЕ. $my_url , $my_uri и $base_url перезаписываются значениями по умолчанию перед каждым запросом, поэтому, если вы хотите их изменить, обязательно установите для этой переменной значение true или ссылку на код, вносящую желаемые изменения.

Эта переменная имеет значение только при использовании постоянных веб-сред, которые обслуживают несколько запросов с использованием одного экземпляра gitweb, например mod_perl, FastCGI или Plackup.

Other variables

Обычно вам не нужно изменять (настраивать) какие-либо переменные конфигурации, описанные ниже; они должны быть автоматически установлены gitweb на правильное значение.

$version

Версия Gitweb, устанавливается автоматически при создании gitweb.cgi из gitweb.perl. Возможно, вы захотите изменить его, если вы используете модифицированный gitweb, например.

our $version .= " with caching";

если вы запускаете модифицированную версию gitweb с поддержкой кэширования. Эта переменная является чисто информационной и используется, например, в мета-заголовке "generator" в заголовке HTML.

$my_url
$my_uri

Полный URL-адрес и абсолютный URL-адрес скрипта gitweb; в более ранних версиях gitweb вам, возможно, приходилось устанавливать эти переменные, но теперь в этом нет необходимости. См. $per_request_config , если вам нужно их установить.

$base_url

Базовый URL-адрес для относительных URL-адресов на страницах, созданных gitweb (например, $logo , $favicon , @stylesheets , если они являются относительными URL-адресами), необходим и используется <base href="$base_url"> только для URL-адресов с непустым PATH_INFO. Обычно gitweb устанавливает свое значение правильно, и нет необходимости устанавливать эту переменную, например, в $my_uri или "/" . См. $per_request_config , если вам все равно нужно его переопределить.

Настройка функций gitweb

Многие функции gitweb можно включить (или отключить) и настроить с помощью хеша %feature . Имена функций gitweb являются ключами этого хеша.

Каждый хеш-элемент %feature является хэш-ссылкой и имеет следующую структуру:

"<feature_name>" => {
        "sub" => <feature-sub (subroutine)>,
        "override" => <allow-override (boolean)>,
        "default" => [ <options>... ]
},

Некоторые функции нельзя переопределить для каждого проекта. Для этих функций структура соответствующего хеш-элемента %feature имеет более простую форму:

"<feature_name>" => {
        "override" => 0,
        "default" => [ <options>... ]
},

Как можно видеть, в нем отсутствует элемент «sub».

Значение каждой части конфигурации функции описано ниже:

default

Список (ссылка на массив) параметров функции (если таковые имеются), используемый также для переключения (включения или отключения) данной функции.

Обратите внимание, что в настоящее время это всегда ссылка на массив, даже если функция не принимает никаких параметров конфигурации, а значение «по умолчанию» используется только для ее включения или выключения. В таком случае вы включаете эту функцию, устанавливая для этого элемента значение [1] , и отключаете его, устанавливая для него значение [0] . См. также отрывок о функции "blame" в разделе "Examples" .

Чтобы отключить функции, которые принимают параметры (настраиваются), вам необходимо установить для этого элемента пустой список, т.е. [] .

override

Если это поле имеет значение true , то данная функция является переопределяемой, что означает, что ее можно настроить (или включить/отключить) для каждого репозитория.

Обычно данный "<feature>" настраивается с помощью переменной конфигурации gitweb.<feature> в файле конфигурации Git для каждого репозитория.

Обратите внимание, что по умолчанию ни одна функция не может быть переопределена.

sub

Внутренние детали реализации. Важно то, что если это поле отсутствует, то переопределение для каждого репозитория для данной функции не поддерживается.

Вам не нужно будет когда-либо менять его в файле конфигурации gitweb.

Возможности %feature

Функции gitweb, которые можно настроить с помощью хеша %feature , перечислены ниже. Это должен быть полный список, но в конечном итоге авторитетный и полный список находится в исходном коде gitweb.cgi, а функции описаны в комментариях.

blame

Включите представления BLOB-объектов "blame" и "blame_incremental" , показывая для каждой строки последний коммит, который ее изменил; см. git-blame[1] . Это может быть очень ресурсоемким, поэтому по умолчанию отключено.

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.blame (логическое значение).

snapshot

Включите и настройте действие "snapshot" , которое позволяет пользователю загружать сжатый архив любого дерева или коммита, созданный git-archive[1] и, возможно, дополнительно сжатый. Это потенциально может привести к высокому трафику, если у вас большой проект.

Значение «по умолчанию» представляет собой список названий форматов снимков, определенных в хеше %known_snapshot_formats , которые вы хотите предложить. Поддерживаемые форматы: "tgz" , "tbz2" , "txz" (сжатый tar-архив gzip/bzip2/xz) и "zip" ; пожалуйста, обратитесь к источникам gitweb для получения окончательного списка. По умолчанию предлагается только "tgz" .

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.snapshot , которая содержит список форматов, разделенных запятыми, или "none" для отключения снимков. Неизвестные значения игнорируются.

grep

Включить поиск grep, в котором перечислены файлы в выбранном в данный момент дереве (каталоге), содержащие заданную строку; см. git-grep[1] . Конечно, это может потенциально нагружать процессор. Включено по умолчанию.

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.grep (логическое значение).

pickaxe

Включите так называемый поиск киркой, который будет перечислять коммиты, которые ввели или удалили данную строку в файле. Это может быть практичной и довольно быстрой альтернативой действию "blame" , но оно по-прежнему потенциально требует большого количества ресурсов ЦП. Включено по умолчанию.

Поиск кирки описан в git-log[1] (описание опции -S<string> , более подробное описание которого относится к записи кирки в gitdiffcore[7] ).

Эту функцию можно настроить для каждого репозитория, задав переменную конфигурации репозитория gitweb.pickaxe (логическое значение).

show-sizes

Включите отображение размера больших двоичных объектов (обычных файлов) в представлении "tree" в отдельном столбце, аналогично тому, что делает ls -l ; см. описание опции -l на странице руководства git-ls-tree[1] . Это стоит немного ввода-вывода. Включено по умолчанию.

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.showSizes (логическое значение).

patches

Включите и настройте представление "patches" , которое отображает список коммитов в формате вывода по электронной почте (обычный текст); см. также git-format-patch[1] . Значение представляет собой максимальное количество исправлений в наборе исправлений, созданном в представлении "patches" . Установите в поле default список, содержащий один элемент, или пустой список, чтобы отключить просмотр исправлений, или список, содержащий одно отрицательное число, чтобы удалить любое ограничение. Значение по умолчанию — 16.

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.patches (целое число).

avatar

Поддержка аватаров. Когда эта функция включена, такие представления, как "shortlog" или "commit" , будут отображать аватар, связанный с адресом электронной почты каждого коммиттера и автора.

В настоящее время доступны поставщики "gravatar" и "picon" . Одновременно можно выбрать только одного провайдера ( default — список из одного элемента). Если указан неизвестный поставщик, функция отключена. Обратите внимание, что некоторые провайдеры могут потребовать установки дополнительных пакетов Perl; более подробную информацию см. в gitweb/INSTALL .

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации gitweb.avatar репозитория.

См. также %avatar_size с размерами пикселей для значков и аватаров ( "default" используется для однострочных, например "log" и "shortlog" , "double" используется для двухстрочных, например "commit" , "commitdiff" или "tag" ). Если размеры шрифта или высота строк по умолчанию изменены (например, путем добавления дополнительной таблицы стилей CSS в @stylesheets ), возможно, будет целесообразно изменить эти значения.

email-privacy

Редактируйте адреса электронной почты из сгенерированного HTML-контента и т. д. Это скрывает адреса электронной почты, полученные из разделов автора/коммиттера и комментариев журнала Git . Он предназначен для предотвращения веб-сканеров, которые собирают и злоупотребляют адресами. Такие сканеры могут не учитывать файл robots.txt. Обратите внимание, что пользователи и пользовательские инструменты также видят адреса как отредактированные. Если Git web не является последним шагом рабочего процесса, последующие шаги могут работать неправильно из-за отредактированной информации, которую они получают. По умолчанию отключено.

highlight

Поддержка подсветки синтаксиса на стороне сервера в представлении "blob" . Для этого требуется наличие программы $highlight_bin (см. описание этой переменной в разделе "Configuration variables" выше), поэтому по умолчанию она отключена.

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.highlight (логическое значение).

remote_heads

Включите отображение удаленных головок (ветвей удаленного отслеживания) в списке "heads" . В большинстве случаев список ветвей удаленного отслеживания является ненужной внутренней деталью private , и поэтому эта функция по умолчанию отключена. git-instaweb[1] , который обычно используется для просмотра локальных репозиториев, включает и использует эту функцию.

Эту функцию можно настроить для каждого репозитория с помощью переменной конфигурации репозитория gitweb.remote_heads (логическое значение).

Остальные функции нельзя переопределить для каждого проекта.

search

Включить текстовый поиск, в котором будут перечислены коммиты, соответствующие автору, коммиттеру или тексту коммита для заданной строки; см. описание опций --author , --committer и --grep на странице руководства git-log[1] . Включено по умолчанию.

Переопределение для конкретного проекта не поддерживается.

forks

Если эта функция включена, gitweb считает проекты в подкаталогах корня проекта (базовое имя) ветвями существующих проектов. Для каждого проекта $projname.git проекты в каталоге $projname/ и его подкаталогах не будут отображаться в основном списке проектов. Вместо этого рядом с $projname отображается знак «+», который ссылается на представление "forks" , в котором перечислены все ответвления (все проекты в подкаталоге $projname/ ). Кроме того, представление проекта "forks" связано со страницей сводки проекта.

Если список проектов взят из файла ( $projects_list указывает на файл), ответвления распознаются только в том случае, если они указаны после основного проекта в этом файле.

Переопределение для конкретного проекта не поддерживается.

actions

Вставляйте пользовательские ссылки в панель действий на всех страницах проекта. Это позволяет вам ссылаться на сторонние скрипты, интегрируемые в gitweb.

Значение "default" состоит из списка троек в форме ("<label>", "<link>", "<position>") , где "position" — это метка, после которой вставляется ссылка, "link" — это строка формата, где %n расширяется до имени проекта, %f — до пути проекта в файловой системе (т. е. "$projectroot/$project" ). , %h — текущий хеш (параметр gitweb «h») и %b — текущая база хэша (параметр gitweb «hb»); %% расширяется до «%».

Например, на момент написания этой страницы хостинг-сайт https://repo.or.cz Git установил для нее следующее значение, чтобы включить графический журнал (с использованием стороннего инструмента git-browser):

$feature{'actions'}{'default'} =
        [ ('graphiclog', '/git-browser/by-commit.html?r=%n', 'summary')];

Это добавляет ссылку с названием "graphiclog" после ссылки "summary" , ведущую к сценарию git-browser , передавая r=<project> в качестве параметра запроса.

Переопределение для конкретного проекта не поддерживается.

timed

Включите отображение того, сколько времени и сколько команд Git потребовалось для создания и отображения каждой страницы в нижнем колонтитуле (внизу страницы). Например, нижний колонтитул может содержать: "This page took 6.53325 seconds and 13 Git commands to generate." Отключено по умолчанию.

Переопределение для конкретного проекта не поддерживается.

javascript-timezone

Включите и настройте возможность изменения общего часового пояса для дат в выводе gitweb через JavaScript. Даты в выходных данных gitweb включают в себя дату автора и дату фиксации в представлениях "commit" , "commitdiff" и "log" , а также дату taggerdate в представлении "tag" . Включено по умолчанию.

Значение представляет собой список из трех значений: часовой пояс по умолчанию (если клиент не выбрал другой часовой пояс и не сохранил его в файле cookie), имя файла cookie, в котором будет храниться выбранный часовой пояс, и используемый класс CSS. для разметки дат манипуляций. Если вы хотите отключить эту функцию, установите для "default" пустой список: [] .

Типичные файлы конфигурации gitweb меняют только начальный (по умолчанию) часовой пояс и оставляют для других элементов значения по умолчанию:

$feature{'javascript-timezone'}{'default'}[0] = "utc";

Представленный здесь пример конфигурации гарантированно имеет обратную и прямую совместимость.

Значения часового пояса могут быть "local" (для местного часового пояса, который использует браузер), "utc" (то, что gitweb использует, когда JavaScript или эта функция отключены) или числовые часовые пояса в форме "+/-HHMM" , например "+0200" .

Переопределение для конкретного проекта не поддерживается.

extra-branch-refs

Список дополнительных каталогов под "refs" , которые будут использоваться в качестве ссылок на ветки. Например, если у вас есть настройка gerrit, где все ветки под refs/heads/ являются официальными, ветки, открываемые после проверки, а ветки под refs/sandbox/, refs/wip и refs/other являются пользовательскими, где разрешения намного шире, тогда вы можете установить эту переменную следующим образом:

$feature{'extra-branch-refs'}{'default'} =
        ['sandbox', 'wip', 'other'];

Эту функцию можно настроить для каждого репозитория после установки для $feature{ extra-branch-refs }{ override } значения true с помощью переменной конфигурации gitweb.extraBranchRefs репозитория, которая содержит список ссылок, разделенных пробелами. Пример:

[gitweb]
        extraBranchRefs = sandbox wip other

gitweb.extraBranchRefs на самом деле является многозначной переменной конфигурации, поэтому следующий пример также верен, и результат такой же, как и в приведенном выше фрагменте:

[gitweb]
        extraBranchRefs = sandbox
        extraBranchRefs = wip other

Указание ссылки, которая не прошла проверку "git check-ref-format" , является ошибкой. Повторяющиеся значения фильтруются.

Examples

Чтобы включить поддержку обвинений, поиска киркой и поддержку снимков (разрешить создание снимков "tar.gz" и "zip" ), одновременно позволяя отдельным проектам отключать их, поместите следующее в свой файл GITWEB_CONFIG:

$feature{'blame'}{'default'} = [1];
$feature{'blame'}{'override'} = 1;

$feature{'pickaxe'}{'default'} = [1];
$feature{'pickaxe'}{'override'} = 1;

$feature{'snapshot'}{'default'} = ['zip', 'tgz'];
$feature{'snapshot'}{'override'} = 1;

Если вы разрешите переопределение функции моментального снимка, вы можете указать, какие форматы моментальных снимков будут отключены глобально. Вы также можете добавить любые параметры командной строки (например, настройку уровня сжатия). Например, вы можете отключить сжатие снимков Zip и настроить gzip(1) для запуска на уровне 6, добавив следующие строки в файл конфигурации gitweb:

$known_snapshot_formats{'zip'}{'disabled'} = 1;
$known_snapshot_formats{'tgz'}{'compressor'} = ['gzip','-6'];

Bugs

Отладка была бы проще, если бы резервный файл конфигурации ( /etc/gitweb.conf ) и переменная среды для переопределения его местоположения ( GITWEB_CONFIG_SYSTEM ) имели имена, отражающие их роль "fallback" . Текущие имена сохраняются во избежание нарушения рабочих настроек.

Environment

Расположение файлов конфигурации для каждого экземпляра и общесистемной конфигурации можно переопределить с помощью следующих переменных среды:

GITWEB_CONFIG

Устанавливает расположение файла конфигурации для каждого экземпляра.

GITWEB_CONFIG_SYSTEM

Устанавливает расположение резервного общесистемного файла конфигурации. Этот файл доступен только для чтения, если он не существует для каждого экземпляра.

GITWEB_CONFIG_COMMON

Устанавливает расположение общего общесистемного файла конфигурации.

Files

gitweb_config.perl

Это имя по умолчанию для файла конфигурации каждого экземпляра. Формат этого файла описан выше.

/etc/gitweb.conf

Это имя по умолчанию резервного общесистемного файла конфигурации. Этот файл используется только в том случае, если переменная конфигурации для каждого экземпляра не найдена.

/etc/gitweb-common.conf

Это имя по умолчанию общего общесистемного файла конфигурации.
