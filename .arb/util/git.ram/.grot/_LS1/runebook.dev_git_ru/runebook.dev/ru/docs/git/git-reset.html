git-reset

Name

git-reset — Сбросить текущий HEAD в указанное состояние

Synopsis

git reset [-q] [<tree-ish>] [--] <pathspec>…​
git reset [-q] [--pathspec-from-file=<file> [--pathspec-file-nul]] [<tree-ish>]
git reset (--patch | -p) [<tree-ish>] [--] [<pathspec>…​]
git reset [--soft | --mixed [-N] | --hard | --merge | --keep] [-q] [<commit>]

Description

В первых трех формах скопируйте записи из <tree-ish> в индекс. В последней форме установите текущий заголовок ветки ( HEAD ) на <commit> , при необходимости изменив индекс и рабочее дерево для соответствия. По умолчанию <tree-ish> / <commit> имеет значение HEAD во всех формах.

git сброс [-q] [<дерево>] [--] <путь>…
git сброс [-q] [--pathspec-from-file=<file> [--pathspec-file-nul]] [<tree-ish>]

Эти формы сбрасывают записи индекса для всех путей, соответствующих <pathspec> , до их состояния в <tree-ish> . (Это не влияет на рабочее дерево или текущую ветвь.)

Это означает, что git reset <pathspec> является противоположностью git add <pathspec> . Эта команда эквивалентна git restore [--source=<tree-ish>] --staged <pathspec>... .

После запуска git reset <pathspec> для обновления записи индекса вы можете использовать git-restore[1] для проверки содержимого индекса в рабочем дереве. Альтернативно, используя git-restore[1] и указав фиксацию с помощью --source , вы можете скопировать содержимое пути из фиксации в индекс и в рабочее дерево за один раз.

git сброс (--patch | -p) [<дерево>] [--] [<путь>…​]

В интерактивном режиме выберите фрагменты разницы между индексом и <tree-ish> (по умолчанию HEAD ). Выбранные фрагменты применяются обратно по отношению к индексу.

Это означает, что git reset -p является противоположностью git add -p , т.е. его можно использовать для выборочного сброса ханков. См. раздел «Интерактивный режим» git-add[1] , чтобы узнать, как работать в режиме --patch .

git сброс [<режим>] [<фиксация>]

Эта форма сбрасывает текущий заголовок ветки на <commit> и, возможно, обновляет индекс (сбрасывая его на дерево <commit> ) и рабочее дерево в зависимости от <mode> . Перед операцией ORIG_HEAD устанавливается на кончик текущей ветки. Если <mode> опущен, по умолчанию используется --mixed . <mode> должен быть одним из следующих:

--soft

Вообще не трогает индексный файл или рабочее дерево (но сбрасывает голову на <commit> , как и все режимы). При этом все ваши измененные файлы останутся "Changes to be committed" , как выразился бы git status .

--mixed

Сбрасывает индекс, но не рабочее дерево (т. е. измененные файлы сохраняются, но не помечаются для фиксации) и сообщает, что не было обновлено. Это действие по умолчанию.

Если указан -N , удаленные пути помечаются как добавляемые (см. git-add[1] ).

--hard

Сбрасывает индекс и рабочее дерево. Любые изменения в отслеживаемых файлах в рабочем дереве, начиная с <commit> , отбрасываются. Любые неотслеживаемые файлы или каталоги на пути записи любых отслеживаемых файлов просто удаляются.

--merge

Сбрасывает индекс и обновляет файлы в рабочем дереве, которые различаются между <commit> и HEAD , но сохраняет те, которые различаются между индексом и рабочим деревом (т. е. имеют изменения, которые не были добавлены). Если файл, который отличается между <commit> и индексом, имеет неустановленные изменения, сброс прерывается.

Другими словами, --merge делает что-то вроде git read-tree -u -m <commit> , но переносит неслитые записи индекса.

--keep

Сбрасывает записи индекса и обновляет файлы в рабочем дереве, которые различаются между <commit> и HEAD . Если файл, который отличается между <commit> и HEAD , имеет локальные изменения, сброс прерывается.

--[no-]recurse-submodules

Когда рабочее дерево обновляется, использование --recurse-submodules также рекурсивно сбрасывает рабочее дерево всех активных подмодулей в соответствии с коммитом, записанным в суперпроекте, а также устанавливает отсоединение HEAD подмодулей при этом коммите.

См. "Reset, restore and revert" в git[1] , чтобы узнать о различиях между тремя командами.

Options

-q
--quiet

Молчите, сообщайте только об ошибках.

--refresh
--no-refresh

Обновите индекс после смешанного сброса. Включено по умолчанию.

--pathspec-from-file=<file>

Спецификация пути передается в <file> вместо аргументов командной строки. Если <file> — это точно - , то используется стандартный ввод. Элементы Pathspec разделяются LF или CR/LF. Элементы Pathspec можно заключать в кавычки, как описано для переменной конфигурации core.quotePath (см. git-config[1] ). См. также --pathspec-file-nul и глобальный --literal-pathspecs .

--pathspec-file-nul

Имеет смысл только с --pathspec-from-file . Элементы Pathspec разделяются символом NUL, а все остальные символы воспринимаются буквально (включая символы новой строки и кавычки).

--

Не интерпретируйте дополнительные аргументы как варианты.

<pathspec>…

Ограничивает пути, на которые влияет операция.

Дополнительные сведения см. в записи pathspec в gitglossary[7] .

Examples

Undo add
$ edit                                     (1)
$ git add frotz.c filfre.c
$ mailx                                    (2)
$ git reset                                (3)
$ git pull git://info.example.com/ nitfol  (4)

  1. Вы с удовольствием над чем-то работаете и обнаруживаете, что изменения в этих файлах в порядке. Вы не хотите видеть их при запуске git diff , поскольку вы планируете работать с другими файлами и изменения в этих файлах отвлекают.

  2. Кто-то просит вас потянуть, и изменения кажутся достойными слияния.

  3. Однако вы уже испортили индекс (т.е. ваш индекс не соответствует коммиту HEAD ). Но вы знаете, что вытягивание, которое вы собираетесь сделать, не влияет на frotz.c или filfre.c , поэтому вы отменяете изменения индекса для этих двух файлов. Ваши изменения в рабочем дереве остаются там.

  4. Затем вы можете извлечь и объединить, оставив изменения frotz.c и filfre.c в рабочем дереве.

Отменить фиксацию и повторить
$ git commit ...
$ git reset --soft HEAD^      (1)
$ edit                        (2)
$ git commit -a -c ORIG_HEAD  (3)

  1. Чаще всего это делается, когда вы помните, что то, что вы только что зафиксировали, является неполным, или вы допустили ошибку в сообщении о фиксации, или и то, и другое. Оставляет рабочее дерево таким, каким оно было до "reset" .

  2. Внесите исправления в файлы рабочих деревьев.

  3. "reset" копирует старую головку в .git/ORIG_HEAD ; повторить фиксацию, начав с ее сообщения в журнале. Если вам не нужно дальше редактировать сообщение, вы можете вместо этого указать опцию -C .

См. также вариант --amend для git-commit[1] .

Отменить коммит, сделав его тематической веткой
$ git branch topic/wip          (1)
$ git reset --hard HEAD~3       (2)
$ git switch topic/wip          (3)

  1. Вы сделали некоторые коммиты, но понимаете, что их появление в ветке master было преждевременным. Вы хотите продолжить их доработку в тематической ветке, поэтому создайте ветку topic/wip на основе текущей HEAD .

  2. Перемотайте ветку master, чтобы избавиться от этих трёх коммитов.

  3. Переключитесь на ветку topic/wip и продолжайте работу.

Отменить фиксацию навсегда
$ git commit ...
$ git reset --hard HEAD~3   (1)

  1. Последние три коммита ( HEAD , HEAD^ и HEAD~2 ) были плохими, и вы не хотите видеть их снова. Не делайте этого, если вы уже передали эти коммиты кому-то другому. (См. раздел "RECOVERING FROM UPSTREAM REBASE" в git-rebase[1] , чтобы узнать о последствиях этого.)

Отменить слияние или извлечение
$ git pull                         (1)
Auto-merging nitfol
CONFLICT (content): Merge conflict in nitfol
Automatic merge failed; fix conflicts and then commit the result.
$ git reset --жесткий (2)
$ git pull . topic/branch          (3)
Updating from 41223... to 13134...
Fast-forward
$ git reset --hard ORIG_HEAD (4)

  1. Попытка обновления из апстрима привела к множеству конфликтов; вы не готовы тратить много времени на слияние прямо сейчас, поэтому решаете сделать это позже.

  2. "pull" не выполнил фиксацию слияния, поэтому git reset --hard , который является синонимом git reset --hard HEAD , удаляет беспорядок из индексного файла и рабочего дерева.

  3. Объедините ветку темы с текущей веткой, что привело к перемотке вперед.

  4. Но вы решили, что ветка темы еще не готова к использованию public . "pull" или "merge" всегда оставляет исходную вершину текущей ветки в ORIG_HEAD , поэтому жесткий сброс возвращает ваш индексный файл и рабочее дерево обратно в это состояние и сбрасывает вершину ветки до этой фиксации.

Отменить слияние или втянуть внутрь грязного рабочего дерева
$ git pull                         (1)
Auto-merging nitfol
Merge made by recursive.
 nitfol                |   20 +++++----
 ...
$ git reset --объединить ORIG_HEAD (2)

  1. Даже если у вас могут быть локальные изменения в вашем рабочем дереве, вы можете смело говорить git pull , когда знаете, что изменение в другой ветке не пересекается с ними.

  2. Проверив результат слияния, вы можете обнаружить, что изменение в другой ветке неудовлетворительно. Запуск git reset --hard ORIG_HEAD позволит вам вернуться туда, где вы были, но отменит ваши локальные изменения, которые вам не нужны. git reset --merge сохраняет ваши локальные изменения.

Interrupted workflow

Предположим, вас прервал запрос на срочное исправление, когда вы находитесь в процессе большого изменения. Файлы в вашем рабочем дереве еще не готовы к фиксации, но вам нужно перейти в другую ветку для быстрого исправления ошибки.

$ git switch feature  ;# вы работали в ветке  "feature"  и
$ work work work      ;# меня прервали
$ git commit -a -m "snapshot WIP"                 (1)
$ git switch master
$ fix fix fix
$ git commit ;# фиксируем с реальным журналом
$ git switch feature
$ git reset --soft HEAD^ ;# вернуться в состояние WIP (2)
$ git reset                                       (3)

  1. Этот коммит будет уничтожен, поэтому выброшенное сообщение журнала — это нормально.

  2. Это удалит фиксацию WIP из истории фиксации и установит ваше рабочее дерево в состояние непосредственно перед созданием этого моментального снимка.

  3. На этом этапе индексный файл все еще содержит все незавершенные изменения, которые вы зафиксировали как snapshot WIP . Это обновит индекс, чтобы ваши файлы WIP отображались как незафиксированные.

См. также git-stash[1] .

Сбросить один файл в индексе

Предположим, вы добавили файл в свой индекс, но позже решили, что не хотите добавлять его в свой коммит. Вы можете удалить файл из индекса, сохранив изменения, выполнив сброс git .

$ git reset -- frotz.c                      (1)
$ git commit -m "Commit files in index"     (2)
$ git add frotz.c                           (3)

  1. Это удалит файл из индекса, сохранив его в рабочем каталоге.

  2. Это фиксирует все остальные изменения в индексе.

  3. Снова добавляет файл в индекс.

Сохраняйте изменения в рабочем дереве, отбрасывая при этом некоторые предыдущие коммиты.

Предположим, вы работаете над чем-то и фиксируете это, а затем продолжаете работать еще немного, но теперь думаете, что то, что у вас есть в вашем рабочем дереве, должно находиться в другой ветке, которая не имеет ничего общего с тем, что вы зафиксировали ранее. Вы можете запустить новую ветку и сбросить ее, сохранив изменения в рабочем дереве.

$ git tag start
$ git switch -c branch1
$ edit
$ git commit ...                            (1)
$ edit
$ git switch -c branch2                     (2)
$ git reset --keep start                    (3)

  1. Это зафиксирует ваши первые изменения в branch1 .

  2. В идеальном мире вы могли бы понять, что предыдущий коммит не принадлежал к новой теме, когда вы создали и переключились на branch2 (т. е. git switch -c branch2 start ), но никто не идеален.

  3. Но вы можете использовать reset --keep для удаления нежелательной фиксации после перехода на branch2 .

Разделить коммит на последовательность коммитов

Предположим, вы создали множество логически отдельных изменений и зафиксировали их вместе. Затем позже вы решаете, что, возможно, было бы лучше, чтобы каждый логический фрагмент был связан с отдельным коммитом. Вы можете использовать сброс git , чтобы перемотать историю без изменения содержимого ваших локальных файлов, а затем последовательно использовать git add -p для интерактивного выбора, какие фрагменты включать в каждый коммит, используя git commit -c для предварительного заполнения сообщения о коммите.

$ git reset -N HEAD^                        (1)
$ git add -p                                (2)
$ git diff --cached                         (3)
$ git commit -c HEAD@{1}                    (4)
...                                         (5)
$ git add ...                               (6)
$ git diff --cached                         (7)
$ git commit ...                            (8)

  1. Сначала сбросим историю на один коммит, чтобы мы удалили исходный коммит, но оставили рабочее дерево со всеми изменениями. Параметр -N гарантирует, что любые новые файлы, добавленные с помощью HEAD , по-прежнему будут помечены, чтобы git add -p мог их найти.

  2. Затем мы в интерактивном режиме выбираем фрагменты различий для добавления с помощью средства git add -p . При этом вам будет последовательно задаваться вопрос о каждом фрагменте различий, и вы сможете использовать простые команды, такие как "yes, include this" , "No don’t include this" или даже очень мощную функцию "edit" .

  3. Если вы удовлетворены фрагментами, которые хотите включить, вам следует проверить, что было подготовлено для первого коммита, с помощью git diff --cached . Здесь показаны все изменения, которые были перенесены в индекс и скоро будут зафиксированы.

  4. Затем зафиксируйте изменения, хранящиеся в индексе. Опция -c указывает на предварительное заполнение сообщения фиксации из исходного сообщения, с которого вы начали первую фиксацию. Это полезно, чтобы избежать повторного ввода. HEAD@{1} — это специальная запись для фиксации, на которой находился HEAD до исходной фиксации сброса (1 изменение назад). Дополнительную информацию см. в git-reflog[1] . Вы также можете использовать любую другую действительную ссылку на фиксацию.

  5. Вы можете повторить шаги 2–4 несколько раз, чтобы разбить исходный код на любое количество коммитов.

  6. Теперь вы разделили многие изменения на отдельные коммиты и можете больше не использовать режим исправлений git add , чтобы выбрать все оставшиеся незафиксированные изменения.

  7. Еще раз проверьте, включили ли вы то, что хотите. Вы также можете убедиться, что в файле diff git нет оставшихся изменений, которые необходимо зафиксировать позже.

  8. И, наконец, создайте финальный коммит.

Discussion

В таблицах ниже показано, что происходит во время работы:

git reset --option target

чтобы сбросить HEAD на другую фиксацию ( target ) с различными параметрами сброса в зависимости от состояния файлов.

В этих таблицах A , B , C и D — это несколько разных состояний файла. Например, первая строка первой таблицы означает, что если файл находится в состоянии A в рабочем дереве, в состоянии B в индексе, в состоянии C в HEAD и в состоянии D в целевом, то git reset --soft target покинет файл. в рабочем дереве в состоянии A и в индексе в состоянии B . Он сбрасывает (т.е. перемещает) HEAD (т.е. кончик текущей ветки, если вы находитесь на ней) в target (файл которого находится в состоянии D ).

----------------------------------------------------
 A       B     C    D     --soft   A       B     D
                          --mixed  A       D     D
                          --hard   D       D     D
                          --merge (disallowed)
                          --keep  (disallowed)
working index HEAD target         working index HEAD
----------------------------------------------------
 A       B     C    C     --soft   A       B     C
                          --mixed  A       C     C
                          --hard   C       C     C
                          --merge (disallowed)
                          --keep   A       C     C
working index HEAD target         working index HEAD
----------------------------------------------------
 B       B     C    D     --soft   B       B     D
                          --mixed  B       D     D
                          --hard   D       D     D
                          --merge  D       D     D
                          --keep  (disallowed)
working index HEAD target         working index HEAD
----------------------------------------------------
 B       B     C    C     --soft   B       B     C
                          --mixed  B       C     C
                          --hard   C       C     C
                          --merge  C       C     C
                          --keep   B       C     C
working index HEAD target         working index HEAD
----------------------------------------------------
 B       C     C    D     --soft   B       C     D
                          --mixed  B       D     D
                          --hard   D       D     D
                          --merge (disallowed)
                          --keep  (disallowed)
working index HEAD target         working index HEAD
----------------------------------------------------
 B       C     C    C     --soft   B       C     C
                          --mixed  B       C     C
                          --hard   C       C     C
                          --merge  B       C     C
                          --keep   B       C     C

reset --merge предназначен для использования при сбросе после конфликтного слияния. Любая операция слияния гарантирует, что файл рабочего дерева, который участвует в слиянии, не имеет локальных изменений по отношению к индексу до его запуска, и что он записывает результат в рабочее дерево. Поэтому, если мы видим некоторую разницу между индексом и целью, а также между индексом и рабочим деревом, то это означает, что мы не выходим из состояния, в котором операция слияния осталась после неудачи с конфликтом. Вот почему в этом случае мы запрещаем опцию --merge .

reset --keep предназначен для использования при удалении некоторых последних коммитов в текущей ветке с сохранением изменений в рабочем дереве. Если могут возникнуть конфликты между изменениями в коммите, который мы хотим удалить, и изменениями в рабочем дереве, которые мы хотим сохранить, сброс не допускается. Вот почему это запрещено, если есть изменения как между рабочим деревом и HEAD , так и между HEAD и целью. В целях безопасности это также запрещено при наличии несвязанных записей.

В следующих таблицах показано, что происходит при наличии несвязанных записей:

working index HEAD target         working index HEAD
----------------------------------------------------
 X       U     A    B     --soft  (disallowed)
                          --mixed  X       B     B
                          --hard   B       B     B
                          --merge  B       B     B
                          --keep  (disallowed)
