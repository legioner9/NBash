git-merge-base

Name

git-merge-base — найти как можно более хороших общих предков для слияния

Synopsis

git merge-base [-a | --all] <фиксация> <фиксация>…​
git merge-base [-a | --all] --octopus <фиксация>…​
git merge-base --is-ancestor <фиксация> <фиксация>
git merge-base --independent <commit>…​
git merge-base --fork-point <ref> [<commit>]

Description

git merge-base находит наилучшего общего предка между двумя фиксациями для использования в трехстороннем слиянии. Один общий предок - это better , а другой общий предок, если последний является предком первого. Общий предок, у которого нет лучшего общего предка, — это best common ancestor , то есть merge base . Обратите внимание, что для пары коммитов может быть более одной базы слияния.

Operation modes

В наиболее распространенном частном случае указание только двух коммитов в командной строке означает вычисление базы слияния между данными двумя коммитами.

В более общем смысле, среди двух коммитов, на основе которых вычисляется база слияния, один указывается первым аргументом коммита в командной строке; другой коммит — это (возможно, гипотетический) коммит, который представляет собой слияние всех остальных коммитов в командной строке.

Как следствие, merge base не обязательно содержится в каждом аргументе фиксации, если указано более двух коммитов. Он отличается от git-show-branch[1] при использовании с опцией --merge-base .

--octopus

Вычислите лучших общих предков всех предоставленных коммитов для подготовки к n-стороннему слиянию. Это имитирует поведение git show-branch --merge-base .

--independent

Вместо печати баз слияния распечатайте минимальное подмножество предоставленных коммитов с теми же предками. Другими словами, среди данных коммитов перечислите те, до которых невозможно добраться ни из одного другого. Это имитирует поведение git show-branch --independent .

--is-ancestor

Проверьте, является ли первый <commit> предком второго <commit>, и выйдите со статусом 0, если это правда, или со статусом 1, если нет. Об ошибках сигнализирует ненулевой статус, отличный от 1.

--fork-point

Найдите точку, в которой ветвь (или любая история, ведущая к <commit>) отделилась от другой ветки (или любой ссылки) <ref>. При этом не просто ищется общий предок двух коммитов, но также учитывается рефлог <ref>, чтобы увидеть, не возникла ли история, ведущая к <commit>, от более раннего воплощения ветки <ref> (см. обсуждение этот режим ниже).

Options

-a
--all

Выведите все базы слияния для коммитов вместо одной.

Discussion

Учитывая два коммита A и B , git merge-base A B выведет коммит, который доступен как из A , так и из B через родительское отношение.

Например, с этой топологией:

         o---o---o---B
        /
---o---1---o---o---o---A

базой слияния A и B является 1 .

Учитывая три коммита A , B и C , git merge-base A B C вычислит базу слияния между A и гипотетическим коммитом M , который представляет собой слияние B и C . Например, с этой топологией:

       o---o---o---o---C
      /
     /   o---o---o---B
    /   /
---2---1---o---o---o---A

результат git merge-base A B C — 1 . Это связано с тем, что эквивалентная топология с фиксацией слияния M между B и C :

       o---o---o---o---o
      /                 \
     /   o---o---o---o---M
    /   /
---2---1---o---o---o---A

и результат git merge-base A M — 1 . Commit 2 также является общим предком между A и M , но 1 является лучшим общим предком, поскольку 2 является предком 1 . Следовательно, 2 не является базой слияния.

Результатом git merge-base --octopus A B C является 2 , поскольку 2 — лучший общий предок всех коммитов.

Когда история включает перекрестные слияния, для двух коммитов может быть более одного общего предка best . Например, с этой топологией:

---1---o---A
    \ /
     X
    / \
---2---o---o---B

и 1 , и 2 являются базами слияния A и B. Ни одна из них не лучше другой (оба являются базами слияния best ). Если опция --all не указана, не указано, какой из них будет лучшим.

Распространенной идиомой проверки "fast-forward-ness" между двумя коммитами A и B является (или, по крайней мере, раньше) вычисление базы слияния между A и B и проверка, совпадает ли она с A, и в этом случае A является предком Б. Вы увидите, что эта идиома часто используется в старых сценариях.

A=$(git rev-parse --verify A)
if test "$A" = "$(git merge-base A B)"
then
        ... A is an ancestor of B ...
fi

В современном git это можно сказать более прямо:

if git merge-base --is-ancestor A B
then
        ... A is an ancestor of B ...
fi

instead.

Обсуждение режима разветвления

После работы над веткой topic , созданной с помощью git switch -c topic origin/master , история ветки удаленного отслеживания origin/master могла быть перемотана и перестроена, что привело к истории такой формы:

                 o---B2
                /
---o---o---B1--o---o---o---B (origin/master)
        \
         B0
          \
           D0---D1---D (topic)

где origin/master раньше указывал на коммиты B0, B1, B2, а теперь указывает на B, и ваша ветка topic была запущена поверх нее еще тогда, когда origin/master был на B0, и вы создали три коммита, D0, D1 и D, на самое верхнее. Представьте, что теперь вы хотите перенести работу, которую вы проделали по теме, поверх обновленного источника/мастера.

В таком случае git merge-base origin/master topic вернет родителя B0 на рисунке выше, но B0^..D — это не тот диапазон коммитов, который вы хотели бы воспроизвести поверх B (он включает B0, а это не то, что вы написали; это фиксация, отброшенная другой стороной при перемещении кончика с B0 на B1).

git merge-base --fork-point origin/master topic призван помочь в таком случае. Он принимает во внимание не только B, но также B0, B1 и B2 (т. е. старые подсказки ветвей удаленного отслеживания, о которых знает рефлог вашего репозитория), чтобы увидеть, на каком коммите была построена ваша тематическая ветка, и находит B0, позволяя вам воспроизводить только коммиты по вашей теме, за исключением коммитов, которые другая сторона позже отбросила.

Hence

$ fork_point=$(git merge-base --fork-point origin/master topic)

найдет B0, и

$ git rebase --onto origin/master $fork_point topic

будет воспроизводить D0, D1 и D поверх B, чтобы создать новую историю этой формы:

                 o---B2
                /
---o---o---B1--o---o---o---B (origin/master)
        \                   \
         B0                  D0'--D1'--D' (тема - обновлена)
          \
           D0---D1---D (тема - старая)

Предостережение: срок действия старых записей журнала репозитория в вашем репозитории может истекать к моменту git gc . Если B0 больше не появляется в рефлоге ветки удаленного отслеживания origin/master , режим --fork-point , очевидно, не может его найти и завершается сбоем, избегая выдачи случайного и бесполезного результата (например, родительского элемента B0, как и та же команда без опции --fork-point ). дает).

Кроме того, ветвь удаленного отслеживания, с которой вы используете режим --fork-point , должна быть той, от которой ваша тема развилась. Если вы выполнили форк из более старого коммита, чем совет, этот режим не найдет точку форка (представьте, что в приведенном выше примере истории B0 не существовало, источник/мастер начинался с B1, перемещался в B2, а затем в B, и вы разветвили свою тему. в origin/master^, когда origin/master был B1; форма истории будет такой же, как указано выше, без B0, а родительским элементом B1 является то, что git merge-base origin/master topic правильно находит, но режим --fork-point не будет, потому что он не один коммитов, которые раньше находились на вершине источника/мастера).
