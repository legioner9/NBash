git-merge

Name

git-merge — объединить две или более истории разработки вместе

Synopsis

git merge [-n] [--stat] [--no-commit] [--squash] [--[no-]edit]
        [--no-verify] [-s <strategy>] [-X <strategy-option>] [-S[<keyid>]]
        [--[no-]allow-unrelated-histories]
        [--[no-]rerere-autoupdate] [-m <msg>] [-F <file>]
        [--into-name <branch>] [<commit>…​]
git merge (--continue | --abort | --quit)

Description

Включает изменения из именованных коммитов (с момента, когда их история отклонилась от текущей ветки) в текущую ветку. Эта команда используется git pull для включения изменений из другого репозитория и может использоваться вручную для объединения изменений из одной ветки в другую.

Предположим, существует следующая история и текущая ветка — master :

          A---B---C topic
         /
    D---E---F---G master

Затем git merge topic воспроизведет изменения, внесенные в ветку topic с момента ее расхождения с master (т. е. E ) до текущего фиксации ( C ) поверх master , и запишет результат в новую фиксацию вместе с именами двух родительских коммитов. и сообщение журнала от пользователя, описывающее изменения. Перед операцией ORIG_HEAD устанавливается на кончик текущей ветки ( C ).

          A---B---C topic
         /         \
    D---E---F---G---H master

Слияние прекращается, если возникает конфликт, который невозможно разрешить автоматически, или если при запуске слияния был указан --no-commit . На этом этапе вы можете запустить git merge --abort или git merge --continue .

git merge --abort прервет процесс слияния и попытается восстановить состояние до слияния. Однако если на момент начала слияния были незафиксированные изменения (и особенно если эти изменения были дополнительно изменены после начала слияния), git merge --abort в некоторых случаях не сможет восстановить исходные изменения (до слияния). Поэтому:

Предупреждение. Запуск git merge с нетривиальными незафиксированными изменениями не рекомендуется: хотя это и возможно, это может оставить вас в состоянии, из которого будет трудно выйти в случае конфликта.

Options

--commit
--no-commit

Выполните слияние и зафиксируйте результат. Эту опцию можно использовать для переопределения --no-commit.

С --no-commit выполните слияние и остановите его непосредственно перед созданием фиксации слияния, чтобы дать пользователю возможность проверить и дополнительно настроить результат слияния перед фиксацией.

Обратите внимание, что ускоренные обновления не создают фиксации слияния, и поэтому невозможно остановить эти слияния с помощью --no-commit. Таким образом, если вы хотите гарантировать, что ваша ветка не будет изменена или обновлена ​​командой слияния, используйте --no-ff с --no-commit.

--edit
-e
--no-edit

Перед выполнением успешного механического слияния вызовите редактор для дальнейшего редактирования автоматически созданного сообщения о слиянии, чтобы пользователь мог объяснить и обосновать слияние. Опцию --no-edit можно использовать для приема автоматически сгенерированного сообщения (обычно это не рекомендуется). Параметр --edit (или -e ) по-прежнему полезен, если вы отправляете черновик сообщения с параметром -m из командной строки и хотите отредактировать его в редакторе.

Старые сценарии могут зависеть от исторического поведения, запрещающего пользователю редактировать сообщение журнала слияния. Они увидят открытый редактор при запуске git merge . Чтобы упростить настройку таких сценариев на обновленное поведение, в их начале для переменной среды GIT_MERGE_AUTOEDIT можно установить значение no .

--cleanup=<mode>

Эта опция определяет, как будет очищено сообщение о слиянии перед фиксацией. Дополнительную информацию см. в git-commit[1] . Кроме того, если <mode> присвоено значение scissors , ножницы будут добавлены к MERGE_MSG перед передачей механизму фиксации в случае конфликта слияния.

--ff
--no-ff
--ff-only

Указывает, как обрабатывается слияние, если объединенная история уже является потомком текущей истории. --ff используется по умолчанию, если не происходит объединение аннотированного (и, возможно, подписанного) тега, который не хранится на своем естественном месте в иерархии refs/tags/ , и в этом случае предполагается --no-ff .

С помощью --ff по возможности разрешайте слияние как ускоренную перемотку вперед (обновляйте только указатель ветки, чтобы он соответствовал объединенной ветке; не создавайте фиксацию слияния). Если это невозможно (когда объединенная история не является потомком текущей истории), создайте фиксацию слияния.

С помощью --no-ff создайте фиксацию слияния во всех случаях, даже если вместо этого слияние можно было разрешить как ускоренную перемотку вперед.

С помощью --ff-only по возможности разрешайте слияние как ускоренную перемотку вперед. Если это невозможно, откажитесь от слияния и выйдите с ненулевым статусом.

-S[<keyid>]
--gpg-sign[=<keyid>]
--no-gpg-sign

Подпишите GPG полученный коммит слияния. Аргумент keyid является необязательным и по умолчанию соответствует идентификатору коммиттера; если указано, оно должно быть прикреплено к опции без пробела. --no-gpg-sign полезен для отмены как переменной конфигурации commit.gpgSign , так и более ранней версии --gpg-sign .

--log[=<n>]
--no-log

Помимо названий ветвей, заполните сообщение журнала однострочными описаниями не более чем из <n> фактических коммитов, которые объединяются. См. также git-fmt-merge-msg[1] .

С помощью --no-log не перечисляйте однострочные описания фактических объединяемых коммитов.

--signoff
--no-signoff

Добавьте трейлер Signed-off-by от коммиттера в конце сообщения журнала фиксации. Значение подписания зависит от проекта, за который вы беретесь. Например, он может удостоверить, что коммиттер имеет права представить работу по лицензии проекта или согласен на какое-либо представление участника, такое как Сертификат происхождения разработчика. (См. https://developercertificate.org , который используется в ядре Linux и проектах Git .) Обратитесь к документации или руководству проекта, в который вы вносите свой вклад, чтобы понять, как подписи используются в этом проекте.

Опцию --no-signoff можно использовать для отмены более ранней опции --signoff в командной строке.

--stat
-n
--no-stat

Покажите diffstat в конце слияния. diffstat также контролируется параметром конфигурации merge.stat.

С -n или --no-stat не показывать статистику различий в конце слияния.

--squash
--no-squash

Создайте рабочее дерево и состояние индекса, как если бы произошло реальное слияние (за исключением информации о слиянии), но на самом деле не делайте фиксацию, не перемещайте HEAD и не записывайте $GIT_DIR/MERGE_HEAD (чтобы заставить следующую команду git commit создать фиксацию слияния). Это позволяет вам создать один коммит поверх текущей ветки, эффект которого аналогичен слиянию другой ветки (или большего количества в случае осьминога).

С помощью --no-squash выполните слияние и зафиксируйте результат. Эту опцию можно использовать для переопределения --squash.

С --squash --commit не разрешен и завершится ошибкой.

--[no-]verify

По умолчанию запускаются перехватчики pre-merge и commit-msg. Когда задан --no-verify , они обходятся. См. также githooks[5] .

-s <strategy>
--strategy=<strategy>

Используйте данную стратегию слияния; могут быть предоставлены более одного раза, чтобы указать их в том порядке, в котором они должны быть опробованы. Если опция -s отсутствует, вместо нее используется встроенный список стратегий ( ort при объединении одной головы, octopus в противном случае).

-X <option>
--strategy-option=<option>

Передайте конкретную опцию стратегии слияния в стратегию слияния.

--verify-signatures
--no-verify-signatures

Убедитесь, что фиксация объединяемой боковой ветки подписана действительным ключом, т. е. ключом с действительным идентификатором пользователя: в модели доверия по умолчанию это означает, что ключ подписи подписан доверенным ключом. Если кончик фиксации боковой ветки не подписан действительным ключом, слияние прерывается.

--summary
--no-summary

Синонимы --stat и --no-stat; они устарели и будут удалены в будущем.

-q
--quiet

Работайте тихо. Подразумевается --no-progress.

-v
--verbose

Be verbose.

--progress
--no-progress

Включите/выключите прогресс явно. Если ни один из них не указан, прогресс отображается, если к терминалу подключена стандартная ошибка. Обратите внимание, что не все стратегии слияния могут поддерживать отчеты о ходе выполнения.

--autostash
--no-autostash

Автоматически создайте временную запись в тайнике перед началом операции, запишите ее в ref MERGE_AUTOSTASH и примените после завершения операции. Это означает, что вы можете запустить операцию на «грязном» рабочем дереве. Однако используйте с осторожностью: финальное приложение stash после успешного слияния может привести к нетривиальным конфликтам.

--allow-unrelated-histories

По умолчанию команда git merge отказывается объединять истории, не имеющие общего предка. Эту опцию можно использовать для обхода этой безопасности при объединении историй двух проектов, которые начали свою жизнь независимо. Поскольку это очень редкий случай, переменная конфигурации, позволяющая включить это по умолчанию, не существует и не будет добавлена.

-m <msg>

Установите сообщение фиксации, которое будет использоваться для фиксации слияния (в случае, если оно создано).

Если указан --log , к указанному сообщению будет добавлен краткий журнал объединяемых коммитов.

Команду git fmt-merge-msg можно использовать для задания подходящего значения по умолчанию для автоматических вызовов git merge . Автоматизированное сообщение может включать описание филиала.

--into-name <branch>

Подготовьте сообщение о слиянии по умолчанию, как при слиянии с веткой <branch> , вместо имени реальной ветки, с которой производится слияние.

-F <file>
--file=<file>

Прочтите сообщение фиксации, которое будет использоваться для фиксации слияния (в случае, если оно создано).

Если указан --log , к указанному сообщению будет добавлен краткий журнал объединяемых коммитов.

--rerere-autoupdate
--no-rerere-autoupdate

После того, как механизм rerere повторно использует записанное разрешение текущего конфликта для обновления файлов в рабочем дереве, разрешите ему также обновить индекс с результатом разрешения. --no-rerere-autoupdate — хороший способ перепроверить то, что сделал rerere , и выявить возможные ошибки слияния, прежде чем заносить результат в индекс с помощью отдельного git add .

--overwrite-ignore
--no-overwrite-ignore

Тихо перезаписать игнорируемые файлы из результата слияния. Это поведение по умолчанию. Используйте --no-overwrite-ignore для прерывания.

--abort

Прервите текущий процесс разрешения конфликта и попытайтесь восстановить состояние до слияния. Если запись autostash присутствует, примените ее к рабочему дереву.

Если при запуске слияния присутствовали незафиксированные изменения рабочего дерева, git merge --abort в некоторых случаях не сможет восстановить эти изменения. Поэтому рекомендуется всегда фиксировать или сохранять изменения перед запуском git merge .

git merge --abort эквивалентен git reset --merge , когда присутствует MERGE_HEAD , если только MERGE_AUTOSTASH не присутствует также, и в этом случае git merge --abort применяет запись тайника к рабочему дереву, тогда как git reset --merge сохраняет спрятанные изменения в списке тайника.

--quit

Забудьте о текущем слиянии. Оставьте индекс и рабочее дерево как есть. Если присутствует MERGE_AUTOSTASH , запись тайника будет сохранена в списке тайника.

--continue

После остановки git merge из-за конфликта вы можете завершить слияние, запустив git merge --continue (см. раздел "HOW TO RESOLVE CONFLICTS" ниже).

<commit>…

Обязывает, обычно других руководителей филиалов, присоединиться к нашему филиалу. Указание более одного коммита создаст слияние с более чем двумя родительскими объектами (ласково называемое слиянием Octopus).

Если в командной строке не получена фиксация, объедините ветки удаленного отслеживания, которые текущая ветка настроена для использования в качестве восходящей ветки. См. также раздел конфигурации на этой странице руководства.

Если указан FETCH_HEAD (и никакая другая фиксация), ветки, записанные в файле .git/FETCH_HEAD при предыдущем вызове git fetch для слияния, объединяются с текущей ветвью.

Pre-merge checks

Прежде чем применять внешние изменения, вы должны привести свою собственную работу в хорошую форму и зафиксировать ее локально, чтобы она не была испорчена в случае возникновения конфликтов. См. также git-stash[1] . git pull и git merge остановятся, ничего не делая, если локальные незафиксированные изменения перекрываются с файлами, которые git pull / git merge , возможно, потребуется обновить.

Чтобы избежать записи несвязанных изменений в фиксации слияния, git pull и git merge также прерывают работу, если в индексе зарегистрированы какие-либо изменения относительно фиксации HEAD . (Особые узкие исключения из этого правила могут существовать в зависимости от того, какая стратегия слияния используется, но обычно индекс должен соответствовать HEAD.)

Если все именованные коммиты уже являются предками HEAD , git merge выйдет раньше с сообщением "Already up to date." .

Fast-forward merge

Часто текущий глава ветки является предком именованного коммита. Это наиболее распространенный случай, особенно при вызове из git pull : вы отслеживаете вышестоящий репозиторий, не внесли никаких локальных изменений и теперь хотите обновиться до более новой исходной версии. В этом случае новый коммит для хранения объединенной истории не требуется; вместо этого HEAD (вместе с индексом) обновляется, чтобы указать на именованный коммит, без создания дополнительного коммита слияния.

Такое поведение можно подавить с помощью опции --no-ff .

True merge

За исключением быстрого слияния (см. выше), объединяемые ветки должны быть связаны вместе коммитом слияния, в котором они обе являются родительскими.

Объединенная версия, согласующая изменения из всех объединяемых ветвей, фиксируется, и ваш HEAD , индекс и рабочее дерево обновляются до нее. В рабочем дереве можно вносить изменения, если они не перекрываются; обновление сохранит их.

Когда не очевидно, как согласовать изменения, происходит следующее:

  1. Указатель HEAD остался прежним.

  2. Ссылка MERGE_HEAD настроена на указание на другую голову ветки.

  3. Пути, которые были объединены без ошибок, обновляются как в индексном файле, так и в рабочем дереве.

  4. Для конфликтующих путей индексный файл записывает до трех версий: на этапе 1 сохраняется версия от общего предка, на этапе 2 — из HEAD , а на этапе 3 — из MERGE_HEAD (вы можете проверить этапы с помощью git ls-files -u ). Файлы рабочего дерева содержат результат операции слияния; т.е. результаты трехстороннего слияния со знакомыми маркерами конфликта <<< === >>> .

  5. Записывается ссылка с именем AUTO_MERGE , указывающая на дерево, соответствующее текущему содержимому рабочего дерева (включая маркеры конфликтов для текстовых конфликтов). Обратите внимание, что эта ссылка записывается только при использовании стратегии слияния ort (по умолчанию).

  6. Никаких других изменений не производится. В частности, локальные изменения, внесенные до начала слияния, останутся прежними, а записи индекса для них останутся прежними, т. е. будут соответствовать HEAD .

Если вы попробовали слияние, которое привело к сложным конфликтам и хотите начать все сначала, вы можете восстановить его с помощью git merge --abort .

Merging tag

При слиянии аннотированного (и, возможно, подписанного) тега Git всегда создает фиксацию слияния, даже если возможно ускоренное слияние, и шаблон сообщения о фиксации подготавливается с сообщением тега. Кроме того, если тег подписан, проверка подписи отображается в виде комментария в шаблоне сообщения. См. также git-tag[1] .

Если вы хотите просто интегрироваться с работой, ведущей к фиксации, которая помечена тегами, например, синхронизироваться с вышестоящей точкой выпуска, вы можете не захотеть делать ненужную фиксацию слияния.

В таком случае вы можете самостоятельно передать тег "unwrap" перед передачей его в git merge или передать --ff-only , когда у вас нет самостоятельной работы. например

git fetch origin
git merge v1.2.3^0
git merge --ff-only v1.2.3

Как представлены конфликты

Во время слияния файлы рабочего дерева обновляются, чтобы отразить результат слияния. Среди изменений, внесенных в версию общего предка, непересекающиеся изменения (то есть вы изменили область файла, в то время как другая сторона оставила эту область нетронутой, или наоборот) дословно включаются в окончательный результат. Однако, когда обе стороны внесли изменения в одну и ту же область, Git не может случайным образом выбрать одну сторону перед другой и просит вас решить эту проблему, оставив то, что обе стороны сделали с этой областью.

По умолчанию Git использует тот же стиль, что и программа "merge" из пакета RCS, для представления такого конфликтующего фрагмента, например:

Here are lines that are either unchanged from the common
ancestor, or cleanly resolved because only one side changed,
or cleanly resolved because both sides changed the same way.
<<<<<<< yours:sample.txt
Conflict resolution is hard;
let's go shopping.

Область, где произошла пара конфликтующих изменений, отмечена маркерами <<<<<<< , ======= и >>>>>>> . Часть перед ======= обычно является вашей стороной, а часть после — обычно их стороной.

Единственное, что вы можете сказать, это то, что ваша сторона хочет сказать, что это сложно, и вы предпочитаете ходить по магазинам, в то время как другая сторона хочет утверждать, что это легко.

Помимо маркеров <<<<<<< , ======= и >>>>>>> , он использует еще один маркер ||||||| , за которым следует исходный текст. Вы можете сказать, что оригинал просто констатировал факт, а ваша сторона просто поддалась этому утверждению и сдалась, в то время как другая сторона попыталась занять более позитивную позицию. Иногда вы можете найти лучшее разрешение, просмотрев оригинал.

Как разрешать конфликты

Увидев конфликт, вы можете сделать две вещи:

  • Решаем не объединяться. Единственные необходимые очистки — это сбросить индексный файл до фиксации HEAD для обратного 2. и очистить изменения рабочего дерева, сделанные 2. и 3.; Для этого можно использовать git merge --abort .

  • Разрешите конфликты. Git будет отмечать конфликты в рабочем дереве. Отредактируйте файлы в форму и git add внесите их в индекс. Используйте git commit или git merge --continue , чтобы заключить сделку. Последняя команда проверяет, происходит ли (прерванное) слияние перед вызовом git commit .

Разобраться в конфликте можно с помощью ряда инструментов:

  • Используйте инструмент слияния. git mergetool для запуска графического инструмента слияния, который будет работать с вами через слияние.

  • Посмотрите на различия. git diff покажет трехходовой дифференциал, подчеркивающий изменения по сравнению с версиями HEAD и MERGE_HEAD . git diff AUTO_MERGE покажет, какие изменения вы уже внесли для разрешения текстовых конфликтов.

  • Посмотрите на различия в каждой ветке. git log --merge -p <path> сначала покажет различия для версии HEAD , а затем для версии MERGE_HEAD .

  • Посмотрите оригиналы. git show :1:filename показывает общего предка, git show :2:filename показывает версию HEAD , а git show :3:filename показывает версию MERGE_HEAD .

Examples

  • Объедините ветки fixes и enhancements поверх текущей ветки, создав объединение осьминога:

    $ git merge fixes enhancements

  • Объединить ветку obsolete с текущей веткой, используя стратегию слияния ours :

    $ git merge -s ours obsolete

  • Объедините ветку maint с текущей веткой, но не делайте новый коммит автоматически:

    $ git merge --no-commit maint

    Это можно использовать, если вы хотите включить дополнительные изменения в слияние или написать собственное сообщение о фиксации слияния.

    Вам следует воздерживаться от злоупотребления этой опцией для внесения существенных изменений в коммит слияния. Небольшие исправления, такие как изменение названия выпуска/версии, будут приемлемыми.

Merge strategies

Механизм слияния (команды git merge и git pull ) позволяет выбирать серверную часть merge strategies с опцией -s . Некоторые стратегии также могут иметь свои собственные параметры, которые можно передать, передав аргументы -X<option> в git merge и/или git pull .

ort

Это стратегия слияния по умолчанию при извлечении или объединении одной ветки. Эта стратегия может разрешить только две головы, используя алгоритм трехстороннего слияния. Если существует более одного общего предка, который можно использовать для трехстороннего слияния, создается объединенное дерево общих предков и используется в качестве эталонного дерева для трехстороннего слияния. Сообщается, что это приводит к меньшему количеству конфликтов слияния, не вызывая неправильных слияний, согласно тестам, выполненным на реальных коммитах слияния, взятых из истории разработки ядра Linux 2.6. Кроме того, эта стратегия может обнаруживать и обрабатывать слияния, связанные с переименованиями. Он не использует обнаруженные копии. Название этого алгоритма является аббревиатурой ( "Ostensibly Recursive’s Twin" ) и произошло от того, что он был написан в качестве замены предыдущего алгоритма по умолчанию recursive .

Стратегия ort может принимать следующие варианты:

ours

Эта опция заставляет конфликтующие фрагменты автоматически разрешаться, отдавая предпочтение версии our . Изменения из другого дерева, которые не конфликтуют с нашей стороной, отражаются в результате слияния. Для бинарного файла все содержимое берется с нашей стороны.

Это не следует путать со стратегией слияния ours , которая вообще не рассматривает содержимое другого дерева. Он отбрасывает все, что сделало другое дерево, объявляя, что история our содержит все, что в ней произошло.

theirs

Это противоположность ours ; обратите внимание, что, в отличие от ours , не существует стратегии слияния theirs , с которой можно было бы спутать этот вариант слияния.

ignore-space-change
ignore-all-space
ignore-space-at-eol
ignore-cr-at-eol

Считает строки с указанным типом изменения пробелов неизмененными для трехстороннего слияния. Изменения пробелов, смешанные с другими изменениями в строке, не игнорируются. См. также git-diff[1] , -b , -w , --ignore-space-at-eol и --ignore-cr-at-eol .

  • Если версия their вводит в строку только изменения пробелов, используется версия our ;

  • Если версия our содержит изменения в пробелах, а версия their включает существенные изменения, используется версия their ;

  • В противном случае слияние происходит обычным способом.

renormalize

При этом выполняется виртуальное извлечение и возврат всех трех этапов файла при разрешении трехстороннего слияния. Эта опция предназначена для использования при объединении ветвей с различными чистыми фильтрами или правилами нормализации конца строки. Подробности см. в "Merging branches with differing checkin/checkout attributes" и gitattributes[5] .

no-renormalize

Отключает опцию renormalize . Это переопределяет переменную конфигурации merge.renormalize .

find-renames[=<n>]

Включите обнаружение переименования, при необходимости установив порог сходства. Это значение по умолчанию. Это переопределяет переменную конфигурации merge.renames . См. также git-diff[1] --find-renames .

rename-threshold=<n>

Устаревший синоним find-renames=<n> .

subtree[=<path>]

Этот вариант представляет собой более продвинутую форму стратегии subtree , где стратегия предполагает, как два дерева должны быть сдвинуты, чтобы они совпадали друг с другом при слиянии. Вместо этого к указанному пути добавляется префикс (или удаляется с самого начала), чтобы форма двух деревьев совпадала.

recursive

Это может разрешить только две головы, используя алгоритм трехстороннего слияния. Если существует более одного общего предка, который можно использовать для трехстороннего слияния, создается объединенное дерево общих предков и используется в качестве эталонного дерева для трехстороннего слияния. Сообщается, что это приводит к меньшему количеству конфликтов слияния, не вызывая неправильных слияний, согласно тестам, выполненным на реальных коммитах слияния, взятых из истории разработки ядра Linux 2.6. Кроме того, это может обнаруживать и обрабатывать слияния, связанные с переименованиями. Он не использует обнаруженные копии. Это была стратегия по умолчанию для разрешения двух головок от Git v0.99.9k до v2.33.0.

Стратегия recursive использует те же параметры, что и ort . Однако есть три дополнительных параметра, которые ort игнорирует (не описаны выше), которые потенциально полезны для стратегии recursive :

patience

Устаревший синоним diff-algorithm=patience .

diff-algorithm=[patience|minimal|histogram|myers]

Используйте другой алгоритм сравнения при слиянии, что поможет избежать ошибок слияния, возникающих из-за неважных совпадающих строк (например, фигурных скобок из разных функций). См. также git-diff[1] --diff-algorithm . Обратите внимание, что ort специально использует diff-algorithm=histogram , а recursive по умолчанию использует настройку конфигурации diff.algorithm .

no-renames

Отключите обнаружение переименования. Это переопределяет переменную конфигурации merge.renames . См. также git-diff[1] --no-renames .

resolve

Это может разрешить только две головы (т. е. текущую ветвь и другую ветвь, из которой вы извлекли) с использованием алгоритма трехстороннего слияния. Он пытается тщательно обнаружить неоднозначности перекрестного слияния. Он не обрабатывает переименования.

octopus

Это разрешает случаи с более чем двумя головками, но отказывается выполнять сложное слияние, требующее разрешения вручную. В первую очередь он предназначен для объединения заголовков тематических ветвей. Это стратегия слияния по умолчанию при извлечении или объединении более чем одной ветки.

ours

При этом разрешается любое количество голов, но результирующее дерево слияния всегда является деревом текущей головы ветки, эффективно игнорируя все изменения из всех других ветвей. Он предназначен для замены старой истории развития побочных ветвей. Обратите внимание, что это отличается от опции -Xours стратегии слияния recursive .

subtree

Это модифицированная стратегия ort . При слиянии деревьев A и B, если B соответствует поддереву A, B сначала настраивается так, чтобы оно соответствовало древовидной структуре A, вместо чтения деревьев на одном уровне. Эта корректировка также вносится в дерево общего предка.

В стратегиях, использующих трехстороннее слияние (включая стандартную ort ), если изменение внесено в обе ветви, но позже отменено в одной из ветвей, это изменение будет присутствовать в объединенном результате; некоторых людей такое поведение сбивает с толку. Это происходит потому, что при выполнении слияния учитываются только головы и база слияния, а не отдельные фиксации. Поэтому алгоритм слияния считает отмененное изменение вообще отсутствием изменений и вместо этого заменяет измененную версию.

Configuration

branch.<name>.mergeOptions

Устанавливает параметры по умолчанию для слияния с веткой <имя>. Синтаксис и поддерживаемые параметры такие же, как у git merge , но значения параметров, содержащие символы пробелов, в настоящее время не поддерживаются.

Все, что находится выше этой строки в этом разделе, не включено в документацию git-config[1] . Следующее содержимое такое же, как и найденное там:

merge.conflictStyle

Укажите стиль, в котором конфликтующие фрагменты записываются в файлы рабочего дерева при слиянии. По умолчанию используется "merge" , который показывает маркер конфликта <<<<<<< , изменения, внесенные одной стороной, маркер ======= , изменения, внесенные другой стороной, а затем маркер >>>>>>> . Альтернативный стиль "diff3" добавляет маркер ||||||| и исходный текст перед маркером ======= . Стиль "merge" имеет тенденцию создавать меньшие конфликтные области, чем diff3, как из-за исключения исходного текста, так и потому, что когда подмножество строк совпадает с двух сторон, они просто вытягиваются из конфликтной области. Другой альтернативный стиль, "zdiff3" , похож на diff3, но удаляет совпадающие линии с двух сторон конфликтной области, когда эти совпадающие линии появляются либо в начале, либо в конце конфликтной области.

merge.defaultToUpstream

Если слияние вызывается без какого-либо аргумента фиксации, объедините восходящие ветки, настроенные для текущей ветки, используя их последние наблюдаемые значения, хранящиеся в их ветвях удаленного отслеживания. Значения branch.<current branch>.merge , которые называют ветки на удаленном компьютере, названные branch.<current branch>.remote , анализируются, а затем они сопоставляются через remote.<remote>.fetch с соответствующими ветвями удаленного отслеживания, а кончики этих ветвей отслеживания объединяются. По умолчанию истинно.

merge.ff

По умолчанию Git не создает дополнительную фиксацию слияния при слиянии фиксации, которая является потомком текущей фиксации. Вместо этого кончик текущей ветки перематывается вперед. Если установлено значение false , эта переменная сообщает Git о необходимости создания дополнительной фиксации слияния в таком случае (эквивалентно указанию опции --no-ff из командной строки). Если установлено значение only , разрешены только такие быстрые слияния (эквивалентно указанию параметра --ff-only из командной строки).

merge.verifySignatures

Если это правда, это эквивалентно параметру командной строки --verify-signatures. Подробности см. в git-merge[1] .

merge.branchdesc

Помимо имен ветвей, заполните сообщение журнала текстом описания ветки, связанным с ними. По умолчанию ложь.

merge.log

Помимо имен ветвей, заполните сообщение журнала не более чем указанным количеством однострочных описаний из фактических коммитов, которые объединяются. По умолчанию установлено значение false, а true является синонимом 20.

merge.suppressDest

При добавлении glob, который соответствует именам ветвей интеграции, к этой многозначной переменной конфигурации, сообщение слияния по умолчанию, вычисляемое для слияний с этими ветвями интеграции, будет опускать "into <branch name>" в своем заголовке.

Элемент с пустым значением можно использовать для очистки списка glob, накопленного из предыдущих записей конфигурации. Если переменная merge.suppressDest не определена, для обратной совместимости используется значение по умолчанию master .

merge.renameLimit

Количество файлов, которые следует учитывать при исчерпывающей проверке переименования во время слияния. Если не указано, по умолчанию используется значение diff.renameLimit. Если не указаны ни merge.renameLimit, ни diff.renameLimit, в настоящее время по умолчанию используется значение 7000. Этот параметр не действует, если обнаружение переименования отключено.

merge.renames

Определяет ли Git переименования. Если установлено значение "false" , обнаружение переименования отключено. Если установлено значение "true" , включено базовое обнаружение переименования. По умолчанию используется значение diff.renames.

merge.directoryRenames

Определяет ли Git переименование каталогов, влияющее на то, что происходит во время слияния с новыми файлами, добавленными в каталог на одной стороне истории, когда этот каталог был переименован на другой стороне истории. Если для merge.directoryRenames установлено значение "false" , обнаружение переименования каталога отключено, а это означает, что такие новые файлы останутся в старом каталоге. Если установлено значение "true" , включено обнаружение переименования каталога, что означает, что такие новые файлы будут перемещены в новый каталог. Если установлено значение "conflict" , для таких путей будет сообщаться о конфликте. Если merge.renames имеет значение false, merge.directoryRenames игнорируется и рассматривается как false. По умолчанию "conflict" .

merge.renormalize

Сообщите Git , что каноническое представление файлов в репозитории со временем изменилось (например, более ранние фиксации записывают текстовые файлы с окончаниями строк CRLF, но в последних используются окончания строк LF). В таком репозитории Git может преобразовать данные, записанные в коммитах, в каноническую форму перед выполнением слияния, чтобы уменьшить ненужные конфликты. Дополнительную информацию см. в разделе "Merging branches with differing checkin/checkout attributes" в gitattributes[5] .

merge.stat

Следует ли печатать разницу между ORIG_HEAD и результатом слияния в конце слияния. True по умолчанию.

merge.autoStash

Если установлено значение true, автоматически создается запись временного тайника перед началом операции и применяется ее после завершения операции. Это означает, что вы можете запустить слияние на «грязном» рабочем дереве. Однако используйте с осторожностью: финальное приложение stash после успешного слияния может привести к нетривиальным конфликтам. Эту опцию можно переопределить опциями --no-autostash и --autostash для git-merge[1] . По умолчанию ложь.

merge.tool

Управляет тем, какой инструмент слияния используется git-mergetool[1] . В списке ниже показаны допустимые встроенные значения. Любое другое значение рассматривается как пользовательский инструмент слияния и требует определения соответствующей переменной mergetool.<tool>.cmd.

merge.guitool

Управляет тем, какой инструмент слияния используется git-mergetool[1] , если указан флаг -g/--gui. В списке ниже показаны допустимые встроенные значения. Любое другое значение рассматривается как пользовательский инструмент слияния и требует определения соответствующей переменной mergetool.<guitool>.cmd.

merge.verbosity

Управляет объемом вывода, отображаемого стратегией рекурсивного слияния. Уровень 0 не выводит ничего, кроме окончательного сообщения об ошибке, если обнаружены конфликты. Уровень 1 выводит только конфликты, 2 выводит конфликты и изменения файлов. Уровень 5 и выше выводит отладочную информацию. По умолчанию используется уровень 2. Может быть переопределен переменной среды GIT_MERGE_VERBOSITY .

merge.<driver>.name

Определяет удобочитаемое имя для пользовательского низкоуровневого драйвера слияния. Подробности см. в gitattributes[5] .

merge.<driver>.driver

Определяет команду, реализующую пользовательский низкоуровневый драйвер слияния. Подробности см. в gitattributes[5] .

merge.<driver>.recursive

Называет низкоуровневый драйвер слияния, который будет использоваться при выполнении внутреннего слияния между общими предками. Подробности см. в gitattributes[5] .
