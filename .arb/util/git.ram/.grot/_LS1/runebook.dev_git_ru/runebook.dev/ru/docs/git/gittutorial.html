gittutorial

Name

gittutorial — вводное руководство по Git

Synopsis

git *

Description

В этом руководстве объясняется, как импортировать новый проект в Git, вносить в него изменения и делиться изменениями с другими разработчиками.

Если вместо этого вы в первую очередь заинтересованы в использовании Git для получения проекта, например, для тестирования последней версии, вы можете начать с первых двух глав The Git User’s Manual .

Во-первых, обратите внимание, что вы можете получить документацию для такой команды, как git log --graph , с помощью:

$ man git-log

or:

$ git help log

В последнем случае вы можете использовать ручной просмотрщик по вашему выбору; дополнительную информацию см. в git-help[1] .

Перед выполнением какой-либо операции рекомендуется представиться Git , указав свое имя и адрес электронной почты public . Самый простой способ сделать это:

$ git config --global user.name  "Your Name Comes Here"
$ git config --global user.email [email protected]

Импорт нового проекта

Предположим, у вас есть архив project.tar.gz с вашей первоначальной работой. Вы можете поместить его под контроль версий Git следующим образом.

$ tar xzf project.tar.gz
$ cd project
$ git init

Git ответит

Initialized empty Git repository in .git/

Теперь вы инициализировали рабочий каталог — вы можете заметить, что создан новый каталог с именем .git .

Затем скажите Git сделать снимок содержимого всех файлов в текущем каталоге (обратите внимание на . ), используя git add :

$ git add .

Этот снимок теперь хранится во временной промежуточной области, которую Git называет "index" . Вы можете постоянно хранить содержимое индекса в репозитории с помощью git commit :

$ git commit

Это предложит вам сообщение о фиксации. Теперь вы сохранили первую версию своего проекта в Git.

Making changes

Измените некоторые файлы, а затем добавьте их обновленное содержимое в индекс:

$ git add file1 file2 file3

Теперь вы готовы взять на себя обязательства. Вы можете увидеть, что будет зафиксировано, используя git diff с опцией --cached :

$ git diff --cached

(Без --cached git diff покажет вам любые изменения, которые вы внесли, но еще не добавили в индекс.) Вы также можете получить краткое описание ситуации с git status :

$ git status
On branch master
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)

        modified:   file1
        modified:   file2
        modified:   file3

Если вам нужно внести какие-либо дополнительные изменения, сделайте это сейчас, а затем добавьте в индекс все вновь измененное содержимое. Наконец, зафиксируйте изменения с помощью:

$ git commit

При этом вам снова будет предложено ввести сообщение с описанием изменения, а затем записать новую версию проекта.

Альтернативно, вместо предварительного запуска git add , вы можете использовать

$ git commit -a

который автоматически обнаружит любые измененные (но не новые) файлы, добавит их в индекс и зафиксирует, и все это за один шаг.

Примечание о сообщениях о фиксации: Хотя это и не обязательно, рекомендуется начинать сообщение о фиксации с одной короткой (не более 50 символов) строки, в которой суммируется изменение, за которой следует пустая строка, а затем более подробное описание. Текст до первой пустой строки в сообщении о фиксации рассматривается как заголовок фиксации, и этот заголовок используется во всем Git. Например, git-format-patch[1] преобразует коммит в электронное письмо и использует заголовок в строке «Тема», а остальную часть коммита — в теле.

Git отслеживает контент, а не файлы

Многие системы контроля версий предоставляют команду add , которая сообщает системе начать отслеживать изменения в новом файле. Команда Git add делает нечто более простое и мощное: git add используется как для новых, так и для вновь измененных файлов, и в обоих случаях она делает снимок данных файлов и помещает это содержимое в индекс, готовое для включения в следующий коммит.

Просмотр истории проекта

В любой момент вы можете просмотреть историю ваших изменений, используя

$ git log

Если вы также хотите видеть полные различия на каждом этапе, используйте

$ git log -p

Часто обзор изменений полезен, чтобы прочувствовать каждый шаг.

$ git log --stat --summary

Managing branches

Один репозиторий Git может поддерживать несколько ветвей разработки. Чтобы создать новую ветку с именем experimental , используйте

$ git branch experimental

Если вы сейчас запустите

$ git branch

вы получите список всех существующих ветвей:

  experimental
* master

Ветка experimental — это та, которую вы только что создали, а ветка master — это ветка по умолчанию, которая была создана для вас автоматически. Звездочкой отмечена ветка, в которой вы сейчас находитесь; тип

$ git switch experimental

для перехода на ветку experimental . Теперь отредактируйте файл, зафиксируйте изменения и вернитесь к ветке master :

(edit file)
$ git commit -a
$ git switch master

Убедитесь, что внесенное вами изменение больше не отображается, поскольку оно было сделано в ветке experimental , а вы вернулись в ветку master .

Вы можете внести другие изменения в ветку master :

(edit file)
$ git commit -a

на этом этапе две ветви разошлись, в каждой из которых были внесены разные изменения. Чтобы объединить изменения, внесенные в experimental , в master , запустите

$ git merge experimental

Если изменения не конфликтуют, все готово. Если есть конфликты, в проблемных файлах останутся маркеры, показывающие конфликт;

$ git diff

это покажет. После того как вы отредактировали файлы для разрешения конфликтов,

$ git commit -a

зафиксирует результат слияния. Окончательно,

$ gitk

покажет красивое графическое представление полученной истории.

На этом этапе вы можете удалить ветку experimental с помощью

$ git branch -d experimental

Эта команда гарантирует, что изменения в ветке experimental уже находятся в текущей ветке.

Если будете разрабатывать на ветке crazy-idea , то пожалеете, ветку всегда можно удалить с помощью

$ git branch -D crazy-idea

Филиалы — это дешево и просто, так что это хороший способ что-то попробовать.

Использование git для совместной работы

Предположим, что Алиса начала новый проект с репозиторием Git в /home/alice/project , и что Боб, у которого есть домашний каталог на той же машине, хочет внести свой вклад.

Боб начинает с:

bob$ git clone /home/alice/project myrepo

При этом создается новый каталог myrepo , содержащий клон репозитория Алисы. Клон находится на равных с оригинальным проектом, обладая собственной копией истории оригинального проекта.

Затем Боб вносит некоторые изменения и фиксирует их:

(edit files)
bob$ git commit -a
(repeat as necessary)

Когда он будет готов, он приказывает Алисе извлечь изменения из репозитория по адресу /home/bob/myrepo . Она делает это с помощью:

alice$ cd /home/alice/project
alice$ git pull /home/bob/myrepo master

Это объединит изменения из ветки master Боба с текущей веткой Алисы. Если Алиса за это время внесла свои собственные изменения, возможно, ей придется вручную исправить любые конфликты.

Таким образом, команда pull выполняет две операции: извлекает изменения из удаленной ветки, а затем объединяет их с текущей веткой.

Обратите внимание, что в целом Алиса хотела бы, чтобы ее локальные изменения были зафиксированы до запуска этого pull . Если работа Боба противоречит тому, что делала Алиса с момента разветвления их историй, Алиса будет использовать свое рабочее дерево и индекс для разрешения конфликтов, а существующие локальные изменения будут мешать процессу разрешения конфликтов (Git все равно выполнит выборку, но откажется от слияния — Алисе придется каким-то образом избавиться от своих локальных изменений и сделать это снова, когда это произойдет).

Алиса может просмотреть, что сделал Боб, без предварительного слияния, используя команду fetch ; это позволяет Алисе проверить, что сделал Боб, используя специальный символ FETCH_HEAD , чтобы определить, есть ли у него что-нибудь стоящее, например:

alice$ git fetch /home/bob/myrepo master
alice$ git log -p HEAD..FETCH_HEAD

Эта операция безопасна, даже если у Алисы есть незафиксированные локальные изменения. Обозначение диапазона HEAD..FETCH_HEAD означает "show everything that is reachable from the FETCH_HEAD but exclude anything that is reachable from HEAD " . Алиса уже знает все, что приводит к ее текущему состоянию ( HEAD ), и просматривает с помощью этой команды то, что Боб имеет в своем состоянии ( FETCH_HEAD ), чего она еще не видела.

Если Алиса хочет визуализировать то, что сделал Боб с момента разделения их историй, она может ввести следующую команду:

$ gitk HEAD..FETCH_HEAD

Здесь используется то же обозначение диапазона из двух точек, которое мы видели ранее с git log .

Алиса может захотеть просмотреть, что они оба сделали после форка. Она может использовать форму с тремя точками вместо формы с двумя точками:

$ gitk HEAD...FETCH_HEAD

Это означает "show everything that is reachable from either one, but exclude anything that is reachable from both of them" .

Обратите внимание, что эти обозначения диапазонов могут использоваться как с gitk , так и с git log .

После проверки того, что сделал Боб, если нет ничего срочного, Алиса может решить продолжить работу, не отрываясь от Боба. Если в истории Боба действительно есть что-то, что немедленно понадобится Алисе, Алиса может сначала спрятать свою незавершенную работу, выполнить pull , а затем, наконец, извлечь свою незавершенную работу поверх результирующей истории.

Когда вы работаете в небольшой сплоченной группе, нет ничего необычного в том, чтобы снова и снова взаимодействовать с одним и тем же репозиторием. Определив сокращение репозитория remote , вы можете упростить задачу:

alice$ git remote add bob /home/bob/myrepo

Благодаря этому Алиса может выполнить первую часть операции pull самостоятельно, используя команду git fetch , не объединяя их со своей собственной веткой, используя:

alice$ git fetch bob

В отличие от полной формы, когда Алиса получает данные от Боба, используя сокращенную версию удаленного репозитория, настроенную с помощью git remote , то, что было получено, сохраняется в ветке удаленного отслеживания, в данном случае bob/master . Итак, после этого:

alice$ git log -p master..bob/master

показывает список всех изменений, которые Боб сделал с тех пор, как он отделился от ветки master Алисы.

Изучив эти изменения, Алиса могла бы объединить их в свою ветку master :

alice$ git merge bob/master

Этот merge также можно сделать с помощью pulling from her own remote-tracking branch , вот так:

alice$ git pull . remotes/bob/master

Обратите внимание, что git pull всегда объединяется с текущей веткой, независимо от того, что еще указано в командной строке.

Позже Боб может обновить свой репозиторий последними изменениями Алисы, используя

bob$ git pull

Обратите внимание, что ему не нужно указывать путь к репозиторию Алисы; когда Боб клонировал репозиторий Алисы, Git сохранила местоположение своего репозитория в конфигурации репозитория, и это местоположение используется для извлечения:

bob$ git config --get remote.origin.url
/home/alice/project

(Полную конфигурацию, созданную git clone , можно увидеть с помощью git config -l , а на странице руководства git-config[1] объясняется значение каждой опции.)

Git также хранит нетронутую копию ветки master Алисы под именем origin/master :

bob$ git branch -r
  origin/master

Если позже Боб решит работать с другого хоста, он все равно сможет выполнять клонирование и извлечение данных, используя протокол ssh:

bob$ git clone alice.org:/home/alice/project myrepo

Альтернативно, Git имеет собственный протокол или может использовать http; подробности см. в git-pull[1] .

Git также можно использовать в режиме, подобном CVS, с центральным репозиторием, в который различные пользователи отправляют изменения; см. git-push[1] и gitcvs-migration[7] .

Exploring history

История Git представлена ​​как серия взаимосвязанных коммитов. Мы уже видели, что команда git log может вывести список этих коммитов. Обратите внимание, что первая строка каждой записи git log также содержит имя фиксации:

$ git log
commit c82a22c39cbc32576f64f5c6b3f24b99ea8149c7
Author: Junio C Hamano <junkio@cox.net>
Date:   Tue May 16 17:18:22 2006 -0700

    merge-base: Clarify the comments on post processing.

Мы можем дать это имя git show , чтобы увидеть подробности об этом коммите.

$ git show c82a22c39cbc32576f64f5c6b3f24b99ea8149c7

Но есть и другие способы ссылки на коммиты. Вы можете использовать любую начальную часть имени, достаточно длинную, чтобы однозначно идентифицировать коммит:

$ git show c82a22c39c        # первые несколько символов имени
                        # обычно достаточно
$ git show HEAD                # кончик текущей ветки
$ git show experimental        #кончик ветки  "experimental"

Каждый коммит обычно имеет один коммит "parent" , который указывает на предыдущее состояние проекта:

$ git show HEAD^  # чтобы увидеть родителя HEAD
$ git show HEAD^^ # чтобы увидеть предка HEAD
$ git show HEAD~4 # чтобы увидеть прапрадедушку HEAD

Обратите внимание, что коммиты слияния могут иметь более одного родителя:

$ git show HEAD^1 # показать первого родителя HEAD (так же, как HEAD^)
$ git show HEAD^2 # показать второго родителя HEAD

Вы также можете дать коммитам собственные имена; после бега

$ git tag v2.5 1b2e1d63ff

вы можете обратиться к 1b2e1d63ff по имени v2.5 . Если вы собираетесь поделиться этим именем с другими людьми (например, для идентификации версии выпуска), вам следует создать объект "tag" и, возможно, подписать его; подробности см. в git-tag[1] .

Любая команда Git , которой необходимо знать фиксацию, может принимать любое из этих имен. Например:

$ git diff v2.5 HEAD         # сравниваем текущий HEAD с версией 2.5
$ git branch stable v2.5 # запускаем новую ветку с именем  "stable"  на основе
                         # в версии 2.5
$ git reset --hard HEAD^ # сбросьте текущую ветку и работайте
                         # каталог в его состояние HEAD^

Будьте осторожны с последней командой: она не только потеряет все изменения в рабочем каталоге, но и удалит все последующие коммиты из этой ветки. Если эта ветка — единственная ветка, содержащая эти коммиты, они будут потеряны. Кроме того, не используйте git reset в общедоступной ветке, из которой берутся другие разработчики, так как это заставит других разработчиков выполнять ненужные слияния для очистки истории. Если вам нужно отменить внесенные вами изменения, используйте вместо этого git revert .

Команда git grep может искать строки в любой версии вашего проекта, поэтому

$ git grep "hello" v2.5

ищет все вхождения "hello" в v2.5 .

Если вы пропустите имя фиксации, git grep будет искать любые файлы, которыми он управляет, в вашем текущем каталоге. Так

$ git grep "hello"

— это быстрый способ поиска только тех файлов, которые отслеживаются Git.

Многие команды Git также принимают наборы коммитов, которые можно указать разными способами. Вот несколько примеров с git log :

$ git log v2.5..v2.6            # коммитов между v2.5 и v2.6
$ git log v2.5..                # коммитов начиная с версии 2.5
$ git log --since="2 weeks ago" # коммитов за последние 2 недели
$ git log v2.5.. Makefile       # коммиты начиная с версии 2.5, которые изменяют
                                # Makefile

Вы также можете передать git log коммиты "range" , где первый не обязательно является предком второго; например, если вершины ветвей stable и master некоторое время назад разошлись от общего коммита, то

$ git log stable..master

будет перечислены коммиты, сделанные в ветке master , но не в стабильной ветке, а

$ git log master..stable

покажет список коммитов, сделанных в стабильной ветке, но не в ветке master .

У команды git log есть недостаток: она должна представлять коммиты в списке. Когда в истории есть линии развития, которые расходились, а затем снова сливались вместе, порядок, в котором git log представляет эти коммиты, не имеет смысла.

В большинстве проектов с несколькими участниками (например, ядро ​​Linux или сам Git ) часто происходят слияния, и gitk лучше справляется с визуализацией их истории. Например,

$ gitk --since="2 weeks ago" drivers/

позволяет просматривать любые коммиты за последние 2 недели, в которых были изменены файлы в каталоге drivers . (Примечание: вы можете настроить шрифты gitk, удерживая клавишу управления и одновременно нажимая "-" или "+" .)

Наконец, большинство команд, принимающих имена файлов, при необходимости позволяют вам предварять любое имя файла коммитом, чтобы указать конкретную версию файла:

$ git diff v2.5:Makefile HEAD:Makefile.in

Вы также можете использовать git show , чтобы увидеть любой такой файл:

$ git show v2.5:Makefile

Next steps

Этого руководства должно быть достаточно для выполнения базового распределенного контроля версий в ваших проектах. Однако, чтобы полностью понять глубину и мощь Git , вам необходимо понять две простые идеи, на которых он основан:

  • База данных объектов — это довольно элегантная система, используемая для хранения истории вашего проекта — файлов, каталогов и коммитов.

  • Индексный файл представляет собой кэш состояния дерева каталогов, используемый для создания коммитов, проверки рабочих каталогов и хранения различных деревьев, участвующих в слиянии.

Во второй части этого руководства описываются объектная база данных, индексный файл и некоторые другие мелочи, которые вам понадобятся, чтобы максимально эффективно использовать Git. Вы можете найти его по адресу gittutorial-2[7] .

Если вы не хотите сразу продолжать эту тему, вот несколько других отступлений, которые могут быть интересны на этом этапе:

  • git-format-patch[1] , git-am[1] : они преобразуют серии коммитов git в патчи, отправляемые по электронной почте, и наоборот, что полезно для таких проектов, как ядро ​​​​Linux, которые в значительной степени полагаются на патчи, отправляемые по электронной почте.

  • git-bisect[1] : Если в вашем проекте наблюдается регрессия, один из способов отследить ошибку — просмотреть историю, чтобы найти точный коммит, который виноват. git bisect может помочь вам выполнить двоичный поиск этого коммита. Он достаточно умен, чтобы выполнять поиск, близкий к оптимальному, даже в случае сложной нелинейной истории с множеством объединенных ветвей.

  • gitworkflows[7] : Дает обзор рекомендуемых рабочих процессов.

  • giteveryday[7] : Git на каждый день с примерно 20 командами.

  • gitcvs-migration[7] : Git для пользователей CVS.
