git-fast-export

Name

git-fast-export — экспортер данных Git

Synopsis

git fast-export [<options>] | git fast-import

Description

Эта программа сохраняет данные версии в форме, подходящей для передачи в git fast-import .

Вы можете использовать его как удобочитаемую замену пакета (см. git-bundle[1] ) или как формат, который можно редактировать перед отправкой в ​git fast-import для перезаписи истории (возможность, на которую опираются такие инструменты, как git filter-repo ).

Options

--progress=<n>

Вставьте операторы progress в каждый объект <n>, который будет отображаться git fast-import во время импорта.

--signed-tags=(verbatim|warn|warn-strip|strip|abort)

Укажите, как обрабатывать подписанные теги. Поскольку любое преобразование после экспорта может изменить имена тегов (что также может произойти при исключении ревизий), подписи не будут совпадать.

При запросе abort (по умолчанию) эта программа закроется при обнаружении подписанного тега. При использовании strip теги будут автоматически делаться беззнаковыми, при использовании warn-strip они будут делаться беззнаковыми, но будет отображаться предупреждение, при использовании verbatim они будут экспортироваться автоматически, а при использовании warn они будут экспортироваться, но вы увидите предупреждение.

--tag-of-filtered-object=(abort|drop|rewrite)

Укажите, как обрабатывать теги, помеченный объект которых отфильтровывается. Поскольку версии и файлы для экспорта могут быть ограничены по пути, помеченные объекты могут быть полностью отфильтрованы.

При запросе abort (который стоит по умолчанию) эта программа умрет при встрече с таким тегом. При использовании drop такие теги будут пропущены в выводе. При использовании rewrite , если помеченный объект является фиксацией, он перезапишет тег, чтобы пометить фиксацию предка (посредством перезаписи родителя; см. git-rev-list[1] ).

-M
-C

Выполните обнаружение перемещения и/или копирования, как описано на странице руководства git-diff[1] , и используйте его для генерации команд переименования и копирования в выходном дампе.

Обратите внимание, что более ранние версии этой команды не выдавали жалоб и не давали неверных результатов, если вы указали эти параметры.

--export-marks=<file>

После завершения выгружает таблицу внутренних меток в <file>. Метки записываются по одной в строке как :markid SHA-1 . Сбрасываются только оценки за доработки; отметки для больших двоичных объектов игнорируются. Серверные части могут использовать этот файл для проверки импорта после его завершения или для сохранения таблицы оценок при дополнительных запусках. Поскольку <file> открывается и усекается только при завершении, тот же путь можно безопасно указать --import-marks. Файл не будет записан, если ни один новый объект не был отмечен/экспортирован.

--import-marks=<file>

Перед обработкой любого ввода загрузите метки, указанные в <file>. Входной файл должен существовать, быть читаемым и иметь тот же формат, что и --export-marks.

--mark-tags

Помимо маркировки BLOB-объектов и коммитов идентификаторами меток, также маркируйте теги. Это полезно в сочетании с --export-marks и --import-marks , а также полезно (и необходимо) для экспорта вложенных тегов. В других случаях это не повредит и будет использоваться по умолчанию, но многие интерфейсы быстрого импорта не готовы принимать теги с идентификаторами меток.

Любые уже отмеченные коммиты (или теги) не будут экспортированы повторно. Если серверная часть использует аналогичный файл --import-marks, это позволяет выполнять инкрементальный двунаправленный экспорт репозитория, сохраняя одинаковые метки при каждом запуске.

--fake-missing-tagger

В некоторых старых репозиториях есть теги без тегера. Протокол быстрого импорта был довольно строгим в этом отношении и не позволял этого. Итак, подделайте теггер, чтобы иметь возможность быстро импортировать выходные данные.

--use-done-feature

Запустите поток с помощью раздела feature done и завершите его командой done .

--no-data

Пропустите вывод объектов BLOB-объектов и вместо этого обратитесь к BLOB-объектам через их исходный хэш SHA-1. Это полезно при перезаписи структуры каталогов или истории репозитория, не затрагивая содержимое отдельных файлов. Обратите внимание, что полученный поток может использоваться только тем репозиторием, который уже содержит необходимые объекты.

--full-tree

Эта опция приведет к тому, что быстрый экспорт будет выдавать директиву "deleteall" для каждого коммита, за которым следует полный список всех файлов в коммите (в отличие от простого перечисления файлов, которые отличаются от первого родительского коммита).

--anonymize

Анонимизируйте содержимое репозитория, сохраняя при этом форму истории и сохраненного дерева. См. раздел о ANONYMIZING ниже.

--anonymize-map=<from>[:<to>]

Преобразуйте токен <from> в <to> в анонимизированном выводе. Если <to> опущен, сопоставьте <from> с самим собой (т. е. не анонимизируйте его). См. раздел о ANONYMIZING ниже.

--reference-excluded-parents

По умолчанию выполнение такой команды, как git fast-export master~5..master , не будет включать в себя мастер фиксации master~5 и приведет к тому, что master~4 больше не будет иметь master~5 в качестве родительского (хотя и старый master~4, и новый master~4 будут иметь одинаковые файлы). Используйте --reference-excluded-parents, чтобы вместо этого поток ссылался на коммиты в исключенном диапазоне истории по их sha1sum. Обратите внимание, что полученный поток может использоваться только репозиторием, который уже содержит необходимые родительские коммиты.

--show-original-ids

Добавьте в выходные данные дополнительную директиву для коммитов и больших двоичных объектов, original-oid <SHA1SUM> . Хотя такие директивы, скорее всего, будут игнорироваться импортерами, такими как git-fast-import, они могут быть полезны для промежуточных фильтров (например, для перезаписи сообщений о фиксации, которые ссылаются на старые фиксации, или для удаления больших двоичных объектов по идентификатору).

--reencode=(yes|no|abort)

Укажите, как обрабатывать заголовок encoding в объектах фиксации. При запросе abort (по умолчанию) эта программа завершится при обнаружении такого объекта фиксации. С помощью yes сообщение фиксации будет перекодировано в UTF-8. При использовании no исходная кодировка будет сохранена.

--refspec

Примените указанную спецификацию ссылки к каждой экспортированной ссылке. Их можно указать несколько.

[<git-rev-list-args>…​]

Список аргументов, приемлемый для git rev-parse и git rev-list , который определяет конкретные объекты и ссылки для экспорта. Например, master~10..master вызывает экспорт текущей главной ссылки вместе со всеми объектами, добавленными с момента ее 10-го предка, и (если не указан параметр --reference-excluded-parents) со всеми файлами, общими для master~9 и master~10.

Examples

$ git fast-export --all | (cd /empty/repository && git fast-import)

Это позволит экспортировать весь репозиторий и импортировать его в существующий пустой репозиторий. За исключением коммитов перекодирования, не относящихся к UTF-8, это будет зеркало «один к одному».

$ git fast-export master~5..master |
        sed "s|refs/heads/master|refs/heads/other|" |
        git fast-import

Это создаст новую ветку под названием other из master~5..master (т.е. если master имеет линейную историю, она возьмет последние 5 коммитов).

Обратите внимание, что это предполагает, что none больших двоичных объектов и сообщений фиксации, на которые ссылается этот диапазон ревизий, содержит строку refs/heads/master .

Anonymizing

Если указана опция --anonymize , git попытается удалить всю идентифицирующую информацию из репозитория, сохраняя при этом достаточное количество исходного дерева и шаблонов истории для воспроизведения некоторых ошибок. Цель состоит в том, чтобы ошибка git , обнаруженная в репозитории private , сохранялась в анонимизированном репозитории, и последним можно было поделиться с разработчиками git , чтобы помочь устранить ошибку.

С помощью этой опции git заменит все имена ссылок, пути, содержимое больших двоичных объектов, сообщения о фиксации и тегах, имена и адреса электронной почты в выходных данных анонимными данными. Два экземпляра одной и той же строки будут заменены эквивалентно (например, два коммита с одним и тем же автором будут иметь в выходных данных одного и того же анонимного автора, но не будут иметь сходства с исходной строкой автора). Связь между коммитами, ветвями и тегами сохраняется, а также временные метки коммитов (но сообщения о коммитах и ​​имена ссылок не имеют сходства с оригиналами). Относительный состав дерева сохраняется (например, если у вас есть корневое дерево с 10 файлами и 3 деревьями, выходные данные будут такими же), но их имена и содержимое файлов будут заменены.

Если вы считаете, что нашли ошибку git , вы можете начать с экспорта анонимизированного потока всего репозитория:

$ git fast-export --anonymize --all >anon-stream

Затем подтвердите, что ошибка сохраняется в репозитории, созданном из этого потока (многие ошибки не сохраняются, поскольку они действительно зависят от точного содержимого репозитория):

$ git init anon-repo
$ cd anon-repo
$ git fast-import <../anon-stream
$ ... test your bug ...

Если в анонимном репозитории обнаружена ошибка, возможно, стоит поделиться anon-stream вместе с обычным отчетом об ошибке. Обратите внимание, что анонимный поток очень хорошо сжимается, поэтому рекомендуется использовать его сжатие. Если вы хотите проверить поток и убедиться, что он не содержит данных private , вы можете просмотреть его непосредственно перед отправкой. Вы также можете попробовать:

$ perl -pe 's/\d+/X/g' <anon-stream | sort -u | less

который показывает все уникальные строки (с числами, преобразованными в "X" , чтобы свернуть "User 0" , "User 1" и т. д. в "User X" ). Это дает гораздо меньший результат, и обычно легко быстро убедиться, что в потоке нет данных private .

Воспроизведение некоторых ошибок может потребовать ссылки на определенные коммиты или пути, что становится затруднительным после того, как имена ссылок и пути были анонимизированы. Вы можете попросить оставить конкретный токен как есть или сопоставить его с новым значением. Например, если у вас есть ошибка, которая воспроизводится с помощью git rev-list sensitive -- secret.c , вы можете запустить:

$ git fast-export --anonymize --all \
      --anonymize-map=sensitive:foo \
      --anonymize-map=secret.c:bar.c \
      >stream

После импорта потока вы можете запустить git rev-list foo -- bar.c в анонимном репозитории.

Обратите внимание, что пути и имена ссылок разбиваются на токены по границам косой черты. Приведенная выше команда анонимизирует subdir/secret.c как что-то вроде path123/bar.c ; затем вы можете выполнить поиск bar.c в анонимном репозитории, чтобы определить окончательный путь.

Чтобы упростить ссылку на окончательный путь, вы можете сопоставить каждый компонент пути; поэтому, если вы также анонимизируете subdir в publicdir , окончательный путь будет publicdir/bar.c .

Limitations

Поскольку git fast-import не может помечать деревья, вы не сможете полностью экспортировать репозиторий linux.git, поскольку он содержит тег, ссылающийся на дерево, а не фиксацию.
