gitworkflows

Name

gitworkflows — обзор рекомендуемых рабочих процессов с Git .

Synopsis

git *

Description

В этом документе предпринята попытка описать и мотивировать некоторые элементы рабочего процесса, используемые для самого git.git . Многие идеи применимы в целом, хотя полный рабочий процесс редко требуется для небольших проектов с меньшим количеством задействованных людей.

Сформулируем набор rules для быстрого ознакомления, а проза пытается мотивировать каждый из них. Не всегда воспринимайте их буквально; вы должны ценить веские причины для своих действий выше, чем такие страницы руководства, как эта.

Separate changes

Как правило, вам следует попытаться разбить изменения на небольшие логические шаги и зафиксировать каждый из них. Они должны быть последовательными, работать независимо от последующих коммитов, проходить набор тестов и т. д. Это значительно упрощает процесс проверки, а историю — более полезной для последующей проверки и анализа, например, с git-blame[1] и git-bisect[1] .

Чтобы добиться этого, попробуйте с самого начала разбить свою работу на маленькие шаги. Всегда легче объединить несколько коммитов, чем разделить один большой коммит на несколько. Не бойтесь делать слишком маленькие или несовершенные шаги на этом пути. Вы всегда можете вернуться позже и отредактировать коммиты с помощью git rebase --interactive , прежде чем публиковать их. Вы можете использовать git stash push --keep-index для запуска набора тестов независимо от других незафиксированных изменений; см. раздел ПРИМЕРЫ git-stash[1] .

Managing branches

Существует два основных инструмента, которые можно использовать для включения изменений из одной ветки в другую: git-merge[1] и git-cherry-pick[1] .

Слияния имеют много преимуществ, поэтому мы стараемся решить как можно больше проблем с помощью одних лишь слияний. Сбор вишен все еще иногда бывает полезен; см. пример "Merging upwards" ниже.

Самое главное, что слияние работает на уровне ветки, а выборка — на уровне коммита. Это означает, что слияние может с одинаковой легкостью перенести изменения из 1, 10 или 1000 коммитов, что, в свою очередь, означает, что рабочий процесс намного лучше масштабируется для большого количества участников (и участников). Слияния также легче понять, поскольку коммит слияния представляет собой "promise" , в который теперь включены все изменения всех его родительских элементов.

Конечно, есть компромисс: слияния требуют более тщательного управления ветвями. В следующих подразделах обсуждаются важные моменты.

Graduation

Поскольку данная функция переходит из экспериментальной в стабильную, она также "graduates" между соответствующими ветвями программного обеспечения. git.git использует следующий integration branches :

  • maint отслеживает коммиты, которые должны войти в следующий "maintenance release" , т. е. обновление последней выпущенной стабильной версии;

  • master отслеживает коммиты, которые должны войти в следующий выпуск;

  • next предназначен как ветка тестирования для тем, проверяемых на стабильность для мастера.

Существует четвертая официальная ветка, которая используется немного иначе:

  • seen (исправления, видимые сопровождающему) — это ветвь интеграции для вещей, которые еще не совсем готовы к включению (см. "Integration Branches" ниже).

Каждая из четырех ветвей обычно является прямым потомком той, что находится над ней.

Концептуально эта функция появляется в нестабильной ветке (обычно next или seen ) и с "graduates" по master для следующего выпуска, как только он будет признан достаточно стабильным.

Merging upwards

Однако описанный выше "downwards graduation" не может быть реализован путем фактического слияния вниз, поскольку это приведет к объединению изменений all в нестабильной ветке со стабильной. Отсюда следующее:

Правило: слияние вверх

Всегда вносите исправления в самую старую поддерживаемую ветку, которая их требует. Затем (периодически) сливайте ветви интеграции вверх друг с другом.

Это обеспечивает очень контролируемый поток исправлений. Если вы заметили, что вы применили исправление, например, к master , которое также требуется в maint , вам нужно будет выбрать его (с помощью git-cherry-pick[1] ) вниз. Это произойдет несколько раз, и не о чем беспокоиться, если вы не делаете это очень часто.

Topic branches

Для реализации любой нетривиальной функции потребуется несколько исправлений, и в течение ее существования могут быть внесены дополнительные исправления или улучшения.

Коммит всего непосредственно в ветках интеграции приводит ко многим проблемам: неправильные коммиты невозможно отменить, поэтому их приходится откатывать один за другим, что создает запутанную историю и дополнительную вероятность ошибок, если вы забываете отменить часть группы изменений. Параллельная работа смешивает изменения, создавая еще большую путаницу.

Использование "topic branches" решает эти проблемы. Название говорит само за себя, с оговоркой, вытекающей из правила "merge upwards" , приведенного выше:

Правило: Тематические ветки

Создайте боковую ветку для каждой темы (функция, исправление ошибки и т. д.). Разветвите его на самой старой ветке интеграции, с которой вы в конечном итоге захотите его объединить.

Тогда многие вещи можно будет сделать очень естественно:

  • Чтобы добавить функцию/исправление ошибки в ветку интеграции, просто объедините ее. Если за это время тема развилась дальше, объединитесь еще раз. (Обратите внимание, что вам не обязательно сначала объединить его с самой старой веткой интеграции. Например, вы можете сначала объединить исправление ошибки с next , дать ему некоторое время на тестирование и объединить его с maint , когда вы будете уверены, что оно стабильно.)

  • Если вы обнаружите, что вам нужны новые возможности из ветки other для продолжения работы над вашей темой, объедините other с topic . (Однако не делайте этого "just habitually" , см. ниже.)

  • Если вы обнаружите, что выбрали не ту ветку и хотите переместить ее "back in time" , используйте git-rebase[1] .

Обратите внимание, что последний пункт противоречит двум другим: тему, которая была объединена с другим местом, не следует перебазировать. См. раздел «ВОССТАНОВЛЕНИЕ ИЗ UpSTREAM REBASE» в git-rebase[1] .

Мы должны отметить, что "habitually" (регулярно без какой-либо реальной причины) объединяет ветку интеграции с вашими темами - и, как следствие, регулярное слияние чего-либо восходящего потока с чем-либо нижестоящим - не одобряется:

Правило: объединяйтесь с нисходящим потоком только в четко определенных точках.

Не объединяйтесь с нисходящей ветвью, кроме случаев, когда есть веская причина: изменения API восходящей ветки влияют на вашу ветку; ваша ветка больше не сливается с восходящей ветвью; и т. д.

В противном случае тема, которая была объединена, внезапно содержит более одного (хорошо разделенного) изменения. Множество возникших в результате мелких слияний сильно засорят историю. Любой, кто позже исследует историю файла, должен будет выяснить, повлияло ли это слияние на тему, находящуюся в разработке. Восходящий канал может даже случайно быть объединен с ветвью "more stable" . И так далее.

Throw-away integration

Если вы следовали последнему абзацу, у вас теперь будет много небольших тематических веток, и вы иногда будете задаваться вопросом, как они взаимодействуют. Возможно, результат их слияния даже не сработает? Но с другой стороны, мы хотим избежать их слияния где-либо, "stable" , потому что такое слияние невозможно легко отменить.

Решение, конечно, состоит в том, чтобы сделать слияние, которое мы можем отменить: слить в одноразовую ветку.

Правило: выбрасывать ветки интеграции

Чтобы протестировать взаимодействие нескольких тем, объедините их в одноразовую ветку. Вы никогда не должны основывать какую-либо работу на такой ветви!

Если вы (очень) ясно дадите понять, что эта ветка будет удалена сразу после тестирования, вы даже можете опубликовать эту ветку, например, чтобы дать тестировщикам возможность поработать с ней или другим разработчикам возможность увидеть, незавершенная работа будет совместима. У git.git есть такая официальная одноразовая ветка интеграции под названием seen .

Руководство филиалом для релиза

Предполагая, что вы используете описанный выше подход слияния, при выпуске проекта вам нужно будет выполнить некоторую дополнительную работу по управлению ветвями.

Выпуск функции создается на основе ветки master , поскольку master отслеживает фиксации, которые должны войти в следующий выпуск функции.

Предполагается, что ветка master будет надмножеством maint . Если это условие не выполняется, то maint содержит некоторые коммиты, которых нет в master . Поэтому исправления, представленные этими коммитами, не будут включены в ваш выпуск функций.

Чтобы убедиться, что master действительно является надмножеством maint , используйте журнал git :

Рецепт: убедитесь, что master является расширенным набором maint.

git log master..maint

Эта команда не должна отображать какие-либо коммиты. В противном случае проверьте master и объедините с ним maint .

Теперь можно приступить к созданию функционального релиза. Примените к кончику master метку с указанием версии выпуска:

Рецепт: Отпустить метку

git tag -s -m "Git X.Y.Z" vX.Y.Z master

Вам необходимо отправить новый тег на сервер public Git (см. "DISTRIBUTED WORKFLOWS" ниже). Это сделает тег доступным для других, отслеживающих ваш проект. Это также может вызвать перехватчик после обновления для выполнения элементов, связанных с выпуском, таких как создание архивов релиза и предварительно отформатированных страниц документации.

Аналогично, для выпуска обслуживания maint отслеживает коммиты, которые будут выпущены. Поэтому на приведенных выше шагах просто пометьте и нажмите maint , а не master .

Управление ветвью обслуживания после выпуска функции

После выпуска функции вам необходимо управлять ветвями обслуживания.

Во-первых, если вы хотите продолжать выпускать исправления обслуживания для выпуска функций, выпущенного до последнего, вам необходимо создать другую ветку для отслеживания коммитов для этого предыдущего выпуска.

Для этого текущая ветка обслуживания копируется в другую ветку, названную по номеру версии предыдущего выпуска (например, maint-XY(Z-1), где XYZ — текущий выпуск).

Рецепт: Копировать мейнт

git branch maint-X.Y.(Z-1) maint

Теперь ветку maint следует быстро перейти к недавно выпущенному коду, чтобы можно было отслеживать исправления обслуживания для текущей версии:

Рецепт: обновить maint до новой версии.

  • git checkout maint

  • git merge --ff-only master

Если слияние не удалось из-за того, что оно не является перемоткой вперед, возможно, некоторые исправления maint были пропущены в выпуске функции. Этого не произойдет, если содержимое ветвей было проверено, как описано в предыдущем разделе.

Управление ветвями для следующих и видимых после выпуска функций

После релиза фичи ветка интеграции next при желании может быть перемотана и пересобрана из верхушки master , используя сохранившиеся темы по next :

Рецепт: Перемотка назад и восстановление следующего.

  • git switch -C next master

  • git merge ai/topic_in_next1

  • git merge ai/topic_in_next2

  • …​

Преимущество этого в том, что история next будет чистой. Например, некоторые темы, объединенные в next , возможно, изначально выглядели многообещающе, но позже были признаны нежелательными или преждевременными. В таком случае тема откатывается из next но факт остается в истории, что когда-то ее слили и откатили. Воссоздавая next , вы даете еще одному воплощению таких тем возможность повторить попытку, а выпуск новой функции — хороший исторический момент для этого.

Если вы это сделаете, то у вас должно появиться объявление public о том, что next был перемотан и пересобран.

Тот же процесс перемотки и восстановления можно выполнить для seen . Объявление public не требуется, поскольку seen является одноразовой ветвью, как описано выше.

Distributed workflows

После последнего раздела вы должны знать, как управлять темами. В общем, вы будете не единственным человеком, работающим над проектом, поэтому вам придется поделиться своей работой.

Грубо говоря, есть два важных рабочих процесса: слияние и исправление. Важным отличием является то, что рабочий процесс слияния может распространять полную историю, включая слияния, а исправления — нет. Оба рабочих процесса могут использоваться параллельно: в git.git только специалисты по сопровождению подсистемы используют рабочий процесс слияния, в то время как все остальные отправляют исправления.

Обратите внимание, что сопровождающие могут налагать ограничения, такие как требования "Signed-off-by" , которым должны соответствовать все коммиты/исправления, представленные для включения. Для получения дополнительной информации обратитесь к документации вашего проекта.

Merge workflow

Рабочий процесс слияния работает путем копирования ветвей между восходящим и нисходящим потоком. Upstream может объединять вклады в официальную историю; ниже по течению основывают свою работу на официальной истории.

Для этого можно использовать три основных инструмента:

  • git-push[1] копирует ваши ветки в удаленный репозиторий, обычно в тот, который может быть прочитан всеми участвующими сторонами;

  • git-fetch[1] , копирующий удаленные ветки в ваш репозиторий; и

  • git-pull[1] , который извлекает и объединяет данные за один раз.

Обратите внимание на последний пункт. Используйте not git pull , если вы действительно не хотите объединить удаленную ветку.

Внести изменения легко:

Рецепт: Push/pull: Публикация ветвей/тем.

git push <remote> <branch> и скажите всем, откуда можно взять.

Вам все равно придется сообщать людям об этом другими способами, например по почте. (Git предоставляет git-request-pull[1] для отправки предварительно отформатированных запросов на включение вышестоящим разработчикам, чтобы упростить эту задачу.)

Если вы просто хотите получать новейшие копии ветвей интеграции, оставаться в курсе последних событий тоже легко:

Рецепт: Push/Pull: быть в курсе событий.

Используйте git fetch <remote> или git remote update , чтобы оставаться в курсе событий.

Затем просто создайте ветки тем из стабильных удаленных версий, как объяснялось ранее.

Если вы являетесь специалистом по сопровождению и хотели бы объединить тематические ветки других людей с ветками интеграции, они обычно отправят запрос на это по почте. Такой запрос выглядит

Please pull from
    <URL> <branch>

В этом случае git pull может выполнить выборку и объединение за один раз, следующим образом.

Рецепт: Push/Pull: объединение удаленных тем

git pull <URL> <branch>

Иногда у сопровождающего могут возникнуть конфликты слияния, когда он пытается получить изменения из нисходящего потока. В этом случае они могут попросить нижестоящих выполнить слияние и самостоятельно разрешить конфликты (возможно, они будут лучше знать, как их разрешать). Это один из редких случаев, когда нисходящий should сливается с восходящим.

Patch workflow

Если вы являетесь участником, который отправляет изменения в виде электронных писем, вам следует использовать ветки тем, как обычно (см. выше). Затем используйте git-format-patch[1] для создания соответствующих электронных писем (настоятельно рекомендуется форматировать их вручную, поскольку это облегчает жизнь сопровождающему).

Рецепт: format-patch/am: Публикация веток/тем

  • git format-patch -M upstream..topic , чтобы превратить их в предварительно отформатированные файлы исправлений.

  • git send-email --to=<recipient> <patches>

Дополнительные примечания по использованию см. на страницах руководства git-format-patch[1] и git-send-email[1] .

Если сопровождающий сообщает вам, что ваш патч больше не применяется к текущему исходному коду, вам придется перебазировать вашу тему (вы не можете использовать слияние, потому что вы не можете слиять форматирование патчей):

Рецепт: format-patch/am: Поддержание тем в актуальном состоянии.

git pull --rebase <URL> <branch>

Затем вы можете исправить конфликты во время перебазирования. Предположительно вы не публиковали свою тему кроме как по почте, поэтому перебазировать ее не проблема.

Если вы получили такую ​​серию патчей (как сопровождающий или, возможно, как читатель списка рассылки, которому она была отправлена), сохраните письма в файлы, создайте новую ветку темы и используйте git am для импорта коммитов:

Рецепт: format-patch/am: Импорт патчей

git am < patch

Стоит отметить одну особенность — трехстороннее слияние, которое может помочь в случае возникновения конфликтов: git am -3 будет использовать индексную информацию, содержащуюся в патчах, для определения базы слияния. Другие варианты см. в git-am[1] .
