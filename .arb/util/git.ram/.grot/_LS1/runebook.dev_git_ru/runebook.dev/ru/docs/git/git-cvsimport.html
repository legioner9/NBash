git-cvsimport

Name

git-cvsimport — спасение ваших данных от другого SCM, который люди любят ненавидеть

Synopsis

git cvsimport [-o <branch-for-HEAD>] [-h] [-v] [-d <CVSROOT>]
              [-A <author-conv-file>] [-p <options-for-cvsps>] [-P <file>]
              [-C <git-repository>] [-z <fuzz>] [-i] [-k] [-u] [-s <subst>]
              [-a] [-m] [-M <regex>] [-S <regex>] [-L <commit-limit>]
              [-r <remote>] [-R] [<CVS-module>]

Description

ВНИМАНИЕ: git cvsimport использует cvsps версии 2, которая считается устаревшей; он не работает с cvsps версии 3 и более поздних. Если вы выполняете однократный импорт репозитория CVS, рассмотрите возможность использования cvs2git или cvs-fast-export .

Импортирует репозиторий CVS в Git. Он либо создаст новый репозиторий, либо постепенно импортирует его в существующий.

Разделение журнала CVS на наборы исправлений выполняется cvsps . Требуется как минимум версия 2.1.

ВНИМАНИЕ: в определенных ситуациях импорт приводит к неверным результатам. Дополнительную информацию см. в разделе ISSUES .

Никогда не следует выполнять какую-либо собственную работу с ветвями, созданными git cvsimport . По умолчанию первоначальный импорт создаст и заполнит ветку "master" из основной ветки репозитория CVS, с которой вы можете свободно работать; после этого вам необходимо выполнить добавочный импорт git merge или любые ветки CVS самостоятельно. Целесообразно указать именованный удаленный сервер через -r, чтобы разделить и защитить входящие ветки.

Если вы собираетесь настроить общий репозиторий public , который все разработчики смогут читать/записывать, или если вы хотите использовать git-cvsserver[1] , то вы, вероятно, захотите создать простой клон импортированного репозитория и использовать его в качестве общего репозитория. См. gitcvs-migration[7] .

Options

-v

Подробность: пусть cvsimport сообщит, что он делает.

-d <CVSROOT>

Корень CVS-архива. Может быть локальным (простой путь) или удаленным; в настоящее время поддерживаются только методы доступа :local:, :ext: и :pserver:. Если не задано, git cvsimport попытается прочитать его из CVS/Root . Если такого файла не существует, он проверяет переменную среды CVSROOT .

<CVS-module>

Модуль CVS, который вы хотите импортировать. Относительно <CVSROOT>. Если он не указан, git cvsimport пытается прочитать его из CVS/Repository .

-C <target-dir>

Репозиторий Git для импорта. Если каталог не существует, он будет создан. По умолчанию — текущий каталог.

-r <remote>

Пульт Git для импорта этого репозитория CVS. Перемещает все ветки CVS в Remotes/<remote>/<branch> аналогично тому, как git clone использует origin по умолчанию.

-o <branch-for-HEAD>

Если удаленный доступ не указан (через -r), ветка HEAD из CVS импортируется в ветку origin в репозитории Git , поскольку HEAD уже имеет специальное значение для Git . Если указан удаленный компьютер, ветвь HEAD называется Remotes/<remote>/master, что отражает поведение git clone . Используйте эту опцию, если вы хотите импортировать в другую ветку.

Используйте -o master для продолжения импорта, который изначально был выполнен старым инструментом cvs2git.

-i

Только импорт: не выполнять оформление заказа после импорта. Эта опция гарантирует, что рабочий каталог и индекс останутся нетронутыми и не будут создавать их, если они не существуют.

-k

Ключевые слова Kill: извлечет файлы с -kk из архива CVS, чтобы избежать шумных наборов изменений. Настоятельно рекомендуется, но по умолчанию отключено, чтобы сохранить совместимость с ранними импортированными деревьями.

-u

Преобразуйте подчеркивания в именах тегов и ветвей в точки.

-s <subst>

Замените символ "/" в названиях ветвей на <subst>.

-p <options-for-cvsps>

Дополнительные опции для cvsps. Опции -u и -A являются неявными и не должны использоваться здесь.

Если вам нужно передать несколько параметров, разделите их запятой.

-z <fuzz>

Передайте в cvsps коэффициент нечеткости временной метки в секундах. Если параметр не установлен, cvsps по умолчанию равен 300 с.

-P <cvsps-output-file>

Вместо вызова cvsps прочитайте предоставленный выходной файл cvsps. Полезно для отладки или когда cvsps обрабатывается вне cvsimport.

-m

Попытайтесь обнаружить слияния на основе сообщения фиксации. Эта опция включит регулярные выражения по умолчанию, которые пытаются получить имя исходной ветки из сообщения фиксации.

-M <regex>

Попытайтесь обнаружить слияния на основе сообщения фиксации с помощью специального регулярного выражения. Его также можно использовать с -m для включения регулярных выражений по умолчанию. Вы должны избегать косых черт.

Регулярное выражение должно зафиксировать имя исходной ветки в $1.

Эту опцию можно использовать несколько раз, чтобы обеспечить несколько регулярных выражений обнаружения.

-S <regex>

Пропустить пути, соответствующие регулярному выражению.

-a

Импортируйте все коммиты, включая недавние. cvsimport по умолчанию пропускает коммиты с меткой времени менее 10 минут назад.

-L <limit>

Ограничьте количество импортируемых коммитов. Обходной путь для случаев, когда cvsimport вызывает утечку памяти.

-A <author-conv-file>

CVS по умолчанию использует имя пользователя Unix при записи журналов фиксации. Использование этой опции и файла Author-Conv сопоставляет имя, записанное в CVS, с именем автора, адресом электронной почты и дополнительным часовым поясом:

        exon=Andreas Ericsson <[email protected]>
        spawn=Simon Pawn <[email protected]> America/Chicago

git cvsimport сделает это видимым, поскольку у этих авторов все время были правильно установлены GIT_AUTHOR_NAME и GIT_AUTHOR_EMAIL. Если указан часовой пояс, к GIT_AUTHOR_DATE будет применено соответствующее смещение.

Для удобства эти данные сохраняются в $GIT_DIR/cvs-authors каждый раз, когда предоставляется опция -A , и считываются из того же файла каждый раз, когда запускается git cvsimport .

Не рекомендуется использовать эту функцию, если вы собираетесь позже снова экспортировать изменения обратно в CVS с помощью git cvsexportcommit .

-R

Создайте файл $GIT_DIR/cvs-revisions , содержащий сопоставление номеров ревизий CVS с вновь созданными идентификаторами фиксации Git . Сгенерированный файл будет содержать одну строку для каждой импортированной пары (имя файла, редакция); каждая строка будет выглядеть так

src/widget.c 1.1 1d862f173cdc7325b6fa6d2ae1cfd61fd1b512b7

Данные ревизии добавляются к файлу, если он уже существует, для использования при добавочном импорте.

Эта опция может быть полезна, если у вас есть номера ревизий CVS, хранящиеся в сообщениях о фиксации, системах отслеживания ошибок, архивах электронной почты и т.п.

-h

Распечатайте короткое сообщение об использовании и выйдите.

Output

Если указан -v , сценарий сообщает, что он делает.

В противном случае успех обозначается способом Unix, т.е. простым выходом с нулевым статусом выхода.

Issues

Проблемы, связанные с метками времени:

  • Если временные метки коммитов в репозитории CVS недостаточно стабильны, чтобы их можно было использовать для упорядочивания коммитов, изменения могут отображаться в неправильном порядке.

  • Если какие-либо файлы когда-либо были "cvs import" ed более одного раза (например, импорт выпусков более чем одного поставщика), HEAD содержит неправильное содержимое.

  • Если порядок временных меток различных файлов пересекается с порядком редакций в пределах временного окна соответствия фиксации, порядок коммитов может быть неправильным.

Проблемы, связанные с филиалами:

  • Ветки, в которых не было сделано никаких коммитов, не импортируются.

  • Все файлы из точки ветвления добавляются в ветку, даже если они никогда не добавлялись в CVS.

  • Это относится к файлам, добавленным в исходную ветку после создания дочерней ветки: если ранее в дочерней ветке не было сделано никаких коммитов, они будут ошибочно добавлены в дочернюю ветку в git.

Проблемы, связанные с тегами:

  • Несколько тегов одной и той же версии не импортируются.

Если вы подозреваете, что какая-либо из этих проблем может относиться к репозиторию, который вы хотите импортировать, рассмотрите возможность использования cvs2git:

  • cvs2git (часть cvs2svn), https://subversion.apache.org/
