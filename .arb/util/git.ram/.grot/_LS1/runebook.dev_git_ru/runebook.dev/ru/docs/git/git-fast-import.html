git-fast-import

Name

git-fast-import — серверная часть для быстрых импортеров данных Git .

Synopsis

frontend | git fast-import [<options>]

Description

Эта программа обычно не является тем, что конечный пользователь хочет запускать напрямую. Большинство конечных пользователей хотят использовать одну из существующих внешних программ, которая анализирует внешний источник определенного типа и передает хранящееся там содержимое в git fast-import .

fast-import считывает смешанный поток команд/данных из стандартного ввода и записывает один или несколько паковочных файлов непосредственно в текущий репозиторий. Когда EOF принимается на стандартный ввод, быстрый импорт записывает обновленные ссылки на ветки и теги, полностью обновляя текущий репозиторий вновь импортированными данными.

Серверная часть быстрого импорта сама может импортировать в пустой репозиторий (который уже был инициализирован git init ) или постепенно обновлять существующий заполненный репозиторий. Поддерживается ли добавочный импорт из конкретного внешнего источника, зависит от используемой внешней программы.

Options

--force

Принудительно обновить измененные существующие ветки, даже если это приведет к потере коммитов (поскольку новый коммит не содержит старый коммит).

--quiet

Отключите вывод, отображаемый --stats, чтобы быстрый импорт обычно не работал в случае успеха. Однако если поток импорта содержит директивы, предназначенные для отображения вывода пользователя (например, директивы progress ), соответствующие сообщения все равно будут отображаться.

--stats

Отображение некоторой базовой статистики об объектах, созданных быстрым импортом, пакетных файлах, в которых они были сохранены, и памяти, используемой быстрым импортом во время этого запуска. Отображение этого вывода в настоящее время используется по умолчанию, но его можно отключить с помощью --quiet.

--allow-unsafe-features

Многие параметры командной строки могут быть предоставлены как часть самого потока быстрого импорта с помощью команд feature или option . Однако некоторые из этих опций небезопасны (например, позволяют быстрому импорту получить доступ к файловой системе за пределами репозитория). Эти параметры отключены по умолчанию, но их можно разрешить, указав этот параметр в командной строке. В настоящее время это влияет только на функциональные команды export-marks , import-marks и import-marks-if-exists .

Only enable this option if you trust the program generating the
fast-import stream! This option is enabled automatically for
remote-helpers that use the `import` capability, as they are
already trusted to run their own code.

Опции для интерфейсов

--cat-blob-fd=<fd>

Записывайте ответы на запросы get-mark , cat-blob и ls в файловый дескриптор <fd> вместо stdout . Позволяет отделить вывод progress , предназначенный для конечного пользователя, от другого вывода.

--date-format=<fmt>

Укажите тип дат, которые интерфейс будет предоставлять для быстрого импорта с помощью команд author , committer и tagger . Подробную информацию о поддерживаемых форматах и ​​их синтаксисе см. ниже в разделе «Форматы дат».

--done

Завершить с ошибкой, если в конце потока нет команды done . Эта опция может быть полезна для обнаружения ошибок, из-за которых внешний интерфейс завершает работу до того, как он начал запись потока.

Расположение файлов меток

--export-marks=<file>

После завершения выгружает таблицу внутренних меток в <file>. Метки записываются по одной в строке как :markid SHA-1 . Внешние интерфейсы могут использовать этот файл для проверки импорта после его завершения или для сохранения таблицы оценок при дополнительных запусках. Поскольку <file> открывается и усекается только на контрольной точке (или завершении), тот же путь можно безопасно передать --import-marks.

--import-marks=<file>

Перед обработкой любого ввода загрузите метки, указанные в <file>. Входной файл должен существовать, быть читаемым и иметь тот же формат, что и --export-marks. Для импорта более одного набора меток могут быть предоставлены несколько опций. Если метке присвоены разные значения, побеждает последний файл.

--import-marks-if-exists=<file>

Аналогично --import-marks, но вместо ошибки автоматически пропускает файл, если он не существует.

--[no-]relative-marks

После указания --relative-marks пути, указанные с помощью --import-marks= и --export-marks=, относятся к внутреннему каталогу в текущем репозитории. В git-fast-import это означает, что пути указаны относительно каталога .git/info/fast-import. Однако другие импортеры могут использовать другое местоположение.

Относительные и неотносительные метки могут быть объединены путем переплетения --(no-)-relative-marks с опциями --(import|export)-marks=.

Submodule Rewriting

--rewrite-submodules-from=<name>:<file>
--rewrite-submodules-to=<name>:<file>

Перепишите идентификаторы объектов для подмодуля, указанного в <name>, со значений, использованных в from <file>, на значения, используемые в <file>. Метки from должны были быть созданы git fast-export , а метки to должны были быть созданы git fast-import при импорте того же подмодуля.

<имя> может быть любой произвольной строкой, не содержащей символа двоеточия, но при указании соответствующих меток с обоими параметрами должно использоваться одно и то же значение. Можно указать несколько подмодулей с разными значениями <name>. Не использовать эти опции в соответствующих парах является ошибкой.

Эти параметры в первую очередь полезны при преобразовании репозитория из одного алгоритма хеширования в другой; без них быстрый импорт завершится неудачно, если он встретит подмодуль, поскольку у него нет возможности записать идентификатор объекта в новый алгоритм хеширования.

Настройка производительности и сжатия

--active-branches=<n>

Максимальное количество ветвей, которые можно поддерживать активными одновременно. Подробности см. в разделе «Использование памяти» ниже. По умолчанию — 5.

--big-file-threshold=<n>

Максимальный размер большого двоичного объекта, для которого быстрый импорт попытается создать разность, выражается в байтах. По умолчанию — 512 м (512 МБ). Некоторые импортеры могут пожелать снизить это значение в системах с ограниченным объемом памяти.

--depth=<n>

Максимальная глубина дельта для дельтификации больших двоичных объектов и деревьев. По умолчанию 50.

--export-pack-edges=<file>

После создания файла пакета распечатайте строку данных в <file>, в которой будет указано имя файла пакета и последний коммит в каждой ветке, записанный в этот файл пакета. Эта информация может быть полезна после импорта проектов, общий набор объектов которых превышает лимит пакетного файла в 4 ГиБ, поскольку эти фиксации могут использоваться в качестве краевых точек во время вызовов git pack-objects .

--max-pack-size=<n>

Максимальный размер каждого выходного пакетного файла. По умолчанию неограничено.

fastimport.unpackLimit

See git-config[1]

Performance

Конструкция быстрого импорта позволяет импортировать большие проекты с минимальным использованием памяти и временем обработки. Если предположить, что интерфейс способен поддерживать быстрый импорт и подавать ему постоянный поток данных, время импорта для проектов с более чем 10-летней историей и содержащих более 100 000 отдельных коммитов обычно выполняется всего за 1-2 часа при довольно скромном ( ~ 2000 долларов США) аппаратное обеспечение.

Большинство узких мест, по-видимому, связаны с доступом к внешним исходным данным (источник просто не может достаточно быстро извлекать версии) или дисковым вводом-выводом (быстрый импорт записывает так быстро, как диск принимает данные). Импорт будет выполняться быстрее, если исходные данные хранятся на другом диске, чем целевой репозиторий Git (из-за меньшего количества конфликтов ввода-вывода).

Development cost

Типичный интерфейс для быстрого импорта обычно занимает около 200 строк кода Perl/Python/Ruby. Большинству разработчиков удалось создать работающие импортеры всего за пару часов, хотя это их первое знакомство с быстрым импортом, а иногда даже с Git. Это идеальная ситуация, учитывая, что большинство инструментов преобразования одноразовые (используйте один раз и никогда не оглядывайтесь назад).

Parallel operation

Как и git push или git fetch , импорт, обрабатываемый быстрым импортом, можно безопасно выполнять вместе с параллельными вызовами git repack -a -d или git gc или любой другой операцией Git (включая git prune , поскольку свободные объекты никогда не используются при быстром импорте).

fast-import не блокирует ссылки на ветки или теги, которые он активно импортирует. После импорта, на этапе обновления ссылки, быстрый импорт проверяет каждую существующую ссылку на ветку, чтобы убедиться, что обновление будет быстрым обновлением (фиксация, хранящаяся в ссылке, содержится в новой истории фиксации, которая будет записана). Если обновление не является обновлением с ускоренной перемоткой вперед, быстрый импорт пропустит обновление этой ссылки и вместо этого выведет предупреждающее сообщение. fast-import всегда будет пытаться обновить все ссылки на ветки и не останавливается при первой неудаче.

Обновления веток можно принудительно выполнить с помощью --force, но рекомендуется использовать это только в тихом репозитории. Использование --force не требуется для первоначального импорта в пустой репозиторий.

Technical discussion

fast-import отслеживает набор ветвей в памяти. Любую ветвь можно создать или изменить в любой момент процесса импорта, отправив команду commit во входной поток. Такая конструкция позволяет фронтенд-программе обрабатывать неограниченное количество ветвей одновременно, генерируя коммиты в том порядке, в котором они доступны из исходных данных. Это также значительно упрощает интерфейсные программы.

fast-import не использует и не изменяет текущий рабочий каталог или какой-либо файл в нем. (Тем не менее, он обновляет текущий репозиторий Git , как указано в GIT_DIR .) Поэтому интерфейс импорта может использовать рабочий каталог для своих целей, таких как извлечение версий файлов из внешнего источника. Такое незнание рабочего каталога также позволяет быстрому импорту выполняться очень быстро, поскольку ему не нужно выполнять какие-либо дорогостоящие операции обновления файлов при переключении между ветками.

Input format

За исключением необработанных данных файла (которые Git не интерпретирует), формат ввода быстрого импорта основан на тексте (ASCII). Этот текстовый формат упрощает разработку и отладку интерфейсных программ, особенно когда используется язык более высокого уровня, такой как Perl, Python или Ruby.

fast-import очень строг в отношении ввода. Когда мы говорим SP ниже, мы имеем в виду ровно один пробел. Аналогично, LF означает один (и только один) перевод строки, а HT — одну (и только одну) горизонтальную табуляцию. Предоставление дополнительных символов пробелов приведет к неожиданным результатам, таким как имена ветвей или имена файлов с начальными или конечными пробелами в их именах, или досрочное завершение быстрого импорта при обнаружении непредвиденного ввода.

Stream Comments

Чтобы облегчить отладку интерфейсов, быстрый импорт игнорирует любую строку, начинающуюся с # (фунт/хэш ASCII) до строки, заканчивающейся LF . Строка комментария может содержать любую последовательность байтов, не содержащую LF, и поэтому может использоваться для включения любой подробной отладочной информации, которая может быть специфичной для внешнего интерфейса и полезна при проверке потока данных быстрого импорта.

Date Formats

Поддерживаются следующие форматы даты. Интерфейсный интерфейс должен выбрать формат, который он будет использовать для этого импорта, передав имя формата в параметре командной строки --date-format=<fmt>.

raw

Это собственный формат Git и <time> SP <offutc> . Это также формат по умолчанию для быстрого импорта, если параметр --date-format не указан.

Время события определяется <time> как количество секунд с начала эпохи UNIX (полночь, 1 января 1970 г., UTC) и записывается как десятичное целое число ASCII.

Локальное смещение определяется <offutc> как положительное или отрицательное смещение от UTC. Например, EST (которое отстает от UTC на 5 часов) будет выражаться в <tz> как «-0500», а UTC — «+0000». Локальное смещение не влияет на <time> ; он используется только в качестве рекомендации, помогающей процедурам форматирования отображать временную метку.

Если локальное смещение недоступно в исходном материале, используйте «+0000» или наиболее распространенное локальное смещение. Например, во многих организациях есть репозиторий CVS, доступ к которому имеют только пользователи, находящиеся в том же месте и часовом поясе. В этом случае можно предположить разумное смещение от UTC.

В отличие от формата rfc2822 , этот формат очень строгий. Любое изменение в форматировании приведет к тому, что быстрый импорт отклонит значение, и также могут быть выполнены некоторые проверки правильности числовых значений.

raw-permissive

Это то же самое, что и raw , за исключением того, что проверки работоспособности числовой эпохи и локального смещения не выполняются. Это может быть полезно при попытке отфильтровать или импортировать существующую историю, например, с поддельными значениями часового пояса.

rfc2822

Это стандартный формат электронной почты, описанный в RFC 2822.

Пример значения: «Вторник, 6 февраля, 11:22:18 2007 -05:00». Парсер Git точен, но немного мягок. Это тот же парсер, который используется git am при применении исправлений, полученных по электронной почте.

Некоторые неверные строки могут быть приняты как допустимые даты. В некоторых из этих случаев Git все равно сможет получить правильную дату из искаженной строки. Существуют также некоторые типы неверных строк, которые Git будет неправильно анализировать, но при этом считать допустимыми. Строки с серьезными искажениями будут отклонены.

В отличие от формата raw , описанного выше, информация о часовом поясе/сдвиге UTC, содержащаяся в строке даты RFC 2822, используется для корректировки значения даты в формате UTC перед сохранением. Поэтому важно, чтобы эта информация была максимально точной.

Если в исходном материале используются даты в стиле RFC 2822, интерфейс должен позволять быстрому импорту выполнять синтаксический анализ и преобразование (вместо того, чтобы пытаться сделать это самостоятельно), поскольку анализатор Git был хорошо протестирован в реальных условиях.

Интерфейсным службам следует отдавать предпочтение формату raw , если исходный материал уже использует формат UNIX-эпохи, может быть уговорен предоставить даты в этом формате или его формат легко конвертируется в него, поскольку при синтаксическом анализе нет двусмысленности.

now

Всегда используйте текущее время и часовой пояс. Буквальное значение now всегда должно быть указано для <when> .

Это игрушечный формат. Текущее время и часовой пояс этой системы всегда копируются в идентификационную строку во время ее создания посредством быстрого импорта. Нет возможности указать другое время или часовой пояс.

Этот конкретный формат предоставляется, поскольку он прост в реализации и может быть полезен для процесса, который хочет создать новый коммит прямо сейчас, без необходимости использовать рабочий каталог или git update-index .

Если в commit используются отдельные команды author и committer , временные метки могут не совпадать, поскольку системные часы будут опрашиваться дважды (по одному разу для каждой команды). Единственный способ гарантировать, что идентификационная информация об авторе и коммиттере имеет одинаковую временную метку, — это опустить author (таким образом копируя из committer ) или использовать формат даты, отличный от now .

Commands

fast-import принимает несколько команд для обновления текущего репозитория и управления текущим процессом импорта. Более подробное обсуждение (с примерами) каждой команды будет позже.

commit

Создает новую ветку или обновляет существующую ветку, создавая новую фиксацию и обновляя ветку, чтобы она указывала на вновь созданную фиксацию.

tag

Создает аннотированный объект тега из существующей фиксации или ветки. Эта команда не поддерживает упрощенные теги, поскольку их не рекомендуется использовать для записи значимых моментов времени.

reset

Сбросьте существующую ветку (или новую ветку) до определенной версии. Эту команду необходимо использовать для изменения ветки на определенную ревизию без фиксации ее.

blob

Преобразуйте необработанные данные файла в большой двоичный объект для будущего использования в команде commit . Эта команда является необязательной и не требуется для выполнения импорта.

alias

Запишите, что метка относится к данному объекту без предварительного создания нового объекта. Использование --import-marks и ссылка на отсутствующие метки приведут к сбою быстрого импорта, поэтому псевдонимы могут предоставить способ установить в противном случае сокращенные коммиты на допустимое значение (например, ближайший необрезанный предок).

checkpoint

Принудительно выполняет быстрый импорт, чтобы закрыть текущий пакетный файл, сгенерировать его уникальную контрольную сумму и индекс SHA-1 и запустить новый пакетный файл. Эта команда является необязательной и не требуется для выполнения импорта.

progress

Заставляет быстрый импорт отображать всю строку на собственный стандартный вывод. Эта команда является необязательной и не требуется для выполнения импорта.

done

Отмечает конец потока. Эта команда является необязательной, если функция done не была запрошена с использованием параметра командной строки --done или команды feature done .

get-mark

Заставляет быстрый импорт печатать SHA-1, соответствующий метке дескриптора файла, установленной с помощью --cat-blob-fd или stdout , если не указано.

cat-blob

Заставляет быстрый импорт печатать большой двоичный объект в формате cat-file --batch в дескриптор файла, установленный с помощью --cat-blob-fd или stdout , если он не указан.

ls

Заставляет быстрый импорт печатать строку, описывающую запись каталога в формате ls-tree , в дескриптор файла, установленный с помощью --cat-blob-fd или stdout , если он не указан.

feature

Включите указанную функцию. Для этого необходимо, чтобы быстрый импорт поддерживал указанную функцию и прерывался, если это не так.

option

Укажите любые параметры, перечисленные в разделе ОПЦИИ, которые не меняют семантику потока в соответствии с потребностями внешнего интерфейса. Эта команда является необязательной и не требуется для выполнения импорта.

commit

Создайте или обновите ветку новым коммитом, записав одно логическое изменение в проект.

        'commit' SP <ref> LF
        mark?
        original-oid?
        ('author' (SP <name>)? SP LT <email> GT SP <when> LF)?
        'committer' (SP <name>)? SP LT <email> GT SP <when> LF
        ('encoding' SP <encoding>)?
        data
        ('from' SP <commit-ish> LF)?
        ('merge' SP <commit-ish> LF)*
        (filemodify | filedelete | filecopy | filerename | filedeleteall | notemodify)*
        LF?

где <ref> — имя ветки, в которой необходимо выполнить коммит. Обычно имена ветвей имеют префикс refs/heads/ в Git , поэтому при импорте символа ветки CVS RELENG-1_0 будет использоваться refs/heads/RELENG-1_0 для значения <ref> . Значение <ref> должно быть допустимым именем ссылки в Git . Поскольку LF недопустим в refname Git , здесь не поддерживается синтаксис кавычек или экранирования.

Опционально может появиться команда mark , запрашивающая быстрый импорт для сохранения ссылки на вновь созданный коммит для будущего использования внешним интерфейсом (формат см. ниже). Интерфейсы очень часто отмечают каждый созданный ими коммит, тем самым позволяя в будущем создавать ветки из любого импортированного коммита.

Команда data , следующая за committer , должна предоставить сообщение фиксации (синтаксис команды data см. ниже). Чтобы импортировать пустое сообщение фиксации, используйте данные длиной 0. Сообщения коммитов имеют произвольную форму и не интерпретируются Git. В настоящее время они должны быть закодированы в UTF-8, поскольку быстрый импорт не позволяет указывать другие кодировки.

Ноль или более команд filemodify , filedelete , filecopy , filerename , filedeleteall и notemodify могут быть включены для обновления содержимого ветки перед созданием фиксации. Эти команды могут подаваться в любом порядке. Однако рекомендуется, чтобы команда filedeleteall предшествовала всем командам filemodify , filecopy , filerename и notemodify в одном коммите, поскольку filedeleteall очищает ветку (см. ниже).

LF после команды необязателен (раньше он был обязательным). Обратите внимание, что по соображениям обратной совместимости, если фиксация заканчивается командой data (т. е. в ней нет команд from , merge , filemodify , filedelete , filecopy , filerename , filedeleteall или notemodify ), то в конце команды могут появиться две команды LF . вместо одного.

author

Команда author может появиться дополнительно, если информация об авторе может отличаться от информации о коммиттере. Если author опущен, то быстрый импорт будет автоматически использовать информацию коммиттера для части автора коммита. Ниже приведено описание полей в author , поскольку они идентичны committer .

committer

Команда committer указывает, кто и когда сделал этот коммит.

Здесь <name> — отображаемое имя человека (например, «Com M Itter»), а <email> — адрес электронной почты человека («[email protected]»). LT и GT — это буквальные символы «меньше» (\x3c) и «больше» (\x3e). Они необходимы для отделения адреса электронной почты от других полей в строке. Обратите внимание, что <name> и <email> имеют произвольную форму и могут содержать любую последовательность байтов, кроме LT , GT и LF . <name> обычно имеет кодировку UTF-8.

Время изменения указывается <when> с использованием формата даты, выбранного параметром командной строки --date-format=<fmt>. См. выше «Форматы дат» для получения информации о наборе поддерживаемых форматов и их синтаксисе.

encoding

Необязательная команда encoding указывает кодировку сообщения фиксации. Большинство коммитов имеют формат UTF-8, и кодировка опущена, но это позволяет импортировать сообщения коммитов в git без предварительного их перекодирования.

from

Команда from используется для указания фиксации для инициализации этой ветки. Эта ревизия будет первым предком нового коммита. Состояние дерева, построенного при этом коммите, начнется с состояния коммита from и будет изменено в результате изменений содержимого в этом коммите.

Если пропустить команду from в первом коммите новой ветки, быстрый импорт создаст этот коммит без предка. Обычно это желательно только для первоначальной фиксации проекта. Если интерфейс создает все файлы с нуля при создании новой ветки, вместо from можно использовать команду merge , чтобы начать фиксацию с пустым деревом. Обычно желательно пропустить команду from в существующих ветвях, поскольку текущий коммит в этой ветке автоматически считается первым предком нового коммита.

Поскольку LF недопустим в ссылочном имени Git или выражении SHA-1, в <commit-ish> не поддерживается синтаксис кавычек или экранирования.

Здесь <commit-ish> — это любой из следующих вариантов:

  • Имя существующей ветки, уже находящейся во внутренней таблице ветвей быстрого импорта. Если быстрый импорт не знает имени, оно рассматривается как выражение SHA-1.

  • Ссылка на знак :<idnum> , где <idnum> — номер знака.

    Причина, по которой быстрый импорт использует : для обозначения ссылки на метку, заключается в том, что этот символ недопустим в имени ветки Git . Ведущий : позволяет легко отличить метку 42 ( :42 ) и ветвь 42 ( 42 или refs/heads/42 ) или сокращенный SHA-1, который состоит только из цифр с основанием 10.

    Метки должны быть объявлены (через mark ), прежде чем их можно будет использовать.

  • Полная 40-байтовая или сокращенная фиксация SHA-1 в шестнадцатеричном формате.

  • Любое допустимое выражение Git SHA-1, которое разрешается как фиксация. Подробности см. в разделе «УКАЗАНИЕ ВЕРСИЙ» в gitrevisions[7] .

  • Специальный null SHA-1 (40 нулей) указывает, что ветвь должна быть удалена.

Особый случай перезапуска добавочного импорта из текущего значения ветки следует записать так:

        from refs/heads/branch^0

Суффикс ^0 необходим, поскольку быстрый импорт не позволяет ветке начинаться с самого себя, а ветвь создается в памяти еще до того, как команда from будет прочитана из ввода. Добавление ^0 заставит быстрый импорт разрешить фиксацию через библиотеку анализа ревизий Git, а не через внутреннюю таблицу ветвей, тем самым загружая существующее значение ветки.

merge

Включает один дополнительный коммит предка. Дополнительная ссылка на происхождение не меняет способ построения состояния дерева при этом коммите. Если команда from опущена при создании новой ветки, первая фиксация merge будет первым предком текущей фиксации, и ветка начнется без файлов. Неограниченное количество команд merge на одну фиксацию разрешено с помощью быстрого импорта, тем самым устанавливая n-стороннее слияние.

Здесь <commit-ish> — любое из выражений спецификации фиксации, также принимаемое from (см. выше).

filemodify

Включен в команду commit для добавления нового файла или изменения содержимого существующего файла. Эта команда имеет два разных способа указания содержимого файла.

Внешний формат данных

Содержимое файла уже было предоставлено предыдущей командой blob . Интерфейсу просто нужно его подключить.

        'M' SP <mode> SP <dataref> SP <path> LF

Здесь обычно <dataref> должен быть либо ссылкой на метку ( :<idnum> ), установленной предыдущей командой blob , либо полным 40-байтовым SHA-1 существующего BLOB-объекта Git . Если <mode> — 040000` , то <dataref> должен быть полным 40-байтовым SHA-1 существующего объекта дерева Git или ссылкой на метку, установленной с помощью --import-marks .

Встроенный формат данных

Содержание данных для файла еще не предоставлено. Интерфейсная часть хочет предоставить его как часть этой команды изменения.

        'M' SP <mode> SP 'inline' SP <path> LF
        data

Подробное описание команды data см. ниже.

В обоих форматах <mode> — это тип записи файла, указанный в восьмеричном формате. Git поддерживает только следующие режимы:

  • 100644 или 644 : обычный (неисполняемый) файл. Большинство файлов в большинстве проектов используют этот режим. Если сомневаетесь, это то, что вам нужно.

  • 100755 или 755 : обычный, но исполняемый файл.

  • 120000 : символическая ссылка, целью ссылки будет содержимое файла.

  • 160000 : ссылка gitlink, SHA-1 объекта относится к фиксации в другом репозитории. Ссылки Git можно указать только с помощью SHA или метки фиксации. Они используются для реализации подмодулей.

  • 040000 : подкаталог. Подкаталоги можно указать только с помощью SHA или через метку дерева, установленную с помощью --import-marks .

В обоих форматах <path> — это полный путь к файлу, который нужно добавить (если он еще не существует) или изменить (если он уже существует).

Строка <path> должна использовать разделители каталогов в стиле UNIX (косая черта / ), может содержать любой байт, отличный от LF , и не должна начинаться с двойной кавычки ( " ).

Путь может использовать кавычки в стиле C; это допускается во всех случаях и является обязательным, если имя файла начинается с двойной кавычки или содержит LF . При кавычках в стиле C полное имя должно быть заключено в двойные кавычки, а любые символы LF , обратную косую черту или двойные кавычки должны быть экранированы предшествующей им обратной косой чертой (например, "path/with\n, \\ and \" in it" ).

Значение <path> должно быть в канонической форме. То есть нельзя:

  • содержать пустой компонент каталога (например, foo//bar недействителен),

  • заканчиваться разделителем каталогов (например, foo/ недействителен),

  • начните с разделителя каталогов (например, /foo недействителен),

  • содержат специальный компонент . или .. (например, foo/./bar и foo/../bar недействительны).

Корень дерева может быть представлен пустой строкой как <path> .

Рекомендуется всегда использовать кодировку <path> с использованием UTF-8.

filedelete

Включен в команду commit для удаления файла или рекурсивного удаления всего каталога из ветки. Если в результате удаления файла или каталога его родительский каталог станет пустым, родительский каталог также будет автоматически удален. Это каскадно перемещается вверх по дереву до тех пор, пока не будет достигнут первый непустой каталог или корень.

        'D' SP <path> LF

здесь <path> — полный путь к файлу или подкаталогу, который необходимо удалить из ветки. Подробное описание <path> см. выше в filemodify .

filecopy

Рекурсивно копирует существующий файл или подкаталог в другое место ветки. Существующий файл или каталог должен существовать. Если пункт назначения существует, он будет полностью заменен содержимым, скопированным из источника.

        'C' SP <path> SP <path> LF

здесь первый <path> — это исходное местоположение, а второй <path> — место назначения. Подробное описание того, как может выглядеть <path> , см. выше в filemodify . Чтобы использовать исходный путь, содержащий SP, этот путь должен быть заключен в кавычки.

Команда filecopy вступает в силу немедленно. После того как исходное местоположение будет скопировано в место назначения, любые будущие команды, применяемые к исходному местоположению, не повлияют на место назначения копии.

filerename

Переименовывает существующий файл или подкаталог в другое место ветки. Существующий файл или каталог должен существовать. Если пункт назначения существует, он будет заменен исходным каталогом.

        'R' SP <path> SP <path> LF

здесь первый <path> — это исходное местоположение, а второй <path> — место назначения. Подробное описание того, как может выглядеть <path> , см. выше в filemodify . Чтобы использовать исходный путь, содержащий SP, этот путь должен быть заключен в кавычки.

Команда filerename вступает в силу немедленно. После переименования исходного местоположения в место назначения любые будущие команды, применяемые к исходному расположению, будут создавать там новые файлы и не влиять на место назначения переименования.

Обратите внимание, что filerename — это то же самое, что filecopy , за которым следует filedelete исходного местоположения. Использование filerename дает небольшое преимущество в производительности, но это преимущество настолько мало, что никогда не стоит пытаться преобразовать пару удаления/добавления в исходном материале в переименование для быстрого импорта. Эта команда filerename предназначена только для упрощения интерфейсов, которые уже имеют информацию о переименовании и не хотят беспокоиться о ее разложении на filecopy , за которым следует filedelete .

filedeleteall

Включен в команду commit для удаления всех файлов (а также всех каталогов) из ветки. Эта команда сбрасывает внутреннюю структуру ветки, чтобы в ней не было файлов, что позволяет интерфейсу впоследствии добавлять все интересующие файлы с нуля.

        'deleteall' LF

Эта команда чрезвычайно полезна, если интерфейсная часть не знает (или не хочет знать), какие файлы в данный момент находятся в ветке, и поэтому не может генерировать правильные команды filedelete для обновления содержимого.

Выдача filedeleteall с последующими необходимыми командами filemodify для установки правильного содержимого даст те же результаты, что и отправка только необходимых команд filemodify и filedelete . Однако подход filedeleteall может потребовать быстрого импорта, чтобы использовать немного больше памяти на активную ветвь (менее 1 МБ даже для большинства крупных проектов); поэтому веб-интерфейсам, которые могут легко получить только затронутые пути для коммита, рекомендуется делать это.

notemodify

Включен в команду commit <notes_ref> для добавления нового примечания, аннотирующего <commit-ish> , или изменения содержания этого примечания. Внутренне он похож на filemodify 100644 по пути <commit-ish> (возможно, разделенный на подкаталоги). Не рекомендуется использовать какие-либо другие команды для записи в дерево <notes_ref> , кроме filedeleteall для удаления всех существующих заметок в этом дереве. Эта команда имеет два разных способа указания содержания примечания.

Внешний формат данных

Содержимое примечания уже было предоставлено предыдущей командой blob . Интерфейсу просто нужно подключить его к коммиту, который нужно аннотировать.

        'N' SP <dataref> SP <commit-ish> LF

Здесь <dataref> может быть либо ссылкой на метку ( :<idnum> ), установленной предыдущей командой blob , либо полным 40-байтовым SHA-1 существующего объекта большого двоичного объекта Git .

Встроенный формат данных

Содержимое примечания еще не предоставлено. Интерфейсная часть хочет предоставить его как часть этой команды изменения.

        'N' SP 'inline' SP <commit-ish> LF
        data

Подробное описание команды data см. ниже.

В обоих форматах <commit-ish> — это любое из выражений спецификации фиксации, также принимаемое from (см. выше).

mark

Организует быстрый импорт для сохранения ссылки на текущий объект, позволяя интерфейсу вызвать этот объект в будущем, не зная его SHA-1. Здесь текущий объект — это команда создания объекта, внутри которой появляется команда mark . Это могут быть commit , tag и blob , но наиболее часто используется commit .

        'mark' SP ':' <idnum> LF

где <idnum> — номер, присвоенный интерфейсом этой метке. Значение <idnum> выражается как десятичное целое число ASCII. Значение 0 зарезервировано и не может использоваться в качестве метки. В качестве оценок могут использоваться только значения, превышающие или равные 1.

Новые отметки создаются автоматически. Существующие метки можно переместить на другой объект, просто повторно используя ту же команду <idnum> в другой команде mark .

original-oid

Предоставляет имя объекта в исходной системе управления версиями. fast-import просто проигнорирует эту директиву, но процессы фильтрации, которые работают с потоком и изменяют его перед подачей в fast-import, могут использовать эту информацию.

        'original-oid' SP <object-identifier> LF

где <object-identifier> — любая строка, не содержащая LF.

tag

Создает аннотированный тег, ссылающийся на конкретную фиксацию. Чтобы создать облегченные (неаннотированные) теги, используйте команду reset ниже.

        'tag' SP <name> LF
        mark?
        'from' SP <commit-ish> LF
        original-oid?
        'tagger' (SP <name>)? SP LT <email> GT SP <when> LF
        data

где <name> — имя создаваемого тега.

Имена тегов автоматически имеют префикс refs/tags/ при сохранении в Git, поэтому при импорте символа ветки CVS RELENG-1_0-FINAL вместо <name> будет использоваться только RELENG-1_0-FINAL , а при быстром импорте соответствующая ссылка будет записана как refs/tags/RELENG-1_0-FINAL .

Значение <name> должно быть допустимым именем ссылки в Git и, следовательно, может содержать косую черту. Поскольку LF недопустим в refname Git , здесь не поддерживается синтаксис кавычек или экранирования.

Команда from аналогична команде commit ; подробности см. выше.

Команда tagger использует тот же формат, что и committer внутри commit ; еще раз см. выше подробности.

Команда data , следующая за tagger , должна предоставить аннотированное сообщение тега (синтаксис команды data см. ниже). Чтобы импортировать пустое сообщение тега, используйте данные длиной 0. Сообщения тегов имеют произвольную форму и не интерпретируются Git. В настоящее время они должны быть закодированы в UTF-8, поскольку быстрый импорт не позволяет указывать другие кодировки.

Подписание аннотированных тегов во время импорта из функции быстрого импорта не поддерживается. Не рекомендуется включать собственную подпись PGP/GPG, поскольку интерфейс не имеет (легко) доступа к полному набору байтов, который обычно входит в такую ​​подпись. Если требуется подпись, создайте упрощенные теги из функции быстрого импорта с помощью reset , а затем создайте аннотированные версии этих тегов в автономном режиме с помощью стандартного процесса git tag .

reset

Создает (или воссоздает) именованную ветку, возможно, начиная с определенной ревизии. Команда сброса позволяет интерфейсу выдать новую команду from для существующей ветки или создать новую ветку на основе существующего коммита без создания нового коммита.

        'reset' SP <ref> LF
        ('from' SP <commit-ish> LF)?
        LF?

Подробное описание <ref> и <commit-ish> см. выше в разделах commit и from .

LF после команды необязателен (раньше он был обязательным).

Команду reset также можно использовать для создания облегченных (неаннотированных) тегов. Например:

reset refs/tags/938
from :938

создаст облегченный тег refs/tags/938 , ссылающийся на любую ссылку на метку фиксации :938 .

blob

Запрашивает запись одной ревизии файла в пакетный файл. Ревизия не связана ни с каким коммитом; это соединение должно быть сформировано в последующей команде commit путем ссылки на большой двоичный объект через назначенную метку.

        'blob' LF
        mark?
        original-oid?
        data

Команда mark здесь необязательна, поскольку некоторые внешние интерфейсы решили генерировать Git SHA-1 для большого двоичного объекта самостоятельно и передавать его непосредственно в commit . Однако обычно это требует больше работы, чем того стоит, поскольку знаки недороги в хранении и просты в использовании.

data

Предоставляет необработанные данные (для использования в качестве содержимого больших двоичных объектов/файлов, сообщений о фиксации или сообщений с аннотациями тегов) для быстрого импорта. Данные могут быть переданы с использованием точного количества байт или разделены завершающей строкой. Настоящие интерфейсы, предназначенные для преобразований производственного качества, всегда должны использовать точный формат подсчета байтов, поскольку он более надежен и работает лучше. Формат с разделителями предназначен в первую очередь для тестирования быстрого импорта.

Строки комментариев, появляющиеся в части <raw> команд data , всегда считаются частью тела данных и поэтому никогда не игнорируются при быстром импорте. Это делает безопасным импорт любого содержимого файла/сообщения, строки которого могут начинаться с # .

Точный формат подсчета байтов

Интерфейс должен указать количество байтов данных.

        'data' SP <count> LF
        <raw> LF?

где <count> — точное количество байтов, встречающихся в <raw> . Значение <count> выражается как десятичное целое число ASCII. LF по обе стороны от <raw> не включен в <count> и не будет включен в импортированные данные.

LF после <raw> не является обязательным (раньше он был обязательным), но рекомендуется. Его всегдае включение упрощает отладку потока быстрого импорта, поскольку следующая команда всегда начинается в столбце 0 следующей строки, даже если <raw> не заканчивается на LF .

Delimited format

Строка-разделитель используется для обозначения конца данных. fast-import вычислит длину путем поиска разделителя. Этот формат в первую очередь полезен для тестирования и не рекомендуется для реальных данных.

        'data' SP '<<' <delim> LF
        <raw> LF
        <delim> LF
        LF?

где <delim> — выбранная строка-разделитель. Строка <delim> не должна появляться в отдельной строке внутри <raw> , иначе при быстром импорте будет считаться, что данные заканчиваются раньше, чем на самом деле. LF , следующий за <raw> , является частью <raw> . Это одно из ограничений формата с разделителями: невозможно предоставить фрагмент данных, последний байт которого не содержит LF.

LF после <delim> LF является необязательным (раньше он был обязательным).

alias

Запишите, что метка относится к данному объекту без предварительного создания нового объекта.

        'alias' LF
        mark
        'to' SP <commit-ish> LF
        LF?

Подробное описание <commit-ish> см. выше в разделе from .

checkpoint

Принудительно выполняет быстрый импорт, чтобы закрыть текущий пакетный файл, запустить новый и сохранить все текущие ссылки на ветки, теги и метки.

        'checkpoint' LF
        LF?

Обратите внимание, что быстрый импорт автоматически переключает файлы пакетов, когда текущий файл пакета достигает --max-pack-size или 4 ГиБ, в зависимости от того, какой предел меньше. Во время автоматического переключения файлов пакетов быстрый импорт не обновляет ссылки, теги или метки ветвей.

Поскольку checkpoint может потребовать значительного количества процессорного времени и дискового ввода-вывода (для вычисления общей контрольной суммы пакета SHA-1, создания соответствующего индексного файла и обновления ссылок), выполнение одной команды checkpoint может легко занять несколько минут.

Внешние интерфейсы могут создавать контрольные точки во время очень больших и длительных импортов или когда им необходимо разрешить другому процессу Git доступ к ветке. Однако, учитывая, что репозиторий Subversion объемом 30 ГиБ можно загрузить в Git посредством быстрого импорта примерно за 3 часа, явная установка контрольных точек может не потребоваться.

LF после команды необязателен (раньше он был обязательным).

progress

Заставляет быстрый импорт печатать всю строку progress без изменений в ее стандартный выходной канал (дескриптор файла 1), когда команда обрабатывается из входного потока. В противном случае команда не влияет ни на текущий импорт, ни на какое-либо внутреннее состояние быстрого импорта.

        'progress' SP <any> LF
        LF?

Часть команды <any> может содержать любую последовательность байтов, не содержащую LF . LF после команды не является обязательным. Вызывающая сторона может захотеть обработать вывод с помощью такого инструмента, как sed, чтобы удалить начальную часть строки, например:

frontend | git fast-import | sed 's/^progress //'

Размещение команды progress сразу после checkpoint сообщит читателю, когда checkpoint будет завершен, и он сможет безопасно получить доступ к ссылкам, которые обновляются с помощью быстрого импорта.

get-mark

Заставляет быстрый импорт печатать SHA-1, соответствующий отметке на стандартный вывод или дескриптору файла, предварительно заданному с помощью аргумента --cat-blob-fd . В противном случае команда не повлияет на текущий импорт; его цель — получить SHA-1, на которые последующие коммиты могут захотеть ссылаться в своих сообщениях о фиксации.

        'get-mark' SP ':' <idnum> LF

Подробную информацию о том, как безопасно читать этот вывод, см. ниже в разделе «Ответы на команды».

cat-blob

Заставляет быстрый импорт печатать большой двоичный объект в дескриптор файла, предварительно заданный с помощью аргумента --cat-blob-fd . В противном случае команда не повлияет на текущий импорт; его основная цель — получить большие двоичные объекты, которые могут находиться в памяти быстрого импорта, но недоступны из целевого репозитория.

        'cat-blob' SP <dataref> LF

<dataref> может быть либо ссылкой на метку ( :<idnum> ), установленной ранее, либо полным 40-байтовым SHA-1 большого двоичного объекта Git , уже существующего или готового к записи.

Вывод использует тот же формат, что и git cat-file --batch :

<sha1> SP 'blob' SP <size> LF
<contents> LF

Эту команду можно использовать там, где может появиться директива filemodify , что позволяет использовать ее в середине коммита. Для filemodify , использующего встроенную директиву, она также может появиться непосредственно перед директивой data .

Подробную информацию о том, как безопасно читать этот вывод, см. ниже в разделе «Ответы на команды».

ls

Печатает информацию об объекте по пути к файловому дескриптору, предварительно указанному с помощью аргумента --cat-blob-fd . Это позволяет распечатать большой двоичный объект из активного коммита (с помощью cat-blob ) или скопировать большой двоичный объект или дерево из предыдущего коммита для использования в текущем (с помощью filemodify ).

Команду ls также можно использовать там, где может появиться директива filemodify , что позволяет использовать ее в середине коммита.

Чтение из активного коммита

Эту форму можно использовать только в середине commit . Путь называет запись каталога в активной фиксации быстрого импорта. В этом случае путь должен быть указан в кавычках.

        'ls' SP <path> LF
Чтение из именованного дерева

<dataref> может быть ссылкой на метку ( :<idnum> ) или полным 40-байтовым SHA-1 тега Git , фиксации или объекта дерева, уже существующего или ожидающего записи. Путь указывается относительно верхнего уровня дерева, указанного <dataref> .

        'ls' SP <dataref> SP <path> LF

Подробное описание <path> см. выше в filemodify .

Вывод использует тот же формат, что и git ls-tree <tree> -- <path> :

<mode> SP ('blob' | 'tree' | 'commit') SP <dataref> HT <path> LF

<dataref> представляет объект blob, дерево или фиксацию в <path> и может использоваться в более поздних командах get-mark , cat-blob , filemodify или ls .

Если по этому пути нет файла или поддерева, git fast-import вместо этого сообщит

missing SP <path> LF

Подробную информацию о том, как безопасно читать этот вывод, см. ниже в разделе «Ответы на команды».

feature

Требовать, чтобы быстрый импорт поддерживал указанную функцию, или прервать его, если это не так.

        'feature' SP <feature> ('=' <argument>)? LF

Часть <feature> команды может быть любой из следующих:

date-format
export-marks
relative-marks
no-relative-marks
force

Действуйте так, как будто в командной строке был передан соответствующий параметр командной строки с ведущим -- (см. ОПЦИИ выше).

import-marks
import-marks-if-exists

Аналогично --import-marks, за исключением двух моментов: во-первых, для каждого потока разрешена только одна команда "feature import-marks" или "feature import-marks-if-exists" ; во-вторых, параметр командной строки --import-marks= или --import-marks-if-exists переопределяет любую из этих команд "feature" в потоке; в-третьих, "feature import-marks-if-exists" , как и соответствующий параметр командной строки, молча пропускает несуществующий файл.

get-mark
cat-blob
ls

Требуйте, чтобы серверная часть поддерживала команды get-mark , cat-blob или ls соответственно. Версии быстрого импорта, не поддерживающие указанную команду, завершатся с соответствующим сообщением. Это позволяет выявить ошибку импорта на ранней стадии с четким сообщением, вместо того, чтобы тратить время на раннюю часть импорта, прежде чем будет обнаружена неподдерживаемая команда.

notes

Требуйте, чтобы серверная часть поддерживала подкоманду notemodify (N) для команды commit . Версии быстрого импорта, не поддерживающие заметки, завершатся с соответствующим сообщением.

done

Ошибка, если поток завершается без команды done . Без этой функции ошибки, приводящие к внезапному завершению работы интерфейса в удобной точке потока, могут остаться незамеченными. Это может произойти, например, если интерфейс импорта прекращает работу в середине операции, не выдавая SIGTERM или SIGKILL на подчиненном экземпляре быстрого импорта git .

option

Обрабатывает указанный параметр, чтобы быстрый импорт git работал в соответствии с потребностями внешнего интерфейса. Обратите внимание, что параметры, указанные внешним интерфейсом, переопределяются любыми параметрами, которые пользователь может указать для самого быстрого импорта git .

    'option' SP <option> LF

Часть команды <option> может содержать любые параметры, перечисленные в разделе ОПЦИИ, которые не меняют семантику импорта, без ведущего -- и обрабатываются таким же образом.

Дополнительные команды должны быть первыми командами на входе (не считая команд функций), чтобы дать дополнительную команду после того, как любая команда, не являющаяся опцией, является ошибкой.

Следующие параметры командной строки изменяют семантику импорта и поэтому не могут быть переданы в качестве параметра:

  • date-format

  • import-marks

  • export-marks

  • cat-blob-fd

  • force

done

Если функция done не используется, считается, что EOF был прочитан. Это можно использовать, чтобы указать быстрому импорту завершиться раньше.

Если используется параметр командной строки --done или команда feature done , команда done является обязательной и отмечает конец потока.

Реакции на команды

Новые объекты, записанные с помощью быстрого импорта, доступны не сразу. Большинство команд быстрого импорта не имеют видимого эффекта до следующей контрольной точки (или завершения). Интерфейсный интерфейс может отправлять команды для заполнения входного канала быстрого импорта, не беспокоясь о том, как быстро они вступят в силу, что повышает производительность за счет упрощения планирования.

Однако для некоторых интерфейсов полезно иметь возможность считывать данные из текущего репозитория по мере его обновления (например, когда исходный материал описывает объекты в виде исправлений, которые будут применены к ранее импортированным объектам). Этого можно добиться, подключив интерфейс и быстрый импорт через двунаправленные каналы:

mkfifo fast-import-output
frontend <fast-import-output |
git fast-import >fast-import-output

Интерфейс, настроенный таким образом, может использовать команды progress , get-mark , ls и cat-blob для чтения информации из текущего импорта.

Чтобы избежать взаимоблокировки, такие внешние интерфейсы должны полностью использовать все ожидающие выходные данные от progress , ls , get-mark и cat-blob , прежде чем выполнять записи в быстрый импорт, которые могут блокироваться.

Crash reports

Если для быстрого импорта предоставлены недопустимые входные данные, он завершится с ненулевым статусом выхода и создаст отчет о сбое на верхнем уровне репозитория Git , в который он импортировал. Отчеты о сбоях содержат снимок внутреннего состояния быстрого импорта, а также самые последние команды, которые привели к сбою.

Все последние команды (включая комментарии к потоку, изменения файлов и команды выполнения) отображаются в истории команд в отчете о сбое, но необработанные данные файла и сообщения о фиксации исключаются из отчета о сбое. Это исключение экономит место в файле отчета и уменьшает объем буферизации, которую должен выполнять быстрый импорт во время выполнения.

После написания отчета о сбое быстрый импорт закроет текущий пакетный файл и экспортирует таблицу меток. Это позволяет разработчику внешнего интерфейса проверить состояние репозитория и возобновить импорт с того места, где он произошел сбой. Измененные ветки и теги не обновляются во время сбоя, поскольку импорт не завершился успешно. Информацию о ветках и тегах можно найти в отчете о сбое, и при необходимости обновления их необходимо применять вручную.

Следующие советы и рекомендации были собраны от различных пользователей быстрого импорта и предлагаются здесь в качестве рекомендаций.

Используйте одну отметку за фиксацию

При преобразовании репозитория используйте уникальную метку для каждой фиксации ( mark :<n> ) и укажите параметр --export-marks в командной строке. fast-import создаст дамп файла, в котором перечислены все метки и соответствующий им объект Git SHA-1. Если внешний интерфейс может привязать метки обратно к исходному репозиторию, легко проверить точность и полноту импорта, сравнив каждый коммит Git с соответствующей исходной версией.

В такой системе, как Perforce или Subversion, это должно быть довольно просто, поскольку метка быстрого импорта также может быть номером набора изменений Perforce или номером версии Subversion.

Свободно пропускать ветки

Не пытайтесь оптимизировать интерфейс, чтобы при импорте использовать одну ветку за раз. Хотя это может быть немного быстрее для быстрого импорта, это имеет тенденцию значительно увеличивать сложность кода внешнего интерфейса.

LRU ветки, встроенный в быстрый импорт, обычно ведет себя очень хорошо, а стоимость активации неактивной ветки настолько низка, что переключение между ветвями практически не влияет на производительность импорта.

Handling Renames

При импорте переименованного файла или каталога просто удалите старые имена и измените новые имена во время соответствующей фиксации. Git выполняет обнаружение переименования постфактум, а не явно во время фиксации.

Используйте ветки исправления тегов

Некоторые другие системы SCM позволяют пользователю создавать тег из нескольких файлов, которые не входят в один и тот же набор коммитов/изменений. Или создать теги, которые являются подмножеством файлов, доступных в репозитории.

Импорт этих тегов «как есть» в Git невозможен без выполнения хотя бы одного коммита, который «исправляет» файлы, чтобы они соответствовали содержимому тега. Используйте команду reset быстрого импорта, чтобы сбросить фиктивную ветвь за пределами обычного пространства ветвей до базовой фиксации для тега, затем зафиксировать одну или несколько фиксаций исправления файла и, наконец, пометить фиктивную ветвь.

Например, поскольку все обычные ветки хранятся под refs/heads/ , назовите ветку исправления тега TAG_FIXUP . Таким образом, ветка исправления, используемая импортером, не может иметь конфликтов пространства имен с реальными ветвями, импортированными из источника (имя TAG_FIXUP не является refs/heads/TAG_FIXUP ).

При фиксации исправлений рассмотрите возможность использования merge для подключения коммитов, которые передают версии файлов в ветку исправлений. Это позволит таким инструментам, как git blame , отслеживать реальную историю коммитов и правильно комментировать исходные файлы.

После завершения быстрого импорта интерфейсу потребуется выполнить rm .git/TAG_FIXUP , чтобы удалить фиктивную ветку.

Импортируйте сейчас, перепакуйте позже

Как только быстрый импорт завершится, репозиторий Git станет полностью действительным и готов к использованию. Обычно это занимает очень короткое время, даже для значительно больших проектов (более 100 000 коммитов).

Однако переупаковка репозитория необходима для улучшения локальности данных и производительности доступа. Это также может занять несколько часов в очень больших проектах (особенно если используется -f и большой параметр --window). Поскольку перепаковку безопасно выполнять вместе с читателями и авторами, запустите перепаковку в фоновом режиме и дайте ей завершиться после завершения. Нет причин ждать, чтобы изучить ваш новый проект Git !

Если вы решите дождаться перепаковки, не пытайтесь запускать тесты производительности или тесты производительности до завершения перепаковки. fast-import выводит неоптимальные паковочные файлы, которые просто никогда не встречаются в реальных ситуациях использования.

Переупаковка исторических данных

Если вы переупаковываете очень старые импортированные данные (например, старше прошлого года), подумайте о том, чтобы потратить дополнительное время процессора и указать --window=50 (или выше) при запуске git repack . Это займет больше времени, но при этом будет создан пакфайл меньшего размера. Вам нужно приложить усилия только один раз, и каждый, кто использует ваш проект, выиграет от меньшего репозитория.

Включите некоторые сообщения о ходе выполнения

Время от времени ваш интерфейс отправляет сообщение progress для быстрого импорта. Содержимое сообщений полностью произвольное, поэтому можно было бы предложить выводить текущий месяц и год каждый раз, когда текущая дата фиксации перемещается на следующий месяц. Ваши пользователи будут чувствовать себя лучше, зная, какая часть потока данных была обработана.

Packfile optimization

При упаковке BLOB-объекта быстрый импорт всегда пытается выполнить дельтификацию по последнему записанному BLOB-объекту. Если это специально не оговорено интерфейсом, это, вероятно, не будет предыдущей версией того же файла, поэтому сгенерированная дельта не будет минимально возможной. Полученный пакетный файл будет сжат, но не будет оптимальным.

Внешние интерфейсы, которые имеют эффективный доступ ко всем версиям одного файла (например, чтение файла RCS/CVS,v), могут выбрать предоставление всех версий этого файла в виде последовательности последовательных команд blob . Это позволяет при быстром импорте сравнивать различные версии файлов друг с другом, экономя место в окончательном пакетном файле. Метки можно использовать для последующей идентификации отдельных редакций файла во время выполнения последовательности команд commit .

Пакетные файлы, созданные с помощью быстрого импорта, не поддерживают хорошие шаблоны доступа к диску. Это вызвано быстрой записью данных при импорте в том порядке, в котором они были получены на стандартном вводе, в то время как Git обычно организует данные в пакетных файлах, чтобы самые последние (текущие подсказки) данные отображались перед историческими данными. Git также кластеризует коммиты, ускоряя прохождение ревизий за счет лучшей локальности кэша.

По этой причине пользователям настоятельно рекомендуется повторно упаковать репозиторий с помощью git repack -a -d после завершения быстрого импорта, что позволит Git реорганизовать паковочные файлы для более быстрого доступа к данным. Если дельты больших двоичных объектов неоптимальны (см. выше), то добавление опции -f для принудительного пересчета всех дельт может значительно уменьшить окончательный размер паковочного файла (вполне типично меньше на 30-50%).

Вместо запуска git repack вы также можете запустить git gc --aggressive , который также оптимизирует другие вещи после импорта (например, упаковывает свободные ссылки). Как отмечено в разделе "AGGRESSIVE" в git-gc[1] , опция --aggressive обнаружит новые различия с опцией -f по сравнению с git-repack[1] . По причинам, изложенным выше, использование --aggressive после быстрого импорта является одним из немногих случаев, когда это, как известно, имеет смысл.

Memory utilization

Существует ряд факторов, влияющих на то, сколько памяти требуется для быстрого импорта для выполнения импорта. Как и критические разделы ядра Git, быстрый импорт использует собственные распределители памяти для амортизации любых накладных расходов, связанных с malloc. На практике быстрый импорт имеет тенденцию амортизировать любые накладные расходы malloc до 0 из-за использования выделения больших блоков.

per object

fast-import поддерживает структуру в памяти для каждого объекта, записанного в этом выполнении. В 32-битной системе структура составляет 32 байта, в 64-битной системе — 40 байт (из-за большего размера указателя). Объекты в таблице не освобождаются до тех пор, пока не завершится быстрый импорт. Импорт 2 миллионов объектов в 32-битной системе потребует примерно 64 МБ памяти.

Таблица объектов на самом деле представляет собой хеш-таблицу, основанную на имени объекта (уникальный SHA-1). Эта конфигурация хранилища позволяет быстрому импорту повторно использовать существующий или уже написанный объект и избегать записи дубликатов в выходной пакетный файл. Дубликаты больших двоичных объектов на удивление часто встречаются при импорте, обычно из-за слияния ветвей в источнике.

per mark

Метки хранятся в разреженном массиве, используя 1 указатель (4 байта или 8 байтов, в зависимости от размера указателя) на каждую метку. Несмотря на то, что массив разреженный, во внешних интерфейсах по-прежнему настоятельно рекомендуется использовать метки от 1 до n, где n — общее количество меток, необходимых для этого импорта.

per branch

Филиалы делятся на активные и неактивные. Использование памяти двумя классами существенно различается.

Неактивные ветки хранятся в структуре, которая использует 96 или 120 байт (32-битные или 64-битные системы соответственно), плюс длину имени ветки (обычно менее 200 байт) на каждую ветвь. fast-import легко обработает до 10 000 неактивных ветвей, занимая менее 2 МБ памяти.

Активные ветки имеют те же накладные расходы, что и неактивные, но также содержат копии каждого дерева, которое было недавно изменено в этой ветке. Если поддерево include не было изменено с момента, когда ветвь стала активной, его содержимое не будет загружено в память, но если поддерево src было изменено фиксацией с момента, когда ветвь стала активной, то его содержимое будет загружено в память.

Поскольку активные ветки хранят метаданные о файлах, содержащихся в этой ветке, размер их хранилища в памяти может вырасти до значительного размера (см. ниже).

fast-import автоматически переводит активные ветки в неактивное состояние на основе простого алгоритма, который использовался реже всего. Цепочка LRU обновляется при каждой команде commit . Максимальное количество активных ветвей можно увеличить или уменьшить в командной строке с помощью --active-branches=.

за активное дерево

Деревья (также известные как каталоги) используют всего 12 байт памяти сверх памяти, необходимой для их записей (см. «на активный файл» ниже). Стоимость дерева практически равна нулю, поскольку его накладные расходы амортизируются по отдельным записям файла.

за активную запись файла

Файлы (и указатели на поддеревья) в активных деревьях требуют 52 или 64 байта (32/64-битные платформы) на запись. Для экономии места имена файлов и деревьев объединяются в общую таблицу строк, что позволяет имени файла «Makefile» использовать всего 16 байт (после включения служебных данных заголовка строки), независимо от того, сколько раз оно встречается в проекте.

LRU активной ветки в сочетании с пулом строк имен файлов и отложенной загрузкой поддеревьев позволяет осуществлять быстрый импорт для эффективного импорта проектов с более чем 2000 ветвями и 45 114+ файлами при очень ограниченном объеме памяти (менее 2,7 МБ на активную ветвь).

Signals

Отправка SIGUSR1 в процесс git fast-import досрочно завершает текущий пакетный файл, имитируя команду checkpoint . Нетерпеливый оператор может использовать эту возможность для просмотра объектов и ссылок в процессе импорта, ценой некоторого дополнительного времени работы и худшего сжатия.

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

fastimport.unpackLimit

Если количество объектов, импортированных git-fast-import[1] , меньше этого предела, объекты будут распакованы в отдельные объектные файлы. Однако если количество импортированных объектов равно этому пределу или превышает его, пакет будет сохранен как пакет. Сохранение пакета из быстрого импорта может ускорить выполнение операции импорта, особенно на медленных файловых системах. Если не установлено, вместо него используется значение transfer.unpackLimit .
