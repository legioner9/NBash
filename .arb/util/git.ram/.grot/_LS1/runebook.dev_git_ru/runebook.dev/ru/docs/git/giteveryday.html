giteveryday

Name

giteveryday — Минимальный полезный набор команд для Everyday Git

Synopsis

Git на каждый день с примерно 20 командами

Description

Пользователей Git можно условно сгруппировать в четыре категории, чтобы описать здесь небольшой набор полезных команд для повседневного использования Git .

  • Команды Individual Developer (Standalone) необходимы для всех, кто делает коммит, даже для тех, кто работает в одиночку.

  • Если вы работаете с другими людьми, вам также понадобятся команды, перечисленные в разделе Individual Developer (Participant) .

  • Людям, играющим роль Integrator , необходимо выучить еще несколько команд в дополнение к вышеперечисленным.

  • Команды Repository Administration предназначены для системных администраторов, которые отвечают за обслуживание и подачу репозиториев Git .

Индивидуальный разработчик (автономный)

Автономный индивидуальный разработчик не обменивается патчами с другими людьми и работает один в одном репозитории, используя следующие команды.

  • git-init[1] для создания нового репозитория.

  • git-log[1] , чтобы узнать, что произошло.

  • git-switch[1] и git-branch[1] для переключения ветвей.

  • git-add[1] для управления индексным файлом.

  • git-diff[1] и git-status[1] , чтобы увидеть, чем вы занимаетесь.

  • git-commit[1] для перехода к текущей ветке.

  • git-restore[1] для отмены изменений.

  • git-merge[1] для объединения локальных ветвей.

  • git-rebase[1] для поддержки тематических ветвей.

  • git-tag[1] для обозначения известной точки.

Examples

Используйте архив в качестве отправной точки для нового репозитория.
$ tar zxf frotz.tar.gz
$ cd frotz
$ git init
$ git add . (1)
$ git commit -m "import of frotz source tree."
$ git tag v2.43 (2)

  1. добавьте все в текущий каталог.

  2. сделайте легкий, неаннотированный тег.

Создайте ветку темы и развивайтесь.
$ git switch -c alsa-audio (1)
$ edit/compile/test
$ git restore curses/ux_audio_oss.c (2)
$ git add curses/ux_audio_alsa.c (3)
$ edit/compile/test
$ git diff HEAD (4)
$ git commit -a -s (5)
$ edit/compile/test
$ git diff HEAD^ (6)
$ git commit -a --amend (7)
$ git switch master (8)
$ git merge alsa-audio (9)
$ git log --since='3 days ago' (10)
$ git log v2.43.. curses/ (11)

  1. создайте новую ветку темы.

  2. отмените неудачные изменения в curses/ux_audio_oss.c .

  3. вам нужно сообщить Git , добавили ли вы новый файл; удаление и модификация будут отловлены, если вы сделаете git commit -a позже.

  4. чтобы увидеть, какие изменения вы совершаете.

  5. зафиксируйте все, как вы проверили, с вашей подписью.

  6. просмотрите все ваши изменения, включая предыдущий коммит.

  7. отредактируйте предыдущий коммит, добавив все новые изменения, используя исходное сообщение.

  8. переключитесь на главную ветку.

  9. объедините ветку темы с основной веткой.

  10. просмотреть журналы коммитов; другие формы ограничения вывода можно комбинировать и включать -10 (для отображения до 10 коммитов), --until=2005-12-10 и т. д.

  11. просматривать только те изменения, которые касаются содержимого каталога curses/ , начиная с тега v2.43 .

Индивидуальный застройщик (участник)

Разработчик, работающий в качестве участника группового проекта, должен научиться общаться с другими и использовать эти команды в дополнение к тем, которые нужны автономному разработчику.

  • git-clone[1] из восходящего потока для загрузки вашего локального репозитория.

  • git-pull[1] и git-fetch[1] вместо "origin" , чтобы быть в курсе последних событий.

  • git-push[1] в общий репозиторий, если вы используете рабочий процесс общего репозитория в стиле CVS.

  • git-format-patch[1] для подготовки отправки по электронной почте, если вы используете рабочий процесс форума public в стиле ядра Linux.

  • git-send-email[1] для отправки вашего электронного письма без искажений вашим MUA.

  • git-request-pull[1] для создания сводки изменений для восходящего сервера.

Examples

Клонируйте восходящий поток и работайте над ним. Подача изменений в восходящий поток.
$ git clone git://git.kernel.org/pub/scm/.../torvalds/linux-2.6 my2.6
$ cd my2.6
$ git switch -c mine master (1)
$ edit/compile/test; git commit -a -s (2)
$ git format-patch master (3)
$ git send-email --to="person <[email protected]>" 00*.patch (4)
$ git switch master (5)
$ git pull (6)
$ git log -p ORIG_HEAD.. arch/i386 include/asm-i386 (7)
$ git ls-remote --heads http://git.kernel.org/.../jgarzik/libata-dev.git (8)
$ git pull git://git.kernel.org/pub/.../jgarzik/libata-dev.git ALL (9)
$ git reset --hard ORIG_HEAD (10)
$ git gc (11)

  1. Оформить заказ на новую ветку mine от мастера.

  2. повторите по мере необходимости.

  3. извлеките патчи из вашей ветки относительно мастера,

  4. и отправьте им электронное письмо.

  5. вернитесь к master , готовы увидеть, что нового

  6. git pull по умолчанию извлекает данные из origin и объединяет их с текущей веткой.

  7. сразу после вытягивания посмотрите на изменения, внесенные выше по течению с момента последней проверки, только в той области, которая нас интересует.

  8. проверьте имена ветвей во внешнем репозитории (если они не известны).

  9. извлеките из определенной ветки ALL из определенного репозитория и объедините ее.

  10. вернуть тягу.

  11. сбор мусора, оставшиеся объекты от возвращенного извлечения.

Перенесите в другой репозиторий.
satellite$ git clone mothership:frotz frotz (1)
satellite$ cd frotz
satellite$ git config --get-regexp '^(remote|branch)\.' (2)
remote.origin.url mothership:frotz
remote.origin.fetch refs/heads/*:refs/remotes/origin/*
branch.master.remote origin
branch.master.merge refs/heads/master
satellite$ git config remote.origin.push \
           +refs/heads/*:refs/remotes/satellite/* (3)
satellite$ edit/compile/test/commit
satellite$ git push origin (4)

mothership$ cd frotz
mothership$ git switch master
mothership$ git merge satellite/master (5)

  1. На машине Mothership есть репозиторий frotz в вашем домашнем каталоге; клонируйте его, чтобы запустить репозиторий на спутниковой машине.

  2. clone устанавливает эти переменные конфигурации по умолчанию. Он организует git pull для извлечения и сохранения ветвей машины-носителя в локальных ветвях удаленного отслеживания remotes/origin/* .

  3. организовать git push для отправки всех локальных веток в соответствующую ветвь материнской машины.

  4. push спрячет всю нашу работу на ветках удаленного отслеживания remotes/satellite/* на машине-матке. Вы можете использовать это как запасной метод. Аналогично, вы можете сделать вид, что от вас Mothership "fetched" (полезно, когда доступ односторонний).

  5. на материнской машине объедините работу, выполненную на спутниковой машине, с основной веткой.

Ответвление от определенного тега.
$ git switch -c private2.6.14 v2.6.14 (1)
$ edit/compile/test; git commit -a
$ git checkout master
$ git cherry-pick v2.6.14..private2.6.14 (2)

  1. создайте ветку private на основе хорошо известного (но несколько устаревшего) тега.

  2. перенаправить все изменения в ветке private2.6.14 в ветку master без формального "merging" . Или от руки
    git format-patch -k -m --stdout v2.6.14..private2.6.14 | git am -3 -k

Альтернативный механизм отправки участников использует git request-pull или механизмы запроса на включение (например, используемые на GitHub (www.github.com) для уведомления вышестоящей стороны о вашем вкладе.

Integrator

Достаточно центральный человек, действующий как интегратор в групповом проекте, получает изменения, внесенные другими, просматривает и интегрирует их, а также публикует результат для использования другими, используя эти команды в дополнение к тем, которые необходимы участникам.

Этот раздел также может использоваться теми, кто отвечает на git request-pull или запрос на включение на GitHub (www.github.com), чтобы интегрировать работу других в свою историю. Ответственный за репозиторий будет действовать как участник, так и интегратор.

  • git-am[1] для применения исправлений, полученных по электронной почте от ваших участников.

  • git-pull[1] для слияния с вашими доверенными помощниками.

  • git-format-patch[1] для подготовки и отправки предложенной альтернативы участникам.

  • git-revert[1] для отмены неудачных коммитов.

  • git-push[1] , чтобы опубликовать передовую информацию.

Examples

Типичный день интегратора Git .
$ git status (1)
$ git branch --no-merged master (2)
$ mailx (3)
& s 2 3 4 5 ./+to-apply
& s 7 8 ./+hold-linus
& q
$ git switch -c topic/one master
$ git am -3 -i -s ./+to-apply (4)
$ compile/test
$ git switch -c hold/linus && git am -3 -i -s ./+hold-linus (5)
$ git switch topic/one && git rebase master (6)
$ git switch -C seen next (7)
$ git merge topic/one topic/two && git merge hold/linus (8)
$ git switch maint
$ git cherry-pick master~4 (9)
$ compile/test
$ git tag -s -m "GIT 0.99.9x" v0.99.9x (10)
$ git fetch ko && for branch in master maint next seen (11)
    do
        git show-branch ko/$branch $branch (12)
    done
$ git push --follow-tags ko (13)

  1. посмотрите, чем вы занимались, если что.

  2. посмотрите, какие ветки еще не объединены в master . Аналогично для любых других ветвей интеграции, например maint , next и seen .

  3. читать письма, сохранять те, которые применимы, и сохранять другие, которые не совсем готовы (доступны другие программы для чтения почты).

  4. применяйте их в интерактивном режиме во время подписания.

  5. при необходимости создайте ветку темы и подайте заявку, опять же с подписями.

  6. перебазировать внутреннюю ветку темы, которая не была объединена с основной или представлена ​​как часть стабильной ветки.

  7. перезагружайте seen каждый раз со следующего.

  8. и групповые ветки тем, которые все еще готовятся.

  9. перенести критическое исправление.

  10. создать подписанный тег.

  11. убедитесь, что мастер не был случайно перемотан дальше того, что уже вытолкнуто.

  12. В выводе git show-branch master должно иметь все, что есть у ko/master , а next должно иметь все, что есть у ko/next , и т. д.

  13. вытолкнуть передовую крайность вместе с новыми тегами, указывающими на выдвинутую историю.

В этом примере сокращение ko указывает на репозиторий сопровождающего Git на kernel.org и выглядит следующим образом:

(in .git/config)
[remote "ko"]
        url = kernel.org:/pub/scm/git/git.git
        fetch = refs/heads/*:refs/remotes/ko/*
        push = refs/heads/master
        push = refs/heads/next
        push = +refs/heads/seen
        push = refs/heads/maint

Repository administration

Администратор репозитория использует следующие инструменты для настройки и поддержания доступа к репозиторию со стороны разработчиков.

  • git-daemon[1] , чтобы разрешить анонимную загрузку из репозитория.

  • git-shell[1] можно использовать как restricted login shell для пользователей общего центрального хранилища.

  • git-http-backend[1] обеспечивает реализацию Git-over-HTTP на стороне сервера ( "Smart http" ), позволяющую выполнять как выборку, так и отправку данных.

  • gitweb[1] предоставляет веб-интерфейс для репозиториев Git , который можно настроить с помощью сценария git-instaweb[1] .

update hook howto имеет хороший пример управления общим центральным репозиторием.

Кроме того, существует ряд других широко распространенных решений для хостинга, просмотра и просмотра, таких как:

  • gitolite, проверка кода Gerrit, cgit и другие.

Examples

Мы предполагаем следующее в /etc/services
$ grep 9418 /etc/services
git                9418/tcp                # Система контроля версий Git
Запустите git-daemon для обслуживания /pub/scm из inetd.
$ grep git /etc/inetd.conf
git        stream        tcp        nowait        nobody \
  /usr/bin/git-daemon git-daemon --inetd --export-all /pub/scm

Фактическая строка конфигурации должна находиться на одной строке.

Запустите git-daemon для обслуживания /pub/scm из xinetd.
$ cat /etc/xinetd.d/git-daemon
# по умолчанию: выключено
# описание: Сервер Git предоставляет доступ к репозиториям Git.
service git
{
        disable = no
        type            = UNLISTED
        port            = 9418
        socket_type     = stream
        wait            = no
        user            = nobody
        server          = /usr/bin/git-daemon
        server_args     = --inetd --export-all --base-path=/pub/scm
        log_on_failure  += USERID
}

Проверьте документацию и настройки xinetd(8), это из системы Fedora. Другие могут быть другими.

Предоставьте доступ только для push/pull разработчикам, использующим git-over-ssh.
