gitcvs-migration

Name

gitcvs-migration — Git для пользователей CVS

Synopsis

git cvsimport *

Description

Git отличается от CVS тем, что каждое рабочее дерево содержит репозиторий с полной копией истории проекта, и ни один репозиторий по своей сути не является более важным, чем любой другой. Однако вы можете эмулировать модель CVS, назначив один общий репозиторий, с которым люди смогут синхронизироваться; в этом документе объясняется, как это сделать.

Требуется некоторое базовое знакомство с Git . Пройдя через gittutorial[7] и gitglossary[7] , должно быть достаточно.

Разработка на основе общего репозитория

Предположим, что общий репозиторий настроен в /pub/repo.git на хосте foo.com. Затем как отдельный коммиттер вы можете клонировать общий репозиторий через ssh с помощью:

$ git clone foo.com:/pub/repo.git/ my-project
$ cd my-project

и взломать. Эквивалент cvs update :

$ git pull origin

который объединяется с любой работой, которую другие могли бы выполнить после операции клонирования. Если в вашем рабочем дереве есть незафиксированные изменения, сначала зафиксируйте их, прежде чем запускать запрос git .

Note

Команда pull знает, откуда получать обновления, благодаря определенным переменным конфигурации, заданным первой командой git clone ; подробности см. в git config -l и на справочной странице git-config[1] .

Вы можете обновить общий репозиторий своими изменениями, сначала зафиксировав изменения, а затем используя команду git push :

$ git push origin master

до "push" , которые фиксируются в общем репозитории. Если кто-то другой недавно обновил репозиторий, git push , как и cvs commit , выдаст сообщение об ошибке, и в этом случае вам придется извлечь любые изменения, прежде чем повторять попытку отправки.

В приведенной выше команде git push мы указываем имя удаленной ветки для обновления ( master ). Если мы оставим это без внимания, git push попытается обновить все ветки в удаленном репозитории, имеющие то же имя, что и ветка в локальном репозитории. Итак, последний push можно сделать с помощью любого из:

$ git push origin
$ git push foo.com:/pub/project.git/

до тех пор, пока общий репозиторий не имеет никаких веток, кроме master .

Настройка общего репозитория

Мы предполагаем, что вы уже создали репозиторий Git для своего проекта, возможно, созданный с нуля или из архива (см. gittutorial[7] ), или импортированный из уже существующего репозитория CVS (см. следующий раздел).

Предположим, что ваш существующий репозиторий находится в /home/alice/myproject. Создайте новый репозиторий "bare" (репозиторий без рабочего дерева) и загрузите в него свой проект:

$ mkdir /pub/my-repo.git
$ cd /pub/my-repo.git
$ git --bare init --shared
$ git --bare fetch /home/alice/myproject master:master

Затем предоставьте каждому члену команды доступ на чтение/запись к этому репозиторию. Один из простых способов сделать это — предоставить всем членам команды ssh-доступ к машине, на которой расположен репозиторий. Если вы не хотите предоставлять им полную оболочку на машине, существует ограниченная оболочка, которая позволяет пользователям выполнять только операции Git push и pull; см. git-shell[1] .

Поместите всех коммиттеров в одну группу и сделайте репозиторий доступным для записи этой группе:

$ chgrp -R $group /pub/my-repo.git

Убедитесь, что коммиттеры имеют umask не более 027, чтобы в создаваемых ими каталогах были доступны запись и поиск для других членов группы.

Импорт архива cvs

Note
 В этих инструкциях используется сценарий git-cvsimport , который поставляется с git, но другие импортеры могут обеспечить лучшие результаты. Другие варианты см. в примечании к git-cvsimport[1] .

Сначала установите версию 2.1 или выше cvsps от https://github.com/andreyvit/cvsps и убедитесь, что она есть на вашем пути. Затем перейдите в извлеченный рабочий каталог CVS интересующего вас проекта и запустите git-cvsimport[1] :

$ git cvsimport -C <destination> <module>

При этом архив Git указанного модуля CVS помещается в каталог <destination>, который будет создан при необходимости.

При импорте из CVS проверяется каждая редакция каждого файла. Сообщается, что cvsimport может выполнять в среднем около двадцати изменений в секунду, поэтому для проекта среднего размера это не должно занимать более пары минут. Более крупные проекты или удаленные репозитории могут занять больше времени.

Основная магистраль хранится в ветке Git с именем origin , а дополнительные ветви CVS хранятся в ветках Git с такими же именами. Самая последняя версия основного канала также остается извлеченной в ветке master , поэтому вы можете сразу же приступить к добавлению собственных изменений.

Импорт является инкрементным, поэтому, если вы вызовете его снова в следующем месяце, он получит все обновления CVS, которые были сделаны за это время. Чтобы это работало, вы не должны изменять импортированные ветки; вместо этого создайте новые ветки для собственных изменений и при необходимости объедините импортированные ветки.

Если вам нужен общий репозиторий, вам нужно будет создать чистый клон импортированного каталога, как описано выше. Затем рассматривайте импортированный каталог как еще один клон разработки для целей объединения добавочного импорта.

Расширенное управление общим репозиторием

Git позволяет вам указать сценарии под названием "hooks" , которые будут запускаться в определенных точках. Вы можете использовать их, например, для отправки всех коммитов в общий репозиторий в список рассылки. См. githooks[5] .

Вы можете обеспечить более детальные разрешения, используя перехватчики обновлений. См. Controlling access to branches using update hooks .

Предоставление доступа cvs к репозиторию git

Также возможно предоставить true CVS доступ к репозиторию Git , чтобы разработчики могли по-прежнему использовать CVS; подробности см. в git-cvsserver[1] .

Альтернативные модели развития

Пользователи CVS привыкли предоставлять группе разработчиков доступ к общему репозиторию. Как мы видели, это также возможно с Git . Однако распределенный характер Git допускает использование других моделей разработки, и вы, возможно, захотите сначала подумать, может ли одна из них лучше подойти для вашего проекта.

Например, вы можете выбрать одного человека для обслуживания основного репозитория public проекта. Затем другие разработчики клонируют этот репозиторий и каждый работает над своим собственным клоном. Когда у них есть серия изменений, которые их устраивают, они просят сопровождающего извлечь их из ветки, содержащей эти изменения. Сопровождающий просматривает внесенные изменения и переносит их в основной репозиторий, из которого другие разработчики извлекают их по мере необходимости для координации. Ядро Linux и другие проекты используют варианты этой модели.

В небольшой группе разработчики могут просто извлекать изменения из репозиториев друг друга без необходимости участия центрального сопровождающего.
