gitnamespaces

Name

gitnamespaces — пространства имен Git

Synopsis

GIT_NAMESPACE=<namespace> git upload-pack
GIT_NAMESPACE=<namespace> git receive-pack

Description

Git поддерживает разделение ссылок одного репозитория на несколько пространств имен, каждое из которых имеет свои собственные ветки, теги и HEAD. Git может предоставлять каждое пространство имен как независимый репозиторий для извлечения и отправки, при этом совместно используя хранилище объектов и предоставляя все ссылки для таких операций, как git-gc[1] .

Хранение нескольких репозиториев как пространств имен одного репозитория позволяет избежать хранения дубликатов одних и тех же объектов, например, при хранении нескольких ветвей одного и того же источника. Механизм альтернатив обеспечивает аналогичную поддержку для предотвращения дублирования, но альтернативы не предотвращают дублирование между новыми объектами, добавляемыми в репозитории без текущего обслуживания, в отличие от пространств имен.

Чтобы указать пространство имен, задайте для переменной среды GIT_NAMESPACE это пространство имен. Для каждого пространства имен ссылок Git сохраняет соответствующие ссылки в каталоге под refs/namespaces/ . Например, GIT_NAMESPACE=foo будет хранить ссылки в refs/namespaces/foo/ . Вы также можете указать пространства имен с помощью опции --namespace для git[1] .

Обратите внимание, что пространства имен, включающие / , расширяются до иерархии пространств имен; например, GIT_NAMESPACE=foo/bar будет хранить ссылки в refs/namespaces/foo/refs/namespaces/bar/ . Это приводит к тому, что пути в GIT_NAMESPACE ведут себя иерархически, так что клонирование с помощью GIT_NAMESPACE=foo/bar дает тот же результат, что и клонирование с помощью GIT_NAMESPACE=foo и клонирование из этого репозитория с помощью GIT_NAMESPACE=bar . Это также позволяет избежать двусмысленности со странными путями пространства имен, такими как foo/refs/heads/ , которые в противном случае могли бы вызвать конфликты каталогов и файлов внутри каталога refs .

git-upload-pack[1] и git-receive-pack[1] переписывают имена ссылок, как указано в GIT_NAMESPACE . git-upload-pack и git-receive-pack будут игнорировать все ссылки за пределами указанного пространства имен.

Интеллектуальный HTTP-сервер git-http-backend[1] передаст GIT_NAMESPACE серверным программам; см. git-http-backend[1] для примера конфигурации, чтобы представить пространства имен репозитория как репозитории.

Для простого локального теста вы можете использовать git-remote-ext[1] :

git clone ext::'git --namespace=foo %s /tmp/prefixed.git'

Security

Протоколы выборки и отправки не предназначены для предотвращения кражи одной стороной данных из другого репозитория, которые не предназначены для совместного использования. Если у вас есть данные private , которые необходимо защитить от злонамеренного узла, лучший вариант — сохранить их в другом репозитории. Это касается как клиентов, так и серверов. В частности, пространства имен на сервере неэффективны для управления доступом для чтения; вам следует предоставлять доступ на чтение к пространству имен только тем клиентам, которым вы доверяете доступ на чтение ко всему репозиторию.

Известные векторы атак следующие:

  1. Жертва отправляет строки "have" , объявляющие идентификаторы имеющихся у нее объектов, которые явно не предназначены для совместного использования, но могут использоваться для оптимизации передачи, если они также есть у узла. Злоумышленник выбирает идентификатор объекта X для кражи и отправляет ссылку на X, но не обязан отправлять содержимое X, поскольку оно уже есть у жертвы. Теперь жертва считает, что у злоумышленника есть X, и позже отправляет содержимое X обратно злоумышленнику. (Эту атаку наиболее просто выполнить клиент на сервере, создав ссылку на X в пространстве имен, к которому клиент имеет доступ, а затем извлекая ее. Наиболее вероятный способ, которым сервер может выполнить эту атаку на клиенте, — это "merge" . X в ветку public и надеяться, что пользователь проделает дополнительную работу над этой веткой и отправит ее обратно на сервер, не заметив слияния.)

  2. Как и в № 1, злоумышленник выбирает объект с идентификатором X для кражи. Жертва отправляет объект Y, который уже есть у злоумышленника, и злоумышленник ложно утверждает, что у него есть X, а не Y, поэтому жертва отправляет Y как дельту против X. Дельта показывает злоумышленнику области X, похожие на Y.
