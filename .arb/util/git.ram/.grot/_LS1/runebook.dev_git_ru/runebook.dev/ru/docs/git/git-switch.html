git-switch

Name

git-switch — переключить ветку

Synopsis

git switch [<options>] [--no-guess] <branch>
git switch [<options>] --detach [<start-point>]
git switch [<options>] (-c|-C) <new-branch> [<start-point>]
git switch [<options>] --orphan <new-branch>

Description

Переключиться на указанную ветку. Рабочее дерево и индекс обновляются в соответствии с веткой. Все новые коммиты будут добавляться в начало этой ветки.

При желании новая ветка может быть создана либо с помощью -c , -C , автоматически из удаленной ветки с тем же именем (см. --guess ), либо отделить рабочее дерево от любой ветки с помощью --detach вместе с переключением.

Переключение ветвей не требует чистого индекса и рабочего дерева (т.е. никаких отличий по сравнению с HEAD ). Однако операция прерывается, если она приводит к потере локальных изменений, если иное не указано в --discard-changes или --merge .

Options

<branch>

Ветка, на которую нужно переключиться.

<new-branch>

Название новой ветки.

<start-point>

Отправная точка для новой ветки. Указание <start-point> позволяет вам создать ветвь, основанную на какой-либо другой точке истории, отличной от текущей точки HEAD. (Или, в случае с --detach , позволяет осмотреть и отсоединиться от какой-либо другой точки.)

Вы можете использовать синтаксис @{-N} для ссылки на N-ю последнюю ветвь/фиксацию, на которую было переключено использование операции "git switch" или "git checkout" . Вы также можете указать - , который является синонимом @{-1} . Это часто используется для быстрого переключения между двумя ветвями или для отмены ошибочного переключения ветвей.

В частном случае вы можете использовать A...B в качестве ярлыка для базы слияния A и B , если существует ровно одна база слияния. Вы можете исключить не более одного из A и B , в этом случае по умолчанию используется HEAD .

-c <new-branch>
--create <new-branch>

Перед переключением на ветку создайте новую ветку с именем <new-branch> , начиная с <start-point> . Это удобный ярлык для:

$ git branch <new-branch>
$ git switch <new-branch>
-C <new-branch>
--force-create <new-branch>

Аналогичен --create , за исключением того, что если <new-branch> уже существует, он будет сброшен до <start-point> . Это удобный ярлык для:

$ git branch -f <new-branch>
$ git switch <new-branch>
-d
--detach

Переключитесь на фиксацию для проверки и отбрасываемых экспериментов. Подробности см. в разделе "DETACHED HEAD" в git-checkout[1] .

--guess
--no-guess

Если <branch> не найден, но существует ветвь отслеживания ровно на одном удаленном компьютере (назовем его <remote> ) с совпадающим именем, рассматривайте его как эквивалент

$ git switch -c <branch> --track <remote>/<branch>

Если ветвь существует на нескольких удаленных устройствах и один из них назван конфигурационной переменной checkout.defaultRemote , мы будем использовать ее в целях устранения неоднозначности, даже если <branch> не уникален для всех удаленных устройств. Установите его, например, на checkout.defaultRemote=origin , чтобы всегда получать оттуда удаленные ветки, если <branch> неоднозначен, но существует на удаленном origin . См. также checkout.defaultRemote в git-config[1] .

--guess — поведение по умолчанию. Используйте --no-guess , чтобы отключить его.

Поведение по умолчанию можно установить с помощью переменной конфигурации checkout.guess .

-f
--force

Псевдоним --discard-changes .

--discard-changes

Продолжайте, даже если индекс или рабочее дерево отличаются от HEAD . И индекс, и рабочее дерево восстанавливаются в соответствии с целью переключения. Если указан --recurse-submodules , содержимое подмодуля также восстанавливается в соответствии с целью переключения. Это используется для отмены локальных изменений.

-m
--merge

Если у вас есть локальные изменения в одном или нескольких файлах, которые различаются в текущей ветке и ветке, на которую вы переключаетесь, команда откажется переключать ветки, чтобы сохранить ваши изменения в контексте. Однако при использовании этой опции трехстороннее слияние текущей ветки, содержимого вашего рабочего дерева и новой ветки выполнено, и вы окажетесь в новой ветке.

Когда происходит конфликт слияния, записи индекса для конфликтующих путей остаются несвязанными, и вам необходимо разрешить конфликты и пометить разрешенные пути с помощью git add (или git rm , если слияние должно привести к удалению пути).

--conflict=<style>

То же, что и опция --merge выше, но меняет способ представления конфликтующих фрагментов, переопределяя переменную конфигурации merge.conflictStyle . Возможные значения: "merge" (по умолчанию), "diff3" и "zdiff3" .

-q
--quiet

Тихо, подавляйте сообщения обратной связи.

--progress
--no-progress

Статус выполнения по умолчанию сообщается в стандартном потоке ошибок, когда он подключен к терминалу, если не указан --quiet . Этот флаг включает отчет о ходе выполнения, даже если он не подключен к терминалу, независимо от --quiet .

-t
--track [direct|inherit]

При создании новой ветки настройте конфигурацию "upstream" . Подразумевается -c . Подробности см. в --track и git-branch[1] .

Если опция -c не указана, имя новой ветки будет получено из ветки удаленного отслеживания путем просмотра локальной части спецификации ссылки, настроенной для соответствующего удаленного устройства, а затем удаления начальной части до "*" . Это подскажет нам использовать hack в качестве локальной ветви при ответвлении от origin/hack (или remotes/origin/hack , или даже refs/remotes/origin/hack ). Если данное имя не имеет косой черты или в результате приведенного выше угадывания имя остается пустым, угадывание прерывается. В таком случае вы можете явно указать имя с помощью -c .

--no-track

Не настраивайте конфигурацию "upstream" , даже если переменная конфигурации branch.autoSetupMerge имеет значение true.

--orphan <new-branch>

Создайте новую нерожденную ветку с именем <new-branch> . Все отслеживаемые файлы удаляются.

--ignore-other-worktrees

git switch отказывается, когда требуемая ссылка уже извлечена другим рабочим деревом. Эта опция в любом случае заставляет его проверять ссылку. Другими словами, ссылка может храниться более чем в одном рабочем дереве.

--recurse-submodules
--no-recurse-submodules

Использование --recurse-submodules обновит содержимое всех активных подмодулей в соответствии с коммитом, записанным в суперпроекте. Если ничего (или --no-recurse-submodules ) не используется, рабочие деревья подмодулей обновляться не будут. Как и в случае с git-submodule[1] , это позволит отключить HEAD от субмодулей.

Examples

Следующая команда переключается на ветку "master" :

$ git switch master

После работы не в той ветке переключение на правильную ветку будет осуществляться с помощью:

$ git switch mytopic

Однако ваша ветка "wrong" и правильная ветка "mytopic" могут отличаться в файлах, которые вы изменили локально, и в этом случае вышеуказанный переключатель выйдет из строя следующим образом:

$ git switch mytopic
error: You have local changes to 'frotz'; not switching branches.

Вы можете указать флаг -m команде, которая попытается выполнить трехстороннее слияние:

$ git switch -m mytopic
Auto-merging frotz

После этого трехстороннего слияния локальные изменения not регистрируются в вашем индексном файле, поэтому git diff покажет вам, какие изменения вы внесли с момента начала новой ветки.

Чтобы вернуться к предыдущей ветке до того, как мы переключились на mytopic (т.е. ветку "master" ):

$ git switch -

Вы можете вырастить новую ветку из любого коммита. Например, переключимся на "HEAD~3" и создадим ветку "fixup" :

$ git switch -c fixup HEAD~3
Switched to a new branch 'fixup'

Если вы хотите запустить новую ветку из удаленной ветки с тем же именем:

$ git switch new-topic
Branch 'new-topic' set up to track remote branch 'new-topic' from 'origin'
Switched to a new branch 'new-topic'

Чтобы получить коммит HEAD~3 для временной проверки или эксперимента без создания новой ветки:

$ git switch --detach HEAD~3
HEAD is now at 9fc9555312 Merge branch 'cc/shared-index-permbits'

Если окажется, что все, что вы сделали, стоит сохранить, вы всегда можете создать для него новое имя (не переключаясь):

$ git switch -c good-surprises

Configuration

Все, что находится ниже этой строки в этом разделе, выборочно включено из документации git-config[1] . Содержание такое же, как и найденное там:

checkout.defaultRemote

Если вы используете git checkout <something> или git switch <something> и у вас есть только один пульт дистанционного управления, он может неявно вернуться к проверке и отслеживанию, например origin/<something> . Это перестанет работать, как только у вас будет более одного пульта с кодом <something> . Этот параметр позволяет указать имя предпочтительного пульта, который всегда должен выигрывать, когда дело доходит до устранения неоднозначности. Типичный вариант использования — установить для этого параметра значение origin .

В настоящее время это используется git-switch[1] и git-checkout[1] , когда git checkout <something> или git switch <something> проверяет ветку <something> на другом удаленном устройстве, и git-worktree[1] , когда git worktree add ссылается на удаленную ветку. Этот параметр может использоваться для других команд или функций, подобных оформлению заказа, в будущем.

checkout.guess

Предоставляет значение по умолчанию для параметра --guess или --no-guess в git checkout и git switch . См. git-switch[1] и git-checkout[1] .

checkout.workers

Количество параллельных рабочих процессов, используемых при обновлении рабочего дерева. По умолчанию установлено одно, т.е. последовательное выполнение. Если установлено значение меньше единицы, Git будет использовать столько рабочих процессов, сколько доступно логических ядер. Этот параметр и checkout.thresholdForParallelism влияют на все команды, выполняющие проверку. Например, проверка, клонирование, сброс, разреженная проверка и т. д.

Примечание. Параллельная проверка обычно обеспечивает более высокую производительность для репозиториев, расположенных на твердотельных накопителях или через NFS. Для репозиториев на вращающихся дисках и/или машинах с небольшим количеством ядер последовательная проверка по умолчанию часто работает лучше. Размер и уровень сжатия репозитория также могут влиять на производительность параллельной версии.

checkout.thresholdForParallelism

При параллельном извлечении небольшого количества файлов затраты на создание подпроцессов и межпроцессное взаимодействие могут перевесить выгоды от распараллеливания. Этот параметр позволяет определить минимальное количество файлов, для которых следует попытаться выполнить параллельное извлечение. По умолчанию — 100.
